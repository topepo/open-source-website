<h3>Package Manager authentication</h3>

<p><a data-t="878">So, hi.</a> <a data-t="878">I wanted to talk about one of the newest features that we just released some more support</a> <a data-t="884">for in Package Manager, one that we've been excited about because it's been long overdue</a> <a data-t="895">and one of the most popular requests from many of our customers, is support for authentication</a> <a data-t="895">in Package Manager.</a></p>

<p><a data-t="908">So, there's several reasons why authentication can be useful in any product,</a> <a data-t="908">especially Package Manager.</a> <a data-t="914">One, for being able to restrict access to repositories that may not</a> <a data-t="914">be wanted to use in a public context, even within your own company, to specific teams or groups.</a> <a data-t="919">The ability to grant more granular access to individual trusted users, your power users who</a> <a data-t="925">may want to be able to grant individual access to publish packages directly to Package Manager,</a> <a data-t="931">and even just for delegating other administrative tasks rather than being able to be forcing that</a> <a data-t="936">all to be done kind of from an IT administrator perspective as well.</a> <a data-t="942">And, of course, being able</a> <a data-t="942">to securely integrate with other tools outside of Package Manager in a way that you really shouldn't</a> <a data-t="947">do in a more public context.</a></p>

<p><a data-t="953">So, we've introduced three types of authentication now with our latest</a> <a data-t="953">release.</a> <a data-t="962">One, what we started earlier this year around API tokens, then the latest single sign-on,</a> <a data-t="962">as well as some identity federation, and I'm going to talk just briefly through kind of the</a> <a data-t="966">around all three of these areas and what value they bring to the product here.</a></p>

<p><a data-t="971">Starting with token authentication, so this is where an administrator manually creates an</a> <a data-t="976">authentication token, generates it from Package Manager, shares it with the user to then use to</a> <a data-t="980">authenticate.</a> <a data-t="986">Really useful for shared or server level authentication.</a> <a data-t="986">For example, if you want to</a> <a data-t="986">just authenticate your Workbench or Connect server as a whole to Package Manager rather than at the</a> <a data-t="992">per user level with long live keys or other services outside of that there.</a> <a data-t="997">The nice part</a> <a data-t="997">about token authentication is that it's really compatible with pretty much all existing tooling.</a> <a data-t="1001">You can just attach it to the basic credentials that are used to access Package Manager.</a> <a data-t="1006">So, existing R and Python client tooling makes it really easy to kind of connect that in there.</a></p>

<p><a data-t="1012">But, the limitations certainly there of having to manage all those keys manually can definitely be</a> <a data-t="1017">a headache there.</a> <a data-t="1023">So, with the latest release, we just introduced single sign-on using OpenID Connect</a> <a data-t="1023">and this really allows you to log in with a corporate identity provider that supports</a> <a data-t="1028">OpenID Connect, such as Okta, Microsoft Enter ID, also formally known as Azure Active Directory.</a> <a data-t="1036">But really, I mean, these are two implementations that we've really focused on, but really any</a> <a data-t="1040">OpenID Connect compliant identity provider should work fairly well here.</a></p>

<p><a data-t="1046">Big advantage of that is you can actually manage your user and group assignment in your identity</a> <a data-t="1050">provider rather than having to manage them directly within, you know, court setting your users and</a> <a data-t="1054">groups within Package Manager itself, and then just map those groups to the various permission</a> <a data-t="1058">scopes you want to have in Package Manager.</a> <a data-t="1062">Big advantage on the user side, it just allows for seamless login support through, you know,</a> <a data-t="1066">logging in directly through the web UI in Package Manager, the command line interface for Package</a> <a data-t="1071">Manager, and package installation tools themselves just being able to use those standard OAuth-type</a> <a data-t="1077">workflows to log in.</a></p>

<p><a data-t="1080">And then finally, we have identity federation, which is kind of a useful way, especially by configuring</a> <a data-t="1067">Package Manager to accept tokens issued by other OpenID Connect providers.</a> <a data-t="1077">So this would be cases like, you know, running a non-interactive job by a CI-CD system, say you're</a> <a data-t="1077">using GitHub Actions to generate things and you want to actually use the credentials that GitHub</a> <a data-t="1081">provides to authenticate with Package Manager and trust those credentials rather than having to go</a> <a data-t="1085">through separate workflows to generate and, you know, obtain new credentials just to access</a> <a data-t="1089">Package Manager.</a></p>
