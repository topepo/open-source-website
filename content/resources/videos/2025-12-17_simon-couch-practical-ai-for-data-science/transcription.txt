<h3>Tool calling</h3>

<p><a data-t="793">The next capability that I want to talk about is something called tool calling.</a> <a data-t="799">And practically, this is a super helpful capability when you're trying to get LLMs to help you with certain tasks.</a></p>

<p><a data-t="808">So one thing that LLMs can't do is access real time information.</a> <a data-t="815">This just isn't how they work.</a> <a data-t="817">They have a knowledge cut off.</a> <a data-t="819">They're trained on a bunch of information.</a> <a data-t="822">Natively, they have no way to learn what the current date is.</a> <a data-t="827">So if I initialize this chat object and I say, what day is it today?</a> <a data-t="832">Cloud will respond to me and say, I don't know.</a></p>

<p><a data-t="836">If you're using a model like chat2pt.com or cloud.ai or whatever,</a> <a data-t="840">they'll often inject that information into the prompt so that it appears that they do know what the date is.</a> <a data-t="846">But that's like scaffolding on top of the model rather than the raw model itself.</a></p>

<p><a data-t="854">So what we can do is provide a tool to the model.</a> <a data-t="860">A tool is just a function and then some documentation on how to use the function.</a> <a data-t="865">So, for example, I could make this tool called today, which gets today's date.</a> <a data-t="871">And all it does when it's called is just calls the sys.date function.</a> <a data-t="877">And so we describe this function as get today's date to the model.</a> <a data-t="882">The function has no arguments.</a> <a data-t="884">So we just provide an empty list.</a></p>

<p><a data-t="886">And then we use yet another method on this chat object, which is the register tool method.</a> <a data-t="893">This tool function here is exported from the ellmer package,</a> <a data-t="897">and it just allows you to attach some metadata to a given function.</a> <a data-t="902">And once you've done that, you can register it with the chat.</a></p>

<p><a data-t="906">So then if I go back to that chat object and I ask the same question, I say, what day is it today?</a> <a data-t="913">The model can choose to return a message back to me in this structured way,</a> <a data-t="920">which ellmer will see and say, oh, this isn't a message that's intended for the user.</a> <a data-t="925">This is a message that I can process myself and immediately return a response back.</a> <a data-t="931">So the LLM decides to send me this special formatted message that says call this function today.</a></p>

<p><a data-t="939">And the function returns the date at the moment at which it was called.</a> <a data-t="944">So this is August 8th, 2025.</a> <a data-t="948">So then it's again the model's turn to do whatever it wants to do.</a> <a data-t="952">And in this case, it's just going to situate this response from the tool inside of its message back to you.</a> <a data-t="959">So it can say today is August 8th, 2025.</a> <a data-t="962">And it happens to know that it's Friday because LLMs are weird.</a></p>

<p><a data-t="969">This is a strange concept to wrap your head around at first.</a> <a data-t="972">So I just want to like actually diagram this out and make sure this is clear what's actually happening here.</a> <a data-t="979">So this is me last week.</a> <a data-t="982">And last week, I was having a conversation with this humanoid robot typing on a laptop slash desktop.</a> <a data-t="989">So last week, I send this message over to the humanoid robot.</a> <a data-t="994">And I say, what day is it today?</a></p>

<p><a data-t="997">So the only thing that the model can do is respond back with another HTTP request.</a> <a data-t="1005">Except that it can format that request in such a way that ellmer will see the response.</a> <a data-t="1012">And it will realize that the response shouldn't be shown to me.</a> <a data-t="1017">So I never see this message, please call today.</a> <a data-t="1020">ellmer sees it and it processes it automatically and returns the result back to the model.</a></p>

<p><a data-t="1026">So again, the humanoid robot sends this specially formatted message that says, please call today.</a> <a data-t="1033">The response is returned to the robot.</a> <a data-t="1036">And then the humanoid robot can situate that response in a response back to me.</a></p>

<p><a data-t="1043">So this is what we call a complete tool calling loop, where a model chooses to call a tool.</a> <a data-t="1052">It gets a response back, and then it chooses what to do next.</a> <a data-t="1056">So often, a model can call a tool.</a> <a data-t="1060">It can observe the result.</a> <a data-t="1062">And then once it observes the result, it can either respond back to me, as it did.</a> <a data-t="1070">Or it can just continue to call tools until it's accomplished the task that it would like to do.</a></p>

<h3>Coding agents</h3>

<p><a data-t="1085">And now we'll situate tool calling inside of a scaffold where we're using tool calls to write and evaluate code.</a></p>

<p><a data-t="1096">So LLMs, through their hazy recollection of the entire Internet, are pretty good at generating code.</a> <a data-t="1105">And if we give LLMs the ability to run code in a thoughtful way, we provide them with a scaffold that helps them assist us as best as they can.</a> <a data-t="1117">Then these models can be pretty helpful at generating data science code.</a></p>

<p><a data-t="1122">So again, this is a callback to the tool calling loop.</a> <a data-t="1127">When people say coding agent, which is a pretty popular term now, this can refer to agents like Cloud Code or Codex, and also agents built by Posit, like </a><a href="/software/positron/">Positron</a><a data-t="1127"> Assistant or Databot.</a> <a data-t="1142">This is LLMs calling tools in a loop.</a></p>

<p><a data-t="1148">And usually, when we say something is an agent, we mean that those tools give the model the ability to read and write state.</a> <a data-t="1157">So what do I mean?</a> <a data-t="1160">If I make a fresh chat object, and in that chat I say, delete the CSV files in my working directory.</a> <a data-t="1169">If the model were given the ability to run code, it could probably figure out how to do that pretty easily.</a></p>

<p><a data-t="1176">So, for example, this model has chosen to show me how to do this in Bash if it were to be given the ability to run code, or it can just tell me this is the code that you would use to make that happen.</a> <a data-t="1190">What this model is not actually doing is going and deleting the CSV files, because it doesn't have tools that give it the ability to read and write state.</a></p>

<p><a data-t="1201">And so in order to carry this out, the model needs to be able to actually find the files that it wants to delete in the first place, and then go and delete those files, which is writing state.</a></p>

<p><a data-t="1220">We can do this with two ellmer tools.</a> <a data-t="1220">The first one we will call ls, and that just calls the dir function in R, which will list the files inside of the current directory.</a> <a data-t="1229">So this tool will give the model the ability to read state.</a></p>

<p><a data-t="1235">Then we can provide a second tool called rm, and given a path, that path is just provided to the unlink function in R, and this tool will give the LM the ability to delete one or more files.</a></p>

<p><a data-t="1262">So using these two tools, first the model can read state to find the CSV files that it wants to delete, and then given this rm tool, it can actually go and delete those files.</a></p>

<p><a data-t="1280">So if we return to that fresh chat again and ask the same question, delete all the CSV files in the current directory,</a> <a data-t="1291">the model can choose to call the ls tool, which will return a bunch of information about the objects that are inside of that directory.</a> <a data-t="1303">The model can identify which of those are CSVs and call another tool.</a></p>

<p><a data-t="1310">So this is an example of that full tool calling loop, where the model calls a tool, it gets a response back, and before it says something and yields control back to the user, it decides to call another tool before it does so.</a> <a data-t="1330">So it will call this remove tool, and it will provide these two paths to the CSV files that it found, and then the tool will return.</a></p>

<p><a data-t="1341">So in this case, because all the rm tool is doing is writing state, it just says, okay, I did that, done, which is what this true represents.</a> <a data-t="1356">And only at this point will the model complete the tool calling loop and yield back to the user saying, done, I deleted those files.</a></p>
