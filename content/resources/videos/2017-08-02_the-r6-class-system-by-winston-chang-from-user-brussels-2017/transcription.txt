<h3>Functional vs. encapsulated OOP</h3>

<p><a data-t="99">So, I'm going to start off with talking about R's built-in object-oriented programming systems.</a> <a data-t="104">There's a classification that John Chambers uses in his book, Extending R.</a> <a data-t="109">Functional object-oriented programming systems and encapsulated object-oriented programming systems.</a> <a data-t="115">So, for the functional object-oriented programming, that's exemplified by the S3 and S4 systems in R.</a> <a data-t="122">And for encapsulated object-oriented programming, that's exemplified by reference classes in R.</a> <a data-t="130">And these are the ones that are built into R.</a></p>

<p><a data-t="132">So, the differences between them include this.</a> <a data-t="135">For functional object-oriented programming, the objects contain data,</a> <a data-t="140">but class methods that operate on those objects, they're separate.</a> <a data-t="145">The methods themselves are separate from the objects, and the objects are not mutable.</a> <a data-t="149">So, whenever you have a function that operates on an object, and if it modifies the object,</a> <a data-t="155">what actually happens is you get a copy of the object, and then you can choose to store that modified copy or not.</a></p>

<p><a data-t="163">For encapsulated object-oriented programming, the objects contain both the data and the methods.</a> <a data-t="169">The objects are mutable.</a> <a data-t="169">They can change when you call a method or if you modify things in it.</a> <a data-t="175">And it's also known as classical object-oriented programming.</a> <a data-t="178">So, this is the type of system you'd see with Java, C++, or Python.</a> <a data-t="184">This is much more common in the wider programming world.</a></p>

<p><a data-t="189">Okay, so for using functional object-oriented programming, you might do something like,</a> <a data-t="194">if you've got your object X, you'd call summary of X, and there might be a method for your particular class that gives you a summary.</a> <a data-t="202">If you want to modify X, you'd call some function, modify, here, modifyX.</a> <a data-t="208">As I said, that doesn't modify X in place.</a> <a data-t="208">It returns a new object, which you would have to save back in X.</a></p>

<p><a data-t="216">Now, for encapsulated object-oriented programming, the methods are part of the object.</a> <a data-t="221">The dollar sign is the subset operator, so you'd say X$summary.</a> <a data-t="225">You might have a method that modifies the object to X$modify.</a> <a data-t="228">That could change X in place.</a> <a data-t="228">You don't have to save that back in X.</a></p>

<p><a data-t="233">If you pass X to a function, like I have this object called trim, it may change X, or it might not change X.</a> <a data-t="240">So, from the outside, you can't tell.</a> <a data-t="243">Because of this, it can be easier to reason about functional object-oriented programming,</a> <a data-t="250">where every time you want to modify an object, you always have an assignment operator.</a> <a data-t="256">Whereas with encapsulated, you don't know if it's going to change.</a> <a data-t="259">But encapsulated object-oriented programming is useful for representing data that evolves over time.</a></p>

<blockquote><a data-t="259">But encapsulated object-oriented programming is useful for representing data that evolves over time.</a></blockquote>

<p><a data-t="264">Some real-world examples of this, where R6 is used, are </a><a href="/software/shiny-r/">Shiny</a><a data-t="264">.</a> <a data-t="272">Their Shiny session is represented with an R6 object.</a> <a data-t="275">It represents the state of the application as the user interacts with it.</a> <a data-t="280">In dplyr, R6 objects are used to represent database connections.</a> <a data-t="285">In a relatively new package called </a><a href="/software/processx/">ProcessX</a><a data-t="285"> by Gabor, R6 objects are used to represent system processes.</a> <a data-t="294">In this case, these are all things that have connections to things that are outside of R.</a> <a data-t="298">This is one very common use case for encapsulated object-oriented programming.</a></p>
