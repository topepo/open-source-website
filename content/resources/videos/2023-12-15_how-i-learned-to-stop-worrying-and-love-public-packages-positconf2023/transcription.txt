<h3>Security risks: package quality</h3>

<p><a data-t="214">Let's jump into security risks, because that's the interesting</a> <a data-t="214">part.</a> <a data-t="220">I grew up with some security risks.</a> <a data-t="220">This is one of the first initial things that</a> <a data-t="220">a lot of people care about with packages is, how do I assess package quality?</a> <a data-t="227">Or really,</a> <a data-t="227">does this package that I'm seeing on the public repository do what it's supposed to do and</a> <a data-t="232">do it accurately and give me the right results?</a> <a data-t="237">It's a hard question to ask, because we didn't</a> <a data-t="237">write the package.</a> <a data-t="241">Often we're trying to use packages that we may not be domain experts</a> <a data-t="241">in ourselves.</a> <a data-t="244">That's why we're going to get a package that someone else has written to</a> <a data-t="244">do it.</a></p>

<h3>Security risks: vulnerabilities</h3>
