<h3>What is a medical device?</h3>

<p><a data-t="201">Now to get back to the medical device part, what is a medical device?</a> <a data-t="209">Well, we all know that this and this and this is a medical device, but in this talk, I'm going to focus on what is software as a medical device.</a> <a data-t="220">And more specifically, I'm going to talk about clinical decision support tools.</a></p>

<p><a data-t="228">If you decide to build a medical device, most likely you will go through the following stages.</a> <a data-t="234">We're in the pre-development stage.</a> <a data-t="235">You basically decide which class of medical device you're going to be, what are the claims you are making.</a> <a data-t="240">And this is also the stage where you set up a quality system.</a> <a data-t="247">And then in the next stage, you'll begin development of your products under design controls and the quality system that you developed.</a> <a data-t="262">And then in later stages, you'll apply the verification framework that you set up.</a> <a data-t="270">And then before you file with the FDA, you do some validation with actual users.</a> <a data-t="280">And then you should be good to release this into the wild.</a></p>

<p><a data-t="279">I would say the most important thing in this whole process is probably the risk analysis that you will do.</a> <a data-t="289">You'll do that really early on.</a> <a data-t="289">And in this case, you scope out what are all the risks associated with the medical device.</a> <a data-t="296">And once you have that, you can actually develop a document called the software specifications.</a> <a data-t="304">And that is just a long list of things like the software shall be able to do this and that basically 100 times or more.</a> <a data-t="313">So all these requirements, what you can do then is map these to actual verification steps.</a></p>

<h3>Computations and the PKPDSim package</h3>

<p><a data-t="539">So then we can go into some of the choices that we made and what we developed.</a> <a data-t="553">And the first thing here is the computations.</a> <a data-t="553">And like I said in the introduction, we use pharmacology models.</a> <a data-t="567">And those are often defined as differential equations.</a> <a data-t="547">And when we started out five years ago, there was not really a package or anything in R to actually handle that.</a> <a data-t="555">So that's why we wrote a package called PKPDSim.</a> <a data-t="565">And the main thing that it's doing, it can do a prediction of the concentration of a specific drug in the body over time.</a> <a data-t="572">That's basically the core of our platform.</a> <a data-t="580">Again, this is also open source.</a></p>

<p><a data-t="594">And the lesson that we learned here over time is that, well, at first, it took a lot of work to actually implement this ourself.</a> <a data-t="600">But over time, we realized that this actually has been very good for us because that really helped us in the regulatory process because we know the ins and outs of this package.</a> <a data-t="615">We know the weaknesses and the strengths.</a> <a data-t="615">So if the FDA ever comes to a door and asks us questions about this package, it will be very easy for us to provide them with answers.</a></p>

<h3>Architecture and API choices</h3>

<p><a data-t="698">So then a little bit about the architecture.</a> <a data-t="698">Well, we're speaking at an R conference and we're speaking about web applications.</a> <a data-t="709">So the elephant in the room that we have to address is </a><a href="/software/shiny-r/">Shiny</a><a data-t="709">.</a> <a data-t="722">And of course, I think it's fine to write a medical device in Shiny.</a> <a data-t="737">There's absolutely no objection to that.</a> <a data-t="737">And if you're worried about the scalability of that, you can always split off the API so that it scales independently from the web application.</a> <a data-t="747">So that's completely fine.</a> <a data-t="752">But really early on, we made the decision to not go with Shiny.</a> <a data-t="752">And that was mostly because we thought in the end, the application would be too complex.</a> <a data-t="762">It would be too hard to develop in Shiny.</a> <a data-t="772">That's why we went with a sort of more default JavaScript stack.</a></p>

<p><a data-t="781">If we then go into the API itself, which we of course left in R, what you have to keep in mind is the scale and the load requirements for the API.</a> <a data-t="790">Well, we don't have the load requirements that were presented before, like a million users a day or a million API calls a day.</a> <a data-t="797">We have a little bit less.</a> <a data-t="797">We are growing, but we have about a few hundred API calls an hour.</a></p>
