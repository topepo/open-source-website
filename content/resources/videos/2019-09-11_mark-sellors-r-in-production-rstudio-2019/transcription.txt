<h3>Defining production</h3>

<p><a data-t="106">So much like Joe, who kind of, you know, stealing my thunder this morning,</a> <a data-t="114">we need to start out with a working definition of what production actually means.</a> <a data-t="121">And mine's kind of similar to Joe, there's a lot of overlap.</a></p>

<p><a data-t="126">But basically, when people say production,</a> <a data-t="131">they tend to sort of have this idea in their head of what production means.</a> <a data-t="136">And production is the same as massive, right?</a> <a data-t="139">That's kind of the basis for all these production kind of conversations.</a> <a data-t="144">And when people historically have said, oh, well, you know, you can't run R in production,</a> <a data-t="148">quite often what they mean is you can't run R at scale.</a> <a data-t="152">And even that's not true anymore.</a></p>

<p><a data-t="184">So for me, production is anything that's run repeatedly,</a> <a data-t="188">and that the business relies on.</a> <a data-t="190">So hopefully, I'm kind of lining up with Joe Chang on that one.</a> <a data-t="194">And the important bits of that are that it's repeated or continuous,</a> <a data-t="200">kind of an extension of the same idea, and that it's relied upon.</a></p>

<h3>Why R is great in production</h3>

<p><a data-t="247">So actually, R is great in production.</a> <a data-t="251">There are a number of reasons for this.</a> <a data-t="254">R doesn't have a lot of the baggage that other programming languages do.</a> <a data-t="260">And I'm thinking specifically things like the Python 2.3 schism.</a> <a data-t="266">There's still an awful lot of stuff in Python 2,</a> <a data-t="269">which is slowing adoption of Python 3.</a> <a data-t="272">And that's because it's used so heavily on Linux systems</a> <a data-t="277">where vendors are perhaps a little slower to upgrade things.</a></p>

<p><a data-t="281">Obviously, R doesn't have that baggage,</a> <a data-t="285">so you can install whatever version or versions of R you want to use.</a> <a data-t="289">And I've found overall that performance is exceptional.</a> <a data-t="294">On top of that, you've obviously got a fantastic package ecosystem</a> <a data-t="298">and things like that that we can draw on to build great things</a> <a data-t="303">like we saw from the previous talk.</a></p>

<h3>Getting R into production</h3>

<p><a data-t="306">So what I really want to talk about is actually how we get there.</a> <a data-t="311">How do we get R into production?</a> <a data-t="314">So maybe you're working at an organization now.</a> <a data-t="317">You've perhaps got R on your desktop, something like that.</a> <a data-t="320">But you're not running anything R-based in production.</a> <a data-t="324">There are traditionally a number of reasons for this.</a></p>

<h3>Two paths to adoption</h3>

<p><a data-t="383">So there are two kind of distinct approaches that have emerged</a> <a data-t="390">in the work that I've done and the businesses that we've talked to</a> <a data-t="393">over the last, so I've been doing this sort of work for about six years.</a> <a data-t="398">There are two paths that I've seen people be really successful with</a> <a data-t="403">for getting R into their organizations.</a> <a data-t="406">And I want to talk a little bit about those now.</a></p>

<p><a data-t="409">So the left-hand path is the path of magic, essentially.</a> <a data-t="416">So if you think about the great talk that we saw a moment ago,</a> <a data-t="423">part of what happened was that a fantastic shiny app</a> <a data-t="430">was presented to somebody important, and then that drove</a> <a data-t="435">the kind of decision-making over whether to use R</a> <a data-t="437">or something like that downwards in the organization.</a></p>

<p><a data-t="441">So the path of magic is a great path.</a> <a data-t="446">It can be a very short path, but it can also sort of</a> <a data-t="450">fraught with difficulty, I guess, because what can happen</a> <a data-t="453">is it can backfire a little bit.</a> <a data-t="456">So this path, I think, is brilliant and, like I say,</a> <a data-t="461">can be a really good shortcut, and especially if you've got</a> <a data-t="463">something really powerful that you can demonstrate.</a> <a data-t="467">But you can get kicked back later on when we come to talk</a> <a data-t="471">to IT teams about actually getting that stuff</a> <a data-t="474">into the production ecosystem within the business.</a></p>

<p><a data-t="477">So that's one option, and hopefully that's kind of</a> <a data-t="481">fairly easy to understand.</a> <a data-t="484">The right-hand path, though, is...</a> <a data-t="488">So the right-hand path has, I guess, I don't want to say</a> <a data-t="493">guarantees, because there are never any guarantees,</a> <a data-t="494">but what you're looking for with the right-hand path</a> <a data-t="499">is for the business to have confidence in the work</a> <a data-t="503">that you're doing.</a> <a data-t="505">And historically, that's the area where getting R</a> <a data-t="508">into production kind of falls down.</a></p>

<p><a data-t="510">So the IT team might look at the work that a data scientist</a> <a data-t="513">is doing and go, well, I'm not confident in what you're doing</a> <a data-t="517">because where's the test team?</a> <a data-t="520">Where's your requirements document and things like that?</a> <a data-t="522">All these kind of historical software engineering things</a> <a data-t="526">that have been sort of adopted by businesses over the last</a> <a data-t="530">50, 60 years, whatever it may be.</a></p>

<h3>Data science vs. software engineering</h3>

<p><a data-t="534">And so we need to kind of address that elephant in the room,</a> <a data-t="541">which is essentially to come to the understanding</a> <a data-t="546">and to sort of agree that data science isn't software engineering.</a> <a data-t="550">We know it isn't software engineering,</a> <a data-t="553">and that's absolutely fine.</a></p>

<p><a data-t="556">In terms of the kind of rigor that goes into these</a> <a data-t="561">different disciplines, the rigor goes into a different place</a> <a data-t="564">in data science.</a> <a data-t="566">And so what I usually see from data science teams</a> <a data-t="570">is that they're very good at ensuring their code does</a> <a data-t="574">what they think it does.</a> <a data-t="576">That's obviously extremely important.</a> <a data-t="577">But also that they're kind of methodologically sound, I guess.</a> <a data-t="583">And then data science will often stop there.</a></p>

<p><a data-t="587">Software engineering, on the other hand,</a> <a data-t="590">obviously has a different set of drivers</a> <a data-t="592">and a different set of goals.</a> <a data-t="595">And so software engineering can very quickly mushroom</a> <a data-t="600">into a much bigger thing where you have things</a> <a data-t="603">like QA departments and requirements documents</a> <a data-t="605">and all those sorts of things,</a> <a data-t="607">which you hopefully try and avoid being too onerous</a> <a data-t="612">in the data science world.</a></p>

<p><a data-t="614">So there's this friction between our starting point,</a> <a data-t="618">which is the data science,</a> <a data-t="620">and then software engineering,</a> <a data-t="622">which is kind of the IT and the business.</a></p>

<p><a data-t="626">So in terms of tackling things head on,</a> <a data-t="631">which is obviously what we're kind of aiming to do,</a> <a data-t="633">and this was just my really quick thoughts,</a> <a data-t="636">and there might be different things in your environments,</a> <a data-t="639">but these are some of the things that, as a data scientist,</a> <a data-t="643">you might be pushed into understanding</a> <a data-t="647">in order to get stuff into production.</a></p>

<h3>Production readiness checklist</h3>

<p><a data-t="958">So I've put together, and again,</a> <a data-t="961">this list is by no means exhaustive,</a> <a data-t="964">but I've put together a kind of production list</a> <a data-t="967">or a production readiness checklist</a> <a data-t="970">for some of this stuff.</a> <a data-t="972">So when R goes into production,</a> <a data-t="974">you need to think about the target environment.</a> <a data-t="977">What is the target environment?</a> <a data-t="979">Is it Windows?
