<h3>What we gain from multiple languages</h3>

<p><a data-t="1037">I am now going to, and to start and talking about what is it that we gain when we use</a> <a data-t="1047">multiple programming languages.</a> <a data-t="1049">And I am going to start back at the organization level because I know that's top of mind when</a> <a data-t="1055">people try to think about, oh, like, will we support using XYZ language in our research</a> <a data-t="1062">group or company or department?</a> <a data-t="1064">So the, at any given time, the people that are in your organization are better at some</a> <a data-t="1071">task using one, one tool versus another tool.</a> <a data-t="1077">And when that person can make the choice of what tool they're going to use, they are,</a> <a data-t="1083">they are going to be more productive.</a> <a data-t="1085">And so when we use multiple programming languages, this makes, this makes everybody more productive.</a></p>

<p><a data-t="1091">But this, this dynamic is in tension with the consistency, consistency, complexity thing,</a> <a data-t="1101">like, like, oh, we're having problems around consistency and complexity.</a> <a data-t="1105">This ends up, this ends up being intention or balancing everyone actually being better</a> <a data-t="1110">at the work that they are trying to do.</a> <a data-t="1112">And in my experience, like there is no a priori way of knowing how that balance is going to</a> <a data-t="1118">play out.</a> <a data-t="1119">Like there is, there's no hard and fast rule about how, like which one of those will win</a> <a data-t="1126">in the end.</a> <a data-t="1127">It depends on scale of your, you know, it depends on the, like the specifics of what</a> <a data-t="1133">the infrastructure is like in your organization.</a></p>

<p><a data-t="1138">This leading to people to just be like, okay, well, let's figure it out.</a> <a data-t="1147">Let's take a pragmatic attitude to how it is that we, how we choose tools.</a> <a data-t="1152">This leads to like a belief or a value that you might express like this, practicality</a> <a data-t="1158">beats purity.</a> <a data-t="1159">This is also from the Zen of Python, and this is a, this is a, like a reflection of the</a> <a data-t="1165">idea that we can, we can, we can in our specific circumstances decide what works best at this</a> <a data-t="1174">point and not have an overly purist view about tooling or how people should be doing</a> <a data-t="1179">things.</a></p>

<p><a data-t="1180">This is really related to this idea of the right tool for the job.</a> <a data-t="1184">And what I observe is that it is, it is very difficult and, you know, like maybe plain</a> <a data-t="1191">impossible to make, to make overarching general statements about what a right tool for a job</a> <a data-t="1198">is.</a> <a data-t="1199">But instead, what the right tool for the job is, is always specific to a circumstance,</a> <a data-t="1204">to a person, even to a time.</a> <a data-t="1206">So the right question isn't what is the right tool for the job, but what is the right tool</a> <a data-t="1211">for the job for this person at this time in this organization?</a></p>

<blockquote><a data-t="1206">So the right question isn't what is the right tool for the job, but what is the right tool</a> <a data-t="1211">for the job for this person at this time in this organization?</a></blockquote>

<p><a data-t="1221">The project that I worked on that, where I've seen this play out a lot is, is, is Vetiver.</a> <a data-t="1227">So Vetiver is software for Python and R that, for MLOps.</a> <a data-t="1232">So for about the, the process of deploying and operationalizing models.</a> <a data-t="1236">So the, when you say MLOps, people mean a lot of different things.</a> <a data-t="1240">So I'm going to get a little specific here by looking at this diagram of a model lifecycle.</a> <a data-t="1245">So if we start with, you know, we start by collecting data.</a> <a data-t="1249">The first thing we'll need to do when we have that data is to understand and clean it.</a> <a data-t="1254">And there's lots of great open source software, open source tools for understanding and cleaning</a> <a data-t="1260">your data.</a> <a data-t="1261">Next, the next thing you do is it's time to train and evaluate a machine learning model.</a> <a data-t="1267">And again, there's lots of really fantastic open source packages across different languages</a> <a data-t="1271">to, to train and evaluate the model.</a></p>

<p><a data-t="1275">At that point, it, the situation becomes much less clear.</a> <a data-t="1281">And that's both because there, there's a lot less community understanding of what, what</a> <a data-t="1286">is the right next thing to do once you have a machine learning model trained.</a> <a data-t="1292">And also because there are fewer open source tools for what needs to happen next.</a> <a data-t="1298">Especially fewer open source tools built for a scientist, data analyst, data scientist</a> <a data-t="1304">type user, as opposed to what I might call a generalist software engineer type user.</a> <a data-t="1311">And this is where Vetiver sits.</a> <a data-t="1313">Vetiver has a opinionated idea of what are the next things you need to do.</a> <a data-t="1319">You need to version your model, deploy your model, and then monitor your model.</a> <a data-t="1325">And also provides, provides functions and infrastructure for, for taking those next</a> <a data-t="1340">steps.</a></p>

<p><a data-t="1341">And it's really about this, like giving people the opportunity to choose the right tool for,</a> <a data-t="1348">for what they are needing to do.</a> <a data-t="1349">So person A might decide, oh, I'm going to train a random forest model using Scikit-learn</a> <a data-t="1355">and Python.</a> <a data-t="1356">And they can deploy, they can version, deploy, and monitor their model with Vetiver.</a> <a data-t="1361">And person B might say, oh, I need to create a survival analysis model.</a> <a data-t="1367">And I think the best way to do that is going to be to use tidy models and R.
