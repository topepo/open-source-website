<h3>Ways to add CSS to Shiny</h3>

<p><a data-t="219">So, how would we actually go about doing this?</a> <a data-t="219">So, with Shiny itself, there's three very specific ways that you can do it.</a> <a data-t="225">So, you can add the style directly to your tags when you're building your UI.</a> <a data-t="232">You can add it as, let's say, a separate tag in the header of your page, or your application, but you can also store it in files and link those files directly to the Shiny application.</a></p>

<p><a data-t="247">So, for the first one, whatever you create in your UI, as long as it's an HTML tag,</a> <a data-t="254">you can always pass a special style attribute, which lets you basically pass any kind of</a> <a data-t="262">CSS statement that will affect that specific element.</a> <a data-t="262">So, here, there's no statements, there's no classes, there's nothing.</a> <a data-t="266">This is style that you want to apply to that specific element.</a> <a data-t="277">And, this is fine. This is okay.</a> <a data-t="277">But, to be honest, this is probably, this should be your last resort when it comes to adding CSS to Shiny.</a></p>

<p><a data-t="285">The truth is that, by adding, it's fine if you have one or two elements, but when you get to 10, 20, 50 elements, when you start having very complex UIs, you lose track of where you style the specific thing.</a> <a data-t="299">You cannot reuse any rules.</a> <a data-t="299">So, if you have three titles, you need to style the three titles.</a> <a data-t="304">It's very hard to keep consistency between all the elements.</a> <a data-t="311">This is something that, if you use BSLib, for example, it's all for you, because there's general rules that affect every element, but if you're doing these individual changes, you won't be able to actually keep track of them as your project grows.</a></p>

<p><a data-t="323">The second way would be to add CSS to your header.</a> <a data-t="330">So, this is slightly better.</a> <a data-t="337">You can add a style tag with some text inside, and that text can be CSS text.</a> <a data-t="337">This is slightly better.</a> <a data-t="345">This does mean that you can use those selectors.</a> <a data-t="345">You can apply the same styling to a lot of elements at the same time, but at the same time, you won't really be able to leverage the browser here, because you're generating the CSS whenever the page is created.</a> <a data-t="359">This actually means that your browser won't be able to cache these.</a> <a data-t="367">By caching, I mean saving them for future usage when you reopen the application later.</a> <a data-t="374">So, you end up having a slightly better approach, but at the same time, it's still not the ideal approach.</a></p>

<h3>Introducing Sass</h3>

<p><a data-t="450">So, one way that you can improve this workflow even better is to use Sass.</a> <a data-t="459">So, Sass is, as Tom mentioned before, this is a language built on top of CSS, and the idea here is that it's</a> <a data-t="467">preprocessor to CSS, so any Sass that you create always ends up being CSS, but there's this layer</a> <a data-t="476">of obstruction where you have a few different tools, and you don't really need to worry about</a> <a data-t="483">creating very complex or managing very complex CSS, because Sass gives you a couple of ways of managing it a bit better.</a></p>

<p><a data-t="494">So, a couple of differences here between Sass and CSS are</a> <a data-t="494">Sass is object-oriented, so this means that you can have nesting, it's more about the concept of what is a specific element that you're trying to target, not the specific element that you're targeting.</a> <a data-t="513">It also allows variables.</a> <a data-t="513">I know that CSS3 is actually pushing forward with variables in CSS.</a> <a data-t="521">This is, I think, a very nice bonus, but it's one of the things that Sass, at least my understanding, still does much better than just CSS.</a> <a data-t="541">And also, you can always use Sass to actually generate CSS variables to reuse in the future.</a></p>

<p><a data-t="557">So, a couple of really interesting things that you can do with Sass.</a> <a data-t="557">So, we talked about nesting, so what nesting means is that you can simplify a lot of the rules that you do by nesting them.</a> <a data-t="569">So, as you can see, CSS doesn't, CSS on the right side doesn't really allow for this, so if you're targeting two different elements, in this case, for example, a UL and an LI element inside of an app bar, you're going to need to create the full selector every time you create a rule.</a> <a data-t="586">Sass kind of solves this by allowing nesting, so if you have two elements inside the same elements, you can kind of nest them together and simplify your code even a tiny bit more.</a></p>

<p><a data-t="583">We also talked about variables, so CSS does allow variables, but Sass does it in a way that you can kind of declare them, even in separate files, completely different.</a> <a data-t="595">There's no scope required, which is something that CSS variables need, and you can simply use them by assigning a name to a property that starts with a dollar sign, and then just this becomes a variable that you can use throughout your code.</a></p>

<p><a data-t="628">It also has two very important mechanisms, which are extents and mixins, so extents is kind of, and this is something that is very well explained with an example, so it basically allows you to share styles between selectors, and the way that you would do this normally in CSS, if you would have two elements that you would like to style, even though they are very similar, is that you would simply do the styling for both of them.</a> <a data-t="642">By using extend, you can actually create a partial part of code by using the percentage as a marker for the name of that block of code, and you can then extend for each element, you can simply extend the name, the extent rule that you created, and the code will actually be copied into that selector, so this means that you can reuse parts of styling that you use very often by simply extracting them into a separate name that you can then reuse.</a></p>

<p><a data-t="684">The second one is mixins, so mixins are slightly similar, but these work more like functions, so the idea here is that instead of simply taking out a piece of a block of code and saving it with a name, with a mixin, you're actually having a function, so this means that you can pass specific arguments to those functions that you're creating, so in this case, for example, we've identified that the only difference between these two components is actually the border size, so we can create a mixin that has an argument of border size and simply call those functions in each of our declarations for each specific element.</a></p>

<h3>Speeding up Shiny apps</h3>

<p><a data-t="912">Let's talk about speeding up Shiny apps.</a> <a data-t="912">So, small disclaimer, this is going to be just some of the, I think, very important details that you always need to take care of.</a> <a data-t="920">It doesn't mean that you'll be able to speed up everything using just these tips, but hopefully it will get you started when it comes to at least identify some of the issues that your Shiny app might have.</a> <a data-t="935">So, I'm going to talk about three very important details when it comes to speeding up Shiny applications.</a> <a data-t="940">The first one is update functions.</a> <a data-t="945">I will also talk a bit about proxy objects and proxy functions, and then I will finish up by letting you know a bit about custom messages and browser offloading.</a> <a data-t="957">I won't get into too much detail, and, of course, every package is different and needs different types of optimisations, but hopefully this will help you get started.</a></p>

<p><a data-t="971">So, update functions. What are update functions?</a> <a data-t="972">So, update functions are very simple functions that let you update existing Shiny widgets.</a> <a data-t="980">So, the important thing here is that base Shiny has these, but it's very possible that you're not always using them.</a> <a data-t="989">So, as an example, we have here a very simple Shiny application that has just a checkbox and a drop-down, and you can see that in our UI, we define a checkbox and we define a UI output, and then in our server, we simply observe that checkbox and we change our select input label and values depending on the state of the checkbox.</a> <a data-t="1017">This feels very standard.</a> <a data-t="1017">This is something that you probably did quite a lot of times already.</a></p>
