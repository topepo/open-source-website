<h3>How quickr works under the hood</h3>

<p><a data-t="362">So if you just want to use quickr, that's roughly all you need to know.</a> <a data-t="366">But I'm sure many of you are curious, how does it work under the hood?</a> <a data-t="370">So what happens when you pass a function to quickr is it takes that function and it translates it to Fortran, and then it lays down the plumbing to make that Fortran routine callable from R.</a></p>

<p><a data-t="385">So this R function becomes this Fortran subroutine.</a> <a data-t="391">And then this C function is built.</a> <a data-t="393">This is the bridge between Fortran and R, and here we unpack the R arguments.</a> <a data-t="398">This is where we do the type checking, call the Fortran subroutine, and then return the result.</a> <a data-t="405">And then this compiled C function gets wrapped in an R closure, and that's returned to the user at runtime.</a></p>

<p><a data-t="412">Now the sequence of events here has two modes of operating, which I'll call the JIT mode and the AOT mode.</a> <a data-t="420">So JIT stands for adjust in time, and this is what happens when you just call Quick in a normal R session.</a> <a data-t="428">So it'll generate the Fortran and C code, write it out to a temporary directory, call the compiler, load the shared object, build the closure, and return it.</a> <a data-t="439">And that happens quickly.</a> <a data-t="441">It's fine.</a> <a data-t="442">It's fractions of a second.</a></p>

<p><a data-t="444">But in an R package, Quick does almost nothing, right?</a> <a data-t="447">Something very different happens.</a> <a data-t="449">Quick returns just the closure with basically a placeholder for the pointer to the compiled code.</a> <a data-t="457">And the code generation of the Fortran and C actually happens in a separate motion when you do </a><a href="/software/devtools/">devtools</a><a data-t="457"> load all.</a> <a data-t="464">So I think this is something most package developers use.</a> <a data-t="469">If you don't, quickr also has a helper to help you invoke this code path.</a></p>

<p><a data-t="474">So quickr collates all the Quick functions that are in your package, it builds them into a submodule or a shared compilation unit, and then dumps them out into these C and Fortran files.</a> <a data-t="487">And that's it.</a> <a data-t="488">It doesn't do anything else after that point, because the R package machinery already knows what to do with an F90 file and a C file.</a> <a data-t="498">So from the perspective of the R package machinery or external reviewers like CRAN, your package just has some Fortran code in it.</a> <a data-t="507">It's not doing any kind of runtime compilation.</a></p>

<p><a data-t="511">And so this overall design, I think, sidesteps a lot of issues that you might have with other approaches to building a compiler or a runtime compiler.</a> <a data-t="521">Probably the two most successful ones that come to mind for me are Numba in Python and Julia, the language.</a> <a data-t="531">So both of these ship basically most of LLVM as part of what they offer.</a> <a data-t="538">And that comes with a lot of complexity.</a> <a data-t="540">And it's like a lot of work.</a> <a data-t="543">And here we're basically short-circuiting that.</a> <a data-t="545">And we're using Fortran, the existing infrastructure that's already there, so we don't have to ship a full compiler as part of quickr.</a></p>

<p><a data-t="552">And if you really want LLVM, you can also use Flang as basically a frontend to LLVM, which gives you a lot of the same things.</a> <a data-t="564">The other thing this does is there's no compilation overhead at startup time.</a> <a data-t="569">So Numba and Julia and sort of things in that space, when they're used to a sufficient amount, it really degrades the interactive experience.</a> <a data-t="579">Because every time you start up your REPL, you're waiting many seconds for things to recompile every time.</a> <a data-t="586">And so using the R package machinery like this, there's no degradation of the startup time.</a></p>

<h3>Why Fortran?</h3>

<p><a data-t="596">So that's how it works in broad strokes.</a> <a data-t="600">And now I'll talk about sort of the why and how.</a> <a data-t="604">So if you're not familiar, actually, how many of you, show of hands, know what Fortran is or have used it in the past?</a> <a data-t="611">Okay.</a> <a data-t="612">About half.</a> <a data-t="613">That's great.</a> <a data-t="614">Fortran is the first high-level language that was...</a> <a data-t="617">Before that, people wrote machine code directly.</a> <a data-t="620">It predates R and Python and C by over two decades.</a></p>

<p><a data-t="627">And many, many languages have come and aimed explicitly to displace Fortran and have failed and are just in the history bin.</a> <a data-t="635">It consistently ranks in one of the top ten languages that's still in use today.</a> <a data-t="640">And it's used especially in the physics community and the high-performance computing community.</a> <a data-t="645">So any time you're doing any kind of simulation, like climate modeling, weather prediction, or plasma physics, whatever, things like that, you're probably running Fortran code.</a> <a data-t="655">And about 23% of R itself is today Fortran sources.</a></p>

<p><a data-t="660">So Fortran is very, very fast.</a> <a data-t="662">And it's fast for a couple of reasons.</a> <a data-t="664">One is the people writing Fortran code and running Fortran code really care about speed.</a> <a data-t="669">And they've cared about it for a long time.</a> <a data-t="671">There are decades of effort behind making Fortran fast.</a> <a data-t="674">The language itself also makes it really easy for the compiler.</a> <a data-t="680">So the arrays are first-class citizens in Fortran with declared sizes and types and a memory layout.</a> <a data-t="688">And the array semantics means that the compiler can assume arrays don't overlap.</a> <a data-t="693">So this leaves a lot of low-hanging fruit for the compiler to do things like pipeline instructions or use SIMD or skip bounce checks and so on.</a> <a data-t="701">And these are things that you don't have to be aware of as an author.</a> <a data-t="704">You just kind of drop into this pit of success for optimization.</a></p>

<p><a data-t="710">And the language is very, very active still.</a> <a data-t="712">There are new language additions every couple of years.</a> <a data-t="714">And it continues to see compiler improvements and also new compilers coming out every few years.</a> <a data-t="719">It is actually quite an active space.</a></p>

<p><a data-t="723">So that's one reason.</a> <a data-t="724">Fortran is fast.</a> <a data-t="725">The second reason is Fortran comes with very, very strong support from R.</a> <a data-t="731">So this is a talk John Chambers gave at this very conference in 2006 showing the scale sketch of S, the initially proposing S. And from conception, from day one, the goal was to provide an interface to Fortran subroutines.</a> <a data-t="750">And this comes through in a lot of ways, including how R was designed and evolved.</a> <a data-t="756">And one of the ways it comes through is that everything with Fortran just works.</a> <a data-t="760">You don't have the growing pains that you use with something like Rust today or what C++ experienced in the past and sort of the inherent complexity that comes with that.</a> <a data-t="770">Fortran just works.</a> <a data-t="771">It works simply and it works well in an R package.</a></p>

<p><a data-t="775">The other way it comes through is that there is quite a bit of overlap in semantics and syntax and even culture between Fortran and R.</a> <a data-t="784">And this comes through in some technical ways, like, for example, one-based indexing, first class support for arrays with the same exact memory layout and very similar slicing operator or subsetting operator.</a> <a data-t="797">First class complex values, vectorized operations on those arrays, scalar broadcasting.</a> <a data-t="804">So if you multiply a scalar in an array, that just works.</a> <a data-t="809">The base namespaces of both these languages come fully populated with everything you need to do math.</a></p>

<p><a data-t="816">Both of these languages, I think, also are designed for practicing researchers, which comes through as some design decisions that maybe people who aren't practicing researchers are dismissive of sometimes.</a> <a data-t="833">And both Fortran and base R have a strong aversion to breaking changes.</a> <a data-t="837">So if you come across Fortran code written 30 years ago, it probably runs just fine today, and the same holds true for R, for well-written against base R.</a></p>

<p><a data-t="847">So the way to think about it is R is a big language with a lot of features.</a> <a data-t="851">And Fortran is a big language with a lot of features.</a> <a data-t="855">And the areas where they overlap is substantial.</a> <a data-t="858">And this is the space where quickr sits.</a> <a data-t="861">quickr will let you compile the parts of R that overlap with Fortran.</a></p>

<blockquote><a data-t="847">R is a big language with a lot of features.</a> <a data-t="851">And Fortran is a big language with a lot of features.</a> <a data-t="855">And the areas where they overlap is substantial.</a> <a data-t="858">And this is the space where quickr sits.</a></blockquote>
