<h3>The case for dependency isolation</h3>

<p><a data-t="201">So we have to look elsewhere for a definition of what</a> <a data-t="201">a project is. And one of the things that I found really inspiring when I was trying to</a> <a data-t="206">figure out how to more reliably deliver software was a website called 12factor.net. This website</a> <a data-t="213">was published by people who work on the deployment platform Heroku, which helped people deploy</a> <a data-t="218">software as a service applications to the public. But a lot of the lessons that they</a> <a data-t="224">published there, I think, are useful even in the data professional context. And one</a> <a data-t="228">of the things they really stress is the importance of environment isolation, right? So the 12</a> <a data-t="233">factor apps never rely on the implicit existence of system-wide packages. And there are a couple</a> <a data-t="238">of things that they say are important to making this happen, right? So you have to declare</a> <a data-t="242">all your dependencies, and then you have to isolate them from other things on the system.</a> <a data-t="247">And you have to make sure that no implicit dependencies leak in.</a></p>

<p><a data-t="251">And I want to dwell</a> <a data-t="251">on the leak in thing for a moment, because I'm going to give a sort of stylized example</a> <a data-t="256">from my past to show you how this kind of thing can happen, right? So let's say, you</a> <a data-t="260">know, or let's say I was working on a project. I did a lot of work on the 2020 census for</a> <a data-t="266">an advertising agency I worked on at the time. And I delivered some dashboard, and people</a> <a data-t="272">looked at the results, and it was good. And then someone asked me to do some data analysis</a> <a data-t="279">on the census tract level. It needed to do between a one and seven day look back. There</a> <a data-t="284">were like 80,000 census tracts. We were doing data analysis for, at that point, several</a> <a data-t="290">months worth of data. And so I was like, OK, I need to do something. This might be a good</a> <a data-t="295">case for the latest DTPlier. And so I installed DTPlier. I was able to work on the analysis,</a> <a data-t="304">and everything was fine.</a></p>

<p><a data-t="312">Except I accidentally, by installing DTPlier, upgraded the version</a> <a data-t="312">of DPlier that was being used in my dashboard, because the thing I haven't highlighted so</a> <a data-t="316">far is that we have a dependency on a shared user library.</a> <a data-t="329">The answer is not in production.</a></p>

<p><a data-t="334">And so I think I'm going to borrow Julia's excellent metaphor from yesterday. We want</a> <a data-t="339">to find a way of working that stops it from being possible for us to lock our keys in</a> <a data-t="339">the car.</a></p>

<h3>How to use renv</h3>

<p><a data-t="345">So let's talk about how we do that, right? The first step is going to be to isolate</a> <a data-t="346">your dependencies. And so in order to do that, you're going to need to install the renv package.</a> <a data-t="351">And then you're going to initialize it. You have a couple of options for doing that. You</a> <a data-t="354">can call the init function. There's also a little box you can check as you're creating</a> <a data-t="358">a new project in the RStudio IDE. And that will create, and this will sort of overwrite</a> <a data-t="363">your libpaths, right? So by default, you have kind of a user library and a system library.</a> <a data-t="369">And renv will take that over and say, OK, now you have a project-specific library. And</a> <a data-t="374">then your system library.</a></p>

<p><a data-t="379">And so what that looks like, if we go back in time, I properly</a> <a data-t="379">isolate my libraries. I have this user library in my dashboard. And then the project library</a> <a data-t="385">gets the upgraded versions. And that means that if I start working on new projects, say</a> <a data-t="391">I start doing experimental things with new packages and tidy models, I'm not really concerned</a> <a data-t="396">about breaking other things in my workflow. And the other thing that's nice to note here</a> <a data-t="401">is that you can adopt this incrementally. So you don't have to move all of your projects</a> <a data-t="406">at once. You can see that my dashboard is still pointing in my user library. I can go</a> <a data-t="411">and upgrade that surgically later on if I want to.</a></p>

<p><a data-t="414">And you might be asking yourself, OK, well, you're making me maintain this isolated library.</a> <a data-t="419">Does that mean I need to reinstall every package from scratch each time? The answer is no.</a> <a data-t="424">Renv maintains a package cache. So once you install a package, it's available to you to</a> <a data-t="429">reuse in other projects if they have the same dependency. So that first installation</a> <a data-t="433">might take a little bit of time. But subsequent installations will go much faster.</a></p>

<p><a data-t="440">Once we've isolated the dependencies on our system, the next thing we need to do is to</a> <a data-t="444">discover what they are and write that down somewhere. And so calling the renv snapshot</a> <a data-t="450">function is going to give us some information about the versions of packages we use and</a> <a data-t="454">which ones we have, where we got those packages from. Because as we've heard, shared drives,</a> <a data-t="462">RStudio Package Manager, the R universe, all of these are different places where you can</a> <a data-t="467">get R packages. And so you want to be able to keep track of where you got them, as well</a> <a data-t="471">as the version of R you were using for your project.</a></p>

<p><a data-t="474">And so you can see here, this is a package I've installed from the R universe. And so</a> <a data-t="479">we're keeping track of the information about the git commit. And we're writing all of this</a> <a data-t="483">down in the renv lock file. And I think the renv lock file is the project artifact that</a> <a data-t="488">we're looking for.</a></p>

<blockquote><a data-t="483">I think the renv lock file is the project artifact that we're looking for.</a></blockquote>

<p><a data-t="493">Part of the thing that it's going to enable</a> <a data-t="493">us to do is to collaborate more effectively. And so renv is itself very helpful in giving</a> <a data-t="498">you status messages about what you need to do next. And this is going to be really helpful</a> <a data-t="502">for contexts where either you're picking up an old project or you're sharing a project</a> <a data-t="506">with a teammate who doesn't have the same package set on their computer.</a></p>

<p><a data-t="511">So you can see in this case, I've installed some packages, or I'm sorry, I've pulled down</a> <a data-t="516">a lock file. And packages are in the lock file, but they're not yet on my system. And</a> <a data-t="520">renv is going to tell me what to do. And this is a similar kind of situation where I started</a> <a data-t="524">using a package, but I haven't installed it yet. Calling renv status is, again, going</a> <a data-t="529">to tell me what to do to get right. And if I remove a package from my project,</a> <a data-t="534">then I want to remove those dependencies as well, because I don't want to ship unnecessary</a> <a data-t="539">things to my deployment target, whatever that is.</a></p>

<p><a data-t="544">And so this is going to be the workflow.</a> <a data-t="544">We're going to initialize a project in renv, we'll install the packages we need, we snapshot,</a> <a data-t="550">we ship, and then later we can restore that environment. Once we have the project in renv,</a> <a data-t="556">install, snapshot, ship, and restore.</a></p>
