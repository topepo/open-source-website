<h3>Reducing collaborative friction</h3>

<p><a data-t="428">So our goal with the formatter is that once you start using it,</a> <a data-t="433">it will be hard for you to go back to your formatter-less lifestyle.</a> <a data-t="438">And that's something we've certainly managed to achieve with Hadley,</a> <a data-t="441">because now when he saves a file and he does a new format, it feels broken for him.</a> <a data-t="446">And we kind of knew that Hadley would be an easy win for us</a> <a data-t="448">because he has always cared a lot about style.</a></p>

<p><a data-t="452">But now I would like to talk a little bit about what happens</a> <a data-t="455">when you have a room full of programmers who all care a lot about style.</a> <a data-t="460">Something like this happens.</a></p>

<p><a data-t="463">So these are excerpts from our GitHub organization</a> <a data-t="467">where a pull request came in,</a> <a data-t="469">and the reviewer is focusing part of the review</a> <a data-t="473">on the style of the code rather than on the contents.</a> <a data-t="476">So that's not a good place to be in, and we call this collaborative friction.</a></p>

<p><a data-t="482">And the nice thing about the formatter workflow</a> <a data-t="485">is that if everyone on your team gets on the same page</a> <a data-t="489">and is using the formatter as configured in a project,</a> <a data-t="492">and the formatter kicks in every time a file is saved,</a> <a data-t="496">well, the code is never going to go out of sync</a> <a data-t="499">with the expectations around the style.</a> <a data-t="503">And so that fully solves the problem of collaborative friction</a> <a data-t="505">that we saw in the last slide.</a></p>

<p><a data-t="508">So you might think, but what about an external contributor?</a> <a data-t="512">They might not be using the formatter.</a> <a data-t="516">Well, for this scenario, we have a GitHub action</a> <a data-t="519">that you can set up in your repository.</a> <a data-t="521">And when you do, a robot will come in for every contribution</a> <a data-t="525">and check the style for you.</a> <a data-t="527">And it will even suggest fixes when something is not right,</a> <a data-t="532">and the contributor can merge the fixes by themselves.</a> <a data-t="536">And so there's no human in the loop anymore for checking the style.</a></p>

<p><a data-t="540">And so hopefully, when the reviewer comes in,</a> <a data-t="543">they can focus on the contents of the code</a> <a data-t="546">and get the contribution much faster.</a> <a data-t="548">So now there's another aspect to collaborative friction,</a> <a data-t="552">and that has to do with configuration.</a></p>

<p><a data-t="557">And to see this, here's an example of a configuration file</a> <a data-t="560">for a formatter called clang-format.</a> <a data-t="564">And that's for C code.</a> <a data-t="565">And as you can see, it has a lot of options,</a> <a data-t="568">extremely configurable.</a> <a data-t="571">And that's only a small sample of the option</a> <a data-t="574">that you can set to specify how you want your code to look.</a></p>

<p><a data-t="578">And that's how you want your code to be formatted.</a> <a data-t="582">So when it comes to configure the formatter for a project,</a> <a data-t="587">that's a lot of complexity.</a> <a data-t="590">And maybe some of you would appreciate this flexibility,</a> <a data-t="593">but we would like to argue that there is a hidden cost to it.</a></p>

<p><a data-t="597">And to see that, let's see an example with Davis and I</a> <a data-t="601">trying to configure a formatter for Rust code,</a> <a data-t="603">because we write a lot of Rust code these days.</a> <a data-t="606">Suggesting that we might be able to use a particular option.</a> <a data-t="609">But then I come in, and I'm like,</a> <a data-t="610">well, actually, I don't like that option.</a> <a data-t="613">I think it's better if the code looks this way.</a> <a data-t="615">And then we're just back here again.</a> <a data-t="617">So we're back to collaborative friction,</a> <a data-t="619">because now we have to agree as a team</a> <a data-t="622">on all the options that we want to use in the project.</a></p>

<p><a data-t="626">So that's why we made the formatter</a> <a data-t="628">more like a live switch.</a> <a data-t="630">It's either on or off.</a> <a data-t="632">You're either using the formatter with all the rules</a> <a data-t="634">that we have implemented, or you're not.</a></p>

<h3>Controlling code layout</h3>

<p><a data-t="656">So we have kind of a strict formatter.</a> <a data-t="658">But does that mean that you have absolutely no say</a> <a data-t="662">as to how the code is laid out?</a> <a data-t="665">With the formatter.</a> <a data-t="667">Well, not quite, because one of our goals</a> <a data-t="669">was to give you power and control</a> <a data-t="671">over how the code is laid out.</a> <a data-t="673">And this happens in two ways.</a></p>

<p><a data-t="677">And the first way is persistent line breaks.</a> <a data-t="680">And here we have an example with a couple of expressions,</a> <a data-t="683">a list call and a ggplot pipeline.</a> <a data-t="686">And they are laid out in what we call the flat layout,</a> <a data-t="689">which is like horizontal.</a> <a data-t="691">And let's say that you, as a programmer,</a> <a data-t="693">want to go to the expanded layout,</a> <a data-t="694">which is more vertical.</a></p>

<p><a data-t="697">The idea of persistent line breaks</a> <a data-t="698">is that you are going to find the first element</a> <a data-t="700">in that expression, insert a line break, save the file.</a> <a data-t="704">And it's going to be taken as the line break</a> <a data-t="707">is an instruction for the formatter to go to expand it.</a> <a data-t="710">So let's see what it looks like.</a> <a data-t="712">With the list call, you can find the first argument,</a> <a data-t="715">insert a line break, you save, goes to expanded.</a> <a data-t="719">And with the ggplot pipeline, you find the first element,</a> <a data-t="721">insert a line break, you save, and it goes to expanded.</a></p>

<p><a data-t="725">And the nice thing is that this process is reversible,</a> <a data-t="727">so you can remove the line break,</a> <a data-t="729">you save, it goes back to flat.</a> <a data-t="732">And same with the list call.</a> <a data-t="735">So that's one of our favorite features with the formatter</a> <a data-t="738">because it really gives you a grip on your code.</a> <a data-t="741">And we hope that you will like it too.</a></p>

<p><a data-t="746">So the second way you have control is with magic commands.</a> <a data-t="750">So here we have an example with the igraph package,</a> <a data-t="752">which is a package to work with graphs.</a> <a data-t="755">And it has a domain-specific language to specify graphs</a> <a data-t="759">with plus and minus operators.</a> <a data-t="762">And if you try to use the formatter with this code,</a> <a data-t="765">it's not going to know how to do a good job with this code.</a> <a data-t="770">And you get something that doesn't look like a graph anymore.</a></p>

<p><a data-t="773">So what you can do in these situations</a> <a data-t="775">is use a magic command with a skip directive.</a> <a data-t="778">And that will instruct to ignore the next expression.</a> <a data-t="783">So if you use igraph,</a> <a data-t="785">you're not going to want to sprinkle your code with magic commands.</a> <a data-t="788">So there's also a way in the configuration file</a> <a data-t="791">to specify which function should be skipped</a> <a data-t="793">across the whole project.</a></p>

<p><a data-t="796">Now, magic commands are not only for turning off the formatter.</a> <a data-t="800">And we are starting to look into new ways</a> <a data-t="803">to specify how to layout code.</a> <a data-t="807">And one feature that is currently in development</a> <a data-t="810">is a tabular directive.</a> <a data-t="813">And that's going to be extremely useful</a> <a data-t="814">with Tribble calls, for example.</a></p>

<p><a data-t="817">So if you don't know,</a> <a data-t="818">Tribble is a tidy-verse way of creating a data frame</a> <a data-t="822">where you enter tabular data in your script.</a> <a data-t="827">And historically, it's always been kind of a pain</a> <a data-t="830">to format these Tribble calls</a> <a data-t="832">because you have to manually align all of the columns.</a> <a data-t="836">But with the formatter,</a> <a data-t="840">that becomes automatic.</a> <a data-t="842">You just save the file</a> <a data-t="843">and everything goes in the right place.</a> <a data-t="845">And if you edit one of the fields,</a> <a data-t="848">create some misalignments,</a> <a data-t="850">you only need to save the file</a> <a data-t="851">and everything is perfect</a> <a data-t="853">and you get a nice little Tribble call.</a></p>

<p><a data-t="856">So that feature is still in development</a> <a data-t="858">but coming soon to the formatter.</a> <a data-t="859">But as you can see,</a> <a data-t="861">even though we have a strict configuration,</a> <a data-t="865">we still give you many ways to control</a> <a data-t="868">how the code should be laid out.</a></p>

<p><a data-t="872">So in general,</a> <a data-t="874">we think that the formatter will save you time,</a> <a data-t="876">that it will make your code pretty,</a> <a data-t="879">that it will reduce collaborative friction,</a> <a data-t="881">and in general,</a> <a data-t="882">just spark joy in your development workflow.</a></p>
