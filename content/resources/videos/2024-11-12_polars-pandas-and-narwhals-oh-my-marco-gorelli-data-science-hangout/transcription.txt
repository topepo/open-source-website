<h3>Introducing pandas, Polars, and Narwhals</h3>

<p><a data-t="489">So pandas and polars both expose objects known as data frames.</a> <a data-t="497">A data frame is a two-dimensional object with which you can store data, typically in columns.</a> <a data-t="503">So you've got a collection of columns.</a> <a data-t="505">Each column has to have the same number of elements, and within each column, all the elements need to have the same data type.</a> <a data-t="513">People in the chat are saying that they are two cute bears, and this is true.</a> <a data-t="517">It does seem to be a bit of a law that in data science you need to name your tools after animals.</a> <a data-t="522">I tried to follow that tradition with narwhals.</a></p>

<p><a data-t="524">But pandas actually derives from a panel data analysis.</a> <a data-t="544">A panel was some structure that was in the library, but it's no longer around.</a> <a data-t="553">And polars was, so it's written in a programming language called Rust, where the file extension is usually .rs.</a> <a data-t="559">So that's why polars is named like this.</a> <a data-t="561">It ends with .rs and also makes a reference to the pandas library.</a></p>

<p><a data-t="575">Narwhals, on the other hand, tries to bring them together.</a> <a data-t="583">So traditionally, most of data science, let's say that pandas has carried a lot of the weight of the world of data science on its back for the best part of a decade.</a> <a data-t="608">And most of the time when people write tools intended for data scientists, they are writing them with pandas in mind.</a> <a data-t="634">So maybe people can pass in arrays, maybe people can pass in pandas data frames, and quite often that's about it.</a> <a data-t="648">But things are starting to change.</a> <a data-t="650">A lot of users are demanding that their tools natively support polars data frames because it's newer, it's trendier, it's got a bunch of improvements over pandas.</a> <a data-t="660">Like it's a lot stricter, it has some parallelization by default, it's got a lazy API, it's generally speaking noticeably faster and stricter in a lot of respects.</a> <a data-t="672">It helps you avoid making a lot of bugs that you would otherwise make quite easily using pandas.</a></p>

<p><a data-t="677">So there's a lot of demand now on people making data tools to not just support pandas, but also to support polars data frames.</a> <a data-t="683">So how do people do that?</a> <a data-t="685">You can either, there's a few different strategies.</a> <a data-t="689">So let's talk about the different strategies.</a></p>

<p><a data-t="719">So if you want to make a tool that supports both pandas and polars, what are your strategies?</a> <a data-t="723">So one strategy is to just write your logic using pandas.</a> <a data-t="728">And then if your input isn't pandas, you convert to pandas and you continue like that.</a> <a data-t="733">This kind of works, but it's a missed opportunity.</a> <a data-t="736">It could be a lot better to just keep things native to the library that people are starting with.</a> <a data-t="741">Another strategy could be to duplicate your logic.</a> <a data-t="745">So you've got your logic for pandas dataframes, your logic for polars dataframes,</a> <a data-t="749">but then what happens? What happens next year when somebody comes across with belugas dataframes?</a> <a data-t="754">You're going to have to have yet another set of logic for those dataframes.</a></p>

<p><a data-t="759">So what we're trying to address with narwhals is a third way of doing this, which is just express your dataframe logic once using a nice little unified API that's extensive enough to be useful, but not too extensive that it becomes unmaintainable.</a> <a data-t="812">And like this, it can just dispatch to whatever input your user provides and you enable your users to achieve the concept of BYODF,</a> <a data-t="820">which is not a system of a down song. It stands for bring your own dataframe.</a></p>

<h3>Lazy vs eager execution</h3>

<p><a data-t="574">So with Pandas, everything happens eagerly.</a> <a data-t="579">You give the library some instructions and it evaluates your instructions the moment you give them.</a> <a data-t="584">Whereas Polars, it's got an option whereby you can give it some instructions, it can wait a bit, and then it can do them all together in the most optimal way that it can detect.</a> <a data-t="610">If I was to tell you to cook me a recipe and I gave you the steps one at a time, and you have to do the steps one at a time, and for each step go out and buy the ingredients that you need for that step, it's going to be a pretty inefficient way of making a cake.</a> <a data-t="633">But if I can give you the recipe beforehand and you can buy all the ingredients together, then maybe do two steps at the same time, it's going to be a much nicer cooking experience.</a></p>

<p><a data-t="650">On the question about, I don't deal with big datasets, why should I care about lazy execution? I would say that performance doesn't matter until it does.</a> <a data-t="655">And that when you've got something that's categorically faster, it changes the kinds of questions you can ask.</a> <a data-t="661">So in a previous company I was working at, I remember that there was some workflow where we were making weekly forecasts or something.</a> <a data-t="666">There was like a very complicated process, like the forecasting process took like hours.</a> <a data-t="671">But the moment when we modernized the tool chain, used some better algorithms and we got the time from hours to minutes, then all of a sudden the questions that product management started asking were, well, maybe we can run daily or by daily forecasts.</a> <a data-t="688">So yeah, categorically better performance really changes the kinds of questions you ask.</a> <a data-t="692">And then at some point, even if your data isn't big now, maybe it will be later.</a> <a data-t="697">So it's a good thing to start with scalability in mind.</a></p>

<blockquote><a data-t="650">Performance doesn't matter until it does. And that when you've got something that's categorically faster, it changes the kinds of questions you can ask.</a></blockquote>

<h3>PyArrow and dependency management</h3>

<p><a data-t="793">And some people suggest this as a solution to data frame interoperability. <h3>Narwhals vs Ibis</h3>

<p><a data-t="1714">So perhaps for people who don't know, Ibis is a project which describes itself as being a portable data frame library.</a> <a data-t="1720">So the idea is that they've got this Python API, which then gets translated to their intermediate representation, and they can then translate it either to SQL engines or to Polars.</a> <a data-t="1730">And like this, with just one API, you can target different engines.</a></p>

<p><a data-t="1742">And if you're an end user, if you need to do some complicated analysis, if you're an ML engineer or whatever, then I'm not sure that Narwhals is necessarily the best tool for that task.</a> <a data-t="1757">So with Narwhals, we're trying really to target library maintainers.</a> <a data-t="1766">And to do that, we're making decisions based on what library maintainers need.</a> <a data-t="1772">And I don't think that our API is extensive enough or that it will be extensive enough for the kinds of really customized, really complicated analyses that a lot of analysts and ML engineers are doing.</a> <a data-t="1791">So a SQL front end like Ibis might be a better choice for that.</a></p>

<p><a data-t="1797">Conversely, for the kinds of tools that people are building using Narwhals, I don't think that Ibis is a very good choice.</a> <a data-t="1803">And a few reasons for that are that a lot of tools that use Narwhals really do a lot with eager execution, whereas with Ibis, everything is lazy.</a> <a data-t="1818">And with Narwhals, we've got both the lazy and the eager execution, so you can choose.</a> <a data-t="1822">But at some point, you may well need to do things eagerly, and we allow that.</a></p>

<p><a data-t="1831">The other is that with Ibis, every back end requires Pandas and PyArrow as required dependencies, whereas with Narwhals, we've been very strict from the beginning by saying no back end should have any extra dependencies.</a> <a data-t="1845">Another is that with Ibis, there's no support for categorical data.</a></p>
