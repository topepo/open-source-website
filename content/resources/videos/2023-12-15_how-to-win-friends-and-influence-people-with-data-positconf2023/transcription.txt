<h3>Influencing leadership</h3>

<p><a data-t="105">So how do you ensure that your data science project goes into production and gets widely</a> <a data-t="110">adopted?</a> <a data-t="112">I have found an enormous amount of value from a small book written 90 years ago called How</a> <a data-t="117">to Win Friends and Influence People.</a> <a data-t="121">This is a book that kept getting recommended over and over again for probably two decades</a> <a data-t="126">before I finally picked it up, and I think it took me a long time to pick it up because</a> <a data-t="131">my initial reaction to the title was, ick.</a> <a data-t="135">It sounds like a shmarmy, manipulative way to deal with leaders and colleagues.</a></p>

<h3>Winning over your colleagues</h3>

<p><a data-t="571">So this brings us to step two, winning over your colleagues.</a> <a data-t="575">So leadership's endorsed your solution, and they're now asking, like, great, we want to</a> <a data-t="579">endorse this, we want people to adopt it.</a> <a data-t="583">You don't want this to be something that's jammed down your colleague's throat, or is</a> <a data-t="586">just miserable for them to implement.</a> <a data-t="589">And this is where design is going to come in heavily, is where your colleagues are concerned.</a> <a data-t="592">You want to design a user-friendly tool, and then provide adequate support for adoption.</a></p>

<p><a data-t="609">So I can proudly say our first sequential testing apps were unusable.</a> <a data-t="613">But luckily, we had done lots of user testing with the analysts that went into it, and they</a> <a data-t="617">told us this early on.</a> <a data-t="620">They would say things like, I don't even know where to look, I don't know where to start,</a> <a data-t="623">I don't know what I should do.</a></p>

<h3>Support and adoption</h3>

<p><a data-t="786">So summary for winning over your colleagues.</a> <a data-t="790">You need to promote your colleagues' adoption of your solution by designing user-friendly</a> <a data-t="794">interface and ensuring adequate support for adoption.</a> <a data-t="799">So to do this, we use the user-friendly design.</a> <a data-t="802">But this second step was really important, is that we chose to do team-by-team interactive</a> <a data-t="807">trainings.</a></p>

<p><a data-t="809">Most data science projects I see rolled out, they may have one or two mass trainings with</a> <a data-t="815">300 or 500 people present.</a> <a data-t="817">That's not an environment where people are going to ask vulnerable questions or try to</a> <a data-t="822">reconcile their needs with what you're offering them.</a> <a data-t="826">Team-by-team interactive trainings, these are like 10 people or less.</a> <a data-t="830">People get very comfortable asking those dumb, vulnerable questions that actually ensure</a> <a data-t="835">that they're going to feel confident using the tool.</a></p>

<p><a data-t="838">The second key piece is that it's interactive.</a> <a data-t="841">If you're merely talked at for an hour on how to use an application, there's very low</a> <a data-t="845">likelihood that you're going to retain or feel confident applying any of that.</a> <a data-t="849">We always use the show, then do rhythm.</a> <a data-t="852">So we would demonstrate a capability of the app.</a> <a data-t="854">And then we would have them, for instance, change the primary metric and then perform</a> <a data-t="858">the same steps themselves.</a> <a data-t="860">This makes the training a lot stickier.</a></p>

<p><a data-t="863">Finally, we offered ongoing support in a Slack channel, sequential testing support.</a> <a data-t="871">And I want to emphasize that because we took those first two steps of it was a user-friendly</a> <a data-t="875">design and we had provided adequate training, we were not overwhelmed with minutiae requests</a> <a data-t="882">in the Slack channel.</a> <a data-t="884">A lot of times, the questions we were getting in the Slack channel were actually really</a> <a data-t="887">provocative or really exemplary requests that other people could benefit from seeing the</a> <a data-t="893">conversation around.</a></p>

<p><a data-t="894">The tone in which you respond in the Slack channel is also going to set a lot of expectation</a> <a data-t="899">around whether people are irritating you with their help requests or whether this is a collaborative</a> <a data-t="904">effort that they're participating in.</a> <a data-t="906">And I think, you know, if you take these three steps, it is a lot more work than just</a> <a data-t="913">doing, you know, rapid rollout and mass training.</a> <a data-t="916">But this is what it takes, you know, to see widespread adoption.</a></p>

<h3>Implement, track, and report</h3>

<p><a data-t="922">Okay.</a> <a data-t="922">So step three, implement, track, and report.</a> <a data-t="926">So if leadership has endorsed your solution and your colleagues feel empowered to use</a> <a data-t="930">it, you're going to get widespread adoption.</a> <a data-t="933">You need to be thinking ahead and have a measurement plan in place beforehand for how</a> <a data-t="938">you're going to track adoption rate and estimate the benefits of your project.</a></p>

<p><a data-t="942">So we reached 100% adoption, but now we needed to go back to leadership and tell them, well,</a> <a data-t="947">how do we fare against this goal to reduce test duration by 25%?</a> <a data-t="951">And importantly, what was that worth?</a></p>

<p><a data-t="955">So the way that we did this was that every sequential test under the hood was computing</a> <a data-t="961">a fixed sample power analysis.</a> <a data-t="963">So we weren't just going to rely on simulations anymore.</a> <a data-t="965">We would know for every single test how long it actually took and how long it would have</a> <a data-t="969">taken if they had run the same analysis in a fixed sample test.</a> <a data-t="974">And what we found, on average, was right in line with our simulations.</a> <a data-t="977">We had an average savings of 63%, and you can see in the negative territory that some</a> <a data-t="983">tests did run longer.</a> <a data-t="985">But on average, this was an enormous time savings, well exceeding our 25% goal.</a></p>

<p><a data-t="991">But when tests end 63% faster, that means that there's more time in season for customers</a> <a data-t="998">to enjoy the best experiences.</a> <a data-t="1002">So what we then did is we calculated the incremental revenue gain from faster rollout on every</a> <a data-t="1007">sequential test.</a> <a data-t="1009">And so what you're seeing is it's obviously the sequential tests are on the Y axis.</a> <a data-t="1014">The blue section is, I should say the full length of the bar, is the revenue associated</a> <a data-t="1020">with that rollout.</a> <a data-t="1023">And if you go back to the blue section, that's how much the revenue would have been if we</a> <a data-t="1026">had done the delayed rollout on a fixed sample test.</a></p>

<p><a data-t="1031">So now we could sum up the green incremental revenue gain from sequential testing across</a> <a data-t="1035">all of our tests, and we could report back that not only did we save 63% time-wise, but</a> <a data-t="1042">that this method had increased revenue by 38% relative to the methods that we replaced.</a></p>
