<h3>How reactivity works</h3>

<p><a data-t="166">The main strength of Shiny is</a> <a data-t="166">reactivity, and what that means is that the only things that re-render in a Shiny app are the</a> <a data-t="171">things that need to re-render. So it doesn't run the application code top to bottom, it just</a> <a data-t="176">keeps track of the relationships between all the different inputs and outputs in your application</a> <a data-t="181">and re-renders the outputs which need to re-render if an input changes.</a></p>

<p><a data-t="187">What reactivity means is that</a> <a data-t="187">when you're writing an application, you basically kind of decorate various different function calls</a> <a data-t="194">which generate a relationship between all of those different function calls. So this ends up as a</a> <a data-t="199">what's called a computation graph. And the main virtue of computational graphs is that you can</a> <a data-t="204">minimally re-render them so the application can do the smallest amount of work that it needs to do</a> <a data-t="208">to generate the right output. What makes this very powerful for Shiny is that if you have, for example,</a> <a data-t="214">a multi-page app with many different components, the only components that actually cause your</a> <a data-t="219">computer to do anything are the ones that are visible to the user. So this kind of means that</a> <a data-t="223">you can have apps load fast initially, and then you can put expensive computations in places that</a> <a data-t="230">people don't necessarily go to right away to speed up the overall user experience.</a></p>

<p><a data-t="238">So reactivity</a> <a data-t="238">hits this sweet spot where it allows you to accommodate really, really complex things,</a> <a data-t="243">but your code doesn't really need to be that complicated because most of the work</a> <a data-t="246">about the wiring and how those relationships are generated between the different components</a> <a data-t="252">is really a side effect of writing the application. It's not something you need to</a> <a data-t="256">think about that much, especially for a beginner.</a></p>

<blockquote><a data-t="238">So reactivity hits this sweet spot where it allows you to accommodate really, really complex things,</a> <a data-t="243">but your code doesn't really need to be that complicated because most of the work</a> <a data-t="246">about the wiring and how those relationships are generated between the different components</a> <a data-t="252">is really a side effect of writing the application. It's not something you need to</a> <a data-t="256">think about that much, especially for a beginner.</a></blockquote>

<h3>Shiny vs. Streamlit</h3>

<p><a data-t="263">Streamlit is a web application framework that</a> <a data-t="263">allows you to take an existing Python script and add special function calls that turn it into</a> <a data-t="269">interactive web application. And the main virtue of Streamlit is that it's a really natural next</a> <a data-t="275">step from developing a Python script. So the idea behind Streamlit is you take your existing Python</a> <a data-t="282">script, and just by adding some functions to that script, it turns it into a web application. So you</a> <a data-t="288">don't need to reorganize your code very much initially to get something that does run as a</a> <a data-t="294">useful interactive web application. So that initial experience is very good. The app looks</a> <a data-t="300">very nice right out of the gate. And the main thing is that for the brand new user, they're</a> <a data-t="306">not asked to learn very much to start with Streamlit. And that's kind of, I think, one of</a> <a data-t="309">the main reasons why it's been such a phenomenally successful company.</a></p>

<p><a data-t="315">The main technical difference</a> <a data-t="315">between Shiny and Streamlit is that Streamlit re-renders the whole application code start to</a> <a data-t="321">finish every time any input changes. Whereas Shiny only re-executes the parts of the application</a> <a data-t="327">that need to be executed in order to display something. So this kind of comes up a lot of</a> <a data-t="333">times when you maybe have a plot, which is based on reading in a big dataset. If you change some</a> <a data-t="338">option that just affects how the plot looks, like the color or the scales, by default, Streamlit</a> <a data-t="345">will just read in the whole application again. It'll start from the very beginning of the script</a> <a data-t="348">and read the whole thing. So there's various different ways of accommodating that on a</a> <a data-t="354">Streamlit application and avoiding that problem of just reading in everything all the time.</a></p>

<p><a data-t="360">But all of them kind of add a lot of complexity to the application. So Streamlit apps, I would</a> <a data-t="364">say, they start easy and end hard. So that as the app grows in complexity, the things that you have</a> <a data-t="370">to do to get it to work become really difficult and it stops being actually an easy experience.</a></p>

<p><a data-t="377">Shiny takes a different approach, which is that because it has this sort of reactive framework,</a> <a data-t="385">you have to reorganize your code a little bit to get it to work initially. So that very,</a> <a data-t="389">very first experience, you have to learn a little bit to get there. But the lessons that</a> <a data-t="394">you learn at the beginning will carry you all the way through your application's journey,</a> <a data-t="399">its lifecycle. So there's never a point once you kind of have a Shiny app running</a> <a data-t="404">where the next ask, the next question becomes this monumental task that requires you to</a> <a data-t="410">refactor your application. So the way that it accomplishes that is by executing things</a> <a data-t="419">in a computation graph instead of re-rendering everything whenever anything changes.</a></p>

<h3>Strengths of Shiny for Python</h3>

<p><a data-t="426">So there's a few strengths of Shiny for Python. So one of them is Shiny applications are websites.</a> <a data-t="431">So they're simple, like at the front end, it's a simple HTML site, which means that</a> <a data-t="436">all of the sort of HTML or JavaScript knowledge that you or your team can bring to bear on the</a> <a data-t="441">problem can be used right away. Similarly, if you have an existing R app that has some CSS,</a> <a data-t="448">that CSS should work, should be imported, should be importable without modification</a> <a data-t="454">to your Shiny for Python app. Streamlit, by contrast, their way that they kind of run</a> <a data-t="459">styling means that it's much more difficult to add truly customized user experiences on top of it.</a> <a data-t="465">It's kind of optimized for making the initial app look really good and have some small changes that</a> <a data-t="470">you can make. But if you really want to make a customized sort of look and feel of your application,</a> <a data-t="475">it becomes pretty difficult.</a></p>

<p><a data-t="480">The second main thing is the relative computational efficiency of Shiny</a> <a data-t="480">over Streamlit. So because Streamlit has made the choice to run everything start to finish,</a> <a data-t="484">you kind of necessarily are going to have code which is running, which doesn't strictly speaking</a> <a data-t="489">need to run in order to display the correct output for a given set of inputs. Shiny, by contrast,</a> <a data-t="495">minimally re-renders components, which means that you don't really need to think so hard about</a> <a data-t="501">how your app is connected to itself or making sure that only the right things are cached.</a> <a data-t="507">Shiny handles all of that for you. So with Streamlit, in order to build a kind of complicated</a> <a data-t="512">app or app that reads in large data sets, you usually have to kind of manually manage callback</a> <a data-t="518">functions or caching and caching validation, which can turn into a bit of a headache if you want to</a> <a data-t="522">get a performant application.</a></p>

<p><a data-t="528">So of course, all of these things to some degree are a matter of taste,</a> <a data-t="528">like what people find difficult or not. But if you have experienced some of these problems with</a> <a data-t="533">Streamlit, you found that execution model limiting or you found caching or callbacks confusing or</a> <a data-t="539">fraught with bugs, I'd really encourage you to give Shiny a try. We're a new framework in this</a> <a data-t="544">world, but we think we do have something really special to offer and we'd love your feedback on it.</a></p>

<h3>Shiny vs. Dash</h3>

<p><a data-t="571">So Dash is a slightly older framework in the Python ecosystem, a web application framework</a> <a data-t="577">that's built around this idea of stateless functions, which means that each callback or</a> <a data-t="582">each thing that's creating a graph or a table is independent of all the other pieces. And</a> <a data-t="588">its main goal, and it does, I think, a remarkably good job at accomplishing this goal, is to build</a> <a data-t="594">applications which are really, really efficient to deploy. So I would say a lot of the</a> <a data-t="600">things that Dash asks you to do as a developer are there so that the deployment of a Dash app</a> <a data-t="605">and the way it's served by servers to the user is really, really efficient. And that's kind</a> <a data-t="611">of the main thing that it optimizes for. The value of that is that it's easy to scale apps</a> <a data-t="617">horizontally by adding a lot more servers, a lot more processes, and because you have</a> <a data-t="622">these kind of independent callbacks. When you have done the work to develop a good Dash app,</a> <a data-t="627">you've gone through that process, the app that you end up with is really, really efficient to serve</a> <a data-t="631">to users. So the kind of user experience is fast, and most importantly, the process of scaling</a> <a data-t="638">that app is really, really easy for a DevOps person.</a></p>

<p><a data-t="646">So stateful versus stateless. So stateful</a> <a data-t="646">apps are apps where the server is kind of maintaining some information about the user</a> <a data-t="652">session. Mostly when you're working on a script on your computer, you have some kind of global</a> <a data-t="656">variable or information that is passed, that is held in memory on your computer and passed to a</a> <a data-t="662">bunch of different functions. So in stateful applications, that information is held in memory</a> <a data-t="667">by the server, and it's passed to the individual function calls. Stateless applications take a</a> <a data-t="673">different approach, which is that the server is not maintaining any kind of state about the user,</a> <a data-t="678">like each component of the app kind of needs to go from start to finish to deliver that particular</a> <a data-t="684">graph. So if you have three graphs or three different components being rendered on a website</a> <a data-t="689">in a stateless framework, they might each read in a data set, each read in the same data set</a> <a data-t="694">in order to generate that component. And the value of that is that multithreading becomes really,</a> <a data-t="702">really easy because you don't need to keep any of those components in sync.</a></p>

<p><a data-t="707">The issue with developing stateless</a> <a data-t="707">applications is that it's not how you usually program. So if you're programming on your computer,</a> <a data-t="713">you usually are dealing with one process on one laptop, one computer. And as a result,</a> <a data-t="730">it's really natural to use variables to store information in server state and to have functions</a> <a data-t="718">read data in one place and have it be used in many places by rendering functions. And so when</a> <a data-t="723">you're moving from that sort of stateful way that you've been developing a project to a stateless</a> <a data-t="728">dash application, it can be really painful, and you end up needing to do quite a lot of work</a> <a data-t="733">to move your process into that world in a kind of reasonably efficient way.</a></p>

<p><a data-t="741">When you compare Shiny to Dash, the main thing that Shiny has is just a richer set of server</a> <a data-t="746">states. Shiny is stateful, and what that means is that you can read data in one place, you can have</a> <a data-t="753">reactive calculation that does some work on it, and then pass the result of that reactive</a> <a data-t="758">calculation to as many different rendering functions as you want. So it really does a</a> <a data-t="762">better job of respecting the kind of do-not-repeat-yourself principle of computer programming,</a> <a data-t="766">where you're reading the data in one place, you're processing it in another function, and passing it</a> <a data-t="772">to rendering functions only afterwards.</a></p>

<p><a data-t="778">With Dash, you kind of are forced, if you have like an</a> <a data-t="778">application with three graphs, you have a couple of choices that you can make. You can have</a> <a data-t="782">three functions which each read and process the data, or you can have one sort of mega function</a> <a data-t="788">that reads in the data and then updates all three plots. And both of them are kind of a little</a> <a data-t="794">awkward because that's not usually how you're used to developing applications. Usually what</a> <a data-t="800">you want to do is have each function do one thing and do it well, and you want to have each</a> <a data-t="804">calculation happen in one place. You don't want to have that code be repeated anywhere in your</a> <a data-t="809">application. And when you're moving to Dash, you kind of have to pick between a few options,</a> <a data-t="814">none of which are terribly appealing.</a></p>

<h3>Design philosophy</h3>

<p><a data-t="819">So whenever I've developed a web application, usually I need to start developing it very,</a> <a data-t="824">very quickly. I have almost no time to get a minimal thing out the door. But it never stays</a> <a data-t="830">simple. It always gets more complicated. There's another ask, there's another requirement from some</a> <a data-t="836">user, and the application slowly becomes more complicated. It's used by more people, more people</a> <a data-t="842">want more things from it. So our design philosophy for Shiny is that we want Shiny to support any of</a> <a data-t="847">those asks. So even though it's something that you can start, get something up in an afternoon</a> <a data-t="852">or a couple of hours, you know that you have all the tools to make it a business-critical</a> <a data-t="857">web application that's used by many, many people. And we don't want you to end up in a place where</a> <a data-t="863">there's some task, something you really want this application to do, and the framework just won't</a> <a data-t="867">let you do it.</a></p>

<p><a data-t="873">So that's kind of what we mean when we say that it strikes a great balance between</a> <a data-t="873">simplicity and complexity. It starts simple, but that application can grow to accomplish</a> <a data-t="880">really, really complicated problems without needing to sort of fight the framework.</a></p>

<blockquote><a data-t="873">So that's kind of what we mean when we say that it strikes a great balance between</a> <a data-t="873">simplicity and complexity. It starts simple, but that application can grow to accomplish</a> <a data-t="880">really, really complicated problems without needing to sort of fight the framework.</a></blockquote>

<p><a data-t="887">To sum up, most of the popular Python web application frameworks are focused on a kind</a> <a data-t="892">of narrow band of problem complexity. Streamlit is focused on your initial application just right</a> <a data-t="897">when you start. Dash is focused on applications which are where deployment efficiency is really,</a> <a data-t="904">really important. There's not really a single framework that lets you grow from very, very</a> <a data-t="909">simple problems to very, very complicated ones. The killer feature of Shiny is that it does support</a> <a data-t="915">that type of complexity. So it's a simple set of principles to learn about building your initial</a> <a data-t="920">Shiny app, but once you learn them, you don't really need to change them as your problem gets</a> <a data-t="924">more complicated. It supports a full range of CSS and JavaScript customization and also has an</a> <a data-t="931">execution model that's efficient enough to handle data science web applications, which can often be</a> <a data-t="936">quite computationally intensive.</a></p>

<p><a data-t="942">Our main hope is that you all try this framework, let us know what</a> <a data-t="942">works and what doesn't, show us what you're building with it, and help us develop some of the</a> <a data-t="947">community contributions that we really need to make this a successful project. So give it a try</a> <a data-t="953">and we can't wait to see what you build with it.</a></p>
