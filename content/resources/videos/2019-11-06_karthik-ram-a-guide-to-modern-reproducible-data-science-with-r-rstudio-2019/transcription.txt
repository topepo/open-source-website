<h3>The research compendium</h3>

<p><a data-t="95">And so the main thing I want to talk to you about today is this idea of a research compendium.</a> <a data-t="102">And this is an idea that was put forth back in 2004 by Robert Gentleman and Duncan Temple-Lang.</a> <a data-t="109">And the general idea was that you can create sort of this container for your data, your</a> <a data-t="115">code and your text, but also take advantage of software engineering principles to make</a> <a data-t="122">sure that this package that you have or compendium that you have can easily be maintained, updated</a> <a data-t="128">and then shared with everyone else.</a></p>

<p><a data-t="130">And so what I want to encourage you after this talk is to go back and set up a compendium</a> <a data-t="135">for your project as a way to make it easily shareable and useful to other people.</a> <a data-t="143">Even though setting up a compendium was very complex and painful back in 2004, thanks to</a> <a data-t="149">Tidyverse plus </a><a href="/software/devtools/">DevTools</a><a data-t="149"> and friends, it's a lot easier to do now.</a></p>

<p><a data-t="164">And so Ben Marwick, Carl Bettinger and Lincoln Mellon wrote a very nice paper last year on</a> <a data-t="174">how to set up a research compendium or a practical guide to setting up a research compendium.</a> <a data-t="179">And I just want to talk through a few of these ideas that came out of that paper.</a> <a data-t="184">The first one being that you should just organize your project in a way that makes sense to</a> <a data-t="178">you.</a> <a data-t="182">There are no hard and fast rules about how to set up a compendium.</a></p>

<p><a data-t="187">Many of you attended Jim and Jenny's workshop over the last couple of days on how to set</a> <a data-t="187">up your R projects.</a> <a data-t="192">All the advice you got from there is excellent for a compendium.</a> <a data-t="197">And so just do it in a way that makes sense for people that you might collaborate with.</a> <a data-t="202">Once you've got that going for you, make sure you have all of your code and your data and</a> <a data-t="207">all of the artifacts separate from the outputs that you might want to generate.</a></p>

<p><a data-t="218">And then the last bit that is not necessarily about Docker itself, although I'll spend most</a> <a data-t="223">of my time talking about Docker, is figure out some way to describe your computing environment</a> <a data-t="229">as well as you can so that someone in the future can actually run your analysis or then</a> <a data-t="234">extend it one way or another.</a></p>

<p><a data-t="232">And so to make your compendium shareable and useful, you have to have four separate things.</a> <a data-t="239">One is make sure there's a license that tells people how to use it.</a> <a data-t="244">Make sure it's under some kind of version control, preferably Git.</a> <a data-t="247">Make sure there's enough metadata as part of your compendium.</a> <a data-t="251">And then the last thing, which is often ignored everywhere, is that you need to have a long-term</a> <a data-t="256">archive.</a> <a data-t="257">So GitHub is not a long-term archive, but Zenodo.org, for example, is a long-term place</a> <a data-t="263">where you can deposit your compendium so that it continues to persist even after GitHub</a> <a data-t="268">disappears or any other services disappear.</a></p>

<blockquote><a data-t="257">So GitHub is not a long-term archive, but Zenodo.org, for example, is a long-term place</a> <a data-t="263">where you can deposit your compendium so that it continues to persist even after GitHub</a> <a data-t="268">disappears or any other services disappear.</a></blockquote>

<h3>Using the R package structure</h3>

<p><a data-t="272">So it turns out that the R package structure is already a very fantastic way to organize</a> <a data-t="277">a compendium, and there's not really very much that you need to do.</a> <a data-t="282">So many of you in the room are package developers or everyone in this room is a package user.</a> <a data-t="290">And so if you've ever looked at the contents of an R package, you might see a description</a> <a data-t="293">file that looks like this.</a> <a data-t="295">This is a standard Debian control format that contains a lot of generally useful metadata.</a> <a data-t="300">It has a title, an author, a version number, a date, and then somewhere in the bottom it</a> <a data-t="305">contains useful bits like these are the dependencies for this project.</a></p>

<p><a data-t="309">But since everybody is binge-watching Marie Kondo on Netflix, I can make a Marie Kondo</a> <a data-t="314">reference and say, Marie Kondo, the shit out of this file, and just make it a very, very</a> <a data-t="318">minimal description file.</a> <a data-t="321">You just need to say that it's a compendium, have a title, possibly a version number, and</a> <a data-t="325">just your dependencies that are either on CRAN or CRAN plus GitHub.</a></p>

<p><a data-t="332">Simply adding this file to a project and calling it a description file makes it a compendium,</a> <a data-t="337">and you already have a research compendium now.</a> <a data-t="340">And once you turn something into a compendium, now you can take advantage of DevTools and</a> <a data-t="345">Friends.</a> <a data-t="346">So DevTools can help you make sure that your packages or your compendium can be installed.</a> <a data-t="352">You can take advantage of all the tools that help you generate documentation and websites</a> <a data-t="357">and also link everything up to continuous integration services to make sure that your</a> <a data-t="361">compendium is still working over time.</a></p>

<p><a data-t="366">So how do you set up a compendium?</a> <a data-t="368">Depends on the scale of your project.</a> <a data-t="370">If you have a very simple analysis that is just a simple R Markdown notebook, you can</a> <a data-t="375">have this very minimal structure, just a description, a readme file, a license, and then the key</a> <a data-t="381">bits here that are different than a package are that you have a folder containing your</a> <a data-t="385">R Markdown file and then a data folder containing, say, a small amount of data.</a></p>

<p><a data-t="390">And this would work for a small project or a small report, but then for a lot of you,</a> <a data-t="395">you might end up with a slightly more complex compendium.</a> <a data-t="399">So the only thing that's different here from the previous version is that you have decided</a> <a data-t="403">to write a few custom functions for this project.</a> <a data-t="406">And this is a situation where these functions are not broadly useful outside the scope of</a> <a data-t="411">this project, which is why you haven't written a package.</a> <a data-t="415">So these exist within the compendium, and now you have a namespace file that captures</a> <a data-t="420">some of these dependencies.</a></p>

<p><a data-t="422">But for a lot of you, your actual real-world long-term research project is going to be</a> <a data-t="426">far more complex.</a> <a data-t="428">So in this case, you have taken all of your functions and moved those out into a separate</a> <a data-t="432">package except for a few that are critical only to this project.</a> <a data-t="436">But the new elements that you'll start to see are that you have issues with keeping</a> <a data-t="440">track of all of the outputs that you have to generate, all of the scripts that you have</a> <a data-t="444">to deal with.</a> <a data-t="445">So you need some sort of workflow.</a> <a data-t="448">And in this case, that is a makefile.</a> <a data-t="450">And then because your project's dependencies can change over time and packages can change</a> <a data-t="455">over time and the same package can have API changes over time, you need to capture your</a> <a data-t="460">computing environment.</a> <a data-t="461">And in that case, I have a Docker file.</a></p>

<h3>Managing data</h3>

<p><a data-t="470">So you will have to decide what level of complexity you want for your compendium.</a> <a data-t="476">And so for the rest of my talk, I want to focus on three separate components of a compendium</a> <a data-t="476">and how to think about them and how to use tools for that particular purpose.</a> <a data-t="480">I'd like to talk about data.</a> <a data-t="482">I'm from industry.</a> <a data-t="483">I don't have any good answers or solutions from you.</a> <a data-t="485">You've already got a data engineering team that can help you.</a> <a data-t="489">So the advice that I have is mostly for small to medium-sized projects.</a> <a data-t="493">The second one is how to isolate your computing environment.</a> <a data-t="496">And then the third one is talking about simple and complex workflows.</a></p>

<p><a data-t="501">So I'll start with data.</a> <a data-t="502">So how do you manage data in the context of, like, a small to a medium-sized project?</a> <a data-t="507">Well, a very simple answer is if the data are small enough, they're just tabular data,</a> <a data-t="512">you can stick them inside of your compendium itself.</a> <a data-t="516">Because GitHub lets you have files up to 150 megabytes each, and a compendium is not a</a> <a data-t="520">package that is going to go on CRAN, you can just have arbitrary data files.</a> <a data-t="525">And if you want to keep things very useful, you can even have a data raw folder that contains</a> <a data-t="529">the raw data and then some scripts that turn it into useful data.</a></p>

<p><a data-t="533">If you write a package alongside your compendium and you want it to go on CRAN, you can include</a> <a data-t="538">data in there, but your overall package size cannot exceed 5 megabytes.</a> <a data-t="544">Nick Tierney and I started poking around all the data that exists inside of CRAN packages,</a> <a data-t="548">and right now about 37% of them have a ton of data in there, and some projects have hundreds</a> <a data-t="553">of tabular data files that are shipped inside of an R package.</a> <a data-t="557">So that is one simple, easy solution that is often overlooked.</a></p>

<p><a data-t="562">Another one that I want to talk about is this fun little package by Carl Bettinger</a> <a data-t="565">called Piggyback, and Piggyback allows you to attach arbitrary files, these can be any</a> <a data-t="572">kind of files, up to 2 gigabytes each, and as many files as you would like to a GitHub</a> <a data-t="578">release.</a> <a data-t="579">So what is a GitHub release?</a> <a data-t="581">If you have written software before and you release a new version of your software, you</a> <a data-t="585">can tag a release on GitHub.</a> <a data-t="587">When you do that, you get binaries attached, and Piggyback just allows you to attach more</a> <a data-t="594">data files on to that release, and the functionality for this package is very, very simple.</a> <a data-t="600">There are only five functions.</a></p>

<p><a data-t="601">You can create a new release for your repo, tag it with a version number, and then just</a> <a data-t="606">upload whatever files that you would like to upload, and when you do that and go to</a> <a data-t="611">the release section of your GitHub repo, it's just your repo slash release, or releases,</a> <a data-t="618">you can see these files show up there.</a> <a data-t="620">So you can set up your compendium in such a way that when you're running the code, you</a> <a data-t="625">can have all these files downloaded back straight from GitHub, and because GitHub is a very</a> <a data-t="629">fast CDN, you can get access to these files pretty easily, no matter where you're running</a> <a data-t="635">your compendium.</a></p>

<p><a data-t="637">If you've got a medium data situation where you have a large number of highly compressed</a> <a data-t="643">text files that contain data locally, you can use this package called ArcDB that is</a> <a data-t="648">built upon </a><a href="/software/dbi/">DBI</a><a data-t="648">, and it allows you to chunk the data using your memory limits in and out</a> <a data-t="656">of any database backend that DBI supports.</a> <a data-t="659">So this is a nice way to either get lots of data back from a database, or if you've already</a> <a data-t="665">acquired data through some other means, this is a way to use the limitations of your setup</a> <a data-t="671">to get data into a database.</a></p>

<h3>Isolating your computing environment</h3>

<p><a data-t="676">The next bit I want to talk about is isolating your computing environment, and this is really</a> <a data-t="680">important because even I can never actually run any of my old analysis, especially things</a> <a data-t="685">that involve </a><a href="/software/ggplot2/">ggplot2</a><a data-t="685"> from last year or the year before or the year before that.</a> <a data-t="690">I have all the packages.</a> <a data-t="691">The code looks the same, but nothing really runs.</a></p>

<p><a data-t="695">This is a paper in Nature Biotechnology that talks about how having just different versions</a> <a data-t="699">of R can produce entirely different results from your same analysis or your same code.</a> <a data-t="705">So those of you that are familiar with Docker, Docker is a very nice way to just containerize</a> <a data-t="710">your existing environment so that even if you change machines or upgrade machines or</a> <a data-t="715">move organizations, you should still be able to run the same code with the same set of</a> <a data-t="719">dependencies and then get the same output back.</a></p>

<p><a data-t="724">And adding a Docker file to your existing R project is quite easy these days.</a> <a data-t="729">There's an R package called containerit.</a> <a data-t="731">It's not on CRAN yet.</a> <a data-t="733">It's only on GitHub, but it can take any arbitrary folder or get repo and generate a nice Docker</a> <a data-t="739">file for you.</a> <a data-t="741">The Jupyter project from the Jupyter Notebook team has a more general solution called repo</a> <a data-t="746">to Docker that will do it for either a Python project or an R project or projects that contain</a> <a data-t="751">both.</a></p>

<p><a data-t="753">And adding a Docker file to your project has many advantages.</a> <a data-t="756">One very simple straight up one is that you can then Dockerize that particular project,</a> <a data-t="761">launch a container, run your analysis, and then get back out.</a> <a data-t="765">But I want to spend a few minutes talking about this fantastic open source project called</a> <a data-t="769">Binder, or I should rather say Binder Hub.</a></p>

<p><a data-t="773">Binder Hub is an open source project that allows you to launch a live notebook from</a> <a data-t="779">any GitHub repo on demand and let someone else run this analysis, poke through your</a> <a data-t="786">code, make some changes, see the results for themselves, and then when they close the browser</a> <a data-t="790">window, that particular instance is immediately killed off.</a></p>

<p><a data-t="806">And because it's easier to show you how this thing</a> <a data-t="809">works, this is an example real project from my collaborator Carl.</a> <a data-t="813">I'm not on this particular collaboration.</a> <a data-t="823">There's a button called launch Binder, which once I click will either build a Docker image</a> <a data-t="823">for the first time or if it's already found an image from before, pull it out of the cache</a> <a data-t="829">and then go on to install any other additional R packages that you might need that is not</a> <a data-t="834">already captured in the Docker file and then drop you into an </a><a href="/software/rstudio/">RStudio</a><a data-t="834"> server.</a></p>

<p><a data-t="839">And as it's par for a live demo, it doesn't actually work as quickly as you expect it</a> <a data-t="845">to.</a> <a data-t="846">But in about 30 seconds or maybe a few minutes, this will actually drop you into a RStudio</a> <a data-t="852">server.</a> <a data-t="853">I'm just going to let that run while I go back to my talk.</a></p>

<p><a data-t="864">And so you end up in an RStudio server now, and here are the two beautiful things that</a> <a data-t="865">are happening.</a> <a data-t="865">All of the R packages that you need for this analysis have already been installed, and</a> <a data-t="869">all of your scripts, data, and code have already been copied in.</a> <a data-t="872">And there's nothing more to do other than to open up a notebook and then start running</a> <a data-t="876">through it.</a></p>

<p><a data-t="878">And the way this works is that JupyterHub does 80% of this work, and BinderHub just</a> <a data-t="883">does a little bit of coordination.</a> <a data-t="885">It reads in a repo, sees if there's a Docker file, if there's a Docker file, it checks</a> <a data-t="890">to see if there's a Docker image.</a> <a data-t="892">If not, it builds one and then hands it off to JupyterHub to allocate some resources to</a> <a data-t="896">it and launch a server on demand.</a> <a data-t="899">This service is free.</a> <a data-t="900">And then mybinder.org is an example of a BinderHub.</a></p>

<p><a data-t="905">And so how do you set this up for your R project?</a> <a data-t="908">Well, there are many different ways to do this, and in fact, I'm not going to go through</a> <a data-t="911">all the different ways because that is going to be part of my readme.</a> <a data-t="915">A simple way is just to add a text file called runtime.txt and then add R and then a particular</a> <a data-t="921">date that corresponds to the version of R that you might want to capture.</a> <a data-t="927">This also means that the Rocker project will pull all the CRAN packages that correspond</a> <a data-t="933">to this exact date.</a> <a data-t="934">So if your project moves along or your system moves along, everything will be locked in</a> <a data-t="938">time.</a></p>

<p><a data-t="939">And then you have another text file called install.r that just contains a list of package</a> <a data-t="944">install commands.</a> <a data-t="945">This is the simple, most basic way to set this up.</a> <a data-t="949">Then you can go to mybinder.org, paste the URL for your GitHub repo, click launch, add</a> <a data-t="954">that Binder button, and then you should be good to go.</a></p>

<p><a data-t="959">So this is the basic approach, adding an install.r, runtime.txt, and then if you want to add any</a> <a data-t="968">Linux dependencies, you can add an app.txt.</a> <a data-t="971">The only downside is it's very slow and every time you change anything, it will take hours</a> <a data-t="975">to rebuild your Docker image.</a> <a data-t="977">But then there's a faster way to do this is by adding a Docker file using one of those</a> <a data-t="981">two packages, like container.it or repo.docker.</a> <a data-t="985">And because we're going to use a Docker image that already contains tidyverse, things will</a> <a data-t="990">move along very quickly.</a></p>

<p><a data-t="992">And then the pro option is because now you're all experts on Research Compendia and you</a> <a data-t="996">already have a description file, then you just need to add a very, very short Docker</a> <a data-t="1001">file.</a> <a data-t="1002">It could even just be four lines that says start at this base image, move all my files</a> <a data-t="1006">into my container, and then install these few additional packages for my description.</a> <a data-t="1013">Those of you that don't know about the Rocker project, it contains daily versions of different</a> <a data-t="1018">instances of R, including ones that contain RStudio and then specialized ones for geospatial</a> <a data-t="1023">data and so on.</a></p>

<h3>Workflows with Drake</h3>

<p><a data-t="1026">The last bit of that I want to talk about are workflows.</a> <a data-t="1029">So I imagine that a lot of people in this room are quite familiar with Makefiles, and</a> <a data-t="1035">Carl Broman, who's here in this room, has a fantastic tutorial on Makefiles.</a> <a data-t="1039">And so having a Makefile somewhere inside of your compendium allows you to keep track</a> <a data-t="1043">of all of your scripts and all of your inputs and outputs.</a> <a data-t="1046">But those of you that have used Makefiles probably know that it can get cumbersome very,</a> <a data-t="1051">very quickly.</a></p>

<p><a data-t="1056">The wildcard syntax is not intuitive to remember.</a> <a data-t="1059">Like I have to Google this thing every damn time.</a> <a data-t="1059">And then it generates so much output that it just clutters your compendium every single</a> <a data-t="1063">time you run something.</a></p>

<p><a data-t="1065">So I want to give a shout out to a fantastic project called Drake, which is also part of</a> <a data-t="1070">ROpenSci.</a> <a data-t="1071">The author for this project, Will Landau, is in the room somewhere, and you should go</a> <a data-t="1074">talk to him if you're interested in talking about Drake.</a> <a data-t="1077">And what Drake is, is if you want to think about a piece of software that mimics Make</a> <a data-t="1084">but it's very R-centric, that's what Drake is.</a> <a data-t="1088">And Drake stands for Data Frames in R for Make.</a></p>

<p><a data-t="1092">And of course, it is impossible for me to do a demo in the amount of time that I have</a> <a data-t="1097">left.</a> <a data-t="1097">But I want to mention that what Drake allows you to do is generate these very, very complex</a> <a data-t="1101">Makefiles without writing them out.</a> <a data-t="1103">It does beautiful wildcard expansions and then takes advantage of numerous parallel</a> <a data-t="1109">backends including things like future.</a> <a data-t="1112">It gives you a nice dependency graph with all of your inputs expanded.</a> <a data-t="1117">And then it'll also estimate runtimes.</a> <a data-t="1119">And because it uses a system called Store, S-T-O-R-R, as the backend, it does not clutter</a> <a data-t="1125">your entire environment and allows you to quickly read outputs back as and when you</a> <a data-t="1128">need them.</a></p>

<p><a data-t="1130">And then this would be a beautiful dependency graph that it generates.</a> <a data-t="1134">It's an interactive graph.</a> <a data-t="1135">You can actually see runtimes, missing objects, and all the relationships in your workflow.</a> <a data-t="1142">So having Drake as part of your compendium instead of a Makefile can be a nice way for</a> <a data-t="1147">you to scale very large projects.</a> <a data-t="1149">And then the author will actually develop this for a real-world use case for a very</a> <a data-t="1153">complex Bayesian analysis project.</a></p>

<h3>Take-home messages</h3>

<p><a data-t="1157">So I have two take-home messages for you.</a> <a data-t="1159">One is that turning your project into a research compendium is as simple as throwing in a disk</a> <a data-t="1165">description file in there.</a> <a data-t="1166">And it can be a very, very minimal description file.</a> <a data-t="1169">The tiny bits of metadata, dependencies.</a> <a data-t="1172">And then you can use these modern tools like Binder to launch notebooks that make it more</a> <a data-t="1178">accessible for people trying to understand your analysis.</a></p>
