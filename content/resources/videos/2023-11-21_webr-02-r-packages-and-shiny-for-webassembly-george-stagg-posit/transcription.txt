<h3><a href="/software/shiny-r/">Shiny</a> for WebAssembly with <a href="/software/httpuv/">httpuv</a> shimming</h3>

<p><a data-t="480">And finally, the last thing I want to talk about with R packages is Shiny and httpuv,</a> <a data-t="486">which is one of the support packages that Shiny uses.</a> <a data-t="492">So the httpuv package, again, is based on network connections.</a> <a data-t="492">It's a server that can listen for HTTP and </a><a href="/software/websocket/">WebSocket</a><a data-t="492"> traffic</a> <a data-t="502">and then respond to those requests.</a> <a data-t="510">So a package shim has been created for WebR,</a> <a data-t="517">which is available on the RWASM organization, if you want to read the source code for that.</a> <a data-t="517">And what that is, is it's a package that acts like httpuv, but is re-implemented to be used with WebR.</a> <a data-t="528">And in particular, what it's doing is taking advantage of a JavaScript feature known as service workers.</a> <a data-t="534">And service workers are nice because they can act as network proxies.</a> <a data-t="534">So normally when that traffic that httpuv would handle over the network is instead being handled entirely client-side, inside a JavaScript service worker instead.</a> <a data-t="546">And that service worker acts as our network server when really it's all running client-side.</a></p>

<p><a data-t="553">With this shim, Shiny apps are able to run under WebR, and in particular, they can run entirely client-side.</a> <a data-t="560">So the way this works is that the R Shiny server software thinks it's talking to httpuv through its normal R package API.</a> <a data-t="572">And at the other side, we have the Shiny web client that the user is interacting with.</a> <a data-t="572">That thinks it's talking over the network to the Shiny server, but instead that is being intercepted, as I say, by the JavaScript service worker.</a> <a data-t="585">And then that proxy in the middle between the Shiny server and the Shiny client converts those network requests, captures them, converts them, and sends them to WebR.</a> <a data-t="600">And WebR works with httpuv to handle those messages.</a> <a data-t="600">And the responses from Shiny, again, are sent back to the client at the front end through that network proxy,</a> <a data-t="606">through that service worker.</a> <a data-t="610">So through this package shimming process, we're able to run Shiny apps entirely within the browser with both the front end and the back end server, all running inside a JavaScript worker.</a></p>

<p><a data-t="629">This is a similar scheme, by the way, to how </a><a href="/software/shinylive/">shinylive</a><a data-t="629"> for Python works.</a> <a data-t="634">It's not exactly the same, but it is a similar scheme.</a> <a data-t="634">And so we're hoping that in the near future, we'll be able to integrate this kind of thing directly into shinylive to create a shinylive for R,</a> <a data-t="646">which will take the complication of deploying apps like this away so that the general user will be able to use shinylive to deploy R Shiny apps in this way,</a> <a data-t="655">rather than having to learn the intricacies of how to use a JavaScript service worker or deploy their own application in this way.</a></p>
