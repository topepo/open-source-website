<h3>What is renv?</h3>

<p><a data-t="137">So first, what is renv?</a> <a data-t="141">And here is the sort of sales pitch of what renv is.</a> <a data-t="145">It's a toolkit used to manage project local libraries of R packages.</a></p>

<p><a data-t="152">So you can use renv to make your projects more isolated, isolated in that every project gets its own library of R packages, because each project has its own library, you don't have to worry about updating packages in that library, potentially breaking your other projects.</a></p>

<p><a data-t="164">It'll help you make your projects a bit more reproducible or portable, because it captures the state of your R library and the packages installed in that library, it captures those into a lock file, and that lock file is kind of like a shopping list of the packages used in your project, and renv knows how to use that thing to recreate your library in the future.</a></p>

<p><a data-t="189">And that ties into the reproducibility story.</a> <a data-t="193">You use snapshot to create that lock file, and later you use restore to restore your library.</a></p>

<p><a data-t="201">Now, renv tries to prescribe a default workflow that just works for the average user, but remains flexible enough that alternate workflows can still be built on top of renv.</a> <a data-t="216">And so one of the hardest things for me in building renv is just everyone has a different idea in how projects should be managed, how dependencies should be managed, and ultimately renv has to take an opinionated take on how to approach this problem that should at least work for the average user.</a></p>

<h3>Understanding library paths</h3>

<p><a data-t="232">And so before we talk about renv, I just want to give a lay of the land of what it looks like when you're working on a project without renv.</a></p>

<p><a data-t="239">So you've started a new project.</a> <a data-t="243">You're ready to use dplyr to analyze some data, and so you're going to load and use in your project.</a> <a data-t="248">You're calling library dplyr.</a> <a data-t="250">And so what happens when we actually run this code?</a> <a data-t="254">R is going to search your library paths for a package called dplyr that you've installed, and then it's going to load it.</a></p>

<p><a data-t="261">So the natural questions that arise from that are what is a library path?</a> <a data-t="265">What are the active library paths?</a> <a data-t="267">And how does R search those library paths when loading a package?</a></p>

<p><a data-t="274">So first, a raise of hands.</a> <a data-t="275">Who thinks they can tell me what a library is?</a> <a data-t="278">You don't have to tell me, but just if you think you know what it is.</a></p>

<p><a data-t="292">I'm happy to say that the definition is quite simple here.</a> <a data-t="296">It is simply a directory.</a> <a data-t="298">It's a directory in which packages get installed.</a> <a data-t="301">Nothing more, nothing less.</a> <a data-t="303">Let any remaining mystique be dispelled.</a> <a data-t="305">It's just a directory.</a> <a data-t="306">It's a folder on your system.</a> <a data-t="308">It's where packages go to live and get loaded.</a></p>

<p><a data-t="312">Now where a bit of the extra complexity comes in is that each R session can be configured with multiple library paths.</a> <a data-t="322">That is, a bunch of folders where R will search for your installed packages.</a> <a data-t="327">You might have seen the .libpaths function, which is used to find those library paths.</a> <a data-t="333">So for example, on my system, I've got two of these.</a></p>

<p><a data-t="338">And just to give some terminology for these libraries, the first one you see is most often called the user library, because it's your library.</a> <a data-t="348">It's where the packages that you download and install go.</a> <a data-t="352">The last one you have in that list is the system library.</a> <a data-t="356">And that's most typically where the packages that come with R are installed.</a></p>

<p><a data-t="361">And if you're on Linux, you might see another set of libraries, which are often called site libraries.</a> <a data-t="371">You can think of these as the library paths that your administrator might take care of for you.</a> <a data-t="377">So if you're in some organization, your R administrator might want to install some set of packages in the site libraries that would then be available to all users on the system.</a></p>

<p><a data-t="394">And then next, so when R wants to load a package, how does it search these things to find it?</a></p>

<p><a data-t="424">And so what happens is the first installation of the package that is discovered and loaded, no size say the first.</a> <a data-t="430">So if you've ever, say, run into a situation where you have multiple copies of a package installed, say one in your user library, one in your system library, the first one that it finds, the one in your user library, is the one that gets loaded.</a></p>

<p><a data-t="451">But if you've ever thought, like, wait, where is this package actually coming from?</a> <a data-t="456">You can use the find.package to ask R, hey, where does this package that I want to use, where does it actually live?</a> <a data-t="463">And so for me, dplyr, I've made sure to install it in my user library.</a></p>

<p><a data-t="467">Now, I should say, if you're on macOS, you can actually install CRAN packages directly into your system library, which I would argue is not a great idea because you want to have this separation of packages that belong to you that you've installed versus packages that belong to your R installation.</a> <a data-t="493">So mixing them up can cause some surprises.</a></p>

<h3>The challenge of shared library paths</h3>

<p><a data-t="496">All right, so what is the challenge here?</a> <a data-t="500">So by default, each R session is going to use the same set of library paths.</a> <a data-t="505">So you've got this library, it's really multiple libraries, a set of library paths, and all the projects on your system, unless you're already managing your library paths in some way, are going to go and use those same library paths.</a></p>

<p><a data-t="519">It means that, say, if you had dplyr 0.8.2 in that library, that's the package that you're using in every one of your projects.</a></p>

<p><a data-t="529">But of course, different projects can have different package dependencies.</a> <a data-t="533">For example, suppose project one, maybe something you were working on a while ago, that uses dplyr 0.7.8.</a> <a data-t="541">And that's, say, an older version.</a> <a data-t="543">You got everything tidy, wrapped up, maybe it's associated with a paper that you published, and you're thinking, okay, I'm done with this, it works, I don't want to change it.</a></p>

<p><a data-t="552">Project two, on the other hand, you're using some more modern version of the package, you've upgraded to dplyr 0.8.2.</a> <a data-t="560">And say, if you were a developer, maybe you're even using the development version of dplyr in another project.</a> <a data-t="565">Unfortunately, in the current state of the world, these projects all share the same library path.</a> <a data-t="571">So if you were to install one of these packages, you're changing the version of dplyr that's used in all three of those projects.</a></p>

<p><a data-t="579">So in this world, if you were trying to switch between projects, you'd have to go back and say, okay, wait, I need to go back and install dplyr 0.7.8 for project one.</a> <a data-t="588">Oh, wait, but now I need to reinstall 0.8.2 for project two.</a> <a data-t="592">And as you can imagine, this could spell disaster.</a></p>

<h3>renv's solution: project local libraries</h3>

<p><a data-t="623">So the solution, at least the solution posed by renv, is to ensure that each project gets its own unique library of packages.</a> <a data-t="633">By using project local libraries, you can rest assured that upgrading packages in one project will not risk breaking your other projects.</a></p>

<p><a data-t="642">And so it's from this idea, the use of project local libraries, that the renv package is born.</a> <a data-t="648">We give each project its project local library, we make it simple and straightforward for R sessions to use that local library, we provide tools for managing the R packages installed in those project local libraries, and we try to make that experience as seamless as possible.</a></p>

<blockquote><a data-t="633">By using project local libraries, you can rest assured that upgrading packages in one project will not risk breaking your other projects.</a></blockquote>

<h3>The renv workflow</h3>

<p><a data-t="670">And so if you've seen PackRat, this will feel familiar.</a> <a data-t="675">Once again, the API is quite similar, but I'll walk through it.</a> <a data-t="678">The first step in activating renv for a project is just calling renv init.</a> <a data-t="683">And one thing that it does that is different from PackRat is it actually forks the state of your current world of R libraries into a project local library.</a></p>

<p><a data-t="697">So the idea is you're working on a project, you have it working.</a> <a data-t="701">Say you haven't adopted renv as your default workflow or you don't manage your library paths in a specific way, but you're saying, okay, the state of the world is good right now, I want to capture that in this project.</a> <a data-t="717">So what you want to call is you call renv init, and it takes the packages you have installed in that library and moves them into the project library.</a> <a data-t="725">So it's like forking that state into your project.</a></p>

<p><a data-t="731">The other thing is infrastructure-related is we create a project local .r profile.</a> <a data-t="735">And that basically makes sure that every time you start R in that session, renv automatically makes sure you use that project library.</a> <a data-t="742">So you don't have to worry about managing your library paths.</a> <a data-t="745">Renv does it for you automatically when you launch R.</a></p>

<p><a data-t="750">And this mirrors what I said before around making sure that after you call renv init, you can work exactly the same as you were before, with the only difference being now you have your own project local library.</a></p>

<p><a data-t="772">There are two main differences you'll see after you call renv init in a project.</a> <a data-t="776">The first, you'll see a nice little banner printed saying, like, OK, renv has activated this project for you, and this is the version of renv that it's using.</a> <a data-t="784">And the second thing is, as I've said, the library paths will now be changed.</a></p>

<h3>Snapshot and restore</h3>

<p><a data-t="793">So that is the first goal, making it easy to use project local libraries.</a> <a data-t="798">The second goal is to make it easy to save and load the state of those libraries.</a> <a data-t="803">Or in the parlance of renv, snapshot to save the state of a library and restore to reinstall those packages into a library.</a></p>

<p><a data-t="811">You can capture the state of your project library using snapshot.</a> <a data-t="815">And so when you call snapshot, it'll give you a little log of what's changing, of what dependencies it's capturing when you call snapshot.</a> <a data-t="823">So in this case, I'm just capturing three packages for a project, let's say, and it's capturing from CRAN the markdown package, the </a><a href="/software/rmarkdown/">rmarkdown</a><a data-t="823"> package, and the yaml package.</a></p>

<p><a data-t="837">And this star is just as a placeholder to say, we didn't know about this before.</a> <a data-t="842">Now we know that using version 1.1 of markdown, now using 2.1 of the rmarkdown package, 2.2.0 of the yaml package.</a> <a data-t="850">And I say, do you want to proceed?</a> <a data-t="852">Yes.</a> <a data-t="853">Please write those.</a> <a data-t="854">And it writes it to this thing we call the lock file.</a></p>

<p><a data-t="857">So we're serializing the state of your project library to a file called a lock file, which we'll name on the file system as renv.lock.</a> <a data-t="867">So this thing is just a text file.</a> <a data-t="870">It's a shopping list.</a> <a data-t="872">It's the packages that you had installed in your library, their versions, and where they came from.</a></p>

<p><a data-t="876">So why do we want these lock files?</a> <a data-t="880">There's three cases that I outlined specifically here.</a> <a data-t="883">One is for time capsules.</a> <a data-t="885">This is the case where you've, OK, I'm done a project.</a> <a data-t="888">I want to make sure that this project still runs a year from now.</a> <a data-t="891">To do that, I want to make sure I know what packages I have, and I want to make sure those packages exist over time so I can revisit later.</a></p>

<p><a data-t="897">For collaborative workflows, if you're working with a set of collaborators, you want to probably make sure everyone is working with the same environment.</a> <a data-t="905">If people have different computer machines, one way to make sure you synchronize the environment is share the lock file, have everyone restore from that lock file, and that way you're sure everyone's using the same set of R packages.</a></p>

<p><a data-t="919">And also for deployment.</a> <a data-t="921">So if you're working on something locally and it works on your machine, you might use renv to give yourself some extra guarantees that it works on the machine you're deploying on by replicating your library on that other machine.</a></p>

<p><a data-t="936">So given a lock file, you create your lock file with snapshot, and you restore the state of your library later with restore.</a> <a data-t="946">And so restore will give you a very similar kind of output.</a> <a data-t="949">It'll tell you what packages are going to be installed, how they differ from the package that was already installed, if any, and what's the version.</a></p>

<p><a data-t="957">And fortunately, renv is able to restore from a number of sources.</a> <a data-t="961">If you've used remotes or dev tools, it basically understands all those same sources, CRAN, Bioconductor, GitHub, GitLab, Bitbucket.</a> <a data-t="969">If you have another remote source that you'd like renv to support, you can let me know on the GitHub issue tracker, and we can see if that can become a possibility.</a></p>

<p><a data-t="979">Another big thing is I've made an effort to ensure that you can authenticate with private repositories.</a> <a data-t="983">So if you have your own, say, CRAN-like repository hidden behind some kind of authentication mechanism, renv has some tools to make it easier to authenticate.</a></p>

<p><a data-t="994">So these are the three main features of renv, and I think these are the three things you need to know if you want to get started.</a> <a data-t="1002">The rest of the talk will be kind of extra on top of that.</a> <a data-t="1005">Init to initialize your project, give yourself a project local library, snapshot to save the state of your library once you are ready to save the state, and restore if you need to restore your project library from a previously generated lock file.</a></p>

<h3>The global cache</h3>

<p><a data-t="1031">And so one major issue with project local libraries is the duplication of identical packages across projects.</a> <a data-t="1038">So if you've used PackRat before, you've probably seen this issue before.</a> <a data-t="1042">So if you had ten projects that used dplyr 0.9.2, then you have ten project libraries that also have dplyr 0.9.2 installed.</a> <a data-t="1052">And this can be costly, both in terms of disk space to use and installation time, especially if, say, you're on a Linux machine, you're installing these things from sources.</a></p>

<p><a data-t="1063">So imagine having to install the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="1063"> every time you start a new project.</a> <a data-t="1070">And so this is what I got from running it on my Linux VM, and it took just over six or seven minutes, which is too long when you want to get started on something.</a> <a data-t="1083">You know, when you're starting a new project, you want to just get going.</a> <a data-t="1087">You don't want to spend ten minutes just to get up and running.</a></p>

<p><a data-t="1092">So the way renv solves this problem is with a global cache.</a> <a data-t="1097">And what this basically is is a giant bucket of all the packages you have installed.</a> <a data-t="1101">And in each of your project libraries, rather than having the actual installation of that package, it's a link into that cache.</a> <a data-t="1109">So the idea is that rather than having ten installations of dplyr 0.9.2, you install that package once, you put it in the cache, and then renv knows how to pull that out whenever it needs to.</a></p>
