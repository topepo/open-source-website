<h3>What is OpenTelemetry?</h3>

<p><a data-t="206">This is all being done through OpenTelemetry.</a> <a data-t="208">And OpenTelemetry, according to their website, is highly high quality, ubiquitous and portable telemetry to enable effective observability.</a> <a data-t="238">If we're just looking at our app the way we were the first time, we can see what's happening on the surface but really what's happening underneath where the computations are being done, that's the important part to the developers and during production.</a> <a data-t="272">OpenTelemetry provides a set of APIs and libraries and agents and instrumentation to provide observability for your applications.</a> <a data-t="282">In this case, OpenTelemetry is being integrated into the R ecosystem and is currently under development.</a></p>

<p><a data-t="293">So let's look at this a little bit more generically.</a> <a data-t="295">OpenTelemetry is really good at a burst of an action such as a user clicked the website or </a><a href="/software/shiny-r/">Shiny</a><a data-t="295">'s reactivity just started.</a> <a data-t="313">After this action you'll have different services be engaged, in this case the green, then the blue, then the yellow, which then calls a database and then they undo their work since they're done processing.</a> <a data-t="337">So we can see how long the blue, then how long the yellow take, and then finally accessing that database how long of your computation time is performing that.</a></p>

<h3>OpenTelemetry vs <a href="/software/profvis/">Profvis</a> and <a href="/software/reactlog/">ReactLog</a></h3>

<p><a data-t="375">So you're probably seeing these flame graph looking things and you're like well, what about Profvis? What about ReactLog?</a> <a data-t="383">This is how I debug my applications already.</a> <a data-t="385">And Profvis it allows for local R code profiling and it also displays some memory usage as well.</a> <a data-t="395">In ReactLog you can replay all of the local reactivity that you have captured and it always displays a full dependency graph for every reactive component within your application.</a> <a data-t="415">So if you have a very large application you'll have a very large connected graph.</a> <a data-t="419">I highly recommend it if things aren't working the way you expect it.</a></p>

<p><a data-t="426">The bad part about both of these packages is that they only support the main R process.</a> <a data-t="430">So if you're using Mirai which does computing on a different R process we don't have access to that with Profvis or ReactLog.</a> <a data-t="439">But with OpenTelemetry we can pass through some extra headers and Mirai can do the proper reporting.</a> <a data-t="447">Also Profvis and ReactLog are not for production.</a> <a data-t="453">By definition both of those packages produce a memory leak.</a> <a data-t="457">They're constantly recording.</a> <a data-t="459">Even if we were to write out to disk you're going to run out of space eventually.</a> <a data-t="463">So both of these packages are great for local development, great for debugging, but I do not recommend them for production.</a></p>

<h3>The 2025 approach to Shiny in production</h3>

<p><a data-t="657">When looking at a lot of GitHub issues, I'm constantly referring back to </a><a href="/people/joe-cheng/">Joe Cheng</a><a data-t="657">'s keynote in 2019 of Shiny in production.</a> <a data-t="667">This is where he went through and described how you should iterate and try to make your apps in better performance but I would like to have this be updated now to be leveraging open telemetry so let's walk through that.</a> <a data-t="683">This is the 2025 version.</a></p>

<p><a data-t="685">First, enable open telemetry to see your span durations.</a> <a data-t="691">Second, let's look at those long spans that are run in R and use Profvis to see where that code is slow.</a> <a data-t="701">For when the code is slow, let's try to optimize it.</a> <a data-t="705">We can, one, move work outside of Shiny.</a> <a data-t="709">In the standard case, if you have consistent computing being done, maybe lift it outside into your global.R or try to do some preprocessing and have that be loaded before your server function.</a> <a data-t="723">Just relocating it sometimes makes a big, big difference.</a></p>

<p><a data-t="727">Two, make your code faster.</a> <a data-t="729">If you can't move it outside the function, instead maybe use DuckDB as a backend.</a> <a data-t="735">Even if it's still being processed in memory, it'll possibly be faster than what you have already with </a><a href="/software/dplyr/">dplyr</a><a data-t="735">.</a> <a data-t="743">Use caching.</a> <a data-t="745">If it applies, if you're returning the same answer for the same input, maybe we can add a layer of caching on top of that and then your spans will just disappear.</a> <a data-t="757">And then finally, use some non-blocking reactivity.</a> <a data-t="761">In the demo, I happened to have the tool call use Mirai to pull it off the main R process so that others could do work.</a> <a data-t="771">I wasn't showing it in the use case, but it did pull it off the main R process, and then that way it wasn't blocking others to stream or to do any calculations.</a></p>

<p><a data-t="781">And then repeat.</a> <a data-t="783">When you do make some of these changes, you may have to go back up to number two and see Profvis and reevaluate your app, because sometimes just by enabling caching, spans will just disappear and your computation will be very different in shape compared to the first time.</a></p>
