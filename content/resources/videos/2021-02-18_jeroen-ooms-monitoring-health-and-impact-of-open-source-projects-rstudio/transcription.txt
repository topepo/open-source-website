<h3>Choosing the right R packages</h3>

<p><a data-t="168">So, how do we discover and choose the best R packages? Because R is amazing and you can do</a> <a data-t="173">so many cool things with it and there's very few rules or limitations. But this also makes</a> <a data-t="178">our ecosystem sometimes feel a bit like the Wild West of research software, where it can be very</a> <a data-t="184">hard to find the good stuff or to judge if something is trustworthy.</a> <a data-t="190">Suppose you're a data</a> <a data-t="190">scientist and you need to do a particular analysis or you need to read a particular file format or</a> <a data-t="195">you need an HTTP client to interface to some open data web servers. How do you know what's out there</a> <a data-t="202">and how do you know if it's any good? Maybe you find an article or a blog post that mentions</a> <a data-t="207">an interesting package. So, you read the description and you maybe run an example and it seems to work.</a> <a data-t="213">So, problem solved, right?</a></p>

<p><a data-t="213">Well, as you probably know, anyone can create an R package. You just put</a> <a data-t="219">your code into the right format and then you pull it up on GitHub or you submit it to CRAN.</a> <a data-t="223">Now, CRAN will actually check if the package can be installed on Windows and Mac and Linux</a> <a data-t="227">and if it does not violate any policies. But that's about it. There's no gatekeeping or judgment on</a> <a data-t="234">the contents of the R package. And this is what makes open source software so great because anyone</a> <a data-t="240">can participate. But obviously, the quality of these contributions can vary.</a></p>

<p><a data-t="248">And what is perhaps</a> <a data-t="248">less obvious at first is that different package authors have very different ideas on what you can</a> <a data-t="255">expect from them in terms of maintenance and support and participation. And these things</a> <a data-t="261">become more relevant as you start using R more seriously. Now, of course, when you're just toying</a> <a data-t="266">around for a homework assignment or a one-off problem, you just do whatever works and move on.</a></p>

<p><a data-t="272">But if you start to really depend on a package, I think the dynamic changes a bit. Suppose you're</a> <a data-t="278">really going to build an R package in your dissertation research or your company's data</a> <a data-t="282">pipeline. At some point, you should probably ask yourself some questions like who made this package?</a> <a data-t="288">What's their background? Is this an expert? What other things have they made? And how can you trust</a> <a data-t="294">the results are correct? Has there been any peer review or scientific publication to accompany this</a> <a data-t="299">package? Is it reliable? Does it get tested a lot? Because if it works for one example, that doesn't</a> <a data-t="305">always mean it's going to work for any data. And what if you have a question about the software or</a> <a data-t="310">you find a bug? Is the maintainer going to be available to help you? Is the project still</a> <a data-t="315">actively developed? And what are the maintenance expectations? Is the package even going to be</a> <a data-t="321">around three years from now when you plan to finish your dissertation? And who else is using</a> <a data-t="326">the software? Is this like an established project? Is there a large user community where you can maybe</a> <a data-t="332">ask questions?</a></p>

<p><a data-t="339">So what you do not want is that you become fully dependent on a piece of software</a> <a data-t="344">in your research or in your business. And then at some point, a problem arises and you run into a</a> <a data-t="350">really bad bug or the thing is crashing or it's depending on some other thing that has been retired</a> <a data-t="356">or it just doesn't work anymore on the latest version of Windows or Mac. And then it turns out</a> <a data-t="362">there is no issue tracker. And then you contact the maintainer, but you get no response. Or worse,</a> <a data-t="368">you get a response like, well, this package is part of a research project that no longer exists</a> <a data-t="374">and I've since then transferred to another university and I don't have time to work on</a> <a data-t="377">this anymore.</a></p>

<p><a data-t="377">And this happens a lot. I've certainly been there. And it's good to remember</a> <a data-t="383">that the package author really doesn't owe you anything. Most maintainers are very nice and</a> <a data-t="389">very friendly and they're very interested in helping you while also using your feedback to</a> <a data-t="394">improve their package. That is what open source software is all about. But sometimes you find out</a> <a data-t="400">that some software that you were using may actually not be in very good shape.</a></p>

<blockquote><a data-t="383">the package author really doesn't owe you anything. Most maintainers are very nice and</a> <a data-t="389">very friendly and they're very interested in helping you while also using your feedback to</a> <a data-t="394">improve their package. That is what open source software is all about.</a></blockquote>

<h3>The lifecycle of open-source software</h3>

<p><a data-t="400">I think a lot of this</a> <a data-t="400">also comes down to the unique life cycle of research software. All software has a life cycle,</a> <a data-t="406">but for open source projects, this is often somewhat unclear and unpredictable. Most</a> <a data-t="412">projects start out as an experiment and then a small fraction of those projects, if it works well</a> <a data-t="417">and the maintainer has enough time to put into it, it may gradually develop into a mature established</a> <a data-t="423">piece of software. But eventually one day all software becomes obsolete and it gets retired</a> <a data-t="431">or replaced with something better. And in commercial projects, there's usually a license</a> <a data-t="435">that states that this product is officially supported until at least 20 blah, blah, blah.</a> <a data-t="441">But in open source projects, there is no such thing. Things are supported basically for the</a> <a data-t="447">extent and the duration that the author has an interest in maintaining that thing.</a></p>

<p><a data-t="453">For example, a lot of research software only exists as a proof of concept to accompany a</a> <a data-t="459">scientific paper. And once the thing has been published, the author has really no intention</a> <a data-t="464">of ever touching the software again. So if you want to actually build on that software, then</a> <a data-t="468">you're basically on your own. But many R packages these days, they are the exact opposite and they</a> <a data-t="475">are not published for their scientific merit, but they do something very useful. And it is in the</a> <a data-t="480">interest of the package author to get as many people as possible to use the thing to maximize</a> <a data-t="486">the impact of the work.</a> <a data-t="492">So when we're using a piece of open source software, it can be helpful</a> <a data-t="492">to think for a second about where in the lifecycle this project is. Is this an experiment or a mature</a> <a data-t="500">project? Is it still actively being developed or is it something of the past?</a></p>

<h3>Indicators of a healthy project</h3>

<p><a data-t="506">So how should we</a> <a data-t="506">handle this? What are things you could look for when you're shopping for R packages? What are</a> <a data-t="511">indicators that reveal something about the health of a project or that may give you a sense of what</a> <a data-t="518">to expect? And what are red flags to avoid common pitfalls? I think we can roughly distinguish</a> <a data-t="525">three categories of indicators. The first category of indicators are technical indicators.</a> <a data-t="531">And those are things that we can measure relatively objectively. So for example, we can</a> <a data-t="537">look at the dependency network and the CRAN homepage of a package already shows you the</a> <a data-t="541">reverse dependencies of a package. And so those are other packages that depend on that package.</a> <a data-t="546">And so that gives you a sense of if the package is trusted by other developers.</a></p>

<p><a data-t="552">But just counting the number of reverse dependencies doesn't always tell you the</a> <a data-t="556">whole story because some of these reverse dependencies are way more important than others.</a> <a data-t="560">Or sometimes you find a package that is used by 10 other packages all by the same author,</a> <a data-t="566">just like with scientific citations. And so an alternative metric that you could look at</a> <a data-t="572">are to count the recursive reverse dependencies. Or you could weigh the reverse dependencies by</a> <a data-t="578">their own relative importance. So you get something of a page rank statistic.</a></p>

<p><a data-t="584">And besides dependency relations, other technical indicators that you could look at</a> <a data-t="589">are like download statistics or project activity in terms of releases or commit activity.</a> <a data-t="597">And I think it's especially interesting to look at how these numbers change over time so that you</a> <a data-t="603">get a sense of the lifecycle of a project to see if the product is gaining traction,</a> <a data-t="608">or if it's on its way out, or if it has found like a stable role within the ecosystem.</a></p>

<p><a data-t="615">Let's actually look at an example that's close to my heart. And this is the curl package.</a> <a data-t="619">So if you need a web client in R, you probably need something with curl bindings.</a> <a data-t="623">But there's actually two CRAN packages that can do this. One is called rcurl and the other one</a> <a data-t="627">is just called curl. So which one should you use? Well, if you look at the CRAN homepage,</a> <a data-t="632">you find a very similar description and even a similar number of reverse dependencies. And you</a> <a data-t="638">can confirm this in R as well. But if we start looking at the recursive reverse dependencies,</a> <a data-t="643">we can see a much bigger difference. And the reason for this is that the curl package is</a> <a data-t="648">used as the foundation for some important web frameworks in R, such as the header package.</a></p>

<p><a data-t="654">And also, if we look at the reverse dependencies over time, we can see that the use of curl has</a> <a data-t="660">been steadily rising over the past few years, whereas the number of packages that use rcurl</a> <a data-t="666">has been pretty stale. And we can see a similar trend if we look at the download statistics for</a> <a data-t="671">these packages over the years. And the story here is that the rcurl package is very old.</a> <a data-t="677">It has been on CRAN since 2004 and has a paper about it from 2006. And it was some very pioneering</a> <a data-t="683">work at the time. However, a lot has changed since then. And the package has some very fundamental</a> <a data-t="690">problems. And the maintainer of rcurl is not very active anymore. So we wrote the curl package</a> <a data-t="696">specifically as a replacement for rcurl with a modern design to be simpler and more robust.</a> <a data-t="702">And it works very well. And most users have switched over to this new package since then.</a></p>

<p><a data-t="708">But the old rcurl package is still around. However, it's not in very good shape. And if</a> <a data-t="713">you use it today, you quickly start running into some of these problems. So I think this is one</a> <a data-t="718">example where metrics can help you make a more informed decision and potentially save you a lot</a> <a data-t="725">of problems down the line.</a></p>

<p><a data-t="732">But of course, these technical metrics don't always work. For example,</a> <a data-t="732">many R packages are very niche and they're useful only to a very particular specialized group of</a> <a data-t="738">people. So the package may not have many downloads or reverse dependencies. But if this is something</a> <a data-t="744">in your field, it may be exactly what you need.</a></p>

<p><a data-t="749">And the second category of indicators I want to</a> <a data-t="749">talk about are social indicators. I consider these to be all the things that have to do something</a> <a data-t="754">with the people behind the project and the way in which the development and the participation</a> <a data-t="760">gets organized. And in my experience, these are by far the most important things to look for</a> <a data-t="765">in a healthy open source project. But these are not easy to quantify. A lot of it just comes down</a> <a data-t="771">to getting to know the community a little bit.</a></p>

<p><a data-t="779">So open source projects are, for better or worse,</a> <a data-t="779">most of the time maintained by one or two people. And there's this fantastic book by Nadia Ekpal</a> <a data-t="786">from earlier this year that I really like that talks about the reality behind open source</a> <a data-t="791">development and why this is the case and the consequences that this has. And this is again</a> <a data-t="796">one of those big differences with commercial software. A lot of people are used to thinking</a> <a data-t="801">of software as written by anonymous engineers from a big company. And that is just not how</a> <a data-t="807">it works for us. Open source software is written by people like you and me. And you should think</a> <a data-t="813">of R packages more like a scientific publication or a piece of work by a local artist or a musician.</a> <a data-t="822">It really matters who is making that thing.</a></p>

<blockquote><a data-t="807">Open source software is written by people like you and me. And you should think</a> <a data-t="813">of R packages more like a scientific publication or a piece of work by a local artist or a musician.</a> <a data-t="822">It really matters who is making that thing.</a></blockquote>

<p><a data-t="830">So the most important social indicator is basically</a> <a data-t="830">who is the author, what organizations they are part of, what people they collaborate with a lot,</a> <a data-t="838">if they're still active or not, and what they're currently working on. And based on the type of</a> <a data-t="842">package, you may care about their formal qualifications and if they have an online</a> <a data-t="847">presence like a blog or a social media. Another social indicator is basically how is the product</a> <a data-t="853">managed. Is there a public place for reporting bugs and do these typically get resolved?</a> <a data-t="858">And if the product is open to contributions from external people and what that process looks like.</a></p>

<p><a data-t="864">To me, this is very important. I believe every R package should have a public place where everybody</a> <a data-t="869">can post their issues so you can see what problems other users are running into and people can help</a> <a data-t="875">each other out. But let me be clear that you should not expect every R package author to be a</a> <a data-t="881">full-time tech support person. Answering all these questions and reviewing suggestions can be very</a> <a data-t="886">exhausting and sometimes you just don't have time to work on a project for a few days or a few weeks</a> <a data-t="892">or a few months. But for me, the important thing is transparency. If I have no way of seeing what</a> <a data-t="900">are the common problems or questions with a package, that is a red flag.</a> <a data-t="907">There may be some</a> <a data-t="907">exceptions if the package is relatively small and maintained by a professor that's really the expert</a> <a data-t="914">in the field. It may be okay. But generally speaking, I think a functional issue tracker is</a> <a data-t="920">really a minimal requirement for a healthy open source project.</a></p>

<p><a data-t="926">And finally, other social indicators</a> <a data-t="926">you could think of are available resources for learning about the package. For example, if there's</a> <a data-t="932">good documentation either by the package author themselves or by users writing blog posts.</a> <a data-t="939">But for example, also if there's many questions and answers on Stack Overflow, that may be a great</a> <a data-t="945">source of information and a sign that other people are using this package.</a></p>

<p><a data-t="951">And then there's a third</a> <a data-t="951">category of indicators and these are perhaps more specific to R because we have a little research</a> <a data-t="956">software. And these are scientific indicators. So for packages that are specifically implementing</a> <a data-t="962">some scientific method, you are probably interested in the validity of those results.</a> <a data-t="968">And for example, if the package is fitting some model, you want to be sure that the estimates are</a> <a data-t="973">accurate and that the package is handling all edge cases properly. And this is of course very difficult</a> <a data-t="980">to judge, but there's some things we can look at. For example, some packages have gone through a</a> <a data-t="986">peer review process. For example, in the Journal of Systical Software or in R OpenSci. That's a</a> <a data-t="992">good sign. It's not a guarantee that the package will be in good standing forever, but at least</a> <a data-t="997">the author has gone through an effort at some point to have a colleague have a critical look at the</a> <a data-t="1003">code.</a></p>

<p><a data-t="1011">And of course, we can look at citations of packages in scientific publications. This is</a> <a data-t="1016">very tricky because many researchers don't cite software and if they cite, they probably just cite</a> <a data-t="1024">R or </a><a href="/software/tidyverse/">tidyverse</a><a data-t="1024"> and not the individual packages. But what is interesting is that many journals</a> <a data-t="1024">these days are starting to require researchers to include the analysis code with</a> <a data-t="1040">their publication for reproducibility purposes. And so you can imagine if a lot of code becomes</a> <a data-t="1036">available this way, we can start to analyze some of this code to see which R packages are</a> <a data-t="1042">commonly used in scientific research.</a> <a data-t="1051">So I think the scientific quality of R packages is by far</a> <a data-t="1051">the most difficult to judge, but it's very important.</a></p>

<h3>R Universe</h3>

<p><a data-t="1058">All right. So why am I talking about all of this?</a> <a data-t="1058">So in R OpenSci, we want to help researchers to get the most out of R. So we do curation and</a> <a data-t="1064">we organize peer review of R packages and we provide some development infrastructure.</a> <a data-t="1070">And recently, we've been increasingly feeling the need for better tools to actively monitor</a> <a data-t="1077">the quality of R packages. And this is both for our internal projects, but also for the R ecosystem</a> <a data-t="1086">as a whole, to be able to better advise researchers on what to use.</a></p>

<p><a data-t="1092">So we've started working on this</a> <a data-t="1092">new project called R Universe. And on the one hand, R Universe is an extension of our internal</a> <a data-t="1097">build infrastructure. So in R OpenSci, we have a doc server, which automatically builds package</a> <a data-t="1103">documentation for all of our packages. And we have a CRAN-like server, which automatically</a> <a data-t="1108">builds the binaries for all of our packages. And this works very well. And with R Universe, we</a> <a data-t="1113">basically want to enable other organizations to take advantage of this same tooling. But we want</a> <a data-t="1119">R Universe to not become just a build system, but also a place where you can learn more about these</a> <a data-t="1126">packages. And so we're going to try to combine some of these hard data that you get from the build</a> <a data-t="1131">system with other sources of information about authors and activity and the life cycle and all</a> <a data-t="1138">these things that we've talked about.</a></p>

<p><a data-t="1143">And then we have someone else to work on the scientific metrics.</a> <a data-t="1143">So they are developing tools that can help us find citations and mentions of packages in scientific</a> <a data-t="1150">publications. And besides showing information about a single project, we also want to help with</a> <a data-t="1156">the discovery of other software. And so by cross-referencing similar packages or other work by</a> <a data-t="1164">the same author or the same organization, maybe we can build something of a recommender system.</a></p>

<p><a data-t="1170">And finally, we want this to be a place that says something about the impact and the</a> <a data-t="1176">importance of an R project in order to help package authors get credit for research software</a> <a data-t="1185">in the same way as this happens for scientific publications. So this was my talk. Thank you for</a> <a data-t="1190">listening. I hope I got you thinking a little bit about the world behind open source development.</a> <a data-t="1197">And if I got you interested in the R-Universe project, keep an eye on our website,</a> <a data-t="1200">r-universe.dev, where we will start adding more of the things that we're working on</a> <a data-t="1206">over the upcoming year.</a></p>
