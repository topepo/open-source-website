<h3>What is Posit Connect and the old approach</h3>

<p><a data-t="95">So really briefly, Posit Connect, for those of you who are unfamiliar, it's a way, it facilitates being able to publish and share various data science artifacts,</a> <a data-t="109">whether those are static content like a </a><a href="/software/quarto/">Quarto</a><a data-t="109"> document, or interactive applications like a Shiny app.</a> <a data-t="117">And this feature specifically works with interactive content, so like a </a><a href="/software/shiny-r/">Shiny</a><a data-t="117"> or a Streamlit or a Dash app.</a> <a data-t="125">And it is designed and built to make it easier to interact with the data that you might have in your Databricks environment or your Snowflake environment.</a></p>

<p><a data-t="140">So it's important to call out here, the way that customers and users would try to implement this type of interaction previously,</a> <a data-t="150">is that you would set perhaps environment variables that are using service account like credentials to interact with resources like Databricks or Snowflake.</a> <a data-t="162">So the problem that this flow ran into pretty quickly is that these deployments really only have one identity that they're using when they're interacting with the external resource.</a> <a data-t="178">So having fine grained access control that respects, say, how different users or user personas might interact with that external resource would be really, really difficult.</a></p>

<p><a data-t="190">Some of the workarounds that people would use would be maybe deploying the same content item with either different business logic,</a> <a data-t="199">where it's running different queries against the dataset, or perhaps different service account credentials to interact with that external resource as a different account identity.</a> <a data-t="211">But this means that you're deploying the same content over and over and over.</a> <a data-t="215">And it's the responsibility of the app to ensure that data is being accessed securely and that the permissions that are applied to a given user are being respected.</a> <a data-t="229">So this moves the locus of sort of secure access out of the data source and into the applications that are querying that data.</a></p>

<blockquote><a data-t="229">So this moves the locus of sort of secure access out of the data source and into the applications that are querying that data.</a></blockquote>

<h3>How OAuth integrations solve the problem</h3>

<p><a data-t="239">So what this new feature supports is that where in Databricks Unity Catalog, for example, you can set up fine grained row and column based access control,</a> <a data-t="252">and you can set up like really robust access protocols against those data sources.</a> <a data-t="261">It's now possible to deploy content to connect that transparently relies on what is set up on top of the data in one centralized location.</a> <a data-t="271">So you can deploy a single application to connect, which serves many users and many different data access patterns.</a> <a data-t="279">And the app is transparently relying on what the security protocol is at the data source itself.</a></p>

<p><a data-t="287">So in this workflow demo, we're going to set up a data source with fine grained access controls in Databricks Unity Catalog.</a> <a data-t="296">And then we're going to show how those permissions can then be delegated to connect using this new OAuth integrations feature.</a></p>
