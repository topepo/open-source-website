<h3>Why DuckDB?</h3>

<p><a data-t="28">DuckDB is an embedded analytical database, SQLite for analytics.</a> <a data-t="33">Because of that, there is a variety of ways that you can package DuckDB as a command-line interface, as a Python, Rust, WebAssembly, and also as an R package.</a></p>

<p><a data-t="46">I see three main reasons why you may wish to use DuckDB.</a> <a data-t="51">If your data is large, DuckDB can read straight from Parquet and from other formats.</a> <a data-t="57">If your data analysis or data pipeline is just a little too slow, DuckDB will make it faster.</a> <a data-t="59">And then the third reason, DuckDB comes with a lot of exciting functions and data types, and there are extensions that add to that.</a></p>

<p><a data-t="70">All these are valid reasons to use DuckDB as a command-line, to use the DuckDB R package.</a> <a data-t="77">Now, duckplyr, to me, this is the interface that's more native to R, and I've tried to make this more accessible so that it blends in with all the other tools in the R ecosystem.</a></p>

<h3>Accessing DuckDB from R</h3>

<p><a data-t="97">Now, looking at ways to access DuckDB from R, you could do the command-line at the bottom right.</a> <a data-t="101">The more traditional way is through </a><a href="/software/dbi/">DBI</a><a data-t="101">.</a> <a data-t="107">DBI is a generic package to connect to a variety of databases, including DuckDB.</a> <a data-t="112">You would write SQL to interact through DBI or through the command-line.</a> <a data-t="121">If you enjoy dplyr-like interfaces, as I do, you would use </a><a href="/software/dbplyr/">dbplyr</a><a data-t="121"> that connects with DBI.</a></p>

<p><a data-t="121">So dbplyr translates your </a><a href="/software/dplyr/">dplyr</a><a data-t="121"> pipelines into SQL, runs it on the database, collects the results, but only if you tell it to.</a> <a data-t="154">So dbplyr works with lazy tables, and I forgot to mention, all of this will be available after the talk.</a> <a data-t="169">You can run the code, and the slides are on GitHub.</a></p>

<p><a data-t="176">So dbplyr translates to SQL, gets the data back as a lazy table.</a> <a data-t="196">You can collect, but you also need to collect to get this as a data frame.</a> <a data-t="204">Duckplyr is similar but different in a few respects.</a> <a data-t="166">So first, there is no SQL involved.</a> <a data-t="172">We use a lower-level interface so that we don't have to translate to text and back from binary formats.</a> <a data-t="177">And second, you don't need to collect.</a></p>

<p><a data-t="177">So the data comes back as a true and proper data frame, and it will be materialized on first access.</a></p>

<h3>Recommendations for large data</h3>

<h3>Speeding up existing pipelines</h3>

<p><a data-t="521">Of course, analyzing large data is slow.</a> <a data-t="523">The same can be true for smaller data sets.</a> <a data-t="531">This is a completely synthetic example, 1 million rows doing a simple summary over 40,000 loops, took five seconds on my machine.</a> <a data-t="538">Imagine multiplying this by ten.</a></p>

<p><a data-t="544">Now, if you have an existing pipeline, and this is one of the strengths and one of the original purposes that that was intended for, if you have a pipeline and you just want to get that a little bit faster, you have basically three strategies, you could go all in, try to do end-to-end with DuckDB.</a> <a data-t="567">I think this amounts to a rewrite of the entire code.</a> <a data-t="569">The best effort mode is the default mode.</a> <a data-t="575">My recommendation would be to do a very targeted optimization of just the slow parts.</a></p>

<p><a data-t="1015">So again, I think this is the interface that's more native to R. Dbplyr doesn't return data frames.</a> <a data-t="1021">Dbplyr returns a proxy object that you just cannot pass into ggplot unless ggplot takes measures to do so, which I'm not sure it does today.</a> <a data-t="1034">And if ggplot does, many other packages will not know how to handle a lazy table from dbplyr, but they will know how to handle data frame.</a></p>

<p><a data-t="1042">How does duckplyr interact with arrow?</a> <a data-t="1042">I have no idea.</a> <a data-t="1043">We'll save an answer for later.</a></p>
