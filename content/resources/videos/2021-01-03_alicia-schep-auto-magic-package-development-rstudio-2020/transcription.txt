<h3>The VegaWidget ecosystem</h3>

<p><a data-t="443">I think my biggest contribution was actually suggesting to Ian that maybe he could split</a> <a data-t="449">off one part of the package into something else, which is that he was using Altair to</a> <a data-t="456">use some code to make that specification for what the chart should be.</a> <a data-t="459">But then another big part of the code was just taking that specification and rendering</a> <a data-t="463">it in R so you could actually see the plot.</a> <a data-t="465">And I said, well, maybe we should make that a separate package to just do the render.</a> <a data-t="470">And so that became the VegaWidget package.</a></p>

<p><a data-t="472">So VegaWidget just takes a JSON spec and then uses the HTML widgets package in R to turn</a> <a data-t="478">it into a plot.</a> <a data-t="480">And this was primarily developed by Ian, who actually at this very minute is giving a talk</a> <a data-t="484">in another room, which is kind of unfortunate timing.</a></p>

<p><a data-t="489">But he's a really smart, very nice guy.</a> <a data-t="494">You should go find him and talk to him if you're interested in any of this stuff.</a> <a data-t="498">And myself and Stuart Lee helped with some of the JavaScript functionality in this package.</a></p>

<p><a data-t="505">And so what this package also does is it provides interfaces from that HTML widget to other</a> <a data-t="511">parts of the R ecosystem, in particular </a><a href="/software/shiny-r/">Shiny</a><a data-t="511">.</a> <a data-t="514">So it has functions to help you listen to events, to connect to what in Vega world is</a> <a data-t="520">called a signal, which is basically a dynamic data value, and also to let you update the</a> <a data-t="525">data that's in the plot without completely re-rendering it.</a></p>

<p><a data-t="530">What VegaWidget doesn't do is it doesn't help you make the spec.</a> <a data-t="532">So it basically expects some JSON to be given to it, which in R form is going to be basically</a> <a data-t="539">a big, complicated, nested list.</a> <a data-t="541">So this isn't a package that you're usually going to want to use directly.</a> <a data-t="544">You're not going to want to go and make this kind of list because it's a little bit clunky</a> <a data-t="548">and you'd have to store in your head exactly how to make this kind of thing.</a></p>

<p><a data-t="553">But this package is designed to be a helper to other packages, where those other packages</a> <a data-t="558">would help you make the spec.</a> <a data-t="559">And then they can rely on the shared interface to actually render it.</a> <a data-t="564">So one of those packages is an R port to the Altair package, which uses Altair via reticulate</a> <a data-t="570">to actually make that spec and then gets rendered.</a></p>

<p><a data-t="574">Ian and Haley and others are now working on ggvega, which is a package to take a ggplot</a> <a data-t="580">object, turn it into one of those specs, and then have it render.</a> <a data-t="585">And then I've been working on VLBuilder, which is basically, in some ways, like a spiritual</a> <a data-t="589">successor to that VegaLite package, which tries to make a native R API to make these</a> <a data-t="595">specs, which then can be rendered via VegaWidget.</a></p>

<h3>Goals of VLBuilder</h3>

<p><a data-t="599">So the goals of VLBuilder were they wanted a pipe-friendly approach for building up VegaLite</a> <a data-t="603">specs.</a> <a data-t="604">They wanted something that felt sort of natural to an R user who might be familiar with the</a> <a href="/software/tidyverse/">tidyverse</a><a data-t="607">.</a> <a data-t="608">They wanted to stay pretty close to the VegaLite spec, so not inventing new abstractions, not</a> <a data-t="614">inventing new terminology, but just kind of taking what is in VegaLite and making it accessible</a> <a data-t="621">in R.</a> <a data-t="621">But maybe a little syntactic sugar here and there for when you have common things</a> <a data-t="625">to do that might be a little clunky and trying to make those a little easier.</a></p>

<p><a data-t="629">Another key thing I wanted was to have autocomplete and documentation help, so that, you know,</a> <a data-t="633">if you're trying to add an encoding, you can see what are the different parameters I can</a> <a data-t="637">give to that.</a> <a data-t="638">And that's something that actually is a little bit missing from the Altair package, because</a> <a data-t="642">it's going through Articulate.</a> <a data-t="644">You kind of end up having to really use it effectively.</a> <a data-t="646">You often have to end up going to the Python docs.</a> <a data-t="648">They're not quite as easy to access for an R user in </a><a href="/software/rstudio/">RStudio</a><a data-t="648">.</a></p>

<p><a data-t="656">And the most important thing, well, in my mind, developing this, was that it had to</a> <a data-t="660">be fun to develop and easy to update as the VegaLite scheme evolved, because I learned</a> <a data-t="663">from that other PR that, like, you know, if you just, you know, try to build something</a> <a data-t="667">up manually that tries to match VegaLite, it's going to take a lot of work when VegaLite</a> <a data-t="674">evolves.</a> <a data-t="676">And I didn't want to get caught in that same bind again.</a></p>

<h3>Auto-generating the API from the schema</h3>

<p><a data-t="680">So here I took inspiration from Altair and the VegaLite API packages, because both of</a> <a data-t="687">those approaches in Python and JavaScript respectively actually auto-generate big parts</a> <a data-t="691">of their API using VegaLite itself.</a> <a data-t="696">So VegaLite has a schema which can derive what a VegaLite specification should look</a> <a data-t="702">like.</a> <a data-t="703">You can use that to build up an API.</a> <a data-t="705">So I thought, why don't I try to do this in R too?</a></p>

<p><a data-t="709">So just to go a little bit more into, like, what a schema is and what I mean by that,</a> <a data-t="714">the VegaLite schema is basically telling you in that, like, kind of JSON, that list object</a> <a data-t="718">where you're specifying the different parts of the plot, what is an acceptable sort of</a> <a data-t="725">format for each of those parts?</a> <a data-t="725">So for example, you know, you might say here an aggregate can be one of these other three</a> <a data-t="729">types and then this type can be one of these 12 different values.</a> <a data-t="734">Hopefully, it also includes descriptions for what different fields mean.</a> <a data-t="737">So in terms of trying to build up the documentation, a lot of what you'd want as documentation</a> <a data-t="742">is actually embedded in this schema document.</a></p>

<p><a data-t="748">And so the approach I took for building an R API from a JSON schema was one of semi-automation.</a> <a data-t="755">So early on, I was like, oh, maybe I'll try to automate it as much as possible.</a> <a data-t="758">But at some point, like, making something very generalized and totally automated becomes</a> <a data-t="763">a very hard problem.</a> <a data-t="765">And so I try to strike the right balance between using some human curation and some automation.</a></p>

<p><a data-t="771">So the human curation part is sort of identifying what families of functions do I want.</a> <a data-t="776">So I know that, you know, an encoding is going to be a very important part to make a VegaLite</a> <a data-t="782">spec.</a> <a data-t="783">I'm going to want a bunch of different functions that let you add different kinds of encoding.</a> <a data-t="788">I'm also going to want that for a mark.</a> <a data-t="790">So the mark is sort of like the geom and geom.</a> <a data-t="792">So I want, like, a VL mark point, VL mark bar, VL mark line, et cetera.</a> <a data-t="798">And so I'm going to want all these different kinds of families of functions, and they're</a> <a data-t="801">going to have this kind of format.</a></p>

<p><a data-t="803">But in terms of figuring out exactly which ones I need and what's going to go inside</a> <a data-t="808">them, what are the acceptable parameters, there I'm going to use automation.</a> <a data-t="812">I'm going to pull those things out of the schema and plop them into my function rather</a> <a data-t="817">than doing that manually by copying and pasting.</a> <a data-t="820">I'm going to write functions to do that.</a></p>

<p><a data-t="823">And so all of this sort of functions to build these functions lives in a package within</a> <a data-t="828">my package.</a> <a data-t="829">So if you look in VL builder, there's a build directory, and inside of that, there's a package</a> <a data-t="833">I call VL metabuilder, which is the package that builds VL builder, and the main function</a> <a data-t="840">is one that's called just create API, and it takes as an argument the schema.</a> <a data-t="844">So as the schema updates, as there's a new version of that schema, I can give it that</a> <a data-t="849">new version, and I'll create a new API.</a></p>

<p><a data-t="856">And so as an example of how to make a family of functions, I have that create API function</a> <a data-t="862">called something called create encoding functions, which will make all those encoding functions.</a> <a data-t="866">It first will pull all the possible encoding, so it'll take a function that says, okay,</a> <a data-t="871">this is where in the schema I'm going to look for them, and has a function to then pull</a> <a data-t="875">out all the different options that would be available.</a> <a data-t="878">It then iterates through all of them and makes the appropriate functions.</a></p>

<p><a data-t="883">So here, actually, I make sort of for each encoding, I want both a function that will</a> <a data-t="887">sort of add one to spec or one that just makes that object on its own.</a> <a data-t="891">You can see there's some deprecation, so I have even functions to generate my deprecated</a> <a data-t="896">versions of my functions as well.</a></p>

<p><a data-t="901">And so here, in terms of what a function to make a function is, here I have my create</a> <a data-t="907">encoder function that calls another function, because a lot of them actually share a lot</a> <a data-t="911">of properties.</a> <a data-t="912">So I have a generalized make function that takes in, most importantly, where to look</a> <a data-t="918">in the schema.</a> <a data-t="919">So this is a path to a particular place in that specification for what can go in a particular</a> <a data-t="925">part of a Vega-Lite spec, and it looks like what are all the available options.</a></p>

<p><a data-t="930">And then there's a couple extra things where I specify, as you can tell, like one thing</a> <a data-t="934">that gets used a lot in this effort is glue.</a> <a data-t="937">The glue package is really handy for this kind of effort, because you're basically trying</a> <a data-t="941">to make a bunch of strings where you're putting different pieces together.</a> <a data-t="945">So that has come in really handy in this effort.</a></p>

<p><a data-t="949">And so then it ends up making a function that also has documentation, because it actually</a> <a data-t="953">pulls all those little pieces of documentation out of that schema object.</a> <a data-t="956">And so I don't have to write any of it.</a> <a data-t="958">I can just use what's already been created, but make it accessible in R in a way that</a> <a data-t="964">the R user doesn't have to go to some website.</a> <a data-t="965">They can just type in help and get it right there in their RStudio pane.</a></p>
