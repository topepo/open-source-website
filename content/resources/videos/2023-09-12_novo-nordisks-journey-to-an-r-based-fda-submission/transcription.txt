<h3>Package management strategy</h3>

<p><a data-t="955">And I thought we would have had such a great success</a> <a data-t="957">in the user uptake on the system,</a> <a data-t="959">if we didn't have a really great strategy</a> <a data-t="961">on handling packages.</a> <a data-t="963">And now, Anders will take us through</a> <a data-t="965">the handling of these packages from CRAN.</a></p>

<p><a data-t="969">Yeah.</a> <a data-t="970">Thank you for that, Stephan.</a> <a data-t="972">So the NN package management really started in 2020</a> <a data-t="976">with the...</a> <a data-t="981">Remember that our organization was used to SAS and SAS only.</a> <a data-t="987">So introducing a new language</a> <a data-t="988">that is very much unlike SAS, that's challenging.</a> <a data-t="991">So SAS is nice and simple.</a> <a data-t="993">You need to install SAS 9.4.</a> <a data-t="996">It's 10 years old.</a> <a data-t="997">It has been patched a couple of times,</a> <a data-t="999">but it's still in use.</a> <a data-t="1000">R on the other hand, that's not as simple.</a> <a data-t="1003">R is evolving at a higher pace.</a> <a data-t="1006">And R also has all of these packages.</a> <a data-t="1009">And each of these packages,</a> <a data-t="1011">they come with their own versions,</a> <a data-t="1013">they come at different times,</a> <a data-t="1014">they have their own life.</a></p>

<p><a data-t="1016">So these packages are what makes R great and powerful,</a> <a data-t="1019">but there is no free lunch.</a></p>

<p><a data-t="1025">So dependencies between packages,</a> <a data-t="1027">they quickly become too complex</a> <a data-t="1028">if you just pick and choose your versions arbitrarily.</a> <a data-t="1031">You need, you'll quickly create a set of packages</a> <a data-t="1034">that does not work as intended together.</a> <a data-t="1037">So we wanted to create an environment</a> <a data-t="1039">that is sufficiently fixed and stable,</a> <a data-t="1042">but still flexible.</a> <a data-t="1043">And we opted for this shared baseline model</a> <a data-t="1045">as it has now become known.</a></p>

<p><a data-t="1048">So here you lock each version of R</a> <a data-t="1050">to a particular date and time,</a> <a data-t="1053">and you install packages in versions</a> <a data-t="1055">that were available at that given date.</a> <a data-t="1057">So we do this so that we know</a> <a data-t="1059">that any pair of dependent packages,</a> <a data-t="1062">they were tested at that point in time,</a> <a data-t="1064">and they were tested that they work together.</a> <a data-t="1067">So that means that, for example,</a> <a data-t="1068">our R version 4.02,</a> <a data-t="1071">that's locked to some date in September, 2020,</a> <a data-t="1074">that we will always get </a><a href="/software/dplyr/">dplyr</a><a data-t="1074"> 1.0.2,</a> <a data-t="1078">despite how many releases of dplyr that has been since.</a> <a data-t="1082">So teams simply just choose their version of R.</a> <a data-t="1086">They can deviate from the standard ones</a> <a data-t="1088">that we have in the system,</a> <a data-t="1090">but if they do so, they must use </a><a href="/software/renv/">renv</a><a data-t="1090">.</a> <a data-t="1093">In general, we actually recommend that you always use renv.</a></p>

<p><a data-t="1098">So what packages can and should we use?</a> <a data-t="1101">Well, they can be roughly categorized in this way.</a> <a data-t="1105">So they are the base and recommended packages.</a> <a data-t="1108">Those are the ones that ship with R.</a> <a data-t="1110">They are low risk, and they are very widely used.</a> <a data-t="1114">The contributed ones are the one</a> <a data-t="1115">we need to do a risk assessment on.</a> <a data-t="1117">So of these, there are the widely used ones.</a> <a data-t="1120">They are low risk by virtue of the enormous user base.</a> <a data-t="1127">There is the well-tested and documented ones.</a> <a data-t="1130">They may be more niche,</a> <a data-t="1132">but they are still considered low risk.</a> <a data-t="1134">It is the risk that pose a problem.</a></p>

<p><a data-t="1137">So when we do a risk assessment,</a> <a data-t="1139">we lean heavily on the R Validation Hub's</a> <a data-t="1142">recommendations on how to categorize.</a> <a data-t="1145">So the main points of categorization is the type of use,</a> <a data-t="1148">the maintenance, the popularity, and so on.</a> <a data-t="1152">So at the end of the day,</a> <a data-t="1153">this is just a human risk assessment,</a> <a data-t="1155">and the way we prompt this assessment to get made</a> <a data-t="1159">is by a Shiny app.</a></p>

<h3>Internal R packages</h3>

<p><a data-t="1233">So we now have a GXP environment</a> <a data-t="1236">and we have a strategy for packages,</a> <a data-t="1239">but we still do not have R code</a> <a data-t="1241">for producing tables, figures, and listings.</a> <a data-t="1243">We do not have batch execution and logging capabilities</a> <a data-t="1246">in R, what about those?</a> <a data-t="1249">This is also handled by the packages,</a> <a data-t="1253">our internal packages,</a> <a data-t="1254">and these internal packages,</a> <a data-t="1255">they started as a single package,</a> <a data-t="1257">one package to rule them all in 2018,</a> <a data-t="1261">and this was developed on a local laptop,</a> <a data-t="1264">and it was simply a gathering</a> <a data-t="1265">of all the usual functions that have been made.</a></p>

<p><a data-t="1269">In 2020, we introduced the nnBiostat package</a> <a data-t="1274">that was simply defined as the accepted packages.</a> <a data-t="1279">So users would write library in nnBiostat</a> <a data-t="1281">and get the most useful packages.</a> <a data-t="1286">In 2021, we realized that nnR had become too big</a> <a data-t="1291">and it was split into packages related to specific tasks.</a> <a data-t="1296">So nnR then became just a simple umbrella</a> <a data-t="1299">of all of these packages,</a> <a data-t="1300">like you know from </a><a href="/software/tidyverse/">Tidyverse</a><a data-t="1300">.</a></p>

<p><a data-t="1304">So the most important of our internal packages</a> <a data-t="1307">is nnAccess, that is for getting data</a> <a data-t="1308">without worrying about paths or data formats.</a> <a data-t="1311">It's nnPlot for creating NovoStyle plots.</a> <a data-t="1315">It's nnTable, which defines the grammar of tables</a> <a data-t="1319">for laying out tables and creating text-based output.</a> <a data-t="1323">So we wanted to mimic what we already did in SAS.</a> <a data-t="1327">It can also produce HTML outputs.</a> <a data-t="1330">We also have nnExport for exporting table figures</a> <a data-t="1334">and lists that fit into our production pipeline.</a> <a data-t="1338">And we have nnLog for batch execution and logging.</a> <a data-t="1342">Here we actually use niraspin to create nice-looking logs.</a></p>

<p><a data-t="1348">So, and although these packages,</a> <a data-t="1350">they fit together in a complicated network,</a> <a data-t="1352">each package is just like in the real world,</a> <a data-t="1355">it has its own repo.</a> <a data-t="1357">It is managed in an open-source way internally.</a> <a data-t="1361">And we have version numbers, a maintainer group,</a> <a data-t="1364">we have pipelines, and we have boards</a> <a data-t="1367">for managing bugs, features, ideas, and much more.</a></p>

<p><a data-t="1373">So we aimed to follow a common</a> <a data-t="1376">and good software principles of modularity,</a> <a data-t="1379">and we tried to build up functionality like Lego bricks.</a> <a data-t="1383">So we tried to segregate the responsibility of the packages</a> <a data-t="1387">and define appropriate extraction levels.</a> <a data-t="1390">So at the bottom here,</a> <a data-t="1391">you see the packages that are close to the system.</a> <a data-t="1394">They are system-dependent, if you will.</a> <a data-t="1397">And on top, you see the packages</a> <a data-t="1399">that are close to the task at hand.</a> <a data-t="1402">At the top, users shouldn't worry</a> <a data-t="1404">about how the data is stored,</a> <a data-t="1406">where is it stored, and so on.</a> <a data-t="1409">They should just concentrate on the task.</a></p>

<p><a data-t="1412">So for example, a programmer wishes to use nntfl</a> <a data-t="1417">to make a mean plot that will call nnAccess</a> <a data-t="1421">to fetch ADAM data through a connection set up</a> <a data-t="1423">by nnRemote.</a> <a data-t="1425">And after the plot has been made,</a> <a data-t="1426">nnExport will take care of exporting the data</a> <a data-t="1429">to an appropriate data format</a> <a data-t="1431">and associated files that we need.</a> <a data-t="1435">So this has the benefit</a> <a data-t="1436">that we can change underlying architectures in the system</a> <a data-t="1441">and simply just do small changes</a> <a data-t="1444">in the lower-level packages, if you will,</a> <a data-t="1448">and the users won't see a difference.</a> <a data-t="1450">So this is also the reason</a> <a data-t="1451">that we could inject a caching system into the system.</a></p>

<p><a data-t="1458">So let's talk about the tool chain that we use in Novo.</a> <a data-t="1462">So we use git for version control</a> <a data-t="1465">and Azure Pipelines for executing code upon code changes.</a> <a data-t="1471">We use useless for the description file and the news file.</a> <a data-t="1475">We use package down for generating a website</a> <a data-t="1480">of documentation,</a> <a data-t="1481">and we use our oxygen to write the documentation.</a> <a data-t="1486">We use lifecycle to convey the stability</a> <a data-t="1489">or the lifecycle of a function within the packages.</a> <a data-t="1494">And we use test.uncover to test the packages.</a> <a data-t="1498">We also use our markdown</a> <a data-t="1499">for writing longer format documentation</a> <a data-t="1502">in the form of vignettes.</a> <a data-t="1504">And we use </a><a href="/software/devtools/">DevTools</a><a data-t="1504"> to do the normal system,</a> <a data-t="1508">sorry, the normal package development.</a></p>

<p><a data-t="1511">So upon a code change,</a> <a data-t="1514">we will test again the package in a matrix way.</a> <a data-t="1519">So we test against all our libraries,</a> <a data-t="1521">all our versions of R</a> <a data-t="1523">and across all of our system environments.</a> <a data-t="1529">So we know that if, for example,</a> <a data-t="1531">an upcoming change in development to the system</a> <a data-t="1534">will break something in our packages.</a> <a data-t="1537">We then build the package and then we release it.</a> <a data-t="1540">So in the process of releasing,</a> <a data-t="1542">we are updating the package down website</a> <a data-t="1545">and we ship it to the internal repo in the package manager.</a></p>

<p><a data-t="1550">So we realized some time ago,</a> <a data-t="1553">but we implemented it this year</a> <a data-t="1556">that we need the ability to deprecate our versions.</a> <a data-t="1560">So say we no longer want to support</a> <a data-t="1563">the oldest version of R we have and stop,</a> <a data-t="1566">then we have made version specific repos</a> <a data-t="1572">so we can stop updating that.</a> <a data-t="1575">This also allows for hotfixes if system changes are needed.</a></p>

<p><a data-t="1581">So we try to ensure a high level of code quality</a> <a data-t="1584">by enforcing a main branch policy.</a> <a data-t="1587">Nobody can change the main branch directly.</a> <a data-t="1590">Only pull requests to the main branch are accepted.</a> <a data-t="1594">The pipeline needs to run with success and no errors.</a> <a data-t="1598">A feature or bot needs to be linked to the pull request</a> <a data-t="1602">so we know why we have to change.</a> <a data-t="1604">And we need at least one maintainer to approve</a> <a data-t="1608">to ensure a code review.</a></p>
