<h3>Differences between DevOps and MLOps</h3>

<p><a data-t="169">And I think one of the ways that we can think about that is to have a look at some of the differences between traditional development processes, DevOps, and machine learning building, MLOps.</a> <a data-t="182">Now in both cases, we're working with files, you know, that source code.</a> <a data-t="187">You know, it might be C++ or .NET files on the regular application side of things.</a> <a data-t="192">But with machine learning, we're also working with other artifacts, things like data files.</a> <a data-t="198">There's a whole thread I could go down to get down in at this point about thinking about what does it mean to have a 55 megabyte data file that you're analyzing and putting that into Git?</a> <a data-t="211">Short hashtag don't do that.</a> <a data-t="213">But lots of issues around managing data that don't come up in traditional development processes.</a> <a data-t="219">Same things if you're working with notebooks or RMD documents or R Markdown documents.</a> <a data-t="225">How do we deal with those and how do they manage in source code control systems?</a></p>

<p><a data-t="229">A good practice when building applications is rather than managing infrastructure directly by pointing and clicking in consoles and terminals is to manage that infrastructure with commands in code.</a> <a data-t="244">And those same principles apply on the machine learning side as well, with an additional wrinkle that we often have to manage environments.</a> <a data-t="253">Think for in the R case, managing the packages and the package versions that need to exist for your R model to run in a reliable fashion.</a> <a data-t="263">So we want to be able to manage those as code as well.</a></p>

<p><a data-t="265">I touched a little bit about those issues of working with source code control systems.</a> <a data-t="270">Those also apply in the machine learning operations scenario.</a> <a data-t="273">But there are other things we need to track changes in as well, not just the source code.</a> <a data-t="278">We also want to track changes in our experiments.</a> <a data-t="281">And by that I mean, as we try out different types of models, perhaps different transformations of our data, we're looking for some kind of outcome, perhaps the model with the best accuracy.</a> <a data-t="292">And we'd like to be able to track that as well in a reproducible way so that we can always go back to prior experiments if we end up using that one instead.</a></p>

<p><a data-t="305">When we're building applications in a DevOps environment, typically we're building binary executables.</a> <a data-t="311">Those builds might take minutes, maybe hours.</a> <a data-t="313">That's kind of the typical limit for most types of applications.</a> <a data-t="316">You're typically building on sort of commodity computing infrastructure.</a> <a data-t="320">But on the machine learning side, rather than building, in addition to building executables rather, we're building the models, we're training the models that become part of those executables.</a> <a data-t="332">And that model training, especially in sort of deep learning type of environments, could likely take hours, might well take days.</a> <a data-t="340">I've seen examples where model training takes months.</a> <a data-t="343">And so how do we put that into a traditional DevOps type of environment?</a> <a data-t="348">And also how do we incorporate the exotic computing environments like the GPU computing environments that we need for deep learning?</a></p>

<p><a data-t="355">When it comes to version control, version management, when we're building applications, we like to give versions to the applications that we build and release.</a> <a data-t="366">On the machine learning side, we would like to assign versions to models so we can track when we change a model and then go back to the underlying data and code that created that model and do so reproducibly.</a> <a data-t="380">And another sort of side branch that I could go into for hours is this whole concept of tests when it comes to machine learning as opposed to developing applications.</a> <a data-t="392">On the application side, tests are fairly deterministic.</a> <a data-t="395">Did it break? Did it cause an error? Did it give the right response?</a> <a data-t="399">But in machine learning, tests tend to be a little more probabilistic.</a> <a data-t="403">You know, think of the question of, is this a picture of a cat?</a> <a data-t="406">That's not so much of a yes or a no type question.</a> <a data-t="409">It's more of a probabilistic thing that we need to determine.</a> <a data-t="412">And so the way we do testing is different.</a></p>

<p><a data-t="414">I can't go into all of these topics today, but I'm just going to touch on a few of them.</a> <a data-t="419">But I just wanted to give you this as just an illustration that there are quite a few differences, in my opinion, between machine learning operations and traditional DevOps.</a> <a data-t="430">And we need to incorporate those differences into any process that we're using when we build applications with machine learning.</a></p>

<h3>Azure Machine Learning Service</h3>

<p><a data-t="437">Now, Microsoft has an application to help with that.</a> <a data-t="440">It's called the Azure Machine Learning Service.</a> <a data-t="442">It's based on a process that we developed internally for building the machine learning models that we have in things like Bing and Xbox and all sorts of other applications.</a> <a data-t="453">But now we brought it out so that people can use it externally as well.</a> <a data-t="457">And the workflow that's built into it has really been driven by a lot of the engagements that we have had with customers that are building machine learning models at scale and building them into very high-scale production applications.</a></p>

<p><a data-t="477">There are lots of things in this that I don't have time to talk about today, things like being able to use notebooks, automated machine learning, being able to just give it a data set and it tries a bunch of models.</a> <a data-t="490">There's a nice drag-and-drop visual interface that I can't go into today.</a> <a data-t="495">But instead, I'm going to focus on these concepts, these abstractions, which help us build a nice process for machine learning operations.</a> <a data-t="507">Things like data sets, being able to version a set of rows and columns of data, and being able to get back to that version of data at any time in a reproducible way when we want to recreate our models in our experiments.</a></p>

<p><a data-t="528">Experiments in this context are training runs, a model, I'm thinking of a statistical model here that we would try against a specific version of data, and the metrics like accuracy that we get back from that model, being able to track all of those.</a> <a data-t="548">Pipelines, I'm not going to cover too much here, but being able to combine together little pieces of R code, little pieces of Python code, that together form a complete workflow we might want to execute as part of a continuous integration or continuous delivery process.</a> <a data-t="568">Models, we're all familiar with the concept of a model from a statistical standpoint, but a framework that allows us to register models that we might want to use and have versions associated with those models.</a> <a data-t="580">And endpoints, this is where the ops part of it comes in.</a> <a data-t="585">Real-time endpoints that we can use to pass data to those models and get results back, make predictions from models.</a> <a data-t="592">And also endpoints for those pipelines that I mentioned just a minute ago, so that we can automate the process of kicking off sequences of doing computations in R and do that as part of a larger build process for a complete application.</a></p>
