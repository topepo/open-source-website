<h3>From TensorFlow 1 to TensorFlow 2</h3>

<p><a data-t="511">So TensorFlow 1 with the Python implementation was using wonky things called sessions.</a> <a data-t="517">We've learned a lot since then.</a> <a data-t="520">Sessions were a little bit frustrating.</a> <a data-t="522">I'm not sure how many people in the audience are familiar with Python code.</a> <a data-t="525">But the way that the logic for this worked was that you would define some variables.</a> <a data-t="531">But you wouldn't be able to use them immediately.</a> <a data-t="533">You would have to initialize something called a session.</a> <a data-t="535">You would have to then initialize the variables.</a> <a data-t="538">You would have to start a queue runner to be able to even understand what this input</a> <a data-t="543">data would look like.</a> <a data-t="544">You would have to create these weird batch sticks.</a> <a data-t="547">And it just felt like a very wonky experience.</a></p>

<p><a data-t="550">The transition from TensorFlow 1 is using Keras as the recommended high level API.</a> <a data-t="556">Every single addition that we make to the TensorFlow ecosystem going forward is going</a> <a data-t="561">to be Keras focused.</a> <a data-t="563">So I think TF, the text preprocessing layers was mentioned in the previous talk.</a> <a data-t="570">We're going to also see Keras integration with something called TensorFlow extended.</a> <a data-t="575">So all of the tooling there will be supported as well.</a> <a data-t="579">And you'll still be able to use low level operations.</a> <a data-t="584">So if you need the kind of close to the machine manipulation, you can use that as well.</a></p>

<p><a data-t="591">We've also removed duplicate functionality.</a> <a data-t="598">We are trying to make consistent and intuitive syntax across the APIs.</a> <a data-t="602">So again, focusing on Keras is the standard interface.</a> <a data-t="606">And creating compatibility throughout the entire TensorFlow ecosystem with something</a> <a data-t="610">called saved model.</a> <a data-t="611">And you are able to output saved models with all of your R TensorFlow code.</a></p>

<h3>Symbolic vs. imperative API and saved model format</h3>

<p><a data-t="712">So technical differences in TensorFlow 2.</a> <a data-t="717">We have something called the symbolic API as well as the imperative API.</a> <a data-t="721">And the difference between the two is kind of it isn't immediately obvious, right?</a> <a data-t="731">So if you're using the Keras functional API, your model turns into byte code, whereas sequential</a> <a data-t="737">API, your model is more a graph of layers.</a> <a data-t="740">And mixing functionality between the two is not at all recommended.</a> <a data-t="745">So if you are intending to use the Keras API, I strongly suggest sticking with one</a> <a data-t="752">and sort of staying with it.</a></p>

<p><a data-t="755">For training loops and for kind of understanding the differences between the different Keras</a> <a data-t="760">implementations, you might have heard of standalone Keras historically.</a> <a data-t="765">That has since ceased development.</a> <a data-t="768">So now all development work on Keras is sort of supporting the TensorFlow API backend.</a> <a data-t="775">There's still the sort of historic implementation that supports things like Theano and CNTK.</a> <a data-t="781">But for the most part, all development for Keras is going to be focused on TensorFlow</a> <a data-t="785">going forward.</a> <a data-t="788">And then the Keras R bindings for TF Keras are created by the good folks here at RStudio.</a></p>

<p><a data-t="833">So building pipelines with TF datasets is like you would use pretty much the same style</a> <a data-t="839">of interface that you've focused on before.</a> <a data-t="859">So the reason why TF datasets exists is because if you have sort of complex performance hardware</a> <a data-t="866">architectures, so if you have GPUs or if you have TPUs or TPU pods that you're attempting</a> <a data-t="871">to saturate, you can't really do that with sort of the common NumPy interface or the</a> <a data-t="879">traditional data pumping formats that you see today.</a> <a data-t="885">So TF data kind of gives you the ability to build high-performance data input pipelines</a> <a data-t="891">using just Python as opposed to having to hand roll these input streams in C++ or something</a> <a data-t="899">a little bit more, at least to me, a little bit more frightening.</a></p>

<p><a data-t="913">So TF data is sort of a platform agnostic data input pipeline that you could use with</a> <a data-t="920">TensorFlow, but also if you have R code that you do want, you know, a different R package</a> <a data-t="927">that does need a similar pipeline structure for its work, you could use TF datasets for</a> <a data-t="933">that as well.</a></p>

<p><a data-t="955">Moving along, compatibility with the TensorFlow ecosystem.</a> <a data-t="965">So I mentioned before that we're trying to standardize the interface for TensorFlow,</a> <a data-t="971">and we're really focusing on something called the saved model format.</a> <a data-t="975">The saved model gives you the ability to have your model generated in R or in Python, packaged</a> <a data-t="981">up into a friendly format that's kind of a defined directory structure that's also pretty</a> <a data-t="987">human readable.</a> <a data-t="988">There's a proto, some weights, you know, some additional metadata associated with the model,</a> <a data-t="995">and you can take that and you can put it in a cellular device.</a> <a data-t="998">You can put it on, like, an M series embedded device.</a> <a data-t="1002">You can use it in a browser, as a web application, or pretty much anything else.</a> <a data-t="1009">We've also started work on improving our C++ API for TensorFlow.</a> <a data-t="1014">You'll be seeing an announcement about that in a couple of months.</a></p>

<p><a data-t="1019">With sort of the roadmap for the C++ inference API, as well as the C++ API for training.</a> <a data-t="1027">And the idea is that all of it is focused around this concept of a saved model.</a> <a data-t="1032">And again, with your R code, you can generate this as an output and deploy your models anywhere</a> <a data-t="1038">that you could with a TensorFlow Python model.</a></p>

<p><a data-t="1042">We've also integrated TensorBoard into Jupyter notebooks.</a> <a data-t="1046">Jupyter notebooks have support for R, of course, if you wanted to use an R kernel, or you can</a> <a data-t="1051">also use TensorBoard within RStudio, I believe, with built in performance profiling, both</a> <a data-t="1057">for GPU and for TPU.</a> <a data-t="1060">And so under the hood, autograph and TF function, I mentioned this a little bit earlier, but</a> <a data-t="1064">what they're doing is even though we're giving you the ability to do eager computation, kind</a> <a data-t="1072">of, you know, the...</a> <a data-t="1074">You add numbers, you get a response, that sort of thing.</a> <a data-t="1077">Autograph and TF function sort of take whatever arbitrary Python code you've created and they</a> <a data-t="1082">build a graph out of it.</a> <a data-t="1085">So that means that your models, even though they're debuggable and they're easily understandable</a> <a data-t="1091">when you need them to be, they're also hyper performant if you need to run them on accelerated</a> <a data-t="1095">hardware or at scale using TF distribute.</a></p>
