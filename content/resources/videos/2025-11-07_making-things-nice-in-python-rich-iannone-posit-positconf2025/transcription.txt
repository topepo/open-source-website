<h3>Supporting many table types in <a href="/software/pointblank/">pointblank</a></h3>

<p><a data-t="42">First off, let's get a quick introduction to pointblank.</a> <a data-t="45">It's a package for validating tables, like do we have duplicate rows?</a> <a data-t="48">Are cum values in the defined range?</a> <a data-t="51">We can find this out.</a> <a data-t="52">We define a validation plan, we can interrogate the data table, and you get a nice report in the end.</a></p>

<p><a data-t="57">But let's look at line 3 of this pointblank validation code.</a> <a data-t="60">Now this is the part I really wanted to make nice, this data parameter.</a> <a data-t="64">Many other libraries that accept a table, they tend to support only one or a few types of tables.</a> <a data-t="70">It gets kind of restrictive.</a> <a data-t="71">The problem with this is that if a user uses different types of tables that aren't supported, they are generally less likely to use the package.</a></p>

<p><a data-t="80">I really wanted to, just like the R version of pointblank, support many types of tables.</a> <a data-t="84">This includes a variety of data frames and a plethora of database tables.</a> <a data-t="89">Luckily, I didn't really have to write too much custom code to support each of these table types,</a> <a data-t="94">because there are actually two Python libraries that could help make bring your own table pretty painless.</a></p>

<p><a data-t="100">The implementation uses two nice libraries, narwhals and ibis.</a> <a data-t="104">Narwhals can handle all sorts of data frames like polars, pandas, and spark,</a> <a data-t="108">and ibis handles a huge number of other types of tables, like many database tables.</a> <a data-t="113">By incorporating these libraries into pointblank,</a> <a data-t="115">it was actually a cinch to support all the tables that narwhals and ibis can work with.</a></p>

<p><a data-t="120">Supporting all sorts of tables meant that we had to build two internal processing pipelines,</a> <a data-t="125">one for narwhals and the other for ibis.</a> <a data-t="127">This doubled the amount of code to maintain, but it still felt like a win, because it was so nice for the user.</a> <a data-t="132">We can see in this diagram that the user just has to provide a data table,</a> <a data-t="136">and the program will handle it in one of two code paths.</a> <a data-t="139">The way it was set up ensured that the user will see the correct results no matter what type of table was used,</a> <a data-t="144">and crucially, the user will get output artifacts in the same format as the input.</a></p>

<p><a data-t="149">Not very long ago, narwhals gained the ability to directly use ibis tables.</a> <a data-t="154">This was amazing, since it meant that pointblank, with a little work,</a> <a data-t="157">would no longer need branching code for both narwhals and ibis.</a> <a data-t="161">It just goes to show that narwhals is itself a very nice Python package.</a></p>

<p><a data-t="166">Anyway, we jumped at the opportunity to simplify our code,</a> <a data-t="169">and this is not the only way that narwhals got nicer.</a> <a data-t="173">Narwhals gained the ability to use spark dataframes,</a> <a data-t="176">so pointblank no longer needed a sparkdf to be an ibis table.</a> <a data-t="180">For a spark user, this is so much nicer.</a> <a data-t="183">No need to transform the dataframe before using it in pointblank.</a> <a data-t="186">In the future, we can go even further by removing the user's need to prepare ibis tables for other table types.</a></p>

<h3>Adding datasets to a Python package</h3>

<p><a data-t="751">Well, that's all I wanted to say about making documentation great for a package website.</a> <a data-t="757">Let's move on to the fourth section,</a> <a data-t="759">and the topic there is adding lots of datasets to a Python package.</a></p>

<p><a data-t="763">Great Tables has a lot of datasets.</a> <a data-t="767">Here they all are.</a> <a data-t="768">There are 16 datasets in total,</a> <a data-t="770">and each one has a decorative medallion.</a> <a data-t="773">Those serve to make the datasets nice and fun.</a></p>

<p><a data-t="778">Having lots of datasets in Great Tables means we can provide great examples with them.</a> <a data-t="783">For example, if I wanted to show a super text-heavy table,</a> <a data-t="786">I would use the Peeps dataset.</a> <a data-t="789">If the demonstration involves adding graphics to the table,</a> <a data-t="792">then I would likely use the Metro dataset.</a> <a data-t="794">I use the formatImage method to get those SVG graphics into the table cells.</a> <a data-t="800">If I wanted to show users how to make nanoplots,</a> <a data-t="802">I would probably reach for the Illness dataset.</a> <a data-t="805">And as a final example on how datasets make for nice examples,</a> <a data-t="809">I like the Films dataset when showing tables with country flag icons within them.</a></p>

<p><a data-t="850">In pointblank, it's a little different.</a> <a data-t="852">That package provides the loadDataset function.</a> <a data-t="855">You can choose from multiple datasets and output table types with that function.</a> <a data-t="859">Again, there's no real conventions for datasets in Python packages,</a> <a data-t="862">but both these approaches are nice enough.</a></p>
