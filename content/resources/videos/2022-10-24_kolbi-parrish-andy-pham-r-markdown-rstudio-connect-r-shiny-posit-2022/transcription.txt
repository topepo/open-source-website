<h3>Error alerts and logging</h3>

<p><a data-t="224">So great.</a> <a data-t="225">Now we have automation.</a> <a data-t="226">But imagine one of those many automated scripts results in an error.</a> <a data-t="231">How are you going to see what error occurred and where?</a> <a data-t="235">Doing that can kind of be like finding a single noodle in a giant bowl of messy spaghetti.</a> <a data-t="241">Just a little bit frustrating.</a></p>

<p><a data-t="244">But error alerts and logging are there to help.</a> <a data-t="247">The way that we implemented that was through a combination of R Markdown, </a><a href="/software/blastula/">blastula</a><a data-t="247">, Logit, and RStudio Connect.</a> <a data-t="254">So the process was really similar for us to the way that we automated our data processing, except we added in a few additional layers.</a> <a data-t="263">And those layers were logging, email alert logic, and an error report template.</a> <a data-t="269">And for us, that was in the form of an R Markdown document, which added some structure to those email alerts.</a></p>

<h3>Process monitoring app</h3>

<p><a data-t="595">So I'm going to turn the floor over to Andy to cover how we took those log files that were in JSON-based format and turned them into something a little more digestible.</a> <a data-t="602">All right.</a> <a data-t="603">Thank you, Colby.</a> <a data-t="607">So now you have your automated processes set up and you have your email alerts for error logging.</a> <a data-t="611">Now let's say we get an email saying that there's an error.</a> <a data-t="614">The next step is to troubleshoot, right?</a></p>

<p><a data-t="617">However, errors rarely occur in isolation.</a> <a data-t="621">In order to fix an error efficiently and to prevent it from occurring again in future states, you want to be able to find if this error is occurring in your development and production environment, has this error occurred multiple times throughout your processes?</a></p>

<p><a data-t="635">And so before the process monitoring app, what we would have to do is we have to go through the error and process logs.</a> <a data-t="637">And as Colby mentioned before, these are in JSON format.</a> <a data-t="641">And there's probably hundreds to thousands of lines of this.</a> <a data-t="645">So you'd be pretty much abusing your Control-F to try to find what errors occurred and where it occurred.</a> <a data-t="651">And that's also not to mention that since we're logging almost every day, we'd have to go through all these files to see if there's some kind of historic pattern coming on from these errors.</a></p>

<p><a data-t="662">And so going through all this, I would equate it to sort of trying to eat a melting ice cream sundae.</a> <a data-t="668">Everything's mixed together, different ice cream flavors are mixing, and you're tasting like two, three different flavors, and you're not sure if you like it or not.</a> <a data-t="678">It's very interesting.</a> <a data-t="679">However, our approach was to season in some R magic and make a more beautiful ice cream sundae where all the components are easily seeable and you can get to where you need to be quickly and efficiently.</a></p>

<p><a data-t="694">And so we did this with this was our process for our process monitoring app.</a> <a data-t="699">We took a bunch of messy logs, applied some Logit, R Shiny, Plotly, data tables, any R visualization packages, and turned it into a dashboard where we can view both how our processes were doing and the errors we got within one single dashboard.</a></p>

<p><a data-t="716">However, we first needed to be able to read all of these logs into something that was R-readable.</a> <a data-t="722">Fortunately for us, Logit provides this function already.</a> <a data-t="727">There's a function called read underscore logs where if you pass the path of your error file, it'll read in that error file and return an R data table that's easily filtered.</a> <a data-t="737">So now we could filter based on if we wanted to find an error and then by which specific process and then within a specific time frame.</a></p>

<p><a data-t="746">So this was very useful for putting up the dashboard initially.</a> <a data-t="750">And so we fed this data into a data table call and built error data tables where we could view all of our error data errors in relation to each other, when they occurred, and if they occurred within any other processes.</a> <a data-t="763">We additionally fed our process logs into another data table.</a> <a data-t="767">And because we had set up our Logit calls within each R Markdown chunk, we can now see how long each R Markdown chunk took within the process and be able to identify any bottlenecks as needed.</a></p>

<p><a data-t="780">What we also did was we also fed some of the process log data into Plotly.</a> <a data-t="786">And so we created this graph where we could usually be able to see how our processes were doing in relation to each other and in relation to our development and production environments.</a> <a data-t="795">We could also see if any processes were running much, much longer than we expected so that we can get in there and fix it before anything broke.</a></p>

<p><a data-t="806">And so we took all of these different pieces, the error data tables, the graphs, and we used R Shiny to put it all into a dashboard.</a> <a data-t="814">And so now we could get where we needed to be really quickly and be able to see both of our development and production errors and processes in one setting.</a></p>
