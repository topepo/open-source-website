<p><a data-t="77">So okay, so to go kind of all the way down the hole, we get down into the Shiny package</a> <a data-t="83">and I suspect that most of you are familiar with Shiny, but basically Shiny is, as you</a> <a data-t="90">know, an R package, but what's important to realize is that Shiny speaks WebSocket.</a> <a data-t="95">And so what that means is that any browser that's compatible with WebSocket can connect</a> <a data-t="100">directly to a Shiny process and do, you know, the communications there back and forth between</a> <a data-t="106">those two entities.</a></p>

<p><a data-t="107">So that's great.</a> <a data-t="108">And you probably notice this when you're developing applications locally.</a> <a data-t="112">So if you're using </a><a href="/software/rstudio/">RStudio</a><a data-t="112">, for instance, and you press run app, you're going to get</a> <a data-t="114">a little viewer.</a> <a data-t="115">That viewer is communicating directly to your R process or, you know, if you open in a browser,</a> <a data-t="120">you're communicating directly with an R process and there's no Shiny server intermediary between</a> <a data-t="124">those two.</a> <a data-t="125">And that's because Shiny speaks WebSocket.</a> <a data-t="128">So it is possible to open a direct connection from a browser into the Shiny process.</a></p>

<p><a data-t="135">So that's great.</a> <a data-t="136">But there are a couple of limitations on this.</a> <a data-t="137">First of all, as I mentioned, that only works for browsers that can communicate over WebSockets.</a> <a data-t="142">And so that rules out Internet Explorer 8 and 9 and some other versions of Firefox and</a> <a data-t="148">Chrome and things like that.</a></p>

<h3>What Shiny Server sets out to accomplish</h3>

<p><a data-t="198">So Shiny server sets out to accomplish a few goals.</a> <a data-t="203">The couple that are interesting for this discussion are, first of all, it should start and stop</a> <a data-t="208">Shiny processes on demand and as needed, and also it should translate non-WebSocket traffic</a> <a data-t="215">into WebSocket traffic that Shiny can understand.</a> <a data-t="218">And then finally, it can also map different URLs to particular applications.</a></p>

<p><a data-t="223">So if you were just to run an army of Shiny processes on your bare machine, all of those</a> <a data-t="228">would have to run on different ports, and so your users would need to enter a port number</a> <a data-t="231">in order to get to an application, and it would be, you know, a little sloppy.</a> <a data-t="235">And so Shiny server offers a kind of a cleaner, more convenient way to be able to navigate</a> <a data-t="241">between the different applications, and we do that by offering a few different hosting</a> <a data-t="244">modes.</a> <a data-t="245">So you could just host one application.</a> <a data-t="246">You could allow users to host their own applications, or you could host an entire directory of applications,</a> <a data-t="251">and you can do these at different URL subspaces.</a> <a data-t="253">So it just allows you to kind of more meaningfully manage, you know, many Shiny applications.</a></p>

<p><a data-t="262">And so a couple of these to kind of revisit, so starting and stopping Shiny processes,</a> <a data-t="267">so again, this is, you know, if you're running this naively, you would need to be running</a> <a data-t="271">that Shiny process all the time, but because you're running Shiny servers, kind of this</a> <a data-t="274">long running, you know, always on daemon, what that means is that when a user goes to</a> <a data-t="279">connect into the Shiny process, if it's not running now, Shiny server can kind of hold</a> <a data-t="285">that request for a second while it goes to start the Shiny process, and then once the</a> <a data-t="290">Shiny process is ready, it can let the traffic through into that process.</a> <a data-t="294">So it can kind of broker and manage the communications between applications that may or may not be</a> <a data-t="298">running at the moment they're requested, and then also if it notices that all the users</a> <a data-t="303">have left and that application's been idle for some configurable length of time, then</a> <a data-t="307">it can go ahead and reap those processes to free up some more resources for other applications.</a></p>

<p><a data-t="313">And again, like I mentioned, older versions of Internet Explorer are not going to be able</a> <a data-t="315">to handle WebSockets natively, and so Shiny server uses a library called sock.js, which</a> <a data-t="322">basically allows you to work as if you were working with WebSockets, but it will kind</a> <a data-t="326">of gracefully fall back into other protocols as needed, and that's actually really important,</a> <a data-t="332">if you're not planning to support older versions of Internet Explorer, because many networks,</a> <a data-t="338">if you're using older network equipment that was designed before WebSockets gained popularity,</a> <a data-t="343">then you'll actually end up in a situation where they may be dropping or blocking WebSocket</a> <a data-t="349">traffic, and we see this quite a bit, especially at universities and academic settings.</a> <a data-t="353">They for some reason tend to have network equipment that isn't friendly with WebSockets.</a></p>

<p><a data-t="357">So you can kind of end up in the situation where even if you're using all the latest</a> <a data-t="363">and greatest technology, there may be some router between you and your client that is</a> <a data-t="368">going to interrupt your WebSocket traffic, and so it's nice to have a technology that</a> <a data-t="372">will allow you to fall back to something that is supported if needed.</a> <a data-t="376">So that ends up being a pretty important feature.</a></p>

<h3>Architecture overview</h3>

<p><a data-t="385">Okay, so kind of an overview of the architecture.</a> <a data-t="388">You can envision it looking something like this.</a> <a data-t="390">So you have a variety of users over here.</a> <a data-t="393">I don't know if you can see the mouse.</a> <a data-t="395">Let's try the pen here.</a> <a data-t="397">You have your variety of users over here, which could be a handful, or we run servers</a> <a data-t="402">where there are hundreds of concurrent connections from different users around the world, but</a> <a data-t="407">they're all opening connections into your Shiny server instance, which, again, is kind</a> <a data-t="412">of this always-on server process that's running as a daemon on your server.</a> <a data-t="417">And then it's going to open and close these, or start and stop these Shiny applications</a> <a data-t="423">and kind of proxy the traffic as it goes through from users' browsers into the Shiny applications.</a></p>

<p><a data-t="430">And so what this typically looks like, just to give you a feel if you're going to get</a> <a data-t="433">down to the network level, the first request, of course, is just for kind of the root URL</a> <a data-t="438">of some application.</a> <a data-t="439">A Shiny server would go in and inspect, is that application already online?</a> <a data-t="442">So let's say it maps some URL to app one.</a> <a data-t="448">So it would go through and say, is app one already online?</a> <a data-t="451">If it is, it'll proxy the traffic through.</a> <a data-t="453">If it's not, it'll start app one and then proxy the traffic through.</a> <a data-t="458">And then just kind of continue to keep these channels open between the user and the application.</a></p>

<p><a data-t="464">So what this looks like at a network level, the first request is going to be for kind</a> <a data-t="468">of the root URL of that application.</a> <a data-t="470">And then that's going to return some HTML that is going to kind of point the browser</a> <a data-t="475">to different static assets that may be of relevance to CSS files and JavaScript files</a> <a data-t="479">and things like that.</a> <a data-t="481">And so it'll open up more connections for all of those files.</a> <a data-t="485">And then eventually it'll actually open a WebSocket or WebSocket-like connection that</a> <a data-t="490">will be persistent between that browser and that Shiny application.</a> <a data-t="494">And that's kind of how Shiny, the Shiny metric works is via that WebSocket or that WebSocket-like</a> <a data-t="499">channel that allows the user and the Shiny process to communicate back and forth for</a> <a data-t="505">as long as that connection is open.</a></p>
