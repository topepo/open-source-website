<h3>API basics</h3>

<p><a data-t="58">So API basics.</a> <a data-t="58">There are a lot</a> <a data-t="58">of data accessible over the Internet these days and a lot of it is made available via</a> <a data-t="63">APIs, which are application programming interfaces.</a> <a data-t="69">These allow programs to talk to each other</a> <a data-t="69">based on a communication contract.</a> <a data-t="75">If you ask me a question in a way I understand, I</a> <a data-t="75">will give you an answer.</a> <a data-t="81">So there are two important parts to HTTP communication over</a> <a data-t="81">the Web.</a> <a data-t="86">There's the request, which is data sent to the server from the client, and then</a> <a data-t="86">the response, which is the data that the server sends back to that client.</a> <a data-t="92">Now this is a very</a> <a data-t="92">oversimplified view of the client-server communication, but the basics of it are ask a question and</a> <a data-t="97">get a response.</a></p>

<p><a data-t="97">And this is predicated on the fact that the client and server need to</a> <a data-t="102">understand each other's language, so to speak.</a> <a data-t="108">The client can only send a request to the</a> <a data-t="108">server that the server knows how to handle, and then the server will send that response</a> <a data-t="112">back to the client, and the client will need to know how to parse that response to get</a> <a data-t="115">the data out based on the question that was asked.</a> <a data-t="122">Web APIs usually provide read and or</a> <a data-t="122">write access to data stores when you're talking about Web APIs, and they allow you to access</a> <a data-t="128">data on an ongoing basis.</a> <a data-t="133">This is significant since it allows you to write scripts that</a> <a data-t="133">pull an API to get the most up-to-date data.</a> <a data-t="140">Not only will your analyses be current, but</a> <a data-t="140">when the data changes, your scripts don't have to.</a> <a data-t="145">Depending on how you design that</a> <a data-t="145">script, data frames and plots will update automatically when your data changes without</a> <a data-t="149">you having to download a new CSV or import data manually.</a></p>

<p><a data-t="155">So as an example of this request</a> <a data-t="155">response, I've put down here a curl request.</a> <a data-t="163">Curl is a utility in the Unix world that allows</a> <a data-t="163">you to make HTTP requests.</a> <a data-t="170">In this case, I'm calling the OMDB API.</a> <a data-t="170">This is the Online Movie</a> <a data-t="170">Database API, and I'm giving it a request with a query parameter, T for title and R for</a> <a data-t="177">response format.</a> <a data-t="184">So in this case, I'm saying, OMDB, give me the data for the title, the</a> <a data-t="184">movie named Clue, and send it back to me in JSON format.</a> <a data-t="189">The server understands this</a> <a data-t="189">request because you've put it into the format that it expects, and so it sends a response</a> <a data-t="194">that contains the data you asked for, the title, the year, the rating, et cetera.</a> <a data-t="199">Again,</a> <a data-t="199">this is a very simplistic example, but it gives you an idea of sort of how simple it</a> <a data-t="204">is to ask a question and receive an answer and pull that data into the scripts that you're</a> <a data-t="209">writing every day.</a></p>

<p><a data-t="217">Web APIs are organized into resources that are often in a particular</a> <a data-t="217">hierarchy.</a> <a data-t="223">In this example, there's a map that if you follow down from the top, you</a> <a data-t="223">can get information about accounts, a particular account by its ID, the lists that a particular</a> <a data-t="230">account has, the ID of those lists, the campaigns, the subscribers, the web forms, et cetera.</a> <a data-t="236">So these are often organized in maps or webs of information, and if you know how to get</a> <a data-t="241">to those different levels of information, you can construct a URL and ask for very specific</a> <a data-t="246">things all the way down at the bottom, like clicks and tracked events in this case.</a></p>

<p><a data-t="251">Now that presumes that you understand a little bit about the API that you're trying to use.</a> <a data-t="257">Fortunately, most of it has, many of them, I should say most of them, have decent documentation.</a> <a data-t="263">This example is from the Star Wars API.</a> <a data-t="263">So in this case, you can see across the left</a> <a data-t="268">the kinds of information you can learn from this documentation.</a> <a data-t="272">There's a getting started</a> <a data-t="272">guide here on the first page.</a> <a data-t="278">But the things that you need to understand are the resources</a> <a data-t="278">towards the bottom, so that you know what you can get and how to get it.</a> <a data-t="283">For example,</a> <a data-t="283">this being the Star Wars API, you can get information about people, films, starships,</a> <a data-t="287">planets, vehicles, et cetera.</a> <a data-t="292">And the beauty of this documentation, this particular documentation</a> <a data-t="292">is very good, so it'll tell you exactly what URL to use to get to that information, and</a> <a data-t="298">it'll give you an example response as well.</a> <a data-t="303">So it will tell you how to ask the question</a> <a data-t="303">and what format the answer will be in, so that you can then parse it appropriately in</a> <a data-t="307">your script.</a></p>

<p><a data-t="313">The important part here is that you need to learn that this, learn this API contract in</a> <a data-t="317">order to use it effectively.</a> <a data-t="321">You need to know how to ask the question and how the answer</a> <a data-t="321">will be sent back to you.</a> <a data-t="327">Fortunately, again, these documentation examples are often quite</a> <a data-t="327">good, because they want you to be able to use it successfully as well.</a></p>

<h3>Tools for accessing APIs in R</h3>

<p><a data-t="339">So to access these APIs and pull this information into R, you have a couple of different options.</a> <a data-t="343">In some cases, people have already written packages that will wrap those API calls for</a> <a data-t="343">a given service.</a> <a data-t="350">The examples I've provided here are for AWS S3.</a> <a data-t="350">This is S3 buckets in</a> <a data-t="350">the Amazon Web Services universe.</a> <a data-t="356">Our Google Analytics will get you information about Google</a> <a data-t="356">Analytics based on the ID code that you have for your website.</a> <a data-t="363">ACS is for the American</a> <a data-t="363">Census information, et cetera.</a> <a data-t="367">So some people have already wrapped these APIs into packages</a> <a data-t="367">that are very easy to use.</a> <a data-t="373">You do, I mean, if you can use them, by all means do.</a> <a data-t="373">They</a> <a data-t="373">make your life a lot easier.</a> <a data-t="376">You do need to be aware that the people writing them may</a> <a data-t="376">not have the same goals as you do.</a> <a data-t="382">So they've written it to solve their problem and then</a> <a data-t="382">made it public.</a> <a data-t="386">But if you have other questions that you need to ask or if there's particular</a> <a data-t="386">end points that aren't covered in that package, you may need to write that yourself.</a> <a data-t="391">Or there</a> <a data-t="391">could be no package for the data source that you're trying to reach, in which case you'll</a> <a data-t="394">write it yourself with some other tools.</a></p>

<p><a data-t="401">I recommend </a><a href="/software/httr/">HTTR</a><a data-t="401"> or Hitter for making the requests.</a> <a data-t="407">And then JSON Lite or </a><a href="/software/xml2/">XML2</a><a data-t="407"> for parsing the response.</a> <a data-t="407">The most common response formats</a> <a data-t="407">of API calls are JSON and XML, and these two packages are extremely good at parsing that</a> <a data-t="413">data for you.</a> <a data-t="423">Once you have the data in your script, you can wrangle the data the way that</a> <a data-t="423">you normally do with data that comes in using the various packages in the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="423"> to get</a> <a data-t="428">your data in an orderly rectangle so that you can use it for your analyses and for your</a> <a data-t="433">plots.</a></p>
