<h3>One-way vs two-way widget bindings</h3>

<p><a data-t="106">So a great example of this is the quack package, which is a new Python package</a> <a data-t="111">that makes it super easy to, you just give it a data frame and it creates a</a> <a data-t="115">much more sophisticated interactive table.</a> <a data-t="120">And this table sort of just comes with the ability to drill down</a> <a data-t="124">into certain values of your data.</a> <a data-t="127">So it gives you these nice graphics at the top where you can choose certain values</a> <a data-t="131">here, I'm choosing certain values of sex and a certain range of height.</a> <a data-t="135">And that will drill down and just to the rows that, um, match those queries.</a></p>

<p><a data-t="142">And in addition here, notice that I also executed another code cell after</a> <a data-t="147">interacting with the table to get the SQL attributes, uh, of this widget,</a> <a data-t="153">which actually captures the filtered state of the data.</a> <a data-t="158">So it's almost as if we can in Python through the Python object, access</a> <a data-t="163">information about what the user is doing with the widget, right?</a></p>

<p><a data-t="170">So these are two really great examples that helped me kind of frame this idea</a> <a data-t="175">that there are two broad categories of widgets when you're working in Jupyter.</a></p>

<p><a data-t="181">Um, the most broad category, which really encompasses all widgets</a> <a data-t="185">is the idea of a one-way binding.</a> <a data-t="188">Which really all I mean by that is that you have some Python code that you're</a> <a data-t="192">executing to get a web-based results.</a> <a data-t="195">And any interactivity that you get from that widget, it's all</a> <a data-t="199">happening inside of JavaScript.</a> <a data-t="201">There's no communication back to Python.</a> <a data-t="204">Everything is just kind of baked in and predefined.</a> <a data-t="207">And, uh, if there's any interactivity, it's likely JavaScript.</a></p>

<p><a data-t="212">Now the two-way binding has the additional ability to send data and</a> <a data-t="217">information back and forth between Python and JavaScript.</a> <a data-t="223">And actually with the Jupyter widget two-way binding, they have this</a> <a data-t="227">concept of state, both existing, both in JavaScript and Python.</a> <a data-t="234">And, um, the framework that makes this possible, uh, will sort of</a> <a data-t="240">automatically in some sense, detect when state changes on the front end,</a> <a data-t="244">say in JavaScript through drilling down in our table, automatically, there will</a> <a data-t="249">be some messages sent back to Python to update the Python object with that state.</a> <a data-t="254">And also vice versa, you can also update, uh, Python objects</a> <a data-t="259">to then, uh, update the front end.</a></p>

<p><a data-t="266">So one of the nice things about working with a one-way binding, as you</a> <a data-t="271">can imagine with the two-way binding, there's a whole lot more complexity</a> <a data-t="274">and overhead with, you know, passing data back and forth between Python and</a> <a data-t="280">JavaScript, if the one-way binding kind of does the job for you, that's a great</a> <a data-t="286">option because you can do kind of better, simpler things with it.</a> <a data-t="290">Like, you know, not even working in notebook, you could take that Python</a> <a data-t="294">code to create like a static webpage, maybe inside of a </a><a href="/software/quarto/">Quarto</a><a data-t="294"> document.</a></p>

<p><a data-t="301">But it's also kind of fundamentally more limited in this sense.</a> <a data-t="306">Uh, mainly, you know, because we can't access any information about what's</a> <a data-t="312">happening on the front end with our Python code, like we can't take, you</a> <a data-t="317">know, what the user is doing to inform, um, other things in a web application.</a></p>

<p><a data-t="325">So really what I mean by that, like, uh, the, the thing that makes, um, uh, this</a> <a data-t="332">interesting from the interactivity perspective is that we can kind of access</a> <a data-t="337">the state of what's happening on the front end and link multiple views in the</a> <a data-t="341">sense of like they're sharing the state of what's happening to that Python object.</a></p>

<p><a data-t="349">So if we go back to our interactive table with drill down capabilities,</a> <a data-t="354">uh, you can imagine since we can actually access like what filtered data</a> <a data-t="358">has been filtered in this table, we could take that data and route it to</a> <a data-t="363">another view that maybe gives us like a higher dimensional picture of what's</a> <a data-t="367">happening just in that filtered data.</a></p>

<p><a data-t="375">The other big thing that the two way binding, um, gives us as a</a> <a data-t="380">feature is this idea of mutable state.</a> <a data-t="384">Which really means, you know, imagine you have a widget that you've just rendered</a> <a data-t="389">and maybe you want to, um, just, uh, change a particular value of a particular</a> <a data-t="397">attributes, which you can do is just mutate that Python objects, give it a</a> <a data-t="401">new value for certain attributes.</a> <a data-t="404">And just that change in states is sent to the front end.</a> <a data-t="408">So in order to update a view, you can do it a lot more performantly by just</a> <a data-t="412">sending only what has changed, um, from Python to JavaScript, right?</a></p>

<p><a data-t="420">So to give you a simple, concrete example of this, imagine I had a</a> <a data-t="424">dropdown where I could choose different cities and just update the center of</a> <a data-t="429">a map from, um, one city to the next.</a> <a data-t="433">If I were to try to do this by re rendering the map from scratch, every</a> <a data-t="438">time you would get a little bit of graying out and flickering every time</a> <a data-t="442">that map updates, but by just telling the front end that only the center of</a> <a data-t="446">the map has changed, it's able to smoothly transition from one location to the next.</a></p>

<h3>The IPyWidgets standard</h3>

<p><a data-t="457">So going back to our diagram here, there's actually like a formal name</a> <a data-t="461">and standard and implementation of this two-way binding, which is</a> <a data-t="465">provided by the IPyWidgets package.</a> <a data-t="469">And you as a user might not actually end up importing from IPyWidgets.</a> <a data-t="475">Uh, you may, maybe you just want to use, um, the Quack package or Plotly or Altair.</a> <a data-t="480">Um, and all of these packages that provide this two-way binding interface</a> <a data-t="485">are going to inherit from this widget class from IPyWidgets, because it kind</a> <a data-t="491">of provides the foundation for passing that state back and forth</a> <a data-t="495">between Python and JavaScript.</a></p>

<p><a data-t="499">And as we'll see shortly, the ShinyWidgets package has a render widget</a> <a data-t="503">decorator that allows us to render IPyWidgets in </a><a href="/software/shiny-r/">Shiny</a><a data-t="503"> and leverage all of</a> <a data-t="509">those nice things that come with the two-way binding.</a></p>

<p><a data-t="516">And most of the projects that you've heard of, um, are going to</a> <a data-t="520">provide two-way binding interfaces.</a> <a data-t="523">So big names like Plotly, </a><a href="/software/leaflet/">Leaflet</a><a data-t="523">, Bokeh, Vega, Altair, Mapbox, um,</a> <a data-t="529">has a binding called PyDeck.</a> <a data-t="530">All of these projects tend to provide a two-way binding interfaces.</a></p>

<p><a data-t="537">And more than that, a lot of these big projects will also provide a</a> <a data-t="541">one-way binding interface, which, you know, makes a whole lot of sense, uh,</a> <a data-t="546">for the, especially from their perspective that sometimes, you know,</a> <a data-t="551">the one-way binding does the job.</a> <a data-t="553">Maybe you don't even want to work, uh, in a Jupyter notebook.</a> <a data-t="557">Maybe you just want a script that, um, creates a static webpage or</a> <a data-t="562">embedded in a Quarto document.</a> <a data-t="564">Um, just having basically the static results with JavaScript, um, you</a> <a data-t="570">know, requires a lot less overhead.</a></p>

<p><a data-t="575">So, um, ShinyWidgets can actually handle both of these cases.</a> <a data-t="579">And what it will do is it will automatically transform a one-way binding.</a> <a data-t="582">If you give it, if you give this one-way binding to, um, ShinyWidgets, it will</a> <a data-t="588">automatically transform the one-way binding to a two-way binding.</a> <a data-t="593">And this makes getting started with ShinyWidgets, um, a whole lot easier</a> <a data-t="598">because a lot of these packages like Altair, I'm just using Altair as an</a> <a data-t="602">example here, but this is true for like Plotly and a lot of other packages.</a> <a data-t="606">Where the one-way binding is kind of at the heart of their</a> <a data-t="611">documentation and code examples.</a> <a data-t="613">So if you actually look up Altair, you'll see Altair chart all over the place.</a> <a data-t="618">So it's very convenient to just kind of dump those code examples</a> <a data-t="622">into ShinyWidgets and it just works.</a></p>

<h3>Why ShinyWidgets works: the architecture</h3>

<p><a data-t="901">All right, and this point, I would just kind of like to take a step</a> <a data-t="904">back and have us take a moment to appreciate that IPy widgets were</a> <a data-t="911">designed for notebooks, not for Shiny.</a> <a data-t="917">And the fact that we can even create something like Shiny widgets and,</a> <a data-t="922">you know, leverage over a decade's worth of work in the Python community</a> <a data-t="928">inside of Shiny for Python like this is really a matter of technical</a> <a data-t="933">coincidence, and like there was no guarantee when we started Shiny for</a> <a data-t="938">Python, like we were kind of scratching our heads, how are we going to, you</a> <a data-t="943">know, allow people to use these rich interactive outputs in a way that</a> <a data-t="947">they're kind of familiar with already?</a></p>

<blockquote><a data-t="950">And it's kind of a miracle almost in some sense that there was two kind of</a> <a data-t="953">parallel worlds evolving over the last decade.</a></blockquote>

<p><a data-t="950">And it's kind of a miracle almost in some sense that there was two kind of</a> <a data-t="953">parallel worlds evolving over the last decade.</a> <a data-t="957">IPy widgets kind of came out around the same time as Shiny, like</a> <a data-t="961">2012 or something like that.</a> <a data-t="964">And there's been over a decade of a lot of development and work</a> <a data-t="968">in building upon this standard.</a></p>

<p><a data-t="973">So to give you a little bit of a picture of how we're able to kind of</a> <a data-t="977">hook into the IPy widgets architecture, let's actually kind of take a kind of</a> <a data-t="985">a simplified view of the architecture that's necessary to power IPy widgets.</a> <a data-t="993">And here I have this diagram where on the left-hand side, you know, we have</a> <a data-t="998">the Python end and on the right-hand side, we have the front end in JavaScript.</a> <a data-t="1004">And whenever you create and display a new Jupyter widget in Python, what</a> <a data-t="1011">happens is the widget object sends some data and some information to what the</a> <a data-t="1016">IPy widgets developers call a com layer, which stands for communication layer,</a> <a data-t="1022">that will kind of orchestrate a message passing in a way that will power all</a> <a data-t="1029">of the widgets necessary in a notebook.</a></p>

<p><a data-t="1032">So it kind of receives some data from the widget, passes that along to a front</a> <a data-t="1037">end where there's kind of a similar communication layer.</a> <a data-t="1041">And then that communication layer will then kind of know how to get to the</a> <a data-t="1046">necessary JavaScript libraries and take that data and actually</a> <a data-t="1050">render it as a interactive view.</a></p>

<p><a data-t="1056">So once we're rendered and the user kind of maybe comes in and interacts with the</a> <a data-t="1062">widget and introduces some changes to state, that widget view will know how to</a> <a data-t="1067">send data to the com layer to say, Hey, I have some state that has changed.</a> <a data-t="1072">Com layer passes that back all the way back to the Python object on the back end.</a></p>

<p><a data-t="1082">And one of the interesting bits here is, you know, the widget implementations</a> <a data-t="1086">like Plotly, Leaflet, Altair, they're all going to be kind of responsible for</a> <a data-t="1090">this part of the stuff that happens that's very close to the actual widget</a> <a data-t="1095">object and the JavaScript view.</a> <a data-t="1099">But kind of more of the backend architecture of the com layer, this is</a> <a data-t="1104">all provided by IPyWidgets and is just kind of generically handles all</a> <a data-t="1110">different widget implementations.</a></p>

<p><a data-t="1114">And it turns out these IPyWidget developers were very thoughtful about how</a> <a data-t="1118">they actually design the software for these com layers, where they've designed</a> <a data-t="1122">them to be abstract and pluggable by other environments so that you can</a> <a data-t="1126">actually kind of, they've been thoughtful about somebody like me coming in and</a> <a data-t="1131">wanting to use these widgets in another environment.</a> <a data-t="1137">And it just so happens, like that's not necessarily good enough to, you know,</a> <a data-t="1142">leverage widgets in a framework like Shiny, but it just so happens that Shiny</a> <a data-t="1146">works through message passing in a very similar way that allows us to create</a> <a data-t="1152">something like Shiny widgets and have it be interoperable with Shiny.</a></p>

<p><a data-t="1160">So in summary, widgets are either one-way or two-way bindings.</a> <a data-t="1164">The two-way bindings are called IPyWidgets and are much more capable.</a> <a data-t="1170">And, you know, you're able to achieve a lot more with interactivity</a> <a data-t="1174">and performance with these widgets.</a> <a data-t="1177">And thanks to their open standard and the thoughtfulness of the IPyWidget</a> <a data-t="1180">developers, we can leverage IPyWidgets inside of Shiny for Python.</a></p>
