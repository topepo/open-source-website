<h3>Factors in R and their quirks</h3>

<p><a data-t="294">And in R, you probably know that we represent categorical data as factors.</a> <a data-t="301">And factors consist of a set of values and then an ordered set of valid levels.</a></p>

<p><a data-t="351">And when I started thinking about the paper that I was going to write, I was thinking about all these problems that come up when you're doing data wrangling in real data problems from this experience of working with my students.</a> <a data-t="401">And so I said, you know, I could write a paper on a couple different subjects, but here's one, you know, I could talk about factors and how to recode them.</a></p>

<p><a data-t="413">So we started off this paper by showing some of the ways in which factors behave unexpectedly.</a> <a data-t="422">So one of the ways is that if you have some numeric data and you turn it into a factor, it sort of looks the same because it has those same values, but then it has the levels associated with it.</a> <a data-t="437">And if you convert back to a numeric, then you don't get the numbers that you expect.</a> <a data-t="444">Okay? So instead of getting back 20, 20, 10, 40, 10, I get back 2, 2, 1, 3, 1.</a> <a data-t="449">And this is the sort of thing that can really mess you up when you're doing analysis with factors.</a> <a data-t="458">Because you think that you're going to get back the value, but you get back sort of the numeric level instead.</a></p>

<p><a data-t="464">There's other things like this where, you know, if you create a factor variable and give levels that don't match, you're going to get an NA.</a> <a data-t="469">Again, you don't get a warning or a message or an error.</a> <a data-t="475">If I had saved this into an object, it just would have seemed like it worked, but I've lost that level of A.</a> <a data-t="481">So there are things like this that you learn to kind of work around when you're working with factor data.</a></p>

<h3>The stringsAsFactors problem</h3>

<p><a data-t="491">And I think these frustrating issues were much more common back in the days of Yore when we were using read.csv to read in our data files.</a> <a data-t="498">Because read.csv has this argument of strings as factors, and that was by default set to true.</a> <a data-t="505">So any time you had any character or string variable, it was getting read in as a factor.</a> <a data-t="511">And so people were running into these frustrating issues a lot.</a></p>

<p><a data-t="517">Roger Peng has a great blog post that's strings as factors, an unauthorized biography, and he talks about some of the historical reasons why this default was set to be true.</a> <a data-t="529">And I think the most salient one is that it used to save space in memory.</a> <a data-t="534">So instead of having to save in quotes, you know, like high, medium, low, many, many times as strings, it would save one, two, three, the numerals many times, and then just keep those labels once.</a> <a data-t="546">It turns out that even base R is no longer doing that, so it's not a memory or storage issue anymore.</a></p>

<p><a data-t="554">But people were getting frustrated with strings as factors not defaulting to false.</a> <a data-t="561">This led to the strings as factors equal hell no movement.</a> <a data-t="567">At JSM one year, I think </a><a href="/people/jenny-bryan/">Jenny Bryan</a><a data-t="567"> made these ribbons that you could put on your name tag to represent how you felt about that issue.</a></p>

<p><a data-t="503">And so when we moved into the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="503"> kind of world where people are using read underscore CSV, that got baked into the philosophy that you shouldn't be forcing some data format on your string or character vectors, because that might not be what you want, and you could run into these icky issues where things can get disconnected in a strange way.</a> <a data-t="540">So now if you read underscore CSV, you're not going to encounter factors that much anymore.</a></p>

<h3>Why we still need factors</h3>

<p><a data-t="507">So the problem is you sometimes still need factors.</a> <a data-t="513">We can't get rid of them altogether.</a> <a data-t="518">In particular, if you're doing modeling, even just using a simple linear model, but you're including a categorical variable, a factor variable in your analysis, those need to be formatted as factors so that you can choose which level is your reference level.</a> <a data-t="548">And you can't do that with character strings because R will just pick the first level alphabetically.</a></p>

<p><a data-t="557">You also need to work with factors if you want to reorder elements that are maybe in your ggplot.</a> <a data-t="567">So if you have a number of items on your plot and you want them in some other order, you need those to be factors so that you can reorder them properly.</a></p>

<p><a data-t="554">So we still need to use factors, and up until reasonably recently, this meant doing a bunch of ugly base R code.</a> <a data-t="567">So this is an example of some ugly base R code that I wrote for another fully reproducible journal article in 2013.</a> <a data-t="573">So if you want to look at some vintage R code, I recommend it.</a> <a data-t="579">This paper will maybe come out this year, so it's now six years old and it's really getting pretty classic.</a> <a data-t="586">So if you remember working with factors, this is the sort of thing that you would be doing, right?</a> <a data-t="591">You'd be overriding the levels or you'd be reordering the levels by using square brackets and the C function to reorder them.</a></p>

<h3>Bad approaches and fragile base R code</h3>

<p><a data-t="618">So R users love to talk about how Excel forces data to be dates, for example.</a> <a data-t="629">When you open a spreadsheet, then anything that looks vaguely like a date becomes a date.</a> <a data-t="636">But R has similar issues where factors can ruin your data in sort of a horrifying way.</a></p>

<p><a data-t="642">So in the paper that Nick Horton and I wrote, we were focusing on the GSS data, and so what we showed were some of the bad approaches that you can use in base R.</a> <a data-t="655">So if you have a variable about income and you want to order this so that it's in an appropriate order, right?</a> <a data-t="665">We want far above average to come first, and then above average, average, et cetera, one thing that you can do is overwrite the levels.</a> <a data-t="675">So that's what I was doing just a minute ago, and you can watch and you can see how this is wrecking my data.</a> <a data-t="680">So watch like average, about 1,000 observations, and now average 666, okay?</a> <a data-t="692">So because I'm overwriting the levels outside of the factor command, I've now broken the relationship between my levels and labels, and all of my data is now invalid.</a> <a data-t="703">And again, it's not going to give you an error or a warning.</a></p>

<p><a data-t="709">Another bad approach in base R, again, you could watch the average category, is you use the levels there and do it numerically in some way.</a> <a data-t="716">So again, this is looking similar to my code that I was showing, except I think mine happened to continue to work.</a> <a data-t="728">And so Nick and I, you know, said you shouldn't be using this kind of base R code.</a> <a data-t="733">If you're going to use base R, you should be using a more robust approach.</a> <a data-t="742">So you should be doing something where you're going to use the factor command, you're going to put levels within it, and now things are working much better.</a></p>

<p><a data-t="758">If you watch, for example, above average 483, that count is going to stay appropriate, right?</a> <a data-t="763">But something has happened to average.</a> <a data-t="763">Average actually did not come through.</a> <a data-t="768">Even though I'm using a more robust base R approach, I'm still losing my average values because I made a typo.</a> <a data-t="776">I put a trailing space after the word average when I was recoding my levels.</a></p>

<p><a data-t="776">So we called this a fragile approach, sort of a reference to the idea of software brittleness, where if one little thing changes, then your code can break.</a> <a data-t="790">And so we're thinking about if you had a collaborator who sent you a new spreadsheet and they happened to change the phrasing of one of the levels, or if you are a little bit fat fingering and you hit a space, then you're going to, again, mess up your data.</a> <a data-t="810">But this approach at least is not losing the counts for everything.</a> <a data-t="815">It's just that one place where I made the typo.</a></p>

<h3>Defensive coding and takeaways</h3>

<p><a data-t="1081">So part of my, the goal of this talk is to evangelize for forcats and the tidyverse and get you to use it.</a></p>

<p><a data-t="1086">I think there's also something to be said about defensive coding here.</a> <a data-t="1092">So the way that you make sure that your examples are working is you have to, like, look at your data over and over again.</a> <a data-t="1104">So this is from an issue on that same paper.</a> <a data-t="1104">This was someone's example that had to do with categorical data, factor data, splitting the data into testing and training data.</a> <a data-t="1109">You can get different factor levels in the two datasets, and then, again, when you do the modeling and the predictions, it won't throw an error, but everything is going to be wrong.</a> <a data-t="1119">So there's lots of ways that working with factors can go wrong.</a></p>

<p><a data-t="1124">And so one of your methods of defense is just summary, summary, summary.</a> <a data-t="1131">So you saw every one of those examples, I ran the summary first, then I did something with my factors, and then I ran it again.</a> <a data-t="1134">And you could use count or something from the tidyverse if you wanted to.</a> <a data-t="1138">This is just a little more compact to show on the screen.</a></p>

<p><a data-t="1142">The other thing that we talked about is maybe you could get some testing worked into your paper, right?</a> <a data-t="1148">So you have the number of levels, assert that the level should be equal, equal to three, or test that the levels should be female and male, just so that you can keep track of your analysis.</a> <a data-t="1159">Expect equivalent doesn't work if the levels are the right levels, but out of order, so that's on my wish list for future versions of the package.</a></p>

<p><a data-t="1173">And so my takeaways from this talk are essentially that you should use forcats.</a> <a data-t="1179">That's going to solve the vast majority of your problems.</a> <a data-t="1185">But like I said with the testing and training dataset, it doesn't always solve every single problem, so you really need to be practicing defensive coding.</a> <a data-t="1196">And that probably means that summary is your friend, and maybe you want to be asserting that and testing that things in your analysis.</a> <a data-t="1201">And if you want to see more horror stories and have a good way to explain to people why the tidyverse is better, not just for people and their thinking, but actually for making your data work, that's the paper, and thank you very much.</a></p>
