<h3>LLMs and traditional ML in tandem</h3>

<p><a data-t="715">So I think we can use LLMs in tandem with traditional machine learning development and production models to help our workflows.</a> <a data-t="725">So I think we can, for example, when developing a traditional machine learning model, we can use the LLM to classify data initially, a human to review and correct that data.</a> <a data-t="735">And then traditional machine learning algorithms to train a model, which is really nice because those models will typically be free to use.</a> <a data-t="745">And in production, we can add a step at the beginning to use our machine learning, current existing one machine learning model to classify the data.</a> <a data-t="755">And then like the crimes example that I showed, also use an LLM to classify the data.</a> <a data-t="763">And then we would compare them for disagreement and different levels of certainty to have a human compare and review those responses and then fill it in with the correct decision that they end up making.</a> <a data-t="779">And then we can retrain the machine learning model so that it's better.</a></p>

<h3>Caveats and limitations</h3>

<p><a data-t="783">So what's the catch?</a> <a data-t="783">Frontier models are a black box.</a> <a data-t="787">Pretty much everything that's happening inside them, the decision making are obscured by billions of parameters, unknown training data sets, secretive training and fine tuning algorithms that really require just using very that only have documentation for very small pieces of this.</a> <a data-t="809">So some of the frontier model companies are more transparent about this than others.</a> <a data-t="809">But even sometimes they have to do research on their models to understand what's going on inside.</a> <a data-t="822">And on the other hand, local open source models tend to perform poorly.</a> <a data-t="822">So it's best to be really careful or even avoid using them altogether.</a></p>

<p><a data-t="833">Last, LLMs are not deterministic, so they're not always consistent or reliable.</a> <a data-t="833">And here's an example.</a> <a data-t="839">If we ask an LLM, in this case, chatGBT5, to tell us a joke, it gives us in the first run, why don't scientists trust atoms?</a> <a data-t="839">Because they make up everything.</a> <a data-t="849">And then in the second run, they give us a different joke, or the model provides a different joke, which is, why did the scarecrow win an award?</a> <a data-t="849">Because he was outstanding in his field.</a> <a data-t="861">And so basically, we provide the same input and get a different output.</a> <a data-t="861">And this tends to happen more when we give it a general prompt like this.</a> <a data-t="872">And it becomes less frequent, I think, when we ask it, for example, for structured data, we provide a little bit more prompting around it.</a> <a data-t="880">But what it means is that we won't always get the same answer.</a> <a data-t="880">And we really need to validate and look at our responses and understand that they could change even from one run to another.</a></p>
