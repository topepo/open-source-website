<h3>Intro to renv and reproducibility</h3>

<p><a data-t="155">This is from the first sentence of the website.</a> <a data-t="158">The renv package helps you create reproducible environments</a> <a data-t="162">for your R projects.</a> <a data-t="164">So let's break down this important word, reproducible.</a></p>

<p><a data-t="168">There's a spectrum of reproducibility.</a> <a data-t="170">On the lowest end of the spectrum, we have code,</a> <a data-t="173">and the middle ground, we have code</a> <a data-t="176">and associated package versions at the time of execution.</a> <a data-t="180">And on the highest order of reproducibility,</a> <a data-t="182">we have code, package versions, R version,</a> <a data-t="186">your system dependencies, and your operating system.</a> <a data-t="189">And where renv fits is in the middle,</a> <a data-t="192">with code and package versions,</a> <a data-t="195">which is both the beauty and the challenge of renv.</a> <a data-t="198">The beauty being that it doesn't have all of the overhead</a> <a data-t="200">of the upper end of the spectrum,</a> <a data-t="202">and the challenge being that it doesn't address</a> <a data-t="205">the overhead of the upper end of the spectrum.</a></p>

<p><a data-t="209">We also need to talk about this really important word helps.</a> <a data-t="213">The renv package helps you</a> <a data-t="215">create reproducible environments.</a> <a data-t="217">It does not do it for you.</a> <a data-t="219">When you use renv,</a> <a data-t="221">there is an immense amount of user responsibility.</a></p>

<p><a data-t="226">So here's an example project.</a> <a data-t="227">I'm going to call it jsonlite-example,</a> <a data-t="229">and I initialized it in February of this year.</a> <a data-t="233">At this point, renv is not activated on this project.</a> <a data-t="236">I have an rproj file,</a> <a data-t="238">and I have a single file called script.r,</a> <a data-t="240">which has one line on it, library jsonlite.</a> <a data-t="244">I can initialize renv with renv init,</a> <a data-t="247">in which case I get the message</a> <a data-t="248">that I'm recording versions of R</a> <a data-t="250">and packages in my lock file,</a> <a data-t="252">and in my project directory,</a> <a data-t="253">I additionally get a .er profile,</a> <a data-t="255">an renv lock, and an renv directory.</a></p>

<p><a data-t="259">And then when I'm iterating on this project,</a> <a data-t="261">I'll do some combination of installing new packages</a> <a data-t="264">and snapshotting to record those to my lock file,</a> <a data-t="266">and this is pretty smooth sailing.</a></p>

<p><a data-t="269">And so what I have is I have this initial project,</a> <a data-t="273">and my R versions and my package versions,</a> <a data-t="275">they are all released at different points in time.</a> <a data-t="279">They are all available for different lengths of time,</a> <a data-t="282">and I'm taking a single slice of that point in time</a> <a data-t="285">that represents latest available versions at that date.</a> <a data-t="290">So I have my initial project.</a></p>

<h3>Package installation and libraries</h3>

<p><a data-t="293">Now let's talk about package installation.</a> <a data-t="297">A library is a directory containing installed packages.</a> <a data-t="300">It's just a folder on your machine</a> <a data-t="302">where those installed packages go.</a> <a data-t="304">When I don't have renv activated on my project,</a> <a data-t="307">I have two library paths available to me.</a> <a data-t="310">One's a system path, one's a user path,</a> <a data-t="313">and I want you to notice</a> <a data-t="314">that they're both associated with R version of 4.4,</a> <a data-t="318">which is why when R 4.5 is released,</a> <a data-t="320">Hadley posts on social media,</a> <a data-t="322">happy reinstalling all your R packages day</a> <a data-t="325">to all those who celebrate,</a> <a data-t="326">because you will physically get a new directory</a> <a data-t="329">on your machine in which you need to install those packages.</a></p>

<p><a data-t="334">And so when I don't have renv activated on my project,</a> <a data-t="337">I have a shared package environment.</a> <a data-t="340">I have one library that all of my projects point to.</a> <a data-t="344">That one library contains a single version</a> <a data-t="346">of JSON Lite 1.8.9,</a> <a data-t="348">and all of my projects are forced</a> <a data-t="350">to use that single version.</a> <a data-t="353">And again, this is all under the umbrella of R 4.4.</a></p>

<p><a data-t="359">When I do have renv activated on my project,</a> <a data-t="362">I again have two library paths available.</a> <a data-t="365">And again, these are associated with R version of 4.4.</a> <a data-t="369">Additionally, one of these is project-specific</a> <a data-t="372">for my JSON Lite example,</a> <a data-t="374">and the other one points to my global cache.</a></p>

<p><a data-t="378">So when I do have renv activated,</a> <a data-t="380">I have an isolated package environment.</a> <a data-t="383">I have a global cache that contains every version</a> <a data-t="386">of JSON Lite I could possibly want to use</a> <a data-t="388">across all of my projects,</a> <a data-t="390">and now all of my projects can be associated</a> <a data-t="393">with a different version of JSON Lite,</a> <a data-t="395">and this is under the umbrella of R 4.4.</a></p>

<p><a data-t="399">And when we think about any given project,</a> <a data-t="402">they have project layers.</a> <a data-t="404">We'll call this the project onion.</a> <a data-t="406">In our innermost layer,</a> <a data-t="407">we have our .rparge file and renv lock.</a> <a data-t="410">Then we have the packages that are associated</a> <a data-t="412">with our analysis, the R version,</a> <a data-t="414">the system dependencies, and the operating system.</a> <a data-t="417">So keep in mind where renv operates</a> <a data-t="419">in those innermost layer touching the packages.</a></p>

<p><a data-t="424">Now the renv lock is a JSON file</a> <a data-t="426">that contains information</a> <a data-t="427">from the installed packages description file.</a> <a data-t="430">Let's walk through this a bit.</a> <a data-t="432">It contains information about the versions</a> <a data-t="434">of R and JSON Lite and renv.</a> <a data-t="438">It contains information about the repositories,</a> <a data-t="440">which we just heard a lot about.</a> <a data-t="441">That's where you actually installed your package from.</a> <a data-t="444">If you haven't done any configuration on your machine,</a> <a data-t="447">your default repository probably points to CRAN</a> <a data-t="450">at the URL cloudrproject.org.</a> <a data-t="454">There's other repositories out there</a> <a data-t="457">that you can install packages from.</a> <a data-t="458">And we'll be talking more about</a> <a data-t="459">the Posit Public Package Manager, or P3M.</a></p>

<p><a data-t="464">We also should talk about this field, needs compilation.</a> <a data-t="467">JSON Lite says needs compilation is yes,</a> <a data-t="470">and renv says needs compilation is no.</a> <a data-t="474">Needs compilation means that this requires</a> <a data-t="477">compiling tools outside of R,</a> <a data-t="479">which touches on your system dependencies</a> <a data-t="481">and your operating system,</a> <a data-t="483">which are layers that renv does not touch.</a></p>

<p><a data-t="488">Now we have to talk about binaries.</a> <a data-t="491">So binaries are compiled R packages.</a> <a data-t="494">They are very specific to your operating system,</a> <a data-t="497">your R version, your package version, and architecture.</a> <a data-t="502">Typically, when we install packages,</a> <a data-t="503">we either install them from source or binary.</a> <a data-t="507">If we start from binary, it's a straight shot to install.</a> <a data-t="511">If we start from source, then our machine</a> <a data-t="514">has to compile it into a binary,</a> <a data-t="516">and then we get to install it.</a> <a data-t="518">So that needs compilation step happens</a> <a data-t="520">in between going from source to binary,</a> <a data-t="523">which means when we install from source,</a> <a data-t="525">this is a harder and slower process,</a> <a data-t="528">whereas if we install from binary,</a> <a data-t="530">this is an easier and faster process.</a> <a data-t="533">And typically, the reason why we'll do one over the other</a> <a data-t="536">is just simply the availability of the binaries.</a> <a data-t="539">They're not always available.</a></p>

<h3>Projects over time: freeze, manage, or update</h3>

<p><a data-t="542">So now, let's talk about projects over time.</a> <a data-t="547">I'm gonna pick up this project six months later,</a> <a data-t="550">in August of this year, and I'm gonna actively work on it.</a> <a data-t="554">And when I pick it up, the entire ecosystem has changed.</a> <a data-t="557">There's a new version of R, a new version of renv,</a> <a data-t="560">and a new version of Jsonlite.</a> <a data-t="563">And so now, I have to make some project decisions,</a> <a data-t="565">freeze, manage, or update.</a></p>

<p><a data-t="569">And we're gonna think about this</a> <a data-t="570">in this versions over time map,</a> <a data-t="572">where on one axis, we have our R version increasing,</a> <a data-t="576">and on the other axis,</a> <a data-t="577">we have our package version increasing.</a> <a data-t="579">And where we're starting with our initial project</a> <a data-t="582">is right here in the top left-hand corner</a> <a data-t="584">with R 4.4.2 and Jsonlite 1.8.9.</a></p>

<p><a data-t="590">So let's talk about the freeze option.</a> <a data-t="592">What does this mean in the context of renv?</a> <a data-t="596">So I'm gonna pose the question</a> <a data-t="598">that I am really excited about the new release of base R,</a> <a data-t="602">and I want to advance to R 4.5.1.</a> <a data-t="605">And so can I do this?</a> <a data-t="606">Can I advance to R 4.5.1,</a> <a data-t="609">but still use older versions of renv and Jsonlite</a> <a data-t="612">when I pick up my active project?</a></p>

<p><a data-t="615">And so if we're thinking about what's happening here,</a> <a data-t="618">we have moved forward in time, R 4.51 is out,</a> <a data-t="621">and I want to use it,</a> <a data-t="622">but I wanna move backwards in time</a> <a data-t="625">and use an older version of Jsonlite.</a> <a data-t="628">Can I do this?</a></p>

<p><a data-t="630">And let's be explicit about what we're doing here.</a> <a data-t="632">If we're using a new R version, we will get a new cache.</a> <a data-t="636">I had everything working just the way I wanted to</a> <a data-t="638">under R 4.4, and now when I'm using R 4.5,</a> <a data-t="642">my cache is empty, so I will be forced</a> <a data-t="644">to install new packages.</a></p>

<p><a data-t="647">And so we're going to open up this project</a> <a data-t="649">under R 4.5.1, and we're going to push the only button</a> <a data-t="653">we know how to push, which is renv restore.</a> <a data-t="657">And unfortunately, I get my failure message, right?</a> <a data-t="660">It says I am attempting to install Jsonlite</a> <a data-t="664">from source, and it failed.</a></p>

<p><a data-t="667">Instead of crying, let's push our second button.</a> <a data-t="670">Evaluate our project state</a> <a data-t="672">against our current, our configuration.</a> <a data-t="676">So my first question is, do I have the configuration</a> <a data-t="679">required for source compilation?</a> <a data-t="683">I'm on Windows 11, I have RTools 4.5,</a> <a data-t="687">and the answer is yes, I do.</a> <a data-t="689">All the machinery is there.</a> <a data-t="692">And I can programmatically check that</a> <a data-t="694">with a function package build check builds tools,</a> <a data-t="696">which yields the result, your system is ready</a> <a data-t="699">to build packages.</a> <a data-t="700">So that's kind of weird.</a></p>

<p><a data-t="703">Well, if installing from source is a problem,</a> <a data-t="706">can I just install from binary instead, right?</a> <a data-t="708">I can just bypass this whole compilation issue.</a> <a data-t="712">And that gets a little muddy.</a> <a data-t="714">So let's think about when binaries are generally available.</a> <a data-t="718">When I started this project in February of this year,</a> <a data-t="722">Jsonlite 1.8.9 was the latest and greatest</a> <a data-t="725">that was available on CRAN,</a> <a data-t="726">and the binary for this package</a> <a data-t="728">was available on CRAN at that time,</a> <a data-t="730">and also on Posit Public Package Manager P3M.</a> <a data-t="733">And the gray squares were future versions.</a> <a data-t="735">Now, six months later, CRAN hosts binaries</a> <a data-t="739">for latest available package versions.</a> <a data-t="741">So Jsonlite 2.0 is gonna be on CRAN, not 1.8.9.</a></p>

<p><a data-t="747">If I do wanna go back in time,</a> <a data-t="749">P3M does actually host binaries for older releases.</a> <a data-t="753">But where I'm at in the upper right-hand corner</a> <a data-t="758">of this matrix, there is no binary available</a> <a data-t="761">for this combination of things.</a></p>

<p><a data-t="766">So can I install from binary instead?</a> <a data-t="769">The answer is no.</a> <a data-t="771">Which leads me to think, am I doing something weird?</a></p>

<p><a data-t="775">So you get a little hint of this at the top of your console.</a> <a data-t="778">RM tells you, hey, you're using R 4.51,</a> <a data-t="782">but your log file was generated with R 4.4.2.</a> <a data-t="786">And that picture we just showed about binary availability</a> <a data-t="789">speaks to a general reflection</a> <a data-t="791">of what's going on in the R ecosystem.</a></p>

<p><a data-t="795">So in the R ecosystem, things work really well</a> <a data-t="799">at any given point in time when we're looking</a> <a data-t="801">at latest releases available at that point in time.</a> <a data-t="805">The diagonal is a good place to be.</a> <a data-t="809">The lower triangle is also an okay place to be.</a> <a data-t="814">We can do things like stay on R 4.4</a> <a data-t="816">and use a newer version of a package.</a> <a data-t="819">The upper triangle, no.</a> <a data-t="822">We don't want to be here.</a> <a data-t="823">This is undefined, uncharted territory.</a></p>

<p><a data-t="865">So am I doing something weird?</a> <a data-t="867">Yes.</a></p>

<p><a data-t="870">So the correct way to freeze a project</a> <a data-t="873">is to return to its initial state.</a> <a data-t="876">I want to revert back to that original R version.</a> <a data-t="881">So let's go back in time.</a> <a data-t="882">Now, if you're not aware, you can have multiple versions</a> <a data-t="886">of R installed on your machine,</a> <a data-t="887">and a wonderful tool for switching</a> <a data-t="890">between those R versions is called </a><a href="/software/rig/">rig</a><a data-t="890">,</a> <a data-t="892">which is known as the R installation manager.</a> <a data-t="895">So in your terminal, you can submit rig default 4.4.2.</a> <a data-t="900">So now, I've reverted back to my original version of R,</a> <a data-t="904">and at this point, RM restore becomes a trivial</a> <a data-t="907">and easy process.</a></p>

<p><a data-t="908">So I can successfully install package versions</a> <a data-t="912">from my repository specified in the lock file.</a> <a data-t="914">And remember, that was originally set to CRAN</a> <a data-t="917">when I started this six months ago.</a> <a data-t="919">Now, I've advanced in the future,</a> <a data-t="922">and when I'm looking at my binary availability chart,</a> <a data-t="925">if I'm really trying to go back in time</a> <a data-t="927">to this version 1.8.9,</a> <a data-t="930">that's no longer available on CRAN in binary.</a> <a data-t="934">So if I want to take advantage of that binary,</a> <a data-t="937">I can go to P3M to get that binary.</a> <a data-t="940">And you can do that by supplying the repose argument</a> <a data-t="943">in the RM restore function.</a> <a data-t="945">And there's a third way you can do this,</a> <a data-t="948">which is with the RM checkout function.</a> <a data-t="951">So in RM checkout, you supply a date,</a> <a data-t="955">and then RM will install date-based package versions</a> <a data-t="958">such that they fetch the latest available versions</a> <a data-t="961">up to that date.</a></p>

<p><a data-t="965">Let's talk about the manage decision.</a> <a data-t="968">So I'm calling it manage when I want to maintain</a> <a data-t="970">my R version of 4.4.2,</a> <a data-t="973">but I want to advance to JSON Lite 2.0</a> <a data-t="977">to use the latest and greatest.</a> <a data-t="980">And remember, this is an okay place to be</a> <a data-t="982">in the R ecosystem.</a> <a data-t="986">This is pretty simple.</a> <a data-t="987">In this case, I would do install of JSON Lite,</a> <a data-t="990">and then snapshot to record that to my lock file.</a></p>

<p><a data-t="994">And lastly, what about update?</a> <a data-t="998">Update means I want to bring everything forward</a> <a data-t="1000">into the future, including my R and my package versions.</a> <a data-t="1003">And in our R ecosystem map, this is a great place to be.</a> <a data-t="1008">There's three steps to updating.</a> <a data-t="1009">First, you'll execute RM upgrade</a> <a data-t="1012">to upgrade RM to the latest available version.</a> <a data-t="1015">Then you'll execute RM install</a> <a data-t="1017">to install all latest package versions.</a> <a data-t="1019">And lastly, snapshot to record those versions</a> <a data-t="1022">to the lock file.</a> <a data-t="1024">And I want to be clear here</a> <a data-t="1025">that if you're actively working on a project,</a> <a data-t="1028">regularly updating is the happy path.</a> <a data-t="1031">It allows you to reckon with upgrading</a> <a data-t="1034">your package versions in a controlled manner.</a></p>

<h3>Summary and recommendations</h3>

<p><a data-t="1039">So to summarize,</a> <a data-t="1044">RM places an immense amount of responsibility on the user.</a> <a data-t="1048">And it is a package that you should truly</a> <a data-t="1054">read the documentation front to end.</a> <a data-t="1054">Because what we've talked about here</a> <a data-t="1055">in terms of what RM does and does not do for you</a> <a data-t="1058">is listed in the caveats.</a> <a data-t="1060">So please, read the documentation.</a></p>

<p><a data-t="1064">And another framework that's in the RM documentation</a> <a data-t="1067">is how you handle active projects.</a> <a data-t="1070">It gives you these kind of decision-making frameworks.</a> <a data-t="1073">And if you're using RM on an active project,</a> <a data-t="1076">if you're thinking about the freeze option,</a> <a data-t="1077">I just want to keep those package versions stable,</a> <a data-t="1081">that's really just short-term sustainable.</a> <a data-t="1083">It's gonna come back to bite you,</a> <a data-t="1084">and it's gonna be harder the more time that elapses.</a> <a data-t="1087">Managing by doing one-off upgrades of a package</a> <a data-t="1090">is also short-term sustainable.</a> <a data-t="1092">It's gonna really kind of lead to a fragile R ecosystem.</a> <a data-t="1096">And then updating is truly</a> <a data-t="1098">the only long-term sustainable way to go.</a></p>

<p><a data-t="1102">And if you think about the reproducibility spectrum,</a> <a data-t="1104">just remember where RM sits, right in the middle.</a> <a data-t="1108">There might be times and places where you need to go</a> <a data-t="1110">to the upper end of that reproducibility spectrum</a> <a data-t="1113">and use something like a container, like Docker.</a> <a data-t="1116">And I love this example from Andrew Heiss.</a> <a data-t="1118">He's got a wonderful repository</a> <a data-t="1121">that shows you how you can execute code</a> <a data-t="1124">from a published paper using all methods of the spectrum,</a> <a data-t="1127">using Docker, using renv,</a> <a data-t="1130">or just using packages installed locally system-wide.</a></p>
