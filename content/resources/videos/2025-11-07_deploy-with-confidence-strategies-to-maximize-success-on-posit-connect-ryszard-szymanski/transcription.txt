<h3>Blue-green deployments</h3>

<p><a data-t="77">So, usually, you host your app on some kind of a server.</a> <a data-t="81">While your users access it through a client. For example, a browser in their laptops.</a> <a data-t="87">And the idea behind blue-green deployments is that we have two separate app instances.</a> <a data-t="93">One called blue, and the other called green.</a> <a data-t="97">And in the beginning, your users access the app through the blue instance.</a></p>

<p><a data-t="103">When you're ready to release a new version, you deploy it to the green instance.</a> <a data-t="108">You verify if everything's working as expected there.</a> <a data-t="112">And if it does, you switch the traffic.</a> <a data-t="116">So now your users access the new version, but this time through the green instance.</a></p>

<p><a data-t="121">But that was the optimistic scenario where everything worked as expected.</a> <a data-t="126">What if something goes wrong on the green instance?</a> <a data-t="130">Well, in these cases, you find yourself in a much more comfortable situation.</a> <a data-t="134">Where you can calmly investigate what's wrong in the green instance.</a> <a data-t="137">While your end users continue to use the stable version of your app on the blue instance.</a></p>

<p><a data-t="146">Let's see how we can use this strategy on Posit Connect.</a> <a data-t="150">So the way we can reverse traffic is by using vanity URLs.</a> <a data-t="154">And I'll explain what those are in a bit.</a> <a data-t="157">And separate app instances can be achieved by having separate content items.</a></p>

<p><a data-t="162">So usually, when you deploy something on Posit Connect, the URL would look sort of like this.</a> <a data-t="168">So it's your connect URL slash content slash the GUID of your app.</a> <a data-t="173">Posit Connect allows you to configure a user-friendly URL called a vanity URL.</a> <a data-t="178">Where you can configure the exact path where your app is available.</a> <a data-t="184">You can configure a vanity URL using the content settings in the Posit Connect UI.</a> <a data-t="191">And separate app instances can be achieved by having separate content items.</a> <a data-t="196">So you can create two separate deployments on Posit Connect and call one blue and the other green.</a></p>

<p><a data-t="203">Let's see how an example blue-green deployment could look like on Posit Connect.</a> <a data-t="208">So let's say our users are accessing the app through the my underscore URL vanity URL.</a> <a data-t="214">Which is initially configured for the blue content item.</a> <a data-t="218">Now we deploy the new version to the green content item.</a> <a data-t="222">And once we confirmed everything's working as expected, we reconfigure the same vanity URL.</a> <a data-t="230">But this time for the green content item.</a></p>

<h3>Feature flags</h3>

<p><a data-t="267">Now let's move on to feature flags.</a> <a data-t="269">So feature flags are also often called feature toggles or feature switches.</a> <a data-t="274">They're essentially a mechanism for turning features on and off in your application.</a> <a data-t="280">It's just a clever if statement.</a></p>

<p><a data-t="282">So a feature flag can be just this simple logical value.</a> <a data-t="287">And depending on whether it's true or false, we would enable or disable features.</a> <a data-t="291">So in this case, if that feature flag is enabled, we would welcome the users to the new experience.</a> <a data-t="298">But if it's disabled, we would welcome them back to the good and old experience.</a></p>

<p><a data-t="304">We can apply the same idea to applications.</a> <a data-t="307">So let's say we have an app where we have two different tabs with two different features.</a> <a data-t="313">We can hide one of the tabs behind the feature flag.</a> <a data-t="317">And this is how it could look like.</a> <a data-t="319">So based on the feature flag value, we could insert a tab containing our feature or not.</a></p>

<p><a data-t="327">Of course, there are other types of feature flags.</a> <a data-t="330">These don't have to be simple logical values.</a> <a data-t="333">But you can enable a feature for specific users or groups based on an environment variable or at a specific time.</a></p>

<p><a data-t="345">Let's have a look at an example of how we can use Posit Connect.</a> <a data-t="350">Let's get back to our example of an app with two different tabs with two different features.</a> <a data-t="356">And in the Posit Connect UI, you can find the People tab where you can find information on users and groups.</a> <a data-t="364">And the reason I mention this is Posit Connect provides you with information about the logged in user when accessing your app.</a> <a data-t="373">And that information includes their username and the groups to which they belong to.</a> <a data-t="380">And in case of </a><a href="/software/shiny-r/">Shiny</a><a data-t="380">, that information is available through the session parameter of the server function.</a></p>

<p><a data-t="388">Let's create a group called FeatureBGroup.</a> <a data-t="392">And what we want to do here is to only show the FeatureB tab for the members of that group.</a> <a data-t="400">So we can use the FeatureFlag package, which I have developed, which allows you to create a ConnectGroupFeatureFlag,</a> <a data-t="406">where you specify the groups for which this flag should be enabled.</a> <a data-t="409">And here we'll specify the FeatureBGroup we just created.</a> <a data-t="414">Now, in our code, we can check if that flag is enabled.</a> <a data-t="419">And if it is, we would insert the tab containing FeatureB.</a></p>

<p><a data-t="423">So what we just achieved here is a situation where different users accessing the same app might see a slightly different UI.</a> <a data-t="431">Members of the FeatureBGroup would see the FeatureB tab, while others who do not belong to that group would only see the FeatureA tab.</a></p>

<h3>Handling failed releases</h3>

<p><a data-t="560">Okay, but what if we use blue-green deployments or FeatureFlags and their release still failed?</a> <a data-t="567">So while those strategies do help you minimize some risks, they're not silver bullets.</a> <a data-t="574">So usually we have at least two options.</a> <a data-t="577">One of them is to fix things forward.</a> <a data-t="580">So the idea here is you see an issue, you fix it, you release a new version containing the fix.</a> <a data-t="586">And that's useful in scenarios where the issue is small, does not disrupt users, and you exactly know how to fix it quickly.</a> <a data-t="594">A good example of that is a typo in the UI.</a></p>

<p><a data-t="598">Or the other option is to roll back.</a> <a data-t="601">And that means going back to a previous stable version of your app.</a> <a data-t="605">And that's useful in cases where the issue prevents your users from working in the app,</a> <a data-t="613">and you really don't know what's going on and you need time to investigate.</a></p>

<p><a data-t="619">Thankfully, you can easily perform rollbacks in the Posit Connect UI.</a> <a data-t="623">So if you navigate in the top right corner to the source versions option,</a> <a data-t="627">you will be listed all of the bundle versions that have been used for this specific content item.</a> <a data-t="633">And in there, you can pick a previous bundle and click the reactivate button,</a> <a data-t="639">and that way bring back a previous version of your code.</a></p>

<p><a data-t="644">In the end, deployments don't have to be scary.</a> <a data-t="648">By adopting the right strategies, like blue-green deployments or feature flags,</a> <a data-t="653">and by knowing how to fix things forward or rollback quickly, we gain control.</a> <a data-t="659">Shipping to production should feel safe.</a> <a data-t="662">And with the right approach, it can be.</a></p>
