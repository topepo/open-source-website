<h3>Combining datasets with binds</h3>

<p><a data-t="40">So there's one last thing that you will probably want to do as you wrangle your data with these</a> <a data-t="44">packages and that's combine multiple datasets into the same dataset. There's a simple way</a> <a data-t="50">to combine datasets together in R. If your datasets were meant to go together, you can</a> <a data-t="55">add them together by pasting one as extra columns onto the first with the bind calls</a> <a data-t="60">command from </a><a href="/software/dplyr/">dplyr</a><a data-t="60">. There's also a cbind command in base R. Bind calls is just a little more</a> <a data-t="65">optimized. So that's why I'm showing you bind calls.</a></p>

<p><a data-t="70">This is somewhat dangerous to do. R has no way of checking whether the rows are in the</a> <a data-t="74">same order between the datasets or whether when you combine the datasets together, the</a> <a data-t="79">row describes an observation that makes sense. For example, here you probably want B to be</a> <a data-t="86">on the second line as in the first line. So use column binding with caution. Make sure</a> <a data-t="90">the data is arranged for you to do what you want to do.</a></p>

<p><a data-t="94">You can add the second dataset as additional rows to the bottom of your first dataset with</a> <a data-t="98">bind rows or rbind in base R. Again, you need to make sure that the variables are in the</a> <a data-t="103">same order and mean the same thing, but it's probably less risky than column binding.</a></p>

<h3>Joining data</h3>

<p><a data-t="139">Now normally when you want to put datasets together, you need to do something a little</a> <a data-t="142">more involved, and that's what I call joining data. So here's a trivial example. Here's</a> <a data-t="148">two data frames. They both talk about the Beatles. One's a list of songs and who wrote</a> <a data-t="152">each of the songs, and then one's a list of the artists in the Beatles and what each one</a> <a data-t="157">played or one of the instruments they played. And if you wanted to put these things together</a> <a data-t="161">into the same dataset, so you match up the instrument that each Beatle played with, you</a> <a data-t="166">know, what song they wrote, you can do that with a left join.</a></p>

<p><a data-t="172">What left join does is it takes the names of two data frames, and then it takes some joining criteria. These would</a> <a data-t="177">be the name or names of columns that both datasets share. And then what it will do is</a> <a data-t="182">it'll go through the second data frame, find the pieces of information that apply to rows</a> <a data-t="188">in the first data frame, and add them where they belong in that data frame. So order is</a> <a data-t="193">not preserved. The second data frame's order is not preserved. First data frame, yes, that</a> <a data-t="196">is preserved. But things are moved around to supply a play column that actually makes</a> <a data-t="201">sense with your data frame.</a></p>

<p><a data-t="207">And at the end, you have rows in your output that describe</a> <a data-t="207">a unified observation. John both wrote Across the Universe and played guitar. Guitar had</a> <a data-t="213">to appear twice because John was also in the second row, and so it appears twice. There's</a> <a data-t="218">no information about this last row, Buddy Holly, who made Peggy Sue. So for the place</a> <a data-t="223">column that gets an NA, there's nothing we could put there.</a></p>

<p><a data-t="228">Left joins aren't the only</a> <a data-t="228">way to join data together in an intelligent fashion. You could do a right join. Well,</a> <a data-t="235">this would be the syntax for if you want to join based on two columns, both first and</a> <a data-t="239">last. Use the C to make a vector of column names and pass that to by.</a></p>

<p><a data-t="256">Left join isn't the only way to join data sets together. You could do an inner join,</a> <a data-t="256">which is a little more exclusive. So an inner join still puts the data sets together in</a> <a data-t="260">the same fashion, but if there's any row from the first data frame that doesn't have information</a> <a data-t="265">in the second data frame, like Peggy Sue and Buddy Holly didn't appear in the second</a> <a data-t="268">data frame, inner join will exclude that from the results. It just contains the rows that</a> <a data-t="273">appeared in both data frames.</a></p>

<p><a data-t="286">So</a> <a data-t="289">we just released two new joins in the slides. You'll find them on the cheat sheet. Those</a> <a data-t="294">are right join and outer join. So right join is like a left join, but it has the same effect</a> <a data-t="297">as if you switch the order the data frames appear. If you put artist first and song second,</a> <a data-t="302">that would give you the equivalent of a right join. The second data frames prioritize are</a> <a data-t="306">the first. An outer join is the complement of an inner join. An inner join only keeps</a> <a data-t="311">the rows that appear in both data frames. An outer join keeps every row that appears</a> <a data-t="315">in any data frame, and you get a very big data frame at the end, but that probably has</a> <a data-t="319">a bunch of missing values. You could look those up on the cheat sheet.</a></p>

<h3>Filtering joins</h3>

<p><a data-t="325">Then we have</a> <a data-t="325">two filtering joins, semi-join and anti-join. These don't actually join data sets together</a> <a data-t="329">at all. These take the first data set and remove rows from it based on what appears</a> <a data-t="335">in the second data set. So what semi-join will do is it'll take the first data set,</a> <a data-t="339">look at the data set and see which rows have an equivalent in the second data set and return</a> <a data-t="344">just those rows that have an equivalent. It doesn't add anything to the rows, it just</a> <a data-t="347">shows you what rows will actually be joined should you do a join. Anti does the opposite.</a> <a data-t="353">It shows you which rows do not have equivalent in the second data set. So you can use anti-join</a> <a data-t="357">to see which rows will not be joined when you do a join. Semi-join and anti-join are</a> <a data-t="361">good ways to sort of debug or get a feel for how the join's going to work ahead of time.</a></p>
