<h3>Parquet files</h3>

<p><a data-t="26">My first favorite thing about the Arrow package are parquet files.</a> <a data-t="30">So parquet files are a little bit like CSVs in that you can read and write tables, but</a> <a data-t="35">they're binary, so they're smaller, and they're faster to read and write, and they remember</a> <a data-t="39">your data type.</a> <a data-t="40">So if you write a date to a parquet file, you get a date back.</a> <a data-t="43">If you write a number to a parquet file, you get a number back.</a></p>

<p><a data-t="47">With GeoArrow, if you write an sf object to a parquet file, you get an sf object back,</a> <a data-t="52">which is really useful, and it turns out that it's also really fast.</a> <a data-t="54">So if you find yourself waiting for your files to load, try parquet.</a> <a data-t="58">If you find yourself waiting for your geospatial files to load, try GeoArrow read-view-parquet.</a></p>

<h3>Datasets and parallel queries</h3>

<p><a data-t="63">My second favorite thing about the Arrow package are datasets.</a> <a data-t="69">Datasets let you take a whole lot of files that you create, and you can split them up</a> <a data-t="73">however you want, and you can query them like they're a database.</a></p>

<p><a data-t="79">In this example, I have a whole bunch of parking interactions from the city of Philadelphia,</a> <a data-t="84">and I have them hosted on an S3 bucket.</a> <a data-t="86">They could be on my computer, too.</a> <a data-t="89">Here, using the Arrow package, I can open them as a dataset, and I can query them using</a> <a data-t="93">DeepLyer, and when I hit collect, Arrow takes care of the details.</a></p>

<p><a data-t="98">And there's a lot of details to take care of.</a> <a data-t="100">Arrow pulls the data from a remote data source or from your computer, and Arrow runs your</a> <a data-t="105">query in parallel using all the cores on your computer, and it splits up the data into manageable</a> <a data-t="109">chunks just in case the dataset is bigger than your memory.</a> <a data-t="112">A lot of geospatial data is really big, and a lot of it is bigger than memory, so this</a> <a data-t="116">lets you use that workflow with geospatial data, and it just works.</a></p>

<h3>Simplifying the rspatial ecosystem</h3>

<p><a data-t="123">The final thing is something that a lot of users don't think about, but it's something</a> <a data-t="127">that I'm really passionate about as an rspatial package maintainer, which is simplifying possibly</a> <a data-t="133">how to maintain all of the components that make up the rspatial ecosystem and how they</a> <a data-t="137">fit together.</a></p>

<p><a data-t="138">There's a lot of components, and they all come together, hopefully silently, to make</a> <a data-t="142">your experience as an rspatial user awesome.</a> <a data-t="146">But there's a lot of interactions, and there's a lot of people that spend a lot of time managing</a> <a data-t="150">all of those interactions and maintaining them.</a></p>

<p><a data-t="154">GeoArrow uses a data structure that we can pass between all of those, which means that</a> <a data-t="158">we can define an interaction for every single component rather than every single pair of</a> <a data-t="163">components, which is a lot less maintenance work, and it means that rspatial developers</a> <a data-t="168">like me can spend more time adding features, which is fun, and less time fixing bugs, which</a> <a data-t="173">is less fun.</a></p>

<p><a data-t="177">And finally, because that thing that we're passing around is an Arrow array, we get all</a> <a data-t="181">of the Arrow ecosystem at our disposal, which includes the Arrow package, Parquet files</a> <a data-t="186">and datasets, but also the wider Arrow ecosystem, which includes bindings for Python and Rust</a> <a data-t="191">and Julia and a lot of other languages and a lot of other frameworks that have adopted</a> <a data-t="195">Arrow.</a></p>
