<h3>Product conception</h3>

<p><a data-t="147">Let's start with the product conception.</a> <a data-t="152">The first question that we ask is how we are going to collaborate, given that we came from the data analysis that we were doing individually.</a> <a data-t="162">Well, now we needed to code together.</a> <a data-t="165">You may know the answer, so I'm just going to say it.</a> <a data-t="168">We just Git.</a></p>

<p><a data-t="171">Some of the team members were learning about Git, like how to push and pull code.</a> <a data-t="181">Well, luckily, we had this plugin in </a><a href="/software/rstudio/">RStudio</a><a data-t="181"> that is the learning curve.</a> <a data-t="189">Because it can be tricky sometimes.</a> <a data-t="193">And why use Git?</a> <a data-t="195">Well, you want to be able to share your code, integrate everyone's contribution, and keep different versions of your code.</a> <a data-t="205">This is on your local, but also in the Git platform.</a> <a data-t="208">And you can use any Git platform, like GitLab, or GitHub, or Bitbucket.</a></p>

<p><a data-t="213">In our case, we asked the infrastructure team which was the better tool, because what we wanted to do is just focus on the development and not worry too much about the hosting.</a> <a data-t="222">We wanted some pipeline, some automation where we were able to push our code, and then it was going to be deployed to the cloud.</a> <a data-t="231">So they chose GitLab because of the continuous integration and continuous development features.</a> <a data-t="238">So in this way, they were able to use the tool that they felt more comfortable with while we were using the tool that we were comfortable with in RStudio.</a></p>

<p><a data-t="248">Then we started using Golem.</a> <a data-t="252">Golem is a Shiny framework to build production-level, robust Shiny applications.</a> <a data-t="260">At the beginning, I wasn't very sold on using this tool because it will add a lot of R packages and scaffolding code that I wasn't very sure that we were needing it.</a> <a data-t="272">But then I really liked it.</a></p>

<p><a data-t="275">It sets you a very basic web application where you can build on the top of it.</a> <a data-t="283">And one of the things that it adds, for example, is kind of like these scripts that it will guide you step by step so you can start building your Shiny app.</a> <a data-t="302">For example, it will help you configure your application name, your unit testing, your favicon, and then it will take you to the next step.</a> <a data-t="314">So it is a very nice tool, and I highly recommend using Golem or a framework like this one, especially if this is your first Shiny application.</a></p>

<h3>Dependency management</h3>

<p><a data-t="325">This is a concept that if you're not very used to it, it can be hard to understand it.</a> <a data-t="333">But it is very useful.</a> <a data-t="335">So we were facing the issue that maybe my code was working on my local, but it wasn't working on another team member environment, and that's because they didn't have the R package that was needed.</a> <a data-t="350">Or maybe they have it, but it was an older version that wasn't supporting the function that I had.</a> <a data-t="357">So for that, you want to use a tool like a dependency management tool.</a> <a data-t="363">And there are several options.</a></p>

<p><a data-t="365">You might as well just use an R script that will call your list of dependencies, or there are more sophisticated options like background or RMP.</a> <a data-t="374">Whatever the option you choose, try to do it from the beginning so you won't have to do some reverse engineering.</a> <a data-t="386">Remember all the tools that you have installed and then add it to the dependency management tool.</a> <a data-t="393">If you do it from the beginning, you won't have that problem.</a></p>

<p><a data-t="400">Also, it is okay to explore and innovate or there are packages that are out there, but for your application, just add the tools that you really need for your application to run.</a> <a data-t="414">So when it is on the cloud, you are going to use packages, just the dependencies that you need.</a> <a data-t="422">So just keep it simple.</a> <a data-t="423">Every package that you add is used, potential bugs or issues are more maintained.</a> <a data-t="430">So it's a good idea to keep it simple.</a></p>

<h3>Development time</h3>

<p><a data-t="434">At this point, we were ready for the development time.</a> <a data-t="437">So everything is already set up.</a> <a data-t="440">So we were ready to build our product.</a></p>

<p><a data-t="446">So how were we going to distribute the workload?</a> <a data-t="450">Well, Golem helped you to create these Shiny modules.</a> <a data-t="454">You can think of them as mini Shiny applications that you are going to call from the main script.</a> <a data-t="461">In our case, we had five dashboards, so we created one Shiny module per dashboard and we distributed them among the team members.</a> <a data-t="470">The way that you are going to separate the work is going to depend on your use case, your application or your team members.</a></p>

<p><a data-t="481">Integrating the backend.</a> <a data-t="483">So this is how we did it.</a> <a data-t="485">We collected the data and created some R data frames.</a> <a data-t="492">Then we loaded them into an SQLite database, which is a file.</a> <a data-t="499">And then set up the database connection.</a> <a data-t="502">We create all the queries that we needed and then connect them, connect those results to our dashboard charts.</a></p>

<p><a data-t="511">But that wasn't real data.</a> <a data-t="513">It was just like sample data.</a> <a data-t="515">So we needed to really be collecting data on a daily basis.</a> <a data-t="518">We needed to build some data pipelines and then loading that data into an Amazon Redshift and then do it all over again.</a> <a data-t="528">So what is the recommendation here?</a> <a data-t="530">Well, connect to your database from the beginning.</a> <a data-t="534">Even if that database does not contain all the production code, it can be sample data.</a> <a data-t="540">But it is very important that you start building your product thinking that it's going to be in the cloud and that it's going to be in production.</a></p>

<h3>Keeping credentials secure</h3>

<p><a data-t="551">Keep it secure.</a> <a data-t="553">So at the beginning, we were... This is very bad, I know.</a> <a data-t="557">But we were hard-coding passwords, AWS keys, and sensitive information.</a> <a data-t="562">So what happened?</a> <a data-t="563">Every time that we were pushing database credentials, we needed to go to that database that was already compromised and change the credentials.</a></p>

<p><a data-t="575">But what if we had pushed some sensitive information that we cannot really change?</a> <a data-t="581">Well, you need to go to Git and remove that from your code.</a> <a data-t="586">But Git, one of the main features is that you are going to keep several versions of your code.</a> <a data-t="592">So even, let's say that we are talking about the password, even if the latest version does not contain that password, you can just go back to the previous commit and the password is going to be there.</a> <a data-t="605">So you have to go all the way back to the Git history where that password is present and remove all those commits from the Git history.</a> <a data-t="612">And by doing that, you are going to remove some production code.</a> <a data-t="617">So it's going to be a mess.</a></p>

<p><a data-t="619">So how to solve that?</a> <a data-t="620">Well, use environment variables.</a> <a data-t="623">You can use this .r environment file that's going to be kept in your local.</a> <a data-t="629">That's going to have these key value pairs.</a> <a data-t="632">And you are going to call the keys in your code like so.</a> <a data-t="638">And not only is it going to be secure, but it's going to be ready for the cloud as well.</a> <a data-t="643">Because what if you have a database per environment?</a> <a data-t="646">Those credentials are going to be different.</a> <a data-t="648">It doesn't matter because your code is already prepared.</a> <a data-t="651">It's only using the keys.</a> <a data-t="652">The values are different in every environment.</a> <a data-t="656">So the only thing that you need to tell your infrastructure engineers are what are the environment variables that you need for your application to run.</a> <a data-t="665">And they are going to set up with something similar to that.</a></p>

<h3>Going to the cloud</h3>

<p><a data-t="670">Now, going to the cloud.</a> <a data-t="672">This is the moment of truth.</a> <a data-t="674">Because what you want at this point is just push your code and hope it runs.</a> <a data-t="679">You want to build, release, and run.</a> <a data-t="682">Well, this is rare that it happens the first time.</a> <a data-t="686">And for us, it didn't happen like the first ten times.</a></p>

<p><a data-t="690">So how we solved that?</a> <a data-t="692">What we did is that every time we were doing a commit, we were validating that commit.</a> <a data-t="698">GitLab has these nice pipelines where you can add some steps to validate the code that is being pushed.</a> <a data-t="706">So, for example, what you can do is the first step, install the dependencies, package your application.</a> <a data-t="714">And I think this was the selling point for me about Golem.</a> <a data-t="718">Because I wasn't really sure about using it.</a> <a data-t="721">And when I saw that Golem needs to package your application and then run it, it was weird to me.</a> <a data-t="729">Because I was thinking like a package like dplar or ggplot that those are there for you to use it.</a> <a data-t="736">And I didn't want my application to be out there.</a> <a data-t="739">But don't think it like that.</a></p>

<p><a data-t="742">When you package your application, what will happen is that it will run all your code.</a> <a data-t="748">So that means that if you forgot to add a dependency, it's going to fail.</a> <a data-t="753">If you forgot to remove some line of code that didn't have to be there, it's going to fail.</a> <a data-t="759">But if it can be packaged, then it means that every line of code is a working line.</a> <a data-t="766">So that's a very good step for quality assurance.</a></p>

<blockquote><a data-t="759">But if it can be packaged, then it means that every line of code is a working line.</a> <a data-t="766">So that's a very good step for quality assurance.</a></blockquote>

<p><a data-t="771">And then run your unit test.</a> <a data-t="773">You can validate the presence of a title, a legend, or a button, just to know that your dashboard is up and running.</a> <a data-t="782">Also, have a development cloud environment.</a> <a data-t="785">This is an environment that is meant to be broken.</a> <a data-t="789">So it's for you to break.</a> <a data-t="791">And if it is not broken, then it means that you can push it to production.</a> <a data-t="798">This is especially important when you already went live.</a> <a data-t="802">And you can also have a middle environment, like a staging or validation, where it's going to be an exact replica of your production environment.</a> <a data-t="812">And you can run some end-to-end testings there, just to be sure that all the functionality is there.</a></p>
