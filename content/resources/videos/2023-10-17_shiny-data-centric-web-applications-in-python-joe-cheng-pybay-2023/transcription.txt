<h3>Shiny vs. Streamlit</h3>

<p><a data-t="639">So when we came out with Shiny for Python,</a> <a data-t="644">a lot of people out there are already using Shiny for R.</a> <a data-t="647">So some people had this kind of a reaction.</a> <a data-t="653">And other people said something like this.</a> <a data-t="657">But it's just so easy to throw something up</a> <a data-t="660">with Streamlit.</a> <a data-t="661">How many people here have heard of Streamlit?</a> <a data-t="664">Okay, plenty of you.</a></p>

<p><a data-t="666">And we've gotten this question a lot</a> <a data-t="668">since the introduction of Shiny for Python,</a> <a data-t="670">because Streamlit is,</a> <a data-t="673">as smilodon138 says,</a> <a data-t="676">just so easy, right?</a> <a data-t="678">So I wanted to just take a moment</a> <a data-t="680">to unpack this claim a little bit.</a> <a data-t="682">And the promise of Streamlit</a> <a data-t="685">is that you can start with a normal script.py</a> <a data-t="689">that's just a straight linear set</a> <a data-t="692">of data analysis function calls.</a> <a data-t="696">And with Streamlit, you can take exactly that script,</a> <a data-t="700">sprinkle in a few Streamlit statements,</a> <a data-t="702">and boom, you have an interactive application.</a></p>

<p><a data-t="706">And even if you've never seen Shiny before this talk,</a> <a data-t="708">you already know that Shiny doesn't do that, right?</a> <a data-t="710">That Shiny has this form of a UI and a server</a> <a data-t="715">and these little functions with decorators.</a> <a data-t="717">It is asking you to pour your code into its form.</a></p>

<p><a data-t="722">So the way Streamlit is able to do this,</a> <a data-t="727">because that's super cool, right?</a> <a data-t="728">Very nice to be able to just take your script</a> <a data-t="730">and sprinkle in some Streamlit.</a> <a data-t="731">And the way they're able to do this is this.</a> <a data-t="733">Streamlit's architecture allows you to write apps</a> <a data-t="735">the same way you write plain Python scripts.</a> <a data-t="737">To unlock this, Streamlit apps have a unique data flow.</a> <a data-t="740">Anytime something must be updated on the screen,</a> <a data-t="742">Streamlit reruns your entire Python script</a> <a data-t="744">from top to bottom.</a> <a data-t="745">Nobody else does this.</a> <a data-t="746">This is only Streamlit.</a></p>

<p><a data-t="749">And it really is that easy.</a> <a data-t="752">It is true.</a> <a data-t="753">I'm not here to debunk Streamlit's ease of use.</a> <a data-t="756">It is like you might learn Streamlit by accident.</a> <a data-t="759">You could write a Streamlit app falling out of bed.</a> <a data-t="763">It's that easy.</a></p>

<p><a data-t="766">Now, granted, it's running the entire script</a> <a data-t="769">every time you move a slider,</a> <a data-t="771">every time you click a button.</a> <a data-t="772">So as a result, by default,</a> <a data-t="775">it does have slow performance.</a> <a data-t="776">And there's only so much control you can have</a> <a data-t="779">when it's running the entire script</a> <a data-t="781">every time anything changes.</a> <a data-t="782">You can't opt out of that behavior.</a> <a data-t="784">So that means as your Streamlit apps</a> <a data-t="787">get just a little bit more complicated,</a> <a data-t="789">you are actually left in quite a bad place.</a> <a data-t="792">So they have introduced some features</a> <a data-t="793">like caching callbacks and session state</a> <a data-t="796">that are designed to let you work around</a> <a data-t="799">the top-to-bottom execution model.</a></p>

<p><a data-t="801">So I can very much endorse Streamlit for simple apps.</a> <a data-t="805">And I think they made some really fantastic,</a> <a data-t="807">interesting choices.</a> <a data-t="809">But for the complicated apps,</a> <a data-t="811">you're putting yourself and your users in danger</a> <a data-t="813">because it is very tricky to get</a> <a data-t="815">that kind of application correct.</a></p>

<p><a data-t="818">So in contrast, Shiny is easy,</a> <a data-t="821">though not as easy as Streamlit.</a> <a data-t="823">But it is scalable.</a> <a data-t="825">And the history of Shiny is a history of people</a> <a data-t="829">getting so much value out of this framework</a> <a data-t="832">without being web developers,</a> <a data-t="834">but building sometimes quite complex apps</a> <a data-t="838">and getting huge results from those apps.</a> <a data-t="842">And the kinds of stories I could tell you</a> <a data-t="844">about the kinds of accomplishments</a> <a data-t="846">people have made on their Shiny apps</a> <a data-t="848">would, you know, gives me goosebumps</a> <a data-t="849">just thinking about it.</a></p>

<p><a data-t="850">So I would sum it up by saying</a> <a data-t="854">the meantime from getting started</a> <a data-t="856">to having something of value,</a> <a data-t="858">it's Streamlit, it's incredible.</a> <a data-t="860">Like within 10 seconds, you have an interactive plot.</a> <a data-t="863">And with Shiny, maybe it takes you 60 seconds, okay?</a></p>
