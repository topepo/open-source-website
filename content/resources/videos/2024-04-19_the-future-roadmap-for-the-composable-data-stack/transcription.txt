<h3>Composable data systems</h3>

<p><a data-t="391">Last year at VLDB, I collaborated with fine folks at Meta to write this paper called</a> <a data-t="402">The Composable Data Management System Manifesto.</a> <a data-t="406">It's a great paper.</a> <a data-t="408">I want to help you in this talk by presenting some of the key ideas from the paper and why</a> <a data-t="415">they matter.</a> <a data-t="417">I do recommend that you check out this paper.</a> <a data-t="419">I think it's really well written.</a></p>

<p><a data-t="438">What is a composable data system?</a> <a data-t="438">The way that I think about this is that we are building systems based on open standards</a> <a data-t="444">and protocols.</a> <a data-t="446">We're designing for modularity, reuse, and interoperability with other systems that share</a> <a data-t="451">those common interfaces in open standards and protocols.</a> <a data-t="455">In particular, we're resisting this idea of vertical integration where every layer of</a> <a data-t="462">your system is bespoke and you build it yourself.</a> <a data-t="465">It's this homespun thing where you build everything top to bottom.</a></p>

<p><a data-t="469">It does create more work in some ways because you have to deal with coordination and open</a> <a data-t="474">source diplomacy.</a> <a data-t="476">You have to think about moving along these different pieces that you share governance</a> <a data-t="480">and you share ownership with other developers and people who are working on different layers</a> <a data-t="484">of the stack.</a> <a data-t="487">The hope is that over time that each of the component pieces that we use to build these</a> <a data-t="492">systems become so much better, easier to use, faster, more interoperable, and that you end</a> <a data-t="499">up creating a solution that is a lot more future-proof and delivers better results over</a> <a data-t="508">time.</a></p>

<p><a data-t="510">It's a little bit painful because you have to go through the painful growing pains of</a> <a data-t="518">creating these new systems and making them all work together as well as negotiating with</a> <a data-t="522">all of the open source developers that are involved in these projects.</a> <a data-t="526">The way that it was put in the words in the paper is that we envision by decomposing data</a> <a data-t="531">management systems into a more modular stack of reusable components, the development of</a> <a data-t="537">engines can be streamlined while reducing maintenance costs and ultimately providing</a> <a data-t="541">a more consistent user experience.</a></p>

<p><a data-t="545">By clearly outlining APIs and encapsulating responsibilities, data management software</a> <a data-t="551">could more easily be adapted, for example, to leverage novel devices and accelerators</a> <a data-t="556">as the underlying hardware evolves.</a> <a data-t="559">That was one of the reasons why we founded Voltron Data.</a> <a data-t="563">We observed that it's hard for people to take advantage of GPUs to accelerate their systems.</a> <a data-t="569">We also saw that there's many new hardware companies creating new types of hardware.</a> <a data-t="575">We need to make a corresponding investment in the software layer to make it easier to</a> <a data-t="580">take advantage of new developments in hardware, whether that's faster computing, faster</a> <a data-t="585">networking, faster storage.</a></p>

<p><a data-t="588">Ideally, from the standpoint of you as a user, you just want to be able to write Python code</a> <a data-t="592">and as the hardware improves, all your software gets faster and you don't have to think about</a> <a data-t="596">the messy details of, well, how do I take advantage of this bleeding edge development</a> <a data-t="600">in computing hardware?</a></p>

<p><a data-t="604">The hope is that by relying on a modular stack that reuses execution engines, language front-ends,</a> <a data-t="610">the data systems can provide a more consistent experience and semantics to users from</a> <a data-t="616">transactional to analytic systems, stream processing to machine learning workloads.</a> <a data-t="620">That sounds very nice in principle.</a></p>

<h3>Why composability is happening now</h3>

<p><a data-t="624">One question people have had is, why is this happening now?</a> <a data-t="628">Why is it happening in the 2020s?</a> <a data-t="632">Why didn't this happen in the 2010s or even earlier?</a> <a data-t="636">It helps to go back and think about the generational eras of big data technology going back to the</a> <a data-t="640">original Google MapReduce paper in the mid-2000s.</a></p>

<p><a data-t="646">The clone of Google MapReduce and Google File System as HDFS and Apache Hadoop,</a> <a data-t="652">that was the first release of Hadoop from Yahoo was in 2006.</a> <a data-t="658">The MapReduce paper popularized this idea of disaggregated storage and compute.</a> <a data-t="664">You have a big distributed file system, you have all of your data sets live as files in that file</a> <a data-t="668">system, and then you have a bunch of compute engines that can process on those files.</a> <a data-t="674">You process files in, the systems write files out back into that distributed file system.</a> <a data-t="680">But what this created was this collection of monolithic vertically integrated systems</a> <a data-t="686">that were part of the Hadoop ecosystem.</a></p>

<p><a data-t="690">Just getting these components to a place where they were viable and you could use them to</a> <a data-t="696">productively work was hard enough.</a> <a data-t="700">Thinking about a hierarchy of needs thing, thinking about how we could modularize components</a> <a data-t="706">of those systems and reuse large pieces of software between those systems,</a> <a data-t="712">that was pie-in-the-sky thinking back in the late 2000s, early 2010s.</a></p>

<p><a data-t="718">But there was a big shift in the 2010s where a particular large software vendor shifted from</a> <a data-t="724">selling proprietary vertically integrated software components to delivery of services in a cloud</a> <a data-t="730">environment.</a> <a data-t="734">That also led to an emergence of open source standards, file formats like Parquet,</a> <a data-t="740">as well as there was major progress in computing, networking, and storage.</a> <a data-t="746">Disk drives got a lot faster, networking got a lot faster, processor units.</a> <a data-t="752">In GPUs a little bit, but in GPUs it's continued to go off to the moon</a> <a data-t="758">as Moore's Law has continued in more specialized processor chips.</a></p>

<p><a data-t="772">Through that period, as open standards began to emerge, we started thinking about</a> <a data-t="778">how we could create technologies that would help better take advantage</a> <a data-t="784">of the innovation in the hardware and the hardware layer.</a> <a data-t="790">That has led us in this current era to think about re-architecting all of our systems</a> <a data-t="796">on interoperable standards that will enable us to get much better performance</a> <a data-t="802">and efficiency across a heterogeneous application stack that's doing a mixture</a> <a data-t="808">of ML, feature engineering, machine learning training, and AI inference.</a></p>

<p><a data-t="814">We've had this emergence of new standards for composability, which I will talk about.</a> <a data-t="820">We're seeing simultaneously an emergence of new systems which are built to be</a> <a data-t="826">composable from the ground up, as well as retrofitting existing pieces</a> <a data-t="832">of systems with new components which are created with this new composable mindset.</a></p>

<h3>Apache Arrow and the new stack</h3>

<p><a data-t="916">A lot of involvement in this was helping create the Apache Arrow project.</a> <a data-t="928">We realized in the mid-2010s that we needed to create a cross-language</a> <a data-t="934">fast memory format which could be used for fast in-memory computing</a> <a data-t="940">as well as fast data interchange across disparate systems.</a> <a data-t="946">We started imagining what if we had this shared data science run time,</a> <a data-t="952">a set of libraries and computational systems that we could use interchangeably</a> <a data-t="958">across programming languages.</a></p>

<p><a data-t="964">It didn't matter which language front-end you're using, what programming language.</a> <a data-t="970">At the same time, in the database world,</a> <a data-t="976">there was also a gnashing of teeth around this problem</a> <a data-t="982">of how inefficient it is to import and export data from databases.</a> <a data-t="988">If you're using just a single database, single data warehouse,</a> <a data-t="994">you can't get a thick milkshake through a straw that's too small</a> <a data-t="1000">if you've ever had that experience.</a></p>

<p><a data-t="1010">What we've done is create technologies to provide an open standard</a> <a data-t="1016">for Arrow native APIs for interacting with databases</a> <a data-t="1022">using a standard wire protocol out of the box</a> <a data-t="1028">rather than using a proprietary or system-specific wire protocol</a> <a data-t="1034">that has to be serialized when you're getting data out.</a> <a data-t="1040">Our hope is that, firstly, with existing systems,</a> <a data-t="1046">but at the same time they can also build support</a> <a data-t="1052">for the new wire protocol directly into the database.</a> <a data-t="1056">They have their legacy protocol as well as the new open standard protocol,</a> <a data-t="1062">and that yields significantly better performance.</a></p>

<p><a data-t="1079">DuckDB is one example of now a very successful project,</a> <a data-t="1085">which is putting a super-fast columnar analytical database</a> <a data-t="1091">into a browser pretty much anywhere that can run C++ code or WASM.</a></p>

<h3>Architecture of composable systems</h3>

<p><a data-t="1200">If we think about the architecture of what new composable systems look like,</a> <a data-t="1210">this is a diagram from the paper.</a> <a data-t="1216">We have a modular execution engine,</a> <a data-t="1222">a runtime which provides for distributed computing and coordination.</a> <a data-t="1228">That could be Spark, that could be Ray,</a> <a data-t="1234">or another distributed runtime that makes use of one of these modular engines</a> <a data-t="1240">like DuckDB or Velox or another.</a></p>

<p><a data-t="1260">There are now several modular execution engines</a> <a data-t="1266">written in different programming languages</a> <a data-t="1272">such as the now Apache DataFusion.</a> <a data-t="1278">It's a Rust-based modular query engine.</a> <a data-t="1284">It has a query optimizer on the front end as well,</a> <a data-t="1290">but can also be just as an execution engine in Rust.</a></p>

<p><a data-t="1301">One of the things that we've developed in this community</a> <a data-t="1307">to make connecting these engines to the language front end easier</a> <a data-t="1313">is to have an intermediate representation for queries.</a> <a data-t="1318">The idea is that your language front end,</a> <a data-t="1322">whether that's a DataFrame API or a SQL API,</a> <a data-t="1326">we translate that into a substrate plan</a> <a data-t="1330">and we can optimize that substrate plan</a> <a data-t="1334">and then the execution engine can take that plan</a> <a data-t="1338">and turn that into a physical execution plan</a> <a data-t="1342">that is sent off to the cluster to be executed.</a></p>

<p><a data-t="1350">We also retrofit existing popular query engines</a> <a data-t="1354">with these new modular accelerators.</a> <a data-t="1358">Three of them that you can get access to</a> <a data-t="1362">or you can look at or contribute to right now.</a> <a data-t="1366">One is Prestissimo, which is retrofitting PrestoDB,</a> <a data-t="1370">I should clarify, not Trino, with Velox.</a> <a data-t="1374">There's Comet, which has been just donated</a> <a data-t="1378">into the DataFusion project from Apple,</a> <a data-t="1382">which is accelerating Spark with DataFusion.</a> <a data-t="1386">There's also Apache Gluten, part of the Apache Incubator,</a> <a data-t="1390">which is accelerating Spark with Velox.</a></p>

<p><a data-t="1394">Eventually we'll have DataFusion and Presto</a> <a data-t="1398">and probably every combination of modular accelerators</a> <a data-t="1402">in our query engine projects.</a> <a data-t="1406">All of this begins to pose the question of</a> <a data-t="1410">why as a user should I be locked into using</a> <a data-t="1414">a particular full-stack query engine like Spark or Presto?</a> <a data-t="1418">The cost, performance, and latency</a> <a data-t="1422">of these different systems varies greatly.</a></p>

<p><a data-t="1426">We can do a lot with DuckDB.</a> <a data-t="1430">But when you have an actual big problem</a> <a data-t="1434">that's too big for DuckDB,</a> <a data-t="1438">you want to be able to transition gracefully</a> <a data-t="1442">to using that larger scalable execution engine.</a> <a data-t="1446">Of course, last year at Data Council</a> <a data-t="1450">we had Jordan Tagani on the stage in a duck costume</a> <a data-t="1454">talking about how big data is dead.</a> <a data-t="1458">Big data still is big data. Big data does still exist.</a> <a data-t="1462">I believe that we should be able to gracefully transition</a> <a data-t="1466">from working at the single machine,</a> <a data-t="1470">could be a large single machine, but the small scale</a> <a data-t="1474">to a large scalable data processing engine</a> <a data-t="1478">without as much pain and suffering.</a></p>

<h3>SQL portability and the Ibis project</h3>

<p><a data-t="1482">This isn't that easy, though, in part because</a> <a data-t="1486">it advertises just standard SQL.</a> <a data-t="1490">Because SQL dialects, while SQL is a standard,</a> <a data-t="1494">in practice SQL dialects are non-portable</a> <a data-t="1498">and they feature a wide spectrum of different features.</a> <a data-t="1502">There's a tool like SQLGLOT, which is amazing,</a> <a data-t="1506">which helps with transpiling and converting from one SQL dialect</a> <a data-t="1510">to another, but this does create a problem.</a> <a data-t="1514">You have to solve this problem because if you have a mountain</a> <a data-t="1518">of existing SQL queries and you want to migrate them</a> <a data-t="1522">to use a different data processing engine,</a> <a data-t="1526">you have to rewrite all those queries and deal with all the quirky edge cases</a> <a data-t="1530">of type coalescing, conversion, and whatnot.</a> <a data-t="1534">Knowing which engine to use in every scenario is non-trivial</a> <a data-t="1538">and may not always be obvious.</a></p>

<p><a data-t="1542">A few years ago in CIDR 2021,</a> <a data-t="1546">folks at Microsoft approached this problem from the standpoint</a> <a data-t="1550">of the Azure data platform to say,</a> <a data-t="1554">could we create a unified API for interacting with all these different database engines</a> <a data-t="1558">that we have available in the Azure platform and intelligently choose</a> <a data-t="1562">which one to use based on the type of workload,</a> <a data-t="1566">like how big the data set is, what statistics we have about the data,</a> <a data-t="1570">performance, good total cost of ownership for the query.</a> <a data-t="1574">They described a research project that they did called Magpie</a> <a data-t="1578">where they created a pandas-like data frame API.</a> <a data-t="1582">They have a compiler and optimizer at the middle.</a> <a data-t="1586">They have a common data layer, which is based on Arrow,</a> <a data-t="1590">that provides for a distributed data fabric to coordinate</a> <a data-t="1594">these different execution engines within the Azure platform.</a></p>

<p><a data-t="1606">I love this idea and I would really like to help it become</a> <a data-t="1610">more mainstream and a reality for data engineers everywhere.</a> <a data-t="1614">A project that I've been involved in,</a> <a data-t="1618">I started about nine, almost 10 years ago,</a> <a data-t="1622">and that we've been putting a lot of resources into is the Ibis project in Python,</a> <a data-t="1626">which had been an effort to harmonize the best features of modern SQL</a> <a data-t="1630">with Python's fluent data frame APIs.</a> <a data-t="1634">We also wanted in Ibis to fix shortcomings in the pandas API</a> <a data-t="1638">and make things that are hard to do with pandas a lot easier to do with Ibis,</a> <a data-t="1642">so we made a deliberate choice to depart from the pandas API.</a></p>

<p><a data-t="1646">We wanted to make it easy to build complex,</a> <a data-t="1650">large analytical SQL queries by using the benefits of a modern</a> <a data-t="1654">programming language, functions and parameters and things like that,</a> <a data-t="1658">things that don't exist in SQL.</a> <a data-t="1662">Over the years, we've been really focused on portability and compatibility</a> <a data-t="1666">across a wide spectrum of engines who now support over 20 backends</a> <a data-t="1670">to provide a unified API that could be the basis</a> <a data-t="1674">of building a multi-engine data stack.</a></p>

<p><a data-t="1678">Ibis code, this produces a pretty complex SQL query</a> <a data-t="1682">and everything is based on pipe-like</a> <a data-t="1686">sequences of methods that compose with each other</a> <a data-t="1690">to create complex expressions.</a></p>

<h3>Dev to prod and GPU-scale workloads</h3>

<p><a data-t="1694">One of the workflows that we hear a lot of pain points about</a> <a data-t="1698">and people wanting to work in this ecosystem</a> <a data-t="1702">and to create a composable data stack is how to go from</a> <a data-t="1706">dev to prod.</a> <a data-t="1710">Nobody wants to burn a ton of snowflake credits while they're developing</a> <a data-t="1714">and the ideal scenario would be to be able to develop,</a> <a data-t="1718">build your model, do all of your development locally on your laptop</a> <a data-t="1722">with DuckDB and whenever you're ready to go into prod</a> <a data-t="1726">in Spark SQL or BigQuery or Snowflake,</a> <a data-t="1730">then it's about as easy as flipping the switch.</a> <a data-t="1734">Run this workflow but in Snowflake and to not have to rewrite all of your queries.</a> <a data-t="1738">That's one of the key things that we're trying to facilitate</a> <a data-t="1742">in this project.</a></p>
