<h3>Introducing <a href="/software/mirai/">mirai</a></h3>

<p><a data-t="125">So I'm going to use mirai to demonstrate the two ways that we can deal with async.</a> <a data-t="133">mirai is a package that I wrote that runs R code in the background very efficiently without blocking the current R session.</a> <a data-t="142">It's rather straightforward to use.</a> <a data-t="145">You simply wrap the expression you want to run with mirai and that returns a mirai object immediately without blocking your session.</a></p>

<p><a data-t="157">So once you have this object, how do you get the result of the background computation or how do you even know whether that computation has finished or not?</a> <a data-t="168">So if we take the polling approach, the way to do this would be to call unresolved on the mirai object.</a> <a data-t="175">And if that's still ongoing, then it will return true, as we have here.</a></p>

<p><a data-t="182">Now, how do we use that to use mirai in an asynchronous fashion?</a> <a data-t="190">Well, we can use a loop like this.</a> <a data-t="194">So while the mirai is unresolved, sleep for a small amount of time and then check again.</a> <a data-t="202">And when you exit from that loop, you can actually access the data.</a> <a data-t="209">And instead of just printing the data, you would likely pass that to another function which uses the result.</a></p>

<p><a data-t="216">And similarly as well, instead of just sleeping for a while, you can actually do work concurrently with while this mirai is being computed in a background process.</a> <a data-t="231">So if you look at this while loop, it's very much like checking email every 15 minutes.</a> <a data-t="238">Except here, we're checking much more often than that because the computations we typically deal with are in the seconds or minutes.</a> <a data-t="247">So we want to have lower latency.</a></p>

<h3>How mirai handles task completion</h3>

<p><a data-t="256">Now, before we go too far with this email analogy, I want to delve a little bit deeper about what actually happens with mirai when tasks complete.</a> <a data-t="270">Because actually a mirai is much more asynchronous.</a> <a data-t="274">And what I mean is with email, when you check if there's new email, you're actually sending a request to the server and then that replies.</a> <a data-t="284">If it has email, then it actually sends it back to you.</a> <a data-t="288">That is not what a mirai does.</a></p>

<p><a data-t="291">When you call unresolved, it is not sending anything to the background process.</a> <a data-t="298">Instead, what happens is when the background task completes, it will send the result straightaway back to the main process.</a> <a data-t="309">And this is handled at the C level by a background thread.</a> <a data-t="315">A completion callback checks the error code.</a> <a data-t="319">If that's successful, it stores a pointer to the binary data that's already been received and it stores the completion status in the mirai object itself.</a> <a data-t="330">This is all done on the background thread in parallel to whatever's happening on the main R thread.</a></p>

<p><a data-t="337">This means that later when you call the R function unresolved or try to access the data element of a mirai, all that does is check the completion status of the object.</a> <a data-t="351">If that's success, then it checks if there's binary data.</a> <a data-t="355">If there is, then it simply un-serializes this into an R object.</a></p>

<p><a data-t="364">We can see that this is much more efficient because we're doing as much as possible ahead of time.</a> <a data-t="370">The only reason we actually have two stages is because we can't do the un-serialization and creating of an R object on the background thread.</a> <a data-t="380">Because R is single-threaded, that needs to be at a time when we're actually on the main thread.</a></p>

<h3>Event-driven async with promises and later</h3>

<p><a data-t="406">Does this help us also implement event-driven async?</a> <a data-t="410">What can we do there?</a> <a data-t="413">Well, we can, for example, use the brackets method for a mirai object to actually wait for and collect the result.</a> <a data-t="423">This is efficient in that there's no constant checking in a loop, but it also blocks the session.</a> <a data-t="430">So this in itself is not a solution for event-driven async.</a></p>

<p><a data-t="437">For that, I will need to add to my async toolkit.</a> <a data-t="442">So the packages that I will need here are promises and later.</a> <a data-t="449">Now, you can see that there are no hex logos for either of them.</a> <a data-t="454">So that should tell you immediately that these are serious low-level packages that do the real heavy lifting here.</a></p>

<p><a data-t="463">So in terms of promises, that provides a high-level interface for creating these async functions.</a> <a data-t="471">And what a typical signature of the key function there looks like is you wrap an object like a mirai in as.promise, turn that into promise, and you can then access the then method.</a> <a data-t="489">And you pass a function to that.</a> <a data-t="493">So what this means is as soon as a mirai completes, then call this function on it.</a></p>

<p><a data-t="502">Later is a lower-level package, and it actually provides some of the key implementation for the async.</a> <a data-t="510">And the signature for the main function there is later function and seconds.</a> <a data-t="515">And what this does is it schedules a function to run with a certain delay.</a></p>

<p><a data-t="525">Now, how does mirai promises and later work together to give you event-driven async?</a> <a data-t="530">Well, let me give you an example.</a> <a data-t="533">First, not of event-driven async, but of polling async.</a> <a data-t="540">So what you see here is the actual as.promise method for a mirai as it was about two years ago.</a> <a data-t="549">Pretty much the same.</a> <a data-t="550">Simplified a little bit for clarity, but the core logic is the same.</a></p>

<p><a data-t="555">Let me go through this line by line.</a> <a data-t="558">So this is the as.promise.mirai method.</a> <a data-t="563">We create a promise, and we pass it a function with resolve and reject.</a> <a data-t="569">I won't go into that details of why.</a> <a data-t="572">But just note, inside that function, we define another function called check.</a> <a data-t="577">And what this does is it calls unresolved.</a> <a data-t="583">And if the mirai is unresolved, then it will use later to schedule check again in 0.1 seconds.</a> <a data-t="593">And then if it has resolved, it just takes the value, extracts the value, and it resolves or rejects the value.</a> <a data-t="601">And that's the definition of the check function.</a> <a data-t="603">And we actually call check to kick things off.</a> <a data-t="606">But essentially, you can see that this schedules itself to check every 0.1 seconds via later, whether the mirai has resolved or not.</a></p>

<p><a data-t="621">So apart from polling, you can see that it's also not very efficient, because if we have something that takes quite a long time, then we're checking potentially hundreds, thousands of times.</a> <a data-t="634">And this is just one promise.</a> <a data-t="637">If we have a lot of promises in whatever we're building, then we could be spending a lot of time doing these checks.</a></p>

<p><a data-t="646">So how do we turn this into event-driven promises?</a> <a data-t="654">Well, we can see that this is, again, it's a simplified version, so it's not the entire story, but the core logic is accurate.</a> <a data-t="665">You can see the stretch is maintained pretty much the same.</a> <a data-t="669">But just scanning this, you can see there's no check function that repeats itself.</a> <a data-t="675">There's no 0.1 parameter anyone in here.</a></p>

<p><a data-t="680">So what's changed is the core logic of your promise function is if the mirai is unresolved, then you call this dot keep.</a> <a data-t="693">And dot keep is a special internal function which tells essentially the mirai completion callback.</a> <a data-t="703">So if you remember that from a few slides ago, this is something that's happening independently of the R thread, which processes as much of the completion as possible.</a> <a data-t="714">It basically tells that callback to additionally call into later via the C interface to resolve the promise.</a> <a data-t="722">So later has a C interface, which means that we're continually dealing with C code.</a> <a data-t="729">We're not calling any R code, and that lets us do this completely independently of whatever's happening in R.</a> <a data-t="739">And what this means is that the action is scheduled as soon as the mirai completes.</a> <a data-t="744">We don't need to wait until unresolved is called or anything like that.</a> <a data-t="749">And you can also see that dot keep is called once only.</a> <a data-t="754">So not only do we have event driven, it is much more efficient as well because we're not potentially calling unresolved thousands of times.</a></p>
