<h3>Recap and thinking about interactivity</h3>

<p><a data-t="2196">All right, so let's pause.</a> <a data-t="2196">I mean, there's quite a bit there.</a> <a data-t="2205">I'd like to recap what we looked at.</a> <a data-t="2205">When you're making computational documents with R Markdown, the opportunities you should pay attention to are actually generating the text of the document with code, using inline code, because if you do that, it will pay off when you parameterize the document.</a> <a data-t="2218">Next, don't think of it in terms of R.</a> <a data-t="2218">Think of Python, SAS, JavaScript, whatever you like to use.</a> <a data-t="2225">You can put that into an R Markdown document.</a> <a data-t="2225">Some more exotic languages will take some work on your part on the back end, but things used for data science like Python are ready to go.</a> <a data-t="2236">Hide the code from your users if they're not looking for a code experience, and then add parameters to reports to make them flexible.</a> <a data-t="2245">Your entire report becomes like a function in R that takes some input and returns some output, and all you have to do is change the input to change the output.</a> <a data-t="2249">If you're generating the text with inline code, you're able to change the results that are reported to the user as the parameters themselves change.</a></p>

<p><a data-t="2261">Consider adding HTML widgets for self-contained interactive pieces, and then consider going to adding Shiny components if you need to do some sort of custom interactions based on your unique R algorithms.</a> <a data-t="2275">If you look at these things that we went over, you'll notice here at the end these last three options are sort of like a hierarchy of ways to make your report more and more interactive, and you can think of this hierarchy of interactivity from static R Markdown documents on one end to Shiny apps on the other end, which are completely interactive tools.</a></p>

<p><a data-t="2296">All of these are available for you to use, and so now as someone sophisticated who can make computational documents, you need to start thinking about your strategy for supplying needs using these different levels of interactivity, so let's look at that, thinking about interactivity.</a> <a data-t="2315">Not only does each rung on this hierarchy create a more interactive document, it also requires more work on your part.</a> <a data-t="2322">Interactivity is complicated to make, and when it comes to making something like a Shiny app, those of you who have made Shiny apps, you know that there is a lot going on there, and that is a very specialized undertaking to make a Shiny app, but you don't have to make a Shiny app every time you want interactivity.</a></p>

<p><a data-t="2337">For example, if you just want to generate the same report but on a new drug or on a new market segment or on a new updated database, you might be able to do that with the simple R Markdown document that has been parameterized with parameters.</a> <a data-t="2346">Alternatively, if you want your user to be able to poke and prod the document, you might want to create a Shiny app, or you might be fine with an R Markdown document that uses HTML widgets, or even an R Markdown document with Shiny components.</a> <a data-t="2360">What I recommend is that you aim for the level of interactivity that's as simple as possible to do the task that you want, because that leaves less room for things to go wrong and need to be debugged on your part, less room for things to go wrong and for the user to have a poor experience, and less time needed to create those documents.</a></p>

<blockquote><a data-t="2360">What I recommend is that you aim for the level of interactivity that's as simple as possible to do the task that you want, because that leaves less room for things to go wrong and need to be debugged on your part, less room for things to go wrong and for the user to have a poor experience, and less time needed to create those documents.</a></blockquote>

<p><a data-t="2381">Here's some simple bright lines that you could use to think about this hierarchy.</a> <a data-t="2409">If you ask yourself, how many times do I need to change the data in my report during my presentation or during the viewer's presentation?</a> <a data-t="2417">How many times do they need to change the data for themselves?</a> <a data-t="2421">If it's less than once, then you don't need anything that involves Shiny.</a> <a data-t="2427">This is a very important question to ask yourself, because we tend to see people learn Shiny and then use it for everything.</a> <a data-t="2432">It's very hard to walk back from Shiny once you've adopted a Shiny workflow, and that's okay.</a> <a data-t="2437">Shiny is excellent, but it's very easy to overestimate what you need to accomplish the results you want, and if you're not updating your data more than once per session, then you can just use a simple non-Shiny R Markdown document, and you can create a new document before each session, and that document will suffice for that session.</a></p>

<p><a data-t="2459">The next thing to ask is, do you need to create a custom UI for your app?</a> <a data-t="2467">If what you want to report to your user would do fine in a document format or a dashboard format, you can do that with an R Markdown document, and you don't need to invest in creating a Shiny or an HTML UI to contain your interactive components.</a></p>
