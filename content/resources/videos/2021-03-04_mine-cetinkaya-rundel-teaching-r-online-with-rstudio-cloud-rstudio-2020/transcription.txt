<p><a data-t="1420">The base project applies to all projects created after the base project has been specified.</a> <a data-t="1424">It does not apply retroactively, and that should be an attractive feature.</a> <a data-t="1429">Once people create a project in the class, the settings of that project shouldn't change if you later decide to change how your base project looks.</a> <a data-t="1435">I think most importantly, the base project can be updated as many times as you like throughout the course without having to convince any IT people.</a> <a data-t="1444">There might be a new version of a package that you want your students to learn about, and you can choose to install that package to your base project at 2 a.m. the night before you teach, and tomorrow morning when they get in and start working on an assignment, it will be available to them.</a></p>

<p><a data-t="1465">How does it work to create a base project?</a> <a data-t="1466">There are a few steps for it.</a> <a data-t="1470">You start by creating just like any old project.</a> <a data-t="1474">In this base project, you're going to want to install any packages that you may want to have.</a> <a data-t="1480">I like renaming the project right away as base project.</a> <a data-t="1484">You really can use any name you want, it just needs to be identifiable so that you can choose it from a list later.</a> <a data-t="1488">As I said before, I always like creating an R Markdown document, because that list of packages, I forget what they are.</a> <a data-t="1495">All the list of packages you need to get R Markdown up and running, at least RStudio does it automatically for me, so even though I'm not putting in R Markdown document here, I go through the motions of doing so.</a></p>

<h3>Common pitfalls</h3>

<p><a data-t="2041">A few common pitfalls using either paradigm, sharing projects or sharing a workspace.</a> <a data-t="2048">It's important to remember that when a student makes a copy of your project, or when they start an assignment, RStudio Cloud is making a literal copy of that project at that time point.</a> <a data-t="2060">So, changes you make after a student launches a project under either model won't propagate to their project.</a> <a data-t="2069">This is not like collaborative editing of Google Docs, it is more like you distributing, handing out paper to your students, and then if you make changes to your own copy, that's not going to propagate to theirs.</a> <a data-t="2082">What does this mean?</a> <a data-t="2083">Until you're ready to share an assignment with your students, I would recommend keeping the access level to be available only to you, which is the default anyway, so you could be working on what your next week's homework is going to look like, and your students don't need to see that work until you're ready to actually make that project available to everyone.</a></p>

<p><a data-t="2111">Another common pitfall that I see is that packages in the base project will be installed for everyone, but they will not be loaded.</a> <a data-t="2115">If you need students to run code in the console that requires a package, your instructions should remind them to load the package first, so run something like a library command.</a> <a data-t="2126">They never should need to run install packages, unless that's part of your learning objective of course, but they do need to run the library command, and also, if using an R Markdown document that you're giving them as a template, you might have a chunk on top that loads the packages that you want.</a></p>
