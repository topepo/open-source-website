<h3>The prime directive: correctness, transparency, reproducibility</h3>

<p><a data-t="113">So if we all have the ability to do this,</a> <a data-t="117">that doesn't answer the question of whether we should,</a> <a data-t="119">whether it's a good idea.</a> <a data-t="121">And this is a question that is very near and dear</a> <a data-t="124">to my heart and to the heart of Posit,</a> <a data-t="126">and to talk about why,</a> <a data-t="128">I wanna talk a little bit about where Posit comes from.</a> <a data-t="131">So we are a public benefit corp,</a> <a data-t="134">and if you are not familiar with that concept,</a> <a data-t="136">you can sort of think of it as being a compromise</a> <a data-t="138">between a for-profit company and a nonprofit organization.</a> <a data-t="143">So we have all of the flexibility of a for-profit company,</a> <a data-t="148">but we are motivated by a mission.</a></p>

<p><a data-t="152">And as a PBC, we have to state out loud what that mission is.</a> <a data-t="157">And for us, this is sort of an abridged version</a> <a data-t="159">of the beginning of our official mission.</a> <a data-t="162">Our mission is to create open source software</a> <a data-t="164">for data science, scientific research,</a> <a data-t="166">and technical communication.</a></p>

<p><a data-t="169">And I really wanna focus on this data science</a> <a data-t="172">and scientific research.</a> <a data-t="174">A lot of the use cases that are really important to us</a> <a data-t="178">are things like people doing academic</a> <a data-t="180">and scientific research,</a> <a data-t="182">people who are really using this stuff</a> <a data-t="185">for high stakes kinds of scenarios,</a> <a data-t="189">people who are analyzing healthcare data,</a> <a data-t="192">or doing drug development, deciding public policy,</a> <a data-t="198">environmental studies, epidemiology,</a> <a data-t="201">things that it's really important</a> <a data-t="202">that your answers are correct</a> <a data-t="204">or people could really get hurt.</a></p>

<p><a data-t="237">Both the data analysts, that's probably most of you,</a> <a data-t="241">and the software provider, that's us</a> <a data-t="244">and people who write packages for R,</a> <a data-t="246">therefore have a strong responsibility</a> <a data-t="248">to produce a result that is trustworthy</a> <a data-t="251">and if possible, one that can be shown to be trustworthy.</a> <a data-t="255">This obligation I label the prime directive.</a></p>

<blockquote><a data-t="255">This obligation I label the prime directive.</a></blockquote>

<p><a data-t="281">So if we have this principle in mind</a> <a data-t="284">and for us, it really is, it's more than a best practice.</a> <a data-t="287">It's really a moral and ethical obligation</a> <a data-t="290">to adhere to this prime directive.</a> <a data-t="292">What does that mean concretely?</a> <a data-t="293">Like how does that translate into real concerns</a> <a data-t="297">as we work with data science software?</a></p>

<p><a data-t="300">So number one, correctness is obviously paramount</a> <a data-t="304">at all times.</a> <a data-t="305">We wanna do our best to write software without bugs,</a> <a data-t="307">especially in the code paths that produce answers.</a> <a data-t="311">Secondly, transparency.</a> <a data-t="313">We want the methods of our analysis</a> <a data-t="316">to be able to be inspected</a> <a data-t="318">and having open source code goes a long way</a> <a data-t="322">towards those ends.</a> <a data-t="324">And third, reproducibility.</a> <a data-t="326">The ability for the analysis to be repeated</a> <a data-t="329">on the same data in the future,</a> <a data-t="330">hopefully producing the same results.</a> <a data-t="333">So these three principles of correctness,</a> <a data-t="335">transparency and reproducibility</a> <a data-t="337">are first and foremost in our minds</a> <a data-t="339">any time we are working on a new tool</a> <a data-t="343">for helping with data analysis.</a></p>

<h3>Why LLMs are a poor fit — and what that means</h3>

<p><a data-t="347">And when we hold these three principles against LLMs,</a> <a data-t="351">this is not an obvious fit.</a> <a data-t="353">Like it's actually quite an obvious misfit</a> <a data-t="356">because when it comes to correctness,</a> <a data-t="358">LLMs are infamous for giving convincing but wrong answers.</a> <a data-t="362">They're getting better all the time,</a> <a data-t="364">but this phenomenon of very convincingly telling you</a> <a data-t="368">the wrong thing is certainly still very prevalent.</a> <a data-t="372">Secondly, transparency.</a> <a data-t="374">I mean, LLMs are the ultimate black box.</a> <a data-t="377">We don't really understand what they do.</a> <a data-t="380">We don't really understand how they do it.</a> <a data-t="382">And in fact, the more you understand about them,</a> <a data-t="385">the less what they are already capable of seems possible.</a></p>

<p><a data-t="390">And then finally, they are kind of inherently</a> <a data-t="394">not reproducible.</a> <a data-t="395">They're inherently non-deterministic</a> <a data-t="397">just by virtue of the way that they work.</a> <a data-t="400">Some of the model providers will give you the ability</a> <a data-t="403">to set a random seed, and even they will warn you</a> <a data-t="407">that this is not fully reproducible</a> <a data-t="409">because the answers will still vary</a> <a data-t="411">based on the underlying hardware</a> <a data-t="413">or their software configuration.</a> <a data-t="414">So we inherently do not have the ability</a> <a data-t="417">to reproduce conversations or actions that LLMs take.</a></p>

<p><a data-t="422">And we've all seen this in action.</a> <a data-t="425">Like, we've all asked LLMs things like,</a> <a data-t="428">draw an intricate piece of ASCII art,</a> <a data-t="431">and it says, here's an intricate piece of ASCII art</a> <a data-t="433">of a wolf, penguin.</a> <a data-t="436">I mean, and it's very confidently saying,</a> <a data-t="439">I've done this for you.</a> <a data-t="441">Why does it do that?</a> <a data-t="443">We don't really know.</a></p>

<p><a data-t="446">And I wanna make sure you understand that my point here</a> <a data-t="448">is not that LLMs are bad or that they can't do things.</a> <a data-t="452">They can do a lot of things.</a> <a data-t="453">And in fact, the shape of the things that they can</a> <a data-t="456">and can't do is quite surprising.</a> <a data-t="458">I think you could intuitively imagine</a> <a data-t="461">that LLMs work like this,</a> <a data-t="463">that there is a spectrum of tasks</a> <a data-t="465">that they could possibly do that range from very easy,</a> <a data-t="469">according to some criteria or intuition, to very hard.</a> <a data-t="473">And you would think that the easy tasks they excel at,</a> <a data-t="477">and as these tasks get harder, they get worse and worse,</a> <a data-t="480">which is what you see here,</a> <a data-t="481">kind of a smooth, relatively smooth capability curve.</a></p>

<p><a data-t="485">And that is not the world that we live in.</a> <a data-t="487">Today's LLMs do not reflect this kind of behavior.</a> <a data-t="493">In fact, it's more like this,</a> <a data-t="495">where there are very easy tasks that they excel at</a> <a data-t="500">and very easy tasks that they are terrible at</a> <a data-t="502">and very difficult tasks that they excel at</a> <a data-t="505">and very difficult tasks that they're terrible at.</a> <a data-t="507">And it is very, very difficult to intuit</a> <a data-t="510">without trying to know what category</a> <a data-t="515">or whether this will be inside or outside</a> <a data-t="517">of their capability curve</a> <a data-t="519">when you're thinking of a new task.</a></p>

<p><a data-t="521">Just to take two examples, they're quite good at coding.</a> <a data-t="525">And I think you would think, as a human,</a> <a data-t="528">that coding is a pretty hard thing to do.</a> <a data-t="530">There's a lot of logic and reasoning.</a> <a data-t="533">There's a lot of nuance.</a> <a data-t="534">There's abstractions.</a> <a data-t="535">There's a lot of context that you need to hold in your head.</a> <a data-t="539">There's a lot of intent that needs to be inferred</a> <a data-t="541">from reading code.</a> <a data-t="543">And yet, these models are incredibly good at it.</a> <a data-t="546">But at the same time, the very simplest data tasks,</a> <a data-t="550">they are really surprisingly bad at.</a> <a data-t="552">And let me just give you a quick demonstration of that.</a></p>

<p><a data-t="556">So I tried to think of,</a> <a data-t="557">what is the very easiest data operation that you could do?</a> <a data-t="562">And the one I thought of was length.</a> <a data-t="564">Given an array of values, how long is the array?</a> <a data-t="567">How many elements are in the array?</a> <a data-t="570">So to do this in R, we create an array of length n.</a> <a data-t="575">These are random floating point values between zero and one.</a> <a data-t="579">And then we ask, in this case, GPT 4.1,</a> <a data-t="584">we ask it, how long is this array?</a> <a data-t="586">And then we pass it those random numbers</a> <a data-t="589">encoded as a JSON array.</a></p>

<p><a data-t="593">And if you pass it 10 values to its credit,</a> <a data-t="596">it will get it correct.</a> <a data-t="598">At 100, it gets it correct.</a> <a data-t="599">And even 1,000, it gives the correct answer.</a> <a data-t="603">But then at 10,000, it says 1,000.</a> <a data-t="606">I mean, it's not even close.</a> <a data-t="609">And even these are cherry-picked round numbers.</a> <a data-t="611">They're just powers of 10.</a> <a data-t="614">When I gave it 103 elements, it said there were 100.</a> <a data-t="617">So this is a pretty disappointing showing</a> <a data-t="620">that even such a fundamental operation as length,</a> <a data-t="623">these models are not reliable for.</a></p>

<p><a data-t="626">In fairness, when I made this slide,</a> <a data-t="629">this was definitely true,</a> <a data-t="630">and GPT 4.1 was state-of-the-art.</a> <a data-t="633">With GPT 5, it takes a very long time,</a> <a data-t="635">but I have mostly seen it give correct answers,</a> <a data-t="638">and the same with opus 4.1.</a> <a data-t="640">So the state-of-the-art is changing,</a> <a data-t="643">but the fact remains that an operation</a> <a data-t="645">that feels like it should be simple and fundamental,</a> <a data-t="650">we cannot trust these LLMs.</a> <a data-t="653">And that does not bode well</a> <a data-t="654">for all the other possible data science operations,</a> <a data-t="658">the fundamental operations you'd wanna do.</a></p>

<h3>Three approaches to responsible LLM use</h3>

<p><a data-t="661">So given these inherent challenges,</a> <a data-t="664">these challenges around correctness,</a> <a data-t="666">transparency, reproducibility,</a> <a data-t="668">are there responsible ways to use LLMs</a> <a data-t="670">in the service of data science?</a> <a data-t="672">Are there responsible ways that we can leverage</a> <a data-t="674">these kind of magical new tools?</a></p>

<p><a data-t="677">So what I am gonna show you now</a> <a data-t="680">are three different approaches that we have taken.</a> <a data-t="683">And I wanna emphasize that these are not what I consider,</a> <a data-t="687">like these are principles</a> <a data-t="689">that everybody should be abiding by.</a> <a data-t="690">These are just experiments that we ran</a> <a data-t="692">and approaches that we kind of observed</a> <a data-t="694">have worked more or less well</a> <a data-t="697">for these different scenarios that we tried.</a> <a data-t="699">So this is all evolving very quickly.</a> <a data-t="703">Every model behaves differently.</a> <a data-t="706">So we are approaching this</a> <a data-t="708">with a spirit of experimentation and humility</a> <a data-t="711">and openness for each generation of LLMs</a> <a data-t="714">to bring us a different set of opportunities.</a> <a data-t="715">And I highly encourage you</a> <a data-t="717">to think of these approaches in that spirit.</a></p>

<p><a data-t="721">So the first approach I wanna talk about</a> <a data-t="723">is to constrain these LLMs, okay?</a> <a data-t="726">And what that looks like is we identify some things</a> <a data-t="730">that these LLMs can do,</a> <a data-t="732">at least take a particular model that you're targeting.</a> <a data-t="736">And we decide, based on our empirical experience,</a> <a data-t="739">okay, this LLM is very, very good at this one task</a> <a data-t="743">or this one skill.</a> <a data-t="745">It is firmly inside the jagged frontier of their capability.</a> <a data-t="751">Then in order to make this one skill more useful,</a> <a data-t="754">we kind of augment the LLM</a> <a data-t="756">with other kinds of software we write,</a> <a data-t="758">like deterministic software</a> <a data-t="761">that kind of leverage that LLM skill</a> <a data-t="764">and make it more useful.</a> <a data-t="766">And then we instruct the LLM,</a> <a data-t="768">just stick to the task we are giving you,</a> <a data-t="771">stick to doing this thing that we've decided</a> <a data-t="773">that you're very good at.</a> <a data-t="775">And resist the urge to add features</a> <a data-t="778">that would pull the LLM in directions</a> <a data-t="780">that it's still okay at,</a> <a data-t="783">but maybe not quite as solidly good at.</a></p>

<p><a data-t="786">So the example I have in mind for this</a> <a data-t="789">is to use an LLM to drive SQL, to drive a dashboard.</a> <a data-t="794">Let me show you what that looks like.</a> <a data-t="796">So this is a </a><a href="/software/shiny-r/">Shiny</a><a data-t="796"> dashboard.</a> <a data-t="798">This is not an LLM-powered dashboard.</a> <a data-t="799">This is just a regular Shiny dashboard.</a> <a data-t="801">This is actually one of the templates for </a><a href="/software/shiny-python/">Shiny for Python</a><a data-t="801">.</a></p>

<p><a data-t="806">And just like a typical Shiny application,</a> <a data-t="808">you can see that there are some reactive outputs,</a> <a data-t="810">and then there are some inputs on the side.</a> <a data-t="812">You know, if this was a real dashboard,</a> <a data-t="814">there would probably be a lot more inputs than this.</a> <a data-t="816">But as you change these values,</a> <a data-t="818">you can see that the values on the right update, okay?</a></p>

<p><a data-t="823">So we asked ourselves,</a> <a data-t="825">could we leverage these LLM's abilities</a> <a data-t="829">to write SQL very well,</a> <a data-t="831">and use that to enhance our dashboard</a> <a data-t="835">with more capabilities?</a> <a data-t="836">So here we've replaced all the filters with a sidebar.</a> <a data-t="840">This is now written in R with Elmer and </a><a href="/software/shinychat/">ShinyChat</a><a data-t="840">.</a> <a data-t="843">And we can ask these filtering questions</a> <a data-t="846">like show only male smokers who had dinner on Saturday.</a> <a data-t="850">And this request goes out to an LLM.</a> <a data-t="854">The LLM comes back with a select statement.</a> <a data-t="858">And that select statement is shown to the user</a> <a data-t="861">in two places.</a> <a data-t="863">And then that select statement is applied</a> <a data-t="866">to a data source by Shiny using reactive code.</a> <a data-t="871">And everything that you see down here,</a> <a data-t="872">this code has not changed.</a> <a data-t="874">It is the same reactive Shiny code.</a> <a data-t="877">It's just keyed off of a different reactive data frame,</a> <a data-t="879">one that is driven by this SQL.</a> <a data-t="882">This technique works with any SQL speaking database</a> <a data-t="886">that you can get to from R.</a></p>

<p><a data-t="889">And what this means is that we do not have to worry</a> <a data-t="891">about these values being hallucinated.</a> <a data-t="894">We don't have to worry about this plot</a> <a data-t="896">or this table of numbers being hallucinated.</a> <a data-t="899">All we have to worry about is whether this SQL is correct.</a> <a data-t="903">And if you are someone who speaks SQL fluently,</a> <a data-t="908">then great, that's a very easy task for you.</a> <a data-t="909">But even if this is something that you're handing off</a> <a data-t="912">to say your boss who doesn't speak SQL as well,</a> <a data-t="917">there's a really good chance that they can either</a> <a data-t="919">sort of intuit by reading the SQL.</a> <a data-t="924">They can intuit whether it's right or wrong.</a> <a data-t="926">And at worst, they can take that SQL and bring it to you</a> <a data-t="928">and ask you to verify it.</a></p>

<p><a data-t="931">But importantly, modern LLMs are quite good</a> <a data-t="934">at writing SQL, so it's very infrequent</a> <a data-t="936">that they make mistakes here.</a> <a data-t="938">And we can use this capability to achieve filtering</a> <a data-t="941">that we could not do.</a> <a data-t="943">We could not do with traditional Shiny controls.</a> <a data-t="946">This is a very easy thing to reproduce in normal Shiny,</a> <a data-t="949">that select these three criteria,</a> <a data-t="951">but something as simple as invert this filter</a> <a data-t="955">and saying the ones that we're showing now</a> <a data-t="957">make those the only ones we are not showing.</a> <a data-t="960">That is not an easy thing to do with most dashboard UIs.</a></p>

<p><a data-t="976">This approach, this LLM SQL dashboard approach</a> <a data-t="979">turns out to be incredibly, incredibly useful.</a> <a data-t="983">And I want to use this illustration</a> <a data-t="988">from Rick and Morty.</a> <a data-t="990">If you haven't seen the show, there's this one scene</a> <a data-t="992">where Rick, who is a scientist, he's</a> <a data-t="995">tinkering at the breakfast table,</a> <a data-t="996">and he builds this little robot.</a> <a data-t="999">And the robot wakes up, and it immediately</a> <a data-t="1001">asks, what is my purpose?</a> <a data-t="1003">And instead of answering, Rick asks the robot</a> <a data-t="1006">to pass the butter.</a> <a data-t="1007">And the robot brings the butter over, and Rick's very happy.</a> <a data-t="1013">And then other things happen in this scene.</a> <a data-t="1015">They're having conversation while the robot just</a> <a data-t="1017">waits there.</a> <a data-t="1017">And then when there's a pause, the robot</a> <a data-t="1020">says, what is my purpose?</a> <a data-t="1022">And Rick says, you pass butter.</a> <a data-t="1026">And this poor robot goes, oh my god.</a></p>

<p><a data-t="1032">And this is sort of how I think of this scenario, where</a> <a data-t="1036">you have this LLM that is good at a lot of things.</a> <a data-t="1039">It has this very general capability,</a> <a data-t="1041">this very general intelligence.</a> <a data-t="1044">And we are really saying, you pass butter.</a> <a data-t="1047">You do this one thing, you just write SQL,</a> <a data-t="1049">and we do not want you to do anything else.</a></p>

<p><a data-t="1055">So this approach of massively constraining</a> <a data-t="1058">what the model can do, is this a responsible approach?</a> <a data-t="1061">So in this case, in terms of correctness,</a> <a data-t="1063">we are asking it to generate SQL,</a> <a data-t="1066">and it generally does that incredibly well.</a> <a data-t="1068">Certainly, I would say, at a superhuman level</a> <a data-t="1072">compared to the median, or even well</a> <a data-t="1075">above median data analysts.</a> <a data-t="1079">In terms of transparency, all the analysis</a> <a data-t="1082">that you're seeing on the screen is</a> <a data-t="1084">driven off of a SQL query that is very clearly displayed.</a> <a data-t="1088">And in terms of reproducibility, the fact</a> <a data-t="1090">that you can take that SQL and apply it later on,</a> <a data-t="1094">and get the same result as long as the data has not changed,</a> <a data-t="1097">I think that qualifies.</a></p>
