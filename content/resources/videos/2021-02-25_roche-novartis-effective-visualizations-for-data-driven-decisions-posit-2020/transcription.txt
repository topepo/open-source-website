<h3>The three laws of effective visual communication</h3>

<p><a data-t="929">This is one of the main motivations behind the collaboration that we are presenting today and the initial design of the Bizarre Package.</a> <a data-t="938">But how do we become more effective?</a> <a data-t="940">Myself and colleagues at Novartis have been thinking about this topic for a number of years.</a> <a data-t="944">We've tried a number of approaches to try and address this question, from reading all the literature available,</a> <a data-t="950">defining best practices, developing galleries of code, newsletters to highlight the importance of the topic, to handing out cheat sheets.</a> <a data-t="958">And we've had various levels of success. But to make this concept stick and to be easier to remember,</a> <a data-t="1004">we recently defined what we call the three laws for effective visual communication.</a> <a data-t="1008">Have a clear purpose, show the data clearly and make the message obvious.</a></p>

<p><a data-t="1013">You can think of these three laws as superheroes. So have a clear purpose.</a> <a data-t="1018">This is really about why do we need this graph? So we should really know the purpose of creating the graph.</a> <a data-t="1024">And this is really related also to data analysis itself.</a> <a data-t="1027">We should identify the quantitative evidence to support the purpose and also identify the audience and focus the design to support their needs.</a> <a data-t="1036">Show the data clearly. This is about avoiding misrepresentation, choosing the appropriate graph type to display the data, maximizing data to ink ratio.</a> <a data-t="1046">So this is really just a complex way of saying keep it simple.</a> <a data-t="1049">Finally, make the message obvious. So using proximity and alignment to aid in comparisons, minimize mental arithmetic.</a> <a data-t="1057">This is really going back to the waterfall plot where you really don't want the reader to be doing summations in their head and using colors and annotations to highlight important details.</a></p>

<p><a data-t="1117">So I'll now go through each of these three laws in more detail. So law one, have a clear purpose.</a> <a data-t="1124">This first law is really advanced common sense. Is it to explore data or to deliver a message?</a> <a data-t="1138">What evidence is available to support this purpose? This could be data sets, new experiments, simulations, for example.</a> <a data-t="1144">Whatever. OK. Who is my intended audience? Is it a mix of specialists or non-specialists or both?</a> <a data-t="1152">You really need to think about who this audience is and support their needs. And the where, where will I publish and what are the constraints there?</a></p>

<p><a data-t="1162">So for the where, we may identify certain formats such as a Word document, a PowerPoint, PowerPoint presentation or a Markdown notebook or a </a><a href="/software/shiny-r/">Shiny</a><a data-t="1162"> app.</a> <a data-t="1202">And really the design and the purpose and the principles that we have really, really is important for all of these different formats.</a> <a data-t="1210">But depending on the format, some of these principles may apply and the design choices may vary.</a></p>

<p><a data-t="1219">Another aspect to this law is to identify what are the underlying questions you're trying to address.</a> <a data-t="1224">So here I've highlighted an article by Roger Penn that discusses his thoughts on John Tukey, exploratory data analysis and design thinking.</a> <a data-t="1233">He argues that better analyses are related to the quality of the question.</a> <a data-t="1237">So more time spent on defining better questions is time well spent.</a> <a data-t="1241">And this is related to the famous Tukey quote, an approximate answer to the right question is worth a great deal more than the precise answer to the wrong question.</a> <a data-t="1250">So the message here is to spend more time thinking about your question prior to an analysis or designing a visualisation.</a></p>

<p><a data-t="1259">Another aspect to this law is why are you creating visualisation?</a> <a data-t="1263">So thinking about this aspect can help identify the audience and their needs, which in turn helps with the design.</a> <a data-t="1309">So, for example, if you want to dig into data, get familiar with the data or find the story in my data, then the audience is normally you.</a> <a data-t="1317">If you want to communicate the results or to tell the story behind the data, then the audience is someone else.</a> <a data-t="1323">So being clear on the purpose is important here.</a> <a data-t="1326">For example, if you're designing a visualisation to be presented to a group of stakeholders where you have three minutes to get your recommendations across,</a> <a data-t="1334">do you really want the audience to play? <p><a data-t="1352">Law two, show the data clearly.</a> <a data-t="1400">The key part of this law is not to lie or represent with data. <h3>Architecture and design of VisR</h3>

<p><a data-t="2955">So that's why we are now presenting the vis R package in its current form, which is actually is still very early experimental phase.</a> <a data-t="2966">Or you could actually call it a prototype, maybe which we which we developed and would like to hear feedback of you all later on as well.</a> <a data-t="2974">We thought to there were there are a few architecture considerations we put in in order to implement those considerations and requirements that we got in our earliest stakeholder interviews.</a></p>

<p><a data-t="2987">So the first one was really about this integration, seamless integration into the tidyverse in our analysis workflow.</a> <a data-t="2994">So we thought whatever we do, it should interact with the plier and the modeling packages and plotting should be built on ggplot.</a> <a data-t="3001">We also want to keep full transparency on data modifications, first of all, for the reproducible reporting aspect, but also for being able to scale projects across different stages.</a> <a data-t="3015">And that led us to this decoupled design that you can see on the right where we actually take our input data, let's say, for example, one row per sample, one row per patient data set.</a> <a data-t="3026">And then we have dedicated functions that take care of the data wrangling and the modeling before another set of functions takes care of the visualization.</a> <a data-t="3038">And that visualization ideally adheres to the graphical principles that Mark just described before.</a> <a data-t="3044">And then last but not least, we have a separate set of functions, again, that takes care of the separate that takes care of the styling of the outputs.</a> <a data-t="3058">So there shouldn't be any hard coded theming, for example, in the plotting function.</a> <a data-t="3065">But that is then taken care of and allows us, for example, to adapt the visualization to corporate designs.</a> <a data-t="3071">But it also allows us to adapt the visualization to be consistent across the report where, for example, the treatments are always colored in the same way across.</a></p>

<p><a data-t="3092">We mentioned before as well that in addition to being flexible, though, the coding of and adhering to those graphical principles shouldn't add a lot of overhead to the coding work you're doing as a data scientist anyways.</a> <a data-t="3112">So we thought it would be great to have, in addition to those individual decoupled functions, the wrapper functions that allow you to explore different visualizations for typical analysis questions.</a> <a data-t="3124">So one typical question in clinical development when working with a patient cohort in the beginning is, for example, to see how many of those patients in the data set adhere to a set of inclusion and exclusion criteria.</a> <a data-t="3137">And you might want to show the cohort attrition.</a> <a data-t="3141">And there are different ways of doing that. And Diego will talk about it a bit later.</a> <a data-t="3145">But having a wrapper function that allows you to explore these different visualizations can then help you quickly find out which visualization might be the best one to convey your message clearly to the target audience you're currently designing your report for.</a></p>

<p><a data-t="3193">So this was all very theoretical so far, and it's going to stay a bit theoretical for the next two slides before I hand over to Diego.</a> <a data-t="3205">But what I want to show here is really how this would look if you implemented it for a typical question, another typical question around time to event analysis.</a> <a data-t="3225">And so by having decoupled functions, if we had our initial input data model for patient data that has, for example, intervention like treatment and then the time of follow up or time to event and the status at the end,</a> <a data-t="3240">then we wouldn't go directly from this input data model to the plot, but we would go through three individual steps.</a> <a data-t="3248">The first one is really taking this input data and have an estimator function that computes the estimate, in this case, the survival model.</a> <a data-t="3257">And after cleaning that with Broom, we get a data table or a table as an interim data model, which we can then build our visualizations on.</a> <a data-t="3270">And in this case, for example, would contain the time, the stratum, as well as the estimates at different time points and the confidence intervals.</a> <a data-t="3278">And these can then be easily passed on to ggplot, for example, for visualization.</a></p>

<p><a data-t="3284">As I said before, we didn't want to reinvent the wheel where this was necessary.</a> <a data-t="3289">So we are building a lot of those functions on existing packages of the tidyverse and beyond, such as Broom or in this case, also survival.</a> <a data-t="3299">The idea is by having this multi-tier approach that you can define a lot of custom features that might be required, such as custom time windows or adding hypothesis tests.</a> <a data-t="3312">And so after you have this interim data model, you have the visualization function and then the styling in the end that helps you allow to adhere to those graphical principles that we spoke a bit about before.</a></p>

<p><a data-t="3388">So this were a lot of the design considerations that are definitely not set in stone.</a> <a data-t="3392">So if people have ideas and thoughts about that, we would love to hear them.</a> <a data-t="3395">But Diego will show you now in the next few slides how we implemented the prototype and what those outputs look like for now.</a> <a data-t="3400">And with that, I would like I would like to hand over to you, Diego.
