<h3>Notebooks vs. IDE: when to use each</h3>

<p><a data-t="1988">Let's talk for just a second, and I touched on it briefly, but there's a difference</a> <a data-t="1993">between notebooks, whether it's Jupyter notebooks or Databricks notebooks in some of the cases,</a> <a data-t="1999">and a true sort of integrated development environment. And this is not comprehensive,</a> <a data-t="2004">and I think there's reasons you'll use one over the other, and depending on what you're trying to</a> <a data-t="2008">do, there's a tool that might be better suited for the job.</a></p>

<p><a data-t="2012">But there's sort of a well-known</a> <a data-t="2016">dialogue around a few common things with notebooks that you have. They're messy when working with</a> <a data-t="2021">version control because you have a lot of JSON data that's attached versus standard Python files</a> <a data-t="2026">where your diffs are a little bit easier to understand. Working in and sort of sharing</a> <a data-t="2032">code across notebooks, particularly within Databricks, can be a little bit of a challenge</a> <a data-t="2036">versus Python. You can structure things in modules, and you can have shared imports.</a> <a data-t="2041">There's code review can be a little bit of a challenge with notebooks. Again, going back to</a> <a data-t="2045">sort of the Git diff challenges that we have there. Unit testing can be a little bit tricky,</a> <a data-t="2049">although there are some frameworks that I think can help out there. But if you're using just</a> <a data-t="2053">standard Python files, you've got access to standard testing frameworks that you can leverage</a> <a data-t="2058">to build testing around those files. Debugging in notebooks can be a little bit of a challenge.</a> <a data-t="2063">In an IDE, you have a full catalog of debugging tools that can be at your disposal.</a></p>

<p><a data-t="2070">And like we</a> <a data-t="2071">kind of highlighted a little bit today, Positron Assistant does a really great job if it's going</a> <a data-t="2077">and encounters any sort of errors at addressing those errors in real time. And then we also have,</a> <a data-t="2082">you know, notebooks are a little bit notorious for the fact that cells can be executed in</a> <a data-t="2086">arbitrary order. And so if you're not careful in the construction of the notebook, you can end up</a> <a data-t="2090">in a state where it's not it's no longer linear. Stateless functions and the way in which a Python</a> <a data-t="2095">file natively executes can help prevent some of that from creeping in.</a></p>

<p><a data-t="2107">If you're looking at, for example, a code diff with a notebook versus a</a> <a data-t="2113">standard Python file, you end up with quite a lot of noise in a notebook diff versus a Python file</a> <a data-t="2119">that's a little bit cleaner to understand what's changed and what those changes might introduce.</a> <a data-t="2124">So reviewing changes through version control can be a little bit more seamless with standard</a> <a data-t="2132">Python files. You have testing that we talked about previously. You can define tests in one</a> <a data-t="2136">place in your developer environment, and those tests can then be applied anywhere those Python</a> <a data-t="2140">files run.</a></p>

<p><a data-t="2146">And the idea here really is to say, look, let's have one collection of code. We can</a> <a data-t="2151">build tests around that. We can run that code in our local environment in Positron. And then we can,</a> <a data-t="2155">using Databricks asset bundles and some of these other tools that Databricks provides,</a> <a data-t="2160">we can then move that code, push that out into Databricks, and run that as part of a pipeline</a> <a data-t="2165">or a scheduled job inside of our workspace.</a></p>

<p><a data-t="2170">Now, again, this isn't to say that we shouldn't</a> <a data-t="2175">ever use notebooks. Notebooks can be great for ad hoc data analysis, doing quick experiments,</a> <a data-t="2180">sharing results, right? Notebooks have a nice sort of format to them. It's a lot easier to sort of</a> <a data-t="2184">look at a notebook. You've got inline plots than it is to like look at just a Python file if you're</a> <a data-t="2189">trying to walk somebody through something. And for interactive tutorials or just kind of</a> <a data-t="2193">interacting in real time, I think notebooks provide a really great interface. But when you're</a> <a data-t="2208">doing sort of these production workloads, defining multiple different pieces at a time, building out</a> <a data-t="2214">these complex pipelines and jobs, using a true developer environment, and we think Positron is</a> <a data-t="2219">very well suited for this task, can start to have a distinct advantage over trying to consolidate</a> <a data-t="2225">and run everything outside of a notebook or collection of notebooks.</a></p>
