<h3>What is validation?</h3>

<p><a data-t="50">So a common question that any of us, if we've ever tried to introduce new software into</a> <a data-t="55">the pharma world is, is it validated?</a> <a data-t="60">That's a question that we're always going to be getting.</a> <a data-t="60">Is it validated?</a> <a data-t="66">But what is validation?</a> <a data-t="66">It can mean different things to different people.</a> <a data-t="66">Yes, there's an official definition, but it means different things to different people.</a></p>

<p><a data-t="71">So as I understand it, the definition of validation is establishing documentary evidence that</a> <a data-t="77">our software performs some sort of process, procedure, activity, and compliance with our</a> <a data-t="82">specifications with a high degree of assurance.</a> <a data-t="86">So in layman's terms, we're creating documents</a> <a data-t="86">to prove that our software does what we say it does.</a></p>

<p><a data-t="91">But why do we even care about validation?</a> <a data-t="95">Why is that question even asked?</a> <a data-t="95">Well, a lot</a> <a data-t="95">of people refer to the fact that we'd have to validate our software for FDA submission.</a> <a data-t="101">But there's a lot of unspoken benefits around validation that don't always get talked about,</a> <a data-t="105">such as improved quality and safety of our code, because we've gone through and checked</a> <a data-t="108">it.</a> <a data-t="113">We're confident in the quality of the code, as well as the fact that it'll return</a> <a data-t="113">proper results.</a> <a data-t="117">It results in faster processing.</a> <a data-t="117">Because we already have this code now set</a> <a data-t="117">up, we can use it across multiple projects and trust that it'll work.</a> <a data-t="123">And then it also</a> <a data-t="123">promotes trust.</a> <a data-t="127">Because we've performed this validation, we've gone through and vetted</a> <a data-t="127">it the best to our abilities.</a> <a data-t="132">So we are confident that when we give it an input, it'll give</a> <a data-t="132">us a consistent output, as well as throw an error when things go out of bounds.</a></p>

<p><a data-t="138">So validation practice can be a really high bar.</a> <a data-t="143">And there's a lot of documents that go</a> <a data-t="143">into creating validation.</a> <a data-t="148">Because first, you have to fill out a form for specifications,</a> <a data-t="152">planned uses, environments you're going to be using it in.</a> <a data-t="152">Then write out some of your</a> <a data-t="152">code based on those specifications.</a> <a data-t="156">Record the function authorship in some external file,</a> <a data-t="156">potentially an Excel file.</a> <a data-t="161">Then get another form to document your test cases, test environment,</a> <a data-t="161">how you plan on doing your testing.</a> <a data-t="166">Then maybe the last form to show how your testing plan</a> <a data-t="166">is comprehensive, that you're proving that all your specifications are being met through</a> <a data-t="170">your testing plan.</a> <a data-t="175">Have a third party come through and manually evaluate your tests and</a> <a data-t="175">get screenshots for results.</a> <a data-t="179">And then you have to review your documentation, combine</a> <a data-t="179">it into a final validation packet for release.</a> <a data-t="183">But information is shared across all of these</a> <a data-t="183">documents.</a> <a data-t="187">So any time one of them updates, you need to go back and update it.</a> <a data-t="187">And if</a> <a data-t="187">you manually evaluated all your tests and had to make an update, you're going to have</a> <a data-t="190">to go through and rerun everything.</a> <a data-t="194">So it's incredibly inefficient.</a> <a data-t="194">And at any time, it</a> <a data-t="194">can feel like game over because you're just redoing your work over and over again.</a></p>
