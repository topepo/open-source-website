<p><a data-t="1653">And you really want two of these things.</a> <a data-t="1656">They're basically identical, but I think the main difference is like your staging environment, like data scientists look at that.</a> <a data-t="1663">And decision makers look at the deployment environment.</a> <a data-t="1666">They're otherwise identical, but it means you can like put something, send something to staging and look at it to see if it's working or not</a> <a data-t="1670">without accidentally breaking the things that the people in your organization use to make decisions.</a></p>

<p><a data-t="1677">And I think these, I've kind of put, they're pretty thick lines here because there's some fairly big challenges in each of these transitions.</a> <a data-t="1686">You know, kind of one of the things that, I think like one of the goals of Posit, the company that I work for, is to help with each of these.</a> <a data-t="1697">Like the idea of Posit Workbench, that helps you run something like, like run the </a><a href="/software/rstudio/">RStudio</a><a data-t="1697"> IDE, Jupyter Notebooks, Visual Code, like in a standardized way that's centrally maintained pretty easily.</a> <a data-t="1715">RStudio Connect allows you to deploy things like apps, Shiny apps, like Markdown documents, </a><a href="/software/quarto/">Quarto</a><a data-t="1715"> documents, Dash apps, Flask apps,</a> <a data-t="1725">in a way where you can kind of work on your laptop or central compute and just get things published as easily as possible.</a></p>

<p><a data-t="1769">Then as Mark pointed out, there's some major pain points moving from an environment where you can debug interactively to an environment where you run a script.</a> <a data-t="1777">And 30 minutes later, you get an error message back.</a> <a data-t="1779">And there's also some similar challenges that arise related to like who is doing the analysis.</a> <a data-t="1787">Like typically, on your laptop or in central compute, like the person doing the analysis is you.</a> <a data-t="1793">It's your access to the database that chooses what data to come back.</a> <a data-t="1798">When you start moving into a staging environment or a deployment environment, you need to think about like whose access is being used to get the data.</a> <a data-t="1808">This is like particularly important when the data is sensitive.</a> <a data-t="1810">And like different people in your organization can see different views of the data set.</a></p>

<h3>Windows vs. Linux and desktop vs. server</h3>

<p><a data-t="1815">So the first challenge to overcome is if you're a Windows user on your laptop, there's a bunch of things that are just different on Linux for like historic reasons.</a> <a data-t="1823">None of them are particularly important, but they're all annoying and they will all catch you out.</a> <a data-t="1827">Like Windows uses different line endings.</a> <a data-t="1829">It uses a different character encoding in R, and it uses a different path separator.</a> <a data-t="1834">And all of these are just like little things you will stub your toe on again and again and again and again.</a></p>

<p><a data-t="1840">And I guess I stub my toe on these things too because I generally write my code on a Mac, which for these purposes is functionally a Linux.</a> <a data-t="1848">And then when I test my code on Windows machines, I'm like, oh, the path separator is wrong.</a> <a data-t="1852">Oh, there's some weird thing in my test snapshots because the line endings are different.</a></p>

<p><a data-t="1861">Also differences because your computer is a desktop and the central computer is a server.</a> <a data-t="1868">Like a desktop is set up for like you.</a> <a data-t="1870">It has set up to use your time zone.</a> <a data-t="1873">It's set up to use your language of preference.</a> <a data-t="1875">It has all of the fonts you've installed, and it has graphics devices that lots of people use and a lot of care has been put into.</a> <a data-t="1883">When you move to a server, it's probably defaulting to a UTC time zone.</a> <a data-t="1888">So that's sort of basically the same as Greenwich Mean Time.</a> <a data-t="1892">It's probably using a C locale.</a> <a data-t="1895">So it just sorts things in kind of the ASCII order.</a> <a data-t="1900">Probably doesn't have many fonts installed, and it probably has kind of crappy graphics devices.</a> <a data-t="1905">The first two are gonna cause potentially problems for your code.</a></p>

<p><a data-t="1911">One of the things like we've tried to do in the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="1911"> is make sure as much as possible you don't automatically derive those things.</a> <a data-t="1920">For example, there's a really interesting problem that in different, some different, some locales sort the letters of the alphabet differently to other locales.</a> <a data-t="1930">The order of a factor in R by default is alphabetical ordering, and that determines your contrast of a linear model.</a> <a data-t="1937">So it is possible, it's pretty unusual, but it's possible and it has happened to people that by running code in a different locale, you get a bunch of different contrasts outputted from your model.</a> <a data-t="1947">Like the model's still the same, but it's confusing for people looking at the outputs.</a></p>

<p><a data-t="1952">And the last two are just pain points.</a> <a data-t="1954">Like if you're trying to make sure all of your plots look exactly the same using your corporate style guide, your approved fonts,</a> <a data-t="1959">this is just a pain to get right when you can't easily install fonts on your computer.</a></p>

<h3>Package installation and interactive vs. batch debugging</h3>

<p><a data-t="1972">So one of the big differences between the way our packages work on Windows and Mac and Linux is that on Windows and Mac,</a> <a data-t="1980">when you install a package like </a><a href="/software/xml2/">XML2</a><a data-t="1980">, which uses the libxml2 library,</a> <a data-t="1985">on Windows and Mac, libxml is like bundled inside that package.</a> <a data-t="1989">And you can just use it without having to separately worry about using libxml.</a> <a data-t="1993">Servers, generally, you're gonna install a package from source, which means you need to compile it,</a> <a data-t="1999">which means you need to have all these additional bits of software installed on your computer.</a> <a data-t="2003">That's changing a little bit, like we've done a lot of work with PositPackageManager and PositPublicPackageManager to make this easier.</a> <a data-t="2012">And there's some great tools in the </a><a href="/software/pak/">pak</a><a data-t="2012"> package, which will tell you like all of the system dependencies that you need to install this package and all of its recursive dependencies,</a> <a data-t="2020">but tends to be a bit more painful and that might be something you need to get a server admin involved with.</a> <a data-t="2026">It's not something you can necessarily do by yourself anymore.</a></p>

<p><a data-t="2030">On your laptop, you typically just install all of your R packages in one library.</a> <a data-t="2035">Obviously, now on a shared server, you're gonna have lots of different libraries, because not everyone's gonna agree on that.</a> <a data-t="2041">Similarly, it's a wild west on your laptop.</a> <a data-t="2044">You install packages from GitHub, you install them from CRAN, you install them from Bioconductor,</a> <a data-t="2047">you install them from like some random site you found on the internet.</a> <a data-t="2052">Hopefully on your central compute, that's gonna be a little bit more locked down.</a></p>

<p><a data-t="2059">So that was kind of the transition from your laptop to central compute.</a> <a data-t="2063">There's another transition from central compute to deployment.</a> <a data-t="2067">And that is you are no longer in an interactive environment.</a> <a data-t="2070">So when something goes wrong, when you get an error, you can no longer say, oh, give me a trace back,</a> <a data-t="2076">or let me browse into this function and fix it.</a></p>

<p><a data-t="2081">So debugging something that's failing on some other computer is extremely frustrating.</a> <a data-t="2091">If you develop R packages, like the way you experience this often is that something fails in your automatic checks on GitHub Actions.</a> <a data-t="2099">And it fails in a way that you can't reproduce locally, and now your iteration speed,</a> <a data-t="2104">instead of like typing a few ideas of the console and getting them back in seconds,</a> <a data-t="2110">it's now like a 20 minute or a 30 minute loop,</a> <a data-t="2113">which is bad, not just because it's so long,</a> <a data-t="2115">but it's long enough that you do something else and then you forget about the thing you're trying to do</a> <a data-t="2119">and then you come back to it two hours later and you've lost all the state you had in your head.</a></p>

<p><a data-t="2123">So there's definitely some like techniques that are different.</a> <a data-t="2126">Like when you move into a, when you're in an interactive debugging environment,</a> <a data-t="2131">it's all about like very quickly coming up with hypotheses about what's going wrong, testing them out, and then discarding them.</a> <a data-t="2137">And because you're in that short iteration cycle, you can do that very simply.</a> <a data-t="2141">When you're in a batch scenario, it's more about like brainstorming every possible thing that could be causing this problem,</a> <a data-t="2149">trying to write some code that can identify all of those cases at once, and then deploying that</a> <a data-t="2153">so that when you get that information back in 20 minutes, you haven't just answered one possible question, you've answered like 10 possible questions.</a></p>

<p><a data-t="2186">And the other kind of technique that starts to get really useful is this idea of logging,</a> <a data-t="2201">just recording like what is happening in your code?</a> <a data-t="2204">Where have you got to?</a> <a data-t="2206">So when it goes wrong, you know, have you have some sense of like, how did I get to this point?</a></p>

<h3>Authentication and access control</h3>

<p><a data-t="2213">There's some certainly like really cool technologies out there about like with time travel debugging,</a> <a data-t="2218">where it is theoretically possible in some languages to kind of capture a failure on a server</a> <a data-t="2226">and then bring it back to your local computer in a way that you can interactively debug.</a> <a data-t="2230">That seems like a dream, but not something that is easy to port to R, unfortunately.</a></p>

<p><a data-t="2218">And then the other kind of issue with where this difference comes up is when you're authenticating against some server.</a> <a data-t="2245">Now, the easiest way and most, the way that works just about everywhere is to stuff credentials and environment variables.</a> <a data-t="2253">That works locally, that works in your deployment environment, that is all fine and dandy,</a> <a data-t="2259">but it is not really a great way to do auth because a lot of these credentials ideally should be changing over time.</a> <a data-t="2266">They should be rotating and that means you need to update them.</a> <a data-t="2268">And if you've relied on like that password and like 15 different data science projects,</a> <a data-t="2274">now you're going to go through and update all 15 of those.</a></p>

<p><a data-t="2277">So ideally, you're using something a bit more interactive, something like OAuth,</a> <a data-t="2282">where you're going to do like a little interactive dance to give the server some permission for you to do stuff on its behalf.</a> <a data-t="2289">It's going to give you something back interactively.</a> <a data-t="2291">This kind of all happens behind the scenes for you, all very easy to do interactively.</a> <a data-t="2298">But if you're doing this on a server, you can't do that kind of interactive dance.</a> <a data-t="2302">And that also forces you to think about, well, like, who is doing this?</a> <a data-t="2306">Who is seeing this analysis?</a></p>

<p><a data-t="2308">And typically, like if you're doing that locally, you are doing the analysis,</a> <a data-t="2312">you're going to pull down the data that you are allowed to see and work with that.</a> <a data-t="2318">If you're deploying something like a </a><a href="/software/shiny-r/">Shiny</a><a data-t="2318"> app, it's also possible that it should be using the data that the person viewing the Shiny app is allowed to see.</a> <a data-t="2326">I think that kind of the easiest scenario to think about this is like, if you're writing an HR app,</a> <a data-t="2331">like you should only be able to see the data for yourself and people report that report to you.</a> <a data-t="2337">You shouldn't be able to see random data for people in the organization.</a></p>
