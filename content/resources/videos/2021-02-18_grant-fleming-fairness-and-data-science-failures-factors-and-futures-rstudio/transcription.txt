<p><a data-t="0">My name is Grant Fleming, and I'm a data scientist at Elder Research, where I work on data science</a> <a data-t="13">consulting projects with various public sector clients.</a> <a data-t="17">And in that work, along with a lot of the work that I'm doing outside of work work,</a> <a data-t="23">I'm having to wrestle with the issue of fairness and data science, specifically things like</a> <a data-t="28">how do we make sure that the predictions being generated by our models are fair?</a> <a data-t="33">How do we make sure we're actually recording the correct information in our data and not</a> <a data-t="39">leaving certain important factors out?</a> <a data-t="42">And if we suspect that something may have snuck in our model, how do we make sure that</a> <a data-t="48">we're able to detect that element before we deploy our model and cause harm intentionally?</a></p>

<p><a data-t="54">Because as we've seen probably in some of the news over the past few years, this is</a> <a data-t="58">an issue that's, if not become more widespread, become more widely known.</a> <a data-t="62">For example, in 2018, Joy Buolamwini and her research team at MIT published the GenderShades</a> <a data-t="69">work that they did on facial recognition algorithms from companies like Amazon, IBM, and others.</a> <a data-t="78">And in this work, they found out that these facial recognition algorithms were actually</a> <a data-t="85">having much higher error rates, detecting darker skin faces versus lighter skin faces.</a> <a data-t="91">And that obviously is something that we would not want to see in any sort of fair model.</a></p>

<p><a data-t="96">Similarly, we saw something similar in 2019 in the healthcare sector, where an algorithm</a> <a data-t="103">furnished to hospitals throughout the US, which was tasked with helping to allocate</a> <a data-t="110">treatment to patients within the hospitals, was systematically underestimating the healthcare</a> <a data-t="117">needs for black patients relative to white patients or patients from other ethnic groups.</a> <a data-t="123">Again, likely a case of unintended bias, but nonetheless, something that led to harms in</a> <a data-t="129">the real world.</a></p>

<h3>Why these failures happen</h3>

<p><a data-t="131">And I could spend the rest of the session talking about these issues across the various</a> <a data-t="136">different examples, which I'm not including, but that's not productive.</a> <a data-t="140">Maybe what's more productive then is coming to an understanding of why this is actually</a> <a data-t="144">happening.</a></p>

<p><a data-t="146">So from my perspective, at least, it seems like this is in large part the result of us</a> <a data-t="152">as data scientists thinking that our tools alone, the technical tools, the code and the</a> <a data-t="157">data and the models that we were able to build are enough to solve these complex social problems.</a> <a data-t="164">Like duct tape, we use them wherever we can to fix whatever we can.</a> <a data-t="169">And some situations are helped by duct tape, but duct tape cannot alone solve everything.</a> <a data-t="176">I think that's a good metaphor here for how we should think about it.</a></p>

<p><a data-t="180">How we should think about it instead is our toolkit is not duct tape that should be used</a> <a data-t="188">wildly wherever we can, but rather our toolkit is one that, along with people who have the</a> <a data-t="193">subject matter and lived expertise in the areas that we seem to operate in, they, along</a> <a data-t="201">with us, can lay brick by brick a solution to a problem or at least get closer to solving</a> <a data-t="207">it.</a></p>

<p><a data-t="208">And this is twofold.</a> <a data-t="210">We from them can get a greater appreciation for the factors that can't be encoded in our</a> <a data-t="215">data or aren't encoded in our data, but are nonetheless relevant to our work.</a> <a data-t="221">And they, in turn, can help us to better diagnose issues within our models before they're deployed.</a></p>

<h3>A framework for responsible data science</h3>

<p><a data-t="230">Sounds good, but how do we actually do this?</a> <a data-t="232">How do we actually do responsible, ethical, trustworthy, however you want to call it,</a> <a data-t="236">data science in practice?</a> <a data-t="238">It's the million dollar question, and a number of researchers are working to try and answer</a> <a data-t="243">this.</a> <a data-t="244">And to my reading, it seems like they've coalesced around the three main factors that go into</a> <a data-t="249">making a data science project, we'll just say a responsible data science project.</a></p>

<p><a data-t="254">Those being, first, fairness in procedures.</a> <a data-t="259">So basically, was the project designed in such a way that it can be conducted fairly?</a> <a data-t="264">Did it include a diverse range of voices and all relevant stakeholders?</a> <a data-t="270">Next would be fairness in predictions.</a> <a data-t="272">So when the model was actually built, does the model perform equally well across different</a> <a data-t="278">groups within the data?</a> <a data-t="279">And the final one being fairness and outcomes.</a> <a data-t="282">So once the actual process of the model is set in stone and is more or less fair, and</a> <a data-t="289">even if we get relative fairness in our predictions, does the impact of those predictions actually</a> <a data-t="297">improve fairness in the real world?</a> <a data-t="300">Are the impacts of those predictions equitable?</a></p>

<h3>Fairness in procedures</h3>

<p><a data-t="324">And where better to start thinking about that than ensuring inclusion.</a> <a data-t="329">If you don't have a diverse perspective or diverse group of perspectives on your project,</a> <a data-t="335">in all likelihood, something could slip through that would cause an unintended harm when your</a> <a data-t="360">model was deployed.</a> <a data-t="361">This is especially relevant for someone like me, who not only is a consultant, who's often</a> <a data-t="366">coming into a data science project in a subject matter context where I have less expertise</a> <a data-t="372">than maybe some of the people I'm working with, but also because I'm a white man.</a> <a data-t="378">And that bears with it certain biases, some of them benign, some of them harmful, in how</a> <a data-t="387">I see the world and operate within it.</a></p>

<p><a data-t="390">And it's important for me and the people I work with on projects and the clients I work</a> <a data-t="396">with that those biases are checked by the views that other people can bring to the table.</a> <a data-t="402">And I think critically, this is something that is worthy in all projects and besides</a> <a data-t="408">being the right thing to do.</a></p>

<p><a data-t="411">Next though, would be reproducible code.</a> <a data-t="414">We don't want code that just runs on our own machines.</a> <a data-t="418">This is maybe easier said than done, but there are fortunately within the R ecosystem a number</a> <a data-t="424">of packages available for this.</a> <a data-t="426">I won't go too much into detail on what those are because so many other people have already</a> <a data-t="432">spoken in depth about these.</a> <a data-t="433">I'll reference two of them here.</a> <a data-t="436">The first would probably be Dr. Anna Cristalli.</a> <a data-t="439">She has a great talk called Putting the R into Reproducible Research, which you can</a> <a data-t="444">find by Googling.</a> <a data-t="445">It should come up.</a></p>

<p><a data-t="447">And then another person, Karthik Ram, delivered a talk at the previous </a><a href="/software/rstudio/">RStudio</a><a data-t="447"> conference</a> <a data-t="452">in 2019.</a> <a data-t="453">Google that.</a> <a data-t="454">You should find it.</a> <a data-t="455">I don't remember the title, but between the two of them, they covered a lot of really</a> <a data-t="458">great materials for doing reproducible work within R.</a></p>

<p><a data-t="463">But back to fairness and procedures, finally, to wrap this all up, we want to make sure</a> <a data-t="468">that we document everything we do and every decision we make in the same way that we want</a> <a data-t="473">to have our code be reproducible by technical people.</a> <a data-t="477">We also want non-technical people who may not be able to actually access our code directly</a> <a data-t="482">to be able to pour through our results in a way that's easy for them to interpret and</a> <a data-t="486">that they can actually evaluate.</a></p>

<p><a data-t="489">And the Google Brain Ethical AI team has done a lot of good work on this.</a> <a data-t="495">They've come up with data sheets, model cards, and audit reports.</a> <a data-t="499">So three different sort of documentation modalities that you can all think of as nutrition labels,</a> <a data-t="506">I guess, for each part of the project.</a> <a data-t="509">And even though we don't yet have any way to do these within R, hopefully soon there</a> <a data-t="514">will be some sort of R markdown templates that make this easier for us.</a> <a data-t="519">I'm certainly on my own end trying to put together a package for it, and we'll hopefully</a> <a data-t="522">have some more updates on that in the future.</a></p>

<h3>Fairness in predictions</h3>

<p><a data-t="553">And so you're in good shape to start now thinking about fairness and predictions, starting with</a> <a data-t="559">predicting accurately.</a> <a data-t="560">This doesn't need a whole lot of development, but I include this here to make sure to point</a> <a data-t="570">out that if you do not have high predictive performance overall on your model, by no conception</a> <a data-t="577">can it be fair.</a> <a data-t="578">Even if you perform equally across groups, getting a 10% level of performance across</a> <a data-t="583">all of them is meaningless, and that's going to do you and the people this model is hoping</a> <a data-t="589">to serve no good whatsoever.</a></p>

<p><a data-t="592">But more subtle and probably more relevant to the work we do is predicting equally.</a> <a data-t="598">And to explore this in practical detail and how we can actually work on this in our own</a> <a data-t="603">projects, we're going to go through an example with the Compass Recidivism dataset.</a> <a data-t="608">So this is a dataset released by the public interest group ProPublica, and what it tracks</a> <a data-t="615">is demographic and criminal records information of defendants.</a> <a data-t="622">And the goal of this is to take all this information and use it to predict whether these defendants</a> <a data-t="627">will commit another crime in the next two years.</a></p>

<p><a data-t="630">Because of the demographic information within this data, it's often used as sort of the</a> <a data-t="635">iris or titanic dataset for fairness methods and evaluation of fairness maintaining approaches</a> <a data-t="642">within data science, as we're about to do here, and as I already did here.</a></p>

<p><a data-t="648">So on this plot, we can see two different bars, one for the accuracy of a logistic regression</a> <a data-t="654">and one for the accuracy of a random forest model.</a> <a data-t="657">And again, these are taking demographic and criminal history features and predicting a</a> <a data-t="661">binary yes, no, will commit a crime, will not commit a crime within two years.</a> <a data-t="667">We can see that they perform roughly the same, nothing really jumping out here, which is</a> <a data-t="673">why maybe we want to dig a little deeper.</a></p>

<p><a data-t="676">Maybe we can look at per group metrics.</a> <a data-t="679">So in this case, instead of just looking at overall model performance, we'll still look</a> <a data-t="685">at the performance of one model versus the other, but we're going to look at it with</a> <a data-t="690">respect to a protected feature or sensitive feature, in this case, race, and then the</a> <a data-t="695">protected groups, aka the levels of that feature, which in this case would be Hispanic, other</a> <a data-t="701">Caucasian or African-American.</a> <a data-t="703">And already here, we can see that there's some differences or disparities in performance</a> <a data-t="707">coming to the fore, with African-Americans specifically, the model getting the lowest</a> <a data-t="714">accuracy for African-Americans relative to other groups.</a></p>

<p><a data-t="718">This is already a pretty compelling result, I think, goes to show that overall metrics</a> <a data-t="725">alone are not enough to be able to identify when you may have some issues or unfairness</a> <a data-t="730">in your model, but we still don't necessarily know why these disparities exist.</a> <a data-t="735">These disparities have to be in part due to different rates of error, and we don't know</a> <a data-t="740">whether that's more false positives or more false negatives for the different groups.</a> <a data-t="744">So let's actually look at plots of error direction.</a></p>

<p><a data-t="748">And here's where we start to see the real compelling causes come to the fore here, with</a> <a data-t="755">in this case, the model hitting a almost four times higher false positive rate error for</a> <a data-t="761">African-Americans than Caucasians, and similarly high rates for other non-Caucasian groups.</a> <a data-t="769">This is clearly an issue of bias, and if we were not to rectify this or communicate this</a> <a data-t="774">to our clients or the non-technical people we're working with in this project, and we</a> <a data-t="780">were to just put a stamp on this and let it go, in the real world, this would perpetuate</a> <a data-t="786">all sorts of biases that would be harmful, and we don't want that.</a></p>

<blockquote><a data-t="748">And here's where we start to see the real compelling causes come to the fore here, with</a> <a data-t="755">in this case, the model hitting a almost four times higher false positive rate error for</a> <a data-t="761">African-Americans than Caucasians, and similarly high rates for other non-Caucasian groups.</a></blockquote>

<p><a data-t="792">So to get a better context to that, before we actually communicate those conclusions,</a> <a data-t="797">maybe we want to be able to say, okay, we've made every comparison between every different</a> <a data-t="801">group, so Caucasians with African-Americans, Caucasians with Hispanics, and maybe we care</a> <a data-t="806">specifically about the relative performance, so we want to look at the percent differences.</a> <a data-t="811">That's where fairness metrics come in.</a> <a data-t="813">They don't measure absolute performance, they measure relative.</a></p>

<p><a data-t="816">When we look at it in this plot here, it's still the same y-axis, still the same x-axis,</a> <a data-t="821">but now we get the percentages of false positive rates for each group relative to the privileged</a> <a data-t="827">group, Caucasians.</a> <a data-t="829">And again, we can see here that there's quite a lot of disparity.</a> <a data-t="834">Why though?</a> <a data-t="836">What is it in the features that are going into the model that are causing these disparities</a> <a data-t="841">to exist?</a></p>

<p><a data-t="843">It's hard to get at causality at any point within data science, but if we want to at</a> <a data-t="848">least understand more about how the features impact these predictions differently, we're</a> <a data-t="853">going to want to look at interpretability methods.</a> <a data-t="856">These are often explained in the context of the black box, hence the image on the right.</a> <a data-t="862">What they allow us to do is open this black box, because a lot of these interpretability</a> <a data-t="867">methods are model agnostic, and they can be used even on models like XGBoost or Random</a> <a data-t="872">Forest to generate the sort of explanations, maybe not quite to the same level of fidelity,</a> <a data-t="879">but approximately, that we would get from a linear model, like linear regression or</a> <a data-t="884">something like logistic regression.</a></p>

<p><a data-t="887">I'm not going to go through these in detail within this presentation, because the visuals</a> <a data-t="891">for them are a little complex and we don't have a lot of time, but I will mention that</a> <a data-t="894">they come in two main varieties, the first being global methods, which explain the average</a> <a data-t="899">effect of features globally.</a> <a data-t="900">You can think of this as like the coefficient of a regression, and if you want to read more</a> <a data-t="905">about them, you can look up something like permutation feature importance, which ought</a> <a data-t="909">to be well known to a number of people, partial dependence plots, or individual conditional</a> <a data-t="914">expectation plots.</a></p>

<p><a data-t="917">We can also look at something in the category of the local methods, and this could be say</a> <a data-t="923">LIME or Shapley values or derivatives of either of those, and these are really powerful</a> <a data-t="929">because they allow us to say for an individual prediction on one observation, each feature</a> <a data-t="936">value, we can quantify the exact amount that it contributed to the prediction.</a> <a data-t="941">So if someone were to ask, hey, why did I get this specific classification from your</a> <a data-t="945">model?</a> <a data-t="946">You could tell them this feature contributed this much, this contributed this much, etc.</a> <a data-t="950">A lot of useful explanation there.</a></p>

<p><a data-t="953">Hopefully this is something that we can end up applying to most of our models, and it's</a> <a data-t="957">certainly a set of techniques which are well developed within the R package ecosystem currently.</a> <a data-t="963">The IML package by Christoph Molnar is available, and I use it in my work and I think it's fantastic,</a> <a data-t="969">but there's also the VIP and DALEX, D-A-L-E-X, I believe, packages which include a number</a> <a data-t="976">of interpretability methods as well.</a></p>

<h3>Fairness in outcomes</h3>

<p><a data-t="1009">Outcomes are all about the consequences and how we consider fairness in the context of</a> <a data-t="1015">the real world.</a> <a data-t="1017">Because these outcomes and the impacts they have on people are not necessarily something</a> <a data-t="1021">that we can measure in the data, or at least can, or at least do measure in the data, it's</a> <a data-t="1027">something that requires conversations not just with the people who maybe we're working</a> <a data-t="1033">with who are non-technical, who have the subject matter expertise for the projects we're working</a> <a data-t="1038">on, but also the people impacted by the models.</a></p>

<p><a data-t="1042">And talking with all these different groups and coming up with some conception of fairness</a> <a data-t="1047">to say, okay, maybe we're going to optimize for certain predictions despite these predictions</a> <a data-t="1055">coming at a loss of predictive performance for our model, because ultimately that's the</a> <a data-t="1059">equitable thing to do, that's something that's very project-specific.</a> <a data-t="1063">It doesn't make a lot of sense for me or anyone else to give any sort of real definitive</a> <a data-t="1069">guidance on that.</a></p>

<p><a data-t="1071">And just for a little bit more detail, maybe an example to make this clear, you could think</a> <a data-t="1076">of fairness and outcomes as being relevant in a case where maybe we're deciding whether</a> <a data-t="1082">someone deserves a scholarship, binary yes-no, a yes would benefit someone who comes from</a> <a data-t="1089">maybe a more disadvantaged background, much more than someone who comes from a more privileged</a> <a data-t="1094">background.</a> <a data-t="1095">And again, that's not necessarily something we'd measure within our model.</a> <a data-t="1099">But if it's something we want to try and work towards, then we need to make sure to mitigate</a> <a data-t="1103">the bias that we see in our model in a specific way to serve that end.</a></p>

<p><a data-t="1108">And that's where bias mitigation techniques come in.</a> <a data-t="1112">And I have the star on the word there in the previous slide, because these techniques,</a> <a data-t="1117">they sometimes work well, they sometimes don't, but by and large, they all have trade-offs.</a> <a data-t="1122">And that's because you're either modifying the data in the case of pre-processing that</a> <a data-t="1126">goes into your models, the model itself, or the predictions that come out of the model</a> <a data-t="1132">to reach some notion of fairness.</a> <a data-t="1134">And while that may improve that specific notion of fairness, you may see that there are losses</a> <a data-t="1140">in predictive performance or fairness across other dimensions.</a></p>

<p><a data-t="1144">And to show this, we'll go back and look at the compass example.</a> <a data-t="1147">So here, I have a plot, again, it shows protected groups on the y-axis and false positive rate</a> <a data-t="1154">on the x-axis.</a> <a data-t="1157">But now we're looking, instead of at logistic regression and random forest, we're looking</a> <a data-t="1161">at just the random forest.</a> <a data-t="1163">But we're looking at two sets of predictions here, one where the prediction thresholds</a> <a data-t="1168">are all set at 0.5, for if you have a higher than 0.5, we'll predict that you will commit</a> <a data-t="1175">a crime, lower the opposite.</a> <a data-t="1178">And one that's set to an optimal level for each class, such that it minimizes false positive</a> <a data-t="1184">rate.</a> <a data-t="1185">And we can see here that it does that well, all groups actually have a large decrease</a> <a data-t="1189">in false positive rate, which is great, at least relative to each other.</a></p>

<p><a data-t="1195">We need to actually go back a few levels to look at the per group performance metrics</a> <a data-t="1201">on the absolute basis to see what happened there.</a> <a data-t="1205">And when we do that, unfortunately, we see that most of the gains in fairness that we</a> <a data-t="1210">made in that previous plot are actually due to the error rate, mostly increasing for the</a> <a data-t="1217">groups, but increasing for the other groups more than it did for the African American</a> <a data-t="1222">group.</a> <a data-t="1224">So basically, what happened here is the model said, OK, you want to make it more fair across</a> <a data-t="1229">groups for false positive rate, we'll make it so that the other groups have more error</a> <a data-t="1235">and make it so that the African American group, their error relative to the other groups is</a> <a data-t="1239">lower than it was originally.</a> <a data-t="1241">Again, if fairness is our goal here, that's fine, that's not a bad result.</a> <a data-t="1247">However, it clearly did come with a trade off.</a> <a data-t="1251">And whenever we use these methods, this is just something we're going to have to consider.</a></p>

<p><a data-t="1256">Lastly, for fairness and outcomes, we want to consider the possibility of algorithmic</a> <a data-t="1262">recourse.</a> <a data-t="1263">So if our models are making decisions about people, we don't want them to lose the ability</a> <a data-t="1269">to actually have some influence over that decision.</a> <a data-t="1272">They should be able to appeal it, they should be able to ask for explanations as to why</a> <a data-t="1276">they were given that decision, and they should be given, if asked, or if they choose to ask,</a> <a data-t="1282">recommendations for what they can do to change it.</a> <a data-t="1284">For more info on this, I recommend looking at the work of Ruha Benjamin, William Isaac,</a> <a data-t="1290">or Shakir Mohammed.</a></p>
