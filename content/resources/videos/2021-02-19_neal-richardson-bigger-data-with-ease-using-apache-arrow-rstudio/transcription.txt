<h3>What makes Arrow so fast</h3>

<p><a data-t="178">So how were we able to do that?</a> <a data-t="183">So what I want to talk about today is how characteristics of the Arrow project, and</a> <a data-t="184">particularly the community around it, enable us to do things that we might otherwise have</a> <a data-t="190">thought would be impossible to do in R.</a></p>

<p><a data-t="195">So a little history.</a> <a data-t="197">So Arrow was started in 2016.</a> <a data-t="200">A group of database developers and data frame library maintainers got together and realized</a> <a data-t="207">that they were all trying to solve similar problems.</a> <a data-t="210">And rather than saying, this is fine, and we're just going to keep duplicating each</a> <a data-t="214">other's work, what if we work together and create a shared foundation for our work?</a> <a data-t="219">And we would all benefit from that.</a></p>

<p><a data-t="222">So Arrow is fundamentally about three things.</a> <a data-t="225">First is it is a format, is a specification for how data is represented in memory.</a> <a data-t="231">It's columnar, and it's designed to take advantage of features of modern CPUs and GPUs and other</a> <a data-t="237">hardware.</a> <a data-t="240">There's also a set of 12 libraries that implement it in different languages.</a> <a data-t="245">Python and R use the C++ library.</a> <a data-t="248">But there are many others in the Arrow project.</a> <a data-t="254">And then third, there's this broader ecosystem of packages and projects around Arrow that</a> <a data-t="260">use Arrow in some form, whether it's as their internal data model, or is it just a means</a> <a data-t="265">of exchange between projects.</a></p>

<p><a data-t="268">So how do these characteristics of Arrow lead us to get such unbelievable performance and</a> <a data-t="276">be able to enable us to do things that we didn't think were possible in R?</a> <a data-t="282">So in terms of modern hardware, Arrow is designed to take advantage of many features of CPUs</a> <a data-t="287">that exist now that didn't exist when R was developed initially.</a> <a data-t="292">So R was first released in 1995.</a> <a data-t="296">Of course, computer technology has progressed quite a bit since then.</a> <a data-t="299">And there's things that I can do on my laptop now that was not something that was a thing</a> <a data-t="305">back then.</a></p>

<p><a data-t="306">So for example, my laptop here has got eight cores.</a> <a data-t="311">That's CPUs that you can run in parallel.</a> <a data-t="313">You can get a host on AWS with 96 cores.</a> <a data-t="317">That's pretty good.</a> <a data-t="318">But R is generally only using one of those.</a> <a data-t="320">So we're leaving a lot of performance on the table.</a> <a data-t="324">Newer CPUs also can take advantage of what's called SIMD, single instruction multiple data.</a> <a data-t="330">My laptop, for example, can take up to 256 bits at a time.</a> <a data-t="333">So that would be the size of eight integers in R. Other newer ones can do 512 bits.</a> <a data-t="339">So you can feed a lot more data into the CPU at a time if your code is designed to take</a> <a data-t="344">advantage of that, but R generally is not.</a></p>

<p><a data-t="348">So it's like you have this super fast sports car and you just want to take it for a spin</a> <a data-t="353">around the track and see what you can do with it.</a> <a data-t="356">But for whatever reason, you have to keep it under 25 miles an hour, drive, keep your</a> <a data-t="364">hands at 10 and 2 on the steering wheel, nothing too crazy.</a> <a data-t="369">And it's a really missed opportunity.</a></p>

<p><a data-t="371">So what do you do about that?</a> <a data-t="372">So you could write some C or C++ code in your R package to do multi-threading to take advantage</a> <a data-t="380">of all those extra cores that you have available.</a> <a data-t="383">You could write code to do SIMD optimizations, but that's really hard, frankly.</a> <a data-t="391">And the number of people that know how to do that well isn't that great, probably.</a> <a data-t="397">And the number of people who do that and are going to be writing R packages is even smaller.</a></p>

<h3>The power of the Arrow community</h3>

<p><a data-t="404">So here's where the large Arrow community comes in handy here.</a> <a data-t="408">So the Arrow project, since it was created in 2016, has had a steady growth of contributors</a> <a data-t="416">to the project over time, well over 500 by now.</a> <a data-t="421">And what this means is all of these people are working together on this project and the</a> <a data-t="426">benefits are shared.</a> <a data-t="427">So in order to have smart multi-threading in the R package, I didn't have to write that.</a> <a data-t="434">Anyone else who understands how to do that much better than I would did that.</a> <a data-t="438">I didn't have to write SIMD code either.</a> <a data-t="440">There are other people, people who work on hardware that's the actual CPUs themselves</a> <a data-t="446">that know what those options are.</a> <a data-t="449">They wrote that code.</a> <a data-t="450">And they're all contributing it in.</a> <a data-t="451">And they're not contributing it in because they want to make the R package faster.</a> <a data-t="456">They're contributing it in because they are using the C++ library for some other project</a> <a data-t="461">or they're using Python, which uses the same C++ library that the R package does.</a> <a data-t="466">So we benefit from all these other communities' contributions.</a></p>

<p><a data-t="472">So what does that look like?</a> <a data-t="474">So to give an example of how this work plays out in reality, I want to talk about this</a> <a data-t="480">example, this demo that I gave last year at RStudio conference.</a> <a data-t="484">I was introducing the Arrow R package and I was going through this example with the</a> <a data-t="491">data set where you take about 10.5 years of New York City taxi data, 2 billion rows, and</a> <a data-t="500">you could scan that and get some results on that on your laptop.</a> <a data-t="504">So I did some kind of sample </a><a href="/software/dplyr/">dplyr</a><a data-t="504">, filtering and selecting and grouping and aggregating</a> <a data-t="511">on this data set that I'd opened with the Arrow package where I pointed at a directory</a> <a data-t="515">of these files and then I could query it.</a> <a data-t="518">And I got a result in four seconds over 2 billion rows.</a> <a data-t="521">That was pretty good.</a></p>

<p><a data-t="528">I did a talk this past summer and I redid the result just on the latest version of the</a> <a data-t="529">code and it was twice as fast.</a> <a data-t="532">And I hadn't really done anything in the R package to do that.</a> <a data-t="535">This is all based on improvements in the underlying C++ code that all the rest of the community</a> <a data-t="540">was working on.</a> <a data-t="543">Right before I did this talk now, I ran the code again on the latest version and it's</a> <a data-t="547">another 25% faster.</a> <a data-t="549">Again, I didn't do anything to make that happen.</a> <a data-t="553">It's just based on the Arrow community and its work.</a></p>

<h3>Arrow as a universal standard</h3>

<p><a data-t="558">So there's other reasons, other ways that this ecosystem plays out and has benefits</a> <a data-t="564">for us in our community.</a> <a data-t="568">So when you hear that a bunch of database developers got together and decided that there</a> <a data-t="574">wasn't a good standard for columnar data and they were going to create their own standard</a> <a data-t="580">to unify all of these, it might bring this XKCD comic to mind where the creation of standards</a> <a data-t="588">to solve the lack of standards just makes more problems, more standards, more competing</a> <a data-t="593">standards.</a> <a data-t="594">And there certainly is that risk.</a> <a data-t="596">But five years on from the creation of Arrow, we can see now how the benefits of this approach</a> <a data-t="604">have played out.</a></p>

<p><a data-t="607">So just to give an example, so suppose I've got data in Spark and I want really quick</a> <a data-t="615">access to that in Python and R. What you could do is that you could from Spark, you could</a> <a data-t="621">write a special adapter for Python, a special adapter for R that understands, in R's case,</a> <a data-t="630">you understand R's vector types and specific, the bits in the vectors that make up a data</a> <a data-t="637">frame.</a> <a data-t="638">And you have to do the same thing for Python, for NumPy or Pandas.</a> <a data-t="643">And Spark and its Java code would have to understand how to do this.</a> <a data-t="648">I put another arrow in the chart there to connect the interchange between Python and R.</a> <a data-t="652">That's obviously something you'd want to do too.</a> <a data-t="654">And it's another place where you'd need someone on the other side of the language barrier</a> <a data-t="658">to understand the internals of your format.</a></p>

<p><a data-t="662">So that's feasible if this is all you care about.</a> <a data-t="665">But in reality, the system is, there's many more languages and there's many more projects</a> <a data-t="669">and too many to fit on the slide.</a> <a data-t="673">And as you can imagine, if you try to draw all of those lines to connect the various</a> <a data-t="677">adapters to the different things, it gets crazy really quickly.</a> <a data-t="681">So what do you do?</a> <a data-t="682">You're obviously not going to do that.</a> <a data-t="684">That's too much work.</a> <a data-t="687">And so what you want is to have a ubiquitous standard that everyone can write to and read</a> <a data-t="692">from and then you don't have to re-implement everything every time.</a></p>

<p><a data-t="699">So prior to Arrow, what you do is probably just dump out a CSV.</a> <a data-t="706">Every language knows how to read a CSV.</a> <a data-t="708">Every database knows how to read in a CSV.</a> <a data-t="711">But this comes with some costs.</a> <a data-t="714">CSVs can be expensive to read and write because you have to convert between a string format</a> <a data-t="720">on disk to the arrays of bits that R and other languages use in memory to do work.</a> <a data-t="730">And that conversion is costly.</a> <a data-t="732">And CSV also doesn't have rich support for types.</a> <a data-t="735">You may have timestamps and you don't have any way of indicating that this is a timestamp.</a> <a data-t="740">You have to guess whether it should be just a regular string or a timestamp.</a> <a data-t="744">And so there's lots of costs and penalties that come from doing this.</a> <a data-t="748">But CSV is kind of the lowest common denominator and so that's why it gets picked up.</a></p>

<p><a data-t="753">But Arrow is kind of a happier version of that where you have a standard and it's columnar</a> <a data-t="760">and it's binary.</a> <a data-t="761">So it matches very closely with how R or Julia or Spark or any of these languages actually</a> <a data-t="768">work with data in memory.</a> <a data-t="770">So the conversion is minimal and in some cases doesn't actually involve copying memory because</a> <a data-t="776">it where it does actually line up as being the same shape.</a> <a data-t="780">So it's very efficient.</a> <a data-t="782">And from our perspective in the R community, what this means is once we have and now that</a> <a data-t="787">we have an Arrow R package, we have access to all of these other projects and databases</a> <a data-t="793">and languages very efficiently as long as they also can write to Arrow.</a> <a data-t="799">And over the last five years, we've seen an increasing number of projects that have picked</a> <a data-t="802">up on Arrow for this because it's really efficient.</a></p>

<p><a data-t="816">So going back to Spark for our example there, early on in Arrow's life, we added support</a> <a data-t="824">in PySpark, the Python Spark library, to communicate with Spark the database which is written in</a> <a data-t="831">Java very efficiently using Arrow.</a> <a data-t="833">And so you'd get a hundredfold speed up on certain operations whether it's with user</a> <a data-t="833">defined functions or whether you're just pulling a large amount of data into your Python environment</a> <a data-t="839">to do further analysis on it.</a> <a data-t="842">And that worked because Java has an Arrow library, Python has an Arrow library, and</a> <a data-t="847">they write the same format.</a> <a data-t="849">The beauty of this is that once we finally got the R package going, we could pick up</a> <a data-t="854">and do the same thing.</a> <a data-t="856">These are both references to blog posts on the Arrow website.</a> <a data-t="859">So we were able to take advantage of the same thing.</a> <a data-t="863">And because we can read and write Arrow from R, we didn't need to do any other special</a> <a data-t="869">Spark business in order to get those 100x speed ups of pulling data to and from Spark</a> <a data-t="878">in R.</a></p>
