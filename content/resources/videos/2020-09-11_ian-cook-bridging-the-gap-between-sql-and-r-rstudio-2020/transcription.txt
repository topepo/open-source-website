<h3>SQL and <a href="/software/dplyr/">dplyr</a>: two sides of the same coin</h3>

<p><a data-t="137">So if you know some SQL and you know R and the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="137">, one of the things you'll notice is that SQL and dplyr are really like two implementations of the same abstract concept.</a> <a data-t="153">And they have a whole lot in common.</a></p>

<p><a data-t="156">First of all, a table in SQL is essentially equivalent to the concept of a tidy data frame in R. So each variable forms a column.</a> <a data-t="165">Each observation forms a row.</a> <a data-t="167">Each type of observational unit forms a table.</a> <a data-t="171">That's what we call tidy data in R. But in the world of SQL, that is really the essence of Edgar F. Codd's third normal form, which is the most widely used way of structuring the tables in a database.</a></p>

<p><a data-t="215">And once you have your data in this kind of structure, whether you call it tidy or third normal form, the types of operations you can apply to the data are basically the same in dplyr as in SQL.</a></p>

<p><a data-t="227">So dplyr, as I'm sure most of us know, provides a set of functions, so-called verbs.</a> <a data-t="233">And the idea is that you can use just a few of these to do most of the kinds of data manipulation that you'd ever need to do.</a> <a data-t="244">So when Hadley created dplyr, essentially what he did was he took the semantics of the SQL SELECT statement and he brought it into R.</a> <a data-t="260">Of course, the syntax is different, and Hadley did depart from SQL in a few notable ways.</a> <a data-t="263">But the similarities are really striking.</a></p>

<p><a data-t="266">So if you're familiar with SQL, then you'll recall the clauses of the SQL SELECT statement.</a> <a data-t="271">SELECT FROM, WHERE, GROUP BY, HAVING, ORDER BY, and LIMIT are the main clauses.</a> <a data-t="279">And each of these maps onto a dplyr verb.</a> <a data-t="282">But it's not exactly a clean one-to-one mapping.</a></p>

<p><a data-t="293">The FROM clause in SQL, I'll start there, just specifies what table to use.</a> <a data-t="298">So in dplyr, you do that by specifying just the name of a data frame.</a> <a data-t="255">The SELECT statement in SQL has several different roles.</a> <a data-t="261">So it corresponds to different dplyr verbs, depending on what you're trying to do.</a> <a data-t="261">If you're just grabbing existing columns, then that's a SELECT in dplyr.</a> <a data-t="266">If you're using expressions to define new columns, that's MUTATE or TRANSMUTE.</a> <a data-t="272">And if you're using aggregate functions, in that case, SELECT corresponds to SUMMARIZE in dplyr.</a></p>

<p><a data-t="279">The WHERE clause in a SQL SELECT statement corresponds to FILTER.</a> <a data-t="284">GROUP BY corresponds very nicely to GROUP BY.</a> <a data-t="289">HAVING in SQL, that's the clause that you use to filter by groups after aggregating.</a> <a data-t="295">And in dplyr, you would use FILTER for that, but you'd use it after GROUP BY.</a> <a data-t="302">And then ORDER BY, that corresponds to ARRANGE in dplyr.</a> <a data-t="306">And finally, LIMIT, that's just HEAD or SLICE, if you prefer.</a></p>

<p><a data-t="315">And incidentally, you know, it's not just dplyr and SQL that have this same underlying abstraction layer.</a> <a data-t="320">This has proven to be such a successful one that it's been widely adopted.</a> <a data-t="325">Pandas and Python, Spark in the big data world, many, many more tools for working with data all follow the same semantics to varying degrees.</a></p>

<h3><a href="/software/dbplyr/">dbplyr</a>: translating dplyr to SQL</h3>

<p><a data-t="340">Anyway, one of the great things about this common abstraction underlying both SQL and dplyr is that it's enabled dplyr code to be translated into SQL, allowing you to use dplyr to query databases.</a> <a data-t="349">And this is enabled by the package dbplyr.</a></p>

<p><a data-t="353">With dbplyr, you can connect to a database, refer to a specific table in the database, and then chain on dplyr verbs, just as if you were working with an in-memory R dataframe.</a> <a data-t="363">And what dbplyr does is it translates all those dplyr operations into a SQL SELECT statement, sometimes with the help of a custom backend package, like, specific to that, you know, SQL engine.</a> <a data-t="379">And then finally, it uses the </a><a href="/software/dbi/">DBI</a><a data-t="379"> package to run that SQL SELECT statement on the database and return the result back to R.</a></p>

<p><a data-t="414">So you first define a variable.</a> <a data-t="416">It's effectively just a pointer to a database table.</a> <a data-t="421">So in this example, this is just creating a small in-memory SQLite database, and then it's copying the iris dataframe to it.</a> <a data-t="429">Then you can use dplyr on that table in the SQL engine, just as if it were an in-memory dataframe.</a> <a data-t="435">And if you're ever curious about what SQL code your dplyr code is being translated into, dbplyr has a handy function, show query, that will show you the SQL that it's generating.</a></p>

<h3>The idea: translating SQL to dplyr</h3>

<p><a data-t="432">So I've been using dbplyr for several years.</a> <a data-t="436">I've made a few small contributions to it, and I developed a custom backend package for it.</a> <a data-t="437">But I've always wondered, like, what about going in the other direction?</a> <a data-t="440">What about translating SQL to dplyr code?</a> <a data-t="444">I always thought it would be really cool to have this.</a> <a data-t="447">It could enable you to use SQL statements to query dataframes in R. And I searched all around, and there was no existing package that did quite this, SQL to dplyr translation.</a></p>

<p><a data-t="458">Well, there's a clear answer to this question, evidenced by the popularity of another R package, sqldf.</a> <a data-t="479">So this is a well-established package.</a> <a data-t="482">It's very popular, created by this guy, Gabor Grotendijk.</a> <a data-t="486">And it lets you run SQL queries on R dataframes.</a> <a data-t="490">But to do this, it does not parse your SQL statement.</a> <a data-t="493">Instead, it uses a clever trick.</a> <a data-t="495">The package includes an embedded database.</a> <a data-t="499">And it loads your R dataframe into the database, and then runs SQL queries on the database, and then returns the result as an R dataframe.</a></p>

<p><a data-t="508">So here's an example of sqldf at work.</a> <a data-t="510">The main function has the same name as the package, sqldf.</a> <a data-t="514">And you pass it just one argument, which is a SQL statement enclosed in quotes.</a> <a data-t="518">And the one part of this SQL statement that it does look at is the from clause to identify which dataframes, you know, you're referring to, and it loads those into the database.</a> <a data-t="528">But it doesn't do any further parsing of the SQL statement.</a> <a data-t="531">It just passes it over to the database to run.</a></p>

<p><a data-t="535">I think sqldf is really cool.</a> <a data-t="537">It eliminates an obstacle for users who are proficient in SQL and not so proficient in dplyr, and who just need to get some data manipulation done in R.</a> <a data-t="550">So I work as a data science and machine learning educator at Cloudera.</a> <a data-t="554">And I encounter a lot of learners who just love this, they eat it up.</a> <a data-t="558">They typically have some experience with SQL, and they're really excited that they can reuse those skills inside R to do the kind of boring data manipulation piece, and they can move on to the more exciting stuff, like machine learning and </a><a href="/software/ggplot2/">ggplot2</a><a data-t="558"> and shiny apps, which were, you know, the reasons they were excited to learn R in the first place.</a></p>

<p><a data-t="577">But there are a couple of fundamental limitations to sqldf, just inherent to the way it works, that it's important to recognize.</a> <a data-t="583">Like I said, it works by copying data out of R into a database.</a> <a data-t="586">That's okay for smaller, maybe medium-sized data.</a> <a data-t="590">It does introduce a bit of latency on the first query for a particular data frame, because it has to copy it.</a> <a data-t="597">However, with bigger data, it could be really impractical to make a copy like this.</a> <a data-t="604">Just like architecturally, copying data to a whole different system like this seems like a bit of an anti-pattern.</a></p>

<p><a data-t="620">Second thing, what sqldf provides feels less like a bridge from SQL to R and more like a dead end.</a> <a data-t="625">I mean, you could just, you know, refuse to learn dplyr and just do all your data manipulation in R using SQL.</a> <a data-t="632">I mean, I think it's important to learn dplyr at some point.</a></p>

<h3>Building queryparser</h3>

<p><a data-t="637">So a few months ago, I set out to create something that doesn't have these shortcomings, and it was immediately apparent to me that this problem breaks down into two separate pieces.</a> <a data-t="651">One, parsing a SQL query into some kind of abstract representation of the data manipulation specified in the query.</a> <a data-t="659">And then two, actually applying those data manipulation steps to the data in R data frames.</a></p>

<p><a data-t="660">So the first challenge then was to create a SQL query parser for R. I mean, how hard could that be?</a> <a data-t="671">It turns out it's pretty hard.</a> <a data-t="673">So to achieve it, I ended up building kind of a Rube Goldberg machine.</a></p>

<p><a data-t="674">I created this package called QueryParser, and it's on CRAN.</a> <a data-t="677">The main function in QueryParser is named parseQuery.</a> <a data-t="681">It takes a quoted SQL SELECT statement as input, and what it returns is a list with sublists that are named according to the structure of the SELECT statement.</a> <a data-t="691">So you can see in this example, using the iris table, the SQL statement has a SELECT clause, a FROM clause, and a GROUP BY clause, and the list returned by parseQuery has sublists that are named the same.</a></p>

<p><a data-t="705">And then the items within the sublists, what they are are unevaluated R expressions.</a> <a data-t="709">In this example, see how there are two expressions in the SELECT list in the SQL query?</a> <a data-t="713">One returns the species column, one computes the average of the sepal length column for each species.</a> <a data-t="718">And these have been translated from SQL to R, and they're each returned within the SELECT sublist as an unevaluated R expression.</a></p>

<p><a data-t="731">So I'm really proud of a few things about how QueryParser works.</a> <a data-t="737">One, it is implemented entirely as R code, with no imports, no depends, no compiled code.</a> <a data-t="742">It's just R code calling base R functions.</a> <a data-t="744">And this makes it easy to maintain.</a> <a data-t="745">No dependencies are gonna break over time.</a></p>

<p><a data-t="753">However, there is an option in this function to use functions from several tidyverse packages in the translations that it outputs.</a> <a data-t="762">This option defaults to false, but you can set it to true.</a> <a data-t="764">So in this example, for instance, COUNTSTAR translates to the dplyr function n to count the number of rows, which is handy in many cases.</a></p>

<p><a data-t="773">QueryParser performs semantic translations, not literal translations.</a> <a data-t="778">And a good example of this is how it translates the SQL aggregate expression SUM of 1.</a> <a data-t="782">This is commonly used by folks in the SQL community just to count rows.</a> <a data-t="785">It does the same thing as COUNTSTAR.</a> <a data-t="787">But the SUM function in R doesn't work like that.</a> <a data-t="792">So QueryParser translates this to the dplyr n function.</a></p>

<p><a data-t="799">Also, the parseQuery function checks all the expressions in all the clauses of the query that you pass to it against a list of allowed functions and operators.</a> <a data-t="807">And it throws an error like this if there's a function or operator that's not on that list.</a> <a data-t="810">And the reason I did this is because I'm imagining that someone will eventually build this into a </a><a href="/software/shiny-r/">shiny</a><a data-t="810"> app where there's a box that allows users to input a SQL query.</a> <a data-t="821">And in this case, QueryParser would allow any malicious inputs from getting through.</a></p>

<p><a data-t="824">So the details about how all this works are enough to fill a whole different talk.</a> <a data-t="829">But in the simplest sense, it kind of goes like this.</a> <a data-t="832">First, it splits the query into clauses.</a> <a data-t="837">Then it splits the clauses into strings containing different SQL expressions.</a> <a data-t="843">And then it manipulates the strings to change them from SQL expressions to R-ish expressions.</a> <a data-t="847">Things that R will recognize as R code.</a></p>

<p><a data-t="851">This is the most laborious and unsatisfying part of the whole process.</a> <a data-t="856">The code in QueryParser needs to step through every single expression, character by character, keeping track of when it's inside quotes, when it's inside parentheses, looking for different kinds of SQL syntax and replacing them with the equivalent R syntax.</a> <a data-t="868">It uses this thing called raw connections inside base R to do that.</a> <a data-t="872">It's not particularly elegant, but it works.</a></p>

<p><a data-t="880">The next step is it takes those R-like expressions and it parses them using str2lang, which creates an unevaluated R expression from a string containing R code.</a> <a data-t="889">And now this is the step that is satisfying.</a> <a data-t="894">After that, what happens is this magical combination of quoting, substitution, unquoting, deparsing, string manipulation, reparsing.</a> <a data-t="902">Some of this is what you might call quasi-quotation in an </a><a href="/software/rlang/">Rlang</a><a data-t="902"> context.</a> <a data-t="907">But here it's all done with base R. And this is the part that I'd love to do a whole talk about, because it's sort of fascinating how it all works.</a></p>

<p><a data-t="915">After that, I have to do a few more manipulations of the abstract syntax tree to make everything work exactly right.</a> <a data-t="927">Then I perform some basic validity checks and then return that list that you saw in the previous slide.</a> <a data-t="935">So all this takes about only 100 milliseconds for a simple query.</a> <a data-t="940">Maybe like 200 or 300 milliseconds for a more complex query.</a> <a data-t="943">It could be much faster, for sure.</a> <a data-t="947">But to the end user, just parsing one query, it feels instantaneous.</a></p>

<h3>Introducing tidyquery</h3>

<p><a data-t="957">So moving on to the second piece, tidyquery.</a> <a data-t="961">This is also on CRAN.</a> <a data-t="963">And this is the package that most end users will actually use.</a> <a data-t="969">The main function in tidyquery is named query.</a> <a data-t="973">It takes a quoted select statement as input, and it uses the from clause to identify which data frame you want to query.</a> <a data-t="979">And then it uses the rest of the select statement to establish how to manipulate that data frame, performs those manipulations, and returns a new data frame as the result.</a> <a data-t="988">So basically, if you're familiar with SQLDF, then you use tidyquery just the same way.</a> <a data-t="993">But of course, it works differently under the hood, as I'll describe.</a></p>

<p><a data-t="995">But first, there's another option.</a> <a data-t="1001">You can also use this query function like a dplyr verb in a chain with other dplyr verbs above it and below it.</a> <a data-t="1006">The thing is, you just have to leave out the from clause.</a> <a data-t="1013">Because then the data that's passed in, piped in above, becomes the from clause.</a></p>

<p><a data-t="1019">So despite the syntax of this being similar to SQLDF, under the hood, this query function in tidyquery works totally differently.</a> <a data-t="1026">The first thing it does is it calls the parse query function in query parser.</a> <a data-t="1031">Then getting back that list, it determines which sequence of dplyr verbs it needs to call to execute this query.</a> <a data-t="1040">And then it uses dplyr with the help of rlang to evaluate the expressions in the context of a data frame.</a></p>

<p><a data-t="1057">So if you remember the earlier slide about dbplyr, essentially, tidyquery is like the inverse of dbplyr, which Hadley was quick to note.</a></p>

<p><a data-t="1052">Currently query parser and tidyquery work for the most common types of SQL queries.</a> <a data-t="1054">They have some limitations.</a> <a data-t="1057">I'll not go through these point by point in the interest of time.</a> <a data-t="1060">But I do have plans to implement all of this.</a> <a data-t="1062">But I don't have any specific time frames yet.</a> <a data-t="1067">If you feel strongly that you'd like to see one of these features implemented sooner rather than later, let me know.</a> <a data-t="1073">And I've talked to several people at the conference who voted for subqueries.</a></p>

<p><a data-t="1101">So one of the things I'm really excited about is that tidyquery doesn't work only on R dataframes.</a> <a data-t="1104">It can work on any data object that dbplyr can work on.</a> <a data-t="1107">So for example, data.table.</a> <a data-t="1113">With Hadley's recent overhaul of the </a><a href="/software/dtplyr/">dtplyr</a><a data-t="1113"> package, you can use dplyr code with data.table and it uses lazy evaluation to run efficiently on data.table.</a> <a data-t="1120">So tidyquery preserves that lazy evaluation.</a></p>

<p><a data-t="1129">So basically any time dplyr works with some data source, tidyquery can work with it.</a> <a data-t="1137">Maybe I might have to make a couple little changes to recognize the class name.</a> <a data-t="1139">But it'll all work.</a> <a data-t="1139">And it won't move the data.</a> <a data-t="1141">It'll query it in place.</a> <a data-t="1146">I'm excited to try this with the new version of the Apache arrow R package that Neil Richardson spoke about yesterday, which now supports dplyr verbs.</a> <a data-t="1151">I haven't tried it, but I'll do that soon.</a></p>
