<h3>What Connect does</h3>

<p><a data-t="191">So to begin, let's start by talking about Connect.</a> <a data-t="191">What is Connect?</a> <a data-t="191">Connect, I think, I like to try to define things by what they do.</a> <a data-t="203">And what Connect does is it drives down the cost of application publishing.</a> <a data-t="208">I think people at this conference are probably pretty used to the idea that data scientists are people who in the course of doing their work builds web applications.</a> <a data-t="228">But that's not a typical view that most people hold about one of the things that data science is.</a> <a data-t="234">And so part of what Connect does is it makes it easy for data scientists to get work off their machines by reducing the demands on what it asks them for when they're trying to publish something.</a></p>

<p><a data-t="234">So Connect's deployments do lots of work to inspect your local environment, gather up some information about what you have there, and then send that to the server where Connect reconstructs the environment for you.</a> <a data-t="252">And Connect servers, and I have one of those architecture diagrams that James pulled up before, a single Connect server can run multiple versions of R, multiple versions of Python, multiple versions of </a><a href="/software/quarto/">Quarto</a><a data-t="252">.</a> <a data-t="264">It can handle conflicting versions of packages for each of those runtimes.</a> <a data-t="270">And it deploys the applications in an isolated and atomic way.</a> <a data-t="274">So it makes it, again, it's trivial to publish a new thing.</a> <a data-t="274">If you break it, you just publish again.</a> <a data-t="280">You roll forward, you iterate, you make progress.</a> <a data-t="280">And for most, many workflows, that's good enough.</a></p>

<h3>When more complexity is required</h3>

<p><a data-t="289">There are, however, circumstances where more complexity is required.</a> <a data-t="296">That might be because you're working with packages that are for some reason difficult to install.</a> <a data-t="302">They might have complicated dependencies that are difficult for something running at the application layer to reconstruct.</a> <a data-t="307">There might be conflicting package dependencies.</a> <a data-t="313">You might have data that's stored somewhere that demands something else about the way you access that data.</a> <a data-t="318">So the Linux file system is no longer enough.</a> <a data-t="325">And so for all of those reasons, we've put a lot of work into making it possible to run Connect in Kubernetes.</a></p>

<p><a data-t="335">We've decoupled where Connect runs and where applications run.</a> <a data-t="342">Applications sit in their own container context, and Connect is responsible for spawning them and managing some other things.</a> <a data-t="347">And we call this mode off-host execution.</a> <a data-t="353">And so what I want to do now is I want to show you around some of the Connect interfaces that reflect this change so that you can understand what kinds of things are possible when Connect is running in this mode.</a></p>

<h3>Connect interfaces for off-host execution</h3>

<p><a data-t="363">So I'll jump back to our shiny application here.</a> <a data-t="368">And if you look at the right side, you'll see that below the settings where we can control the access to the application or the sub-path we serve it on, we have some additional configuration settings.</a> <a data-t="400">And so in addition to controlling the, for example, the user executing the content, we might also provide a specific Kubernetes service account, which would have permissions to access particular data sources in a way that isn't going to sort of correspond to how we do that in the Linux file system.</a> <a data-t="436">And that's going to make it easier to provide that fine-grained permission.</a></p>

<p><a data-t="445">The other new sort of option you're seeing here is the ability to configure execution environments on a per-deployment basis.</a> <a data-t="453">And so what you're seeing here is I pushed a container to the GitHub container registry that just has a version of Python in it, but I can change that environment to be something more complicated.</a> <a data-t="425">It might be the case that, for example, I have an image that has R and Python in it.</a> <a data-t="430">And if I'm thinking about making changes to my application, I can select a different image for subsequent execution.</a> <a data-t="440">And I can also select that as a default environment image.</a></p>

<p><a data-t="445">If I come back to the Connect home page, you'll see there's an additional tab that explains the execution environments that are available.</a> <a data-t="451">And so in addition to the default images that Connect ships with in the Helm chart I used to provision this instance, I've added this Python 3.10 image and also an image that has a version of Quarto 1.4 in it.</a> <a data-t="467">I am extremely excited about Quarto 1.4.</a> <a data-t="472">And if you want to understand why, you should go to </a><a href="/people/carlos-scheidegger/">Carlos Scheidegger</a><a data-t="472">'s talk.</a></p>

<p><a data-t="481">But that's not all, right?</a> <a data-t="481">So in addition to wanting to, say, constrain the environment that we're constructing as Connect is being built, one of the other things we might need is the ability to control the resources that are being consumed by the application while it runs.</a> <a data-t="501">And so in addition to memory limits, which you can apply now in both local and off-host execution mode, you can also apply CPU limits.</a> <a data-t="507">This means that if you have applications that have a way of spinning out, taking down servers, and forcing you to buy people beers, you can put a little bit more guardrails around that in a way that's going to make your application safer to use.</a></p>

<p><a data-t="522">And then the last thing, and this is one of the newest pieces of functionality that we've introduced to Connect, is the ability to bypass certain elements of environment reconstruction.</a> <a data-t="533">You can see here on both the R environment and Python environment management, we have the option to select a setting which gives us control over who has responsibility for environment construction.</a> <a data-t="546">So I pushed a container that has just a Python image in it, and so I allowed Connect to build the content in my container.</a> <a data-t="557">It's also the case that I could construct an environment that has all the dependencies that my application requires, and then push that container to Connect, and Connect doesn't have to actually do that environment restoration step.</a> <a data-t="569">And so that's giving you, again, more fine-grained control over what exactly is executing and where.</a></p>
