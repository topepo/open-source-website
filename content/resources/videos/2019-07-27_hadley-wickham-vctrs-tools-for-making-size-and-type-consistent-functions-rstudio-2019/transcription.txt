<h3>The rules for combining types</h3>

<p><a data-t="200">And so kind of the motivation for this vectors package was to think about, well, like, what</a> <a data-t="205">should we do?</a> <a data-t="206">This is clearly crazy what currently happens.</a> <a data-t="209">How can we do better?</a> <a data-t="211">What should we do?</a> <a data-t="213">And I think we can get some hint of that by looking at the rules for C, for concatenate,</a> <a data-t="219">for atomic vectors, because the rules here are, like, pretty simple and pretty reasonable.</a></p>

<h3>Type stability</h3>

<p><a data-t="750">So the underlying idea behind vectors is this idea of type stability, which I'm still trying</a> <a data-t="757">to figure out how to express, to articulate as clearly as possible.</a> <a data-t="761">But I think there's this principle.</a> <a data-t="763">The way I sort of think about it is, like, when you're reading R code, what you are actually</a> <a data-t="768">doing is running that R code in your internal mental model of R.</a> <a data-t="773">And obviously, your internal mental model of R is much, much simpler than the real thing.</a> <a data-t="779">But I think in your internal mental model of R, one thing that's really useful to think</a> <a data-t="783">about is the type of each variable.</a> <a data-t="786">And I notice that when I read other people's code, when I'm reviewing pull requests, for</a> <a data-t="790">example, if I read the code and I look at a variable and I have no idea what type of</a> <a data-t="795">variable that is, I don't know if it's a list or a character or a data frame, that code</a> <a data-t="800">just feels very dangerous to me, because I cannot accurately predict what's going to</a> <a data-t="804">happen.</a></p>

<p><a data-t="805">So the first principle of type stability is that the output type should only depend on</a> <a data-t="810">the types of the inputs.</a> <a data-t="813">And so one kind of violation of this in base R is the if else function.</a> <a data-t="818">So here I've got three calls to if else.</a> <a data-t="821">The first argument is always logical.</a> <a data-t="823">The second argument is always an integer.</a> <a data-t="826">The third argument is always a character vector.</a> <a data-t="829">And yet, depending on the value of the logical vector, the output could be a logical vector,</a> <a data-t="835">it could be a double vector, it could be a character vector.</a> <a data-t="840">And I think this is an undesirable property, because when you read code that involves if</a> <a data-t="844">else, it makes it hard to know what type of output you're going to get.</a></p>

<p><a data-t="849">And again, I don't want to say, like, the tidyverse is perfect, it's far from perfect.</a> <a data-t="853">Deply if else actually is equally annoying, but in the opposite direction, it's too strict,</a> <a data-t="860">which forces you, if you want to insert a missing value, to know about all of these</a> <a data-t="863">different types of missing values that you should never be forced to confront, and it's</a> <a data-t="868">pretty annoying.</a></p>

<p><a data-t="870">The second principle is that if you're just combining a bunch of vectors or your argument</a> <a data-t="875">has dot, dot, dot in it, your function has dot, dot, dot in it, the order of the inputs</a> <a data-t="879">should not affect the output type.</a> <a data-t="882">So I kind of showed you an example of this already, lots of examples of this with C,</a> <a data-t="886">but ideally, the order in which you supply arguments to C should not affect the output.</a> <a data-t="893">This is the idea that gave us that symmetry in the table.</a> <a data-t="897">This makes predicting the output much, much, much simpler, because there's just a much</a> <a data-t="901">smaller set of rules that you need to remember and internalize.</a></p>

<p><a data-t="905">And again, the goal here is not that you should ever have to explicitly memorize the rules,</a> <a data-t="911">but hopefully by having this small set of reasonable and consistent rules, you will</a> <a data-t="915">just come to learn them over time, and when you predict, when you guess, your naive guess</a> <a data-t="922">about what a function will return should just be more accurate from the get-go.</a></p>

<p><a data-t="929">And then the final principle is almost so obvious that you shouldn't have to say it,</a> <a data-t="936">and that is ideally, there should be one set of rules that is applied everywhere.</a> <a data-t="941">So again, just to illustrate this, if you concatenate two factors together, you get</a> <a data-t="946">an integer vector, but if you put those factors together in a list and then you unlist them,</a> <a data-t="952">you will get a factor as you expected.</a> <a data-t="955">So the fact that we have these inconsistencies just makes it hard to build up that accurate</a> <a data-t="960">mental model, and it gets confusing.</a></p>

<p><a data-t="963">And I think there's sort of both this first-order effect that when you look at some code, you're</a> <a data-t="969">like, oh, I don't know what this is going to return, but you also get the second-order</a> <a data-t="972">effect that's even worse, that over time you sort of get this learned helplessness.</a> <a data-t="977">You're like, I have no idea, I've been burned so many times before, I've got no idea what</a> <a data-t="981">this is going to return, so I'm just not even going to try to guess, which I think</a> <a data-t="985">is even more harmful.</a></p>

<blockquote><a data-t="977">You're like, I have no idea, I've been burned so many times before, I've got no idea what</a> <a data-t="981">this is going to return, so I'm just not even going to try to guess, which I think</a> <a data-t="985">is even more harmful.</a></blockquote>
