<h3>Things to remember and future ideas</h3>

<p><a data-t="848">So some things to remember for the reactive coding.</a> <a data-t="852">Any outputs that are not visible are not calculated.</a> <a data-t="852">This is on purpose.</a> <a data-t="853">This is a direct design decision by Shiny, so you may see some output elements that just</a> <a data-t="859">hang out invalidated forever.</a> <a data-t="861">They're not on display.</a> <a data-t="863">So you may need to open that second tab or do something else with that.</a></p>

<p><a data-t="929">If new reactive elements are created for every user interaction, such as a click or an input</a> <a data-t="935">value change, you've probably coded what's referred to as an anti-solution.</a> <a data-t="941">Joe talks about this in his Shiny DevCon presentation.</a> <a data-t="946">And what it will do with reactlog is it will cause the reactlog to explode.</a> <a data-t="951">It'll get very big because more reactives will be done for every user interaction.</a> <a data-t="956">So if this happens, it's okay.</a> <a data-t="958">Your Shiny app isn't exploding on memory, I promise.</a> <a data-t="961">There's garbage collection going on, but the graph will be quite big right now.</a> <a data-t="967">And then reactlog is not a performance debugger.</a> <a data-t="970">It's a reactivity debugger.</a> <a data-t="972">If you want performance debugging or inspection, please use </a><a href="/software/profvis/">Profvis</a><a data-t="972">.</a> <a data-t="976">It's very good at that and it will tell you very quickly where to change your code.</a></p>
