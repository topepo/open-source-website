<h3>What is querychat?</h3>

<p><a data-t="96">So, what exactly is querychat?</a> <a data-t="96">Well, querychat makes it easy to query data using</a> <a data-t="103">natural language.</a> <a data-t="103">It offers a drop-in component for multiple popular frameworks,</a> <a data-t="109">including </a><a href="/software/shiny-r/">Shiny</a><a data-t="109">, which we'll use here in this video.</a> <a data-t="109">It also offers a console interface and</a> <a data-t="114">other programmatic building blocks.</a> <a data-t="121">You ask questions, querychat translates it to SQL,</a> <a data-t="121">executes it, and returns the results.</a> <a data-t="128">The results are then available as a reactive data frame,</a> <a data-t="128">which makes it easy to display or further process the data.</a> <a data-t="134">There's no clicking, no predefined</a> <a data-t="134">filters.</a> <a data-t="140">It's just you, your data set, and, you know, questions like, show me the matches where</a> <a data-t="140">the Netherlands played in the World Cup.</a> <a data-t="147">Or can you filter the matches from the 2025 UEFA Euro?</a> <a data-t="152">Or which team has the best win rate in the Canada versus United States matchup?</a></p>

<h3>How querychat works</h3>

<p><a data-t="381">You could think about querychat as, you know, just a chat window sitting on top of a dataset, but</a> <a data-t="386">it's more.</a> <a data-t="386">With querychat, you never have to think about how to get to an answer.</a> <a data-t="393">You don't have to</a> <a data-t="393">think about what filters to apply or how to extract information from the filter data.</a> <a data-t="401">No, you simply</a> <a data-t="401">ask a question, and querychat just handles the rest.</a> <a data-t="407">And okay, okay, I hear you thinking.</a> <a data-t="407">You</a> <a data-t="407">could also ask all those questions in a typical LLM chat tool.</a> <a data-t="413">Yeah.</a> <a data-t="413">But querychat is a bit</a> <a data-t="413">different.</a> <a data-t="419">You're not relying on the LLM to invent answers or reason about the data internally.</a> <a data-t="419">No,</a> <a data-t="419">with querychat, every question is translated into SQL.</a> <a data-t="426">The SQL is then executed on the actual</a> <a data-t="426">dataset and returned exactly as the data dictates.</a> <a data-t="433">And also very important, the SQL is always shown.</a> <a data-t="433">This means you can see precisely what is being executed.</a></p>

<p><a data-t="442">So there are four benefits of querychat.</a> <a data-t="442">One, reliability.</a> <a data-t="450">The LLM does not touch the raw data itself.</a> <a data-t="450">It doesn't analyze or, you know,</a> <a data-t="450">transform it.</a> <a data-t="458">It only generates SQL.</a> <a data-t="458">querychat handles the execution of that SQL via tool calling,</a> <a data-t="458">and all results come from the real data engine and not from the model's internal guesswork.</a> <a data-t="464">Two, transparency.</a> <a data-t="471">Every run reveals the full SQL statement.</a> <a data-t="471">Nothing is hidden, nothing is</a> <a data-t="471">adjusted, and you always know how the answer was actually produced.</a> <a data-t="479">Three, reproducibility.</a> <a data-t="479">Since every SQL query is visible, analysis can be reused, shared, and audited.</a> <a data-t="485">And lastly,</a> <a data-t="485">four, safety.</a> <a data-t="492">querychat's tools are designed with read-only actions in mind, meaning the LLM</a> <a data-t="492">is essentially unable to perform any destructive actions.</a></p>

<p><a data-t="500">Great.</a> <a data-t="500">Now, I mentioned querychat handles</a> <a data-t="500">execution of the SQL via tool calling.</a> <a data-t="508">Tool calling is essentially a bridge between an LLM</a> <a data-t="508">and your Python session.</a> <a data-t="514">The model itself does not execute code.</a> <a data-t="514">Instead, it requests your Python</a> <a data-t="514">session to execute a certain function with certain inputs.</a> <a data-t="522">Once Python gets the results, the result</a> <a data-t="522">is passed back to the model for interpretation.</a> <a data-t="528">But, you know, you can't leave the LLM completely</a> <a data-t="528">clueless and in the dark.</a> <a data-t="533">The LLM does need to know something about your data, of course.</a> <a data-t="540">For example, the columns, their meaning, their type.</a> <a data-t="540">Now, this schema information is shared with</a> <a data-t="540">the model, but not the raw data itself.</a> <a data-t="546">With this schema information, it produces a SQL query as a</a> <a data-t="546">tool call.</a> <a data-t="552">Now, running SQL requires a database engine, and querychat's weapon of choice is DuckDB.</a> <a data-t="554">If you have your own database, that's no problem, too.</a> <a data-t="554">You can also bring your own.</a> <a data-t="558">If your database is SQLAlchemy supported, it can be used by querychat.</a> <a data-t="558">That's pretty smart, right?</a></p>

<h3>Safety and enterprise use</h3>

<p><a data-t="888">In just 10 minutes, you learned how to use querychat to build your next application,</a> <a data-t="892">but before you take off, I want to come back to one important aspect of querychat, though,</a> <a data-t="898">and that is safety.</a> <a data-t="898">Because working with LLMs, you'll always wonder, is this safe?</a> <a data-t="904">It's a fair</a> <a data-t="904">question, but luckily, querychat is designed entirely around control.</a> <a data-t="912">The LLM never executes</a> <a data-t="912">anything itself.</a> <a data-t="918">It never touches your data or your database, and it never sees your raw data.</a> <a data-t="918">It only sees the schemas and the extra information that you provide yourself.</a> <a data-t="925">Its only job is to</a> <a data-t="925">propose read-only SQL.</a> <a data-t="932">If you ask to edit the data or to drop tables, it will not do it.</a> <a data-t="932">It's</a> <a data-t="932">instructed to do so.</a> <a data-t="938">Now, combine that with an underlying database, like could be DuckDB or</a> <a data-t="938">could be your own, that only provides read-only access and your data will be left untouched.</a> <a data-t="945">No destructive actions on your production database when you set querychat's database</a> <a data-t="950">permissions to read-only.</a></p>

<blockquote><a data-t="912">The LLM never executes anything itself.</a> <a data-t="918">It never touches your data or your database, and it never sees your raw data.</a> <a data-t="925">Its only job is to</a> <a data-t="925">propose read-only SQL.</a></blockquote>

<p><a data-t="957">The other nice thing is that querychat doesn't do black box magic.</a> <a data-t="958">Every query can be logged, inspected, audited, and if you need to use private or managed LLMs,</a> <a data-t="965">you're also covered.</a> <a data-t="965">Azure, AWS Bedrock, Google Vertex AI all provide versions of popular models</a> <a data-t="971">that support tool calling and can thus work with querychat.</a> <a data-t="978">So yes, you can use querychat</a> <a data-t="978">in enterprise regulated environments.</a></p>
