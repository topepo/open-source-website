<h3>Building internal R packages</h3>

<p><a data-t="113">So first, technology.</a> <a data-t="115">The first tool that I'm going to share with you out of two is building your own internal R packages.</a> <a data-t="119">We speak a lot about building R packages for a wider community, sharing them as open source software, which is great, but we can't always do that, and if we're working on proprietary projects, we still will want to be able to take advantage of the features of R packages.</a></p>

<p><a data-t="145">So why would we want to do that?</a> <a data-t="147">We don't want to reinvent the wheel when we're reusing code, and we'd like to reduce boilerplate.</a> <a data-t="151">So when we're working on the clinical data that Picnic Health produces, it might come up that we want to calculate the age of our patients on a certain date, and to do that, we'll need the birth date and, if relevant, the death date.</a> <a data-t="161">The calculation is fairly straightforward, but we'd rather make sure that we're doing it correctly once, and then using that correct solution throughout our code, instead of copying and pasting, possibly with errors, throughout many scripts.</a> <a data-t="176">It's also nice that updating the package updates your code.</a> <a data-t="180">So if in our internal data source, our birth date column changes name to date of birth, we can update in our package what column we're referring to, and then our code doesn't need to think about that further.</a> <a data-t="195">And finally, it's always nice when team members are using the same functionality, even when they're working on different projects, because it increases interoperability, so when I'm handing off work to a teammate, they can pick up running, and when I'm reading code in a code review, or I'm trying to figure out what a co-worker did who's no longer at the company, I have more familiarity with what they were doing.</a></p>

<p><a data-t="220">One sign that you might need an R package is if your scripts all begin with a laundry list of functions that you wrote elsewhere.</a> <a data-t="226">So in this example, you're actually already 80% of the way to your package, because all of your content is already there.</a></p>

<p><a data-t="240">When you're building your internal R package, I encourage you to start small with a list of a handful of functions that you've already written.</a> <a data-t="246">This way, you're focusing on the process of building the package and not getting bogged down in the content, which you can always expand later.</a> <a data-t="258">So start small and iterate.</a> <a data-t="259">Get your package to build without errors so that it works, and use your package yourself.</a> <a data-t="264">Then you can share it with others.</a> <a data-t="266">Beyond that, there are all sorts of fancy things you can do, like building a package gem website or writing tests, but before you can run, you can walk.</a> <a data-t="275">So begin small.</a> <a data-t="276">It's good enough if you have tons of warnings and notes when you run </a><a href="/software/devtools/">devtools</a><a data-t="276"> check, which tells you how well your package is building, as long as we have zero errors for now.</a></p>

<p><a data-t="294">For further reading, the standard text by </a><a href="/people/hadley-wickham/">Hadley Wickham</a><a data-t="294"> and Jenny Bryan is called R packages, and it's available for free online.</a> <a data-t="301">There is a package called devtools, which you use, and this is all the same literature, whether you're writing an open source package or an internal package.</a> <a data-t="312">There was a talk at this conference last year about writing your first R package, and for a slightly more advanced approach, you can host your package on Posit Package Manager, which is a paid Posit product.</a> <a data-t="320">This is what we do at Picnic Health, so once we've set up our options, we just have to write install.packages with the package name, just as we would for a package from CRAN.</a></p>

<h3>Cultural solutions: wiki and knowledge sharing</h3>

<p><a data-t="427">All right, we've made it through the business in the front, and now we have our party.</a> <a data-t="434">So now I'm talking about cultural solutions to upskilling our team together.</a> <a data-t="438">And the first one is a wiki.</a> <a data-t="441">So by a wiki, I just mean a website that is editable by its users, in this case, your teammates.</a> <a data-t="447">Wikis are good because they are highly discoverable.</a> <a data-t="450">So if you're not sure what you don't know, you can go to your wiki and read through it, and see what your teammates have deemed a useful and necessary knowledge for you.</a> <a data-t="461">I also advise that when you're contributing to the wiki, advertise your contributions to your team.</a> <a data-t="467">This will encourage them to do the same.</a></p>

<p><a data-t="468">As an example, at our company, this is what our wiki looks like.</a> <a data-t="473">We're using a product called Gitbook to host it, which there are a lot of paid products that will do this.</a> <a data-t="480">I don't necessarily care which one you choose, but this meets our needs.</a> <a data-t="485">You can see we have sections about troubleshooting common errors, how to access our data, how to use our Git and GitHub setup, and much more.</a> <a data-t="499">When you're advertising your contributions, let your teammates know what you're adding, maybe give them a link to the new page, and then they can learn from what you just wrote, and they are encouraged to do the same when they edit the wiki.</a></p>

<p><a data-t="512">I also want to point out that just like in our package, you might already be 80% of the way there.</a> <a data-t="518">A Google doc can be a wiki.</a> <a data-t="520">All we need is an editable page or website that your teammates can view, and a Google doc can be that.</a> <a data-t="529">So why not start with taking a list of bug fixes that you might have posted in Slack already and putting them in a Google doc and slapping the name of wiki on it.</a></p>

<h3>Office hours and pair programming</h3>

<p><a data-t="536">The next technological solution I want to share is office hours and pair programming.</a> <a data-t="543">So by setting aside an optional meeting that we have once or twice a week with my team, and I'm coming from a team of seven, if you have a team of 50 or two, a cadence might be different, and invite folks to come with their problems, their questions, maybe some work that they need a little help collaborating on, and share your knowledge with one another.</a> <a data-t="573">For example, you might talk about resolving merge conflicts in Git, or updating a function in your package, or even set differences between vectors in R.</a></p>

<p><a data-t="590">We can also share our knowledge directly with our teammates, not waiting for a meeting.</a> <a data-t="597">We don't want to hoard our knowledge.</a> <a data-t="599">We want to share it.</a> <a data-t="600">And we can use whatever format our team usually uses to communicate.</a> <a data-t="603">So if you are an in-person team and you always hang by each other's cubicles, go ahead and chat.</a> <a data-t="609">If you are on a remote-first team like me, share in Slack or Teams, whatever you use, email, the format doesn't really matter.</a> <a data-t="617">What matters is that we're sharing our knowledge.</a> <a data-t="619">For example, maybe you found a blog post about ordering the colors of your legend so they match the order of colors from top to bottom in your plot.</a> <a data-t="629">Or maybe you found that two functions that you use a lot can sometimes return multiple values instead of just one.</a> <a data-t="635">Let your teammates know the same since they're likely to be using the same functions, encountering the same issues, or wanting to pretty up their plots in the same way.</a></p>

<p><a data-t="648">The inverse of sharing a public service announcement is asking for help.</a> <a data-t="653">If you're wondering something and you haven't found the answer yet, well, others might be monitoring the same thing.</a> <a data-t="659">Set an example that it's okay to need help.</a> <a data-t="668">And make yourself approachable.</a> <a data-t="669">This goes especially for senior team members.</a> <a data-t="671">So again, on a remote team, especially a bigger remote team, it can be difficult to post in a public forum or chat where everyone can see that you don't know what you're doing.</a> <a data-t="684">And as a senior team member, you can set an example, show that's okay, and become more approachable.</a></p>

<h3>What these solutions have in common</h3>
