<p><a data-t="0">Hi, my name is Sean.</a> <a data-t="9">I'm a product manager at </a><a href="/software/rstudio/">RStudio</a><a data-t="9">.</a> <a data-t="12">As a product manager, I get to talk to data science teams, both small and large, all over the world.</a> <a data-t="18">And unfortunately, in interacting with these teams, we often hear that they're confronted by a false choice to either pick R or Python.</a></p>

<p><a data-t="33">This choice might occur when they're writing a job description to hire a new data scientist.</a> <a data-t="39">It might occur when they're trying to decide what to learn as individuals who want to upskill themselves, or it might occur when they're interacting with IT, asking for resources to support a project.</a></p>

<p><a data-t="49">So why do teams face this false choice?</a> <a data-t="53">Do we really need to pick between one language or another?</a> <a data-t="55">Well, the answer is no.</a> <a data-t="56">And today we're going to talk about why.</a> <a data-t="57">We're going to look at some of the common myths that lead to this choice, how to debunk those myths, and ultimately how data science teams can be most effective when they choose to use R and Python together.</a></p>

<h3>The screwdriver analogy</h3>

<p><a data-t="75">Now, to set some context, imagine you're a craftsman, someone who is handy and builds things.</a> <a data-t="76">As a craftsman, you'd probably be familiar with screwdrivers.</a> <a data-t="82">I mean, who hasn't used a screwdriver to put something together?</a> <a data-t="83">And you might be aware that screwdrivers come in a variety of different shapes and forms.</a> <a data-t="87">You can have a flathead screwdriver, a star bit, a Phillips head, and each of these screwdrivers is designed to serve a specific purpose.</a></p>

<p><a data-t="100">Now imagine as a craftsman, if you were told that you have to choose for the rest of your career between using one type of screwdriver or another, you'd probably look at that person and say, that's crazy.</a> <a data-t="109">There's no way I can practically make this choice.</a> <a data-t="111">For some projects, I'm going to need a flathead.</a> <a data-t="113">For other projects, I'll need a Phillips.</a> <a data-t="115">It doesn't make sense for me to pick one screwdriver to use for the rest of my career.</a></p>

<p><a data-t="120">Instead, what you might do as a craftsman is to say, I want to opt in to using a smarter tool, a tool that's going to be more powerful and allow me to take advantage of all these different bits that are out there.</a> <a data-t="141">Specifically, as a craftsman, you might be interested in something like a drill, a tool that regardless of what bit you're going to need for a project, allows you to work faster and to accomplish more and allows you to work in an easier fashion.</a></p>

<p><a data-t="156">So craftsmen have this drill.</a> <a data-t="157">What about data science teams?</a> <a data-t="157">Well, I believe as data scientists, we should refuse that same false choice between R and Python and other languages, just as the craftsman refuses to pick one type of screwdriver.</a> <a data-t="170">Instead, we should work alongside of folks in IT and the leaders of our teams to build something like a drill, something that regardless of what language we use, is going to give us the power to accomplish our projects faster and easier.</a></p>

<h3>Debunking myth one: more languages means more work</h3>

<p><a data-t="191">But first, I want to address some of these common objections that you'll hear.</a> <a data-t="195">People that say, no, no, no, there's no such thing as a drill for data science.</a> <a data-t="199">We have to pick a single language or a single screwdriver.</a> <a data-t="204">So where do these objections come from?</a> <a data-t="206">What's the biggest objection that data science teams face?</a></p>

<p><a data-t="209">Well, the first one is this belief that if we are to support more than one language, we'll end up doing a lot of duplicative work.</a> <a data-t="219">So for example, if I were a team that wanted to use R and Python, IT might be worried that now I have two times the amount of work to do.</a> <a data-t="228">Instead of supporting one language, I now have to support two.</a> <a data-t="231">That means twice the number of installs, twice the number of support tickets, twice the money spent on IT resources.</a></p>

<p><a data-t="239">And luckily, this line of thinking simply isn't accurate.</a> <a data-t="243">The reason for that is because regardless of what language you use, the core things that IT needs to support are the same.</a> <a data-t="252">Things like computation, logs, authentication, security, data access.</a> <a data-t="258">These provide a common core, that drill, that we can invest in regardless of which drill bit, which language, we end up using.</a></p>

<blockquote><a data-t="258">These provide a common core, that drill, that we can invest in regardless of which drill bit, which language, we end up using.</a></blockquote>

<h3>Development: RStudio Server Pro</h3>

<p><a data-t="282">So the first thing I want to take a look at is in the development space.</a> <a data-t="287">When we're data scientists going to write code in either R or Python, often what we want is to choose an editor that's purpose-built for one of those languages.</a> <a data-t="301">So in R, that might be the RStudio IDE.</a> <a data-t="304">In Python, that might be something like a Jupyter Notebook, or more recently, Jupyter Labs, or perhaps I want to use a full-fledged development environment like VS Code.</a></p>

<p><a data-t="319">Now IT might be thinking, oh no, that means we have to support all these different editors, all these different environments.</a> <a data-t="329">But luckily, that's not true.</a> <a data-t="332">With tools like RStudio Server Pro, you can install a single infrastructure that supports those different editors.</a> <a data-t="340">And so what you're seeing on the screen here is the homepage of RStudio Server.</a> <a data-t="345">When a data scientist enters through that common front door, they're able to pick the different editors that they might want to use for a certain project.</a></p>

<p><a data-t="358">So IT only has to stand up one server environment, they only have to do one set of configuration, they only have to supply data access to one common entry point, but a data scientist is still able to use whatever editor makes the most sense for their project.</a> <a data-t="373">You'll also see here that we're taking advantage of cloud-native tools like Kubernetes to provide elastic scale, and a Docker backend to provide explicitly those dependencies that we might need for a project.</a></p>

<p><a data-t="401">So what this all amounts to is that in development, we're able to get to our first line of code really quickly, regardless of what language that line of code is written in, without doubling the IT headaches.</a></p>

<h3>Production: RStudio Connect</h3>

<p><a data-t="413">Now that's development.</a> <a data-t="414">What about production?</a> <a data-t="414">What about when it comes time to create these different artifacts and share those with others?</a> <a data-t="420">Well, luckily, the same concept applies.</a> <a data-t="423">So at RStudio, what that looks like is a tool like RStudio Connect.</a> <a data-t="428">So RStudio Connect allows you to deploy a wide variety of data artifacts, regardless of what language they're written in.</a></p>

<p><a data-t="436">So for web applications in R, that might be something like </a><a href="/software/shiny-r/">Shiny</a><a data-t="436">.</a> <a data-t="441">For web applications in Python, that might be something like Dash, Streamlit, or Bokeh.</a> <a data-t="446">But regardless of which of those engines you choose to use, RStudio Connect allows you to quickly deploy them onto a web server so that you get a URL that you can share with others.</a> <a data-t="457">And Connect takes care of things like authentication, security, logging, and scale.</a></p>

<p><a data-t="462">Similarly, if you wanted to create an API, say you have a model that you want to expose to other services, well, that exposure might take place in R through a package like </a><a href="/software/plumber/">Plumber</a><a data-t="462">, or might take place in Python through a tool like Flask.</a> <a data-t="476">But either way, Connect provides that common core infrastructure, that drill.</a> <a data-t="482">Finally, the same thing applies for automated reporting.</a> <a data-t="486">So in the Python side, that might be Jupyter Notebooks.</a> <a data-t="488">On the R side, it might be something like R Markdown.</a> <a data-t="491">Regardless, again, we're able to deploy those things to Connect.</a></p>

<p><a data-t="495">And Connect can handle things like scheduling those notebooks to be re-rendered on a regular basis, emailing stakeholders with the new results, and even customizing those emails so that you can send exactly what you need to a stakeholder's inbox, regardless of what language you're using for a project.</a> <a data-t="452">And again, IT is only setting up this infrastructure once.</a> <a data-t="476">They're not doubling their work just because you're multiplying by five or tenfold the number of different data products that you can use.</a></p>

<p><a data-t="487">And as a data scientist, that flexibility is key.</a> <a data-t="492">When it comes to communicate your work, getting buy-in to the modeling that you're doing, or working alongside of a domain expert in real time in a meeting, you need to have the flexibility to use all these different tools to be an effective data science team.</a></p>

<h3>Debunking myth two: multilingual teams can't collaborate</h3>

<p><a data-t="604">Let's talk about a second one, which is that multilingual teams can't collaborate.</a> <a data-t="610">Imagine you are a data science team leader, and you have three or four data scientists on your team that have built out something really awesome in Python.</a> <a data-t="620">In fact, I was speaking to a data science team leader named Wayne recently, who was exactly in this situation.</a> <a data-t="627">But he needed to hire someone new.</a> <a data-t="630">His context was a marketing project.</a> <a data-t="631">He really wanted to hire someone with marketing domain knowledge.</a></p>

<p><a data-t="636">But unfortunately, he couldn't find that candidate who had both the Python expertise and the marketing background.</a> <a data-t="643">But he did know a few people who had strong marketing chops and knew a little bit of R.</a> <a data-t="645">Is he presented with a false choice here?</a> <a data-t="651">I think so, because it turns out that even if your code base is mostly in Python, you can bring someone who has a background in R up to speed really quickly.</a> <a data-t="660">And again, it comes down to having those right tools, a common drill bit, to allow for that type of inter-team collaboration.</a></p>

<p><a data-t="671">And so I want to quickly show you what investments we've made in RStudio to make that type of collaboration possible.</a> <a data-t="675">So what we're looking at here is the RStudio IDE.</a> <a data-t="678">And the new version of the IDE has a what-you-see-is-what-you-mean editor for R Markdown.</a> <a data-t="685">And inside of that editor, you're able to incorporate both R and Python code chunks.</a> <a data-t="691">So here you're looking at Python code that we're able to run, but that code takes advantage of objects coming from R.</a> <a data-t="699">See the r.cars object.</a></p>

<p><a data-t="701">So once we have that Python code executed, it's actually going to write objects back into the environment that we can see in the IDE.</a> <a data-t="709">So in the environment pane in the object explorer, we can view that pandas dataframe we created.</a> <a data-t="715">Once we're happy with the result that we have in Python, we can use the what-you-see-is-what-you-mean editor to insert an R code chunk that takes advantage of that Python dataframe.</a> <a data-t="728">So just like we used the R object in Python, we can use the Python object in R and create a plot with </a><a href="/software/ggplot2/">ggplot2</a><a data-t="728"> of that Python data.</a></p>
