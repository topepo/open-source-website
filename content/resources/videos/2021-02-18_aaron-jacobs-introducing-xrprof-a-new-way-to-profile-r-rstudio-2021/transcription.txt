<p><a data-t="61">But with this increased opportunity has come much increased complexity and increased responsibility.</a> <a data-t="68">The reality is that for many R users, presenting these applications means that these applications</a> <a data-t="74">have users other than their authors for the first time.</a> <a data-t="80">And often they're running in environments</a> <a data-t="80">that are not the same local development environments that we're used to.</a> <a data-t="86">This means that you now</a> <a data-t="86">get bug reports like, hey, why is your </a><a href="/software/shiny-r/">Shiny</a><a data-t="86"> app slow?</a> <a data-t="93">Or this report is taking an hour</a> <a data-t="93">to run, but it used to take two minutes.</a> <a data-t="99">What's happening?</a></p>

<h3>Profiling in R</h3>

<p><a data-t="99">Now, traditionally, to answer performance</a> <a data-t="99">related questions like this, you would make use of a profiling tool.</a> <a data-t="103">Profiling itself</a> <a data-t="103">is a term from computer science.</a> <a data-t="107">It effectively means that we collect data about where our</a> <a data-t="109">program is spending its time.</a> <a data-t="112">Effectively, which functions are being called most often.</a></p>

<p><a data-t="115">Now, for R users, this is actually kind of a comfortable situation because profiling produces</a> <a data-t="120">data.</a> <a data-t="120">And R users know how to deal with data.</a> <a data-t="122">We can use all of our favorite tools for analysis,</a> <a data-t="127">visualization, summary of profiling data, just like we could with any other kind.</a> <a data-t="133">And profiling</a> <a data-t="133">itself is a fundamentally empirical approach to performance analysis.</a> <a data-t="138">Don't guess, measure.</a> <a data-t="138">And</a> <a data-t="138">then use those measurements to make informed decisions.</a></p>

<blockquote><a data-t="138">Don't guess, measure.</a> <a data-t="138">And</a> <a data-t="138">then use those measurements to make informed decisions.</a></blockquote>

<p><a data-t="144">Profiling is such an important field</a> <a data-t="144">for performance analysis that actually R has its own built-in sampling profiler.</a> <a data-t="150">It's available</a> <a data-t="150">through the RProf function.</a> <a data-t="153">And there's a variety of functions in base R and the wider ecosystem</a> <a data-t="156">that allow you to make use of RProf's format to produce, again, these analysis, these summaries,</a> <a data-t="162">these visualizations.</a> <a data-t="162">And so the prof tools package or the prof is package.</a> <a data-t="167">There's also</a> <a data-t="167">AProf and GUIProfiler, which are slightly less popular.</a> <a data-t="172">These all do exactly this summary,</a> <a data-t="172">analysis, visualization.</a></p>

<p><a data-t="181">And actually, because there's a wider profiling ecosystem beyond R,</a> <a data-t="181">lots of these packages can actually produce conversions to popular formats that are used</a> <a data-t="186">in external tools.</a> <a data-t="188">So for example, the ability to convert RProf formats to KCacheGrind or Google's</a> <a data-t="192">PProf tool, or the SpeedScope format, or also the original FlameGraph software format.</a> <a data-t="199">These are all available in the R ecosystem.</a> <a data-t="203">So there's a big existing set of tools you can use</a> <a data-t="205">for performance analysis using the RProf function.</a></p>

<h3>The problem with local profiling</h3>

<p><a data-t="211">Now, all of this sounds great, but let's get back</a> <a data-t="215">to that whole Shiny performance issue for a moment.</a> <a data-t="218">One of the interesting things about Shiny</a> <a data-t="218">is that it's often used by coworkers or peers or otherwise as an internal tool, which means that</a> <a data-t="223">the users are often people who essentially would send you an email or a message on Slack if they</a> <a data-t="227">were issuing you some kind of report.</a> <a data-t="231">And for example, if you receive some sort of performance</a> <a data-t="231">bug report, it's basically going to come down to, hey, your app is slow.</a> <a data-t="238">Can you make it faster so</a> <a data-t="238">that I can do my job?</a></p>

<p><a data-t="245">And of course, this is not really a helpful or informative thing for them to</a> <a data-t="250">tell you.</a> <a data-t="250">So you start asking follow-up questions like, hey, what were you doing?</a> <a data-t="254">And they'll tell you,</a> <a data-t="254">you know, I had clicked X and Y.</a> <a data-t="258">You'll ask them things like, okay, that's not really enough</a> <a data-t="258">information for me to recreate your issue.</a> <a data-t="262">Do you remember what you set these inputs to?</a></p>

<p><a data-t="261">And sometimes they can remember and tell you perfectly, but much more often,</a> <a data-t="265">usually they either don't remember or they misremember and they tell you what they did,</a> <a data-t="269">but really they didn't.</a> <a data-t="271">And you spent a long time running down this rabbit hole of trying to</a> <a data-t="273">recreate their issue locally.</a> <a data-t="278">And the fundamental reason for this is that when you're using</a> <a data-t="278">RProf, you have to be doing profiling locally.</a> <a data-t="286">So you're trying to reproduce an issue or an</a> <a data-t="286">environment and then profiling locally to collect data that you can use to analyze your program.</a></p>

<p><a data-t="293">Now, there's all kinds of reasons why this is frustrating, but in addition to the frustration,</a> <a data-t="297">there's also the problem that like your local environment may kind of differ from wherever your</a> <a data-t="304">code ends up really running for these users.</a> <a data-t="310">So a classic example of this is, say for myself,</a> <a data-t="310">I often write and develop applications in Canada, but deploy them to servers that are in Europe for</a> <a data-t="316">European users.</a> <a data-t="318">This means that, you know, the kind of databases that they're connecting to,</a> <a data-t="322">or the other network services that they might contact, might have very, very different latency</a> <a data-t="328">profiles when they're actually deployed, as opposed to when I'm working on them locally.</a></p>

<p><a data-t="333">And this can kind of create both bugs, but also high performance issues where stuff that works</a> <a data-t="338">fast for you locally doesn't work very fast at all for your users.</a> <a data-t="343">So in addition to the</a> <a data-t="343">difficulty of trying to recreate what your user was doing, you're also trying to recreate this</a> <a data-t="347">environment.</a> <a data-t="352">And overall, this is a fairly error-prone process, and it really slows down</a> <a data-t="352">tracking down performance issues.</a></p>

<h3>Introducing xrprof</h3>

<p><a data-t="359">So this kind of disconnect between where code is really running</a> <a data-t="359">and where you want to profile it is actually the fundamental motivation behind a new project that</a> <a data-t="364">I've been working on called XRProf.</a> <a data-t="370">Now, XRProf is not a clever name.</a> <a data-t="374">It just means external</a> <a data-t="374">RProf, and it is exactly that.</a> <a data-t="377">It's a standalone program that allows you to profile R code that is</a> <a data-t="377">already running.</a></p>

<p><a data-t="385">So the fundamental way of thinking about this is that RProf allows you to profile by</a> <a data-t="385">modifying your R code, usually locally in an interactive session.</a> <a data-t="391">XRProf, on the other hand,</a> <a data-t="391">is designed and oriented around profiling R code that is already running without modification.</a> <a data-t="398">And this is very powerful if, for example, you want to profile code that's running</a> <a data-t="402">in a production environment, say on Shiny Server, on RStudio Connect, for R applications that are</a> <a data-t="407">running under Docker.</a> <a data-t="412">XRProf can see and understand R programs that are running in all of those</a> <a data-t="412">conditions and profile them for you without modifying them.</a> <a data-t="417">This allows you to actually do</a> <a data-t="417">your profiling with real users and real data.</a> <a data-t="423">Makes it much easier to track down performance</a> <a data-t="423">issues because you have data that's close to the source.</a></p>

<p><a data-t="431">Now, XRProf is fundamentally designed to</a> <a data-t="431">be a compatible drop-in replacement for RProf, which means it produces the same format,</a> <a data-t="437">and it works with all of the existing R ecosystem tools, and by extension,</a> <a data-t="442">through conversions to other formats, all of the wider profiling world outside of R.</a> <a data-t="449">Because of this compatibility, it's really, really easy to use, and it should be very familiar if</a> <a data-t="454">you've used profiling data in R before.</a></p>

<h3>Profiling C and C++ code</h3>

<p><a data-t="458">Now, there's another piece of this puzzle.</a> <a data-t="461">One of the</a> <a data-t="461">other major things that's happening in the R ecosystem is that more and more R packages are</a> <a data-t="463">making use of C and C++ code.</a> <a data-t="469">Now, it's always been possible to embed native code in R.</a> <a data-t="473">It's</a> <a data-t="475">actually one of the strengths of R as a programming language.</a> <a data-t="482">But I think partly because of</a> <a data-t="482">democratizing tools like Rcpp, which have made it much easier for R users to write C++ code,</a> <a data-t="482">it's now much more common to include C++ in an R package.</a></p>

<p><a data-t="489">This is great on the one hand for</a> <a data-t="496">performance, but it does create a problem, which is that RProf can actually only see functions</a> <a data-t="496">that are happening inside of the R code.</a> <a data-t="503">So, it can only see R functions.</a> <a data-t="505">That means that the more</a> <a data-t="505">and more of this code that is happening in C and C++, the more is totally invisible to RProf.</a></p>

<p><a data-t="509">And this creates serious issues when the C and C++ code is actually the source of the performance</a> <a data-t="515">problem.</a> <a data-t="521">Now, I know you might be thinking, hold on, C and C++ are much faster than R.</a> <a data-t="528">And this is true, but in practice, C and C++ code can make exactly the kind of poor performance</a> <a data-t="528">decision you can make in R.</a> <a data-t="533">You can do stuff you don't really need to do.</a> <a data-t="535">You can do things much</a> <a data-t="535">more often than you really need to do them.</a> <a data-t="538">Or you can choose, you know, a kind of inefficient</a> <a data-t="538">data structure to represent the problem or an algorithm to actually solve the problem.</a></p>

<p><a data-t="549">So, just because code is written in C and C++ doesn't mean that it somehow</a> <a data-t="549">becomes free of these issues.</a> <a data-t="557">And I think that if you were trying to track down a C or C++</a> <a data-t="557">related performance issue in an R package, it's actually been kind of painful so far.</a> <a data-t="563">Either you just guess what the problem is and keep trying various approaches, or you try and</a> <a data-t="569">use some sort of C or C++ level profiling tool and an R level profiling tool at the same time,</a> <a data-t="576">and then try and cross-reference what's happening.</a> <a data-t="581">This is actually in practice a very inexact</a> <a data-t="581">and frustrating process.</a> <a data-t="586">There's just not enough information on either side for you to be able to</a> <a data-t="586">knit these things together.</a></p>

<p><a data-t="593">And this secondary frustration was actually a big technology</a> <a data-t="593">motivation of XRProf as well.</a> <a data-t="600">So, XRProf can actually see what is happening in your C and C++</a> <a data-t="600">code.</a> <a data-t="606">And what it can do is append that to your R level functions so you can see a more holistic</a> <a data-t="606">picture of what's going on.</a> <a data-t="611">I think this is a really, really powerful tool, especially for R</a> <a data-t="611">package developers who need to do performance analysis that crosses this language barrier.</a> <a data-t="617">And don't forget that R itself is actually written in C.</a> <a data-t="622">So, in practice, sometimes when you're</a> <a data-t="622">having performance issues that touch base R code, actually the issue is in the C.</a></p>

<h3>A real-world example: mongolite</h3>

<p><a data-t="628">So, again, what does this kind of look like?</a> <a data-t="635">This is a more involved example.</a> <a data-t="635">And in fact,</a> <a data-t="635">it was a very real issue that I first encountered a couple of years ago.</a></p>

<p><a data-t="639">So, at work, we have a whole bunch of semi-structured data in a database called MongoDB.</a> <a data-t="647">And there's an existing R package called mongolite, which can be used to query MongoDB.</a> <a data-t="653">And it actually works really, really well the vast majority of the time.</a></p>

<p><a data-t="657">Now, if you're not familiar with MongoDB, effectively what it does is store data as a</a> <a data-t="663">kind of JSON.</a> <a data-t="670">Which means that tabular data, the kind you would see in a CSV or in a SQL table,</a> <a data-t="671">is essentially represented as an array, a JavaScript array of JavaScript objects.</a> <a data-t="677">So, this effectively looks like a big long list of dictionaries of key value pairs,</a> <a data-t="683">where every key value or every row is represented as a dictionary.</a></p>

<p><a data-t="688">Okay. Now, it turns out that there's a known performance issue with mongolite for large</a> <a data-t="695">databases with lots of columns.</a> <a data-t="700">The queries can take a long time.</a> <a data-t="700">A surprisingly long time,</a> <a data-t="700">at least.</a> <a data-t="705">And when I first encountered this a couple of years ago, I wanted to figure out why.</a></p>

<p><a data-t="715">So, I profiled it using Rprof.</a> <a data-t="715">But I actually discovered that I couldn't really see anything.</a> <a data-t="721">And the reason for this is that all of the interesting parts of mongolite are actually</a> <a data-t="721">implemented in C.</a> <a data-t="727">Which means that it was very hard to figure out what was going on.</a></p>

<p><a data-t="731">And so, at the time, I did exactly this very awkward dance of using C and C++</a> <a data-t="731">related tools and R related tools and trying to cross reference them.</a> <a data-t="735">And it was very painful and took me a long time.</a> <a data-t="740">And I wasn't even really ever sure if</a> <a data-t="741">that was measuring the right thing.</a> <a data-t="746">So, when I redid the analysis for this talk,</a> <a data-t="746">it was kind of comforting that what had taken me several days a few years ago</a> <a data-t="750">took me something like 90 seconds when I used XRprof.</a></p>

<p><a data-t="758">And so, this is what happens when you use Rprof.</a> <a data-t="766">This is a flame graph of what you would see.</a> <a data-t="766">Which is not that informative because you just don't see what's happening inside of the R code</a> <a data-t="770">that's mostly implemented in C.</a> <a data-t="776">But if you can see the C code, this is what you will see.</a> <a data-t="776">So, here the R functions are in blue and the C functions are in orange.</a></p>

<p><a data-t="781">Now, of course, I know this is much too small for you to actually be able to read function names.</a> <a data-t="784">And that's not really the point.</a> <a data-t="789">So, I'll just point out a couple of things that I was able to</a> <a data-t="790">pick up on once I had this view.</a> <a data-t="799">The first is that there's a certain amount of time that R</a> <a data-t="799">is communicating with Mongo itself.</a> <a data-t="805">And this is important because effectively this represents the</a> <a data-t="805">total upper bound of how fast we could ever possibly make this R code.</a> <a data-t="810">Because the reality</a> <a data-t="810">is we still need to spend time transmitting data from the database over the network.</a> <a data-t="815">And there's</a> <a data-t="815">nothing really we can do about the time that it takes to do that.</a> <a data-t="819">Other than, say, querying less</a> <a data-t="819">data.</a> <a data-t="827">So, this kind of serves as a total upper bound.</a></p>

<p><a data-t="827">But there were two kind of odd things</a> <a data-t="827">that I noticed or picked up on when I looked at these profiles.</a> <a data-t="835">Which is that R is spending a lot</a> <a data-t="835">of time allocating memory.</a> <a data-t="842">And if you don't know what that means, well, mostly it doesn't matter.</a> <a data-t="842">Because allocating memory is normally really fast.</a> <a data-t="848">And in order for allocating memory to show</a> <a data-t="848">up on a profile, it means it needs to be happening a lot.</a> <a data-t="853">Or at least happening really inefficiently.</a> <a data-t="858">So, when I saw this, it immediately piqued my interest.</a> <a data-t="862">And it made me think what could be</a> <a data-t="862">causing this many allocations?</a> <a data-t="864">The second thing that I noticed is that</a> <a data-t="865">MongoLite seems to be spending a ton of time transposing lists and checking for uniqueness.</a></p>

<p><a data-t="873">Which seems like an odd operation to be doing.</a> <a data-t="877">Because, again, checking uniqueness is normally</a> <a data-t="877">pretty fast.</a> <a data-t="882">In order to spend so much time doing it, you have to be doing it a lot.</a></p>

<p><a data-t="883">So, originally, actually, I puzzled over this for a while.</a> <a data-t="888">Before I realized that, ultimately,</a> <a data-t="889">it actually totally made sense.</a> <a data-t="895">So, remember how I said that JSON data represents tabular</a> <a data-t="895">data as an array of objects?</a> <a data-t="901">Well, it turns out that how MongoLite works is that when it reads</a> <a data-t="901">in rows, it takes these JSON objects and turns them into R lists.</a> <a data-t="908">Which is a very natural</a> <a data-t="908">representation of a row.</a> <a data-t="916">Basically, you know, you have 20 columns.</a> <a data-t="916">And that gives you a list</a> <a data-t="916">of length 20.</a> <a data-t="920">And, you know, every element of that list can have a different type.</a> <a data-t="920">And, you know,</a> <a data-t="920">the names of your lists are the names of your column.</a> <a data-t="927">And then what happens at the end is,</a> <a data-t="927">basically, it takes all of these lists and smooshes them together in a data frame.</a></p>

<p><a data-t="941">Now, this is kind of interesting because if you imagine, if we were to create a data frame in</a> <a data-t="941">advance, we knew how many rows we were getting, we would essentially just do one allocation for</a> <a data-t="947">every column.</a> <a data-t="953">Right? So, if we had 20 columns, we do 20 allocations.</a> <a data-t="953">But if we read in a million</a> <a data-t="953">rows of 20 columns using this approach, we do something like 20 million allocations.</a> <a data-t="961">So, you can kind of see where, you know, maybe some of this was showing up in the profile because</a> <a data-t="965">instead of doing 20 allocations, we're doing 20 million.</a></p>

<p><a data-t="970">And the second thing is, because we have all these lists, when we smoosh them together,</a> <a data-t="976">we have to do a bunch of things like check that all of the names are the same.</a> <a data-t="980">And all of the</a> <a data-t="980">types are the same.</a> <a data-t="987">And we have to do this for every single row we add against all the data</a> <a data-t="987">that we've previously added.</a> <a data-t="994">So, again, you can see how this adds up.</a> <a data-t="994">Now, actually, this story</a> <a data-t="994">doesn't have a super happy ending.</a> <a data-t="999">I did propose changing MongoLite so that it used a different</a> <a data-t="999">algorithm.</a> <a data-t="1004">The problem was it was a very complex patch, something like 700 lines of new C code or</a> <a data-t="1004">something like that.</a> <a data-t="1009">So, it wasn't accepted into MongoLite.</a> <a data-t="1010">But it was about five times faster.</a> <a data-t="1013">Which just goes to show that even though you're writing something in a fast language,</a> <a data-t="1013">like C or C++, doesn't mean you can't make exactly the same kind of performance issues</a> <a data-t="1018">that you'd encounter in R code.</a></p>

<blockquote><a data-t="1013">Which just goes to show that even though you're writing something in a fast language,</a> <a data-t="1013">like C or C++, doesn't mean you can't make exactly the same kind of performance issues</a> <a data-t="1018">that you'd encounter in R code.</a></blockquote>
