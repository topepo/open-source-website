<h3>The R Validation Hub and white paper</h3>

<p><a data-t="174">Now the R Validation Hub is a cross-industry initiative to enable the use of R by the pharmaceutical</a> <a data-t="182">industry in a regulatory set where the output may be used in submissions to regulatory agencies.</a> <a data-t="191">Now to support our mission, we wrote a white paper titled A Risk-Based Approach for Assessing</a> <a data-t="197">R Package Accuracy Within a Validated Infrastructure.</a> <a data-t="201">This paper was written by Andy Nichols from JSK, Paolo Vargo from Janssen and John Sims</a> <a data-t="209">from Pfizer and the aim of this paper is to propose a possible risk-based approach for</a> <a data-t="217">assessing R package accuracy within a validated infrastructure.</a></p>

<p><a data-t="223">This white paper addresses concerns raised by a statistician, programmers, quality assurance</a> <a data-t="230">teams and others within the pharmaceutical industry about the use of R and selected R</a> <a data-t="237">packages as a primary tool for a statistical analysis for regulatory submission work.</a></p>

<p><a data-t="245">Now the white paper is based, or rather the definition that we use of validation in the</a> <a data-t="250">white paper is based on the FDA definition of validation, which you can read more about</a> <a data-t="258">in the linked article.</a> <a data-t="262">And there validation can be separated in three main components, accuracy, reproducibility</a> <a data-t="268">and traceability.</a></p>

<p><a data-t="271">In the white paper, we focus on accuracy.</a> <a data-t="275">We focus on this qualitative assessment of correctness or freedom from error.</a> <a data-t="283">And in the white paper, we propose a risk-based approach to establish accuracy of those</a> <a data-t="290">packages that are contributed packages.</a> <a data-t="292">And so we make a decision between core versus contributed R packages.</a></p>

<p><a data-t="298">And core packages, we say that those packages that are part of the official R distribution</a> <a data-t="307">as formally released by the R Foundation.</a> <a data-t="309">So these software is commonly referred to as base R plus recommended packages.</a> <a data-t="316">Now these packages are considered to be low risk and we make this determination based on</a> <a data-t="324">how the R Foundation develops the R language.</a></p>

<p><a data-t="328">And you can read more about how the R Foundation describes and how the R Foundation takes their</a> <a data-t="338">approach to developing the R language, their testing approach and how they release versions</a> <a data-t="343">in the link document.</a> <a data-t="346">And we can, by reading this document, you may agree with us that the R-Core team controls</a> <a data-t="352">R like any of the companies that produces the commercial software that you may buy in</a> <a data-t="359">this space.</a> <a data-t="360">So the R packages are, the core R packages are very well controlled and so can be considered</a> <a data-t="366">to be low risk.</a></p>

<p><a data-t="368">Now the complement, we have those packages that are contributed that are made by the</a> <a data-t="374">community, which are about 70,000 packages now on CRAN.</a> <a data-t="380">The risk of these packages really depends on various factors, like how well is the package</a> <a data-t="386">developed?</a> <a data-t="387">Is it well tested?</a> <a data-t="390">The package may have varying software development life cycles that you should be aware of.</a> <a data-t="396">And you also can check for how much exposure has the package had within the user community.</a></p>

<h3>Assessing contributed packages</h3>

<p><a data-t="406">And in the white paper, we propose, as I mentioned, an approach to assess the accuracy of the</a> <a data-t="411">contributed packages as those core are considered to be low risk.</a> <a data-t="417">So once someone in your company wants to use one of these contributed packages, the question</a> <a data-t="424">is, what steps can you take to start to assess the risk of those packages?</a> <a data-t="427">So the white paper can aid you in taking the first steps to start assessing the risk of</a> <a data-t="434">using an R package.</a></p>

<p><a data-t="437">So once you have decided that you want to use a package, the next step that we propose</a> <a data-t="445">for you to do is to assess the package based on four criteria, purpose, maintenance, good</a> <a data-t="452">practice, community usage and testing.</a></p>

<p><a data-t="457">And so when talking about purpose, it is assumed that any package that we're talking about</a> <a data-t="463">is intended to be used within a GXP workflow.</a> <a data-t="468">So package directly manipulating data, creating graphics, or running a statistical model that</a> <a data-t="475">forms part of the submission, we're referring to those.</a> <a data-t="479">So in terms of risk, any package should be weighted equally.</a> <a data-t="485">But we do recognize that modeling statistical packages deserve special attention.</a> <a data-t="491">And we know that there is an innate difficulty to identify errors in those packages in those</a> <a data-t="497">modeling packages versus those say on the manipulated data.</a></p>

<p><a data-t="502">So for instance, if you would like to test the arranged function of the plier, it is</a> <a data-t="508">easy to do.</a> <a data-t="509">You can or relatively easy to do.</a> <a data-t="511">You can use a small data set to test this function and check that the corresponding</a> <a data-t="516">column has been sorted correctly.</a> <a data-t="519">On the other hand, a statistical models are harder to test, and a statistician can roughly</a> <a data-t="526">check that a model is correct, but not correct to the decimal places.</a></p>

<p><a data-t="532">Now in terms of maintenance, good practice is good to to address the questions on how</a> <a data-t="538">is the documentation of the package is a source code public on GitHub or any other medium.</a> <a data-t="545">And we do we do know that when you when you expose a package to the community, you helps</a> <a data-t="552">reduce the risk as even if the package has no tests, but it's popular, you know that</a> <a data-t="558">it has been widely used by the community.</a> <a data-t="561">And so lastly, testing is a crucial part of the SDLC and another maintenance best practice</a> <a data-t="572">metric.</a></p>
