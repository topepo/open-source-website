<h3>A story about manual testing pain</h3>

<p><a data-t="47">So with that in mind, I'd like to begin with a story</a> <a data-t="49">about something that happened to me back in May 2021.</a> <a data-t="53">So I was in my home office working on a new feature for the app,</a> <a data-t="57">but of course I was multitasking because I was also keeping an eye</a> <a data-t="60">on the activity in our team chat window.</a> <a data-t="63">So one of the users was saying she was having some issues with the app.</a> <a data-t="67">It wasn't populating the drop-down menu with the values that she was expecting.</a> <a data-t="71">So the values were in the app before,</a> <a data-t="73">and then now all of a sudden she didn't see them anymore.</a></p>

<p><a data-t="76">So one of the other developers identified the issue</a> <a data-t="81">and determined that we would have to quickly release a new version of the app</a> <a data-t="84">that resolves the issue with the drop-down menu.</a> <a data-t="88">And that's when I panicked because I realized</a> <a data-t="90">that meant two things were about to happen.</a></p>

<p><a data-t="94">First, it meant that I would have to put on another hat,</a> <a data-t="101">my code reviewer hat.</a> <a data-t="101">So as a code reviewer, I'm responsible for reviewing GitHub pull requests,</a> <a data-t="106">have to read the code to make sure it adheres to our coding standards,</a> <a data-t="109">and then I have to run the app in a local environment</a> <a data-t="112">to make sure that all the old features work with the new features.</a> <a data-t="115">I have to check the radio buttons, the drop-down menus,</a> <a data-t="118">the contents of the tables.</a> <a data-t="120">I have to make sure that the database API and the model API connections</a> <a data-t="124">are still working as they used to as we added these new features.</a> <a data-t="128">And so if it passes those tests, then we deploy the app to the test server.</a></p>

<p><a data-t="133">But then that means I have to put on yet another hat.</a> <a data-t="137">This time I have to put on my app tester hat.</a> <a data-t="140">So as a member of the app testing team,</a> <a data-t="142">I was one of several people that had to manually</a> <a data-t="148">test the deployed version of the app.</a> <a data-t="150">So I would have to open the app in a web browser</a> <a data-t="153">and then run a lot of the same tests as before,</a> <a data-t="155">check the radio buttons, check the drop-down menus,</a> <a data-t="158">check the API connections, database connections,</a> <a data-t="160">all those other things that I did before in the code review.</a> <a data-t="164">And so now the only difference is that this</a> <a data-t="166">is on the deployed version of the app as opposed</a> <a data-t="168">to the pre-deployment version.</a></p>

<p><a data-t="170">Now, testing is very important.</a> <a data-t="172">It does allow us to catch bugs so that we can fix them</a> <a data-t="175">before the new version of the app is released to production.</a> <a data-t="179">However, as one of the people who was responsible for actually</a> <a data-t="183">running the test, I could say that this</a> <a data-t="185">was a really extremely tedious, painful experience for me.</a> <a data-t="191">It was very stressful.</a></p>

<p><a data-t="193">I also felt undervalued because I'm thinking,</a> <a data-t="195">OK, I have a PhD in statistics, and yet my job now</a> <a data-t="199">is to click this radio button or check this drop-down menu.</a> <a data-t="203">So I felt like this wasn't a good use of my time.</a> <a data-t="206">Another issue is that I and the other app testers</a> <a data-t="209">had other priorities.</a> <a data-t="211">So we weren't full-time app testers.</a> <a data-t="213">We could only run the test when we</a> <a data-t="215">had time to squeeze testing in.</a></p>

<blockquote><a data-t="193">I also felt undervalued because I'm thinking,</a> <a data-t="195">OK, I have a PhD in statistics, and yet my job now</a> <a data-t="199">is to click this radio button or check this drop-down menu.</a> <a data-t="203">So I felt like this wasn't a good use of my time.</a></blockquote>

<p><a data-t="217">And that meant that managing all of those schedules</a> <a data-t="221">across the different testers meant</a> <a data-t="222">that it could take a long time for us</a> <a data-t="224">to get through those tests and actually go to the next stage</a> <a data-t="227">of our software release.</a> <a data-t="230">So that was increasing the amount of time</a> <a data-t="232">it took us to deploy the app.</a> <a data-t="235">Because it was so time-intensive,</a> <a data-t="236">we had to sparingly apply these tests.</a> <a data-t="239">So we couldn't afford to run them</a> <a data-t="240">as new features were coming in.</a> <a data-t="242">We had to use multiple scenarios.</a> <a data-t="244">We could only do them really once if we could do that.</a></p>

<p><a data-t="249">And then lastly, we found that different testers were</a> <a data-t="252">running the tests with slightly different conditions.</a> <a data-t="256">So there was some subjective assessment</a> <a data-t="258">with whether or not the test passed or failed.</a> <a data-t="261">So one tester could say it passed,</a> <a data-t="263">and another person running the same test</a> <a data-t="265">would say that it failed.</a> <a data-t="266">So that was a problem.</a></p>

<h3>Discovering automation tools</h3>

<p><a data-t="268">So because these releases were such a heavy burden for me,</a> <a data-t="271">I started looking for a way to make this process more</a> <a data-t="274">efficient for me and for our team.</a> <a data-t="276">I was already familiar with Selenium.</a> <a data-t="278">I knew it was a great tool for automating interactions</a> <a data-t="281">with the web browser.</a> <a data-t="282">And I figured this might help automatically check</a> <a data-t="284">whether the UI elements were still</a> <a data-t="286">working on the deployed version of the app.</a></p>

<p><a data-t="289">I was somewhat familiar with the </a><a href="/software/rvest/">rvest</a><a data-t="289"> package</a> <a data-t="292">and that it was great for web scraping.</a> <a data-t="294">So I figured this might help to programmatically extract</a> <a data-t="297">the contents of the web page.</a> <a data-t="299">And I could run some sanity checks</a> <a data-t="300">against it to make sure everything's rendering</a> <a data-t="302">properly so Selenium and rvest could work together</a> <a data-t="304">on the deployed version of the app.</a></p>

<p><a data-t="307">My colleague Eric Nant recommended</a> <a data-t="309">that I use the </a><a href="/software/shinytest2/">shinytest2</a><a data-t="309"> package</a> <a data-t="312">to test the code that other developers were submitting</a> <a data-t="314">in their pull requests.</a> <a data-t="316">So he figured that that might help to automate</a> <a data-t="319">some of the code reviews.</a> <a data-t="321">And then, of course, later I found out</a> <a data-t="322">that there was another version of the </a><a href="/software/shinytest/">shinytest</a><a data-t="322"> package</a> <a data-t="326">called shinytest2 that somebody will speak about later.</a> <a data-t="329">And then finally, the shinytest documentation</a> <a data-t="332">was recommending that we also use the </a><a href="/software/testthat/">testthat</a><a data-t="332"> package.</a></p>

<p><a data-t="336">And so now I had all these tools.</a> <a data-t="339">There was a bit of a learning curve,</a> <a data-t="340">but I took about three months of my time.</a> <a data-t="342">And eventually, over the next three months,</a> <a data-t="344">I was able to use those tools to create an automated testing</a> <a data-t="348">pipeline.</a></p>

<p><a data-t="350">Let's see how the software release workflow changed</a> <a data-t="353">after introducing these tools.</a> <a data-t="357">So in the code review stage in blue,</a> <a data-t="359">we see the manual pre-deployment testing during code reviews</a> <a data-t="364">was replaced with the shinytest2 script that</a> <a data-t="367">automates these same tasks.</a> <a data-t="369">And now the code reviewer hat sits</a> <a data-t="371">on the head of the bot that runs the test script.</a></p>

<p><a data-t="375">And then in orange, in the post-deployment testing stage,</a> <a data-t="379">I'm running the manual regression test.</a> <a data-t="384">I'm replacing those with a suite of Selenium test scripts.</a> <a data-t="387">And then I get to remove that heavy app tester hat</a> <a data-t="389">from my head and place it on the bot who</a> <a data-t="391">is happy to run those test scripts.</a> <a data-t="393">At least he doesn't complain about it if he does not</a> <a data-t="396">feel happy about it.</a></p>

<p><a data-t="398">So those two changes had a significant impact</a> <a data-t="400">on our software release workflow.</a> <a data-t="402">Our code reviewers and app testers had less work to do.</a> <a data-t="406">They weren't feeling stressed out with each new release.</a> <a data-t="409">We didn't have to worry about testers balancing their testing</a> <a data-t="412">duties with other high-priority work,</a> <a data-t="413">so we ended up shortening our time to deployment</a> <a data-t="416">from five days to five hours, which was pretty</a> <a data-t="420">huge for our testing team.</a></p>

<blockquote><a data-t="413">so we ended up shortening our time to deployment</a> <a data-t="416">from five days to five hours, which was pretty</a> <a data-t="420">huge for our testing team.</a></blockquote>

<h3>The automated testing framework</h3>

<p><a data-t="450">So I've talked about this before on the previous slides,</a> <a data-t="452">but here's where I get to formally introduce</a> <a data-t="454">the automated testing pipeline that I'm proposing.</a> <a data-t="457">It contains four key principles.</a> <a data-t="460">First, I believe you should automate the tests.</a> <a data-t="463">You should actually have some sort of programmatic</a> <a data-t="465">representation of your test logic.</a></p>

<p><a data-t="468">And this could obviously be, we could easily</a> <a data-t="471">fill an hour or maybe a day talking about this,</a> <a data-t="473">but lots of different things like how do you</a> <a data-t="475">recover safely from errors?</a> <a data-t="476">How do you appropriately log the test results?</a> <a data-t="479">How do you have this repository of all your test artifacts,</a> <a data-t="481">maybe your screenshots or things</a> <a data-t="483">that were downloaded in the course of running the app?</a> <a data-t="486">That's what I mean by automating the test, all of that.</a></p>

<p><a data-t="489">Next, it's important to have one or more development servers</a> <a data-t="493">that are completely separate from your production server.</a> <a data-t="496">And that allows us to see how the updated app works</a> <a data-t="499">without breaking the production version that end users</a> <a data-t="502">are interacting with.</a> <a data-t="503">And then the next two principles have</a> <a data-t="505">to do with when we would run the test.</a> <a data-t="508">The first is that I recommend running</a> <a data-t="511">some pre-deployment tests so that you can integrate</a> <a data-t="513">these features safely in a safe space on your development</a> <a data-t="517">server without crashing the version of your app</a> <a data-t="519">that the end users are using.</a> <a data-t="522">And then finally, I would recommend</a> <a data-t="524">running the post-deployment check also</a> <a data-t="528">on the development server.</a> <a data-t="530">And this is where we can more closely mimic</a> <a data-t="532">what the end users are going to experience when they log</a> <a data-t="534">into the app using the web browser.</a></p>

<p><a data-t="537">So again, pre-deployment and post-deployment,</a> <a data-t="540">that's really important.</a> <a data-t="542">One question that comes up is whether automation</a> <a data-t="545">makes sense for a particular use case or a particular web app.</a> <a data-t="548">And to answer this question, I usually</a> <a data-t="552">follow up with a couple of follow-up questions.</a> <a data-t="554">The first would be, is the logic consistent?</a> <a data-t="558">The test logic, is it consistent from one run to the next?</a> <a data-t="562">So if you can say, step one, step two, step three,</a> <a data-t="565">if you can actually have all of that logic</a> <a data-t="568">and not have to worry about a change from one to the next,</a> <a data-t="571">that's one consideration.</a></p>

<p><a data-t="573">And then the second question would be</a> <a data-t="575">whether the test logic can be completely scripted</a> <a data-t="578">so that you can get the result without having</a> <a data-t="580">to use any sort of subjective judgment</a> <a data-t="581">or having some sort of human involved making that decision.</a> <a data-t="584">There are other considerations, obviously.</a> <a data-t="586">So there's, do you have the infrastructure</a> <a data-t="588">to support these automation tools like shinytest2</a> <a data-t="590">or Selenium?</a> <a data-t="592">Do you have time to do this?</a> <a data-t="593">There's a little bit of a learning curve</a> <a data-t="595">and a little bit of time investment</a> <a data-t="596">in terms of writing these test scripts.</a> <a data-t="598">But in general, I would say if you answered yes</a> <a data-t="600">to both of those questions, then I</a> <a data-t="601">would say your app is a prime candidate for automation.</a></p>
