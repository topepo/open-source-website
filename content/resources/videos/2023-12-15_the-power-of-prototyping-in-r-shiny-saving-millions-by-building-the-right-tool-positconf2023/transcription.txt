<h3>A cautionary tale</h3>

<p><a data-t="47">Today I will be talking mostly from my own personal experience.</a> <a data-t="52">I would like to start with a short story.</a> <a data-t="52">You can see a spoiler here, it won't be a success story.</a></p>

<p><a data-t="59">It started as a typical project.</a> <a data-t="62">There was a manager who had this great idea for a new tool that he wanted to introduce for his colleagues.</a> <a data-t="68">He started development on his own.</a> <a data-t="71">After a year, he realized that he would need some help from more experienced developers</a> <a data-t="76">to fix a few bugs and also implement more complex features.</a></p>

<p><a data-t="81">He decided to contact us.</a> <a data-t="81">We started working together, but he was our only point of contact.</a> <a data-t="86">He was the one deciding on the priorities.</a> <a data-t="86">He was also validating our work.</a> <a data-t="91">This is why we asked him a few times if it would be possible for us to talk with the users,</a> <a data-t="97">mostly to understand the product better, but also maybe to suggest some improvements.</a> <a data-t="103">But this manager was very hesitant about it.</a></p>

<p><a data-t="107">He always repeated, not yet, let's finish this one additional functionality,</a> <a data-t="111">or let's improve this one small feature.</a> <a data-t="115">So we continued the development.</a></p>

<p><a data-t="118">After an additional three months, we released the first version of the application.</a> <a data-t="124">Only then we started talking with the users.</a> <a data-t="128">We were able to meet with them to see how they are interacting with the application</a> <a data-t="132">and also ask them a few questions.</a></p>

<p><a data-t="135">As a result of those meetings, we prepared a report.</a> <a data-t="138">Here you can see the result of this report.</a> <a data-t="141">Nine out of ten people had problems with usability.</a> <a data-t="146">In the end, they told us that they won't be using the application in the current state.</a></p>

<p><a data-t="151">On the next slide, you can see a few more detailed comments, but they are quite similar.</a> <a data-t="155">People were mentioning that the app is overcomplicated</a> <a data-t="158">and it is very hard to use it without proper training.</a></p>

<p><a data-t="163">After over a year of development altogether,</a> <a data-t="166">we created something that people were not going to use.</a> <a data-t="169">As you can guess, the project was closed and the app was never really on the production.</a></p>

<p><a data-t="174">Of course, after a project like this, we started asking ourselves,</a> <a data-t="178">what could we have done better?</a> <a data-t="181">The first answer that we had is that we should probably talk with the users sooner.</a> <a data-t="185">Of course, even before contacting us, this manager could have some talks with the users.</a> <a data-t="189">He could just deploy the first version of the application</a> <a data-t="192">and ask them to provide feedback,</a> <a data-t="194">but maybe also we could be more pushy when asking for contact with the users.</a></p>

<h3>The case for prototyping</h3>

<p><a data-t="201">That's why currently when I'm starting any project,</a> <a data-t="205">this is the one phrase that I'm trying to follow</a> <a data-t="208">and I'm trying to convince all of the clients to release as fast as possible.</a></p>

<p><a data-t="213">To be able to do this, you need to start simple.</a> <a data-t="216">For me, the best way to be able to release fast is to start with a prototype.</a></p>

<p><a data-t="222">When I say prototype, I mean the first version of your application</a> <a data-t="226">with only a few main functionality fully developed,</a> <a data-t="229">but it is important here to focus on the default user path.</a> <a data-t="233">Whenever you have your business case,</a> <a data-t="235">just focus on the one main thing you want your users to be able to do in your application</a> <a data-t="239">and then strip it out of everything else.</a></p>

<p><a data-t="242">There are, of course, some things that can be mocked</a> <a data-t="244">to just show to the users that it is possible to implement it,</a> <a data-t="247">but it does not need to be fully functional.</a> <a data-t="250">Because only then you can have something that you can show to your users</a> <a data-t="255">in just a few weeks rather than a few months.</a></p>

<h3>Why <a href="/software/shiny-r/">Shiny</a> for prototyping</h3>

<p><a data-t="258">Now, some of you may ask me,</a> <a data-t="260">what is this amazing tool that you are using to have prototype in just a few weeks?</a> <a data-t="264">In our case, especially for data-driven applications,</a> <a data-t="268">Shiny would be the tool that we are using.</a></p>

<p><a data-t="271">There are probably many reasons that a lot of people from the Shiny team could share,</a> <a data-t="275">but for us it is, first of all, the fact that you can prototype very fast.</a> <a data-t="280">So, you don't need to have any web development knowledge.</a> <a data-t="284">It is enough to know R and Shiny will take care of everything else for you.</a></p>

<p><a data-t="288">The second point is that you have a lot of out-of-the-box packages</a> <a data-t="291">like </a><a href="/software/bslib/">bslib</a><a data-t="291"> that was just presented.</a> <a data-t="294">And also there are even ready-to-use templates that you can download</a> <a data-t="297">like on our apps and on our webpage you can find some of them.</a> <a data-t="299">And then there can be your starting point.</a></p>

<p><a data-t="303">Another point is that you can deploy very fast in Shiny.</a> <a data-t="308">It is very easy to deploy your app,</a> <a data-t="309">especially if you have Posit Connect and it's just one click of a button,</a> <a data-t="312">but there are also alternatives there.</a></p>

<p><a data-t="315">And the last point that, in my opinion, is underappreciated</a> <a data-t="319">is the ease to show the real data.</a> <a data-t="321">And, of course, on one side you have a lot of ready-to-use components</a> <a data-t="324">that are allowing you to create an interactive application</a> <a data-t="328">that your users can use to explore the data,</a> <a data-t="332">but also we can pre-process the data directly in R.</a> <a data-t="335">So, you don't need to have any additional tools</a> <a data-t="337">and we all know that usually when we are starting our project</a> <a data-t="341">the data is not perfect and it needs some cleanup or modifications.</a></p>

<h3>Why real data matters in prototypes</h3>

<p><a data-t="346">And I would like to focus a bit more on this last point</a> <a data-t="349">and tell you why I think that it is important to show the real data to the users</a> <a data-t="353">when you are even just prototyping.</a></p>

<p><a data-t="360">So, first of all, it allows the users to focus on the functionalities</a> <a data-t="366">rather than discussing the data in accuracy.</a> <a data-t="370">So, I've been in a situation that I had to present a prototype with only mock data</a> <a data-t="375">and it still brought value, but we spent a lot of time discussing</a> <a data-t="379">why this 15 is not 17.</a> <a data-t="381">And even if I explained to the users that the data is mocked,</a> <a data-t="386">then I had to explain myself where do I plan to take the data from.</a> <a data-t="391">So, in the end they were just focusing on this one part.</a></p>

<p><a data-t="395">Another point is that it is just easier to compare to the current process.</a> <a data-t="400">So, whenever you are introducing a new tool in your organization</a> <a data-t="404">you are probably trying to support or even replace a process that already exists,</a> <a data-t="410">maybe this is just sending an Excel file to your manager.</a> <a data-t="413">But then if you are showing the users the same data in both places,</a> <a data-t="417">they can easily compare those two ways of doing the same task</a> <a data-t="421">and give you a better feedback.</a></p>

<p><a data-t="423">And the last point is that it allows to select the best way to present the data.</a> <a data-t="430">So, we all know that if we would use some default sets like Iris,</a> <a data-t="435">your data will look probably decent.</a> <a data-t="437">But then when you are starting to use production data, it may differ a lot.</a> <a data-t="441">So, one of the examples I have is that we created a big table in one of my projects</a> <a data-t="446">with rows and cells colored based on some business logic.</a> <a data-t="450">And it looked very well with the mocked data, we were very happy with it.</a> <a data-t="453">But then when we started using the production data,</a> <a data-t="456">the colors were all over the place, it was just not possible to read this</a> <a data-t="460">and we had to completely redesign this feature.</a></p>

<h3>Gathering user feedback</h3>

<p><a data-t="464">So, let's now assume that we were able to build a prototype,</a> <a data-t="469">we are even able to show the real data.</a> <a data-t="473">So, what is next?</a></p>

<p><a data-t="478">So, next we need to start talking with the users.</a> <a data-t="481">And today I will be talking about two ways of gathering user feedback</a> <a data-t="485">that worked best in my project.</a> <a data-t="488">So, I will start with user interviews and then I will go to user workshops.</a></p>

<p><a data-t="493">So, when I say user interviews, I mean a set of one-on-one meetings</a> <a data-t="498">between the users and the interviewer.</a></p>

<p><a data-t="501">And here it is important, first of all, for the meetings to be really one-on-one</a> <a data-t="505">because only then the user can fully focus on the application</a> <a data-t="509">and also answer honestly for questions.</a> <a data-t="513">So, if his manager is in the room, it will be a bit more difficult sometimes to do so.</a></p>

<p><a data-t="518">The second point is that the user has to interact with the application.</a> <a data-t="523">So, this is not us showcasing the app and just explaining new features,</a> <a data-t="526">it is just sending a link to a user and asking them to perform some actions.</a> <a data-t="531">And it is good to keep the instructions as simple as possible.</a> <a data-t="535">So, for example, avoid saying something like,</a> <a data-t="538">could you please go to the page three and just edit data in the row four.</a> <a data-t="543">It is much better to focus on this business aspect.</a> <a data-t="545">So, for example, asking something like,</a> <a data-t="548">could you please modify the number of chamomile shampoos</a> <a data-t="552">in the magazine in Chicago in November.</a> <a data-t="555">Because then you can see how easy or difficult it is first for the user</a> <a data-t="559">to find the information about this chamomile shampoo,</a> <a data-t="562">but then also to modify it.</a></p>

<p><a data-t="565">And another point is to limit explanations to minimum.</a> <a data-t="569">And it may be challenging at first,</a> <a data-t="571">especially if you see people struggling with the app,</a> <a data-t="574">you want to help them.</a> <a data-t="575">But it is crucial to just observe and learn from it.</a></p>

<p><a data-t="580">And the last point here is to remember to give opportunities</a> <a data-t="584">to provide open feedback.</a> <a data-t="586">So, one thing that I'm always doing is I'm asking the users</a> <a data-t="590">to speak aloud about the thinking process.</a> <a data-t="593">Because then they can tell me not only what they are doing and why,</a> <a data-t="597">but they are also sharing their expectations quite often.</a> <a data-t="600">And this is a great source of information</a> <a data-t="602">on how we can improve your application.</a></p>

<p><a data-t="605">And also after this first part of the interview,</a> <a data-t="608">so after the user stopped interacting with the application</a> <a data-t="611">and performing those tasks,</a> <a data-t="612">it is good to save some time to ask additional questions.</a> <a data-t="616">And here the only rule is to just avoid suggesting the answer.</a> <a data-t="620">So, I would avoid questions like,</a> <a data-t="623">don't you think that this functionality is great or the most useful?</a> <a data-t="627">It is much better to ask which of the functionalities</a> <a data-t="630">is the most useful.</a> <a data-t="631">Or even better, how often are you going to use</a> <a data-t="635">different functionalities here?</a></p>

<h3>User workshops</h3>

<p><a data-t="637">So, this was the first part about those one-on-one meetings,</a> <a data-t="640">user interviews.</a> <a data-t="641">And now I will go to the second part about user workshops,</a> <a data-t="645">which is a set of meetings with a larger group of users,</a> <a data-t="650">ideally five to seven, but I think ten is still manageable.</a> <a data-t="655">And all those meetings,</a> <a data-t="656">people are working with the application individually.</a> <a data-t="659">They are then sharing their experience,</a> <a data-t="662">which is later discussed, and some improvements are suggested.</a></p>

<p><a data-t="668">And on those meetings,</a> <a data-t="669">it is good to split into sessions focused on one functionality.</a> <a data-t="675">Here you can see an example agenda from a workshop I did a few months back.</a> <a data-t="682">I just replaced the functionality names with placeholders.</a></p>

<p><a data-t="686">And you may be surprised that for each functionality,</a> <a data-t="691">we have two hours dedicated.</a> <a data-t="696">And it may seem a lot, especially that those are quite simple things,</a> <a data-t="700">like editing the data or even filtering it.</a> <a data-t="703">But it was really needed,</a> <a data-t="706">mostly because a lot happens during these two hours.</a></p>

<p><a data-t="709">So, first of all, there is some individual work.</a> <a data-t="712">We are sending the link to the users,</a> <a data-t="715">and we are asking them to, again, go through the app,</a> <a data-t="718">to focus on this one particular functionality,</a> <a data-t="721">to perform the tasks that they would normally do.</a> <a data-t="724">And then I am also in the room answering questions,</a> <a data-t="727">looking how they are interacting with the app.</a> <a data-t="729">So, it is quite similar to what I described before.</a> <a data-t="733">But then during this process,</a> <a data-t="734">they have time to note down some feedback that is later discussed,</a> <a data-t="739">prioritized, and then, again, discussed.</a> <a data-t="742">So, you can imagine that if there is like 10 people in the room,</a> <a data-t="744">probably the discussion can take some time.</a></p>

<p><a data-t="748">And another point is to give an easy way to provide feedback.</a> <a data-t="752">But you should also think about yourself here,</a> <a data-t="754">to make it easy later for you to gather it and summarize it.</a> <a data-t="759">What we usually do is,</a> <a data-t="761">we are preparing for every separate functionality</a> <a data-t="764">to be a board like this,</a> <a data-t="765">with the things that people liked in a given functionality,</a> <a data-t="770">they didn't like,</a> <a data-t="771">the things that are missing and are obligatory,</a> <a data-t="773">they have to have them to be able to use the app,</a> <a data-t="776">and the things that are missing</a> <a data-t="777">and can be implemented later somewhere in the future.</a></p>

<p><a data-t="782">What I also started doing is that I also created another space</a> <a data-t="787">called parking lot,</a> <a data-t="789">that people can add some random thoughts that they had in mind,</a> <a data-t="792">just to not interrupt during each session.</a></p>

<p><a data-t="796">And the last point is to save some time to prioritize.</a> <a data-t="802">So, after all of the sessions,</a> <a data-t="806">it is good to have additional one to just take,</a> <a data-t="809">you know, three to five cards from every board like this,</a> <a data-t="814">put them, again, in one place,</a> <a data-t="816">and prioritize them and discuss this prioritization.</a> <a data-t="820">And this discussion is extremely important.</a></p>

<p><a data-t="823">So, for example, I've been in a situation that,</a> <a data-t="826">you know, after this exercise, I looked at the cards,</a> <a data-t="828">and I saw that something that I thought is super important</a> <a data-t="831">based on the conversations was somewhere at the bottom.</a> <a data-t="834">So, I asked the users, why is this the case?</a> <a data-t="838">And they told me that this is obvious,</a> <a data-t="840">that it needs to be implemented.</a> <a data-t="842">And it wasn't obvious for me.</a> <a data-t="844">So, it would probably not be that obvious</a> <a data-t="846">for the development team as well.</a> <a data-t="849">That's why I think this time to talk about it</a> <a data-t="852">is very important.</a></p>

<p><a data-t="854">And usually the results of such workshops</a> <a data-t="856">are just a report with a list of priorities</a> <a data-t="859">and a short description about each priority,</a> <a data-t="862">sometimes also if you can add some simple visuals</a> <a data-t="867">that can help.</a> <a data-t="870">And this report is later also very useful</a> <a data-t="872">during the development.</a> <a data-t="873">So, if you have reviews with the users,</a> <a data-t="874">you can even go back to it.</a> <a data-t="876">You can discuss it and show them</a> <a data-t="877">that you are listening to them</a> <a data-t="879">and trying to implement the improvements.</a></p>

<p><a data-t="882">And I told you at the start that user workshops</a> <a data-t="885">and user interviews work best in my projects.</a> <a data-t="888">So, now I want to just defend it a bit.</a> <a data-t="891">So, first of all, it allows for a detailed discussion,</a> <a data-t="895">something that I mentioned, I think,</a> <a data-t="897">a few times during this talk.</a> <a data-t="899">But because of this, you can understand</a> <a data-t="901">not only what people want you to do, but also why.</a> <a data-t="905">And then you can suggest some alternatives.</a> <a data-t="909">But also have a bigger picture</a> <a data-t="911">and make sure that when you are designing a new feature,</a> <a data-t="914">you are taking this into consideration.</a> <a data-t="916">So, both in the architecture, but also in the UX design.</a></p>

<p><a data-t="921">Another point is that you need only five to ten users,</a> <a data-t="925">which I think is not a lot.</a> <a data-t="927">One thing to remember here</a> <a data-t="928">is that this group should be diverse.</a> <a data-t="930">So, if you are building something</a> <a data-t="932">that should work across departments,</a> <a data-t="934">probably you should have people from different departments.</a> <a data-t="937">The same from people from different regions.</a></p>
