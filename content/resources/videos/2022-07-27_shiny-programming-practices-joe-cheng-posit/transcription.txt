<h3>Reasoning locally about the reactive graph</h3>

<p><a data-t="60">So what I want to say about this reactive graph issue, where you start out with an input</a> <a data-t="66">and an output, right?</a> <a data-t="66">Not even a reactive in the middle, just input, output, multiple inputs to one output, multiple</a> <a data-t="73">outputs to one input.</a> <a data-t="75">Like all of that is pretty understandable.</a> <a data-t="77">And then you add one or two reactive calculations in the middle, but it doesn't take much for</a> <a data-t="82">that to start becoming very crazily looking, especially when you have multiple levels of</a> <a data-t="90">reactive calculations.</a></p>

<p><a data-t="91">I think like a most important thing to understand is that you're not supposed to understand</a> <a data-t="96">the whole graph.</a> <a data-t="97">The goal is not to create a graph that you look at and say, oh, that's definitely right.</a> <a data-t="102">Think of this as the same exact thing with functions in a normal piece of software, a</a> <a data-t="108">normal package, which functions call which other functions.</a> <a data-t="112">And you could draw a graph with that, right?</a> <a data-t="113">You can draw a function called graph.</a> <a data-t="115">And people do.</a> <a data-t="116">There's software that does that for you, helps you make those connections.</a> <a data-t="119">The goal is never to be able to look at that entire graph of functions pointing at each</a> <a data-t="126">other and be like, oh, yep, it's right.</a></p>

<p><a data-t="131">That's not the point.</a> <a data-t="132">If your software is well-written, the point is to be able to look at any one node, any</a> <a data-t="137">one node in that graph and prove to yourself that that node is correct.</a> <a data-t="143">To look at one output and just look at the things that are calling into that output.</a> <a data-t="149">Just one level, right?</a> <a data-t="151">Like don't go beyond the arrows that are going directly into that output and say, are those</a> <a data-t="157">correct?</a> <a data-t="157">And if the answer is yes, then you can move on.</a> <a data-t="161">You can move on to look at each of the other nodes in the graph.</a></p>

<p><a data-t="166">So the point is not to be able to reason globally.</a> <a data-t="170">The point is to be able to reason locally.</a> <a data-t="173">This is exactly the same thing that we say about functions, right?</a> <a data-t="178">When you write a function, the goal is not to be able to take the entire rest of your</a> <a data-t="184">code base that might indirectly or directly call or be called by a function that's under</a> <a data-t="190">a question.</a> <a data-t="191">The goal is to be able to look at this function and to be able to confidently say, if the</a> <a data-t="197">functions that this calls are written correctly, then this logic is correct.</a></p>

<blockquote><a data-t="191">The goal is to be able to look at this function and to be able to confidently say, if the</a> <a data-t="197">functions that this calls are written correctly, then this logic is correct.</a></blockquote>

<p><a data-t="202">So you need to be able to look at this function and reason about just this function in isolation.</a> <a data-t="208">Now you might be wrong.</a> <a data-t="209">Like you might be assuming that those other functions are written correctly and that might</a> <a data-t="213">not be true.</a> <a data-t="214">But that problem is a much better problem to have to tackle than the possibility that</a> <a data-t="220">all those functions are correctly written and you still don't know if your code is correct.</a> <a data-t="226">Because not only do you have to know that they're correct, but you have to know about</a> <a data-t="229">really subtle interactions that those functions have with each other.</a></p>

<p><a data-t="232">With reactivity, this is a way that that same quality of I can look at a small piece of</a> <a data-t="239">code and just by knowing that the reactives that are being called, if those just assume</a> <a data-t="246">that those are going to execute at the right time, then this is going to execute at the</a> <a data-t="251">right time.</a> <a data-t="252">If there's something that I'm calling and I don't want to update when it updates, then</a> <a data-t="256">I need to know to call isolate around that.</a> <a data-t="259">So not calling isolate when I don't want a reactive relationship, that would be a bug</a> <a data-t="264">in this code, right?</a></p>

<p><a data-t="266">But if I don't have that bug, if I, my relationship with my immediate things that I depend on</a> <a data-t="274">is correct, then I'm satisfied that this piece of code is correct.</a> <a data-t="278">And now I can go look at another piece of code and make sure that it's correct.</a> <a data-t="282">The bottom line being, if you have an app that's written out of 10 such pieces of code,</a> <a data-t="289">then you have to make sure that 10 pieces of code are correct.</a> <a data-t="293">And if you add an 11 piece of code, you have to make sure that 11 piece of code is correct.</a> <a data-t="297">As opposed to having to make sure that each piece of code and all of the possible interactions</a> <a data-t="302">it could ever have with any of the rest of the pieces.</a></p>

<h3>Building Shiny apps: top down vs. bottom up</h3>

<p><a data-t="606">There's so many different ways I want to answer this question.</a> <a data-t="610">Only between building the front end and the back end, I don't think there's a right order.</a> <a data-t="614">If what is going to get you excited about the project enough to sort of build that momentum</a> <a data-t="620">is that you have this cool UI concept in mind, or that just motivates you to look at something</a> <a data-t="626">concrete and see it sort of one piece after another light up, absolutely go crazy with</a> <a data-t="632">the UI first.</a> <a data-t="633">You know, I don't think there's anything wrong with that.</a></p>

<p><a data-t="654">If you want to end up with a bunch of well-named, well-reasoned, nicely factored reactives and</a> <a data-t="661">outputs and whatever, is that something you sort of do on a whiteboard first and then</a> <a data-t="667">sort of stub each one out and make it so?</a> <a data-t="670">Or is it more something that you just start coding and then extract out the things that</a> <a data-t="674">you need?</a> <a data-t="675">Again, I don't think there's a wrong way to do it.</a> <a data-t="677">But I would draw a direct analogy to when you're writing either a package or maybe a</a> <a data-t="683">complicated script.</a> <a data-t="685">When and how do you introduce functions?</a> <a data-t="687">Because it's exactly the same process that applies here as well, I think.</a></p>
