<h3>Data wrangling with Polars</h3>

<p><a data-t="144">So, first, let's talk about data exploration, which is one area where many of</a> <a data-t="150">us R users coming from a tidyverse background have really been spoiled with dplyr and </a><a href="/software/ggplot2/">ggplot2</a><a data-t="150">.</a> <a data-t="156">While many think of pandas as the go-to data wrangling language, I've actually found that</a> <a data-t="162">Polars checks many more of the boxes for me of an ergonomic tool.</a> <a data-t="162">We have that similar workflow</a> <a data-t="167">with a functional paradigm and this really expressive and expansive rich API,</a> <a data-t="172">but it isn't on its own a very stand alone impressive and successful project.</a> <a data-t="178">It's seen great adoption in the Python community, so much so that it's natively supported in Seaborn</a> <a data-t="184">and Scikit-learn, and there are those signals with the communities really latching on to it.</a></p>

<p><a data-t="188">And it's no wonder why.</a> <a data-t="188">It's zero dependencies and written in ultra fast, highly performant Rust</a> <a data-t="194">code.</a> <a data-t="194">Looking at a surface level, I think it's clear to see some of the superficial similarities</a> <a data-t="199">between </a><a href="/software/dplyr/">dplyr</a><a data-t="199"> and Polars just in the way that it provides data frame in, data frame out methods</a> <a data-t="205">with similar kind of function names.</a> <a data-t="207">And beyond those naming conventions, the feature parity is</a> <a data-t="212">also very similar, not only to dplyr, but also to the broader tidyverse.</a> <a data-t="213">So, we have nice functions</a> <a data-t="218">to support things like string and date wrangling, restructuring and reshaping our data, and working</a> <a data-t="224">iteratively.</a></p>

<p><a data-t="225">But a lot of different tools can have feature parity without having that same feel,</a> <a data-t="231">without having the ergonomics.</a> <a data-t="231">So, what I want to do is go a little bit deeper.</a> <a data-t="237">We can look at the</a> <a data-t="237">actual official tidyverse design principles and see how well those stack up in Polars.</a> <a data-t="243">We can</a> <a data-t="243">think about how our dplyr tools are composable.</a> <a data-t="248">We have these small atomic functions that allow</a> <a data-t="248">us to express really complex tasks.</a> <a data-t="254">They're also consistent.</a> <a data-t="254">So, the API that we learn in one</a> <a data-t="255">package and one function easily transports and expands our knowledge across other tools.</a> <a data-t="260">And</a> <a data-t="260">they're human centered.</a> <a data-t="261">They always kind of know what we're going for and kind of help us do that</a> <a data-t="265">super easily.</a></p>

<p><a data-t="267">So, let's see how well these hold up in Polars.</a> <a data-t="267">In dplyr, of course, we have our</a> <a data-t="271">piping and those data frame in and data frame out methods that make things very composable.</a> <a data-t="277">Really across Python with method chaining, you can achieve that similar sort of narrative flow</a> <a data-t="283">where we can read our code top down in the exact order of execution.</a> <a data-t="288">Now, in R, there's no reason</a> <a data-t="288">that we couldn't keep that piping paradigm going within each function.</a> <a data-t="294">But our coding get kind of</a> <a data-t="294">long and lengthy and we don't tend to do that.</a> <a data-t="299">Instead, we tend to go back to that traditional</a> <a data-t="299">nested function paradigm.</a> <a data-t="304">But with Polars, because we have a lot of nice methods to operate</a> <a data-t="305">on individual columns, we can actually expand that sense of composability and make our code</a> <a data-t="310">just as readable from left to right as it is from top to bottom.</a></p>

<p><a data-t="312">Similarly, in dplyr, I always value</a> <a data-t="317">the consistency both for the purposes of reading and writing.</a> <a data-t="317">All of the functions look sort of</a> <a data-t="322">similar whether we're adding new columns or subsetting rows or columns.</a> <a data-t="322">So, it's just easy</a> <a data-t="328">both to read and to write.</a> <a data-t="328">I never found the same to be as true about pandas where to do those same</a> <a data-t="333">three operations, you might have to use three very different mindsets.</a> <a data-t="338">So, using an anonymous</a> <a data-t="338">function to alter a column.</a> <a data-t="344">Then articulating how you want to subset your data with a string.</a> <a data-t="344">And then just using bracket notation.</a> <a data-t="349">It just didn't feel organic.</a> <a data-t="349">It didn't feel like language.</a></p>

<p><a data-t="352">But again, Polars really brings back the same kind of consistency to the language as dplyr with</a> <a data-t="359">a lot of just very readable code.</a> <a data-t="365">And finally, the human centeredness.</a> <a data-t="365">dplyr spoils us with so</a> <a data-t="370">much syntactic sugar and I feel like Polars is very the same in this regard as well.</a> <a data-t="375">Just one</a> <a data-t="375">example I'll point to here is Polars implementing select helpers which lets us easily, much like in</a> <a data-t="382">R, identify variables by a part of a variable name or a variable type and apply the same</a> <a data-t="389">transformation to all of those variables at once instead of duplicating our code.</a></p>

<p><a data-t="385">Similarly, Polars also knows that as data scientists, we work with data of very different</a> <a data-t="389">sizes and in very different places.</a> <a data-t="395">So, whether we're reading data in batch or streaming, in memory</a> <a data-t="395">or out of memory, we can run roughly the same Python code and let Polars handle the details</a> <a data-t="401">of optimizing that workflow for us.</a></p>

<h3>Visualization with <a href="/software/plotnine/">plotnine</a> and Seaborn objects</h3>

<p><a data-t="407">But of course, data wrangling is only one half of exploration.</a> <a data-t="412">We also have visualization and here we actually now have two really great options.</a> <a data-t="412">First, there's plotnine which is a very high fidelity code clone of ggplot that is well</a> <a data-t="418">supported by Posit.</a> <a data-t="423">Sometimes I worry a little bit about a lot of packages that try to clone</a> <a data-t="423">existing packages, that they may not be well supported, they may be hobby projects,</a> <a data-t="433">but having that institutional support is huge.</a> <a data-t="433">On the other hand, we have Seaborn objects</a> <a data-t="438">which lives inside Seaborn, one of the most popular Python packages for data visualization,</a> <a data-t="445">and this provides a novel experimental object-oriented programming interface</a> <a data-t="450">with unique ggplot2 inspirations that allows us an alternative way to bring the grammar of graphics</a> <a data-t="456">into creating the underlying matplotlib figure specifications.</a></p>

<p><a data-t="459">So, again, if we think about</a> <a data-t="459">what makes ggplot2 special, we have that sense of a grammar mapping different components in our data</a> <a data-t="465">to different aesthetic elements and doing that incrementally in layers where we can control each</a> <a data-t="470">layer as much as we want.</a> <a data-t="476">It shouldn't be hard to convince you that we can achieve that thing</a> <a data-t="476">in plotnine because this plotnine code, believe it or not, is actually Python code, even though it</a> <a data-t="481">looks almost identical to the ggplot2 code that we know and love.</a> <a data-t="487">But if we want, we can get a</a> <a data-t="487">little more creative and give that a little bit of its own Pythonic flavor.</a> <a data-t="492">Using Python's</a> <a data-t="492">incremental addition operator, we can still use all the same ggplot2 functions we know,</a> <a data-t="497">but break our code up a little bit more without that method chaining to make it easier to debug</a> <a data-t="502">and experiment with.</a></p>

<p><a data-t="562">And always, any of these tools, you always worry, what if I reach the limit of this tool?</a> <a data-t="567">Will I get kind of stuck and stranded if I'm using a tool that isn't the language standard?</a> <a data-t="572">But both of these tools make it really easy to get out that underlying matplotlib figure.</a> <a data-t="577">So,</a> <a data-t="577">if we need to customize further, we can always fill up our sleeves and do so.</a> <a data-t="583">We'll never get</a> <a data-t="583">stranded.</a></p>
