<h3>Explain plans, distribution styles, and sort keys</h3>

<p><a data-t="326">Imagine for a second that you're going to explore a dark cave.</a> <a data-t="330">What items might you want to help you?</a> <a data-t="336">For me, I would want a map.</a> <a data-t="338">I would want to know where I'm going.</a> <a data-t="341">Does this cave even have an exit?</a> <a data-t="343">How many tunnels are there?</a></p>

<p><a data-t="346">I would also want a team or a cavern crew.</a> <a data-t="349">I would want people to help carry equipment and make the journey easier.</a> <a data-t="356">It's also very dark in this cave, so a flashlight would be helpful to be able to see where I'm</a> <a data-t="359">going.</a></p>

<p><a data-t="365">An explain plan is like a map.</a> <a data-t="368">It returns a step-by-step list of actions that your query will take to return your dataset.</a> <a data-t="376">It also returns the estimated computational cost of that query.</a> <a data-t="382">It's very easy to run.</a> <a data-t="384">You only have to put the word explain in front of your query and run it.</a></p>

<p><a data-t="390">The first time I saw an explain plan, I was very overwhelmed.</a> <a data-t="393">I swear my head exploded.</a> <a data-t="396">But it turns out that it's a very helpful tool because that computational cost can help</a> <a data-t="401">you understand if changes that you're making to your query are helpful.</a> <a data-t="406">So in my instance, where I would run my query, wait an hour, kill it, run it, wait an hour,</a> <a data-t="413">kill it, if I had run explain plans, I could see that the changes that I was making would</a> <a data-t="420">reduce the computational cost without having to wait that extra hour.</a> <a data-t="424">So explain plans are very useful.</a></p>

<p><a data-t="430">For our cavern crew or cave team, we want everyone to be contributing equally.</a> <a data-t="438">We don't want one person to be carrying all of the backpacks and the other people to not</a> <a data-t="442">be carrying anything.</a> <a data-t="444">That doesn't seem, one, very nice, or two, very efficient.</a></p>

<p><a data-t="450">From a query perspective, compute nodes are like your cave team.</a> <a data-t="455">You want all of your compute nodes contributing equally and leveraging all of their resources.</a> <a data-t="463">To make sure that happens, you need to appropriately specify the distribution style.</a> <a data-t="471">Distribution means that your data is properly distributed across the compute nodes and</a> <a data-t="475">does not have significant skew.</a></p>

<p><a data-t="481">But how do you do that?</a> <a data-t="482">How do you know what's the right distribution style?</a> <a data-t="486">You think about which column you're going to join on.</a></p>

<p><a data-t="493">If you're planning on joining on a unique ID, such as an order ID or a customer ID,</a> <a data-t="500">then dist style key is what you want to use.</a> <a data-t="504">And the choice of your dist key depends on which column you are going to use to join</a> <a data-t="509">to another dataset.</a></p>

<p><a data-t="513">Dist style all is very dangerous.</a> <a data-t="517">It copies your data across each of the compute nodes.</a> <a data-t="521">So in the case where you're not going to join your table to anything ever and it's a small</a> <a data-t="527">dataset, you can get away with dist style all, but I personally wouldn't recommend it.</a> <a data-t="534">And if you're not sure what you're going to join on, maybe you have a table that's a lot</a> <a data-t="538">of rows, but it doesn't have a good unique ID or a candidate for a dist key, and maybe</a> <a data-t="546">it's a combination of fields that's making it have a lot of rows, then dist style even</a> <a data-t="550">is a good candidate.</a></p>

<p><a data-t="553">And the beauty of the explain plan is that when we make these changes to our table and</a> <a data-t="559">try to run our query, we can run an explain plan to know if those changes actually made</a> <a data-t="564">a difference.</a></p>

<p><a data-t="577">Your sort key is like a flashlight.</a> <a data-t="580">It helps you only search relevant data.</a> <a data-t="583">Imagine that you have a dataset that has three years of data, 2020, 2021, and 2022.</a> <a data-t="591">I only care about 2022.</a> <a data-t="592">I'm living in the present.</a> <a data-t="594">So I have a where clause that filters on where year equals 2022.</a> <a data-t="600">And my result set would be filtered.</a></p>

<p><a data-t="603">However, if I'm not, if the table I'm querying from does not have a sort key, even though</a> <a data-t="611">my result set is filtered, the query had to scan the entire table to find it, which is</a> <a data-t="618">not very efficient.</a> <a data-t="621">And so when you specify a sort key, it helps your query know immediately where to find</a> <a data-t="625">the data that you're looking for, so it reduces the computation needed to pull back your result</a> <a data-t="630">set.</a></p>

<p><a data-t="635">To decide on a sort key, you think about which column you will filter on.</a> <a data-t="640">A lot of times that's a date.</a></p>

<p><a data-t="709">Pulling together everything that we have learned, on the left-hand side, I have a query that</a> <a data-t="714">takes that page view level data that we were just looking at and aggregates it to the session</a> <a data-t="719">level.</a> <a data-t="722">If you can imagine landing on a website, you can have multiple page views during a session,</a> <a data-t="726">and so I cared about what marketing is at the session level.</a></p>

<p><a data-t="765">On the right, I have a query that leverages the table that I just created and joins it</a> <a data-t="769">to another table called web orders.</a> <a data-t="774">And I'm joining to that table on user ID and session ID.</a> <a data-t="781">Because we have specified session ID as a distribution key, all of the compute nodes</a> <a data-t="787">will be used in this join, and it will make it run much more efficiently.</a> <a data-t="794">Because we have the sort key specified as session start time, I'm filtering on anything</a> <a data-t="798">greater than January 1st of 2022, and so therefore, my query doesn't have to search for data that</a> <a data-t="805">it doesn't care about.</a> <a data-t="807">And we've saved time there as well.</a></p>
