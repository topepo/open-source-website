<h3>readxl and the <a href="/software/tidyverse/">Tidyverse</a></h3>

<p><a data-t="61">So readxl is a package that finds itself inside a larger group of packages, which we're calling</a> <a data-t="71">the Tidyverse, and this I believe is still a subset of the Tidyverse, but the idea of</a> <a data-t="77">the Tidyverse is to take a collection of packages and make sure that they operate with a common</a> <a data-t="83">philosophy and a really similar interface so that it's easy to transfer expertise from</a> <a data-t="91">one package to the other.</a> <a data-t="92">It's easy to move objects from one package to another, and so readxl is the Tidyverse</a> <a data-t="101">solution for getting data out of Excel spreadsheets and into R.</a></p>

<h3>History of readxl</h3>

<p><a data-t="197">So it was first released on CRAN in April of 2015, and that release was written entirely</a> <a data-t="204">by Hadley, and then about a year later it had a patch release, which I think was just</a> <a data-t="209">triggered by the need to update for compatibility with the package, so that didn't have a lot</a> <a data-t="215">of changes in it.</a> <a data-t="216">And so then it got its first major maintenance release about a month ago, which is when I</a> <a data-t="244">took over as the new maintainer and we moved the version up to one.</a> <a data-t="248">I feel like it is pretty feature complete, and we squashed lots of bugs and added a few</a> <a data-t="255">new features, which I will talk about today.</a></p>

<p><a data-t="261">So one way to get Read Excel installed on your computer is to install the entire Tidyverse.</a> <a data-t="267">So we now have a package called Tidyverse, which is kind of different from the concept</a> <a data-t="272">of the Tidyverse.</a> <a data-t="272">It's a meta package that will install the whole set of packages on your computer, or</a> <a data-t="277">you could install Read Excel all by itself.</a> <a data-t="283">Then either way, you must load Read Excel individually.</a> <a data-t="288">You can actually call library Tidyverse, and it will load a subset of the Tidyverse packages</a> <a data-t="295">that are kind of called the core.</a> <a data-t="298">And Read Excel is not part of that core, because you wouldn't need it in almost every analysis.</a> <a data-t="303">It's only when you have a spreadsheet.</a> <a data-t="305">So you'll always have to load it explicitly on its own.</a></p>

<h3>The landscape of Excel-reading packages</h3>

<p><a data-t="322">So I promised in my abstract to give a little bit of the landscape of our packages that</a> <a data-t="327">Read Excel lives in.</a> <a data-t="331">And the reason I wanted to do that is I figure anyone who's willing to spend an hour with</a> <a data-t="336">us talking about Excel must have some slightly non-trivial need to read Excel files.</a> <a data-t="343">And it's kind of interesting just to know where the packages fit in and why some packages</a> <a data-t="347">are harder to use than others or what the different features are.</a></p>

<p><a data-t="352">So there's a huge set of packages that read Excel files, because this has been an important</a> <a data-t="357">area for a very long time, obviously.</a> <a data-t="360">So Read Excel might actually be the most recent entry into this group.</a> <a data-t="368">OpenXLSX is also fairly recent, and it uses a lot of the same stuff under the hood as</a> <a data-t="375">Read Excel.</a> <a data-t="377">Packages that have been around much longer are XLConnect, XLSX, and GData, RODBC.</a></p>

<p><a data-t="386">So when you're thinking about where does Read Excel fit in and which of these packages</a> <a data-t="388">are relevant to me, I think there's at least three things you need to consider.</a> <a data-t="395">So one, there's two broad classes of Excel files.</a> <a data-t="401">And unless someone's been evil, you'll be able to identify them from their file extension.</a> <a data-t="406">So .xls are the older Excel files.</a> <a data-t="410">In our documentation, we call them legacy files.</a> <a data-t="413">And those are files from Excel up to about 2007.</a> <a data-t="418">But there's plenty of systems today that are still producing .xls files, and in fact, Excel</a> <a data-t="423">itself is still quite content to produce these files.</a> <a data-t="427">And then the more modern file format is XLSX.</a> <a data-t="433">And that's what you're going to get by default coming out of Excel now.</a></p>

<p><a data-t="447">So the main question to ask is not about the files, but it's about the R package.</a> <a data-t="451">And there's sort of two main categories of R packages.</a> <a data-t="451">There are those that are self-contained versus the ones that have an external dependency.</a> <a data-t="457">So neither of these file formats is something that you want to parse by hand, all by yourself</a> <a data-t="468">from the ground up.</a> <a data-t="469">So all of the packages that read Excel files are standing on the shoulders of others and</a> <a data-t="475">have some sort of dependency where we're bringing in an existing library to parse the file.</a></p>

<p><a data-t="481">And it turns out that the most comprehensive package for doing that is a Java system called</a> <a data-t="490">the Apache POI.</a> <a data-t="493">And so the packages that wrap Java have great functionality in the sense that you can almost</a> <a data-t="500">say if Excel can read this file, this package can read it because of all the accumulated</a> <a data-t="507">conventional wisdom in Apache POI.</a> <a data-t="510">And so that's why when you're writing an Excel reading package for R, it's very tempting</a> <a data-t="516">to use Java because you can take advantage of this pretty amazing resource.</a> <a data-t="521">And likewise, there are some sort of Perl scripts that have been around for a long time</a> <a data-t="526">that have a lot of smarts about reading Excel files.</a> <a data-t="530">And so we've taken another approach in ReadExcel, which is to bake everything you need into</a> <a data-t="534">the package.</a></p>

<p><a data-t="536">And then the third huge component to think about is whether the package you're using</a> <a data-t="541">is oriented towards bringing in a rectangular piece of data, so a data frame usually, or</a> <a data-t="549">is it more focused on or capable of focusing on individual cells and formatting.</a></p>

<p><a data-t="561">So here's where ReadExcel comes out on all of these points.</a> <a data-t="565">It is able to read XLS and XLSX.</a> <a data-t="567">It is self-contained.</a> <a data-t="568">So once you get it installed, which should be relatively trivial, and definitely let</a> <a data-t="574">us know if it is not, it will just work.</a> <a data-t="578">You don't need to configure an external dependency in the right place for it to work.</a> <a data-t="582">And it is explicitly focused on bringing in a rectangle of data.</a> <a data-t="587">So it is not a package that's going to reveal formatting for you, whereas some of those</a> <a data-t="593">others will.</a></p>
