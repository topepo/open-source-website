<h3>Package dependency conflicts</h3>

<p><a data-t="1837">Okay, so now that we've covered packages, libraries, and repositories,</a> <a data-t="1840">let's talk about the not-so-fun stuff.</a> <a data-t="1843">And that is package dependency conflicts,</a> <a data-t="1846">or basically what happens when things go wrong.</a> <a data-t="1849">So let me run through a scenario with you.</a> <a data-t="1852">I'm going to pick a random date. This is completely arbitrary.</a> <a data-t="1855">We'll say January 1, 2020.</a> <a data-t="1858">You open up R and RStudio, and you install a package.</a> <a data-t="1861">We'll focus on the tbl package, which some of you may be familiar with.</a> <a data-t="1865">So we go and install .packages, the tbl package.</a> <a data-t="1869">It is very common in R, and also in Python,</a> <a data-t="1872">for packages to depend on other packages.</a> <a data-t="1876">So by installing the tbl package, it actually, by default,</a> <a data-t="1880">will install three dependencies, the </a><a href="/software/rlang/">rlang</a><a data-t="1880">, cli, and cran packages.</a></p>

<p><a data-t="1889">Now, again, on this date, we installed tbl and all those dependencies,</a> <a data-t="1892">so they're all being installed at the exact same time.</a> <a data-t="1895">And there's actually a cool thing that CRAN does.</a> <a data-t="1898">So again, that big repository of R packages.</a> <a data-t="1901">CRAN always tests to ensure that the latest packages</a> <a data-t="1904">always work together.</a> <a data-t="1907">So if you install these packages on the exact same date,</a> <a data-t="1911">all the packages and the versions and dependencies,</a> <a data-t="1914">they should all work nicely together. You shouldn't get any conflicts.</a></p>

<p><a data-t="1917">But let's fast forward a month.</a> <a data-t="1920">I'm just taking some arbitrary time.</a> <a data-t="1923">And you go back into that same R environment where you install tbl.</a> <a data-t="1926">And you now install another package,</a> <a data-t="1929">and we'll focus on the packagedown package.</a> <a data-t="1932">So you install packagedown. It turns out packagedown</a> <a data-t="1935">actually has the same three dependencies,</a> <a data-t="1938">rlang, </a><a href="/software/cli/">cli</a><a data-t="1938">, and cran.</a> <a data-t="1941">But because this is being installed a month later,</a> <a data-t="1944">it turns out there's actually some new versions of these various dependencies,</a> <a data-t="1947">which is why they're kind of shaded in this darker gray color.</a> <a data-t="1950">So by installing packagedown on February 1st,</a> <a data-t="1953">it'll automatically update the rlang, cli, and cran packages.</a> <a data-t="1956">But what it will not do is update the tbl package itself.</a> <a data-t="1959">And now you're in a bit of a broken state</a> <a data-t="1962">because you have an old version of tbl</a> <a data-t="1965">that's trying to use new versions of its dependencies,</a> <a data-t="1968">and you may run into some issues where functions aren't behaving as expected.</a></p>

<p><a data-t="1971">So how do we get around this?</a> <a data-t="1974">Well, I mentioned before that CRAN tests to ensure</a> <a data-t="1977">that the latest packages always work together.</a> <a data-t="1980">So because of that, it's very common for data science teams</a> <a data-t="1983">to do something what's known as freezing a repository</a> <a data-t="1986">to a set date.</a> <a data-t="1989">So let me explain how this works.</a> <a data-t="1992">So again, we have two dates right here, and we have January 1st, 2020.</a> <a data-t="1995">And let's say that you as the user or your administrator</a> <a data-t="1998">decides to freeze your repository</a> <a data-t="2001">to this date, February 1st.</a> <a data-t="2004">You install tbl, things go great, and then you fast forward</a> <a data-t="2007">to February 1st. But even though it's February 1st,</a> <a data-t="2010">your repository is still frozen to January 1st.</a> <a data-t="2013">So you go to install packagedown in February,</a> <a data-t="2016">but it still installs it as if it's January 1st.</a> <a data-t="2019">So no matter when you installed any subsequent packages,</a> <a data-t="2022">they're all being installed as if it's the same time.</a> <a data-t="2025">So all these packages, again, should work together</a> <a data-t="2028">because that's the added service that CRAN provides.</a></p>

<h3>Package management strategies</h3>

<p><a data-t="2031">Okay, now that we've talked about a bunch of different things,</a> <a data-t="2034">you might have some questions kind of circulating in your head</a> <a data-t="2037">as you start to devise your own package management strategy.</a> <a data-t="2040">You know, things like who is going to be responsible</a> <a data-t="2043">for reproducing the environment?</a> <a data-t="2046">Is that going to be up to the data scientists, the users,</a> <a data-t="2049">or is it going to be up to my system administrators?</a> <a data-t="2052">Also, how open is the environment? Should we be using these shared</a> <a data-t="2061">system libraries or just specific user libraries?</a> <a data-t="2064">And really, you have to weigh the pros and cons</a> <a data-t="2067">specific to your team. It's nice to talk to</a> <a data-t="2070">other teams as well and to kind of see what they're doing, but ultimately</a> <a data-t="2073">it's just to find a solution that kind of meets your team's</a> <a data-t="2076">needs.</a></p>

<p><a data-t="2076">Now having said that, we have worked with a bunch of teams here at Posit</a> <a data-t="2082">where we've seen package management strategies succeed</a> <a data-t="2085">and we've seen some strategies not so succeed.</a> <a data-t="2088">And so what you're seeing on this plot right here,</a> <a data-t="2091">we're showing you a few different strategies that we're going to</a> <a data-t="2094">touch upon. On the y-axis, we're looking at package</a> <a data-t="2097">access. So that's asking, can a user</a> <a data-t="2100">install whatever package they want? So is it open,</a> <a data-t="2103">or is it fairly locked down? So they're kind of constrained to specific</a> <a data-t="2106">packages or specific versions. And then</a> <a data-t="2109">on the x-axis, we're looking at who is responsible for</a> <a data-t="2112">reproducing the environment. Is that going to be up to your system</a> <a data-t="2115">administrators, or is it going to be up to the data scientists?</a></p>

<p><a data-t="2118">We find teams have the most success</a> <a data-t="2121">somewhere along this diagonal. And we're</a> <a data-t="2124">actually going to talk about these three strategies. We're going to start with snapshot,</a> <a data-t="2127">then we'll talk about shared baseline, and then finally a validated</a> <a data-t="2130">strategy. And then we'll just briefly touch</a> <a data-t="2133">upon some of these other strategies where we see things go a little</a> <a data-t="2136">bit haywire.
