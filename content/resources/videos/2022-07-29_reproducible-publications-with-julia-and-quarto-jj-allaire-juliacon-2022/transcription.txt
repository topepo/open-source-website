<h3>What is Quarto?</h3>

<p><a data-t="72">So I'd start with the very basics, which is what specifically is Quarto.</a> <a data-t="78">You might actually be familiar with other systems that are similar to Quarto.</a> <a data-t="86">I would call it a literate programming system in the tradition of Org Mode or Sweave or in Julia you</a> <a data-t="86">may have seen Weave.jl, R Markdown, JupyterBook.</a> <a data-t="94">Basically the fundamental idea is that we want to create a scientific and technical publishing system that uses markdown and enhances it with</a> <a data-t="100">features that are required for scientific communication.</a> <a data-t="105">So first and foremost of those is the incorporation of computations into documents and publications.</a> <a data-t="111">Again at the core we're going to use markdown.</a> <a data-t="117">Traditionally in scientific communication there's a substantial tradition of using LaTeX and as you'll see the flavor of markdown that is used within Quarto</a> <a data-t="123">and within Pandoc borrows a lot from LaTeX and in fact is able to incorporate LaTeX directly.</a> <a data-t="129">And then we'll talk about the again as I said the different types of output that can be created.</a></p>

<p><a data-t="134">So where did this project come from?</a> <a data-t="141">It's an open source project.</a> <a data-t="141">It's sponsored primarily by RStudio and it actually comes out of about 10 years of experience with another system,</a> <a data-t="147">a similar system called R Markdown.</a> <a data-t="152">That as I said that was actually announced 10 years ago almost to the day here.</a> <a data-t="160">And we worked on R Markdown for about 10 years and I developed a lot of good sound core ideas and folks in the R community got a huge amount of value out of the system.</a> <a data-t="173">But as we're all aware the number of languages and runtimes and environments used for scientific discourse is very broad.</a> <a data-t="180">There's R, there's Julia, there's Python, there's Scala, and there will be many more.</a> <a data-t="187">And what we really wanted to do was take all of the experience you had with R Markdown, build a new system kind of reimagined with the benefit of hindsight and make</a> <a data-t="194">that system fundamentally multi-language and multi-engine.</a> <a data-t="201">So not tied to R in any way and not tied to Python in any way or Julia in any way.</a> <a data-t="207">It sort of is able to work with lots of different languages and engines that are available now and in the future.</a></p>

<h3>Goals of the project</h3>

<p><a data-t="214">And so I also before I get into more of the mechanics and the how and the what I wanted to talk a little bit about the goals of the project.</a> <a data-t="226">One is this idea of computational documents and that is documents that incorporate the source code required for their production directly inside the document.</a> <a data-t="237">And this can take the form of notebooks which we're all familiar with but also there are plain text flavors of computational documents.</a> <a data-t="243">The goal here is principally reproducibility.</a> <a data-t="247">The ability to replicate the studies and documents and analyses that we create.</a> <a data-t="251">And also automation.</a> <a data-t="251">Automation leads to reproducibility but it's also a significant practical benefit to working this way.</a> <a data-t="261">So we want to provide a system that makes it easier than not to create computational documents.</a></p>

<p><a data-t="268">We have this also this idea of scientific markdown and if I'm sure many or most of you are familiar with preparing scientific manuscripts and you can see here if you try to use a tool like Microsoft Word it might be relatively smooth going at the beginning but the requirements of technical</a> <a data-t="301">documents quickly escalate and make a tool like Word quite unwieldy.</a> <a data-t="308">You may have used LaTeX which starts out you can see the red line here at a higher bar of difficulty of use but it's a relatively flat curve once you once you climb up to that bar.</a> <a data-t="317">So LaTeX has been a great system for creating technical documents but it is not the most accessible.</a> <a data-t="328">Markdown is in some ways has some similarities to LaTeX in that it's a plain text format that's sort of compiled or rendered to a final document and it's not nearly as capable as LaTeX.</a> <a data-t="340">It's easier to start with kind of approaching as easy as using something like Word but then it it doesn't quite have the features and functionality of LaTeX.</a> <a data-t="352">So our goal here really is to take Markdown, hopefully take that line all the way down to where Word starts, make it extremely easy to work with Markdown but then</a> <a data-t="357">have a system whose complexity scales a lot more like LaTeX.</a> <a data-t="363">Once you learn the basics of it then it's very straightforward to do sophisticated things.</a></p>

<blockquote><a data-t="363">Once you learn the basics of it then it's very straightforward to do sophisticated things.</a></blockquote>

<p><a data-t="369">And another goal is is this idea of single-source publishing.</a> <a data-t="374">The content that we create oftentimes needs to go to multiple variations of HTML, they may need to go to print, it may need to go to become a Word document or presentation, EPUB books, there's lots of different ultimate locations that we need to deploy our content to and we really like to write it one time.</a> <a data-t="395">And that's what a system like Markdown or an approach to publishing that is afforded by Markdown leads to.</a></p>

<h3>Using Quarto with Julia</h3>

<p><a data-t="1052">All right, so let's talk a little bit about iJulia.</a> <a data-t="1052">If Quarto sees a Julia code cell inside your document, it's automatically going to assume that it wants to use the iJulia Jupyter kernel.</a> <a data-t="1066">It'll find the most recent version of the Julia kernel on your system, but you can also pick a specific version by specifying in this example, for example, Julia 1.7 directly.</a> <a data-t="1080">iJulia executes Julia code and then transforms it to either plain text, graphics, markdown, HTML.</a> <a data-t="1087">It knows how to take Julia output and render it into something that can then be published.</a> <a data-t="1092">One piece that I'll get into in a little more depth in a minute is that for interactive sessions, we don't want to absorb the startup time of the kernel for every document render, so Quarto will keep the Jupyter kernel resident to mitigate this, and then revise is used to make sure that if there are changes to dependent files or packages that occur while that long-running session exists, that they're updated and refreshed.</a></p>

<h3>Managing performance</h3>

<p><a data-t="1116">I want to talk a little bit about how we manage two types of performance.</a> <a data-t="1116">So one is startup performance, which is how long does it take to load the interpreter and packages and how often I need to do that, and then rendering performance, how expensive are computations and how frequently do we need to run them.</a> <a data-t="1128">Startup performance, I talked about a little bit of this idea of keeping a kernel daemon around to mitigate startup costs.</a> <a data-t="1139">That HelloQMD example from earlier takes about 30 seconds on first run on my machine, but it takes less than a half a second on subsequent runs, so keeping that kernel around is really valuable, but then that creates the problem of stale code, and that, again, the solution of that is revise, which I'm sure most of you are familiar with for just long-running REPL sessions, and you can add revise so that it always runs inside iJulia by adding this code.</a> <a data-t="1176">This is all documented on the revise website, adding this code to the iJulia startup.</a> <a data-t="1176">So revise, people have found to be a great addition to iJulia when working with Quarto.</a></p>

<p><a data-t="1182">Thinking about rendering performance, you know, and this has nothing to do with startup performance, this is just how long do my computations take, and if I'm working iteratively on a document and focusing on content, I don't always want to rerun on my computations, so there's a few approaches to that.</a> <a data-t="1196">One is just authoring inside a notebook allows you to control exactly when code execution occurs and actually cache the results in the notebook, so that's one approach.</a> <a data-t="1209">Jupyter Cache, which is a package you can install separately, will actually do caching of all your cell outputs.</a> <a data-t="1215">It's all or nothing, so if any of your code changes it's got to re-execute the whole document, but again, if you've done your computations and now you're writing and doing analysis, this allows you to re-render with zero computations.</a> <a data-t="1224">And then I won't get into all the mechanics of it, but Quarto has a freeze feature that allows you to also, separate from cache, it's a little more explicit and durable, you can permanently save and reuse computational output.</a> <a data-t="1237">So for example, a blog post that you wrote three years ago, you don't want to have to keep re-rendering that, and so that can be saved.</a> <a data-t="1247">Or if you're deploying code to a server that may or may not have the permissions and software required to render, you may want to render everything locally, have it freeze, and then deploy the kind of frozen execution results onto the server.</a></p>

<p><a data-t="1267">We picked iJulia mostly because we had made a bunch of investments to make the working of Quarto and Jupyter to work really well.</a> <a data-t="1274">One was this kernel demonization and caching work.</a> <a data-t="1282">iJulia also had implemented a lot of primitives for supporting MIME outputs from Julia results, and that's including the ability to output raw LaTeX, which ends up being important for some more sophisticated tables, and the fact that as a format, IPyMB was supported in a bunch of popular notebook front ends, JupyterLab, VS Code, etc.</a> <a data-t="1307">So I think it aligned well with a lot of infrastructure that both we had and that users were taking advantage of.</a> <a data-t="1312">But it's definitely conceivable that the other notebook or literate programming systems like Pluto or Neptune, they could be integrated as an alternative to iJulia.</a> <a data-t="1318">So that's something we'd certainly be interested in talking about.</a> <a data-t="1329">The execution engine system is intended for future extensions, and it's pluggable.</a> <a data-t="1333">So it's something that we could definitely do.</a></p>
