<p><a data-t="838">So how do you exactly make maintainable code because that's kind of a magic thing.</a> <a data-t="845">These are principles that are, well, not necessarily for Shiny but for any app that you're developing in any software.</a> <a data-t="850">It's important that you write code which is easy to understand and debug, meaning that you think about coding style, commenting your code, keep it stupid simple, the KISS principle.</a> <a data-t="861">So that is always easy to understand.</a> <a data-t="861">You can choose to write really complex code but that honestly never helped us because if you developed something very complex with a very complex piece of code and then look at it a couple of months later, you're like, I don't know what was actually happening here, some magic.</a></p>

<p><a data-t="1873">So we've written a code, and we've made it maintainable, we have built a structure, and we have thought about user experience.</a> <a data-t="1883">So we've deployed it somewhere.</a> <a data-t="1883">So now it's time to go look into a bit more detail into how we can optimize Shiny, to kind of make sure that everybody who wants to access the application at a given point in time can also do so.</a> <a data-t="1903">And the first thing I'm going to talk about with you is process configuration.</a> <a data-t="1908">And I quickly touched upon this when I said that we always have five R processes running for our Shiny app and for our Plumber API.</a> <a data-t="1914">Because, as you might know, a Shiny app or a Plumber API requires an R process, meaning that R needs to be up and running, and that needs to be ready to do stuff.</a></p>

<p><a data-t="1924">And you can configure these kind of settings.</a> <a data-t="1928">And these kind of settings that I mentioned here, maximum processes, minimum processes, maximum connections per process, load factor, you can set them in most cases yourself.</a> <a data-t="1939">So in </a><a href="/software/shinyapps/">shinyapps</a><a data-t="1939">.io, which is not an enterprise solution, but if you're still there, these settings will also be there in RStudio Connect.</a> <a data-t="1949">And it's just very important, also for if you are deploying your app in for any other cloud service provider, it's important to understand how this works and how you are kind of redirecting your users to an R process.</a></p>

<p><a data-t="1966">Because a Shiny app requires an R process.</a> <a data-t="1970">Okay, so then there is an R process in the Shiny app.</a> <a data-t="1970">And then there's also a user.</a> <a data-t="1976">So a user can basically hop on to that R process.</a> <a data-t="1976">And you can say, okay, I want to have only one user for our process, or I want to have multiple and that is kind of going to determine your scalability of the application.</a> <a data-t="1991">So things that you can tweak, for example, are the maximum number of processes that you are allowing to run for your Shiny app.</a></p>

<p><a data-t="2014">The minimum number of processes means you have, as in our case, a minimum number of processes running for your Shiny app at any point in time.</a> <a data-t="2034">So if you set this higher, let's say 10, 15, 15, it means that you also consume more resources.</a> <a data-t="2039">Because all these processes are running, they consume memory of your server.</a> <a data-t="2044">So the higher you set this, the more resource consuming it will be.</a> <a data-t="2049">So you need to think about that.</a> <a data-t="2049">In our case, we chose for five minimum processes.</a> <a data-t="2054">Because if we're looking at the number of users that are coming at the simultaneous time to the app, and are doing stuff, this is more than enough.</a></p>

<p><a data-t="2060">But if you have the minimum number of processes set, it means that there's always an R process running for your application, meaning that the startup in time will increase drastically.</a> <a data-t="2074">Because and that's kind of the downside of building a Shiny app, the R process needs to be startup.</a> <a data-t="2082">So besides loading your app, it also needs to load an R session.</a> <a data-t="2086">So together, that can take a lot of time and people don't like to wait.</a> <a data-t="2092">And especially if you are building apps for a larger audience, apps that people pay for, you have to have high expectations, people can't wait 40 seconds for the app to load, needs to be 10 seconds max.</a></p>

<h3>Smart coding: cache, background processes, and Feather</h3>

<p><a data-t="2234">So that's one thing that you can optimize.</a> <a data-t="2234">But most of the game really is to be made in the app itself.</a> <a data-t="2239">So how are you going to make your app fast?</a> <a data-t="2239">How are you going to make your app enjoyable?</a> <a data-t="2245">Because that is part of building enterprise-grade apps.</a> <a data-t="2245">They need to work great.</a> <a data-t="2245">People expect a lot from them.</a></p>

<p><a data-t="2253">So smart coding.</a> <a data-t="2253">How are we going to speed up our Shiny application?</a> <a data-t="2259">One way we can do that is by using cache.</a> <a data-t="2259">And cache is an interesting way of kind of reducing the time it takes to get output.</a> <a data-t="2267">So imagine that people can hit a button in your Shiny app.</a> <a data-t="2271">And once they hit that button, there will be a very lengthy calculation running.</a> <a data-t="2276">I don't know.</a> <a data-t="2276">Maybe you have some kind of machine learning model or something.</a> <a data-t="2282">But they choose one input.</a> <a data-t="2282">And the calculation runs and it gets an output.</a> <a data-t="2287">It's obviously a shame if then a second user comes in, chooses the exact same input, does the calculation again, and gets the exact same output again.</a> <a data-t="2293">It's kind of a waste of time.</a> <a data-t="2293">Why should you do that calculation twice?</a></p>

<p><a data-t="2302">So this is where cache comes in.</a> <a data-t="2302">Because what you also can do is say, okay, well, the first user requested a result for this input.</a> <a data-t="2311">Input A, we call it.</a> <a data-t="2311">We do that lengthy calculation.</a> <a data-t="2315">Yes, it takes some time.</a> <a data-t="2315">But for the next ones, the next users, we're not going to do that.</a> <a data-t="2320">No.</a> <a data-t="2320">Instead, we're going to immediately use the results that were being cached from that calculation.</a> <a data-t="2326">So cache is basically a way of storing your output somewhere, in this case, on your server or wherever you choose to store it.</a> <a data-t="2333">And then retrieving it from storage instead of doing the calculation.</a></p>

<p><a data-t="2338">So especially if you have a large audience for your app, it can save a lot of time.</a> <a data-t="2344">And especially if a large audience is doing many of the same things.</a> <a data-t="2351">And not only if you have a large audience, even if you have just one user using your app, but it's constantly switching between input A, B, C, A, B, C, B again, C again.</a> <a data-t="2359">It's kind of requesting constantly that same calculation because he's just shifting inputs and he forgets kind of what the outcome of A was, so goes back again.</a> <a data-t="2370">So even for single users, it can be very beneficial to just store these results.</a></p>

<p><a data-t="2377">There are two ways you can implement cache in Shiny.</a> <a data-t="2377">The first is the memoirs package, which kind of helps you to store functions and the result of those functions as cache.</a> <a data-t="2389">And you have Shiny's bind cache function.</a> <a data-t="2391">I have an example of both of them on my GitHub page, and I will share that afterwards.</a></p>

<p><a data-t="2400">So cache sounds nice, but a couple of disclaimers.</a> <a data-t="2400">It's obviously only useful when your output does not change and does not change often.</a> <a data-t="2412">So let's say that your output changes, I don't know, because there's a random seed in there or whatever.</a> <a data-t="2417">It changes every time, yet there's no point.</a> <a data-t="2417">It also is not that much point if your data behind it changes like every, I don't know, six hours or every day.</a> <a data-t="2424">It only is useful when it doesn't change regularly, so that is actually a point of storing your output.</a> <a data-t="2430">Then the downside of using cache is that it requires space, because obviously output needs to be stored somewhere, and that requires some space.</a> <a data-t="2441">And you need to think about sensitive data as well, because if you are kind of sharing your cached results with your other users, you need to think about, okay, is user A allowed to see whatever user C did?</a> <a data-t="2450">But cache can be a game changer in the performance of your app.</a></p>

<p><a data-t="2458">Another smart thing you can do is using background processes, and I touched upon it a bit by saying that if you have multiple users per connection, that they have to wait for each other because the session is occupied.</a> <a data-t="2470">Well, that sometimes can happen, and we don't want that, because we don't want to have parallel execution.</a> <a data-t="2476">No, we want to have asynchronous execution of code, especially if you're developing an app for multiple users.</a> <a data-t="2480">You want them to be able to do things at the same time and not be affected.</a> <a data-t="2490">And in fact, you also don't want a user to have to wait while something is being calculated and that he or she cannot do anything in the app.</a></p>

<p><a data-t="2497">So let's say a user hits a button to kind of render a graph, and then in the background, some data is being retrieved from the database, and then calculations are being done, and then the graph is being rendered.</a> <a data-t="2508">Well, maybe the user doesn't want to wait those, I don't know, 30 seconds for the graph to show up, but maybe in between, the user wants to look at another table and maybe hit another button for a lighter calculation or another calculation.</a> <a data-t="2518">You don't want people to wait.</a> <a data-t="2525">So this is where </a><a href="/software/callr/">callr</a><a data-t="2525"> comes into play, because with callr, you can spin a background R process.</a> <a data-t="2533">And this is really cool, because your Shiny app is running on one R process, and then you can use the callr to kind of set up an additional R process, do the calculations there, then fetch the result, and put it back into the current R process.</a> <a data-t="2546">So this is especially useful if you have queries to databases, which take a very long time.</a></p>
