<h3>What is MLOps?</h3>

<p><a data-t="1053">And so those best practices for machine learning operations to operationalize your model are called MLOps.</a> <a data-t="1061">And so there's a lot of hype around this word that I find online.</a> <a data-t="1066">It's like, I don't even know what MLOps is.</a> <a data-t="1068">You can read all of these different, like, startups and you read their documentation and you don't really know what they're doing at the end of it.</a> <a data-t="1075">It's kind of stressful to get started.</a> <a data-t="1077">But here is a kind of one-liner.</a> <a data-t="1079">MLOps is a set of practices to deploy and maintain machine learning models in production reliably and efficiently.</a></p>

<p><a data-t="1087">So where we had those best practices like setting a seed or splitting your training and testing data to build a model, MLOps are the same different set of practices.</a> <a data-t="1099">But, you know, it's a set of practices to make sure that the model, when it leaves your computer, can still be reproducible and shared in a safe way with others.</a></p>

<blockquote><a data-t="1099">it's a set of practices to make sure that the model, when it leaves your computer, can still be reproducible and shared in a safe way with others.</a></blockquote>

<h3>Versioning models with pins</h3>

<p><a data-t="1297">So the first step of this is versioning.</a> <a data-t="1300">And when people think of versioning, it's normally in the context of Git, where you have some sort of central repository of files that people can read and write to, like push and pull to, but people actually version a lot of things.</a> <a data-t="1318">And they normally version things pretty badly, actually.</a></p>

<p><a data-t="1324">I think of kind of how I am building models, kind of my past, and I would version, you know, I would start with a model and I save it probably as a RDS file or as a Joblib file.</a> <a data-t="1337">So I save my model, I name it model, or maybe something a little better.</a> <a data-t="1342">And then maybe you update some of your features and you get new data, and then you re-save it and it's model final.</a> <a data-t="1349">And then maybe you realize that it needs a little bit more tweaking, and so you have your model, final, final, and you guys see where this goes.</a> <a data-t="1358">You end up with all of these versions of one object that maybe you need information from multiple files of these, but it's not really scalable and these aren't really reproducible.</a></p>

<p><a data-t="1374">Versioning really is the foundation for success in machine learning deployments.</a> <a data-t="1383">And also maybe not just for machine learning deployments, but a lot of reproducibility problems.</a> <a data-t="1393">A well-versioned system, a well-versioned file is normally going to be easier to share.</a> <a data-t="1401">It's going to be easier for you to share with yourself six months from now as well.</a></p>

<p><a data-t="1407">So when we think about our ideal versioning system, it would be nice if all of these files lived in a central location, so you don't have to search between different directories.</a> <a data-t="1418">Maybe if they were discoverable by a team, so something that's not just on your computer, but something that can be hosted in a cloud environment between people.</a> <a data-t="1430">And it's also really nice if things can be loaded right into memory, so you don't have to, like, go download the data if you can't in an easy way, load it onto your local computer, open up RStudio, load the data in that.</a> <a data-t="1451">So it's a few extra steps.</a></p>

<h3>Creating a vetiver model</h3>

<p><a data-t="1730">And this is where vetiver comes in.</a> <a data-t="1734">So let's say we are in our data science workflow.</a> <a data-t="1738">We have created our model.</a> <a data-t="1741">It's called RF pipe.</a> <a data-t="1743">This is like a random forest pipeline that I created.</a> <a data-t="1747">And we're ready to deploy it.</a> <a data-t="1749">So we want to create this deployable model object called a vetiver model.</a> <a data-t="1754">And our inputs into this vetiver model are going to be the model itself that we've created.</a> <a data-t="1760">And then we're going to give it a name as well.</a></p>

<p><a data-t="1763">And the things that are useful about using this vetiver model object is that it holds on to a lot of information for you.</a> <a data-t="1770">Inside this vetiver model, you're going to have things like an input data prototype.</a> <a data-t="1778">So when you created your model, you, of course, fed it some input data.</a> <a data-t="1784">And vetiver actually can like slurp that up on its own.</a> <a data-t="1789">And it knows the data that it should expect as an input.</a> <a data-t="1792">So if you have five columns and you accidentally put four when you're making a prediction, vetiver will tell you, like, no, that's wrong.</a> <a data-t="1802">Your data should be five columns or if it has like a wrong date format or something like that.</a> <a data-t="1809">This becomes super useful when your model is deployed somewhere that's not on your computer.</a></p>

<p><a data-t="1817">It also will say things like what packages are needed to recreate this model and a little bit of metadata about the model itself.</a> <a data-t="1826">And then with your model board and then your vetiver model created, you can do a function that's just vetiver pin write, your model board, your vetiver model.</a> <a data-t="1839">And it will actually version your model automatically.</a> <a data-t="1843">So this will be wherever you want to place it.</a> <a data-t="1848">And your board will collect this information, collect this vetiver model.</a></p>

<p><a data-t="1855">And kind of there it is.</a> <a data-t="1858">It's meant to be only a few lines of code.</a> <a data-t="1860">There's a little bit of magic that happens under the surface, but it's supposed to be kind of like a natural extension of what people are already doing.</a> <a data-t="1869">If you wanted to see the metadata that it saves, you can see it has the title.</a> <a data-t="1875">It has a description when the model was created, a hash, a little bit more robust versioning system than a bunch of underscore finals, as well as the required packages.</a></p>

<h3>Model cards and documentation</h3>

<p><a data-t="1916">There's other things that are beyond just creating these models in vetiver.</a> <a data-t="1922">MLOps is not just about creating models that are, you know, like in different locations.</a> <a data-t="1931">It's a lot about documenting and making sure you have good models that are well documented, that are reproducible and can really show what you've learned when you've created this model.</a></p>

<p><a data-t="1943">We believe that, you know, when you've created the model, this is the time where you've spent all of your research.</a> <a data-t="1949">And there's no better person to write information about this model than the person who created it.</a> <a data-t="1955">So there are a few templates that you can pull up in RStudio.</a> <a data-t="1963">One of them is a model card.</a> <a data-t="1965">Actually, when you write the model itself, you'll get a little feedback that the model card is a framework for transparent, responsible reporting and to use this Quarto framework as a place to start.</a></p>
