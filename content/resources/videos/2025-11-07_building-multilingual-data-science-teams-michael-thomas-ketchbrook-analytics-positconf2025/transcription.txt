<h3>The secret to multilingual teams: technical empathy</h3>

<p><a data-t="103">So now that I've clearly convinced you that a multilingual data science team can have a lot of benefits, I'm going to let you in on the secret to actually making that work.</a> <a data-t="113">And this is going to sound non-technical, but I've found that the key to making multilingual data science teams work is an extra dose of empathy.</a> <a data-t="123">And I'd argue that empathy is an important ingredient in any data science team, but especially so for teams that are trying to use R and Python together.</a></p>

<p><a data-t="131">And I want to focus on the technical things that you can do that are empathetic towards others on your team.</a> <a data-t="142">And I'm going to break it down into three areas.</a> <a data-t="145">Environment management, package choices, and documentation.</a> <a data-t="150">And I like to think of these as the three pillars of technical empathy, and we'll revisit this pillar concept throughout this presentation.</a></p>

<blockquote><a data-t="113">I've found that the key to making multilingual data science teams work is an extra dose of empathy.</a></blockquote>

<h3>Pillar one: environment management</h3>

<p><a data-t="160">And let's start with the first pillar, environment management.</a> <a data-t="165">In the pillar of environment management, empathy means helping your team spend more time creating value and less time troubleshooting technical issues.</a></p>

<p><a data-t="179">But it works on my machine.</a> <a data-t="183">Have any of you ever heard this phrase, maybe even uttered it yourself?</a> <a data-t="188">A little show of hands out there.</a> <a data-t="190">I won't ask you if you've said it yourself, but at least heard it.</a> <a data-t="195">There are still lots of folks dealing with this today, day in and day out.</a> <a data-t="201">And there are also people that don't care because they're essentially a lone wolf.</a> <a data-t="208">And I still think that those people should care about this.</a> <a data-t="211">It's a short-sighted approach.</a> <a data-t="213">But today we're focusing on data science teams, not lone wolves.</a> <a data-t="217">And the idea that you want the code that you're writing on your own laptop to also run successfully somewhere else.</a></p>

<p><a data-t="225">I believe successful environment management boils down to consistency across three things.</a> <a data-t="231">The operating system, the version of Python or R that you're using, and the versions of the individual R or Python packages that you're using on that project.</a></p>

<p><a data-t="242">So let's talk about the first two.</a> <a data-t="247">On the operating system and Python or R version side, you can typically use the same tooling for managing both of those.</a> <a data-t="255">There are two best approaches to doing this, leveraging third-party tools like Posit Workbench, hosted on a dedicated server or cloud environment, that allow you to set up your environment in a more simplified kind of point-and-click approach.</a></p>

<p><a data-t="287">Or you can leverage open source tooling where you will essentially script your environment.</a> <a data-t="291">And if you choose to go that open source approach, know that it's going to require some DevOps skill sets.</a> <a data-t="299">So you'll need to ask yourself if that's something you want to undertake or not, and weigh that against the cost benefit of something like a Posit Workbench that abstracts a lot of that away from you.</a></p>

<p><a data-t="307">On the package versioning side, using virtual environments, which allow you to manage package versions on a project-by-project basis, instead of having some sort of global package library, can be really efficient for collaboration and ensuring that you and the rest of your team are working within the same setup.</a> <a data-t="268">And I think that there are essentially two leaders these days in that area, UV on the Python side and </a><a href="/software/renv/">Renv</a><a data-t="268"> on the R side.</a></p>

<h3>Pillar two: package choices</h3>

<p><a data-t="301">So we just talked about package versions, but now let's talk about choosing which packages to use at all.</a> <a data-t="309">In the second pillar of package choices, empathy means syntax similarity for easier code review and collaboration.</a></p>

<p><a data-t="318">Here's a big question that I ask myself every day.</a> <a data-t="321">How easy would it be for someone who knows R to review my Python code, or vice versa?</a> <a data-t="329">And the choice of packages you use can really impact how successful your multilingual data science team will be.</a></p>

<p><a data-t="331">In these next few slides, you'll see R code examples on the left-hand side of the screen and Python code examples on the right side of the screen.</a></p>

<p><a data-t="340">And this somewhat goes back to my slide where I mentioned that everything great in each language had been ported to the other language.</a> <a data-t="348">But I really think it actually goes beyond that.</a> <a data-t="351">I think the great ideas are being borrowed from each language.</a></p>

<p><a data-t="353">And the Python package Polars is a great example.</a> <a data-t="360">The author of Polars, Richie Vink, he's mentioned in a bunch of interviews that Polars takes a lot of inspiration from the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="360"> in R.</a> <a data-t="369">And in a minute here, I think we'll show you just how true that is.</a></p>

<p><a data-t="375">And we know that data prep is 80% of data science.</a> <a data-t="379">So the methodology and the opinions that you choose are a significant investment for your team.</a></p>

<p><a data-t="385">At Ketchbrook, our choice is typically Polars for Python data prep and </a><a href="/software/dplyr/">dplyr</a><a data-t="385"> for R data prep because not only are they two of the most popular data wrangling libraries in their respective languages, but they're also syntactically really, really similar.</a></p>

<p><a data-t="395">If you want to filter data in your data frame, use the filter function in either package and specify the column and its constraint.</a> <a data-t="407">Do you need to select a column or a specific set of columns?</a> <a data-t="411">Use the select verb.</a> <a data-t="413">And because nothing is completely perfect, we have the arrange verb in dplyr and sort in Polars with some slight differences in direction such that descending is actually a function in dplyr, whereas it's an argument on the Polars side.</a></p>

<p><a data-t="425">But lastly, we have the trusty head function from Base R and in Polars for grabbing the first five rows of data after the upstream ETL takes place.</a> <a data-t="436">And these are just a few functions, but if you went out today and you further explored the dplyr and the Polars packages, you'll continue to find these syntax similarities all over the place.</a></p>

<p><a data-t="447">Now, Polars is already pretty good at this, but in situations where you may need to manipulate quote, unquote, big data, whatever that means these days, you could opt for the arrow R package to continue to leverage a lot of that dplyr syntax that you may be used to.</a></p>

<p><a data-t="463">But on multilingual teams, there's another interesting option that's perhaps even more language agnostic.</a> <a data-t="471">It's actually courtesy of a PositConf keynote speaker last year, and that's DuckDB.</a> <a data-t="479">If you haven't heard of it, DuckDB is a zero-dependency analytics database engine that allows you to do data wrangling in SQL at lightning-fast speeds.</a> <a data-t="490">I'm underselling it.</a> <a data-t="491">If you haven't tried it, I really encourage you to check it out.</a></p>

<p><a data-t="496">But beyond that, we have great libraries in both R and Python that allow us to call DuckDB.</a> <a data-t="503">In each language, there's a bit of setup in the front to connect to the database and some small syntactic differences at the end of the script.</a> <a data-t="513">But everything in the middle is exactly the same.</a> <a data-t="516">The year is 2025 and SQL will not die.</a> <a data-t="520">In fact, I'd argue that perhaps it's as popular and hot as ever.</a> <a data-t="527">And using DuckDB is a great way to still have a high-powered analytics engine that's easily portable across languages.</a></p>

<blockquote><a data-t="516">The year is 2025 and SQL will not die.</a> <a data-t="520">In fact, I'd argue that perhaps it's as popular and hot as ever.</a></blockquote>

<p><a data-t="533">If I had a longer version of this data wrangling script in R and suddenly I needed to port my project over to Python instead, the only things I would need to change would be that little tiny bit of code at the beginning of my script and at the end of my script.</a></p>

<p><a data-t="549">Let's take a look at another example, ggplot2 in R and </a><a href="/software/plotnine/">plotnine</a><a data-t="549"> in Python.</a> <a data-t="555">With the exception of quoting column names and wrapping the entire expression in parentheses on the Python side, the code is literally identical.</a> <a data-t="565">And one last example I'll give is </a><a href="/software/gt/">gt</a><a data-t="565"> in R versus greattables in Python.</a> <a data-t="572">Cue the office meme with Pam saying they're literally the same picture.</a></p>

<h3>Pillar three: documentation</h3>

<p><a data-t="599">And our final pillar today, documentation.</a> <a data-t="603">Empathy means providing clear guides and road maps for collaborators, including your future self.</a> <a data-t="610">So here's my public service announcement.</a> <a data-t="614">Please treat documentation as a first-class citizen in your software development practices.</a></p>

<p><a data-t="620">Years ago, I heard someone say, I think it was J.D. Long, that software shouldn't be a bunch of code with a little bit of documentation.</a> <a data-t="630">Actually, the ratio should be the inverse.</a> <a data-t="633">It should be a lot of documentation with a little bit of code.</a></p>

<p><a data-t="635">Now, I'm going to talk about documentation in the context of, A, function documentation, and, B, what I'm going to call Git-based documentation, such as issues and pull requests.</a></p>

<p><a data-t="651">Consider this simple function I've put together that rounds a number to a certain number of digits and forces it to round up.</a> <a data-t="660">You don't need to care about what it's doing, and the syntax is very similar across both R and Python.</a> <a data-t="667">But the point here is going to be around the documentation of the function.</a> <a data-t="673">In R, we have ROxygen as the prevailing framework for function documentation.</a> <a data-t="679">In Python, we have a couple of options and a little bit more flexibility.</a> <a data-t="684">The prevailing two options seem to be either Google or SciPy's approach to doc strings.</a> <a data-t="691">Within our team at Ketchbrook, we tend to go with the SciPy approach, really only because I find it a little bit more readable, and I know SciPy, as an organization, is data science-focused.</a></p>

<p><a data-t="712">So let's compare the ROxygen framework to the Python SciPy framework.</a> <a data-t="717">In both cases, you can kind of just stick your function description at the beginning of your documentation.</a> <a data-t="724">Function arguments are defined in param tags in R and under a parameter section in Python.</a> <a data-t="730">And you might note that Python tends to traditionally be more explicit about supplying argument data types than we're used to in R.</a> <a data-t="741">While it's not required by ROxygen, we've tried to adopt that in R as well.</a> <a data-t="747">And the output of the function is documented in a returns tag or section in both languages.</a> <a data-t="751">And examples are always critical and similarly structured in both cases.</a> <a data-t="757">So when we leverage these parallels in how we document our functions across the two languages, it's yet another relatively simple way that we can make it easier to cross the language aisle and work together.</a></p>

<p><a data-t="771">So jumping now to our approach to what I'll call GitHub documentation via issues and pull requests.</a> <a data-t="778">If you're on a data science team, there are probably a few approaches to how you could structure your collaborative workflows.</a> <a data-t="784">I'm going to show you our current approach, which I believe is not only ours, but very similar to organizations that we've seen, including Posit.</a> <a data-t="792">So the first step would be a user or a developer identifies a bug in the software or comes up with an enhancement that they're suggesting we implement.</a> <a data-t="803">And they write this up in a detailed issue.</a> <a data-t="808">Once that issue has been created, when we're ready to actually work on making the required code changes, we first create a new Git branch in the repository, specifically for the purposes of addressing that issue.</a> <a data-t="820">Then the developer goes off and writes or modifies the code that fixes the bug or adds the enhancement.</a> <a data-t="826">And once the developer feels like the updated code is in a good spot, they can write up a pull request or a PR for short that details how they addressed the issue, what changed, and more.</a></p>

<p><a data-t="842">So let's focus first on issues.</a> <a data-t="845">In my opinion, I think a good issue is comprised of three different parts.</a> <a data-t="851">First, an overview of the bug that you're encountering or the enhancement that you're proposing, including the rationale for why you feel a change to the code base is necessary.</a> <a data-t="862">Second, a </a><a href="/software/reprex/">reprex</a><a data-t="862">, a reproducible example, a.k.a. some code that someone else can run to show that current shortfall that the proposed enhancement will overcome or identify the bug.</a> <a data-t="874">And lastly, a discussion which may or may not include code regarding possible solutions.</a> <a data-t="880">So I thought I'd show an example of a real-world GitHub issue we've authored that contains this structure.</a> <a data-t="887">An overview section, a reproducible example section with some code that can be run, and a possible solution that's being included as well.</a></p>

<p><a data-t="919">On the pull request side, it's very similar.</a> <a data-t="920">A good pull request, in my opinion, is also comprised of three parts.</a> <a data-t="925">First, the purpose of the pull request and the associated issue or issues that it addresses.</a> <a data-t="932">Second, a more in-the-weeds section that details the technical aspects of how the issue was addressed in code as well as any design decisions that were made and any hurdles that were overcome along the way.</a> <a data-t="950">Lastly, instructions and code that a reviewer can run in order to see the impact of the changes that you made.</a> <a data-t="956">So I'm going to pick on Ivan on our team in the front row for a minute here because he does an awesome job at this.</a> <a data-t="965">And here's an example of a GitHub pull request he authored containing those three components I just mentioned.</a> <a data-t="971">An overview section, a detail section, a test section, and even a discussion section that, in this case, provided some additional context that he felt was important for the reviewer to know.</a></p>

<p><a data-t="983">In the interest of time, I won't show more examples beyond issues and pull requests today, but we also take a standardized approach to our READMEs in each repository as well, making sure to document at a bare minimum the purpose of the repository and the project's background, the prerequisites or dependencies that you'll need to run the code, and how to use the software.</a></p>
