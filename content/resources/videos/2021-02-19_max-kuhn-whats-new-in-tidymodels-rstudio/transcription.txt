<h3>Racing methods</h3>

<p><a data-t="203">Now, the first method we'll talk about is something called racing methods.</a> <a data-t="203">And this is sort of an alteration of grid search that's been around for quite a while.</a> <a data-t="207">Now in ordinary grid search, what you would do is come up with a candidate set of parameter</a> <a data-t="212">values that you want to try. And then you'd measure their performance, usually using resampling,</a> <a data-t="217">and then pick the one that seems to be best or has the characteristics that you like. Now,</a> <a data-t="222">the problem with this is you have to evaluate a potentially large number of models to get</a> <a data-t="226">performance estimates until you can make a decision. So let's say we do 20 values of</a> <a data-t="231">the support vector machine tuning parameters, and we use tenfold cross-validation. That</a> <a data-t="236">would be about 200 total model fits, and we can't make a decision on which tuning parameters</a> <a data-t="240">are good or bad until we do all that work.</a></p>

<p><a data-t="244">So what racing methods do is they sort of</a> <a data-t="244">incrementally evaluate the resamples. So in the first couple resamples, you proceed as</a> <a data-t="249">normal and measure their performance. And then what you do subsequently is do an interim</a> <a data-t="254">analysis after each resample to figure out, basically, are there tuning parameters that</a> <a data-t="259">have a low probability of being called the best results? And so one way we could do that</a> <a data-t="263">is we could fit an ANOVA model to those results, where the outcome is, let's say, the errand</a> <a data-t="268">of the RC curve, or whatever performance metric you want to use. And the predictive factor</a> <a data-t="273">is the qualitative tuning parameter combinations, like, you know, model 1, model 12, and so</a> <a data-t="279">on. And so what we can do is we can evaluate those and see which ones are statistically</a> <a data-t="284">different from the best possible. And if they are, we can discard them.</a> <a data-t="290">Now, there's a variety of ways to do racing methods. We have a second one implemented in FineTune</a> <a data-t="294">that uses win-loss statistics and something called the Bradley-Terry model, and treats</a> <a data-t="298">these data as if they were a sports competition. And if you want to learn more about these</a> <a data-t="302">methods and particularly how we implemented them here, there's a paper there from 2014</a> <a data-t="308">that has the details. And for our particular example, we'll use a grid of 20 support vector</a> <a data-t="313">machine candidate values that are generated with a space-filling design. And we'll show</a> <a data-t="318">how racing can be used to efficiently evaluate them compared to ordinary grid search.</a></p>

<p><a data-t="324">So let's illustrate the race. So let's say we take our 10 folds and we randomly arrange</a> <a data-t="329">them so they're in a random order. We would take the first fold and do what we'd normally</a> <a data-t="334">do. We'd compute our models and then make predictions and measure, let's say, the area</a> <a data-t="340">of the RC curve. Now, in the plot, what you see is on the y-axis is just the model number,</a> <a data-t="347">which is just sort of arbitrarily numbered. There's 20 of them. And on the x-axis, we</a> <a data-t="350">see the measure of performance. And so the blue dot for model 12 indicates that it had</a> <a data-t="357">the best performance across all the tuning parameter values. And then the rest of the</a> <a data-t="361">points show the loss of performance compared to the current best. So you can see model</a> <a data-t="366">12 was the current best, but model 17 was close behind. And on the bottom there, model</a> <a data-t="371">16 did not have very good performance. And so this is after we've evaluated the first</a> <a data-t="375">resample.</a></p>

<p><a data-t="378">Now, in the second resample, we do the same</a> <a data-t="378">thing. We just resample and compute the performance metrics for all 20 tuning parameter values.</a> <a data-t="385">Now, in this plot, instead of the individual RC values, these points represent the average</a> <a data-t="390">of two ROC area under the curve values for each model. Now, the leaders switch from model</a> <a data-t="397">12 to model 17. But you can still see that there's a variety of different models that</a> <a data-t="402">do not have very good performance compared to those.</a></p>

<p><a data-t="411">Now, in iteration 3, we start doing the interim analysis. In this case, we're going to be</a> <a data-t="411">talking about the ANOVA method. And what that allows us to do is put a confidence interval</a> <a data-t="415">on the loss of performance compared to the leader for each tuning parameter value. And</a> <a data-t="420">so you can see the ones that are in black, there's no evidence to reject that they're</a> <a data-t="424">statistically different from the best value in blue. But the ones in gray that you can</a> <a data-t="428">see are statistically different from that. And therefore, we think it would be a pretty</a> <a data-t="432">good idea to discard them from further consideration. So basically, we use the ANOVA statistics</a> <a data-t="437">to figure out which tuning parameter values to keep resampling and which ones we can get</a> <a data-t="442">rid of. And in this particular case, in the third iteration, we're getting rid of 17 different</a> <a data-t="446">tuning parameter values.</a></p>

<p><a data-t="448">Now, on iteration 4, the confidence interval is tightened up pretty considerably. And model</a> <a data-t="454">1 there falls out to where we're left with two models. And as we keep resampling, you</a> <a data-t="458">can see that we're getting closer and closer to having one fall out. And then iteration</a> <a data-t="464">9 of 10, model 12 finally is discarded because it's not statistically or because it is statistically</a> <a data-t="470">different than the best.</a></p>

<p><a data-t="472">So this is on the ninth fold out of 10. What we normally do is if we have any situation</a> <a data-t="478">where there's only one left over, we keep resampling it till it has the full precision</a> <a data-t="483">that we would like to measure performance. So after nine iterations, basically, we filter</a> <a data-t="489">out all but one tuning value combination. And then we just resample model 17 once. And</a> <a data-t="494">what the result is, instead of fitting the full set of 200 model fits, we actually considered</a> <a data-t="499">74 model fits. So it's a pretty substantial reduction in both time and evaluations of</a> <a data-t="504">models.</a></p>

<p><a data-t="505">And since our interval analysis method is not very slow, we're able to call these values</a> <a data-t="510">pretty quickly. <p><a data-t="524">The code for racing is shown here. It's very similar to the tune grid function in the tune</a> <a data-t="529">package. <h3>Simulated annealing</h3>

<p><a data-t="588">So as I've mentioned, racing is really a grid search method that's done very efficiently.</a> <a data-t="593">And again, in grid search, you predefine all the values you want to look at. So the next</a> <a data-t="597">method we'll talk about is an iterative search method based on simulated annealing where</a> <a data-t="601">new values are determined after each current value. So you don't predefine the tuning parameter</a> <a data-t="607">values.</a></p>

<p><a data-t="609">So simulated annealing is really a pretty old search routine. And it conducts a bias</a> <a data-t="613">random walk around our tuning parameter space. And the bias comes in because once it creates</a> <a data-t="618">a new tuning parameter candidate and evaluates it, it may discard that from the search and</a> <a data-t="625">pretend like it never existed. Or if the results are not too far from optimal or a new best,</a> <a data-t="632">it would accept that and proceed from there.</a></p>

<p><a data-t="634">So we'll show that in detail in a little bit as we show the example. One thing we should</a> <a data-t="640">talk about is how simulating comes up with its next value to evaluate. So on the right-hand</a> <a data-t="645">side, we sort of have a plot where the black line shows a search that's gone through six</a> <a data-t="649">iterations and is slowly going down towards the bottom right. And then once we are at</a> <a data-t="655">that last point, we need to generate values that are within a local neighborhood of the</a> <a data-t="660">current best. So what simulated annealing would do would find values that are not too</a> <a data-t="664">different from the one you're currently evaluating.</a></p>

<p><a data-t="666">So what we've done is we've generated a variety of random combinations that are shown in red</a> <a data-t="673">there that are within some sort of radius around the current best. And then we would</a> <a data-t="677">select one from random as the next point that we should evaluate. Now the bias-random</a> <a data-t="682">walk part comes in because if we select one of those and the performance is not as good</a> <a data-t="686">as the current value, we have two choices. The algorithm can either accept it and proceed</a> <a data-t="692">from it going forward or discard it and again, pretend as if it was never generated.</a></p>

<p><a data-t="697">And the determining factors for whether we discard or accept a result are based on two</a> <a data-t="702">things. One is how early you are in the search process. So you're much more likely to accept</a> <a data-t="707">suboptimal values at the early parts of the search progress. Um, that's how simulated</a> <a data-t="712">annealing works. And as you get to the later iterations, you're much less likely to accept</a> <a data-t="717">poor values. Now, the second factor in that decision is how bad was the current result.</a> <a data-t="723">So if the current result is just slightly worse than the current best, it's more likely</a> <a data-t="727">that you would accept it. But if it's a really awful point, um, probabilistically it's less</a> <a data-t="732">likely that you would consider that as the current point and base your next point off</a> <a data-t="736">of that one. So we'll show this in some detail with our current example.</a></p>

<p><a data-t="742">So here's our search space that we looked at earlier. Um, I've generated a grid of four</a> <a data-t="746">different, uh, tuning perimeter values and evaluate them using resampling. Uh, the current</a> <a data-t="751">best from that grid is the green dot. Um, so simulated annealing will start from that</a> <a data-t="756">green dot and proceed from there. So on the first iteration, we're going to define again</a> <a data-t="761">this neighborhood around that green dot and randomly generate a point. And so we did that.</a> <a data-t="766">It actually turns out it's a new best. It's going towards the better results. And so we</a> <a data-t="770">keep that and then base the iteration two results on this value. Now in iteration two,</a> <a data-t="777">we go in kind of a poor direction and get actually fairly bad results. And they're low</a> <a data-t="781">enough to where we would discard this as being suboptimal. And so when we go to iteration</a> <a data-t="785">three, it's based on the results from iteration one, as you'll see in just a second. So discarding</a> <a data-t="791">it means we do not use it to generate the next point in the search.</a></p>

<p><a data-t="796">So in iteration three, you see we get another new best and then we can keep going around</a> <a data-t="801">the search by finding things that are in a local neighborhood. Now, one of the things</a> <a data-t="806">about simulated annealing is it's a global search method, which means it can go back</a> <a data-t="810">and reevaluate areas that it might have already evaluated. And so in this particular result</a> <a data-t="816">on iteration eight, it goes back in a different direction and we accepted a suboptimal. And</a> <a data-t="822">as you can see, we're going to go into an area that's not very good and we're getting</a> <a data-t="826">a series of just poor results.</a></p>

<p><a data-t="830">Now, simulated annealing also has this feature</a> <a data-t="830">where you can do a restart. So if you don't have good results within a certain number</a> <a data-t="834">of iterations, you can trigger the search to start from its last known best point and</a> <a data-t="839">proceed from there. And that is actually what happens here, that we still have another</a> <a data-t="843">suboptimal result. And then we start the search again from the current best.</a></p>

<p><a data-t="849">And so we proceed along here. It happens again. We go off in a bad direction and then we start.</a> <a data-t="857">And so you can see we're close to the number of iterations. So I did 30 iterations here</a> <a data-t="861">and it would just continue to look until it finds better and better values.</a></p>

<p><a data-t="867">So that's a little graphical representation of how simulated annealing works. If you want</a> <a data-t="871">to use this in code, the code is very similar to the tune base function in the tune package.</a> <a data-t="876">The arguments are almost all the same. You could tell a number of iterations to proceed</a> <a data-t="880">from. Again, there's another control parameter that determines things like the size of local</a> <a data-t="886">neighborhood and so on. So it has some of the numerical properties that you can tweak</a> <a data-t="890">for simulated annealing. It also has a verbose method. So if you set that to true, you get</a> <a data-t="895">some nice logging as the search continues. What's nice about this is if you're running</a> <a data-t="900">interactively and it's been running for a while and you feel like maybe it's done as</a> <a data-t="904">best as it could do, you can manually interrupt the execution of the tune semineal function</a> <a data-t="910">and it will return all the results that it's achieved so far. So by manually interrupting</a> <a data-t="914">it, you don't lose all the results that you've generated. It will give you the ones that</a> <a data-t="918">you've completed to date.</a></p>
