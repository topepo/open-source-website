<h3>Listening vs. watching users</h3>

<p><a data-t="847">So it's really important to bring together the people that are using your software with the people that are making it.</a> <a data-t="852">Like, that direction connection is invaluable.</a> <a data-t="856">But listening to your users alone can actually result in really bad software.</a></p>

<p><a data-t="862">Users approach you with feedback.</a> <a data-t="869">You listen to them.</a> <a data-t="871">You do what they asked you to do.</a> <a data-t="874">You built exactly the right tool.</a> <a data-t="878">This is how a lot of people kind of approach software development.</a> <a data-t="881">And we've learned that this actually is not always effective.</a></p>

<p><a data-t="887">So we're all, I think, statisticians in the audience.</a> <a data-t="891">Can anyone tell me a problem or a bias that might creep in with the very first step of this process?</a> <a data-t="906">I would argue that this is a sampling bias.</a> <a data-t="910">You know, if you... This is sort of like having a survey.</a> <a data-t="915">And the first question being, do you love responding to surveys?</a> <a data-t="920">You know, if you only listen to the users who approach you, you are actually going to get a very not representative sample of all of the users who are using your tool.</a></p>

<p><a data-t="932">Let me show you a very specific example from RStudio.</a> <a data-t="936">So if you go on RStudio today and you look at the GitHub issues in RStudio and just sort them by the ones that people want the most, you'll find that the most upvoted issue, number one, with 104 upvotes, is support for more Linux packaging systems.</a> <a data-t="951">And so it seems like this is the number one thing that users want us to do.</a> <a data-t="955">And this is actually a really great idea.</a> <a data-t="958">But I'll point out that we should consider the source.</a> <a data-t="961">You know, where are we looking at this?</a> <a data-t="964">This is our GitHub issue tracker, which means that everybody on it is a GitHub user.</a> <a data-t="969">And GitHub users, as some of you may know, actually tend to prefer alternative operating systems.</a> <a data-t="975">You want to guess how many Linux users as a percentage there are of RStudio?</a> <a data-t="982">It's about 5%.</a></p>

<p><a data-t="985">So again, if you use Linux, this is not a dunk on you.</a> <a data-t="988">You are awesome.</a> <a data-t="989">This feature is a great idea.</a> <a data-t="992">What I want to point out here is that if you only listen to the loudest voices in the room and you only listen to the people that come and talk to you, you're actually not going to build a product that's good for everybody.</a> <a data-t="1003">It's not really enough to simply listen to your users.</a></p>

<p><a data-t="1009">So when you listen to them, you can introduce another kind of bias.</a> <a data-t="1013">I would just call this one missing values.</a> <a data-t="1016">Because even though users will talk to you, it's also true that users mostly don't know what they want.</a> <a data-t="1024">You know, some of my favorite features to work on in RStudio have been features that literally nobody asked for.</a> <a data-t="1031">This is one that I worked on a few years back.</a> <a data-t="1034">It was one of my favorite little pet projects.</a> <a data-t="1036">It became a very popular feature.</a> <a data-t="1038">But literally no one asked for this.</a> <a data-t="1041">Your users don't necessarily know what they want.</a> <a data-t="1043">And so you need to kind of fill in the blanks when you're listening to them.</a></p>

<p><a data-t="1052">So if you want to avoid this bias, what you need to do is actually watch people use your tool.</a> <a data-t="1054">And I promise you.</a> <a data-t="1054">I promise you if you watch people use your tool, you will always be surprised 100% of the time.</a> <a data-t="1061">People do things that you have no idea that they would do.</a></p>

<p><a data-t="1066">I just want to be clear here.</a> <a data-t="1066">We do not collect telemetry on individual users at Posit.</a> <a data-t="1071">That's just not our thing.</a> <a data-t="1073">So everything that I'm about to say is based on things that were shared on YouTube.</a> <a data-t="1078">Turns out people really love recording themselves using RStudio and putting it on YouTube.</a> <a data-t="1085">Observations of colleagues and so forth.</a></p>

<p><a data-t="1089">Here's an example of somebody using RStudio on YouTube.</a> <a data-t="1093">One of the things that we noticed when we watched people using RStudio is this.</a> <a data-t="1099">So you can put your hand up or not.</a> <a data-t="1101">But how many of you have ever opened RStudio, one of the most advanced statistical environments ever invented, in order to add two numbers?</a> <a data-t="1115">People love doing this.</a> <a data-t="1116">Yeah, they treat it like a calculator, right?</a> <a data-t="1119">This is a thing that people love.</a> <a data-t="1121">And you'll never see this on GitHub issues because no one's going to open an issue and say, I wish this thing was a calculator.</a></p>

<p><a data-t="1129">And so by watching people do this, we actually were able to bring that same feature to Positron.</a> <a data-t="1134">So when you open up Positron, you'll notice that it has exactly the same feature.</a> <a data-t="1140">When you open up, it will try very hard to make sure that it is a calculator.</a> <a data-t="1145">You can type stuff into it and get an evaluation.</a> <a data-t="1149">Again, just kind of based on watching people.</a></p>

<p><a data-t="1152">The same thing is true of the data viewer, right?</a> <a data-t="1154">So RStudio has got a really nice data viewer.</a> <a data-t="1159">And one thing we noticed from watching people use the system is that people have this data viewer open all the time.</a> <a data-t="1167">It's a fixture.</a> <a data-t="1168">In fact, the amount of time that people spend using this data viewer is way out of proportion to the amount of time that we spent making this data viewer.</a> <a data-t="1177">This thing is the result of maybe like two or three months of effort.</a></p>

<p><a data-t="1183">And so based on that observation, when we built Positron, we were like, we are going to make a really, really good data viewer.</a> <a data-t="1192">You can go play with this today in the Positron lounge if you're interested.</a> <a data-t="1194">But this thing has support for a million rows and a million columns.</a> <a data-t="1198">It's super high performance.</a> <a data-t="1200">There's really nice column profiles.</a> <a data-t="1202">It has excellent searching and sorting and filtering and row and column pitting and all kinds of great stuff.</a> <a data-t="1208">And again, this is based on the observation that the data viewer is where people spend a lot of their time.</a> <a data-t="1213">It's not something you can learn by just listening to the problems that people report.</a></p>

<h3>Hiding complexity</h3>

<p><a data-t="1218">So it's really important to listen to people and to watch them.</a> <a data-t="1222">And another thing you've got to do if you want to make tools like this is you need to take things that are complicated and make them simple.</a></p>

<p><a data-t="1240">So again, this is the Malling-Hansen writing ball.</a> <a data-t="1240">And I just want to invite you to stare at this thing for a minute.</a> <a data-t="1243">And I think the longer you look at it, the more complicated this thing looks.</a> <a data-t="1248">I could probably tell my children that this is a steam powered radio telescope.</a> <a data-t="1253">And I think they would believe me.</a> <a data-t="1257">But if you use a keyboard today, it actually probably looks a lot more like the keyboard on the right than the one on the left.</a> <a data-t="1265">And I just ask you, you know, where did that complexity go?</a></p>

<p><a data-t="1279">And someone much smarter than me once pointed out that complexity is a little bit like entropy, which is to say that you can never get rid of it.</a> <a data-t="1282">And the best you can ever hope to do is just kind of move it around.</a> <a data-t="1288">In fact, if you look at this somewhat simpler keyboard, you'll find that it's not actually simpler.</a> <a data-t="1295">It's considerably more complicated than that thing that Nietzsche was using.</a> <a data-t="1298">It's just that all the complexity is hidden from you.</a> <a data-t="1301">The complexity is over there.</a> <a data-t="1302">It's hidden where you don't have to deal with it.</a> <a data-t="1307">The tool has taken the complexity away from you and kind of embedded it in the tool itself.</a></p>

<p><a data-t="1312">It's probably not too hard to see where I'm going here with data science tools.</a> <a data-t="1317">An example from the RStudio world is breakpoints.</a> <a data-t="1321">In RStudio, you can click on a line number and you get a cute little red dot.</a> <a data-t="1328">And the next time that line of code runs, it will stop there and you can look around.</a> <a data-t="1333">This functionality is built on a lot of complexity.</a> <a data-t="1338">In fact, did anyone know that R itself has a set breakpoint function?</a> <a data-t="1345">I'm guessing that most of you did not know this because even though R has this, it's really complicated to use.</a> <a data-t="1354">RStudio takes that complexity away and puts it in the tool so you can just kind of click in the gutter.</a></p>

<p><a data-t="1360">Search is another good example of this.</a> <a data-t="1362">If you use a search or a search and replace in RStudio or here this is Positron, the interface almost could not be simpler.</a> <a data-t="1372">You just literally type what you're looking for and you see a list of results.</a> <a data-t="1376">And again, this is not something that you have to use an IDE to do.</a> <a data-t="1380">You can do this at the command line with something like grep.</a> <a data-t="1383">But if you really want to do this well, you need to understand things like text encodings and recursion and ignore files and regular expressions.</a> <a data-t="1395">And we take all that complexity away from you.</a> <a data-t="1397">We put it in the tool so you can just type a word.</a></p>

<p><a data-t="1401">And so that's kind of the thing that you need to remember here is, when at all possible.</a> <a data-t="1406">You can't get rid of complexity, but you can take it away from people and put it in your tool.</a></p>

<h3>Empowering users through extensibility</h3>

<p><a data-t="1668">So it's really important to listen to people.</a> <a data-t="1668">It's really important that you make your tool easy to use and simple.</a> <a data-t="1672">Another thing that you've really got to do is empower your users.</a> <a data-t="1675">Back in 2016, we created these things called RStudio add-ins.</a> <a data-t="1684">And the RStudio add-in API is actually fairly low-powered as extension APIs go.</a> <a data-t="1688">But the community took it and made just kind of an explosion of creativity with it.</a></p>

<p><a data-t="1692">We have this, this is one of my favorite add-ins called Datapasta.</a> <a data-t="1696">What this thing does is it, you should try it if you haven't, it lets you copy data from anything on the web and paste it into RStudio and it turns it into R code that creates that same data set.</a> <a data-t="1704">There's one called Shortcuts that literally lets you make anything into an RStudio shortcut.</a> <a data-t="1708">There's a really nice one called RegExplain.</a> <a data-t="1711">And what this one does is explains regular expressions.</a> <a data-t="1713">There's a color picker one.</a> <a data-t="1717">There's a styler one for styling your R code.</a> <a data-t="1720">There's a theme assister and there's a bazillion of these.</a> <a data-t="1725">And the community has just really surprised us with their creativity.</a></p>

<p><a data-t="1729">This is also true in VS Code land.</a> <a data-t="1733">And that is one of the reasons that Positron is built on VS Code is because VS Code has got something like 80,000 extensions.</a> <a data-t="1742">They have really managed to make it very easy for people to contribute experiences to the editor surface.</a> <a data-t="1750">And these extensions, they really run the gamut of possibility.</a> <a data-t="1755">All the way from something like a C and C++ development compilation and inspection environment to a cute little cat that you can adopt and put in your IDE.</a></p>

<p><a data-t="1770">In fact, if you'll allow me to nerd out just a little bit, bear with me for a minute here.</a> <a data-t="1775">I think that VS Code's extension model is actually a core reason for it being so successful.</a> <a data-t="1782">Prior to VS Code, a lot of editors, and by the way, I blame Emacs for this, a lot of editors load their extensions right into the editor itself.</a> <a data-t="1794">Emacs started this trend, as far as I know, and it gives the extensions a lot of power and literally makes the editor something that you can extend infinitely in a way that's as native as the editor itself.</a> <a data-t="1809">Emacs actually is a really great IDE.</a> <a data-t="1811">It's just missing a good text editor.</a></p>

<p><a data-t="1822">And this is the approach that Adam takes, for example, and the problem with this approach is that when you do this, you allow the extensions to compromise the core IDE experience.</a> <a data-t="1831">And so VS Code works a little differently.</a> <a data-t="1833">Extensions run actually separately from the IDE.</a> <a data-t="1833">They run in this thing called the extension host, and they only get to talk to the UI and what's effectively remote procedure calls.</a> <a data-t="1841">And this design has allowed VS Code, I think, to make a very good tradeoff between making a system that is extensible by the community, but also having a reputation for reliability and performance.</a> <a data-t="1855">This is why VS Code has 80,000 extensions and is still fast, and it's one of the design principles that we adopted for Positron.</a></p>

<p><a data-t="1864">So in Positron, we kind of took this idea and we extended it.</a> <a data-t="1870">Just like VS Code has an API that allows you to build extensions, Positron has that API.</a> <a data-t="1876">We also added another API that makes it possible to add language packs.</a> <a data-t="1880">So in Positron, the R and the Python systems are actually extensions.</a> <a data-t="1886">I find it's easiest to think of Positron as almost like a data science workbench that has these kind of core features like a data view and a console.</a> <a data-t="1897">And then R and Python are more like extensions that plug into that workbench and give you specific language capabilities.</a> <a data-t="1904">But this thing is built on an API that allows anyone to make these, and we hope that you as a community are able to take this thing and build your own creative solutions on top of it.</a></p>

<p><a data-t="1916">This is another way that I like to think about this.</a> <a data-t="1920">If you think about the parts of the product that you build, certainly the core functionality is something that belongs inside your product.</a> <a data-t="1929">But if you allow users to build extensions to your system, what you'll find is that they can build out kind of this long-tail functionality that is not or maybe even shouldn't be part of the core of your system.</a> <a data-t="1944">So it is very important to give people the ability to extend your tool.</a> <a data-t="1948">I promise you will always be surprised at their creativity if you do this.</a></p>
