<h3>Requirements and tooling choices</h3>

<p><a data-t="296">So time was of the essence here.</a> <a data-t="298">So we took a look at what are the basic requirements,</a> <a data-t="302">what are the absolute musts that we need to cover</a> <a data-t="305">if we're building a solution,</a> <a data-t="307">and what tooling is already available for us</a> <a data-t="310">to help us out.</a></p>

<p><a data-t="312">So taking a look at our traditional looking output,</a> <a data-t="315">first and foremost, </a><a href="/software/gt/">gt</a><a data-t="315"> compatibility.</a> <a data-t="318">So gt, the R package for making tables</a> <a data-t="321">developed by Rich from Posit.</a> <a data-t="324">This is really, really important to Pharma.</a> <a data-t="326">We use it a lot for tables.</a> <a data-t="327">It's very, very powerful with formatting.</a> <a data-t="330">So whatever solution we created,</a> <a data-t="333">we wanted to make sure it worked well with gt.</a> <a data-t="336">Didn't want to disrupt that.</a></p>

<p><a data-t="338">Headers and footers.</a> <a data-t="340">These are really, really important,</a> <a data-t="342">part of this whole traditional appearance and traceability.</a> <a data-t="348">So we want to know what study this table came from,</a> <a data-t="351">for instance, when the display was created,</a> <a data-t="354">things like that.</a> <a data-t="355">We want to have control over where</a> <a data-t="356">this metadata goes as well.</a></p>

<p><a data-t="359">And finally, this one may seem obvious,</a> <a data-t="361">but we want a workflow that's based in R.</a> <a data-t="364">This is for a few reasons.</a> <a data-t="366">One is we have some teams who are very, very new to R,</a> <a data-t="370">so we don't want to throw another thing in.</a> <a data-t="373">We want to just kind of keep it consistent.</a> <a data-t="375">The other is we're, as developers,</a> <a data-t="379">we already have some tooling we're using</a> <a data-t="380">and developing in R.</a> <a data-t="382">So if we're trying to build something very, very quickly,</a> <a data-t="385">we want to minimize the amount of change</a> <a data-t="387">that we're making for ourselves</a> <a data-t="389">and make it easy to develop and maintain.</a></p>

<p><a data-t="391">So, gt.</a> <a data-t="394">I took this image from the website here.</a> <a data-t="396">It just shows the workflow for gt.</a> <a data-t="399">I think many of us know it by now,</a> <a data-t="401">but you start with your input data,</a> <a data-t="404">and then you work with it in R</a> <a data-t="406">to create a nice formatted table using the gt package,</a> <a data-t="410">and you end up with an object that's in your R session</a> <a data-t="413">that you're just kind of looking at in the </a><a href="/software/rstudio/">RStudio</a><a data-t="413"> viewer,</a> <a data-t="417">and you can then save it out to a document if you like.</a> <a data-t="421">And there's a few different choices.</a></p>

<p><a data-t="423">So we kind of thought,</a> <a data-t="424">okay, what's going to get us closest to PDF?</a> <a data-t="427">And if we consider the first option listed there, HTML,</a> <a data-t="432">Pharma's not quite ready for HTML yet.</a> <a data-t="434">We're still making our paginated displays.</a> <a data-t="437">You could also, you could take an image,</a> <a data-t="440">like a screenshot of an HTML table</a> <a data-t="444">and have a nice little pretty picture of it</a> <a data-t="446">and put it in a PDF,</a> <a data-t="448">but traceability is very important for us.</a> <a data-t="451">So we wanted to make sure our values</a> <a data-t="454">were embedded in that PDF.</a> <a data-t="456">So HTML was unfortunately off the table.</a></p>

<p><a data-t="460">RTF is something that has a pretty rich history in Pharma,</a> <a data-t="465">so you could imagine that being an option,</a> <a data-t="468">but going from RTF to PDF is not, it's not easy or smooth.</a> <a data-t="472">It's kind of a manual step.</a> <a data-t="474">We don't want manual.</a> <a data-t="476">So that leaves us with LaTeX,</a> <a data-t="478">and we just heard about LaTeX from Keaton.</a> <a data-t="480">That was a nice primer.</a> <a data-t="481">But LaTeX is a typesetting system</a> <a data-t="483">that has been around for a long time,</a> <a data-t="487">and once you get things into LaTeX,</a> <a data-t="489">you're not quite to PDF, but you're very, very close.</a> <a data-t="493">It just needs to be compiled,</a> <a data-t="494">and that's a pretty, pretty simple step.</a> <a data-t="497">So LaTeX puts us in the best position for PDF here.</a></p>

<p><a data-t="503">Speaking of LaTeX, there's a lot of libraries in LaTeX.</a> <a data-t="508">gt uses a few of those to format those tables,</a> <a data-t="512">so we took a look at what else might be available there,</a> <a data-t="515">and you saw it flash up on the screen</a> <a data-t="518">in the last presentation, but Fancy Header</a> <a data-t="521">is a pretty flexible LaTeX library</a> <a data-t="525">for doing our headers and footers in different places,</a> <a data-t="528">in the top and bottom of the page,</a> <a data-t="530">with controlling all of those bits.</a> <a data-t="533">So a really, really promising option for us.</a></p>

<p><a data-t="538">And making it placed in R.</a> <a data-t="539">So we're already, our teams are already developing</a> <a data-t="543">their tables in R, their figures in R,</a> <a data-t="545">they're doing it in the R script.</a> <a data-t="547">How are we gonna get them into the PDF?</a> <a data-t="550">And how does LaTeX come into play?</a> <a data-t="553">Well, that's where R Markdown and </a><a href="/software/quarto/">Quarto</a><a data-t="553"> come in.</a> <a data-t="557">Instead of an R script, we can use these tools</a> <a data-t="561">to render a PDF directly through the PDF formatting engine,</a> <a data-t="566">and under the hood, it's doing the LaTeX,</a> <a data-t="568">and so it makes it very easy for us</a> <a data-t="570">to also throw in some LaTeX as well.</a></p>

<h3>Headers, footers, and flexibility</h3>

<p><a data-t="693">Because headers and footers are so important,</a> <a data-t="695">I wanted to zoom in even closer.</a> <a data-t="697">Here is our original display, just blown up a little bit,</a> <a data-t="701">so we can see the headers more easily.</a> <a data-t="704">So we have a couple lines of headers here</a> <a data-t="706">that we can specify through the header argument</a> <a data-t="710">and doc writer, and through the header argument</a> <a data-t="712">and doc writer, we use a fancy head function,</a> <a data-t="714">and this is totally an interface</a> <a data-t="716">to the fancy header LaTeX library.</a> <a data-t="719">And fancy head takes a series of fancy rows as an argument,</a> <a data-t="723">so row by row in the header, you can say,</a> <a data-t="726">what's my left, center, and right?</a> <a data-t="728">And so here, we want something on the left,</a> <a data-t="731">the study name, we want some automatic page numbering</a> <a data-t="734">on the right through our little helper.</a> <a data-t="737">In the second row, we just want something on the left,</a> <a data-t="740">the population, so we leave everything else empty.</a> <a data-t="743">And then in the final row,</a> <a data-t="745">we want to put our centered table title.</a></p>

<h3>Evolving the framework</h3>

<p><a data-t="881">We also realized pretty early on</a> <a data-t="884">that our framework was generalizable,</a> <a data-t="888">and useful in other scenarios for other document types.</a> <a data-t="892">So we care a lot about PDFs,</a> <a data-t="894">but there's other things that need to be made,</a> <a data-t="900">not just for us, but for others as well,</a> <a data-t="902">and so we started with just a render PDF function</a> <a data-t="906">that was kind of all Docorator did in the beginning,</a> <a data-t="908">and we decided to break it into pieces,</a> <a data-t="912">and that's where the asDocorator function came to be.</a> <a data-t="915">So we split up Docorator creation, rendering,</a> <a data-t="919">and produced an intermediate Docorator object,</a> <a data-t="922">and optionally a permanent file as well.</a> <a data-t="925">And so that holds the display itself,</a> <a data-t="927">and all of that important metadata</a> <a data-t="929">that can be picked up and reused in different templates.</a> <a data-t="933">So we have a couple templates now,</a> <a data-t="935">the original PDF, the RTF,</a> <a data-t="938">which is kind of very basic compared to the PDF,</a> <a data-t="942">but it's in a position now,</a> <a data-t="945">the framework, that it could expand to future output types,</a> <a data-t="949">just need the template for it.</a></p>

<p><a data-t="954">The ability to evolve is really, really important as well,</a> <a data-t="959">and so we wanna be able to take our framework</a> <a data-t="963">and adapt it as technologies change,</a> <a data-t="966">as new things come in,</a> <a data-t="968">much like if you're doing the kitchen part</a> <a data-t="970">of your home renovation,</a> <a data-t="971">you might make certain aesthetic choices today</a> <a data-t="974">that you, you know, the white countertops</a> <a data-t="976">and cabinets and everything.</a> <a data-t="979">Five, 10 years from now,</a> <a data-t="979">that may not be what you want anymore,</a> <a data-t="981">there may be totally different trends.</a> <a data-t="983">And so with Quarto in particular,</a> <a data-t="987">we're well-poised to adapt to these trends.</a></p>

<p><a data-t="991">Something like Docorator for Python may be useful one day,</a> <a data-t="994">and so it's an easier transition</a> <a data-t="996">to building something like that.</a> <a data-t="999">And, you know, today we're using LaTeX,</a> <a data-t="1002">which is a little bit finicky with the syntax,</a> <a data-t="1007">it comes with the burden of system dependencies as well,</a> <a data-t="1012">so like Keaton, we're looking forward</a> <a data-t="1014">to the capabilities of types,</a> <a data-t="1017">it's a little bit of early days with gt,</a> <a data-t="1021">and because that's so important to us,</a> <a data-t="1022">we're not quite there,</a> <a data-t="1024">but, you know, looking forward to that</a> <a data-t="1026">potentially being a better option down the line,</a> <a data-t="1030">and having, you know,</a> <a data-t="1032">just a smoother maintenance process as well.</a></p>
