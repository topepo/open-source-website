<h3>The problem with scaling Shiny</h3>

<p><a data-t="98">So this talk, okay, this talk is going to be a story about two things.</a> <a data-t="103">One is going to be about a toy app that struggled to scale to a large number of users.</a> <a data-t="110">And then we're also going to talk about how we use JavaScript to augment Shiny to overcome</a> <a data-t="116">that struggle.</a></p>

<p><a data-t="119">So this is the app that we start with.</a> <a data-t="121">It's a very simple Shiny application with two interactive hex bin plots.</a> <a data-t="127">Simple easy and nice.</a> <a data-t="129">So with an app like this, you know, what seems to be the issue?</a> <a data-t="133">The issue happens when we go from one concurrent user, which is me operating this app locally,</a> <a data-t="139">to when we have 100 concurrent users accessing this app at the same time.</a></p>

<p><a data-t="147">And that problem is underscored by this plot that I'm going to show you all right here.</a> <a data-t="152">Now for some of you that aren't familiar with kind of like load testing tools for Shiny,</a> <a data-t="157">this plot may be a little bit opaque.</a> <a data-t="159">So let's try to break it down a little bit.</a> <a data-t="161">So this is what's known as a session duration plot.</a> <a data-t="163">And it's showing 100 simulated users using the Shiny app.</a></p>

<p><a data-t="170">The ticks along the Y axis are basically representing your simulated users as they go through the</a> <a data-t="177">workflow on the previous slide.</a> <a data-t="180">And then each of these segments that are stretching across the X axis, that's the amount of time</a> <a data-t="187">that it took a particular step in that user's workflow.</a> <a data-t="192">And then we have, it's a, sorry, it might be skinny for those in the back, but there's</a> <a data-t="196">a red line here that is kind of a reference point that indicates how long the total workflow</a> <a data-t="201">took for a single concurrent user.</a> <a data-t="204">And here, for this workflow, is about 93 seconds.</a></p>

<p><a data-t="208">So there's two things that stand out in this plot.</a> <a data-t="212">One is that we're noticing some inconsistency in when our users are finishing the workflow.</a> <a data-t="219">From a UX perspective, the same workflow should take people the same amount of time</a> <a data-t="223">if they're accessing the app at the same time.</a> <a data-t="225">That's just fair.</a></p>

<p><a data-t="228">Another thing to notice is that the amount of time that it's taking to go through this</a> <a data-t="233">workflow is, has gotten longer.</a> <a data-t="237">And in the worst case, we're seeing that some users are finishing close to 150 seconds,</a> <a data-t="242">as opposed to the 93 seconds for the one user.</a> <a data-t="245">There's no bueno.</a></p>

<p><a data-t="247">So we have this problem, why is it happening?</a> <a data-t="251">To understand why it's happening, we want to quickly just run through some basics about</a> <a data-t="255">how Shiny works.</a> <a data-t="257">Whenever we update one of the dropdowns in those hex bin plots, the browser is going</a> <a data-t="262">to send a signal over to R and say, hey, can you please generate another plot for me?</a> <a data-t="266">And then R is going to be like, sure, buddy, I got you.</a> <a data-t="269">And it's going to generate another PNG, which then the browser goes to render.</a> <a data-t="273">So we have this nice bidirectional communication going.</a></p>

<p><a data-t="277">And that's nice when you have just one user, like when you're developing locally and whatnot.</a> <a data-t="282">But this becomes a problem when you have many browsers connecting to your app simultaneously.</a> <a data-t="289">Most of the time, when you have your Shiny app deployed, there's ultimately a single</a> <a data-t="294">R process that's underlying that app.</a> <a data-t="297">And when all of these browsers are connecting to it, that single R process is doing the</a> <a data-t="302">brunt of the work to serve up your plots, tables, et cetera.</a></p>

<p><a data-t="307">Now, R is a very fast language.</a> <a data-t="309">But R is still a single-threaded language.</a> <a data-t="311">And so because of that, every Shiny app out of the box is going to reach a certain critical</a> <a data-t="316">point where the R process becomes saturated.</a> <a data-t="321">And it develops a backlog of requests that it has to mow through.</a> <a data-t="324">And that backlog is what causes that increase in the session duration that we saw in the</a> <a data-t="330">plot earlier.</a></p>

<blockquote><a data-t="311">And so because of that, every Shiny app out of the box is going to reach a certain critical</a> <a data-t="316">point where the R process becomes saturated.</a> <a data-t="321">And it develops a backlog of requests that it has to mow through.</a></blockquote>

<h3>Using JavaScript to fix the problem</h3>

<p><a data-t="333">So that's, you know, we have a sense for what the problem is.</a> <a data-t="336">So what are we going to do about this?</a> <a data-t="338">Now, there's been plenty of talks about scaling Shiny.</a> <a data-t="341">And there's a lot of good strategies that have come from those.</a> <a data-t="344">We can cache different assets.</a> <a data-t="347">That's a good idea.</a> <a data-t="348">We should be doing that, if we're not already.</a> <a data-t="350">There's also asynchronous operations, you know, writing out files, like large IOE-type</a> <a data-t="356">stuff.</a> <a data-t="357">But that's not what we're here to talk about.</a> <a data-t="358">We're here to talk about something a little weirder.</a> <a data-t="360">We're going to talk about using JavaScript to fix this problem.</a></p>

<p><a data-t="368">So what's the, like, use JavaScript, what are we going to do?</a> <a data-t="372">The big idea here is that we can scale our Shiny applications by taking work from that</a> <a data-t="379">R process, that R session, and moving it into the browser.</a></p>

<p><a data-t="387">If we think back to this previous diagram where all of the browsers are hitting this</a> <a data-t="392">R process at the same time, like, please give me a plot, please give me a plot, please give</a> <a data-t="396">me a plot, we're going to handle it a little differently now.</a> <a data-t="399">We're going to have R take the data set that's to be plotted, and we're going to have R kick</a> <a data-t="405">it out to every connecting browser.</a> <a data-t="408">And then from there, the browser will become responsible for regenerating the plots as</a> <a data-t="412">you update those dropdowns.</a> <a data-t="414">So it's all happening within those individual browsers.</a></p>

<h3>DuckDB-WASM and Observable Plot</h3>

<p><a data-t="420">So that's the idea, cool, but how do we pull this off?</a> <a data-t="424">So much like we use R packages to accomplish different needs that we have, there's some</a> <a data-t="428">JavaScript libraries that we need to get the job done.</a> <a data-t="432">And so we're going to need, what are we going to need?</a> <a data-t="433">We're going to need a data manipulation library, something to select columns, filter data,</a> <a data-t="438">aggregate if needed, and then we're going to need a data visualization library as well.</a></p>

<p><a data-t="444">So for the data manipulation library, I chose DuckDB Wasm.</a> <a data-t="449">So you might be thinking, like, DuckDB Wasm, you know, like, what does all that mean?</a> <a data-t="452">Well, DuckDB, if you haven't heard already, it's a database, but it's not just any database.</a> <a data-t="457">I think of DuckDB as, like, SQLite, but for analytics workloads.</a> <a data-t="462">It's designed from the ground up to munch through the sort of queries that our shiny</a> <a data-t="466">apps love to send.</a></p>

<p><a data-t="470">But then what's the Wasm part?</a> <a data-t="472">Wasm is a shorthand for something called WebAssembly, which you can roughly think of as, you know,</a> <a data-t="477">assembly but tailor-made for the browser.</a> <a data-t="480">And if those are still kind of, like, opaque descriptors, the important thing about, to</a> <a data-t="487">take away from this slide, is that the synthesis of these two technologies, you take DuckDB</a> <a data-t="491">and WebAssembly and you put that in the browser, you get stupid fast queries.</a> <a data-t="495">You're not going to lose any speed in manipulating data and whatnot if we move to the browser,</a> <a data-t="501">which is good.</a></p>

<p><a data-t="505">For data visualization, I chose, because there's a lot of different JavaScript libraries for</a> <a data-t="509">visualizing data, so I chose something called Observable Plot.</a> <a data-t="513">It's a project that's built on top of a more mature project called D3.js, which is known</a> <a data-t="520">for, you know, being able to create any sort of bespoke data visualization you could think</a> <a data-t="525">of.</a> <a data-t="526">Now, I ultimately chose this one as opposed to something like D3 for two reasons.</a> <a data-t="531">One is that, look, I'm not really a JavaScript developer.</a> <a data-t="535">Like, I'm an R developer primarily, and so I'd like for my JavaScript library to have</a> <a data-t="540">as, you know, like, I don't want it to have a very steep learning curve, right?</a></p>

<p><a data-t="545">So the nice thing about Observable Plot is that it's very similar in spirit to </a><a href="/software/ggplot2/">ggplot2</a><a data-t="545">.</a> <a data-t="550">If we look at the code samples here, on the left we have, you know, some garden variety</a> <a data-t="554">ggplot2 code in R, and on the right is the equivalent code using the Observable Plot</a> <a data-t="561">API.</a> <a data-t="562">And you can see that there's a lot of good similarities here, right?</a> <a data-t="567">The whole idea being that in both cases, we have this plot that we're going to be composing</a> <a data-t="572">by mapping different, you know, attributes of our data set to different geomes that come</a> <a data-t="577">together to build this final, you know, composition that's the plot that we're after.</a></p>

<p><a data-t="587">So the other thing is that it is flexible and expressive.</a> <a data-t="591">We all love ggplot2 here, right?</a> <a data-t="593">We love how it doesn't limit us in its ability to create visualizations and whatnot.</a> <a data-t="599">And I didn't want to lose that when I moved from R into the browser.</a> <a data-t="606">What's fortunate is that Observable Plot has a very expressive API that makes it easy to</a> <a data-t="612">develop all the sorts of plots that we're used to making with ggplots.</a> <a data-t="616">We can have our scatter plots.</a> <a data-t="618">We can have our column charts.</a> <a data-t="620">We can have our difference plots that are annotated every which way.</a> <a data-t="624">And just recently, tool tips have made its way into the project.</a> <a data-t="628">So that adds another layer of interactivity that you can inject into your visualizations.</a></p>

<h3>Benefits and caveats</h3>

<p><a data-t="782">So we've already talked a little bit about why this is awesome, but to kind of, like,</a> <a data-t="789">put a bow on it, right?</a> <a data-t="791">One thing that we've done is we've found a way to clear the prototype purgatory hurdle,</a> <a data-t="795">right?</a> <a data-t="797">And we all know what this is as Shiny developers.</a> <a data-t="799">You build an app, it's great, people like using it, but then there's this does it scale</a> <a data-t="802">question and stuff like that that we all have to deal with.</a> <a data-t="807">Well, this is a nice solution to that.</a></p>
