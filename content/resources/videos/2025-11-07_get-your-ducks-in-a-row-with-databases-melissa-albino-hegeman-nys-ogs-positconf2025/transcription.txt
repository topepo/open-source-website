<h3>The original system and its problems</h3>

<p><a data-t="164">So to give you an idea about what our system looked like</a> <a data-t="166">at the beginning, it started out with a SQL server.</a> <a data-t="169">We had to get all our data in one place.</a> <a data-t="170">We were previously using distributed access databases,</a> <a data-t="173">which were terrible because we needed</a> <a data-t="176">one for a year's worth of data.</a> <a data-t="178">Nothing fit.</a> <a data-t="180">Once we got the data in the server,</a> <a data-t="182">the next part was to create an internal package</a> <a data-t="184">for us to use.</a> <a data-t="185">We had to make sure everyone was counting fish</a> <a data-t="187">the exact same way.</a></p>

<p><a data-t="190">And once that was done,</a> <a data-t="193">it made sense to make a Shiny front-end</a> <a data-t="194">so that we didn't have to look at the code all the time.</a> <a data-t="197">We could view it in a browser.</a></p>

<p><a data-t="201">And that was wonderful for a while.</a> <a data-t="203">Until, we did have some constraints on us.</a></p>

<p><a data-t="212">One, all of our team was running a Shiny app locally.</a> <a data-t="214">We did not have access to a Shiny server,</a> <a data-t="216">so everyone had to clone the repo,</a> <a data-t="219">make sure it ran on their machine.</a> <a data-t="221">And that was okay.</a> <a data-t="223">People were using it every day.</a></p>

<p><a data-t="227">Doing the queries, running the reports,</a> <a data-t="229">and they would get ideas about how it could be better.</a> <a data-t="232">And they'd be really good ideas.</a> <a data-t="235">And since we were in control of app development,</a> <a data-t="238">we got to decide what we wanted to add and what we didn't.</a> <a data-t="241">And so the cycle continued.</a> <a data-t="242">In every iteration, the app got better and better.</a></p>

<p><a data-t="247">But eventually, things started to catch up to us.</a> <a data-t="251">Imagine, if you've ever waited tables,</a> <a data-t="254">you're probably familiar with this problem,</a> <a data-t="255">or maybe it's a problem at your dinner table</a> <a data-t="257">with your own family.</a> <a data-t="259">But back in college, I waited tables,</a> <a data-t="260">and you'd go to your four-top,</a> <a data-t="262">and someone would say,</a> <a data-t="263">hey, can I have a glass of water?</a> <a data-t="265">You'd say, sure, no problem.</a> <a data-t="266">You'd go get the glass of water, and you'd bring it back.</a> <a data-t="267">And then the second person would ask for a glass of water.</a> <a data-t="270">You'd go, you'd get the glass of water,</a> <a data-t="272">and you'd bring it back.</a> <a data-t="273">And now the third person also wants a glass of water.</a> <a data-t="276">And at this point, you're just like, you know, come on.</a> <a data-t="277">Couldn't we have done this all in one shot?</a> <a data-t="280">So you wise up, and you also ask the fourth person</a> <a data-t="282">at the same time if they want a glass of water, too.</a> <a data-t="285">And this was a pretty good analogy</a> <a data-t="286">on how our app was running.</a> <a data-t="288">It wasn't well thought out,</a> <a data-t="291">because it was built to do a very small thing to start,</a> <a data-t="293">and we kept adding all these features.</a></p>

<p><a data-t="296">So in terms of the app, as with all shiny apps,</a> <a data-t="300">you know, we had an interface that took inputs</a> <a data-t="302">and then returned results.</a> <a data-t="304">And every time you changed an input,</a> <a data-t="305">that was a new query to the SQL server.</a></p>

<p><a data-t="308">And we found out each query took like eight seconds.</a> <a data-t="311">And that's an eternity.</a> <a data-t="313">I thought about just sitting in silence</a> <a data-t="315">for eight seconds up here,</a> <a data-t="315">but I thought it would be too uncomfortable for all of us.</a></p>

<p><a data-t="321">So where was this bottleneck happening?</a> <a data-t="324">It was with all these calls to the SQL server.</a> <a data-t="328">Every time we had a new question,</a> <a data-t="330">we'd go back to the source to get the answer,</a> <a data-t="332">recalculate everything, and display results.</a> <a data-t="334">So too many queries, run too often,</a> <a data-t="338">and most of them were really unnecessary.</a></p>

<h3>Introducing DuckDB as a cache layer</h3>

<p><a data-t="341">And this is where DuckDB came in.</a> <a data-t="344">We decided we should try pre-processing all this data</a> <a data-t="348">and doing everything in one fell swoop at the beginning</a> <a data-t="350">when people opened the app.</a> <a data-t="352">So DuckDB gave us an easy way to create</a> <a data-t="354">a single file of that data, again, with no cost to us,</a> <a data-t="358">no additional infrastructure needs.</a> <a data-t="360">It worked out of the box</a> <a data-t="362">with everything we were already using.</a> <a data-t="364">And since it was approved by IT,</a> <a data-t="365">we could implement it ourselves and didn't have to wait.</a> <a data-t="367">And if that wasn't enough,</a> <a data-t="369">and that was enough for us to decide to give this a try,</a> <a data-t="374">it also added spatial capabilities to our data</a> <a data-t="376">that weren't available in the SQL server we were using.</a> <a data-t="378">So in the future, we had the option to increase</a> <a data-t="382">some of our QA, QC, and mapping capabilities that way.</a></p>

<p><a data-t="386">So where does this fit in with the workflow we had?</a> <a data-t="392">Our SQL server is still the database of record.</a> <a data-t="395">That is where all the data entry goes.</a> <a data-t="397">That's where all the corrections go.</a> <a data-t="398">When we find mistakes, but it's just really slow.</a> <a data-t="403">So we wanted our app to call that server only once</a> <a data-t="407">on opening, and then write a file locally</a> <a data-t="411">that holds all of that data.</a></p>

<p><a data-t="415">So now, what is in the DuckDB file?</a> <a data-t="418">It's a series of tables that has the raw data,</a> <a data-t="421">the row-level data that the app needs to be able to display.</a> <a data-t="424">It also has the results of all our calculations</a> <a data-t="427">and aggregations stored as separate tables.</a> <a data-t="429">So everything is happening immediately</a> <a data-t="431">when the app is opened, and the app no longer</a> <a data-t="435">has to do calculations or queries on the fly.</a> <a data-t="439">So we have essentially asked for all four glasses of water</a> <a data-t="442">at the beginning when we opened the app.</a></p>

<p><a data-t="477">So we didn't have to change anything on that app</a> <a data-t="480">that people are opening and running.</a> <a data-t="481">What we did have to change was some of the files</a> <a data-t="484">and modules we were calling at the beginning of the app.</a> <a data-t="487">So we have our functions for the app,</a> <a data-t="489">and we have all our modules, but we wanted to add</a> <a data-t="493">a script that did that data pull</a> <a data-t="495">and all those calculations for us, and stored it locally.</a> <a data-t="499">So we created a very creatively named app</a> <a data-t="501">called the CreateDuckDB database.</a></p>

<p><a data-t="536">Additionally, we just had to make sure</a> <a data-t="537">all of our modules were referencing,</a> <a data-t="539">pointing to the right database.</a> <a data-t="541">And since that was defined at the top</a> <a data-t="541">of each of those scripts, it was very easy.</a> <a data-t="544">This was probably an afternoon of work</a> <a data-t="546">to get a testable version ready for everybody else.</a></p>

<h3>Results and caveats</h3>

<p><a data-t="552">And the results were great.</a> <a data-t="555">We went from an eight-second query many, many times</a> <a data-t="559">throughout the app, because people could be using this</a> <a data-t="562">for hours, they usually had it open all day,</a> <a data-t="564">to pretty much instantaneous changes</a> <a data-t="566">when we changed the input.</a> <a data-t="568">The boot-up time for the app was like two seconds longer,</a> <a data-t="571">so even if they only opened it to a run-run query,</a> <a data-t="573">we were still ahead by six seconds.</a> <a data-t="576">So it made the app feel much better to use.</a> <a data-t="578">It was less frustrating.</a> <a data-t="580">We were able to do it ourselves.</a> <a data-t="581">And again, I can't emphasize the low-cost, no-cost enough,</a> <a data-t="587">because we don't really have a budget</a> <a data-t="589">to make these changes on the fly.</a> <a data-t="591">I'm sure if any of you are working</a> <a data-t="592">in the public sector in government,</a> <a data-t="593">you understand how long it takes</a> <a data-t="595">to get those things established.</a></p>

<p><a data-t="598">But there are some big caveats I want to talk about.</a> <a data-t="601">And it has to do with storing the data locally on the PCs.</a> <a data-t="605">Because remember, everyone's making a copy</a> <a data-t="606">and putting it on their machine for that shiny app to run.</a> <a data-t="610">So depending on when they open up their app in the morning,</a> <a data-t="612">there is a chance they could have different data sets</a> <a data-t="614">between two people in the office.</a> <a data-t="616">Most of the time, that's not a problem,</a> <a data-t="618">but there could be situations where</a> <a data-t="621">they're getting different results for the same task,</a> <a data-t="624">and the most likely reason is that</a> <a data-t="627">something happened, there was a big data upload</a> <a data-t="628">between the times they started the app.</a></p>

<p><a data-t="632">Because we are uploading hundreds of records every day,</a> <a data-t="638">so in the grand scheme of things,</a> <a data-t="640">that's not really a difference</a> <a data-t="640">when you're looking at hundreds of thousands of records,</a> <a data-t="642">but just by bad luck, it could make a difference</a> <a data-t="645">if you're helping a specific customer.</a> <a data-t="648">And the other issue, again,</a> <a data-t="649">with that local storage of the file,</a> <a data-t="651">is that you need to be conscious</a> <a data-t="653">that it meets the permissions your user is allowed to have.</a> <a data-t="657">In our situation, everybody could look at</a> <a data-t="659">row-level and trip-level data as part of their job,</a> <a data-t="663">but if your users were only supposed to look at</a> <a data-t="666">calculations or aggregations of the data,</a> <a data-t="668">you don't want to store the row-level data</a> <a data-t="670">in the DuckDB database,</a> <a data-t="671">because then they could look at it</a> <a data-t="674">even if they're not supposed to.</a></p>
