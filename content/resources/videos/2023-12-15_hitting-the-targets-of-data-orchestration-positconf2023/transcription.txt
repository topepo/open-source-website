<h3>The efficiency problem: Groundhog Day</h3>

<p><a data-t="322">And we followed best practices.</a> <a data-t="325">We write our unit tests, our integration tests,</a> <a data-t="328">data validations.</a> <a data-t="330">But something is missing here.</a> <a data-t="333">And what about efficiency?</a></p>

<p><a data-t="336">So, the next day, the pipeline triggers</a> <a data-t="339">and everything is computed correctly.</a> <a data-t="342">But do I really want to recompute everything?</a> <a data-t="346">Because, for example, if the inputs have not changed,</a> <a data-t="349">then I already know the results from the previous day.</a></p>

<p><a data-t="354">So, looking closer to Posit Connect,</a> <a data-t="357">at the bottom line, it is a Cloud Worker.</a> <a data-t="360">And the Cloud Worker means that it triggers a process,</a> <a data-t="364">it does some computations,</a> <a data-t="366">and then it writes, of course, some data in the file system.</a> <a data-t="370">But when the Cloud Worker stops, then finishes the process,</a> <a data-t="374">then everything from the file system vanishes.</a></p>

<h3>Directed acyclic graphs and the desired behavior</h3>

<p><a data-t="417">So, let's see what went wrong here</a> <a data-t="419">and see what options we have to get out of this trap.</a> <a data-t="423">This is a concept from ETL Pipeline.</a> <a data-t="426">It's a very easy concept.</a> <a data-t="428">It's called Directed Acyclic Graph.</a> <a data-t="430">So, consider two inputs, two tables,</a> <a data-t="433">called Data1 and Data2.</a> <a data-t="435">I read this data, I process this data,</a> <a data-t="438">and we join this data.</a> <a data-t="440">And you can see the flow of the logic from the left of the right</a> <a data-t="444">and this is a Directed Acyclic Graph.</a></p>

<p><a data-t="447">I can go now even further and generalize this notion</a> <a data-t="451">where A, B, C, D can be any kind of steps.</a> <a data-t="454">And here, in our case, B can be</a> <a data-t="457">I load some big genomic files to the S3 bucket.</a> <a data-t="462">And I compute some correlation and regression</a> <a data-t="465">in the database for genes that can take hours, et cetera.</a></p>

<p><a data-t="469">So, what I have created here is an all or nothing scenario.</a> <a data-t="473">And this means that whatever,</a> <a data-t="475">whenever I trigger this pipeline,</a> <a data-t="477">everything is invalidated.</a> <a data-t="479">Everything is processed again.</a> <a data-t="481">And what is the desired behavior?</a> <a data-t="484">So, consider here this G node</a> <a data-t="486">that it is like a kind of a logic</a> <a data-t="490">that I just read a small file from the file system</a> <a data-t="494">that can take seconds.</a> <a data-t="496">And when this file changes, this node is invalidated.</a> <a data-t="500">So, what I want to do is just to recompute</a> <a data-t="503">the nodes that are related to this part in my workflow.</a></p>

<p><a data-t="510">And omit the other steps</a> <a data-t="512">that can be like big computations that can take hours.</a></p>

<p><a data-t="517">So, going back to my previous experience</a> <a data-t="520">from research and data science,</a> <a data-t="522">what usually we tend to do</a> <a data-t="524">is like we introduce some kind of an abstraction</a> <a data-t="527">that we call the job or a chapter or something like that.</a> <a data-t="531">And what you tend to do is to break this down</a> <a data-t="534">to some kind of steps that you load some data,</a> <a data-t="537">you process some data, and then you save some data.</a> <a data-t="540">And then you continue on if you want to extend your project.</a></p>

<p><a data-t="545">You introduce a new job,</a> <a data-t="547">you load some data again, process data, save data.</a> <a data-t="551">And this goes on and on,</a> <a data-t="553">and you start creating something like a higher order graph</a> <a data-t="556">where you do things manually.</a> <a data-t="559">But still here you have somehow</a> <a data-t="562">breaked out from this Sisyphean cycle</a> <a data-t="565">for development process,</a> <a data-t="567">but still you have to do some custom scripting</a> <a data-t="570">to declare the sequence of jobs.</a> <a data-t="573">You should do some custom scripting to skip steps,</a> <a data-t="577">and you should also do some custom scripting,</a> <a data-t="580">for example, to shake workflow data.</a></p>

<p><a data-t="582">So, you should be careful.</a> <a data-t="584">All the saved files, for example,</a> <a data-t="586">if you are in a cloud worker,</a> <a data-t="588">to push them to an external file storage.</a> <a data-t="591">That it is persistent.</a> <a data-t="593">And this is just part of a real pipeline in the wild.</a> <a data-t="598">This is a small pipeline,</a> <a data-t="600">and these are the components of the pipeline,</a> <a data-t="603">how they are connected together.</a> <a data-t="605">I think you will see in the next presentation</a> <a data-t="607">a bigger view of this.</a></p>

<h3>Targets to the rescue</h3>

<p><a data-t="630">So, here is where Targets comes into play.</a> <a data-t="633">Targets is an opinionated framework</a> <a data-t="637">about how to structure your data science workflows</a> <a data-t="639">and your data pipelines.</a> <a data-t="642">And if you follow its conventions</a> <a data-t="645">and the way that it suggests you to build your workflows,</a> <a data-t="648">then you get out of the box many benefits.</a></p>
