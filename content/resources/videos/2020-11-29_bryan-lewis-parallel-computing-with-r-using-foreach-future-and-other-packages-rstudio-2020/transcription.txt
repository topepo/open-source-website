<h3>The foreach philosophy</h3>

<p><a data-t="350">This is the situation we found ourselves in in 2008.</a> <a data-t="355">And my friend and, at the time, my colleague, Steve Weston, he looked back at this problem,</a> <a data-t="361">and he thought, there's got to be a better way, a different way, a different approach.</a> <a data-t="366">What if package authors and code writers could decide which parts of their program can run</a> <a data-t="375">in parallel, and separately, later, the users of those packages decide how those codes should</a> <a data-t="382">run in parallel based on their circumstances and available resources?</a></p>

<p><a data-t="388">This is the philosophy of foreach.</a> <a data-t="391">And it's a philosophy shared by Henrik's much more recent future package.</a> <a data-t="397">And it's a very, very important point, because we abstract away the implementation of a particular</a> <a data-t="404">parallel portion of our code and leave that up to the user to decide at runtime how things</a> <a data-t="409">should run.</a></p>

<blockquote><a data-t="366">What if package authors and code writers could decide which parts of their program can run</a> <a data-t="375">in parallel, and separately, later, the users of those packages decide how those codes should</a> <a data-t="382">run in parallel based on their circumstances and available resources?</a></blockquote>

<p><a data-t="411">Now, back in 2008, Steve made the decision to syntactically structure foreach in the</a> <a data-t="417">form of a for loop.</a> <a data-t="419">And because back then Steve was really enamored of list comprehensions and set comprehensions</a> <a data-t="425">and all these kinds of Haskell-y things, he was really into that.</a></p>

<h3>How foreach works</h3>

<p><a data-t="454">But back to foreach, despite its for loop syntax, foreach really kind of works like</a> <a data-t="457">a reduce of a map.</a> <a data-t="460">And let's make that concrete by looking at a really kind of simple contrived example.</a> <a data-t="464">Here's a foreach loop, and I'll explain each part.</a></p>

<p><a data-t="464">foreach is a function, and these iteration variables i and j in the blue circle, they</a> <a data-t="469">kind of play the role of any old iterator variable in a regular old for loop.</a> <a data-t="476">And we can supply a reduction function using this .combine argument, and that can be any</a> <a data-t="481">function you want of two or more arguments.</a> <a data-t="483">And in this case, I'm just using the rbind function.</a> <a data-t="486">And then in the purple circle, there's just a regular old R expression.</a> <a data-t="489">It can be any R expression you want, and it can use those iterator variables.</a> <a data-t="493">But in our case, we're just going to run over the unique values of i and j.</a> <a data-t="497">Each iteration of the loop is going to produce a single row data frame, and we'll pass it</a> <a data-t="500">to rbind.</a></p>

<p><a data-t="502">And if we run...</a> <a data-t="508">But wait, there's this weird mysterious operator, dopar, that takes a function call on one side</a> <a data-t="508">and an R expression on the other.</a> <a data-t="510">It's kind of bizarre.</a> <a data-t="511">If we run this, well, big surprise.</a> <a data-t="515">We have two loop iterations.</a> <a data-t="516">Each one emits a single row of a data frame.</a> <a data-t="520">They're passed into rbind, and our ultimate output is a two-row data frame.</a> <a data-t="525">Rocket science.</a></p>

<p><a data-t="526">But we also get this interesting warning message from dopar.</a> <a data-t="530">And that's dopar telling us, look, man, you didn't tell me that you could run it in parallel</a> <a data-t="536">or you wanted to run it in parallel, so just so you know, this was run on a plain old sequential</a> <a data-t="541">R session.</a> <a data-t="543">And that's what dopar does.</a> <a data-t="544">That strange operator is effectively the API of the foreach package.</a> <a data-t="550">It's the glue that takes your R statements and variables and puts them together with</a> <a data-t="557">various ways to run those things in parallel, like MPI or whatever.</a> <a data-t="563">And that's the abstraction layer, the API.</a></p>

<p><a data-t="568">And so here's an example of that.</a> <a data-t="575">A user then can register a particular parallel adapter package, in this case doparallel,</a> <a data-t="575">with this registration function, and dopar sees that through a hidden global variable</a> <a data-t="579">somewhere.</a> <a data-t="580">And now the loop, without changing the code in any way, runs and gives you the same answer,</a> <a data-t="585">but this time it's run in parallel.</a> <a data-t="587">In this example, two separate R processes were started for each loop iteration.</a> <a data-t="592">Their output was combined through the rbind function to give you the same result.</a> <a data-t="598">That's the philosophy of foreach and the future package in a nutshell.</a></p>

<h3>The future package</h3>

<p><a data-t="603">Thanks to Henrik's amazing work, the future package is like a new extension of this idea.</a> <a data-t="615">It just doesn't give you any...</a> <a data-t="616">There's no...</a> <a data-t="622">Unlike foreach, where you have a very opinionated syntactical structure of a for loop, future</a> <a data-t="622">package lets you do things like this using any type of R syntax that you want.</a> <a data-t="626">You can use regular old for loops, elliplies, maps, the other map, replicate, no loop at</a> <a data-t="631">all.</a> <a data-t="631">You can do anything you want to do.</a></p>

<p><a data-t="632">And the cool thing is both the future package and the foreach package use magic.</a> <a data-t="637">They do.</a> <a data-t="638">They automagically detect and make sure that lexically scoped R objects are available to</a> <a data-t="645">the worker processes.</a> <a data-t="647">So in this case, it's the same loop we were running before, except notice there's this</a> <a data-t="650">K variable at the top that's not defined within the body, the expression that's in the loop.</a> <a data-t="655">And foreach and dopar are smart enough to identify that and say, wait a minute, that</a> <a data-t="659">worker process is going to need that value.</a> <a data-t="661">So I make sure that it has it, even though it's defined in a lexical scope.</a> <a data-t="666">And sure enough, it works.</a> <a data-t="667">And that'll work if you're running it on the same computer, or if you're running it across</a> <a data-t="670">your departmental network, or if you're using AWS to run it across the country, or even</a> <a data-t="675">if you're running it on another planet.</a> <a data-t="676">It'll all just work magically.</a> <a data-t="678">It's really quite amazing.</a></p>

<p><a data-t="681">There are a few other kind of interesting things about the foreach package that are</a> <a data-t="684">more esoteric.</a> <a data-t="686">You can compose two foreach loops together with a composition operator, and that produces</a> <a data-t="690">a third foreach loop.</a> <a data-t="692">And that can be really useful for nested parallelism, to make it work, actually, in a more predictable</a> <a data-t="698">way.</a></p>

<p><a data-t="700">And going back to the Haskell-y ideas, you can compose a foreach loop with a predicate</a> <a data-t="705">filtering operation to produce the when function, in this case, to produce a set comprehension,</a> <a data-t="711">a very Haskell-y kind of thing.</a> <a data-t="714">And Steve was very much into those ideas.</a></p>

<p><a data-t="724">Now, again, thanks to this amazing guy right here who's going to talk about the future</a> <a data-t="724">package just in the very next talk, so you all should stick around.</a> <a data-t="729">Future works interoperably with foreach through the do future parallel adapter package, as</a> <a data-t="735">do a number of other older parallel adapters that are listed there.</a></p>
