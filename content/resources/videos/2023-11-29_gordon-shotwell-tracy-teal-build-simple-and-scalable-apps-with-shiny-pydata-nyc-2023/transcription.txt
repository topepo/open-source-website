<h3>Shiny's reactive graph algorithm</h3>

<p><a data-t="416">So what's a better way?</a> <a data-t="418">A better way is, you know, you can think about DAGs, right?</a> <a data-t="421">So rectidacyclic graphs are great because you know that for any particular upstream change,</a> <a data-t="426">which nodes need to change downstream, right?</a> <a data-t="429">And that's wonderful.</a> <a data-t="430">This is something you use in make files, Airflow, all over the place, right?</a> <a data-t="434">It's a very good way of figuring out that problem of what needs to re-execute.</a> <a data-t="438">But the problem is you still have to write this.</a> <a data-t="440">If you write this wrong, your application is going to be wrong.</a> <a data-t="442">So is there a way we can maybe avoid writing this at all?</a></p>

<p><a data-t="446">So here's Shiny's strategy.</a> <a data-t="448">Shiny's strategy is to </a><a href="/software/infer/">infer</a><a data-t="448"> all the relationships between your component from your application.</a> <a data-t="454">And use that inference to build a computation graph.</a> <a data-t="458">And use that computation graph to minimally re-execute your application.</a></p>

<p><a data-t="464">So telling you that I'm doing this kind of magic inference,</a> <a data-t="467">I would just suggest that that should create a little bit of suspicion, right?</a> <a data-t="471">Because oftentimes you get these inferences and they're like 85%, right?</a> <a data-t="476">And that's much worse than just not doing it at all, right?</a> <a data-t="479">I don't want to have an inference if it's not rock-solid, right?</a> <a data-t="482">So this has to work all the time.</a> <a data-t="488">100% of the time.</a> <a data-t="490">Because if it misses one, then you're not going to be able to trust it.</a> <a data-t="493">And you're going to have to do the event handling anyway.</a> <a data-t="495">And if you can't trust that inference, it's not useful to you.</a></p>

<p><a data-t="500">So one question then is, like, okay, how would we do this, right?</a> <a data-t="503">Let's just start from scratch.</a> <a data-t="504">Like, how would you do this?</a> <a data-t="505">And one answer you might think of is, well, we could analyze the source code, right?</a> <a data-t="508">So I have these inputs here.</a> <a data-t="510">Those inputs are likely user inputs.</a> <a data-t="512">So maybe I could read all the source code.</a> <a data-t="514">I could parse it.</a> <a data-t="515">Find all those inputs.</a> <a data-t="516">And use those to draw that down.</a> <a data-t="518">And you could sort of think, okay, that might work.</a> <a data-t="520">It would be hard.</a> <a data-t="522">But it might work.</a> <a data-t="523">But it won't work.</a> <a data-t="524">And the reason it won't work is because Shiny allows dynamic user interfaces.</a></p>

<p><a data-t="529">So this is a to-do list.</a> <a data-t="531">So if I, like, say buy groceries, I can add a bunch of these tasks.</a> <a data-t="535">I can complete these tasks and clear them.</a> <a data-t="538">And things go away, right?</a> <a data-t="540">So at any given point in this application, the application's state depends on all the things that the user has done up until that point.</a> <a data-t="548">And the reason we're doing that is that this particular application state is not in the source code.</a> <a data-t="553">And it's very difficult, therefore, to use static code analysis to get that application state.</a> <a data-t="558">So it's not going to work.</a></p>

<p><a data-t="561">Instead, what Shiny does is runtime tracing.</a> <a data-t="564">So at runtime, while your application is running, it just watches what all of the different components ask for.</a> <a data-t="570">And it writes that down.</a> <a data-t="572">It says, okay, you asked for this thing last time.</a> <a data-t="574">I'm going to know that when that changes, I'm going to ask you to re-render.</a> <a data-t="579">Because I remember that you asked it.</a> <a data-t="581">I don't really care about what's happening in the code.</a> <a data-t="583">I just know that you made a request for this thing.</a> <a data-t="585">And that thing is a reactive thing.</a> <a data-t="587">So I'm going to, like, just keep track of that.</a> <a data-t="589">But what happens inside is not really my business.</a></p>

<p><a data-t="593">So how does this kind of just walk you through a really basic example?</a> <a data-t="596">So user asks for this output.</a> <a data-t="598">So this is an output on the screen called text.</a> <a data-t="600">It's the ID.</a> <a data-t="602">Shiny then goes and finds the rendering function that matches that ID.</a> <a data-t="605">It fires it.</a> <a data-t="607">And when it fires the rendering function, you see there's this input call.</a> <a data-t="610">So that's a call out to an input from the UI.</a> <a data-t="615">And then Shiny goes and gets the value of that UI.</a> <a data-t="618">It records all of those steps, right?</a> <a data-t="620">And in a simplified way, you get this little graph.</a> <a data-t="622">So you get one input to one output.</a> <a data-t="625">Most of the rest of this talk is just going to be about these beautiful little mermaid graphs.</a></p>

<p><a data-t="631">So let's just refresh our memory about what this looks like.</a> <a data-t="635">So I have two inputs and two outputs, like the count input, the metric input,</a> <a data-t="641">and then the model metrics plot and the distribution here.</a> <a data-t="646">And when Shiny starts out, this is all it knows.</a> <a data-t="649">It just knows that there are four things.</a> <a data-t="651">Two of them are inputs and two of them are outputs.</a> <a data-t="653">It has no idea about any of the relationships between them.</a> <a data-t="655">And so it's like, okay, I don't have any idea what's going to happen.</a> <a data-t="657">So let's just try to get one of these outputs.</a> <a data-t="659">I'm going to pick the metric plot.</a></p>

<p><a data-t="661">So it calculates the metric plot, and then the metric plot goes out</a> <a data-t="664">and requests the selector and the account selector.</a> <a data-t="668">Those are the two things it needs to generate its plot.</a> <a data-t="671">Shiny does that and draws these two lines.</a> <a data-t="674">And the distribution plot fires, and that only needs the account selector.</a> <a data-t="677">So the model metric doesn't matter.</a> <a data-t="680">And there we have our graph.</a> <a data-t="682">No staticated code analysis is required.</a></p>

<p><a data-t="684">So when the account changes, Shiny does something called invalidate.</a> <a data-t="689">And what that means is it tells the downstream dependencies,</a> <a data-t="692">you're not good anymore, you need to recalculate.</a> <a data-t="695">So it invalidates those two outputs.</a> <a data-t="697">And when things are invalidated,</a> <a data-t="699">it just forgets everything that it knows about those relationships.</a> <a data-t="703">Because, again, this is all happening at runtime.</a> <a data-t="705">So while those relationships that was the relationships before</a> <a data-t="708">might not be the relationship now when I recalculate.</a> <a data-t="711">So it recalculates, and in this case it gets the same graph.</a> <a data-t="714">So it gets these two inputs, tries to calculate the distribution,</a> <a data-t="718">gets the account selector, and we're back.</a></p>

<p><a data-t="724">If the metric changes, though, something different happens, right?</a> <a data-t="727">Because the metric only has one child node.</a> <a data-t="730">So that's the only one it invalidates,</a> <a data-t="732">but the distribution plot doesn't need to invalidate, right?</a> <a data-t="734">It didn't depend on it, right?</a> <a data-t="736">So it's still good, it doesn't need to recalculate.</a> <a data-t="738">We still forget the dependencies from the invalidated nodes,</a> <a data-t="741">and then recalculate it.</a> <a data-t="743">So go get the two values, and we're done.</a></p>

<blockquote><a data-t="750">So this is the kind of secret of how Shiny is able to let you build</a> <a data-t="755">very performant applications with very low workload as a developer.</a> <a data-t="760">I don't need to, as a developer, keep track of this graph, really, in any way.</a> <a data-t="765">I need to tell Shiny, I need to declare to Shiny,</a> <a data-t="767">this is how these things should be drawn.</a> <a data-t="771">But then Shiny is able to infer from that, as that application runs,</a> <a data-t="775">how those things should be re-rendered.</a></blockquote>

<h3>Dynamic reactive graphs</h3>

<p><a data-t="786">So this graph we showed, everything was the same.</a> <a data-t="789">It came to the same conclusion, so it seems like a little bit of a lot of work,</a> <a data-t="793">but it's forgetting everything.</a> <a data-t="795">But Shiny actually can do graphs that do change,</a> <a data-t="798">and this is pretty common.</a> <a data-t="800">So here I have a really simple example where I have one slider.</a> <a data-t="803">I have a button which tells Shiny which slider it should use,</a> <a data-t="807">it should listen to.</a> <a data-t="809">And when I change this slider, nothing changes.</a> <a data-t="811">But if I want to change slider one, the text changes.</a> <a data-t="813">And if I switch this, and I go to slider two,</a> <a data-t="817">now slider one is ignored, and slider two responds.</a></p>

<p><a data-t="820">And here's the code that's doing that rendering.</a> <a data-t="823">And basically it's just a simple conditional,</a> <a data-t="825">where if it's slider one, I'm returning one slider value,</a> <a data-t="828">and otherwise I'm returning slider one.</a> <a data-t="831">So again, Shiny knows nothing about this application,</a> <a data-t="833">it just knows that there are three inputs and one output.</a> <a data-t="836">It tries to calculate the text output.</a> <a data-t="838">Initially, the first thing it needs is, like,</a> <a data-t="840">I'm doing that conditional, which slider should I call in,</a> <a data-t="842">go get the button's value.</a> <a data-t="844">And then it knows to go get slider value, and we're finished, right?</a> <a data-t="848">It never got to input slider two.</a></p>

<p><a data-t="851">So then when slider two changes, nothing happens, right?</a> <a data-t="855">And I just want to pause for a second and explain why that's correct.</a> <a data-t="859">So when we ran that first function,</a> <a data-t="862">it said it came to its conclusion without ever asking for the value of slider two.</a> <a data-t="867">What that means is that for all possible values of slider two,</a> <a data-t="871">this output is the same.</a> <a data-t="873">And if that's true, that means that if slider two changes,</a> <a data-t="875">we can just ignore it, right?</a> <a data-t="877">For all possible values of that, the output ended up being the same.</a></p>

<p><a data-t="881">So if slider one changes, this is kind of the same process</a> <a data-t="884">where the text output invalidates, it goes and gets the button,</a> <a data-t="887">the button's still slider one, so it changes to slider one.</a> <a data-t="891">But if the buttons change, when it invalidates the text output</a> <a data-t="895">and goes and gets the new value of the button,</a> <a data-t="897">which it says, go look at slider two,</a> <a data-t="899">this time it's going to get slider two, right?</a> <a data-t="901">And we have a different graph.</a> <a data-t="903">So that's kind of why we have this process of things changing,</a> <a data-t="905">because the reactive graph is going to change</a> <a data-t="907">over the course of that user's application.</a> <a data-t="909">And again, for those dynamic UI elements,</a> <a data-t="911">where you're adding and removing things,</a> <a data-t="913">where things are changing based on other values,</a> <a data-t="915">this is really important.</a> <a data-t="917">Because you might actually add or remove nodes from this graph</a> <a data-t="919">based on what has happened previously in that session.</a></p>

<h3>Scaling with reactive calculations</h3>

<p><a data-t="935">So what's great about this pattern is that it scales.</a> <a data-t="940">So every single Shiny app uses this exact pattern.</a> <a data-t="943">There's no way of developing complex Shiny apps</a> <a data-t="947">versus simple Shiny apps.</a> <a data-t="949">They all fundamentally use this type of transparent reactivity.</a> <a data-t="952">And you can kind of imagine how this scales, right?</a> <a data-t="954">DAGs are very efficient at rendering.</a> <a data-t="956">So if I have lots and lots of components,</a> <a data-t="958">assuming that Shiny is going to be able to use this same process</a> <a data-t="960">to put it into a DAG,</a> <a data-t="962">it's going to be able to render those very complicated applications.</a> <a data-t="966">This works for dynamic UI.</a> <a data-t="968">And lastly, it's lazy.</a> <a data-t="970">So because it's starting out at the bottom of that graph,</a> <a data-t="972">where the user is asking for something,</a> <a data-t="974">if something is hidden from the user,</a> <a data-t="976">it never gets calculated</a> <a data-t="978">until the user actually goes and asks for it.</a></p>
