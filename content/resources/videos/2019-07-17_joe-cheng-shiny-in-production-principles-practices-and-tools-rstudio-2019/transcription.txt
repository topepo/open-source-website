<h3>What does "production" mean?</h3>

<p><a data-t="482">Okay, so before we go any further, let's talk about what production even means.</a> <a data-t="488">I mean, we throw around this word a lot, right?</a> <a data-t="493">And I define production here as there's software environments that are used and relied on by real users with real consequences if things go wrong, right?</a></p>

<p><a data-t="503">So real users, not developers and testers that, you know, they understand if something goes down.</a> <a data-t="513">We're talking about people who really need your stuff to work, and there are real stakes and real consequences if things go down.</a> <a data-t="522">So the sort of antonym or some not production environments would be proof of concepts, prototypes, sandboxes.</a> <a data-t="532">These are places where you expect things to go wrong, and in some cases, like testing and staging, you kind of want things to go wrong.</a> <a data-t="540">That is not the case with production.</a> <a data-t="542">You're always having a good day if you find bugs and or have outages on production.</a></p>

<p><a data-t="551">So I'm using the word software environments.</a> <a data-t="554">What does that even mean?</a> <a data-t="555">Well, it really means the network hardware and software and your code, all of that together when you make a deployment of a Shiny app.</a> <a data-t="565">So this is just one example of what a production environment might look like.</a> <a data-t="568">You've got RStudio Connect in the middle running your Shiny application.</a> <a data-t="573">That could be Shiny server instead.</a> <a data-t="575">You have Nginx, which is just a web server sitting in front if you like.</a> <a data-t="580">And this particular Shiny app is running on Connect, and it's connecting to an Oracle database and a Spark cluster, say.</a> <a data-t="589">So all that together, if that's being accessed by real users, that is your production environment.</a></p>

<p><a data-t="597">And what we generally do if you're following best practices is you have an identical setup for staging.</a> <a data-t="608">You want it to match as closely as possible to what you're running in production.</a> <a data-t="612">So you duplicate that entire stack so that when you are using staging, you're really doing kind of like a dress rehearsal.</a> <a data-t="620">And you want to find all the bugs and work them out in dress rehearsal before it's showtime.</a></p>

<h3>Goals for production environments</h3>

<p><a data-t="636">When we're running code in production, when we're running production environments, we have several goals.</a> <a data-t="645">So first goal is to keep it up.</a> <a data-t="649">We want to keep our production applications up and running.</a> <a data-t="655">Number two, we want to keep them safe.</a> <a data-t="658">The data may be private, the functionality might be privileged, and the code might be proprietary.</a> <a data-t="665">We want to keep all that safe from unauthorized access.</a> <a data-t="668">Now, that may not be the case for you.</a> <a data-t="671">Maybe you're dealing with all open data and your code is open source, but you still don't want your server to get hacked, for example.</a></p>

<p><a data-t="682">Number three, we want to keep it correct.</a> <a data-t="685">We want our applications to be relatively free of bugs.</a> <a data-t="688">We want them to give answers that are correct.</a> <a data-t="692">So we want to be sure that we can verify that.</a></p>

<p><a data-t="702">And finally, we want to keep it snappy.</a> <a data-t="705">When you're working individually with R on your own laptop, the performance might be a little less important.</a> <a data-t="713">But when you put an application into production, you are probably going to have multiple people simultaneously using your code, which means performance is a much bigger concern than it normally is.</a> <a data-t="730">So we want to make sure that our applications run fast enough for the traffic that we expect to get and have the ability to scale if we need to.</a></p>

<h3>Cultural challenges</h3>

<p><a data-t="814">So talking about cultural challenges first, Shiny apps are developed by R users.</a> <a data-t="820">Well, I certainly hope so.</a> <a data-t="824">R users, by and large, don't come from a software engineering background.</a> <a data-t="831">And if you don't come from that background, if you don't come from a background of deploying production software, you may not even realize that you are dealing with a production situation.</a> <a data-t="842">You may not think about the fact that people are going to be relying on this code to be running day after day.</a></p>

<p><a data-t="849">And even if you do know what the stakes are, if you realize what the stakes are, you may not know what the best practices are for writing production applications and deploying them.</a></p>

<p><a data-t="871">And even if you do know what those best practices are, often people don't anticipate or budget for the amount of time and effort it takes to productionize and follow those best practices.</a> <a data-t="883">If it only takes you three weeks to create a Shiny app, and you're getting ready to deploy it, and then now you realize that there's three more weeks of work to do the appropriate automated testing, to do performance and scale testing, to think about security and do those kinds of reviews, it can be really tempting to just cut the corner and just push out to production, especially if nobody's standing there kind of looking over your shoulder.</a> <a data-t="928">And we have definitely seen people do that, and sometimes they pay the price.</a></p>

<h3>Plot caching</h3>

<p><a data-t="2056">So in order to make this faster, we now have to turn our attention to optimizing plots.</a> <a data-t="2059">And that is a particularly difficult thing to do for various reasons.</a> <a data-t="2064">What we've done with Shiny 1.2 is introduce this feature called Plot Caching.</a> <a data-t="2071">And if you're not familiar with the concept of caching, what it means is if you have an operation that is going to be performed multiple times with exactly the same results, maybe don't execute it every time.</a> <a data-t="2083">Execute it the first time, save the result, and in future times, you recognize that, oh, somebody's looking for the same result, and you serve up your already cached result.</a> <a data-t="2093">So we're going to do exactly this with plots.</a> <a data-t="2096">It requires Shiny 1.2, so if you want to play with this, you'll need to install from CRAN.</a></p>

<p><a data-t="2103">Now, plot caching is not for every Shiny application, so there are some criteria you need to meet before you can decide that this is going to work for you.</a> <a data-t="2111">Number one, you have to have slow plots.</a> <a data-t="2114">If your plots are super fast, then caching them is not going to make a huge difference.</a> <a data-t="2116">But we do have slow plots.</a> <a data-t="2117">There are several hundred milliseconds each.</a> <a data-t="2119">And those plots have to be a significant fraction of the overall slowness of your app.</a></p>
