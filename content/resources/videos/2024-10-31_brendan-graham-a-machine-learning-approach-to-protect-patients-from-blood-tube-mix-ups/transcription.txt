<h3>What is a wrong blood in tube error?</h3>

<p><a data-t="15">So to set the stage, every day thousands of tubes of blood make their way into the lab to be</a> <a data-t="20">analyzed.</a> <a data-t="20">And these tubes of blood serve a really important purpose, which is to inform the care</a> <a data-t="25">of patients in the hospital.</a> <a data-t="31">Clinicians need to make decisions about diagnoses, treatment plans,</a> <a data-t="31">and just monitor the status of their patients.</a> <a data-t="36">And in the hallway in the fifth floor of the lab,</a> <a data-t="36">there's a poster reminding us that behind every specimen, behind every tube of blood, is a patient.</a></p>

<p><a data-t="43">And oftentimes that's a sick patient.</a> <a data-t="48">Let's call this patient John Smith.</a> <a data-t="48">And John Smith's clinician is worried that he might have an infection, so they order some labs to check</a> <a data-t="52">for it.</a> <a data-t="58">So John Smith's blood is collected, put into a tube, it makes its way into the lab to be</a> <a data-t="58">analyzed.</a> <a data-t="64">Somebody reviews those results, one of the lab technicians.</a> <a data-t="64">Those results are then</a> <a data-t="64">published to the medical record, where they're accessed by the clinician caring for John Smith.</a> <a data-t="69">This forms kind of the typical life cycle of a blood test.</a> <a data-t="75">But sometimes the wrong decision is</a> <a data-t="75">made, or the wrong treatment plan is implemented, and it's because there's wrong results in the chart.</a></p>

<p><a data-t="80">And those wrong results are not because that machine makes a mistake, and it's not a human</a> <a data-t="86">error mistake by somebody reviewing their results.</a> <a data-t="92">But rather, the wrong blood was put into the tube</a> <a data-t="92">in the first place before it ever even got to the lab.</a> <a data-t="98">And when that happens, this workflow looks</a> <a data-t="98">something like this, where now there's a second patient in the mix.</a> <a data-t="103">Let's call them Jane Smith.</a></p>

<p><a data-t="104">And Jane Smith's blood was accidentally put into the tube labeled for John Smith.</a> <a data-t="109">And once that happens, that tube proceeds along this pipeline, this workflow.</a> <a data-t="114">And now we have the</a> <a data-t="114">clinician making treatment decisions for John Smith based on the blood results for Jane Smith.</a></p>

<p><a data-t="120">And this is what's called a wrong blood in tube error, or a WBIT.</a> <a data-t="125">And as you can guess,</a> <a data-t="125">the consequences of a WBIT can be really severe.</a> <a data-t="131">The wrong medications can be administered,</a> <a data-t="132">the decision not to administer a medication could be taken, and there could even be life-threatening</a> <a data-t="137">reactions from transfusion complications.</a></p>

<p><a data-t="144">To make this even more complicated or hard</a> <a data-t="145">of a problem to solve, these errors are often silent, meaning once the blood is put into the</a> <a data-t="149">tube, there's nothing stopping the downstream processes from proceeding, right?</a> <a data-t="155">It looks like</a> <a data-t="155">blood in the tube, and everything can proceed as normal.</a> <a data-t="160">And so this is a really potentially</a> <a data-t="160">difficult problem to detect, but potentially very severe in terms of patient harm and outcomes.</a></p>

<blockquote><a data-t="144">To make this even more complicated or hard of a problem to solve, these errors are often silent, meaning once the blood is put into the tube, there's nothing stopping the downstream processes from proceeding, right?</a> <a data-t="160">And so this is a really potentially difficult problem to detect, but potentially very severe in terms of patient harm and outcomes.</a></blockquote>

<h3>Current detection methods</h3>

<p><a data-t="167">But the goal of my talk is not to scare everybody.</a> <a data-t="171">There are measures in place to</a> <a data-t="171">prevent this from happening.</a> <a data-t="176">So at the time of collection, there's a barcode scan that can happen.</a> <a data-t="176">So on the wristband, there's a barcode.</a> <a data-t="181">On the tube of blood, there's a barcode.</a> <a data-t="181">Both are scanned at the time of collection to make sure that a mistake hasn't happened.</a></p>

<p><a data-t="186">But human error, you can't account for everything, and so mistakes do end up happening.</a> <a data-t="191">So in the lab itself, there's an automated comparison that takes place.</a> <a data-t="196">And lab machines</a> <a data-t="196">can perform what's called a single analyte delta check, which is really just a fancy way of saying,</a> <a data-t="201">let's take one component of the blood, one analyte, in this case, in this example,</a> <a data-t="206">a red blood cell count.</a> <a data-t="211">So let's look at that analyte and compare it from the current sample</a> <a data-t="211">from the patient to this previous sample from a patient over some short enough time frame.</a> <a data-t="216">And if the delta or the difference between those red blood cell counts is small enough,</a> <a data-t="220">then we can conclude that it's most likely the same patient sample, right?</a> <a data-t="224">Over a short</a> <a data-t="224">enough time frame, certain characteristics of the blood shouldn't change too much.</a></p>

<p><a data-t="229">And we were actually really interested in this automated approach because in the tubes of blood</a> <a data-t="235">and in everybody's blood right now, there's more than just red blood cells, right?</a> <a data-t="240">So we have</a> <a data-t="240">lymphocytes and monocytes and white blood cells and platelets and hemoglobin and all these different</a> <a data-t="246">components of the blood.</a> <a data-t="251">So could we use all of them?</a> <a data-t="251">Could we develop a multi-analyte approach</a> <a data-t="251">to detect wrong blood and tube errors using machine learning?</a> <a data-t="256">Could we implement this in</a> <a data-t="256">the clinical workflow to ultimately prevent patient harm from occurring?</a> <a data-t="262">So with these goals in mind,</a> <a data-t="262">I'm going to talk through a project that I've been working on with a few colleagues in the labs</a> <a data-t="267">to do just this and some of the constraints that we came across and how we overcame them.</a></p>

<h3>Building the dataset</h3>

<p><a data-t="274">So my job as the data scientist on the team is to start building up our data set.</a> <a data-t="279">And we wanted</a> <a data-t="279">to focus on complete blood count tests or CBCs.</a> <a data-t="284">This is a very commonly used test in the hospital</a> <a data-t="284">and all over.</a> <a data-t="290">I would imagine everybody in this room has had a CBC test at some point, most likely.</a></p>

<p><a data-t="341">And before I kind of walk through that, I want to kind of orient everybody to what our data set</a> <a data-t="345">really looked like.</a> <a data-t="351">So before our simulation, we had in this fake data set here, we have</a> <a data-t="351">columns for patient ID and when the collection was taken.</a> <a data-t="357">So when the specimens were taken,</a> <a data-t="357">we also have the actual results from the blood test.</a> <a data-t="361">So white blood cell count, red blood cell</a> <a data-t="361">count, a dozen and a half other results.</a> <a data-t="368">Then for every sample, for every patient, we grab their</a> <a data-t="368">previous sample, right?</a> <a data-t="373">And we get their previous results.</a> <a data-t="373">And then we take the difference between</a> <a data-t="373">those two results across all of the analytes.</a> <a data-t="377">Remember, this is our multi-analyte approach.</a></p>

<p><a data-t="378">And these analytes, sorry, these deltas actually form the predictors in the model.</a> <a data-t="384">So we have our predictors, but we don't have an outcome.</a> <a data-t="389">So to simulate these outcomes in a way</a> <a data-t="389">that kind of approximates how they occur in practice, we use the matching algorithm from</a> <a data-t="394">the matchit package that matched temporally and on department.</a> <a data-t="400">So if we have a patient in our</a> <a data-t="400">data set in the neonatal ICU, the NICU, what we want to do is find another patient in the NICU</a> <a data-t="405">who had their sample taken around the same time.</a> <a data-t="411">And then once we have a match, we overwrite the</a> <a data-t="411">results of one patient with the matched patient.</a> <a data-t="416">And we kind of introduce a synthetic WBIT error.</a></p>

<p><a data-t="416">And that looks something like this.</a> <a data-t="421">So we have, let's say we ran our algorithm on this fake data</a> <a data-t="421">set and we've matched patient 1, 2, 3 with patient 7, 8, 9, because they had their samples collected</a> <a data-t="427">only a half hour apart.</a> <a data-t="430">And let's say, for the sake of example, they're in the same department.</a> <a data-t="432">We take the results from patient 7, 8, 9 and overwrite the original results of patient 1,</a> <a data-t="436">2, 3.</a> <a data-t="442">And then when we take the deltas, we can see that specifically the white blood cell count delta</a> <a data-t="442">in the simulated data set is 9.7 compared to 1.6 in sort of the unmodified data set.</a> <a data-t="448">So we've introduced this WBIT error, which is reflected in this really high delta,</a> <a data-t="452">which allows us to create a new column or WBIT indicator column.</a></p>

<p><a data-t="460">And this approach,</a> <a data-t="461">we repeated for our entire data set.</a> <a data-t="466">And it took us in this kind of stylized example here,</a> <a data-t="466">on the left, of a situation where we had, you know, no WBITs in our data set to running it on</a> <a data-t="472">our 50% of the samples and finding a match.</a> <a data-t="477">And now we have like a perfectly balanced data set</a> <a data-t="477">in terms of the outcome, which is great for model training.</a> <a data-t="483">But, you know, keep this in the back of</a> <a data-t="483">your mind, because as we talk about model assessment, this is going to play a pretty</a> <a data-t="486">big role in how we assess our models.</a></p>

<h3>Thinking about clinical implementation</h3>

<p><a data-t="592">So it's not totally surprising, but also, you know,</a> <a data-t="596">this isn't just a model development exercise.</a> <a data-t="597">We're not just concerned with the fact that</a> <a data-t="597">it's cool to make models with nice data.</a> <a data-t="601">We actually want to think about implementing this,</a> <a data-t="601">right?</a> <a data-t="605">Like who could use these models and how would they use them in their workflow?</a> <a data-t="606">What happens if we get a prediction wrong?</a> <a data-t="612">And so this kind of switched our mindset from</a> <a data-t="612">like model development.</a> <a data-t="616">Like I said, we're not just concerned with developing classification</a> <a data-t="616">models, but how do we implement these into the clinical workflow?</a> <a data-t="621">And so we kind of switched</a> <a data-t="621">gears into the clinical workflow and faced a new set of constraints, specifically around</a> <a data-t="627">false positives.</a></p>

<p><a data-t="634">So here I have on the screen kind of the classic confusion matrix, a two-by-two</a> <a data-t="634">table comparing predicted versus actual values.</a> <a data-t="640">And on the top row, we have true positives and</a> <a data-t="640">false positives.</a> <a data-t="645">Bottom row, we have false negatives and true negatives.</a> <a data-t="645">And the false</a> <a data-t="645">positives are highlighted for an important reason, because you can think about a hypothetical</a> <a data-t="649">scenario in the future where we have this model in production and we make a prediction.</a> <a data-t="654">We see</a> <a data-t="654">a sample and we think, oh, this is a WBIT.</a> <a data-t="659">So now we're kind of living in that top row.</a> <a data-t="659">We're</a> <a data-t="659">either correct, it's a true positive, or we've made a prediction and it's a false positive.</a> <a data-t="664">It's</a> <a data-t="664">not a real WBIT.</a> <a data-t="669">And in the context of like the clinical lab, the false positive translates to a</a> <a data-t="669">lot of extra work that somebody now has to do.</a> <a data-t="674">They probably have to rerun that test to make</a> <a data-t="674">sure that they get the same weird result, which delays all the other tests in line and which has</a> <a data-t="680">downstream impacts on all those patients and the clinicians waiting for those results.</a></p>

<p><a data-t="686">They might have to find the ordering provider, call them, get in touch with them somehow, and</a> <a data-t="691">say, hey, did you really order this blood test for John Smith?</a> <a data-t="697">Because there have been anecdotal</a> <a data-t="697">evidence in the past where you get a blood result published in the chart and the provider says,</a> <a data-t="702">I never ordered this.</a> <a data-t="706">And so one quick way to check for a WBIT is just confirming, hey, did</a> <a data-t="706">you actually order this for this patient?</a> <a data-t="712">Again, that's extra work on the backs of the lab staff.</a> <a data-t="713">And at the scale that the CHOP lab operates, it has real impacts on turnaround time and just</a> <a data-t="717">overall lab efficiency.</a></p>

<h3>Evaluating models on realistic prevalence</h3>

<p><a data-t="755">So with these two kind of ideas in mind, we were thinking as we move to evaluate the models,</a> <a data-t="762">maybe we should move on from focusing on more of the traditional classification metrics like</a> <a data-t="768">accuracy and ROC that I showed earlier and think about positive predictive value,</a> <a data-t="773">which is the proportion of all positives that are true positives.</a> <a data-t="779">And so if we can have a model or</a> <a data-t="779">identify a model that has a high positive predictive value, we can be as confident as we</a> <a data-t="784">can that when we raise an alert, that it will hopefully not result in a lot of unnecessary</a> <a data-t="789">extra work.</a> <a data-t="793">I mean, there will always be false positives, but we can try our best to reduce that.</a></p>

<p><a data-t="794">One of the potential issues that we ran into, or not potential issue, an issue that we ran into</a> <a data-t="799">with this though, is that PPV is extremely sensitive to the prevalence of the outcome.</a> <a data-t="805">And if you remember a few slides ago, when I talked about how we took our training data and</a> <a data-t="809">made it kind of perfectly balanced in this nice bar charts here on the left, this will have major</a> <a data-t="816">implications if we were to assess our models with the holdout data, with data the models</a> <a data-t="823">haven't seen yet, that if it's structured the same way, we would vastly overestimate the prevalence</a> <a data-t="829">of the outcome.</a> <a data-t="835">We would probably overestimate the model PPV.</a> <a data-t="835">And that has, like I said,</a> <a data-t="835">real-world implications for generating a lot of false positives.</a></p>

<p><a data-t="842">So to help get over this constraint,</a> <a data-t="843">we chose to downsample in the holdout data set, the WBIT proportions, such that, excuse me,</a> <a data-t="849">the WBIT prevalence in our holdout data set matched the real-world, excuse me, prevalence of</a> <a data-t="854">the WBITs that are reported in the literature.</a> <a data-t="862">And the real-world, the reported prevalence is</a> <a data-t="862">like one in 3,000 to one in 5,000 samples.</a> <a data-t="867">So it's pretty low.</a> <a data-t="867">So what that means for us,</a> <a data-t="867">like practically in our data set, is that we have about 11,000 or so samples in the holdout data set</a> <a data-t="874">and we downsample them so that only three WBITs remain out of 11,000.</a> <a data-t="880">So the idea is to like</a> <a data-t="880">unleash the models on this unbalanced, downsampled, low-prevalence validation data set and then see</a> <a data-t="887">how the models perform in terms of PPV.</a></p>

<p><a data-t="893">Another thing to think about though, as we assess the</a> <a data-t="893">models, is what's the threshold that we want to raise an alert at?</a> <a data-t="901">You know, the default 50%</a> <a data-t="901">threshold is probably too low.</a> <a data-t="906">It might generate a lot of false positives.</a> <a data-t="906">But what is the optimal</a> <a data-t="906">threshold?</a> <a data-t="913">Is it 90%, 95%?</a> <a data-t="913">We don't know that right now.</a> <a data-t="913">It's not obvious.</a> <a data-t="919">And so using the</a> <a data-t="919">model validation data set, we actually use the probability package to iterate across a bunch of</a> <a data-t="919">different probability thresholds and then reassessed the model performance at each</a> <a data-t="924">threshold, again on this unbalanced, low-prevalence validation data set, which looked something like</a> <a data-t="930">this.</a></p>

<p><a data-t="939">So if we used, again in this unbalanced data set, using a probability threshold of 0.95,</a> <a data-t="940">we have 61 false positives and a PPV of 0.05, which is pretty low, even at this high threshold.</a> <a data-t="947">And which makes sense, we have a really low prevalent outcome.</a> <a data-t="952">But one of our goals here was</a> <a data-t="952">to see how high could we push this threshold while still capturing those three true positives that we</a> <a data-t="957">know are in our validation data set.</a> <a data-t="962">And so again, using this probability package, we're able to</a> <a data-t="962">iterate across thresholds.</a> <a data-t="968">Here, we're bringing it up to 0.99.</a> <a data-t="968">We see a reduction in the false</a> <a data-t="968">positives and a slight increase in PPV to just under 0.09.</a> <a data-t="976">And as we increase the threshold even</a> <a data-t="976">more, we see that same pattern.</a> <a data-t="983">We see fewer false positives and a PPV of about 0.21, which again</a> <a data-t="983">is pretty modest.</a> <a data-t="990">But given the low prevalence of this outcome, we were actually pretty happy with</a> <a data-t="990">this.</a> <a data-t="995">And actually, if we increase the threshold even higher, we start to drop one of those three</a> <a data-t="995">true positives.</a> <a data-t="1000">And that validates one of the criteria that we really care about, which is</a> <a data-t="1000">making sure that we can flag the ones that we know exist in our holdout data set.</a></p>
