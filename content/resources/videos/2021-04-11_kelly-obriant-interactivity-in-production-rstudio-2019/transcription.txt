<h3>Optimizing: moving work out of Shiny</h3>

<p><a data-t="988">So, I'll spare you the first two points of the workflow.</a> <a data-t="997">Again, go watch Joe's talk for those.</a> <a data-t="1006">And skip to number three where I'm optimizing my code.</a> <a data-t="1010">So, at this point, the number one thing that you do very often to optimize some of these performance issues is to move work out of Shiny.</a></p>

<p><a data-t="1021">And Joe says there are two important things to consider when you're moving work out of Shiny.</a> <a data-t="1027">Things that you don't want to do.</a> <a data-t="1027">So, you don't want to do work in Shiny while the user is waiting.</a> <a data-t="1033">And the second thing is you don't want to do the same work for every user.</a> <a data-t="1040">That's very wasteful.</a></p>

<p><a data-t="1041">So, he had this example of loading the CSV data into a Shiny application every time a user connects.</a> <a data-t="1047">And this application I actually designed for the purpose of highlighting these two anti-patterns.</a> <a data-t="1056">But it does a very similar thing.</a> <a data-t="1057">I am making 11 API requests to the trending R packages table to create that data for the user.</a> <a data-t="1066">And then once a user is on the application, I'm calling Rvest two times to perform that metadata scraping for each row selection that occurs.</a> <a data-t="1076">So, that would be repeating that same work over and over again for every user.</a> <a data-t="1083">Which, again, very wasteful.</a> <a data-t="1084">So, how do we solve this problem?</a></p>

<p><a data-t="1090">There are a number of ways I can think of to solve this.</a> <a data-t="1097">I could generate the data ahead of time and publish it in the bundle along with my application code.</a> <a data-t="1106">And this is a totally valid workflow and makes sense to do for a lot of different types of Shiny applications.</a> <a data-t="1111">Sometimes you can just pull the data together and throw it in that bundle and send it along as you're deploying to staging, to production, and it's fine.</a> <a data-t="1126">The problem here is that I want users to see the latest and greatest trending R packages.</a> <a data-t="1132">It's literally an application to explore trends.</a></p>
