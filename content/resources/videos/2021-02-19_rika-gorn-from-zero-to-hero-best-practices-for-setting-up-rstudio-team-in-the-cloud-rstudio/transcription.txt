<h3>A roadmap for data scientists learning engineering</h3>

<p><a data-t="183">So today what I want to do is I want to share with you my own roadmap for how to start the</a> <a data-t="189">process of learning data engineering.</a> <a data-t="191">So if you're looking to set up RStudio team, then this talk should be very helpful for</a> <a data-t="196">you.</a> <a data-t="197">But even if your first engineering product includes a different server setup, then most</a> <a data-t="202">of this talk should also apply to you as well.</a></p>

<p><a data-t="206">So my roadmap is framed around three different parts.</a> <a data-t="209">People, so how you can use people around you and in your organization to help you set up</a> <a data-t="214">your first engineering project.</a> <a data-t="216">Learning, so the most important things for you to learn as a data scientist as you go</a> <a data-t="221">on your data engineering journey, and implementation, what to do when you're actually in the thick</a> <a data-t="227">of things, in the weeds of your project.</a></p>

<h3>People: partnering with an engineer</h3>

<p><a data-t="231">Okay, so you're about to start your first data engineering project.</a> <a data-t="237">What now?</a> <a data-t="239">So even if you're going at this alone or as part of a small group, it's super helpful</a> <a data-t="244">to have an engineer in your corner.</a> <a data-t="247">It's important to understand that your relationship shouldn't be combative.</a></p>

<p><a data-t="252">I think as a data scientist, very often you want all the data possible at your fingertips,</a> <a data-t="259">whereas engineers, a lot of times they have particular security concerns and don't give</a> <a data-t="265">you the data that you need as quickly as you may want it.</a> <a data-t="269">So it's important to start a good relationship with your engineer and to learn from them</a> <a data-t="273">what their security concerns may be for your server and to understand where their concerns</a> <a data-t="280">are coming from.</a></p>

<p><a data-t="281">It's also important before you start your project to really understand the value of</a> <a data-t="286">your project.</a> <a data-t="287">Why are you doing this?</a> <a data-t="289">Who will benefit?</a> <a data-t="290">Who will use this new server, this new product?</a> <a data-t="293">Are there people in your organization that you need to train or that you need to bring</a> <a data-t="296">on into your side?</a> <a data-t="299">It's important to make sure that there's kind of a pot of gold at the end of your rainbow,</a> <a data-t="303">because what you're going to be doing is taking time away from your regular data science tasks</a> <a data-t="309">and focusing on a new skill set on data engineering.</a> <a data-t="313">So it's important that before you start, the work that you're doing is going to very clearly</a> <a data-t="319">bring value to you and to your team.</a></p>

<h3>Learning: Bash, architecture, and security</h3>

<p><a data-t="323">So technology and learning, specifically for engineering projects, my biggest weakness</a> <a data-t="329">when I started in the beginning was that I wasn't comfortable using Bash.</a> <a data-t="334">For me, it was this kind of scary, black, blinking command line, and I found a lot of</a> <a data-t="340">it very unintuitive.</a> <a data-t="342">There was one-letter commands and arguments that were unintuitive as compared to R.</a></p>

<p><a data-t="348">So here we have on the left-hand side some R code, and you can see it reads very easily,</a> <a data-t="353">almost like a sentence.</a> <a data-t="354">You can see what's being grouped, what's being counted, whereas the Bash on the right-hand</a> <a data-t="358">side can be kind of confusing.</a> <a data-t="360">There's one-letter commands.</a> <a data-t="362">It's unclear what's happening.</a> <a data-t="364">So it's important to start learning a little bit about this.</a> <a data-t="366">You don't have to learn everything, but start.</a> <a data-t="369">Learn how to move around in your directories.</a> <a data-t="372">Learn how to copy and paste files.</a> <a data-t="374">Learn how to log into your server using something called SSH protocols.</a> <a data-t="379">And make sure that you actually have access to the correct files, and if you don't, learn</a> <a data-t="383">how to change the access for your files using Bash.</a></p>

<p><a data-t="390">Another piece of learning that's important, it's important to draw out your actual server</a> <a data-t="397">architecture.</a> <a data-t="398">And when I say draw it out, I mean actually take out a piece of paper and draw it out.</a> <a data-t="404">It can even help to look at AWS architecture diagrams on the internet and try to decipher</a> <a data-t="409">what they mean.</a> <a data-t="411">AWS has a million and a half services, and as a data scientist entering that world, a</a> <a data-t="416">lot of it can be super daunting to understand.</a> <a data-t="419">And so understanding what all the various terms to all the different softwares and products</a> <a data-t="423">can be really helpful.</a> <a data-t="425">And that's why drawing out your diagram, even if it's super simple, can be helpful in organizing</a> <a data-t="431">your work before you start.</a> <a data-t="433">Also sharing that diagram with your engineer can also help you prevent making errors in</a> <a data-t="438">your system before you even start.</a></p>

<p><a data-t="443">So here's an example of a simplified AWS architecture diagram.</a> <a data-t="448">When you first are drawing it out, it's important to think about which parts can actually talk</a> <a data-t="453">to each other, what is public, what is private, and which parts of the server point to what</a> <a data-t="459">other parts of the server.</a> <a data-t="461">So here we have the VPC or the Virtual Private Cloud.</a> <a data-t="465">This is your virtual server and it's hosted on AWS, but you can also host servers on Azure</a> <a data-t="471">or Aptable or other managed services.</a> <a data-t="475">Here is where we have the meat of what makes RStudio Team, RStudio Team.</a> <a data-t="480">We have three EC2 instances or three smaller servers that have all the data that make up</a> <a data-t="488">RStudio Server Pro, Connect, and Package Manager.</a> <a data-t="491">So these EC2 instances live in a private subnet, which create a layer of added security so</a> <a data-t="496">that random people can just access your server.</a></p>

<p><a data-t="501">So here we have a bunch of extra layers of security.</a> <a data-t="504">We have a load balancer, which helps route the correct requests to your servers.</a> <a data-t="509">We also have a bastion or jump box, which adds yet another security layer so that you</a> <a data-t="513">don't jump directly into the RStudio Server.</a> <a data-t="516">You have to go through an added jump box in order to get into your server.</a> <a data-t="521">Now once again, this is a simplified model, but it starts to show you what AWS products</a> <a data-t="526">you as the data engineer have to start setting up in the cloud.</a></p>

<p><a data-t="533">Now one last piece is another important learning topic that's critical before you start your</a> <a data-t="538">project, and this is learning a little bit about ports and security.</a> <a data-t="543">So this is a huge topic, but it's helpful to know just a little bit about how computers</a> <a data-t="549">network and a little bit about security management and the most common internet protocols that</a> <a data-t="554">are used.</a> <a data-t="556">This is one of those topics that engineers seem to know a lot about, but when I was learning,</a> <a data-t="561">I found there weren't that many resources available specifically for data scientists.</a></p>

<p><a data-t="566">So ports are communication endpoints that allow computers to talk to each other in various</a> <a data-t="572">secure ways.</a> <a data-t="573">We have port 22, 25, 80, 443.</a> <a data-t="578">These are things like HTTP, HTTPS, things that you may have heard about, but it's important</a> <a data-t="584">to learn just a little bit about what ports are available, which ones you have to turn</a> <a data-t="588">on and off in your server.</a></p>

<p><a data-t="591">SSL and TLS protocols are protocols that make secure internet communication possible by</a> <a data-t="598">encrypting internet traffic.</a> <a data-t="600">I like to think of this as a handshake between different computers.</a> <a data-t="605">Setting up these protocols requires learning a little bit about different certificates</a> <a data-t="609">and configurations, so it's important to learn what protocol you're going to be using when</a> <a data-t="614">setting up your project.</a></p>

<h3>Implementation: getting started</h3>

<p><a data-t="617">Okay, so you've done your learning and now it's time for implementation.</a> <a data-t="623">My main piece of advice is to get started.</a> <a data-t="626">Obviously, do your research, but don't fall into the trap of analysis paralysis.</a> <a data-t="631">I think for me as a data scientist, I kind of want to know everything available before</a> <a data-t="634">I started, but what happened is that I did two weeks of research and then realized I</a> <a data-t="639">had nothing to show for it other than theoretical knowledge.</a></p>

<p><a data-t="644">It's helpful here to take a lesson from your engineering teams and use a more agile approach</a> <a data-t="651">where you get started quickly, you may fail quickly, but then you can get started again</a> <a data-t="656">just as quickly.</a></p>

<p><a data-t="663">All right, so you've officially started your project, what do you do now?</a> <a data-t="668">Use your engineer as a guide and to check in on your work.</a> <a data-t="673">They can tell you when you've done something horribly wrong, like when I accidentally opened</a> <a data-t="677">all of my instances to the entire world, and they can also help you to finish tasks</a> <a data-t="681">that you may not have the appropriate security clearances for.</a> <a data-t="686">So for example, if you want to point your server at a particular domain name, they can</a> <a data-t="691">help you to do that and they could also help you to estimate costs for your server.</a></p>

<p><a data-t="697">Learning, so it's important to start looking at the new data formats and files that you're</a> <a data-t="701">going to be using in your server.</a> <a data-t="703">So to set up RStudio Teams specifically, RStudio uses something called the Cloud Formation</a> <a data-t="708">Template to set up push button deployment.</a> <a data-t="711">Now when I started looking at this, I was very excited because it looked like something</a> <a data-t="715">very similar to something in the R world, and that is that it looked basically like</a> <a data-t="716">a YAML file.</a> <a data-t="718">So it's important to get to know these files, don't just deploy them, look at them, read</a> <a data-t="723">through them, understand their defaults.</a> <a data-t="725">So when I started, I did not look at the defaults of my Cloud Formation file, and then</a> <a data-t="729">I had to destroy my entire stack and start from scratch because I didn't change a really</a> <a data-t="734">important default.</a> <a data-t="736">So get to know your files, get to know your file formats, read through them, understand</a> <a data-t="740">their defaults.</a></p>

<p><a data-t="743">So implementation, you are going to be making mistakes.</a> <a data-t="746">So make sure that you set up guardrails and fail-safes for those potential mistakes.</a> <a data-t="753">First, take screenshots of your instances.</a> <a data-t="756">This is something that you can do in AWS.</a> <a data-t="759">When you set up your server, after you configure your server, you're actually able to take</a> <a data-t="763">a screenshot of it.</a> <a data-t="764">So if you ever mess anything up and you need to go back, you don't have to start over from</a> <a data-t="768">scratch.</a></p>

<p><a data-t="769">As you're going to be setting up your server, there's going to be lots of passwords, PEM</a> <a data-t="775">keys, .pk keys, certificates, save all of these in a secure way.</a> <a data-t="781">For example, saving them in something like one pass.</a> <a data-t="784">Don't just save them on your local computer.</a> <a data-t="787">And lastly, write good documentation.</a> <a data-t="791">Write good documentation that an engineer can follow, that another data scientist can</a> <a data-t="795">follow, because at the moment, while you're the admin, you may not always be the admin</a> <a data-t="799">for this project.</a> <a data-t="800">So it's important to write good documentation as you go along.</a></p>

<h3>Configuring your server and training your team</h3>

<p><a data-t="805">All right.</a> <a data-t="805">So your server is set up, whether it's for RStudio team or for another project.</a> <a data-t="811">Now it's time for the fun part, which is to configure all of your settings and defaults.</a></p>

<p><a data-t="818">So when you're configuring and after you've set up the main parts of your server, you</a> <a data-t="823">need to get buy-in from the data scientists on your team, because they're going to be</a> <a data-t="827">the ones who are going to be testing your deployment.</a> <a data-t="830">So it was helpful for me, at least, to have a testing log to write down what deployments</a> <a data-t="835">are working, which ones are failing.</a> <a data-t="838">And this is helpful for later, when I was actually writing my documentation.</a></p>

<p><a data-t="843">Understand who's going to need to be trained in the company alongside your data scientists.</a> <a data-t="848">Don't bring everyone in until you've figured out most of the bugs.</a> <a data-t="851">You don't want to be doing demos for your product when there's still bugs in it.</a> <a data-t="855">So take time to set up training and set up testing way before you're going to be introducing</a> <a data-t="861">it to folks at your company.</a> <a data-t="864">You don't want to discourage your users and the consumers of your product before they</a> <a data-t="869">can get started and get really excited for your new project.</a> <a data-t="873">Set up time for training and make sure that there's a little bit of time before you actually</a> <a data-t="877">start demoing.</a></p>

<p><a data-t="879">All right, so once you've set up your server, you're now officially the root user with pseudo-access.</a> <a data-t="887">It's important that you use this power for good and not for evil.</a> <a data-t="891">So learn about different types of authorizations, about Google Auth, multi-factor authentication,</a> <a data-t="897">API keys, basic authorization.</a> <a data-t="900">As you're going to be configuring your product, you're going to need to be giving folks in</a> <a data-t="904">your company access to it.</a> <a data-t="906">And this has to be done in a secure manner.</a> <a data-t="908">Your engineer can definitely help you here.</a> <a data-t="910">But knowing a little bit about the differences for different authorization protocols can</a> <a data-t="914">be super duper helpful.</a> <a data-t="916">And once again, you may be the root user now, but you may not be the root user later.</a></p>

<p><a data-t="922">Implementation.</a> <a data-t="924">This is the fun part for me especially.</a> <a data-t="926">Get inside your server and play around.</a> <a data-t="929">Look at the important main files.</a> <a data-t="931">So for RStudio Server Pro, Studio Connect, and Arc Package Manager, I found that these</a> <a data-t="937">five files were super duper important in making sure that deployment was actually working.</a> <a data-t="943">So when I first started, I had a lot of deployment errors, and that's because I realized that</a> <a data-t="948">I hadn't changed some of the defaults in these five files.</a> <a data-t="952">So get into your server, understand what the paths are like, understand what the important</a> <a data-t="957">files are, understand what their defaults are, and how they should be changed, and if</a> <a data-t="962">and when they should be changed.</a></p>

<h3>Summary</h3>

<p><a data-t="967">All right.</a> <a data-t="967">So this is my entire system for setting up a data engineering project as a data scientist.</a> <a data-t="972">So before you start, partner with an engineer and understand their security concerns as</a> <a data-t="977">well as the larger value that your project brings to the entire team.</a> <a data-t="982">Learn a little bit about Bash, about ports and network administration, and draw out your</a> <a data-t="987">AWS infrastructure diagram on a piece of paper.</a> <a data-t="990">And when you're ready to implement, get started as soon as you can.</a></p>
