<h3>The edition concept</h3>

<p><a data-t="54">So the big news with this release of testthat, this upcoming release, testthat 3.0, is we're</a> <a data-t="60">going to introduce the idea of this 3rd edition.</a> <a data-t="63">And I should say anything you're going to hear about today is also explained in more</a> <a data-t="68">detail in vignettes on the testthat website, so if you want to learn more about anything</a> <a data-t="72">I say today, feel free to take a look at those vignettes, and if they don't explain, they</a> <a data-t="78">don't help you understand what's going on, please file an issue so we can learn better.</a></p>

<p><a data-t="83">So the basic idea of the addition is that we want to make breaking changes to testthat.</a> <a data-t="89">Testthat's like 10 years old or something, and there's a bunch of things in the package</a> <a data-t="95">that we now regret.</a> <a data-t="96">But testthat's kind of a victim of its own popularity.</a> <a data-t="100">There's something like 4,000 or 5,000 packages on CRAN that use testthat, and many of them</a> <a data-t="106">are not actively developed.</a> <a data-t="107">So there's no way we can change testthat without breaking a bunch of packages, and many of</a> <a data-t="112">these packages, since they wouldn't be maintained, they'd just end up off CRAN because people</a> <a data-t="118">wouldn't have the energy to fix them.</a></p>

<p><a data-t="122">So the idea of an addition is that this is something you have to deliberately opt into.</a> <a data-t="128">So if you want to use many of the new features or all of the new features of testthat I talk</a> <a data-t="132">about today, you have to deliberately choose to use a 3rd edition of testthat.</a> <a data-t="138">And to sort of do that, you basically have to add a line in your description file, </a><a href="/software/config/">config</a> <a data-t="142">slash testthat slash addition.</a> <a data-t="144">So if you don't do this, you'll continue to use the second edition, that's the existing</a> <a data-t="148">behavior of testthat.</a> <a data-t="149">If you do this, you'll get, if you do change this, you'll get a bunch of new features and</a> <a data-t="155">a bunch of things, which I'll explain shortly, which have gone away.</a></p>

<p><a data-t="158">So the kind of idea is that it should take hopefully, you know, less than 30 minutes</a> <a data-t="166">to convert a package to use the 3rd edition.</a> <a data-t="168">You know, it's a little bit of work, hopefully not too much work, but it just generally gets</a> <a data-t="173">you up to date with all of our best current best practices.</a> <a data-t="176">And I should say, if this kind of idea of this addition idea is successful, it's something</a> <a data-t="182">we're likely to try out in other packages, a way of allowing existing code to continue</a> <a data-t="187">working while giving you the choice to opt into a new set of behaviors.</a></p>

<p><a data-t="194">So additions will always be coupled with a major version.</a> <a data-t="197">So that means testthat2 and everything else before this counts as the 2nd edition, testthat3.0</a> <a data-t="205">can use the 2nd edition, that's the default behavior, or you can opt into using the 3rd edition.</a> <a data-t="210">And if one day, you know, many, probably multiple years down the line, we decide to introduce</a> <a data-t="214">a 4th edition, then that would be tied to testthat4.0.</a></p>

<h3>What's going away in the 3rd edition</h3>

<p><a data-t="220">So what's going away in the 3rd edition?</a> <a data-t="222">These are the things that are most likely to lead you to change your tests.</a> <a data-t="226">So while the context function is going away, we've kind of been moving away from this for</a> <a data-t="231">quite some time in favor of just using the name of the test file.</a> <a data-t="235">There's no need to kind of duplicate the information in the test file and another function, which</a> <a data-t="240">you have to remember to update if you ever move your test files.</a></p>

<p><a data-t="244">We've also been lately in </a><a href="/software/devtools/">devtools</a><a data-t="244"> and use this building kind of a stronger coupling</a> <a data-t="249">between files in the R directory and files in the test directory.</a> <a data-t="253">So that is a kind of a one-to-one correspondence, which gives you a bunch of handy keyboard</a> <a data-t="258">shortcuts and development tools.</a></p>

<p><a data-t="261">The very old expect that function is going away.</a> <a data-t="264">This was the kind of first API, the first excessively clever API that testthat used</a> <a data-t="271">with every test kind of had the flavor.</a> <a data-t="274">You could read it like a sentence like testthat blah is true.</a> <a data-t="279">And I pretty quickly decided that that was excessively clever, not very good.</a> <a data-t="285">And so we're finally deprecating that for good.</a></p>

<p><a data-t="289">Expect is we're deprecating in favor of expect S3 class, expect S4 class and expect type</a> <a data-t="297">of just to be more precise about what exactly you're trying to express.</a> <a data-t="301">Expect equivalent.</a> <a data-t="302">I'll talk about a little bit later, but we're basically that's just going to be an argument</a> <a data-t="305">to expect equal or expect identical.</a> <a data-t="308">Some of the mocking functions going away in favor of more featureful packages.</a> <a data-t="314">And then finally, we are moving away from set up and tear down in favor of test fixtures,</a> <a data-t="320">which I'm not going to talk about today, but there's a whole vignette about them if you</a> <a data-t="324">want to learn more.</a></p>

<p><a data-t="327">And hopefully that's kind of all I need to say.</a> <a data-t="329">Just these are the functions that are going away.</a> <a data-t="332">Like if you really, really believe any of these functions are important, like now's</a> <a data-t="336">the time to speak up and say you want to keep these.</a> <a data-t="338">Otherwise, when you switch to the third edition, rerunning your tests, you'll get deprecation</a> <a data-t="345">messages telling you what to replace these functions with.</a></p>

<h3>Waldo: clearer comparison output</h3>

<p><a data-t="350">So that's kind of the stuff that's going away.</a> <a data-t="353">Now I want to focus on the cool new features.</a> <a data-t="356">I think the feature you are most likely to encounter and the most likely to give you</a> <a data-t="361">pleasure in your life is that comparisons with expect equal and expect identical or</a> <a data-t="366">test failures with expect equal and expect identical are now much easier to understand.</a></p>

<p><a data-t="375">So previously expect equal uses the base all.equal function.</a> <a data-t="379">This was never really what all.equal was intended for, but it kind of did an okay job.</a> <a data-t="384">But there's a few cases where it just doesn't give very useful output.</a> <a data-t="388">And so here I'm comparing empty cars without the first column to empty cars, the full data set.</a> <a data-t="397">And you'll notice I get a lot of output.</a> <a data-t="400">What tells me that there's 10 string mismatches in the names, that doesn't really help me.</a> <a data-t="404">It tells me there's a length mismatch and then it gives me a bunch of mean relative differences.</a> <a data-t="409">So this certainly tells me that there's a difference, but it doesn't kind of concisely</a> <a data-t="415">point me to the fact there's a column missing in this data frame.</a></p>

<p><a data-t="419">So now we use waldo.</a> <a data-t="422">So to use this, I'm just going to use this function called local edition.</a> <a data-t="425">This is not something you generally need to use in your tests because you'll convert a</a> <a data-t="430">whole package to use a third edition.</a> <a data-t="433">But we provide this local edition function for writing stuff like this, for writing vignettes</a> <a data-t="438">so you can explain about this new edition.</a> <a data-t="441">This just temporarily sets the edition to the third edition.</a></p>

<p><a data-t="445">So now we use the third edition and now this is the new waldo output.</a> <a data-t="451">So the kind of the first principle is that waldo is going to give you the most important</a> <a data-t="456">differences first.</a> <a data-t="458">And in this case, I think the most important difference is that the lengths are different.</a> <a data-t="463">And then it tells us what the names are, how the names have changed, lining up the old</a> <a data-t="470">names, lining up the actual names with the expected names.</a> <a data-t="477">And then it uses color to highlight the fact that in the expected, there's another column</a> <a data-t="482">called MPG.</a> <a data-t="483">And then finally down here, we get an exact description that, well, the names are different,</a> <a data-t="487">but the values are also different.</a></p>

<p><a data-t="502">So here I've switched to use the third edition.</a> <a data-t="507">Waldo then shows the most important differences first.</a> <a data-t="512">It uses color to help highlight differences, and it always uses the names of things we</a> <a data-t="518">have possible, just because that makes it easier to understand exactly where the differences lie.</a></p>

<p><a data-t="534">So another little example here, I have a factor.</a> <a data-t="539">This is an ordered factor with one extra level.</a> <a data-t="543">When you look at the expect equal output from the second edition, you'll see like it kind</a> <a data-t="548">of tells you that they're different, but it's hard because it doesn't include the values</a> <a data-t="552">to know exactly what's changed.</a> <a data-t="555">When we switch to the third edition, now we can see, well, there's a new element of</a> <a data-t="558">the class vector.</a> <a data-t="559">It's ordered, and there's a new element of the levels, which is D, and hopefully this</a> <a data-t="564">will make it much, much easier to see exactly what's gone wrong.</a> <a data-t="568">And certainly in my own, this is like, this is the number one reason I've been converting</a> <a data-t="572">packages to use the third edition to test that, just because it makes things so much</a> <a data-t="576">easier to see when something's gone wrong.</a></p>

<blockquote><a data-t="568">And certainly in my own, this is like, this is the number one reason I've been converting</a> <a data-t="572">packages to use the third edition to test that, just because it makes things so much</a> <a data-t="576">easier to see when something's gone wrong.</a></blockquote>

<p><a data-t="582">So if you want to learn more about waldo, you can go to the waldo website, which shows</a> <a data-t="586">a bunch of kind of examples, shows some of the principles.</a> <a data-t="591">A lot of the differences of the values are powered by this really nice package called</a> <a data-t="596">diffopt by Brody Graslam, which uses the same algorithm as the diff utility on Linux, which</a> <a data-t="604">just makes it really easy to kind of narrow in exactly what has changed between two vectors.</a></p>

<p><a data-t="611">This change also makes it now possibly precise about what's the difference between expect</a> <a data-t="617">equal, expect identical, and expect equivalent, which are always a little vague in the past.</a> <a data-t="622">Now all of these functions are equivalent to expect identical, but with some extra arguments set.</a> <a data-t="629">So if you use expect equal, that's equivalent to expect identical, except that it ignores</a> <a data-t="634">small floating point differences.</a> <a data-t="638">Or if you use expect equivalent, which is now deprecated, it's just the same as using</a> <a data-t="642">expect equal or expect equivalent with ignore attribute equals true.</a> <a data-t="646">That's the only thing that expect equivalent does is it just ignores attributes.</a> <a data-t="650">Now we're going to make that precise in a function argument.</a> <a data-t="653">And then any other arguments in expect identical or expect equal are passed on to waldo compare,</a> <a data-t="659">which gives you the ability to kind of fine tune your comparisons as needed.</a></p>

<p><a data-t="666">Now this is such a great idea.</a> <a data-t="668">You might wonder, well, why can't it work for the second edition as well?</a> <a data-t="671">Well, unfortunately, when I implemented expect equal, it turns out that I made a rather silly</a> <a data-t="679">mistake when implementing the tolerance comparison.</a> <a data-t="683">So depending on exactly which code path it goes down, it computes the tolerance in slightly different ways.</a> <a data-t="689">One way it always uses the absolute tolerance, and one way it uses absolute or relative tolerance.</a> <a data-t="695">So there's basically no way I did a few experiments.</a> <a data-t="698">There's no way, regardless of which one I pick, it causes like hundreds of CRAN packages to break.</a> <a data-t="703">So that's the main reason that this has to be in the third edition.</a></p>

<h3>Snapshot testing</h3>

<p><a data-t="728">The next big feature I want to talk about is snapshot testing.</a> <a data-t="732">Again, this comes with a vignette, so if you want to learn more about it, you can read the vignette.</a> <a data-t="737">If the vignette doesn't make sense, please file an issue so we can make it better.</a> <a data-t="742">But the basic idea is this is a new type of testing.</a></p>

<p><a data-t="753">So normally in unit tests, you describe the expected output using code.</a> <a data-t="760">And in the vast majority of cases, this is a really, really good idea because it allows</a> <a data-t="764">you to kind of describe what you expect.</a> <a data-t="767">And in some sense, the tests help document, because the tests are code and code is a means of communication.</a> <a data-t="774">They kind of help describe the expected behavior of the functions as well.</a> <a data-t="779">But sometimes describing the expected output is just like really annoying, like if it contains</a> <a data-t="783">a bunch of special characters, like quotes or backslashes, you have to spend a bunch</a> <a data-t="788">of time like carefully escaping them.</a> <a data-t="791">And then when something goes wrong, it's hard to see exactly, like you've got to unescape</a> <a data-t="795">them in your head, and it's just a pain.</a> <a data-t="797">Or maybe it's very large, like maybe you want to check an entire HTML page or multiple paragraphs</a> <a data-t="803">of text, as you expect.</a> <a data-t="806">Or maybe it's not even something that you can easily describe with text.</a> <a data-t="810">It's an image.</a></p>

<p><a data-t="813">So the idea of a snapshot or golden test, which idea used in other programming languages</a> <a data-t="820">and other testing packages, and test that inspiration draws primarily from Jest, which</a> <a data-t="832">is a JavaScript testing package that Joe Chang shared his experiences with me a bunch</a> <a data-t="838">and persuaded me that this was something really useful.</a></p>

<p><a data-t="842">But so the snapshot test, the key idea is that instead of recording the results inline</a> <a data-t="847">in the test itself, they're stored in a separate file.</a> <a data-t="851">And test that provides a bunch of tools for looking after that, for managing that file.</a> <a data-t="857">So it will create it automatically the first time you run it, and then it gives you tools</a> <a data-t="861">to update it when you decide there really has to be a change.</a> <a data-t="865">And if you've used verify output or expect known output, which we've never really advertised</a> <a data-t="869">because we've never really been particularly happy with them, the snapshot tests basically</a> <a data-t="874">supersede those functions.</a></p>

<p><a data-t="877">So what does a snapshot test look like?</a> <a data-t="881">So here I'm just going to give you a quick simulation in the presentation, and I'll show</a> <a data-t="885">you what this looks like in a real package.</a> <a data-t="888">So here I've got two files, foo.r, I've just got this very silly and simple function, which</a> <a data-t="894">has a mistake in it, which you can see, which we'll fix shortly, and then I've got a test.</a> <a data-t="900">So I run foo, I expect it as a character vector, and then I'm going to use this new expectation,</a> <a data-t="908">expect snapshot output, and this doesn't have what the expected value is, because it's going</a> <a data-t="913">to save it to test.</a></p>

<p><a data-t="915">So the first time I run this, it's going to say, warning, create snapshot reference,</a> <a data-t="921">and it creates a new file, and I forgot to put the correct directory in here, inside</a> <a data-t="931">the test that directory, it's going to create a new directory called snaps, underscore snaps,</a> <a data-t="936">and inside that, it's going to have a file called foo.md, right?</a> <a data-t="940">So our R file is called foo.r, our test file is called test-foo.r, and then our snapshot</a> <a data-t="947">for that, the snapshot file is going to be called foo.md.</a> <a data-t="951">So all of these are just named, have a strong naming convention, so you can easily find</a> <a data-t="956">which snapshot corresponds to which test, which corresponds to which R file.</a></p>

<p><a data-t="961">So what does that snapshot contain?</a> <a data-t="963">Well, it's a markdown file, I'll kind of explain the syntax shortly, but we use a heading to</a> <a data-t="968">indicate that this is the test, and then it's going to put the output of that test directly in that file.</a> <a data-t="978">So now if I run it again, if I run that test again, the test is going to work, because</a> <a data-t="983">nothing's changed, and so the test will pass.</a> <a data-t="988">If I change it to fix that typo, I run the test again, I'm getting an error, saying that</a> <a data-t="995">the previous value was something conflicted, and the new value is something complicated.</a></p>

<p><a data-t="1001">Now the downside of snapshot tests is that there's no way to know what is correct, so</a> <a data-t="1007">you as a human now have to step in and intervene.</a> <a data-t="1010">So if this is correct, if you really did mean to make this change, you can run snapshot</a> <a data-t="1016">accept, and that will accept the change.</a> <a data-t="1020">And the way that this works is that when you have this value, when something is changed,</a> <a data-t="1024">there will be a new markdown file, which contains the new value.</a> <a data-t="1029">If you accept it, then it will replace that with the old value.</a></p>

<p><a data-t="1037">So let's just dive into that for a slightly more realistic example.</a> <a data-t="1043">Start with a simple one.</a> <a data-t="1045">So I have this package, I have the same test which I copied before, and I'm going to run this test.</a> <a data-t="1061">And I'm going to run this test by pressing commands T, which is a shortcut for devtools test file.</a> <a data-t="1069">So this keyboard shortcut takes advantage of that convention, that if you've got a file</a> <a data-t="1076">called foo.r, the corresponding test will be test.foo.</a></p>

<p><a data-t="1090">And so this says I've added a new snapshot, this is the value of that snapshot, I can</a> <a data-t="1094">look at it, there's this markdown file again, which I'll explain shortly.</a> <a data-t="1101">And then if I run that test again, you'll see that all the tests have passed.</a></p>

<p><a data-t="1102">The other thing, if you've used devtools test file before, which I'll talk about briefly</a> <a data-t="1106">later, is there's now a slightly different display, a slightly more compact display to</a> <a data-t="1111">hopefully make it easier to run tests for a single file interactively.</a></p>

<p><a data-t="1117">So I'm going to change this, I'm going to correct that typo, I'm going to test again,</a> <a data-t="1124">and now the test fails because the snapshot has changed.</a> <a data-t="1128">So it shows you the current value, which is something complicated, and it shows you the</a> <a data-t="1133">previous value, which is something complicated.</a> <a data-t="1136">Now, if this is a deliberate change, you can run snapshot accept.</a> <a data-t="1141">If this wasn't a deliberate change, oh, that was just a typo, I can fix it and rerun and</a> <a data-t="1147">all my tests pass again.</a></p>

<p><a data-t="1149">So if you've ever used verify output, this is a little bit different because verify output</a> <a data-t="1153">automatically updates the kind of true known value on disk, which forced you to use it</a> <a data-t="1159">with GIT, basically.</a> <a data-t="1161">So I'm going to fix this, the test fails, you can see now in my snaps directory, I've</a> <a data-t="1168">got foo, that's the foo.md, that's the previous value, and I've got foo.new.md, which is the new value.</a></p>

<p><a data-t="1262">Okay, so let's look at a slightly more complicated example.</a> <a data-t="1266">So this is a bullets function, and it's basically used to create HTML bullets.</a></p>

<p><a data-t="1280">And this is like a little annoying to test, because if you're going to put this in a test,</a> <a data-t="1290">you'd have to escape all of these new lines and kind of carefully manage all the white</a> <a data-t="1294">spaces, it's just a pain.</a> <a data-t="1296">To test this code, it's a little easier to use a snapshot test.</a></p>

<p><a data-t="1322">So I can run this test, and now here is a test snapshot file with multiple expectations in it.</a> <a data-t="1333">The first one, I am just using a bullet with a single bullet, and the second also has a</a> <a data-t="1340">single bullet and also sets the ID.</a> <a data-t="1342">So if I later want to change this bullets function, maybe I've decided actually I don't want this indent.</a> <a data-t="1348">Let's get rid of that.</a> <a data-t="1350">I rerun the test.</a> <a data-t="1353">Again, the changes use waldo to highlight what's changed.</a> <a data-t="1356">So it's a little bit hard to see this, but this is the thing that's the same.</a> <a data-t="1361">It's in gray.</a> <a data-t="1362">What's changed?</a> <a data-t="1364">We can see there is now no space where there was before.</a></p>

<p><a data-t="1380">So I look at this, I decide, yep, that's a deliberate change, and then I run snapshot</a> <a data-t="1385">except bullets to update that snapshot.</a></p>

<p><a data-t="1391">And that basically is snapshot testing.</a> <a data-t="1396">Of course, there's more documentation, and I think it will take a little while to get your head around it all.</a> <a data-t="1402">I've shown you expect snapshot output here.</a> <a data-t="1402">There's also expect snapshot, which, as well as printed output, also captures messages,</a> <a data-t="1405">warnings, errors.</a> <a data-t="1407">There's also expect snapshot error if you want to capture specifically just error messages.</a> <a data-t="1412">And then we've got expect snapshot value, which captures return value.</a> <a data-t="1416">So this is a little bit different.</a> <a data-t="1418">This is if you, for example, wanted to test the output from a complicated function, you</a> <a data-t="1424">just want to make sure it doesn't change without warning.</a> <a data-t="1426">You can use expect snapshot value.</a></p>

<p><a data-t="1435">So the only other thing to mention is just what do these files look like?</a> <a data-t="1435">Why are they markdown files?</a> <a data-t="1437">Well, if you're writing this, this package is a collaborative project and you're using</a> <a data-t="1442">GitHub or some other tool where you do code review, it's really important to make sure</a> <a data-t="1448">it's really important that these snapshots be human readable because when someone goes</a> <a data-t="1452">to review your code, they need to go look at the snapshot and say, is this a reasonable</a> <a data-t="1456">change or not?</a> <a data-t="1457">So they use markdown.</a> <a data-t="1459">Each file, again, is there's one snapshot file per test file, which normally corresponds</a> <a data-t="1465">to our file in the R directory.</a> <a data-t="1468">There's a heading for the test name.</a> <a data-t="1470">And then if you have multiple snapshot expectations and a single test, they're separated by the</a> <a data-t="1475">horizontal row with three dashes.</a></p>

<p><a data-t="1487">One thing that I'm working on at the moment and some help from Joshua Kunst is a </a><a href="/software/shiny-r/">Shiny</a> <a data-t="1492">app that will help you review all of these differences and accept them or reject them</a> <a data-t="1497">with by clicking buttons rather than typing stuff at the console.</a> <a data-t="1501">And the other big part of that is providing tools for doing image snapshots as well.</a> <a data-t="1506">So this is something that we've implemented in two places and Shiny test, which is used</a> <a data-t="1511">for testing Shiny and VDiffer, which is used for testing </a><a href="/software/ggplot2/">ggplot2</a><a data-t="1511">.</a> <a data-t="1515">The idea is they're going to pull out the common code, centralize and test that and</a> <a data-t="1519">invest a bunch in this whole snapshotting idea.</a> <a data-t="1522">So you've got a really nice workflow if you do need to do image tests.</a></p>

<p><a data-t="1525">Image tests are complicated because they can change for all sorts of reasons unrelated to your code.</a> <a data-t="1532">But sometimes they're all you have and they are a really important part of testing both</a> <a data-t="1537">Shiny itself and ggplot2 itself.</a></p>

<h3>Parallel testing</h3>

<p><a data-t="1572">Next, I wanted to talk briefly about parallel tests.</a> <a data-t="1576">This is still work in progress by Gabor Chadi.</a> <a data-t="1580">And again, you're going to have to activate this specifically for your package again by</a> <a data-t="1585">putting another line in your description, config, test that parallel true.</a> <a data-t="1590">But the payoff for this is pretty big.</a> <a data-t="1592">It's going to run your tests on multiple processes.</a> <a data-t="1606">So if you have long running tests, this is going to make a big improvement to the total</a> <a data-t="1614">running time of your tests.</a></p>

<p><a data-t="1616">Now, the downside is like there's a little bit of overhead associated with that starting</a> <a data-t="1620">out multiple R packages, multiple R processes, loading all the code in those.</a> <a data-t="1627">So there's a little bit of overhead.</a> <a data-t="1628">So if your tests are very fast, like if your tests all run in under a second, it's probably</a> <a data-t="1632">not going to have a huge impact.</a> <a data-t="1635">But if you've got tests that take five or ten seconds, the whole test suite takes minutes,</a> <a data-t="1640">then this should hopefully have a really big positive impact on your workflow.</a></p>

<p><a data-t="1645">There are some downsides.</a> <a data-t="1649">As well as this big upside of speed, tests will now kind of effectively run in stochastic</a> <a data-t="1654">order because you'll have like four processes and each process takes kind of the next test</a> <a data-t="1661">in the queue.</a> <a data-t="1662">And so depending on exactly how long each test takes to run, the tests might get run</a> <a data-t="1668">in a different order.</a> <a data-t="1670">And this basically means if you have any dependency between your tests, which is relatively</a> <a data-t="1676">easy to introduce because normally they will run in alphabetical order.</a> <a data-t="1680">If there is any dependency between your test files, you'll get like random test failures</a> <a data-t="1684">that occur sometimes but don't occur other times depending on exactly which order the tests are run in.</a> <a data-t="1691">In other words, this is like a dependency debugging nightmare.</a></p>

<p><a data-t="1695">So we're still thinking about tools to kind of ensure that if that happens to you, we've</a> <a data-t="1699">got some mode debugging mode you can switch on to get more insight.</a> <a data-t="1704">And then also, if you have used like global setup or teardown, for example, like you set</a> <a data-t="1708">up a database or some CSV file for all of your tests, you're going to need to think</a> <a data-t="1715">that through in a little bit more detail because those setup and teardown files are</a> <a data-t="1720">now run by multiple processes.</a> <a data-t="1724">So there's going to be a little bit of work to convert your tests to use Parallel.</a> <a data-t="1728">We're still working on this.</a> <a data-t="1729">You know, there's a vignette if you want to learn more and want to try it out.</a> <a data-t="1732">But we're hopeful that this will make a big impact if you've got long running tests.</a></p>
