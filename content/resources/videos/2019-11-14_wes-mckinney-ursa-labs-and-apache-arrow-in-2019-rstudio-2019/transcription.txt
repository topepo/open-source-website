<h3>The problem with current data science tools</h3>

<p><a data-t="129">So the executive summary of what this is all about is that if you look at how data science</a> <a data-t="135">tools work now, like Base R, NumPy, SciPy, the Python ecosystem, MATLAB, a lot of the</a> <a data-t="143">semantics of how these tools work are based on the way that computational systems worked</a> <a data-t="149">in the 1980s and 1990s, so everything's built on a foundation of 30- to 40-year-old Fortran</a> <a data-t="156">code.</a></p>

<p><a data-t="158">We have seen tools augmented with multi-core algorithms and some more sophisticated execution,</a> <a data-t="167">but in general, you have things that are single-core, that evaluate eagerly.</a> <a data-t="174">The language ecosystems are pretty fragmented, so you very rarely see R programmers collaborating</a> <a data-t="180">with Python programmers and sharing libraries of code to power their applications.</a></p>

<p><a data-t="187">So the world that I would like to see is to be able to take advantage of all this fancy</a> <a data-t="190">hardware that we have now.</a> <a data-t="191">We have graphics cards, we have computers, you can buy a desktop with 16 physical cores</a> <a data-t="197">and 32 virtual cores, so you should be able to take advantage of that.</a> <a data-t="202">We can compile code at runtime and generate custom functions that are specialized for</a> <a data-t="207">your hardware, but a big part also is that I would like to see the language ecosystems</a> <a data-t="213">able to collaborate and share code and data.</a></p>

<p><a data-t="217">So one reason that not as much of this work has happened yet has been because of Moore's</a> <a data-t="222">Law and, you know, if your code runs slow, you can just wait because in a couple of years</a> <a data-t="228">your processor will be twice as fast and that has essentially stopped happening, so we need</a> <a data-t="233">to get more out of the hardware that we have because it's not getting that much faster</a> <a data-t="239">year over year.</a> <a data-t="241">Although we do have graphics cards, which seemingly are, you know, continuing to scale</a> <a data-t="246">and have thousands of cores on a single graphics card.</a></p>

<h3>Apache Arrow's vision</h3>

<p><a data-t="251">So the Arrow project, so I helped launch the project about three years ago and our central</a> <a data-t="258">idea is that we wanted to define a language independent open standard for data frames.</a> <a data-t="265">And you might say, well, we have data frames in R and data frames in Python, but if you</a> <a data-t="271">look under the hood how those data frames are implemented, how they're arranged in memory,</a> <a data-t="278">the memory representation is very different.</a> <a data-t="280">And so as a result of that, you can't really share code and if you want to share data from</a> <a data-t="285">one runtime environment to another, you have to serialize, which is very expensive.</a></p>

<p><a data-t="291">So essentially the view of the world that we would like to see is to have tools able</a> <a data-t="296">to collaborate and work on a common data representation that is not tied to a particular runtime environment</a> <a data-t="304">or programming language.</a></p>

<blockquote><a data-t="291">So essentially the view of the world that we would like to see is to have tools able</a> <a data-t="296">to collaborate and work on a common data representation that is not tied to a particular runtime environment</a> <a data-t="304">or programming language.</a></blockquote>

<p><a data-t="307">So kind of the bird's eye view, if you think, and where the project is going, and since</a> <a data-t="314">I have zero time now, but what I would like to do and the way that we've been building</a> <a data-t="321">the community is to bring together the data science world with the analytic database community.</a> <a data-t="328">So a lot of the performance and scalability work in working with big data, I don't know</a> <a data-t="335">if we're still saying big data now, a lot of that work has happened in the analytic</a> <a data-t="339">database world and relatively little work has made its way into the data science ecosystem.</a></p>

<p><a data-t="344">And if you look at the way that database engines are built, typically you interact with them</a> <a data-t="349">through SQL or some other frontend API, usually it's SQL queries, but those queries then interact</a> <a data-t="356">with a bunch of components that deal with query execution, in memory, data management,</a> <a data-t="363">and interactions with storage and essentially whether it's disk or cloud storage or whatever.</a> <a data-t="370">So what we were doing in the Arrow project is building a set of high-quality components</a> <a data-t="375">that have public APIs that can be used as the basis of building different kinds of essentially</a> <a data-t="381">embedded database engines that can be run on a laptop or run kind of in a distributed</a> <a data-t="387">cluster.</a></p>

<p><a data-t="389">But we aren't really being prescriptive about the frontend, so if you want to build a database,</a> <a data-t="394">you can build a database, if you want to build a data frame library or build something that</a> <a data-t="399">can be used by an R or Python programmer, you can do that as well.</a></p>

<h3>Primary use cases</h3>

<p><a data-t="405">So we have three primary use cases, and I'm going to tell you very briefly about how this</a> <a data-t="410">is relevant to R programmers, but the three big things that we're concerned with is getting</a> <a data-t="416">faster access to data, so being able to read and write datasets in the most important file</a> <a data-t="423">formats as fast as possible.</a> <a data-t="426">We want to be able to move around data as efficiently as possible within a single node,</a> <a data-t="431">so between one process and another through shared memory, or moving data between nodes</a> <a data-t="436">in a network through remote procedure calls, so through sockets or other inter-node communication.</a></p>

<p><a data-t="446">The last, so the last thing is that we want to be able to compute analytics, and so if</a> <a data-t="452">you're using R, you're using Python, you're already doing analytics every day when you</a> <a data-t="456">use data frame libraries, so these would be standardized libraries of algorithms that</a> <a data-t="464">could be used in many different scenarios, and we want to be able to take advantage of</a> <a data-t="469">the latest and greatest in compiler technology to be able to do specialization of functions</a> <a data-t="475">to go even faster based on prior knowledge of what you're doing with your data.</a></p>

<h3>Relevance to R</h3>

<p><a data-t="481">So how is this all relevant to R?</a> <a data-t="483">Well, we are building bindings for the Arrow libraries, which are mostly written in C++,</a> <a data-t="491">so that work on the R side has been led by Romain Francois, who you may know from the</a> <a href="/software/dplyr/">dplyr</a><a data-t="496"> project.</a></p>

<p><a data-t="500">So I will talk about R data frames, because I have no time, but the key use cases where</a> <a data-t="508">there's a lot of interest in the R world is being able to memory map and interact with</a> <a data-t="513">very large on-disk data sets that don't fit into memory.</a> <a data-t="517">We want to be able to, this is not about replacing dplyr or replacing data.table, it's</a> <a data-t="522">about having tools that can be used to accelerate these projects while leaving the user interface,</a> <a data-t="530">like with the code that you write, largely unchanged.</a></p>

<p><a data-t="534">So you may know about dplyr, dplyr is already a domain-specific language where you write</a> <a data-t="541">a sequence of dplyr verbs and expressions, and then those expressions are effectively</a> <a data-t="548">compiled into whatever backend where dplyr is executing.</a> <a data-t="553">So dplyr can execute against SQL engines or it can execute using its in-memory engine.</a> <a data-t="561">And so what our goal is to do is if you do have a very large data set which is coming</a> <a data-t="567">out of code in the arrow ecosystem, whether it's memory mapped on-disk data, or maybe</a> <a data-t="572">you have a directory of parquet files, or maybe a set of cloud buckets that contain</a> <a data-t="578">parquet files.</a></p>

<p><a data-t="580">So using dplyr, you may have an arrow data set and a collection of dplyr verbs, and so</a> <a data-t="588">we can translate these standard dplyr expressions into computation graphs that are evaluated</a> <a data-t="595">by parallel runtime coming out of the arrow project, and we can also do fancy things like</a> <a data-t="600">look at non-standard evaluation, kind of unevaluated R expressions.</a> <a data-t="606">So when you write this in dplyr, these expressions don't get evaluated immediately, but we can</a> <a data-t="612">use LLVM to compile these expressions to go even faster against the data sets.</a></p>
