<h3>What testthat does</h3>

<p><a data-t="119">So testthat's been around for about 15 years,</a> <a data-t="122">and its job is to be a code tester.</a> <a data-t="125">So you want to use this during package development</a> <a data-t="128">to test your code for specific conditions,</a> <a data-t="130">like warnings and errors,</a> <a data-t="132">or test for expected outputs, like specific numbers.</a></p>

<p><a data-t="138">The thing is, you're probably already doing testthat's job.</a> <a data-t="142">So as an example, we're going to look into</a> <a data-t="145">what I've been developing for our package.</a> <a data-t="149">This is Tidy Water.</a> <a data-t="150">It's on CRAN if you want to check it out.</a> <a data-t="153">And as an environmental engineer,</a> <a data-t="155">we work with drinking water treatment plants</a> <a data-t="158">to optimize their treatment processes</a> <a data-t="161">to provide better water quality to the general public.</a></p>

<p><a data-t="166">So this first function here, define water,</a> <a data-t="169">is what we start with for every single modeling scenario.</a> <a data-t="173">And this creates a water.</a> <a data-t="175">This is like making a cup of water</a> <a data-t="177">with specific parameters like pH, temperature, and alkalinity.</a> <a data-t="182">I can then pipe my define water function into other models.</a> <a data-t="187">So in this case, chem dose pH,</a> <a data-t="189">which calculates the pH and alkalinity</a> <a data-t="191">after we dose a chemical.</a> <a data-t="193">So here we're dosing alum.</a> <a data-t="196">We dose this at the beginning of most treatment processes</a> <a data-t="199">to coagulate out a bunch of dissolved material in the water</a> <a data-t="202">that you don't want to ingest.</a></p>

<p><a data-t="205">So if I want to test that this chem dose pH function</a> <a data-t="208">is operating as expected,</a> <a data-t="210">I might run this code chunk and then say,</a> <a data-t="213">what's my water at alkalinity?</a> <a data-t="216">And I can see that it's dropped from that 33 starting value</a> <a data-t="219">down to 17.</a> <a data-t="221">But with testthat, all I have to do is wrap all of this code</a> <a data-t="225">in a testthat function,</a> <a data-t="227">label it so I remember what I'm testing.</a> <a data-t="229">So test that chem dose pH returns the expected values,</a> <a data-t="233">and then replace that water alkalinity</a> <a data-t="236">with a testthat function like expect equal.</a> <a data-t="239">And here I'm expecting that my calculation</a> <a data-t="242">from that code chunk equals a specific number, 17.87.</a></p>

<p><a data-t="248">So testthat seems kind of easy to use,</a> <a data-t="251">kind of interested so far.</a> <a data-t="253">And if you're interested too,</a> <a data-t="255">you can get started at this QR code,</a> <a data-t="257">and that will take you to that package development book</a> <a data-t="261">by Jenny and Hadley, specifically to this testing basics.</a> <a data-t="265">It's a little bit more involved</a> <a data-t="267">than just doing install packages, testthat.</a> <a data-t="271">But you can go to this link,</a> <a data-t="273">and it'll tell you the exact steps to get through,</a> <a data-t="276">and it's not too difficult.</a></p>

<h3>First date: testing inputs and outputs</h3>

<p><a data-t="278">So to get to know testthat a little bit more,</a> <a data-t="280">we're going to go on a few dates.</a> <a data-t="282">So our first date will be a coffee date, super simple,</a> <a data-t="285">get to know testthat a little superficially.</a> <a data-t="288">Then we'll go on a second date hike</a> <a data-t="290">where we start to trust testthat a little bit more.</a> <a data-t="293">And finally, we're really going to commit to testthat</a> <a data-t="296">and get into a late relationship dinner with the parents.</a></p>

<p><a data-t="300">So this first date,</a> <a data-t="302">we're just going to test function inputs and outputs</a> <a data-t="305">and understand some of the functions</a> <a data-t="307">inside of the testthat package.</a> <a data-t="309">So this first test is testing</a> <a data-t="311">that define water warns when no pH is input.</a> <a data-t="315">So every water will have a pH,</a> <a data-t="319">and because of that,</a> <a data-t="321">all of our downstream models require pH.</a> <a data-t="324">So if I don't include pH in my define water,</a> <a data-t="327">I should get a warning,</a> <a data-t="329">and the reason it's giving a warning</a> <a data-t="331">is because in my define water code,</a> <a data-t="334">I have this code that says,</a> <a data-t="336">if you're missing pH, throw a warning.</a></p>

<p><a data-t="339">But you can see I have a second warning on the bottom</a> <a data-t="341">if I'm missing alkalinity.</a> <a data-t="343">So if I want to be really specific</a> <a data-t="345">and make sure that the correct warning being thrown</a> <a data-t="347">is related to pH, I can add some regex,</a> <a data-t="350">or I could add the entire warning message.</a> <a data-t="354">Similarly, you can use expect error.</a> <a data-t="356">In this case, I've entered pH as a character</a> <a data-t="359">instead of a number.</a> <a data-t="361">So expect warning and expect error</a> <a data-t="363">are really useful for evaluating the outcomes</a> <a data-t="365">of your conditional statements.</a> <a data-t="368">It's also useful for figuring out</a> <a data-t="370">when your user is breaking your code.</a></p>

<p><a data-t="375">Next we have testing that just define water works.</a> <a data-t="380">So testthat has a lot of really intuitive function names.</a> <a data-t="384">So expect S3 class.</a> <a data-t="387">Well, we expect the code inside of this function</a> <a data-t="390">to output an S3 class.</a> <a data-t="392">You can also expect S4 class or an S7 class.</a> <a data-t="396">You can also expect true.</a> <a data-t="398">So here I'm saying my water pH should be a numeric.</a> <a data-t="402">There's also an expect false function.</a> <a data-t="405">And finally, there's this expect LT, expect less than.</a> <a data-t="409">So my pH should be less than 14.</a> <a data-t="411">Similarly, there's an expect </a><a href="/software/gt/">GT</a><a data-t="411">,</a> <a data-t="413">which means expect greater than.</a> <a data-t="415">So you can see these testthat functions</a> <a data-t="417">are really intuitive, easy to understand</a> <a data-t="419">without even getting into the documentation too much.</a></p>

<p><a data-t="423">Finally, we have what I showed you earlier.</a> <a data-t="425">You can write a whole chunk of code</a> <a data-t="427">inside of a test that function</a> <a data-t="429">and then evaluate the outputs from that code chunk.</a> <a data-t="432">In this case, we're using expect equal again,</a> <a data-t="434">but you'll notice I've set that alkalinity to 200,</a> <a data-t="437">which is really high.</a> <a data-t="439">So we should be getting an error.</a> <a data-t="441">So we're going to take a little sidestep</a> <a data-t="443">into how testing is useful for error catching.</a></p>

<p><a data-t="447">So in your console, you can run </a><a href="/software/devtools/">DevTools</a><a data-t="447"> test.</a> <a data-t="451">And what this does is starts running</a> <a data-t="453">through all of your testing code.</a> <a data-t="456">And what's going to show up is a yellow F for failure,</a> <a data-t="461">a pink W for warning, and an S for skipped.</a> <a data-t="465">And in this left column, you can see the number of tests</a> <a data-t="468">that are run for each function you've developed.</a> <a data-t="471">So I ran 33 tests for balanced ions,</a> <a data-t="473">25 tests for biofilters,</a> <a data-t="475">and every single one of those tests,</a> <a data-t="477">my team manually wrote.</a></p>

<h3>Second date: testing interconnected functions</h3>

<p><a data-t="513">So this is pretty cool.</a> <a data-t="515">What else can testthat do?</a> <a data-t="517">So on a second date, we're going on this hike.</a> <a data-t="519">We're trusting that it's not going to push us</a> <a data-t="521">off a cliff in the wilderness,</a> <a data-t="523">but rather look for obstacles</a> <a data-t="526">and prevent us from tripping over them.</a></p>

<p><a data-t="528">So as I mentioned earlier,</a> <a data-t="530">we always start with define water in our package,</a> <a data-t="532">creating that first cup of water.</a> <a data-t="534">And into that cup of water,</a> <a data-t="536">I can dose chemical A and chemical B</a> <a data-t="538">and get a resulting pH and alkalinity.</a> <a data-t="541">But there's another way I can get to this same answer.</a> <a data-t="544">In one cup of water, I can dose chemical A.</a> <a data-t="547">In a second cup of water, I can dose chemical B.</a> <a data-t="550">I can mix them together using our blend waters function,</a> <a data-t="553">and I should get that same pH and alkalinity.</a></p>

<p><a data-t="556">So this shows how your package</a> <a data-t="559">can be really interconnected.</a> <a data-t="561">And we really want our tests to fail.</a> <a data-t="564">If something changes in that upstream function,</a> <a data-t="566">like chem dose pH,</a> <a data-t="568">I want my blend waters to either reflect that change,</a> <a data-t="571">or if it doesn't, my test should fail.</a></p>

<p><a data-t="578">So what does this look like using testthat?</a> <a data-t="581">So in this chunk of code,</a> <a data-t="581">we're going to test that this blend waters</a> <a data-t="583">handles our equilibrium chemistry correctly.</a> <a data-t="585">So first, I make that starting water quality.</a> <a data-t="588">I dose two chemicals into that first cup.</a> <a data-t="591">Then I dose them separately and blend them</a> <a data-t="593">using blend waters.</a> <a data-t="595">And then I expect that my pH using water B</a> <a data-t="598">should be the same as my pH from my blended waters.</a> <a data-t="601">So this is a step up from testing hard-coded inputs.</a> <a data-t="607">So instead of saying my pH should be 7,</a> <a data-t="610">why is it 7, what does 7 mean,</a> <a data-t="612">what else can I compare this to?</a> <a data-t="614">So this is really helping us</a> <a data-t="616">make our package more resilient</a> <a data-t="618">and account for the interconnectivity.</a></p>

<h3>Late relationship: committing to tests during refactoring</h3>

<p><a data-t="622">So now that we know that tests</a> <a data-t="624">can catch hard-coded inputs like numbers,</a> <a data-t="627">it can test our conditional statements,</a> <a data-t="630">it can test the interconnectivity,</a> <a data-t="632">we can really start to commit to testthat</a> <a data-t="635">and see how far we can fall.</a></p>

<p><a data-t="638">So here we're in the late relationship stage.</a> <a data-t="641">And we're just going to trust that our tests catch everything.</a> <a data-t="645">In the example here, we're going to do a refactor.</a> <a data-t="648">And refactoring is just changing your code</a> <a data-t="653">to maybe be more organized,</a> <a data-t="655">but your outputs shouldn't change.</a> <a data-t="657">So this might mean changing your naming consistency</a> <a data-t="661">or adjusting your functions</a> <a data-t="663">to reflect the standards of </a><a href="/software/tidyverse/">Tidyverse</a><a data-t="663">.</a></p>

<p><a data-t="668">The detriment is that you could introduce bugs,</a> <a data-t="671">and hopefully your tests will catch that.</a> <a data-t="673">In our case, we needed to really speed up our package.</a> <a data-t="677">It was very slow when we were running thousands of scenarios.</a> <a data-t="680">So to do this, we changed all of our functions</a> <a data-t="683">from Tidyverse functions to Base R.</a> <a data-t="686">The thing is, our entire team learned R using Tidyverse,</a> <a data-t="690">so we're a little rusty on Base R.</a> <a data-t="693">So you can imagine there's a lot of room for errors to sneak in.</a></p>

<p><a data-t="697">So as an example, we have this code chunk here</a> <a data-t="700">where I've highlighted the different Tidyverse functions</a> <a data-t="704">that we know and love.</a> <a data-t="706">But when we change that to Base R,</a> <a data-t="708">suddenly there's three separate lines of code,</a> <a data-t="711">there's a lot of brackets and dollar signs.</a> <a data-t="714">But the thing is, as I'm running this,</a> <a data-t="717">and I'm a little unsure about Base R,</a> <a data-t="720">I'll make sure that this works by running each line of code.</a> <a data-t="723">And the thing is, it does.</a> <a data-t="725">I don't run into any errors.</a> <a data-t="727">The problem is, when I run my tests, I do get an error.</a> <a data-t="730">And that's because this merge is replacing a left join.</a> <a data-t="735">The merge default behavior is an inner join,</a> <a data-t="738">and to get that left join behavior,</a> <a data-t="741">I need this allX equals true.</a> <a data-t="743">So my tests were catching that I was dropping some data</a> <a data-t="746">that a left join would have kept</a> <a data-t="748">that this merge default was losing.</a></p>

<blockquote><a data-t="731">So my point is that you can code,</a> <a data-t="734">you might not see an error,</a> <a data-t="736">you might not account for edge cases</a> <a data-t="739">that your tests should catch.</a></blockquote>

<p><a data-t="751">So my point is that you can code,</a> <a data-t="753">you might not see an error,</a> <a data-t="756">you might not account for edge cases</a> <a data-t="759">that your tests should catch.</a> <a data-t="762">You might be lulled into a false sense of security as well</a> <a data-t="766">as you're coding that there's,</a> <a data-t="769">I don't need to run my tests, my code's working fine,</a> <a data-t="771">it's not breaking.</a></p>

<h3>Takeaways</h3>

<p><a data-t="912">So my takeaways are that you should use testthat</a> <a data-t="915">at the beginning of package development</a> <a data-t="918">because your tests help you manage complexity</a> <a data-t="921">as you get your package more interconnected and complicated.</a> <a data-t="927">It's easier to build coverage as you go</a> <a data-t="930">and a little harder to reverse engineer that later.</a> <a data-t="933">You might miss out on those different edge cases.</a> <a data-t="936">And building tests into your package</a> <a data-t="939">allows you to focus on cool development opportunities.</a> <a data-t="942">For us, that meant being able to speed up our package significantly</a> <a data-t="946">by refactoring without having to worry too much</a> <a data-t="949">about the consequences.</a></p>

<p><a data-t="951">For you, it might look like an open-source project</a> <a data-t="954">where you can crowdsource cool ideas and contributions</a> <a data-t="957">from people who aren't as familiar with the intricacies of your package</a> <a data-t="961">and not worry about how they might break your package.</a> <a data-t="965">Your tests should catch that.</a></p>

<p><a data-t="967">So overall, I was wrong to judge testthat so harshly.</a> <a data-t="972">Luckily, we committed to the package early</a> <a data-t="975">and developed a really positive relationship</a> <a data-t="978">with testthat and with package development.</a> <a data-t="981">So I'm hoping that if you're considering your first package</a> <a data-t="985">or you're in the early stages of developing a package,</a> <a data-t="988">that you too can find love with testthat</a> <a data-t="990">and start to feel like more of a real programmer.</a> <a data-t="993">Thank you.</a></p>
