<h3>The problem with assembling tools one by one</h3>

<p><a data-t="81">My name is Kamil.</a> <a data-t="85">I'm a full stack engineer at Appsilon.</a> <a data-t="89">We specialize in building </a><a href="/software/shiny-r/">Shiny</a><a data-t="89"> applications.</a> <a data-t="95">Let me tell you of a typical situation that we faced many times over in our work.</a></p>

<p><a data-t="101">So a client comes to us and asks whether we could build an application for them.</a> <a data-t="105">Using wonderful capabilities of Shiny, we create something within a couple of hours</a> <a data-t="109">that is interactive and impresses the client.</a> <a data-t="109">Contract signed.</a> <a data-t="111">The project starts.</a> <a data-t="112">A couple of weeks pass.</a> <a data-t="114">The code base grows 20 times.</a> <a data-t="116">And realize that we are spending a lot of time discussing the code style, where the</a> <a data-t="122">opening brackets should be.</a></p>

<p><a data-t="128">We figure that we need some tool that will automatically check this for us.</a> <a data-t="129">We need a linter.</a> <a data-t="132">So what do we do?</a> <a data-t="137">We find linter, we configure it properly, we add it to the project, we start linting</a> <a data-t="140">our code, we're good.</a></p>

<p><a data-t="142">Some time passes.</a> <a data-t="148">We realize that we touch one place in the application and something else breaks.</a> <a data-t="151">So we figure that we should start writing tests.</a> <a data-t="152">We need unit tests.</a> <a data-t="153">What do we do?</a> <a data-t="153">We find test that, we configure it, we add it to the project, we start writing unit tests.</a> <a data-t="159">We're good.</a></p>

<p><a data-t="161">Some time passes.</a> <a data-t="163">And you get the point.</a> <a data-t="165">Needs keep arising and we need to keep addressing them.</a> <a data-t="168">You start wondering, wouldn't it make sense to start with a complete development toolbox</a> <a data-t="174">instead of assembling one tool by tool?</a></p>

<p><a data-t="178">This goes deeper, though.</a> <a data-t="180">In case of development tools, introducing them later on in the project is typically</a> <a data-t="185">more costly.</a> <a data-t="186">For example, if you want to add linter when you already have hundreds of linter errors,</a> <a data-t="191">you'll first have to fix them all.</a> <a data-t="193">And that's not only a lot of work, but you need to synchronize it with your team.</a></p>

<p><a data-t="199">But we should take even a wider perspective, though.</a> <a data-t="201">It's not just one project.</a> <a data-t="203">It's multiple teams dealing with multiple projects at the same time.</a> <a data-t="208">And similar needs arise in each of those projects.</a> <a data-t="211">And each team keeps configuring the same tools.</a> <a data-t="214">Wouldn't it be wonderful to have just one universal toolbox?</a> <a data-t="218">All this effort could be directed into really polishing it instead of reinventing the wheel</a> <a data-t="223">all the time.</a></p>

<p><a data-t="224">Moreover, in an organization, when you have just one universal toolbox, people get proficient</a> <a data-t="230">using it.</a> <a data-t="231">So when you start a new project or move to a different one, everybody's up to speed right</a> <a data-t="236">from the beginning.</a> <a data-t="237">This can really bring the productivity to a whole new level.</a></p>

<h3>The origin of Rhino</h3>

<p><a data-t="241">So you can see what kind of problem we have here.</a> <a data-t="245">And this is where my personal story begins.</a> <a data-t="248">Right here in the middle.</a> <a data-t="250">So I joined Epsilon two years ago in July.</a> <a data-t="253">It was a hot month.</a> <a data-t="255">My birthday was coming.</a> <a data-t="257">It was the last month of our only office being open.</a> <a data-t="260">Pandemic was raging.</a> <a data-t="261">Lots of things were happening.</a></p>

<p><a data-t="264">So my first project was called Project Patterns.</a> <a data-t="268">What was that?</a> <a data-t="269">Well, technically speaking, it was a Git repository with snippets of code on branches and a related</a> <a data-t="276">Python tool that could be used to combine them into project structure for new and existing</a> <a data-t="281">projects.</a> <a data-t="282">Sounds complicated?</a> <a data-t="283">It was.</a> <a data-t="285">And it was my task to improve it.</a></p>

<p><a data-t="291">So first of all, I tried to understand the purpose of all this.</a> <a data-t="296">And I started researching what we tried to do about this in the past.</a> <a data-t="299">And it turned out that we did a lot.</a> <a data-t="302">Like years before already, we had discussion groups.</a> <a data-t="307">And we had a repository of code snippets that could be reused to some extent.</a> <a data-t="310">Directly before Project Patterns, we had something called...</a> <a data-t="314">We had a setup involving </a><a href="/software/rstudio/">RStudio</a><a data-t="314"> in Docker and a related bash tool that could be used</a> <a data-t="317">to automate some of the things.</a></p>

<p><a data-t="323">So I realized that already many people before me tried to solve these problems that we had.</a> <a data-t="325">And I had a week to improve that.</a> <a data-t="329">You can guess that I didn't manage to fix everything within that week.</a> <a data-t="333">But for the following two years, we kept improving what we had.</a> <a data-t="339">For example, we went through a phase of simplification, where instead of having all this complex set</a> <a data-t="343">of tools, we created just a simple, shiny template.</a> <a data-t="347">A Git repository that could be cloned and used as a basis for your project.</a></p>

<p><a data-t="349">We also researched other tools.</a> <a data-t="351">For example, we took a look at Golem.</a> <a data-t="354">We tried to learn how it differs from our approach.</a> <a data-t="355">We tried to learn from it.</a> <a data-t="355">We considered whether we could just use it.</a> <a data-t="359">For two years, we kept addressing needs as they were arising, resolving issues as they</a> <a data-t="364">were surfacing.</a> <a data-t="365">And finally, in the year 2022, we started feeling that we have something that we want</a> <a data-t="370">to share with the world.</a> <a data-t="372">And thus, Rhino was born.</a></p>

<p><a data-t="375">No, not just Rhino.</a> <a data-t="377">Rhino, the free and open-source R package, which we announced in May this year during</a> <a data-t="383">Appsilon Shiny Conference.</a></p>

<h3>What's inside Rhino</h3>

<p><a data-t="388">So it's an R package.</a> <a data-t="389">Means it has functions.</a> <a data-t="392">For example, this Rhino init function.</a> <a data-t="394">It creates a new Rhino project for you.</a> <a data-t="398">So what do you get inside?</a> <a data-t="403">Perhaps the most obvious part is the full project structure of files and directories.</a> <a data-t="409">So we have places for configuration, source files, dependencies, tests.</a> <a data-t="414">Everything is covered from the beginning.</a> <a data-t="416">So let's take a deeper look at some of those.</a></p>

<p><a data-t="420">Any project will need other R packages to do its job.</a> <a data-t="425">And it's very important to be explicit about what these other packages are and what versions</a> <a data-t="427">you need.</a> <a data-t="431">If you are not explicit about that, you might find that when you try to launch a project</a> <a data-t="434">after a couple of months, it doesn't work.</a> <a data-t="435">Why?</a> <a data-t="436">Who knows?</a> <a data-t="436">Maybe some package got updated and it broke.</a> <a data-t="439">If you have an explicit list of dependencies, the chances that it will just work are much</a> <a data-t="444">higher.</a> <a data-t="445">And we have R package to deal with this.</a></p>

<p><a data-t="449">For a similar issue but related to the R sources themselves, we have the box package.</a> <a data-t="454">This one allows you to use Python-style imports at the top of each file so that for each file</a> <a data-t="461">it is immediately obvious what packages, functions, and other files it needs.</a></p>

<p><a data-t="467">So moving on to the quality of R code.</a> <a data-t="472">We have three packages which deal with that.</a> <a data-t="475">For writing tests, we have test that package.</a> <a data-t="479">Then we have lint R, which can be used to automatically test the style of your codes</a> <a data-t="483">and also detect some errors that you might have.</a> <a data-t="487">And we have the styler package, which is able to fix many of those problems automatically.</a> <a data-t="491">All these tools can be run using Rhino functions out of the box.</a> <a data-t="495">You don't need to configure anything.</a> <a data-t="497">Just do, for example, Rhino lint R, and you get your code linted.</a></p>

<p><a data-t="504">So in enterprise settings, typically you will have some needs that will be quite common.</a> <a data-t="509">For example, one of them is logging.</a> <a data-t="512">It is very important to output good log messages from your application.</a> <a data-t="516">If it breaks, for example, in production, this kind of log messages can really help</a> <a data-t="520">you to find what the issue is.</a> <a data-t="523">And we have the logger package, which deals with that.</a></p>

<p><a data-t="526">Then you typically will have more than one environment.</a> <a data-t="530">You will have the local one when you develop the application.</a> <a data-t="533">Then you will have staging where you will be testing it before pushing it to production.</a> <a data-t="538">And to manage the differences between these different environments, we have the </a><a href="/software/config/">config</a> <a data-t="542">package.</a></p>

<p><a data-t="545">Now in any big project, we realize that sooner or later we need JavaScript.</a> <a data-t="551">We need to customize the application beyond what's provided by our packages.</a> <a data-t="555">And we figured, why not use the state-of-the-art tech for that?</a> <a data-t="559">So for JavaScript, we will run your code through Babel and Webpack.</a> <a data-t="564">This to ensure that you can use the latest features without worrying about browser support.</a> <a data-t="569">We have SLint that you can use to check the quality of your JavaScript code and also fix</a> <a data-t="575">many of the errors automatically.</a> <a data-t="577">While these things need Node.js to be installed on your machine, when using it from Rhino,</a> <a data-t="582">you will just call the package functions, build.js, lint.js.</a></p>

<p><a data-t="587">Now similarly, we typically need to add some custom styling.</a> <a data-t="591">And we chose SAS to do that.</a> <a data-t="593">It's an extension for the CSS language, which allows you to structure your code in a more</a> <a data-t="598">maintainable and better way.</a> <a data-t="601">And again, here, to compile your SAS into something understandable by the browser, we</a> <a data-t="606">use the state-of-the-art latest implementation of SAS.</a> <a data-t="610">We also have style lint, which is able to lint your style files automatically.</a></p>

<p><a data-t="616">Now what else do you need in a big project?</a> <a data-t="619">Well, certainly more testing.</a> <a data-t="621">And as we already are using Node.js, we are using also Cypress for end-to-end testing</a> <a data-t="627">of your application.</a> <a data-t="629">This framework allows you to launch your whole application, interact with it, like press</a> <a data-t="634">buttons, type text, and then verify if it behaved as expected.</a> <a data-t="638">For example, that some text appeared or a window got opened.</a> <a data-t="642">So again, this can be used simply by running a function of the package called test end-to-end.</a></p>

<p><a data-t="649">And finally, as a cherry on top, automation.</a> <a data-t="654">All these tools are useless if you are not actually running them.</a> <a data-t="658">And if it's not happening automatically, sooner or later, you will probably forget about it.</a> <a data-t="663">So if you use GitHub, Rhino comes with a working out-of-the-box setup for GitHub Actions.</a> <a data-t="671">These will run all the linters, unit tests, end-to-end tests for you automatically once</a> <a data-t="676">you initialize your project.</a></p>

<h3>What Rhino is</h3>

<p><a data-t="680">All right, so to summarize, what is Rhino?</a> <a data-t="684">Well, certainly you can think of it as this complete development toolbox with all those</a> <a data-t="690">things that I've just listed and mentioned.</a> <a data-t="691">But it's more than that.</a> <a data-t="695">We call it a framework.</a> <a data-t="697">And if you know other frameworks for Python or Ruby on Rails, we took a slightly different</a> <a data-t="704">angle here.</a> <a data-t="705">While you do get the project structure, packages to use, and basic structure for your application,</a> <a data-t="712">we also have all these tools around it for ensuring the quality of your code.</a> <a data-t="716">So we can think of Rhino as a framework, but for the whole development process.</a></p>

<p><a data-t="722">Then Rhino is certainly the absolute way to do things.</a> <a data-t="727">It is an opinionated framework based on years of experience that we have in building Shiny</a> <a data-t="732">applications.</a> <a data-t="733">It is not the only way to do things, but it is the way that we believe in and that we</a> <a data-t="738">recommend.</a></p>
