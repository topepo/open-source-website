<h3>What is Quarto?</h3>

<p><a data-t="101">So, most of you probably already know this, but the simplest way to think of what Quarto is,</a> <a data-t="106">it's the next generation of R Markdown.</a> <a data-t="112">We saw R Markdown in the last video.</a> <a data-t="112">Most, I think all of you know what that is.</a> <a data-t="116">And really, Quarto has really two goals,</a> <a data-t="117">one of which is to make R Markdown work across different computing ecosystems.</a> <a data-t="124">So, the Jupyter ecosystem, the R ecosystem, the Julia ecosystem.</a> <a data-t="130">So, broaden it in terms of computational environments, and also take the things that we learned over the first 10 years of working on</a> <a data-t="135">R Markdown and really kind of create a next generation of those capabilities.</a></p>

<p><a data-t="142">So, again, open source scientific and technical publishing system.</a> <a data-t="147">We do support quite a few different computational environments.</a> <a data-t="152">We support Knitter and R, Jupyter, observable JavaScript.</a> <a data-t="155">We actually have some folks from the Julia community working on adding a Pluto engine right now.</a> <a data-t="160">So, the idea is that we're going to build all these publishing capabilities, and then there will be lots of different compute engines over time, and we'll be able to work with all of them</a> <a data-t="169">rather than being tied to a single one of those.</a> <a data-t="174">Obviously, the underlying Markdown engine is Pandoc, but we've added lots of enhancements for scientific publishing.</a> <a data-t="180">Pandoc obviously has some very robust citation capabilities, but we've added lots of other things.</a> <a data-t="185">And then tried to let you create lots of different types of output, both simple documents, but also presentations and</a> <a data-t="190">blogs and websites, books, etc.</a></p>

<p><a data-t="198">So, lots of other projects have had similar goals and similar features.</a> <a data-t="203">I list a few of those at the bottom.</a> <a data-t="203">So, we're very much in debt to all that work and continue to learn from those projects as we build Quarto.</a> <a data-t="211">Again, I've covered most of this open source project sponsored by Posit, building on R Markdown, and just I would say I was frustrating.</a> <a data-t="219">I think we really had a lot of conviction that the ideas behind R Markdown were really sound and useful, and had the chance to positively impact how science was done, but it was frustrating</a> <a data-t="230">that it was only in the R ecosystem that the tools could be used.</a> <a data-t="237">And so, we really wanted to, you know, our observation was that certainly Jupyter was quite broadly used.</a> <a data-t="241">There were other new environments like Julia, and we wanted our work to be applicable across all of those.</a> <a data-t="247">So, that's really kind of how we got started working on Quarto, which was about three or four years ago.</a></p>

<p><a data-t="253">And in terms of the really high level goals of the project, this is one I don't need to explain or sell very much here, but make scientific work more reproducible and more automatable.</a></p>

<p><a data-t="267">Another thing we like to think about is that if you think about tools for writing scientific manuscripts, you know, Word, you can see the blue line, it starts off very easy and very quickly it gets totally out of control.</a> <a data-t="283">LaTeX is harder to start with, but it's actually once you learn LaTeX and learn the basic mechanics of it, the curve is actually relatively flat.</a> <a data-t="288">It scales really nicely.</a> <a data-t="296">And then traditionally Markdown has had, it's sort of a mixed story.</a> <a data-t="296">It's simple at first, and then you try to do advanced things, and you get into all kinds of weird hacks and</a> <a data-t="301">things like that.</a> <a data-t="305">So, I think with Quarto, really what we really want to do, and I'll show some work that I think aims at this, we'd like to have an environment that is as close to Word as possible</a> <a data-t="310">in terms of getting started in the initial ramp, but that also has the scaling characteristics</a> <a data-t="317">of LaTeX, that as you do more and more complicated things, it works really, it grows seamlessly with you.</a></p>

<p><a data-t="323">And then of course, this idea of single source publishing, which the manuscripts feature</a> <a data-t="330">will really underscore the idea that we want to author our content in a universal format</a> <a data-t="336">that can be repurposed across lots and lots of different mediums.</a> <a data-t="340">Because obviously scientific papers, it's still very convenient to have them in the PDF, but also great to have them on the web</a> <a data-t="345">and have them on mobile and other places.</a></p>

<blockquote><a data-t="305">So, I think with Quarto, really what we really want to do, and I'll show some work that I think aims at this, we'd like to have an environment that is as close to Word as possible</a> <a data-t="310">in terms of getting started in the initial ramp, but that also has the scaling characteristics</a> <a data-t="317">of LaTeX, that as you do more and more complicated things, it works really, it grows seamlessly with you.</a></blockquote>

<p><a data-t="353">Okay.</a> <a data-t="353">I think I've mostly covered this.</a> <a data-t="353">Why did we create a new system?</a> <a data-t="360">It's really just this idea that the languages and runtimes used for scientific discourse is very broad, and we really wanted something that could match that.</a> <a data-t="366">And kind of the idea is like do all these deep publishing capabilities and markdown capabilities and tooling, do it kind of once and make it very broadly applicable.</a></p>

<p><a data-t="380">So, as I mentioned before, there are several different sort of compute engines.</a> <a data-t="386">Knitter engine essentially gives us very, very close compatibility with our markdown.</a> <a data-t="392">Most RMD documents can just be run in Quarto without modification.</a> <a data-t="399">Jupyter is another engine.</a> <a data-t="399">And then we've done something with ObservableJS, and as I said, others are possible, and there is active work on a Pluto engine for Julia.</a></p>

<p><a data-t="404">And the way the Knitter engine works is exactly, this is the exact same diagram we used to use to explain our markdown, but instead of RMD, there's a QMD, uses Knitter, makes it very compatible</a> <a data-t="418">with all the existing RMD files.</a> <a data-t="426">One difference to highlight is that chunk options are generally encouraged to use YAML rather than put them inline.</a> <a data-t="434">And there's just an example of a Knitter engine document, and you can see, very familiar for people who've used our markdown,</a> <a data-t="440">you can see cross-references in here, you can see the label and caption in the code chunk.</a></p>

<p><a data-t="453">So, that's the Knitter engine.</a> <a data-t="453">The Jupyter engine actually has two different modes.</a> <a data-t="453">One is you can use a QMD, which is like an RMD, so it's plain text.</a> <a data-t="460">You can use any text editor to edit it, works very similar to RMD, and in that case, what happens, we take the QMD, we actually turn it into a Jupyter notebook, and then run it through our engine.</a> <a data-t="471">So, that's extremely analogous to how RMD works.</a> <a data-t="478">And there's another modality, which is I have a notebook, I have an existing Jupyter notebook, and I just want to render it, and that works too.</a> <a data-t="486">So, in that case, you use all the tools that you normally use for editing Jupyter notebooks, whether it be JupyterLab, or VS Code, or what have you,</a> <a data-t="491">Google Colab, and then you can put it through the same pipeline.</a></p>

<p><a data-t="498">Now, in this case, there's no, by default, we do not re-execute the Jupyter notebook.</a> <a data-t="502">This is one characteristic of Jupyter notebooks that people like a lot, is that for very, very expensive computations, you can control exactly when the computation is done, and then essentially, by not re-executing a cell,</a> <a data-t="512">you don't have to pay for the computation again.</a> <a data-t="518">It creates reproducibility problems, so it's not a total panacea, but it's certainly something that lots of folks in the Jupyter ecosystem do take advantage of.</a></p>

<p><a data-t="529">And then, in terms of tooling, we have the ability to do side-by-side preview with VS Code, and Jupyter notebooks, and things like that.</a> <a data-t="535">We have a JupyterLab extension for Quarto, and we have a VS Code extension for Quarto.</a> <a data-t="540">There's a NeoVim mode that's pretty good, and there's some integration with ESS as well.</a></p>

<h3>Notebooks and scientific publishing</h3>

<p><a data-t="667">So, now I want to kind of zoom out a little bit and talk about kind of the notebooks in Markdown and scientific publishing, how these things all relate together, how we would like them to relate.</a></p>

<p><a data-t="679">And I'm going to do a little bit of a sleight of hand here, and I'm going to assert kind of a rough equivalence between a QMD and a notebook, because QMDs really are very similar in that they mix prose and code and output.</a> <a data-t="691">They serve a similar purpose.</a> <a data-t="691">They work differently.</a> <a data-t="703">Notebooks bundle together output and code in one file.</a> <a data-t="703">QMD and RMD files are a text file, and then output is generated separately.</a> <a data-t="707">But I think we can think of these as notebooks, and in fact, there are some industry-level initiatives about notebooks and scientific publishing, and it's more useful when you're talking to folks who aren't familiar with all</a> <a data-t="716">the technical details to simply refer to these things all as notebooks versus trying to break</a> <a data-t="722">it down into this taxonomy of things that have to do with kind of somewhat incidental technical details.</a></p>

<p><a data-t="736">So the idea is that notebooks are a big part of how scientific communication happens, how scientific manuscripts are shared.</a> <a data-t="743">However, none of the workflows for publications make notebooks a first-class entity.</a> <a data-t="750">They're just sort of dropped on the floor, and then the content goes through.</a> <a data-t="755">But how do I reproduce this article, or could I even just see the computations underlying the article?</a> <a data-t="760">That's really not possible, and when you get into things like archiving, so archiving a scientific manuscript so that we can</a> <a data-t="765">reference it again or compute on it in 20 or 30 years, again, the notebooks and the computations are completely gone.</a> <a data-t="770">So that's a problem.</a></p>

<p><a data-t="779">I think what would be good to have would be an end-to-end workflow that treats notebooks, either classic Jupyter notebooks or sort of Quarto documents, as a more primary element of the scientific record, and a workflow that encourages authors</a> <a data-t="793">and rewards authors for making their work transparent and reproducible, and also acknowledges</a> <a data-t="800">lots of the work that goes into scientific articles that are a lot more than just the composition of the article.</a></p>

<h3>Notebooks Now initiative</h3>

<p><a data-t="812">So this is aspirational, and it might sound kind of fanciful, like, well, that's never going to happen.</a> <a data-t="818">Great to talk about it, but where are these incentives going to come from?</a> <a data-t="823">How are we going to get publishers on board?</a> <a data-t="823">But I would tell you that there is quite a bit more hope for this than you might imagine.</a> <a data-t="828">There is an initiative that was funded by the Sloan Foundation.</a> <a data-t="833">It was a grant given to the American Geophysical Union, the AGU, called Notebooks Now, and the idea is elevating computational notebooks as primary elements of the scientific record.</a> <a data-t="849">So this was a grant that resulted in a workshop initially that happened last fall.</a> <a data-t="854">There were about 50 in-person attendees, and I'll get into who was there,</a> <a data-t="860">but there were people from the R community, R OpenSci, the Jupyter community,</a> <a data-t="861">Public Library of Science, Journal of Open Source Software, major publishers, all together to talk about how we can make notebooks a fundamental part of the whole workflow and process.</a></p>

<p><a data-t="875">So we had an in-person meeting.</a> <a data-t="875">There's a steering committee.</a> <a data-t="880">You probably know more than one of the people who are on the steering committee.</a> <a data-t="880">These are folks that are from the R community.</a> <a data-t="886">Some of these folks are from the Jupyter community.</a> <a data-t="890">Some of these folks are from the Open Science community, and then the scientific publishing community.</a> <a data-t="894">So it's kind of the right group of people to get together to talk about how we can make notebooks and computations a more fundamental part of the publishing process.</a></p>

<p><a data-t="905">So what's happened is we had the in-person workshop, created a bunch of working groups, and there's actually a couple of implementations of this system, or these ideas, planned.</a> <a data-t="917">Quarto is doing one implementation, and then MIST is the Markdown dialect used in JupyterBook, is doing another implementation.</a> <a data-t="923">So what I want to show you today is this feature, Quarto Manuscripts, is really built to implement the kind of spec and ideas in this Notebooks Now effort.</a> <a data-t="937">Part of the idea is that the AGU is going to start by having one of their journals do an entire issue where all of the manuscripts have notebooks behind them and use all these tools, and the hope is that lots of other journals will follow suit, and even people who are creating</a> <a data-t="956">new journals will find this makes it quite a bit easier to create journals that are more interactive,</a> <a data-t="961">that use the web better, that collapse a bunch of the workflows that exist in the current journal publishing process.</a></p>
