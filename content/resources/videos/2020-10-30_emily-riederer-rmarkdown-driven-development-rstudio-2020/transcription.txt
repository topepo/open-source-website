<h3>Step 1: Cleaning up your RMarkdown</h3>

<p><a data-t="216">So now we'll go through the steps of our RMarkdown driven development one at</a> <a data-t="221">a time.</a> <a data-t="221">The first step's pretty easy because we don't actually have to generate anything</a> <a data-t="226">new.</a> <a data-t="226">We just simply have to delete a lot of things that shouldn't be in our RMarkdown</a> <a data-t="231">to begin with.</a> <a data-t="231">So for example, some of these things might include hard-coded variables,</a> <a data-t="237">which are common when we're doing a one-off analysis.</a> <a data-t="242">But thinking about ourselves or</a> <a data-t="242">a future user that might want to update this analysis later on, it can be really detrimental</a> <a data-t="247">to have hard-coded variables late in our analysis that users may not realize when</a> <a data-t="253">or where they have to change.</a> <a data-t="253">Or even worse, they might change them inconsistently some</a> <a data-t="259">places and not the others and introduce inconsistencies in their analysis.</a></p>

<p><a data-t="259">Instead, RMarkdown parameters</a> <a data-t="265">let us front load all of this in the YAML header of your RMarkdown document, which means</a> <a data-t="271">that it's very easy for a future user to see exactly in front of them what changes they</a> <a data-t="276">need to make and have everything flow on through your analysis.</a> <a data-t="281">Effectively then, your entire</a> <a data-t="281">RMarkdown is acting kind of like one mega function through which everything can flow.</a> <a data-t="287">This is also an effective strategy for dealing with credentials that you might not want in</a> <a data-t="292">your final output in case you end up sharing your code with a colleague and even just for</a> <a data-t="297">good data security practices.</a> <a data-t="297">There's really great synergy here with the </a><a href="/software/rstudio/">RStudio</a><a data-t="297"> IDE because</a> <a data-t="302">if you click knit with parameters, you can get a nice little pop-up box and replace dummy</a> <a data-t="307">credentials in your header with your actual credentials, so you've never hard-coded them</a> <a data-t="312">anywhere.</a></p>

<p><a data-t="312">Next up, I think we already learned from Jenny a little bit this morning about</a> <a data-t="317">file paths, and we know that we don't want to include global file paths in our RMarkdown</a> <a data-t="323">because it makes it very unresilient to working in the future.</a> <a data-t="328">Local file paths are a slight</a> <a data-t="328">improvement over these, but can still break as we change across operating systems or later</a> <a data-t="334">on as we move our files around inside of our working directory.</a> <a data-t="339">So best of all, we can</a> <a data-t="339">use the here package to create very resilient file paths to make sure our analysis always</a> <a data-t="345">works as planned.</a></p>

<p><a data-t="346">And finally, and this step is by far the hardest for me, sometimes we</a> <a data-t="351">have to kill our darlings.</a> <a data-t="351">You can't let your RMarkdown be a junk drawer for code that you</a> <a data-t="355">wrote, rabbit holes you went down, that you really, really thought would work, but then</a> <a data-t="360">didn't end up being the direction your analysis went in.</a> <a data-t="365">Take this time to remove package</a> <a data-t="365">loads that you didn't need and delete code experiments that you maybe have commented</a> <a data-t="370">out in your code just to create a very clean code script with a lot of hygiene on which</a> <a data-t="376">to proceed with your refactoring.</a></p>

<h3>Step 2: Rearranging code chunks</h3>

<p><a data-t="377">The next step in RMarkdown-driven development is simply</a> <a data-t="383">rearranging your code chunks.</a> <a data-t="383">RMarkdown is great because it allows us to capture our</a> <a data-t="388">thought process as we go through our analysis, but the one downside of this is our thought</a> <a data-t="393">processes are rarely linear and as a result, often neither are the resulting markdowns.</a> <a data-t="399">The strategy here is to move to the top a lot of the heavy-duty infrastructure and computing</a> <a data-t="404">chunks.</a> <a data-t="404">These can be things that set up our environment, such as library loads, data loads,</a> <a data-t="410">and a lot of our heavy-duty computation with data wrangling, data cleaning, model fitting,</a> <a data-t="415">et cetera.</a> <a data-t="415">And let's sink down to the bottom, the more communication and narrative aspects.</a></p>

<p><a data-t="421">This has many benefits.</a> <a data-t="421">First of all, it allows us users to clearly see the main dependencies</a> <a data-t="427">of our script at the top as we have things like external file references and package loads</a> <a data-t="433">right where the user opens the document.</a> <a data-t="433">Similarly, by grouping like pieces of code</a> <a data-t="438">together, we can more easily start to see patterns throughout the code and get a sense</a> <a data-t="443">where we can later consolidate.</a> <a data-t="447">And finally, often we'll be working on an analysis with</a> <a data-t="447">colleagues that don't code and having all of the more narrative elements consolidated</a> <a data-t="453">at the bottom may make it easier for them to find places where they can actually jump</a> <a data-t="457">in and edit the text parts in RMarkdown without being kind of intimidated by large blocks</a> <a data-t="463">of code sprinkled throughout.</a></p>

<p><a data-t="463">Again, there's another great opportunity here to leverage</a> <a data-t="469">the RStudio IDE for even more synergy and take advantage of some tips like naming your</a> <a data-t="475">chunks and commenting your code with the quadruple dash, as you see in this example.</a> <a data-t="482">The combination</a> <a data-t="482">of these two features allows you to create a really friendly, nice table of contents</a> <a data-t="488">that you can pop out to help a user navigate your script better and jump to the exact component</a> <a data-t="494">that they want to work on.</a></p>

<h3>Step 3: Extracting functions</h3>

<p><a data-t="494">So after we've done all of this rearranging, one of the main</a> <a data-t="500">benefits of that is it can more easily help you understand related parts of your script</a> <a data-t="506">and find places where we can cut out duplicate code and replace these with functions for</a> <a data-t="511">an even more streamlined approach.</a> <a data-t="512">As a somewhat trivial example of this, in an exploratory</a> <a data-t="517">data analysis part of your workflow, you probably end up visualizing the relationship between</a> <a data-t="523">multiple variables and a lot of related plots.</a> <a data-t="523">Doing this without functions leads to a lot</a> <a data-t="528">of code that's both repeated and efficiently, but also makes it harder to tell exactly what</a> <a data-t="533">piece of those lines is changing.</a> <a data-t="533">All I'm actually changing there is the variable being</a> <a data-t="538">plotted on the y-axis, but it kind of takes some staring at it to tell.</a></p>

<p><a data-t="538">Instead, breaking</a> <a data-t="543">these apart into separate chunks to define my function and then to use my function gives</a> <a data-t="548">me numerous benefits.</a> <a data-t="548">First, if I want to change something about those plots, like theme,</a> <a data-t="554">I can easily just change that code in one place instead of multiple places.</a> <a data-t="559">And secondly,</a> <a data-t="559">from a more literate programming perspective, it's much easier in the view on the right</a> <a data-t="565">to see what stayed the same versus what's changed between iterations.</a></p>

<p><a data-t="572">This is also a</a> <a data-t="572">good time to start thinking about adding in R Oxygen 2 comments to your R Markdown.</a> <a data-t="578">For</a> <a data-t="578">those that haven't ever built a package, R Oxygen 2 is the tool that you'll use to document</a> <a data-t="584">functions and create the help pages that we've all likely seen at some point when we've pulled</a> <a data-t="589">out up R documentation.</a> <a data-t="594">But even before you're building a package, you can easily in the</a> <a data-t="594">RStudio IDE add in this R Oxygen 2 skeleton and use that to document your function.</a> <a data-t="601">This</a> <a data-t="601">is nice because it's a format that users are already used to seeing and gives them the</a> <a data-t="606">content they expect to know about a function.</a> <a data-t="606">It can also help you think more critically</a> <a data-t="610">about the design of your functions because it forces you to almost make a contract with</a> <a data-t="615">yourself about precisely what those inputs and those outputs ought to look like.</a></p>

<h3>Step 4: Cleaning up style</h3>

<p><a data-t="620">So now</a> <a data-t="620">that we've cleaned up the actual content of our code a lot, one final step to consider</a> <a data-t="625">at this point is also cleaning up the style.</a> <a data-t="625">And we have many great R packages to help</a> <a data-t="630">out here, including the lintar, styler, and spelling packages.</a> <a data-t="637">Lintars and stylers help</a> <a data-t="637">us adhere to a specific style guide for our script by either flagging for us to change</a> <a data-t="645">or by proactively themselves editing our script to make them adhere to certain rules</a> <a data-t="651">such as rules about spacing and indentation.</a> <a data-t="652">And spelling similarly can help us catch typos</a> <a data-t="657">in our code for the cleanest possible view of our report.</a></p>

<h3>From file to project</h3>

<p><a data-t="659">So going through all these</a> <a data-t="665">steps, we've now really done a lot of work to move our R Markdown to something that's</a> <a data-t="670">pretty sustainable, pretty readable, pretty navigable.</a> <a data-t="675">And in some cases, this in and</a> <a data-t="675">of itself may be a really good stopping point.</a> <a data-t="680">In resource-constrained environments, if you</a> <a data-t="680">don't have a good way to share files with a colleague, a single file can be very easy</a> <a data-t="685">to send.</a> <a data-t="685">If you don't have access to a good version control system, it can be very easy</a> <a data-t="690">to just take a diff between versions.</a> <a data-t="695">So there are some benefits to leaving things in a single</a> <a data-t="695">file.</a> <a data-t="700">Finally, you can also quickly refresh your code by just running one single file</a> <a data-t="700">in isolation.</a></p>

<p><a data-t="706">However, these still can be sometimes lengthy, monolithic, and sort of</a> <a data-t="711">intimidating to new users.</a> <a data-t="711">And you may actually be forcing R Markdown to do a lot more computation</a> <a data-t="717">than necessary any time you want to know.</a> <a data-t="722">So if these sound like problems to you, we</a> <a data-t="722">may want to consider moving from a file to an R project.</a></p>

<p><a data-t="726">An R project looks something</a> <a data-t="726">like this.</a> <a data-t="731">And at a high level, our goal is to take a lot of the assets we have in</a> <a data-t="731">our single file R Markdown document and break them apart and modularize them into many different</a> <a data-t="737">folders with semantic meaning.</a> <a data-t="737">This can be very helpful if you or your organization uses</a> <a data-t="743">a very standardized file structure across projects, because everyone will know where</a> <a data-t="748">to find specific data assets within a project.</a> <a data-t="752">For example, one structure I like is shown</a> <a data-t="752">here on the screen, where I would save my actual R Markdown report and the output of</a> <a data-t="757">that in my analysis folder.</a> <a data-t="757">I put a lot of those functions that I defined and other little</a> <a data-t="763">bits of R code, such as those that read and process data, in a variety of R scripts in</a> <a data-t="769">my SRC folder.</a> <a data-t="769">And I can save both my raw data as well as data artifacts that I create</a> <a data-t="777">throughout the process in different data folders.</a> <a data-t="781">And finally, it's a good place to have</a> <a data-t="781">catch-all files for different types of documentation or external pieces of context.</a></p>

<p><a data-t="788">This</a> <a data-t="788">allows me to create a much leaner R Markdown, where I just read in the minimal amount of</a> <a data-t="793">resources from these files that I actually need.</a> <a data-t="798">To take two more specific examples of</a> <a data-t="798">what this looks like, instead of defining both a function and calling that function in the</a> <a data-t="804">same R Markdown, I can break that up and define my R function in a modularized script that</a> <a data-t="811">could be easily stolen and repurposed in another analysis.</a> <a data-t="816">And I could simply load that in</a> <a data-t="816">by executing that code remotely with the source function in my R Markdown.</a></p>

<p><a data-t="824">Another big kind of step change benefit by going through this process is in your actual</a> <a data-t="829">data processing.</a> <a data-t="829">Instead of doing all of your data processing within your R Markdown, I</a> <a data-t="835">tend to think about doing it in three different steps.</a> <a data-t="835">First, I use some R scripts to access</a> <a data-t="840">my data from my database or my API or wherever that data is coming from, and save that raw</a> <a data-t="846">data in a read-only data file.</a> <a data-t="851">Then, through, again, a variety of R scripts that might help</a> <a data-t="851">me with a lot of my data cleaning and wrangling and model fitting, I can create a lot of smaller</a> <a data-t="857">data artifacts and have only those be what I read into my R Markdown.</a> <a data-t="863">This means that</a> <a data-t="863">I can run my R Markdown much faster without having to, like, repeat many long, lengthy</a> <a data-t="868">simulations or model builds or whatever it is you may be doing if I just want to make</a> <a data-t="873">some small change.</a> <a data-t="873">And it also removes upstream dependencies on external systems, like APIs</a> <a data-t="879">or databases, to be live and be working.</a> <a data-t="884">If you just want to do something really small,</a> <a data-t="884">like change it, fix the typo in your R Markdown, you don't want to not be able to do that just</a> <a data-t="889">because for some reason your database wasn't running.</a></p>

<p><a data-t="893">And one final note on projects.</a> <a data-t="893">There exists a lot of great packages out there to help</a> <a data-t="898">you set up these projects, and some of them also have different opinionated file structures,</a> <a data-t="904">a little different maybe than the exact one I showed here, but very similar in spirit.</a> <a data-t="908">So I definitely encourage you to check out use this project template and starters to</a> <a data-t="913">find the one that works best for you.</a> <a data-t="918">And also making a project is a great time to think</a> <a data-t="918">more about managing your package dependencies that are going into that project.</a></p>

<h3>From project to package</h3>

<p><a data-t="931">So again, stepping back a minute to take a view of where we are, we now have a pretty</a> <a data-t="936">well refactored analytical project as opposed to a file.</a> <a data-t="941">And some benefits of this are that</a> <a data-t="941">we can easily, like, we've really broken apart all sorts of pieces that might be independently</a> <a data-t="948">useful and put them in a place where they're very easily found and accessed to be repurposed.</a> <a data-t="954">But at the same time, because we're in sort of project mode, we've really preserved a</a> <a data-t="959">lot of the context specific to our individual analysis while doing this.</a> <a data-t="964">So for people that</a> <a data-t="964">are more interested in building off of our analysis or taking the actual, like, learnings</a> <a data-t="969">from what we were trying, the question we were trying to answer, it's still very easy</a> <a data-t="974">to do that.</a></p>

<p><a data-t="975">However, for that very reason, projects can still cause some confusion because</a> <a data-t="980">they kind of blur the line sometimes between what's a tool versus what's part of a higher</a> <a data-t="985">level process that solves a class of problems versus your individual problem.</a> <a data-t="991">If you want</a> <a data-t="991">to focus mostly on solving that broad class of problems, that's when we can move to a</a> <a data-t="996">full-blown R package.</a></p>

<p><a data-t="997">To me, one magical thing about R packages</a> <a data-t="1002">that I didn't realize when I first started using R was they're really as simple as saving</a> <a data-t="1008">a lot of R code in the right places.</a> <a data-t="1008">And there's pretty clear one-to-one mapping between the</a> <a data-t="1013">kind of structure you might use for a project that I've talked about previously to the kind</a> <a data-t="1018">of structure you'd use for an R package.</a> <a data-t="1019">All of the functions that we just broke up into</a> <a data-t="1019">separate R scripts translate easily, go right into a single R folder.</a></p>

<p><a data-t="1024">The actual R Markdown that we've been working with this whole time can actually serve numerous</a> <a data-t="1029">purposes.</a> <a data-t="1029">It can either go in your package as an R Markdown template, so users of your</a> <a data-t="1035">package immediately have some code they can use to start playing with and experimenting</a> <a data-t="1040">with and a working R Markdown example of how to use what you've built.</a> <a data-t="1046">And they can also</a> <a data-t="1046">be repurposed as vignettes and long-form documentation to really teach people more</a> <a data-t="1052">deeply about how to think about the problem that they're working on.</a></p>

<p><a data-t="1055">Similarly, all of the data artifacts we created, if we're careful to anonymize them and take</a> <a data-t="1060">out anything that we shouldn't be sharing with the public, can be very useful as examples</a> <a data-t="1065">in your data package.</a> <a data-t="1065">By shipping example data in the data folder, this gives us a great</a> <a data-t="1070">way to build more vignettes, to conduct unit testing, and to give examples in your R Oxygen</a> <a data-t="1076">documentation.</a> <a data-t="1082">Really, the only two things we're missing at this stage are a man folder</a> <a data-t="1082">where the documentation manual lives and the test folder for unit tests.</a></p>

<p><a data-t="1088">And at this point, I doubt it should surprise anyone here that there are more great R packages</a> <a data-t="1093">to help us go this last mile.</a> <a data-t="1099">First of all, once again, the use this package is phenomenal</a> <a data-t="1099">for helping us set up the package infrastructure we need and making sure we do, in fact, save</a> <a data-t="1104">all those individual package assets in the right places with the right file structures</a> <a data-t="1108">and configuration files to make everything flow through seamlessly.</a></p>

<p><a data-t="1113">Dev tools can automatically generate our R documentation from the R Oxygen comments that</a> <a data-t="1120">we have already wrote many steps back in this process.</a> <a data-t="1125">And </a><a href="/software/testthat/">testthat</a><a data-t="1125"> provides a really friendly</a> <a data-t="1125">interface for doing unit testing, which, quite honestly, objectively, if we want to trust</a> <a data-t="1132">the outcome of any analysis, we probably even could have thought about doing many steps</a> <a data-t="1137">earlier in this process.</a> <a data-t="1143">And beyond all of those absolute requirements,</a> <a data-t="1143">R even goes above and beyond what we need.</a> <a data-t="1148">And there also exists Packagedown, which can</a> <a data-t="1148">similarly reuse all of those great assets you made throughout your project and throughout</a> <a data-t="1154">the R Markdown-driven development process and create a very user-friendly, attractive,</a> <a data-t="1160">polished website to help people learn about and use your analytical tool.</a></p>

<h3>Wrapping up</h3>
