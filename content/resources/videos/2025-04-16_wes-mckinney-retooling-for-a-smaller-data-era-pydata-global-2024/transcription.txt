<h3>Data size is relative</h3>

<p><a data-t="604">One of the things that we encounter over and over is that data size is relative, and what we think of as big data or small data has changed a lot over the last 20 years.</a> <a data-t="621">So what used to be big data can now fit on your laptop.</a> <a data-t="626">So we used to think that a gigabyte or 100 gigabytes or a terabyte of data was big data.</a> <a data-t="632">But now a terabyte of data may compress down to a set of parquet files that are not that big and can easily fit on your laptop and be queried very effectively with many of the tools that we have today.</a></p>

<p><a data-t="646">And so if you go back and think about the original big data paper from Google, the MapReduce paper from 2004, I don't know if anybody knows how many processing cores, CPU cores, were standard in top-of-the-line servers in 2004.</a> <a data-t="670">So the answer is one. So in 2004, all servers and data centers had a single processor core.</a> <a data-t="678">So maybe they had hyper-threading in that era so you could run two threads simultaneously, but the processing capabilities 20 years ago were much more modest than they are today.</a></p>

<p><a data-t="690">And if you look at the servers that you can buy and rent now in AWS or in other cloud services, you can get servers that have 96 physical cores.</a> <a data-t="704">And so if you have hyper-threading, that's 192 concurrent threads of processing on a single CPU socket.</a> <a data-t="711">And servers can have two of these or even sometimes more than two sockets.</a> <a data-t="717">And so you could have a dual socket server in the cloud with 384 simultaneous threads of processing.</a> <a data-t="724">And so that in a single server is bigger than many of the big distributed clusters of machines that Google folks were talking about in their MapReduce paper 20 years ago.</a> <a data-t="740">So this is a massive change.</a></p>

<p><a data-t="742">We've also seen similar evolution in hard disk drive performance.</a> <a data-t="750">So 20 years ago, everything was spinning rust hard disk drives with high seek latency.</a> <a data-t="759">We moved on to solid state drives and got massive improvement in seek performance, much higher read and write bandwidth.</a> <a data-t="770">And then more recently, we've moved on to non-volatile memory NVMe, which has brought the read and write bandwidth starting to approach 10 gigabytes a second.</a> <a data-t="780">And I expect that disk will continue to get faster and faster.</a></p>

<p><a data-t="784">And we've seen similar trends in networking performance as well.</a> <a data-t="789">So this chart shows basically state-of-the-art networking performance over time.</a> <a data-t="797">And so we've gone from gigabit Ethernet to terabit Ethernet in less than 20 years, which is pretty impressive.</a></p>

<p><a data-t="809">For a long time, people were talking about how Moore's law is dead.</a> <a data-t="814">So Moore's law is the idea that every 18 months, the number of the transistor density in CPUs doubles, and effectively, the processing capability of CPUs goes up by a factor of two.</a> <a data-t="828">And that started to plateau in CPUs at some point, but we've continued to see core counts go up, especially in GPUs and specialized silicon, which has enabled us to continue to have that exponential increase in processing capabilities.</a></p>

<h3>Ergonomics and the cost of big data systems</h3>

<p><a data-t="846">But one of the challenges in thinking about the development of big data systems compared with our nice, friendly Python tools, PyData tools, is that people thought about the ergonomics, the usability, the user experience of big data systems in a very different way.</a> <a data-t="865">Whereas in Python, we really value our library ergonomics, the code that we write, that it should be very easy to read, that it should be easy to type, very fluent and natural.</a></p>

<p><a data-t="880">And so that emphasis on developer productivity, user productivity, was comparatively less important in the big data ecosystem, where it was really all about how do we make it feasible to process terabytes and petabytes of data.</a> <a data-t="901">And so, not just the usability and the ergonomics, but also the overhead and the cost of processing data at scale was also often an afterthought.</a></p>

<p><a data-t="916">This was highlighted in this famous paper from 2015 by some former Microsoft research developers who've gone on to have illustrious careers working on </a><a href="/software/tensorflow/">TensorFlow</a><a data-t="916"> and Materialize and other projects.</a> <a data-t="934">But they pointed out that while many of the big data systems had achieved scalability and the ability to process large data sets, they also introduced a lot of overhead that makes the cost for processing each terabyte or each gigabyte significantly higher than what you could do on smaller scale data sets on a single machine.</a></p>

<p><a data-t="959">So the way that they described it in the paper is these systems have achieved impressive scalability, but to what extent are they truly improving performance as opposed to parallelizing the overhead that they introduce into the process?</a></p>

<blockquote><a data-t="959">So the way that they described it in the paper is these systems have achieved impressive scalability, but to what extent are they truly improving performance as opposed to parallelizing the overhead that they introduce into the process?</a></blockquote>

<p><a data-t="973">And so to make this, what we're talking about a little bit more complete, is think about a SQL query that aggregates a very large table. Maybe we're grouping by one column, computing the average of another column.</a> <a data-t="991">We may very well write similar code in pandas or similar code in R. And these are all basically doing the same thing, but under the hood, the architecture of the systems that evaluates this code, depending on the scale of the data and the underlying execution engine, is very different.</a></p>

<p><a data-t="1012">So in pandas and R, we're loading the whole data set into memory, whereas in the SQL engine, it might be some distributed MapReduce job that's running on a big cluster of machines if you have a massive data set that can't possibly fit on a single node.</a></p>

<p><a data-t="1027">And so this has led to this sort of hierarchy of needs when thinking about building these systems, when looking at things from the big data perspective, where the ability to scale to say, I can process a terabyte of data or 10 terabytes or a petabyte of data, that starts out as being the primary concern.</a> <a data-t="1050">And only after that point can you begin to think about, okay, how do we make it faster? Just wall clock time. We want it to take less time from start to finish to complete the job.</a></p>

<p><a data-t="1064">After you've made it fast, you can start thinking about efficiency, both from a resource like the amount of hardware you use, but also increasingly we're beginning to think about processing efficiency in terms of power utilization.</a> <a data-t="1078">And so how many kilowatt hours and thus how much money does it cost to execute a particular workload? And when you're paying for compute time by the core hour in the cloud, this starts to become a big deal because the queries that you're running are converting into a bill that you're getting from AWS or from Google Cloud.</a></p>

<p><a data-t="1103">And then beyond these performance and efficiency considerations, as we start building more heterogeneous data systems that are doing raw data processing, feature engineering, as well as AI training and model serving, we have a set of components that is solving many different problems.</a> <a data-t="1134">That can't be achieved within a single system. And so how those systems fit together, how they are composed with each other, and how efficient is composing them together is another concern which has come more to the forefront in recent years, especially with the growth and adoption of AI.</a></p>

<h3>Composable data systems and open standards</h3>

<p><a data-t="1154">And so when we think about these composable data systems, one of the things that enables us to plug the systems together and do it in an efficient way is to develop open standards and protocols for connectivity at the data level as well as transmitting and understanding of what you want each system to do.</a> <a data-t="1175">And so we are beginning to design around this concept of modularity, reuse, and interoperability so that if we adopt an open standard or protocol for plugging systems together, we can greatly reduce the amount of overhead that is present in distributed systems that make use of different processing components.</a> <a data-t="1197">And in doing so, we can create a virtuous cycle with the different open source projects that can then work better together to build more efficient, heterogeneous distributed systems.</a></p>

<p><a data-t="1212">And so a lot of this work is broken down into a number of different areas that are common in the world of data processing engines and database systems from the processing engines themselves to the protocols which connect systems together.</a> <a data-t="1228">Those are generally data protocols like Arrow, but also there's new things like Open Lineage, which provides an open standard for metadata so you can talk about what the system is doing with the data in a standard way so you can reason about schemas and transformations across different processing components.</a> <a data-t="1250">Increasingly, how we store our data and doing so in an open standard way has become an important area. We've seen significant investment there in open source projects like Iceberg and Fileform. That's like Parquet to make that easier for people.</a></p>

<p><a data-t="1270">For me, this is a problem that I've been thinking about for many years, going back to 2015, the same year that the scalability but at what cost paper came out. You can see a lot of people were thinking about this problem in this era because we had recognized that there was a problem, but we didn't have a lot of great solutions to it.</a> <a data-t="1291">The idea is that we want to facilitate decoupling the query user layer, the programming layer, the code that you write, how you express what you want the systems to do.</a> <a data-t="1301">You want to decouple that from the underlying storage and the execution so that we can enable the respective developers of each layer of the stack to really hyper, hyper specialize and make the different pieces as powerful and efficient and composable as possible.</a></p>

<p><a data-t="1321">When you do this, you're enabling components to be shared across different types of systems. You enable interchangeability so you can swap out an execution engine or bring in a new query interface.</a> <a data-t="1336">We've made significant progress toward this. That's been really exciting to see happen and to be a part of.</a></p>

<p><a data-t="1344">The way that I described the problem to the PyData and also the broader data science community in 2017, this was at JupyterCon in 2017, was wouldn't it be amazing if we had a powerful computational runtime that could be shared across programming languages so that it isn't all the Python developers working for themselves to rebuild everything just for Python in the same way?</a> <a data-t="1374">With the R ecosystem and the same with the Julia ecosystem, but that we could have a set of shared libraries and components that could be portably reused in a portable way across programming languages so that the programming language becomes an interchangeable front end for the computational back ends.</a> <a data-t="1394">We can make improvements to our computational systems and reap the benefits everywhere, not just in Python or not just in R or another language.</a></p>

<h3>Apache Arrow, DuckDB, and embeddable engines</h3>

<p><a data-t="1404">One of the things that we've built to make this easier is Apache Arrow, which has provided a language-independent fast interchange layer for data, for tabular data, data frames, effectively database tables.</a> <a data-t="1421">And another component which has helped lead the way in thinking about embeddable and interchangeable execution engines is DuckDB, which you can think about affectionately is like SQLite for analytics.</a> <a data-t="1437">It's a super fast columnar database engine that can be embedded into existing systems, deployed and used everywhere.</a></p>

<p><a data-t="1448">This has become a wildly popular project and pretty much has shut down the cottage industry of people building their own database and data processing engines.</a> <a data-t="1461">It used to be that if you wanted to add some form of querying or SQL capability to a web application or just some other application, you might build some subset of the features that you need just for your application.</a> <a data-t="1473">But now you can pick up DuckDB off the shelf and have something that is state of the art.</a> <a data-t="1479">I'm a huge fan of this project and has helped socialize these ideas of modularity and reuse of systems.</a></p>

<p><a data-t="1488">There are some other data processing engines that have been built that are being used along this mantra of modularity and interchangeability.</a> <a data-t="1499">The Data Fusion project, which is similar to DuckDB. It's written in Rust.</a> <a data-t="1504">Polars, which many of you are probably familiar with, a new data frame library.</a> <a data-t="1509">That ultra-fast or Rust-based data frame library for Python, which has become really popular.</a> <a data-t="1515">But there's a number of projects that I'm aware of that are using the Rust execution engine of Polars to build other types of systems.</a></p>

<p><a data-t="1523">And that's really exciting to see. <h3>Toward a multi-engine data stack</h3>

<p><a data-t="1569">But this kind of begs the question, why be locked into a single execution engine, along with a full stack of tools, including storage and query interface?</a> <a data-t="1581">And so, based on the size of your dataset, as well as different requirements of your workload, you may want to use one stack of tools that's optimized for smaller datasets and maybe programming in Python.</a> <a data-t="1597">Whereas in another environment, you may have much larger datasets in your programming in SQL.</a> <a data-t="1602">And so you'd like to have the flexibility to be able to work in different modalities on top of your storage, which may be all Parquet files living in iceberg tables in S3.</a></p>

<p><a data-t="1617">And so traditionally, the way that people would approach working at different scales and many different engines processing on the same data was you would use SQL.</a> <a data-t="1629">But SQL was conceived as a standard, but in practice, it is fragmented and being different for every database engine.</a> <a data-t="1638">And choosing which engine to use can often be non-trivial.</a></p>

<p><a data-t="1644">And so there's tools like SQL Glot for Python, which helps with transpiling SQL queries from one dialect to another.</a> <a data-t="1651">But we'd also like to be able to not be stuck using SQL for everything, because having SQL strings littered around your Python code base is not ideal.</a></p>

<p><a data-t="1663">So one of the, I know I'm about to run out of time, but some projects which are helping with this, we've seen also a proliferation of DataFrame APIs in Python.</a> <a data-t="1674">But there are some projects which are trying to help make things simpler.</a> <a data-t="1677">There's a new project called Narwhals, which is a Polars API transpiler that can use Polars, but also transpile to different SQL dialects or other query backends.</a> <a data-t="1689">I've been involved with the Ibis project, which is a kind of a new, it's a DataFrame API, which transpiles into 20 different backends.</a> <a data-t="1701">There's the Modin project, which is another query transpiler that targets the Pandas API.</a></p>

<p><a data-t="1706">And so if you're interested in this problem at the query layer level, or interested in having a nice DataFrame API that targets lots of different backends, I encourage you to check out the Ibis project.</a> <a data-t="1718">It's now a nine-year-old project and has become pretty mature and easy to use.</a> <a data-t="1726">You write DataFrame-like expressions, but everything is an expression and can be composed with each other to build more complicated expressions.</a> <a data-t="1735">So if you want to run things in memory with Pandas or with Polars, you can do that.</a> <a data-t="1740">But if you want to run things out of core with DuckDB or BigQuery, you can do that as well.</a></p>
