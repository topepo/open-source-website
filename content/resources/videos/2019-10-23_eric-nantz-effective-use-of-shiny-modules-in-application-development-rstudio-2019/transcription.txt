<h3>Avoiding the kitchen sink approach</h3>

<p><a data-t="848">Next, it's not just about what you should do.</a> <a data-t="851">I'm going to tell you about a technique that you definitely do not want to do.</a> <a data-t="855">I somewhat affectionately call this the kitchen sink approach.</a> <a data-t="858">So let's imagine you have this reactive values object called rdata, and this is a place where</a> <a data-t="865">in essence in each of these modules I have in this application, instead of following</a> <a data-t="869">that advice I just gave you, imagine I just want to have these reactives here and there,</a> <a data-t="874">and when I need to, I just toss some stuff into that reactive values object.</a> <a data-t="879">Who cares about returning anything out of it?</a> <a data-t="882">Because the best part is when I do the module calls, I just have that one argument retaining</a> <a data-t="888">to that name rdata, that reactive values slot.</a> <a data-t="891">Boy, it seems all rosy unicorns, right?</a></p>

<p><a data-t="895">Well guess what?</a> <a data-t="896">This is a bad idea, because you've now ventured into the dangerous world of hidden state.</a> <a data-t="902">This is where it's difficult to figure out what is happening to these individual objects</a> <a data-t="907">in the rdata kitchen sink, I would call it, and only certain objects from this are needed</a> <a data-t="912">as you pass from one module to another, and even more importantly, it's losing that contract</a> <a data-t="918">that you've established with these modules.</a> <a data-t="921">So this is a really bad idea, and even in typical R programming, you wouldn't see people</a> <a data-t="926">passing around environments back and forth either, that just is ugly stuff, so don't</a> <a data-t="931">do it.</a></p>
