<h3>The reproducibility crisis in finance</h3>

<p><a data-t="23">Our story actually begins in the year of 2015.</a> <a data-t="28">Back then, Professor Kemp Harvey, who is a well-known figure in the field of finance, published a quite provocative paper.</a> <a data-t="38">In the abstract of his paper, he stated that most claimed research findings in financial economics are likely false.</a> <a data-t="48">As you can imagine, this triggered quite a bit of a debate in finance and even the Financial Times reported about it back then.</a></p>

<blockquote><a data-t="38">In the abstract of his paper, he stated that most claimed research findings in financial economics are likely false.</a></blockquote>

<p><a data-t="58">But one of the most important and long-lasting effects this paper had is that top journals introduced code and data sharing policies.</a> <a data-t="73">So now, as a researcher, when you want to publish a paper, you have to provide your code and at least some mock data and ideally the actual data that is used to create your results.</a> <a data-t="87">The Journal of Finance was the first journal to implement this because Kemp Harvey was the editor at the time, so this helped a lot.</a> <a data-t="95">But the other journals followed as well in the coming years.</a></p>

<h3>Origins of Tidy Finance</h3>

<p><a data-t="100">One important consequence of these code and data sharing policies was that as young researchers, we kind of faced a hard time.</a> <a data-t="113">So you see here a picture of my brilliant colleague and co-author and friend Stefan Voigt and me at the time in 2015 when we started our PhD.</a></p>

<p><a data-t="127">There was hardly any public code or data available.</a> <a data-t="131">So this really sucked because we wanted to do empirical work.</a> <a data-t="135">So you have to use proprietary data and come up with the code yourself.</a> <a data-t="141">It was really hard to reproduce papers.</a> <a data-t="143">I mean, obviously, people are not sharing all the artifacts that you need to replicate the results.</a> <a data-t="149">It's very hard to do so.</a></p>

<p><a data-t="153">And since we loved working with data, we found ourselves in this well-known position of data scientists that we spend a lot of time preparing data that actually people have already worked with thousands of times.</a> <a data-t="167">We felt like we are reinventing the wheel every time.</a> <a data-t="171">So that was annoying.</a></p>

<p><a data-t="173">And Stefan and I, we had different coping mechanisms with this situation.</a> <a data-t="179">I decided to leave academia, actually.</a> <a data-t="185">Before I did that, I started writing blog posts.</a> <a data-t="189">I figured that's a great opportunity to signal to potential future employers that I know different things.</a> <a data-t="199">I can do asset pricing stuff, for instance.</a></p>

<p><a data-t="199">So what I did is I picked up a textbook that is used for teaching all over the world.</a> <a data-t="206">And I love the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="206">. <p><a data-t="221">And what happened is that over time, people reached out to me thanking me for putting this stuff online.</a> <a data-t="230">They're like, oh, my God, you make my life so much easier.</a> <a data-t="233">Thank you for doing that.</a> <a data-t="236">And I mean, they were also asking me to do more stuff, obviously.</a> <a data-t="239">But I was really happy to receive that kind of feedback.</a> <a data-t="242">It's like, OK, I mean, people find my stuff, they use it.</a> <a data-t="246">This is great feedback.</a></p>

<p><a data-t="250">Stefan took another route.</a> <a data-t="253">He stayed in academia. <p><a data-t="286">And he created some lecture notes, used some of my stuff.</a> <a data-t="289">And at some point, we kind of jokingly said, well, you know, we have so much stuff, we could create a textbook ourselves.</a> <a data-t="298">This is what we did, actually.</a></p>

<h3>Benefits of open source educational content</h3>

<p><a data-t="362">And this is the first point that I want to make in my presentation.</a> <a data-t="365">There are tremendous benefits of this open source educational content.</a> <a data-t="370">First, fairly obvious, everybody can access it.</a> <a data-t="374">Wherever you are, regardless of your financial constraints, you can just visit the website, use the stuff.</a> <a data-t="382">That's brilliant.</a></p>

<p><a data-t="385">As an author, it's also great that you can just keep it up to date.</a> <a data-t="390">I mean, you will immediately, once you send stuff off to the printing press, you will find a mistake like every time.</a> <a data-t="397">But you can just correct it. You can keep a changelog.</a> <a data-t="400">When a new package comes around, you can just update all the relevant chapters.</a> <a data-t="404">You can add new stuff as you go and collect feedback. That's amazing.</a></p>

<p><a data-t="410">And I want to stress this feedback collection very much because readers can actually contribute.</a> <a data-t="415">So we have our book on GitHub so people can create issues.</a> <a data-t="421">And it has happened frequently that people raised issues around, oh, are you sure that this is the way to do it?</a> <a data-t="431">Yes, we were most of the time.</a> <a data-t="434">But sometimes they also ask clarifying questions because the text is not clear enough or the code is not clear enough.</a> <a data-t="443">Or they come up with new edge cases that we have not thought about yet.</a> <a data-t="449">And that's amazing for me as an author to have this kind of feedback.</a></p>

<h3>Transparent code: narrative, code, and results</h3>

<p><a data-t="478">Now, as a next step, I want to show you a typical page in our Tidy Finance book.</a> <a data-t="483">Don't focus so much on the details here. It's just it's about the outline.</a> <a data-t="488">So on the top of this example, you have Narrative.</a> <a data-t="493">So we economists, we love regression models.</a> <a data-t="498">So this is an example where there isn't a regression model.</a> <a data-t="502">So we explain the equation, what's happening, what do we want to estimate?</a> <a data-t="507">And this is usually where typical textbooks in finance stop.</a></p>

<p><a data-t="511">What we do is we talk about the package, why we're using it, what potential alternatives might be.</a> <a data-t="520">So for each code chunk, we try to provide much more context than just describing what's going to happen next from an empirical perspective.</a></p>

<p><a data-t="530">So second piece, the code chunk.</a> <a data-t="536">We try to keep code chunks as small as possible to solve a very specific problem.</a> <a data-t="542">Not too many things at the same time in a code chunk, which is sometimes challenging.</a> <a data-t="549">And we try to keep the object naming and the syntax consistent across all the chapters.</a> <a data-t="557">So people should feel familiar with the syntax when they browse through the book.</a> <a data-t="563">And here we're just implementing exactly the regression model that we described above and we're using the function that we described.</a></p>

<p><a data-t="569">And as a third part, we have the results.</a> <a data-t="574">And we, as much as I love </a><a href="/software/gt/">GT</a><a data-t="574"> and GT Extra to create amazing tables,</a> <a data-t="579">in our book, we wanted to print the results as they are for people who would just copy the code into their console.</a> <a data-t="586">So we didn't want to put many bells and whistles around it because we want to be easily reproducible.</a> <a data-t="594">People can just go to a website, copy the stuff that they find interesting and ideally get exactly the same results.</a></p>

<p><a data-t="606">And we call this combination of narrative code and results transparent code.</a> <a data-t="616">And I think there are also a lot of tremendous benefits.</a> <a data-t="622">I've just talked about the reproducibility.</a> <a data-t="625">People can just copy the stuff.</a> <a data-t="627">They can understand what exactly is happening.</a> <a data-t="631">We can bridge the gap between theory and applications very easily.</a> <a data-t="637">We also have a chapter on machine learning where we do a brief introduction and then immediately jump into applying machine learning models.</a> <a data-t="645">And the content is extendable.</a> <a data-t="647">So when people understand what's happening in a chapter in specific code chunks, they can just copy it, modify it for their specific application.</a></p>

<h3>Trade-offs of multi-language support</h3>

<p><a data-t="808">At this stage, I don't want to just talk about benefits.</a> <a data-t="813">I feel like there's a stronger trade-off here that you have to face when you're creating this type of content.</a> <a data-t="820">And one big pro to me is enhanced reproducibility.</a> <a data-t="826">I mean, we had our stuff in R and we just reproduced the stuff in Python.</a> <a data-t="831">That's great.</a> <a data-t="833">We found some mistakes, obviously, in the R version.</a> <a data-t="837">But we now have two consistent versions of the same applications.</a></p>

<p><a data-t="845">It increases the accessibility.</a> <a data-t="848">And now the University of Cambridge, they can use our stuff.</a> <a data-t="854">There's also learning flexibility.</a> <a data-t="859">So for instance, Stefan extended his course by telling his students, I'm just teaching these topics.</a> <a data-t="871">You might pick your own programming language.</a> <a data-t="874">So in the same course, you have some people working with R, some with Python.</a> <a data-t="879">And they are able to kind of communicate with each other.</a> <a data-t="883">And they are super happy that they can pick the tool that they find most valuable.</a></p>

<p><a data-t="891">And I want to mention, </a><a href="/software/quarto/">Quarto</a><a data-t="891"> makes it really easy.</a> <a data-t="894">We were super grateful that the Quarto team worked on features that we needed to publish the book just as we wrote it.</a> <a data-t="903">So we actually sent, we used pre-release versions to get the final version of the book to the printing press.</a> <a data-t="912">But it's a great support.</a></p>

<p><a data-t="917">But the cons are huge, because this type of consistency is quite challenging.</a> <a data-t="923">You have a numerical problem, for instance, an optimization problem.</a> <a data-t="928">And then you implement it in a different programming language.</a> <a data-t="930">And all of a sudden, you get different results.</a> <a data-t="933">So is it the optimizer? Is it the language? Should I make a mistake with the data?</a> <a data-t="938">You have to really tinker things until everything works out.</a> <a data-t="945">And that's the most painful part.</a></p>

<p><a data-t="948">You have now double the code maintenance.</a> <a data-t="950">So if somebody discovers a potential mistake in your code and you have to correct it, you have to do it twice.</a> <a data-t="959">Or another example, I updated the data that we use in the book for another year.</a> <a data-t="966">And now instead of looking at 12 chapters, I had to check 24 chapters.</a> <a data-t="971">So that was kind of a painful experience.</a> <a data-t="974">But overall, it's totally worth it for us.</a> <a data-t="978">So we're really happy that now we have both R and Python version on the website.</a> <a data-t="984">And we are really looking forward to taking this to the next level by creating new content.</a></p>
