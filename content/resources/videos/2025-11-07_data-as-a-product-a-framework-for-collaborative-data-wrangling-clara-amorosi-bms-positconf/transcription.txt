<p><a data-t="0">Hi, thanks everyone.</a> <a data-t="2">So I'm a data scientist at Bristol-Myers Squibb,</a> <a data-t="6">and I'm here to talk about a framework</a> <a data-t="7">for reproducible and collaborative data wrangling.</a></p>

<p><a data-t="12">But first, I'd like to ask if you've ever heard</a> <a data-t="14">of type one versus type two fun.</a> <a data-t="17">If you haven't, type one fun is an activity</a> <a data-t="20">that's enjoyable while it's happening.</a> <a data-t="22">So this could be sitting on a sunny beach on vacation.</a> <a data-t="26">Type two fun, on the other hand,</a> <a data-t="27">is an activity that's not as enjoyable</a> <a data-t="30">while it's happening.</a> <a data-t="31">You might even be miserable.</a> <a data-t="33">But then, when the activity is over,</a> <a data-t="35">you look back on it fondly.</a></p>

<p><a data-t="37">And so, classic examples of type two fun</a> <a data-t="40">include exercise, or my personal favorite,</a> <a data-t="43">jumping into an icy alpine lake.</a> <a data-t="46">And so, today I'll talk to you about how,</a> <a data-t="48">just like type two fun, taking the plunge</a> <a data-t="51">into reproducible research is worth it.</a></p>

<p><a data-t="55">So I often think about reproducible research</a> <a data-t="58">in the context of ad hoc analysis requests.</a> <a data-t="62">I work with a lot of clinical trial data</a> <a data-t="63">in a large organization, and I might get a request</a> <a data-t="67">from an investigator such as this one.</a> <a data-t="69">So maybe they ask me to look at baseline albumin levels</a> <a data-t="72">across two different clinical trials.</a> <a data-t="75">And albumin is a relatively common lab measurement,</a> <a data-t="78">so I'll say, okay, I'll take a look.</a></p>

<p><a data-t="80">But then, when I actually start looking into the data,</a> <a data-t="83">I realize that there's a whole bunch of questions</a> <a data-t="85">that I'll need to answer in order to actually go back</a> <a data-t="88">to the investigator with the solution.</a> <a data-t="90">So maybe I need to figure out which specimen type</a> <a data-t="93">I'm interested in, or maybe there's different units used,</a> <a data-t="97">so I need to perform a unit conversion step.</a> <a data-t="100">And inevitably, most of the work goes into</a> <a data-t="102">finding, cleaning, and structuring the data.</a></p>

<p><a data-t="108">And this is really the classic 80-20 rule of data science,</a> <a data-t="110">where most of the work goes into this data wrangling step,</a> <a data-t="115">and then you only have the last 20%</a> <a data-t="117">where you do the actual analysis</a> <a data-t="119">and essentially get the answer.</a></p>

<p><a data-t="122">And oftentimes, or sometimes,</a> <a data-t="125">this can be disposable data work.</a> <a data-t="127">You maybe just need to get to the answer</a> <a data-t="129">as quickly as possible, and so you essentially</a> <a data-t="131">just turn in your homework and go on with your day.</a> <a data-t="134">But I feel like fairly often, maybe you're asked</a> <a data-t="138">to return to this analysis six months later,</a> <a data-t="141">or maybe you want to share your analysis with somebody else.</a></p>

<p><a data-t="145">And in this case, having disposable data work</a> <a data-t="147">isn't a really good solution here.</a> <a data-t="150">So one way to make this data wrangling more reusable</a> <a data-t="154">is to package it into a data product.</a></p>

<h3>What is a data product?</h3>

<p><a data-t="157">And so what do I mean by a data product?</a> <a data-t="160">In this case, a data product is an analysis-ready data set,</a> <a data-t="165">or a group of data sets that's versioned,</a> <a data-t="168">reproducible, and shareable.</a> <a data-t="170">And this is how you can essentially</a> <a data-t="173">package your data wrangling work</a> <a data-t="175">and collaborate with others</a> <a data-t="176">on creating this data product object.</a></p>

<p><a data-t="179">The only small wrinkle here is that this is more work.</a> <a data-t="183">This is definitely the type two track here,</a> <a data-t="186">but I'd like to talk through a framework</a> <a data-t="188">for creating these data products</a> <a data-t="189">and hope to convince you why it can be worth it</a> <a data-t="192">to create these data products.</a></p>

<h3>Introducing DAPR</h3>

<p><a data-t="195">So at BMS, we've developed a suite of packages</a> <a data-t="200">called DAPR, or data as a product in R,</a> <a data-t="204">that provides a framework for creating</a> <a data-t="206">these reproducible data products.</a> <a data-t="209">These were developed by a group of us at BMS</a> <a data-t="212">and are used across different types</a> <a data-t="214">of scientific analysis questions,</a> <a data-t="216">mostly in the exploratory space.</a></p>

<p><a data-t="219">The idea behind DAPR is that it allows multiple users</a> <a data-t="222">to pool their data wrangling efforts</a> <a data-t="224">and then create this data product object</a> <a data-t="226">that they can collaborate on.</a> <a data-t="229">The overall goal is to be able to build</a> <a data-t="230">these analysis-ready data products</a> <a data-t="233">that follow sound data engineering principles.</a> <a data-t="236">And so I'll talk through what I think</a> <a data-t="238">is the core principle behind DAPR,</a> <a data-t="240">and then I'll go through how you would actually use</a> <a data-t="242">the DAPR framework to build a data product.</a></p>

<p><a data-t="246">So the key principle behind DAPR is pretty simple.</a> <a data-t="251">It's just to version all the things.</a> <a data-t="253">So this comes in three different flavors.</a> <a data-t="256">You version your code, you version your environment,</a> <a data-t="258">and you version your data.</a> <a data-t="260">I've arbitrarily ranked these by how hard I think they are,</a> <a data-t="264">but DAPR attempts to make it easier for the user</a> <a data-t="268">to do all these things and also forces them</a> <a data-t="271">to version all of these things.</a></p>

<blockquote><a data-t="251">It's just to version all the things.</a></blockquote>

<p><a data-t="272">So in terms of code, DAPR uses Git and GitHub,</a> <a data-t="275">so users make sure to version control</a> <a data-t="278">and commit and push their data wrangling code.</a> <a data-t="282">In terms of versioning your environment,</a> <a data-t="284">it uses the </a><a href="/software/renv/">renv</a><a data-t="284"> package, so users track and install</a> <a data-t="288">all their required package dependencies.</a> <a data-t="290">This is also very useful when you're sharing code</a> <a data-t="293">with somebody else and you need them to start</a> <a data-t="295">from the same point as you.</a></p>

<p><a data-t="297">And then in terms of versioning your data,</a> <a data-t="299">DAPR tracks the metadata of all of the input files</a> <a data-t="302">that you're using.</a> <a data-t="304">And under the hood, it's using the pins package to do this.</a> <a data-t="307">So this is a package that's, a wonderful package</a> <a data-t="312">that's used for publishing and sharing datasets.</a> <a data-t="314">And one of the really nice things it does</a> <a data-t="316">is it creates versioned datasets.</a> <a data-t="319">And so DAPR uses this to track which versions</a> <a data-t="321">of all of your input datasets you're using.</a></p>

<h3>The DAPR workflow</h3>

<p><a data-t="326">Okay, so that's the key principle behind DAPR,</a> <a data-t="328">but how does it actually work?</a> <a data-t="330">How do you build a data product?</a></p>

<p><a data-t="333">This is the DAPR workflow, and I'll build out these steps</a> <a data-t="336">as I walk through them.</a> <a data-t="337">But the general idea is that you're going to start,</a> <a data-t="339">initialize your product, create an input data snapshot,</a> <a data-t="344">do the actual building step where you add derivation code,</a> <a data-t="348">then you do the version everything step,</a> <a data-t="350">and then finally you can deploy and share your data product.</a></p>

<p><a data-t="355">So I'll jump into the first step here,</a> <a data-t="357">which is how do you actually initialize a data product?</a> <a data-t="361">So you'll need a few things in order to start.</a> <a data-t="364">The first is you need to specify the remote board</a> <a data-t="368">that will be used to store your data.</a> <a data-t="370">This could be cloud storage,</a> <a data-t="372">or a shared server, or a networked device.</a> <a data-t="374">And you'll basically just need to gather the credentials</a> <a data-t="376">or configuration details needed</a> <a data-t="378">to connect to this remote board.</a></p>

<p><a data-t="381">You can then use one DAPR command</a> <a data-t="383">to initialize your local build environment.</a> <a data-t="386">So this will initialize Git locally,</a> <a data-t="389">it will start RN for you</a> <a data-t="392">and track all your package dependencies,</a> <a data-t="394">and it will save all the configuration details</a> <a data-t="396">you need to connect to your remote.</a></p>

<p><a data-t="401">At this step, you can create an input data snapshot.</a> <a data-t="404">So you would gather the input files that you need</a> <a data-t="407">that you're going to process.</a> <a data-t="409">In my world, this is a lot of clinical data sets,</a> <a data-t="412">but this could be any type of input file</a> <a data-t="414">as long as it's in rectangular format.</a></p>

<p><a data-t="418">DAPR then pins and syncs these input files</a> <a data-t="421">to your remote board and saves the versions</a> <a data-t="424">of all of the files that you're working with.</a></p>

<p><a data-t="431">Okay, we're now on the build step.</a> <a data-t="433">So this is where you actually build the data product</a> <a data-t="436">by adding derivation code.</a> <a data-t="438">And there's quite a lot of flexibility here</a> <a data-t="440">in how you approach this.</a> <a data-t="440">DAPR provides essentially scaffolding code</a> <a data-t="443">for how you might go about this,</a> <a data-t="444">but often the way we approach this</a> <a data-t="447">is to create derivation functions</a> <a data-t="450">where each function creates a data set</a> <a data-t="451">that you might use for analysis.</a> <a data-t="453">I'm showing an example here</a> <a data-t="455">where you have a deriveAlbumen function.</a></p>

<p><a data-t="458">And within the function, it will take a link</a> <a data-t="461">to the input data that's snapshotted on your remote board</a> <a data-t="464">and read that in and do all of the,</a> <a data-t="466">essentially, data processing work.</a> <a data-t="469">And each of these derivation functions</a> <a data-t="471">might reference one or more input files,</a> <a data-t="473">and then you might have one or more derivation functions</a> <a data-t="475">that all build on top of each other.</a></p>

<p><a data-t="480">And I should mention here</a> <a data-t="480">that we often use the targets package to help with this.</a> <a data-t="484">This is a really cool data pipeline package.</a> <a data-t="486">It's kind of make-like,</a> <a data-t="487">and it allows you to kind of set up all of your dependencies</a> <a data-t="491">in terms of the derivation functions.</a></p>

<p><a data-t="495">So as I said before, there's quite a lot of flexibility</a> <a data-t="497">in how you actually build your data product.</a> <a data-t="501">I'm showing here an example data product structure.</a> <a data-t="504">So there's a lot going on here.</a> <a data-t="506">The main thing I want you to focus on</a> <a data-t="508">is that the data product object itself</a> <a data-t="511">is structured into three main pieces.</a> <a data-t="513">So you have your inputs, your outputs, and your metadata.</a></p>

<p><a data-t="517">The inputs are all of your input files</a> <a data-t="519">that are already synced to your remote board,</a> <a data-t="522">and then the outputs are all of the data sets</a> <a data-t="525">that you've created from derivation functions.</a> <a data-t="528">And then you can optionally have metadata,</a> <a data-t="530">such as a data dictionary,</a> <a data-t="531">to define what your column names are, for example.</a></p>

<p><a data-t="536">The other thing I'll note here</a> <a data-t="538">is that we attempt to make these data products</a> <a data-t="540">a little bit more lightweight</a> <a data-t="542">by having the inputs return links to the data themselves,</a> <a data-t="547">not actually the data.</a> <a data-t="548">This is helpful because,</a> <a data-t="550">at least in the clinical trial world,</a> <a data-t="553">I would say we're working with medium-sized data,</a> <a data-t="556">not truly big data, but it is still helpful</a> <a data-t="558">to not have to read in the whole data sets at the same time.</a></p>

<p><a data-t="564">Okay, so at this point,</a> <a data-t="565">you've structured your data product however you want.</a> <a data-t="568">You've done a bunch of derivation code and added that.</a> <a data-t="572">Now you get back to the version everything step.</a> <a data-t="574">And so, Dapr will help you commit and push, essentially,</a> <a data-t="578">all of your code and other versionable things</a> <a data-t="582">to your GitHub repo.</a> <a data-t="583">So, this is the derivation code</a> <a data-t="586">and all of the data product structuring logic.</a> <a data-t="590">This is your RN files to track your package dependencies.</a> <a data-t="593">And this is also all of the metadata</a> <a data-t="596">around your input and output files.</a></p>

<p><a data-t="599">Okay, and then one final step,</a> <a data-t="601">which is that now you're ready to deploy it.</a> <a data-t="603">So, at this point,</a> <a data-t="604">you can push your data product to the remote board,</a> <a data-t="607">and it's ready to share, to use for your own analysis,</a> <a data-t="610">to share with somebody else to use for their own analysis,</a> <a data-t="614">and you're ready to go here.</a></p>

<p><a data-t="618">There's actually a sixth step here,</a> <a data-t="621">which is to iterate over this.</a> <a data-t="623">So, at this point, you can go back to step two,</a> <a data-t="625">add new input data,</a> <a data-t="628">deploy a new version of the data product,</a> <a data-t="630">or you could add more derivation code,</a> <a data-t="632">like maybe you have a new function you want to add,</a> <a data-t="634">and then you can, for each of these,</a> <a data-t="636">you'd go back and go through the cycle</a> <a data-t="638">and deploy a new version of the data product.</a></p>

<p><a data-t="641">The other helpful thing here is that all of this is version,</a> <a data-t="644">so if you have an analysis script</a> <a data-t="646">that references the data product,</a> <a data-t="648">you can refer to it by version</a> <a data-t="650">so that it doesn't break when a new version is published.</a> <a data-t="653">This is also where you can bring in collaborators</a> <a data-t="656">to work on the data product with you.</a> <a data-t="659">So, it doesn't have to be just you</a> <a data-t="660">going through this iterative process.</a> <a data-t="662">You can bring in somebody else to do this as well.</a></p>

<h3>Getting people onboarded</h3>

<p><a data-t="727">So, I hope this shows, really,</a> <a data-t="729">how Dapr can be used as a collaborative framework.</a> <a data-t="733">I do want to switch tracks a little bit now</a> <a data-t="735">and perhaps address the elephant in the room,</a> <a data-t="738">which is, how do you actually get people to try this?</a> <a data-t="741">So, I walked through, admittedly,</a> <a data-t="744">a number of different steps to create a data product,</a> <a data-t="748">and for people trying this out,</a> <a data-t="749">they might need to learn to use new tools</a> <a data-t="751">that they're not as comfortable with.</a> <a data-t="753">So, how do you actually get people</a> <a data-t="755">onboarded to a framework like this?</a></p>

<p><a data-t="758">So, I have a few tips here.</a> <a data-t="760">I will note that most of them are educational</a> <a data-t="762">or focused on reducing barriers to entry.</a> <a data-t="765">We're not forcing anyone to use this framework.</a></p>

<p><a data-t="769">The first tip I have is to really provide</a> <a data-t="771">comprehensive training.</a> <a data-t="773">So, this might be hands-on demos with real data products.</a> <a data-t="778">I would say it's been very effective</a> <a data-t="780">to have one-on-one demos,</a> <a data-t="782">so you can kind of look over somebody's shoulder</a> <a data-t="784">and see when they run into issues.</a> <a data-t="786">We also have asynchronous learning materials,</a> <a data-t="789">so this is things like video tutorials, vignettes,</a> <a data-t="793">and for all of these, just being really clear</a> <a data-t="794">and providing dedicated debugging support,</a> <a data-t="799">that's really been helpful.</a></p>

<p><a data-t="801">The second tip I have is to let people ease into the process.</a> <a data-t="805">So, you know, we do this by providing multiple DAPR roles.</a> <a data-t="809">So, the first role is a reader role.</a> <a data-t="811">So, at this point, people could essentially interact</a> <a data-t="815">with the data product in read-only mode,</a> <a data-t="817">and they could use it for their analysis,</a> <a data-t="819">but they don't have to fully learn the DAPR framework.</a> <a data-t="822">They could then graduate to a writer role</a> <a data-t="824">where they would then learn the full suite</a> <a data-t="827">of DAPR packages and tools</a> <a data-t="828">and would contribute to derivation code.</a></p>

<p><a data-t="831">And then the third role is an owner.</a> <a data-t="833">So, at this point, this is more of an admin role.</a> <a data-t="836">They would maybe manage access to the remote</a> <a data-t="839">and do some provisioning activity,</a> <a data-t="841">and we don't have as many people getting fully to the owner role,</a> <a data-t="843">which is fine.</a></p>

<p><a data-t="846">The third tip I have is really to show off the results,</a> <a data-t="850">like show off what you can get by working within this framework.</a> <a data-t="855">It's not very convincing to show, for example,</a> <a data-t="858">just the number of lines of derivation code</a> <a data-t="860">you've added to a repo.</a> <a data-t="861">That's not going to convince anyone.</a> <a data-t="863">Instead, sharing things like maybe you have a </a><a href="/software/quarto/">Quarto</a><a data-t="863"> report</a> <a data-t="867">that's linked to the data product</a> <a data-t="868">that has key figures that get shown in stakeholder meetings,</a> <a data-t="872">or maybe you have a </a><a href="/software/shiny-r/">Shiny</a><a data-t="872"> app</a> <a data-t="874">that lets you explore the data product</a> <a data-t="876">and which derivations are involved in it.</a></p>

<p><a data-t="880">And then finally, showing off the results of the collaboration.</a> <a data-t="884">So, if you find somebody</a> <a data-t="885">who's not already a reproducibility nerd,</a> <a data-t="888">but has tried out the framework and has had success with it,</a> <a data-t="892">and then they can kind of champion the framework</a> <a data-t="894">to other people, that's really key as well.</a></p>

<h3>Common roadblocks</h3>

<p><a data-t="900">So, despite all of these tips,</a> <a data-t="902">we do run into a lot of common roadblocks</a> <a data-t="905">getting people onboarded to this framework.</a> <a data-t="907">And I think a lot of these will be familiar</a> <a data-t="909">to many folks here as well.</a> <a data-t="910">They're not specific to Dapr.</a> <a data-t="912">And so, I just wanted to walk through</a> <a data-t="914">how we approach some of these.</a></p>

<p><a data-t="917">So, the first common roadblock</a> <a data-t="919">is that managing credentials is annoying.</a> <a data-t="922">So, you know, maybe you have like a Git token,</a> <a data-t="925">maybe you have like S3 keys,</a> <a data-t="928">maybe you have another data platform credential</a> <a data-t="930">you need to provide.</a> <a data-t="931">And all of these probably expire at a different time,</a> <a data-t="934">which, you know, you can never really control that.</a> <a data-t="937">We've had good success using the </a><a href="/software/keyring/">keyring</a><a data-t="937"> package</a> <a data-t="940">to manage all types of credentials,</a> <a data-t="941">but there's a lot of different</a> <a data-t="943">credential management packages out there.</a></p>

<p><a data-t="947">The second common roadblock is focused on</a> <a data-t="950">how do you really impart Git best practices?</a> <a data-t="954">I will also say there's a lot of resources out here</a> <a data-t="956">for getting started with Git.</a> <a data-t="958">I just wanted to shout out </a><a href="/people/jenny-bryan/">Jenny Bryan</a><a data-t="958">'s Happy Git with R,</a> <a data-t="961">which is an excellent resource.</a></p>

<p><a data-t="964">But I think the thing I want to focus on here</a> <a data-t="966">is that we've all been at the learning stage</a> <a data-t="968">for, you know, learning how to use Git and GitHub.</a> <a data-t="971">We've all accidentally created a merge conflict</a> <a data-t="973">or committed a file that we shouldn't.</a> <a data-t="975">And so, I think for all of these,</a> <a data-t="977">you know, leading with solutions when you're teaching people</a> <a data-t="979">how to work through these issues,</a> <a data-t="981">that's really key because we were all learning this once.</a></p>

<p><a data-t="985">And then the third common roadblock</a> <a data-t="987">kind of relates to the corporate enterprise environment.</a> <a data-t="990">So, maybe you have a corporate firewall</a> <a data-t="992">or maybe you have to configure your internal package manager</a> <a data-t="996">to install packages.</a> <a data-t="998">And for these, I would say it's been most helpful</a> <a data-t="1000">just to provide example setup code,</a> <a data-t="1003">especially across different operating systems.</a> <a data-t="1006">And I would say the key point</a> <a data-t="1008">for walking through these common roadblocks</a> <a data-t="1010">is that they will happen,</a> <a data-t="1012">and the goal is really to try to empower people</a> <a data-t="1014">to be able to work through them</a> <a data-t="1015">when they encounter them the next time.</a> <a data-t="1017">So, that's really what we try to focus on</a> <a data-t="1019">when we have people come to us with these issues.</a></p>
