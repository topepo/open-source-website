<h3>Three things organizations don't understand</h3>

<p><a data-t="523">And this kind of, like, very basic problem is not unusual.</a> <a data-t="526">I talk to other data scientists and data leaders.</a> <a data-t="526">These are the kinds of things we face a lot.</a> <a data-t="529">So there's three buckets of things, I think, that we need that organizations don't really understand coming in.</a></p>

<p><a data-t="539">One is analysis-ready data.</a> <a data-t="542">And this can go wrong in several ways.</a> <a data-t="544">One is you can't get access to the data, or it's not in a form that's safe for analytics.</a> <a data-t="550">Another might be that the data you need for the project you're doing doesn't exist.</a> <a data-t="554">Not really even being collected.</a> <a data-t="556">Or maybe it is being collected, and you can get access to it, but the data management practices are so nonexistent that it's messy to the point where it's almost unusable.</a></p>

<p><a data-t="609">Another thing we need is cross-team support.</a> <a data-t="612">So in the situation I was in at the transit company, the engineering team could have solved my problem.</a> <a data-t="617">They had the ability to get the data into an analytics data warehouse, right?</a> <a data-t="622">But they weren't expecting to need to do this work.</a> <a data-t="625">No one realized the data scientists would need any support.</a> <a data-t="628">And they had a list of high-priority items that they were already struggling to be able to address with the capacity they had.</a> <a data-t="634">So they weren't going to prioritize this.</a></p>

<p><a data-t="637">And this can also happen in the other direction.</a> <a data-t="639">Sometimes people are thrilled to see you arrive as the data person, because they think, my team has data problems.</a> <a data-t="644">We spend way too much time reporting.</a> <a data-t="648">Here's someone who can solve this for us.</a> <a data-t="651">And then you can get a lot of incoming requests, which can be helpful in understanding what your organization needs.</a> <a data-t="657">But if you can't control that, you end up spending all your time reacting to these requests, and never get to those potentially really high-impact projects, which are what people really wanted data scientists for in the first place.</a></p>

<p><a data-t="712">And the third thing is space for uncertainty.</a> <a data-t="715">So unlike almost everything else that organizations do, data science projects depend on the contents of the data in a way that makes it hard to know whether they're worth doing until we've kind of already done them.</a> <a data-t="729">And so this can make it really hard to fit into the normal way that your organization plans work.</a></p>

<h3>Guerrilla data science tactics</h3>

<p><a data-t="737">So if you find yourself dealing with these kinds of problems, what do you do?</a> <a data-t="744">And I think in my experience, there's a set of tactics that work here that I think of as guerrilla data science tactics.</a> <a data-t="752">So this means maybe not doing things ideally the way that we should be doing them, but doing what we have to do to make it work in the situation we're in.</a> <a data-t="801">So being scrappy and clever and working around the obstacles in our way.</a></p>

<p><a data-t="808">But guerrilla, of course, comes from guerrilla warfare, and warfare is not the right mindset for us to be in, right?</a> <a data-t="816">So as frustrated as we might get with some of our colleagues, they aren't trying to make things difficult for us.</a> <a data-t="822">We're all wanting to make it work, we just haven't figured out how to do that yet.</a> <a data-t="828">So instead, maybe we think about this as guerrilla data science, powerful but friendly.</a></p>

<p><a data-t="841">So what does this actually mean in practice?</a> <a data-t="841">There's a series of steps that I have found success with in getting data science off the ground.</a> <a data-t="848">The first step is scan for opportunity.</a> <a data-t="851">So this means being heads up in your organization and looking for the overlap of what's valuable to the organization and what you're able to do with data.</a></p>

<p><a data-t="863">And once you've found that opportunity, the next step is show, don't tell.</a> <a data-t="869">So it can be tempting to explain to people what you're thinking and what you're planning to do and what you need from them and how this is going to play out to get buy-in, but in my experience, at least early on, this is usually a waste of time.</a> <a data-t="923">What you need to do is build the thing, at least the skinny version of the thing, so that they can see the vision you have in a more concrete way and have something to react to.</a></p>

<p><a data-t="936">Next, take the data and run.</a> <a data-t="940">So this doesn't mean stealing the data or hacking into the data, but it means that if you can't get access in the way that you ultimately will need to, you have to do whatever it takes to get an initial data set.</a> <a data-t="952">Maybe this means befriending an engineer who does have access and asking them to just help you get a one-time data set that you can get started with.</a></p>

<p><a data-t="1002">And then once you've used that data set to build something that delivers on that opportunity, nail the landing.</a> <a data-t="1011">And what this means is communicate.</a> <a data-t="1014">So all of these things you've done up to this point don't matter if no one knows what you did or doesn't understand why it's valuable.</a> <a data-t="1023">And I think it's almost impossible to overinvest in communication, communicating things different ways until it connects with people and they really get it, and also communicating broadly.</a> <a data-t="1034">So not just your primary stakeholder or your team, but everyone around the organization.</a> <a data-t="1040">Make friends and talk to them all the time about what you're doing.</a></p>

<p><a data-t="1045">And then once you've done this, it's time to up the ante.</a> <a data-t="1049">So once you've done an initial project that shows value and people know about it and understand it, you want to use that organizational goodwill to do something that's maybe higher risk but higher impact the next time.</a> <a data-t="1063">And so you go through this cycle again.</a></p>

<p><a data-t="1067">So when I found myself in this position of not being able to access the data that I was there to find value in, I was at the scan for opportunity stage.</a> <a data-t="1078">And the opportunity that presented itself, we had just raised money, we had a growing employee base, people needed to understand what was going on with the company.</a> <a data-t="1087">And so we worked on business metrics.</a> <a data-t="1091">Not what I had come to do, but it was the opportunity we had at the time.</a> <a data-t="1095">And so we started building a FLEX dashboard and deployed it and were sharing it with people so they could see what we were doing.</a></p>

<p><a data-t="1106">The data here did not come from that production system, but we were able to find scrappy ways to pull information in from around the company to feed this dashboard.</a> <a data-t="1113">And then we communicated it to everyone.</a> <a data-t="1113">We showed it in company meetings, we talked to teams about how they could use it.</a> <a data-t="1118">And that got us some goodwill to do the next step.</a></p>

<p><a data-t="1122">So now we're back to scan for opportunity.</a> <a data-t="1125">And this time, the company was getting into a new product in the area of microtransit, which is like an Uber pool for transit agencies and universities.</a> <a data-t="1136">And this runs on an algorithm.</a> <a data-t="1137">So from our data science viewpoint, we thought, well, of course we understand how the algorithm works, but that's not the same as understanding how it will play out in the real world.</a> <a data-t="1148">And we thought we could figure that out, figure out where it works well and where it doesn't in which situations by using simulation.</a></p>

<p><a data-t="1156">And so now we had this goodwill.</a> <a data-t="1158">So I thought, this time we can do it right.</a> <a data-t="1160">I went to the product manager and talked about our idea and why we thought it would be valuable and what we needed from the engineering teams to do this.</a> <a data-t="1169">And she said, oh, that's interesting.</a> <a data-t="1172">We're not going to prioritize this.</a> <a data-t="1174">No one's asking me for it.</a> <a data-t="1176">We have a long list of high-priority items, and I already can't do them all with the team that I have.</a> <a data-t="1182">So we're just not going to be able to help you with this.</a></p>

<p><a data-t="1185">So back to show, don't tell.</a> <a data-t="1188">How could we build a version of this that we could show people on our own?</a> <a data-t="1194">So in this case, because we were using simulation, we were creating our own data.</a> <a data-t="1200">So that's problem solved.</a> <a data-t="1200">But we still needed access to the algorithm.</a> <a data-t="1203">So we went to engineering with this one-time ask.</a> <a data-t="1206">Just help us get access so that we can run this algorithm offline.</a> <a data-t="1210">It won't interfere with your production system, and we'll leave you alone.</a> <a data-t="1213">And so we got that.</a></p>

<p><a data-t="1216">So then we built our </a><a href="/software/shiny-r/">Shiny</a><a data-t="1216"> app to create inputs.</a> <a data-t="1218">We scheduled all our simulations.</a> <a data-t="1220">We created reports for different scenarios.</a> <a data-t="1224">So then we had something really cool to show people.</a> <a data-t="1226">So we went back to our team and the product manager and showed them this.</a> <a data-t="1230">And within our department, people thought, you know, this is interesting, but it's kind of like a toy.</a> <a data-t="1235">It's not really, you know, part of our software.</a> <a data-t="1238">It's just kind of an interesting side project.</a> <a data-t="1241">But we kept talking about it.</a> <a data-t="1242">And one day, I showed it to the chief revenue officer.</a> <a data-t="1246">And he immediately saw the value in this because he's thinking about the customer, right?</a> <a data-t="1250">The customer is uncomfortable with this algorithm.</a> <a data-t="1253">They need to be able to see how this is going to work for them before they're willing to put this out in the world.</a> <a data-t="1258">And we very quickly became part of our marketing and sales process, and also part of how we communicated to our investors why this was important and what it actually was.</a> <a data-t="1267">So now we had gotten to that point of this high impact.</a> <a data-t="1271">This was the kind of thing that the CEO had wanted to bring the data science team in for to begin with.</a></p>

<h3>Reflections on the process</h3>

<p><a data-t="1280">So a few reflections about what's important in this process.</a> <a data-t="1285">One is to maximize speed and autonomy.</a> <a data-t="1289">So even though we're in this privileged position where people have great faith that we can do amazing things, even though they don't quite know what they are, we still have to start delivering eventually, right?</a> <a data-t="1299">So we have a limited amount of time.</a> <a data-t="1302">And in my experience, the best way to maximize speed is to maximize autonomy of the team.</a></p>

<p><a data-t="1309">And this doesn't mean working in a silo or not collaborating with other teams.</a> <a data-t="1314">But what it means to me is that that iterative data science development process, you need to be able to do that without running into places where you need help from another team, because that's what's going to slow you down.</a></p>

<p><a data-t="1328">The other thing is to build foundations wherever possible along the way.</a> <a data-t="1332">So even though we're doing these scrappy things in our guerrilla data science tactics, everywhere we have a chance, we want to start building things that we can build again on later.</a> <a data-t="1341">So building data pipelines that we can schedule, building R packages that we can reuse, all of those things.</a></p>

<p><a data-t="1350">And these are the things where my teams have found Posit Connect to be really helpful.</a> <a data-t="1354">So instead of every time we need to deploy a dashboard or we need to deploy an app or we need to run a cron job, having to go ask engineering for help with getting that set up, it was just a one-time ask, help us set up Posit Connect.</a> <a data-t="1388">And then we're off and running on our own.</a> <a data-t="1391">And we can do all these things without constantly needing support.</a></p>

<p><a data-t="1397">So if you've gone through this process and you've gotten to that first really high-impact result, pat yourself on the back.</a> <a data-t="1404">You can declare victory.</a> <a data-t="1406">And now you've completed step one, because this is just the beginning to create momentum.</a> <a data-t="1412">But going forward, you can't do things this way and have a sustainable and scalable data science practice.</a> <a data-t="1421">Particularly the take the data and run step needs to evolve.</a> <a data-t="1426">And David, I would love to hear what you've learned about doing data science at scale and how you've done some of these things.</a></p>
