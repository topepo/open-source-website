<h3>Design considerations</h3>

<p><a data-t="669">There are a few considerations we have to take</a> <a data-t="669">into account. The first one is why not just use AI? I mean, it's 2025, and LLMs can write code.</a> <a data-t="678">But I think the main problem here is that AI is not deterministic. The same input is going to</a> <a data-t="684">produce different outputs, which can be confusing to users.</a> <a data-t="690">Also, we want guaranteed correctness.</a> <a data-t="691">Because this GUI is fairly simple, we already know all of the types of operations that are</a> <a data-t="697">available. We already know the type of column that's available. We can just use these code</a> <a data-t="702">templates to fill in the blanks, and we can make sure that this is correct 100% of the time.</a> <a data-t="707">An LLM might generate code that's mostly right, but using this template, we're able to guarantee</a> <a data-t="713">our output.</a> <a data-t="719">Speed matters. People don't want to either pay for every single time they want to</a> <a data-t="723">generate this code. It will take longer if you're making these API calls to a different</a> <a data-t="729">LLM that's hosted somewhere else.</a> <a data-t="731">So, for this, this deterministic rule-based system</a> <a data-t="731">worked really well for this highly structured UI component.</a></p>

<p><a data-t="737">Now, we can think about, you know, it might be hard once we get into more complex filters.</a> <a data-t="743">There are a few things that really helped out with this.</a> <a data-t="749">Some of them are on the side of the</a> <a data-t="749">GUI interface. All of the filters in Positron are AND filters, which means that the order doesn't</a> <a data-t="756">matter. So, we can sort for chocolate chip recipes and then see things that are less than 60 minutes</a> <a data-t="764">long or do things in the opposite direction, and it will give the same information.</a> <a data-t="771">This simplifies a lot of our filters. So, perhaps if this continues to build out and</a> <a data-t="777">Positron supports or logic, I'll have to update this code to support the new UI.</a> <a data-t="783">But for now, we don't have to deal with nested filters as much.</a></p>

<p><a data-t="789">Another thing to think about was what syntax to show. Should we create the most concise code</a> <a data-t="794">or should it be more readable? Do we always want to use method chains or break operations</a> <a data-t="799">up into multiple steps?</a> <a data-t="806">So, my philosophy when I was building this was that the generated code</a> <a data-t="806">should be idiomatic for each backend. It should look like what an expert would write.</a> <a data-t="811">It could also be a teachable moment so that people can see how to express operations in the code.</a> <a data-t="817">So,</a> <a data-t="817">we want something that scales well but is also readable.</a> <a data-t="823">And most crucially, it needs to be copy</a> <a data-t="823">and paste ready. Users should be able to take this code and use it immediately and generate the exact</a> <a data-t="830">same results without any sort of modification.</a> <a data-t="835">Think of it like writing a recipe. You want it</a> <a data-t="835">to be clear enough that a beginner can follow it but also concise enough that an experienced baker</a> <a data-t="841">doesn't get bored reading it or even make it written in a way that people can modify this</a> <a data-t="846">code easily.</a> <a data-t="852">Sometimes these goals conflict and we have to make careful choices.</a> <a data-t="857">The tradeoff</a> <a data-t="857">between ease of use and reproducibility is not fundamental. It's a design choice.</a></p>
