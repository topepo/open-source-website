<h3>Technology choices</h3>

<p><a data-t="391">So, this is the dashboard we made, and now we're going to talk about how we got them.</a> <a data-t="397">So, in terms of how we made the dashboard, we decided to use </a><a href="/software/quarto/">Quarto</a><a data-t="397"> dashboards.</a> <a data-t="402">So, this is a feature of Quarto that's been available since January this year, and it allows you to create dashboards, but using Quarto kind of syntax.</a> <a data-t="411">So, in terms of the different technologies we were trying to choose between before we decided on Quarto dashboards,</a> <a data-t="418">we considered </a><a href="/software/shiny-r/">Shiny</a><a data-t="418">, creating it as a Quarto document, or the one that we went with in the end, a Quarto dashboard.</a></p>

<p><a data-t="425">So, here are the few kind of criteria that we had to make this decision.</a> <a data-t="430">So, the first thing was that we wanted something with a dashboard aesthetic.</a> <a data-t="434">We didn't want a document that we had to scroll through.</a> <a data-t="436">We just wanted to be able to look at it, click on a couple of things, and get the information that we needed.</a></p>

<p><a data-t="443">The next thing that was important to us was the fact that with Quarto dashboards, you can work with markdown syntax.</a> <a data-t="450">So, we were thinking about what is this project about?</a> <a data-t="454">And really, the information in it, it doesn't really change that often.</a> <a data-t="459">We were refreshing the data like once a day, and we didn't need a huge amount of interactivity.</a> <a data-t="464">And Shiny is great, but it felt like just too much for this project, because the idea here was to reduce the maintenance burden.</a> <a data-t="472">And so, being able to work with markdown syntax made life a lot easier for us.</a></p>

<p><a data-t="478">Another thing we had to consider was deployment of the dashboard.</a> <a data-t="482">So, we decided we wanted to deploy it with GitHub Actions, mainly because that's something that I'm familiar with,</a> <a data-t="488">and that was something we could get working quickly.</a> <a data-t="490">And actually, you can deploy Shiny apps when it's Shiny live on GitHub Actions,</a> <a data-t="495">but it still just felt simpler to go with the Quarto dashboard option.</a></p>

<p><a data-t="500">But actually, the most important thing for us was that we wanted to work in something where we could have both R and Python combined.</a> <a data-t="507">We didn't want to divide the project up between, okay, this bit is yours, so you can do that in Python, and this bit is mine, so I'll do that in R.</a> <a data-t="516">We wanted to be able to both work on all of the different bits.</a> <a data-t="518">So, we could just have fun and enjoy it.</a></p>

<h3>Combining R and Python in Quarto</h3>

<p><a data-t="523">So, just having a bit of a think about the technological decisions and choices we had to make,</a> <a data-t="529">when combining R and Python in a Quarto dashboard, I think there's one decision that has to be made up front,</a> <a data-t="535">and that's whether you start from R or you start from Python.</a> <a data-t="538">So, starting from R, you're using the NitR engine and </a><a href="/software/reticulate/">reticulate</a><a data-t="538"> to pass data between R and Python.</a> <a data-t="545">Starting from Python, it's using Jupyter as the engine and using RPy2.</a></p>

<p><a data-t="552">And we were thinking, okay, can we do a bit of a compare and contrast of these?</a> <a data-t="556">But to be honest, when we were kind of building this, we found they're pretty similar approaches.</a> <a data-t="560">In terms of the project, we went starting from R solely because I was saying that at the time, and it was what I was familiar with.</a> <a data-t="567">So, thinking about kind of the structure of the dashboard and the code inside it,</a> <a data-t="571">we started off having a mix of scripts.</a> <a data-t="574">So, some were in R, some were in Python.</a> <a data-t="577">And then we had this index Quarto markdown document that orchestrated all of that.</a> <a data-t="582">So, that was in a mix of R and Python.</a></p>

<p><a data-t="585">But then we had this data Quarto markdown document,</a> <a data-t="588">and that actually enabled us to use a really cool feature of Quarto.</a> <a data-t="591">So, there's this concept of Quarto includes.</a> <a data-t="594">So, includes allow you to reuse information or code across multiple documents,</a> <a data-t="599">or in our case, multiple times in the same document.</a> <a data-t="603">So, this code snippet you can see now is from that index QMD file.</a> <a data-t="608">And we've got their high-level headings, pyro or R.</a> <a data-t="612">We've got a variable, which is a string.</a> <a data-t="619">We've got a variable, lang.</a> <a data-t="622">So, we set that to Python or R.</a> <a data-t="624">And then we include the data Quarto markdown document, and that's it.</a> <a data-t="629">And we get the same dashboard for the different languages.</a></p>

<p><a data-t="632">And the reason this was really important to us is that when we started this project,</a> <a data-t="636">Alenka and I didn't want this to be a collaboration between us that only benefit the two of us.</a> <a data-t="640">We wanted to be able to roll this out to other bits of the Arrow dev community.</a> <a data-t="645">So, let's say the Arrow C++ developers would make good use of this dashboard.</a> <a data-t="650">We could just roll this out, like, in about 30 seconds to them.</a> <a data-t="653">So, that was super important to us.</a></p>

<p><a data-t="656">Okay. So, the last kind of technical question we wanted to ask is,</a> <a data-t="661">like, how much R and how much Python do we end up in the end?</a> <a data-t="664">So, quite conveniently, GitHub on a repository page actually has this information.</a> <a data-t="670">So, we found we ended up actually with a pretty even split, which was pretty cool.</a></p>
