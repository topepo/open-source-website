<p><a data-t="83">And the main insight that I had through this process was that it's not a technical problem, really.</a> <a data-t="88">It's a social problem.</a></p>

<h3>What technical debt actually is</h3>

<p><a data-t="90">Technical debt, for those of you who are not familiar, basically all of the little shortcuts that we take developing things, especially under time pressure, that make the products less stable or less easy to maintain over the long term.</a> <a data-t="103">The idea is that eventually you're going to pay that back.</a> <a data-t="106">That's the idea.</a> <a data-t="107">It's really the practice.</a> <a data-t="109">And when you look at the actual artifacts that we call technical debt, they're almost always social artifacts.</a> <a data-t="116">And I think they fall into two main categories.</a> <a data-t="119">They're a failure of consideration or a failure of communication.</a></p>

<p><a data-t="127">So communication is really just what you think about it.</a> <a data-t="130">It's how we communicate technically with other people.</a> <a data-t="134">So the things that fall into this category are, like, documentation, which is how we communicate use and purpose.</a> <a data-t="141">Tests, which is how we communicate correctness, like what the software is meant to do precisely.</a> <a data-t="148">And then also things like code style and project organization.</a> <a data-t="152">How easy it is to find the thing that you're looking for in a code project and how easy it is to understand that thing, to actually just physically read it.</a></p>

<p><a data-t="161">So those are all quite obviously, I think, things that happen between other people.</a> <a data-t="164">We can't judge whether documentation is good or bad without knowing whether another human being who's relevant to us understands it, right?</a> <a data-t="173">There's no, like, perfect communication that we can transmit to the world.</a> <a data-t="177">What matters is whether somebody understands us.</a></p>

<p><a data-t="181">Consideration are all the things that have to do with how well your software handles things that are not problems that you have, right?</a> <a data-t="188">Whenever we're developing software, it does a really good job of handling the problem that's in front of us, the problem that we're thinking about.</a> <a data-t="196">But we can judge software as being kind of good or bad, robust or not, based on how it sort of handles a broader spectrum of people, right?</a> <a data-t="205">So these are things like the code isn't robust, right?</a> <a data-t="208">Someone throws some arguments at it that you didn't really think about, that weren't something that you would think to throw at a function, and it all falls down.</a></p>

<p><a data-t="217">It can't be updated easily. Maybe there's a lot of repetition and things need to be updated in many places, right?</a> <a data-t="223">It doesn't solve the problems of future people, or even just the problems of other people today.</a> <a data-t="228">And then it has things like fragile dependency structures, or maybe it works on some scales, but it doesn't work at all on larger scales.</a> <a data-t="236">So all of these are basically about how good our imagination is when we're developing these products, right?</a> <a data-t="243">How well we can think about all the different things that somebody might conceivably throw at that,</a> <a data-t="248">and how skilled we are at putting ourselves in other people's shoes to understand how they're going to relate to that problem,</a> <a data-t="255">what they're going to try to solve with it, and how they're going to try to solve it.</a></p>

<p><a data-t="258">So again, these are really social problems, right?</a> <a data-t="261">It's not so much that there's a perfect dependency structure, it's not that there's a perfect kind of robustness.</a> <a data-t="268">It has to do with how we're considering the needs of other people in our organization and the people who will join it in the future.</a></p>

<blockquote><a data-t="268">It has to do with how we're considering the needs of other people in our organization and the people who will join it in the future.</a></blockquote>

<h3>Why paying back technical debt is so hard</h3>

<p><a data-t="275">So this also gives us some insight into why it's so difficult to pay technical debt back, right?</a> <a data-t="283">Because when you realize that actually this is a problem between two people,</a> <a data-t="287">you sort of bring in this other insight, which is that people are really kind of fickle and biased, right?</a> <a data-t="292">So some of the biases that I think you encounter, so one of them is status quo bias.</a></p>

<p><a data-t="297">This is, of course, the 1970s UK rock band, Status Quo. I'm sure it's familiar to you.</a> <a data-t="302">So this is just that when you're working with a system for long enough, right,</a> <a data-t="308">you've paid the price of learning that system, and you've kind of gotten used to the pain you feel when you use it, right?</a> <a data-t="313">So it's sort of all those things kind of disappear for you.</a> <a data-t="316">And so you have this sort of bias that you kind of like the way it is, right,</a> <a data-t="320">and you're not going to really be that interested in changing it.</a> <a data-t="323">You're going to sort of overestimate the cost of change because you've already learned the existing thing,</a> <a data-t="328">and nobody's learned the new things, or you're going to have to do some work to learn the new thing.</a></p>

<p><a data-t="333">The second bias is the IKEA effect, and this is one of my favorites.</a> <a data-t="337">I love this picture because you can really tell that she likes that chair.</a> <a data-t="341">So when you build something from IKEA, basically you overestimate how good it is.</a> <a data-t="346">So you put something together yourself, and you kind of invest a little bit of your ego into that thing,</a> <a data-t="350">and since your ego is your favorite thing, suddenly that thing becomes way better than everything else.</a> <a data-t="355">I think this is a big part of why programmers tend to like building their own databases, right?</a> <a data-t="361">Because even if it's worse objectively than the database you could buy, you built it,</a> <a data-t="366">so there's some pride that goes along with that.</a></p>

<p><a data-t="369">So similarly, whenever you're trying to replace something,</a> <a data-t="372">usually you're replacing something that somebody in your organization who's still there has built, right?</a> <a data-t="377">So they overvalue it.</a> <a data-t="379">And lastly, it's the parenting effect.</a> <a data-t="381">This is my daughter's first experience with spinach.</a> <a data-t="386">When you watch something grow up, you forgive its messes, right?</a> <a data-t="389">You forgive the problems because you knew...</a> <a data-t="391">You sort of see how it's sort of progressed over time,</a> <a data-t="394">and you're like, well, it's so much better than it used to be, right?</a> <a data-t="397">This system can do so much more than it used to be.</a> <a data-t="400">It's so much easier to use than it used to be.</a> <a data-t="402">And you sort of forget about that actually right now it's got some problems.</a></p>

<p><a data-t="407">So all of these things together, I'm sure we could think of other biases,</a> <a data-t="411">but all of these things together means that it's never a fair contest, right?</a> <a data-t="416">Whenever you're trying to pay back technical debt,</a> <a data-t="419">it's not that people are going to rationally evaluate,</a> <a data-t="421">like, which one is objectively a little bit better, right?</a> <a data-t="424">When you're replacing these things, you have to do much, much better</a> <a data-t="427">in order to get over these kinds of humps that you find within an organization.</a></p>

<h3>Building delightful products</h3>

<p><a data-t="432">And the goal, I think, for doing that is to try to build delightful products.</a> <a data-t="437">So something where...</a> <a data-t="439">This is sort of similar to how we relate with external users of products, right?</a> <a data-t="442">We want to build something that actually people get excited about when they use,</a> <a data-t="446">that they really sort of...</a> <a data-t="447">It's a joyful experience when they use this thing.</a> <a data-t="450">And that can mean a lot of different things.</a> <a data-t="452">For me, like, some of the things I really focus on are</a> <a data-t="455">that it sets up in a single step, so like install.packages, right?</a> <a data-t="459">You only have to call one thing, and the setup is complete.</a> <a data-t="463">There's a clear, obvious first problem.</a> <a data-t="467">There's a way to...</a> <a data-t="468">Kind of a clear way to get from the obvious first problem</a> <a data-t="471">to something quite complicated to be a master of that system.</a> <a data-t="475">And then it has sort of help resources, and it doesn't break, right?</a> <a data-t="481">It doesn't sort of just randomly stop working.</a></p>

<p><a data-t="484">So one of the products I really love that fulfills all these things is R Markdown</a> <a data-t="488">because especially it sort of gives you something that, like, right away,</a> <a data-t="491">here's the easy thing that you can do right now.</a> <a data-t="494">But down the road, you know, you can just really take over all publishing, you know?</a> <a data-t="499">Like, you can publish all the websites, all the books, right?</a> <a data-t="501">Using the same framework, and there's a smooth path to that.</a> <a data-t="506">So building delightful products.</a> <a data-t="507">That's the goal.</a> <a data-t="508">And if you do that, then I think you can kind of overcome those biases.</a></p>

<p><a data-t="512">How do you build delightful products?</a> <a data-t="513">Well, I have a three-step plan for you.</a> <a data-t="516">So the first one is to find the right beachhead,</a> <a data-t="519">which is to pick your first battle in this process of paying that technical debt</a> <a data-t="523">and pick that correctly.</a> <a data-t="525">Second is to separate users and maintainers</a> <a data-t="527">and to treat these internal products more like you treat external pacing products.</a> <a data-t="531">And lastly, empathize with the debtor.</a></p>

<h3>Finding the right beachhead</h3>

<p><a data-t="534">So a beachhead, for those of you who are not familiar with the military things,</a> <a data-t="539">is like when you're invading a country,</a> <a data-t="541">you, like, have, like, a little staging area that you have to take first, right?</a> <a data-t="544">So it's not about winning the war, but it's about, like,</a> <a data-t="546">putting yourself in a position where you can, you know,</a> <a data-t="549">deploy your resources and, you know, get supply lines and all that stuff.</a> <a data-t="553">So finding the right beachhead is very much about, like,</a> <a data-t="555">picking the right project.</a></p>

<p><a data-t="557">And so this is a picture of a piece of code base</a> <a data-t="561">that I was working on at Socure when I was first starting,</a> <a data-t="565">and each of these is an R script.</a> <a data-t="566">Each of the dots is an R script,</a> <a data-t="567">and each of the lines is a sourcing relationship</a> <a data-t="569">between the script and another one.</a> <a data-t="572">And Socure at the time was a very small data science team</a> <a data-t="576">and has grown a lot,</a> <a data-t="577">so this was a kind of type of way of sharing code</a> <a data-t="579">that works really pretty well if you're, like, four or five people.</a> <a data-t="582">It works less well if you're, like, 25 or 30 people.</a></p>

<p><a data-t="586">And there's a sort of natural tendency for myself to, like,</a> <a data-t="590">look at this and say,</a> <a data-t="591">I just need to replace this with a package, right?</a> <a data-t="593">I need to take over all of this code and put it into one thing.</a> <a data-t="596">And that's going to fail, right?</a> <a data-t="598">Because there's no way that I'm going to be able to really understand</a> <a data-t="601">what each of these things does,</a> <a data-t="602">and there's no way that actually I even know</a> <a data-t="604">that a lot of these things need to be around at all, right?</a> <a data-t="607">So you have to be strategic about what you're picking.</a> <a data-t="609">Which of these am I going to use to give me kind of something</a> <a data-t="612">which I know I'm going to be able to accomplish,</a> <a data-t="614">and it's also going to build trust in the organization.</a></p>

<p><a data-t="618">So some of these things I think it's really important to focus on</a> <a data-t="620">small, contained projects,</a> <a data-t="622">and something that you can do really, really well at, right?</a> <a data-t="624">You're trying to knock it out of the park,</a> <a data-t="626">and it's easier to do that with a baseball</a> <a data-t="628">than like a bag of Jell-O or, you know, an elephant or something, right?</a> <a data-t="632">So you want to have something that's small,</a> <a data-t="633">because mostly what's going to happen</a> <a data-t="635">is people are going to judge it based on the delta,</a> <a data-t="637">not so much the overall value.</a> <a data-t="639">So it's better to do really, really well in an area of limited scope</a> <a data-t="642">than just a little bit well in an area of big scope.</a></p>

<p><a data-t="646">It's good if it's an orphan or greenfield project,</a> <a data-t="648">like there's nobody in the organization who's super attached</a> <a data-t="650">to how it's like now.</a> <a data-t="652">And if there's external pressure coming from outside of your team,</a> <a data-t="656">that's also really great,</a> <a data-t="657">because nobody likes to respond to that.</a> <a data-t="659">So if you can take that on, it's really good.</a> <a data-t="661">So in my case, what we picked there</a> <a data-t="663">was the database access functions, right?</a> <a data-t="665">They were small, I could understand them well,</a> <a data-t="667">and we were responding to some external pressure</a> <a data-t="669">from our infrastructure team to sort of change</a> <a data-t="671">how we were storing and distributing credentials.</a></p>

<p><a data-t="674">So find the right beachhead.</a> <a data-t="676">Put yourself in a position where you can solve this small problem</a> <a data-t="679">and solve it well so that you can move forward.</a> <a data-t="682">And what this will allow you to do</a> <a data-t="684">is develop trust within the organization.</a> <a data-t="687">It'll allow you to build up the social</a> <a data-t="689">and sort of technical infrastructure that you need</a> <a data-t="691">to solve these problems, like a CI process</a> <a data-t="693">or just learning how to do code review with other people.</a></p>

<h3>Separating users and maintainers</h3>

<p><a data-t="697">The second main point is to separate users and maintainers.</a> <a data-t="701">And this is in order to build delightful products,</a> <a data-t="703">you need somebody who is defining what delight means</a> <a data-t="706">and you need somebody who's trying to make the product do that, right?</a> <a data-t="709">So in other words, you need a user and you need a maintainer.</a> <a data-t="712">So ideally, every system in your team or your organization</a> <a data-t="715">should have a written-down list</a> <a data-t="717">of who is a maintainer of that product</a> <a data-t="719">and everybody else is a user.</a></p>

<p><a data-t="721">These two groups abide by a basic and holy contract.</a> <a data-t="724">That contract is that users get coddled</a> <a data-t="727">and maintainers get opinions, right?</a> <a data-t="730">So you have to pick one of these things.</a> <a data-t="732">And we all want to kind of sit between these.</a> <a data-t="734">Like, we want to be coddled, and we also want to have opinions.</a> <a data-t="737">We want to make these decisions,</a> <a data-t="739">but we don't really want to deal with all the consequences</a> <a data-t="741">that might happen down the road from those decisions.</a> <a data-t="743">We don't want to write the code,</a> <a data-t="745">we don't want to document it or write tests.</a> <a data-t="747">What this says is basically we're not doing that anymore</a> <a data-t="750">with these sort of low-technical problems.</a> <a data-t="752">We're going to have maintainers and we're going to have users.</a></p>

<p><a data-t="755">And if you're a user and you want to be a maintainer,</a> <a data-t="757">you're welcome to, but you're going to get</a> <a data-t="759">all of the other stuff that goes along with that</a> <a data-t="761">to make sure that this product is delightful.</a> <a data-t="764">So being a good maintainer means</a> <a data-t="766">that you're responsible for making it delightful.</a> <a data-t="768">That's your goal, right?</a> <a data-t="770">You want to make this product really, really good,</a> <a data-t="772">really, really easy to use. You want to make it a joy.</a> <a data-t="774">With that, you have the responsibility</a> <a data-t="776">and the authority to do that, right?</a> <a data-t="778">So you're the decision-maker over what that product does</a> <a data-t="780">and how it does it, right?</a></p>

<p><a data-t="782">You decide how it's going to accomplish this goal,</a> <a data-t="784">and you kind of judge yourself</a> <a data-t="786">based on whether you're accomplishing it.</a> <a data-t="788">So along with this means that you don't kind of ask</a> <a data-t="791">the users to do your job for you.</a> <a data-t="793">This comes up a lot for internal products with documentation,</a> <a data-t="797">where the first time that we ask somebody to set up a system,</a> <a data-t="800">and if the documentation is wrong,</a> <a data-t="802">we ask that person to fix the documentation,</a> <a data-t="805">when usually, actually, they're the worst person</a> <a data-t="807">to document something because they understand it the least.</a> <a data-t="809">So this is sort of something where you say,</a> <a data-t="811">like, as a maintainer, I'm going to do all of this stuff</a> <a data-t="813">to make sure that this thing is really good,</a> <a data-t="815">and I'm not going to expect the user</a> <a data-t="817">to be able to do any of the things that I should do.</a></p>

<p><a data-t="820">Being a good user is basically the opposite of this, right?</a> <a data-t="823">You get to define what delight means,</a> <a data-t="825">and importantly, you have to sort of turn off</a> <a data-t="828">the part of your brain that can understand</a> <a data-t="830">computer systems really, really well.</a> <a data-t="832">So as developers, like, it's tempting to basically be like,</a> <a data-t="835">oh, I have a bug with a system I don't maintain.</a> <a data-t="837">I'm going to go read the source code and figure out what that bug is.</a> <a data-t="840">And that's rarely a good use of your time,</a> <a data-t="842">both because you have your own systems to maintain,</a> <a data-t="844">and also because you're not actually fixing the problem</a> <a data-t="847">for the sort of, like, lowest common denominator user, right?</a> <a data-t="851">You want to have the feedback be like,</a> <a data-t="853">actually, this was hard to use.</a> <a data-t="855">I had to think hard to get this to set up.</a> <a data-t="857">Please fix that, right?</a> <a data-t="859">That's not delightful if I have to work too hard at it.</a></p>

<p><a data-t="862">It's also good to ask about the problem, not the solution.</a> <a data-t="865">So rather than proposing, like, you should change</a> <a data-t="867">this technical thing in your system,</a> <a data-t="869">say, like, as a user, I have this problem, right?</a> <a data-t="871">There's a reason why product managers use that a lot.</a> <a data-t="873">It's a really good way of phrasing it.</a> <a data-t="875">And it's just important to not be embarrassed</a> <a data-t="877">about complaining about something</a> <a data-t="879">and to really try to never read source code.</a></p>

<h3>Empathizing with the debtor</h3>

<p><a data-t="935">So the last point is to empathize with the debtor.</a> <a data-t="939">And I think the important thing here</a> <a data-t="941">is whenever you're doing these projects,</a> <a data-t="946">you kind of have to, like, remember</a> <a data-t="949">that what you're doing is kind of inherently threatening</a> <a data-t="952">and sort of insulting, right?</a> <a data-t="954">Because you're saying to somebody</a> <a data-t="956">who has worked hard on something, right,</a> <a data-t="958">that the thing that they've developed, like,</a> <a data-t="960">doesn't work anymore, right?</a> <a data-t="962">Or maybe it never worked, but it doesn't work anymore.</a></p>

<p><a data-t="964">And that is really a problem</a> <a data-t="967">because you're also asking them to learn something.</a> <a data-t="969">And when people are sort of threatened or afraid,</a> <a data-t="972">it turns out that they can't learn anything, right?</a> <a data-t="975">Those two parts of your mind, like,</a> <a data-t="977">they can't work together.</a> <a data-t="979">So you need to have people be kind of, like,</a> <a data-t="981">comfortable and feel supported and understood</a> <a data-t="985">in order to get them to change</a> <a data-t="987">and also in order for you to learn from them, right?</a> <a data-t="989">Because the existing products, like, they're not all bad.</a> <a data-t="991">They've done some good jobs, right?</a> <a data-t="993">So it's important to, like,</a> <a data-t="995">to really, like, try to take their perspective</a> <a data-t="998">to understand, like, where they're coming from,</a> <a data-t="1000">why does this system do things the way it does,</a> <a data-t="1002">and how can we kind of together make it better?</a></p>

<p><a data-t="1006">And then more often than not, this is you, right?</a> <a data-t="1008">Like, you're looking at your own projects</a> <a data-t="1010">from two years ago, paying back technical debt,</a> <a data-t="1013">and you kind of don't want to look at it</a> <a data-t="1015">because you feel bad about yourself,</a> <a data-t="1017">and that's unpleasant, so you never fix the tests, right?</a> <a data-t="1020">Like, you know, so having this, like,</a> <a data-t="1023">I'm kind of using this phrase to, like,</a> <a data-t="1025">talk about other people, but really a lot of the times</a> <a data-t="1027">it's you, right? Like, you're the person</a> <a data-t="1029">on the other side of this when you have some other person</a> <a data-t="1031">coming in and saying, like, this doesn't work,</a> <a data-t="1033">or you're looking at your old stuff and it doesn't work.</a> <a data-t="1035">And all this really applies in the same way, right?</a> <a data-t="1037">Like, you're trying to learn a new thing,</a> <a data-t="1039">and if you feel threatened and afraid,</a> <a data-t="1041">you're not going to learn it, right?</a></p>
