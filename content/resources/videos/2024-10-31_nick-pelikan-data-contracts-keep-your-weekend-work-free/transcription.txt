<h3>Expectations in a data contract</h3>

<p><a data-t="427">And those of you who are familiar with data contracts may be expecting this next one.</a> <a data-t="437">One of the core tenants of data contracts, of a successful data contract, is really well laid out expectations.</a> <a data-t="441">That's when you, the data consumer, and the data producer creating the data, come to an agreement on what form that data should take.</a></p>

<p><a data-t="452">So a couple of things that you should always have in a data contract, I propose, number one, a schema.</a> <a data-t="456">That's what most people think about.</a> <a data-t="456">The data schema is what form does the data take?</a> <a data-t="462">What columns are in the data?</a> <a data-t="462">What data types are those columns?</a> <a data-t="466">Valid values.</a> <a data-t="472">Really quickly, who was in Hadley's art and production workshop on Monday?</a> <a data-t="479">Hadley showed an example of a data input changing to a data output that had a temperature column that changed from Fahrenheit to Celsius.</a> <a data-t="484">That's a great example of some valid values that you should have in a data contract.</a> <a data-t="489">You should be expecting, you should have an expectation of what values are going to go into your output.</a></p>

<p><a data-t="496">SLAs.</a> <a data-t="496">This is super critical, especially for those of you creating very business-like business-critical pieces of information.</a> <a data-t="508">An SLA is around how often data changes.</a> <a data-t="508">What's the expected update frequency of data?</a> <a data-t="512">What's the expected lag time of that data?</a> <a data-t="512">It's really important to put that out because let's say you're building a Shiny app, you're expecting it to be consuming daily data, and that data changes the monthly.</a> <a data-t="527">The people actually looking at that Shiny app might be taken by surprise there.</a></p>

<p><a data-t="531">The interface of data, also super important.</a> <a data-t="531">Interface is where's the data stored?</a> <a data-t="532">Is it in the database?</a> <a data-t="532">Is it in S3 store?</a> <a data-t="532">How do I get it?</a> <a data-t="539">Any needs on query performance, that's super important.</a> <a data-t="542">If you're building a machine learning model, you need data quickly, that should be something you build out in your data contract.</a></p>

<p><a data-t="548">Compliance, also super important.</a> <a data-t="557">Most of us have worked in, most of us are probably working in industries that are regulated or soon to be regulated.</a> <a data-t="562">So does your data need to comply with any regulatory requirements?</a> <a data-t="566">Who here, show of hands, who's in healthcare?</a> <a data-t="569">HIPAA?</a> <a data-t="576">Who here's in finance?</a> <a data-t="576">PCI?</a> <a data-t="576">Who here works with any sort of customer data?</a> <a data-t="582">Most of us, GDPR.</a> <a data-t="582">Almost all data is going to have these days some compliance behind it.</a> <a data-t="583">So it's really important that you make sure that that is noted in your data contract, and you know what those compliance effects are going to be.</a> <a data-t="593">If data gets deleted because it needs to be deleted for compliance, that's something you should be keeping track of and know that that's something your outputs can deal with.</a></p>

<p><a data-t="601">And then probably most critically is interoperability.</a> <a data-t="606">One of the great things, one of the things to look out for as you're starting to build data contracts in your workflow is, can that data contract be consumed by people and machines?</a> <a data-t="621">So can your data inputs, can your data outputs consume the output of your data contract?</a> <a data-t="625">Can they tell if a data contract's failing?</a> <a data-t="629">That can help you create failure statements, that can help you create failure states that your users expect, and help you make sure that your outputs are always giving users a good experience.</a></p>
