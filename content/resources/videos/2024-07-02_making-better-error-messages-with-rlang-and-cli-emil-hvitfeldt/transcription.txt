<h3>Priorities when writing error-throwing functions</h3>

<p><a data-t="571">The way we write these functions, what we think about errors, is mostly like this.</a> <a data-t="578">The data comes</a> <a data-t="578">in at the top and we have a series of checks.</a> <a data-t="584">It can be one, it can be more than one, and then at</a> <a data-t="584">the end, we do something.</a> <a data-t="593">We saw a little bit before, this is the same as this, where if we</a> <a data-t="593">have a lot of checks, we can pull them out into a standalone checker, so we separate where everything</a> <a data-t="602">goes.</a> <a data-t="610">With that, we have some priorities of how we write these functions.</a> <a data-t="612">The first one is</a> <a data-t="612">that, ideally, we should throw the error as early as possible.</a> <a data-t="624">If we need to know it's</a> <a data-t="624">need to be positive, check it right away, instead of doing a bunch of things and then</a> <a data-t="628">deal with what's used.</a> <a data-t="637">That helps the user get back into the swing of things.</a> <a data-t="637">Maybe</a> <a data-t="637">you have a function that needs a value, but also needs to connect to a database.</a> <a data-t="642">If you know the value needs to be a string, check it's a string before you connect to the database,</a> <a data-t="649">because that takes some time, so you can get back as fast as possible.</a></p>

<p><a data-t="654">The check itself, which is what we have here inside the if statement, everything is an if</a> <a data-t="662">statement, and then if it's true, we throw the error.</a> <a data-t="671">The check inside the if should ideally</a> <a data-t="671">calculate as fast as possible.</a> <a data-t="678">It should just detect, are we wrong or not?
