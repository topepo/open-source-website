<h3>The case for upstream accessibility</h3>

<p><a data-t="103">And as you may have guessed from the title of my talk, I'm not here to talk about tampons.</a> <a data-t="108">So in recent years, there's been a growing awareness, understanding, and acceptance of</a> <a data-t="112">the importance of technology accessibility.</a> <a data-t="116">And as we all know, technology doesn't just create itself.</a> <a data-t="119">Data doesn't analyze itself, or at least not yet.</a></p>

<p><a data-t="123">So we have this process of an initial developer, contributor, maintainer, who supports the</a> <a data-t="128">creation of a tool or who is doing the data science themselves.</a> <a data-t="132">So they then produce an output, whether that's an app or a dashboard, a piece of software,</a> <a data-t="139">and then that subsequently ends in the hands of a stakeholder or end user.</a> <a data-t="144">And in most of this dialogue so far, we tend to focus at the end on that end user.</a> <a data-t="149">And we generally do this by focusing in on the output stage.</a></p>

<p><a data-t="152">We do this by asking ourselves questions like, can someone who is colorblind distinguish</a> <a data-t="156">the features of this graph?</a> <a data-t="158">Or have I included sufficient alt text for someone who requires nonvisual access to this</a> <a data-t="163">information?</a> <a data-t="165">But I'm here to insist that this is the wrong place for us to be focusing if our goal is</a> <a data-t="169">to avoid being like that all-male engineering team at NASA.</a></p>

<p><a data-t="174">Instead of focusing primarily on the experience of the end user, we need to look further upstream</a> <a data-t="179">and divert our attention toward the experience of the programmer or data scientist.</a> <a data-t="183">We need to work to reduce unnecessary barriers for disabled coders and continue to support</a> <a data-t="189">their ability to continue to use and contribute to open source projects.</a></p>

<p><a data-t="196">And so to do this, we need to think about the tools, packages, languages, programs,</a> <a data-t="200">et cetera, that data scientists and developers depend on in their daily work.</a> <a data-t="204">We need to figure out how we can improve the accessibility of these tools so that we can</a> <a data-t="208">ensure fewer structural barriers to the continued participation and flourishing of disabled</a> <a data-t="213">members of the open source community.</a></p>

<p><a data-t="216">In the remainder of my talk, I'm going to discuss some simple steps that can be taken</a> <a data-t="219">to begin improving the accessibility of open source projects.</a> <a data-t="224">Importantly, these are steps that can be taken by absolutely anyone and don't require having</a> <a data-t="229">any depth of knowledge in accessibility or web accessibility.</a> <a data-t="234">As a result of that, I'm not going to get into any of the particularly technical aspects</a> <a data-t="238">of technology accessibility.</a></p>

<h3>The power tool metaphor</h3>

<p><a data-t="255">So imagine that you are renovating your kitchen and everything's going perfectly until you</a> <a data-t="259">come across some issue.</a> <a data-t="260">So you do some Googling, you try to figure out what's going on, and eventually you think</a> <a data-t="265">you figured it out and it seems like you can solve it yourself if you just get your hands</a> <a data-t="269">on a specific power tool.</a> <a data-t="271">So you go online and spend some significant effort researching and selecting the right</a> <a data-t="275">tool, making sure it has specific set of features and capabilities that you need.</a></p>

<p><a data-t="279">You order it, and unfortunately, once it arrives, it has plug type F, which is used commonly</a> <a data-t="285">throughout much of Europe, but you're located in the U.S., and therefore your outlets only</a> <a data-t="290">accommodate type A or B plugs.</a> <a data-t="293">It would have been great if this information was included in the product description, but</a> <a data-t="296">it wasn't.</a> <a data-t="297">In fact, it's not present in the description of almost any tool that you can find online.</a> <a data-t="302">So your best bet is to just keep buying new ones and trying them to see if they work.</a></p>

<p><a data-t="306">If you're thinking that this sounds like it would be an incredibly frustrating way to</a> <a data-t="309">get something done, you're right, and it's also the reality for many disabled data scientists</a> <a data-t="315">and programmers when it comes to trying out open source tools in order to solve their</a> <a data-t="319">programming problems.</a> <a data-t="320">In this metaphor, we should be thinking of the power tool as a package or IDE or tool,</a> <a data-t="326">and the plug outlet compatibility is accessibility, right?</a> <a data-t="330">So it's the IDE that someone wants to work in, and the outlet is their screen reader.</a> <a data-t="335">And if you don't in a lot of instances, they have no way of knowing whether or not the</a> <a data-t="339">two will work together until after they've downloaded it and tried it.</a> <a data-t="344">And unlike an incompatible plug, you can't just go on Amazon and buy a travel adapter</a> <a data-t="348">and just circumvent the whole thing with accessibility.</a></p>

<h3>Creating accessibility docs</h3>

<p><a data-t="352">So this brings me to the first thing we can do, which is create accessibility docs.</a> <a data-t="356">What I mean when I say this is creating documentation that helps individuals quickly assess whether</a> <a data-t="361">something meets not only their technical requirements, but crucially also their accessibility requirements.</a> <a data-t="369">This can look like a lot of different things, like just a high-level status of the accessibility</a> <a data-t="373">of the tool.</a></p>
