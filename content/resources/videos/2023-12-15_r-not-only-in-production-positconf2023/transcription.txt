<h3>R at InsightRX: precision medicine</h3>

<p><a data-t="241">So I work at a company called InsightRx.</a> <a data-t="243">And InsightRx, I think, is fairly unique in how pervasive R is to all of what we do.</a> <a data-t="249">So what I'd like to share with you today are the ways that we are doing this.</a> <a data-t="253">How we're using R not only in production,</a> <a data-t="256">but as a first-class language that is essential to fulfilling our mission.</a> <a data-t="261">And to illustrate how we're doing that, I'd like to introduce you to Angela.</a></p>

<p><a data-t="266">So Angela is a 60-year-old patient with a prosthetic heart valve.</a> <a data-t="270">And she comes into her doctor's office one day</a> <a data-t="273">complaining of having fevers kind of off and on over the last week.</a> <a data-t="277">She's just not been feeling very well.</a> <a data-t="280">And her doctor draws some blood samples and sends them off to the lab.</a> <a data-t="284">But in the meantime, Angela is getting sicker and sicker.</a> <a data-t="288">And she gets admitted to the hospital,</a> <a data-t="290">where it turns out that she has contracted an infection called MRSA.</a></p>

<p><a data-t="295">So if you've heard of MRSA before,</a> <a data-t="297">it has likely been in these kind of like hushed, scary tones.</a> <a data-t="301">And that is because MRSA is very difficult to treat</a> <a data-t="305">because it's very resistant to many antibiotics.</a> <a data-t="308">And so it carries with it this specter of antibiotic-resistant super bug scariness.</a> <a data-t="315">But there are antibiotics that can treat MRSA.</a> <a data-t="318">And one of the ones that is used is called vancomycin.</a></p>

<p><a data-t="322">So vancomycin can treat MRSA.</a> <a data-t="324">But vancomycin is very difficult to prescribe.</a> <a data-t="328">It's very difficult to get the right dose for a patient.</a> <a data-t="332">And that is because with vancomycin,</a> <a data-t="335">there is a very narrow window between what is enough to treat the infection,</a> <a data-t="340">but not so much as to cause negative side effects,</a> <a data-t="343">and particularly kidney damage.</a> <a data-t="345">So you need to give enough that the infection is treated.</a> <a data-t="350">But if you go too high, the patient can experience kidney damage.</a></p>

<p><a data-t="354">And this is in contrast to some other antibiotics,</a> <a data-t="358">like let's say an amoxicillin,</a> <a data-t="360">where you might have a lot more leeway in how much you can prescribe.</a> <a data-t="364">For vancomycin, that window is very narrow.</a> <a data-t="367">And not only is it very narrow, it's different for every person.</a> <a data-t="371">When you think about all of the diversity in people who might receive antibiotics,</a> <a data-t="376">which is most people,</a> <a data-t="379">there's so much variation in physiological characteristics</a> <a data-t="383">and all sorts of factors that can affect</a> <a data-t="386">how that drug is going to behave in a particular person's body.</a> <a data-t="390">And so there's no one dose that is going to work for everyone.</a></p>

<p><a data-t="403">So what we do at InsightRx is try to find the exact right dosage</a> <a data-t="407">of medications like vancomycin for every individual patient.</a> <a data-t="412">And we do this by combining peer-reviewed, published pharmacological models</a> <a data-t="416">from the literature, along with the patient's unique characteristics</a> <a data-t="420">and their data, their lab results,</a> <a data-t="422">to find the dose that is going to thread that needle</a> <a data-t="425">and fit right within that window of enough to treat</a> <a data-t="428">whatever condition they're facing,</a> <a data-t="430">but not so much as to cause negative side effects.</a></p>

<p><a data-t="437">And so to do this, we have a web application</a> <a data-t="440">that can access a lot of the patient's data,</a> <a data-t="443">either having it directly inputted by their clinician</a> <a data-t="446">or brought in through their electronic health records.</a> <a data-t="450">And so we get their height and weight, their lab results,</a> <a data-t="454">past doses that they've received.</a> <a data-t="456">And we send all of that data to a REST API that's written in R</a> <a data-t="461">that can wrangle that data, apply these models,</a> <a data-t="465">and identify a dose that is going to get that patient</a> <a data-t="469">to their target level of drug exposure.</a> <a data-t="471">So the goal level amount of drug in their system</a> <a data-t="476">to treat the condition that they're facing.</a></p>

<p><a data-t="480">And our REST API that's written in R</a> <a data-t="482">is going to send that dose, along with various other information</a> <a data-t="487">and plots and so forth, back to the web application,</a> <a data-t="490">where it is then displayed to the patient's clinician,</a> <a data-t="493">and they can use that in combination with their own expert judgment</a> <a data-t="497">to figure out how best to treat their patients.</a></p>

<p><a data-t="501">So R is core to this product,</a> <a data-t="503">because the entire point of its existence is to recommend these doses,</a> <a data-t="508">and R is handling these most important calculations to do that.</a> <a data-t="513">And so because R is core to the product,</a> <a data-t="516">that makes R core to fulfilling our mission,</a> <a data-t="519">which goes beyond treating individual patients,</a> <a data-t="522">but really towards elevating therapies for all patients.</a></p>

<h3>R as a first-class language</h3>

<p><a data-t="784">What is different here is the attitude of R</a> <a data-t="786">as a first-class language</a> <a data-t="788">and the idea that if we have a new problem,</a> <a data-t="791">we can and are willing to solve it in R</a> <a data-t="794">just as easily as we might in any other language.</a> <a data-t="798">And I think that that mindset requires us</a> <a data-t="800">to approach our work in a fundamentally collaborative way</a> <a data-t="803">because all of our systems that I showed in that graph</a> <a data-t="807">have components that are written in R</a> <a data-t="809">and written in other languages</a> <a data-t="811">that need to work together.</a></p>

<p><a data-t="814">So when we're developing a new feature,</a> <a data-t="816">there might be business logic in our R API</a> <a data-t="820">as well as UI changes that are needed</a> <a data-t="822">to support that feature.</a> <a data-t="824">And so in order to make all of this work,</a> <a data-t="826">we have to work very closely.</a> <a data-t="828">We're following the same release cycles,</a> <a data-t="830">the same sprint cycles,</a> <a data-t="832">and adopting many shared practices</a> <a data-t="834">so that all of our pieces of this system</a> <a data-t="837">work together in concert.</a></p>

<p><a data-t="840">And so as a result,</a> <a data-t="841">because our work is so tied together,</a> <a data-t="843">when we have a new problem,</a> <a data-t="845">we can be pretty open-minded about</a> <a data-t="846">who solves that problem and how we solve it.</a> <a data-t="849">And I think that that leads to better solutions overall.</a> <a data-t="852">And so I'll give an example.</a></p>

<p><a data-t="855">Our Nova application is used in multiple different countries</a> <a data-t="859">in the U.S. as well as in Europe.</a> <a data-t="861">And so it needs to operate in multiple different languages.</a> <a data-t="865">And the way that we manage this,</a> <a data-t="867">which is the way that many web applications do,</a> <a data-t="869">is through localization files.</a> <a data-t="872">So localization files list out</a> <a data-t="874">all of the pieces of text that might appear on a page.</a> <a data-t="878">And when the app is running in the U.S.,</a> <a data-t="880">it goes into the English version of the file</a> <a data-t="882">and pulls out the text that it needs to display</a> <a data-t="885">and puts it on the page.</a> <a data-t="886">So at the Change My Password button,</a> <a data-t="888">it's going to extract that text and put it onto the page.</a> <a data-t="891">And if the app is running in a German-speaking country,</a> <a data-t="894">then it will go to the German version of the file</a> <a data-t="896">and pull out that text and display it on the page.</a></p>

<p><a data-t="899">And before every release,</a> <a data-t="901">these files are professionally reviewed</a> <a data-t="904">by fluent speakers, humans.</a> <a data-t="907">But in between releases,</a> <a data-t="909">we need a way to keep all of these localization files up to date.</a> <a data-t="913">And the way that we used to do this</a> <a data-t="915">was that anytime an engineer made a change</a> <a data-t="918">that affected some text,</a> <a data-t="919">they would edit the English version of the file,</a> <a data-t="921">and then they'd copy their new text into Google Translate</a> <a data-t="924">and translate it into all these different languages,</a> <a data-t="926">and then copy and paste it back into the other language files.</a> <a data-t="930">And this was very cumbersome and tedious,</a> <a data-t="932">particularly if you're changing ten or more labels, right?</a> <a data-t="936">You have to translate all of these into multiple languages.</a></p>

<p><a data-t="940">And so my colleague and I decided</a> <a data-t="942">that we could write an R package that could make this easier.</a> <a data-t="946">So our R package</a> <a data-t="949">will look at the English version of the file</a> <a data-t="951">and detect lines that have changed,</a> <a data-t="954">text that has changed.</a> <a data-t="955">And it's going to send each of those updated labels</a> <a data-t="958">to a translation API</a> <a data-t="960">to translate it into the multiple languages.</a> <a data-t="963">And then it can go to the corresponding locale files</a> <a data-t="967">and insert that text</a> <a data-t="969">exactly in the right location where it needs to be.</a></p>

<p><a data-t="973">And so now an engineer doesn't need to</a> <a data-t="975">manually run all these translations.</a> <a data-t="978">They can edit the English version of the file</a> <a data-t="980">and have the rest of the translations</a> <a data-t="982">automatically generated for them.</a> <a data-t="985">And again, before release, these are all reviewed.</a> <a data-t="987">But this speeds up the development process considerably</a> <a data-t="990">and really reduces friction.</a></p>

<p><a data-t="993">And I think that this story illustrates</a> <a data-t="995">how R can be brought in to solve problems</a> <a data-t="997">that don't have that much to do with data science, right?</a> <a data-t="1000">Because these are localization files</a> <a data-t="1002">for a web application that is not primarily written in R.</a> <a data-t="1005">It's not primarily developed by data scientists.</a> <a data-t="1008">But we saw this problem and this point of friction</a> <a data-t="1011">and because we had this mindset,</a> <a data-t="1013">we realized that there was no reason</a> <a data-t="1015">that we couldn't use R to solve it.</a> <a data-t="1017">And in fact, we solved it very quickly and easily using R.</a></p>

<p><a data-t="1023">So our mindset of R as a first-class language</a> <a data-t="1026">empowers us to solve problems.</a> <a data-t="1028">We felt empowered to solve this problem</a> <a data-t="1031">even though it wasn't primarily a data science problem</a> <a data-t="1035">because we know that R is a first-class language at InsightRx.</a> <a data-t="1040">And sometimes the problems that we're solving</a> <a data-t="1042">may be relatively straightforward things,</a> <a data-t="1044">like making it easier to work with localization files.</a> <a data-t="1048">But they can also be bigger things</a> <a data-t="1050">that really affect patient care for patients like Angela.</a></p>

<h3>Building reliable features: testing and CI</h3>

<p><a data-t="1056">So let's check back in with Angela.</a> <a data-t="1059">Angela is in the hospital.</a> <a data-t="1061">She's receiving vancomycin to treat her MRSA infection.</a> <a data-t="1065">And as I mentioned, with MRSA and with vancomycin,</a> <a data-t="1069">if the dose is too high,</a> <a data-t="1071">it can cause damage to the patient's kidneys.</a> <a data-t="1075">And so her doctors are regularly collecting lab results</a> <a data-t="1079">to make sure that Angela's kidneys</a> <a data-t="1082">are functioning correctly, healthily.</a> <a data-t="1085">And we're using those lab results in our calculations</a> <a data-t="1089">to determine what dose to recommend for Angela.</a></p>

<p><a data-t="1093">But we realized that wouldn't it be nice</a> <a data-t="1095">if we could actually alert the clinicians</a> <a data-t="1098">in the event that Angela's kidneys</a> <a data-t="1100">are starting to show signs of acute kidney injury.</a> <a data-t="1104">So we have this lab data in the web application,</a> <a data-t="1107">and we could raise some warning in the UI</a> <a data-t="1111">to let the clinician know,</a> <a data-t="1113">hey, you know, there may be a problem here.</a> <a data-t="1116">Maybe they need to adjust treatment or take some action.</a></p>

<p><a data-t="1121">So because we have this mindset of R as a first-class language,</a> <a data-t="1124">we know that we can do this in R.</a> <a data-t="1127">So we start with this idea for this new feature</a> <a data-t="1129">and the mindset that we can build it in R.</a> <a data-t="1132">And then maybe we write some code</a> <a data-t="1134">to detect acute kidney injury</a> <a data-t="1136">and get something that works on our machine.</a> <a data-t="1141">But then to go from that to an actual feature</a> <a data-t="1144">and a final product takes a little bit more work.</a> <a data-t="1147">And how do we get from something that works on our machine</a> <a data-t="1150">to that final product?</a></p>

<p><a data-t="1154">We have had to invest in certain practices</a> <a data-t="1157">in that space between the prototype and the final product</a> <a data-t="1161">to make sure that our work is reliable and deployable.</a> <a data-t="1165">So the reliability is important for many reasons,</a> <a data-t="1168">the most obvious of which being that</a> <a data-t="1170">there are real human patients</a> <a data-t="1172">whose health is directly affected by what we're doing.</a> <a data-t="1175">And the deployability is important</a> <a data-t="1177">because we need a way to make sure that our code</a> <a data-t="1180">and everything that it needs to run</a> <a data-t="1182">can get to wherever it's going to run.</a></p>

<p><a data-t="1200">So when we're thinking about reliability</a> <a data-t="1207">and how to detect kidney injury reliably,</a> <a data-t="1209">we need to make sure that we're detecting it accurately</a> <a data-t="1212">all of the time.</a> <a data-t="1214">And the way to detect acute kidney injury, or AKI,</a> <a data-t="1218">is relatively straightforward.</a> <a data-t="1220">So in everyone's blood,</a> <a data-t="1222">there's a waste product called creatinine.</a> <a data-t="1225">And a healthy, functioning kidney</a> <a data-t="1227">is going to filter out creatinine over time.</a> <a data-t="1230">So ideally, my body's producing creatinine,</a> <a data-t="1233">my kidneys are filtering it out,</a> <a data-t="1234">and that level is staying relatively stable in my blood.</a> <a data-t="1238">But if my kidneys are not functioning well,</a> <a data-t="1241">then the level of creatinine is going to build up</a> <a data-t="1243">and build up in my blood.</a> <a data-t="1245">And so if that level goes up by a certain amount</a> <a data-t="1248">within a certain period of time,</a> <a data-t="1250">then that constitutes an acute kidney injury.</a></p>

<p><a data-t="1273">So elevated creatinine suggests kidney damage.</a> <a data-t="1276">This is relatively easy to calculate in theory,</a> <a data-t="1280">but in reality, things always get messy.</a> <a data-t="1280">You might have gaps in your data</a> <a data-t="1282">so that you can't really tell</a> <a data-t="1283">how quickly creatinine is changing.</a> <a data-t="1287">You might have data that comes in out of order</a> <a data-t="1289">that you need to sort before you can calculate.</a> <a data-t="1295">You might have data that's reported in different units</a> <a data-t="1297">that are totally different and need to be standardized.</a> <a data-t="1301">You might be missing the baseline creatinine values</a> <a data-t="1304">for a given patient.</a> <a data-t="1306">So you know what their creatinine is now,</a> <a data-t="1307">but if you don't know where they started,</a> <a data-t="1309">then how can you tell how much that's gone up?</a></p>

<p><a data-t="1313">You might have data that's not numeric.</a> <a data-t="1316">I think anyone who has worked with real-world data</a> <a data-t="1319">has experienced the phenomenon of data</a> <a data-t="1321">that really ought to be numbers</a> <a data-t="1323">coming in with other characters that you have to strip out</a> <a data-t="1326">before you can handle it appropriately.</a> <a data-t="1329">And there might be data entry errors,</a> <a data-t="1331">either from the electronic health record system</a> <a data-t="1333">or from manually inputting data.</a></p>

<p><a data-t="1337">And so when we're trying to build a reliable feature,</a> <a data-t="1340">we need to be able to handle all of these scenarios,</a> <a data-t="1343">both now and in the future as our code changes.</a> <a data-t="1346">And to do that, we add tests.</a> <a data-t="1349">So this is a test using the </a><a href="/software/testthat/">testthat</a><a data-t="1349"> package,</a> <a data-t="1352">which is in my top five most indispensable R packages</a> <a data-t="1357">that I cannot live without.</a> <a data-t="1359">And in our test, we're going to lay out</a> <a data-t="1362">the scenario that we want to capture</a> <a data-t="1363">to make sure that our code behaves the way that we expect.</a> <a data-t="1366">So this is a test where we say that we're testing</a> <a data-t="1369">that AKI stage is calculated correctly,</a> <a data-t="1372">and we're going to run our AKI stage calculation function</a> <a data-t="1377">with some sample data that we expect</a> <a data-t="1379">should produce a particular result.</a> <a data-t="1381">In this case, it's a stage one acute kidney injury.</a> <a data-t="1384">And so then in the last line of the test,</a> <a data-t="1387">we'll explicitly state that we expect</a> <a data-t="1389">that the result of our function</a> <a data-t="1392">is going to equal a stage one kidney injury.</a> <a data-t="1394">And if it doesn't, this test is going to fail.</a></p>

<p><a data-t="1397">So we write these tests for all sorts of scenarios</a> <a data-t="1400">that we want to capture, right?</a> <a data-t="1402">Different stages of AKIs, no AKIs,</a> <a data-t="1404">data that looks, you know, in different ways,</a> <a data-t="1407">as well as scenarios that we don't expect to encounter</a> <a data-t="1411">or hope we don't encounter,</a> <a data-t="1413">but would need to handle gracefully nonetheless,</a> <a data-t="1415">like out-of-order data, non-numeric inputs,</a> <a data-t="1418">missing observations.</a></p>

<p><a data-t="1421">And once we have this collection of tests,</a> <a data-t="1424">we are sort of getting at all of our code</a> <a data-t="1426">from many different angles to make sure</a> <a data-t="1428">that it is reliable and it's functioning correctly</a> <a data-t="1431">as long as we are running the tests regularly, right?</a> <a data-t="1435">Because if we're changing our code</a> <a data-t="1437">and not rerunning the test,</a> <a data-t="1438">then maybe we've changed some behavior</a> <a data-t="1440">and we have no idea.</a> <a data-t="1442">So we need to run these tests often.</a></p>

<p><a data-t="1445">And if I am disciplined, I can do that ten times a day.</a> <a data-t="1450">But I'm busy.</a> <a data-t="1452">Sometimes I'm lazy.</a> <a data-t="1454">I have other things that I would like to do with my life</a> <a data-t="1456">other than run these tests all the time.</a> <a data-t="1459">And the reality is that</a> <a data-t="1461">if we're relying on me to run the tests,</a> <a data-t="1463">they're not going to run often enough.</a> <a href="/people/jenny-bryan/">Jenny Bryan</a><a data-t="1466"> once said that</a> <a data-t="1468">if you use software that lacks automated tests,</a> <a data-t="1470">you are the tests.</a> <a data-t="1472">Well, if we build software that lacks automated tests,</a> <a data-t="1475">then that means our users are the tests.</a> <a data-t="1478">And by extension, they're patients.</a> <a data-t="1480">And so that is not acceptable at all.</a> <a data-t="1483">And this is where continuous integration comes in for us.</a></p>

<blockquote><a data-t="1466">Jenny Bryan once said that</a> <a data-t="1468">if you use software that lacks automated tests,</a> <a data-t="1470">you are the tests.</a> <a data-t="1472">Well, if we build software that lacks automated tests,</a> <a data-t="1475">then that means our users are the tests.</a> <a data-t="1478">And by extension, they're patients.</a></blockquote>

<p><a data-t="1487">Continuous integration allows us</a> <a data-t="1491">to automatically run these tests</a> <a data-t="1493">so that we are constantly making sure</a> <a data-t="1495">that our code is behaving as intended.</a> <a data-t="1497">So every time we push new code to GitHub,</a> <a data-t="1500">we kick off a pipeline that gets that latest code,</a> <a data-t="1503">it builds the R package that it contains,</a> <a data-t="1506">and it runs the tests.</a> <a data-t="1508">And then it's going to send its results back to GitHub</a> <a data-t="1510">in a way that we can clearly see.</a></p>

<p><a data-t="1512">So if we were to look at the git history of our project,</a> <a data-t="1515">we might see something like this.</a> <a data-t="1517">We have a commit where we add our function to detect AKIs,</a> <a data-t="1521">and we write our tests along with that</a> <a data-t="1524">so all of those tests are passing.</a> <a data-t="1526">And we get this nice green checkmark,</a> <a data-t="1528">which is very satisfying to me.</a> <a data-t="1531">And then maybe we realize that</a> <a data-t="1534">the performance of our function is lacking.</a> <a data-t="1536">We need to refactor it in some way to make it faster.</a> <a data-t="1538">And so we rewrite it,</a> <a data-t="1540">and we think that we've preserved its previous behavior,</a> <a data-t="1543">but actually we have changed how it handles some edge case</a> <a data-t="1547">like how it deals with missing data.</a> <a data-t="1549">And so tests that used to pass start to fail.</a> <a data-t="1552">And our continuous integration pipeline</a> <a data-t="1554">is going to send this angry red X back to GitHub,</a> <a data-t="1559">which I find incredibly unsettling</a> <a data-t="1561">and impossible to ignore.</a></p>

<p><a data-t="1564">So because I'm alerted in this way</a> <a data-t="1567">that is very, like, pings me right in the brain,</a> <a data-t="1570">I can go in and fix the way we're handling</a> <a data-t="1573">missing data in our function</a> <a data-t="1575">so that the tests pass again</a> <a data-t="1577">and I get my nice green checkmark back.</a></p>

<p><a data-t="1584">So continuous integration allows us to find bugs quickly</a> <a data-t="1587">because it's running automatically</a> <a data-t="1589">as soon as we make changes to the code.</a> <a data-t="1591">It lets us free up time for more important work,</a> <a data-t="1594">and it makes sure that our code continues to behave</a> <a data-t="1596">as we intended.</a> <a data-t="1599">So testing and continuous integration together</a> <a data-t="1601">are two of the practices that are most important for us</a> <a data-t="1604">to ensure the reliability of our work.</a></p>

<h3>Deployability with Docker</h3>

<p><a data-t="1607">And then when it comes to deployability,</a> <a data-t="1609">we need a way to take our code</a> <a data-t="1611">and all of the things it needs to run</a> <a data-t="1613">and get it to some place</a> <a data-t="1615">where it can run off of our laptops.</a> <a data-t="1617">And there are many different ways to do this.</a> <a data-t="1619">If you have access to Posit Connect,</a> <a data-t="1621">that can make it super easy.</a> <a data-t="1623">In our case, we use Docker heavily</a> <a data-t="1626">to be able to deploy our applications.</a></p>

<p><a data-t="1630">So Docker allows you to package up</a> <a data-t="1633">your code and everything it needs to run</a> <a data-t="1635">into a self-contained little bundle called an image.</a> <a data-t="1638">So R itself, any R packages you need,</a> <a data-t="1641">your code, any files your code needs,</a> <a data-t="1644">all get wrapped up into this nice little bundle</a> <a data-t="1647">that can then be sent off to wherever it needs to run.</a></p>

<p><a data-t="1656">And we define these Docker images using a Dockerfile,</a> <a data-t="1659">and so I'll give a brief example of how this might look.</a> <a data-t="1664">In our first line of our Dockerfile,</a> <a data-t="1666">we're going to say that we're going to start</a> <a data-t="1668">with an existing Docker image</a> <a data-t="1670">that already comes with R and Plumber installed.</a> <a data-t="1672">The </a><a href="/software/rstudio/">RStudio</a><a data-t="1672"> Plumber, the 1.1.0 image,</a> <a data-t="1675">comes with R and Plumber.</a> <a data-t="1678">And let's say we're building a Docker image</a> <a data-t="1680">that's going to run a Plumber API,</a> <a data-t="1682">so we need that package installed.</a> <a data-t="1684">So we're going to take this existing image,</a> <a data-t="1686">and then we're going to add to it whatever else we need.</a> <a data-t="1689">So maybe we'll add another package,</a> <a data-t="1691">like this open source package called clinpk that we maintain.</a></p>

<p><a data-t="1697">And after we install our packages,</a> <a data-t="1699">we're going to copy over our source code</a> <a data-t="1701">from our project into the Docker image.</a> <a data-t="1703">So that's what this copy line is doing.</a> <a data-t="1705">It's copying over our code,</a> <a data-t="1707">and then we're going to set our working directory</a> <a data-t="1709">to where we've copied that code,</a> <a data-t="1711">so we have all of our files nicely around us.</a> <a data-t="1713">And then in the last line of this Dockerfile,</a> <a data-t="1716">we say that when we start up a Docker container</a> <a data-t="1719">based on this image,</a> <a data-t="1721">we're going to source this entrypoint.r file,</a> <a data-t="1723">which is going to run our Plumber API.</a></p>

<p><a data-t="1727">I've put this example and a couple of others up on GitHub,</a> <a data-t="1730">so if you feel like playing around</a> <a data-t="1732">with a couple of relatively simple Docker images,</a> <a data-t="1736">Dockerfiles, you can check those out.</a> <a data-t="1738">But the important thing about this</a> <a data-t="1741">is that someone could build and run this Docker image</a> <a data-t="1744">without ever installing R on their machine.</a> <a data-t="1747">You don't need to have R.</a> <a data-t="1749">You don't need to have Plumber.</a> <a data-t="1751">You can build all of this and run this project</a> <a data-t="1753">without installing R yourself on your machine.</a> <a data-t="1756">And so that makes Docker a really useful interface</a> <a data-t="1759">between our data science team and other teams</a> <a data-t="1762">because they don't need to know anything about R</a> <a data-t="1766">to be able to build an R project,</a> <a data-t="1768">and I don't need to know all of the tools</a> <a data-t="1771">and background software for other languages</a> <a data-t="1774">to be able to run those projects.</a></p>

<p><a data-t="1777">And so when it comes to deploying our application,</a> <a data-t="1779">I go to our DevOps engineer</a> <a data-t="1781">who maintains our Kubernetes cluster,</a> <a data-t="1783">and I don't know anything about</a> <a data-t="1785">maintaining a Kubernetes cluster,</a> <a data-t="1787">and he doesn't need to know about</a> <a data-t="1789">what it takes to run an R project.</a> <a data-t="1791">If I can share this working Docker image,</a> <a data-t="1793">then that can be that point of connection</a> <a data-t="1795">and collaboration for us,</a> <a data-t="1797">and that has worked really quite well for us.</a></p>

<p><a data-t="1801">Docker is also actually an important component</a> <a data-t="1804">of how we maintain the reliability of our code as well.</a> <a data-t="1807">So I mentioned that our continuous integration pipeline</a> <a data-t="1810">gets the latest code,</a> <a data-t="1812">builds the R package, and runs the tests.</a> <a data-t="1814">But what it's actually doing is building the Docker image</a> <a data-t="1817">that contains the R package</a> <a data-t="1819">and running the tests in there.</a> <a data-t="1821">And this distinction is important</a> <a data-t="1822">because the Docker image that it builds</a> <a data-t="1824">is the exact one that we are going to run in production.</a> <a data-t="1826">It's exactly the same.</a> <a data-t="1828">And so we can eliminate any errors</a> <a data-t="1832">due to package version mismatches</a> <a data-t="1835">or subtle things that are different</a> <a data-t="1838">in one environment compared to another.</a> <a data-t="1840">We are testing the exact versions of everything</a> <a data-t="1842">that's going to be deployed in production,</a> <a data-t="1844">and that gives us greater confidence in that reliability.</a></p>

<p><a data-t="1849">So we've built our feature. We've tested it.</a> <a data-t="1851">Those tests are running automatically.</a> <a data-t="1853">Now we have our Docker image that contains our feature,</a> <a data-t="1856">and when it comes time to deploy the final product,</a> <a data-t="1859">we'll go to wherever we're hosting it</a> <a data-t="1861">and swap out the previous version of our Docker image</a> <a data-t="1864">with the new one that we've created.</a> <a data-t="1866">And that gets us to the final product.</a> <a data-t="1869">So we started with this idea</a> <a data-t="1871">that we could build this feature and build it in R.</a> <a data-t="1874">We wrote it.</a> <a data-t="1876">We adopted these practices</a> <a data-t="1877">to make sure that it was reliable and deployable.</a> <a data-t="1880">And now not only Angela,</a> <a data-t="1882">but all patients can be monitored on an ongoing basis</a> <a data-t="1885">and their medical teams alerted</a> <a data-t="1887">if they are experiencing acute kidney injury.</a></p>

<h3>What data scientists bring to software development</h3>

<p><a data-t="1895">So I've shared some of the more technical aspects</a> <a data-t="1898">of how we make our work in production for us.</a> <a data-t="1901">And I am aware that it might seem like a little bit of a lot.</a> <a data-t="1905">But for us, these practices are crucial</a> <a data-t="1907">to make sure that we can run our in production</a> <a data-t="1912">and have it be a functioning,</a> <a data-t="1914">first-class system within our organization.</a> <a data-t="1918">And adopting these practices</a> <a data-t="1920">also gives us as data scientists on our team</a> <a data-t="1923">a greater connection to the rest of the organization</a> <a data-t="1926">because we are all following a shared set of practices</a> <a data-t="1930">and working together in a common way.</a></p>

<p><a data-t="1934">And so we can bring our own skills and practices</a> <a data-t="1938">that are useful to us as data scientists</a> <a data-t="1940">to that equation as well.</a> <a data-t="1942">It's not just in the dashboards that we create,</a> <a data-t="1945">the visualizations and the reports that we build</a> <a data-t="1948">that we bring value to our organizations.</a> <a data-t="1952">We as data scientists can bring our skills</a> <a data-t="1955">to the craft and the process of building software as well.</a> <a data-t="1959">And that is something that I think a lot of other people</a> <a data-t="1961">in our organizations can learn a lot from.</a></p>

<p><a data-t="1970">The first one is that we as data scientists</a> <a data-t="1972">are motivated to answer questions.</a> <a data-t="1974">I think that this is something that unites most data scientists.</a> <a data-t="1978">And it's something that we can apply</a> <a data-t="1980">to our programming conundrums</a> <a data-t="1982">just as much as to modeling customer churn</a> <a data-t="1985">or whatever it is that we're doing.</a> <a data-t="1989">Debugging your code is essentially an exercise</a> <a data-t="1991">in testing a hypothesis</a> <a data-t="1993">where you observe some phenomenon</a> <a data-t="1995">where code that you have written</a> <a data-t="1997">is not behaving the way you intended</a> <a data-t="1999">or is showing some undesirable behavior.</a> <a data-t="2002">And you need to develop some kind of hypothesis</a> <a data-t="2005">about why that is the case,</a> <a data-t="2007">gather some data that you can use in some way to analyze</a> <a data-t="2011">and test that hypothesis,</a> <a data-t="2013">and maybe conduct some experiment.</a></p>

<p><a data-t="2015">So I'll give an example of a time where we did exactly that.</a> <a data-t="2020">In one of our applications that the engineering team maintains,</a> <a data-t="2023">we had a database query that was too slow.</a> <a data-t="2026">It was taking too long to return data to a plot.</a> <a data-t="2029">Customers were complaining.</a> <a data-t="2032">And so the idea was to add an index to the database</a> <a data-t="2036">to improve the performance of this query.</a> <a data-t="2041">So we added this index.</a> <a data-t="2044">And then the next day, our batch processing pipeline</a> <a data-t="2047">that uses R to calculate all these metrics</a> <a data-t="2050">that we show to our hospitals in our Apollo product,</a> <a data-t="2054">that data processing pipeline was too slow to run.</a> <a data-t="2058">And it was assumed that because we had added this index,</a> <a data-t="2061">that was slowing down the pipeline.</a></p>

<p><a data-t="2063">And in theory, this is a plausible explanation</a> <a data-t="2065">because when you add an index to a table,</a> <a data-t="2068">typically it can make it faster to read data,</a> <a data-t="2071">but it can make it slower to write data to that table.</a> <a data-t="2074">So we rolled back the change. We removed the index.</a> <a data-t="2077">And the next day, our processing pipeline was fast again.</a> <a data-t="2082">But the day after that, it was kind of slow again.</a> <a data-t="2084">So weird. I don't know. Whatever.</a></p>

<p><a data-t="2088">A few months later, we had another issue of a database query</a> <a data-t="2091">in a different part of the app that was too slow.</a> <a data-t="2094">And again, someone proposed adding an index to the database.</a> <a data-t="2097">But this time, the engineer who had to roll back that change last time</a> <a data-t="2101">said, well, last time it slowed down this other thing.</a> <a data-t="2105">I don't think that we can add an index here.</a> <a data-t="2109">So now we're stuck in a situation where</a> <a data-t="2112">our users are suffering slow queries,</a> <a data-t="2114">but we are wary of making any changes that would improve them.</a></p>

<p><a data-t="2118">And I was not very happy with this scenario</a> <a data-t="2122">because I feel like whenever you're in a situation where you can't make changes</a> <a data-t="2126">because you're afraid of breaking something,</a> <a data-t="2128">that thing is already incredibly broken.</a> <a data-t="2131">And we, between our two teams, have control over this whole system,</a> <a data-t="2136">so there's no reason that we couldn't actually figure out</a> <a data-t="2139">whether the index was the cause of the slowness or not.</a> <a data-t="2143">No one had actually confirmed that.</a> <a data-t="2145">They just rolled back the change.</a></p>

<p><a data-t="2148">So together with a colleague of mine, we ran a little experiment.</a> <a data-t="2152">We have a staging database where we can test out changes</a> <a data-t="2155">before we make them to production.</a> <a data-t="2157">So on that staging database, I ran our data processing pipeline,</a> <a data-t="2162">and I timed it.</a> <a data-t="2164">And then we added the indexes that we wanted to add.</a> <a data-t="2167">And then I ran that same pipeline again and timed it again.</a> <a data-t="2171">And I think that they were within milliseconds of each other.</a> <a data-t="2173">There was absolutely no difference in how long it took to run this pipeline</a> <a data-t="2177">with and without the indexes.</a> <a data-t="2179">And so there was no reason that we couldn't add those indexes.</a></p>
