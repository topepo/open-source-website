<h3>Web API basics</h3>

<p><a data-t="58">So API basics. There are a lot of data accessible over the Internet these days, and a lot of it is made available via</a> <a data-t="63">APIs, which are application programming interfaces.</a> <a data-t="69">These allow programs to talk to each other based on a communication contract.</a> <a data-t="75">If you ask me a question in a way I understand, I will give you an answer.</a> <a data-t="81">So there are two important parts to HTTP communication over the Web.</a> <a data-t="86">There's the request, which is data sent to the server from the client, and then</a> <a data-t="91">the response, which is the data that the server sends back to that client.</a></p>

<p><a data-t="91">Now, this is a very oversimplified view of the client-server communication, but the basics of it are ask</a> <a data-t="97">a question and get a response, and this is predicated on the fact that the client and</a> <a data-t="101">server need to understand each other's language, so to speak.</a> <a data-t="108">The client can only send a request to the server that the server knows how to handle, and then the server will send that</a> <a data-t="112">response back to the client, and the client will need to know how to parse that response</a> <a data-t="115">to get the data out based on the question that was asked.</a></p>

<p><a data-t="122">Web APIs usually provide read and or write access to data stores when you're talking about Web APIs, and they allow you</a> <a data-t="127">to access data on an ongoing basis.</a> <a data-t="133">This is significant since it allows you to write scripts that pull an API to get the most up-to-date data.</a> <a data-t="140">Not only will your analyses be current,</a> <a data-t="145">but when the data changes, your scripts don't have to.</a> <a data-t="149">Depending on how you design that script, data frames and plots will update automatically when your data changes without</a> <a data-t="155">you having to download a new CSV or import data manually.</a></p>

<blockquote><a data-t="133">This is significant since it allows you to write scripts that pull an API to get the most up-to-date data.</a> <a data-t="140">Not only will your analyses be current,</a> <a data-t="145">but when the data changes, your scripts don't have to.</a></blockquote>

<p><a data-t="156">So as an example of this request-response, I've put down here a curl request.</a> <a data-t="163">Curl is a utility in the Unix world that allows you to make HTTP requests.</a> <a data-t="170">In this case, I'm calling the OMDB API. This is the Online Movie Database API,</a> <a data-t="178">and I'm giving it a request with a query parameter, T for title and R for response format,</a> <a data-t="184">so in this case, I'm saying, OMDB, give me the data for the title, the movie named Clue, and send it back to me in JSON format.</a> <a data-t="190">The server understands this request because you've put it into the format that it expects, and so it sends a response that</a> <a data-t="194">contains the data you asked for, the title, the year, the rating, et cetera.</a> <a data-t="199">Again, this is a very simplistic example, but it gives you an idea of sort of how simple it is to</a> <a data-t="205">ask a question and receive an answer and pull that data into the scripts that you're writing every day.</a></p>

<p><a data-t="219">Web APIs are organized into resources that are often in a particular hierarchy.</a> <a data-t="224">In this example, there's a map that if you follow down from the top, you can get information</a> <a data-t="231">about accounts, a particular account by its ID, the lists that a particular account has,</a> <a data-t="232">the ID of those lists, the campaigns, the subscribers, the web forms, et cetera.</a> <a data-t="236">So these are often organized in maps or webs of information, and if you know how to get</a> <a data-t="241">to those different levels of information, you can construct a URL and ask for very specific</a> <a data-t="246">things all the way down at the bottom, like clicks and tracked events in this case.</a></p>

<p><a data-t="252">Now that presumes that you understand a little bit about the API that you're trying to use.</a> <a data-t="257">Fortunately, most of it has many of them, I should say most of them, have decent documentation.</a> <a data-t="263">This example is from the Star Wars API.</a> <a data-t="268">So in this case, you can see across the left the kinds of information you can learn from this documentation.</a> <a data-t="272">There's a getting started guide here on the first page.</a> <a data-t="278">But the things that you need to understand are the resources towards the bottom so that you know what you can get and how to get it.</a> <a data-t="283">For example, this being the Star Wars API, you can get information about people, films, starships, planets, vehicles, et cetera.</a> <a data-t="289">And the beauty of this documentation, this particular documentation is very good,</a> <a data-t="293">so it'll tell you exactly what URL to use to get to that information, and it'll give</a> <a data-t="298">you an example response as well.</a> <a data-t="303">So it will tell you how to ask the question and what format the answer will be in so that you can then parse it appropriately in your script.</a></p>

<h3>Tools for accessing APIs in R</h3>

<p><a data-t="319">So to access these APIs and pull this information into R, you have a couple of different options.</a> <a data-t="323">In some cases, people have already written packages that will wrap those API calls for a given service.</a> <a data-t="330">The examples I've provided here are for AWS S3. This is S3 buckets in the Amazon Web Services universe.</a> <a data-t="336">R Google Analytics will get you information about Google Analytics based on the ID code that you have for your website.</a> <a data-t="343">ACS is for the American Census information, et cetera.</a> <a data-t="347">So some people have already wrapped these APIs into packages that are very easy to use.</a> <a data-t="353">You do, I mean, if you can use them, by all means do. They make your life a lot easier.</a> <a data-t="356">You do need to be aware that the people writing them may not have the same goals as you do.</a> <a data-t="362">So they've written it to solve their problem and then made it public.</a> <a data-t="366">But if you have other questions that you need to ask or if there's particular end points that aren't covered in that package, you may need to write that yourself.</a> <a data-t="371">Or there could be no package for the data source that you're trying to reach, in which case you'll write it yourself with some other tools.</a></p>

<p><a data-t="401">I recommend </a><a href="/software/httr/">HTTR</a><a data-t="401"> or Hitter for making the requests.</a> <a data-t="407">And then JSON Lite or </a><a href="/software/xml2/">XML2</a><a data-t="407"> for parsing the response.</a> <a data-t="413">The most common response formats of API calls are JSON and XML, and these two packages are extremely good at parsing that data for you.</a> <a data-t="420">Once you have the data in your script, you can wrangle the data the way that you normally do with data that comes in using the various</a> <a data-t="425">packages in the </a><a href="/software/tidyverse/">tidyverse</a><a data-t="425"> to get your data in an orderly rectangle so that you can use</a> <a data-t="432">it for your analyses and for your plots.</a></p>

<p><a data-t="441">So Hitter request functions wrap HTTP verbs and fortunately they match them so they're very easy to figure out.</a> <a data-t="447">Get is the one you'll probably be using the most commonly if you're interested in getting information from the web into your scripts.</a> <a data-t="457">Post is to write back to a data store of some kind.</a> <a data-t="457">Patch and put are for updating.</a> <a data-t="462">Delete is fairly obvious.</a> <a data-t="462">And head is identical to get but without the body.</a> <a data-t="466">So in this case you'd just be getting the metadata around the request without getting the body data.</a> <a data-t="472">This can be useful as you're developing to understand sort of everything around the data itself in a much cheaper call.</a></p>

<p><a data-t="480">So to make a request, you first load Hitter with the library function, then call get with the URL.</a> <a data-t="485">In this case I am using the Star Wars API. I'm calling planets with ID one.</a> <a data-t="490">So what I'm trying to do here is get information about planet number one in the Star Wars API database.</a> <a data-t="496">And this will give you a response object.</a> <a data-t="501">If you print that out, you get some really useful information.</a> <a data-t="501">The actual URL that was used after any redirects. The HTTP status. The file or content type. And the size of that response.</a></p>

<p><a data-t="518">So you can use various helpers in Hitter to dig into the response object.</a> <a data-t="522">For example, the status code helper will pull out just the code itself.</a> <a data-t="527">200 is what you want to see. That means okay, successful.</a> <a data-t="532">Anything else may be a problem that you would need to handle in your code.</a> <a data-t="538">Using the headers, helper will get you everything, all the metadata that came around through the header of the response.</a> <a data-t="545">Date, content type, connection, transfer encoding, et cetera.</a> <a data-t="551">And then what most people are interested in is the body itself.</a> <a data-t="556">This is the content of the response. Some of it is URLs.</a> <a data-t="574">So you're going to need to understand all this in order to handle it appropriately in your scripts.</a></p>

<h3>Handling the HTTP response</h3>

<p><a data-t="578">So there are three main parts that you'll probably be most interested in in your HTTP response.</a> <a data-t="585">The first is status. You can get the full status information with the HTTP status function.</a> <a data-t="590">It will tell you everything that it possibly can about the status.</a> <a data-t="595">In most cases, you're only interested in the code.</a> <a data-t="598">So the status code, sorry, object will come back with just the number that you can make decisions in your code based on that number in most cases.</a> <a data-t="609">You can automatically throw a warning or raise an error if it does not succeed, if it was not 200 with the</a> <a data-t="614">warn for status and stop for status helpers in the Hitter package.</a> <a data-t="614">I highly recommend these.</a> <a data-t="620">They convert HTTP errors into R errors, making them much easier to handle in your script.</a> <a data-t="625">And they'll either warn or stop just as they imply.</a></p>

<p><a data-t="629">The second of the components that you'll want to handle are the headers themselves.</a> <a data-t="636">Date, content type, connection, allow, things like that.</a> <a data-t="640">There can be custom headers in here. It really depends on the API that you're working with.</a> <a data-t="645">The one that you'll probably be using most often is the content type.</a> <a data-t="650">Most APIs have a pretty consistent content type in their response.</a> <a data-t="655">In this case, they're using JSON. So you may only have to worry about this once.</a> <a data-t="659">But if you're calling an unfamiliar API or you're calling an API that changes frequently, it may behoove you to take a look at that content</a> <a data-t="664">type to make sure that what you're parsing in your script matches what you're being sent.</a></p>

<p><a data-t="671">You can also get cookies if you're interested. It really depends on your API or what you're trying to get out of it.</a> <a data-t="675">But the thing, again, that most people are interested in is the body.</a> <a data-t="676">You use the content function in Hitter to get the response body.</a> <a data-t="682">In this case, you pass it your object and a modifier.</a> <a data-t="687">So in this case, I want a character vector. So I'm going to pass in the word text.</a> <a data-t="694">And it'll give me back all of that information that we just saw earlier in the structure output.</a> <a data-t="701">If you have non-text responses, you can get the raw content with the raw modifier.</a> <a data-t="710">Or you can give it the parsed. Hitter has some default parsers for common file types, including</a> <a data-t="715">JSON, XML, and a couple of others.</a> <a data-t="717">This can be really useful if you want a quick and dirty look at it.</a> <a data-t="721">But honestly, I tend to want to do the parsing explicitly.</a> <a data-t="725">And so I don't tend to use this very often, but it's there if you want it.</a></p>
