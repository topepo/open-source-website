<h3>Setting the team up for success</h3>

<p><a data-t="38">My first recommendation is to set the team up</a> <a data-t="38">for success by getting infrastructure in order.</a> <a data-t="43">Moving away from historical validation practices</a> <a data-t="43">means updating or generating new SOPs.</a> <a data-t="49">Another area for infrastructure support is to establish</a> <a data-t="49">common development guidelines that get shared across all packages.</a> <a data-t="54">This is especially important</a> <a data-t="54">if packages are implementing legacy code that was previously siloed.</a> <a data-t="59">The ultimate benefit,</a> <a data-t="59">however, is code portability within the overall project.</a></p>

<h3>Addressing development from an empathy perspective</h3>

<p><a data-t="64">My next recommendation is the need</a> <a data-t="64">to address development from an empathy perspective.</a> <a data-t="69">At SHARP, we observe four distinct roles.</a> <a data-t="69">One, users rapidly processing data.</a> <a data-t="69">These would be primarily statistical programmers.</a> <a data-t="74">Two, data analysts reviewing code for context.</a> <a data-t="74">These would be individuals who need to be</a> <a data-t="79">able to rapidly assess context from function names alone.</a> <a data-t="85">Three, stakeholders interacting</a> <a data-t="85">with functions solely from the package validation report.</a> <a data-t="89">So, this could be someone involved</a> <a data-t="89">in writing specifications and test cases but aren't participating in coding activity.</a> <a data-t="96">Four,</a> <a data-t="96">package developers and maintainers who seek to eliminate duplication or re-implementation</a> <a data-t="100">with package updates.</a></p>

<p><a data-t="104">Now, one common need that transcends all of these user groups is</a> <a data-t="104">to have a vocabulary which allows us to place functions within the larger context of assay-specific</a> <a data-t="109">data processing tasks.</a> <a data-t="114">That is, we want to survey what has been implemented, filter either</a> <a data-t="114">by assay or task, and call up help page documentation on demand.</a> <a data-t="120">So, this is actually a reference</a> <a data-t="120">to Schneiderman's information-seeking mantra, which is typically cited in visualization</a> <a data-t="125">literature.</a> <a data-t="129">In this context, we are treating the function names as the identifying handle</a> <a data-t="129">and leveraging </a><a href="/software/rstudio/">RStudio</a><a data-t="129">'s autocomplete as the filtering mechanism as we navigate to</a> <a data-t="134">a particular help page or call the function.</a> <a data-t="139">For example, one step of data processing that</a> <a data-t="139">occurs for all assays is the transformation to a CDISC Atom basic data structure.</a> <a data-t="144">Because</a> <a data-t="144">of the function naming convention we have chosen, a list of all available BDS transformation</a> <a data-t="149">functions are displayed and we can select for the assay of interest.</a></p>

<h3>Package validation and systematic mapping</h3>

<p><a data-t="153">Focusing on the package validation elements, we also need a systematic approach to how</a> <a data-t="157">we map package requirements, test cases, and test code files.</a> <a data-t="162">Validation is a form of user</a> <a data-t="162">acceptance testing, where we have to show that for every software requirement there</a> <a data-t="166">was an appropriate test successfully executed.</a> <a data-t="171">The modular approach we use at SHARP is incredibly</a> <a data-t="171">flexible, which has the development benefits I will discuss later, but it does require</a> <a data-t="175">some organization to track.</a> <a data-t="180">In addition to enforcing a file naming convention, we also</a> <a data-t="180">scrape these files to generate a tabular display in the validation report, which links requirements</a> <a data-t="185">and tests.</a> <a data-t="191">Between these two implementations, both developers and validation report stakeholders</a> <a data-t="191">are able to verify at a glance that complete coverage exists.</a></p>

<h3>Managing technical debt across packages</h3>

<p><a data-t="195">The last couple recommendations I have address the topic of technical debt.</a> <a data-t="200">You may be wondering,</a> <a data-t="200">why are we splitting assays into separate packages?</a> <a data-t="205">A couple reasons.</a> <a data-t="205">First, FDA guidance</a> <a data-t="205">asks us to revalidate the entire system when one element changes.</a> <a data-t="211">Now, change is expected,</a> <a data-t="211">input data structures get updated, analysis needs drift.</a> <a data-t="217">However, an update to one assay</a> <a data-t="217">type should not introduce the appearance of change affecting other assays.</a> <a data-t="222">Secondly, from</a> <a data-t="222">the user perspective, we need to put some conceptual structures around processes.</a> <a data-t="227">Sure,</a> <a data-t="227">every data set will be converted to BDS, but the definition of that basic data structure</a> <a data-t="231">will vary from assay to assay, and we need to capture that distinction in a manner that</a> <a data-t="236">is easy to digest.</a></p>

<p><a data-t="241">What we also want to avoid is excessive re-implementation.</a> <a data-t="241">As we have</a> <a data-t="241">expanded to multiple assay packages, we have identified utility functions that can be shared</a> <a data-t="246">and moved into their own foundation package.</a> <a data-t="251">We can enforce package code dependencies through</a> <a data-t="251">routine use of the description file, but the power of modular validation means that we</a> <a data-t="256">can move the associated specifications, test cases, and test code files as well.</a> <a data-t="262">Validation</a> <a data-t="262">of the lighter assay package with the new utility package is simply compiling the R</a> <a data-t="267">markdown document without any need to rewrite content.</a> <a data-t="272">Likewise, when we need to add additional</a> <a data-t="272">specifications, test cases, or test code to an existing package, an updated validation</a> <a data-t="278">report will add the new child documents to existing sources.</a> <a data-t="283">This has the added benefit</a> <a data-t="283">of being able to use a simple diff to compare validation reports across package versions.</a></p>
