<h3>Best practices for administration</h3>

<p><a data-t="430">So I just want to quickly talk about best practices for administering Jupyter with RStudio</a> <a data-t="436">Server Pro.</a> <a data-t="439">I recommend that you use the default configuration wherever possible, because we've put a lot</a> <a data-t="443">of work into making the configuration defaults just work out of the box with very sane defaults</a> <a data-t="449">that you shouldn't need to be tweaking.</a> <a data-t="451">But two settings that you may actually want to tweak are related to session suspension</a> <a data-t="458">and automatically cleaning up sessions that were forgotten about.</a></p>

<p><a data-t="471">And then when you're mounting your NFS directory, you need to mount NFS directories so that</a> <a data-t="476">users have access to their notebooks.</a> <a data-t="483">And if you're doing that with NFSv3, we recommend using local lock equals all as a mount option</a> <a data-t="487">or just using NFSv4.</a> <a data-t="488">Otherwise you could run into some problems there.</a> <a data-t="491">And then if you're going to be using this with Kubernetes, we recommend that you use</a> <a data-t="495">the R session complete RStudio Docker image.</a> <a data-t="500">So this is basically a Docker image that we published to Docker Hub that has all the RStudio</a> <a data-t="508">binaries that you need to launch Jupyter, and then it has Jupyter itself and an installation</a> <a data-t="513">of Python, and then you can create derivative images from that to provide your users with</a> <a data-t="519">whatever you need in terms of Python versions.</a></p>
