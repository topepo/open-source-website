<h3>Why choose GitHub Actions</h3>

<p><a data-t="200">So this GitHub Actions is not the only service that provides these sort of features.</a> <a data-t="205">So why might GitHub Actions be a good thing for you to try on your repository?</a> <a data-t="210">The first thing is that it's one of the few services that has support for multiple, all three major operating systems, Linux, Windows, and macOS.</a> <a data-t="224">And it also has first class support for Docker containers.</a> <a data-t="227">So if your package or your project, if you're used to working in Docker containers, and you have all of your dependencies already bundled into a Docker container, you can use those on GitHub Actions very easily.</a></p>

<p><a data-t="239">It also has very generous resource allocations for your jobs.</a> <a data-t="244">So you can run 20 concurrent jobs for a given repository.</a> <a data-t="248">And those jobs can take up to six hours.</a> <a data-t="252">And because GitHub Actions is built into GitHub, it doesn't require separate authentication and setup steps apart from your regular GitHub authentication.</a> <a data-t="262">So as soon as you commit the configuration file that describes your workflow to your repository, it's going to start running on GitHub Actions.</a> <a data-t="270">You don't have to do any additional manual setup.</a></p>

<p><a data-t="275">And you might ask, is this service free?</a> <a data-t="277">And it is for open source, academic, and educational repositories.</a> <a data-t="283">If you don't fall into one of those buckets, you can still use GitHub Actions, but you have to pay a fairly reasonable fee.</a></p>

<p><a data-t="292">So I mentioned there were some other services that do similar things to what GitHub Actions provides.</a> <a data-t="299">Some of the most popular that are used in the R community are Travis CI, CircleCI, and AppVeyor.</a> <a data-t="306">You can see compared to these, GitHub Actions is one of the few that has support for all of these three operating systems and really good support for Docker containers.</a> <a data-t="314">And then it also gives you many more concurrent builds than the other services provide for open source projects.</a> <a data-t="319">And the maximum job length is the longest of any of the other options.</a> <a data-t="325">And again, in terms of setup, because this is integrated into GitHub, it's basically painless to set up, whereas some of the other services range from moderately annoying to extremely annoying.</a></p>
