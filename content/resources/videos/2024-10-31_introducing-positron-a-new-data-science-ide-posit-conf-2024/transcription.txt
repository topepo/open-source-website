<h3>The data explorer</h3>

<p><a data-t="1245">My name is Tom Mock.</a> <a data-t="1245">I'm a product manager for Positron, and I'm excited to talk a little bit more deeply about one little area of Positron and how much focus we put on the little design principles within it.</a> <a data-t="1255">So, when you first start opening your data, you might want something like a data viewer, right?</a> <a data-t="1259">In RStudio, there's a data viewer.</a> <a data-t="1261">You can see your grid, or you might be opening things in tools like Excel because you want to kind of walk through your data or feel the data or explore individual cells as opposed to doing group by operations or summarization.</a> <a data-t="1273">So, of course, we built this into Positron and made it available for both Python and R.</a> <a data-t="1278">But we didn't want to stop with just a data viewer.</a> <a data-t="1282">So, we've actually built what we call a next-generation data explorer.</a></p>

<p><a data-t="1287">Of course, it still has that data viewer grid that people are looking for in terms of individual cells and the interaction of an indice, a column, and details about them.</a> <a data-t="1296">But we also have things like a summary panel showing you summary statistics that are appropriate for that data, you know, measures of central tendency for numeric data or unique values for strings or categorical data.</a> <a data-t="1306">There's also missing data and column names there as well as their type.</a> <a data-t="1311">And lastly, we also have a persistent filter bar that allows you to create, delete, and see the filters you've applied to your data as you're exploring it.</a></p>

<p><a data-t="1374">Beyond that, though, we've tried to make it something that's highly scalable.</a> <a data-t="1378">So, again, in this example, I have 31 million rows, 24 columns, and in real time, I can scroll horizontally or vertically across anything in the data frame, and it will kind of instantly load that data.</a> <a data-t="1388">We do a lot of clever things with caching, fetching different windows of data to try and make it efficient in this way.</a> <a data-t="1396">But we've also sweated a little bit of the small stuff.</a> <a data-t="1396">So, things like sorting.</a> <a data-t="1400">It's not just one column.</a> <a data-t="1400">It's multi-sorting.</a> <a data-t="1400">So, you can sort multiple columns at a time and show which order you've sorted them in and remove different sorting as needed so you can modify your data ephemerally in this way.</a> <a data-t="1414">We've used model space fonts in the grid to show you things like nice decimal alignment, problematic things like white space, or differentiating simply between one, I, and L of various capitalization.</a> <a data-t="1423">And then, lastly, for the grid, we've worked really hard on automatic column width so you can fit as much data horizontally and vertically as possible as opposed to these really, really wide columns when you've got a short variable name and short values.</a></p>

<h3>Positron as a Code OSS fork</h3>

<p><a data-t="1751">Positron is a fork of another project called Code OSS.</a> <a data-t="1757">And you can be forgiven for not knowing what Code OSS is, but I promise you, you do know what it is, because Code OSS is the foundation for a proprietary IDE that almost all of you have used, which is called Visual Studio Code.</a></p>

<p><a data-t="1773">We are not the only people to fork Code OSS.</a> <a data-t="1778">Many other companies have forked Code OSS to make IDEs that are specific for other purposes.</a> <a data-t="1783">For example, there's a company called Cursor that makes an IDE called the Cursor IDE with a lot of AI features.</a> <a data-t="1789">So, we are doing something very similar with Positron.</a> <a data-t="1795">We are taking the Code OSS foundation and adding data science tools to it.</a></p>

<p><a data-t="1801">And we've already covered this a little bit in the session, but what that looks like is taking that powerful IDE foundation and adding tools like environment selections, plots and visualization pane, a really first-class console, an amazing data explorer, and so forth.</a></p>

<p><a data-t="1818">So, this brings me to the second elephant in the room, which is, why isn't Positron a VS Code extension?</a> <a data-t="1827">So, to answer that question, I've got my good friend Twilight Sparkle here.</a> <a data-t="1832">So, Microsoft learned a lot from their experience with other IDEs.</a> <a data-t="1837">IDEs like Atom, for example, and Emacs, they allow you to load extensions and they become kind of part of the IDE.</a> <a data-t="1846">This can lead to a lot of clutter and slowness and instability.</a> <a data-t="1850">And so, when Microsoft built Code OSS, they learned to put the extensions in a sandbox and run them separately from the IDE.</a></p>

<p><a data-t="1861">The other thing Microsoft learned is that when you give people the ability to do anything you want with extensions, they just can't help themselves.</a> <a data-t="1871">It is a mess, right?</a> <a data-t="1873">And so, Microsoft learned their lesson and Visual Studio Code extensions are fairly limited.</a> <a data-t="1879">They can't contribute custom UI to the Workbench service.</a> <a data-t="1883">They run on that separate sandbox process.</a> <a data-t="1885">They're ephemeral.</a> <a data-t="1885">They can be restarted at any time.</a> <a data-t="1891">They stop running when the browser is closed.</a> <a data-t="1891">And finally, probably most importantly for us, they don't make sense as a foundation for other extensions, right?</a> <a data-t="1899">R and Python and these other language packs are extensions to Positron.</a> <a data-t="1903">And it's very difficult to build a horizontal layer that provides that extensibility when you are yourself running in the sandbox.</a></p>

<p><a data-t="1916">So, finally, let's talk a little bit about how R and Python talk to these Positron features.</a> <a data-t="1920">So, R and Python are extensions.</a> <a data-t="1920">We call them language packs for Positron.</a> <a data-t="1923">And they talk to the code OSS features in the same way that most other extensions do.</a> <a data-t="1930">So, VS Code has an API, which is what allows extensions to contribute things like auto-completion and diagnostics.</a> <a data-t="1937">So, you've seen a lot of Positron's specific features and we've added a separate parallel API that allows the extensions to not only contribute code completion and diagnostics, but also plug into our console, our plots pane, and the rest of our tools.</a></p>

<p><a data-t="1954">It's important to note, then, that those R and Python sessions live in extensions.</a> <a data-t="1960">And this is important not only for extensibility, but also for this reason.</a> <a data-t="1965">How many of you have seen this screen before?</a> <a data-t="1972">So, this is what happens when R crashes in RStudio.</a> <a data-t="1972">And the reason that it crashes the whole window is because, as we just established, that window is itself R.</a> <a data-t="1978">Everything is coming from R.</a> <a data-t="1980">So, this is what an R crash looks like in RStudio.</a> <a data-t="1983">Here is what an R crash looks like in Positron.</a> <a data-t="1989">It's nothing.</a> <a data-t="1992">It is horribly obliterated.</a> <a data-t="1993">You will lose anything you've got in memory because your R process crashed, but the rest of the IDE will continue to run very smoothly.</a></p>

<p><a data-t="2003">Now, the final reason that it's really important to have all these things in extensions is that this allows Positron to be extended to other languages in the future.</a> <a data-t="2011">And this is something Julia alluded to really well, which is that we want Positron to be a foundation upon which more data science tooling can be built.</a> <a data-t="2020">And all of the R and Python code that we have written plugs into a public API that anybody can write for.</a> <a data-t="2028">So, it's not tightly integrated into the core.</a> <a data-t="2032">Everything in the Positron core can be added to by anybody via a public API.</a></p>

<p><a data-t="2040">So, we have Positron and we have these language packs.</a> <a data-t="2042">I want to talk for just a minute about how those two things talk to each other.</a> <a data-t="2047">So, when we built Positron, we wanted to use standard protocols for standard functionality.</a> <a data-t="2053">And what that means is that when those R and language subsystems talk to Positron, they use protocols that have already been established in the data science community.</a> <a data-t="2064">When we run an execute code, we are using the Jupyter protocol.</a> <a data-t="2067">When we do completion and diagnostics, we're using the language server protocol.</a> <a data-t="2073">We've only added our own protocols when we have Positron specific features that we need to power.</a></p>

<p><a data-t="2084">So, quick recap.</a> <a data-t="2084">It is a polyglot IDE.</a> <a data-t="2086">It's focused on data science.</a> <a data-t="2087">It's a code OSS fork like VS Code is.</a> <a data-t="2090">Those extensions live with language features.</a> <a data-t="2094">We use existing protocols wherever we can to make it easy for other people to plug in.</a> <a data-t="2098">And we have a public API that lets anyone add new languages or features.</a></p>

<h3>ARC: the R kernel for Positron</h3>

<p><a data-t="2108">Hello, my name is Lionel, and I'm here with my colleague Davis to talk to you about ARC, which is a component in Positron.</a> <a data-t="2115">So, Davis and I are from the Tide Divers group, and we've been working for a number of years on packages like </a><a href="/software/dplyr/">dplyr</a><a data-t="2115">.</a> <a data-t="2122">And for the last year, a bit more than that, we've been working on Positron with the mission to make it easier for other people to use Positron.</a> <a data-t="2134">To help making R support the best it can be in Positron.</a> <a data-t="2139">So, we've seen with Jonathan that Positron is a multilingual IDE.</a> <a data-t="2145">And we've seen with Isabel that we've broadened our scope to include Python.</a> <a data-t="2151">And so, the question that we want to answer in this talk is, where does that leave us with R?</a></p>

<p><a data-t="2158">And the answer is that we wanted to make the R support the most modern it could be, and the result is ARC.</a> <a data-t="2165">So, when you use R in Positron, ARC is behind most of what's happening.</a> <a data-t="2172">So, if you type R code and get some completions, ARC is providing the completions.</a> <a data-t="2179">If you need to execute a piece of R code and get some output from R, that goes through ARC as well.</a></p>

<p><a data-t="2188">So, what does it take to make a great development experience for R?</a> <a data-t="2193">There are three main things that ARC is concerned with.</a> <a data-t="2197">First, code execution, so sourcing of script or executing a piece of code.</a> <a data-t="2202">Code assistance, so that ranges anything from completions to diagnostics to let you know that there might be a problem in your code even before you execute it.</a> <a data-t="2213">Or contextual help.</a> <a data-t="2216">And the final piece is the debugger.</a> <a data-t="2218">We want to make the debugging experience in Positron the least frustrating it can be.</a></p>

<p><a data-t="2225">And each of these areas is backed by a standard protocol, and we'll see in this talk why this is important and why it matters.</a> <a data-t="2234">And we'll start with execution.</a></p>

<p><a data-t="2239">So, here you have a classic Positron setup.</a> <a data-t="2242">You have an open R file with a bunch of code.</a> <a data-t="2246">You have a console and a plot pane.</a> <a data-t="2250">And the user here has created a data frame, got some output in a console, and created a plot.</a> <a data-t="2255">So, where does ARC fit in this picture?</a> <a data-t="2260">So, you can think of code that you execute in Positron as a question that Positron asks to ARC.</a> <a data-t="2268">ARC does a bit of magic and then provides an answer back to Positron, which is able to display the output in the console.</a> <a data-t="2277">The same thing when you create a plot.</a> <a data-t="2281">ARC does a bit of magic after getting the question from Positron and provides the answer back to Positron.</a></p>

<p><a data-t="2289">So, the question is, how does this work?</a> <a data-t="2291">What is the language that ARC and Positron use to communicate with each other?</a> <a data-t="2299">And to answer this, we'll take a step back and look at what's going on in the Jupyter world.</a> <a data-t="2305">So, here we have a Jupyter notebook.</a> <a data-t="2312">If you're not familiar with this way of doing data science, you have a document made up of cells.</a> <a data-t="2316">You can write code in the cells and you execute these cells and you get the output included in the document.</a> <a data-t="2324">So, this is very popular in the Python community.</a> <a data-t="2328">But the thing about Jupyter is that it works with multiple languages.</a> <a data-t="2332">It's multilingual, just like Positron.</a> <a data-t="2336">And it's even in the name.</a> <a data-t="2338">So, if you did not know, Jupyter stands for Julia, Python, and R.</a></p>

<p><a data-t="2343">And so, the way they pull this off is not by implementing support for all these languages.</a> <a data-t="2351">They have independent backends called kernels.</a> <a data-t="2355">And so, when a Jupyter notebook needs to execute a piece of code, it's like a question.</a> <a data-t="2361">The kernel gets the question, provides an answer back to the notebook.</a> <a data-t="2366">And Jupyter came up with a standard set of questions and answers.</a> <a data-t="2372">And in this case, it's an execute request that is asked to the Jupyter kernel and the kernel provides an execute response back.</a> <a data-t="2379">And that's exactly what we needed to communicate between Positron and Arc.</a> <a data-t="2385">And so, Arc is a Jupyter kernel.</a> <a data-t="2390">And actually, in this notebook, this is Arc running in the background, running the R code for the notebook.</a></p>

<h3>Code assistance and the debugger</h3>

<p><a data-t="2453">While code execution is certainly a very important part of this, it's not the only piece of the puzzle, or not all pieces of the puzzle.</a> <a data-t="2461">We also think that it's really important to have fantastic code assistance inside of Positron.</a> <a data-t="2467">Now, code assistance just means anything that makes it easier for you to write R code.</a> <a data-t="2472">That could look like diagnostics, where we analyze all the files in your project and can pop up a little tooltip that says, hey, we think something's wrong here.</a> <a data-t="2481">You might want to take a look.</a> <a data-t="2484">Code assistance could also look like a function signature, where you're working with SliceMax.</a> <a data-t="2488">You don't necessarily need to remember all of the arguments.</a> <a data-t="2490">We can pop up a signature and say, here, hey, here's what they are.</a></p>

<p><a data-t="2495">Assistance could also look like jump to definition.</a> <a data-t="2498">Jump to definition looks like this, where you can click on SliceMax, and that takes you straight through to the definition of the function inside of your project.</a> <a data-t="2511">Now, these are features that you might have been familiar with in RStudio, but Positron has a number of code assistance features that are unique to it as well.</a> <a data-t="2519">One of those is help on hover, which I really like.</a> <a data-t="2523">It looks like this, where you hover over SliceMax, and you get this really great inline documentation immediately.</a> <a data-t="2531">This is great whenever you're actually just kind of in the flow state of writing R code.</a> <a data-t="2535">You don't necessarily need a whole help pane.</a> <a data-t="2537">You just kind of need to quickly remember how something works.</a> <a data-t="2540">You hover over it, get what you need to know, and you go back to work.</a></p>

<p><a data-t="2546">Now, these features in Positron work in a very similar way as executing code.</a> <a data-t="2552">Code execution worked through the Jupyter protocol.</a> <a data-t="2557">It allowed us to ask questions and send back answers, and code assistance works in the same way.</a> <a data-t="2564">With the Jupyter protocol, that really allowed us to standardize these questions and answers.</a> <a data-t="2569">With code assistance, we used the language server protocol, which standardizes the questions and answers in a very similar way.</a> <a data-t="2576">We ask a hover request of Arc, and Arc can send back a hover response in return.</a> <a data-t="2584">The reason that, again, that is so cool is because Arc doesn't care who's on the other side.</a> <a data-t="2589">It doesn't care who's asking the question.</a> <a data-t="2591">It could be Positron, it could be a Jupyter notebook, it could be some other IDE.</a> <a data-t="2595">As long as it implements the other half of the language server protocol, you're in business.</a></p>

<p><a data-t="2601">Over on the right-hand side here, our magic 8-ball, Arc, is doing quite a bit of work to actually send back these responses.</a> <a data-t="2608">Part of the smarts of Arc is wrapped up in a technology called TreeSitter.</a> <a data-t="2614">The only thing you need to know about TreeSitter in this talk is that it takes in all of the R code in your project.</a> <a data-t="2622">We run it through TreeSitter, and it spits out a structured representation of that code, which honestly doesn't even look like R at this point.</a> <a data-t="2629">However, we can use that and power all of the assistance features that we've been talking about today.</a> <a data-t="2636">Now, we think TreeSitter is really cool as a project because it works with any language.</a> <a data-t="2641">As long as you have a grammar that teaches TreeSitter how to work with that language, then you're good to go.</a> <a data-t="2649">The problem was we didn't have an R grammar for TreeSitter, so we had to build one.</a></p>
