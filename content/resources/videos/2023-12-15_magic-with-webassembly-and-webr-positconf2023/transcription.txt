<h3>How it works under the hood</h3>

<p><a data-t="573">So how does this work?</a> <a data-t="578">So the RWASM process runs inside something called a JavaScript web worker.</a> <a data-t="583">This is important because if it didn't do that, it meant that every time you asked R</a> <a data-t="587">to do something, your entire browser would freeze.</a> <a data-t="592">And it's not a good user experience, to say the least.</a> <a data-t="596">That allows the main thread to remain responsive.</a> <a data-t="599">And so to make this work, communication between that main browser thread, where your website</a> <a data-t="604">is and where you're interacting with things and pressing buttons, and the R process is</a> <a data-t="609">handled by message passing.</a> <a data-t="609">So input messages go to R, output messages come back to you on the main thread.</a></p>

<p><a data-t="617">So if you're working with code, you could imagine you give WebR the command Rnome, and</a> <a data-t="621">the worker thread thinks for a bit, the main thread remains responsive, and then after</a> <a data-t="625">a while, you get some output back.</a> <a data-t="626">And you can see here's an example of what I meant before about this kind of thing being</a> <a data-t="630">difficult to work with.</a> <a data-t="631">You can see the output there.</a> <a data-t="632">It has an index at the front of it.</a> <a data-t="634">The numbers are space-separated.</a> <a data-t="635">There's no guarantee how long these could be.</a> <a data-t="638">They could be integers, for example.</a> <a data-t="640">So working with text is not great.</a> <a data-t="643">And that's why WebR lets you work directly with R objects.</a></p>

<p><a data-t="648">So R objects are returned to the user as references and handled by something called a JavaScript</a> <a data-t="654">proxy.</a> <a data-t="654">Now, JavaScript proxies are really cool, because what they let you do is work with a certain</a> <a data-t="659">object but then intercept fundamental operations.</a> <a data-t="657">So if you wanted to read a property of that object, or if you wanted to, say, invoke the</a> <a data-t="663">object as if it was a function, if you were doing that and you just had a number, that</a> <a data-t="669">would fail, because a number has no methods, a number cannot be invoked as a function.</a> <a data-t="673">But proxies let you step in and say, OK, I'll do something else instead, whenever you try</a> <a data-t="678">and do these things.</a></p>

<p><a data-t="680">So by storing a reference to an R object and then using the proxy to handle these things,</a> <a data-t="686">it means that you don't need to pass around giant pieces of memory full of numbers.</a> <a data-t="693">So here, for example, if you evaluate some R code to generate a vector, that R code is</a> <a data-t="700">evaluated and then the result is returned back to the user, not as a vector, but as</a> <a data-t="705">a number.</a> <a data-t="705">And that number there, the SS number, which is short for S expression, it's a lispy thing,</a> <a data-t="711">it's old R stuff.</a> <a data-t="714">That pointer, that number, uniquely identifies that object in R memory.</a></p>

<p><a data-t="721">So when you're working with that R object in JavaScript, JavaScript doesn't really need</a> <a data-t="724">to know what that is, it just needs to know this is an R object, and this is where it</a> <a data-t="728">lives.</a> <a data-t="729">It lives here in memory.</a> <a data-t="731">And then when you do need the numbers, say you actually want to convert that number into</a> <a data-t="735">JavaScript or you want to invoke that function, only then does WebR actually go and use RcApi</a> <a data-t="742">to do the hard work to actually get those numbers.</a> <a data-t="745">So here you can see only after I actually ask WebR to convert to JavaScript, only then</a> <a data-t="753">that big long list of numbers is returned.</a> <a data-t="756">Now here there's only five numbers, but you can imagine if this was a million lines long</a> <a data-t="761">or it had something like 10 million points, that would make a real difference because</a> <a data-t="765">otherwise you'd be passing this object around between JavaScript and R all of the time.</a></p>

<h3>Service workers and shinylive</h3>

<p><a data-t="778">One more thing I want to talk about with WebR is service workers.</a> <a data-t="782">So service workers is, again, a JavaScript feature that are very similar to web workers.</a> <a data-t="788">But rather than acting as a proxy on a certain object, it acts as a network proxy.</a> <a data-t="793">And what that means is that when you ask for a website in your browser, if you have a service</a> <a data-t="799">worker loaded, that can step in and say, no, I don't want you to talk to Google.com, instead</a> <a data-t="805">you will connect to Bing or something like that.</a> <a data-t="809">It doesn't even have to be another website.</a> <a data-t="811">This could step in, stop the network traffic, compute something itself, and then return</a> <a data-t="817">that as the result.</a></p>

<p><a data-t="822">And that's how shinylive for Python already works.</a> <a data-t="825">Whenever you have a shinylive session in your page, and a network connection is made</a> <a data-t="831">to what would normally be a </a><a href="/software/shiny-r/">Shiny</a><a data-t="831"> server, a service worker steps in and redirects that</a> <a data-t="838">traffic to a running PyEdit process inside a web worker.</a> <a data-t="844">And the same thing works for R. But here, WebR is acting as the bridge.</a> <a data-t="851">So the service worker intercepts the network traffic, and then all those tricks with WebR,</a> <a data-t="856">all of that magic of interacting with the environment, that can be used by the service</a> <a data-t="860">worker to send that traffic, instead of a Shiny server, into a web worker process.</a> <a data-t="867">And because the web worker process is running an R WebAssembly, loading packages like Shiny,</a> <a data-t="873">it can do that computation and return the result without hitting the network at all.</a> <a data-t="878">And this is exactly how shinylive for R works.</a></p>
