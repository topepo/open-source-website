<h3>Building the automated pipeline</h3>

<p><a data-t="163">The first step in the process was therefore to figure out an automated pipeline.</a> <a data-t="169">We started out with the data in Excel files and various other sources, which were ingested into Google Sheets as a central repository.</a> <a data-t="179">The reason for using Google Sheets instead of a database was that we still wanted it to be easily accessible to non-technical users,</a> <a data-t="188">so they could also access the data and update it as needed.</a></p>

<p><a data-t="193">Each figure is then governed by a central R script where the data is read in, the figure produced in ggplot, and then the main output focus was SVG.</a> <a data-t="206">However, with such an automated workflow, we can also output multiple formats, useful in their various publishing platforms, whether it's EPUB, print, or the web.</a> <a data-t="218">Furthermore, by taking advantage of the Plotly library and adding a single line of code to our scripts,</a> <a data-t="225">we can also create interactive versions of each figure for the web, further enhancing the teaching and learning experience.</a></p>

<p><a data-t="235">This entire process is wrapped up within an R package with several helper functions to help automate this pipeline.</a> <a data-t="243">When we get to integrating with the publishing house's processes, this will likely become an API.</a></p>

<h3>Handling translation</h3>

<p><a data-t="563">Having done this, the publishing house were now quite impressed with our skills</a> <a data-t="568">and starting to believe in us that we could actually do this.</a> <a data-t="572">The next question then was, OK, so can you do this in all languages?</a> <a data-t="577">This economics textbook has been translated into several languages</a> <a data-t="581">and so we needed to figure out how to handle the automatic translation of these figures in our pipeline.</a></p>

<p><a data-t="590">This third step therefore focused on translation</a> <a data-t="593">and the solution that we came up with was to capture the translated strings in a YAML file</a> <a data-t="600">which is then read into the R script for the figure.</a> <a data-t="604">YAML, if you're not familiar with it, used to stand for Yes Another Markup Language</a> <a data-t="610">and it's now become the recursive acronym YAML Ain't Markup Language</a> <a data-t="615">to specifically distinguish its use as being data-oriented</a> <a data-t="620">rather than a traditional markup for documents.</a></p>

<p><a data-t="625">It's often used in configuration files</a> <a data-t="628">and you might have seen it or noticed it at the top of our markdown documents.</a> <a data-t="633">That's YAML.</a> <a data-t="634">It's very structured and importantly, it's human-readable.</a> <a data-t="639">This is important for us because we needed the translators who are non-technical</a> <a data-t="644">to be able to easily work with the format in order to capture the translated strings.</a> <a data-t="649">But at the same time, I have a structure that has a clear format to it</a> <a data-t="654">and it's data-oriented to work with in R.</a></p>

<p><a data-t="658">On the left-hand side here, you can see an example from one of our YAML files.</a> <a data-t="663">Each of the languages is denoted by an abbreviated two-letter word</a> <a data-t="668">for example, EN for English.</a> <a data-t="672">Each of the labels that then requires translation is assigned a variable name</a> <a data-t="677">that's common across all of the languages.</a> <a data-t="680">For example, x-axis label.</a> <a data-t="683">The value for this label is then the text that is the translated string.</a> <a data-t="689">For example, year, ano, or warsi.</a></p>

<p><a data-t="693">In R, we then have the YAML package to work with these files.</a> <a data-t="698">So the strings are read into R using the function read underscore YAML</a> <a data-t="704">as a list of lists, which I then convert into a data frame.</a> <a data-t="709">Within the ggplot code, each of the labels that requires translation</a> <a data-t="714">is then substituted with that variable name.</a> <a data-t="719">The list of languages that needs to be generated</a> <a data-t="722">is obtained from the first level in the hierarchy of the YAML file.</a> <a data-t="727">And then I iterate over those languages</a> <a data-t="729">and replace the variable names with each translated string</a> <a data-t="733">as we generate the output for that translation.</a></p>

<p><a data-t="738">The fourth benefit of this process was that it allowed us</a> <a data-t="742">to give the translators some more custom control over the labels.</a> <a data-t="748">What I mean by this is, for example, in some cases</a> <a data-t="751">we needed some part of the string to be in italics.</a> <a data-t="755">For example, the unit.</a> <a data-t="757">With just English, this is fairly straightforward to do</a> <a data-t="760">within the R script.</a> <a data-t="762">However, if we take multiple languages into account,</a> <a data-t="766">how do we know which part of the string needs to be in italics?</a></p>

<p><a data-t="771">This is where I use the ggtext package.</a> <a data-t="775">This is a line from the theme function which I showed earlier</a> <a data-t="778">where we replaced the element text function</a> <a data-t="782">with element markdown.</a> <a data-t="785">What this enables is that the translators can use</a> <a data-t="788">simple markdown in their strings, which is then converted</a> <a data-t="792">to display the correct output in the final image.</a> <a data-t="796">For example, they could enclose a unit in asterisks</a> <a data-t="800">and this is then converted into italics in the final output.</a></p>

<p><a data-t="805">This worked well for labels that were accessed titles.</a> <a data-t="809">However, we also have many labels that require translation</a> <a data-t="812">that are placed on the plot with geom label or geom text.</a> <a data-t="817">In order to cater for this, what I did was again use</a> <a data-t="820">the ggtext package and create a function that corresponds</a> <a data-t="825">with the original ggplot to override it.</a> <a data-t="828">For example, geom text.</a> <a data-t="830">Here we use instead geom underscore rich text,</a> <a data-t="834">which again enables translators to be able to put</a> <a data-t="839">simple markdown into their strings.</a> <a data-t="841">That's then catered for in the final output.</a></p>

<h3>Handling different numbering systems</h3>

<p><a data-t="846">One of the really interesting challenges to solve</a> <a data-t="849">with the translation was the different numbering systems</a> <a data-t="852">for different languages and how do we automatically</a> <a data-t="855">cater for this in a single R script.</a> <a data-t="858">What do I mean by this?</a> <a data-t="860">Well, here you can see three graphs in three different languages.</a> <a data-t="864">What I want you to do is take note of the y-axis</a> <a data-t="867">and the format of each of the numbers.</a></p>

<p><a data-t="870">On the left we have English and you can see that</a> <a data-t="874">the thousand separator is a comma.</a> <a data-t="877">We then have Spanish, which has a space separator,</a> <a data-t="880">but only once the number is more than four digits long.</a> <a data-t="888">Then we have the Finnish and here you can see</a> <a data-t="891">that the thousand separator is a space.</a> <a data-t="894">So how do we cater for this in the R script?</a></p>

<p><a data-t="898">The solution I came up with was to create</a> <a data-t="901">a number formatting function per language</a> <a data-t="904">which uses the number format function</a> <a data-t="907">to specify what the big mark should be</a> <a data-t="911">for the thousand separator and the decimal mark,</a> <a data-t="915">whether it should be a comma or a point.</a> <a data-t="918">The key thing here is that each function for each language</a> <a data-t="923">has the same name as the two-letter abbreviated version</a> <a data-t="926">in the YAML file.</a> <a data-t="929">This here shows English compared to Finnish,</a> <a data-t="933">which is denoted by F-I.</a></p>

<p><a data-t="936">The Spanish was a slightly more involved example and function.</a> <a data-t="942">And what I did here was to iterate over the list</a> <a data-t="946">of numbers supplied as the labels and first check</a> <a data-t="950">whether the number is more than four digits long.</a> <a data-t="953">If so, then it gets a space and otherwise</a> <a data-t="956">there's no space for the thousand separator.</a></p>

<p><a data-t="962">Within the ggplot code, within the R script,</a> <a data-t="966">the production of each of the languages</a> <a data-t="968">is handled within the for loop and as mentioned previously,</a> <a data-t="972">this list of languages is obtained</a> <a data-t="974">from the first level heading in the YAML file,</a> <a data-t="977">for example, E-N, E-S, F-I.</a> <a data-t="981">In the ggplot code, I then use the very handy</a> <a data-t="985">match.fun function from base R</a> <a data-t="988">that takes the language input as a string</a> <a data-t="992">and then uses the number formatting function</a> <a data-t="995">that corresponds with that</a> <a data-t="997">in order to format the numbers for the Y axis</a> <a data-t="1000">according to what is needed for that language.</a> <a data-t="1004">This here shows the Y axis, but it can just as easily be done</a> <a data-t="1007">for the X axis.</a></p>
