<h3>Introducing Slushy</h3>

<p><a data-t="384">Slushy is an R package that is designed to support our teams that want to use an open R environment and need to figure out how to</a> <a data-t="398">huddle up effectively and efficiently. Our Slushy, we originally created it internally and we decided</a> <a data-t="406">to open source the code to share it with the community. So, I'll share the GitHub link at the</a> <a data-t="413">end. But Slushy makes use of a couple of really awesome existing tools. I mentioned renv already.</a> <a data-t="415">And it also makes use of another tool developed by Posit, which is Posit Package Manager.</a> <a data-t="423">On top of these two things, it gets us, or these two things alone, get our teams a long way.</a></p>

<p><a data-t="431">But they still need some additional supports. And so, in addition to these two things, we have</a> <a data-t="438">some, we're adding some additional functionality to help guide our users through their journey,</a> <a data-t="448">so it's nice and smooth. We're also putting in some guardrails for them,</a> <a data-t="452">so they can stay on track better and make sure everything's nice and easy for them</a> <a data-t="458">as they're adapting to this ever-changing open source landscape, right?</a></p>

<p><a data-t="464">So, we had a few goals when it came to building Slushy. <h3>How Slushy works</h3>

<p><a data-t="491">So, first and foremost, the renv package is absolutely fundamental in helping our users</a> <a data-t="501">begin to form that huddle, begin to form that home base. If you're not super familiar with renv,</a> <a data-t="508">what it does is, well, it does a lot of awesome things, but one of its key features is helping</a> <a data-t="514">everyone on the same team use the same packages. And not just the same packages, but the same</a> <a data-t="520">versions of those packages, so that code that I write will work for you and vice versa.</a></p>

<p><a data-t="526">Now that we have a way to begin to huddle up, we need to help our teams figure out what to huddle</a> <a data-t="532">around. They're looking to us to give them guidance about, you know, what packages do I use?</a> <a data-t="541">So, if we think back to our frozen environments, those centralized environments and all of that</a> <a data-t="546">vetting that happens to decide what packages go in there, that vetting results in a subset of</a> <a data-t="555">packages that are deemed to be accurate and reliable. If that decision has been made on</a> <a data-t="563">an organizational level or department level, we can leverage that and get those packages into those,</a> <a data-t="569">in to be used by that team, and that is their agreed upon set of packages. They can also do</a> <a data-t="576">that legwork themselves. In either case, a vetting step will lead to this subset of packages that the</a> <a data-t="582">team aligns on, and it will help to kick off that huddle.</a></p>

<p><a data-t="589">Also, through that vetting process, we can be sure that each one of these packages is going to work well individually. But with all</a> <a data-t="602">of the moving parts in open source, we want to also be sure that all these packages will be</a> <a data-t="609">compatible with one another. In our case, most of our packages are coming from CRAN.</a> <a data-t="610">So, when a package gets released to CRAN, all of its checks and tests get run automatically,</a> <a data-t="617">and because everything on CRAN is so interconnected, there's also, as a byproduct,</a> <a data-t="623">any other package that's connected into that package some way is checked for consistency</a> <a data-t="628">and compatibility. So, for, you know, everything that's up to date on CRAN as of today is compatible</a> <a data-t="636">together.</a></p>

<p><a data-t="646">Posit Package Manager is a way for us to access different points in time through what's</a> <a data-t="652">called snapshots. <p><a data-t="695">Now, we're still, our teams are still working in the open R environment as part of this paradigm,</a> <a data-t="701">right? So, we're not blocking access to things. <h3>Dynamic huddles: adding packages and time travel</h3>

<p><a data-t="759">But sometimes, teams need to move around a little bit. You know, after all, huddles are dynamic.</a> <a data-t="767">They are responsive to changes in the environment. And projects can be long, and people can change</a> <a data-t="774">their minds about things. Through Slushy, we've given them the ability to make sure that that package can get</a> <a data-t="799">added in to their environment through this function here, Slushy add. And by adding it in</a> <a data-t="807">officially, it makes sure that that package is okay to use. It's no longer going to get flagged</a> <a data-t="813">by that previous monitoring step, and they're able to expand things a little bit.</a></p>

<p><a data-t="820">Another place people might want to move is through time. They commonly do. They've started</a> <a data-t="827">at a snapshot, and, you know, a month later, two months later, things are moving ahead. They want</a> <a data-t="832">access to new features, understandably. <p><a data-t="894">So, making the leap from snapshot to snapshot can be a little bit intimidating. Sometimes it's risky. <h3>Simplifying the experience for users</h3>

<p><a data-t="955">So, it's a lot of different pieces, and we think back to our huddles. Penguins don't read a manual</a> <a data-t="967">on how to huddle. They just know how to do it from the time they're little babies, and we want the</a> <a data-t="971">same experience for our users. We want them to have success and feel like it's easy to get there,</a> <a data-t="978">and feel like they can just focus on their jobs, get their programming done,</a> <a data-t="982">enjoy the process of learning R.</a></p>

<p><a data-t="988">So, one way to simplify things for them is, of course, by creating an R package,</a> <a data-t="994">and we try to be really intentional about, you know, what is the minimum number of steps that</a> <a data-t="1000">our team needs to make and try to really simplify the functionality to reduce the burden on them</a> <a data-t="1006">so they don't have to think too much about all this stuff.</a></p>

<p><a data-t="1010">Another way we try to simplify things is through our training and recommendations about how to use</a> <a data-t="1016">Slushy. So, for instance, instead of allowing or recommending everyone on the team can adjust the</a> <a data-t="1025">environment, we assign that to one or two individuals who are responsible for initializing</a> <a data-t="1034">and maintaining the environment, and everyone else just follows suit, and they don't have to</a> <a data-t="1039">think too much. They just have their library in place, and they can start using it. This decreases</a> <a data-t="1046">the amount of, you know, the learning curve, of course, and the burden on the teams, but it also</a> <a data-t="1054">is going to decrease the risk that that environment might change in some way by somebody who's just</a> <a data-t="1061">trying something out.</a></p>

<p><a data-t="1072">Another way we've tried to simplify is by creating a </a><a href="/software/config/">config</a><a data-t="1072"> file that kind of</a> <a data-t="1079">wraps up all of our settings, and Slushy will read that automatically. So, what this allows,</a> <a data-t="1086">by putting a bunch of settings in a config file, an organization or a team can make</a> <a data-t="1094">a standard config file that they can apply to all of their projects. So, this, you know, taking all</a> <a data-t="1101">of these settings out of Slushy, the teams don't need to be concerned about it. We've done that for</a> <a data-t="1107">them, and through this config file, people can, or we can customize things like the package manager</a> <a data-t="1114">URL, whether or not they should be using a cache, whether there's a cache available, that really</a> <a data-t="1121">makes it faster for the teams to get all of their packages in place, since everyone is installing</a> <a data-t="1128">the same things one after another. They can do all sorts of renv customizations in terms of</a> <a data-t="1135">those settings, special code that should be run before and after Slushy is initialized</a> <a data-t="1143">at the start of each session, and perhaps most importantly is the list of approved packages</a> <a data-t="1150">that should be used from project to project, or the list of agreed upon packages.</a></p>

<p><a data-t="1157">So, again, teams don't need to ask that question of what packages should I use? We can just kind of</a> <a data-t="1162">get that information to them, or they can make that source of truth themselves and have that standardized.</a></p>
