<p><a data-t="378">So we're all data scientists here today.</a> <a data-t="381">And in this talk, we have our data science hat on, and we're looking at our data.</a> <a data-t="385">We can kind of do a visual, you know, exploratory data analysis.</a> <a data-t="389">We see that the data is kind of all trending up and to the right.</a> <a data-t="398">So we've done our bat, we understand our data, we can do some fake feature engineering looking at our data points here.</a> <a data-t="406">And so now we're at our step where we're deciding which model to use.</a></p>

<h3>Limits of explainability</h3>

<p><a data-t="721">So to recap up until now, we know that machine learning models have a different goal than humans.</a> <a data-t="728">We know that explainability comes after thoughtfully engineered models, and we know that we need explainability because of the increasing complexity of models and the increasing use of black box models.</a> <a data-t="743">So I'm not an explainability salesperson, even though it might sound like it, and I also want to make sure that you guys understand or you all understand that you do not need these every time you create a model.</a></p>

<p><a data-t="757">First of all, not all models need explainability.</a> <a data-t="761">Some models are simple.</a> <a data-t="762">We saw this a few slides back in Y equals MX plus B. We don't need to add a complicated computational effort if it's just not needed.</a> <a data-t="773">If we have really robust feature engineering or if we have very in-depth subject matter expertise or if we just have a model that we can fully explain, it's not worth our time.</a></p>

<p><a data-t="786">There's another limitation in use cases.</a> <a data-t="789">Not all use cases need explainability.</a> <a data-t="792">In all honesty, sometimes you just don't care.</a> <a data-t="798">This is most noticeable in some types of forecasting or times where the output just isn't really important enough to justify the added complexity of these algorithms.</a> <a data-t="809">Additionally, there's some use cases where the problem is very well studied and feature interactions are just overall really well understood.</a> <a data-t="819">And so explainability isn't needed there.</a></p>

<p><a data-t="824">Finally, you need to set very healthy boundaries with your explainability algorithms.</a> <a data-t="831">Explanations help when you need to justify how a model came to a prediction.</a> <a data-t="837">Explanations can expose bias.</a> <a data-t="841">Explanations cannot fix biased models.</a> <a data-t="845">They do not make changes to the underlying model or data.</a> <a data-t="850">So you need to really remember what explainability can and cannot do for you.</a></p>

<h3>Global and local explainability algorithms</h3>

<p><a data-t="990">My next step in this problem is to look at the contending algorithms for explaining the prediction of these infections.</a> <a data-t="1000">I do have to claim that all the examples I use in the next slides are fictitious and completely hypothetical.</a> <a data-t="1007">But I want to look at this model at a few different levels.</a> <a data-t="1011">And the first level is at a global scope.</a> <a data-t="1016">This is kind of your 5,000 foot view of your model, understanding how the model itself is making decisions.</a> <a data-t="1024">It can answer questions such as which features are important and what's the interaction between these features.</a></p>

<p><a data-t="1037">Next we have our local scope.</a> <a data-t="1044">So this level of explainability is all about understanding how the model has made a decision for a particular instance.</a> <a data-t="1054">There are many more local explainers than global explainers.</a> <a data-t="1059">And that's partially because global explainers are very computationally complex and therefore very time consuming and very expensive to run.</a> <a data-t="1070">The lack of global explainers also exists because not all instances have the same weights for each feature.</a> <a data-t="1077">So we can't generalize individual explanations to the whole model.</a></p>

<p><a data-t="1085">So maybe a clearer way of saying this is in our healthcare frame of mind, not all patients come in with the same symptoms.</a> <a data-t="1094">You wouldn't want to generalize the symptoms of one patient to the entire hospital since you would lose your ability to have a unique and accurate diagnosis.</a> <a data-t="1103">The same idea applies here.</a> <a data-t="1105">You want to diagnose a particular instance and hope that looking at the smaller area will give better results and explanations to other similar data points that you have.</a> <a data-t="1119">Local explainers can answer questions such as, you know, which features factored most heavily into this classification?</a> <a data-t="1127">And how would the prediction change if certain features changed?</a></p>

<p><a data-t="1137">So I'll first look at some or a global explainability algorithm.</a> <a data-t="1144">As a reminder, this is your super high-level overview of how your model is giving predictions.</a> <a data-t="1148">And the first algorithm we're looking at is called Shapley.</a> <a data-t="1158">And the goal of Shapley is to explain the or I guess contribute the contribution of each feature to the model.</a> <a data-t="1178">In the very simplest terms, Shapley tells me what inputs are important and how important they are.</a></p>

<p><a data-t="1184">So there's some pros and cons to Shapley.</a> <a data-t="1189">And the first pro that is really important is it's one of the few algorithms that can be used globally.</a> <a data-t="1196">Again, not many algorithms are able to be global.</a> <a data-t="1201">There is kind of like a hack because tree-based models are kind of fast.</a> <a data-t="1205">But if you're not wandering through a random forest, it's going to take a long time to train this.</a> <a data-t="1212">So I could see myself using Shapley to ask questions to my model such as, what are the most important features my model is using to predict MRSA?</a> <a data-t="1222">Would it be age, location of admission, or some sort of specific diagnosis?</a></p>

<p><a data-t="1231">So if I'm looking for a less computationally expensive explainability algorithm, I might be looking into more local algorithms.</a> <a data-t="1243">And we'll look at a few different algorithms, but we're going to start at, and surprise, yay, Shapley values can also be used locally.</a> <a data-t="1253">The benefit of this approach is that local versions of this algorithm are a lot quicker.</a> <a data-t="1259">But the negative side, hopefully, is that this is the number one favorite explainer for our evil data scientists.</a> <a data-t="1270">Research has shown that it is possible to hide bias in Shapley with adversarial attacks.</a> <a data-t="1280">This really isn't a big concern for those who are attempting to use this algorithm truthfully, but it could cause mistrust with interpreters of Shap explanations.</a> <a data-t="1291">You'd always have to walk into looking at a Shapley explanation with the possibility that it's fake in the back of your head.</a></p>

<p><a data-t="1301">So whereas for the global Shapley explainer, I would see what features were most important for the model in general, or like the hospital in general, I can use local Shap to see what features were most important for a specific patient to be flagged as someone who would likely become MRSA positive.</a></p>

<p><a data-t="1332">And again, Shapley is still kind of expensive to run.</a> <a data-t="1336">It's still computationally robust.</a> <a data-t="1341">So for those looking for a less computationally expensive technique, local interpretable model agnostic explanations, and that is such a mouthful, you understand why they shorten it to LIME,</a> <a data-t="1354">is an algorithm where you assume linearity around a particular instance and create a simpler model.</a> <a data-t="1361">The key intuition behind this one is that it's much easier to approximate a black box model by using a simple model locally in the neighborhood of the prediction we want to explain as opposed to trying to approximate the entire model globally.</a></p>

<p><a data-t="1381">So it creates a local version of a linear model because Y equals MX plus B is the gift that your middle school teacher just keeps giving.</a> <a data-t="1391">In this case, it's our middle purple line.</a> <a data-t="1394">We might not have any idea of the equation for our red line, but we understand and can easily interpret Y equals 5X plus 1.</a> <a data-t="1405">And this is really a great moment for local explainability to be able to interpret this instance.</a> <a data-t="1412">LIME is also model agnostic, so it doesn't matter what your underlying model is.</a> <a data-t="1416">As long as you have a prediction function, LIME can be used.</a></p>

<p><a data-t="1422">However, there is some danger in using LIME because you don't know how far this explanation holds.</a> <a data-t="1430">We can see that there's endpoints in this image, but the fit is not always as straightforward as this graph may lead you to believe.</a> <a data-t="1440">There is fear that you can be overconfident in your explanations in LIME, and you could have misleading conclusions for unseen but similar instances.</a></p>

<p><a data-t="1448">Anchors actually work a lot like LIME.</a> <a data-t="1451">They both proxy local behavior of the model in a linear way.</a> <a data-t="1456">But remember that LIME broke because we didn't know to what extent the explanation held up.</a> <a data-t="1463">And anchor kind of steps up to the plate and incorporates coverage.</a> <a data-t="1468">I like to think of this coverage as going from a one-dimensional line in LIME to a two-dimensional area, which we can see since our small purple line has turned into a more robust rectangle that's implementing coverage.</a> <a data-t="1486">Anchors explain individual predictions of any black box classification model by finding a decision rule or set of features or range of features that anchors the prediction sufficiently.</a></p>

<p><a data-t="1502">Anchors main selling point is that they address the shortcomings of LIME.</a> <a data-t="1508">But the boundaries are still approximate and occasionally don't even exist, in which case anchors become no different than LIME, and they still carry the same pros and cons.</a> <a data-t="1520">In the context of my MRSA exploration, I could see myself using anchors to see what ranges of different features factor into a MRSA positive prediction.</a> <a data-t="1533">For example, an anchor might be able to tell me that the output is like ages 20 to 30 are at a 20% risk of contraction.</a></p>

<p><a data-t="1540">So LIME, SHAP, and anchors all look at what features are important and look at interpretable models for a particular data point.</a> <a data-t="1549">But contrastive explanation methods kind of shift gears and focus more on cause and effect.</a> <a data-t="1555">So you can test how your model output can be changed for each instance.</a> <a data-t="1561">Contrastive explanation methods focus on explaining instances in terms of pertinent positives and pertinent negatives.</a> <a data-t="1569">Pertinent positives refer to features that should be minimally and sufficiently present to predict the same class as the original instance.</a> <a data-t="1579">And pertinent negatives identify which features should be minimally and necessarily absent from the instance in order to maintain the original prediction class.</a></p>

<p><a data-t="1593">Kind of a mouthful, but humans really think in contrastive explanation methods naturally.</a> <a data-t="1597">If I was trying to get my partner to locate me at a restaurant, I might say that I have a hat on, but I'm not wearing red.</a> <a data-t="1606">If I wanted to have an explanation for this graph, my CEM algorithm would tell me that the star is light purple because Y is greater than 10, but X is less than 12.</a></p>

<p><a data-t="1636">While CEM is useful for human understanding, they get less useful when the classes aren't similar.</a> <a data-t="1643">And this is kind of intuitive.</a> <a data-t="1645">You can distinguish clear yet subtle differences between a cat and a dog, but the differences between a cat and the flu are so numerous that it doesn't even seem logical to list them all out.</a> <a data-t="1659">If my CEM output is that all people who contract MRSA are admitted through the emergency room, but did not show signs of respiratory distress, the location would be a pertinent positive and symptoms would be a pertinent negative.</a></p>

<p><a data-t="1684">Counterfactual explanation is the minimum possible change required to generate the desired output.</a> <a data-t="1690">So kind of think if X had not occurred, Y would not have occurred.</a> <a data-t="1696">In our example here, if I wanted to change my light purple classification to dark purple, the star would have had to move down at least that set amount.</a> <a data-t="1707">On the positive side, counterfactuals don't require access to the data or even the model.</a> <a data-t="1712">They only need the predict function to generate outputs.</a> <a data-t="1716">But the cons for counterfactuals are similar to CEMs.</a> <a data-t="1720">Oftentimes there are so many ways to change the output.</a> <a data-t="1724">In fact, there's so many ways to change the output that it's no longer useful to list all of them out.</a> <a data-t="1732">A counterfactual that I might uncover with a patient is maybe someone who was admitted through the emergency room would not have acquired an infection if they had entered through urgent care instead.</a></p>

<p><a data-t="2072">So to recap, models are complex.</a> <a data-t="2075">Explainable machine learning is necessary to close the gap between the machine goal of output and the human goal of understanding.</a> <a data-t="2083">And we need explainability when it's not enough to just get the prediction.</a> <a data-t="2087">The model must also justify how it came to the prediction.</a> <a data-t="2093">Explainability helps us understand, though, why of models.</a></p>

<p><a data-t="2098">To recap, black boxes are models where you input data and some magic occurs and you get an output.</a> <a data-t="2105">They're really great for speed running a data science workflow and making really high performing models.</a> <a data-t="2111">But they're less great when you're trying to understand what's happening in the some magic occurs portion of the black box.</a> <a data-t="2120">Explainability algorithms help crack open the black box and peer in to see how decisions are being made.</a></p>

<p><a data-t="2129">To recap, we have algorithms that focus on feature importance.</a> <a data-t="2133">Such as Shapley, which gets a gold star for being able to aggregate feature importance both locally and globally.</a> <a data-t="2141">We have Lime, which creates a simple and easily understood y equals mx plus b to explain a local instance.</a> <a data-t="2150">And anchors, which does everything Lime does, but better.</a> <a data-t="2156">To recap, we also learned about algorithms that take a look at the cause and effect view on a model.</a> <a data-t="2162">Such as contrastive explanation methods and counterfactuals.</a></p>
