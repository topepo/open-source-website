<p><a data-t="100">So I'm going to tell you how can you raise an R microservice to thrive in the wild.</a> <a data-t="107">So here we have our R microservice, this cute little giraffe.</a> <a data-t="113">So the first thing we're going to do is we're going to put our R microservice through school.</a> <a data-t="117">We're going to teach it what to do once we release it out into the wild.</a> <a data-t="121">The end result of that is that once we release it into the wild, it can interact with other microservices, other applications written in different languages seamlessly and just have a fun time.</a></p>

<p><a data-t="133">So let's talk a little bit more concretely.</a> <a data-t="135">What is an R microservice?</a> <a data-t="138">So it's basically R code behind </a><a href="/software/plumber/">plumber</a><a data-t="138"> and Docker.</a> <a data-t="143">So we take our R code, our machine learning model, and we put it behind a plumber API.</a> <a data-t="148">This allows any other application to hit our machine learning model, and it only has to provide the expected input, and it gets an output.</a> <a data-t="157">So any other service doesn't need to know anything about the R language.</a></p>

<p><a data-t="163">And then we wrap around that plumber API.</a> <a data-t="167">We put it in a Docker package.</a> <a data-t="170">This makes our API portable.</a> <a data-t="172">So once it's in a Docker package or a Docker image, we can deploy it into the cloud in a variety of destinations.</a></p>

<h3>DevOps principles for robust deployments</h3>

<p><a data-t="241">So in pursuit of building robust R deployments, I want to introduce three DevOps principles.</a> <a data-t="248">The first is continuous integration, and that's where we build and test our packages.</a> <a data-t="254">The second is continuous deployment.</a> <a data-t="257">This is where we take our tested packages and release them into production systems.</a> <a data-t="262">And then the third is cluster computing, which is a common destination for applications in production in the cloud.</a></p>

<h3>Continuous integration</h3>

<p><a data-t="273">So let's start with continuous integration.</a> <a data-t="277">This is a simple continuous integration workflow.</a> <a data-t="280">So we start at the top left, where a data scientist is working in R, in </a><a href="/software/rstudio/">RStudio</a><a data-t="280">, making updates to the model, making updates to the code.</a> <a data-t="288">So whenever a data scientist makes an update to the code base, they will push that to a version tracking tool such as GitHub.</a> <a data-t="298">At that point, the application will be built from the source code, and then we're going to run automated tests on that.</a></p>

<p><a data-t="306">And this happens automatically.</a> <a data-t="308">Once I push, all that happens, and then I'll get a report back.</a> <a data-t="312">If the tests fail, I know I have to go back and fix something.</a> <a data-t="316">If the tests pass, I can then progress.</a> <a data-t="319">I can merge my changes into the master brand, and then we will have an application that is ready for deployment.</a></p>

<p><a data-t="331">So let's look a little bit closer into this first part of the CI pipeline.</a> <a data-t="337">So in the CI build step, basically we have data scientists working in RStudio, writing R code, and we are going to git push, and that's going to trigger GitHub actions to kick off the continuous integration pipeline.</a> <a data-t="352">There's many options for CI tools.</a> <a data-t="355">We use GitHub actions at DV01, and then the GitHub actions is going to build a Docker image based on the updated code, and what's important here is that I'm pushing my code frequently.</a></p>

<p><a data-t="370">That means that I can get instant feedback on what each piece of code that I edit, and so I know where changes happen.</a> <a data-t="377">If I push 500 lines of code at once and something fails, it's going to take me a long time to figure out what goes wrong, but if I push a couple of lines at a time, I can isolate the problems very easily.</a></p>

<h3>Automated testing pyramid</h3>

<p><a data-t="391">So now let's look at this next step.</a> <a data-t="393">We're running the automated tests.</a> <a data-t="396">So within continuous integration tests, we have three broad categories of tests.</a> <a data-t="403">Sort of there's this pyramid structure.</a> <a data-t="405">The base are unit tests.</a> <a data-t="407">These are the most numerous tests that we have.</a> <a data-t="410">Of those, we have integration tests, which are still a good amount of those, but less than unit tests, and finally, there are end-to-end tests, and the test that package is very useful for writing these types of tests.</a></p>

<p><a data-t="431">So starting at unit tests, this is the base of our testing pyramid, and the goal here is to test functions, individual lines of code, and really look at each component of your R package.</a> <a data-t="448">So here I have a trivial example of a unit test, where I define this add 20 function, and then I use test that to assert that if I add 20 to 10, it should equal 30, and I define this, and this is going to run every time I push updates to my package, so if I were to make any changes to this function that would break it, I would get immediate feedback, and these type of tests are something that you should have a lot of.</a></p>

<p><a data-t="468">For example, it's a really good idea to create unit tests for your features to make sure that all the features that go into your models have tests and never break or never give unexpected input.</a> <a data-t="482">If you're imputing missing values at prediction time, that would be a good thing to unit test.</a> <a data-t="490">You could also test data structures, like make sure that functions return data frames of expected structures and things like that, so writing a lot of these unit tests allows you to make sure that your code doesn't break in unexpected ways, and it helps you identify failures.</a></p>

<p><a data-t="510">So the next level is integration tests.</a> <a data-t="512">So for integration tests, the goal is to make sure that our microservice can interact with other services that it will need to, so we may be interacting with databases or APIs, so we're going to write tests to make sure that our R container can interact with every service it needs.</a></p>

<p><a data-t="532">And finally, there are end-to-end tests, and this is where we're looking at the whole microservices application as a whole, so this is, in our case, looking at our whole web app.</a> <a data-t="543">It doesn't really tell us much about our R microservice, but it's necessary to have a holistic view of everything.</a> <a data-t="549">Where we actually get useful input about our R package are the two previous categories of tests.</a></p>

<p><a data-t="558">Now when we're writing tests, we want to write them at the right time.</a> <a data-t="562">If you start writing tests too early, you can waste a lot of time, but they save a lot of time in production.</a> <a data-t="567">So if you start a new project, when you're in the exploratory phase, doing EDA, initial experiments, if you start writing a lot of tests, you're going to slow yourself down.</a> <a data-t="577">But as you progress, as you get your feature set more concrete, as you get closer to production, you should start introducing first unit tests, later integration tests, and once you're really close to production, to releasing a microservice, you're going to want to make sure that the end-to-end tests are properly configured.</a></p>

<h3>Continuous deployment</h3>

<p><a data-t="600">So next we're going to talk about continuous deployment.</a> <a data-t="603">At the end of the continuous integration pipeline, we have our container that's ready for deployment.</a> <a data-t="611">What is going to happen here is that this pipeline is going to take that container and it's going to push it through our different environment.</a> <a data-t="619">Here at DV01, we have a staging, release, and production environment.</a> <a data-t="624">There's many ways to do continuous deployment, but this is just one example.</a></p>

<p><a data-t="629">So once the container is ready, it's going to be pushed to staging, and in staging, there will be some end-to-end tests that run, and we'll just make sure that the whole application looks good as a whole.</a> <a data-t="643">If that goes well, it moves on to release, and then finally it will go to production.</a> <a data-t="649">And this is important because we have a customer-facing application, and we have zero tolerance for any failures or for downtime.</a> <a data-t="658">So we really need to make sure that we have a lot of steps in place to catch any errors that might come up.</a></p>

<blockquote><a data-t="649">And this is important because we have a customer-facing application, and we have zero tolerance for any failures or for downtime.</a></blockquote>

<p><a data-t="665">And each of these environments, the staging, release, and production, is a full replica of the production environment.</a> <a data-t="671">It has our web application, it has the R microservice, as well as all the other services that make up the application.</a></p>

<h3>Cluster computing with Kubernetes</h3>

<p><a data-t="682">So let's talk a little bit about the last concept, which is cluster computing.</a> <a data-t="688">So in the cloud, a common way to run jobs is in Kubernetes.</a> <a data-t="692">So here we have our production cluster, in which we have three replicas of our machine learning application.</a> <a data-t="699">So it's important that we're not just running it once, we're running it several times so we have redundancy.</a></p>

<p><a data-t="706">So when another microservice comes along and says, hey, R, give me some predictions, it's going to first go to a load balancer.</a> <a data-t="715">The load balancer is going to look at all the replicas of our R machine learning microservice, and it's going to route it to one that is free and that is error-free.</a> <a data-t="726">And then the R microservice will respond back.</a> <a data-t="730">And then the full microservice application can continue doing its work.</a></p>

<p><a data-t="739">And so what's important here, as I mentioned, this is a replica set.</a> <a data-t="742">So in Kubernetes, you have replica sets, which basically says that for this application, you're going to have always going to have three instances of this application in this case.</a> <a data-t="756">And if one of the replicas goes bad, it will be torn down, and a new one will be spun up immediately.</a> <a data-t="763">If we're getting high traffic, we can route new traffic to a free replica of our microservice.</a> <a data-t="773">So it's very important.</a> <a data-t="774">It gives us redundancy, and it allows us to handle high request loads.</a> <a data-t="780">This also allows scaling.</a></p>
