<p><a data-t="703">And I think another thing kind of around</a> <a data-t="705">this kind of pull request kind of approach</a> <a data-t="707">as well is I've found</a> <a data-t="709">that coming from a background where I've been working</a> <a data-t="711">in kind of consultancy and a bit of software engineering</a> <a data-t="713">on proprietary projects,</a> <a data-t="715">working on open source libraries</a> <a data-t="717">is a completely different thing</a> <a data-t="719">in some really subtle ways.</a> <a data-t="721">And this is something that I summarize</a> <a data-t="723">as a task-oriented versus code-oriented approach.</a> <a data-t="725">So when I was a consultant,</a> <a data-t="727">I would be working</a> <a data-t="729">building things with libraries.</a> <a data-t="731">So the focus there is getting tasks done.</a> <a data-t="733">You might be building an app for a user</a> <a data-t="735">that needs to do a certain thing as part of their day job.</a> <a data-t="738">You might have a known set of users</a> <a data-t="740">or at least some of them you could maybe go</a> <a data-t="742">sit down with them and see how they'd normally do that task.</a> <a data-t="744">And it makes it a bit easier really</a> <a data-t="746">to kind of see what they're likely to do</a> <a data-t="748">and what they're not likely to do.</a> <a data-t="750">And you can kind of get away with only testing</a> <a data-t="752">kind of like certain paths.</a></p>

<p><a data-t="790">So I think sometimes</a> <a data-t="792">it can be easy to say, right,</a> <a data-t="794">let's just use this other one or a bit from both.</a> <a data-t="796">But when you're working on kind of open source libraries,</a> <a data-t="798">you kind of can't do that.</a> <a data-t="800">But that forces you then</a> <a data-t="802">to learn to work with the tools that you've got in front of you</a> <a data-t="804">and learn that kind of last 10%</a> <a data-t="806">of a library to get tasks done.</a> <a data-t="808">And I think, again, that's been something</a> <a data-t="810">that's been really valuable for me.</a> <a data-t="812">And also just this idea that you're writing code</a> <a data-t="814">that isn't kind of your code.</a> <a data-t="816">This is to be maintained by kind of maybe yourself</a> <a data-t="818">but other people in the community.</a> <a data-t="820">And that means you can't have</a> <a data-t="822">these maybe weird idiosyncrasies</a> <a data-t="824">or kind of code smells.</a> <a data-t="826">You've got to be a bit more strict</a> <a data-t="828">about these kind of things.</a> <a data-t="830">And I think just kind of realizing</a> <a data-t="832">that it's a really different kind of mindset</a> <a data-t="834">between these two things</a> <a data-t="836">kind of makes it a bit easier to accept</a> <a data-t="838">what kind of might seem like pedantry sometimes</a> <a data-t="840">but it's actually for a really good purpose.</a></p>

<h3>Making meaningful contributions</h3>
