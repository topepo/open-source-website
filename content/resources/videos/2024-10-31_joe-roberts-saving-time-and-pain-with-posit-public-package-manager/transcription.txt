<h3>Common package pain points</h3>

<p><a data-t="135">So, let's talk first about what some of those pain points are.</a> <a data-t="138">Let's see if anyone can relate to any of these here.</a> <a data-t="138">How about the first one?</a> <a data-t="138">My code used to work, but now it doesn't.</a> <a data-t="142">Maybe you say, I wrote this code two years ago, but I got a new laptop.</a> <a data-t="147">I'm trying to, I saved my, I seem to get there, but I can't get the same results on my new laptop because something's changed.</a> <a data-t="157">Maybe I updated one package to get some new functionality, and everything broke, and now my whole project doesn't run.</a></p>

<p><a data-t="162">Why do these things happen?</a> <a data-t="165">Well, several reasons could be for the case there.</a> <a data-t="167">One, new versions of those packages may have changes that break old code.</a> <a data-t="171">Most CRAN R package authors are pretty good at keeping backwards compatibility, so you update to a new version, old code should still work, but it's not perfect,</a> <a data-t="180">and sometimes they intentionally break things to enable better functionality down the road.</a> <a data-t="185">Similarly, dependent packages may be out of date.</a> <a data-t="189">So, even though you updated one package, it may expect functions that are in newer versions of other packages that you aren't, that isn't aware of there.</a> <a data-t="199">Because CRAN itself only guarantees package compatibility between packages at any given latest point in time.</a></p>

<p><a data-t="204">So, that's, if you install all these packages at the exact same time from the same date, they should all work together right, and they're tested, they're validated, do that there.</a> <a data-t="216">But if you have them, one installed last week, one installed two months ago, and you're mixing and matching that way, you can get out of sync in ways that just break.</a> <a data-t="228">And third, what we see even more frequently on CRAN, sometimes CRAN packages just disappear.</a> <a data-t="234">They take them off, they get removed for whatever reason, and maybe you can't recreate that in the new.</a></p>

<p><a data-t="242">So, how we help with P3M.</a> <a data-t="247">So, what we do at Posit is we take daily snapshots of CRAN, we also do it for PyPI and Bioconductor,</a> <a data-t="255">and you can actually configure on your side then, R to install packages from a specific date,</a> <a data-t="260">ensuring that even if that date is not the today, like the like you can do by just getting the latest,</a> <a data-t="267">you can go back and say, I want all the packages from February 18th, 2023, because that's what I have this year, I can do that there.</a> <a data-t="277">And then, if you reinstall all those packages from that snapshot in your repository, at least all those packages will be the same, at least compatible with each other, as far as that goes there.</a></p>

<p><a data-t="284">So, you can use those, for example, to recreate the package environment for an old project from your old laptop that crashed.</a> <a data-t="291">As long as you know roughly when the project was written, you can get pretty close by doing it that way, usually.</a> <a data-t="298">But more importantly, you can also future-proof your work by tying it to today's snapshot and saying, I want to always use this snapshot, and I'll manually choose when I want to upgrade these later.</a> <a data-t="314">But that way, if you're sharing with someone else saying, here, here's my code, install all the packages from this snapshot so you have the same things that I built it on when you try to reproduce it yourself.</a></p>

<h3>Slow package installation</h3>

<p><a data-t="293">Next pain point, installing packages is really slow.</a> <a data-t="298">And we see this in certain environments.</a> <a data-t="300">You may recognize this if you are using Posit Workbench or some other Linux-based environment where it's just sent to be slower for whatever reasons we'll talk about.</a> <a data-t="308">Maybe you need to use an older R version.</a> <a data-t="310">Maybe for whatever reason you can't use R44, you need to use R36 even still, or 4.0, because whether it's compatibility, whether it's policy issues or whatever reasons you have in your environment, you still need to use older R.</a> <a data-t="324">Or maybe you're using Docker containers and deploying it in there, and your containers take forever to build every time you try to build them.</a></p>

<p><a data-t="331">So, there's several reasons why this happens.</a> <a data-t="336">One, installing packages from source is slow.</a> <a data-t="339">And most packages, so any source package is where how most are natively available on CRAN, and it may contain C, C++, Fortran code that has to be compiled, which takes some time.</a> <a data-t="349">Other packages, some packages require additional libraries in order to do that or build tools that make it hard.</a> <a data-t="352">You have to install all this extra software just to build the package.</a> <a data-t="356">Still, there's the concept of binary packages, which basically pre-compile all of that work for you.</a> <a data-t="360">And just, and when you go, then you can just download just your pre-built package that has everything ready to go, and then just unzip it and go.</a> <a data-t="369">And CRAN, unfortunately, only provides binary packages for Windows and Mac, and only on the current and previous version of R.</a></p>

<p><a data-t="376">So, if you're on Windows and Mac, you probably don't run into this specific problem very often, if you're on a relatively recent version of R.</a> <a data-t="383">But in different environments, different combinations, this still can be an issue.</a> <a data-t="388">So, again, how does package manager help?</a> <a data-t="392">We provide pre-built binaries for all of CRAN on not just the current and last release, but the current and five old releases.</a> <a data-t="400">So, we actually do provide binary packages for all the way back to R3-6 right now.</a> <a data-t="404">We do it for Windows and Mac and on 12 different Linux distributions.</a> <a data-t="408">So, this actually gives a, for like wherever you're running Workbench, most Docker containers are built on, you know, any number of these different Linux distributions that can really speed up the time here.</a> <a data-t="419">And it's also, it's also complete for a lot of the cloud compute environments, snowflakes, Databricks, because those are tend to be Linux-based containerized environments otherwise, where, you know, you're installing packages repetitively and it just, you're tired of it taking so long.</a></p>

<p><a data-t="432">So, let's talk about why, you know, when it takes, why this is so important here.</a> <a data-t="437">So, what we do, say, let's take a snapshot.</a> <a data-t="440">This is a snapshot of CRAN from today, let's say.</a> <a data-t="444">We, these are, there was, these were the new packages were added to CRAN today.</a> <a data-t="449">So, we every day, we'll look at CRAN, we will go, oh, well, we also then need to look at all the reverse dependencies of the packages, because in order to make sure that the binaries are compatible, we need to make sure that any package that depends on one of those packages is also rebuilt at the same time.</a> <a data-t="465">You know, reverse dependencies are, some have, most have a few, you know, you take, then you take something like </a><a href="/software/dplyr/">dplyr</a><a data-t="465"> that has 3,500 reverse dependencies.</a> <a data-t="473">So, that's a really big major, it's a packages that we have to start rebuilding every, when one package changes.</a></p>
