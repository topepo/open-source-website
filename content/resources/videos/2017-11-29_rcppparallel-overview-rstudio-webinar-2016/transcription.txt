<h3>Parallel computing in R</h3>

<p><a data-t="101">So you might be familiar with some of the packages for parallel computation with R.</a> <a data-t="109">For example, the parallel package which I believe became part of R with 2.15 around that era of R.</a> <a data-t="118">So R provides a number of packages that make it possible to run R code in parallel.</a> <a data-t="123">So you might know about mcapply, clusterapply, parapply, those family of functions.</a> <a data-t="129">And what these functions do is they allow you to take R code and run it in separate R processes.</a> <a data-t="136">These R processes running in parallel and then later collect the results from those processes back into a parent R process.</a></p>

<p><a data-t="144">So if you have a cluster with a number of nodes, you might imagine sending a number of jobs to these separate nodes.</a> <a data-t="151">And each node here would be running this R code computer result and then send that back to the parent node or parent process.</a> <a data-t="160">And there is a CRAN task view that enumerates the number of R packages that help support different forms of parallel computing with R.</a></p>

<p><a data-t="172">Now what RcppParallel does is a bit different from the R family of parallel packages.</a> <a data-t="179">The goal in RcppParallel is not to execute R code in parallel, but to allow R to call lower level C or C++ code that is executed in parallel.</a> <a data-t="190">So whether you're on, you know, say you've got a laptop with two or four cores or a desktop with four cores,</a> <a data-t="197">or you have some supercomputer that has, say, 32, 64 cores and a ton of memory,</a> <a data-t="204">the goal of RcppParallel is to put those cores to their best use for whatever algorithms you're trying to implement.</a></p>

<h3>The challenge of concurrent programming</h3>

<p><a data-t="213">Now, for those of you who haven't done much of this kind of lower level parallel or concurrent programming,</a> <a data-t="222">unfortunately it's quite hard and this is a silly quote from a fellow named Ned Batchelor.</a> <a data-t="229">Some people when confronted with a problem think, I know, I'll use threads.</a> <a data-t="233">Now they have two problems.</a></p>

<p><a data-t="236">And the idea being here you have, if you ever have two threads or multiple threads,</a> <a data-t="241">multiple processes trying to write to standard output, you can get this weird interleaving of output</a> <a data-t="248">where the stuff you get back is just garbage, totally garbled.</a> <a data-t="252">And that's kind of a common theme in parallel computing or concurrent programming.</a> <a data-t="257">The hardest part is managing shared resources.</a> <a data-t="261">In this case, the shared resource being the output stream that people are trying to write to,</a> <a data-t="266">or threads are trying to write to.</a> <a data-t="268">And when you have multiple threads trying to write to standard output without appropriate locking</a> <a data-t="275">or threads taking ownership to say, okay, it's my turn to write and then releasing and let other people write,</a> <a data-t="281">then you get these kinds of problems.</a></p>

<p><a data-t="284">So the goal of RcppParallel is to make it easier to write safe, correct, and perform in multi-threaded code</a> <a data-t="291">using C++ and Rcpp and use that within an R session.</a> <a data-t="296">And more than that, it's to make sure that you, the user, don't have to worry about</a> <a data-t="301">this issues with contention over shared resources.</a> <a data-t="306">For example, if you have an R object that you want to read or use for multiple processes,</a> <a data-t="312">we want to give you some tools that make it easy to safely and performantly use that</a> <a data-t="317">on all of your cores on your machine.</a></p>

<h3>Under the hood: Intel TBB and TinyThread</h3>

<p><a data-t="323">Alright, so the way that RcppParallel does what it does is that it bundles two C++ libraries</a> <a data-t="330">that do most of the grunt work for parallel computation.</a> <a data-t="335">So, the first one, Intel TBB, is a C++ library for task parallelism with a wide variety of parallel algorithms</a> <a data-t="344">and data structures.</a> <a data-t="345">And we've got this working on all of the major platforms, Windows, OSX, Linux, and even Solaris.</a> <a data-t="352">And so, in the most common case, you're going to be using Intel TBB threading building blocks</a> <a data-t="358">and this is the library that behind the scenes of RcppParallel is going to make sure that</a> <a data-t="364">when you designate a job that you want to run in parallel, it's going to do all the work of</a> <a data-t="372">spawning different threads and just making sure that your work runs in parallel and it runs fast.</a></p>

<p><a data-t="380">Now, we also bundle TinyThread which is a very small C++ library for the portable use of operating system threads.</a> <a data-t="387">But nowadays, because Intel TBB is supported on all the platforms that are commonly used by R users,</a> <a data-t="396">in general, you're going to be using Intel TBB.</a> <a data-t="399">Of course, as a user of RcppParallel, you don't need to dive into the internals of TBB</a> <a data-t="405">because we present a nice interface that I'll be showing soon.</a> <a data-t="408">But it's good to know, you know, who's doing the heavy lifting behind the scenes.</a></p>

<h3>parallelFor and parallelReduce</h3>

<p><a data-t="418">Alright, so RcppParallel tries to solve, I guess it tries to provide interfaces for two main functions,</a> <a data-t="426">for two common types of work that you might want to do in parallel.</a> <a data-t="430">The first function we provide is a function called parallelFor.</a> <a data-t="434">The idea is that this will convert the work of a standard serial for loop into a parallel one.</a> <a data-t="441">And the second one is parallelReduce which provides a way to compute and aggregate multiple values in parallel.</a> <a data-t="449">So, these are functions provided by RcppParallel.</a> <a data-t="453">And behind the scenes, they're going to delegate their work to the TBB Intel threading building blocks library.</a></p>

<p><a data-t="462">So, the way that these functions work is effectively that they accept this thing that I'm going to call a worker object.</a> <a data-t="469">And this worker is going to know how to do work on a particular slice of data.</a> <a data-t="474">And I'll get into examples very soon.</a></p>

<p><a data-t="480">Alright, so as a brief outline of how these functions work just to set the stage.</a> <a data-t="486">Each of these functions accept a range of data to operate on.</a> <a data-t="490">And a worker that understands how to do work on slices of that data.</a> <a data-t="495">For example, the parallelFor function.</a> <a data-t="497">And I should just reiterate that this is a C++ function, not an R function.</a> <a data-t="503">So, hopefully you have a bit of familiarity with C or C++.</a> <a data-t="507">But this is just the signature of a function called parallelFor.</a></p>

<p><a data-t="512">It's accepting a begin and an end index.</a> <a data-t="516">Here the size T parameter being like an index parameter.</a> <a data-t="520">And worker being the worker object I briefly discussed before.</a> <a data-t="525">So, you're going to give this parallelFor algorithm a range to operate on.</a> <a data-t="529">And a worker designation that knows how to do work within that range.</a></p>

<p><a data-t="537">So, as a user of RcppParallel, you can implement your own worker by subclassing this RcppParallel worker class.</a> <a data-t="545">And implementing the routines that do work on a slice of data.</a> <a data-t="550">And so after you write the code that does the work and you call parallelFor.</a> <a data-t="556">Behind the scenes, TBB is going to go ahead and generate multiple workers.</a> <a data-t="561">Divide them across your data and make sure that it's going to be responsible for making sure this worker runs in parallel.</a> <a data-t="568">Or more accurately that multiple workers are generated to operate on slices of the data.</a></p>
