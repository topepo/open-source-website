<h3>API authentication and security</h3>

<p><a data-t="2378">Authentication is really important with APIs. You want to make sure that the right people can have access to your API,</a> <a data-t="2392">both for asking for data, but also posting data. So, one thing is that when you create packages</a> <a data-t="2398">for accessing APIs that require authentication, make sure to never hard code your API keys,</a> <a data-t="2405">because someone could take them and impersonate you and possibly do bad, evil things. So, store</a> <a data-t="2411">them in external files, like on your computer or something, or in your R profile as a global</a> <a data-t="2416">variable. This is a nice safeguard. And also, when you publish to Git and things like that,</a> <a data-t="2421">then your keys won't go there, which is also good, too.</a></p>

<p><a data-t="2425">And for those who don't know, the HTTR package has a default place for you to put your authorization code in. It's like a hyperparameter called authorization. And the standard format, generally, is bearer, and then your API key.</a> <a data-t="2434">Also, be careful about public APIs. So, for this one, it's all fun and games.</a> <a data-t="2442">But you don't want people uploading or sending things to your environment that you don't want. So, always try to do as much error handling if you have a public API. And also, if you can do a private API, do that.</a></p>

<p><a data-t="2447">Posit Connect makes it really easy to create authenticated APIs. All you have to do is, just like on a normal thing or document you publish on Posit Connect, switch the sharing</a> <a data-t="2451">settings to all users logon require. And then folks can just use an API key in Posit Connect to authenticate.</a></p>

<p><a data-t="2461">So, I think that's about it. Hopefully, these examples kind of get people more excited about APIs. They're a really great way to both share data, take in data, and just kind of expand your data ecosystem.</a></p>
