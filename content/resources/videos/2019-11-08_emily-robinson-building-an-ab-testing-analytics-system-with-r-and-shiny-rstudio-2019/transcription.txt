<h3>What is A/B testing?</h3>

<p><a data-t="129">So just so this talk will make a little more sense, if you're not familiar with Datacamp, it's an online data science learning platform.</a> <a data-t="135">So there's a little over 200 courses now in R, Python, Shell, SQL, kind of all the big languages.</a> <a data-t="142">This is one of our R courses, and it's a, you know, way to learn interactively in the browser.</a> <a data-t="148">So you have short video lessons, and then you code.</a></p>

<p><a data-t="153">So let's say we have a user, Linda, and Linda logs on to Datacamp one day, and she's interested in learning deep learning.</a> <a data-t="162">So Linda could maybe see this page, and that's one of our course pages.</a> <a data-t="167">There's about a 50% chance of this if we're running an A-B test on it.</a> <a data-t="172">But she also could have seen this page.</a> <a data-t="173">So you might notice there are a couple differences here, right, the color, but a big one is that registration form that's now on the page.</a> <a data-t="179">So what A-B testing is is randomly assigning visitors to your website to one or two or more experiences.</a> <a data-t="186">Sometimes it involves changes on the page here.</a> <a data-t="189">Sometimes it's things like changes to a search algorithm or making the page faster.</a> <a data-t="193">But the idea is that by randomly assigning people and, you know, running these conditions at the same time, you're controlling for all the other factors.</a> <a data-t="201">So this lets you know that if you see a difference between the control and the treatment group and you've run everything properly, any behavioral difference should be because of your change.</a> <a data-t="209">And then you can be really confident it's had the impact you hoped it would.</a></p>

<h3>Lesson 1: Build tools to save yourself time</h3>

<p><a data-t="289">The first is build tools to save yourself time.</a> <a data-t="293">Who here has ever had a first this, then that question?</a> <a data-t="297">Something like who tried X, then did Y?</a> <a data-t="300">What percent of people who did X, then did Y?</a> <a data-t="303">What was the last thing people did before doing Y?</a> <a data-t="306">Or what were all the things people did after doing X, right?</a> <a data-t="309">So they can be, you know, structured differently.</a> <a data-t="311">Maybe you're interested in all the behavior afterwards, just the first time.</a> <a data-t="314">But really, it's you have two sets of behaviors, and you want to follow one user and understand what happened first this, then that.</a></p>

<p><a data-t="320">And this is the type of questions I would ask in A-B testing.</a> <a data-t="323">Say what percentage of the people in the treatment first control registered?</a> <a data-t="326">And this would be registered after starting the experiment.</a> <a data-t="329">So I'd have experiment starts and registration.</a> <a data-t="331">What were the ad clicks that had a course start within two days?</a> <a data-t="336">And my first attempt at doing this involved a lot of lengthy repetitive code.</a> <a data-t="340">There was a ton of copying and pasting, and it was very hard to switch between different types of funnels.</a> <a data-t="344">By which I mean switching between something like I want the first time they did this, and then the first time they did that.</a> <a data-t="348">I want the first time they did this, and everything they did after that.</a> <a data-t="353">It wasn't very easy to go between those types of funnels.</a></p>

<p><a data-t="356">And when you're doing repetitive tasks, you know, a lot of you may know here, it's time to write a function.</a> <a data-t="362">And in my case, a package.</a> <a data-t="365">Unfortunately, I faced a bit of an issue, which was this was me and writing packages.</a> <a data-t="371">I sort of knew this was something.</a> <a data-t="372">I even had written a blog post a couple years ago that was like, you know, writing my first R package, part one, where I talk about how I'm going to write this R package.</a> <a data-t="380">And there was never a part two.</a> <a data-t="382">There actually still is not.</a></p>

<p><a data-t="390">So you know, I was a little, you know, wasn't really sure what to do.</a> <a data-t="393">But fortunately, I had David Robinson at data camp.</a> <a data-t="396">Dave Robinson is our chief data scientist at data camp.</a> <a data-t="404">As I said, also my brother.</a> <a data-t="405">And he is very experienced in writing packages.</a> <a data-t="407">So one day you set up a hackathon for him, me, and one of the other growth team members at data camp to write the package funnel join.</a> <a data-t="416">And if you'd like to try it out yourself, it's available on GitHub.</a> <a data-t="419">And what funnel join is, it's goal is to make it easy to analyze behavior funnels.</a></p>

<p><a data-t="424">So I'm going to walk through how funnel join works.</a> <a data-t="428">The structure of it is you start with two tables.</a> <a data-t="430">The first table is your first set of behaviors.</a> <a data-t="433">And then the second table is your second set.</a> <a data-t="435">The one that you want to happen afterward.</a> <a data-t="437">You tell it what the user column names are, because you want to be able to connect the user across tables.</a> <a data-t="442">You tell it what the time columns are.</a> <a data-t="443">Again, you want to make sure the time in the table on the left-hand side always has to be less than the time on the right-hand side.</a> <a data-t="450">It has to happen before.</a> <a data-t="451">The type of after join, which I'm going to talk about a little bit more.</a> <a data-t="455">And finally, the type of join.</a> <a data-t="456">So like inner, left, right.</a> <a data-t="458">All the </a><a href="/software/dplyr/">dplyr</a><a data-t="458"> joins.</a></p>

<p><a data-t="460">So one example is a first any question.</a> <a data-t="463">What are all the courses people started after visiting the home page for the first time?</a> <a data-t="467">So here I would write my funnel join.</a> <a data-t="470">I write an after inner join, so using the type inner.</a> <a data-t="473">I specify in the type argument the first-any.</a> <a data-t="477">And then I have my time columns and my user columns.</a> <a data-t="479">And so I get out this table, and you'll see, just as we would hope, course started at is always greater than viewed at.</a> <a data-t="485">So now I could say, oh, person three viewed the home page on June 6.</a> <a data-t="491">And we have two entries for them, even for the same home page view, because they started two different courses.</a> <a data-t="495">And that's because I did a first-any join.</a></p>

<p><a data-t="498">But if I wanted to switch it up and say what percent of people saw the pricing page for the first time and then subscribed, I just changed the type of funnel.</a> <a data-t="505">So a very similar structure.</a> <a data-t="507">Change type to first, first after.</a> <a data-t="509">And now because it's a left join, we also have the NA at the subscribe dot.</a> <a data-t="514">What if I want to say, well, that's great, but I really want a tight funnel.</a> <a data-t="517">I want it to happen within four days.</a> <a data-t="520">I can add a max gap argument.</a> <a data-t="522">So I give it a diff time argument.</a> <a data-t="523">And now if I ran the same code, we see a bit more NAs on the subscribe dot, because even though those people subscribed, they didn't do it within four days.</a></p>

<p><a data-t="533">As I said, there are many funnel types supported.</a> <a data-t="535">So this last before, first after, any-any, first-any, supports all types of dplyr joins.</a> <a data-t="541">It works on remote tables, because it uses dplyr, so it takes advantage of </a><a href="/software/dbplyr/">dbplyr</a><a data-t="541">.</a> <a data-t="546">And bug fixes, pull requests, feature requests, welcome.</a> <a data-t="549">Please try it yourself and let me know how it goes.</a></p>

<h3>Lesson 4: Make it easy to do the right thing</h3>

<p><a data-t="849">And my final thing that I learned was to make it easy to do the right thing.</a> <a data-t="854">One best practice for A-B tests is to have a single key metric per experiment.</a> <a data-t="859">It helps clarify decision making.</a> <a data-t="860">Because if you have multiple metrics and one goes up and one goes down, you could be left not really knowing what you should do in that situation.</a> <a data-t="868">It also allows you to run that power analysis.</a> <a data-t="870">So for a power analysis to be run, you need to have the metric.</a> <a data-t="873">You need to know its rate.</a> <a data-t="874">And that can tell you how long to run the experiment for.</a></p>

<p><a data-t="878">So it's great.</a> <a data-t="879">And I really am doing a lot to educate people at Datacamp about A-B testing best practices.</a> <a data-t="884">But even more than that, building it into the process and making it easy for people.</a> <a data-t="889">So this is our </a><a href="/software/air/">air</a><a data-t="889"> table where we put the information about our experiments.</a> <a data-t="892">And you can see at the top here is a success metric.</a> <a data-t="894">And so each experiment is required to have one.</a> <a data-t="897">And only one.</a> <a data-t="898">We can write in the notes maybe other things we care about.</a> <a data-t="900">But this is the key metric.</a> <a data-t="903">And then we can show— you know, then that forces people to make a decision.</a> <a data-t="907">We can match that to something on the dashboard.</a> <a data-t="910">And it can help make decision making at the end a lot easier.</a></p>

<p><a data-t="914">Second guideline is to run the experiment for the length you've planned on.</a> <a data-t="917">This is a graph showing about how— if you simulate thousands of null experiments and you checked every day if the P value was less than .05 and you stopped your experiment at that point, you'd get an over 20% false positive rate.</a> <a data-t="932">So a huge increase.</a> <a data-t="934">You really want to do that power analysis, set the time of your experiment, and stick to it.</a> <a data-t="940">That's why we added to the health checks metric a start date and an end date for experiments.</a> <a data-t="944">Again, it could be really tempting for a stakeholder to message me, hey, I saw— I checked this dashboard on the individual experiment.</a> <a data-t="951">This metric went up.</a> <a data-t="952">Why don't we just go ahead and launch it?</a> <a data-t="954">But now they say, okay, I see the end date is not for another couple days.</a> <a data-t="958">I know I'm supposed to follow this best practice, so I will do it.</a></p>
