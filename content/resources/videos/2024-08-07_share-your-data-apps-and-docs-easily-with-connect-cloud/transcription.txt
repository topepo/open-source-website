<h3>The three-step publishing workflow</h3>

<p><a data-t="169">So when we were putting together Connect Cloud and acknowledging that we have </a><a href="/software/shinyapps/">shinyapps</a><a data-t="169">.io and rpubs and Quartopub, they all sort of have a very IDE-specific way that you're being able to deploy or publish these applications and documents.</a> <a data-t="186">We wanted to think of something that was unified across both R and Python and that wouldn't require a user to learn about a plugin we might have somewhere, to learn about a new package that they might need to install and work into their actual code potentially.</a></p>

<p><a data-t="201">So we came up with a sort of three simple steps for this alpha release.</a> <a data-t="207">And this obviously doesn't mean that this is going to be the only way that you can publish forever on Connect Cloud.</a> <a data-t="212">It's just the starting point that we have right now.</a> <a data-t="214">And what we tried to set off to do is you have three simple steps.</a> <a data-t="218">And our hope is that within three minutes, you're going to be able to create a new account, again, through this GitHub authentication.</a> <a data-t="225">You're going to be able to point the platform to one of your public repos with the source code for what it is you want to publish.</a> <a data-t="232">And then you're going to press go.</a></p>

<p><a data-t="234">And then if we do our job right, we're going to be able to take your code, your dependency files, be able to spin up a working URL that is public in nature that you can start sharing on LinkedIn or with friends, family, whoever it might be.</a> <a data-t="251">And what comes at the end of this process, as I mentioned, is the shareable public URL.</a> <a data-t="255">So this could be an example Quarto dashboard that was put together.</a> <a data-t="260">We're able to deploy it.</a> <a data-t="262">There's a URL associated with it.</a> <a data-t="264">And we'll get into this within the demo in terms of how you as the publisher might interact with some of this.</a></p>

<p><a data-t="270">One neat thing that we've done on Connect Cloud is this then ties back to a portfolio, in essence, a public portfolio of all the diverse type of work that you've put together.</a> <a data-t="285">So this specific thing that I might have published called economic trends is going to end up back on your portfolio or your profile.</a> <a data-t="292">I mean, you can see economic trends here.</a> <a data-t="294">This is the view for you, the user.</a> <a data-t="296">So you still have controls over things.</a> <a data-t="298">But if you were to share this URL out with others, they'd also be able to see what you're working on and find out a little bit more about you.</a></p>

<h3>Dependency files</h3>

<p><a data-t="308">A quick note before we get into the demo portion of today.</a> <a data-t="312">So one way that we need to consider the deployment or the publishing of what you're giving us is for us to know what the system needs to replicate what you did locally.</a> <a data-t="324">So the way that we're able to handle this GitHub workflow, we need a little bit of information about that environment.</a> <a data-t="331">And starting with our projects or our applications in the example that we'll look at today, we need this manifest.json file to be in that public repo.</a></p>

<p><a data-t="342">I know that many people who haven't worked within Connect before or maybe published from the command line to our other tools might not be familiar with this.</a> <a data-t="350">Thankfully, it just takes one function, write manifest.</a> <a data-t="353">You point to where that application, for instance, might be.</a> <a data-t="356">And it's going to give you this file that we'll look at in one of our examples.</a> <a data-t="359">And this is going to tell Connect Cloud the right packages that we should be installing and the right versions.</a> <a data-t="366">And on the R side of things, it's also going to tell us the R version that you were using locally that we're going to be able to spin up on Connect Cloud.</a></p>

<p><a data-t="376">If you're working on a Python application, here we're going to use a requirements.txt file.</a> <a data-t="381">So this is what's going to be required in that repo.</a> <a data-t="386">There are a couple of different ways that you can put together this file when working with Python.</a> <a data-t="390">One, you could just create a blank text file, requirements.txt, and add in your dependencies line by line.</a> <a data-t="397">So you need pandas, you need matplotlib, and you could leave it like that if you wanted to.</a> <a data-t="401">Another approach is using pip freeze from the terminal and then spitting this out into a requirements.txt file.</a> <a data-t="409">This will take all of the packages that you're using to run this thing locally and put that into a very specific requirements.txt file that we can take to upload.</a></p>

<p><a data-t="419">So one difference between the R side and the Python side at the moment is that this manifest file does both the R version and the packages.</a> <a data-t="428">The requirements file only is focused on packages, and you're going to set that Python version from the UI itself.</a></p>
