<h3>The package management prime directive</h3>

<p><a data-t="841">That said, there is a happy path.</a> <a data-t="845">It is possible to make this delicious sandwich, whatever it is.</a> <a data-t="849">And it's not even that hard.</a> <a data-t="853">But it's, you know, you're going to want to do some planning up front.</a> <a data-t="858">You're going to want to do your mise en place.</a> <a data-t="861">And so, we're going to talk for the rest of this talk about two ways that you can make this better.</a> <a data-t="868">And the first is to develop a package management plan with your IT department.</a> <a data-t="875">And the second is to have project specific libraries that you can manage as an individual data scientist.</a></p>

<p><a data-t="882">And so, from here, we're going to get to the package management prime directive, which is a little grandiose.</a> <a data-t="890">And so, the first is that, you know, repositories should be as broad as possible, right?</a> <a data-t="897">These should be managed centrally, if they're being managed at all.</a> <a data-t="902">And they should try and serve as many users as possible.</a></p>

<p><a data-t="911">That is the opposite from libraries, which are specific.</a> <a data-t="915">One other thing I should mention about libraries is that they are pull only, right?</a> <a data-t="920">You have to pull packages to your libraries.</a> <a data-t="923">And so, you know, that means that the data scientists should have control of what they pull into their own libraries.</a> <a data-t="933">This is sometimes a point of friction between IT teams and data scientists.</a> <a data-t="939">And it's worth sort of, you know, having a conversation with your IT team about how you can safely manage libraries as the data science team.</a></p>

<p><a data-t="953">So, on your repository, like, you don't necessarily need to manage a repository.</a> <a data-t="960">A lot of organizations choose not to manage a repository.</a> <a data-t="966">If you are going to manage one, that's generally on IT or admins.</a> <a data-t="972">And, you know, sometimes you have data scientists serving in this role of sort of the data science platform admin.</a> <a data-t="979">And that's fine.</a> <a data-t="980">But it's good to distinguish what you're doing as a data science platform admin versus as a data scientist.</a> <a data-t="989">And so, most often, we see IT admins managing a repository.</a> <a data-t="999">Because the job here is, you know, to stand up a server, to network it correctly, to make sure that it's available to users, to make sure that it stays up, has the appropriate amount of stability and security.</a></p>

<p><a data-t="1013">On the library side, that really falls to the data scientist to manage either a user level library or increasingly, and as I'm going to suggest to do here, a project level library, right?</a> <a data-t="1030">An individual library that corresponds to a single project is the safest path.</a> <a data-t="1037">Because then you know that even if you do other projects, you won't have to worry about breaking your project library.</a></p>
