<h3>What is computer system validation?</h3>

<p><a data-t="57">So before we get going, what is computer system validation?</a> <a data-t="57">So that is establishing documentary evidence that our software performs some sort of task, activity, process, in compliance with the specification we've laid out with a high degree of assurance.</a> <a data-t="68">So there's a couple of very important aspects to that.</a> <a data-t="70">It's documenting the evidence.</a> <a data-t="72">And then it's compliance with the specifications with a high degree of assurance.</a> <a data-t="76">So you are sure that it will continue to behave the way you expect it to.</a></p>

<p><a data-t="81">So part of the reason that we do validation is it's required for FDA submission.</a> <a data-t="85">There's the 21 CFR 11.</a> <a data-t="89">And there's also some general principles of software validation that was released by the FDA in 2002, giving us some guidelines about how we should be performing our validations.</a> <a data-t="97">But that isn't the only reason we should be doing validation.</a> <a data-t="100">Validation has a lot of other benefits as well, such as improved quality and safety with the products that we generate.</a> <a data-t="105">It results in faster processing, because we already have a system in place.</a> <a data-t="108">And it promotes trust with our outputs, because we've already tested it in a variety of inputs.</a></p>

<p><a data-t="113">So we already know that it's going to produce good results, or it'll error out if it doesn't.</a> <a data-t="120">Validation is very different from verification.</a> <a data-t="122">And often, they're confused.</a> <a data-t="123">So we're going to go over that a little bit too.</a> <a data-t="125">So validation is, does the system behave as we expect it to?</a> <a data-t="130">Will it return the correct value given the input?</a> <a data-t="132">Or will it throw an error if the values are outside of our expectations?</a> <a data-t="136">Verification on the other hand is simply do two separate processes on the same data set, for example, give us the same output.</a> <a data-t="143">So it's similar, but different.</a></p>

<p><a data-t="151">But for small companies, validation can be a high bar.</a> <a data-t="154">There's a lot of labor hours in validation.</a> <a data-t="156">And there's tracking the progress.</a> <a data-t="158">There's writing everything around that, and tracking the documentation, which can feel like a lot of work.</a> <a data-t="163">You're constantly filling out this paperwork that's outside of your coding.</a></p>

<p><a data-t="167">For validation, it feels a little excessive, because you first have to develop and get sign-off on your specifications.</a> <a data-t="172">Next, you're going to have to write the actual code following good programming practices and record who wrote which functions.</a> <a data-t="179">Then you have to document and develop your test cases and get sign-off on that.</a> <a data-t="183">Next, write your test code based on your test cases and record who wrote that.</a> <a data-t="188">Manually evaluate and document the test code results.</a> <a data-t="191">Once again, getting a third party to do that.</a> <a data-t="193">And finally, review all this documentation that you have and get final sign-off for release.</a> <a data-t="198">That's a lot of work.</a></p>

<h3>The five steps of validation</h3>

<p><a data-t="275">So here's the five steps that I would say are part of validation.</a> <a data-t="277">First you have to have your specifications.</a> <a data-t="279">Then you write the code.</a> <a data-t="279">Then you write your test cases.</a> <a data-t="280">Then you write the actual test case code.</a> <a data-t="282">And finally you document it all together.</a> <a data-t="284">I'm also assuming that you're going to be writing a package for this validation.</a></p>

<p><a data-t="293">So first, when you're working on your specifications, you're trying to lay out what are the requirements of your package.</a> <a data-t="300">What will the package do?</a> <a data-t="301">What are the expected outputs of your package?</a> <a data-t="304">And what are the high-level steps that are going to be followed?</a> <a data-t="306">So this isn't going into detail.</a> <a data-t="307">This is laying out the groundwork for everything.</a></p>

<p><a data-t="311">So if I were to be writing a specification for my presentation today, I'd say the contents of my presentation will cover my team's approach to validation.</a> <a data-t="319">It'll be roughly 15 to 20 minutes long, and hopefully it'll be entertaining.</a></p>

<p><a data-t="324">So when I go to record these specifications, I'm going to want to use rmarkdown to document all this for me.</a> <a data-t="331">I'm going to use a single file per specification and number those specifications within it.</a> <a data-t="337">In addition, I'm going to use </a><a href="/software/roxygen2/">roxygen2</a><a data-t="337"> to add details around who wrote that specification and when, and what the specification title is, and then I'm going to put this inside my R package.</a></p>

<p><a data-t="353">So the benefit of putting our specifications within our package and writing it in rmarkdown, it allows us to really quickly update the specifications if any of them change.</a> <a data-t="362">Additionally, we can change the ownership.</a> <a data-t="364">If somebody goes and updates the specifications, they are already right there for them to update.</a> <a data-t="369">By putting it inside the package, it's close to the task at hand, so it's really easy to reference.</a> <a data-t="374">You don't have to find a third-party document to pull that information in.</a> <a data-t="377">But also by separating out the specifications in their own files, it forces you to decouple your specifications so they're not reliant on one another, so you can make very clear specifications.</a></p>

<p><a data-t="385">These specifications are Markdown files.</a> <a data-t="387">We suggest putting them under the vignettes folder, creating a validation folder, and finally creating a folder for all your specifications and dropping all of them in there.</a></p>

<h3>Documenting code ownership</h3>

<p><a data-t="416">So typically, when you're documenting the code ownership, you're going to be using an external resource, such as Word, Excel, maybe Smartsheets, and this is really cumbersome because it's outside of your programming world.</a> <a data-t="427">You have to take a minute and step out of it and go and record, oh, I updated this function on this date, and here's what happened.</a> <a data-t="433">And that really just takes you out of their flow of programming.</a></p>

<p><a data-t="437">What I suggest doing is using the section tag in roxygen and documenting the last updated by and the last updated date.</a> <a data-t="445">So you can record who updated it and when, and it's right there within the function.</a> <a data-t="450">You can also do it at function creation.</a> <a data-t="452">The benefit of using these roxygen tags is it doesn't alter how it would behave when you're trying to compile your manuals, and it actually has the added benefit of recording it in the manual who wrote the function and when they last updated it, so you can track that.</a></p>

<p><a data-t="478">But now somebody came in and they updated my function.</a> <a data-t="481">They changed how long the function is going to wait.</a> <a data-t="484">And so this was not done by me, and it was done today.</a> <a data-t="489">But it is right there in the code.</a> <a data-t="490">They didn't have to step outside and remember to update it in the Smartsheet or the Excel file, but it's able to do it right there within the function and in the commit, it's tracked if you're using version control.</a></p>

<h3>Writing test cases</h3>

<p><a data-t="516">Now when you're writing your test cases, so test cases are where you're actually going to draw the connection between the code that you wrote and the actual specifications that you have and how to pair them together.</a> <a data-t="523">These are distinct from unit tests.</a> <a data-t="525">They're often confused with one another, but unit tests are for testing as the function author does your function behave the way you're expecting it to.</a> <a data-t="535">Test cases, on the other hand, are for documenting how you've matched up with your specifications.</a></p>

<p><a data-t="541">Similar but different.</a> <a data-t="541">A single test case can satisfy multiple specifications, conversely, a single specification must match to at least one test case, so you have to prove that each specification has been met.</a> <a data-t="554">So the way you go through and document your test cases is first you record which specification is going to be met, specify the required input data for your test case, record the steps you need to be following but without writing any code, then detail the expectations that they'll need to be testing to prove that you've met these specifications.</a> <a data-t="574">You'll use rmarkdown to perform this task, and you'll use roxygen2 to document all that additional information, such as who wrote that test case and when.</a></p>

<p><a data-t="583">So this is what a test case might look like.</a> <a data-t="585">Once again, for my presentation, I would first see that I met my specifications by gathering a captive audience of coworkers and probably later my significant other, give them the presentation and ask them, if it was informative, ask them what they learned.</a> <a data-t="601">I can then time my presentation to see if I was actually between the 15 and 20 minutes, and then I can see if I was entertaining by the number of eye rolls from my significant other or chuckles from my coworkers.</a></p>

<p><a data-t="613">By using this modular test case format, it makes it easy to shift and update your test cases so you can, as you get more information, you can change them or make sure that you're hitting all the specifications.</a> <a data-t="625">These will also live under the validation folder you created, but create a new folder called test cases.</a></p>

<h3>Test case coding</h3>

<p><a data-t="632">Now test case coding.</a> <a data-t="633">This is where you actually implement the new test cases that you just wrote in rmarkdown.</a> <a data-t="638">You're going to record the results of those test cases, and then you're going to have a third party write this code.</a> <a data-t="646">It's critical to have this third party do it, because they have no prior knowledge of the way the function should be behaving.</a> <a data-t="653">The benefits of this is it helps resolve any interpretation errors before you release it.</a> <a data-t="659">They're able to go through your documentation, because they don't know the way that it's supposed to behave, and see if there's anything that's unclear about your arguments, the way you set that up.</a></p>

<p><a data-t="668">Additionally, they'll be going through your test cases and seeing how that works, and if there's any ambiguity in the way that they're supposed to be processing it, they come back and ask you, and they can improve your test cases, improve your code.</a> <a data-t="682">They can also potentially identify any places for improvements in your code itself.</a> <a data-t="686">They could have a preconceived notion with the way how quickly things should be running, and by going through it, and if it runs slower, they can bring that up to you, and you can go back and update it.</a></p>

<p><a data-t="700">The approach that we're going to take with this test code is we're going to use a familiar format called </a><a href="/software/testthat/">testthat</a><a data-t="700">.</a> <a data-t="706">I think most people that have developed a hack to know that.</a> <a data-t="708">We're also going to augment these test cases with roxygen2.</a> <a data-t="711">I don't know if you've noticed, but there's a bit of a theme.</a></p>

<p><a data-t="718">This looks like any other test case that you might run normally, but we've added this roxygen documentation about what the test is, who wrote that test, and when they wrote it.</a> <a data-t="725">It was not done by me, so it's a third party.</a> <a data-t="727">They did it today, and they wrote out all the expectations that they were supposed to be doing.</a> <a data-t="733">In this case, the test case was rather simple.</a> <a data-t="735">They were just putting in a simple function and seeing what it returned, but this is a simplification of the idea that can be generally applicable.</a></p>

<p><a data-t="743">We're using testthat because it's a familiar framework.</a> <a data-t="746">Most people, if they've written a package before, already know how to write tests using testthat.</a> <a data-t="750">They can develop their test code interactively, meaning they don't have to just send it off to batch and have it run and see what the results are, but it does allow for automation, so we can actually execute all of the test cases at once within a file and capture all that information in what's called a reporter object.</a> <a data-t="767">These are objects output by testthat, and they track each test and expectations.</a></p>

<p><a data-t="772">It will report a success if each expectation passes, and it'll report failure, and for each one of the expectations you have, track all that as well.</a> <a data-t="783">The benefit is also that we can easily access the contents.</a> <a data-t="787">This is a standard output of a reporter object that you're probably familiar with if you're building tests and do control shift T in your </a><a href="/software/rstudio/">RStudio</a><a data-t="787"> IDE, it'll output that, and you could do that if that's acceptable by your organization, but we were required to actually make a nice printed output of a table where we're recording each of our expectations and whether they were as expected, or it'll report the failure mode, and then, additionally, a pass fail surrounding that.</a></p>

<p><a data-t="813">By writing our test code additionally in this modular format, it makes it so that we can update the test code and track who updated that when independently.</a> <a data-t="823">If somebody works on a function, but they can then still do test code development for functions that they haven't worked on yet, these will live within the validation folder, as I said earlier, this time under test code, and number them from however many test cases you have.</a></p>
