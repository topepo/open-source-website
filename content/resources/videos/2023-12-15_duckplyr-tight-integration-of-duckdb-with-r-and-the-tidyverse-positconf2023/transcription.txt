<h3>Introducing duckplyr</h3>

<p><a data-t="164">So can we have the icing and the cake?</a> <a data-t="166">Can we have fast dplyr?</a> <a data-t="168">Yes, this has worked before.</a> <a data-t="171">With </a><a href="/software/dbplyr/">dbplyr</a><a data-t="171">, we can take dplyr code, convert it into SQL, run it on the database, get it back as a data frame, continue operating in R.</a> <a data-t="182">And there are other such systems, such as datatable and </a><a href="/software/dtplyr/">dtplyr</a><a data-t="182">.</a> <a data-t="187">There is arrow, there is collapse, there may be more that are implementing the dplyr syntax.</a></p>

<p><a data-t="194">What is wrong here?</a> <a data-t="195">Well, we inherit too many features from the underlying database.</a> <a data-t="200">DuckDB can read parquet, this is great, dbplyr can also read parquet, duckdb works with datasets larger than memory, perfect, dbplyr works with datasets larger than memory, but when we get the data from dbplyr, it's not a data frame, we have to collect it, transform it into a data frame.</a> <a data-t="240">We don't support all the R functions, we don't support all the R data types, and for some functions even the implementation or the semantics are a little different, bottom line chances are your dplyr code, if you take it and try to run this with dbplyr, will not work out of the box.</a></p>

<p><a data-t="262">Now unlike duckplyr, duckplyr is designed as a full drop-in replacement for dplyr, data frame in, data frame out.</a> <a data-t="273">Support for all data types, all R functions, I mean, yes, we try to do things in duckdb, if we can't we just offload to dplyr and handle it there without breaking your code.</a></p>

<blockquote><a data-t="262">Now unlike duckplyr, duckplyr is designed as a full drop-in replacement for dplyr, data frame in, data frame out.</a></blockquote>

<p><a data-t="271">My goal is to make all of your dplyr code work with duckplyr and make more and more of your code work faster so that you can focus on what matters to you and achieve better results in less time, development time, and computation time.</a></p>

<p><a data-t="291">And now this is me working on duckplyr, so tests are green, everything passes, we need a new feature, maybe we go to Hannes and ask them to build something, sometimes it's the other way around, duckdb changes behavior, it challenges my expectations, but it's a lot of fun iterating between correctness, performance, and completeness.</a></p>

<h3>Design decisions and limitations</h3>

<p><a data-t="361">Now if you've been in Davis's talk, dotpy is the new kid on the block for group operations.</a> <a data-t="369">Groupby we do not support, this is laziness, this is because we want you to use the new syntax and frankly fewer things can go wrong this way.</a></p>

<p><a data-t="382">Now what happens if you compare an integer with a string?</a> <a data-t="386">What happens when you compare an integer with a factor?</a> <a data-t="388">On what day of week does the week start?</a> <a data-t="392">And R and duckdb have different opinions here and on many, many other questions.</a> <a data-t="400">This is why we support only a small subset of the functions right now.</a> <a data-t="405">This is more opt-in than opt-out.</a> <a data-t="409">We know how to handle those.</a> <a data-t="411">All this now takes us time to implement basically the rest of that.</a></p>

<p><a data-t="419">Now one more difference is row order.</a> <a data-t="423">So if you think about a data frame, you sort a data frame by a specific column in dplyr and if there are ties, if there are multiple rows with the same value, the original order of rows will be preserved with dplyr and duckdb just doesn't care.</a> <a data-t="441">Because duckdb cares about performance, they might just shuffle around your rows.</a> <a data-t="446">This is why we have two modes of operation, a slower mode that will maintain the original row order and fast mode that will do whatever duckdb is doing here.</a> <a data-t="459">And we'll see which one will be most useful in the end.</a></p>
