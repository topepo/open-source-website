<h3>Two strategies for API performance</h3>

<p><a data-t="157">API performance is a huge topic that spans</a> <a data-t="157">all sorts of ideas.</a> <a data-t="163">And we won't be able to cover all of them in today's talk.</a> <a data-t="163">So instead,</a> <a data-t="163">we're going to focus on two ideas, two strategies.</a> <a data-t="170">We're going to talk about minimizing serialization</a> <a data-t="170">costs and maximizing responsiveness with async programming.</a> <a data-t="176">And we're going to cover these</a> <a data-t="176">in particular because they will improve your REST API performance.</a> <a data-t="182">They're relatively they</a> <a data-t="182">require relatively minimal code changes, and there's great R packages for implementing</a> <a data-t="187">them.</a></p>

<h3>Minimizing serialization costs</h3>

<p><a data-t="194">So when I say that, what does that mean?</a> <a data-t="203">So the idea here is that serialization</a> <a data-t="203">costs are the time it takes for you to take an R object in memory and turn it into some</a> <a data-t="210">format that you can send over the wire.</a> <a data-t="218">Like JSON.</a> <a data-t="218">Right?</a> <a data-t="218">You can optimize the business</a> <a data-t="218">logic of your API endpoints, but at the end of the day, there's going to be this serialization</a> <a data-t="223">cost that you pay to turn your R object that's going to be the API response into JSON.</a></p>

<p><a data-t="231">So there's an opportunity here to focus on to start to zero in on, like, how your objects</a> <a data-t="238">are being serialized to see if there's an opportunity to improve performance.</a> <a data-t="250">So to think about, like, why this matters, I want you all to consider a scenario.</a> <a data-t="250">I have</a> <a data-t="250">a party, and I'm going to invite 100 people to this party.</a> <a data-t="257">200 doesn't matter.</a> <a data-t="257">And I have</a> <a data-t="257">two options.</a> <a data-t="262">I can either send the invites through a handwritten postcard or I can send</a> <a data-t="262">them through e-mail.</a> <a data-t="268">To send all of these invites through postcard, I have to go get</a> <a data-t="268">the postcards.</a> <a data-t="273">I have to sit there and handwrite 100 postcards.</a> <a data-t="273">Then I have to put them in</a> <a data-t="273">the mailbox and then wait for them to get to their intended recipient.</a> <a data-t="279">Versus with e-mail,</a> <a data-t="279">I can write my message essentially once and then send it to everybody on a distribution</a> <a data-t="285">list and because of the Internet, it's essentially instantaneous.</a></p>

<p><a data-t="300">We want our serialization cost to be closer to e-mail than to postcards.</a> <a data-t="300">Because if we</a> <a data-t="300">do that, we're going to be able to return our responses much faster.</a> <a data-t="305">If we can return</a> <a data-t="305">results faster, we're going to be able to handle more requests, things of that sort.</a></p>

<blockquote><a data-t="300">We want our serialization cost to be closer to e-mail than to postcards.</a> <a data-t="300">Because if we</a> <a data-t="300">do that, we're going to be able to return our responses much faster.</a></blockquote>

<p><a data-t="316">So sounds like a decent deal, good idea.</a> <a data-t="316">How can we do that?</a> <a data-t="316">So I have here a simple</a> <a data-t="316">example of a plumber API that uses a, that uses what's called a serializer function to</a> <a data-t="326">swap out the defaults and then get a performance boost.</a> <a data-t="331">So how we go about doing this, the</a> <a data-t="331">first step is find a package that serializes JSON faster.</a> <a data-t="339">Out of the box, plumber uses</a> <a data-t="339">something called JSON Lite, which is a robust battle-tested package.</a> <a data-t="345">It's great.</a> <a data-t="345">But in</a> <a data-t="345">this example here, I'm using something called YYJSONR, which uses a very fast C library</a> <a data-t="354">under the hood to read and write JSON in record times.</a></p>

<p><a data-t="362">So once you've found your package of choice, then what you're going to do is you're going</a> <a data-t="366">to write this little function, YYJSONRSerializer, and what that's going to be used is it's going</a> <a data-t="374">to be used as a special function that you give to plumber so it knows how to turn your</a> <a data-t="380">R objects into whatever you're serializing it to.</a></p>

<p><a data-t="386">So with that serializer function, from there, it's very simple.</a> <a data-t="393">You can go ahead and just</a> <a data-t="393">use this serializer tag to say that you want to use your new serializer, and then your</a> <a data-t="398">endpoint gets an immediate performance boost.</a> <a data-t="403">There's no need to change the underlying business</a> <a data-t="403">logic of your API endpoint.</a> <a data-t="409">You just get the lift for essentially free.</a></p>

<p><a data-t="415">Now, on this tiny app, you're thinking, like, this isn't a huge deal.</a> <a data-t="420">But if you can imagine</a> <a data-t="420">an API with 100 endpoints, then it starts to become, like, great from a maintenance</a> <a data-t="427">standpoint as well, because now I can just change the serializer on my 100 endpoints,</a> <a data-t="425">and I get improvement across all of them.</a> <a data-t="433">So we go back to our little web service, Appy,</a> <a data-t="433">and we go ahead and we change out his serializer, and he gets a nice speed boost.</a> <a data-t="437">He gets a</a> <a data-t="437">sick set of roller skates.</a> <a data-t="442">So now whereas he was trudging along before, now he's gliding</a> <a data-t="442">through a little quicker.</a></p>

<h3>Maximizing responsiveness with async programming</h3>

<p><a data-t="447">But we're not done yet.</a> <a data-t="447">We can improve Appy</a> <a data-t="447">even more by using the second strategy that we have here, which is maximizing responsiveness</a> <a data-t="452">with async programming.</a> <a data-t="457">Now, to understand this strategy, we need</a> <a data-t="457">to take a quick sidebar to understand how R and plumber works out of the box.</a> <a data-t="465">So when</a> <a data-t="465">you have our API here on the left and our incoming requests on the right, how does plumber</a> <a data-t="472">process these?</a> <a data-t="478">R, I mean, plumber is an R package, and R is single-threaded, meaning</a> <a data-t="478">that it can process one instruction at a time.</a> <a data-t="486">So when those three API requests come in,</a> <a data-t="486">your plumber API is going to munch through each one of them sequentially.</a></p>

<p><a data-t="492">Now for a lot of REST services, this, like, synchronous execution model is perfectly fine.</a> <a data-t="500">Especially if you have your API has, like, low to moderate traffic and the end points</a> <a data-t="504">are relatively snappy.</a> <a data-t="509">There's plenty of synchronous web services out there today.</a></p>

<p><a data-t="509">But when your traffic becomes higher, and your end points can vary in terms of their</a> <a data-t="516">execution time, then this model of execution can start to get hairy pretty quickly.</a> <a data-t="522">And</a> <a data-t="522">to understand why, I want you to imagine that with these three requests, one is slower than</a> <a data-t="528">two and three.</a> <a data-t="537">Say one takes ten seconds, and two and three take two seconds apiece.</a> <a data-t="543">The first request is going to take ten seconds, but now request two doesn't just take two</a> <a data-t="543">seconds, it takes 12, because it has to wait for request one to complete.</a> <a data-t="548">And what's worse,</a> <a data-t="548">request three will have to wait a grand total of 14 seconds, because it has to wait for</a> <a data-t="553">the other two requests to finish.</a> <a data-t="559">And you can imagine beyond three requests, you know,</a> <a data-t="559">if you're getting hit with a thousand requests, then you're going to start to dog pile your</a> <a data-t="563">R process, and it's not going to be a good time.</a></p>

<p><a data-t="568">So we need another execution model.</a> <a data-t="574">And that's where async programming comes in.</a> <a data-t="574">I like to</a> <a data-t="574">think of async programming as just waiting or executing computations without waiting</a> <a data-t="579">for them to complete.</a> <a data-t="587">But, like, intelligently.</a> <a data-t="587">It's computational multitasking.</a></p>

<p><a data-t="591">To make it a little more concrete, I want you to imagine, like, two ways of going about</a> <a data-t="591">your day.</a> <a data-t="598">One is that you go, you brew a pot of coffee, you pour yourself a cup, drink</a> <a data-t="598">it, then you get your bread, put it in the toaster, and then when it's done, you enjoy</a> <a data-t="603">your toast.</a> <a data-t="608">Versus the way I think the way that a lot of us do it, naturally, is you're</a> <a data-t="608">going to wake up, you're going to start a pot of coffee, you're going to throw your</a> <a data-t="612">bread in the toaster, and then you're going to wait for each of those things to complete,</a> <a data-t="616">and, you know, maybe you'll sip on your coffee while the toast is finishing, but you're ultimately</a> <a data-t="619">going to be able to do those two tasks simultaneously without waiting for them to finish.</a></p>

<p><a data-t="625">And so what we want is an execution model that's closer to way two.</a> <a data-t="629">And that's where</a> <a data-t="629">async programming gets us, or what async programming gets us.</a></p>

<p><a data-t="634">So how does this work for something like Appy?</a> <a data-t="640">So what's going to happen is, instead of having</a> <a data-t="640">just one R process that's running our API, what we're going to do instead is we're going</a> <a data-t="645">to spin up a certain number of child processes.</a> <a data-t="652">And what we do then is our main R process</a> <a data-t="652">is going to function essentially as, like, a relay, where as the requests come in, our</a> <a data-t="657">API will direct them to one of these workers where the requests will actually get processed.</a></p>

<p><a data-t="664">And so in doing this, we end up with this nice effect where request one takes ten seconds,</a> <a data-t="672">but now requests two and three now both take two seconds to complete, because they're sent</a> <a data-t="678">to dedicated workers and there's no longer a line, essentially.</a></p>

<h3>Implementing async with <a href="/software/mirai/">mirai</a></h3>

<p><a data-t="691">So that sounds like a good deal.</a> <a data-t="691">That sounds good.</a> <a data-t="691">So how do we go about implementing this</a> <a data-t="691">in code?</a> <a data-t="704">So to do that, you may have heard of something called mirai by now.</a> <a data-t="711">It's a great</a> <a data-t="711">package for doing asynchronous programming, and it's the one we're going to use in this</a> <a data-t="715">example.</a> <a data-t="715">But you may be wondering, well, why not use the future package?</a> <a data-t="720">The future</a> <a data-t="720">package has been around for a while.</a> <a data-t="721">It's another durable battle-tested package that</a> <a data-t="721">works in a variety of contexts, and it was used in the first ‑‑ it has been used</a> <a data-t="727">for asynchronous plumber work and whatnot up until today.</a> <a data-t="735">I'm choosing to use mirai</a> <a data-t="735">in this example for reasons that are kind of like beyond the scope of this talk, but</a> <a data-t="740">I'm happy to talk to folks about it later.</a></p>

<p><a data-t="750">So how can we go about doing this?</a> <a data-t="750">So we have</a> <a data-t="750">this basic example.</a> <a data-t="750">It's another hello world example.</a> <a data-t="756">And the way that we're going to go</a> <a data-t="756">about doing this is we're going to use a function called mirai from mirai called daemons, which</a> <a data-t="763">is going to spin up four workers in this example, four R processes that are going to be lurking</a> <a data-t="768">in the background there waiting to do stuff.</a> <a data-t="775">And then what we're going to do next is our</a> <a data-t="775">business logic, okay, and our API endpoint is then going to basically get bundled up</a> <a data-t="782">and passed into another mirai function called mirai.</a> <a data-t="788">And what that mirai function is going</a> <a data-t="788">to do is it's going to take the R expression or, like, the business logic that you've</a> <a data-t="794">passed it, and it is going to run that code in the child R process.</a></p>

<p><a data-t="801">And then the important thing is that when it's running or after it runs, it's going</a> <a data-t="806">to immediately return something.</a> <a data-t="810">It's going to return something called a promise.</a> <a data-t="810">And</a> <a data-t="810">the way to think about a promise is that it is a placeholder for the eventual result of</a> <a data-t="816">a computation.</a> <a data-t="826">Now, this notion of a promise, I'll kind of hand wave here, but the idea</a> <a data-t="826">of a promise is what allows our API process to remain responsive to other incoming requests.</a></p>

<p><a data-t="836">Because when the promise returns or, like, when we get our promise back, the main R thread</a> <a data-t="841">is free to do other things, like intercept another inbound request.</a> <a data-t="846">And the nice thing</a> <a data-t="846">about mirai is that once this promise is ready to go, when there's a result able to</a> <a data-t="852">be fetched, it's going to inform the main R process directly, and then R is just going</a> <a data-t="858">to send that result back to the client that requested it.</a></p>

<p><a data-t="865">So we go back to Appy, and we instrument some of his code with some async programming inside</a> <a data-t="873">of it, and he gets another speed boost.</a> <a data-t="878">He gets this sick jet pack.</a> <a data-t="878">So now he's not just</a> <a data-t="878">rolling along the street.</a> <a data-t="882">He's rolling with, like, a Mach 8 engine or something like that,</a> <a data-t="882">something real fast.</a> <a data-t="889">And the end of Appy's story is that with these two strategies, it</a> <a data-t="889">was sufficient for the web service to scale to meet the demands that our other teams were</a> <a data-t="896">bringing to it, and all is right with the world again.</a></p>
