<h3>Overview of public package repositories</h3>

<p><a data-t="114">So let's get started. And let's take a look at the main public package repositories for data science.</a></p>

<p><a data-t="116">CRAN is the primary repository for R packages.</a> <a data-t="122">Bioconductor is also widely used for R packages,</a> <a data-t="128">primarily in data science and life sciences.</a> <a data-t="128">And on the Python side, the Python Package Index,</a> <a data-t="134">or PyPI.</a> <a data-t="134">CRAN and Bioconductor take a similar approach to packages.</a> <a data-t="140">And if you've ever submitted</a> <a data-t="140">a package to be published on CRAN, you know exactly how stringent those requirements are.</a> <a data-t="152">Each package has to pass a comprehensive set of checks to ensure the package is properly formed</a> <a data-t="157">and documented, builds properly, doesn't cause any other packages to fail.</a> <a data-t="163">Most importantly,</a> <a data-t="163">from a security standpoint, these packages must be submitted entirely as source code so there's</a> <a data-t="169">full transparency of what is contained within the package, and it will go through an actual</a> <a data-t="175">human verification before being published.</a> <a data-t="175">It's like the exclusive club with the bouncer out front,</a> <a data-t="181">where they may not make sure you can dance, but they definitely are going to make sure you're</a> <a data-t="187">well-dressed before letting you in.</a></p>

<p><a data-t="187">In contrast, PyPI takes the opposite approach.</a> <a data-t="192">PyPI is much more like Wikipedia, a completely open repository where anyone can create an account</a> <a data-t="198">and publish packages.</a> <a data-t="198">Both in source or binary packages, which then become instantly available</a> <a data-t="206">to everyone around the world.</a> <a data-t="206">This isn't necessarily a bad thing, as it does make a far</a> <a data-t="212">wider array of packages available, though it does pose some additional security risks that we'll</a> <a data-t="218">have to take a closer look at.</a></p>

<h3>Package quality risks</h3>

<p><a data-t="224">So let's start with the security risks now, and the first security</a> <a data-t="230">risk is around the more fundamental question of package quality, or does this package accurately</a> <a data-t="236">do what it claims to do?</a> <a data-t="236">And that's not always an easy question to answer, especially if you're</a> <a data-t="242">not a domain expert, but there are some other more objective factors you can look at to help</a> <a data-t="248">you assess quality.</a> <a data-t="248">Things like the number of package downloads, or how many other people are</a> <a data-t="254">using this package, the author's reputation, is this package author well known in the community,</a> <a data-t="260">do they have a lot of other well-used packages that they've published, is this package well</a> <a data-t="266">documented, are there the functions documented, are there instructional guides, vignettes, or other</a> <a data-t="272">tutorials available, and how frequently is the package updated, when was the last update,</a> <a data-t="278">is it actively maintained and frequently updated with fixes, and finally, I mean, does the</a> <a data-t="284">package include a comprehensive set of tests to verify that the functionality is working correctly?</a></p>

<p><a data-t="293">So how do we mitigate package quality concerns?</a> <a data-t="293">Well, the obvious option is for someone to</a> <a data-t="299">carefully scrutinize the package and make a determination, but as we pointed out earlier,</a> <a data-t="305">that's not always easy.</a> <a data-t="305">Fortunately, the community has developed some tools to help</a> <a data-t="311">with those assessments.</a> <a data-t="311">For example, our friends over at the R Validation Hub have produced an R</a> <a data-t="317">package called Risk Metric that takes these that I just talked about and many more quality metrics</a> <a data-t="323">into account to help you get an objective assessment of package quality.</a> <a data-t="329">And once you've</a> <a data-t="329">done your reviews and done your assessments and decided which packages you want to allow</a> <a data-t="335">your data scientists to use, you can leverage a feature in Posit Package Manager called curated</a> <a data-t="341">repositories, allowing you to create a separate repository just containing the CRAN or PyPI</a> <a data-t="347">packages that you've approved for use in your team or organization.</a></p>

<h3>Vulnerabilities in packages</h3>

<p><a data-t="357">So next let's talk about what you typically think of when you say security, and that's</a> <a data-t="363">vulnerabilities.</a> <a data-t="363">And these can occur at any point in the package, from the source code</a> <a data-t="371">introduced in compiled code or vulnerabilities that exist in external libraries that are used</a> <a data-t="377">by the package.</a> <a data-t="377">And it's important to note that while these are sometimes malicious,</a> <a data-t="384">most often these are just due to mistakes in poorly written code.</a> <a data-t="390">But they're still just</a> <a data-t="390">as important to assess and resolve.</a></p>

<p><a data-t="396">So mitigating vulnerabilities is actually a little bit easier</a> <a data-t="400">for public packages, and that's because the packages are known published quantities.</a> <a data-t="406">You don't generally need to worry about scanning them yourself because numerous others are scanning</a> <a data-t="412">these already published packages and reporting any vulnerabilities on them.</a> <a data-t="418">So the best resource</a> <a data-t="418">for mitigating those are through databases of known vulnerabilities.</a> <a data-t="424">In CRAN's case,</a> <a data-t="424">there are relatively few vulnerabilities reported, partially due to that strong curation of packages</a> <a data-t="430">due to the bouncer of the door.</a> <a data-t="430">But Posit has recently started working with the R Consortium</a> <a data-t="436">on building out an advisory database to track known vulnerabilities in a standardized format</a> <a data-t="442">that was first adopted by the widely used Python Packaging Advisory Database that's</a> <a data-t="448">maintained by the Python Packaging Authority.</a> <a data-t="448">Both of these now feed into the OSV open source</a> <a data-t="455">vulnerability database that was created by Google and is the best source for looking</a> <a data-t="461">up known vulnerabilities in both R and Python packages at this point.</a></p>

<p><a data-t="467">Once you have that,</a> <a data-t="467">you can leverage a feature in Package Manager called Package Blocking,</a> <a data-t="477">where instead of taking a bottom-up curated approach to approving every package you want</a> <a data-t="482">to make available, you can instead go top-down and allow the majority of packages available in</a> <a data-t="488">the public repositories, giving your users the most compatibility with everything that's out there,</a> <a data-t="494">but still blocking the ones that have known vulnerabilities and even for blocking packages</a> <a data-t="500">that may have open source licenses that you don't want to make available inside your organization.</a></p>

<h3>Package confusion: typo squatting</h3>

<p><a data-t="507">Now, separate from the security risks in the packages themselves, there are a couple more</a> <a data-t="513">malicious risks that prey on weaknesses in the infrastructure that is used to distribute these</a> <a data-t="519">packages.</a> <a data-t="519">We group these into a category called Package Confusion, which in simplest terms is all</a> <a data-t="525">about deceiving the user into installing the wrong package.</a> <a data-t="525">So instead of getting the package they</a> <a data-t="531">want to use, they are tricked into installing a malicious package that they don't.</a> <a data-t="537">The two risks we'll talk about today are typo squatting and</a> <a data-t="543">dependency confusion, which are some of the more common ones and most high-profile ones that we've</a> <a data-t="549">seen in the last year.</a></p>

<p><a data-t="557">So I'm going to start and give some examples in terms of Python packages,</a> <a data-t="563">as due to that more permissive nature of PyPI, these have shown up in high-profile real-world</a> <a data-t="569">exploits in the past year.</a> <a data-t="569">But the same cases can apply to our package repositories as well.</a></p>

<p><a data-t="577">So first, let's learn about typo squatting.</a> <a data-t="577">And let's start with you, the data scientist or</a> <a data-t="583">developer.</a> <a data-t="595">And you, let's say we want to start a machine</a> <a data-t="601">learning project.</a> <a data-t="601">And so you want to grab your </a><a href="/software/tensorflow/">TensorFlow</a><a data-t="601"> package from PyPI.</a> <a data-t="607">You use the standard</a> <a data-t="607">Python installation utility pip and say pip install TensorFlow, which is the command to</a> <a data-t="613">install TensorFlow, equivalent to doing install.packages and R to install a package from</a> <a data-t="619">CRAN.</a> <a data-t="619">And so pip goes out to PyPI, looks for a package called TensorFlow, downloads it,</a> <a data-t="631">installs it into your environment.</a> <a data-t="631">Perfect. Everything's fine.</a></p>

<p><a data-t="637">But now let's start again,</a> <a data-t="637">back in our clean environment.</a> <a data-t="643">We've still got PyPI there, but this time when we try to install</a> <a data-t="649">TensorFlow, we accidentally make a typo and had two Ns in the name of the package.</a> <a data-t="655">So we have pip</a> <a data-t="655">install TensorFlow with two Ns.</a> <a data-t="661">Pip goes out and looks for it on PyPI and can't find a package by</a> <a data-t="667">that name and returns an error.</a> <a data-t="667">You notice you mistyped the name, fix it and try again.</a> <a data-t="673">All is</a> <a data-t="673">well.</a> <a data-t="673">That's what you expect to happen in those situations and nothing inherently wrong with that.</a></p>

<p><a data-t="680">But here's where things get scary.</a> <a data-t="680">So let's go back to our clean environment here.</a> <a data-t="686">And let's say</a> <a data-t="686">that some evil person, try up to no good, looks at the most popular public packages on PyPI.</a> <a data-t="693">Like TensorFlow, which, for example, gets downloaded from PyPI on average about 600,000</a> <a data-t="699">times a day globally.</a> <a data-t="699">And so they take that and they write their own malicious package</a> <a data-t="705">and upload it to PyPI with a bunch of different similar names that are common things that you</a> <a data-t="712">might make type by accident if you were trying to install the actual TensorFlow package.</a></p>

<p><a data-t="718">So now we let's take our same scenario.</a> <a data-t="718">We try to install TensorFlow with two Ns in it</a> <a data-t="724">and make the same typo as before, but instead of getting an error this time,</a> <a data-t="730">pip goes out to PyPI and finds this malicious version that someone put there</a> <a data-t="737">and with the same name downloads it and we've been exploited now because we've gotten the wrong</a> <a data-t="743">packaging here.</a></p>

<p><a data-t="749">So these are tough to mitigate against, but there are a few things that we can</a> <a data-t="755">do.</a> <a data-t="755">And I should point out again that these are higher risk with PyPI than on CRAN due to the</a> <a data-t="761">more permissive publishing nature we talked about.</a> <a data-t="761">But one thing working to our advantages in that</a> <a data-t="767">case is that there are many organizations and companies in the Python community watching for</a> <a data-t="773">these malicious packages now.</a> <a data-t="773">So they're identified relatively quickly once they are published,</a> <a data-t="780">usually within a few days or less.</a> <a data-t="780">So the simplest mitigation we can do is just avoid the latest</a> <a data-t="788">packages and updates from public repositories and give them time to be discovered, reported,</a> <a data-t="794">and blocked or removed.</a> <a data-t="794">And so latest isn't always the safest there, and sometimes it's</a> <a data-t="800">worth it to sacrifice a bit of cutting edge and compatibility in the name of security.</a></p>

<h3>Package confusion: dependency confusion</h3>

<p><a data-t="837">Finally, I want to talk about</a> <a data-t="837">a different variant that preys particularly on larger teams, especially those who develop and</a> <a data-t="843">share their own internal packages to supplement their work that we call dependency confusion.</a></p>

<p><a data-t="852">So let's go back into our clean environment here, where we're now part of a larger company that has</a> <a data-t="858">its own internal packages.</a> <a data-t="858">Let's say we're at Posit, and we have our internal Posit tools</a> <a data-t="864">package that we use, along with some public packages from PyPI.</a> <a data-t="870">We may even be so sophisticated</a> <a data-t="870">as to have our own internal packages in a PyPI-like repository so that Pip can install from there.</a></p>

<p><a data-t="882">So we want to install Posit tools.</a> <a data-t="882">We add this extra index URL flag to our internal server</a> <a data-t="888">so that it can find where our additional packages are.</a> <a data-t="888">In reality, this extra index URL setting is</a> <a data-t="894">probably set as part of our system configuration, so we don't have to remember to type it every time</a> <a data-t="900">we install something, but I include this here because this is exactly what would actually happen</a> <a data-t="906">in reality.</a> <a data-t="906">So let's say Pip goes out, looks for a Posit tools package on PyPI, doesn't find it,</a> <a data-t="912">realizes it's not there, so it's probably an internal package, and then searches the extra</a> <a data-t="918">index URL, finds it on our internal repository, and installs it.</a> <a data-t="924">I also need pandas, so I do the</a> <a data-t="924">same command, but in this time, it does find it on PyPI and installs it just as expected.</a> <a data-t="930">So everything's perfect until our malicious actor steps in.</a></p>

<p><a data-t="938">So an evildoer is targeting Posit employees, for example, and maybe guesses that we probably have</a> <a data-t="944">an internal package named Posit tools that's not available on PyPI.</a> <a data-t="950">A lot of companies have</a> <a data-t="950">internal packages like this for connecting to internal resources, databases, or other internal</a> <a data-t="956">sources, and so this malicious actor creates their own malicious package and gives it a name they</a> <a data-t="962">think that our company might use internally, like Posit tools, and they publish it on PyPI,</a> <a data-t="968">probably with a large version number, in this case like 9.0, so it looks newer than anything else.</a></p>

<p><a data-t="975">And now we go back to our user who's trying to, say, upgrade Posit tools they have installed,</a> <a data-t="981">and so they, same as before, have their extra index URL, and PIP goes out, and it sees this</a> <a data-t="987">newer version of Posit tools, not from our internal repository, but available now on PyPI,</a> <a data-t="993">and it doesn't know any better, assumes that's what you want, and installs it,</a> <a data-t="999">and you've been exploited.</a></p>

<p><a data-t="1005">So, you know, there's lots of variants of this, not just directly</a> <a data-t="1011">installing packages, but these packages being dependencies of other packages, and all of this</a> <a data-t="1017">can happen in a similar way here.</a> <a data-t="1017">So this one's actually great because Posit package manager can</a> <a data-t="1023">completely insulate you from this case using a unified local and public repository.</a> <a data-t="1029">So in that</a> <a data-t="1029">case, we put all of our packages, access all of our packages through package manager, we put our</a> <a data-t="1035">internal packages in front of the public package repository, and present that to the user as a</a> <a data-t="1041">single repository, taking that decision of which package to install completely out of PIP's hands.</a></p>

<blockquote><a data-t="1041">taking that decision of which package to install completely out of PIP's hands.</a></blockquote>

<p><a data-t="1047">So now when we ask PIP to install Posit tools from our repository here, again,</a> <a data-t="1053">it's probably been pre-configured, so we don't have to add the package manager address directly,</a> <a data-t="1059">but there's only one that PIP knows to pull things from there.</a> <a data-t="1059">And so PIP goes out,</a> <a data-t="1065">asks package manager now, I need the Posit tools package, and package manager knows, hey,</a> <a data-t="1071">I'm always going to give you the internal one, and because I know that the internal package</a> <a data-t="1077">supersedes the public one, and I will never ever serve you the public package, and our</a> <a data-t="1083">case is solved.</a> <a data-t="1083">Similarly, using the same server, I asked for pandas, pandas is not in our internal</a> <a data-t="1089">package source, so package manager says, okay, here you go, here's the public one from PyPI,</a> <a data-t="1095">and still taking advantage of all of the other security measures that we can also use,</a> <a data-t="1101">the curated package blocking or even repository snapshots can all be used in conjunction with</a> <a data-t="1107">these local and public repository can unify.</a> <a data-t="1113">Everyone's happy, and we've solved at least one</a> <a data-t="1119">of many security risks that we have to worry about.</a></p>
