<h3>The reproducibility problem with Shiny</h3>

<p><a data-t="151">So if we focus in on Shiny, really the core problem here is</a> <a data-t="151">that in order to do its thing, Shiny sort of depends on the user events to sort of drive</a> <a data-t="158">the application logic. And this Shiny runtime execution that sort of determines, like, what</a> <a data-t="164">R code to execute in response to, you know, what user events. And at least for a number</a> <a data-t="172">of years, we've been able to mostly work around this problem in focus of user events through</a> <a data-t="179">Shiny bookmarking. So this at least allows us to save the state of the application at</a> <a data-t="185">any point in time and in theory be able to come back to the state of the application</a> <a data-t="192">sort of assuming that, you know, the app is there available and allowing us to sort of</a> <a data-t="198">restore the state of the application.</a></p>

<p><a data-t="205">But especially if we're not, like, if we don't</a> <a data-t="205">have control over the application, if somebody else is hosting that for us, our bookmarking</a> <a data-t="211">URL that we go to to go back to the state of our application, if that app is no longer</a> <a data-t="216">available, we're kind of out of luck. So in that sense, we still lack some sense of permanence</a> <a data-t="223">that we get from reproducible R code. And even if the app is working and we can sort</a> <a data-t="230">of replicate the user events and get back to the state of the application, we still</a> <a data-t="235">sort of lack some sense of transparency in the sense that those outputs are still locked</a> <a data-t="241">behind a graphical user interface. And if, even if I could get at the source code behind</a> <a data-t="248">the application, sort of the core R domain logic that I might be interested in verifying</a> <a data-t="254">is correct, that's still sort of intertwined with Shiny's runtime engine and it's sort</a> <a data-t="262">of hard to know exactly, like, what R code is being executed at any point in time.</a></p>

<blockquote><a data-t="262">it's sort of hard to know exactly, like, what R code is being executed at any point in time.</a></blockquote>

<p><a data-t="269">So</a> <a data-t="269">the goal here is we sort of want to enable you to eliminate that Shiny runtime dependency</a> <a data-t="275">in some sense, where we want to enable you to write Shiny apps that can generate R code</a> <a data-t="283">to essentially mimic the code that it's executing so that you can run that code without Shiny</a> <a data-t="289">and without having to, like, duplicate that logic yourself in multiple places.</a></p>

<p><a data-t="297">So the project that we're working on to help you enable this feature is the shinymeta</a> <a data-t="304">R package. And this provides tools for you as the Shiny app developer to capture</a> <a data-t="310">the core essential domain logic that is powering your application and exposing it as code to</a> <a data-t="318">users that they can run outside of the Shiny app.</a></p>
