<p><a data-t="0">And in doing so, we identified a couple key elements.</a> <a data-t="8">Firstly, we were repeating the same </a><a href="/software/ggplot2/">ggplot2</a><a data-t="8"> theme elements within a file and across multiple</a> <a data-t="14">different files.</a> <a data-t="15">Meaning we were copying code chunks every single time we were creating a new script</a> <a data-t="19">and new plot.</a> <a data-t="20">Importantly, this wasn't standard across the board.</a></p>

<p><a data-t="23">So if I were creating a new plot, I would open up my script, copy in this code chunk,</a> <a data-t="28">and then make a small modification.</a> <a data-t="30">Let's say I wanted to increase the line weight of my axis or something like that.</a> <a data-t="35">That's not to say then that the next person who goes and copies my code is going to keep</a> <a data-t="39">the same things and make the same customizations.</a> <a data-t="41">So we were having these repetitions, but also lacked uniformity across the board.</a></p>

<p><a data-t="45">Secondly, we were repeating hex colors throughout our scripts and had, as Aaron mentioned, multiple</a> <a data-t="52">different colors that were being used that we were explicitly calling out with their</a> <a data-t="55">hex code each time.</a></p>

<p><a data-t="58">And finally, our team had custom export parameters for the size and DPI of our plots that we</a> <a data-t="64">were using for presentations.</a> <a data-t="66">And while we had this ggsave function that was ready to go at the end of each of our</a> <a data-t="70">scripts, we still had to copy that every time we were creating a new script.</a></p>

<p><a data-t="76">And so doing this inventory allowed us to then parse through our code base and make</a> <a data-t="79">decisions about what we wanted to keep that we would then pull into our standard plot</a> <a data-t="85">elements going forward.</a></p>

<h3>Designing the custom theme</h3>

<p><a data-t="86">So here you have a mock-up of our notes and sketches throughout this process and the decisions</a> <a data-t="92">that we made about formatting.</a> <a data-t="93">And we took it a step further and specifically ascribed for each plot element what we wanted</a> <a data-t="99">those decisions to be.</a></p>

<p><a data-t="126">This effort to better document and understand our previous work allowed us to then move</a> <a data-t="131">into ggplot2 seamlessly with this mock-up and start building our custom refinements on top</a> <a data-t="135">of what already existed in ggplot2.</a></p>

<p><a data-t="138">We focused on three main buckets to make these refinements to our custom plot branding.</a> <a data-t="144">Starting with the plot elements themselves, we were really motivated to adhere to data</a> <a data-t="149">visualization principles, namely to declutter our plots and show the data.</a></p>

<p><a data-t="154">To do so, we really liked the minimal from ggplot2 as a great starting point for this</a> <a data-t="159">and leveraged the replace function from ggplot2 to then make small refinements on top of the</a> <a data-t="165">minimal based on our notes from all of our previous code.</a> <a data-t="169">And so this was making adjustments to things from title and subtitle all the way down to</a> <a data-t="175">things like ticks and margins.</a></p>

<p><a data-t="192">And we really wanted to use this font instead of the three default fonts in ggplot2.</a> <a data-t="197">And to do so, we used the extra font package in gr-devices, which allowed us to build in</a> <a data-t="202">our custom font, Source Sans Pro, into our existing R package.</a></p>

<p><a data-t="208">And so by packaging all of these elements into their own central theme and building</a> <a data-t="213">in a custom font for our package, we were able to pull all of these elements together</a> <a data-t="217">into one simple function called si-style.</a> <a data-t="220">And the idea behind si-style is that any user could pick this up and pipe this into</a> <a data-t="225">their existing ggplot2 code and then immediately be able to standardize their plot elements</a> <a data-t="230">to our style.</a> <a data-t="231">So visual hierarchy, axes, whitespace, the likes, while still remaining flexible and</a> <a data-t="236">customizable to the user.</a></p>
