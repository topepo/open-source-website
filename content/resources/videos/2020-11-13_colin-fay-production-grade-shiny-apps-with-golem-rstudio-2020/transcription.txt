<h3>Shiny app as a package</h3>

<p><a data-t="404">So if we go back to what Vincent said about what is production-ready software, it has</a> <a data-t="413">metadata, it is divided in function, it is tested, it lists the requirements, and it's</a> <a data-t="421">documented.</a> <a data-t="422">And if you think about it, this is what a R package is.</a></p>

<p><a data-t="446">So the big idea behind Golem, so the first and the central idea is that a Shiny app should</a> <a data-t="451">be a package.</a> <a data-t="454">It's also very helpful to have a package structure if you want to talk with IT, because if you</a> <a data-t="460">are sending just a file to your IT and say, hey, send it to production, and they are going</a> <a data-t="466">to say, yeah, but which package do I need, do you have system requirements, where do</a> <a data-t="470">you want to deploy it, how do you need things like that?</a> <a data-t="474">But if you are sending a package, a package can be installed and install the dependencies</a> <a data-t="479">and everything is documented, so it helps deploying the application, because everything</a> <a data-t="484">is there.</a></p>
