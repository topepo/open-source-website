<h3>Options for moving computation off the server</h3>

<p><a data-t="260">So what are some ways of not doing the computation on the server?</a> <a data-t="263">One thing you can do is maybe you can precompute the results.</a> <a data-t="267">So sometimes you can do this.</a> <a data-t="271">And sometimes you can't.</a> <a data-t="278">So some of those maps could have been pregenerated, and actually Josh changed his app to do that,</a> <a data-t="283">but that final one with your personalized map could not be.</a></p>

<p><a data-t="288">What about doing computation on the client?</a> <a data-t="290">Well, you have a couple options here.</a> <a data-t="294">So one is to do the computation in the web browser.</a> <a data-t="298">But if you know something about web browsers, you know that the only language they understand</a> <a data-t="303">is JavaScript.</a> <a data-t="306">So therefore, you must write your application in JavaScript.</a> <a data-t="311">And that can be a problem if you're doing data science, because a lot of the tools that</a> <a data-t="314">you're used to might not exist in JavaScript.</a></p>

<p><a data-t="320">So how about another option?</a> <a data-t="322">Send the code and data to the client and let the user run the code on their computer.</a> <a data-t="326">So in order to do this, though, you need to duplicate your computing environment on their</a> <a data-t="331">computer.</a> <a data-t="332">And doing this, duplicating your computing environment is difficult, and it's a lot of</a> <a data-t="336">work.</a> <a data-t="337">And you can't reasonably expect somebody to take that survey, that fun survey, and have</a> <a data-t="342">to install R or Python and a whole bunch of packages just to do that.</a></p>

<h3>WebAssembly and Pyodide</h3>

<p><a data-t="349">Well, this is actually the story of how things were back then, but things have changed.</a> <a data-t="356">So this part here about web browsers can only run JavaScript, that's actually no longer</a> <a data-t="362">true.</a> <a data-t="363">And the corollary is also not true anymore.</a></p>

<p><a data-t="368">So what other language do web browsers support now?</a> <a data-t="372">Well, there's something called WebAssembly, or it's sometimes abbreviated WASM.</a> <a data-t="377">This is a binary format for programs that can run in web browsers.</a> <a data-t="382">It's not a language that you'd write by hand normally.</a> <a data-t="386">Usually you'd write code in C or C++ or some other languages and compile it to WebAssembly,</a> <a data-t="391">and then you can run it in the browser at near-native speeds.</a></p>

<p><a data-t="395">So it turns out Python is written in C, and there's actually a port of Python called Pyodide</a> <a data-t="402">which compiles to WebAssembly and can run in the web browser.</a> <a data-t="406">And of course, I have to mention WebR, which is a project that </a><a href="/people/george-stagg/">George Stagg</a><a data-t="406"> started and</a> <a data-t="410">is working on, which is R compiled to WebAssembly.</a> <a data-t="413">But this is still in the early stages, so I'm not going to talk more about it in this</a> <a data-t="417">presentation.</a></p>

<p><a data-t="419">So let's go back to doing computation on the client.</a> <a data-t="423">So this option two here, this problem here, duplicating a computing environment is difficult.</a> <a data-t="430">Well, actually, if you compile your computing environment to WebAssembly, you can send it</a> <a data-t="434">to the web browser, and this actually becomes very easy to duplicate the computing environment.</a> <a data-t="440">So we'll cross that out.</a> <a data-t="441">And then we have this new combination of options one and two.</a> <a data-t="444">So we send the code and data and the computing environment in WebAssembly to the client and</a> <a data-t="449">then run all of that in the web browser.</a></p>

<p><a data-t="925">Let's take a look at the deployment model here.</a> <a data-t="927">So for traditional Shiny app deployment, you have a server that's running R or running</a> <a data-t="931">Python.</a> <a data-t="932">And computers connect to it.</a> <a data-t="933">And when they connect to it, they maintain this open connection.</a> <a data-t="937">The user interacts with it a little bit.</a> <a data-t="938">It sends data to the server.</a> <a data-t="939">The server does some computation, and it sends some information back.</a> <a data-t="944">For shinylive app deployment, you have a server that doesn't run R or Python.</a> <a data-t="955">When the user connects to it, it sends Python and the app code and Shiny, of course, to</a> <a data-t="961">those computers.</a> <a data-t="962">And all of that stuff runs inside of those client computers.</a> <a data-t="964">And it's very easy for a server to just send those files.</a> <a data-t="967">That's not a computationally intensive thing to do.</a> <a data-t="970">So you can serve a lot of users this way.</a></p>

<h3>Benefits and limitations</h3>

<p><a data-t="974">Let's talk about the benefits of shinylive, to summarize.</a> <a data-t="978">So the entire application runs on the client instead of the server, so there's no computational</a> <a data-t="981">load on the server.</a> <a data-t="984">It's really easy to share these applications.</a> <a data-t="987">It's easy to deploy them to a static web hosting service.</a> <a data-t="990">And it's easy to scale them, because that's what these static web hosting services do.</a> <a data-t="995">That's their bread and butter, and it's very cheap.</a> <a data-t="1000">And it works on devices without Python installed, so like Chromebooks or iPads or phones.</a> <a data-t="1005">And this is not just for viewing the application, but you can even edit the applications and</a> <a data-t="1010">play with them on there.</a></p>

<p><a data-t="1013">So it's not all sunshine and rainbows.</a> <a data-t="1017">There's also a few drawbacks and limitations.</a> <a data-t="1020">So one is that not all packages are available on Pyodide.</a> <a data-t="1024">So Pyodide contains a lot of packages in the Python scientific stack, including pandas</a> <a data-t="1028">and matplotlib, but not all packages for Python.</a> <a data-t="1032">You can install packages from PyPI if they're written in pure Python and don't have any</a> <a data-t="1037">compiled code.</a> <a data-t="1038">So hopefully this situation will improve in the future, but that's what it is right now.</a></p>

<p><a data-t="1044">The code and data for the application is sent to the web browser, so there's no secrets.</a> <a data-t="1049">Somebody can, in principle, look at all the code and data for your application, if they</a> <a data-t="1053">can see your app.</a> <a data-t="1055">It's not good for large amounts of data, like if you've got 100 megabytes of data that your</a> <a data-t="1059">application is operating on, you probably don't want to send that to somebody's web</a> <a data-t="1064">browser.</a> <a data-t="1065">That's a lot.</a> <a data-t="1067">It's not unheard of, but that is a lot.</a></p>

<p><a data-t="1069">So the download size, for a basic application, the browser will download about 13 megabytes</a> <a data-t="1074">of stuff, and if you're loading packages from there, like pandas, it's just going to go</a> <a data-t="1082">up from there.</a> <a data-t="1086">But fortunately, browser caching will speed up subsequent visits.</a> <a data-t="1089">So the first time somebody views it, it might be slow, and especially in this conference</a> <a data-t="1092">Wi-Fi, the first time you visit one of these may be a little bit slow, but then in later</a> <a data-t="1095">visits it should be really, really fast.</a> <a data-t="1099">And finally, the browser security sandbox restricts network connectivity.</a></p>
