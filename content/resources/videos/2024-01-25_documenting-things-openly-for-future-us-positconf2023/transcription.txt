<h3>Practical tips for documenting things</h3>

<p><a data-t="172">Have a place.</a> <a data-t="173">Have an audience in mind designed for readability and accessibility.</a> <a data-t="177">Have a place.</a> <a data-t="178">It doesn't matter where at first.</a> <a data-t="180">Just write it down.</a> <a data-t="181">Write it down somewhere, some kind of software you're using.</a> <a data-t="183">Get it.</a> <a data-t="184">Get it.</a> <a data-t="185">Have a place there.</a></p>

<p><a data-t="187">Then having that place will let you write as you go.</a> <a data-t="190">You'll be able to develop a habit of writing things in that place, and one of the ways</a> <a data-t="194">you'll do that is by copying and pasting things you're already writing.</a> <a data-t="198">Take it from that Slack message, that email.</a> <a data-t="200">This will help you break down documentation that otherwise can seem like a big looming</a> <a data-t="204">task that might take weeks to do.</a></p>

<p><a data-t="208">Keyboard shortcuts are great for cutting and pasting text from one place to another.</a> <a data-t="213">You'll be able to write in a modular way.</a> <a data-t="216">Writing in small bits is less daunting, and it's easier to maintain collaboratively.</a> <a data-t="222">You'll be able to write these little bits in a way that can be networked so that you</a> <a data-t="227">can link back and forth to them because you don't need a linear flow when you're writing</a> <a data-t="231">documentation.</a> <a data-t="232">This will also help you follow the concept of don't repeat yourself, which is something</a> <a data-t="236">that we can borrow from coding, where you're trying to have your code only written once</a> <a data-t="240">that you call by functions.</a> <a data-t="242">There's often also no true order to what you're documenting, and so you'll be moving things</a> <a data-t="246">around as you develop.</a></p>

<p><a data-t="249">Having an audience in mind is key here.</a> <a data-t="251">You're writing this for someone, that someone is probably you and many others, and different</a> <a data-t="257">audiences at different entry points, so you can make your documentation engaging.</a> <a data-t="261">It doesn't have to be dry or distant.</a> <a data-t="265">Writing in an inclusive tone is part of this, and having the mindset of we're in this together</a> <a data-t="270">and that your readers are intelligent people and they're here, and they're here to learn</a> <a data-t="274">from you.</a> <a data-t="275">So you can really consider your goals for your documentation and the style that you</a> <a data-t="279">want to have that can welcome readers.</a></p>

<p><a data-t="282">You can avoid specific words that trivialize pretty complex things, like simply clone the</a> <a data-t="287">repo.</a> <a data-t="288">There's a lot to unpack in a statement like that, and thinking about how to meet folks</a> <a data-t="293">where they are.</a> <a data-t="294">There's a lot of thoughtful resources that you can emulate, and these are just a few.</a></p>

<p><a data-t="300">Narrate code in small chunks if you're presenting code.</a> <a data-t="303">So this is you can narrate your code in a way that you would say it out loud if you</a> <a data-t="309">were teaching it, and you can match that tone with your purpose, especially as you're trying</a> <a data-t="314">to highlight specific parts of your code for learners to focus on.</a> <a data-t="318">We learned this morning that </a><a href="/software/quarto/">Quarto</a><a data-t="318"> has a new code annotation feature to help in this</a> <a data-t="322">as well.</a></p>

<p><a data-t="325">Sharing early is a big part about writing documentation.</a> <a data-t="329">You want to be able to have this be useful for folks as soon as possible, and you want</a> <a data-t="333">to be able to iterate it and receive feedback and incorporate that feedback to improve it.</a> <a data-t="338">This is where the idea of open comes in, and open does not need to mean public.</a> <a data-t="343">Open can mean shared with your team, and you're able to leverage different technologies and</a> <a data-t="349">the permissions that come with them so that you can share what you want broadly and also</a> <a data-t="353">keep things private.</a> <a data-t="355">I think about this as leaving breadcrumbs for yourself, and in this example, this is</a> <a data-t="358">a public Quarto documentation for NOAA fisheries where they're still able to include internal</a> <a data-t="363">links that are just accessible to them, but we're all able to benefit from this documentation.</a></p>

<p><a data-t="368">Lastly, design for readability and accessibility.</a> <a data-t="372">And we're able to leverage a lot of defaults on the software side and best practices on</a> <a data-t="377">the community side in order to support these goals.</a></p>

<p><a data-t="382">Using section headers is one thing we can do here.</a> <a data-t="386">These are important for screen readers to help describe the sections and flow of a document,</a> <a data-t="391">and you're able to anchor to them directly so that you're able to share a URL specifically</a> <a data-t="396">anchored to a certain section of your documentation.</a> <a data-t="399">Naming things can be really key here, and you're able to embrace the idea of the slug</a> <a data-t="404">where the name of the header becomes the URL so that you're able to give a little bit more</a> <a data-t="409">information about what that header is.</a></p>

<p><a data-t="412">Use text formatting in your documentation.</a> <a data-t="414">This will help your readers follow along.</a> <a data-t="417">You're able to hyperlink to the thing that is important for people to look at rather</a> <a data-t="421">than a word like here or this that can be ambiguous and hard to see as well.</a> <a data-t="426">You're also able to distinguish code with fonts and use markdown in this case as well.</a></p>

<p><a data-t="433">Using alternative text for images is really important.</a> <a data-t="437">This is something that describes the details of the take home messages of the visuals.</a> <a data-t="442">So it's different from a caption in that it will actually describe that a round hedgehog</a> <a data-t="448">is knitting a yellow sock, a rabbit with a teal beanie is wearing one yellow sock,</a> <a data-t="454">and it watches in anticipation, and a shelf to the left of them contains yarn in a tote</a> <a data-t="459">labeled text, and knitting patterns in that tote are labeled code.</a></p>

<h3>NASA Openscapes: documentation enabling cloud transition</h3>

<p><a data-t="470">So putting a focus on documentation has enabled NASA to collaborate across divisions and support</a> <a data-t="475">users that are transitioning to Earth Data Cloud.</a> <a data-t="478">So as a reminder, NASA collects an amazing amount of data about our Earth.</a> <a data-t="484">They have many satellites and missions that are collecting data that are freely available</a> <a data-t="488">for users who are studying our planet, who are studying climate change, who are studying</a> <a data-t="492">public health, agriculture, sea ice melt, a lot of awesome and important work.</a></p>

<p><a data-t="508">And these data have been traditionally downloaded by researchers in order to do analyses locally.</a> <a data-t="514">And after many years of effort and planning, these data are being migrated to the cloud</a> <a data-t="520">in AWS West 2, and this means that researchers are going to be taking their compute to the</a> <a data-t="526">cloud in order to analyze these data.</a> <a data-t="529">This means that this is a big shift in the way researchers have worked their entire careers,</a> <a data-t="534">and it's also a shift for the NASA staff who support researchers in terms of how they teach</a> <a data-t="540">users to access and work with these data.</a></p>

<p><a data-t="544">And this teaching this NASA Earth Data Cloud is going on by learning as a community as</a> <a data-t="550">we go.</a> <a data-t="551">So the NASA Openscapes project is a project that I lead with my co-lead, Aaron Robinson,</a> <a data-t="559">and we are supporting a mentor group across the NASA Earth Science Data Centers, and those</a> <a data-t="565">folks are creating and teaching common tutorials that researchers are using to migrate their</a> <a data-t="572">analytical workflows to the cloud.</a></p>

<p><a data-t="575">So this is a huge effort around connecting folks from different parts of an organization,</a> <a data-t="580">having them be in a comfortable environment to learn together, to teach together, to write</a> <a data-t="585">documentation together.</a> <a data-t="587">There's a lot of emphasis that we put on building a psychologically safe space to ask questions,</a> <a data-t="595">to have a growth mindset where we're able to learn and do together.</a></p>

<p><a data-t="611">So first of all, we have a place that researchers can go to to learn when to cloud, how to think</a> <a data-t="619">about using the cloud with R and Python and other approaches.</a> <a data-t="624">So this is the Earth Data Cloud Cookbook.</a> <a data-t="626">It's made with Quarto.</a> <a data-t="627">It's hosted on a GitHub repository that we're able to contribute to with Jupyter Notebooks</a> <a data-t="632">and R Markdown documents and otherwise.</a> <a data-t="635">But so this is the place that we started to have a place to contribute to.</a></p>

<p><a data-t="640">Our audience was really important.</a> <a data-t="642">We realized that it needed to be us first, and then specific researchers.</a> <a data-t="647">And so this diagram shows that we really invested in our NASA OpenScapes mentor community in</a> <a data-t="652">order to write documentation that would let us figure out how to use the cloud, and then</a> <a data-t="656">we were able to then support other staff across the NASA data centers towards a specific workshop</a> <a data-t="662">with researchers, and all the while contributing to the open science community through this</a> <a data-t="667">documentation.</a></p>

<p><a data-t="669">In terms of design, we're really emphasizing meeting people where they are.</a> <a data-t="674">We don't assume experience with GitHub, with Quarto, with cloud, with Python, with R, with</a> <a data-t="679">Quarto, but we do have the expectation that everyone can learn and teach these things.</a> <a data-t="685">And so we're able to support folks coming from where they are and point them to the</a> <a data-t="690">skills that they might need in order to contribute or learn.</a></p>

<p><a data-t="694">We're also able to collaborate on this documentation across workflows because we're using Quarto.</a> <a data-t="699">We're able to share these or combine Jupyter Notebooks, R Markdown documents, as well as</a> <a data-t="704">contribute in our home places that we're most happy working, whether that's VS Code, Jupyter,</a> <a href="/software/rstudio/">RStudio</a><a data-t="711">, MATLAB, or otherwise.</a></p>

<p><a data-t="714">One other design element is that we're really reusing and complementing existing work.</a> <a data-t="719">We found this documentation system framework is a really helpful way for us to write documentation.</a> <a data-t="725">It helped us think about what was learning oriented and writing those as longer form</a> <a data-t="728">tutorials versus what was problem oriented and could be quick how-to guides with copy</a> <a data-t="734">and pasted code.</a></p>

<p><a data-t="738">Another part of complementing existing work and learning from others has been to be a</a> <a data-t="741">part of the broader open science and open source community.</a> <a data-t="745">This has really helped NASA mentors learn what's available, scope out what the bounds</a> <a data-t="751">of the contribution around documentation should be, as well as co-develop and share and get</a> <a data-t="755">that early feedback.</a></p>
