<h3>Shiny vs. Streamlit</h3>

<p><a data-t="465">Okay, so we announced Shiny for Python about a little less than a year ago</a> <a data-t="472">And the news made the rounds</a> <a data-t="475">And on our data science subreddit</a> <a data-t="479">I think we had a pretty typical range of reactions</a> <a data-t="483">Some people were really excited</a> <a data-t="485">But other people were like this</a> <a data-t="493">But it's just so easy to throw something up with Streamlit</a></p>

<p><a data-t="498">How many people here have heard of Streamlit?</a> <a data-t="501">Okay, how many people have used Streamlit?</a> <a data-t="504">Okay, so a few of you</a></p>

<p><a data-t="508">And Streamlit has this reputation for being just so easy</a> <a data-t="513">And I want to examine that, unpack that a little bit</a> <a data-t="516">So imagine that you have a script.py</a> <a data-t="518">A totally normal linear Python script that is doing some data analysis</a> <a data-t="523">Let's say you're loading some data</a> <a data-t="525">You are maybe cleaning it a little bit</a> <a data-t="528">And then you're previewing a pandas data frame and then plotting it</a> <a data-t="532">The promise of Streamlit is that you can take pretty much that exact script</a> <a data-t="538">Sprinkle in some Streamlit to add some UI</a> <a data-t="542">And boom, you have now an interactive data product or what have you</a></p>

<p><a data-t="547">And contrast that to Shiny</a> <a data-t="551">Which, if you've never seen Shiny before, even what I just showed you</a> <a data-t="555">Clearly, Shiny wants you to put code in the places where it tells you to</a> <a data-t="560">So it is a different approach</a> <a data-t="563">And this approach that they have of being able to write Streamlit apps</a> <a data-t="570">In the same way you write Python scripts</a> <a data-t="573">There's a very specific way they make this possible</a> <a data-t="576">Streamlit's architecture allows you to write apps the same way you write plain Python scripts</a> <a data-t="581">To unlock this, Streamlit apps have a unique data flow</a> <a data-t="584">Anytime something must be updated on the screen</a> <a data-t="587">Streamlit reruns your entire Python script from top to bottom</a></p>

<p><a data-t="592">And let me be clear, it is easy</a> <a data-t="597">I mean, for simple Streamlit apps, it is unbelievably easy</a> <a data-t="602">You could learn Streamlit by accident</a> <a data-t="605">You could write a Streamlit app falling down the stairs</a> <a data-t="608">I mean, it is really, really easy</a> <a data-t="611">And they also have really nice-looking output</a> <a data-t="615">I mean, it's pretty hard to make a bad-looking Streamlit app</a></p>

<p><a data-t="620">But this model also has limitations that are quite obvious</a> <a data-t="625">And especially with the early versions of Streamlit</a> <a data-t="628">You had this pretty slow performance</a> <a data-t="630">Because it's always executing the entire thing again and again</a> <a data-t="633">And pretty soon you need to start being pretty deliberate about caching values</a> <a data-t="638">And you have pretty limited opportunities for interactivity</a> <a data-t="642">Other than changing parameters, the whole thing kind of rebuilds</a></p>

<p><a data-t="646">So Streamlit added some capabilities called caching, callbacks, and session state</a> <a data-t="654">And as people develop in their Streamlit journey</a> <a data-t="658">They find themselves relying more and more on these techniques</a> <a data-t="661">And this is where I sort of, I'm out, you know</a> <a data-t="667">The combination of this beautifully elegant top-to-bottom execution model</a> <a data-t="672">Plus these other features, I think this combination is actually quite dangerous</a></p>

<p><a data-t="676">And I'm not going to elaborate too much more on this</a> <a data-t="680">This is not a talk about Streamlit</a> <a data-t="682">But I just want to make it clear that this is not something that I made up</a> <a data-t="688">I came to this conclusion by looking at the feedback that Streamlit's most rabid fans were giving them</a> <a data-t="695">So in the big picture context, I see one critical issue for Streamlit</a> <a data-t="699">The top-to-bottom rerun on every interaction paradigm</a> <a data-t="704">Maintaining state becomes really difficult</a> <a data-t="707">And there are just multiple examples where people found this model</a> <a data-t="711">At first super simple, and then ultimately kind of painted them into a corner</a></p>

<p><a data-t="718">So what we've tried to do with Shiny and this reactive approach</a> <a data-t="723">Is to have a paradigm that's both easy to get started with and it scales</a> <a data-t="728">It scales as the complexity of your apps scale</a> <a data-t="731">It scales as the kinds of things you want to do to extend your apps increase</a> <a data-t="737">And over the last 10 years, as Shiny for R has proliferated on these same ideas of reactivity</a> <a data-t="744">We've seen people build rather complex data apps indeed</a></p>

<p><a data-t="750">And a lot of the time, if you talk to our users and customers</a> <a data-t="756">Who are getting the most value out of Shiny</a> <a data-t="759">They're often building rather complex, not just a simple dashboard</a> <a data-t="764">Not just a simple model demonstrator</a> <a data-t="767">But really using Shiny for those things, yes</a> <a data-t="771">But also integrating into critical workflows</a></p>
