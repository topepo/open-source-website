<h3>A map of an R package</h3>

<p><a data-t="176">So for us to</a> <a data-t="177">go on this journey together, we're going to need a few things.</a> <a data-t="177">And these packages have</a> <a data-t="182">already been mentioned in these previous talks.</a> <a data-t="182">So use this and dev tools are essential to</a> <a data-t="187">the package building tool belt.</a> <a data-t="187">And as Matt has alluded, it's great to store your package</a> <a data-t="193">on GitHub so you can unlock things like using the GitHub issues and all the goodness that</a> <a data-t="198">GitHub has to offer.</a></p>

<p><a data-t="203">So where would we begin?</a> <a data-t="203">Whenever I enter a new landscape for a walk,</a> <a data-t="208">I like to get my hands on a map.</a> <a data-t="208">So here is a really rough sketch of a map of an R package.</a> <a data-t="213">At its core, your R package contains your ideas, your functions or data that you want</a> <a data-t="218">to share with the community.</a> <a data-t="218">Next to those things are your documentation.</a> <a data-t="218">These are your</a> <a data-t="222">help files and vignettes to give your users some guidance on how to interact with your</a> <a data-t="227">R package.</a> <a data-t="227">There's also something to describe your R package.</a> <a data-t="227">So this description file will</a> <a data-t="233">give the user some higher level information about who wrote the R package, what it does</a> <a data-t="238">generally, what dependencies it may have and where to go if they have any questions.</a></p>

<p><a data-t="241">There's</a> <a data-t="241">something called a namespace.</a> <a data-t="241">This controls what comes into your R package.</a> <a data-t="241">So what functions</a> <a data-t="246">you're borrowing from other packages and what goes out of your R package, what gets</a> <a data-t="251">made available to your user when they load your R package into memory.</a> <a data-t="257">Most packages</a> <a data-t="257">will contain tests.</a> <a data-t="257">These, I think, are the most elusive part of an R package.</a> <a data-t="257">These ensure</a> <a data-t="263">that your R package is behaving as it should and to let you know when it isn't.</a> <a data-t="269">What I</a> <a data-t="269">really like about having a map of an R package is that it breaks up what seems like a really</a> <a data-t="274">complex idea into these smaller manageable pieces.</a> <a data-t="274">And so as you're learning, you can</a> <a data-t="279">focus on one thing at a time.</a> <a data-t="279">And so also using this map, you can see the interconnections</a> <a data-t="283">between all these subcomponents.</a></p>

<h3>The science poke: being curious when you get stuck</h3>

<p><a data-t="301">So as you're building your R package, questions will likely come up and this is totally not</a> <a data-t="307">natural and normal.</a> <a data-t="307">And when they do, I invite you to be curious.</a> <a data-t="307">And one way you can do</a> <a data-t="312">that is by using what I call the science poke.</a> <a data-t="312">So the science poke originated from me going</a> <a data-t="317">on nature walks and coming across something new and peculiar and me grabbing a stick and</a> <a data-t="322">investigating it from a safe distance.</a> <a data-t="322">So in your R package, science pokes are really</a> <a data-t="327">simple ways to help you get unstuck and that are really low effort and low stakes.</a></p>

<p><a data-t="332">So for</a> <a data-t="332">example, I was showing my friend Mark how to write his first R package and he goes,</a> <a data-t="337">how do you add package documentation?</a> <a data-t="337">And I'm like, well, what do you mean?</a> <a data-t="341">He said,</a> <a data-t="341">oh, it's that piece of writing that pops up when you call the help file for deep liar,</a> <a data-t="346">for example.</a> <a data-t="346">And I said, off the top of my head, I don't know, but there are functions</a> <a data-t="352">in use this to add a read me, add vignettes and add articles.</a> <a data-t="352">Maybe there's something</a> <a data-t="357">in there for package documentation.</a> <a data-t="357">And so Mark and I poked around the namespace for</a> <a data-t="362">use this and found a relevant looking function for his purposes.</a> <a data-t="369">The next thing we did was</a> <a data-t="369">to call the help file for use package docs to figure out what it does and how to use</a> <a data-t="373">it and if it's really suited for his purposes.</a> <a data-t="373">And that was all it took to get Mark unstuck.</a></p>

<p><a data-t="380">Another way I like to get unstuck is actually to have a poke around in the GitHub repository</a> <a data-t="385">of an existing R package.</a> <a data-t="385">There's something super comforting in seeing another programmer's</a> <a data-t="390">code doing exactly what you want to achieve.</a> <a data-t="390">It's like holding the answers to an exam that</a> <a data-t="395">you never got to prepare for.</a> <a data-t="395">And it just gives you that much bit of confidence to take</a> <a data-t="400">inspiration for their code and to try things out in your own R package.</a> <a data-t="405">Which leads me</a> <a data-t="405">to my next point is that experimenting with your code is a super effective way in getting</a> <a data-t="410">unstuck.</a> <a data-t="410">And the idea is that you change one thing at a time in your code and call the</a> <a data-t="417">functions document, which updates your documentation, or load all, which loads a preview of the</a> <a data-t="422">recent changes.</a> <a data-t="422">And doing this, you can see the immediate effects of the changes you've</a> <a data-t="428">made onto the R package.</a> <a data-t="428">And it's through this iterative cycle that you can figure out</a> <a data-t="433">how mechanics of an R package works.</a></p>
