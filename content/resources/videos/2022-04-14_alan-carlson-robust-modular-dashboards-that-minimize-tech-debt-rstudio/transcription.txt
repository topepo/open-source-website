<h3>The Golem package</h3>

<p><a data-t="417">And so that, when I was researching that problem,</a> <a data-t="420">we learned about what the Golem package was.</a> <a data-t="423">And currently the Golem package</a> <a data-t="426">has a very fleshed out GitHub repo.</a> <a data-t="429">It has a published book</a> <a data-t="432">that you can either purchase or view online.</a> <a data-t="435">But when I looked at it two years ago,</a> <a data-t="437">it was just this little book down</a> <a data-t="442">with five or so sections that had bare bones concepts</a> <a data-t="447">and they're still fleshing out the process.</a></p>

<p><a data-t="450">But the idea of Golem has stayed the same, right?</a> <a data-t="453">They're trying to build modular dashboards</a> <a data-t="456">with a package-based framework.</a> <a data-t="458">And before we go into what that means,</a> <a data-t="460">I just wanna cover those concepts really quick</a> <a data-t="462">with a couple of slides.</a></p>

<p><a data-t="465">So all Shiny apps have the same thing in common, right?</a> <a data-t="468">They use a UI to create a layout</a> <a data-t="471">and then a server to connect data to that layout.</a> <a data-t="474">And so when you build Shiny dashboards,</a> <a data-t="476">there's basically three ways you do so, right?</a> <a data-t="478">The first method is the all-in-one approach, right?</a> <a data-t="480">So you can make a very simple dashboard</a> <a data-t="482">with an app.r file, all your codes in one place,</a> <a data-t="485">and it makes simple dashboards.</a></p>

<p><a data-t="477">When you add more complexity,</a> <a data-t="490">you kind of split the UI and server into their own files</a> <a data-t="494">because that separates their purposes cleanly, right?</a> <a data-t="496">The UI has all the layouts in it,</a> <a data-t="499">the server files all like the code backend</a> <a data-t="503">and so you know that each of those is referring</a> <a data-t="505">to its very specific parts.</a></p>

<p><a data-t="521">The third method is called modularization.</a> <a data-t="524">And what this does is,</a> <a data-t="526">let's say you wanna build a dashboard</a> <a data-t="528">that serves like four different parts of the business, right?</a> <a data-t="531">Sales, product, marketing, and like some executive view.</a> <a data-t="534">If you try and put all of that into one server file,</a> <a data-t="536">that's going to be a very large server file,</a> <a data-t="539">which isn't very helpful.</a> <a data-t="540">So modularization sticks all these sections</a> <a data-t="543">into their own UI and server</a> <a data-t="545">for each individual dashboard piece.</a></p>

<p><a data-t="547">Additionally, this avoids namespace conflicts</a> <a data-t="550">that you can sometimes run into with the other methods.</a> <a data-t="552">But the key point here is that</a> <a data-t="556">when you modularize your dashboards,</a> <a data-t="558">the UI and server become self-contained functions</a> <a data-t="561">that your dashboard will eventually call</a> <a data-t="563">when it builds its dashboard.</a></p>

<p><a data-t="568">R packages, if you've never built an R package,</a> <a data-t="572">you've certainly installed one.</a> <a data-t="574">And what a package actually is,</a> <a data-t="577">is it's a bundle of files, datasets, unit tests,</a> <a data-t="581">and individual functions, right?</a> <a data-t="583">Whether you're using dplyr or </a><a href="/software/ggplot2/">ggplot2</a><a data-t="583">,</a> <a data-t="586">if you look into their GitHub repos,</a> <a data-t="589">all of their functions are tiny little discrete R scripts</a> <a data-t="593">that are bundled together as a package, right?</a></p>

<p><a data-t="596">And so there's that phrase again, self-contained functions.</a> <a data-t="601">And so what Golem does, what it's trying to do basically,</a> <a data-t="604">is it looks at the commonalities</a> <a data-t="606">between modularization and package frameworks.</a> <a data-t="609">In other words, Golem says,</a> <a data-t="609">well, if all my modules are just functions already</a> <a data-t="613">and a package is just a bundle of functions,</a> <a data-t="614">why can't I just make my dashboard a package</a> <a data-t="616">and run it and deploy it that way?</a> <a data-t="618">And the advantage to that is packages</a> <a data-t="620">are very standardized, right?</a> <a data-t="622">There are specific folders and directories and locations</a> <a data-t="626">where files go within an R package.</a></p>

<p><a data-t="630">Because of that though,</a> <a data-t="632">Golem has no shortage of creation options, right?</a> <a data-t="635">So if you just install Golem</a> <a data-t="637">and try to run with their workflow, right,</a> <a data-t="639">there's tons of different deployment types you can do,</a> <a data-t="642">either RPUBs, RStudio Connect, Docker,</a> <a data-t="645">I believe they even do like GitHub Pages at some point.</a> <a data-t="649">There's different unit tests you can create</a> <a data-t="651">either at like the module or the app level.</a> <a data-t="654">Again, various ways to import files</a> <a data-t="656">and different configurations for your actual app.</a></p>

<p><a data-t="659">And so when I was testing this out,</a> <a data-t="661">I'm trying to answer those two questions, right?</a> <a data-t="663">Does it solve those two challenges we're facing?</a> <a data-t="667">Standardization, absolutely, right?</a> <a data-t="669">Packages, like I mentioned,</a> <a data-t="671">they have to be built a certain way in order to work, right?</a> <a data-t="674">They have to have an instance folder, a man folder,</a> <a data-t="676">namespaces, descriptions, all that is standard</a> <a data-t="679">across like every single R package that's ever built.</a></p>

<p><a data-t="682">But because of all these complex options,</a> <a data-t="686">it doesn't do a good job of communication</a> <a data-t="688">to new developers or people that don't have</a> <a data-t="690">a lot of experience with R and R Shiny.</a> <a data-t="693">And that's by design.</a> <a data-t="694">If you look at the first page of the Golem book,</a> <a data-t="698">it actually says, you know, if you're reading this,</a> <a data-t="700">you have some intermediate advanced level</a> <a data-t="701">of how Shiny works on the backend.</a> <a data-t="704">And this is trying to explore more possibilities there.</a></p>
