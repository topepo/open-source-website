<h3>Do and document</h3>

<p><a data-t="472">So you're going to do the thing.</a> <a data-t="474">Complete the work.</a> <a data-t="477">And you're going to document along the way.</a> <a data-t="483">There is one sentence that I think encapsulates why documentation is so important better than</a> <a data-t="485">anything else.</a> <a data-t="487">And that is this quote.</a> <a data-t="493">That documentation is a love letter to your future self.</a> <a data-t="498">And this is the attitude you want to go into documentation with.</a></p>

<p><a data-t="504">And there are folks across this conference that will preach to you the gospel of documentation.</a> <a data-t="508">But there are a couple things I would like to add in.</a> <a data-t="510">One of them is to modularize.</a> <a data-t="515">Which we did hear about a bit this morning in the keynote.</a></p>

<p><a data-t="527">So like this beautiful tart, every slice can be divided up equally and stand on its own.</a> <a data-t="535">You want to treat your code like this.</a> <a data-t="539">Generally, how do you know if you need to modularize your code?</a> <a data-t="533">If you have a single script with more than 500 lines, too long.</a> <a data-t="538">You should consider modularization.</a> <a data-t="544">If you're doing more than two scrolls, it's not going to be sustainable.</a></p>

<p><a data-t="551">So on the left, there is some code that does some spatial processing and will take a while</a> <a data-t="552">to run.</a> <a data-t="558">It also depends on loading in a bunch of datasets and having libraries in your environment.</a> <a data-t="560">And that's fine.</a> <a data-t="561">It will run.</a> <a data-t="564">And the script is going to take a long time.</a> <a data-t="572">But if you modularize, you can go into a few different you can create scripts that are</a> <a data-t="575">named according to what they do.</a> <a data-t="583">So load packages does nothing but load packages for a solid 15 lines.</a> <a data-t="586">Load data only loads the data.</a> <a data-t="589">It doesn't do any extra modeling or spatial processing.</a> <a data-t="592">It just loads the data.</a> <a data-t="594">And they don't have to be completely independent.</a> <a data-t="600">If they need to run sequentially, you can name them 1, 2, 3, and that gives you a signal</a> <a data-t="603">that they need to be run sequentially.</a></p>

<p><a data-t="609">I'd also like to advocate for a bespoke readme or a very fancy readme.</a> <a data-t="617">A readme is an explanatory document that accompanies computer files or software, according to Wikipedia.</a> <a data-t="622">And I think it gets overlooked in data science from my observation.</a> <a data-t="627">There are three parts of a readme that make it truly bespoke and fancy.</a> <a data-t="634">Context, structure, and key information.</a></p>

<p><a data-t="660">The second is structure.</a> <a data-t="662">How is everything organized?</a> <a data-t="663">Where does the data live?</a> <a data-t="670">Where do you actually do the thing to run the analysis and get the output?</a> <a data-t="674">And this will also force you to take a step back from your code and think about it in</a> <a data-t="678">a more structural manner.</a></p>
