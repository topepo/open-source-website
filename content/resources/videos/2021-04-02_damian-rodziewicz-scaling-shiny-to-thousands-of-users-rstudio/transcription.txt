<h3>What: vertical and horizontal scaling</h3>

<p><a data-t="210">Now, before I jump into why, which I think is the most fundamental</a> <a data-t="215">question that you should be asking, let's talk about what so that you have the whole</a> <a data-t="220">context of what we usually do when we want to scale the Shiny application.</a> <a data-t="220">When you think</a> <a data-t="225">about scaling Shiny, there are two groups of actions that you can make.</a> <a data-t="225">The first one</a> <a data-t="231">is called vertical scaling, and this is increasing the amount of users for one machine.</a> <a data-t="231">You can</a> <a data-t="237">do it in two ways.</a> <a data-t="237">The first one is just adding more resources to your machine, so add more</a> <a data-t="243">CPU, add more memory.</a> <a data-t="243">The second way of vertical scaling is making your Shiny application leaner,</a> <a data-t="246">so allowing more users to use it.</a> <a data-t="246">The second group is horizontal scaling, which is just</a> <a data-t="252">adding more machines.</a></p>

<p><a data-t="252">To put it in this context, the first two are actually fairly simple.</a> <a data-t="259">I know I'm very often asked about how to scale Shiny horizontally, how to have multiple servers</a> <a data-t="265">connect.</a> <a data-t="265">Let me tell you this.</a> <a data-t="265">This is actually very, very simple.</a> <a data-t="265">I'm going to share with</a> <a data-t="271">you resources at the end of this presentation so that you can take a look at yourself on</a> <a data-t="276">the contents.</a> <a data-t="276">Basically, the first simple step that some people usually do is just ask</a> <a data-t="282">the DevOps IT if they can increase the size of the server.</a> <a data-t="282">They add memory, they add CPU.</a> <a data-t="288">This right away gives you more users that can use your application.</a> <a data-t="288">Of course, you have</a> <a data-t="293">to pay for that.</a> <a data-t="293">Very often, it is difficult to find additional budget for increasing the</a> <a data-t="298">machine, especially if suddenly you have 10 times as much users and you need a really,</a> <a data-t="304">really big machine then.</a></p>

<p><a data-t="311">The second step that you can do is talk to DevOps IT and ask them</a> <a data-t="316">to add more servers.</a> <a data-t="316">This is as simple as just spinning up additional virtual machines</a> <a data-t="322">if you are using cloud or a little bit more complex if you have physical machines that</a> <a data-t="327">you have to turn on and configure.</a> <a data-t="327">The good news is that </a><a href="/software/rstudio/">RStudio</a><a data-t="327"> Connect is super easy</a> <a data-t="333">to configure.</a> <a data-t="333">There are just simple steps that you need to go through.</a> <a data-t="333">Suddenly, all</a> <a data-t="338">of your machines are going to run RStudio Connect.</a> <a data-t="338">All of them are going to run your</a> <a data-t="344">application.</a> <a data-t="344">The only thing it costs you is actually real money for the machines.</a></p>

<h3>Making your app leaner: the hard part</h3>

<p><a data-t="350">Now, the third thing that you can do is the most difficult one and the one that requires</a> <a data-t="356">you to understand why the application might be slow.</a> <a data-t="356">This is for you as an R Shiny engineer</a> <a data-t="361">to find the bottlenecks and to understand what slows the application down and to make</a> <a data-t="366">the application leaner.</a> <a data-t="366">There are three main things that you can do.</a> <a data-t="366">Of course, there is</a> <a data-t="371">much more techniques that you can apply, but the key parts are first, to leverage the front</a> <a data-t="376">end, to use JavaScript.</a> <a data-t="376">As you saw from the previous presentations, it's not that complex</a> <a data-t="381">and sometimes you don't even need to understand the JavaScript.</a> <a data-t="381">If you don't use server to</a> <a data-t="387">generate HTML and send it back to you when you make changes, it already saves you a very</a> <a data-t="393">valuable time of your processor.</a></p>

<p><a data-t="393">The second one is to extract computations.</a> <a data-t="393">This also</a> <a data-t="399">decreases the CPU usage of your application.</a> <a data-t="399">If your application is doing something heavily,</a> <a data-t="404">think about a way to extract those computations somewhere else to leverage the ability to</a> <a data-t="410">use external services and not put all of the pressure on your Shiny application server.</a> <a data-t="416">The third one, which is I think the most commonly used, is to just use a database.</a> <a data-t="416">I have heard</a> <a data-t="418">a huge amount of success stories when someone just decided to move all of his data or her</a> <a data-t="424">data into the database and suddenly the application was easily scalable up to hundreds or thousands</a> <a data-t="431">of users.</a> <a data-t="431">This gives you two advantages, which is less memory usage and less CPU usage.</a></p>

<h3>Why: understanding Shiny's architecture</h3>

<p><a data-t="437">This is more or less what you need to do to scale your Shiny application.</a> <a data-t="437">Now, I would</a> <a data-t="443">like you to fully understand why we actually have to do this, because when you think about</a> <a data-t="448">it, RStudio Connect is a great product that allows you to scale Shiny applications up</a> <a data-t="452">to tens of thousands of users.</a> <a data-t="452">You might have seen Sean Loop's video where he shows how</a> <a data-t="458">this is possible.</a> <a data-t="458">Also, Shiny is very fast on your local machine when you just run it.</a> <a data-t="465">Why should it be slower for others when there is just more people using it?</a></p>

<p><a data-t="471">Let me tell</a> <a data-t="477">you a story based on a very simple application.</a> <a data-t="477">If you take a look at the code, this is the</a> <a data-t="483">most simple one that you can write.</a> <a data-t="483">There is a slider and there is a text output.</a> <a data-t="483">The</a> <a data-t="488">only thing that the server does here is the server gets the input slider value whenever</a> <a data-t="496">it changes and sends the value back as output text.</a> <a data-t="496">Now, what happens behind the scenes</a> <a data-t="501">when you send such application to the server?</a> <a data-t="501">First thing when the user connects to the</a> <a data-t="508">server is the browser is going to download HTML that is generated by your UI function,</a> <a data-t="515">CSS, JavaScript, all the static files like images.</a> <a data-t="515">At the same time, server is going</a> <a data-t="523">to create a separate process for you so that you can start your Shiny session there and</a> <a data-t="529">you can execute the R code.</a></p>

<p><a data-t="529">Now, the thing about processes is that R is single-threaded</a> <a data-t="535">and you can specify how many users you want to have for one process, but you need to realize</a> <a data-t="541">that if there are two users that have the same process assigned and R is single-threaded,</a> <a data-t="547">then when one process does something, then the other user cannot actually do anything.</a> <a data-t="552">I will show you this in a moment, but what you should understand here is that the server</a> <a data-t="558">creates a process or adds the user to already existing process to handle all of the operations</a> <a data-t="564">on your reactive graph.</a> <a data-t="564">Now, the second thing that happens is the browser creates a connection</a> <a data-t="571">through WebSockets, and through WebSockets, there are going to be data and information</a> <a data-t="577">being sent back and forth.</a> <a data-t="577">WebSockets are slightly different than the typical REST API that you know</a> <a data-t="583">because they allow you to have a bidirectional communication, and that's why the server can</a> <a data-t="590">actually push some messages to the browser, which is not that possible with typical applications</a> <a data-t="596">that have REST API.</a></p>

<p><a data-t="596">So, the second step is the browser actually binds the Shiny inputs and</a> <a data-t="601">outputs, and it starts the </a><a href="/software/websocket/">WebSocket</a><a data-t="601"> connection with the server.</a> <a data-t="601">Now, let's say that you are</a> <a data-t="607">moving the slider and you set the slider to five.</a> <a data-t="607">The browser is going to send this value to the</a> <a data-t="614">server.</a> <a data-t="614">The server is going to check this value, trigger the reactive computation, and return</a> <a data-t="620">the resulting value.</a> <a data-t="620">So, in our example, it is just going to respond to the browser that text</a> <a data-t="627">is five.</a> <a data-t="627">Now, coming back to Pedro's presentation, if instead of just returning a simple value,</a> <a data-t="635">we returned an updated widget, then you would actually have to send the whole HTML if you don't</a> <a data-t="641">use the update function that Pedro was talking about.</a> <a data-t="641">If you used update function, then the</a> <a data-t="647">browser is smart.</a> <a data-t="647">The browser receives only the data that has changed, and on the JavaScript part,</a> <a data-t="654">it's going to update the HTML.</a> <a data-t="654">So, this is a huge value already by using the update functions.</a></p>

<h3>Long-running computations and blocking</h3>

<p><a data-t="660">Now, when it comes to the browser and the server, after all of the computations are done,</a> <a data-t="666">the browser is just waiting for another signal either from the user or from the server.</a> <a data-t="671">Now,</a> <a data-t="672">this is a very simple example.</a> <a data-t="672">Now, let's take a look what happens when you actually start doing</a> <a data-t="678">some difficult computations.</a> <a data-t="678">Instead of just returning the value, I'm going to perform a long</a> <a data-t="684">complex CPU operation based on the input slider value.</a> <a data-t="684">Now, the same thing that happens,</a> <a data-t="689">you send the value through the web sockets to the server, and within that process that is started</a> <a data-t="695">for you, the process is just calculating the long-running computation that you required him to</a> <a data-t="701">do.</a> <a data-t="701">And the problem is that your user no longer is able to make any actions within the application.</a> <a data-t="709">Everything is gray.</a> <a data-t="709">It is waiting for the output.</a> <a data-t="709">And at the same time, new users or old users that</a> <a data-t="715">have already been using this process also cannot interact with the server and have to wait</a> <a data-t="720">for this computation to finish.</a> <a data-t="720">And this is one of the main issues that we see in applications.</a></p>

<p><a data-t="726">When there is plenty of complex computations happening for one users, the other users are</a> <a data-t="731">blocked or the CPU usage is so high that the other users are seeing that things are going</a> <a data-t="737">slower for them.</a> <a data-t="737">So, in order to get rid of this problem, you can use multiple solutions.</a> <a data-t="743">First one,</a> <a data-t="744">extract some computations to the database.</a> <a data-t="744">I will show you this in a moment.</a> <a data-t="744">Second one,</a> <a data-t="750">you can use Shiny Promises, which is a great package by Joe Chang that allows you to</a> <a data-t="757">move those computations to a completely new process and makes Shiny free from any computation.</a> <a data-t="763">You can use Shiny Worker, which is our package for similar computations.</a> <a data-t="763">Or you can simply move</a> <a data-t="769">some of the computations to the JavaScript to the front end.</a></p>

<h3>Memory and the database solution</h3>

<p><a data-t="769">Now, let's talk about the database.</a> <a data-t="774">This is the second biggest problem that I see in applications.</a> <a data-t="774">You might create a successful</a> <a data-t="779">application that works locally fine.</a> <a data-t="779">But in fact, the application, in order to run,</a> <a data-t="786">loads a lot of data into memory.</a> <a data-t="786">And we need to realize that our computers are similar to</a> <a data-t="793">what we have as servers.</a> <a data-t="793">They have their own RAM.</a> <a data-t="793">They have their own CPU.</a> <a data-t="793">And RAM usually is around</a> <a data-t="799">16, 32 gigabytes.</a> <a data-t="799">It's not a lot.</a> <a data-t="799">If in your application, you read one gigabyte of data,</a> <a data-t="806">and then you filter this data to do some actions in the application, let's see what happens in the</a> <a data-t="812">real life.</a></p>

<p><a data-t="812">You can see a machine here.</a> <a data-t="812">You can see five users that are connected to RStudio Connect.</a> <a data-t="819">And in this configuration, we create one process for every two users that access our application.</a> <a data-t="825">This is configurable.</a> <a data-t="825">This is easy to configure.</a> <a data-t="825">But for this purpose, let's assume that we have</a> <a data-t="831">two users per one process.</a> <a data-t="831">Right now, if you have five users, you already have three gigabytes of</a> <a data-t="837">data being loaded, because every process is like a separate box that contains everything that you</a> <a data-t="843">need to trigger all of that to compute everything.</a> <a data-t="843">And now, when you have, for example, 13 users,</a> <a data-t="851">suddenly, you need to have seven processes and use seven gigabytes.</a> <a data-t="851">Let's see if you have 26</a> <a data-t="857">users.</a> <a data-t="857">It already uses up 14 gigabytes of data.</a> <a data-t="857">And when you think about it, 26 users is not a lot.</a></p>

<p><a data-t="864">So you should be very aware of the fact that the usage of your memory is going to multiply by the</a> <a data-t="870">amount of users that you have in your application.</a> <a data-t="870">And you should try to avoid loading too much data</a> <a data-t="877">into memory.</a> <a data-t="877">What you can do instead is to set up a separate database.</a> <a data-t="877">And it doesn't have to be a</a> <a data-t="883">separate SQL server.</a> <a data-t="883">It could be files on your drive, on the machine's drive, that are accessed</a> <a data-t="890">in a different way than just loading all of that into memory.</a> <a data-t="890">You can check out Search Online for</a> <a data-t="896">Christian Ygras' talk about different possible ways of reading data.</a> <a data-t="896">And even Uber is having</a> <a data-t="900">a separate package that basically reads a lot of files, the terabytes of data that you can search.</a> <a data-t="905">This is just files.</a> <a data-t="905">But when you have an external database, that database contains this one gigabyte.</a> <a data-t="912">And you just execute a filter, a query, and you get the result.</a> <a data-t="912">And here, you have two gains.</a> <a data-t="918">First one is, of course, the fact that you don't have to load the data into memory.</a> <a data-t="923">But the second</a> <a data-t="928">one also is the fact that sometimes such filtering is even faster because the database is specially</a> <a data-t="934">indexed to allow you to make the queries very fast.</a></p>
