<p><a data-t="0">Hi everyone, my name is Therese Anders and I'm a postdoctoral research fellow at the</a> <a data-t="12">Hertie School in Berlin.</a> <a data-t="14">And today I will talk to you about how to conduct peer review in data science classes</a> <a data-t="18">using the </a><a href="/software/gh/">GH</a><a data-t="18"> class package.</a></p>

<p><a data-t="23">So why does peer review matter?</a> <a data-t="24">Well, it's a great active learning tool for students and in particular for instructors</a> <a data-t="29">in really large data science classes, peer review enables them to provide their students</a> <a data-t="34">with meaningful, substantive feedback beyond what might be possible with kind of standard</a> <a data-t="39">automated code review tools.</a></p>

<p><a data-t="44">The GH class package now offers software solutions to do this in R and this is work that I did</a> <a data-t="49">during my internship with RStudio this past summer together with </a><a href="/people/mine-çetinkaya-rundel/">Mine Çetinkaya-Rundel</a> <a data-t="53">and Colin Rundell.</a> <a data-t="55">And so in a nutshell, GH class is a package that allows you to manage a class that is</a> <a data-t="60">run as a GitHub organization via R commands interfacing with the GitHub API.</a> <a data-t="67">And as a new feature, we now added the ability to also conduct peer review with this structure.</a></p>

<h3>Key features of GH class</h3>

<p><a data-t="77">So what are some of the key features that GH class allows you to do?</a> <a data-t="81">Well, first and foremost, GH class will allow you to automate the process of distributing</a> <a data-t="84">and subsequently collecting all the reviews on GitHub repositories.</a> <a data-t="89">The package will allow you to create a randomized review roster for your class and you can also</a> <a data-t="95">assign more than one reviewer to an assignment.</a></p>

<p><a data-t="100">The package allows you to automatically collect the scores that students gave to each other.</a> <a data-t="105">You as the instructor can determine whether you want the review to be single or double</a> <a data-t="109">blind.</a> <a data-t="110">And last but certainly not least, this works for a number of different assignments.</a> <a data-t="115">So this could work for a very small coding homework or even an entire final project.</a></p>

<h3>Demo: the four main steps</h3>

<p><a data-t="127">So let me briefly demo for you the four main steps that you need to know in order to do</a> <a data-t="132">this with GH class.</a> <a data-t="134">So suppose we have three students in our class and they submitted their homework one homework</a> <a data-t="139">via GitHub repositories.</a> <a data-t="141">We can initiate the peer review process by creating so-called review repositories for</a> <a data-t="147">each of these students and the peer init function will automatically do this for you.</a></p>

<p><a data-t="149">The second step is to actually assign the review.</a> <a data-t="152">And so in my little graphic on my slides, I'm going to be focusing on the example of</a> <a data-t="156">Bruno as the assignment author in red and Anya as the randomly assigned reviewer for</a> <a data-t="163">this assignment in blue.</a> <a data-t="165">So what peer assign does, it takes Bruno's assignment, makes a copy of it, and stores</a> <a data-t="170">it in a separate folder on Anya's review repository.</a> <a data-t="174">And this folder is anonymized, so Anya doesn't know whose assignment she's reviewing.</a> <a data-t="178">And what peer assign also does is it adds a R markdown feedback form to this repository</a> <a data-t="184">that we're going to ask Anya to fill out.</a></p>

<p><a data-t="191">So here we're going to do a little bit of time traveling here.</a> <a data-t="197">Because this is the part in the review process where our students would actually conduct</a> <a data-t="202">the review.</a> <a data-t="209">And so what our students are going to do is they're going to edit the original assignment</a> <a data-t="214">of the authors and also fill out the review form that we asked them to do.</a></p>

<p><a data-t="220">Once this process is done, you as the instructor can now collect the scores that the students</a> <a data-t="225">gave to each other.</a> <a data-t="226">And so the peer score review form will essentially extract the scores from the R markdown file</a> <a data-t="232">of the reviewers and store the information in either a data frame or a CSV file.</a></p>

<p><a data-t="249">So the last step in this review process is to take all the review files from the reviewers</a> <a data-t="256">and move it all back to the authors.</a> <a data-t="258">And so this is what peer return will do for you.</a> <a data-t="261">So it will essentially clone the author one folder from Anya's review repository and make</a> <a data-t="266">a copy of it on Bruno's original assignment repository.</a> <a data-t="270">And so now Bruno could go and actually take a look at the feedback that he got.</a></p>

<p><a data-t="278">What does this look like from the student's perspective?</a> <a data-t="282">Well, peer return will also automatically open an issue on Bruno's GitHub repository</a> <a data-t="287">that gives him some instructions of what to do and also has links to all the great feedback</a> <a data-t="292">that Anya gave him.</a> <a data-t="295">And so, for example, if he was to click on this assignment.RMD file, what happens is</a> <a data-t="300">he would automatically be seeing a GitHub diff between his original assignment and then</a> <a data-t="305">all the feedback that Anya gave him.</a></p>
