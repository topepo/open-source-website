<h3>Organize</h3>

<p><a data-t="209">So the first step is organize.</a> <a data-t="211">So in a typical project, you have different types of files.</a> <a data-t="213">You may have analysis scripts, data,</a> <a data-t="216">maybe some R functions,</a> <a data-t="218">and maybe some kind of documentation.</a> <a data-t="220">And every person has different ways</a> <a data-t="223">of organizing their projects.</a> <a data-t="224">And even the same person may have different ways</a> <a data-t="226">of organizing different projects.</a> <a data-t="228">And that can make it very difficult</a> <a data-t="231">to orient yourself within any given project.</a></p>

<p><a data-t="234">But by using the R package structure,</a> <a data-t="237">now all of a sudden you have a consistent way</a> <a data-t="239">in which to organize every single project,</a> <a data-t="241">and every type of file has its own place.</a> <a data-t="245">So your function files will go in the R directory,</a> <a data-t="249">your analysis scripts and data can go in the inst directory,</a> <a data-t="253">and documentation can go in vignettes.</a> <a data-t="255">And actually, when you build the package,</a> <a data-t="257">R will yell at you if these are not in the right place.</a></p>

<p><a data-t="261">And so with this organization comes more efficiency,</a> <a data-t="265">which was paramount in my work during the pandemic</a> <a data-t="268">because we did not have a lot of time to do this work.</a> <a data-t="271">And so I was able to orient myself and work efficiently,</a> <a data-t="273">which meant that I could update models rapidly</a> <a data-t="276">as new data became available, which it did consistently.</a> <a data-t="280">It meant that the analysis was delivered on time.</a> <a data-t="282">And so ultimately, my work was actually used</a> <a data-t="286">to directly inform public health policy decisions,</a> <a data-t="289">and the Dutch government did ultimately decide</a> <a data-t="291">to extend the vaccination campaign,</a> <a data-t="293">which affected over a million citizens.</a></p>

<h3>Document</h3>

<p><a data-t="297">But organization alone is not enough.</a> <a data-t="300">We need to be able to understand how, what, and why</a> <a data-t="304">we're doing aspects of a project.</a> <a data-t="307">And so let's go back to that old file structure.</a> <a data-t="310">So here we have a number of different documentation files,</a> <a data-t="312">maybe a code guide, maybe some description</a> <a data-t="315">of the methodology that the code is implementing,</a> <a data-t="317">and maybe a final report.</a></p>

<p><a data-t="320">And it can be very hard to have a separate text file</a> <a data-t="324">in a repo that describes what the code is doing.</a> <a data-t="327">I once inherited a project where I was given</a> <a data-t="330">a repository of code and an academic manuscript.</a> <a data-t="334">The code was uncommented, undocumented,</a> <a data-t="336">and all I had to go off of was a method section</a> <a data-t="338">in an academic manuscript.</a> <a data-t="340">And if you've ever read an academic manuscript,</a> <a data-t="342">they are notoriously concise.</a> <a data-t="344">So I had a very small method section</a> <a data-t="347">with which to try to decipher the code I was looking at.</a> <a data-t="349">And it ended up, I had to end up rerunning all the code</a> <a data-t="352">and looking at intermediate results</a> <a data-t="354">to try to figure out what was going on.</a> <a data-t="355">And that took a lot of time.</a></p>

<p><a data-t="357">But with an R package, you have the advantage</a> <a data-t="360">that documentation lives with the code.</a> <a data-t="362">And I want to highlight two important features</a> <a data-t="365">of R package documentation.</a> <a data-t="367">The first will be the automatically generated help files,</a> <a data-t="370">which gets stored in the man directory.</a> <a data-t="372">And the second are vignettes.</a></p>

<p><a data-t="376">So if you use the ROxygen, or ROxygen, not really sure,</a> <a data-t="381">preamble above your function,</a> <a data-t="383">you can include really important information</a> <a data-t="385">about what that function is doing,</a> <a data-t="387">like what that function does, expected inputs,</a> <a data-t="391">expected outputs, and example usage.</a> <a data-t="395">And a really nice piece of this</a> <a data-t="397">is that this preamble lives with your function code.</a> <a data-t="401">So you can write this as your coding.</a> <a data-t="403">So it alleviates the problem</a> <a data-t="405">of having to write a bunch of code</a> <a data-t="407">and then go back a week, two weeks, a month later</a> <a data-t="410">and try to remember what you did and describe it.</a> <a data-t="412">So this way, you can document your code in real time.</a></p>

<p><a data-t="416">The other very handy aspect of this preamble</a> <a data-t="418">is that when you build the package,</a> <a data-t="420">it creates automatically generated help files.</a> <a data-t="423">And so now, you can access this information</a> <a data-t="426">by running one line of code,</a> <a data-t="428">the question mark and function name.</a> <a data-t="430">So you don't have to go searching</a> <a data-t="432">through some sort of project directory</a> <a data-t="433">to find the information that you need</a> <a data-t="435">in order to run a function or use a code chunk.</a> <a data-t="438">It is easy to access.</a></p>

<p><a data-t="442">And so you may actually find that by using these help files</a> <a data-t="445">that you don't need that code guide anymore.</a> <a data-t="448">That maybe you don't need a piece of documentation</a> <a data-t="450">that you would in a more traditional project.</a> <a data-t="453">And that's one less thing to have to update and manage.</a></p>

<p><a data-t="458">And the second part of documentation</a> <a data-t="461">that I wanna touch on is the vignettes.</a> <a data-t="463">So I have only recently realized</a> <a data-t="465">how powerful these vignettes can be.</a> <a data-t="467">So you remember that project where I had a code base</a> <a data-t="470">and a methods paragraph?</a> <a data-t="472">With a vignette, you can actually incorporate text,</a> <a data-t="477">code and results.</a> <a data-t="479">So now all of a sudden,</a> <a data-t="480">I can actually describe what a code chunk is doing,</a> <a data-t="484">show the code chunk and then show the expected output.</a> <a data-t="487">And that's hugely beneficial</a> <a data-t="490">and it makes it much easier</a> <a data-t="491">to try to document a whole analysis.</a></p>

<p><a data-t="494">The other really nice feature of a vignette</a> <a data-t="497">is that it can be version controlled</a> <a data-t="499">in the same way as your code.</a> <a data-t="500">So whereas if you have a Word file,</a> <a data-t="503">that Word file is being tracked via words track changes</a> <a data-t="506">and it's separate from your code.</a> <a data-t="508">So it increases the likelihood</a> <a data-t="510">that those two documents are gonna be out of sync.</a> <a data-t="513">With a vignette, it can also be version controlled</a> <a data-t="516">using GitHub like your code</a> <a data-t="517">and so it just makes the likelihood</a> <a data-t="520">of the documentation file and the code file</a> <a data-t="522">becoming out of sync much lower.</a> <a data-t="524">Can still happen, you still have to update</a> <a data-t="526">your documentation, but it increases the chance</a> <a data-t="529">that your documentation is current.</a></p>

<p><a data-t="532">And so with the vignette, you may actually find</a> <a data-t="534">that you don't need a separate methodology document</a> <a data-t="539">and that maybe all you need is a final report.</a></p>

<h3>Sharing and collaboration</h3>

<p><a data-t="622">And so the first two features of this R package workflow</a> <a data-t="626">focus mainly on your individual workflow,</a> <a data-t="629">how you move through a project.</a> <a data-t="631">But what I think makes the R package really powerful</a> <a data-t="637">is that when you combine it</a> <a data-t="638">with some sort of file sharing platform,</a> <a data-t="640">whether that's GitHub or a shared network drive</a> <a data-t="643">or something, then all of a sudden</a> <a data-t="645">you have collaborative magic.</a></p>

<p><a data-t="648">Because now all of a sudden you have this beautiful,</a> <a data-t="652">organized, standardized, documented,</a> <a data-t="655">for lack of a better word, package,</a> <a data-t="658">that enables you to install it easily.</a> <a data-t="660">So rather than having to source a bunch of separate scripts,</a> <a data-t="664">you can install a package in one line of code.</a> <a data-t="667">You also have a way to have seamless handovers.</a> <a data-t="670">So it becomes very easy for you to onboard a new person</a> <a data-t="674">or even for yourself to come back to a project</a> <a data-t="676">after six months.</a></p>

<p><a data-t="678">Another important feature is that you can create</a> <a data-t="680">reusable tools that you and your colleagues</a> <a data-t="683">and other people at your organization can use.</a> <a data-t="686">So I found out in my team that actually me</a> <a data-t="688">and two other colleagues were all trying</a> <a data-t="690">to use the same method</a> <a data-t="691">and we were independently coding that.</a> <a data-t="693">But with this approach, someone can code it one time</a> <a data-t="697">and we can all use it.</a> <a data-t="698">And this is also really important in infectious disease work</a> <a data-t="701">because I realized that I'm lucky enough</a> <a data-t="704">to have the resources to build these tools.</a> <a data-t="706">That is not the case everywhere.</a> <a data-t="708">And it's very important that everyone has the capability</a> <a data-t="712">and the resources and the tools</a> <a data-t="714">with which to analyze their own data.</a></p>

<p><a data-t="717">And finally, it facilitates collaboration.</a> <a data-t="721">So it's one thing to have a conversation</a> <a data-t="722">with someone outside your organization</a> <a data-t="725">or someone on another team about things you're working on,</a> <a data-t="728">but it changes the game when you're able to show them</a> <a data-t="731">what you're working on and generate ideas</a> <a data-t="733">for how you can either work on the current project</a> <a data-t="736">or extend it later.</a></p>

<p><a data-t="739">And so with all of this sharing capability</a> <a data-t="741">comes reproducibility, which is a central principle</a> <a data-t="744">in scientific research.</a> <a data-t="746">And how this manifested during the COVID modeling</a> <a data-t="750">was that because I use this approach,</a> <a data-t="752">my model code was actually selected to be included</a> <a data-t="755">in a separate R package called Epidemics,</a> <a data-t="757">which serves as a transmission model library.</a> <a data-t="759">So they were able to take my code, make it better,</a> <a data-t="763">and include it in this other package for others to use.</a> <a data-t="766">Another thing was that I was able to be involved</a> <a data-t="769">in a European-wide effort to provide modeling results</a> <a data-t="774">to help inform COVID public health policy.</a> <a data-t="778">And that would have been much more difficult</a> <a data-t="780">if I hadn't used this approach.</a> <a data-t="781">And so ultimately, one national analysis</a> <a data-t="784">has now become a piece of international infrastructure</a> <a data-t="786">that will live beyond the work that I did.</a></p>
