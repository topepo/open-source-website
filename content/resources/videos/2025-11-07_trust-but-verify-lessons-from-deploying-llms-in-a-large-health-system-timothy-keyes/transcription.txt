<h3>Introducing Chat EHR (Chatter)</h3>

<p><a data-t="128">This is why our team at Stanford</a> <a data-t="132">Healthcare created an application called Chat EHR or Chatter. Chatter's UI is served by an infrastructure layer that handles all</a> <a data-t="150">the technical things that you need when you're working with LLMs, and this</a> <a data-t="154">infrastructure layer is also exposed on the back end as a suite of REST APIs so</a> <a data-t="159">that data scientists like me can build various task-specific automations to</a> <a data-t="163">guide our patients efficiently through the healthcare system. Some of the things</a> <a data-t="168">that we tend to do with these task-specific automations are screen</a> <a data-t="172">patients for eligibility for certain kinds of appointments, schedule those</a> <a data-t="175">appointments, and then write draft documentation for our clinicians once</a> <a data-t="180">those appointments are completed.</a></p>

<p><a data-t="182">However, this talk is actually not about</a> <a data-t="187">Chatter at all. It is not about how it was built, and it is not about how it</a> <a data-t="192">works. Instead, it is about a single question that faced our team once we put</a> <a data-t="197">this system into production and over 4,000 of our clinicians started using it</a> <a data-t="201">out in the real world for real patient care, and that question is this. How do we</a> <a data-t="207">know that an LLM-powered system is behaving as expected over time, or more</a> <a data-t="212">specifically, how do we know that the text that the LLM is generating is</a> <a data-t="216">reliable, useful, and safe for our patients and the clinicians who serve</a> <a data-t="219">them?</a></p>

<p><a data-t="221">I can't tell you that our team or really anybody else has definitively</a> <a data-t="225">answered this question. <h3>Lesson one: the view from space</h3>

<p><a data-t="235">Sound good? Awesome. This brings us then to lesson</a> <a data-t="240">number one, the view from space. I will claim that LLMs really aren't so</a> <a data-t="245">different from other statistical models in terms of what you really care about</a> <a data-t="248">organizationally once they have been deployed. What I mean by this is that our</a> <a data-t="253">organization has an MLOps framework that we use for all of our statistical model</a> <a data-t="257">deployments, and we were really worried when we started working with LLMs that</a> <a data-t="261">they were just too different of a technology, and they would basically</a> <a data-t="264">smash our framework into smithereens and we would have to start over from scratch.</a> <a data-t="269">Luckily, this didn't happen because we just found in practice that regardless</a> <a data-t="275">of the type of model that you're serving, whether it's logistic regression, XGBoost,</a> <a data-t="279">an LLM, or an agent, there's always certain things that you have to do to</a> <a data-t="286">support a model when it's in production that are kind of regardless of the</a> <a data-t="290">underlying technology.</a></p>

<p><a data-t="294">Let me illustrate this clearly by walking you through</a> <a data-t="298">Stanford HealthCare's MLOps framework. Your organization probably has something</a> <a data-t="302">pretty similar, but if you don't, Julia Silke actually gave a really beautiful</a> <a data-t="306">talk about MLOps in her PositConf talk a couple of years ago, and you should go</a> <a data-t="311">watch that. But at Stanford HealthCare, we break down MLOps into three pillars.</a></p>

<p><a data-t="312">The first we call system integrity. This is about monitoring a model's ability to</a> <a data-t="317">find its input data and to post its output to the intended destination with low</a> <a data-t="321">latency and high availability. This has to do with the IT infrastructure supporting</a> <a data-t="326">your model, so regardless of what that model is, it's always on the table and</a> <a data-t="330">it's something you have to keep track of. The second pillar we call performance.</a> <a data-t="335">This is about monitoring the quality and accuracy of the output of the model.</a> <a data-t="339">So how often is the model correct? How often is it incorrect about the things</a> <a data-t="342">that it does or says? Again, regardless of the model, this is something you have to</a> <a data-t="346">care about if you want to be a good data scientist. And then lastly, our third</a> <a data-t="351">pillar is impact. Because we're a hospital, it's really not enough for our</a> <a data-t="355">models to be technically excellent. They also have to have a traceable way of</a> <a data-t="360">serving our number one goal as an organization, which is providing the best</a> <a data-t="364">patient care possible. So we typically also monitor process measures and KPIs</a> <a data-t="369">that are associated with any of our deployments to make sure that they are</a> <a data-t="372">serving their intended purpose towards this mission.</a></p>

<p><a data-t="374">So we found that these three</a> <a data-t="378">principles, kind of luckily for us, are sufficient for monitoring any model after</a> <a data-t="382">it's been deployed, including LLMs. But that all being said, we did find it most</a> <a data-t="387">difficult to tackle this second pillar. How do you define and then monitor the</a> <a data-t="392">accuracy of an LLM? It's a really sophisticated technology, and it can do</a> <a data-t="396">so many things.</a></p>

<h3>Lesson two: key differences in monitoring LLM performance</h3>

<p><a data-t="399">This brings us to our second lesson. So we are now no longer in</a> <a data-t="404">space. We have re-entered the atmosphere. And as we get closer to the Earth, we can</a> <a data-t="408">appreciate that despite their apparent differences at a high level, there are</a> <a data-t="412">some key differences between LLMs and other kinds of statistical models that</a> <a data-t="416">affect how we define and monitor their performance, just like accuracy element.</a> <a data-t="421">There are a bunch of differences that we could pull apart here between LLMs and</a> <a data-t="425">other types of models, but I will talk about three key differences that we are</a> <a data-t="428">most opinionated about at Stanford.</a></p>

<p><a data-t="432">The first is this. Humans simply interact</a> <a data-t="435">with text differently than they do with the outputs from other kinds of models,</a> <a data-t="440">like probabilities or discrete labels. In a clinical environment, machine learning</a> <a data-t="444">models are typically used to make recommendations to clinicians. So they</a> <a data-t="448">might recommend prescribing a particular medication or ordering a</a> <a data-t="451">certain laboratory test. But in practice, what we found is that unless these</a> <a data-t="456">models are highly, highly accurate, which most of them simply are not, because</a> <a data-t="460">medicine is very hard, clinicians learn not to trust them, and so they ignore</a> <a data-t="465">them. So they end up being not so useful. With LLMs, on the other hand, we find</a> <a data-t="470">that this kind of like magical human-computer interaction dynamic</a> <a data-t="474">emerges where clinicians are actually really eager to read the free text</a> <a data-t="478">recommendation of an LLM and to understand its justification for making</a> <a data-t="483">that recommendation. They kind of like see what the LLM is suggesting to them,</a> <a data-t="486">run it through their own clinical reasoning process, and then decide if</a> <a data-t="490">they agree or disagree with the model.</a></p>

<p><a data-t="494">So what we've learned is that monitoring</a> <a data-t="498">the so-called performance of our LLM deployments is really about capturing</a> <a data-t="502">this human-in-the-loop adjudication process, so that we can track over time</a> <a data-t="506">how much our expert clinicians are agreeing with the things that the LLM</a> <a data-t="510">says.</a></p>

<blockquote><a data-t="494">So what we've learned is that monitoring</a> <a data-t="498">the so-called performance of our LLM deployments is really about capturing</a> <a data-t="502">this human-in-the-loop adjudication process, so that we can track over time</a> <a data-t="506">how much our expert clinicians are agreeing with the things that the LLM</a> <a data-t="510">says.</a></blockquote>

<p><a data-t="512">I think this is well illustrated by an example, so I'll walk you through one</a> <a data-t="516">of our LLM deployments to give you a flavor for how this looks in practice.</a> <a data-t="521">The task here is we're asking the LLM to identify patients who might benefit from</a> <a data-t="528">an appointment with a specialist, just a certain kind of physician, basically. So</a> <a data-t="533">our starting point is a prompt, and that prompt contains the criteria for the</a> <a data-t="537">specialist appointment. It usually contains some information about the</a> <a data-t="540">types of illnesses that the specialist can treat. We send this to the LLM along</a> <a data-t="545">with clinical notes describing the health of the patient and ask the LLM to</a> <a data-t="549">output two things. The first is a targeted recommendation for whether the</a> <a data-t="554">patient should get the specialist appointment, and the second is a</a> <a data-t="558">justification with cited evidence from the patient's chart of why it made that</a> <a data-t="563">recommendation. Both of these are then sent to one of our expert clinicians who</a> <a data-t="568">are always in the loop, and that clinician then decides if they agree or</a> <a data-t="571">disagree with the model and if they want to place the referral for the</a> <a data-t="575">specialist appointment.</a></p>

<p><a data-t="576">Our monitoring approach then really hinges on capturing</a> <a data-t="580">this clinician's decision within our electronic health record, and then we</a> <a data-t="584">track it over time so that if we see basically that the disagreement rate of</a> <a data-t="588">our clinicians is increasing for a deployment after some period of</a> <a data-t="593">time, this basically suggests that something is going wrong, maybe</a> <a data-t="596">documentation practices have changed, maybe we need to adjust the prompt, and</a> <a data-t="600">that our data science team then needs to remediate that.</a></p>

<p><a data-t="602">Okay, so moving on to our</a> <a data-t="606">second key difference between LLMs and other statistical models in terms of</a> <a data-t="610">monitoring performance. Whereas the first one was more conceptual, this one is more</a> <a data-t="614">technical. It has to do with like some engineering things, and it's based on the</a> <a data-t="618">observation that frontier LLMs developed by large companies like ChachiPT,</a> <a data-t="622">Claude, and Gemini are externally developed and versioned by third parties.</a> <a data-t="627">So this means that they actually update and deprecate over time without your</a> <a data-t="631">team necessarily being in control. This is in contrast to something like an XGBoost</a> <a data-t="635">model that you may have trained, hosted, and served yourself, which will</a> <a data-t="639">only ever be updated or retrained or deprecated, turned off, when you choose.</a></p>

<p><a data-t="644">You may not think about this as being related to monitoring a model</a> <a data-t="649">post-deployment in the traditional sense, but my point here is that when you're</a> <a data-t="653">working with proprietary LLMs, you don't just have to worry about the data</a> <a data-t="658">distributions of your deployed population shifting over time, you don't</a> <a data-t="662">just have to worry about the user behavior changing over time, you</a> <a data-t="665">actually also have to worry about the model itself changing over time as it's</a> <a data-t="669">continually updated by the third party that has developed it. So what we do</a> <a data-t="674">about this is basically host a suite of internally developed benchmarks that</a> <a data-t="678">help us to monitor task performance for important tasks at Stanford Healthcare</a> <a data-t="682">whenever a new model is released. So for those of you who might be familiar, GPT-5</a> <a data-t="687">was released a couple of weeks ago, and we were curious like, should we</a> <a data-t="691">switch to this model? We're using a couple of legacy models that won't be</a> <a data-t="694">supported in a couple of months, and so this is our approach for dealing with</a> <a data-t="698">that scenario.</a></p>

<p><a data-t="703">For those of you keeping me honest, I have used some jargon here.</a> <a data-t="709">So benchmark is a term that is used commonly in the LLM world, and the LLM</a> <a data-t="714">world like really likes jargon, but to keep it short and sweet, a benchmark is</a> <a data-t="718">essentially a human-labeled gold-standard test data set for a</a> <a data-t="722">specific task. So if you remember our specialty referral example from before,</a> <a data-t="727">the way that you would generate a benchmark for that is basically taking</a> <a data-t="730">your prompts and your clinical notes about a patient, showing them directly to</a> <a data-t="734">a clinician, and then asking, would you place the referral to the specialist for</a> <a data-t="738">this patient, yes or no? If you do that for a hundred patients, you've now</a> <a data-t="742">generated a data set that the model never saw during training because it</a> <a data-t="745">lives within your walled garden of your databasing of your organization, and</a> <a data-t="749">then you can then use that benchmark to evaluate many different LLMs on that</a> <a data-t="754">task, whether they already exist or if they are released in the future.</a></p>

<p><a data-t="760">So what</a> <a data-t="762">we've done in our organization is to basically do this for many different</a> <a data-t="765">tasks that we have identified as high-value, and then bundled them all</a> <a data-t="768">together in a custom Python evaluation harness called Medhelm, or medical</a> <a data-t="773">holistic evaluation of language models, that is actually quite similar under the</a> <a data-t="777">hood to some of the other evaluation frameworks that have been discussed in</a> <a data-t="780">this session. And in fact, we may end up porting it over to something like</a> <a data-t="784">Inspect in the future.</a></p>

<p><a data-t="790">All right, finally, this brings us to our third key</a> <a data-t="794">difference between monitoring LLM performance and monitoring the</a> <a data-t="797">performance of other kinds of statistical models, and this one is the</a> <a data-t="804">hardest. At the end of the day, no matter how big or expansive your</a> <a data-t="812">suite of benchmarks is, you just can't evaluate an LLM on every single task</a> <a data-t="816">that you could potentially apply it to within your organization. It's not</a> <a data-t="819">possible, and even if it were, it'd be prohibitively expensive to do. This</a> <a data-t="824">is because LLM outputs are just so much more flexible and expressive than other</a> <a data-t="828">kinds of models that are trained for one or a small number of specific purposes.</a> <a data-t="832">They can take pretty much anything as input, and they can produce pretty</a> <a data-t="836">much anything as output.</a></p>

<p><a data-t="843">So what do you do? Our approach is to identify important</a> <a data-t="847">and known failure modes across many of your different LLM deployments and try</a> <a data-t="851">to focus on those. So you basically identify things that you don't want the</a> <a data-t="855">LLM to do, and you monitor those specifically. This is helpful not just</a> <a data-t="860">for monitoring purposes, but it also is kind of a step into a process called</a> <a data-t="864">guardrailing your model, which basically means trying to intercept those failure</a> <a data-t="868">modes before they occur.</a></p>

<h3>Monitoring hallucinations with fact decomposition</h3>

<p><a data-t="874">So for example, at our organization, something that we're</a> <a data-t="878">really worried about is this issue of LLM hallucinations. Probably many of you</a> <a data-t="882">who work with LLMs are concerned about these as well. In our context, an LLM</a> <a data-t="886">hallucination just means that the model makes a claim about a patient that is</a> <a data-t="891">not substantiated or supported by any of the information present in that</a> <a data-t="896">patient's chart. So what we do to monitor this is basically we run automated</a> <a href="/software/quarto/">Quarto</a><a data-t="901"> reports that basically conduct a Python analysis using an algorithm</a> <a data-t="906">called fact decomposition, and this estimates the hallucination rate across</a> <a data-t="910">our chatter deployments for us.</a></p>

<p><a data-t="914">So how does fact decomposition work? Bear with</a> <a data-t="918">me, it's like an algorithmic talk. You start with the output text</a> <a data-t="922">that an LLM generates, and then you use either natural language processing or</a> <a data-t="926">another LLM to break it into the individual claims that it contains. So</a> <a data-t="932">for example, if the claim or if the output from the LLM says this patient</a> <a data-t="938">came to the neurology clinic to have their headache evaluated, the individual</a> <a data-t="942">claims would be that the patient has a headache and that they went to the</a> <a data-t="945">neurology clinic, and maybe also a third one that's like that their evaluation of</a> <a data-t="948">the neurology clinic was about their headache. Once you have these individual</a> <a data-t="952">claims, you can then use a powerful reasoning model or a smaller fine-tuned</a> <a data-t="956">language model to cross-reference all of the individual claims with all of the</a> <a data-t="961">clinical notes within the relevant time period of that patient's chart.</a> <a data-t="965">This basically allows you then to flag individual claims that are not supported</a> <a data-t="970">by any of the evidence present in the patient's record, and using simple</a> <a data-t="975">division you can then compute, like estimate a rate at which like how many</a> <a data-t="979">claims or what proportion of claims are supported versus not supported.</a></p>

<p><a data-t="984">So as an</a> <a data-t="985">example, this is a plot that I pulled or that I made in plot 9 that I pulled</a> <a data-t="989">from our most recent partial report that ran for the month of August, and</a> <a data-t="992">basically what we're doing is we're estimating the hallucination rates</a> <a data-t="995">across our different deployments among three of the different models that we</a> <a data-t="999">route to within the chatter infrastructure. What you can see is that</a> <a data-t="1002">one of the models that is becoming a bit dated at this point, GPT-40 mini, was</a> <a data-t="1007">hallucinating at a much much higher rate than the other models that we were using,</a> <a data-t="1011">so this resulted in the recommendation basically to deprecate that model and</a> <a data-t="1015">switch to one that was more reliable.</a></p>
