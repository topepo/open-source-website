<h3>What is an API?</h3>

<p><a data-t="163">What is an API in general?</a> <a data-t="166">You can think of an API as a set of instructions for how a program should interact with a piece</a> <a data-t="170">of software.</a> <a data-t="172">So if you create a piece of software that does something useful and you think someone</a> <a data-t="176">who's writing a program somewhere else might want to use your software, you would set up</a> <a data-t="179">an API which tells that developer how their program should send requests or interact with</a> <a data-t="185">your piece of software.</a></p>

<p><a data-t="187">And an API can be an interface to a database, to a computer's operating system, to a software</a> <a data-t="193">package that does something, or commonly to web applications.</a> <a data-t="198">And we're going to focus on this last case today.</a> <a data-t="200">We're going to focus on how you can use an API to a web data source to collect data into</a> <a data-t="206">R.</a></p>

<p><a data-t="207">And if you wanted to visualize, you know, how APIs work, you could imagine or you could</a> <a data-t="214">think of something that you do every day that involves an API in the background, and that's</a> <a data-t="218">browsing the Internet.</a> <a data-t="220">When you open a web page on your computer, the pictures that you see on the web page,</a> <a data-t="225">the text, the links, all that exists somewhere on a server on the other side of the Internet.</a> <a data-t="231">And your web browser on your computer locates that server that's storing all this information.</a> <a data-t="236">It sends a request to the server using the server's web API, saying, please give me the</a> <a data-t="242">pictures so I can display them on my user's screen, and here she could see what's on your</a> <a data-t="247">web page.</a></p>

<p><a data-t="247">And then the server sends a response back that hopefully contains the pictures and the</a> <a data-t="251">text, everything the web browser needs to build that web page locally for you to look</a> <a data-t="254">at.</a> <a data-t="255">Or the server might send back some sort of error message saying, I can't do that for</a> <a data-t="259">whatever reason.</a> <a data-t="259">The instructions, the requests that your web browser sends to the server, are formatted</a> <a data-t="265">according to the API that that server uses, or that that program uses.</a></p>

<p><a data-t="270">So when you write a piece of software, like I guess Tim Berners-Lee wrote the Internet,</a> <a data-t="279">you can decide how you want this all to work.</a> <a data-t="281">And then anyone who wants to work with that software should follow your rules, otherwise</a> <a data-t="284">your software's not going to respond the way that they want.</a> <a data-t="286">So when your web browser connects to a web server, there's a specific way of doing that</a> <a data-t="290">that your web browser uses, and that is with an API protocol called HTTP.</a> <a data-t="295">And that's what we're going to look at here today.</a></p>

<h3>HTTP and URLs</h3>

<p><a data-t="299">So HTTP is standard for all sorts of web interfaces.</a> <a data-t="304">I mentioned before my background's not in web development.</a> <a data-t="306">I imagine there are some exotic places where you might find something other than HTTP,</a> <a data-t="311">experimental locations, but really I don't know of any.</a> <a data-t="313">So I think HTTP is pretty standard.</a> <a data-t="316">What we're going to try to do today is use the HTTR package to equip R to behave like</a> <a data-t="321">a web browser, in that it can fetch resources from a web server straight into R.</a> <a data-t="327">Those resources could be things you see on a web page like images and text, or they could</a> <a data-t="331">be actual data that's been saved on the Internet for you to download.</a></p>

<p><a data-t="337">So let's look at HTTP, the protocol that we'll use in our web APIs.</a> <a data-t="342">It stands for Hypertext Transfer Protocol.</a> <a data-t="345">The Internet is basically built on hypertext, which is text that's linked to other pieces</a> <a data-t="348">of text and so on.</a> <a data-t="350">And HTTP itself revolves around the idea of URLs, which is another acronym for Uniform</a> <a data-t="357">Resource Locator.</a> <a data-t="358">And a URL is what you know as a web address, it's what you type into your web browsers.</a> <a data-t="363">And you can think of a URL more generally as a file path to a resource on the Internet.</a></p>

<p><a data-t="371">Just like a file path on your computer, there are different parts to this URL that directs</a> <a data-t="377">your computer to the actual resource, the server to the actual resource.</a> <a data-t="380">This is how you identify the resources that you want to request or that you want to alter</a> <a data-t="384">or delete or whatever you want to do through the API.</a></p>

<p><a data-t="388">And just going back to this, there are different parts of this URL that I've put here, and</a> <a data-t="397">many of them are optional.</a> <a data-t="399">You very often won't have query parameters or an ID or whatnot, but all these things</a> <a data-t="404">can work together to form a URL.</a> <a data-t="407">Or you might be able to get resources as simple as www.host.com.</a></p>

<p><a data-t="412">The way HTTP is organized is you make requests, the client makes requests that are sent to</a> <a data-t="421">the server, and then the server responds with responses.</a> <a data-t="425">And both the request message and the response message are basically plain text files, and</a> <a data-t="431">they share the same structure.</a> <a data-t="433">So plain text is really easy to pass over the Internet or over any network.</a> <a data-t="438">And so HTTP is taking advantage of that.</a></p>

<p><a data-t="441">If you wanted to look at an HTTP message that is sent, whether it's a response or a request,</a> <a data-t="447">it's going to have the same three-part structure.</a> <a data-t="448">There's going to be an initial line that contains very important information.</a> <a data-t="452">What's in that line will depend on whether or not this is a request or a response.</a> <a data-t="456">We'll look at those separately in a moment.</a> <a data-t="459">There's going to be, or there can be, a set of optional headers, which are just key value</a> <a data-t="464">pairs separated by a colon.</a> <a data-t="466">Those can contain useful information.</a> <a data-t="468">And then also optionally, there may be a message body at the end of the message, separated</a> <a data-t="472">from the beginning by a blank line.</a> <a data-t="475">And that message body is where contents would be if the message contains an image or a data</a> <a data-t="480">set or whatnot.</a></p>

<h3>HTTP requests and verbs</h3>

<p><a data-t="497">So let's look at a sample request.</a> <a data-t="501">You can't get a response from a server until you send a request, so you need to learn how</a> <a data-t="504">to send a request before you learn how to interpret a response.</a> <a data-t="512">And this would be a real HTTP request going out to the webinars page of RStudio.com, which</a> <a data-t="517">is somewhere you've probably visited if you managed to wind up here, and it's just asking</a> <a data-t="522">for the information that a web browser would need to display this webpage on your computer.</a></p>

<p><a data-t="526">If we look at it, we can see it has that three-part structure, although since this request is</a> <a data-t="531">not really sending information for the server to use, it's just saying, please server, give</a> <a data-t="536">me something back.</a> <a data-t="537">There's no body.</a> <a data-t="538">There's no content to it.</a> <a data-t="539">It's just an initial line and some headers with some metadata.</a></p>

<p><a data-t="544">And this initial line is broken into three parts, and this is how the initial line of</a> <a data-t="548">any HTTP request will be organized.</a> <a data-t="552">The first line will be broken into three parts, each separated by a space, and the last part</a> <a data-t="557">for a request is the most trivial.</a> <a data-t="559">It's the version of HTTP that the request is using.</a> <a data-t="562">We're not going to have to pay attention to that today.</a></p>

<p><a data-t="568">Next part is the URL path to the resource that the message is pertaining to, and you'll</a> <a data-t="573">notice that this isn't a complete URL path, but your computer will have already established</a> <a data-t="578">a connection through a port to the host, which is www.rstudio.com, so the host knows what</a> <a data-t="583">it is.</a> <a data-t="584">The host needs to know where in all of its files is the resource you want to know about,</a> <a data-t="589">and that's the remaining part of the URL.</a> <a data-t="594">So here, resources slash webinars takes you to the webinars page of rstudio.com.</a></p>

<p><a data-t="599">And then the first part of the first line is probably the most important part of your</a> <a data-t="604">request, and that is an HTTP verb that tells the server what you want to do.</a> <a data-t="610">There are a collection, a small collection of verbs that you can use with HTTP, but the</a> <a data-t="615">most important ones are probably the four that I'm going to list here.</a> <a data-t="620">We're using a get verb in that sample, and get is the verb you use when you want to retrieve</a> <a data-t="625">a resource that's specified by a URL.</a> <a data-t="627">You send a get request to the server, and the server knows that it should send you back</a> <a data-t="631">what you're asking for, or tell you that you can't have it, it doesn't exist, you know,</a> <a data-t="636">something like that.</a></p>

<p><a data-t="637">You can also send a post request, which creates the resource out of URL.</a> <a data-t="644">So when you send a post request to a server, you normally attach some data to it, the server</a> <a data-t="648">gets this, and then it uses that data to build a new resource at that URL, which will then</a> <a data-t="652">exist and other people can get it and use it.</a> <a data-t="656">You can use a put request, which is similar to post, but it works with resources that</a> <a data-t="660">already exist.</a> <a data-t="661">You use a put to update that resource with data that you store in the contents of your</a> <a data-t="665">message.</a> <a data-t="666">And then you can also send a delete request, which would delete a resource that exists</a> <a data-t="669">on that server at that URL.</a></p>

<p><a data-t="671">So like I said, there's other verbs, but these are the big four, the most important</a> <a data-t="676">ones.</a> <a data-t="677">And this is really the place where you as an R developer need to start paying attention.</a> <a data-t="681">So the information up to here was all contextual, it's all about how the web is created, but</a> <a data-t="686">these verbs are things that you will use from inside of R.</a></p>
