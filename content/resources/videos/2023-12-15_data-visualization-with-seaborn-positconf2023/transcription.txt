<h3>Origins and early development of Seaborn</h3>

<p><a data-t="59">Seaborn started out with pretty humble ambitions.</a> <a data-t="59">It was essentially a toolbox or an extension to Matplotlib that offered a few plotting functions with statistical capabilities, doing</a> <a data-t="64">things like fitting a kernel density estimate to a histogram or a linear regression line</a> <a data-t="70">to a scatter plot.</a></p>

<p><a data-t="70">And these functions behaved a lot like the rest of Matplotlib.</a> <a data-t="70">They primarily operated on NumPy arrays, because that was the main data structure at the time.</a></p>

<p><a data-t="73">But this was around when pandas was starting to get popular and bringing the idea of a</a> <a data-t="78">labeled columnar data structure to Python.</a> <a data-t="82">And so some of the early Seaborn functions built on that and borrowed some ideas from</a> <a data-t="87">the R community about tidy data, using data frames at first really to make faceted or</a> <a data-t="92">paired small multiple plots with features like conditional color mapping, where the</a> <a data-t="95">color mapping was really thought of as a faceting operation on the third dimension of the plot.</a></p>

<p><a data-t="124">And as Seaborn developed, I was primarily concerned with filling gaps in Matplotlib's</a> <a data-t="128">feature set, especially features that were important for emerging data science applications.</a> <a data-t="135">So Matplotlib's roots are in the physical sciences, and it hasn't always served analytics</a> <a data-t="140">and machine learning use cases as well as it could.</a></p>

<p><a data-t="144">So for instance, one of the first major distinguishing features of Seaborn was its support for categorical</a> <a data-t="151">data and kind of plots that are optimized for categorical data.</a></p>

<p><a data-t="156">And eventually, this landed us in a kind of odd situation where Seaborn had support</a> <a data-t="162">for some fairly complex graphics, things like regression or violin plots, but not simple</a> <a data-t="167">ones, like just basic line or scatter plots.</a> <a data-t="171">And remember, the initial intention or thought was add on to what Matplotlib doesn't have,</a> <a data-t="177">and Matplotlib already had those things.</a></p>

<p><a data-t="181">But it didn't have this tidy API, API where you could just pass in a data frame and name</a> <a data-t="185">some columns and have those automatically mapped to components of the plot.</a> <a data-t="190">So almost, I think, five years or so after the first release of Seaborn, it kind of finally</a> <a data-t="195">gained such basics as a line plot and a scatter plot function.</a></p>

<p><a data-t="201">But not just in their simplest forms, in forms that had support for mapping now up to three</a> <a data-t="205">additional dimensions based on the columns in a data frame, including color and size</a> <a data-t="211">and the style of the elements in the plot.</a></p>

<p><a data-t="216">And so Seaborn's objective remained help people make plots faster so that they can understand</a> <a data-t="222">their data better, but its focus and how it did that shifted from do their statistics</a> <a data-t="229">for them to give them an intuitive interface that lets them rapidly iterate and think about</a> <a data-t="236">their data, not the mechanics of specifying a graphic.</a></p>

<p><a data-t="241">And so that prompted me to revisit even some of the oldest functions in the library and</a> <a data-t="246">to try to impose some API consistency onto these tools that had grown rather organically</a> <a data-t="252">and at the start without much intentional thought about API design.</a></p>

<h3>The classic Seaborn API: strengths and limitations</h3>

<p><a data-t="259">So despite this somewhat haphazard and organic development, the result, the kind of status</a> <a data-t="264">quo is a library that can be used to produce a relatively wide range of informative statistical</a> <a data-t="270">graphics with a reasonably coherent API.</a></p>

<p><a data-t="276">And the API model, the API encourages a mental model where you might say, I want to make</a> <a data-t="283">a line plot, and then there's a function called line plot that you can do to do that.</a> <a data-t="288">Or there's a function called bar plot, and you can probably guess what that does.</a> <a data-t="292">So one objective of Seaborn is that each of these functions should be able to produce</a> <a data-t="296">a usable plot with a bare minimum set of arguments, really just enough to specify the data.</a></p>

<p><a data-t="302">They make some inferences about what you want based on the data types, and they try to use</a> <a data-t="307">sensible defaults, and they produce complete graphics.</a> <a data-t="311">They add labels and legends, so you can very quickly get from an idea or a question you</a> <a data-t="315">have to a plot that you could at least show to one of your colleagues, and they'd understand</a> <a data-t="319">what it is.</a></p>

<p><a data-t="322">But customization is important.</a> <a data-t="325">Good customization requires rapid iteration.</a> <a data-t="328">And so each function exposes a number of additional parameters that afford varying kinds of control.</a> <a data-t="334">So these range from specifying the mappings between your dataset to the visual properties</a> <a data-t="339">in the plot, setting hyperparameters of whatever statistical operations that function's going</a> <a data-t="344">to perform, and tweaking the plot's final appearance.</a></p>

<p><a data-t="349">Now, it's not quite true that you can customize everything just through the parameters of</a> <a data-t="354">the Seaborn functions.</a> <a data-t="356">Recall that it started with the idea of, you know, like Matplotlib, but with some statistics,</a> <a data-t="361">and so the functions were meant to fit into the standard Matplotlib workflow, which means</a> <a data-t="366">that while Seaborn will add default labels to the plot, if you want to customize them,</a> <a data-t="371">you'd use Matplotlib's API for that.</a></p>

<p><a data-t="375">Still, you can get most of the way there by interacting just with the Seaborn API, and</a> <a data-t="380">that's helped, you know, a lot of people make a lot of useful plots.</a> <a data-t="384">But I've also come to appreciate, you know, over the years, some of the limitations that</a> <a data-t="387">this design imposes.</a></p>

<p><a data-t="392">So one obvious consequence is that each of these parameters has a, or each of these functions</a> <a data-t="397">has a lot of parameters.</a> <a data-t="400">And any one function's parameters do a lot of different kinds of things.</a> <a data-t="405">So while on the one hand, you know, it's straightforward to explain how Seaborn works.</a> <a data-t="409">You're just specifying parameters.</a> <a data-t="411">You find a function that you think is going to do what you want, you look at its list</a> <a data-t="414">of parameters to see what options it has.</a></p>

<p><a data-t="417">But with so many parameters, and just in one, you know, linear list, it can be hard to know,</a> <a data-t="423">you know, what bits and pieces you're going to need to solve any specific problem.</a> <a data-t="429">And then because Seaborn is a higher level interface to Matplotlib, some of the parameters</a> <a data-t="434">that you need to set might actually be Matplotlib parameters.</a> <a data-t="439">And even just within Seaborn, some functions are at a higher level than others.</a> <a data-t="444">And so the API relies heavily on keyword argument passing, which means to understand what parameters</a> <a data-t="451">a function takes, you might need to walk through several different pages of documentation across</a> <a data-t="456">multiple project websites.</a></p>

<p><a data-t="461">And the, you know, organic development of the library has led to an expanding scope</a> <a data-t="466">over time has led to some other opportunities for friction or confusion.</a> <a data-t="471">So take, you know, as a case study, the strip plot and the scatterplot functions.</a> <a data-t="477">These are both fundamentally scatterplots, or what they create are fundamentally scatterplots.</a> <a data-t="482">But as I mentioned earlier, the function called strip plot actually came first, because</a> <a data-t="487">it's a scatterplot that's optimized for categorical data.</a> <a data-t="490">It's meant as an alternative or a complement to a box or violin plot.</a> <a data-t="495">So it has capabilities like dodging and jittering that makes sense in that context.</a></p>

<p><a data-t="503">And then the scatterplot function came later, producing plots that are in one sense more</a> <a data-t="507">basic, but on the other hand have more options for mapping properties based on data.</a> <a data-t="513">They allow you to control the size of the dots and the shape of the markers, whereas</a> <a data-t="516">strip plot only controls the color.</a> <a data-t="520">So you know, you might find yourself starting to make a plot using one of these functions,</a> <a data-t="524">realizing that it can't do everything you want, switching to one of the other functions,</a> <a data-t="528">and then realizing you've lost some of the features that you were relying on.</a></p>

<p><a data-t="534">Now you can always add features, but the design makes this hard too.</a> <a data-t="539">And paradoxically, it can feel hardest to add small features, small little nice-to-haves</a> <a data-t="544">that just add a little polish or satisfy a few less common use cases.</a></p>

<p><a data-t="551">So as an example, take the concept of adding a gap or a little spacing between the boxes</a> <a data-t="557">in a box plot when you dodge them.</a> <a data-t="559">It's a totally reasonable thing, you know, I like the way this looks.</a> <a data-t="564">But implementing it would mean adding a parameter to box plot to control whether there's a gap,</a> <a data-t="571">how big the gap is.</a> <a data-t="573">And so that already long list of parameters is going to grow a little bit, and that's</a> <a data-t="577">going to impose an ongoing cost to usability.</a> <a data-t="580">Everyone in the future who comes and looks at the list of parameters is going to have</a> <a data-t="583">one more that they need to sift through to understand what they can do.</a></p>

<p><a data-t="589">And here's the thing, despite the way it's articulated on GitHub here, this isn't really</a> <a data-t="593">just a request about the box plot function.</a> <a data-t="597">Adding this would also mean changing the bar plot function and the violin plot function.</a> <a data-t="604">All of the other functions that have some sort of dodging operation, you either have</a> <a data-t="610">to change all of them or you have to add more inconsistency or accept more inconsistency</a> <a data-t="614">in the library.</a> <a data-t="616">And so beyond implementing that operation in the code itself, each function also has</a> <a data-t="621">tests and documentation that would need to be updated.</a> <a data-t="624">So it just increases the costs of adding even small things.</a></p>

<p><a data-t="630">When Seaborn's functions don't support a feature, one option is to drop down to the underlying</a> <a data-t="634">matplotlib layer or accomplish what you want there.</a> <a data-t="637">Matplotlib's still a Python library, lots of people know how to use it, and this was</a> <a data-t="641">part of the original design, right?</a> <a data-t="642">Seaborn was just adding some things that you couldn't do in matplotlib, so if there was</a> <a data-t="646">something it couldn't do, no harm, you just carry on as if Seaborn wasn't there.</a></p>

<p><a data-t="651">But as Seaborn has developed more capabilities in its own style of API, the gap between the</a> <a data-t="657">two libraries has grown, and so it's no longer a valid assumption, and it's also no longer</a> <a data-t="662">a valid assumption that any Seaborn user is also comfortable using matplotlib.</a> <a data-t="667">Lots of places have started teaching data science using Seaborn because it's a little</a> <a data-t="671">easier, a little bit faster to get started out of the box.</a></p>

<blockquote><a data-t="651">But as Seaborn has developed more capabilities in its own style of API, the gap between the</a> <a data-t="657">two libraries has grown, and so it's no longer a valid assumption, and it's also no longer</a> <a data-t="662">a valid assumption that any Seaborn user is also comfortable using matplotlib.</a></blockquote>

<h3>Introducing the Objects Interface</h3>

<p><a data-t="676">So after nearly a decade, I decided to take a step back and think, if I were designing</a> <a data-t="681">this library from scratch and had from the start a focus on a flexible but consistent</a> <a data-t="687">interface that was easy to use and didn't constrain what you could do, what would that</a> <a data-t="692">look like?</a></p>

<p><a data-t="696">So the result is what I'm calling the Objects Interface, or Seaborn.objects.</a> <a data-t="701">It's a totally new approach, basically a ground-up rewrite that takes inspiration both from things</a> <a data-t="707">that worked well in classic Seaborn and from libraries that are more explicitly based on</a> <a data-t="712">the formal grammar of graphics, like ggplot, also Vega, parts of D3, other libraries that</a> <a data-t="720">are out there that are built on some of these ideas.</a></p>

<p><a data-t="725">So while people have sometimes referred to classic Seaborn as ggplot in Python, I've</a> <a data-t="730">always thought that's wrong, or at least overselling it.</a> <a data-t="734">Seaborn has certainly taken ideas and inspiration from ggplot since the beginning, but it's</a> <a data-t="738">also always had a narrower ambition.</a> <a data-t="740">It's not intended to be a fully flexible system for producing graphics.</a></p>

<p><a data-t="746">The Objects Interface is much more explicitly based on the underlying formalism and also</a> <a data-t="752">has somewhat broader ambitions to be more general.</a> <a data-t="756">It's still not ggplot in Python, though.</a> <a data-t="758">It's not going to be a direct port of the syntax that you might be used to if you're</a> <a data-t="763">a ggplot user.</a> <a data-t="765">It should feel familiar, I hope.</a> <a data-t="767">I think the learning curve should be pretty rapid, but it's a different library.</a></p>

<p><a data-t="773">I always say that if what you want is just to take your ggplot code and have it run in</a> <a data-t="778">Python, you should check out the Plot9 library, and today you're in luck because you just</a> <a data-t="782">have to wait a few minutes and you'll hear about that.</a> <a data-t="785">But for the time being, let me walk you through how I've approached this in Seaborn.</a></p>

<p><a data-t="792">So everything in the Objects Interface is built around an object called plot.</a> <a data-t="797">Plot has a number of methods, and you call these methods to specify how various aspects</a> <a data-t="802">of the graph should appear.</a> <a data-t="803">So that includes data specification, adding layers, parameterizing the scales, customizing</a> <a data-t="808">the decorations, and the theme of the plot.</a> <a data-t="811">Every time you call one of its methods, you're gradually adding on to the spec.</a></p>

<p><a data-t="817">And it's called the Objects Interface because beyond plot, there's a number of other objects</a> <a data-t="821">that you pass to plot's methods to define the graph.</a> <a data-t="826">So there's four basic types of objects, mark, stat, move, and scale.</a> <a data-t="833">A mark is a visual element like a dot or a line, a stat defines a statistical operation,</a> <a data-t="839">a move implements a positional adjustment, and a scale maps from data to visual properties.</a></p>

<h3>How the Objects Interface works</h3>

<p><a data-t="847">So you call the plot constructor with a data source and some assignments from variables</a> <a data-t="854">in the data set to roles in the plot.</a> <a data-t="857">Now this looks a lot like a function call in the classic interface, but if we compiled</a> <a data-t="861">this spec, we would see just an empty graph with some labels on the axes.</a> <a data-t="866">You won't actually get a data visualization until we add layers to the plot.</a></p>

<p><a data-t="872">And you do that, hopefully intuitively, with a method called plot.add.</a> <a data-t="877">So every layer is defined by at least a mark object.</a> <a data-t="881">Here I'm using a dot mark, so it's producing a scatter plot.</a></p>

<p><a data-t="887">And you'll notice that when I define the layer, I'm setting the point size of the dot to a</a> <a data-t="891">specific value, and then also associating the marker, the dot marker, with another column</a> <a data-t="897">from the data frame.</a></p>

<p><a data-t="900">So there's three ways to define visual properties here.</a> <a data-t="903">There's properties that are in the plot constructor that are mapped in all layers.</a> <a data-t="908">There's properties defined in the mark constructor that are set directly to the value you give,</a> <a data-t="914">independent of any of the data.</a> <a data-t="916">And then properties defined in the call to plot.add, which are mapped, but in a layer-specific way.</a></p>

<p><a data-t="924">For this plot, there's no real difference between whether you specify color in the plot</a> <a data-t="928">constructor or when you're calling add, but if we added another layer, such as a line</a> <a data-t="934">representing a regression fit, you'd see that only the colors of the line are mapped.</a></p>

<p><a data-t="944">So as in classic Seaborn, the default scales are chosen for each mapping variable based</a> <a data-t="949">on the data types and other inferences about the plot spec.</a> <a data-t="954">But there's a method called plot.scale that allows you to explicitly override and supply</a> <a data-t="959">the mapping you want to use.</a> <a data-t="961">So for example, here I'm choosing a slightly more intuitive mapping from true-false to</a> <a data-t="966">a circle and an X.</a> <a data-t="969">For full customization, you can pass an instance of a scale object, like a nominal scale or</a> <a data-t="974">a continuous scale.</a> <a data-t="976">There's also a number of shorthands, like just passing a dictionary or a list that imply</a> <a data-t="980">a specific scale type.</a></p>

<p><a data-t="985">The notion of small multiple plots remains very important to Seaborn.</a> <a data-t="989">Any plot can be distributed across a grid of subplots by calling plot.facet and supplying</a> <a data-t="995">one or two additional columns from your data frame that define the rows and the columns</a> <a data-t="1001">in the facet grid.</a></p>

<p><a data-t="1004">And there's also a method called plot.pair, which also produces subplots that look a lot</a> <a data-t="1010">like a facet plot, but much like the classic Seaborn pair grid class or pair plot function,</a> <a data-t="1017">here we're taking multiple columns from the initial data frame and assigning them to the</a> <a data-t="1023">different subplot X or Y axes.</a> <a data-t="1026">So this is like faceting, and you could basically reshape your data and also accomplish this</a> <a data-t="1031">through a facet operation, but sometimes it's more convenient to skip the reshaping step.</a></p>

<p><a data-t="1038">So unlike in the classic Seaborn interface, you use the same term to either map a property</a> <a data-t="1043">based on data or set it directly.</a> <a data-t="1045">What matters is where you do it.</a> <a data-t="1048">And you can map a much wider range of visual properties.</a> <a data-t="1050">So that includes multiple color attributes, the alpha or the opacity of the mark, and</a> <a data-t="1056">whether the mark is filled.</a></p>

<p><a data-t="1058">Where appropriate, you can control style properties like the marker shape or the dash pattern</a> <a data-t="1063">that's used for a line or the edge of a patch.</a> <a data-t="1066">There's a number of size-based properties like the point size of a dot mark, the width</a> <a data-t="1070">of the lines or edges of a patch, and the strokes that define line art markers.</a> <a data-t="1077">If you have a text mark in your plot, you can also control the font size, alignment,</a> <a data-t="1081">and offset from the anchor point.</a> <a data-t="1083">Usually you'd set these directly, but it's possible to map them too.</a> <a data-t="1086">There's no distinction between mappable and unmappable parameters.</a></p>

<p><a data-t="1092">So I think the ergonomics here are a lot better.</a> <a data-t="1094">There's no more keyword redirection across multiple functions.</a> <a data-t="1097">If you're in a notebook or an IDE, all the properties supported by plot or by a specific</a> <a data-t="1102">mark are just going to tab complete out.</a> <a data-t="1104">The documentation is going to be right there.</a></p>

<p><a data-t="1107">If you think back to the discussion about adding a gap parameter to all the functions</a> <a data-t="1111">that support dodging, that's no longer relevant here.</a> <a data-t="1115">There's just a single object.</a> <a data-t="1116">It's called dodge.</a> <a data-t="1117">You add it to any plot spec, and the marks in that layer will get dodged.</a> <a data-t="1122">And the gap parameter can just be defined there.</a> <a data-t="1126">There's only three parameters in this class.</a> <a data-t="1128">It's very easy to understand what it can do, and then it composes with any mark.</a> <a data-t="1134">So we're not in the situation with strip plot and scatter plot where you have to choose</a> <a data-t="1138">between categorical adjustments and mapping additional properties.</a></p>

<p><a data-t="1143">The objects interface lets you combine all of these operations into one plot spec.</a> <a data-t="1149">Now with that said, this particular example is maybe a little overstuffed with information.</a> <a data-t="1155">The classic interface is opinionated, which is a nice way of saying it limits what you</a> <a data-t="1160">can do.</a> <a data-t="1162">And so usually that takes you towards doing things that make sense, but sometimes it prevents</a> <a data-t="1166">you from doing things that make sense in a specific context.</a></p>

<p><a data-t="1171">So the objects interface will make you make plots that maybe don't make so much sense.</a> <a data-t="1177">And that's because there will be some data sets, some contexts where this specific set</a> <a data-t="1181">of operations, when you throw them all together, does produce an informative graph.</a> <a data-t="1184">And so when I was developing the new interface, one objective I had in mind was make it possible</a> <a data-t="1190">to produce graphs that really don't make any sense so that you avoid preventing people</a> <a data-t="1195">from making plots that do.</a></p>
