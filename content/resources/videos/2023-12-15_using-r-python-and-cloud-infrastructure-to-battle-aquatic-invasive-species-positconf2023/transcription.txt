<h3>Background on the AIS Explorer</h3>

<p><a data-t="153">So let's have a little bit of background about this Shiny app.</a> <a data-t="156">So this was initially conceptualized back in 2019 with the support of the people at</a> <a data-t="162">the University of Minnesota, and specifically the Minnesota Aquatic Invasive Species Research</a> <a data-t="167">Center.</a> <a data-t="168">We wouldn't be able to do any of this without the support of these people.</a> <a data-t="172">So special call out, thank you to Nick Phelps, Amy Kinsley, and Alex Badges for all their</a> <a data-t="177">support over the years.</a> <a data-t="178">They've been wonderful.</a></p>

<p><a data-t="180">So the AIS Explorer started its life as a decision support and risk management tool</a> <a data-t="185">to help decision makers on the ground in the state protect their lakes from these invasive</a> <a data-t="192">species by linking up these complex analytics with these practical decisions that they need</a> <a data-t="197">to make day-to-day.</a> <a data-t="200">We finished development of the first version of AIS Explorer back in mid-2020, and it has</a> <a data-t="206">remained in production since then, and it's been actively in use during that time.</a> <a data-t="211">So these decision makers, the county managers, have been using this app.</a> <a data-t="214">It's battle-tested, and it's been through several updates over that time.</a> <a data-t="218">We've continued to maintain it and provide new features for the people who are using</a> <a data-t="222">it.</a></p>

<p><a data-t="223">And there's some pretty complex stuff going on behind the scenes here.</a> <a data-t="227">Now, I'm not a data scientist.</a> <a data-t="229">My background is in software, so I probably can't tell you a lot about the details of</a> <a data-t="233">these models, but I can tell you that we've linked up these kind of collection of Python</a> <a data-t="239">and R models into the application.</a> <a data-t="241">So briefly, we have a risk model written in Python, which is used to determine the risk</a> <a data-t="248">of lakes becoming infested within the next eight years, I think.</a> <a data-t="253">Then we also have an intervention model, which allows users to kind of produce a simulation</a> <a data-t="258">of if they intervene at certain lakes in certain ways, so things like inspections or decontamination</a> <a data-t="265">or education and outreach, how is that going to change the story for these lakes?</a> <a data-t="271">We have an inspection model, which allows people to determine the most cost-effective</a> <a data-t="275">or the most beneficial way that they can place inspectors around their lakes in a county.</a> <a data-t="281">And then finally, the collaboration model, one of the new additions, which is enabling</a> <a data-t="286">county managers to see how the story changes if they pool their resources together.</a></p>

<p><a data-t="292">So these models are pretty complicated, and that makes it a bit of a challenging process</a> <a data-t="296">to keep them fed, watered, operating as they need to.</a> <a data-t="302">The source data that feeds these models updates pretty frequently, and the computations which</a> <a data-t="307">happen to produce the results we present in the app are pretty intensive.</a> <a data-t="312">It takes quite a long time.</a> <a data-t="314">We want to make sure that our Shiny app performs well for the people who are using it.</a> <a data-t="317">So we need to kind of address that in the way that we're hosting this app.</a> <a data-t="322">And even if we wanted to go with kind of a manual update process, that's not going to</a> <a data-t="326">be practical for us in this case either, because we don't actually know when these updates</a> <a data-t="331">are going to need to be applied.</a> <a data-t="332">It's a bit of a variable timing on that one.</a></p>

<h3>Accessibility: hosting on AWS</h3>

<p><a data-t="348">We're going to start off firstly with accessibility.</a> <a data-t="352">This is how we're making sure that these people can actually use the Shiny app that we've</a> <a data-t="356">built and prepared for them.</a> <a data-t="358">And the first question we ask is, where do we actually host this?</a> <a data-t="362">There's usually a couple of options between, do we host it kind of on-prem on some server</a> <a data-t="367">that we have, or do we host it somewhere in the cloud?</a> <a data-t="370">For obvious reasons, it's probably not going to be great if we host it back in our office</a> <a data-t="374">in Wellington.</a> <a data-t="376">So we decided to go with some cloud infrastructure.</a> <a data-t="379">And some discussions with the folks at Mazark led us to Amazon Web Services.</a> <a data-t="384">Over 200 cloud-based services for us to choose from, 32 different regions, 102 availability</a> <a data-t="390">zones, which means that we can take our application and put it right next door, where it's going</a> <a data-t="395">to be nice and accessible for the folks in Minnesota who need to use it.</a></p>

<p><a data-t="401">So now that we've decided on AWS, that leads us to the Elastic Compute Cloud, or EC2 instances.</a> <a data-t="408">So these cloud-based servers, which we create one of these and upload the code for our Shiny</a> <a data-t="413">application into one of these instances.</a> <a data-t="416">And from this, we can do a lot of different configurations if we need to.</a> <a data-t="420">We can adjust the performance.</a> <a data-t="421">We can change security groups.</a> <a data-t="423">We can customize these instances however we need to, really.</a> <a data-t="427">We can also pick from a variety of different performance tiers.</a> <a data-t="430">So we can crank up the performance of the application by just picking a different option</a> <a data-t="436">from a dropdown and restarting the server, which can be really helpful, because this</a> <a data-t="440">app can take quite a bit of effort sometimes.</a> <a data-t="443">The other really useful thing about this is that we get to record our backups, and we</a> <a data-t="447">can take kind of a snapshot of the state of these servers using the Amazon machine images,</a> <a data-t="452">these AMIs.</a> <a data-t="453">And they're going to come in handy later as well.</a></p>

<p><a data-t="457">So we have our Shiny code on this EC2 instance, and then we actually need to have some way</a> <a data-t="462">of running the Shiny app.</a> <a data-t="463">And a little bit of a fun detour we had to take with this one is that we actually run</a> <a data-t="468">the Shiny app directly on this EC2 instance.</a> <a data-t="471">Now traditionally, you would probably go with a Shiny server, but as I painfully found out,</a> <a data-t="477">while we were kind of going through this initial development, there are some edge cases in</a> <a data-t="479">here that come up when you try to throw 18,000 lakes onto a single </a><a href="/software/leaflet/">leaflet</a><a data-t="479"> map, which meant</a> <a data-t="485">that Shiny server actually wasn't really having it for us.</a> <a data-t="488">So we decided to just run the app directly on the EC2 instance.</a> <a data-t="492">And that brings up some questions, which we'll talk about later, about how we deal with multiple</a> <a data-t="496">users.</a></p>

<h3>Relevance: automated data updates</h3>

<p><a data-t="498">But with these solutions, we've made our app accessible to the people in Minnesota, and</a> <a data-t="502">they can use it, and it should be performing reasonably well.</a> <a data-t="505">The next thing we need to think about is how do we keep this relevant?</a> <a data-t="509">So we've got these four complicated models in here.</a> <a data-t="513">And like I said, I'm not going to go too much into the details of these models, but there's</a> <a data-t="517">a couple of things to call out here.</a> <a data-t="519">So if you have a look at the kind of runtime of these different models, you can see it's</a> <a data-t="524">a pretty variable scale here.</a> <a data-t="526">We've got models which take sort of anywhere from one to five minutes, depending on the</a> <a data-t="529">inputs, up to 20 hours or more.</a> <a data-t="534">So it's a pretty variable range of models and how long it takes for these outputs to</a> <a data-t="538">be produced.</a> <a data-t="540">And all of these need to come into our app at the same time.</a></p>

<p><a data-t="543">The other thing to call out here is that there's a common thread running between these models,</a> <a data-t="547">which is the source data, which is coming from the DNR infested waters list.</a> <a data-t="551">So this is produced by the Minnesota Department of Natural Resources.</a> <a data-t="555">And basically, this is feeding everything else which we put into the AIS Explorer.</a> <a data-t="560">All of our models depend on this Excel file, which lives on the DNR website.</a> <a data-t="566">So we obviously need to know when this changes, and that's going to be the trigger for us</a> <a data-t="571">to actually run our models again and produce the results that we need.</a> <a data-t="576">Specifically from that, we need two things.</a> <a data-t="578">We need the species that a lake is infested with, and we need the DOW number.</a> <a data-t="583">It's a unique identifier for each water body which is infested in that list.</a></p>

<p><a data-t="589">I talked earlier about a variable update timing, and that's down to the season.</a> <a data-t="595">So when it's winter in Minnesota, the lakes freeze over, people aren't really inspecting</a> <a data-t="601">all that much.</a> <a data-t="602">So they don't get a lot of new data in this spreadsheet.</a> <a data-t="605">But during the summer, we might be seeing multiple times a week updates to the spreadsheet.</a> <a data-t="610">So we need to have an automated process to know when that's changing.</a> <a data-t="614">And again, for that, we turn to AWS.</a></p>

<p><a data-t="617">So we're using Lambda to write some serverless code which will check the infested waters</a> <a data-t="625">list and see if there are any changes from the last time we looked at it.</a> <a data-t="629">And if there are no changes, that's great.</a> <a data-t="632">We don't need to worry.</a> <a data-t="633">If there are changes, though, then we need to do something, and we're going to spin up</a> <a data-t="637">a processing instance to actually run these models.</a> <a data-t="641">So we use EventBridge to schedule that rule to run every 12 hours, which is how I've ended</a> <a data-t="646">up having an email inbox which twice a day gives me a notification, has this file changed,</a> <a data-t="652">yes or no?</a> <a data-t="653">Which is great for my mental well-being, but that doesn't matter.</a></p>

<p><a data-t="714">But then how do we get it into the app?</a> <a data-t="716">That's the next important question.</a> <a data-t="718">And the way that we do that is on our EC2 instance, which is hosting the Shiny application,</a> <a data-t="725">we have a cron job which, once a day, will go to our S3 bucket, retrieve the latest data</a> <a data-t="730">that's available in there, and then restart the Shiny application.</a> <a data-t="734">And we've got this scheduled for about 5 in the morning Minnesota time, so that hopefully</a> <a data-t="739">nobody's going to be affected when the server restarts.</a> <a data-t="742">And this means that when people come to the application, all of the files that they need</a> <a data-t="747">are already available, loaded in directly from the same server that the application</a> <a data-t="752">is running on.</a> <a data-t="753">So they can get right to what they need to do.</a></p>

<h3>Performance: load balancing and tuning</h3>

<p><a data-t="758">That brings us around to kind of the last point of these challenges that we need to</a> <a data-t="764">solve, and that's the idea of the performance of the application.</a> <a data-t="768">We've already kind of spoken to some of these points as we've gone along.</a> <a data-t="772">So we've made sure that we're hosting the Shiny app in the right place.</a> <a data-t="777">We've made sure to increase the specs of our EC2 instance to make sure it has kind</a> <a data-t="784">of enough power behind it to run maybe some of the more complicated tasks that we need</a> <a data-t="788">to do in the app, things like presenting all those lakes on a leaflet map.</a> <a data-t="793">We've preprocessed our expensive model outputs so that they're all kind of just sitting there</a> <a data-t="798">ready to go whenever someone comes to the application.</a></p>

<p><a data-t="802">But the question then is, what else should we do?</a> <a data-t="807">And that kind of comes back to the way that we're running the Shiny app as well.</a> <a data-t="811">So I mentioned before that we're not using Shiny server on these instances because it</a> <a data-t="816">was kind of just a little bit too much in some cases.</a> <a data-t="820">So that means that we kind of need to handle the question of load balancing ourselves.</a> <a data-t="825">So we want to make sure that if there are multiple concurrent users for the app, that</a> <a data-t="829">they're not going to bump into each other and cause problems in terms of their individual</a> <a data-t="835">user experiences.</a> <a data-t="836">And the way that we do that, again, we're turning to AWS, and we're using the load balancer</a> <a data-t="841">along with target groups and autoscaling groups.</a></p>

<p><a data-t="845">So our load balancer is linked up to the application URL.</a> <a data-t="849">So when someone arrives intending to use the AIS Explorer, instead of just going directly</a> <a data-t="855">to the server which is running the application, they'll instead be directed to the load balancer,</a> <a data-t="860">which we have in the AWS Cloud.</a> <a data-t="863">From there, the load balancer is going to take their connection, and it's going to redirect</a> <a data-t="868">them into the target group, which has one of however many instances we have of the AIS</a> <a data-t="875">Explorer at the time.</a> <a data-t="877">And so it's going to distribute that traffic into the target group, depending on how it's</a> <a data-t="881">currently being used and how that load is currently shared across the instances in that</a> <a data-t="886">target group.</a></p>

<p><a data-t="888">In terms of how many instances we have in that target group, we use the autoscaling</a> <a data-t="893">group to kind of control that for us.</a> <a data-t="895">So we have a minimum number of instances.</a> <a data-t="898">So we always have at least one running.</a> <a data-t="900">And then we have a certain threshold, which if the traffic on that instance gets to a</a> <a data-t="904">certain point, then we'll spin up a new instance automatically so that as new people arrive,</a> <a data-t="911">they will be redirected.</a> <a data-t="913">Instead of going onto the same instance, they'll end up on one of the new ones, and it'll kind</a> <a data-t="917">of share that load out so that people aren't really all crammed into one specific server.</a> <a data-t="925">And as it kind of automatically creates these instances, as that traffic dies down, it will</a> <a data-t="931">also safely tear those down without causing any problems for people who may be using the</a> <a data-t="935">app.</a></p>

<p><a data-t="938">So besides the load balancing, there's one other thing which we want to briefly consider,</a> <a data-t="943">which is around performance tuning.</a> <a data-t="945">And we've already done some of this.</a> <a data-t="946">We want to make sure we're doing as little in the Shiny app as we can.</a> <a data-t="950">We've preprocessed our expensive operations.</a> <a data-t="952">We cache our results in the Shiny app.</a> <a data-t="954">So using functions like bind cache in the app to kind of hold on to expensive operations</a> <a data-t="959">people may have already done.</a> <a data-t="961">But the last question which comes up for me is, what if we need to run one of our expensive</a> <a data-t="966">models?</a></p>
