<h3>Why testing matters</h3>

<p><a data-t="227">So why do I think testing is useful? Firstly, for package authors to ensure that their code as it exists and as they write it is correct.</a> <a data-t="239">Secondly, for future contributions to packages that they do not break existing behavior and that they themselves are correct.</a> <a data-t="251">So this is not only for outside contributions from other authors, but also contributions from the authors themselves,</a> <a data-t="265">so from their future selves, that they don't break existing functionality when they write new functions.</a> <a data-t="274">And lastly, for users as sort of a badge of confidence in a given package and a way to show package quality.</a></p>

<p><a data-t="285">So if a package has tests and a user can be more confident that those tests are covering a large proportion of the package,</a> <a data-t="299">the user can be more confident that the package has been well tested and is less likely to contain bugs that they may run into.</a></p>

<h3>What is test coverage</h3>

<p><a data-t="311">So test coverage. What is test coverage? Test coverage is simply the proportion or lines of your code,</a> <a data-t="320">of your package code that is actually ran or tested by your test code.</a> <a data-t="326">And probably more importantly, it is also the lines of code that are not tested by your package code.</a></p>

<p><a data-t="335">So covr is a R package I started writing in December of 2014.</a> <a data-t="345">It supports local reports of your testing coverage and interactive use, as well as support for use on continuous integration servers such as Travis and also Jenkins.</a> <a data-t="360">It supports R code, including S3, S4 reference class and </a><a href="/software/r6/">R6</a><a data-t="360"> methods, which is all of the primary object oriented systems in R,</a> <a data-t="376">as well as compiled code in C, C++ and Fortran in R packages. That's included in your R package.</a> <a data-t="387">Up until now, we've been talking only about tests, but covr also supports running on example code as well as vignette code.</a> <a data-t="400">So it works not only for your actual test code, but also those other two types as well.</a></p>

<h3>How covr works for R code</h3>

<p><a data-t="410">So how this works for R code is fairly straightforward.</a> <a data-t="415">If we define a function f with two statements and then call the trace calls function on it,</a> <a data-t="426">covr transforms the body of the f function to have two counting statements.</a> <a data-t="435">And these counting statements can be used to track exactly what lines or statements have been executed by that function.</a> <a data-t="447">So basically what covr does is takes all of the functions in your package, replaces the bodies of the function by the modified function definition returned by trace calls,</a> <a data-t="463">and then runs your test code on that modified code.</a> <a data-t="469">You should notice that the counting calls do not actually modify the results of any of the code that they're tracking.</a></p>

<p><a data-t="480">So for a more complicated example, this is a function with a switch statement.</a> <a data-t="484">You can see for each condition in the switch, each one gets its own count call.</a> <a data-t="501">So we can then easily see exactly which conditions were executed by tallying up these counts.</a></p>

<h3>Compiled code coverage</h3>

<p><a data-t="514">So for compiled code, covr uses the gcov extension that is provided in the GCC and Clang compilers.</a> <a data-t="528">So to enable this, you need to compile your code without optimizations and with the coverage argument.</a> <a data-t="542">And then you run your testing code as normal.</a> <a data-t="546">So what that does is instruments the compiled code with coverage information,</a> <a data-t="552">which is output once the process finishes,</a> <a data-t="558">and then can be aggregated by the gcov binary into gcov files,</a> <a data-t="565">which are then parsed by covr and included with the results from the R code in one homogeneous report.</a></p>

<h3>Using covr with Travis CI</h3>

<p><a data-t="575">So I mentioned that covr can be used on Travis and continuous integration servers.</a> <a data-t="581">So how that works for Travis is you can add suggests covr to your package description,</a> <a data-t="589">which will automatically install covr when testing on when you're running your code on Travis.</a> <a data-t="597">And then in the after success block in your Travis </a><a href="/software/config/">config</a><a data-t="597"> file,</a> <a data-t="603">you add either the code cov or coveralls function call, depending on which of those two services you'd like to use.</a> <a data-t="611">And then generates, you can then include a badge in your readme,</a> <a data-t="617">which will make it easy for users to see exactly how much coverage your test coverage your package is using.</a></p>
