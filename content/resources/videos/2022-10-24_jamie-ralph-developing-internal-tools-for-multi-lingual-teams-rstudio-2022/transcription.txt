<h3>Building identical generic functions</h3>

<p><a data-t="158">The first idea I want to talk about is building identical generic functions.</a> <a data-t="162">But what is a generic function?</a> <a data-t="162">Let's look at this from the R perspective.</a> <a data-t="168">Generic functions are very common in the R language.</a> <a data-t="168">For example, functions like print, or summary, or tidy from the broom package.</a> <a data-t="174">The really cool thing about generics is that they behave in different ways depending on the class of an object that is passed to them.</a> <a data-t="185">Under the hood, this is achieved using a process called method dispatch.</a></p>

<p><a data-t="185">As an internal developer, you might want to build a package that uses custom internal classes, and you may want your users to be able to call methods like print or summary on them.</a> <a data-t="195">In some cases, you may even want to define your own generic functions.</a> <a data-t="200">Let's take a look at what generic functions look like in code.</a> <a data-t="204">To do this, we will create a new package which will provide one function, say hello, and this will print a nice friendly greeting to an object that we pass to it.</a></p>

<p><a data-t="217">To start, we define our generic function, say hello, and in the function body, we make a call to use method, passing it the name of our generic as a string.</a> <a data-t="222">And that's it, that's our generic.</a> <a data-t="228">But now we need a class specific implementation of this function.</a> <a data-t="233">And to do this, let's make one for a data frame.</a> <a data-t="233">And to do it, we will use dot notation, say hello dot data frame.</a> <a data-t="240">And in this new function, we print a nice friendly greeting.</a> <a data-t="245">And now if you want to catch all implementations say hello, we will use the default pseudo class, say hello dot default.</a></p>

<p><a data-t="253">Okay, now we know how our R package is going to work.</a> <a data-t="260">We need a Python version.</a> <a data-t="260">Can we write our Python version in the same way as our R version?</a> <a data-t="265">And the answer is yes.</a> <a data-t="265">To do this, we're going to import single dispatch from the functuals module.</a> <a data-t="273">And unlike R, we actually start by defining our fallback method for say hello.</a> <a data-t="278">And we decorate it with single dispatch.</a> <a data-t="278">Now, if you're not familiar with decorators, think of them as wrapper functions that change the behavior of a function.</a> <a data-t="284">And now we're going to create a data frame implementation of say hello in Python, using a data frame object from the pandas library.</a> <a data-t="289">And we do this with the register function.</a> <a data-t="294">And here I am passing that data frame to the register function.</a></p>

<p><a data-t="301">And there we have it.</a> <a data-t="301">But what have we actually achieved here?</a> <a data-t="307">Well, we have two different tools, one in R, one in Python that have the same foundation, a say hello generic with a data frame implementation.</a> <a data-t="313">So both of my packages work the same conceptually.</a> <a data-t="319">This is good for end users, they get a similar API to work with regardless of whether they're using R or Python.</a> <a data-t="326">But also my code is physically quite similar.</a> <a data-t="326">So when I go and make a change in the Python version of my code, I know where to go and make that change in the R version of my code.</a> <a data-t="338">So building identical generic functions is going to save me a lot of time and effort.</a></p>

<blockquote><a data-t="338">So building identical generic functions is going to save me a lot of time and effort.</a></blockquote>

<h3>Identical error handling with classes</h3>

<p><a data-t="344">The next idea I will talk about is identical error handling with classes.</a> <a data-t="350">Let's look at this from the Python perspective.</a> <a data-t="350">Now, if you're a Python user, you're probably familiar with these kinds of exceptions, far not found error, zero division error, and key error.</a> <a data-t="360">Exceptions in Python are themselves classes that inherit from other exception classes.</a> <a data-t="364">And you can probably already tell that the names of these exceptions are quite descriptive, they form part of the error message.</a></p>

<p><a data-t="370">Now as an internal developer, you can create your own custom internal errors, you can give them really descriptive names.</a> <a data-t="375">And the error messages that they give can be as well specific as you want, even telling the user where to go for help if that error has been thrown.</a> <a data-t="388">The powerful thing about exceptions in Python is that they can be handled explicitly.</a> <a data-t="392">For example, with a try except statement, I asked Python to try running some code.</a> <a data-t="396">And in the event of a specific error, in this case, a far not found error, I can run an alternative block of code.</a> <a data-t="402">This gives me maximum control over how my code handles failure.</a></p>

<p><a data-t="409">Okay, so we're building a Python package, and we are handling errors in this way.</a> <a data-t="416">But can we handle errors in the same way in our R packages?</a> <a data-t="422">The answer is yes.</a> <a data-t="422">But let's look at some basic error handling in R first.</a></p>

<p><a data-t="428">If you're an R user, you may have seen one of these two functions stop from base R and abort from the R line package.</a> <a data-t="439">Both functions signal an error.</a> <a data-t="439">And in R, if we want to handle errors in our code, we can use something like a try catch function.</a> <a data-t="444">But notice here on the second line, the error handler is a function that will execute in response to any error.</a> <a data-t="451">It's not designed to handle a specific error.</a> <a data-t="451">And this is a problem.</a> <a data-t="455">I now have a lot less control over my code.</a> <a data-t="455">And more importantly, my Python and R packages handle failure in very different ways.</a></p>

<p><a data-t="469">How can we overcome this?</a> <a data-t="469">Well, the answer to this is within the abort function from </a><a href="/software/rlang/">Rlang</a><a data-t="469">.</a> <a data-t="476">Let me illustrate this with an example.</a> <a data-t="476">I'm going to create a new function abort credentials missing.</a> <a data-t="482">And within called Rlang abort, I'm going to give a specific class name.</a> <a data-t="489">Here it's error credentials missing.</a> <a data-t="498">Now the really great thing about error functions in R is that you can create an equivalent Python exception.</a></p>

<p><a data-t="504">So as an internal developer, you have your Python exceptions and you have your R error functions.</a> <a data-t="508">They're one and the same.</a> <a data-t="508">So when you make a change in one language, you know where to make that change in the other language.</a> <a data-t="514">But what have we gained by giving our error a class name?</a> <a data-t="519">Well, let's go back to try catch.</a> <a data-t="519">You can see here on the second line, I can now write a specific error handler for my new error credentials missing.</a> <a data-t="525">And now my Python package and my R package handle failure in the same way.</a></p>

<p><a data-t="532">I want to round off this section on errors by talking about error chaining.</a> <a data-t="538">In essence, error chaining is throwing an error in response to another error.</a> <a data-t="544">One use for this is to give a high level context to a lower level error.</a> <a data-t="551">Doing this in Python is very easy.</a> <a data-t="551">For example, you can give your original error an alias.</a> <a data-t="555">Here I'm giving it the alias E.</a> <a data-t="555">And then I'm raising another exception from that original exception.</a> <a data-t="562">And this gives me a kind of double error message.</a> <a data-t="562">I get my original error and then Python tells me that this error caused another error.</a></p>

<p><a data-t="567">Can we emulate this in R?</a> <a data-t="573">The answer is yes.</a> <a data-t="573">Now the way that we're going to do this is we're going to use try fetch from the Rlang package instead of a try catch.</a> <a data-t="580">And in my error handler, I'm going to pass my condition object to the abort function, specifically the parent argument.</a> <a data-t="589">And here I get a kind of double error message.</a> <a data-t="595">And there we have it.</a> <a data-t="595">Our Python and R packages now conduct error chaining in the same way.</a></p>

<h3>Creating internal wrappers</h3>

<p><a data-t="601">The third idea I want to talk about is creating your own internal wrappers.</a> <a data-t="607">But what is an internal wrapper?</a> <a data-t="607">Well, it's a package written in one language that wraps functionality from another language.</a> <a data-t="612">But why would you want to create your own internal wrapper?</a> <a data-t="618">Well, imagine a scenario where a piece of functionality is available in one language but not the other.</a> <a data-t="623">A well-known example is a statistical model that's released as an R package but not a Python package.</a> <a data-t="628">But you want all of your analysts to be able to access that functionality.</a> <a data-t="635">Another scenario you might encounter is that both languages have their own implementation of a piece of functionality, like a statistical model, but maybe the underlying computations aren't quite the same.</a> <a data-t="647">And therefore, your analysts get different results.</a></p>

<p><a data-t="652">Okay, let's kick this section off by looking at how we call Python from R.</a> <a data-t="652">There are many ways to do this.</a> <a data-t="658">My method of choice is the </a><a href="/software/reticulate/">reticulate</a><a data-t="658"> package.</a> <a data-t="658">Reticulate gives you a number of different ways to call Python code in your R package.</a> <a data-t="665">Let's look at a couple of examples.</a> <a data-t="665">Now, the first example is importing Python modules into R.</a> <a data-t="670">This works really well if the Python code you're trying to access is contained within a Python package.</a> <a data-t="677">And to do this, I use the import function from reticulate.</a> <a data-t="677">I give it the name of my module.</a> <a data-t="681">Here, I'll call it pymodule, and I assign it to a variable.</a> <a data-t="682">And now, I can access functions from that module using this dollar sign notation.</a></p>

<p><a data-t="689">Now, a really good tip for when you're doing this in an R package is to create a global reference to your Python package.</a> <a data-t="695">And then you can use it throughout your code.</a> <a data-t="701">Notice here, I'm not actually importing anything yet.</a> <a data-t="701">I'm going to specify how my Python module is imported in the onload function.</a> <a data-t="715">This will execute when my R library is loaded.</a> <a data-t="715">And here, I'm specifying delay load to true.</a> <a data-t="720">And what this does is it gives my end user the ability to specify their own custom Python virtual environment to use with my package.</a></p>

<p><a data-t="727">Another way that you can use Python code in your R package is to source Python files.</a> <a data-t="733">This might work well if the code you're trying to access is contained within a Python script.</a> <a data-t="740">To do this, I'm going to use a Python file.</a> <a data-t="740">Here, it's called add.py, which is contained in my R package.</a> <a data-t="745">And inside that file, I define a new function, add to.</a> <a data-t="752">And now, I will use source Python from the reticulate package, and then I can access the add to function.</a></p>

<p><a data-t="762">Okay. We've called Python from R.</a> <a data-t="762">Now, let's call R from Python.</a> <a data-t="769">Again, there are many ways to do this.</a> <a data-t="769">My method of choice is the rpy2 package.</a> <a data-t="769">rpy2 gives you a number of different ways to call R code from your Python package.</a> <a data-t="774">Let's look at some examples.</a></p>

<p><a data-t="776">The first method is to import R libraries into Python.</a> <a data-t="782">To do this, we're going to import the import R function from rpy2.</a> <a data-t="789">And then, in this example, we're actually going to import base R itself, which we can import like any other library.</a> <a data-t="794">To access a function from base R, we use this dot notation.</a> <a data-t="794">Here, I'm accessing the sum function.</a></p>

<p><a data-t="801">Now, to use this function, I'm going to have to do a little bit of work.</a> <a data-t="804">Sometimes, when you're using rpy2, you need to convert objects between R and Python.</a> <a data-t="811">In this example, I need to convert my Python list into an R vector.</a> <a data-t="817">I can do that using the int vector function from rpy2.</a> <a data-t="817">You can check out the rpy2 documentation for more tips on object conversion.</a></p>

<p><a data-t="826">Another way to access R code from Python using rpy2 is to create new functions and access them from the R environment.</a> <a data-t="833">This approach works really well when you have multiple computation steps that need to happen in R.</a> <a data-t="839">Now, to do this, we're going to first import the R objects module from rpy2.</a> <a data-t="848">We're then going to define our R code as a string.</a> <a data-t="853">Here, I'm just creating a new function that returns one.</a> <a data-t="862">I then execute that R code using the R function from R objects.</a> <a data-t="862">And then, I can access that new function using globalenv and assign it to a variable.</a> <a data-t="869">And now, pyfunc is actually a function that I can execute.</a></p>

<p><a data-t="874">Now, a really great utility that rpy2 gives you is the ability to define a temporary package structure.</a> <a data-t="882">Now, we can do this by importing the stap function from rpy2.</a> <a data-t="892">And just like before, we store our R code as a string.</a> <a data-t="892">And then, we use stap to execute that code.</a> <a data-t="893">And then, we can access functions from that code using this dot notation.</a> <a data-t="899">So, this approach works really well when you are concerned about issues like namespace conflicts.</a></p>

<p><a data-t="908">Okay. Now, for the bad news.</a> <a data-t="908">Cross-language wrappers have their own technical quirks and headaches.</a> <a data-t="914">For example, you have twice the number of environments to think about.</a> <a data-t="918">Your users will need to have access to both Python and R.</a> <a data-t="924">And your packages will need to deal with the relevant dependencies.</a> <a data-t="928">From personal experience, Reticulate is a lot more explicit about how it handles Python environments.</a> <a data-t="934">Whereas, with rpy2, you need to do a little bit more of the work yourself.</a></p>

<p><a data-t="940">Now, another issue is object conversion.</a> <a data-t="945">Sometimes, to work with cross-language wrappers, you need to handle the conversion of an object from one language to another, like you saw with rpy2.</a> <a data-t="951">And a third issue are error messages.</a> <a data-t="957">Sometimes, when an error occurs in the language that is being wrapped, the error message doesn't come through in its entirety, which can frustrate your efforts to create nice, helpful error messages.</a> <a data-t="969">But with all this being said, cross-language wrappers are a really useful tool in your internal toolkit.</a> <a data-t="975">They can deal with issues of missing functionality or inconsistent outputs between languages.</a></p>
