<h3>Q&A: alternative approaches for user-specific data</h3>

<p><a data-t="1152">What are some alternative approaches that you have seen customers use when retrieving user-specific data from an external data catalog?</a></p>

<p><a data-t="1179">I think the most common one we've seen, like specifically within the context of Connect, is deploying those multiple instances of the same app</a> <a data-t="1195">and setting static service account-esque credentials on that app, where the service account has the same permissions as the user that is interacting with that duplicated instance of the app that's published,</a> <a data-t="1211">which obviously is very clunky, and that's why we wanted to build this feature.</a></p>

<p><a data-t="1223">There are two others that come to mind.</a> <a data-t="1229">There's the possibility of actually implementing OAuth inside of your content.</a> <a data-t="1229">It is technically possible to handle the OAuth flow from your content, but it would mean you would have to implement it for every piece of content, and that's a lot of work.</a> <a data-t="1240">It's also really difficult to maintain, and it gets really complicated, particularly with high-availability Connect.</a> <a data-t="1247">You never know which Python or R process you're going to connect to, and so that would be weird.</a></p>

<p><a data-t="1254">The other option is using some of the metadata provided by Connect when the content is running.</a> <a data-t="1264">Connect does expose information about the connecting viewer to the content itself.</a> <a data-t="1270">You can get that information out of one of Connect's headers.</a> <a data-t="1273">You could write application-specific logic for every piece of content that says,</a> <a data-t="1278">if this viewer then access the external resource with this other set of credentials, but that doesn't scale very well either.</a></p>

<p><a data-t="1287">So we think that what we've got here is a much better approach for managing that because it's going to scale a little bit better.</a> <a data-t="1296">Rather than maintaining that access control logic inside of your content,</a> <a data-t="1303">you can maintain the access control logic at the data source where they have really robust controls for writing those access control rules.</a></p>

<blockquote><a data-t="1296">Rather than maintaining that access control logic inside of your content,</a> <a data-t="1303">you can maintain the access control logic at the data source where they have really robust controls for writing those access control rules.</a></blockquote>
