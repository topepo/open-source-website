<h3>The <a href="/software/miniui/">miniUI</a> package</h3>

<p><a data-t="854">Now let's look a little at the user experience for this gadget.</a> <a data-t="857">You could see that I'm loading the package MiniUI here for this gadget.</a> <a data-t="863">The UI functions in the gadget come from MiniUI.</a> <a data-t="867">If we look at the MiniUI package,</a> <a data-t="873">you can see it's actually a fairly small package.</a> <a data-t="876">We developed this to work alongside Shiny for making Shiny gadgets,</a> <a data-t="880">and it has a few different pieces of Shiny layout that you can use.</a></p>

<p><a data-t="886">The reason we thought it would be important to develop a new package for gadget layout versus a Shiny app layout</a> <a data-t="892">is that most gadgets tend to be very small.</a> <a data-t="895">The space that you use on the screen when you make your gadget</a> <a data-t="900">is normally limited to either your RStudio viewer pane,</a> <a data-t="903">or you might have a pop-up window or something.</a></p>

<h3>Stopping the app and returning values</h3>

<p><a data-t="1301">All of that's actually probably pretty familiar to you, especially if you're a shiny developer.</a> <a data-t="1305">I'm going to take it for granted that you've been writing our functions and that you've been creating UI and server,</a> <a data-t="1310">because that's what you do as a shiny developer.</a> <a data-t="1312">But this step might be new to everybody involved.</a> <a data-t="1316">That's stopping an app and returning values back out of the app.</a> <a data-t="1320">You can do this with a regular shiny app, but it really becomes essential when you use a shiny gadget.</a> <a data-t="1325">It revolves around a function called stopApp.</a></p>

<p><a data-t="1329">To take advantage of stopApp, you do need to know how to use observeEvent,</a> <a data-t="1333">which is a shiny function for handling events.</a> <a data-t="1337">The way observeEvent works is you call this in the server logic of your function.</a> <a data-t="1341">Just like you'd make a reactive expression or set up an observer or a render statement,</a> <a data-t="1346">you would set up an observeEvent block.</a> <a data-t="1348">There's two observeEvent blocks, one for each of the buttons here.</a> <a data-t="1352">What observeEvent does is it sets up an observer that watches a reactive input.</a></p>

<p><a data-t="1359">Every shiny gadget is going to save input$cancel and input$done as inputs for the two buttons in the title bar,</a> <a data-t="1366">unless you intervene in naming something different.</a> <a data-t="1369">These two observeEvents are going to watch those buttons and wait to see when their values change.</a> <a data-t="1374">Action buttons have an arbitrary value that increments by one every time someone clicks the button.</a> <a data-t="1381">When someone clicks that button, that value will change.</a> <a data-t="1383">These observeEvent statements will notice that that has changed.</a> <a data-t="1387">The next part of the observeEvent call is a code recipe that observeEvent should run whenever the first value changes.</a> <a data-t="1395">Here, when input$done changes, observeEvent is going to run this code, stopApp.</a></p>

<p><a data-t="1403">What stopApp does is, first, it exits the app. It closes the app.</a> <a data-t="1408">Then, second, it passes to the environment that called the app anything that you put as arguments for stopApp.</a> <a data-t="1415">If you put multiple things here, you could create a list for stopApp to return.</a> <a data-t="1418">Here, let's assume that this app created vowels$keep.</a> <a data-t="1424">Perhaps that's a reactive expression.
