<p><a data-t="101">For accessibility, in the UK, all public sector services must follow</a> <a data-t="106">the Web Content Accessibility Guidelines, or WCAG for short.</a> <a data-t="110">As our developers, we've had to show that we understand</a> <a data-t="112">how to develop accessible web applications.</a> <a data-t="115">Otherwise, we're simply not allowed to publish.</a></p>

<h3>Trustworthiness</h3>

<p><a data-t="695">In an age where most people have access to the internet</a> <a data-t="698">and there are swathes of information out there,</a> <a data-t="700">how do they know what to trust?</a> <a data-t="702">And similarly, at a time where the information we release</a> <a data-t="704">is heavily scrutinized</a> <a data-t="706">and new cuts of data are requested frequently,</a> <a data-t="709">how can we convince our digital and IT colleagues</a> <a data-t="711">to let us publish shiny apps that statisticians have created</a> <a data-t="714">when a great deal of risk is involved?</a></p>

<p><a data-t="717">To illustrate this issue,</a> <a data-t="719">which of these would you trust?</a> <a data-t="721">On the left hand here, we've got this cat facts website</a> <a data-t="724">which may or may not send you some spam text messages,</a> <a data-t="728">and on the right hand side,</a> <a data-t="729">we've got the National Geographic page</a> <a data-t="731">which clearly lists out some facts about cats.</a> <a data-t="734">The stakes here are clearly lower</a> <a data-t="735">when we're trying to find out cat facts,</a> <a data-t="737">but for the general public,</a> <a data-t="738">how do they know what they can trust</a> <a data-t="740">for schools, the economy or the environment?</a></p>

<p><a data-t="743">People are aware of misinformation on the web</a> <a data-t="745">but we need to make it clear to our users</a> <a data-t="747">that our statistics are trustworthy</a> <a data-t="749">and can be relied upon.</a> <a data-t="751">We also need to closely work with security and IT experts</a> <a data-t="754">in our departments to clear any external outputs,</a> <a data-t="757">but how can we get their buy-in</a> <a data-t="758">if our pages look more like catfacts.org</a> <a data-t="760">and less like National Geographic?</a></p>

<p><a data-t="862">And there's a really clear link to accessibility here.</a> <a data-t="865">So being able to navigate pages with a similar layout,</a> <a data-t="868">it's really important for those</a> <a data-t="869">who can't visually see a screen.</a> <a data-t="871">Like we saw in Cam's demo earlier,</a> <a data-t="873">it does take a while to orient yourself</a> <a data-t="875">in a new and unfamiliar page layout.</a> <a data-t="877">So using the gov.uk layout benefits both users</a> <a data-t="880">with accessibility needs and those without.</a></p>

<h3>Usefulness</h3>

<p><a data-t="911">The last barrier that we faced</a> <a data-t="912">was something that might be familiar</a> <a data-t="913">to all package developers,</a> <a data-t="915">and that is usefulness.</a> <a data-t="917">So how do we keep the package useful to our users</a> <a data-t="920">without making unnecessary work for ourselves?</a> <a data-t="923">Our package is developed and maintained</a> <a data-t="925">by a team of volunteers</a> <a data-t="926">across lots of different departments</a> <a data-t="928">in UK sort of service.</a> <a data-t="930">So utilizing their resource effectively</a> <a data-t="932">is really important.</a> <a data-t="933">Our GitHub issue log helps us keep on top</a> <a data-t="935">of what users are asking for,</a> <a data-t="937">what bugs have been raised,</a> <a data-t="938">and which volunteer is working on it.</a></p>

<p><a data-t="941">One example of a useful feature</a> <a data-t="943">that we've recently added to the package</a> <a data-t="945">is styled reactable tables</a> <a data-t="947">to something that a user had requested.</a> <a data-t="950">So out of the box,</a> <a data-t="951">these aren't always suitable for GitHub.UK dashboards.</a> <a data-t="954">You might be able to see</a> <a data-t="955">there's a very small sorting arrow</a> <a data-t="957">that appears at the top,</a> <a data-t="959">but these are really hard to make out</a> <a data-t="961">with visual impairments.</a> <a data-t="963">We decided it was worth our time</a> <a data-t="964">to add new CSS for reactables</a> <a data-t="966">to the standard Gov.UK design</a> <a data-t="968">as it is a fairly niche use</a> <a data-t="970">for analysts building R Shiny dashboards</a> <a data-t="972">and unlikely to have any replication across government.</a></p>

<p><a data-t="985">The result of this is a Gov-styled reactable table.</a> <a data-t="989">So we took a lot of elements</a> <a data-t="990">from the current static table in the top left,</a> <a data-t="993">which was designed on the Gov.UK design system.</a> <a data-t="996">So we don't want to diverge too far</a> <a data-t="998">from this user research design,</a> <a data-t="1000">and we want to keep that trust with our users.</a> <a data-t="1003">So this simply adds in things like accessible sorting</a> <a data-t="1006">with the blue bars you can see</a> <a data-t="1008">at the top and the bottom of the column headers</a> <a data-t="1010">and accessible hover overs with that yellow highlight</a> <a data-t="1014">so that it's really clear what a user is focusing on.</a></p>

<p><a data-t="1024">An example of an issue</a> <a data-t="1026">that we decided not to fix is dark mode.</a> <a data-t="1029">So this is a much wider government issue,</a> <a data-t="1032">and it's been raised with the design team for many years.</a> <a data-t="1035">They've been looking at tackling it in the long run,</a> <a data-t="1037">and we don't just want to spin up a version of it</a> <a data-t="1039">that analysts will come to use,</a> <a data-t="1040">which we then need to change to fit</a> <a data-t="1042">with wider government branding later on.</a> <a data-t="1045">So building 50 different versions of dark mode</a> <a data-t="1047">can make it really hard to keep that trust with users</a> <a data-t="1049">that we mentioned earlier across all services.</a></p>

<p><a data-t="1053">Dark mode's been an ongoing conversation since 2018.</a> <a data-t="1056">You can see when this person raised it.</a> <a data-t="1059">And in GovDesign, there are a lot of changes involved</a> <a data-t="1062">that are much bigger than our little package</a> <a data-t="1064">with a fairly small use case for analytical dashboards.</a> <a data-t="1068">This thread shows a lot of people going off</a> <a data-t="1070">and building their own versions,</a> <a data-t="1071">each different from each other,</a> <a data-t="1073">and we don't want to be adding to this list of things</a> <a data-t="1075">to consolidate once a fix is finally in.</a></p>

<p><a data-t="1078">So there's a real challenge here</a> <a data-t="1079">in deciding where best to spend our time</a> <a data-t="1081">as package developers to add the most value.</a> <a data-t="1084">And the solution's been working really closely</a> <a data-t="1086">with both of our users and the wider government</a> <a data-t="1088">to understand how the package is being used</a> <a data-t="1091">and when wider changes might affect it.</a></p>
