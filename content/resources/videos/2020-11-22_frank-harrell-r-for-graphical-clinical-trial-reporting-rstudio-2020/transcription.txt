<h3>Philosophy and structure of clinical trial reports</h3>

<p><a data-t="203">So let's get to clinical trial reports.</a> <a data-t="208">So the applications I'm talking about are where you're doing, say, a randomized clinical</a> <a data-t="213">trial to study a treatment versus some control, or you might be doing a drug study, something</a> <a data-t="219">in the pharmaceutical industry.</a> <a data-t="221">And there's two types of clinical trial reports.</a> <a data-t="223">There's sort of final clinical trial reports and there's interim reports as often used</a> <a data-t="228">in monitoring the conduct of a clinical trial that you might give to a data monitoring committee,</a> <a data-t="234">also called a data and safety monitoring board.</a></p>

<p><a data-t="239">And so the task that I started with was developing high-level abstractions for the job of creating</a> <a data-t="247">reports.</a> <a data-t="248">Most statisticians, when they create a big report, they sort of start over.</a> <a data-t="252">They may have some code they used from previous trials, but they do a lot of new coding and</a> <a data-t="257">it's quite repetitive.</a> <a data-t="259">So if you can make high-level abstractions, I think you can foster better statistical</a> <a data-t="264">analysis and better reporting practices and make everything reproducible and really minimize</a> <a data-t="271">programming.</a></p>

<p><a data-t="273">And the authors of reports about human subjects' research get really tired of reading through</a> <a data-t="281">tables and they usually fall asleep, they don't pay attention to things, and tables</a> <a data-t="286">are not that good for showing patterns anyway.</a> <a data-t="289">And so we really want to get rid of that.</a></p>

<p><a data-t="292">Now there's many standard components to clinical trial reports.</a> <a data-t="296">These are some of them, a summary of how you were able to accrue subjects into the study,</a> <a data-t="303">how did the patient flow occur and what sort of exclusions did you find between the accrual</a> <a data-t="309">or the screening of subjects before you randomized them to experimental treatment, describing</a> <a data-t="316">baseline variables, longitudinal analysis, adverse event analysis, lab safety parameters,</a> <a data-t="325">event timing and incidence, and sequential monitoring of event probabilities.</a></p>

<p><a data-t="331">So the philosophy is that tables really do not lead to pattern recognition and graphics</a> <a data-t="337">are better.</a> <a data-t="339">If you have more than two numbers, I think a graph is better than a table.</a> <a data-t="343">In other words, graph is almost always better.</a> <a data-t="345">And graphs should use features that humans are really good at perceiving, such as position</a> <a data-t="349">along a common scale.</a> <a data-t="352">And then we need side posts on the graphics.</a> <a data-t="354">So when I'm reading a long report, I lose track of where I am in terms of, say, which</a> <a data-t="358">subjects are currently being displayed or analyzed.</a> <a data-t="361">And I'll show you what that means.</a> <a data-t="363">Tables are secondary.</a> <a data-t="364">They can be in an appendix or hyperlinked.</a> <a data-t="367">Or what's better, I think, is to have pieces of tables as hover text from a plot.</a></p>

<p><a data-t="374">And then this was probably a little more controversial.</a> <a data-t="378">Most studies where you're doing an A-B comparison, the way an A-B comparison is really created</a> <a data-t="385">is that confidence intervals for A are not relevant to the design and confidence intervals</a> <a data-t="390">for B are not relevant to the design.</a> <a data-t="392">They're actually inconsistent with a parallel group design.</a> <a data-t="396">But the design is made to give confidence intervals for differences between A and B.</a> <a data-t="400">And that's what we want to emphasize.</a> <a data-t="402">We want to show the entire distribution when possible, favor quantiles over means and standard</a> <a data-t="407">deviations, and then I personally don't like percentages and I do everything I can to get</a> <a data-t="412">rid of percentages.</a> <a data-t="415">That's a personal opinion.</a></p>

<p><a data-t="417">So I have an older package that used the LaTeX and PDF model, and this is a very, very stable</a> <a data-t="422">package.</a> <a data-t="423">It's been around for many years, used in production in some really huge multinational clinical</a> <a data-t="429">trials.</a> <a data-t="430">And then the newer package, which is not on CRAN yet, which is hReport for HTML report.</a> <a data-t="435">So in these packages, there's lots of utility functions, high-level report components, unified</a> <a data-t="441">handling of figure generation and caption.</a> <a data-t="443">So when you write a long report, you spend a lot of time writing figure captions.</a> <a data-t="448">So the figure captions are all automatically generated by these packages.</a> <a data-t="452">And then there's some new graphical elements.</a></p>

<p><a data-t="455">So the high-level functions are accrual report, exclusion report, descriptive statistics,</a> <a data-t="463">event report, and serve report for time to event analysis.</a> <a data-t="469">Now when you're writing a package and you want to implement things at a high level,</a> <a data-t="473">you really have to survey best practices, and it's your opportunity to implement best</a> <a data-t="477">practices and also to create best practices if people like the package.</a> <a data-t="483">And so instead of doing one-off reports, if you start saying, what should be in a patient</a> <a data-t="488">screening report or accrual report, and really nailing that, and then making it so that people</a> <a data-t="494">can join the project on GitHub and they can extend the report with new options or change</a> <a data-t="499">the format, it's much better than just coding from scratch for each report.</a></p>

<p><a data-t="506">Number at risk report, and in the future, there'll be a sequential monitoring boundary</a> <a data-t="511">sort of report.</a> <a data-t="512">And the format of all these functions, the way you call them, is using the formula language.</a> <a data-t="517">You have your analysis variables on the left-hand side of the model, separated by plus, and</a> <a data-t="522">then you have your stratification variables and optional ID variable if you're analyzing</a> <a data-t="527">something where there's multiple records per subject.</a></p>

<p><a data-t="531">So </a><a href="/software/rmarkdown/">rmarkdown</a><a data-t="531"> creates HTML documents, and these allow semi-interactive graphics.</a> <a data-t="536">So if you're using Plotly, this uses HTML widgets, and those are included in your HTML</a> <a data-t="544">without you doing anything special whatsoever, and they'll display beautifully on virtually</a> <a data-t="549">any browser, and they'll automatically resize for viewing on a tablet, and it's very effective</a> <a data-t="554">to view these reports on a tablet.</a> <a data-t="557">And then our functions write HTML.</a> <a data-t="559">They write regular tabular output, hyperlinks, navigation bars, and almost no tables.</a> <a data-t="566">And then hovering over a part of a graphic will display the relevant portion of the table.</a> <a data-t="571">So I don't display whole tables anymore, but just portions of tables on demand.</a></p>
