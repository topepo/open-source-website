<h3>History of async in Shiny</h3>

<p><a data-t="730">Back in 2017 to 2018, we, the Shiny team, introduced async programming to R and then to Shiny.</a> <a data-t="739">And this was an incredibly technically challenging feature to build and resulted in something that I think was pretty conceptually elegant and very satisfying that we were able to bring these two pretty complicated concepts of async programming and reactive programming and to merge them in a way that felt consistent and logical and held together pretty well.</a> <a data-t="779">That approach was to use the future package by Henrik Bengtsson to perform long-running operations in background R processes, not in the current R process.</a> <a data-t="793">Future is a package that existed before async programming existed in Shiny.</a> <a data-t="799">We didn't write it.</a> <a data-t="801">We just took advantage of the fact that Henrik had already created this great package.</a> <a data-t="808">But what we did need to create was a new package called promises to handle the results of these long-running operations back in the original R process in a way that wouldn't block that host R process.</a></p>

<p><a data-t="829">This wasn't something that was really possible to do back in those days.</a> <a data-t="835">And after creating that promises package and integrating it with future, then we had to rewrite big chunks of the Shiny internals to support async programming.</a> <a data-t="845">And this was a process that took months and involved me coding at the very edges of my ability to get this working.</a> <a data-t="855">And at the end of it, we ended up with a feature that worked, but there was always something a little bit unsatisfying about it.</a></p>

<p><a data-t="877">Maybe more than a little bit.</a> <a data-t="880">Number one, and most importantly, it had this incredibly steep learning curve.</a> <a data-t="890">Learning to use Shiny async circa 2018, it was just a really difficult learning journey you'd have to go on.</a> <a data-t="899">First of all, you'd have to be a pretty expert Shiny user to make sense of it.</a> <a data-t="905">Number two, you'd have to get to know future pretty well.</a> <a data-t="909">And this idea of executing code that was going to happen in a background process.</a> <a data-t="913">And then number three, by far the worst, is promises was just really complicated to work with.</a> <a data-t="922">Promises in general are a complicated concept.</a> <a data-t="926">And one of the most difficult things about promises is that they're infectious.</a></p>

<blockquote><a data-t="926">And one of the most difficult things about promises is that they're infectious.</a></blockquote>

<p><a data-t="934">That once you have a function that uses promises, well that function now has to return a promise in order to really work correctly.</a> <a data-t="945">And that means that any function that calls a function that returns a promise, those functions also need to return a promise.</a> <a data-t="952">And you end up in this world where you want to introduce this sort of asynchronicity at this particular point in your code where something slow is happening.</a> <a data-t="960">And then these promises sort of ripple through the whole rest of your code anywhere that directly or indirectly relies on that long-running operations result.</a> <a data-t="972">They all become this promise-oriented syntax that frankly is a pretty weird syntax to begin with.</a> <a data-t="981">So that was the biggest problem, I think, is that it was hard to use because of this sort of infectious property.</a></p>

<p><a data-t="993">But also, even if you went through all of that, if you learned how to use this syntax and learned how to deal with this infectiousness that happens, all of that still didn't solve the problem of intercession concurrency.</a> <a data-t="1015">The way Shiny async was designed, it really only helped you with intercession concurrency.</a> <a data-t="1022">So your app, your experience using an app, was going to still be sort of slow and blocked.</a> <a data-t="1030">But other people could connect and have their own experience that was not going to be blocked by you.</a> <a data-t="1036">That was the best that Shiny async could do.</a></p>

<h3>How extended task changes the reactive graph</h3>

<p><a data-t="1042">And to explain a little further how that worked and how Shiny extended tasks solved this problem, I'll need to use some visuals.</a> <a data-t="1052">And for that, I'll use some diagrams from the amazing book Mastering Shiny by </a><a href="/people/hadley-wickham/">Hadley Wickham</a><a data-t="1052">, which is the best source for understanding reactive programming in Shiny that we have today.</a> <a data-t="1069">Mastering Shiny is only for R.</a> <a data-t="1072">We don't have a Python version right now.</a> <a data-t="1076">But hopefully you still get the idea from reading the chapter on reactive tracking.</a></p>

<p><a data-t="1083">So this is a diagram that we use to illustrate how reactive programming works in Shiny.</a> <a data-t="1091">On the left, these shapes here are reactive values or inputs.</a> <a data-t="1097">They are pointed on the right to show that they have a value that can be read by someone coming in from the right.</a> <a data-t="1104">On the left, we have the opposite.</a> <a data-t="1107">So these are outputs or reactive observers or reactive effects if you're in Python.</a> <a data-t="1114">So these are either outputs or code that is going to execute just for side effects.</a> <a data-t="1120">And in the middle, we have what are called either reactive expressions or reactive calc, depending on whether you're using R or Python.</a> <a data-t="1130">And these are things that can both read values.</a> <a data-t="1134">They can read reactive inputs or reactive values, or they can read other reactive expressions, and they can be read.</a> <a data-t="1141">So they have this sort of shape on both sides.</a> <a data-t="1145">They can both read and be read.</a></p>

<p><a data-t="1149">So this is the view that Shiny has of your app when a user first connects.</a> <a data-t="1155">It knows that there are these reactive expressions or reactive calcs, and it knows that there are these outputs, and it even knows that there are these reactive values or inputs, but it doesn't know how they are related.</a> <a data-t="1167">And what happens is Shiny will look for the first output or observer or effect that it can find, and it just starts executing it.</a> <a data-t="1178">This is denoted in this diagram by turning orange.</a> <a data-t="1182">So something orange is executing.</a> <a data-t="1184">So this thing starts executing, and pretty soon it makes a call to this reactive expression here.</a> <a data-t="1191">That has not executed yet, so it needs to execute.</a> <a data-t="1194">So it turns orange, and that is going to read from this reactive input.</a> <a data-t="1200">It's also going to read from this reactive expression, which reads this input, and pretty soon it's done executing, and it turns green.</a> <a data-t="1208">Everything to the left here has turned green, and next this output is done executing, so it turns green.</a> <a data-t="1215">Repeat that for the next output and the next output, and pretty soon everything is green.</a> <a data-t="1221">Everything has finished executing, and this is what we call being at equilibrium.</a> <a data-t="1228">So Shiny is done doing all the reactive things it knows to do right now, and is just waiting for you to do something at this point.</a> <a data-t="1239">And everything from the first diagram to this one is called a single tick, a reactive tick.</a> <a data-t="1247">Tick meaning like a tick of the clock.</a></p>

<p><a data-t="1254">And that's because somewhere in the depths of Shiny there's some source code that looks like this.</a> <a data-t="1260">There's a loop, an endless loop, that's sitting there waiting for input to appear from the user, taking that input and recomputing anything that's reactive that needs to be recomputed, and then taking any changed outputs that result from that and sending it to the browser.</a> <a data-t="1283">So all of that is called a reactive tick.</a> <a data-t="1287">One trip through this while loop is called a reactive tick.</a> <a data-t="1293">And notice that only at the beginning of this tick do we check for input changes, and only at the end do we send outputs to the client.</a> <a data-t="1302">So in between we cannot respond to the client or the browser in any way.</a></p>

<p><a data-t="1309">That means that our long-running tasks, like in this case I've drawn one of the reactive expressions really large to indicate like it's going to take a really long time, that code executing and taking a super long time is going to block this part of the reactive tick.</a> <a data-t="1330">It's going to block the recomputation, meaning we can't get to the part of the reactive tick where we wait for more input changes or we send outputs.</a> <a data-t="1340">We are stuck in the center of this reactive tick.</a> <a data-t="1347">So in order to fix this problem we need this while loop to be able to keep turning over and over and over without getting hung up in this recompute all affected things step, and yet still have the operation occur, still be able to use the results reactively.</a> <a data-t="1383">So how do we separate the task from the tick?</a></p>

<p><a data-t="1389">So ShinyAsync solves this by running multiple graphs concurrently.</a> <a data-t="1395">So they each have their own while loop going essentially.</a> <a data-t="1400">But you can't run multiple tasks within a single graph because the shape of the graph is unchanged.</a> <a data-t="1409">Like without changing the shape of this graph we will never escape this inability to do intra-session concurrency.</a> <a data-t="1421">So extended task changes the shape of this graph and it does it by taking this long running operation and actually splitting it into two parts.</a> <a data-t="1432">There's the part on the left which is an observer that is going to launch the operation and a totally separate but related piece that is going to hold the result value that can be read.</a> <a data-t="1448">And I've put a little emoji here to represent a background R process that's going to actually do the work for us.</a></p>

<p><a data-t="1458">So when the Shiny app starts the background process is sleeping.</a> <a data-t="1462">It hasn't been told to do any work yet and none of these relationships again have been established.</a> <a data-t="1470">So remember what Shiny does is when it first loads up and it sees all these observers and effects and expressions and things, it picks the first output it finds and starts executing it.</a> <a data-t="1493">And in this case it's going to kick off the extended task and the emoji changes to the sweaty guy because he is now working and super tired.</a> <a data-t="1501">He's working very hard.</a> <a data-t="1502">While that background R process is executing, the rest of the reactive graph is able to proceed as normal and we quickly get to this equilibrium.</a> <a data-t="1517">Everything is done executing as far as the reactive graph is concerned.</a> <a data-t="1522">Although this background task continues to execute in a separate R process.</a> <a data-t="1528">So this is great.</a> <a data-t="1530">This means we've completed the tick.</a> <a data-t="1533">Any outputs that are ready can be sent to the browser and we can start waiting for the next input from the user.</a></p>

<p><a data-t="1541">So let's say the user touches some unrelated slider or input or button or something like that.</a> <a data-t="1547">That's fine.</a> <a data-t="1548">That part of the graph can respond, can re-execute, can get back to equilibrium, can send results back to the user and then start waiting for the next input.</a> <a data-t="1563">So this is like when we added a new Microsoft stock quote to the application that was working with extended task while it was still busy.</a> <a data-t="1573">This still works which is exactly what we wanted.</a></p>

<p><a data-t="1578">Pretty soon this background task is finished so it changes to this smiling angel emoji and takes its result and brings it back into the reactive graph via this second piece of the extended task, this reactive value.</a> <a data-t="1592">And that causes reactivity to trigger in all the right ways and the outputs that depended on that result are now updated and everything is now at true equilibrium.</a> <a data-t="1602">Not only is the reactive graph at equilibrium but we have no more background task running.</a> <a data-t="1608">So now we truly are just waiting for the user to do something else.</a></p>

<h3>Summary and alternatives</h3>

<p><a data-t="1612">So to recap, with this approach we're able to achieve both inter and intra session concurrency unlike the previous approach of ShinyAsync.</a> <a data-t="1619">Extended task doesn't require you to learn a strange syntax like ShinyAsync so we sincerely hope that more people will be able to adopt this strategy for their long-running tasks than ever did with ShinyAsync and it will not be as invasive a refactor as ShinyAsync would sometimes force you to do.</a></p>

<p><a data-t="1650">Now keep in mind that for R, extended task still relies on the future package to put the task in the background.</a> <a data-t="1657">If the task is not happening off of the main R thread for the Shiny R process then nothing we do is going to achieve inter or intra session concurrency.</a> <a data-t="1673">We're just fundamentally limited to doing one thing at a time so we really need to use that future.</a> <a data-t="1679">There are a couple of limitations you should be aware of also if you're going to use this extended task feature.</a></p>

<p><a data-t="1685">In R there is no support currently for canceling a long-running task.</a> <a data-t="1694">Once you have invoked an extended task you can't or there's no built-in way to tell it to stop executing.</a> <a data-t="1702">There's also no built-in progress reporting.</a> <a data-t="1709">There is the ability to use a button that you click to start the operation and that button will say like processing but having like a progress bar that shows you how far along you are that is not currently supported either for R or for Python.</a> <a data-t="1742">We would really like to add both these features.</a> <a data-t="1745">On the R side it's going to involve working with future to make that possible.</a></p>

<p><a data-t="1758">Now one more thing I said you have to use future to launch your long-running task and that's sort of true.</a> <a data-t="1765">You do need to use future or something like future.</a> <a data-t="1769">There are a couple other alternative or complementary ways to run R code in the background.</a> <a data-t="1779">Now future is great.</a> <a data-t="1780">It's very convenient and it has like a very magical API I would say like you just say future and then put some code in it and even if that code refers to variables or packages that are outside of your future code block it'll just sort of figure out how to bring in everything it needs or tries to you know based on some crawling around in your environment and will work pretty hard to automatically make sure all those things are available automatically in the background R process that it launches.</a> <a data-t="1833">That's pretty cool and the other really great thing about future is that it's popular and has been been used for years by many people so there are pitfalls but those pitfalls are somewhat known.</a></p>

<p><a data-t="1846">Now the downsides of future are it has like pretty high runtime overhead so you do lose some performance whenever you invoke a future and the other thing about future is it's quite ambitious as a project.</a> <a data-t="1862">There are tons and tons of options.</a> <a data-t="1864">There are tons of extensions.</a> <a data-t="1866">There are lots of different um policies you can use with future for how it schedules so it's a lot and it can be its own thing to to learn which which can be uh make it a little harder to get started with.</a> <a data-t="1883">And finally it's pretty complex because of that automagicness because it automatically like teleports the data and packages you need to your background R process.</a> <a data-t="1893">That's a little scary and it can be doing things that like you weren't aware of in terms of you know maybe copying more data than you thought it would or accidentally depending on an object that really shouldn't be transported across.</a> <a data-t="1909">So just when it comes to that kind of automagic stuff sometimes it's hard to form a mental model for what's actually happening.</a></p>

<p><a data-t="1924">So one promising alternative slash complement to future is called </a><a href="/software/mirai/">Mirai</a><a data-t="1924">, M-I-R-A-I, which I believe is just Japanese for future.</a> <a data-t="1932">And that it's a new package by </a><a href="/people/charlie-gao/">Charlie Gao</a><a data-t="1932"> that is similar to future in that it can run code in a background R process.</a> <a data-t="1941">The good thing about this Mirai package is that it's super low overhead compared to future.</a> <a data-t="1946">It's very very fast and it's it's designed to be very simple and easy to understand so there's very little magic that it does for you.</a> <a data-t="1953">It doesn't magically slurp in whatever variables you happen to be using inside of your code chunk.</a> <a data-t="1961">It won't automatically load packages for you that it that it can see that you need.</a> <a data-t="1966">Instead you give it and basically expression that's going to be evalved in some R process and if that expression uses variables or even functions that are not just going to be provided by R then you need to provide them yourself.</a> <a data-t="1980">You need to tell Mirai and here are the variables and functions that you're going to need.</a> <a data-t="1985">Please make them available over on the other side.</a></p>

<p><a data-t="1990">The downsides of Mirai are it's still relatively new so it is not as battle-tested as future so there might be things that you know failure modes that we don't know about, might have bugs, and the fact that it has the simpler model and doesn't do all this magic for you is a kind of a double-edged sword.</a> <a data-t="2008">It's nice in that it's fast and simple to understand but it can be less convenient if you do have a lot of functions that you're using or a lot of values that you need to transport to the background R process.</a> <a data-t="2021">You then have to do that all explicitly.</a></p>

<p><a data-t="2028">And finally there's a package called CREW by Will Landau that you might know from the targets package and CREW builds on Mirai so it uses Mirai underneath and it adds basically a convenient way to launch multiple or many tasks both locally and on like standard HPC clusters that you might have especially in like pharma environments.</a> <a data-t="2059">I haven't used CREW that much but it is designed now to integrate with extended tasks somewhat and it has some examples in its documentation about how you can do that so if you have access to a big HPC cluster or you have many many tasks that you want to launch and manage together CREW might be the way to do it and you can do it using extended task as well.</a></p>
