<h3>Understanding ggproto</h3>

<p><a data-t="269">But this is also like the extent of normal code that you will see in extensions.</a> <a data-t="274">There's a reason why I'm here, and I'm talking to you about this, because all the other things</a> <a data-t="279">are based on this entity called ggproto, which is, yay, another object-oriented system in</a> <a data-t="286">R.</a> <a data-t="286">And ggproto is kind of this weird object-oriented system that is only relevant for ggplot2 extensions.</a> <a data-t="297">It is tailor-made for ggplot2.</a></p>

<p><a data-t="301">The reason why it's something new is that it needed to be a reference-based object-oriented</a> <a data-t="307">system, and the current or at that time current object-oriented systems didn't allow subclassing</a> <a data-t="314">objects between packages.</a> <a data-t="315">So if you want to make extensions, of course, you need to be able to subclass between ggplot2</a> <a data-t="320">and into your own package.</a> <a data-t="321">So Winston sat down and made something new.</a></p>

<p><a data-t="325">The good thing about this, in some sense, is that ggproto is completely hidden from</a> <a data-t="329">the user.</a> <a data-t="330">So if this is the first time you hear about ggproto, fear not.</a> <a data-t="333">This is how it should be.</a> <a data-t="336">But on the other hand, it's also weird.</a> <a data-t="338">So when you want to transition into writing extensions, you come into this completely</a> <a data-t="344">new territory that you've never seen before.</a> <a data-t="347">So code may look foreign to you, because it's a new system and some new semantics.</a> <a data-t="352">But don't worry, because it's not really as bad as it should appear.</a></p>

<blockquote><a data-t="352">But don't worry, because it's not really as bad as it should appear.</a></blockquote>

<p><a data-t="356">So basically ggproto is this scaffolding thing that holds all of ggplot together.</a> <a data-t="363">And it is used to orchestrate all the rendering and all the mechanics of how ggplot uses your</a> <a data-t="369">data and transforms it into a plot.</a></p>

<p><a data-t="373">So when you think about how ggproto at least should be used when you're writing extensions</a> <a data-t="377">to ggplot2 and how ggplot2 uses it itself, is that you have to think about ggproto classes</a> <a data-t="384">as somewhat of factories.</a> <a data-t="387">These are classes that are stateless in the sense that you have an object, and it receives</a> <a data-t="395">some data, and it does something to the data, and it spits out the data again.</a> <a data-t="399">It doesn't keep track of what it has been doing in that sense.</a> <a data-t="402">So this is really important, because when you plot a second time, you don't want it</a> <a data-t="407">to change.</a> <a data-t="408">You want it to behave as it should be every time you plot it.</a> <a data-t="411">So all of these ggproto objects are more or less stateless.</a></p>

<p><a data-t="416">There are a few that you shouldn't touch that are not stateless.</a> <a data-t="418">But for every kind of idea of extensions, you should think of these as kind of factories.</a> <a data-t="423">You have this assembly line, and each method is kind of this type of small robot arm or</a> <a data-t="430">whatever.</a> <a data-t="431">It doesn't change what it is doing between the different things that run through the</a> <a data-t="435">assembly line.</a> <a data-t="437">And methods for these classes, they span the assembly line.</a> <a data-t="441">So a class is not an object or a class is not really an area of the assembly line.</a> <a data-t="448">They are grouped based on what type of thing they are doing.</a> <a data-t="451">So we have stat ggproto classes.</a> <a data-t="454">We have geom ggproto classes.</a> <a data-t="456">And they might go in at different times in the rendering process and do their thing.</a></p>

<p><a data-t="464">So more formalized, we have this idea that we have generally in ggplot2.</a> <a data-t="476">We take some data, we run it through some transformation, and there will be a plot.</a> <a data-t="481">And ggproto objects are really these kind of entities that sits at various points through</a> <a data-t="485">this pipeline of data to plot.</a> <a data-t="489">And they have methods, and they modify the data, and then some other object takes over.</a> <a data-t="494">And that's modifying the data, maybe spits it back into the same object and so on and</a> <a data-t="498">so forth.</a> <a data-t="499">And then suddenly a plot appears, hopefully not disappears.</a></p>

<h3>Key ggproto methods</h3>

<p><a data-t="505">So there are a few things in the ggproto classes that are shared between a lot of the different</a> <a data-t="512">implementations.</a> <a data-t="513">And I'm just going to point out a few of them.</a> <a data-t="516">I'm not going to get too much into exactly how you're going to write a new faceting function</a> <a data-t="521">or how you're going to write a new position adjustment or whatever.</a> <a data-t="524">But just kind of highlighting a few of the things that you might potentially stumble</a> <a data-t="528">into when you're starting out with writing extensions.</a></p>

<p><a data-t="531">And one of the things that appears in almost all of the different extension points are</a> <a data-t="535">this setup param methods.</a> <a data-t="538">And the reason why it's there is because, as I said, your objects are stateless.</a> <a data-t="542">But if you want a state, and usually you want something, some kind of state based on the</a> <a data-t="547">data that you have in your plot, this is where you can get it.</a> <a data-t="550">So this receives the data in the layer, and it receives it together with some of the parameters</a> <a data-t="554">that you have defined when the user calls the plot.</a> <a data-t="558">And it can do whatever, and it can spit out a list of parameters that the ggplot2 rendering</a> <a data-t="564">system then takes hold of.</a> <a data-t="565">So it's not changing this data search, but it is returning a list of parameters that</a> <a data-t="569">will get passed in to the other different methods.</a> <a data-t="574">So it's kind of a pseudo stage that you are able to do.</a></p>

<p><a data-t="580">And one of the places where it's getting spit in again is in this setup data method</a> <a data-t="585">that a lot of the different methods has as well.</a> <a data-t="588">So it receives the data as well, and it receives these parameters that you have just calculated.</a> <a data-t="594">So it's stateful, but it's just handled outside of the class.</a> <a data-t="597">And what you can do in the setup data stage is that you can reparameterize data, which</a> <a data-t="603">is actually kind of a powerful concept.</a></p>

<p><a data-t="607">There are a few geoms in ggplot2 which kind of does the same thing.</a> <a data-t="612">For instance, you have geom segment, and then it's lesser well-known sibling called geom</a> <a data-t="616">spoke.</a> <a data-t="617">And the only difference between these two geoms is that geom segments takes the end</a> <a data-t="622">points of a segment and draws a line, while geom spoke takes a start point and an angle</a> <a data-t="626">and a length and draws a line.</a> <a data-t="629">But behind it all, everything it does is really geom spoke is a subclass of geom segment,</a> <a data-t="636">The only thing it does is it has a setup data which takes an angle, a start point, and a</a> <a data-t="640">length and calculates the new end point.</a> <a data-t="643">And then everything else just happens automatically.</a></p>

<p><a data-t="645">So if you have these kind of reparameterized stats or geoms, it's super simple to make</a> <a data-t="651">an extension, because the only thing you have to think about is to subclass it, provide</a> <a data-t="656">a new setup data method, and you're good to go.</a> <a data-t="658">Everything else will just happen automatically.</a> <a data-t="661">So this is a super important part, and you can really save yourself a lot of work by</a> <a data-t="665">just thinking about whether some of the things that you want to achieve is already handled</a> <a data-t="670">by something else.</a></p>

<p><a data-t="672">You can also do stuff like make sure if there are some columns that you need to have, but</a> <a data-t="676">the user doesn't necessarily want to define all the time, you can just inspect your data</a> <a data-t="682">and see, well, does it miss the width column?</a> <a data-t="685">Well, OK, we'll set it to something sensible and so on.</a> <a data-t="689">And just make sure that at this point, the data kind of makes sense for all of the remaining</a> <a data-t="695">steps of the object.</a></p>

<p><a data-t="699">Another quite important thing, especially when it comes to creating stats, which is</a> <a data-t="704">kind of one of the best places to start when creating extensions, is that the computation</a> <a data-t="710">part, like the meat of a stat element in ggplot2 is this kind of tired succession of calls</a> <a data-t="718">where you have a compute layer method, and by default, all it does is it's splitting</a> <a data-t="725">up the layer data by the panels, and then it calls compute panel.</a> <a data-t="729">And compute panel, by default, just splits up the panel data by the group aesthetic and</a> <a data-t="734">calls compute group.</a></p>

<p><a data-t="736">So depending on where you want to sit in that kind of succession, you can define a compute</a> <a data-t="742">group method, which is kind of simple because you know you're only getting data for a single</a> <a data-t="746">group at a time, and it may make it a lot easier to kind of figure out your computations.</a> <a data-t="752">But there's a lot of things that can be vectorized in R, and it might be stupidly inefficient</a> <a data-t="758">to compute the same thing per group.</a> <a data-t="760">So you can just instead define a new compute panel method that does not call compute group</a> <a data-t="765">but does everything in one step.</a> <a data-t="768">So it's usually easier to start at compute group level, but you can quite easily make</a> <a data-t="775">your geoms and stats more efficient by thinking about how can we efficiently do this in a</a> <a data-t="781">vectorized manner.</a></p>

<p><a data-t="783">And these compute things, they simply just take some data along with additional parameters,</a> <a data-t="788">and it spits out a new data frame.</a> <a data-t="789">So this is a very simple part of this kind of assembly line where it just takes data,</a> <a data-t="795">adds something to it or transforms it in some way and just spits out data again for the</a> <a data-t="799">assembly line to take over again.</a></p>

<p><a data-t="802">If you want to do a geom instead of a stat, it's a bit different but kind of the same</a> <a data-t="808">idea.</a> <a data-t="809">We have a tight succession of calls again, and it's just draw layer, draw panel, draw</a> <a data-t="812">group.</a> <a data-t="813">And again, sometimes you can vectorize things out, and you can really, really speed up your</a> <a data-t="819">drawing process.</a> <a data-t="820">For instance, if you think about the geom point, if we had to draw each point once or</a> <a data-t="827">one at a time and create a function that takes the group and just draws one point one</a> <a data-t="833">at a time, it will just be horribly inefficient.</a> <a data-t="836">But thankfully, grid, that is the motor behind ggplot2, can just take a vector of point or</a> <a data-t="842">coordinates.</a> <a data-t="843">So we don't need to do this per group.</a> <a data-t="844">We can do this per panel instead and really speed it up a thousand fold or something like</a> <a data-t="849">that.</a></p>

<p><a data-t="850">So again, it's easier to start at the group level, but think about how you can speed things</a> <a data-t="854">up by vectorizing it properly.</a> <a data-t="857">And this takes in data again, but this does not output data.</a> <a data-t="862">As Klaus talked about, there's this concept of grub in grids, which is graphical objects,</a> <a data-t="867">and the purpose of the draw method is simply to return grubs that will then be rendered</a> <a data-t="872">by grid in the end.</a></p>

<p><a data-t="875">There are a lot of other methods that are being used.</a> <a data-t="878">All of them are predefined.</a> <a data-t="879">You will usually not come up with your own method.</a> <a data-t="882">You will override the preexisting ones.</a> <a data-t="884">And most of these have sensible defaults, and most of these you don't have to worry</a> <a data-t="888">about that much anyway.</a></p>
