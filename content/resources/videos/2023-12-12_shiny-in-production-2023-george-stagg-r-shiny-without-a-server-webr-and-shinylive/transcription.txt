<h3>Existing Shiny hosting options</h3>

<p><a data-t="114">Now, there are some good solutions for running a Shiny server already. <h3>Static web hosting and its limits</h3>

<p><a data-t="351">There is one solution to this kind of thing, and academics like to do this kind of thing, which is static web hosting.</a> <a data-t="357">So you can actually, for free, put educational content or any web content, really, that's static, onto places like GitHub pages, Netify, Cloud for Web pages, or even QuartoPub has a free publishing platform where if you write a </a><a href="/software/quarto/">Quarto</a><a data-t="357"> document, you can upload that Quarto document to QuartoPub for free and get a URL to share with your friends.</a> <a data-t="379">If you haven't used Quarto before, by the way, it's kind of like the new R Markdown.</a> <a data-t="382">It's brilliant.</a> <a data-t="383">Do give it a go.</a> <a data-t="385">I'll be talking more about Quarto later.</a></p>

<blockquote><a data-t="433">But wouldn't it be great if we could run Shiny apps using static hosting?</a> <a data-t="439">And that's what shinylive does.</a></blockquote>

<h3>WebAssembly: the technology behind webR</h3>

<p><a data-t="571">So how does this work, technically?</a> <a data-t="572">Because, you know, I showed a picture of a phone.</a> <a data-t="575">You can't have R installed onto a phone.</a> <a data-t="578">So there's something magical going on here that makes this work.</a></p>

<p><a data-t="582">The main technical detail that allows this to actually happen is something called WebAssembly, which was released in 2017.</a> <a data-t="590">The technical description of what this is is a portable binary code format.</a> <a data-t="594">And what that means is that traditionally, code, when you compile it to run on a machine, needs to be compiled for that specific type of machine.</a> <a data-t="603">So you'll have a piece of code, and you'll compile it for a Windows machine that's a normal laptop.</a> <a data-t="609">Or you'll compile it for a Mac machine, or a Windows machine, or even a phone or a tablet.</a> <a data-t="614">And these are all producing what's called binary format, the output of compilation.</a> <a data-t="619">The code that the CPU actually runs, they're all different, depending on the machine you're running on.</a></p>

<p><a data-t="625">With WebAssembly, that's not true.</a> <a data-t="627">Because what happens is that that output, which is called a binary, doesn't just run on the machine, but it runs inside a web browser.</a> <a data-t="633">And most machines have a web browser.</a> <a data-t="636">And the web browser understands that code format.</a> <a data-t="640">So what you do is you compile your code once, and it will run anywhere, anywhere with a web browser that supports WebAssembly.</a> <a data-t="646">That's what portable means.</a></p>

<p><a data-t="649">And it's surprisingly good.</a> <a data-t="651">You'd expect that this kind of thing would enact a performance impact.</a> <a data-t="655">And it does, but it's much less than you would expect.</a> <a data-t="659">You really can have high-performance applications running inside web pages with WebAssembly.</a> <a data-t="664">There are even games that you can play through a web browser without installing anything to your machine.</a> <a data-t="669">It's really quite impressive.</a> <a data-t="671">And most modern browsers support it.</a> <a data-t="673">Chrome, Firefox, and Safari all support it.</a></p>

<p><a data-t="677">There are other benefits to using WebAssembly.</a> <a data-t="680">There is security implications.</a> <a data-t="682">Code that runs inside WebAssembly runs inside a container and is sandboxed.</a> <a data-t="687">And it inherits all of the years of work of security that web browsers provide.</a></p>

<p><a data-t="701">Another benefit of WebAssembly is something called numerical reproducibility.</a> <a data-t="705">So this is important, particularly for scientific or statistical applications, because there are algorithms that's known as what's called unstable.</a> <a data-t="713">And that means they're very sensitive to changes in the numbers.</a> <a data-t="718">Different computers will actually compute numerical algorithms slightly differently because of something called floating point numbers.</a> <a data-t="726">Which comes about because computers can't represent every number.</a> <a data-t="730">You know, there's an infinite number of them.</a> <a data-t="731">It can't represent all of them.</a> <a data-t="733">So what happens is it has to do approximations.</a> <a data-t="737">And those approximations are slightly different depending on what machine you run your algorithm on.</a> <a data-t="741">And therefore, eventually, the algorithm can give you different answers.</a> <a data-t="744">And that's not reproducible.</a> <a data-t="745">That's kind of bad science, right?</a></p>

<p><a data-t="748">So what WebAssembly does is one of the things it implements is a strict set of instructions for how floating point numbers should be calculated.</a> <a data-t="756">So there are certain guarantees about numerical reproducibility of algorithms that you don't get with normal compiled software.</a> <a data-t="763">So that's a nice bonus that WebAssembly provides.</a></p>

<p><a data-t="767">The way that generally people use WebAssembly these days is either, some people will write it directly, but that's really rare.</a> <a data-t="775">It's not designed to be written.</a> <a data-t="777">It's more designed to be compiled to as a target.</a> <a data-t="780">There are Rust compilers to convert Rust code into WebAssembly.</a> <a data-t="784">But the compiler that I'm particularly interested in is something called Inscripten.</a> <a data-t="788">And Inscripten is magical.</a> <a data-t="790">It's brilliant.</a> <a data-t="791">Because what it does is allows you to take some C or C++ code that's written for Linux and instead compile a WebAssembly binary using that code.</a> <a data-t="802">And it provides a sort of Linux-like environment.</a> <a data-t="805">It's been around for a few years.</a> <a data-t="806">It actually predates WebAssembly.</a> <a data-t="808">I first saw Inscripten in about 2014, I think.</a> <a data-t="812">Back then, it didn't use WebAssembly, but it still worked really, really well.</a></p>

<p><a data-t="817">And it's used for a few different projects.</a> <a data-t="819">So there's SQLite, FFmpeg, DOSBox.</a> <a data-t="821">These things can all run in the browser because they're using Inscripten to compile that C code into WebAssembly code.</a></p>

<h3>Challenges of compiling R for WebAssembly</h3>

<p><a data-t="830">Now, the full process goes kind of like this.</a> <a data-t="834">There are programming languages like Python and R that are written in C or C++.</a> <a data-t="840">Their interpreters that read the input code are written in these languages.</a> <a data-t="844">So, well, we'll just give that to Inscripten.</a> <a data-t="847">Produce a WebAssembly binary.</a> <a data-t="848">Job done.</a> <a data-t="849">Off to the pub.</a> <a data-t="850">But it's not that easy.</a></p>

<p><a data-t="853">It's actually quite difficult because of restrictions that WebAssembly code runs under in the browser.</a> <a data-t="860">So, for example, you can't open local files or any network sockets from inside WebAssembly.</a> <a data-t="869">And that's a shame because things like R, you can do read.csv and give it a URL.</a> <a data-t="873">And it will download that data for you.</a> <a data-t="875">And that's really nice.</a> <a data-t="876">So that's a shame.</a></p>

<p><a data-t="879">Threading and forking is difficult in WebAssembly.</a> <a data-t="882">There's no native threading instructions.</a> <a data-t="886">Graphics and font support is a bit difficult because WebAssembly doesn't have access to something called the JavaScript DOM.</a> <a data-t="891">So you have to kind of work around that.</a> <a data-t="893">And probably the biggest problem is legacy code.</a> <a data-t="896">It turns out that with these programming languages, it's not all C or C++.</a> <a data-t="900">There is code from 1972 or whatever that does a numerical algorithm that has been around for years and nobody's touched.</a> <a data-t="908">Written in something like Fortran.</a> <a data-t="910">I've done my fair share of hacking at Fortran 77 code just to get WebR running.</a> <a data-t="917">And you have to support these things.</a> <a data-t="918">They've been around for a long time and they're a core part of the language.</a></p>

<p><a data-t="923">So that's quite difficult.</a> <a data-t="924">It makes things harder than it really needs to be.</a> <a data-t="928">But it looks like Unix.</a> <a data-t="931">It looks like a Linux environment.</a> <a data-t="933">But there are browser security limitations.</a> <a data-t="935">And the trick with things like this is to replace those older pieces of code with new pieces of code that use browser APIs that allow you to access networks or download files or show graphics on the screen in a safe and secure way.</a> <a data-t="951">That's the game.</a></p>

<p><a data-t="954">And it works.</a> <a data-t="955">PyDyed is a port of Python to WebAssembly.</a> <a data-t="958">It's been around a little while, longer than WebR.</a> <a data-t="961">And you can install and run certain Python packages in a web browser.</a> <a data-t="966">It supports a fair few packages, including the scientific stack of Matplotlib and Numbers and Pandas and various other packages.</a> <a data-t="974">It works really, really well.</a> <a data-t="976">And maybe two years ago at this point, I remember seeing PyDyed, trying it out and being blown away by being able to run Python in a web browser.</a> <a data-t="984">I thought that was brilliant.</a> <a data-t="985">But there was no equivalent for R.</a></p>

<p><a data-t="989">And I kind of needed this because I was teaching at the time.</a> <a data-t="991">And I had students.</a> <a data-t="992">It was just as COVID was kicking off.</a> <a data-t="995">And they didn't have access to a computer cluster anymore.</a> <a data-t="997">They would have their own devices.</a> <a data-t="998">And they'd have something like an iPad.</a> <a data-t="999">And they were doing all their work on an iPad.</a> <a data-t="1001">And I was like, well, how am I going to install RStudio onto an iPad?</a> <a data-t="1005">We didn't have PositCloud or RStudio Cloud at the time.</a> <a data-t="1007">We didn't have access to anything like that.</a> <a data-t="1009">So I thought, well, there must be some way to make this work.</a> <a data-t="1011">It works for Python.</a> <a data-t="1013">It must work for R.</a></p>

<p><a data-t="1015">So I made WebR.</a> <a data-t="1016">It's a version of the R interpreter that's built for WebAssembly.</a> <a data-t="1019">And it allows you to execute R code directly in a web browser without supporting R server.</a> <a data-t="1025">That's because the R process runs directly in your web browser.</a> <a data-t="1029">For techie people who know about these kind of things, it does support Node.js to run R code server side.</a> <a data-t="1036">And it is being used, for example, by the R Universe team to run certain things server side with WebR.</a> <a data-t="1043">But really, we're focusing mainly on the front end, running R in a browser.</a> <a data-t="1049">That's mainly what we're focusing on.</a></p>

<h3>How Shiny runs in the browser with shinylive</h3>

<p><a data-t="1708">The way this works is that web browsers have a feature</a> <a data-t="1712">called service workers.</a> <a data-t="1714">What service workers do, well, they do a few things.</a> <a data-t="1716">But one of the important things that they do</a> <a data-t="1718">is they can capture network traffic.</a> <a data-t="1722">So what you can do is when you open up a Shiny app in your machine,</a> <a data-t="1726">what will happen in the background is the Shiny app will go,</a> <a data-t="1728">oh, I need to connect to the Shiny server.</a> <a data-t="1730">I need to load, you know, johnswebsite.com,</a> <a data-t="1734">colon, whatever port Shiny is running on, or whatever.</a> <a data-t="1737">There's a connection that happens in the background to a Shiny server.</a> <a data-t="1740">Service workers can intercept that connection</a> <a data-t="1743">and say, no, don't go over there.</a> <a data-t="1744">Instead, go over here.</a> <a data-t="1746">In fact, don't even go anywhere.</a> <a data-t="1748">Just run this piece of code.</a> <a data-t="1750">And that's what service workers do.</a> <a data-t="1751">And that's how Shiny is able to run in WebR.</a> <a data-t="1753">What WebR will do is start a service worker</a> <a data-t="1757">and capture that network traffic that would normally be sent to a Shiny server.</a> <a data-t="1760">And instead, rather than sending it over the network,</a> <a data-t="1763">it will send it to a WebAssembly thread where WebR is running</a> <a data-t="1767">and say, WebR, what would Shiny do if it was running on a server?</a> <a data-t="1770">Give me the response that Shiny would give.</a> <a data-t="1773">And that's how that works.</a></p>

<p><a data-t="1775">So what that allows you to do is run the Shiny server itself on the local machine.</a> <a data-t="1781">The Shiny client can be opened in the web browser.</a> <a data-t="1787">And it looks and feels just like a normal Shiny client,</a> <a data-t="1789">because as far as the Shiny client is concerned,</a> <a data-t="1791">it really is connecting to a remote server.</a> <a data-t="1793">But the service worker allows that remote server</a> <a data-t="1797">to actually be running inside the browser.</a></p>

<p><a data-t="1800">Now, you can set this up manually.</a> <a data-t="1802">It's a bit of a pain.</a> <a data-t="1804">I do have a GitHub repo that explains how to do all this</a> <a data-t="1807">and set up a static web server and Shiny in the background.</a> <a data-t="1810">But like I say, you really have to be a JavaScript developer</a> <a data-t="1814">to really understand what's going on there.</a> <a data-t="1817">So instead, we use shinylive.</a> <a data-t="1820">And shinylive makes this much, much easier.</a> <a data-t="1822">And it makes it easy enough that everyone in this room</a> <a data-t="1825">could create their own serverless Shiny app.</a> <a data-t="1827">Even if you've never made a Shiny app before.</a></p>
