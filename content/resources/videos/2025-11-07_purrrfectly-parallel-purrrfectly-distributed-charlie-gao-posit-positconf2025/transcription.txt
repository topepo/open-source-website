<h3>Why parallel purrr, and why now</h3>

<p><a data-t="201">So to answer the question, why do we have parallel purrr</a> <a data-t="203">and why do we have it now?</a> <a data-t="206">The answer is both purrr and mirai</a> <a data-t="208">were designed to be used reliably in programming.</a> <a data-t="212">So if you remember back to the first slide,</a> <a data-t="216">purrr was used for programming.</a> <a data-t="219">And why can I say the same thing about mirai?</a> <a data-t="222">Well, mirai has this design philosophy.</a></p>

<p><a data-t="228">It's based on four pillars.</a> <a data-t="230">It has a modern foundation,</a> <a data-t="231">which gives it the performance that it has.</a> <a data-t="234">And because it was designed for production,</a> <a data-t="237">you can have the confidence to deploy it everywhere.</a></p>

<p><a data-t="241">What do I mean by this?</a> <a data-t="242">Well, in terms of foundation, it's built on NNG.</a> <a data-t="246">This is short for Nano Message Next Generation.</a> <a data-t="249">This is a high-performance messaging library.</a> <a data-t="252">And because it's built on that,</a> <a data-t="253">we have the optimal connection types built in.</a> <a data-t="256">So things like inter-process communications,</a> <a data-t="258">TCP and TLS if you need it.</a> <a data-t="262">We've also engineered base R serialization mechanism</a> <a data-t="267">to better support custom serialization</a> <a data-t="271">of newer cross-language data formats,</a> <a data-t="274">things like Arrow or Safe Tensors.</a></p>

<p><a data-t="279">Because we have this foundation,</a> <a data-t="282">mirai scales to millions of tasks</a> <a data-t="284">over thousands of connections.</a> <a data-t="287">And we can all do this at 1,000 times</a> <a data-t="290">the efficiency we could otherwise.</a> <a data-t="293">The extra bit of engineering that we've done</a> <a data-t="296">are to create these zero-latency promises.</a> <a data-t="299">So if you use mirai together with something like </a><a href="/software/shiny-r/">Shiny</a><a data-t="299">,</a> <a data-t="303">you will notice the extra responsiveness from that.</a></p>

<p><a data-t="308">And we come to this point,</a> <a data-t="309">which is why we have Parallel Pro in the first place.</a> <a data-t="313">mirai was designed to be used in production,</a> <a data-t="317">i.e. reliably.</a> <a data-t="319">And it does this by adopting a very clear evaluation model</a> <a data-t="323">with a clean separation between</a> <a data-t="326">what's in your current environment</a> <a data-t="329">and what's in the environment of the process</a> <a data-t="333">which is actually doing the parallel work.</a></p>

<blockquote><a data-t="319">mirai was designed to be used in production,</a> <a data-t="317">i.e. reliably.</a> <a data-t="319">And it does this by adopting a very clear evaluation model</a> <a data-t="323">with a clean separation between</a> <a data-t="326">what's in your current environment</a> <a data-t="329">and what's in the environment of the process</a> <a data-t="333">which is actually doing the parallel work.</a></blockquote>

<p><a data-t="337">We've minimized the complexity of the package itself,</a> <a data-t="341">and we've eliminated any hidden state.</a> <a data-t="345">What this means for you is the code you write,</a> <a data-t="348">you can expect it to be evaluated transparently</a> <a data-t="351">and reliably over time.</a></p>

<p><a data-t="354">And you can literally deploy this anywhere.</a> <a data-t="357">So locally, on your own laptop,</a> <a data-t="360">on a remote machine over SSH,</a> <a data-t="362">and this is what I'm gonna be covering</a> <a data-t="364">in the second half of the presentation,</a> <a data-t="366">or if you're lucky enough to have access</a> <a data-t="367">to a high-performance compute cluster,</a> <a data-t="369">you can use your scheduler of choice.</a> <a data-t="373">And then mirai has this concept</a> <a data-t="376">of modular compute profiles.</a> <a data-t="378">So you can actually be connected</a> <a data-t="380">to all three resource types at the same time.</a></p>

<p><a data-t="386">And I'm gonna say one last thing about mirai</a> <a data-t="388">before we move on to distributed computing,</a> <a data-t="392">and that we've not just put mirai into purrr,</a> <a data-t="395">we've rolled mirai out across our ecosystem.</a> <a data-t="399">So mirai is now the primary async backend for Shiny.</a> <a data-t="404">It is the only built-in async evaluator for </a><a href="/software/plumber/">Plumber</a><a data-t="404"> 2.</a> <a data-t="410">And it's also an option to run things</a> <a data-t="413">in parallel in tidy models,</a> <a data-t="415">such as hyperparameter tuning.</a></p>

<h3>Distributed purrr</h3>

<p><a data-t="455">So how might you use distributed purrr?</a> <a data-t="459">And all I mean by distributed is,</a> <a data-t="462">have your compute run on another machine,</a> <a data-t="464">so not on the laptop or the local machine you're using.</a> <a data-t="468">Well, firstly, simply you can use it</a> <a data-t="470">to replace local compute.</a> <a data-t="471">So maybe you're working on an underpowered laptop,</a> <a data-t="474">and you actually need to do heavy computations,</a> <a data-t="478">which are more suited to a workstation or a server.</a></p>

<p><a data-t="483">Another way you can use distributed compute</a> <a data-t="485">is to extend your local compute.</a> <a data-t="487">So as well as using all the calls on your laptop,</a> <a data-t="491">you can also use another machine,</a> <a data-t="494">or even multiple machines on your network.</a></p>

<p><a data-t="499">And lastly, I'm gonna show you</a> <a data-t="501">how to use differentiated compute.</a> <a data-t="502">So this is a case where you have a workflow</a> <a data-t="506">where certain calculations may require a GPU,</a> <a data-t="510">which you don't have on your local machine,</a> <a data-t="512">but on another machine.</a></p>

<p><a data-t="517">So before I do that,</a> <a data-t="520">for those of you who have not encountered</a> <a data-t="523">the new parallel functionality in purrr yet,</a> <a data-t="527">what we've done is we've added this in parallel adverb.</a> <a data-t="531">So whereas before you would map a function</a> <a data-t="535">over a list or a vector,</a> <a data-t="537">all you would do now is map in parallel</a> <a data-t="542">a function over a list or a vector.</a></p>

<p><a data-t="549">And again, before we move on to distributed case,</a> <a data-t="554">how would you run that parallel map</a> <a data-t="571">on your local computer,</a> <a data-t="573">using all the calls available to you?</a> <a data-t="576">Well, as a user, you would call this function daemons</a> <a data-t="580">in the mirai package.</a> <a data-t="583">Daemons sets where and how</a> <a data-t="587">you actually do this parallelization.</a> <a data-t="589">And it's for the user to set this,</a> <a data-t="591">because when you write the code,</a> <a data-t="596">your code may be in a package, for example,</a> <a data-t="599">and you don't know the resources</a> <a data-t="602">that are available to the end user.</a> <a data-t="603">So when the user actually runs your code,</a> <a data-t="606">they will call this daemons function.</a> <a data-t="608">And daemons sets where and how</a> <a data-t="610">this parallelization actually happens.</a></p>

<p><a data-t="614">So in this case, to run things on your local laptop,</a> <a data-t="619">all you would do is you would pass a local URL</a> <a data-t="622">to daemons.</a> <a data-t="623">What this does is this creates a socket</a> <a data-t="626">which is listening for connections</a> <a data-t="629">from other processes on your own machine.</a> <a data-t="632">Think of this as setting up a base station.</a> <a data-t="637">You would then call this function launch local.</a> <a data-t="640">This actually spawns local processes,</a> <a data-t="643">which dial in back to your local URL.</a> <a data-t="645">So these are your subglides.</a> <a data-t="647">And this is how mirai actually works under the hood.</a> <a data-t="651">What you would actually do</a> <a data-t="652">is you would actually just do this in one line</a> <a data-t="654">and just say how many processes you want.</a></p>

<h3>Replacing local compute with SSH</h3>

<p><a data-t="663">So in the distributed case,</a> <a data-t="664">you're not just starting processes on your local machine.</a> <a data-t="667">You're starting them on another machine.</a> <a data-t="670">Now, SSH is the most ubiquitous way</a> <a data-t="673">to run things on other machines.</a> <a data-t="675">So you would have come across this</a> <a data-t="676">if you've ever set up a home network, for example.</a> <a data-t="679">Or conversely, if you spun up a cloud instance,</a> <a data-t="683">something like an EC2 on AWS.</a> <a data-t="686">The first thing you would get in that case</a> <a data-t="688">is the IP address of your cloud instance,</a> <a data-t="691">and you would SSH in,</a> <a data-t="693">and you would do all the setup for that instance.</a></p>

<p><a data-t="696">Well, how would you use that in mirai?</a> <a data-t="698">Well, you would take that IP address,</a> <a data-t="701">or if it's just a machine on your local network,</a> <a data-t="704">you can also use the host name,</a> <a data-t="706">and you would pass that to this SSH </a><a href="/software/config/">config</a><a data-t="706"> function.</a> <a data-t="711">What that does is that creates a remote launch configuration.</a> <a data-t="714">This is a simple list.</a> <a data-t="716">There's no other special source in that.</a> <a data-t="718">So you can save that, you can reuse that,</a> <a data-t="720">you can even put this line in your .R profile.</a></p>

<p><a data-t="725">Next, we would call daemons like we did before,</a> <a data-t="729">and in this case, we would use a host URL.</a> <a data-t="733">The only difference here is that a host URL</a> <a data-t="736">is available for connections from your network,</a> <a data-t="740">so not just on your local machine.</a> <a data-t="743">And then instead of launch local,</a> <a data-t="744">we now say launch remote,</a> <a data-t="746">and we pass it the configuration that you've created.</a> <a data-t="751">You can also do all of that on one line.</a></p>

<p><a data-t="757">And once you've done that,</a> <a data-t="762">the, sorry, when you launch those daemons,</a> <a data-t="766">you actually SSH into those machines</a> <a data-t="768">and create those processes,</a> <a data-t="770">which dial back in to your host URL.</a> <a data-t="774">So your host has to be able</a> <a data-t="777">to accept these incoming connections.</a> <a data-t="779">Now, that might already be set up,</a> <a data-t="781">but if you're just using something like a local laptop,</a> <a data-t="785">typically that won't be the case,</a> <a data-t="786">because there'll be a firewall</a> <a data-t="788">that's blocking all of these connections.</a></p>

<p><a data-t="792">What you would do in that case</a> <a data-t="794">is use SSH tunneling to get around this problem.</a> <a data-t="799">Now, I'm gonna explain SSH tunneling</a> <a data-t="802">in the simplest way possible,</a> <a data-t="804">without the use of a diagram.</a> <a data-t="806">So, whereas before, you have a remote machine</a> <a data-t="809">connecting back into your local machine,</a> <a data-t="812">all that's happening now is,</a> <a data-t="813">on both ends, connections are local to local,</a> <a data-t="816">so think of it as a loop,</a> <a data-t="817">local to local,</a> <a data-t="819">and you only have the SSH connection acting as a bridge.</a></p>

<p><a data-t="826">What does that mean in terms of,</a> <a data-t="828">for mirai, well, all we have to do</a> <a data-t="831">is add tunnel equals true</a> <a data-t="833">when you're creating the configuration.</a> <a data-t="835">That's the only change you need.</a> <a data-t="837">But when you get to set up daemons,</a> <a data-t="839">instead of host URL, you're moving back to local URL,</a> <a data-t="842">because all connections are actually local now.</a> <a data-t="845">But you have to remember to set TCP equals true,</a> <a data-t="847">because SSH requires TCP connections.</a> <a data-t="851">And we launch remote the same as before.</a> <a data-t="857">And you can do it in one line,</a> <a data-t="858">and once you've done that,</a> <a data-t="859">you will have six daemons on a remote machine,</a> <a data-t="863">regardless of the firewall settings that you have.</a></p>
