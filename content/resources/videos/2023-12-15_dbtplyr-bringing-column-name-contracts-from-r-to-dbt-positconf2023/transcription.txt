<h3>dplyr's tidy-select helpers</h3>

<p><a data-t="182">But first, let's start with dplyr.</a> <a data-t="184">As many of you may have seen in your own work, dplyr contains a number of helpful tidy-select</a> <a data-t="190">functions, which kind of imply the existence of column names having meaning and being something</a> <a data-t="197">we can act on and operate.</a> <a data-t="199">So we have functions such as starts with, ends with, and contains.</a> <a data-t="204">And how these work is that these selectors, when applied to a dataset, help us pick out</a> <a data-t="210">column names that follow certain patterns.</a></p>

<p><a data-t="213">For example, finding all count variables that maybe start with something like n, or finding</a> <a data-t="218">all variables that relate to some concept, like containing the word spend somewhere within</a> <a data-t="223">them.</a> <a data-t="224">And this simple idea of expecting column names to have a pattern and have a meaning unlocks</a> <a data-t="230">the idea of them being something we can program against and operate on.</a> <a data-t="235">And it incentivizes the strategic use of column names for this general pattern of find columns</a> <a data-t="241">and do something.</a></p>

<p><a data-t="243">And this became very useful in many aspects of my data science, data analysis work.</a> <a data-t="248">In transformation, in modeling, and in validation.</a> <a data-t="252">So just to give a few examples, using something like dplyr for data wrangling, we can find</a> <a data-t="258">columns and then use that for more efficient transformation.</a> <a data-t="262">If we want to do something as simple as the data aggregation, instead of writing the same</a> <a data-t="266">aggregation function, for example, taking the mean of every indicated variable to calculate</a> <a data-t="271">a proportion, we can write that line of code once and have it applied to all relevant variables.</a></p>

<p><a data-t="278">In the modeling context, this unlocks different capabilities of find columns and do something</a> <a data-t="284">else, like prevent model leakage.</a> <a data-t="288">Often when I'm modeling, I like to keep variables in my dataset that should not be used as predictors</a> <a data-t="293">in that model, but might be useful to help me diagnose weak spots in the model as I'm</a> <a data-t="298">looking to enhance future performance.</a> <a data-t="302">But with a package like recipes, to ensure that I don't accidentally include these as</a> <a data-t="306">predictors, I can name those features variables strategically, find those columns, and assign</a> <a data-t="312">them to a role that ensures that they don't enter into the wrong part of the modeling</a> <a data-t="316">pipeline.</a></p>

<h3>Column names as a micro-language</h3>

<p><a data-t="355">So in my own work in the data science and data analysis space, I found this simple idea</a> <a data-t="362">of being able to find column names and expect meaning in those to be really useful, so much</a> <a data-t="367">so that I started to think about column names in and of themselves as a kind of language.</a> <a data-t="373">So as I moved in and out of the data science roles into at times more analytics engineering</a> <a data-t="378">or data engineering type responsibilities, I thought about how I could operationalize</a> <a data-t="384">this concept to not just have meaningful column names for myself, but to better communicate</a> <a data-t="391">between data producers and data consumers by embedding meaning in those column names</a> <a data-t="395">for others.</a></p>

<p><a data-t="397">Column names are themselves a form of abstraction that capture the messiness and the complexity</a> <a data-t="403">and the rich underlying meaning of a dataset.</a> <a data-t="407">So to do this, I started thinking about names not just as words, but more like sentences</a> <a data-t="412">or paragraphs.</a> <a data-t="414">So I started thinking about defining sets of stubs that can encode different types of</a> <a data-t="419">semantics and different types of performance contracts about my data, and then when going</a> <a data-t="425">to name columns for a dataset I intended to publish, permuting those together to create</a> <a data-t="430">a rich combinatorial explosion of intuitive and really meaningful column names.</a></p>

<blockquote><a data-t="397">Column names are themselves a form of abstraction that capture the messiness and the complexity</a> <a data-t="403">and the rich underlying meaning of a dataset.</a></blockquote>

<p><a data-t="438">So to just give a few examples of how this is useful, you can imagine maybe defining</a> <a data-t="443">one layer that captures something like the rough types or intended use cases of variables.</a> <a data-t="450">So for example, ID, indicator, count, date, hopefully many of these seem fairly intuitive.</a> <a data-t="456">That's the goal.</a> <a data-t="457">But beyond intuition alone, we can define very explicit semantics for these variables</a> <a data-t="463">so people know exactly what these things should mean when they look at them.</a></p>

<p><a data-t="467">For example, an indicator variable would be a 0, 1 indicator, and the rest of the name</a> <a data-t="473">would always describe the positive case.</a> <a data-t="476">So we could push towards more specific names and not have ambiguity of which case encodes</a> <a data-t="485">the positive versus the negative.</a> <a data-t="488">But beyond semantics, we can also use these stubs to make promises to the users of our data.</a> <a data-t="494">So for example, indicator variables perhaps always being non-null or dates always being</a> <a data-t="500">in everyone's favorite ISO 8601 format.</a></p>

<p><a data-t="505">And similarly, you can imagine doing the same process as other layers as it best relates</a> <a data-t="510">to your specific use case and tackling some of those gnarly problems in our enterprise</a> <a data-t="515">such as everyone having a different definition of what is the user or what is the customer.</a> <a data-t="520">If we define for a specific stub a singular meaning for something like a user, we can</a> <a data-t="526">then use that concept and that definition consistently throughout all of our datasets</a> <a data-t="531">to push towards more clarity and again that better communication across different job</a> <a data-t="536">families and areas of responsibility.</a></p>

<p><a data-t="539">And beyond encoding semantics and contracts, we can also encode the consequences of those</a> <a data-t="545">choices.</a> <a data-t="546">So for example, if users are identified by IP addresses, sometimes that may result in</a> <a data-t="551">inconsistencies in identification across devices.</a> <a data-t="556">And so forth and so on.</a> <a data-t="557">You can imagine this process playing out differently in different enterprises.</a> <a data-t="561">But the idea being for once, define and align on this common language and then use that</a> <a data-t="568">to create rich, latent communication between different families.</a></p>

<h3>Translating the concept to dbt</h3>

<p><a data-t="574">And the benefits of this pattern really, really cleared me, having come from the perspective</a> <a data-t="579">first of that data science and that data analyst user of how they would interpret and use these</a> <a data-t="585">names.</a> <a data-t="586">But to encourage others on the data production side to use this same pattern, I found that</a> <a data-t="595">I had to make one more set of translation and translate this concept into tooling used</a> <a data-t="602">by analytics engineers and data engineers so they could also benefit and really see</a> <a data-t="608">it and experience for themselves the value of this sort of workflow and this sort of</a> <a data-t="613">pattern.</a></p>

<p><a data-t="615">And that's what encouraged me to take those tidy select helpers that inspired my own workflows</a> <a data-t="620">and translate them into a dbt package called dbtplyr.</a></p>

<p><a data-t="626">Now for those that aren't familiar with dbt, I'll just say a few brief words about what</a> <a data-t="631">that is.</a> <a data-t="632">Dbt is a data engineering framework that largely sits on top of the SQL language and helps</a> <a data-t="637">people build better quality data pipelines by encoding software engineering best practices</a> <a data-t="642">such as enabling clean code with variables and control flow like if statements and for</a> <a data-t="648">loops that you can't necessarily do natively in SQL, promoting more organized projects</a> <a data-t="654">for better structure and reproducibility, and adding some syntactic sugar for better</a> <a data-t="659">developer workflows with things like testing, logging, and orchestration.</a></p>

<p><a data-t="665">So although it's a different language and a different framework, I felt like the dbt</a> <a data-t="669">community and the dbt language already shared a lot of the same underlying values as the</a> <a data-t="674">R community in the way that we valued the scientific quality and reproducibility and</a> <a data-t="680">the idea of bringing these best practices from these different fields into the data</a> <a data-t="684">space.</a></p>

<p><a data-t="686">And just to give one example of how dbt does this, you may be familiar with a traditional</a> <a data-t="691">SQL query that really doesn't follow the dry principles of do not repeat yourself.</a> <a data-t="697">There's a lot of redundancy, a lot of repeated copy-paste business logic, which can cause</a> <a data-t="701">a lot of room for error.</a> <a data-t="704">But dbt, by adding variables and adding for loops, we're able to get a lot closer to that</a> <a data-t="710">dry streamlined workflow that is probably more comfortable to many of us coming from</a> <a data-t="715">more traditional coding languages, as well as encoding a lot of our business logic in</a> <a data-t="721">macros and packages.</a></p>

<p><a data-t="724">And because of this packaging ecosystem, I was able to write a dbt package called dbtplyr,</a> <a data-t="729">which translates all those select helpers that I knew and loved and found so many applications</a> <a data-t="736">for in my R-based workflows to dbt.</a> <a data-t="740">So we have all those same functionalities like starts with, ends with, and contains.</a> <a data-t="746">And what this same pattern looks like as it plays out in the dbt workflow is something</a> <a data-t="751">like this.</a> <a data-t="752">Once again, we can find columns and do stuff, whatever that stuff may be.</a> <a data-t="758">First, we can find column names that match some sort of a pattern.</a> <a data-t="763">Next, in a select statement, we can apply whatever transformation we wish to all of</a> <a data-t="769">the variables matching that pattern.</a> <a data-t="772">And similar to dbtplyr, if we so choose, we can do similar things in filter statements</a> <a data-t="778">and where clauses.</a></p>
