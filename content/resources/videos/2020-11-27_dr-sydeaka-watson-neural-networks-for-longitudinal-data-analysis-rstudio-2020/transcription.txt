<h3>What is longitudinal data?</h3>

<p><a data-t="117">So title, Neural Networks for Longitudinal Data Analysis.</a> <a data-t="119">First let's start with the definition.</a> <a data-t="119">So longitudinal data are recorded on the same units at multiple points in time.</a> <a data-t="125">And so as an example, you could think about a clinical trial where you might have a patient observed over multiple time points, maybe every three months, every six months or something.</a> <a data-t="134">They come in and you take their vital signs, you weigh them, you take blood or tissue samples, and then maybe you record some biomarkers at each of the different time points.</a> <a data-t="145">So that's a healthcare example.</a></p>

<p><a data-t="145">Another example could come from retail where maybe you'd observe a customer's behaviors or characteristics every month, or perhaps you're analyzing user behavior, so maybe you're at Facebook or Twitter, and you're looking at how the users behave or interact with the platform every day.</a> <a data-t="202">So there are lots of other examples.</a></p>

<p><a data-t="205">In those applications, we're looking at people over time, but we could also have a longitudinal data set that stores information about buildings or objects, pets, anything really, whatever it is, where again, we're looking at the same thing over multiple time points.</a> <a data-t="220">And they don't have to be, but usually the time points are evenly spaced, so say we're making observations every day or every month, and then the units don't necessarily have to have the same number of observations.</a> <a data-t="232">So thinking back to that clinical trial example, some patients might come in only twice, and then some patients might come in 10 times.</a> <a data-t="239">And you can think of each unit as having a matrix of data where the variables are observed or across the top of the matrix, and the time points are in the rows.</a></p>

<h3>Modeling considerations</h3>

<p><a data-t="347">So the question is, now that we have these three players in mind with varying performance, with their actual slash lines shown below, we'd like to somehow build a model that can predict these outcomes.</a></p>

<p><a data-t="361">Now for this type of scenario, we have three key modeling considerations.</a> <a data-t="361">First, we know that there is a very strong correlation between past performance and future performance.</a> <a data-t="369">So data from one year to the next are highly correlated, and so we should have some sort of smart way to encode these temporal dependencies and exploit them if possible.</a></p>

<p><a data-t="382">Next we'd like to avoid feature engineering over the different time periods.</a> <a data-t="391">And so if you may have been dealing with this type of scenario in the past, maybe if you had data over multiple time points, you might say, OK, let me aggregate somehow over the different time points how many runs they had or look at some percentage from one period to the next or something like that.</a> <a data-t="407">And then we might also have to manually encode some of the interactions between some of the variables.</a> <a data-t="411">So obviously we're limited by time and our own imaginations, right?</a> <a data-t="416">So there's almost an infinite number of ways that you could do that.</a> <a data-t="419">And so we'd like to, if possible, allow the algorithm to come up with those types of transformations.</a></p>

<p><a data-t="425">And then finally, we would like this model to flexibly accommodate different types of model outputs.</a> <a data-t="434">We might want to predict more than one outcome at a time.</a> <a data-t="437">So such as in our example where we want to predict the three numbers in the slash line.</a> <a data-t="443">And normally in a regression, you would have three separate models, right, where you individually predict those three outcomes.</a> <a data-t="451">And then so each of those individual models are learning separate patterns that are predictive of each of those outcomes.</a> <a data-t="456">But maybe you want to have one model—maybe it's easier to maintain—that looks at individual patterns that can come together and simultaneously predict all of those outcomes.</a> <a data-t="467">And maybe we also want to think about predicting over multiple time points.</a> <a data-t="470">So maybe saying for those three players, could we predict how they performed over the next two or three seasons, right?</a> <a data-t="476">So that's a lot to ask.</a></p>

<h3>Recurrent neural networks for longitudinal data</h3>

<p><a data-t="479">Okay, so recurrent neural network models to the rescue.</a> <a data-t="487">Those for people who have seen those before, you may have been aware that those naturally accommodate temporal dependencies.</a> <a data-t="495">You've probably seen them before in different types of sequential data analyses such as written text or audio.</a> <a data-t="503">There are some variants such as long short-term memory and gated recurrent units that can regulate the flow of information through the network, in particular looking at data from one period to the next.</a></p>

<p><a data-t="515">So this slide shows an example of how information would flow through such a network for our incredible season, 1965 season for our high-performing player Willie Mays.</a> <a data-t="528">And we see that he has the same attributes, age, games, number of hits, number of runs, etc., recorded with different values for each of these years in 1962, 1963, and 1964.</a> <a data-t="544">And so the model is learning all of these season-to-season patterns over all the players that are predictive of these three outcomes in the slash line.</a></p>

<h3>Data preparation</h3>

<p><a data-t="555">To prepare the data, we must break each player's record into these rolling four-season windows.</a> <a data-t="563">And so what that includes is the three seasons, remember, that we're using for prediction.</a> <a data-t="567">So those will be the input years of data.</a> <a data-t="570">And then we've got the one season in the future, right?</a> <a data-t="573">So one of the things that was an issue, of course, because players are not active every single year, is that if they were inactive for a particular season, we have to do some padding.</a> <a data-t="583">And so there are some utilities in R that were helpful for that.</a> <a data-t="586">We have to pad the series to accommodate the skipped years.</a></p>

<p><a data-t="589">But for example, for Willie Mays, we could think about predicting 1962.</a> <a data-t="595">So we're using the data from 59, 60, and 61, and using that to predict his 62 performance.</a> <a data-t="602">And he was also active in 63.</a> <a data-t="605">So we can slide that window to the right and have 60, 60, and 62 predicting 63.</a> <a data-t="612">And we can do it again to predict 64, and slide it over one more time to predict 65.</a> <a data-t="617">Now, this, of course, means that our data samples are not independent.</a> <a data-t="621">And this is OK for now, because we're focusing primarily on the predictions that we're getting from the model at the moment.</a> <a data-t="629">But there are some other aspects of this model where we will have to address this dependence issue head on.</a> <a data-t="635">And so I'll revisit that a little later in the talk.</a></p>

<p><a data-t="639">I focused on data collected between 1916 and 2018.</a> <a data-t="643">And I only looked at the players' first stint.</a> <a data-t="645">So I didn't consider what happened after they were traded to another team.</a> <a data-t="650">And this was really just more of because of the way that the data was collected, it was actually reporting performance on each individual stint.</a> <a data-t="657">I could have also aggregated across the different time points if I was doing this for real.</a> <a data-t="661">But again, this was just an illustration.</a> <a data-t="664">I only looked at players that had at least four consecutive seasons.</a></p>

<p><a data-t="671">And think about the way that I'm constructing these rolling windows.</a> <a data-t="671">So a player that has a longer career has more rolling windows, right?</a> <a data-t="676">I'm sliding that window over a longer period of time.</a> <a data-t="679">So that player will have more points in the data set.</a> <a data-t="683">So to limit their influence, I downsampled players that had longer careers.</a> <a data-t="688">And then to focus on hitters, I required the player to have at least 85 plate appearances.</a></p>

<p><a data-t="693">Now another issue is that, so we also want to avoid data leakage.</a> <a data-t="700">And so think about the fact that we randomly are splitting the data into training and testing and validation sets.</a> <a data-t="705">It's possible to have the same player represented in training and testing and validation if we're not careful, right?</a> <a data-t="714">And so in order to avoid that, one thing we could do is to just randomly split on the players.</a> <a data-t="719">So you have the training players, the validation players, the testing players, and then you can compute the rolling windows for those individual players all across the data splits.</a></p>
