<h3>FunSpotter functions overview</h3>

<p><a data-t="223">So next I want to talk a little bit about each of the different kind of groups of functions that exist within Funspotter.</a> <a data-t="232">So one set are these list files functions that essentially take in some location and return all the different files in that location.</a> <a data-t="238">Then there's a set of functions that spot things.</a> <a data-t="242">So in this case we're talking about spotting functions.</a> <a data-t="245">And then I have a variety of helpers that either manipulate the output of these objects or are helpful for plotting.</a> <a data-t="249">And then there's also a bunch of unexplored functions that I'm not really going to get into today but can help with certain situations you may run</a> <a data-t="253">into if you have like different dependency structures than just kind of like the standard setup.</a></p>

<p><a data-t="259">Okay. So now I'm going to walk through this step by step just so you can see the details of how this works.</a> <a data-t="264">So after running list files GitHub repo what is returned are the file paths for each of your R and R Markdown files</a> <a data-t="270">within that repo and then also the URLs to those.</a> <a data-t="276">Next we pipe this into spot funds files which is going to for each of those individual file paths parse out the actual functions</a> <a data-t="281">and packages within them.</a> <a data-t="285">So if we just hone in on an individual element within that spotted column you'll see this is where our actual function packages that are used within that file exist.</a> <a data-t="293">And then that next step the unnest result is just going to take each of these individual data frames containing the function packages and put them into a more simple data</a> <a data-t="298">frame structure that you can use as a reference table for yourself.</a></p>

<p><a data-t="303">And where most of the work that's going on here is really in this this like spotting step.</a> <a data-t="307">And one thing about R that adds a little bit of complexity here is that when you do a library call you're attaching all the functions within that package.</a> <a data-t="318">So what fun spotter does is essentially try and recreate what it thinks that search space is going to look like.</a> <a data-t="323">It takes a variety of steps to do this so that when it identifies a function it can say I think this is the package this function is actually coming from.</a> <a data-t="332">I'll note too that this takes a few seconds to do for each file.</a> <a data-t="335">So while you can do it for a few for hundreds of files if you try and do it for millions this isn't going to work it's going to take too long.</a> <a data-t="344">If you have any ideas and suggestions for how to do this faster definitely feel free to open a GitHub issue for me if you happen to be an expert in this space.</a></p>

<p><a data-t="348">And I'll also note that if the if you don't have the package installed locally fun spotter is not going to know which package that particular function came from is just going to say unknown.</a> <a data-t="357">So you if you and if you don't want to actually install all of those packages that you may have for some you know set of files that you're running fun spotter through if you don't want</a> <a data-t="365">to install those across all your projects you may want to start off by using something like our end so that you can keep you know the package you're installing for this specific to that project.</a> <a data-t="378">And again this process of recreating the search space is currently set up to be used on like self-contained files and by that I just mean picture like an R</a> <a data-t="382">file and R markdown file that has a bunch of library calls at the top.</a> <a data-t="385">So it's going to work on things like repos for blogs or books and things that more along those lines</a> <a data-t="391">it's not going to work so well on things like targets workflows or you know </a><a href="/software/shiny-r/">shiny</a><a data-t="391"> apps or other types of projects that have more sophisticated dependency structures potentially.</a></p>
