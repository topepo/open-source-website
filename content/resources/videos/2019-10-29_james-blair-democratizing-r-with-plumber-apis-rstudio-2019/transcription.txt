<h3>Plumber annotations and OpenAPI</h3>

<p><a data-t="104">To look at the Plumber specific pieces as we walk through this, Plumber uses specialized</a> <a data-t="108">annotations, and these are just special comments written in the regular comment syntax in R</a> <a data-t="113">with the hash symbol, and then that's followed by a little asterisk.</a> <a data-t="117">And this allows us to annotate our document or our source script in a way that Plumber</a> <a data-t="120">can then understand and interpret our just standard R code and turn it into these APIs.</a> <a data-t="126">So here, this initial, these few initial comments here provide an API name and description from</a> <a data-t="131">an API.</a></p>

<p><a data-t="132">The next step that I have is I have this filter that's defined, and a filter is used when</a> <a data-t="143">I have some sort of logic that I want to take place on an incoming request before it's served</a> <a data-t="148">to an endpoint.</a> <a data-t="149">And so this is where I could do something like if I want to provide some sort of authentication</a> <a data-t="153">on incoming requests, I could use a filter to do that.</a> <a data-t="156">Here this example filter simply takes some information from the incoming request, such</a> <a data-t="160">as where this is coming from, the user ID associated with it, a timestamp, and prints</a> <a data-t="165">that out so that I have a nice log available to me of who's interacting with my API as</a> <a data-t="170">well as the frequency that this API is being interacted with.</a></p>

<p><a data-t="179">Finally we have the function that I started with when we demonstrated this R code, and</a> <a data-t="183">I just have a few different things that I have that I've annotated on this function.</a> <a data-t="187">I've given it a name and a little description, and then I've defined the endpoint that this</a> <a data-t="191">function should respond to.</a> <a data-t="192">So again, I apologize if it's a little small, but I've defined that this function will respond</a> <a data-t="197">or will be called when the predict endpoint is accessed with a post request.</a></p>

<p><a data-t="242">If we look at a comparison between the code that we've written and this file and user</a> <a data-t="246">interface that's been generated, we can start to see how these two things are connected.</a> <a data-t="250">Here I see that my title has been copied from my source code into this user interface.</a> <a data-t="255">I can see that the description matches the description that I defined in my R script.</a> <a data-t="260">And finally, I have this endpoint that's been defined, this predict endpoint that shows</a> <a data-t="264">that it's going to respond to post requests, and it shows the description that I provided</a> <a data-t="269">in my source code.</a></p>

<p><a data-t="272">This sort of magical experience is made possible through the use of OpenAPI.</a> <a data-t="276">Now what OpenAPI is, it was formerly referred to as Swagger.</a> <a data-t="279">This is a specification, so it's a set of guidelines that defines a way that you can</a> <a data-t="283">go through and essentially describe an entire API.</a> <a data-t="287">The endpoints, the attributes of those endpoints, everything from kind of the top to bottom</a> <a data-t="291">of your API can be described using this particular specification.</a> <a data-t="295">There's a few different ways this can happen.</a> <a data-t="297">You can write like a YAML file, which is the example that I have pulled up on the screen.</a> <a data-t="300">You can also create a JSON file that is then ingested and interpreted by OpenAPI to provide</a> <a data-t="306">things like the user interface we were just looking at.</a></p>

<p><a data-t="311">Plumber interacts with OpenAPI in a couple of different ways.</a> <a data-t="313">We've already been exposed to one, and that is through these specialized comments where</a> <a data-t="316">we have the hash symbol followed by an asterisk, and then typically some sort of decorator</a> <a data-t="320">like an at symbol.</a> <a data-t="321">And what happens is Plumber will parse the file, identify these special comments, and</a> <a data-t="326">then insert those into a named list that is finally processed into a JSON file that OpenAPI</a> <a data-t="331">will read and provide us with the user interface.</a> <a data-t="334">Now this functionality has been in Plumber since essentially day one.</a> <a data-t="338">And one of the nice advantages here is that my annotations, my documentation, my description</a> <a data-t="343">is all closely connected to my code.</a> <a data-t="345">They're right next to one another.</a></p>

<p><a data-t="347">However, there are some limitations here, and that is the entire suite of options that</a> <a data-t="352">are available through OpenAPI is very broad, right?</a> <a data-t="356">And so it's not feasible to consider creating a specialized Plumber annotation that provides</a> <a data-t="360">access to every single feature available using the OpenAPI specification.</a> <a data-t="366">And I'm happy to announce that given some recent efforts that have happened with the</a> <a data-t="368">Plumber package, we now have the opportunity to update and adjust the full OpenAPI specification</a> <a data-t="375">at runtime of the Plumber API.</a></p>

<p><a data-t="378">What this looks like is over here on the right-hand side, I have an entry point file that Plumber</a> <a data-t="383">will read, and in this entry point file, I define at runtime a swagger function that</a> <a data-t="389">essentially will take in a few different arguments, and all it needs to return is a named list</a> <a data-t="393">that can be appropriately parsed into JSON that OpenAPI can interpret.</a> <a data-t="397">So I can update anything that I've written, or I can add additional attributes to my API.</a> <a data-t="402">In fact, if I wanted to, I could overwrite everything that Plumber has already automatically</a> <a data-t="406">done and just create my own specification file for my API to be ingested and then rendered</a> <a data-t="410">for the end user.</a></p>
