<p><a data-t="316">And then we get the result of that computation, and then it kicks back to our local and assigns</a> <a data-t="323">that to our reactive value.</a> <a data-t="326">Okay.</a> <a data-t="327">So this is good.</a> <a data-t="328">You'll notice that we assign to a temporary value here the value of our input, and we</a> <a data-t="334">have to do that because the process that you kick off as a child can have no interaction,</a> <a data-t="343">doesn't have access to the reactive context.</a> <a data-t="345">It's an entirely separate process, and these two can't talk to each other.</a> <a data-t="350">And so you need to assign that to a temporary value which is then shipped off to that process</a> <a data-t="358">for utilization.</a></p>

<p><a data-t="379">So with async, now we hit run.</a> <a data-t="381">Oh, God, how long is this going to take?</a> <a data-t="386">I don't know, because we don't have a progress bar anymore, and the reason why we don't have</a> <a data-t="390">a progress bar is because we don't have access to our reactive context.</a> <a data-t="394">There's no communication that can happen.</a> <a data-t="402">And so we finally get our hello world back, but we've had to drop that progress bar.</a></p>

<h3>Introducing the IPC package</h3>

<p><a data-t="429">So that's where the IPC package comes along.</a> <a data-t="432">IPC is a package that's devoted to providing mechanism to communicate back and forth between</a> <a data-t="443">processes and to ship computations and expressions between these threads for execution.</a></p>

<p><a data-t="457">Now here's just a simple code.</a> <a data-t="457">Here we create a queue.</a> <a data-t="458">Now these messaging queues can be backed by some sort of external database that both of</a> <a data-t="463">the processes have access to.</a> <a data-t="464">By default that is done via temporary files using the TXT queue package.</a> <a data-t="473">But other backings are available.</a> <a data-t="474">For instance, Redis is also an implemented option.</a></p>

<p><a data-t="479">Now each queue has a consumer that contains methods for processing things from the queue</a> <a data-t="485">and a producer to fire off messages to the queue.</a> <a data-t="489">So here we have a future.</a> <a data-t="490">We're going to create a future.</a> <a data-t="491">It's going to just basically run forever, periodically having the consumer run consume.</a> <a data-t="498">And then in our main thread we're going to sell the producer to fire off a message to</a> <a data-t="504">evaluate the expression stop that child in the future.</a> <a data-t="512">So when that consume is called and it reads the message, it will process it and execute</a> <a data-t="519">the expression stop that child.</a> <a data-t="523">And then the future will throw an error and the future will stop.</a></p>

<p><a data-t="531">So important methods.</a> <a data-t="533">On the consumer side you've got consume.</a> <a data-t="535">This will process all of the messages on the queues and it will evaluate those messages</a> <a data-t="542">in the environment which you provide it.</a> <a data-t="545">By default that's the environment in which it was called.</a> <a data-t="548">It's similar except it just runs consume periodically as long as R is idle.</a></p>

<p><a data-t="566">Also we have the queue comes shipped with the ability to handle a couple of different</a> <a data-t="574">types of messages, but you can also add your own custom handlers to handle particular signals</a> <a data-t="581">and messages that you want to send.</a> <a data-t="584">On the producer side you fire off signals using the fired method and there's a couple</a> <a data-t="589">of special ones that are useful.</a> <a data-t="594">You can fire eval will signal to evaluate an expression, fire call will signal to call</a> <a data-t="599">a function with arguments dot, dot, dot, or fire do call will fire the function called</a> <a data-t="604">name with parameters list param.</a></p>

<p><a data-t="610">So basically you can have your own custom handlers, but because evaluation and calls</a> <a data-t="614">are so flexible, you can basically do any sort of computational message that you want</a> <a data-t="618">to send via fire call, fire eval or fire call.</a></p>

<p><a data-t="621">So here's an example.</a> <a data-t="622">Before we saw sending a message from the parent to the future.</a> <a data-t="626">Here we're going to send from the future to the child.</a> <a data-t="629">So we create a queue and then in the future we're going to fire an eval, which is going</a> <a data-t="636">to print index.</a> <a data-t="638">And then what we're going to do is over here, so when you fire eval, it will substitute</a> <a data-t="644">the values in the second argument into the expression.</a> <a data-t="648">So we're going to substitute I in for index here and so I is going from 1 to 100.</a> <a data-t="654">So this will basically count from 1 to 100 and it will say, hey, go print this out and</a> <a data-t="660">here in the parent thread.</a></p>

<p><a data-t="676">And so that's pretty easy.</a> <a data-t="679">So you can just run that.</a> <a data-t="686">And so basically while R is idle in the thread, it will just keep on printing out these numbers</a> <a data-t="693">until it gets to 100 and that's all coming from the other process.</a></p>
