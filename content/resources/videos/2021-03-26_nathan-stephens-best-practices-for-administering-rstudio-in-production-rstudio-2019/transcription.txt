<h3>Five best practices</h3>

<p><a data-t="334">So I want to share five best practices for administering RStudio in production.</a></p>

<p><a data-t="342">Number one, keep your system up to date.</a> <a data-t="345">So you're going to want modern tools for your data science platform.</a> <a data-t="349">That means modern operating systems and modern browsers.</a> <a data-t="354">You'll be using the cutting edge tools, you'll be doing cutting edge analysis, you'll be hiring people with very important skills, and you're going to want modern tools for all of those things.</a> <a data-t="369">You'll need a C++11 compiler so that you can build the R packages on your system, and you'll want to think about your internet access, and that's because all of your packages are coming from the internet.</a> <a data-t="393">If you don't have internet access in your server environment, a lot of organizations don't, then it's very important that you talk to RStudio because all of our products are designed to run in an offline mode.</a></p>

<p><a data-t="414">Number two, support multiple versions of R.</a> <a data-t="418">Why do you want to support multiple versions of R?</a> <a data-t="420">Well, because you want to be able to manage upgrades.</a> <a data-t="423">You want to be able to test code on a variety of R versions, and you want to support projects that depend on various versions of R.</a> <a data-t="430">All of our products support multiple versions of R.</a> <a data-t="437">We recommend that you upgrade at least yearly and that you use a modern version of R, so 3.1 or greater.</a> <a data-t="442">And then finally, this is very important that we recommend that you build R from source.</a> <a data-t="447">Building R from source allows you to install multiple versions of R side by side.</a> <a data-t="452">It's not hard to do.</a> <a data-t="453">It follows the standard </a><a href="/software/config/">config</a><a data-t="453"> make install path.</a> <a data-t="459">Most organizations have a process for building software from source already in place, and we have instructions for how to do that.</a></p>

<p><a data-t="425">Number three, organize your R packages.</a> <a data-t="431">So R packages rule the nest.</a> <a data-t="433">Packages will drive your R version, your Linux dependencies, and even your operating system.</a> <a data-t="437">Data scientists will want to access their most beloved packages.</a> <a data-t="442">So say you have a data scientist who wants the most recent modeling package, and they try to install it, and it fails because it needs a new version of R.</a> <a data-t="448">So you want to upgrade that version of R, and then you find that you're going to need some Linux binaries, but those Linux binaries aren't really supported by your operating system.</a> <a data-t="456">So suddenly you're upgrading your whole operating system to support that package of R.</a> <a data-t="460">That is not an unrealistic scenario.</a> <a data-t="462">That's one reason we say keep your platform modern and up to date.</a></p>

<p><a data-t="465">So it turns out that managing packages for a single user is quite easy, like you do on your desktop, but managing packages for an entire platform is really hard.</a> <a data-t="476">And we should know, RStudio knows a lot about managing packages.</a> <a data-t="480">So we created a package manager.</a> <a data-t="482">It's called the RStudio Package Manager, and it solves several problems.</a> <a data-t="487">One problem is when your environment is disconnected from the internet or you're in an </a><a href="/software/air/">air</a><a data-t="487">-gapped environment, you need a way to get packages into that environment, and Package Manager can help with that.</a> <a data-t="497">Another one is being able to serve consistent environments to Docker containers.</a> <a data-t="502">So if your Docker container is running one environment, it's probably not the same environment that you're building on, but the Package Manager can help standardize those environments.</a> <a data-t="512">There's also security control for package repositories.</a> <a data-t="520">A lot of organizations like to curate the packages that gets delivered to their organization, and then also Package Manager helps with sharing internal packages.</a></p>

<h3>Best practices continued</h3>

<p><a data-t="703">All right, so number four, use root privileges.</a> <a data-t="713">So the group in your organization that installs, configures, and manages R in RStudio will need root privileges.</a> <a data-t="720">So RStudio installs require root privileges, and RStudio products also run as root.</a> <a data-t="725">RStudio Server Pro runs as the root user in order to create a new R session on behalf of its users, and RStudio Connect runs as a root user in order to isolate applications and processes.</a> <a data-t="738">I'll also add that system-wide installations of R on Linux oftentimes involve root as well.</a></p>

<p><a data-t="745">So this is very important that the people that have the permissions in your system to install R are able to do so effectively.</a> <a data-t="754">So that means if that person is you, then you need to know how to install the products and the software sufficiently on that system.</a> <a data-t="765">If that person is someone else, then you need to help that person understand what this process involves.</a> <a data-t="771">That means installing R, the R packages, and the products.</a></p>

<p><a data-t="783">Okay, and the last one, five, is securely manage your users.</a> <a data-t="788">There's two types of users.</a> <a data-t="789">There's R programmers and there's end users, and the R programmers will need access to R file shares, databases, and probably many other sensitive systems.</a> <a data-t="800">R processes for the R programmers run as the user under a local account.</a> <a data-t="809">So users need local accounts on the system to be running these types of analyses.</a> <a data-t="815">End users, on the other hand, are people that may or may not know R and just consume apps and reports.</a> <a data-t="821">For these, the R processes typically run under a service account.</a> <a data-t="827">And if you look at our two products, you'll see that RStudio Server Pro is serving R programmers and RStudio Connect is serving end users.</a></p>

<p><a data-t="846">So how do you hook those up?</a> <a data-t="849">Well, that's where authentication comes in.</a> <a data-t="851">Your organization probably has strong opinions on how to authenticate users.</a> <a data-t="855">This space is only getting more fragmented, not less.</a> <a data-t="858">So if you look at like a lot of on-prem solutions today, we'll have like LDAP, Active Directory, and then a lot of the web technologies, the cloud technologies have things like Okta, Duo, and Auth, and Ping, and Shibboleth, and many, many more.</a> <a data-t="872">So we want to support as many systems as possible to work with your authentication system, and we're working hard to do that.</a> <a data-t="883">And then you will go ahead and configure RStudio with whatever authentication system that your organization has.</a></p>

<p><a data-t="891">Now, if our products don't support your particular authentication system, then we recommend using Proxied Auth.</a> <a data-t="897">In Proxied Auth, users don't log in through RStudio, they actually log through a proxy that you set up, and all this is in the guides.</a> <a data-t="903">The supported Auth methods for RStudio Server Pro are PAM, pretty much, and RStudio Connect.</a> <a data-t="909">The supported Auth methods are typically LDAP, Active Directory, or PAM, and SAML, which is coming up soon.</a> <a data-t="915">So we're working on SAML.</a> <a data-t="921">Finally, I want to mention Kerberos, which is an authentication.</a> <a data-t="925">It's access control, but use Kerberos to connect to databases, and those are supported by our drivers.</a> <a data-t="931">Our professional drivers allow you to connect to databases using </a><a href="/software/odbc/">ODBC</a><a data-t="931">.</a></p>

<h3>Summary of best practices</h3>

<p><a data-t="934">All right, so in summary, here's my recommendations.</a> <a data-t="941">Number one, keep your operating systems and browsers up to date.</a> <a data-t="947">Number two, plan to support multiple versions of R by building R from source.</a> <a data-t="951">Number three, organize your R packages for reliability and consistency.</a> <a data-t="955">Number four, use root privileges to install and run RStudio products.</a> <a data-t="960">And number five, securely manage your R programmers and end users.</a> <a data-t="964">So if you stay to these five things as you architect and execute your R platform, you will have a lot more success.</a></p>
