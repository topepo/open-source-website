<h3>The README framework for community building</h3>

<p><a data-t="75">But what if we approached building the community like the README file for our</a> <a data-t="81">project or package? README file is very essential, and I'm sure a lot of you in this room either</a> <a data-t="87">interacted with the README file or created one. This is the first file that programmers</a> <a data-t="93">read when they actually approach your project. So let's look at the typical structure of</a> <a data-t="99">the README file together, and I'll let you know how this translates into the community</a> <a data-t="105">building terms.</a></p>

<p><a data-t="105">When you open the README file, the first thing you look for is the installation</a> <a data-t="111">instructions, how to install the package. Is it on CRAN? Is it on GitHub? Are there</a> <a data-t="116">any dependencies you have to take into account? This is the foundational knowledge you need</a> <a data-t="122">to obtain in order to move forward with installing the package. You can't move forward without</a> <a data-t="128">that. And in the community building terms, this translates into establishing the foundation</a> <a data-t="135">for your community. And to be successful in it, you need to assemble a dedicated team</a> <a data-t="141">of colleagues who will take on the community building project and will make it their priority.</a></p>

<h3>Establishing the foundation: the R-SWAT team</h3>

<p><a data-t="147">And we do have such team at Pfizer, and I'm very lucky to be a part of it. And at Pfizer,</a> <a data-t="153">we have the R-SWAT team. This is our SWAT team. It's part of a bigger RCOE team. And</a> <a data-t="162">the SWAT stands for scientific workflows and analytic tools. We're very proud of this acronym.</a> <a data-t="170">We are the internal consultancy group for those parts of organizations that don't have</a> <a data-t="176">experts on hand. So we are the art experts. And on top of that, we are the professional</a> <a data-t="182">community builders. Building the community is not our side project. This is one of our</a> <a data-t="189">priorities. So we have the community building initiatives in our semester goals, and we</a> <a data-t="195">treat it as a project deliverables. Because if we won't do that, then there always will</a> <a data-t="201">be the project that's going to supersede community building initiatives, and the process</a> <a data-t="207">would be slow.</a></p>

<p><a data-t="211">However, it's very important to acknowledge that we won't be able to make</a> <a data-t="217">the progress that we made without the help of our IT experts. Because we're our community</a> <a data-t="223">builders, and we need our infrastructure to achieve our goals and the goals of our colleagues.</a> <a data-t="229">At Pfizer, we use a suite of professional tools to support our projects, and our IT</a> <a data-t="235">experts support all of them. And the SWAT team works closely with IT to define the best</a> <a data-t="241">practices. We also have regular check-ins to address any questions or concerns we as</a> <a data-t="247">a team have and our colleagues have. So we're this connector between our programmers and</a> <a data-t="253">the IT department. And the relationships we established with the IT colleagues help us</a> <a data-t="259">to support our community.</a></p>

<p><a data-t="259">Having tools is great, but we all know that we need a proper</a> <a data-t="265">documentation for those tools. And we needed a single source of truth and a single source</a> <a data-t="271">of documentation where our colleagues can access all the documentation regardless of</a> <a data-t="277">where they are in the world at any given time. So for us, the R at Pfizer conference became</a> <a data-t="283">this single source of truth. So the conference website is an internal website where we host</a> <a data-t="289">all our documentation, how-to articles. We promote best practices. And this is also the</a> <a data-t="295">place where our IT colleagues post all their best practices, guides, and information.</a></p>

<p><a data-t="300">And the rule of thumb is, if we get the same question from multiple colleagues a few times,</a> <a data-t="306">we need to write a knowledge article on it and post it to the conference website. With</a> <a data-t="311">this, we accomplish a few things. So first of all, we're saving ourselves time in the</a> <a data-t="317">long run because we're only publishing the article once. Of course, we need to maintain</a> <a data-t="322">it, but the bulk of work done just once. And then the second thing is that everyone will</a> <a data-t="327">get the same uniform advice to the question they ask, the advice that we as a team agreed</a> <a data-t="333">on if this is consolidated in the knowledge article.</a></p>

<h3>Usage: training and keeping colleagues updated</h3>

<p><a data-t="338">Perfect. We went through the first part of our readme, the foundation part. So let's</a> <a data-t="344">go back to the typical package readme and see what the next section is. And the next</a> <a data-t="350">section is usage. So you install the package, and now it's time to dive in and learn whether</a> <a data-t="356">the key functionality of your package and learn whether the key functions of the package</a> <a data-t="362">are. And in the community building terms, this translates into creating, delivering,</a> <a data-t="368">and maintaining proper training for your colleagues. Everyone's learning approach is different,</a> <a data-t="373">and everybody can be in different places when it comes to their R journey. So it was very</a> <a data-t="379">important for us to deliver training that will fit everyone.</a></p>

<p><a data-t="387">We all learn programming by doing. Therefore, we create this series of hands-on mini projects</a> <a data-t="392">where our colleagues can get their hands dirty. So far, we have 13 R projects. Those are the</a> <a data-t="400">R Markdown files, and each R Markdown file dedicated to a specific topic can be how to</a> <a data-t="406">create a function or visualize the data. And all of the exercises are relevant to clinical</a> <a data-t="413">programming, so they're relevant to our colleagues' day-to-day job. Trainees can access those</a> <a data-t="418">exercises at any given time, and they can perform them and complete the challenge at</a> <a data-t="423">the end at their own pace.</a></p>

<p><a data-t="423">It's also important to learn from experts.</a> <a data-t="429">Therefore, we make an effort to, once in a while, invite the expert from POSIT or within</a> <a data-t="436">our own Pfizer organization, and they can come in and give a training on a dedicated</a> <a data-t="440">topic. Again, it can be a workflow or a package. And then we record all those sessions. Those</a> <a data-t="446">are typically one- to two-hour sessions, and then we post it to Microsoft Teams channel.</a> <a data-t="452">And we have a bunch of recordings, so we divided them into sections. For example, master the</a> <a href="/software/tidyverse/">tidyverse</a><a data-t="458"> training sessions are the videos where each recording dedicated to, for example,</a> <a data-t="465">the tidyverse package. And the R for beginners, those are the videos with the beginner-friendly</a> <a data-t="472">topics. They're very short videos, actually.</a></p>

<p><a data-t="473">However, we all know that R is moving fast,</a> <a data-t="477">so you recorded the training today, you post it, and then tomorrow they release the new</a> <a data-t="482">version of this package or the new function. So we have a couple of initiatives to keep</a> <a data-t="488">our colleagues updated. In the function of the week initiative, we highlight and we post</a> <a data-t="493">about the function we want to highlight, and we find it very useful into day-to-day</a> <a data-t="497">programmer's life. And in the package of the month initiative, every month or every other</a> <a data-t="503">month, we release a little training, the R markdown for the package of the month that</a> <a data-t="509">colleagues can download and go through.</a></p>

<p><a data-t="511">We also utilize a form of newsletters to distribute</a> <a data-t="516">knowledge. So there are two types of newsletters we distribute. Advances in our newsletter</a> <a data-t="522">is the one that highlights latest and greatest in our community and in our tools, and it's</a> <a data-t="528">also pharma-tailored, so it's very Pfizer-focused. And in external outreach newsletter, our colleagues</a> <a data-t="534">who are involved in external initiatives, like FUSE, our consortium, they highlight</a> <a data-t="540">their group's latest updates, upcoming events, and the working group progress.</a></p>

<h3>Contributing back to the community</h3>

<p><a data-t="548">Perfect. We</a> <a data-t="548">finished the second section of our readme, and now let's move on to the last and my favorite</a> <a data-t="554">part on how to contribute back. So in your typical readme package, there's always a section</a> <a data-t="560">on how to contribute back, because open source projects, they encourage collaboration and</a> <a data-t="567">they encourage innovation. And maybe you downloaded the package, you played with it a bit, and</a> <a data-t="572">then you found a bug and you want to propose the solution, or maybe you would want to request</a> <a data-t="579">the feature. And in the community-building terms, this is actually perfect as is. That</a> <a data-t="584">doesn't translate. And I want to emphasize that I notice this is a common misconception,</a> <a data-t="592">that you have to be a seasoned programmer in order to contribute back to the community.</a></p>

<p><a data-t="599">To prove that this is not true, I want to show you a quick example. This is the issues</a> <a data-t="605">tab of the GitHub page for the very well-known package, </a><a href="/software/usethis/">usethis</a><a data-t="605">. And you can see there are</a> <a data-t="612">a lot of issues filed for this package. This package is by thousands of programmers. And</a> <a data-t="617">each issue has a tag associated. And amongst those tags, you can find a good first issue.</a> <a data-t="625">This shows that even the most complex projects, that they have tasks that can be achieved</a> <a data-t="630">and that can be done by beginner programmers. So by picking up this issue, working on it,</a> <a data-t="638">and contributing the results of your work back, you're helping yourself, you're learning</a> <a data-t="642">new things, as well as helping programmers who are developing the package and programmers</a> <a data-t="647">who are going to use this package.</a></p>

<p><a data-t="652">We kept this in mind as community builders when we</a> <a data-t="657">were creating the initiatives and tools where our colleagues can contribute back. So let's</a> <a data-t="663">take a look at a few of those.</a></p>

<p><a data-t="663">First and foremost, anyone can join the R and</a> <a data-t="669">Pfizer Microsoft Teams channel, comprising over 1,100 colleagues so far. It's a great</a> <a data-t="675">resource for sharing news and answering and asking questions. And sometimes when you're</a> <a data-t="681">battling the programming issue, it might seem like you're the only one in this world facing</a> <a data-t="686">this very niche issue. But a lot of times you're not. There's always someone out there</a> <a data-t="691">who already solved this issue and has a solution for you. Or there's someone who's battling</a> <a data-t="697">the same issue and you can join forces and look for the solution together. So by asking</a> <a data-t="703">the question on Microsoft Teams channel, our colleagues are getting closer to solving the</a> <a data-t="708">issues they have. And then once they build the expertise, they can also contribute to</a> <a data-t="713">the community, give back, and assist others by answering their questions.</a></p>

<p><a data-t="721">Another great initiative I would like to highlight, one of my favorites, is the Book Club. We</a> <a data-t="727">established the Book Club earlier this year and we picked an R for Data Science second</a> <a data-t="733">edition book. And this is a great initiative for our colleagues to collaborate and put</a> <a data-t="738">the time dedicated to learning new things in their day-to-day schedule. So we initially</a> <a data-t="743">had 130 people sign up for the Book Club. And our colleagues were divided into 14 different</a> <a data-t="750">groups. And the geography for those groups spans from North America through Europe to</a> <a data-t="755">Asia. And we divided those groups so that it's the same time zone and also we have different</a> <a data-t="761">expertise level in the same group for our colleagues. And each group has a dedicated</a> <a data-t="766">group lead.</a></p>

<p><a data-t="766">And throughout the time of like working on this initiative and leading one</a> <a data-t="771">of the groups, we learned that one of the biggest obstacle for this particular initiative</a> <a data-t="777">is the day job commitments. As the day job commitments increase, it becomes harder for</a> <a data-t="782">colleagues to join the session and read the book on a dedicated scheduled basis. However,</a> <a data-t="789">for those teams that became tight-knit and for those colleagues who are very consistent</a> <a data-t="795">in reading the book and joining the session, even if they haven't had a chance to read</a> <a data-t="801">the chapter that week, we received incredibly positive feedback and willingness to join</a> <a data-t="807">Book Clubs in the future.</a></p>

<p><a data-t="811">The Community of Practice event is a monthly</a> <a data-t="815">meeting where we have two 20-minute presentations followed by 10-minute questions. And this</a> <a data-t="821">is not an exclusive event. Anyone can get on the schedule and showcase their art projects,</a> <a data-t="827">lessons learned, and ask for the feedback. The presentations vary in scope to keep all</a> <a data-t="834">colleagues engaged. And this is a formal event. So the presentations are recorded and they're</a> <a data-t="840">prepared in advance. So this is one of the ways you can contribute back.</a></p>

<p><a data-t="852">However, there</a> <a data-t="858">is another one which is informal event. We adopted the Data Science Hangout event led</a> <a data-t="865">by Rachel Dempsey at Posit. So if you haven't heard of the Data Science Hangout, it happens</a> <a data-t="871">every Thursday at noon Eastern Standard Time. And anyone from the data science community</a> <a data-t="878">can join. So please join if you haven't already. So similarly, our internal Art Advisor Hangout</a> <a data-t="884">happens every other Wednesday. And anyone from Pfizer interested in our programming</a> <a data-t="890">can join. During this event, we invite a featured colleague. It's our guest. And we ask our</a> <a data-t="896">colleague different questions. For example, their art journey, what excites them about</a> <a data-t="902">art, what projects they're working on. And this is the informal event. So all the questions</a> <a data-t="908">are audience-led. And this is also not recorded, just in case you want to ask interesting questions.</a></p>
