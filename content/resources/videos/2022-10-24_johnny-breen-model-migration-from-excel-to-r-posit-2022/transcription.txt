<h3>Tip 1: Don't copy or replicate Excel</h3>

<p><a data-t="417">And so what's the moral of this story?</a> <a data-t="418">Why am I here?</a> <a data-t="421">What kind of advice can I give you?</a> <a data-t="423">Let's kind of analyze what went wrong here.</a> <a data-t="431">So the first big tip I have, and it's really the biggest fundamental problem that I encountered here is, you don't really want to copy or replicate Excel.</a> <a data-t="436">Okay?</a></p>

<p><a data-t="441">Now, like, this is an obvious point to make to people if you're a seasoned programmer,</a> <a data-t="446">but let's take a step back and put yourselves in the shoes of somebody who hasn't really written much code in their life.</a> <a data-t="454">So somebody who's only ever used Excel before, for somebody like that, everything looks a bit like this.</a> <a data-t="459">And that's a spreadsheet.</a> <a data-t="465">But if you think about it visually, it's a bit like a data frame in R. And that's not a mistake.</a> <a data-t="470">But Excel models a lot of things using data frames.</a></p>

<p><a data-t="476">And whilst this isn't a problem in itself, you end up with a lot of Excel models which have a few questionable design choices.</a> <a data-t="484">So the first issue you encounter typically in the wild is Excel models which start out with, you know, a few number of tabs of slowly increasing levels of information, and you think we're okay, we're stable, and then by the end of it, you've got hundreds of tabs.</a> <a data-t="516">And if you're the creator of this model, you typically end up having a nervous sweat any time anybody else needs to use it.</a> <a data-t="522">It's a very precarious situation to be in.</a></p>

<p><a data-t="524">And all of this comes about because the most natural data structure to use when you're using Excel is a data frame object.</a> <a data-t="534">So this is one issue that you encounter sometimes.</a> <a data-t="538">The second is a lot of these models, you can store data just in a spreadsheet or in a data frame.</a> <a data-t="545">So you don't really need a database, right?</a> <a data-t="548">Well, not quite.</a> <a data-t="556">I mean, a lot of these models probably should be supported by a database, but they're not.</a> <a data-t="557">And that's nobody's fault, right?</a> <a data-t="562">Like a lot of analysts who have built these models don't necessarily have the knowledge or the skills to even think about databases, let alone implement a database.</a></p>

<p><a data-t="575">Now why am I mentioning all this?</a> <a data-t="577">These are issues with Excel models.</a> <a data-t="586">But the point is, if you're migrating an existing model into an R environment, you definitely don't want to be copying over these design choices.</a> <a data-t="597">If you do this, then you're only going to reintroduce the problems in a different context.</a></p>

<h3>Tip 2: Structure your model</h3>

<p><a data-t="600">So what's my next tip off the back of this?</a> <a data-t="606">So what you want to do, instead of kind of copying what's going on in the existing platform, which is Excel in this case, you want to think about the structure of this model.</a> <a data-t="617">So what you want to do is abstract away the implementation that you see in Excel and think about, well, what is it actually trying to do, right?</a> <a data-t="622">What are the functional requirements?</a> <a data-t="624">What is this model trying to achieve?</a></p>

<p><a data-t="630">And the first fundamental step of this process is where does my model begin and where does it end?</a> <a data-t="635">And that can be pretty tricky with some of these Excel models.</a> <a data-t="640">They are so monolithic that it's actually very hard sometimes to know where the calculations begin and where they end.</a> <a data-t="642">But this is a process that you have to go through in order to start unpicking this object and migrating it across.</a></p>

<p><a data-t="649">So this is why I have this A to B presented here.</a> <a data-t="651">So you want to kind of know, how do I get from point A to point B?</a> <a data-t="656">That's the first ingredient.</a> <a data-t="658">The second thing is, well, along that route, what are the operations that I'm applying to the data at point A to get to point B?</a> <a data-t="669">And if you're doing a migration, especially in R or any language, really, these are really important things to identify, because you'll be able to encapsulate these into functions and objects that you can use in your programming environment.</a> <a data-t="683">So this is a good kind of planning scheme to go through before you actually do anything.</a></p>

<p><a data-t="689">And the second thing kind of related to this is you want to think about, what are the data structures that I need to transport the data along this route?</a> <a data-t="699">Okay.</a> <a data-t="700">Just because the Excel model that you're looking at is doing everything in Excel and you've got multiple tabs and multiple bits of information hanging in memory, it doesn't mean that you need to do the exact same thing when you migrate this into R.</a> <a data-t="713">In fact, you could probably save a lot of the hassle, a lot of the overuse of this memory and all the other problems that come with it by just thinking about, what do I need to retain as I go from A to B?</a></p>

<p><a data-t="732">So let me kind of put this into context with an example.</a> <a data-t="737">So I mentioned earlier that I've been working on a simulation model at this time.</a> <a data-t="742">Now, not to go into too much detail, but the simulation model in each outcome, it would generate one simulation which had a set of 30 numeric outputs across 30 different risk classes.</a></p>

<p><a data-t="759">And one stage of this model is involve the task of trying to aggregate these simulations together.</a> <a data-t="766">So the author of the Excel model decided that, well, the best idea that we can use in this context is we'll take the current simulation that you can see on the screen here, and then we'll take the previous simulations, OK, the previous set of simulations, we'll add that on to the existing set, and then we'll get another set of simulations that is one row bigger than the previous.</a> <a data-t="829">And that would go on recursively until you got to, in this case, the 15 millionth iteration.</a></p>

<p><a data-t="833">So yeah, this was something you needed to execute across multiple spreadsheets, and it was a slow process.</a> <a data-t="838">But here's the problem, right?</a> <a data-t="843">If you kind of replicate this approach and you just stick it in an R script, you're going to run into trouble really quickly.</a> <a data-t="849">And it'll look something a bit like this.</a> <a data-t="855">And this is like a classic problem that you'll encounter in the R programming world, which is copying objects in loops unnecessarily.</a></p>

<p><a data-t="867">And unfortunately, like, this is one of the key design flaws that I had first implemented in my script, which meant that this thing would just eat up your memory and it would keep on going for ages.</a> <a data-t="875">You probably wouldn't get to the end of this on 15 million iterations, at least within a few days.</a> <a data-t="881">So this was a really, like, key design flaw in the way I had structured this.</a></p>

<p><a data-t="889">So how did I fix this in the end?</a> <a data-t="896">So I thought about it in a more abstract way, and I abstracted away from whatever the model was doing initially, and I thought, well, what I actually need to generate here is a list of simulation results, and that will be equal to the length of the number of simulations I want to run.</a> <a data-t="850">And for each of these entries in this list, I just need a function that will generate my simulation results across these 30 risk classes, and then I just need to run this again and again until I get to the very end.</a> <a data-t="868">And then we just need to bind these together.</a></p>

<p><a data-t="871">So this looks simple and, like, it was kind of a subtly different way of designing this, but this very different way of doing this was actually, like, a massive improvement on the existing script that I had written.</a> <a data-t="891">And it's small tweaks like that where you think about the requirements, what you're trying to do, which have quite a significant effect on the performance.</a></p>

<h3>Tip 3: Keep things simple</h3>

<p><a data-t="903">So my key tip here, as I've said already, is to structure your model.</a> <a data-t="906">And the final tip I have here is try to keep things simple.</a> <a data-t="911">So one of the problems is, and I don't know what possessed me to do this at the time, but as a budding analyst, I thought it might be a good idea to, you know, try and do a bit of parallel processing in this script.</a> <a data-t="925">And I think this was the straw that broke the camel's back in my case.</a></p>

<p><a data-t="930">Coupled with some of the other design flaws that I'd implemented, I was trying to do multi-threaded processing and it all went a bit wrong.</a> <a data-t="936">I didn't really read the documentation properly.</a> <a data-t="940">And I kind of was overcomplicating things, right?</a> <a data-t="944">Was this really a concern that I needed to address?</a> <a data-t="946">The answer is no.</a> <a data-t="952">What I really needed to do with this process was start simple and start with a sample of iterations that I'm trying to run.</a> <a data-t="960">And then I want to build this up gradually, profile what I'm doing, and work from there.</a> <a data-t="965">So doing the basics right in these cases can be really important.</a></p>
