<h3>What is Rtools?</h3>

<p><a data-t="91">So what's Rtools?</a> <a data-t="94">Well, to understand that, you need to take a step back.</a> <a data-t="101">So in order to compile things, both base R as well as packages, you need a compiler.</a> <a data-t="109">And on Linux and Mac OS, there is a native compiler on the system.</a> <a data-t="115">Linux is usually GCC.</a> <a data-t="118">And on Mac OS, we have Clang with Xcode.</a> <a data-t="123">But on Windows, there's no such thing.</a> <a data-t="136">So on Windows, we need to provide our own compiler tool chain.</a> <a data-t="142">And that is Rtools.</a></p>

<p><a data-t="145">So Rtools, if you've ever had to install a package from source or Windows, it's this thing you install separately.</a> <a data-t="156">And it helps you with building stuff on Windows.</a></p>

<p><a data-t="169">So Rtools has been around for quite a while.</a> <a data-t="171">I didn't invent this.</a> <a data-t="180">As you can see from this retro 90s homepage on CRAN, it has an archive of all of the Rtools going back like a decade or two.</a> <a data-t="202">So this is something that's been supported for quite a while.</a> <a data-t="207">And I've only taken over on maintenance last year.</a> <a data-t="211">So please don't blame me for everything that's suboptimal.</a></p>

<p><a data-t="217">So what does the workflow look like?</a> <a data-t="220">So you install this thing.</a> <a data-t="222">And then you have this installation dialog.</a> <a data-t="224">And you can choose which components you want to install.</a> <a data-t="227">So there's the tool chains, which are like the most important thing.</a> <a data-t="233">Some build utilities, mostly make and shell and tar to extract stuff, which are also needed to install R packages.</a> <a data-t="241">And then the other things are sort of optional.</a> <a data-t="244">The last thing is only needed if you want to build R itself, which you should never do.</a></p>

<p><a data-t="223">So to summarize, Rtools is needed to build base R.</a> <a data-t="255">And if you want to build R packages on Windows, which contain C or C++ or Fortran code, and this is the part that most users are familiar with.</a> <a data-t="264">But then there's also a very big third thing, which is external libraries.</a> <a data-t="270">So a lot of both base R and a lot of R packages are using external system libraries.</a> <a data-t="278">And those system libraries need to be compiled with the same tool chain and same configuration as we're using for base R.</a> <a data-t="285">And that is sort of a dark piece of the ecosystem right now.</a></p>

<p><a data-t="297">So if you've seen any of my previous talks, you've maybe seen this picture.</a> <a data-t="300">So this is what the dependency system looks like.</a> <a data-t="304">You have all of your packages, and they depend on R.</a> <a data-t="306">And R itself depends on a lot of external libraries that we all need to build with the same tool chain in order to build R for Windows.</a> <a data-t="312">And many R packages, they also depend on a bunch of other external libraries.</a> <a data-t="317">And this is a lot of work to build all these things.</a></p>

<h3>The current manual approach</h3>

<p><a data-t="305">So currently we're doing this manually.</a> <a data-t="318">So there's our WinLib organization in which we manually release binaries for the most important external C and C++ libraries, which are used by CRAN packages.</a> <a data-t="331">And there's currently, as you can see there, there's 73 libraries in there right now.</a> <a data-t="338">But that's not counting dependencies.</a> <a data-t="340">So for example, one of these libraries can depend on a dozen more other sub-libraries.</a> <a data-t="346">So there's like hundreds of external libraries that are in there.</a></p>

<p><a data-t="354">And this is kind of suboptimal.</a> <a data-t="360">Maintaining these builds is a lot of work.</a> <a data-t="365">If you want to build an R package that needs any of these libraries, it's kind of a bit of a hack, because the package author or the build server would need to manually download these libs and then put them in a special location and then mingle with the compiler and the linker flags on your system, or you can have a script in your package that automatically downloads them.</a> <a data-t="390">But it's not the way the world's supposed to work.</a></p>

<h3>Introducing Rtools 4.0</h3>

<p><a data-t="390">So for the next R tools, I wanted to sort of improve the situation, and I've looked around a bit and how this is done in other places.</a> <a data-t="402">So we want to take advantage of a proper build system, so that we can more easily build all of these external libraries using the same compiler toolchain and the same R tool system as we use to build R packages or base R itself, and then we want to automate that build process.</a> <a data-t="429">And also, I would like to open and collaboratively maintain these package build scripts, so that it's more easy for other people to contribute a build script of a new external package if they want to use it in their R package.</a></p>

<p><a data-t="451">So that is sort of the goal of R tools 4.0.</a> <a data-t="455">And I want to emphasize again that this is something that we're currently beta testing.</a> <a data-t="459">It's not sure at all if and when it's going to ship, but hopefully it will, maybe for the upcoming release or otherwise the one after that.</a></p>

<p><a data-t="472">So you can already test it.</a> <a data-t="473">If you go to CRAN on Google R tools 4.0, there's a special page, and there you can install R tools 4.0 for your system, and then there's also a special version of R, which I call R testing, and it's like the R devel builds, if you've ever tried this, but it has been configured to use this new toolchain, so it's sort of hardcoded the path to the new toolchain, and it has a custom location where it installs the packages, so you can use this, you can install this, and you can play with the new R tools, and it won't conflict with any other package libraries or so that are using the current stable toolchain.</a></p>

<p><a data-t="512">So what's the experience like?</a> <a data-t="513">You install this thing, and then suddenly in your start menu there's this new thing, which wasn't there before, because before R tools was just sort of a bundle that you install over in system, and now there's actually a shell, and there's a thing, so you're like, what if I click on that?</a> <a data-t="547">So if you click on that, it opens a terminal window, so R tools now actually includes a full terminal with Bash, a shell, and all of the tools that you're used to, or probably not if you're on Windows, but it's a full Unix-like environment based on MSYS2 and Sequin, which has all of the build tools that you need to build all of these open source libraries.</a></p>

<p><a data-t="579">So what's in the tin?</a> <a data-t="580">Well, the most important thing is there's a nice terminal, and there's the shell and everything that comes with that, Bash, and then there's all of these utilities which are part of Sequin, like Make and Set and Perl and whatnot.</a> <a data-t="597">And of course there's the compilers, there's the MinGW compilers, one for 32-bit and one for 64-bit Windows.</a> <a data-t="603">But the most important thing in new R tools is there's a package manager, and there's a real package manager to build and install external libraries, just like apt on Debian or yum on Enterprise Linux, or homebrew on OSX.</a></p>

<h3>Automated builds and deployment</h3>

<p><a data-t="895">All right, so that is how you would do it manually, but of course you want to automate it, so if you send a pull request to this library, or in my case, if you push straight to the master branch, AppVeyor will start building these things, and they look like this, and you can use, you can look at, in the log file of the AppVeyor, which are the latest things that you've built, and this is sort of the most beautiful part of the system, so it automatically deploys, so if you look in the AppVeyor log file, towards the end, and if it's in a pull request or a branch, it will then just say it was complete, but if you're in the master branch, it will then automatically deploy everything to Bintray, so all of these, all of the binaries for these Windows packages, they are hosted on Bintray, and they are automatically deployed once the build is successful from AppVeyor, so there's nothing that I have to do other than merging the pull request to actually release these libraries, right, so once the build is complete, at the bottom here you see it's, AppVeyor starts uploading the binaries to Bintray, and it's sort of recalculating the index file, and then you can see it on this URL, so this is the URL where all of the binary packages are stored, and this is the URL that then Pacman, in our tools, uses to grab the packages, so it's live immediately, there's no manual step anymore, you know, you send the pull request, AppVeyor builds, if it succeeds, it deploys on Bintray, and then the next time any user would do Pacman as update, it starts pulling in these new libraries, so it sort of ties up the circle really nicely.</a></p>

<p><a data-t="1026">Yeah, so I hope that this will make the maintenance much easier, and I should emphasize that once you have installed one of these system libraries using Pacman, our tools will automatically find it when you build an R package that needs a library, so there's no need to set special compiler flags or special linker flags to point to a particular location, because Pacman installs it exactly where the linker expects to find it, and where the compiler expects to find it, right, so it's going to feel much more like apt, or yum, or brew, where if you need a system dependency, you do the Pacman install, and the system dependency is there, and then your R package should just build without any further hacking.</a></p>
