<h3>How plumber handles requests</h3>

<p><a data-t="57">Any questions?</a> <a data-t="59">The ID will represent the order in which the request was sent, so if ID 1 will be sent</a> <a data-t="67">before ID 2.</a> <a data-t="69">Let's see how plumber will handle these requests.</a></p>

<p><a data-t="73">In the receive stage, the request will be unpacked and be prepped for processing.</a> <a data-t="79">In the processing stage, we will execute the user function, your function, such as the</a> <a data-t="84">fast calculation or the slow calculation.</a> <a data-t="87">And then finally, once they are done, we will bundle them up in the respond stage.</a> <a data-t="94">The receive and respond stage must be handled by plumber, because that is very specific</a> <a data-t="99">plumber logic.</a></p>

<p><a data-t="102">So let's look at an example where we have four requests that arrive at roughly the same</a> <a data-t="106">time.</a> <a data-t="107">In order, it'll go a fast, then a slow, then a slow, and then a fast.</a> <a data-t="113">Let's see how these execute.</a></p>

<p><a data-t="117">Fast comes in, is processed really quickly, responded, and then done.</a> <a data-t="123">Slow comes in, is received, processed, but processing takes 10 seconds.</a> <a data-t="129">It is a long time, a very long time, but that's okay.</a> <a data-t="134">That's how long that route takes.</a> <a data-t="136">And then we'll finally respond.</a></p>

<p><a data-t="138">The second slow route has been waiting patiently, but will also take another 10 seconds to calculate.</a> <a data-t="146">Little long, but it's okay.</a> <a data-t="148">That's how long it takes.</a> <a data-t="150">And then finally, it will respond.</a> <a data-t="153">And the fast route has been waiting patiently, processes really quickly, and then responds</a> <a data-t="158">to its original request.</a></p>

<p><a data-t="161">So in total, this will take about 20 seconds.</a> <a data-t="168">The first fast route will be processed roughly instantly.</a> <a data-t="173">The second request, which was a slow route, will take 10 seconds to execute, and then</a> <a data-t="178">it will respond.</a> <a data-t="180">But since R cannot process multiple things at once, slow three will have to wait 10 seconds</a> <a data-t="186">before it can be processed.</a> <a data-t="189">And finally, fast four had to wait a total of 20 seconds before it could even be ingested.</a></p>

<p><a data-t="197">In our current situation, all four requests completed in 20 seconds.</a> <a data-t="202">However, we accumulated 30 seconds of wait time.</a> <a data-t="206">Ideally, we would like this to be zero.</a></p>

<h3>Adding future for async processing</h3>

<p><a data-t="210">Let's explore how Future can solve this for us.</a> <a data-t="215">So here we have our existing plumber API code.</a> <a data-t="219">This is what it would look like if we incorporated it with Future.</a></p>

<p><a data-t="226">Future is an R package made by Henrik, which allows for parallel and distributed computing</a> <a data-t="231">in R.</a> <a data-t="232">Let's talk about the three lines of code that have been added to our plumber API.</a> <a data-t="237">The first line of code tells Future what plans should be used.</a> <a data-t="242">In this case, we're going to use a multi-session plan.</a> <a data-t="245">I'll leave it to you to figure out which plan is best for your situation.</a></p>

<p><a data-t="250">The remaining code is used to tell plumber that, hey, this expression needs to be calculated</a> <a data-t="255">using the Future R package.</a> <a data-t="258">This allows for the main R session to not be blocked.</a> <a data-t="262">A point of note, you must return the result of your Future call in your plumber router</a> <a data-t="268">for it to take effect.</a> <a data-t="270">Otherwise, plumber will not know what's happening.</a></p>

<p><a data-t="274">Let's take a look at the execution time with plumber and Future.</a> <a data-t="278">With just plumber alone, it took 20 seconds total, and we had 30 seconds of accumulated</a> <a data-t="283">wait time.</a> <a data-t="284">However, when we add in Future, everything is run immediately.</a> <a data-t="290">This allows for the total execution time to only take 10 seconds, and we have zero accumulated</a> <a data-t="295">wait time.</a> <a data-t="297">This is the ideal situation, and everything is responding as fast as possible.</a></p>

<h3>How the execution works under the hood</h3>

<p><a data-t="314">Let's dive deep to see how this execution is done.</a> <a data-t="319">Plumber has the three steps that we talked about earlier, Receive, Process, and Respond.</a> <a data-t="324">This is all done on the main R session.</a> <a data-t="327">Only one route can be there at a time.</a></p>

<p><a data-t="330">Future, however, can launch multiple R sessions, and we'll call them Worker 1 and Worker 2.</a> <a data-t="337">In our use case, we're only going to be using them in the Processing step.</a> <a data-t="342">So we must receive and respond from the main plumber R session, but we can offload the</a> <a data-t="348">Processing to the Future Workers.</a></p>

<p><a data-t="353">Let's watch to see how this goes.</a> <a data-t="356">Fast comes in, it is processed, and we respond.</a> <a data-t="360">Nothing new.</a> <a data-t="361">No surprise.</a> <a data-t="362">Slow 2 is offloaded immediately to Worker 1.</a> <a data-t="365">Since the main session is free, then Slow 3 can be offloaded to Worker 2.</a> <a data-t="370">This allows Fast 4 to be processed and respond immediately.</a> <a data-t="374">Once Slow 2 is done, it can respond.</a> <a data-t="377">Same with Slow 3.</a></p>

<p><a data-t="381">That's a lot faster and a lot more going on, but in the end, we have no wait time.</a> <a data-t="387">Let's look at that again.</a> <a data-t="389">This allows Fast 4, since it's not being blocked by Slow 2 or Slow 3, Fast 4 can respond immediately</a> <a data-t="397">at the 0 second mark.</a> <a data-t="400">And since we can do things in parallel, Slow 3 is also being processed at the same time</a> <a data-t="405">as Slow 2.</a> <a data-t="407">Both of these do not block the main R session.</a> <a data-t="410">This is ideal.</a></p>

<h3>Limitations</h3>

<p><a data-t="412">However, there are some limitations to what we can do with plumber in the future.</a> <a data-t="418">One, you have to manually add your Future Call wherever you deem appropriate.</a> <a data-t="423">This is because adding the Future Call adds a little bit of execution time to each route.</a> <a data-t="428">Now, if your execution time is normally in the seconds, and adding a quarter of a second</a> <a data-t="435">does not mean much to you, then great, I think Future is a perfect candidate for that route.</a> <a data-t="439">However, if adding a quarter of a second to your execution time dramatically increases</a> <a data-t="444">it, maybe adding Future is maybe not the best idea, and instead we'll just keep it in the</a> <a data-t="448">main R session.</a></p>

<p><a data-t="451">The other one we have to remember is that processing power is finite and memory is finite.</a> <a data-t="455">We cannot necessarily launch 100,000 workers on one laptop.</a> <a data-t="460">That would be a little bit of overload, either with processing power or the memory.</a> <a data-t="464">One of the two will limit what you can do on your machine.</a></p>
