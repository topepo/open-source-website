<h3>Understanding InfoSec concerns</h3>

<p><a data-t="107">So InfoSec, information security, they care about three things, apparently this is the</a> <a data-t="113">oath they swear to be inducted in the hall of fame of InfoSec agents, I guess.</a> <a data-t="123">Confidentiality, integrity and availability.</a> <a data-t="126">Availability means does this app run?</a> <a data-t="128">Is it available six sigma, 99.99%?</a></p>

<p><a data-t="132">I let them deal with that but I need to be much more aware of confidentiality so if I</a> <a data-t="140">surface new information, is the person that views that information, are they authorised</a> <a data-t="145">to see that?</a> <a data-t="146">And integrity, in other words, is this information accurate and can it be trusted?</a></p>

<p><a data-t="153">So if I have an LNM app, InfoSec flags a number of risks and I'm going to highlight three</a> <a data-t="161">of them.</a> <a data-t="163">You need to work with your own team to see what's relevant for you.</a></p>

<p><a data-t="167">The first one is the risk of data leakage, in other words, if I combine information from</a> <a data-t="172">multiple sources, all of which has potentially sensitive information like PII, personally</a> <a data-t="180">identifiable information or trade secrets, you run the risk of grabbing information from</a> <a data-t="188">multiple places, combining that together and then accidentally surfacing that to somebody</a> <a data-t="192">who should not have had access to that.</a></p>

<p><a data-t="197">The second risk was, well, what if I use an LNM to generate some data and it makes</a> <a data-t="203">up some nonsense and I write that back into a production database?</a> <a data-t="207">Nobody wants that.</a> <a data-t="209">But you also want to have this balance about keeping data in a sandbox, not creating data</a> <a data-t="216">silos.</a> <a data-t="217">So there's a risk that InfoSec is going to want to speak to you about that.</a></p>

<p><a data-t="221">And the third one is a little bit more subtle but everybody is concerned about this.</a> <a data-t="225">If I'm using an LNM agent, and you can use your agent of choice, there is a small risk</a> <a data-t="231">that if you don't control for this carefully, that the data, that your own data will be</a> <a data-t="238">used for training and can accidentally leak to other users, not just on the same server</a> <a data-t="243">but also if it's retrained in the future.</a></p>

<h3>Solutions and recommendations</h3>

<p><a data-t="247">I'm not going to pretend that the solutions I have here is going to work for you.</a> <a data-t="252">But here are a few things to go and think about.</a> <a data-t="255">First of all, make sure that your authorization is correct, that you give minimal permissions,</a> <a data-t="261">so you limit your app to trusted users.</a></p>

<p><a data-t="264">Make sure that you have data security of the LNM itself.</a> <a data-t="269">In our case, we host our models on AWS Bedrock.</a> <a data-t="274">Most of the cloud providers have a service like that where they ensure that your data</a> <a data-t="280">will not leak out and will not be used for retraining.</a> <a data-t="284">And scope to mission.</a> <a data-t="285">So if you write information somewhere else, make sure that where you write it to has the</a> <a data-t="290">same permissions and authorization you had before.</a></p>
