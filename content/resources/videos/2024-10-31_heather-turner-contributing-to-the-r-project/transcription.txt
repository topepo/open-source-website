<h3>Contributing experience and expertise</h3>

<p><a data-t="563">So beyond contributing to the code base, there are other ways that you</a> <a data-t="563">can contribute your experience and expertise.</a> <a data-t="569">One way is testing R before release.</a> <a data-t="569">So you</a> <a data-t="569">may know that R has a major release once a year around April time.</a> <a data-t="573">A couple of weeks</a> <a data-t="573">before that, there'll be a pre-release version that is available on the R project website.</a> <a data-t="578">And you can help by testing your own programs and workflows, your special ways of installing</a> <a data-t="582">or setting up R, things that interact with external libraries or interactive R packages.</a> <a data-t="588">In other words, things that are pretty tricky for the small core team to test themselves.</a> <a data-t="593">And if you find any issues and report those back, you know, maybe they can get fixed before</a> <a data-t="598">the official release.</a></p>

<p><a data-t="605">But it's not too late to report issues, of course, after the release.</a> <a data-t="610">And it's really helpful for anybody to report any issue you find on the bug tracker to help</a> <a data-t="610">maintain R to a high standard.</a> <a data-t="616">So both bug reports and wishlist items, sort of feature</a> <a data-t="616">requests, can be posted on R's Bugzilla.</a> <a data-t="621">And it's really helpful if you're reporting a</a> <a data-t="621">bug to include a minimal reproducible example, which many of you will be familiar with.</a> <a data-t="626">But</a> <a data-t="626">in R, it's particularly helpful to restrict yourself to the base R packages and data.</a> <a data-t="631">So it's clear that it really is an issue with base R, and it can be easily reproduced without</a> <a data-t="636">any add-on packages.</a></p>

<p><a data-t="643">If you've got a little bit more experience and time, then you can</a> <a data-t="643">also help by reviewing bug reports.</a> <a data-t="648">For example, you can help by trying to reproduce the bug</a> <a data-t="648">in different versions of R or in different operating systems, different graphical devices.</a> <a data-t="653">You might be able to simplify the </a><a href="/software/reprex/">reprex</a><a data-t="653"> as another step towards analyzing the root cause,</a> <a data-t="660">or you might be the one that helps find the root cause.</a> <a data-t="664">You know, the few lines of R code</a> <a data-t="664">or C where the bug is actually happening.</a> <a data-t="670">And then you can discuss with others whether</a> <a data-t="670">this is something that actually needs fixing and how to fix it.</a> <a data-t="675">So it helps to have people</a> <a data-t="675">contribute to that discussion rather than it just being one person having to think about</a> <a data-t="679">this.</a></p>

<p><a data-t="685">You can also contribute your expertise via a working group.</a> <a data-t="685">So I've also already</a> <a data-t="685">briefly mentioned the R Contribution Working Group and FWDs.</a> <a data-t="690">The R Consortium also runs</a> <a data-t="690">a number of working groups, and some of these relate to core development or core infrastructure.</a> <a data-t="697">So I mentioned a few of them here.</a> <a data-t="697">There's one on thinking about how to implement multilingual</a> <a data-t="702">R documentation.</a> <a data-t="709">So in other words, having R files, R help files in different languages.</a> <a data-t="709">There's one that's been working for a number of years on the R7 package, which is a potential</a> <a data-t="714">successor to S3 and S4 classes and methods in R.</a> <a data-t="720">And there's the R Repositories Working</a> <a data-t="720">Group, which thinks about how to support CRAN and the development of new package repositories</a> <a data-t="725">for the community.</a></p>
