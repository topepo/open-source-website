<h3>The past: designing gt from scratch</h3>

<p><a data-t="165">So let's go into the past just for a little while. So way back in 2018, early 2018, we looked at</a> <a data-t="173">a number of table packages available in R. I was surprised. There was quite a few. I thought there</a> <a data-t="178">was five. There's more than that. So we looked at all of them. And we noticed they had strengths</a> <a data-t="184">in different areas. Some were known to be a bit domain specific. You know, heavy on the</a> <a data-t="189">stats summary stuff. While others are more general. You can just do whatever you want.</a> <a data-t="195">We noticed that most handled one, maybe two, sometimes a few output formats.</a> <a data-t="208">So we saw these and we thought, can we innovate in this area at all? And yeah, we thought we could. We wanted to make something gt plot like.</a> <a data-t="213">We wanted to focus on noninteractive tables that delivered camera ready output in a wide variety</a> <a data-t="219">of formats.</a></p>

<p><a data-t="225">So we had a few core requirements before we even sort of started, you know,</a> <a data-t="231">making gt. So the first thing we had to have was a declarative but forgiving API.</a> <a data-t="232">So, you know, things we have today are from this. Like, we should always start a gt table</a> <a data-t="239">with the gt function. Any statements could be expressed in any order, give or take.</a> <a data-t="245">Order sometimes matters. But usually not. And each function should act as instructions.</a> <a data-t="251">Gt should then figure out what to render. So it will sort of reconcile</a> <a data-t="255">what you expressed and render the table in the right way.</a></p>

<p><a data-t="262">We wanted to have table generation in multiple output types with the same API. So the same set</a> <a data-t="267">of statements supplied here, gt should always work when taking the same code used for HTML.</a> <a data-t="274">And, you know, you can also make a LaTeX table with the same code.</a> <a data-t="277">We really wanted powerful formatters for cell values. So take, for example, this</a> <a data-t="282">unformatted column of values. We want to have things like formatting to an exact number of</a> <a data-t="288">decimal places, formatting as integers, handling rounding and things like that,</a> <a data-t="292">changing separators, the digit separators, a way to express values in scientific notation,</a> <a data-t="300">a way to use percent signs, perhaps not scaling, ways to force the percent</a> <a data-t="305">the positive sign so it can show changes. We want to have significant digits.</a> <a data-t="318">We wanted to have great support for things</a> <a data-t="318">like currencies, you know, expressing values as bytes, even decorating values using a pattern.</a> <a data-t="324">We wanted it to be flexible and have values formatted the way you wanted them to be.</a></p>

<p><a data-t="330">We also wanted to have it so you can experiment in the API, like move columns to the right,</a> <a data-t="335">move columns around, you know, have GT do things for you, like group columns together</a> <a data-t="342">if they're united under a single spanner, or you can just move columns manually.</a></p>

<p><a data-t="348">We really wanted footnotes. That's a thing we didn't see too much of in other table packages,</a> <a data-t="354">but we know it's a thing and we wanted to have it in there. We wanted them to be pain-free.</a> <a data-t="359">So we didn't want you to have to worry about them and manually sort of reorder them.</a> <a data-t="365">First of all, we wanted it to be straightforward to define table footnotes.</a> <a data-t="368">Then we wanted the ordering to be sort of automatic. And the more advanced features</a> <a data-t="373">are things that are just expected in footnotes, like we should be able to apply the same footnote</a> <a data-t="377">to multiple locations. And if you have multiple footnotes at one location, that should be allowed</a> <a data-t="383">and it should handle the numbering or lettering, you know, in the right way.</a></p>

<p><a data-t="398">So, you know, we worked away at it in most of 2018. We released it in 2018. We had HTML as,</a> <a data-t="398">like, the main output format, but we had LaTeX and RTF as experimental output types.</a> <a data-t="403">Since then, we had tons of conversations with people on social media, inside GitHub issues,</a> <a data-t="409">and through lots of customer conversations. And through those conversations, we learned</a> <a data-t="414">pretty quickly what was, you know, what was working and what wasn't working at all with GT.</a></p>

<p><a data-t="421">So we got to work on fixing stuff. So we improved the RTF output support. We improved LaTeX,</a> <a data-t="427">because, you know, they weren't too good at the beginning. We added additional cell formatters,</a> <a data-t="431">like these ones right here, and we improved the existing cell formatters as well, added more</a> <a data-t="436">options. We provided more ways to style cells, which was very important. It's what people wanted.</a> <a data-t="442">And we had various other quality of life improvements and API enhancements.</a></p>

<p><a data-t="450">Okay. In terms of those customer conversations with people, we had lots of those with people</a> <a data-t="454">in pharma, and we discovered what the biggest requirements were. So we had conversations with</a> <a data-t="458">Yilong Zhang from Work, Tom Tensfeld and Mike K. Smith from Pfizer, Mike Sackhouse from the Taurus,</a> <a data-t="464">Devin Pastore and Evan Eveson from Metrum, and Adrian Waddell and Gabe Becker from Roche.</a> <a data-t="469">So one of the big things we had to do was, again, fix RTF and LaTeX. That was an important</a> <a data-t="474">set of requirements for pharma. We made them quite a bit more reliable. We got much more dialogue</a> <a data-t="480">going about what pharma needs through a working group. It's called the R Tables for Regulatory</a> <a data-t="485">Submissions Working Group, and it's put on by the... well, sponsored by the R Consortium and</a> <a data-t="490">put on by Joe Rickard.
