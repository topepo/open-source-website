<h3>Two guiding principles</h3>

<p><a data-t="365">So in order to avoid having this happen again, I developed a set of approaches guided by</a> <a data-t="369">two principles, don't repeat yourself and talk to yourself.</a> <a data-t="375">And I hope that it's useful for those of you who use R for this kind of repetitive reporting.</a></p>

<p><a data-t="380">So don't repeat yourself is a concept from software engineering aimed at reducing repetition.</a> <a data-t="385">It says that you should abstract away repetitive logic and automate repetitive processes.</a> <a data-t="391">So why do we want to do this?</a> <a data-t="393">Well, copy and pasting your code is really error-prone, and you have to make an update</a> <a data-t="397">everywhere that the code appears.</a> <a data-t="399">And instead of focusing on what changes, we're busy focusing on what stays the same.</a> <a data-t="406">You've probably all heard the mantra that if you copy and paste the same code three</a> <a data-t="422">times, you should turn it into a function.</a> <a data-t="424">So functions and packages are really the core of don't repeat yourself in the R universe.</a></p>

<p><a data-t="429">But repetition is also present in processes.</a> <a data-t="433">Having to redo things, remember where things are, remember what you did is pretty pointless,</a> <a data-t="438">and you might not be able to abstract away that process, but you can definitely try to</a> <a data-t="441">automate things.</a></p>

<p><a data-t="443">Talk to yourself is a concept from me that I thought sounded good with don't repeat yourself.</a> <a data-t="449">And it basically is just aimed at making work easier to use for your future self.</a> <a data-t="453">So it's way easier to document why you're doing something now than to remember why you</a> <a data-t="457">did it a year ago.</a> <a data-t="459">The goal here is to focus on your future self instead of your present self, and just because</a> <a data-t="464">we write easy-to-read code because of the tidyverse doesn't mean that it actually explains</a> <a data-t="467">the motivations of why we're doing something.</a></p>

<blockquote><a data-t="453">So it's way easier to document why you're doing something now than to remember why you</a> <a data-t="457">did it a year ago.</a></blockquote>

<p><a data-t="540">And once I left my files usable as a whole, I had to start thinking about them individually</a> <a data-t="544">and how I could make going through things less manually repetitive and what to abstract</a> <a data-t="549">away.</a> <a data-t="550">So functions are a great way to do this.</a> <a data-t="552">So I had some code to clean and query data that was a good candidate for this.</a></p>

<h3>Artisanal data and generalizable code</h3>

<p><a data-t="672">So I knew that not all of my code had logic that could abstract it away.</a> <a data-t="677">Our projects had a lot of really specific data cleaning steps.</a> <a data-t="680">There was a lot of back and forth and a lot of checking of assumptions and always some</a> <a data-t="684">level of human interaction required.</a></p>

<p><a data-t="686">But a big lesson that I learned is that artisanal data does not require artisanal code.</a> <a data-t="692">So we had very special data, like locally sourced, handcrafted, new mistakes daily artisanal</a> <a data-t="697">data.</a></p>

<blockquote><a data-t="686">But a big lesson that I learned is that artisanal data does not require artisanal code.</a></blockquote>

<p><a data-t="699">But the mistake that I always made was that I was so focused on cleaning that set of data,</a> <a data-t="703">and I wasn't thinking about how to clean a new batch of the data.</a> <a data-t="707">And I only knew why I had done something last time and not what to look for this time.</a> <a data-t="711">So instead of writing super specific code, I started writing generalizable code with</a> <a data-t="716">super specific instructions, including things on data cleaning, who to ask, what to look</a> <a data-t="720">for and what to do.</a> <a data-t="721">And this actually gives you time to deal with cleaning your artisanal data because you're</a> <a data-t="725">not wasting time remembering how to clean it.</a></p>

<p><a data-t="727">So we've seen this before.</a> <a data-t="727">But again, what my code used to look like is that I would take the data, change the</a> <a data-t="730">value of a field for one ID and the value of a different field for a different ID with</a> <a data-t="734">no idea why I had done it.</a> <a data-t="736">But what I replaced it with is something that looks like this.</a> <a data-t="738">So first I'm saying what to do.</a> <a data-t="740">So an example is I want to check for any cases where the province is Ontario, but outside</a> <a data-t="745">of Ontario is true, because that's obviously not right.</a> <a data-t="749">And then why to do it?</a> <a data-t="750">Because some of them may have province wrong, but some may have outside Ontario wrong.</a> <a data-t="754">And then you say who to talk to.</a> <a data-t="756">So email Carla.</a></p>

<p><a data-t="765">And ask them to check paper files to confirm which is correct and then what to do next.</a> <a data-t="770">So based on the response, update any manually until the database is updated.</a> <a data-t="774">Because when I started this job, I found it so difficult to keep track of who to ask for</a> <a data-t="778">different things.</a> <a data-t="778">You know, like you have to email that person and if they're gone, you have to email someone</a> <a data-t="782">else.</a> <a data-t="784">So I decided to just codify it so I wasn't leaving myself wondering.</a> <a data-t="788">And then to avoid repetition, I could write a function that actually automates producing</a> <a data-t="792">the template, again, using the usethis package.</a> <a data-t="795">So you take the template, it makes and saves a copy of it, and you can even pass data to</a> <a data-t="801">it.</a></p>

<h3>Putting it all together in a package</h3>

<p><a data-t="803">And finally, I put everything together into a package.</a> <a data-t="806">A project like this is a great candidate for making a package, and it is not as scary as</a> <a data-t="811">it seems.</a> <a data-t="811">And once I started making packages for this, I started making packages for everything else</a> <a data-t="815">in my life.</a></p>

<p><a data-t="820">So why do we want to make a package?</a> <a data-t="825">I mean, as I said, making a package is really an ultimate form of do not repeat yourself.</a> <a data-t="829">Your functions are centralized in a specific place and it's really clear where to update</a> <a data-t="833">them.</a> <a data-t="834">Your templates also have a place to live and then you can use other functions to copy and</a> <a data-t="837">pass data to them.</a> <a data-t="839">There's robust documentation and you can share code with your coworkers in a better way than</a> <a data-t="843">emailing it or having it on a network drive.</a></p>

<p><a data-t="846">So when all was said and done, I had a package that looked something like this.</a> <a data-t="852">So my functions had a home in the R directory, and then the templates live in a special place</a> <a data-t="857">so you can copy and use them later.</a> <a data-t="859">There's documentation in the form of a readme, function documentation in the man folder,</a> <a data-t="863">and vignettes for detailed usage.</a> <a data-t="865">And then to put it all together, the setup function pulls the setup template and makes</a> <a data-t="870">a new file with the template, which contains other functions that can run through the analysis.</a> <a data-t="875">And then when you run the function to query the data, it does so, and it does so chattily</a> <a data-t="880">so that it's abstracting away the logic but making sure that you know where things are</a> <a data-t="884">saved.</a></p>

<p><a data-t="886">So I use don't repeat yourself and talk to yourself by cleaning up code with clear names</a> <a data-t="892">and ordering.</a> <a data-t="893">I made functions, and I made them chatty to communicate what they were doing.</a> <a data-t="897">I made automated templates, and I made them prescriptive to help me know what to do year</a> <a data-t="900">after year.</a> <a data-t="902">And then I made a package to put it all together.</a></p>

<p><a data-t="907">And using this form of repeated reporting in R, I had a project that was reproducible,</a> <a data-t="912">easy to keep track of, it had minimal manual work, was easily repeatable, and it freed</a> <a data-t="917">up time, which I spent making other R packages.</a></p>

<p><a data-t="921">So my takeaway to you is that if you have a repetitive task that you find confusing</a> <a data-t="925">and you're doing the same work over and over, abstract away or automate what you can and</a> <a data-t="930">loudly communicate the rest.</a> <a data-t="932">Or because it's catchier, don't repeat yourself, talk to yourself.</a> <a data-t="937">Thank you.</a></p>
