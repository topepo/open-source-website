<h3>How Shiny speaks <a href="/software/websocket/">WebSocket</a></h3>

<p><a data-t="77">So okay, so to go kind of all the way down the hole, we get down into the Shiny package,</a> <a data-t="83">and I suspect that most of you are familiar with Shiny, but basically Shiny is, as you</a> <a data-t="90">know, an R package, but what's important to realize is that Shiny speaks WebSocket.</a> <a data-t="95">And so what that means is that any browser that's compatible with WebSocket can connect</a> <a data-t="100">directly to a Shiny process and do, you know, the communications there back and forth between</a> <a data-t="106">those two entities.</a></p>

<p><a data-t="107">So that's great, and you probably notice this when you're developing applications locally.</a> <a data-t="112">So if you're using </a><a href="/software/rstudio/">RStudio</a><a data-t="112">, for instance, and you press run app, you're going to get</a> <a data-t="114">a little viewer.</a> <a data-t="115">That viewer is communicating directly to your R process, or, you know, if you open in a</a> <a data-t="119">browser, you're communicating directly with an R process, and there's no Shiny server</a> <a data-t="123">intermediary between those two, and that's because Shiny speaks WebSockets.</a> <a data-t="128">So it is possible to open a direct connection from a browser into the Shiny process.</a></p>

<p><a data-t="135">So that's great, but there are a couple of limitations on this.</a> <a data-t="137">First of all, as I mentioned, that only works for browsers that can communicate over WebSockets,</a> <a data-t="142">and so that rules out Internet Explorer 8 and 9 and some other versions of Firefox and</a> <a data-t="148">Chrome and things like that.</a> <a data-t="150">The other problem is that the Shiny process has to be running at the moment you try to</a> <a data-t="154">open a connection to it, and so that can be a little complicated in that if you have,</a> <a data-t="159">you know, as an organization, maybe you have, you know, hundreds of Shiny applications that</a> <a data-t="162">you've developed, and at any given time, maybe, you know, a handful of them are actually</a> <a data-t="166">being used and interacted with.</a></p>

<p><a data-t="169">The problem is here that, you know, in order to communicate, Shiny would have to be running</a> <a data-t="173">all of, like, you would have to be running one process for each of those applications</a> <a data-t="177">all the time around the clock, and that creates a lot of problems in terms of resource utilization.</a> <a data-t="182">If you're trying to run these, you know, hundreds or even thousands of processes all the time,</a> <a data-t="187">even though many of them may not be used, you know, but, you know, a few minutes a day.</a> <a data-t="192">So there are a couple of limitations to just running Shiny on the bare metal, but it is possible.</a></p>

<h3>What Shiny Server sets out to accomplish</h3>

<p><a data-t="198">So Shiny server sets out to accomplish a few goals.</a> <a data-t="203">The couple that are interesting for this discussion are, first of all, it should start and stop</a> <a data-t="208">Shiny processes on demand and as needed, and also it should translate non-WebSocket traffic</a> <a data-t="215">into WebSocket traffic that Shiny can understand.</a> <a data-t="218">And then finally, it can also map different URLs to particular applications.</a></p>

<p><a data-t="223">So if you were just to run an army of Shiny processes on your bare machine, all of those</a> <a data-t="228">would have to run on different ports, and so your users would need to enter a port number</a> <a data-t="231">in order to get to an application, and it would be, you know, a little sloppy.</a> <a data-t="235">And so Shiny server offers a kind of a cleaner, more convenient way to be able to navigate</a> <a data-t="241">between the different applications, and we do that by offering a few different hosting</a> <a data-t="244">modes.</a> <a data-t="245">So you could just host one application.</a> <a data-t="246">You could allow users to host their own applications, or you could host an entire directory of applications,</a> <a data-t="251">and you can do these at different URL subspaces.</a> <a data-t="253">So it just allows you to kind of more meaningfully manage, you know, many Shiny applications.</a></p>

<p><a data-t="262">And so a couple of these to kind of revisit.</a> <a data-t="264">So starting and stopping Shiny processes, so again, this is, you know, if you're running</a> <a data-t="269">this naively, you would need to be running that Shiny process all the time, but because</a> <a data-t="273">you're running Shiny server as kind of this long-running, you know, always-on daemon,</a> <a data-t="277">what that means is that when a user goes to connect into the Shiny process, if it's not</a> <a data-t="282">running now, Shiny server can kind of hold that request for a second while it goes to</a> <a data-t="288">start the Shiny process, and then once the Shiny process is ready, it can let the traffic</a> <a data-t="292">through into that process, so it can kind of broker and manage the communications between</a> <a data-t="297">applications that may or may not be running at the moment they're requested.</a></p>

<p><a data-t="301">And then also if it notices that all the users have left and that application's been</a> <a data-t="305">idle for some configurable length of time, then it can go ahead and reap those processes</a> <a data-t="309">to free up some more resources for other applications.</a></p>

<p><a data-t="313">And again, like I mentioned, older versions of Internet Explorer are not going to be able</a> <a data-t="315">to handle WebSockets natively, and so Shiny server uses a library called sock.js, which</a> <a data-t="322">basically allows you to work as if you were working with WebSockets, but it will kind</a> <a data-t="326">of gracefully fall back into other protocols as needed.</a> <a data-t="330">And that's actually really important, even if you're not planning to support older versions</a> <a data-t="334">of Internet Explorer, because many networks, if you're using older network equipment that</a> <a data-t="340">was designed before WebSockets gained popularity, then you'll actually end up in a situation</a> <a data-t="346">where they may be dropping or blocking WebSocket traffic, and we see this quite a bit, especially</a> <a data-t="350">at universities and academic settings, they for some reason tend to have network equipment</a> <a data-t="356">that isn't friendly with WebSockets, so you can kind of end up in the situation where</a> <a data-t="362">even if you're using all the latest and greatest technology, there may be some router between</a> <a data-t="365">you and your client that is going to interrupt your WebSocket traffic, and so it's nice to</a> <a data-t="371">have a technology that will allow you to fall back to something that is supported if needed,</a> <a data-t="376">so that ends up being a pretty important feature.</a></p>

<h3>Shiny Server architecture overview</h3>

<p><a data-t="380">And then as we already discussed, being able to map URLs to different applications.</a> <a data-t="385">So kind of an overview of the architecture, you can envision it looking something like</a> <a data-t="390">this.</a> <a data-t="391">So you have a variety of users over here, I don't know if you can see, you have your</a> <a data-t="397">variety of users over here, which could be a handful, or we run servers where there are</a> <a data-t="403">hundreds of concurrent connections from different users around the world, but they're all opening</a> <a data-t="407">connections into your Shiny server instance, which again is kind of this always-on server</a> <a data-t="413">process that's running as a daemon on your server.</a></p>

<p><a data-t="417">And then it's going to open and close, or start and stop these Shiny applications, and</a> <a data-t="423">kind of proxy the traffic as it goes through from users' browsers into the Shiny applications,</a> <a data-t="430">and so what this typically looks like, just to give you a feel if you're going to get</a> <a data-t="433">down to the network level, the first request, of course, is just for kind of the root URL</a> <a data-t="438">of some application, Shiny server would go in and inspect, is that application already</a> <a data-t="442">online?</a> <a data-t="443">So let's say it maps some URL to app one, so it would go through and say, is app one</a> <a data-t="450">already online?</a> <a data-t="451">If it is, it'll proxy the traffic through, if it's not, it'll start app one, and then</a> <a data-t="455">proxy the traffic through.</a></p>

<p><a data-t="458">And then just kind of continue to keep these channels open between the user and the application.</a> <a data-t="464">So what this looks like at a network level, the first request is going to be for kind</a> <a data-t="469">of the root URL of that application, and then that's going to return some HTML that is going</a> <a data-t="473">to kind of point the browser to different static assets that may be of relevance to</a> <a data-t="478">CSS files and JavaScript files and things like that.</a> <a data-t="481">And so it'll open up more connections for all of those files, and then eventually it'll</a> <a data-t="487">actually open a WebSocket or WebSocket-like connection that will be persistent between</a> <a data-t="492">that browser and that Shiny application, and that's kind of how Shiny, the Shiny magic</a> <a data-t="497">works is via that WebSocket or that WebSocket-like channel that allows the user and the Shiny</a> <a data-t="503">process to communicate back and forth for as long as that connection is open.</a></p>

<p><a data-t="507">So a couple of points here, just as I mentioned, the user's connection to the Shiny Server</a> <a data-t="547">may or may not be a WebSocket, though these connections between Shiny Server and the Shiny</a> <a data-t="551">process must be WebSocket connections, because that's the only protocol that Shiny can understand.</a> <a data-t="557">But these external connections, again, may or may not be WebSockets, they don't have</a> <a data-t="561">to be.</a> <a data-t="565">If you're using an older browser or some network intermediary is blocking WebSockets, that's</a> <a data-t="569">fine, we can handle other protocols as well.</a></p>

<p><a data-t="570">And then the other point here, of course, is that Shiny applications are going to be</a> <a data-t="572">started and stopped as needed, and so the running processes may not encompass all of</a> <a data-t="578">the available applications that we know about, you know, there may be others that we could</a> <a data-t="582">start as a request comes in for that application.</a></p>

<h3>Shiny Server Pro features and architecture</h3>

<p><a data-t="591">So that hopefully gives you a feel for Shiny Server, which is kind of a simplified architectural</a> <a data-t="595">diagram compared to what you're going to encounter in Shiny Server Pro.</a> <a data-t="599">So Shiny Server Pro adds a variety of different features, and of course we enumerate all these</a> <a data-t="604">on the product pages on our website, but a couple that are of relevance today, if you're</a> <a data-t="609">considering doing something like high availability or load balancing between Shiny Server processes,</a> <a data-t="614">there are a couple features that you need to be aware of in Shiny Server Pro.</a></p>

<p><a data-t="619">So again, the Pro product is going to add a few features like authentication and more</a> <a data-t="624">scalability, so being able to support a single Shiny application having multiple processes.</a> <a data-t="630">One of the nice benefits here, if you're looking at doing high availability or load balancing</a> <a data-t="635">across multiple instances, is that there's really not a whole lot of centralized storage</a> <a data-t="640">to coordinate between multiple Shiny Server Pro instances, so you don't necessarily need</a> <a data-t="645">like an NFS share or, you know, some centralized database or something like that.</a> <a data-t="650">Most of the state is actually kept on the user's browser, which I'll kind of explain</a> <a data-t="657">here in a minute, but there's not a whole lot of centralized storage, so the only statefulness</a> <a data-t="662">that you need to worry about is just the information that pertains to the open connections right</a> <a data-t="668">now at that moment, and so you need to know about which user is mapped to which worker</a> <a data-t="673">and how to be able to connect those two entities consistently.</a></p>

<p><a data-t="678">And, you know, as I mentioned, most of the information is actually</a> <a data-t="682">stored in the browser, so the authentication information in particular is actually stored</a> <a data-t="686">in an encrypted cookie that gets placed on the user's browser, so you don't need to worry</a> <a data-t="691">about a session store in, you know, Redis or a database or anything like that.</a> <a data-t="697">That's all just kind of pushed out into the user's browser, which is nice, so you won't</a> <a data-t="701">need to worry about coordinating any central storage like that.</a></p>

<p><a data-t="702">Okay, so this is a very similar diagram for Shiny Server Pro as we saw with Shiny Server,</a> <a data-t="708">but there's one distinction, which is that you can see that a single application may</a> <a data-t="713">be backed by multiple workers, and that's kind of what I was referencing earlier when</a> <a data-t="717">I was saying that, you know, you can scale an application to multiple processes.</a> <a data-t="729">That's one of the big selling points for Pro, so what that means is that you can have different</a> <a data-t="734">workers each with different IDs backing a single application.</a></p>

<p><a data-t="741">It's important that you understand that distinction because when a user opens a connection to</a> <a data-t="746">Shiny Server Pro, but ultimately into a Shiny process, it's very important that that user</a> <a data-t="751">be consistently mapped back to that same Shiny process, and so there are some cases where</a> <a data-t="757">that's really important in Shiny, but it's extremely important in R Markdown, and you'll</a> <a data-t="762">notice if you're hosting any kind of R Markdown documents, you'll notice very quickly that</a> <a data-t="767">things are going to start breaking in kind of weird ways if your users are not being</a> <a data-t="772">deterministically mapped back to the worker that they open the connection on.</a></p>

<p><a data-t="773">So basically what that means is that if I as a user come in and request a page, if I</a> <a data-t="780">request App One, then Shiny Server Pro is going to go through and see, do I have any</a> <a data-t="784">workers associated with App One yet?</a> <a data-t="786">Yes, I have two, and now it's going to pick one of these based on whichever one has the</a> <a data-t="791">least load currently, and so let's say that's Worker 13, so it'll point me to Worker 13.</a> <a data-t="796">Worker 13 will fulfill my original request for the base URL, but then what I'm saying</a> <a data-t="801">is that every HTML request and ultimately the web socket that gets opened between the</a> <a data-t="806">user and that application must always be routed to Worker 13.</a></p>

<h3>Load balancing and high availability</h3>

<p><a data-t="853">So now if we're looking at doing load balancing or high availability of Shiny Server Pro instance,</a> <a data-t="858">this is where you need to start thinking about the implications of some of those things.</a> <a data-t="863">So first of all, if we put a load balancer here in front of two Shiny Server Pro instances,</a> <a data-t="868">you'll notice that both Shiny Server Pro instances may be running processes associated with App1.</a> <a data-t="874">So in this case, we have one Shiny Server Pro over here that's hosting two workers associated</a> <a data-t="878">with App1 and one over here that's also hosting two workers of App1.</a></p>

<p><a data-t="883">And so this is nice because these could be running and typically would be running on</a> <a data-t="887">different physical boxes and may have separate network connections and things like that.</a> <a data-t="892">So this will allow you to do high availability and redundancy.</a> <a data-t="896">But what's nice about this is that then any resources that you're using here, if you're</a> <a data-t="901">doing real-time streaming, for instance, onto one of these Shiny processes, then you</a> <a data-t="905">need to be able to kind of monitor, you know, frequent updates to data or you're storing</a> <a data-t="909">a large amount of memory or something like that, then it can be really nice to be able</a> <a data-t="913">to scale this out to multiple physical boxes, which you can do as in this diagram.</a></p>

<p><a data-t="918">But a couple of things that you'll need to be aware of in doing so.</a> <a data-t="922">So first of all, you'll obviously need a load balancer in front of the two instances that's</a> <a data-t="925">going to balance traffic between the two.</a> <a data-t="929">And keep in mind that there are WebSocket-friendly load balancers, like NGINX, I believe, now</a> <a data-t="932">can support WebSockets and load balance traffic just fine with those.</a> <a data-t="937">And Amazon DLB, if you're on Amazon Web Services and things like that, they can do these well.</a> <a data-t="942">But older proxies or load balancers may not do WebSockets completely, they may not support</a> <a data-t="948">that protocol.</a> <a data-t="949">And so just be aware that if you're using an older load balancer or one that doesn't</a> <a data-t="953">support WebSockets, then all of the WebSocket-like connections coming in from users will be degraded</a> <a data-t="960">down to some other protocol that is compatible.</a></p>

<p><a data-t="963">So it's not necessary that this be WebSocket-friendly, but just be aware that WebSockets are kind</a> <a data-t="970">of going to be the most efficient and lowest latency channel between the Shiny process</a> <a data-t="975">and the user.</a> <a data-t="976">And so just be aware that if you have a non-WebSocket-compatible load balancer, that there's just going to</a> <a data-t="982">be a little bit more latency and a little bit more network overhead between your users</a> <a data-t="985">and your Shiny instances.</a> <a data-t="987">Really it's probably going to be negligible, but just so you're informed.</a></p>
