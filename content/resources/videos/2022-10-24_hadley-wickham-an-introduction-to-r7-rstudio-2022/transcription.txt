<h3>S3, S4, and where R7 fits in</h3>

<p><a data-t="196">But there are already two types of OO systems in R. So how does R7 fit in? Well, first of</a> <a data-t="204">all, we have S3, or Mama Bear. Now, S3 has its name because it was the OOP system developed</a> <a data-t="214">for the third version of the S programming language. It really is the simplest OOP system</a> <a data-t="220">that could possibly work. It works basically mostly on conventions. It doesn't have a lot</a> <a data-t="226">of structure. And it works great, but it's very, very simple.</a></p>

<p><a data-t="230">And then a few years later, as kind of a reaction to S3, along came Papa Bear, or S4. And S4</a> <a data-t="238">is very strict, very rigorous. You get a lot more guarantees, but it's also a lot more</a> <a data-t="244">complicated and a lot more work. So S4, so-called, because it was designed for the fourth version</a> <a data-t="251">of the S programming language.</a></p>

<p><a data-t="255">And so now we've been working on this tool called R7, where we want to try and be like</a> <a data-t="261">Baby Bear. We want to be just right. We want to strike the right balance between complexity</a> <a data-t="266">and power. And we're calling it R7 because it's hopefully 3 plus 4, like the best parts</a> <a data-t="274">of S3 and S4. And it's designed for R, not for backward compatibility with S.</a></p>

<h3>Classes in R7</h3>

<p><a data-t="541">So there are two parts to object-oriented programming systems in R, generics, generic</a> <a data-t="546">functions, which define behavior, and classes, which define data. <p><a data-t="558">So I'm going to create a new class. Surprise, surprise, I'm going to use a function called</a> <a data-t="563">new class to do that. I'm going to call it arrange class. <p><a data-t="585">This class has some properties. So properties are the data that an object possesses. If</a> <a data-t="590">you've used S3 before, this is the same with attributes. If you've used S4, these are the</a> <a data-t="595">same as slots. So these are the data that every object of this range class is going</a> <a data-t="603">to possess. <p><a data-t="611">So now that we've created that object, that class object, that class constructor, we can</a> <a data-t="616">create range objects just by calling that function. <p><a data-t="639">So you can get values, you can set values, but R7 has got some built-in safety rails.</a> <a data-t="645">So that if you try and assign a character vector to that X property that we said had</a> <a data-t="651">to be a numeric, you're going to get an error. <h3>R7 vs S3 and S4</h3>

<p><a data-t="752">So that's R7 in a nutshell. But how does it compare to S3 and S4? Well, I think if we're</a> <a data-t="760">going to compare OO systems, it's good to think about what's the complexity, how hard</a> <a data-t="765">it is to use, and what's the payoff. And S3 has a very, very low complexity. It's very,</a> <a data-t="772">very simple, and the payoff is relatively low. S4, the payoff is much, much higher,</a> <a data-t="779">but the complexity is also much, much higher. And so our goal with R7 is to create something</a> <a data-t="787">that's just a little bit more complex than S3, but has almost all of the features, all</a> <a data-t="793">of the payoff of S4.</a></p>

<p><a data-t="797">Now whenever I hear about someone trying to solve a problem where there's two existing</a> <a data-t="802">solutions by creating a third solution, I always think of this XKCD comic. And we really</a> <a data-t="811">want to avoid this with R7. We don't want to make the whole problem more complicated</a> <a data-t="816">by now giving people three possible choices of object-oriented programming instead of</a> <a data-t="820">the two that are baked into base R, not to mention like the five or six other packages</a> <a data-t="826">on CRAN that have other object-oriented programming systems.</a></p>

<p><a data-t="832">So how are we going to do that? Well, first of all, this is a team effort. The R Consortium</a> <a data-t="836">Working Group on Object-Oriented Programming, that's a mouthful, includes representatives</a> <a data-t="841">from R-Core, from Bioconductor, from ROpenSci, from RStudio, </a><a href="/software/tidyverse/">tidyverse</a><a data-t="841">, and the general R</a> <a data-t="848">community. This is not an effort to solve the problems of just a small group of people,</a> <a data-t="853">but this is an effort to solve the problems faced by a large proportion of the R community.</a></p>

<p><a data-t="860">Next we're trying to take what we believe to be the best parts of S4 and keep them.</a> <a data-t="865">And that's like the formal definition of classes, so when you get an object of a specific</a> <a data-t="869">type you know a bunch about it right off the bat. That that at helper, which gives you</a> <a data-t="875">a useful error message if you try to extract something that does not exist. R7 also provides</a> <a data-t="881">multiple dispatch when needed, which is really important for, for example, arithmetic operations</a> <a data-t="888">where you need to choose the implementation based on the left-hand side and the right-hand</a> <a data-t="894">side of addition. And finally we have this validator function which allows us to apply</a> <a data-t="899">additional conditions to make sure our objects are valid.</a></p>

<p><a data-t="904">And we've tried to do that while keeping R7 100% backward compatible with S3. So if you</a> <a data-t="911">have an existing S3 class, you can convert it to R7 in such a way that everyone who uses</a> <a data-t="918">that object as if it's an R7 object gets all of these new goodies, but no existing code</a> <a data-t="924">breaks.</a></p>

<p><a data-t="928">And one of the ways we've done this is a little, is a little weird in terms of the spectrum</a> <a data-t="933">of object-oriented programming systems and other environments. So one of the core kind</a> <a data-t="938">of precepts of OO programming is normally encapsulation. And that means like only the</a> <a data-t="944">person who like writes a class can access the data inside that class.</a></p>

<p><a data-t="951">And this, this is I think a really good idea in like 99% of programming languages, but</a> <a data-t="957">it doesn't really work in R because the people who use R are data scientists and they want</a> <a data-t="963">that data inside that object and they will do whatever it takes to get that object if</a> <a data-t="968">they want it, to get that data if they want it.</a></p>

<p><a data-t="971">So S4 tried to kind of protect people from this, like that you're never, as a user, you're</a> <a data-t="977">not really supposed to use the at symbol. You're always supposed to use some kind of</a> <a data-t="980">accessor function, but it just didn't really work. People want that data.</a></p>

<p><a data-t="984">And so instead R7 is designed around, like rather than trying to fight this very natural</a> <a data-t="989">tendency, R7 accepts it while giving you the ability to change the internals of your class</a> <a data-t="998">if you need to with these dynamic properties. So if you have to, you can change the internal</a> <a data-t="1003">implementation. You can deprecate fields, properties inside your object and still have</a> <a data-t="1009">existing code work, which is really important property to allow code to evolve over time.</a></p>

<blockquote><a data-t="984">And so instead R7 is designed around, like rather than trying to fight this very natural</a> <a data-t="989">tendency, R7 accepts it while giving you the ability to change the internals of your class</a> <a data-t="998">if you need to with these dynamic properties.</a></blockquote>

<p><a data-t="1016">And then finally we've tried to dig a pit of success with thoughtful function names,</a> <a data-t="1022">argument names, documentation and errors.
