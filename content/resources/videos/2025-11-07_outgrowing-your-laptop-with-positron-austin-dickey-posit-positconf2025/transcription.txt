<h3>Modern data processing tooling</h3>

<p><a data-t="152">So let's talk about modern data processing tooling.</a> <a data-t="154">Basically, who is this for?</a> <a data-t="156">I think this is for people like all the people at my sister's wedding who are computational biologists and were assigned laptops by their organization that they call, quote, crappy and need to run their R scripts for weeks on end and making sure that their laptop is connected to a power source.</a></p>

<p><a data-t="174">This is for people who don't have access to any other machines and are confined to their laptop.</a> <a data-t="180">It might also be for people who maybe have access to other stuff, but they're working really quickly on quick local iterations or proof of concepts.</a> <a data-t="188">This is kind of a different way of thinking how to do data processing.</a> <a data-t="193">But I want to point out that these techniques are always useful, no matter your scenario.</a> <a data-t="197">So I would definitely consider learning these kinds of things as you walk away from this talk.</a></p>

<p><a data-t="204">So as an example, throughout this talk, I'm going to work with the NYC taxi data set.</a> <a data-t="209">Some of you might be familiar with this.</a> <a data-t="210">This is a lot bigger than Iris or Penguins, but it's basically all the trips that New York City taxis have taken since 2009.</a> <a data-t="219">The one that I'm using is hosted by Ursa Labs, and that's a data set in parquet format that goes up until it's ten years of data, 2019, lives in the cloud.</a> <a data-t="228">It's about 36 gigabytes on disk, and if you were to read it all into pandas in Python, that would be 400 gigabytes of RAM, and I don't know about you, but my MacBook does not have 400 gigabytes of RAM.</a></p>

<p><a data-t="241">So what can you do in this scenario?</a> <a data-t="242">I mean, luckily for you, there are a lot of tools that help with this.</a> <a data-t="246">I put up some logos here on the slides.</a> <a data-t="248">You've got DuckDB, Polars, Apache Spark, Dask.</a> <a data-t="252">Python and R work with a lot of these.</a> <a data-t="254">The purpose of this talk is not which one to choose, because often that's determined by whatever your organization is using or, you know, different pros and cons between these things, but instead I want to focus on their similarities.</a></p>

<p><a data-t="268">So they typically use these strategies.</a> <a data-t="270">The first one is deferred evaluation or lazy evaluation, just like these cats, and basically that means you don't query the data until the last possible second, and that's good because you don't want a bunch of intermediate data sets living in your laptop's memory and crashing your process, right?</a></p>

<p><a data-t="292">We also have a strategy called predicate pushdown or column reading, which is you only read the data that you need.</a> <a data-t="300">It's really helpful if you have data, for example, in parquet formats that's partitioned.</a> <a data-t="310">Because when you write a query, and, again, this can be in Python, R, SQL, or anything.</a> <a data-t="316">When you write a query, it can only access the data that it needs in order to execute that query.</a> <a data-t="323">So if you have a million row or million column parquet file and you only need access to a few of those columns for the query that you're doing, you're reading, it only reads those columns, and in some cases, if your parquet file is partitioned the right way, it only reads certain rows from that file as well.</a></p>

<p><a data-t="337">And that's good because you're not wasting time reading a bunch of data that you don't need.</a> <a data-t="342">And then finally, these libraries and tools, they split the work among multiple processors if possible.</a> <a data-t="348">So if your laptop or any computer you're working on has multiple CPUs or maybe even GPUs or VCPUs, whatever, it's going to split up that work, parallelize it, and make things faster.</a> <a data-t="355">So that's great.</a></p>
