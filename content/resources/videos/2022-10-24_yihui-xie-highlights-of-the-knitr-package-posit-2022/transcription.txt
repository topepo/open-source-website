<h3>New way to write chunk options</h3>

<p><a data-t="208">So the first thing I want to highlight is that now you have a new way to write chunk</a> <a data-t="214">options.</a> <a data-t="215">Previously, you must write all chunk options on one line in the chunk header.</a> <a data-t="221">Now you can write them on multiple lines as special comments in the beginning of the</a> <a data-t="228">chunk body.</a> <a data-t="229">So this special comment starts with the hash character followed by the pipe character.</a> <a data-t="236">I'm not sure how would you pronounce this combination.</a> <a data-t="241">Maybe hashpipe?</a> <a data-t="243">I remember someone mentioned to me that hashpipe was a funny coincidence.</a> <a data-t="249">Since I'm not a native English speaker, I had no idea what hashpipe means.</a> <a data-t="254">So I googled for that and I regret it.</a></p>

<p><a data-t="258">Anyway, hashpipe comments, you can use that to write chunk options.</a> <a data-t="264">So as I said previously, you have to force all the chunk options onto a single line,</a> <a data-t="270">which is sometimes very awkward.</a> <a data-t="271">If you have many, many chunk options for a single chunk, there will be an awkward horizontal</a> <a data-t="279">scroll bar.</a> <a data-t="280">But now you can write all of your chunk options can write with your chunk body.</a> <a data-t="287">You can put them inside the chunk after the special comments.</a> <a data-t="292">And you can also hard wrap all these lines freely.</a> <a data-t="297">You can write them on any number of lines.</a></p>

<p><a data-t="305">And alternatively, you can also use the YAML syntax to provide your chunk options.</a> <a data-t="316">So previously I'm showing the old syntax of comma separated chunk options of the form</a> <a data-t="323">option equals value and separated by commas.</a> <a data-t="327">If you're familiar with YAML, you know that syntax will be option colon followed by value.</a> <a data-t="334">And you have to write one option per line.</a> <a data-t="338">For Quarto and RStudio or VS Code users, I strongly recommend that you use the YAML</a> <a data-t="345">syntax because the editors, RStudio and VS Code, they have better support for the YAML</a> <a data-t="353">options.</a> <a data-t="354">For example, they can help you autocomplete the option names.</a> <a data-t="359">They can validate the values of your options.</a> <a data-t="363">And second, the YAML syntax works for other engines in Quarto as well, such as Jupyter.</a></p>

<p><a data-t="371">So if you want to convert the old syntax to the new syntax, there is a function named</a> <a data-t="378">convertChunkHeader in Knitr.</a> <a data-t="381">So you can use that function to convert your old rmarkdown documents to using the new</a> <a data-t="389">syntax for chunk options.</a> <a data-t="390">This requires a very recent version of Knitr.</a></p>

<h3>Reusing chunk options and content</h3>

<p><a data-t="396">And the second thing I want to talk about is how to reuse both chunk options and chunk</a> <a data-t="405">content from other code chunks.</a> <a data-t="411">Previously, there existed several ways to reuse the content of a code chunk.</a> <a data-t="417">For example, through the ref.label option.</a> <a data-t="421">So suppose you have one code chunk with the label chunk A, and then you can write another</a> <a data-t="428">code chunk with the label chunk B, but set the chunk option ref.label equals chunk A.</a> <a data-t="437">ref.label equals chunk A, that means I want to copy the chunk content of chunk A into</a> <a data-t="447">chunk B.</a> <a data-t="450">So now, although you see the chunk B is empty here, but when you execute this</a> <a data-t="457">document, chunk B will have the same content as chunk A.</a> <a data-t="461">You can learn more about reusing the chunk content in the rmarkdown cookbook.</a></p>

<p><a data-t="469">So now, in addition to reusing the content of code chunks, you can also reuse chunk options</a> <a data-t="479">from other code chunks.</a> <a data-t="481">And this can be done through the chunk option ops.label.</a> <a data-t="487">So for example, you have chunk A here, and you have set a couple of chunk options on</a> <a data-t="493">chunk A.</a> <a data-t="497">And if you want to use the same chunk options in chunk B, then you simply can set</a> <a data-t="504">ops.label equals chunk A.</a> <a data-t="507">So instead of repeating all the chunk options from chunk A, you simply</a> <a data-t="513">set a single option, ops.label equals chunk A.</a> <a data-t="517">And then chunk B will use the same chunk</a> <a data-t="524">options as chunk A.</a></p>

<p><a data-t="526">Then, if you want to reuse both chunk options and the content from other code chunks, you</a> <a data-t="535">can set the chunk option ref.label to the label of other code chunks, and then wrap</a> <a data-t="544">the labels in the function I.</a> <a data-t="548">For example, here I have chunk A, which has a few chunk</a> <a data-t="551">options, and then it also has some chunk content.</a> <a data-t="555">And then in chunk B, I set ref.label equals I chunk A.</a> <a data-t="562">That means I want to use both the</a> <a data-t="568">content of chunk A and the chunk options of chunk A.</a> <a data-t="573">So in that way, you don't have</a> <a data-t="578">to repeat everything from chunk A.</a> <a data-t="581">You simply set a single option, and Knitr will copy everything</a> <a data-t="589">to chunk B.</a></p>

<p><a data-t="593">So I also want to share two quick tips.</a> <a data-t="599">First, the chunk options ref.label and ops.label can take a vector of chunk labels.</a> <a data-t="607">That means you can reuse multiple code chunks instead of a single one.</a> <a data-t="613">Second, you can provide further local chunk options.</a> <a data-t="620">If a code chunk has some local chunk options, they will override the reused options of the</a> <a data-t="627">same names.</a></p>

<p><a data-t="632">So in this example, chunk B is basically reusing both chunk options and the chunk content of</a> <a data-t="640">chunk A, but it has set a local chunk option dev equals png.</a> <a data-t="648">So since we have borrowed all the chunk options from chunk A, which includes the chunk option</a> <a data-t="655">dev equals svg.</a> <a data-t="657">So now chunk B, since chunk B has a local chunk option dev equals png, this dev equals</a> <a data-t="665">png will override the dev equals svg from chunk A.</a> <a data-t="671">So basically, local chunk options will have higher priorities if you reuse chunk options</a> <a data-t="679">from other code chunks.</a></p>

<blockquote><a data-t="671">So basically, local chunk options will have higher priorities if you reuse chunk options</a> <a data-t="679">from other code chunks.</a></blockquote>

<h3>New chunk options</h3>

<p><a data-t="682">Third thing I want to mention is the new chunk options in Knitr in recent years.</a> <a data-t="688">We have added a few of them, but today I'm just going to mention one of them here, which</a> <a data-t="695">is the file option.</a> <a data-t="697">So sometimes you may want to develop code in an external script and include it in code</a> <a data-t="706">chunk.</a> <a data-t="707">Previously, what you could do is to use the chunk option code.</a> <a data-t="713">So for example, here I'm using the chunk option code that takes the value from readlines.script.r.</a> <a data-t="721">So that basically means I want to read an external script and pass the content of that</a> <a data-t="726">script to the code option, and then Knitr knows, okay, I will use the content of that</a> <a data-t="732">script as the content of this code chunk.</a> <a data-t="737">That's the previous way of including external scripts into a code chunk.</a></p>

<p><a data-t="744">Now this can be simplified to the chunk option file.</a> <a data-t="750">So you can set file equals a file path, for example, script.r.</a> <a data-t="755">So basically that's equivalent to using code equals readlines.script.r before.</a> <a data-t="762">And this file option also supports a vector of file paths.</a> <a data-t="766">For example, you can pass two scripts to the file option and the file option will read</a> <a data-t="773">both of them into the code chunk.</a></p>

<h3>New engines in Knitr</h3>

<p><a data-t="778">The fourth thing I want to highlight is some new engines in Knitr.</a> <a data-t="785">The first one is the comment engine.</a> <a data-t="787">You can use the comment engine to comment out any content.</a> <a data-t="792">And the content can contain basically anything, including code chunks or inline R expressions</a> <a data-t="801">or other paragraphs or anything.</a> <a data-t="803">So basically there's one rule that you have to remember.</a> <a data-t="808">If the comment contains like three backticks, and then for the comment engine, you have</a> <a data-t="816">to start with four backticks.</a> <a data-t="818">So basically the rule is that if your comment contains N backticks, then the comment engine</a> <a data-t="825">needs to start with N plus one backticks.</a> <a data-t="828">So if the content is commented out, it will not be included in the output.</a></p>

<p><a data-t="835">The second engine is the verbatim engine, and this can be very useful when you want</a> <a data-t="841">to write tutorials on rmarkdown.</a> <a data-t="847">You may want to display some content verbatim, especially the code chunks.</a> <a data-t="852">When you want to display a code chunk verbatim, you can use the verbatim engine.</a> <a data-t="859">And the rule is the same.</a> <a data-t="860">So for example, here I'm including a code chunk with three backticks, and then the verbatim</a> <a data-t="866">engine needs to start with four backticks.</a> <a data-t="870">For the verbatim engine, depending on the language of the content, you can set a syntax</a> <a data-t="880">highlighting language through the chunk option lang.</a> <a data-t="885">So for example, here I'm using lang equals Markdown.</a> <a data-t="888">That means the verbatim content here is Markdown code, but this lang option is optional.</a> <a data-t="899">So it depends on whether you want syntax highlighting in your output for the verbatim content.</a></p>

<p><a data-t="906">Previously, I have introduced some old ugly hacks on how to show content verbatim, and</a> <a data-t="914">now you can forget all of them.</a> <a data-t="920">And here, I just, yeah, I don't want to explain this anymore.</a> <a data-t="925">This is just too ugly.</a> <a data-t="927">Use the verbatim engine and forget about the old ugly hacks.</a></p>

<blockquote><a data-t="927">Use the verbatim engine and forget about the old ugly hacks.</a></blockquote>

<p><a data-t="933">The third engine that I want to show is the embed engine.</a> <a data-t="939">You can use the embed engine to embed external files, text files actually.</a> <a data-t="947">You can embed external text files and display their content verbatim.</a> <a data-t="951">So this is similar to the verbatim engine, but you can use the file option to embed external</a> <a data-t="961">files.</a> <a data-t="962">So you don't have to write the content in the chunk body.</a> <a data-t="967">You just point the file option to some file paths.</a> <a data-t="971">And alternatively, you can write file paths in the chunk body of the embed engine.</a> <a data-t="977">For example, you can write quote foo.r, quote bar.r.</a> <a data-t="983">So the second way has a tiny advantage.</a> <a data-t="987">That is, if you write in the chunk body, if you write the file paths in the chunk body,</a> <a data-t="994">your editor, for example, RStudio, may be able to autocomplete the file paths.</a></p>

<p><a data-t="1006">The last engine that I want to show here is the exec engine.</a> <a data-t="1016">So this engine allows you to execute an arbitrary system command on a file that contains the</a> <a data-t="1025">content of the chunk.</a> <a data-t="1037">So that command can be specified through the chunk option command.</a> <a data-t="1043">For example, I have an exec engine here, and I'm using the command equals bash.</a> <a data-t="1052">That means I want to run the command bash on the command that I specified in the chunk</a> <a data-t="1062">body.</a> <a data-t="1063">So basically, when Knitr sees an exec chunk, it will write the chunk body into a file and</a> <a data-t="1072">execute a command on that file.</a> <a data-t="1076">So for this bash example, actually, you don't have to use the exec engine.</a> <a data-t="1081">You can simply use the existing bash engine in Knitr.</a> <a data-t="1085">So these two code chunks are pretty much equivalent.</a></p>

<p><a data-t="1091">There have existed a lot of command-based engines in Knitr, for example, awk, or go,</a> <a data-t="1097">or perl, or zshell.</a> <a data-t="1100">From now on, we no longer need to add a new engine for every single command.</a> <a data-t="1106">You can basically use the exec engine to execute any command.</a> <a data-t="1112">And by the way, the exec engine is also extensible.</a> <a data-t="1117">If you're interested, you can go back and check out the example through the link here.</a></p>

<p><a data-t="1126">For those who are curious about how the exec engine works, basically, Knitr will execute</a> <a data-t="1135">a command of the form command args1 args2, where args is a file path, and this file contains</a> <a data-t="1150">the content of the code chunk.</a> <a data-t="1153">And args1 and args2 are optional command-line arguments that you may specify through the</a> <a data-t="1159">chunk option engine.ops.</a> <a data-t="1161">So here I have an example that executes the grep command with a few command-line arguments</a> <a data-t="1171">dash i, which means case-insensitive matching.</a> <a data-t="1176">And then the pattern is he.</a> <a data-t="1179">And I want to execute grep on the four lines of text here.</a> <a data-t="1184">So, you know, he will, if you match he in case-insensitivity, it will match the first</a> <a data-t="1191">line, hellos, and the third line, since it has the, which contains he.</a></p>
