<h3>Smart app design</h3>

<p><a data-t="415">So let's talk about how these pillars help us accomplish that. <p><a data-t="447">And as well-intentioned as we are with this approach, sometimes, you know, you build a proof of concept and you end up with something like this, which is maybe somewhere in between, right?</a> <a data-t="456">We have a happy customer, but we've built, like, a tool that has really maybe limited us in how we can scale.</a> <a data-t="467">And I often fall victim to this, and I think it's okay for us to say that, you know, while it's great if our proof of concept scales into our MVP, it's not a necessity. <h3>APIs and internal packages</h3>

<p><a data-t="511">So if design is about planning for scale, the next question is, how do we make that scaling possible?</a> <a data-t="516">And that's where APIs and packages come into play.</a> <a data-t="523">So, you know, I'm sure you're all familiar with, there's a bunch of different variants of it, but first time you write code, you write it. Second time, you copy it. Third time, turn it into a function.</a> <a data-t="531">So APIs and packages are where we implement that, right?</a> <a data-t="539">So it's our place to write code once and make it trustworthy and usable throughout all our environments.</a> <a data-t="544">And internal packages are kind of the helper function or where we kind of house our helper functions that developers use day in and day out, again and again.</a> <a data-t="556">So our approach at Suffolk is to maintain packages in both R and Python that have feature parity across both of them.</a> <a data-t="563">And that ensures that, you know, developers in each of their languages have the tools available to them that they need.</a></p>

<p><a data-t="571">So kind of a real world example of this, you know, we use Databricks as a data warehouse.</a> <a data-t="574">If you've ever seen, like, the Posit Connect cookbook on implementation across, you know, the Python version versus the R version, the mechanisms are similar, but the implementation is quite different.</a> <a data-t="590">So, you know, if I was a Python developer and I went to go to R, at best I might be a little confused. At worst, I might just shut it down and go back to Python.</a> <a data-t="607">So our thought was, you know, was alluded to, like, while there is a lot of similarities, can we make them even more similar?</a> <a data-t="611">So we wrappered those both in functions internally so they're exactly the same across both.</a> <a data-t="618">And that proved to be really successful.</a></p>

<p><a data-t="623">So we took that, looked across, like, the ecosystem for kind of other similar examples and started wrappering some other packages as well.</a> <a data-t="630">So, you know, responses from Vetiver, reading, writing, setting permissions on pins, and most recently, ChatList and Elmer, the interface.</a> <a data-t="637">The other additional benefit of this is because these are internal packages, we can make very opinionated decisions about the functions we expose and the values that we set in them.</a> <a data-t="649">So, you know, model choices, where we write pins, those kind of things, we can remove away and make sure we have a standard implementation.</a> <a data-t="658">But the other ancillary benefit we've seen to this is that, you know, this dual package strategy has really kind of enabled our aspiring polyglot developers to be more comfortable trying their less familiar language because it's one less bit of cognitive load that they have to worry about.</a></p>

<p><a data-t="679">So if packages are kind of our helper functions inside of an app, APIs are our super functions that make that functionality available anywhere.</a> <a data-t="697">As a best practice, you've probably seen this before, we've implemented this call twice, so as a good developer, maybe I move it to a function,</a> <a data-t="703">and kind of the next step we take is the thought of, like, okay, does this function meet two criteria?</a> <a data-t="709">One, is there a value in it besides in this app itself? And do we envision a roadmap of it that makes it — that there's additional complexity that, you know, has the chance to grow, probably would benefit from individual developer focus.</a></p>

<p><a data-t="730">And the idea is that, you know, it's the same input, same outputs, but it's callable from any environment, whether it's R or Python or even somewhere beyond.</a> <a data-t="736">The other benefit of it is that at that point, then, you no longer need to be as concerned about the underlying mechanisms once it's removed from the app.</a> <a data-t="744">You just kind of need to know the schema and the end point and hopefully functionality continues to work.</a></p>

<p><a data-t="749">So this is a real example that we did again where we had, you know, a call built into the app. We extracted it to an API end point deployed on Connect.</a> <a data-t="760">It was originally an R model deployed with Vetiver and </a><a href="/software/plumber/">plumber</a><a data-t="760">.</a> <a data-t="767">It eventually migrated to a Python model deployed with Vetiver and FastAPI,</a> <a data-t="774">and then eventually the complexity of it was a FastAPI that not only used Vetiver but other business logic embedded in it.</a> <a data-t="779">But the key feature is that the apps themselves that were using it would never have to change, right? <h3>Team scalability and contracts</h3>

<p><a data-t="794">So we can have smart design and technical separation, but without a way to work in parallel, we'd still be bottlenecked.</a> <a data-t="806">So how we unblock teams isn't really magic. <p><a data-t="827">So for us, APIs are where those boundaries are defined. They're the agreement that says, you know, you hand me this, I will return that to you,</a> <a data-t="838">and it doesn't matter if the API is like half-baked, hard-coded at the start with, or in various stages of development.</a> <a data-t="844">The important part is we have an agreement so everyone can continue to move forward on their own.</a> <a data-t="852">And in that sense, every kind of API becomes its own mini-MVP,</a> <a data-t="857">which, you know, again, as long as we can agree on the schema, the payload, and the endpoints, it can continue to iterate over time,</a> <a data-t="864">and those other downstream teams can wire up to it knowing that it will evolve without breaking.</a></p>
