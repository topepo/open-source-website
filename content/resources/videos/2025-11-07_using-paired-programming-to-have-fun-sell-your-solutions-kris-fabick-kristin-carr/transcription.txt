<h3>Pair programming strategies</h3>

<p><a data-t="505">Okay, so we were convinced, and we're ready to tackle this.</a> <a data-t="509">So how do we do pair programming?</a> <a data-t="511">Before we get into strategies for pair programming,</a> <a data-t="513">let's talk about what it's not,</a> <a data-t="515">in case you've heard this term before.</a> <a data-t="517">So the first thing is, pair programming</a> <a data-t="518">is not backseat driving.</a> <a data-t="521">The second thing is, it's not 24-7 grind.</a> <a data-t="523">This isn't sitting next to each other,</a> <a data-t="525">looking at the screen for eight hours a day.</a> <a data-t="527">It's also not keyboard hogging.</a> <a data-t="530">So it's sharing the ideas, sharing the coding, all of it.</a> <a data-t="534">And then, lastly, it's not nap time.</a> <a data-t="537">So this is coming to the problem,</a> <a data-t="539">engaged with that mindset, ready to take on the difficulty.</a> <a data-t="542">So now that you know what it's not,</a> <a data-t="544">I'm gonna hand it back to Chris</a> <a data-t="545">to talk about some of the strategies</a> <a data-t="546">that you can use to use pair programming in your projects.</a></p>

<p><a data-t="550">So there are a lot of approaches to pair programming.</a> <a data-t="553">The most common one is called driver navigator.</a> <a data-t="555">It is just what it sounds like.</a> <a data-t="556">One person is driving at the keyboard.</a> <a data-t="557">The other person is helping navigate,</a> <a data-t="559">looking for speed bumps, left turns ahead,</a> <a data-t="562">helping you get through that.</a></p>

<p><a data-t="564">A second approach is called ping pong.</a> <a data-t="566">This is more common in software engineering,</a> <a data-t="568">but I think it's useful for data science as well,</a> <a data-t="570">especially if you have a lot of functions in your code.</a> <a data-t="572">Basically, one person starts with the idea for the function,</a> <a data-t="575">the second person will implement the code for that function,</a> <a data-t="578">and then the first person will go back</a> <a data-t="579">and refactor that function,</a> <a data-t="581">and then the next time they switch roles.</a></p>

<p><a data-t="583">I wanna introduce a really important term here</a> <a data-t="585">called refactoring.</a> <a data-t="586">This is a process I've sort of always done,</a> <a data-t="588">but I didn't actually know it had a name for it.</a> <a data-t="591">It's the process of making the code more readable,</a> <a data-t="593">maintainable, and efficient.</a> <a data-t="595">So you've probably done it too,</a> <a data-t="596">but it's really important to call out here</a> <a data-t="597">because it's a hugely critical component</a> <a data-t="600">of pair programming.</a></p>

<p><a data-t="602">A third approach is called unstructured synchronous.</a> <a data-t="605">Unstructured here does not mean</a> <a data-t="606">like just do whatever you feel like,</a> <a data-t="608">and like there's no plan.</a> <a data-t="610">It just means that it's not so formal.</a> <a data-t="612">You don't have to like your turn, my turn, your turn.</a> <a data-t="615">You can lean into your strengths and your interests</a> <a data-t="617">a little bit more.</a> <a data-t="618">Synchronous means you're sharing time and space.</a> <a data-t="621">So a program, a problem being solved with this approach</a> <a data-t="625">might look something like this,</a> <a data-t="627">where you are working on different project components</a> <a data-t="629">together, you're sharing time and space,</a> <a data-t="631">and you are doing that refactoring together.</a></p>

<p><a data-t="634">A fourth approach is called unstructured asynchronous.</a> <a data-t="636">So same meaning of unstructured,</a> <a data-t="638">asynchronous here just meaning</a> <a data-t="639">you're not sharing time and space.</a> <a data-t="641">And I affectionately call this parallel programming</a> <a data-t="644">because I am a dad and I have to have my dad jokes.</a></p>

<p><a data-t="649">So this approach could look like this,</a> <a data-t="651">where you have one person doing</a> <a data-t="653">maybe these two project components,</a> <a data-t="654">another person doing this project component,</a> <a data-t="656">and maybe they're doing them at the same time,</a> <a data-t="658">but they are not doing them together.</a> <a data-t="659">They're not sharing time and space.</a> <a data-t="661">And I know you think that I'm totally cheating now</a> <a data-t="663">because now I'm not even talking about pair programming.</a> <a data-t="665">You're like, that's just regular programming,</a> <a data-t="667">except we're not done with the project, right?</a> <a data-t="669">Because there's this really important step of refactoring.</a> <a data-t="673">And if you will pause and go back and refactor together,</a> <a data-t="676">maybe even all switching who like takes the lead,</a> <a data-t="680">you're gonna achieve the pairing that you need</a> <a data-t="683">to make this successful.</a> <a data-t="684">The goal at the end of whichever approach</a> <a data-t="686">or approaches you choose is that</a> <a data-t="688">everybody can ride the bike.</a></p>
