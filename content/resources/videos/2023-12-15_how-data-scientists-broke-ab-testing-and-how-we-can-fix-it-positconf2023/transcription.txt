<h3>Non-inferiority test designs</h3>

<p><a data-t="331">So the first approach is non-inferiority test designs, which are not new and not esoteric, but I think they're slept on in the A-B testing context a little more than they ought to be.</a> <a data-t="343">You'll notice the picture here is a guardrail, and that's the metaphor to keep in mind.</a> <a data-t="349">The main idea of this approach is that instead of testing whether the new version of the site is better than the current one, we're just going to test whether it's not worse by some margin.</a> <a data-t="359">And that margin is the delta in the red box there, and we call that the inferiority margin.</a></p>

<p><a data-t="365">Well, when you have a conversation about what these margins ought to be, you are forcing conversations about the sort of things that motivate these, like, launch on neutral type decisions about, you know, well, how much do you want to risk to launch this thing?</a> <a data-t="380">How much do you believe in it?</a> <a data-t="381">How quickly are you going to iterate on it and improve on it after it's launched, right?</a> <a data-t="384">And stakeholders can kind of start to give you, like, meaningful answers to these questions instead of coming up with a fake effect size that they want to find.</a></p>

<p><a data-t="392">And you can start to power against this, like, well, any positive effect is good type of scenario.</a> <a data-t="399">And you can start to have a conversation about, well, look, if you run a test for three weeks and you want good power against any feature that, like, isn't losing us money, then you may have to accept some small risk that it actually will lose us, like, say, 1.5% conversion drop, right?</a> <a data-t="418">And maybe that's an acceptable risk.</a> <a data-t="420">Maybe it's not an acceptable risk.</a> <a data-t="421">But it's an assessment of risk that usually they can reason about a little bit better.</a> <a data-t="425">And you end up with a more productive conversation.</a></p>

<h3>Value of information approach</h3>

<p><a data-t="433">But there's another method that I like that I want to talk to you about.</a> <a data-t="437">And this one directly sort of attacks the core problem, the core question that we have in these test design conversations.</a> <a data-t="444">And that is... what's the hurry?</a> <a data-t="447">Right?</a> <a data-t="447">Like, why do we not have the patience to run, like, an adequately powered A-B test in this organization?</a></p>

<p><a data-t="457">And we sort of know the answer.</a> <a data-t="460">Like, running a test a long time is costly.</a> <a data-t="462">There's an opportunity cost of time.</a> <a data-t="462">This gets back to that, like, the nature of, like, the road mapping and the path dependence amongst features.</a> <a data-t="467">If we're waiting a long time for a test of what feature won, that is going to hold up feature 2, which depends on the launch or the non-launch of feature 1 and what happens.</a></p>

<p><a data-t="478">There's the opportunity cost of sampling and randomization and tests.</a> <a data-t="482">By construction of an A-B test, a bunch of users are not getting the best version of your site.</a> <a data-t="487">If you've ever worked with bandits or seen bandits, you've seen them trying to approach this problem.</a> <a data-t="492">And then lastly, there's just, like, the day-to-day maintenance cost of tests.</a> <a data-t="495">Having a bunch of tests running on the site at once is... you know, engineering effort and, like, code complexity and data storage and whatever.</a></p>

<p><a data-t="506">So the question is... if we know about all these costs and we know they affect how decision makers want to run tests, why aren't we incorporating them into test designs?</a> <a data-t="522">And this is where value of information, experimental designs, can help.</a></p>

<p><a data-t="526">So we know the time we spend running a test longer is costly.</a> <a data-t="529">We know the extra data we get from running a test longer is valuable.</a> <a data-t="533">If we can quantify the cost and we can quantify the value, that should be telling us how long we should be running a test.</a> <a data-t="539">If the value of more data exceeds the cost of more data, you should keep getting data.</a> <a data-t="543">And if the cost of more data exceeds the value of more data, then you should stop getting data.</a></p>

<p><a data-t="551">The longer our tests run, the more data we get, the more valuable that data is.</a> <a data-t="558">Experimental data is less valuable when you have a lot than when you have a little.</a> <a data-t="561">Costs increase as you wait to get that data.</a> <a data-t="564">If it's more valuable to get the data than it is costly, you should get the data.</a></p>

<p><a data-t="571">How do we think about the value of data, though?</a> <a data-t="578">Well, before we run an experiment, before we have any data, we know very little about what the conversion lift of a new feature might be.</a> <a data-t="583">It could be very negative.</a> <a data-t="585">It could be very positive.</a> <a data-t="587">If we make a decision based on our best information now, we could end up launching an awful feature or failing to launch a really, really good one.</a> <a data-t="597">And then as we collect data, we have a better idea of what that conversion lift might be, the range of values it might take narrows.</a> <a data-t="604">We may make an incorrect guess now, but our guess is likely to be wrong by less.</a></p>

<p><a data-t="609">And it turns out that you can put a value on being probably less wrong.</a> <a data-t="617">And again, if that value exceeds the cost of the time it takes to get that data, you should be getting it.</a></p>

<blockquote><a data-t="609">And it turns out that you can put a value on being probably less wrong.</a></blockquote>

<p><a data-t="624">So how do we actually, say, like, compute that value, that value of being potentially less wrong?</a> <a data-t="629">It turns out it has a name.</a> <a data-t="631">It's called the expected value of sample information.</a> <a data-t="635">And I'm gonna show you a really simplified way of how you might estimate it.</a></p>

<p><a data-t="639">So we start with a prior over what we think the conversion lift might be.</a> <a data-t="641">Relatively wide range.</a> <a data-t="641">We don't know very much.</a> <a data-t="641">It could be negative.</a> <a data-t="642">It could be pretty positive.</a> <a data-t="644">We are going to draw a bunch of values out of that prior, a bunch of potential lifts out of that prior.</a></p>

<p><a data-t="651">For each of those lifts that we draw, we're gonna simulate an experiment.</a> <a data-t="657">Let's say we're interested in, like, hey, what if I want to get two more weeks of data?</a> <a data-t="660">So I can simulate a two-week experiment, control and treatment with that lift that I drew out of the prior.</a> <a data-t="667">That data and that prior generates a posterior with a new opinion about what those lifts might be.</a></p>

<p><a data-t="676">Each of those posteriors may change my mind by... may not change my mind at all based on what I was gonna do under the prior.</a> <a data-t="682">They may change my mind a lot based on what I was gonna do under the prior.</a> <a data-t="686">If an experiment generates data... is likely to generate data that changes my mind by a lot, that was a valuable experiment to run.</a> <a data-t="694">If the prior had never had any hope of changing my mind, there was no point in doing it.</a></p>

<p><a data-t="700">And so we run all these simulations, we get all these priors, we get all the posteriors, we get the values, we average those out.</a> <a data-t="705">That's an estimate of this expected value of getting this extra data.</a> <a data-t="711">Even better, this is like an inherently kind of sequential process.</a> <a data-t="715">It's just posterior updating.</a> <a data-t="716">So you can do it over and over again.</a> <a data-t="718">After you get some data, you're really just asking, what's the value of some more data?</a></p>

<p><a data-t="725">And this changes the core decision in an A-B test from... is B better than A?</a> <a data-t="729">As though that's a really hard problem to figure out.</a> <a data-t="731">To... should I stop getting data?</a> <a data-t="732">Or should I keep getting data?</a> <a data-t="733">There's a good fit for A-B tests.</a> <a data-t="737">Because we don't have to recruit subjects for an A-B test.</a> <a data-t="739">We just have to wait.</a></p>

<p><a data-t="745">And then once more data isn't worth it, you just launch the best observed variant.</a> <a data-t="750">The inference problem, the statistical significance problem is irrelevant at that point.</a> <a data-t="752">This is the best information we have.</a> <a data-t="754">And it's not worth getting more.</a></p>

<p><a data-t="760">And it turns out, like, I find this a really compelling way to think about A-B tests with decision makers.</a> <a data-t="767">It directly gets at the core concepts that they think about when they want to make a decision.</a> <a data-t="772">Cost.</a> <a data-t="773">Benefit.</a> <a data-t="774">Time.</a> <a data-t="775">Risk.</a> <a data-t="776">Everything's in dollars.</a> <a data-t="777">The outputs are in dollars.</a> <a data-t="778">They're not like... error rates.</a></p>

<p><a data-t="783">So it's more complicated than traditional testing.</a> <a data-t="791">But it's tractable for a pretty broad range of the kinds of A-B tests I've run in my experience.</a> <a data-t="801">But I've built whole analytics engines on it with R and </a><a href="/software/shiny-r/">Shiny</a><a data-t="801"> and worked with product managers on it who have found it... it gels really well with how they want to make decisions and kind of liberates them into being able to... oh, I can figure out how a test should work with, like, dollar outputs.</a></p>
