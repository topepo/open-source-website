<h3>When to use Shiny for Python: reactivity</h3>

<p><a data-t="570">So another time that you might consider using Shiny for Python apps</a> <a data-t="574">is when you need reactivity.</a> <a data-t="578">Now, what is reactivity?</a> <a data-t="580">Well, to take a step back,</a> <a data-t="581">one thing I think we can all agree on</a> <a data-t="584">is that machine learning is computationally expensive.</a> <a data-t="588">Whether we're doing training, fine tuning, making inferences,</a> <a data-t="592">a lot of the things that we're doing take a lot of computational power.</a> <a data-t="596">And so we don't want an app that is going to run top to bottom</a> <a data-t="601">every time a user changes something.</a> <a data-t="603">Because sometimes, in fact, I might even venture to say oftentimes,</a> <a data-t="607">you are going to be doing things that are computationally expensive</a> <a data-t="611">when building machine learning apps.</a></p>

<p><a data-t="614">So reactivity basically refers to the fact</a> <a data-t="616">that we are going to minimally re-render our apps.</a> <a data-t="620">Only things that absolutely need to change and be re-run</a> <a data-t="623">are going to be changed in response to user input.</a></p>

<p><a data-t="628">So here's the world's cutest app.</a> <a data-t="631">This is looking at a computer vision model</a> <a data-t="634">and it's basically allowing you to test out</a> <a data-t="637">how different parts of the image are important to the model</a> <a data-t="641">by seeing what happens when you completely cover it up.</a> <a data-t="644">By the way, the takeaway from this screenshot</a> <a data-t="647">is that if the computer vision model can't see the fluffy corgi butt,</a> <a data-t="651">it is less confident that the image is a corgi.</a></p>

<p><a data-t="657">This has a bunch of user inputs.</a> <a data-t="660">For instance, it allows users to upload an image</a> <a data-t="662">that they can then use with this app.</a> <a data-t="665">When someone uploads an image,</a> <a data-t="667">pretty much everything on the right-hand side of this app has to update, right?</a> <a data-t="671">We need to update the picture.</a> <a data-t="672">We need to update the occluded picture.</a> <a data-t="674">And we need to make predictions</a> <a data-t="676">both for the unoccluded and the occluded image.</a></p>

<p><a data-t="680">However, if you look at the other inputs,</a> <a data-t="682">we can also decide what is going to be occluded in the image,</a> <a data-t="685">like what range of pixels are going to be removed.</a> <a data-t="688">And when we change that,</a> <a data-t="690">we don't need to update the original image</a> <a data-t="693">and we don't need to update the model's original prediction</a> <a data-t="697">because nothing about our user input actually affects that.</a> <a data-t="701">And so we're saving the computational expense of updating that</a> <a data-t="705">by using reactivity by saying,</a> <a data-t="708">only re-render things that actually need to be re-rendered.</a></p>

<p><a data-t="714">Now, one thing that is really nice about Shiny</a> <a data-t="717">is not just that you get reactivity</a> <a data-t="719">because that's not unique to Shiny.</a> <a data-t="721">One thing that is unique to Shiny is this inferred reactivity.</a> <a data-t="726">Inferred reactivity basically refers to the idea</a> <a data-t="728">that Shiny is deciding what needs to be re-rendered in your app</a> <a data-t="732">based on how you wrote it.</a> <a data-t="734">You don't have to think about that at all.</a></p>

<p><a data-t="738">In order to get a deeper idea of what I mean,</a> <a data-t="740">let's look at this really silly demo app.</a> <a data-t="742">Here we have an app that is plotting a scatterplot</a> <a data-t="745">based on a sample of data.</a> <a data-t="747">It lets users change various aesthetics.</a> <a data-t="749">And then at the bottom left,</a> <a data-t="750">it gives you a summary of the sample of your data.</a></p>

<p><a data-t="754">When we have this app,</a> <a data-t="756">we basically have a computational graph</a> <a data-t="759">that tells you about how all the different parts of the app</a> <a data-t="762">are related to each other.</a> <a data-t="763">For instance, we start off our app with some data.</a> <a data-t="766">And then if we press the action button,</a> <a data-t="769">we take a sample of that data.</a> <a data-t="772">From that sample, we might rely on the fact</a> <a data-t="774">that users are going to choose the color</a> <a data-t="776">and what's on the y-axis of our scatterplot.</a> <a data-t="779">And using that, we then build our scatterplot.</a> <a data-t="783">Then of course, we have that summary table,</a> <a data-t="785">which doesn't rely on the y variable for our plot</a> <a data-t="789">or the color of the dots in our plot,</a> <a data-t="791">but does rely on the sample</a> <a data-t="792">that we've randomly taken from our data.</a></p>

<p><a data-t="795">Now, we just built a very simple computational graph.</a> <a data-t="798">And you can see that this tells us</a> <a data-t="800">how the different parts of the app relate to each other.</a> <a data-t="802">For instance, if I update the dot color of my scatterplot,</a> <a data-t="806">I have no need to update my summary table.</a> <a data-t="809">And that's reflected in my graph.</a> <a data-t="811">Only things downstream of changes that we make</a> <a data-t="815">need to be updated when that input is updated.</a></p>

<p><a data-t="818">Now, this is an incredibly simple app</a> <a data-t="821">and the computational graphs for your apps</a> <a data-t="824">as they grow in complexity is going to become</a> <a data-t="828">at some point unmanageable</a> <a data-t="831">if you had to manage this yourself.</a> <a data-t="833">But the good news is that Shiny manages it for you.</a> <a data-t="837">So instead of you having to design</a> <a data-t="840">and specify the computational graph for your app,</a> <a data-t="843">Shiny is inferring what this computational graph should be.</a> <a data-t="846">So you get to benefit from reactivity</a> <a data-t="849">without having to think about</a> <a data-t="850">the computational graph explicitly.</a></p>

<p><a data-t="852">And I know as I build more complicated apps,</a> <a data-t="855">I am definitely going to mess up</a> <a data-t="857">my computational graph at some point.</a> <a data-t="859">So I'm very grateful that Shiny</a> <a data-t="861">doesn't even make me think about it.</a> <a data-t="863">And you can see with even this Corgi app</a> <a data-t="865">that looks pretty simple when you look at it,</a> <a data-t="867">the computational graph gets more and more complex</a> <a data-t="871">as you add various functionality.</a></p>
