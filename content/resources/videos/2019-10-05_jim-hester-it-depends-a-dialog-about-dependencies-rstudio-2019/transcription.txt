<h3>Weighing the pros and cons</h3>

<p><a data-t="370">So what should we do instead? So I think really when you're deciding whether or not to have</a> <a data-t="377">a dependency in your package or to use a certain package because it has so many dependencies,</a> <a data-t="382">you need to weigh some things. And this, how this scale works really depends on your personal</a> <a data-t="393">use case and the users of your code. So if you're, so if you have more dependencies,</a> <a data-t="401">you gain more features, more bug fixes, more testing of those features because other users</a> <a data-t="409">are using your dependencies, but you, but if you don't, if you have fewer dependencies,</a> <a data-t="417">you instead have less installation time, less, you use less disk space, you potentially</a> <a data-t="423">have less breakage in your code because you don't depend on an external package. So having</a> <a data-t="430">more dependencies gives you more developer time, so it takes you less, you as the developer</a> <a data-t="436">less time, but you lose generality.</a></p>

<p><a data-t="443">So really I think you need to consider your</a> <a data-t="443">users as like the, as when you're making these decisions. If the package you're writing</a> <a data-t="449">or the project you're using is really meant for other package developers to include your</a> <a data-t="453">package in theirs, the installation time becomes more of a factor. So it's more costly if other</a> <a data-t="463">packages are trying to depend on your package. And in addition, smaller, more focused packages</a> <a data-t="470">are easier to include and depend on than larger, broader packages. And in this context,</a> <a data-t="477">I think stability is often more important than features. It's more important if the</a> <a data-t="482">package is stable over time than if it continually gains new additional features.</a></p>

<p><a data-t="489">In contrast,</a> <a data-t="489">if your users are primarily other data scientists or statisticians that are loading your package</a> <a data-t="496">in R scripts, the installation time in that case is very cheap.
