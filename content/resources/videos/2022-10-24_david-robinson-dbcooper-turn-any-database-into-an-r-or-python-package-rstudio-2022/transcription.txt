<h3>How DbCooper works</h3>

<p><a data-t="344">So that's why I think there's a missing puzzle piece in between DBI for creating a connection</a> <a data-t="349">to a database, and Dbplyr for transforming and exploring those tables.</a> <a data-t="354">That's where DbCooper comes in.</a> <a data-t="358">It turns a database into an opinionated collection of functions to let us explore it really fluidly,</a> <a data-t="364">kind of as if it were data that were in memory.</a></p>

<p><a data-t="368">DbCooper takes, you only need to do one function call to set up DbCooper.</a> <a data-t="373">It's DbcInit.</a> <a data-t="374">You pass it the connection object, and you give it a prefix.</a> <a data-t="378">If you've worked with a lot of R packages, you know how useful it can be to have a shared</a> <a data-t="382">prefix.</a> <a data-t="383">DbcInit stores and manages the connection globally.</a> <a data-t="386">It generates a prefixed function for important tasks you would do with that database, and</a> <a data-t="391">really importantly, it generates autocomplete-friendly accessors for tables.</a></p>

<p><a data-t="396">So I'll show how each of those tasks would be achieved in DbCooper.</a> <a data-t="401">Once you've run DbcInit, instead of doing DbListTables, I would just do lamin underscore</a> <a data-t="407">tbl.</a> <a data-t="408">So that prefix that I passed it turned into a prefix for the function.</a> <a data-t="412">LaminList would return the list of tables that are in the database.</a></p>

<p><a data-t="426">So lamin, if I want to access just one table, I use lamin underscore tbl.</a> <a data-t="431">So I say lamin tbl, then I can pass it the name of the table, get that one back.</a> <a data-t="436">If I wanted to run a query, I do lamin underscore query and pass the SQL.</a> <a data-t="443">And if I wanted to execute, I'd use lamin underscore execute.</a> <a data-t="445">So one thing to notice about this is it's taking the connection object entirely out</a> <a data-t="450">of the picture, and it's focusing on an opinionated set of tasks that someone might want to do</a> <a data-t="456">with a database.</a></p>

<p><a data-t="459">My favorite feature about DbCooper is autocomplete.</a> <a data-t="463">So whenever I've worked with a database, it can be hard to remember what the tables are,</a> <a data-t="466">and I get kind of tired of constantly referring back to the documentation or even to a list</a> <a data-t="472">function to see what they are.</a> <a data-t="475">I would maybe run into typos, et cetera, and it just doesn't feel anything like working</a> <a data-t="481">with data that's in memory.</a></p>

<p><a data-t="483">So DbCooper creates one function for each of the tables in your database so that you</a> <a data-t="490">can access them with autocomplete.</a> <a data-t="492">So it's a much more fluid experience where I can say, if I go into a table, I can just</a> <a data-t="496">start typing it and see, oh, yeah, here are the tables that I'm...</a> <a data-t="499">I can't remember exactly what this table in my database is called, but I use autocomplete</a> <a data-t="502">to help me out.</a> <a data-t="503">One thing I like about that is it makes something like a complicated join much more concise</a> <a data-t="512">and much easier to read.</a> <a data-t="513">For me, again, it really means I don't have to remember that I'm working with SQL tables.</a></p>

<p><a data-t="520">Using DbPlyr with remote tables can feel a lot like using Dplyr with local tables.</a></p>

<h3>Building a database-specific package</h3>

<p><a data-t="769">This is a package I put up just as an example of a package that could be created.</a> <a data-t="773">We pick the database.</a> <a data-t="775">Now let's wrap it with a package.</a> <a data-t="777">And the way that we go about doing that is three steps.</a> <a data-t="781">First we write database-specific connection code.</a> <a data-t="784">So anything in terms of setting up credentials, et cetera, that's going to live inside this</a> <a data-t="788">package.</a></p>

<p><a data-t="793">Then critically, in the .onload and .unload functions, those are functions that run every</a> <a data-t="802">time a package is loaded or unloaded, we're going to create one of those database connections</a> <a data-t="809">and then use DbC init.</a> <a data-t="812">So this means that as soon as somebody loads in this package, it's going to populate the</a> <a data-t="817">package's namespace with functions that start with that prefix stack.</a></p>

<p><a data-t="825">Now every database is different, and the ways that we explore them are different, too.</a> <a data-t="830">So the next step would be adding database-specific verbs.</a> <a data-t="835">Something that I love to do with Stack Overflow data is look at, say, patterns of trends of</a> <a data-t="840">tags over time.</a> <a data-t="842">I'm really interested in can I aggregate at the tag level?</a> <a data-t="845">That requires joining together a couple tables in the database and doing a little bit of</a> <a data-t="849">custom logic.</a> <a data-t="851">That kind of custom database-specific logic or problem-specific logic or company-specific</a> <a data-t="856">logic is a really good fit for a package that uses DbCooper because it's going to mean that</a> <a data-t="861">when the person using this package actually does this analysis, everything's going to</a> <a data-t="867">be set up for them.</a></p>
