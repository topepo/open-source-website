<h3>Model workflow overview</h3>

<p><a data-t="260">The workflow for our safety model is fairly standard and can be divided into two distinct processes, model training and model inference.</a> <a data-t="267">Both processes utilize an </a><a href="/software/odbc/">ODBC</a><a data-t="267"> connection to Databricks, which is our data warehouse.</a> <a data-t="273">We employ the Posit suite of tools to train and deploy the model.</a> <a data-t="280">Once deployed, we can make API calls to the model for inference.</a> <a data-t="280">We perform batch inference every week using a script that gathers data from our Databricks tables, creates the model input,</a> <a data-t="286">makes a request to the model's API endpoint, and writes the predictions back to Databricks in a table.</a> <a data-t="298">Once this process is complete, our Shiny app displays the most recent predictions, allowing end users to access the data.</a> <a data-t="304">Additionally, any projects identified as high risk are highlighted in an emailed report, which includes insights from the model explaining why the project is considered high risk.</a></p>

<p><a data-t="317">The Posit suite offers a comprehensive set of tools that allow us to productionize our model workflow.</a> <a data-t="324">The </a><a href="/software/tidymodels/">tidymodels</a><a data-t="324"> library enables us to train multiple models and select the best one based on hyperparameter tuning.</a> <a data-t="331">The </a><a href="/software/dbi/">DBI</a><a data-t="331"> library allows us to establish an ODBC connection to Databricks, enabling us to read data and write predictions back to the database.</a> <a data-t="337">This connection also supports row-level permissioning in Databricks, ensuring that data access is restricted based on predefined permissions, thereby relieving data scientists from managing data access.</a> <a data-t="356">Vetiver allows us to version our models easily and deploy them to Posit Connect, where they can be served as API endpoints.</a> <a data-t="361">Quarto helps us schedule the model workflow and send out key results via email.</a> <a data-t="367">Finally, we use a Shiny app to present the model results in a user-friendly format for our stakeholders, with the app also deployed to Posit Connect.</a></p>

<h3>Model training with tidymodels</h3>

<p><a data-t="378">In this section, I will walk you through the script we use to train our data.</a> <a data-t="383">While I have a separate script dedicated to generating the training data, I will not be showcasing that script or the actual data set used for this public demonstration.</a> <a data-t="392">However, this provides an excellent opportunity to highlight the utility of the PINS library, which allows us to save a version data set and access it from a separate script.</a> <a data-t="403">This is precisely what I am doing here in this cell.</a></p>

<p><a data-t="410">One of the standout features of the tidymodels library is its comprehensive suite of tools for model training.</a> <a data-t="415">I will guide you through some of the key features we utilize.</a> <a data-t="421">Initially, we generate our train-test split to evaluate the model's performance on an independent data set that the model has not encountered before.</a></p>

<p><a data-t="433">Here, we introduce the concept of a model recipe, which I believe is an unsung hero of the tidymodels process.</a> <a data-t="439">Incorporating a model recipe is advantageous because it enables data transformations within the model workflow itself, thereby minimizing the preprocessing required before feeding data into the model when it is served as an API.</a></p>

<p><a data-t="456">In our script, we flag the project number and week columns as IDs, ensuring that the columns are included in the model workflow but not used as features.</a> <a data-t="469">The HadIncident column, which indicates whether an incident occurred, is assigned the outcome role, signaling to the model that this is the target variable we aim to predict.</a> <a data-t="480">We also handle missing values by removing any rows with NA values for numeric features, and perform one-hot encoding for nominal features.</a> <a data-t="490">It's important to note that unless the one-hot parameter is set to true, the encoding will create N-1 features for N values in a nominal column.</a></p>

<p><a data-t="502">Additionally, we create interaction terms between specific features and perform downsampling to address the imbalance in our training data set.</a> <a data-t="511">This imbalance, characterized by a higher number of project weeks without incidents compared to those with incidents, is obviously favorable from a business and safety perspective,</a> <a data-t="523">because it means that incidents are relatively rare.</a> <a data-t="526">However, without downsampling, the model would suffer from poor recall, struggling to correctly identify project weeks with incidents in the testing set.</a></p>

<p><a data-t="538">We specify two models for training, a random forest classifier and an XGBoost classifier.</a> <a data-t="545">To optimize the models, we use the tune function to specify hyperparameters, which we define shortly later in the script.</a> <a data-t="553">A notable feature of tidymodels workflows is the ability to train multiple models using multiple recipes within the same process by using the workflow set function.</a> <a data-t="565">This allows us to test the performance of various recipes on each specified model and select the best one.</a> <a data-t="570">Although we are using a single recipe in this demonstration, it's worth mentioning the flexibility this feature provides.</a></p>

<p><a data-t="581">We define our k-fold cross-validation strategy for training and specify the range of hyperparameters to test.</a> <a data-t="591">For each hyperparameter, we test five different values evenly distributed across the specified range.</a> <a data-t="597">During the training process, we track performance metrics such as precision, accuracy, recall, and area under the curve.</a></p>

<p><a data-t="612">The workflow map function integrates all the components of our script, including the workflow set, cross-validation specification,</a> <a data-t="620">hyperparameter tuning grid, and performance metrics.</a> <a data-t="624">Running this function initiates the training process.</a></p>

<p><a data-t="630">Based on our previous evaluations, we know that the random forest model outperforms the XGBoost model, so we will skip the step of comparing their performance.</a> <a data-t="639">However, examining the training metrics allows us to assess the model's performance.</a> <a data-t="644">Julia Silgi has published numerous insightful blog posts on iteratively tuning model hyperparameters, which I highly recommend for those starting from scratch.</a> <a data-t="653">For the sake of brevity, I know that the model performed best as a random forest, so we will assume that the best model is the random forest model and proceed.</a></p>
