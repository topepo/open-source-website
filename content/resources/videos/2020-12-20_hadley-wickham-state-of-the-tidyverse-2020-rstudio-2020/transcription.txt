<h3>Tidy evaluation: mistakes and lessons</h3>

<p><a data-t="574">So what I did really want to talk about, though, is a little bit about tidy evaluation, because</a> <a data-t="578">I think we did make some mistakes.</a> <a data-t="581">Just here's kind of a provocative question that someone asked a little while ago.</a> <a data-t="586">Will tidy eval kill the tidyverse?</a> <a data-t="589">I'm pretty confident the answer is no.</a></p>

<p><a data-t="594">But just to kind of, if you haven't seen tidy evaluation before, this is kind of one of</a> <a data-t="598">the challenges of programming with functions from dplyr and ggplot2, for example, is introducing</a> <a data-t="605">indirection.</a> <a data-t="606">In dplyr and ggplot2, normally you provide the name of the variable directly.</a> <a data-t="613">But what do you do if you want the user to supply the name of the variable in a function?</a> <a data-t="619">Well, previously we had this technique where you did bang, bang, enquo, bang, bang, required</a> <a data-t="625">that you kind of learned about the theory of quasi-quotation and enquo, made you think</a> <a data-t="630">about these complicated things called quotiers.</a> <a data-t="633">Now we have a system called embracing, which should allow you to solve the vast majority</a> <a data-t="639">of problems, along with a few other techniques, which means that you don't have to learn the</a> <a data-t="645">theory if you don't want to.</a></p>

<p><a data-t="649">So we've also been working on lots of articles to explain this.</a> <a data-t="653">I think I have these up again at the end if you want to grab them.</a> <a data-t="656">But I wanted to talk about kind of the mistakes we made.</a> <a data-t="659">And I think the first mistake we made is that this problem as a whole was much, much harder</a> <a data-t="664">than we thought.</a> <a data-t="666">And I know there have been at least like five points where I'm like, yes, we finally understand</a> <a data-t="672">how all of this should work.</a> <a data-t="673">And on every point, I was wrong, except for today, and I'm reasonably confident, more</a> <a data-t="681">confident than I've ever been before, basically because we've been doing a much better job,</a> <a data-t="686">I think, of creating this problem-oriented documentation.</a> <a data-t="689">And we can see in that that we can solve the problems that most people are having.</a></p>

<p><a data-t="696">I think the other thing, the other mistake, the next mistake we made is the theory really</a> <a data-t="701">is like beautiful and elegant.</a> <a data-t="704">It is, I think, pretty much unique amongst programming languages, because very few programming</a> <a data-t="709">languages have the combination of first-class environments and computing on the language.</a> <a data-t="715">And this just adds, like, it's sort of really interesting.</a> <a data-t="719">And I think it's really cool.</a> <a data-t="720">And Leonel has worked on this a lot with me, thinks it's really cool.</a> <a data-t="725">But most other people do not think it's really cool.</a> <a data-t="728">So totally, I think it's totally still worth it if you want to learn about the theory,</a> <a data-t="731">because you want to learn more about how these things work.</a> <a data-t="735">But generally, kind of the cost-benefit ratio, if you're like a data scientist just trying</a> <a data-t="738">to solve some problem, the cost-benefit for, like, spending the time to learn the theory</a> <a data-t="742">just wasn't there.</a></p>

<h3>Function life cycles</h3>

<p><a data-t="759">So I think two of the kind of takeaway messages with this, like, I think one, just being aware</a> <a data-t="764">of the problem, I think, is important.</a> <a data-t="768">So that we are now kind of more aware that there are things that, like, we get excited</a> <a data-t="772">about that are, like, really exciting to us, but we know, like, the rest of the R community</a> <a data-t="776">are not going to be very excited about them.</a></p>

<p><a data-t="779">So we're going to try and make it more clear, like, what is the status of various things</a> <a data-t="785">that we're working on?</a> <a data-t="786">Is this something that we think is really cool, but it's pretty experimental?</a> <a data-t="789">Maybe it's going to change radically?</a> <a data-t="791">Or is this something that's, like, we now have kind of grave doubts about, and maybe</a> <a data-t="794">it's on its way out in the future?</a> <a data-t="798">And sort of and think about how can we get feedback on ideas that need more feedback</a> <a data-t="803">without forcing, like, everyone to have to think about those issues.</a></p>

<p><a data-t="808">And one of the ways we're doing that is trying to be more clear about where functions live</a> <a data-t="815">in the kind of the life cycle of a function.</a> <a data-t="818">So what is the life cycle of a function?</a> <a data-t="821">Well, the place where most functions live is this kind of stable life cycle.</a> <a data-t="826">Like, we're pretty confident, like, this is a function, like, it does what it says on</a> <a data-t="831">the can.</a> <a data-t="832">It's useful.</a> <a data-t="833">We don't think it's going to change majorly.</a></p>

<p><a data-t="835">Some functions, like, when we introduce them, we're, like, seems like a good idea.</a> <a data-t="839">You know, I think, like, when I first introduced the when we first started introducing the</a> <a data-t="844">pipe, like, I was, like, well, this seems like a really cool idea, but, like, no one's</a> <a data-t="848">going to understand how it works.</a> <a data-t="851">And it turns out, like, it doesn't actually matter.</a> <a data-t="852">The pipe is useful because it does something.</a> <a data-t="854">It allows you to write code in a really clear way.</a> <a data-t="857">It doesn't actually matter if you don't understand how it works.</a> <a data-t="860">You can still use it.</a></p>

<p><a data-t="862">There are other things that we have kind of so sometimes, you know, we think we have a</a> <a data-t="867">really good idea, and then later on we're, like, hmm, maybe that wasn't such a good idea.</a> <a data-t="874">And so we're starting to label those functions now with questioning.</a> <a data-t="877">We're not sure if those functions are a good idea or not.</a> <a data-t="882">Now, sometimes we'll, like, think about it more, and we'll end up back in stable.</a> <a data-t="885">Other times we'll be, like, oh, okay, we finally figured out a better way to implement this</a> <a data-t="889">function or a better way to solve this problem, and those functions become superseded.</a></p>

<p><a data-t="893">So just to give you kind of an example of this in dplyr 1.0, there were two functions,</a> <a data-t="900">row wise and do, that a lot of people really liked that have been in questioning for a</a> <a data-t="904">while.</a> <a data-t="905">We kind of figured out, like, row wise actually can go back to being stable.</a> <a data-t="909">I figured out why I didn't like it and fixed that problem so it's stable again.</a> <a data-t="913">Do we figured out a better way of solving, and so that becomes superseded.</a> <a data-t="917">So superseded functions I'll talk about a little bit more shortly.</a> <a data-t="920">These are functions, like, we don't think they're the best solution anymore, but they're</a> <a data-t="924">not going away.</a> <a data-t="926">We've got, like, a better approach, but you don't need to worry that if you're using these</a> <a data-t="928">functions that we're going to yank the rug out from under you.</a></p>

<p><a data-t="932">We're not going to the only time a function is when we want to be clear that a function</a> <a data-t="936">is going away.</a> <a data-t="937">We'll tell you it's deprecated, and then when eventually it gets removed altogether, then</a> <a data-t="942">it becomes defunct.</a></p>

<p><a data-t="945">So just to kind of contrast deprecated and superseded a little bit more clearly, a deprecated</a> <a data-t="950">function is, like, clearly on its way out in the near future, which kind of probably</a> <a data-t="956">means in the next year or two.</a> <a data-t="959">You'll be warned when you use it.</a> <a data-t="961">So whenever you use a deprecated function, you will get a warning.</a> <a data-t="965">Only you get that warning, like, once per session.</a> <a data-t="967">I think maybe that's not warning you, like, quite enough.</a> <a data-t="971">We don't want to warn you every time you use it, because if we warn you every time you</a> <a data-t="974">use it, it's basically just as annoying as, like, taking the function away in the first</a> <a data-t="978">place.</a> <a data-t="979">So we're still working on trying to get that balance right.</a> <a data-t="981">Like, how do we, like, gently nudge you that maybe you should be moving away from this</a> <a data-t="986">function without, like, getting in your face all the time?</a></p>

<p><a data-t="991">So two examples of that, like, the tbldf function in dplyr, which is one way you used</a> <a data-t="996">to create tbls before we called them tbls, is on its way out.</a> <a data-t="1002">Another function is dplyr do.</a> <a data-t="1006">Generally, these functions, we won't go from, like, it will try, unless it's, like, a very</a> <a data-t="1011">niche function, it will not go from stable to deprecated immediately.</a> <a data-t="1015">Normally, it will go through a questioning phase first, just to make sure.</a> <a data-t="1019">So we've got plenty of notice if we're uncertain about something.</a></p>

<p><a data-t="1022">Now, to contrast this, we're talking about the superseded life cycle.</a> <a data-t="1027">We've also called this retired in the past.</a> <a data-t="1030">My sort of thinking was, like, you know, when you retire, you're not, like, actively working</a> <a data-t="1035">anymore, but you're still a productive member of society, but when people heard that a function</a> <a data-t="1042">was retired, they're, like, oh, they're going to take it out back and shoot it in the head.</a> <a data-t="1047">So we're changing the name to superseded to make it clear that we think there's a better</a> <a data-t="1052">alternative, an alternative that's, like, maybe easier to use or faster or easier to</a> <a data-t="1056">learn or more powerful or something, and we think you should learn that new approach when</a> <a data-t="1063">you've got some free time.</a></p>

<p><a data-t="1065">So these superseded functions, they're not going anywhere.</a> <a data-t="1068">They're going to hang around for a long time, but they're not going to get any new features,</a> <a data-t="1073">and they'll only receive, like, really critical bug fixes.</a> <a data-t="1077">So a really good example of this, spread and gather and reshape and tidy are, you know,</a> <a data-t="1082">like, you know, they've been around for a long time, hundreds of thousands of people</a> <a data-t="1086">rely on them.</a> <a data-t="1088">We think we've got a better approach with pivot longer and pivot wider, an approach</a> <a data-t="1092">that hopefully you can actually remember how to use, but spread and gather are not going</a> <a data-t="1096">away.</a> <a data-t="1097">They will probably eventually go away, but that's, in this case of spread and gather,</a> <a data-t="1102">probably at least five years away.</a> <a data-t="1104">So we do encourage you to switch.</a> <a data-t="1106">We will change them in our documentation, in our books, so that you don't, that we don't</a> <a data-t="1111">teach them, we don't advise them, but those functions will live on for a long time.</a></p>

<p><a data-t="1117">And then on the other end of the spectrum, we've got these experimental functions.</a> <a data-t="1120">These are functions that, like, we have played around with internally, and we think they're</a> <a data-t="1124">kind of interesting or kind of cool, but we're not, like, 100% sure about them.</a> <a data-t="1129">So maybe they'll go away, maybe they'll stay.</a> <a data-t="1132">We just don't know.</a> <a data-t="1133">And so we kind of want you, if you are adventurous, we want you to try these functions out and,</a> <a data-t="1140">like, tell us, like, are these functions, do you love this function, do you hate this</a> <a data-t="1144">function, like, what's going on?</a> <a data-t="1146">So we can use that to kind of inform our decisions.</a> <a data-t="1150">But don't use them in critical code, like, they may get removed in the future.</a> <a data-t="1155">Don't rely on them 100%.</a> <a data-t="1158">But if they are useful to a lot of people, we will, you know, we'll keep them around</a> <a data-t="1162">and hopefully remove that experimental label.</a></p>

<p><a data-t="1167">And really all of this is kind of in service between this tension, like, we want to give</a> <a data-t="1172">you a stable foundation that you can build upon and rely upon, but at the same time,</a> <a data-t="1178">like, just because design is fundamentally iterative, we know we can't get it right the</a> <a data-t="1183">first time.</a> <a data-t="1185">And sometimes we want to be able to have a go at it a few times before we are confident.</a> <a data-t="1191">So we really want to just make it clear, like, where functions are on this so we can</a> <a data-t="1195">continue to build the stable foundation while trying to get it right in the long term.</a></p>
