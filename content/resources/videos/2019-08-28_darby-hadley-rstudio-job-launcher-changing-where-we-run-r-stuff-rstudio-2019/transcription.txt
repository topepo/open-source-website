<h3>How the launcher works</h3>

<p><a data-t="285">Now that you've seen one of the things a launcher can do, let's talk about what is actually</a> <a data-t="289">going on.</a> <a data-t="290">We've added in two additional layers into the system.</a> <a data-t="294">The RStudio launcher is a separate service that is independent of RStudio server and</a> <a data-t="298">it communicates with a plug-in to connect to these different environments.</a></p>

<p><a data-t="306">So what am I talking about with these plug-ins?</a> <a data-t="309">So the core of the job launcher is simply a framework for loading external plug-ins</a> <a data-t="314">which contain the actual logic for communicating with the job's destination.</a> <a data-t="319">So how do these things work together?</a> <a data-t="321">Once your plug-in is configured and the RStudio launcher process starts, it executes your</a> <a data-t="325">plug-in process as a trial of itself and communicates with it via standard in and standard out.</a></p>

<p><a data-t="334">So what kind of plug-ins are available?</a> <a data-t="336">So for the 1.2 release of RStudio, we are releasing three plug-ins, local, Kubernetes,</a> <a data-t="343">and Slurm.</a> <a data-t="345">And side note on Slurm plug-in, it's still in development and it's not really ready to</a> <a data-t="347">be played with yet but it's going to be coming soon.</a></p>

<p><a data-t="352">So what about other plug-ins?</a> <a data-t="354">We're still trying to figure out what comes next but we're planning on adding more plug-ins</a> <a data-t="357">in later RStudio releases.</a> <a data-t="360">But what if you have a super awesome processing platform that you use in your own company?</a> <a data-t="366">Well good news is that you can create your own plug-ins.</a> <a data-t="370">And here's the documentation to do so.</a> <a data-t="372">And that should have everything you need to get up and running with whatever environment</a> <a data-t="375">you want to connect to.</a></p>

<h3>Types of jobs: ad hoc and containerized sessions</h3>

<p><a data-t="378">So there are two types of jobs you can launch from RStudio.</a> <a data-t="380">The first is ad hoc jobs and that's what we were talking about in the first example.</a> <a data-t="384">You can use the jobs pane in the IDE to launch specific scripts to run wherever the plug-in</a> <a data-t="389">is configured for.</a> <a data-t="391">You might see that I snuck a Python job up there.</a> <a data-t="394">The job launcher can also run Python jobs directly from RStudio.</a></p>

<p><a data-t="400">The second is containerized sessions.</a> <a data-t="402">So in this example I developed my code with an R session on the RStudio server that lived</a> <a data-t="407">on that box.</a> <a data-t="409">This is where I wrote and tested my script and it's a session that's tied to the console</a> <a data-t="412">in the IDE.</a> <a data-t="414">Then I launched the script as an ad hoc job to the Kubernetes cluster.</a></p>

<p><a data-t="419">But with the launcher I can also develop and test directly on the cluster with a containerized</a> <a data-t="426">session.</a> <a data-t="427">I can write and debug the code interactively with all the capabilities your code will have</a> <a data-t="431">when it runs.</a> <a data-t="432">That means you can completely separate these R sessions from the RStudio server if you</a> <a data-t="436">would like.</a> <a data-t="437">And this is integrated within the RStudio UI.</a></p>

<p><a data-t="442">So typically you have a single server and everyone is forced to use the same version</a> <a data-t="445">or versions of R and Linux dependencies and an admin has to maintain that box for those</a> <a data-t="449">users if they need updates or new software.</a> <a data-t="453">But with the launcher and containerized session those dependencies don't even have to be installed</a> <a data-t="457">on the server.</a> <a data-t="457">They live in the Docker image itself.</a> <a data-t="459">So various users can be using different images with different versions of R, specific R packages</a> <a data-t="464">installed by default.</a> <a data-t="466">And this allows you to create those truly reproducible environments.</a> <a data-t="470">Also these images can be shared and distributed on Docker Hub or any other container registry.</a></p>

<p><a data-t="478">So on the same RStudio server, user 1 can be running a session with a Docker image that</a> <a data-t="481">contains a specific version of R, get specific Linux packages and user 2 could be running</a> <a data-t="487">a completely different version of R and different R packages installed in shared libraries by</a> <a data-t="493">default.</a> <a data-t="495">Or one user could be running a bunch of different types of images for different sessions for</a> <a data-t="498">various types of work.</a> <a data-t="501">And these dependencies are now self-contained and isolated from other containers being run</a> <a data-t="504">so a user can completely screw up everything in the container and it won't affect the others.</a></p>

<blockquote><a data-t="501">And these dependencies are now self-contained and isolated from other containers being run</a> <a data-t="504">so a user can completely screw up everything in the container and it won't affect the others.</a></blockquote>

<p><a data-t="511">Because of this containerized session, one huge benefit is scaling.</a> <a data-t="514">So currently with RStudio Server Pro, if you'd like to scale, you have to set up multiple</a> <a data-t="518">RSP nodes with all the required dependencies and load balance between them.</a> <a data-t="521">If you get more users and sessions, you have to provision more nodes and reconfigure the</a> <a data-t="526">group.</a> <a data-t="527">Things like autoscaling becomes a much harder problem to solve if that's what you want to</a> <a data-t="530">do.</a></p>

<p><a data-t="531">But using the launcher, you can let a container orchestration platform handle that scaling</a> <a data-t="535">for you.</a> <a data-t="536">As you can see here, we have RSP running on one server with the launcher and the launcher</a> <a data-t="541">communicates with the Kubernetes cluster to create these containerized sessions there.</a> <a data-t="545">Kubernetes has all that functionality to do the scaling for you.</a> <a data-t="550">So this allows you to just have that one RSP server.</a> <a data-t="553">You could have multiple for resiliency, but all the memory and processing load is kicked</a> <a data-t="558">to the cluster and not on the RSP server itself.</a></p>
