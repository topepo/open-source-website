<h3>What's in the Keras kit</h3>

<p><a data-t="142">You can think of Keras as a kit for doing deep learning, kind of like a Marble Run toy</a> <a data-t="147">set.</a> <a data-t="148">If you haven't played with one of these before, when you first get it and open the box, it's</a> <a data-t="151">a little bit overwhelming.</a> <a data-t="152">There are so many parts and some of them are truly complicated, or they seem complicated.</a> <a data-t="158">But pretty quickly you figure out that all these parts compose with each other via a</a> <a data-t="162">very simple interface.</a> <a data-t="163">They just stack right on top of each other.</a> <a data-t="165">It's plug and play.</a></p>

<p><a data-t="167">And when you are composing these parts together, what you're doing is you're creating a path</a> <a data-t="170">for marbles to flow through.</a> <a data-t="173">And Keras is like this.</a> <a data-t="174">It has a lot of parts, but they all compose with each other very nicely.</a> <a data-t="178">And when you're composing them, you're creating a path for data to go through.</a> <a data-t="182">And because everything works so seamlessly, the whole thing feels spi.</a> <a data-t="185">It's fun to design your models and build them, and it's fun to feed them data and watch them</a> <a data-t="189">learn.</a></p>

<p><a data-t="191">So what's in the kit?</a> <a data-t="193">Keras comes with over 100 built-in layers.</a> <a data-t="195">These are the things that you'll use to actually build your model.</a> <a data-t="198">It also has a lot of things that go outside the model.</a> <a data-t="201">You have metrics to track model performance during training, loss functions and optimizers</a> <a data-t="206">which you'll use to actually train your model, data loading utilities to help you work with</a> <a data-t="211">data that doesn't fit in memory, and a family of operations which I'll talk about later.</a> <a data-t="216">And all these pieces fit together in a high-level training API where you just call fit.</a></p>

<h3>Hello World of deep learning</h3>

<p><a data-t="222">To illustrate how all these pieces fit together, I'm going to walk through the Hello World</a> <a data-t="225">of Deep Learning, which is based on the MNIST image dataset of handwritten digits that</a> <a data-t="230">look like this.</a> <a data-t="232">One of the things you can do in a dataset is build a classifier.</a> <a data-t="236">Given an image, predict what digit is it.</a></p>

<p><a data-t="239">The way you would do this in Keras is you would start by creating an input object where</a> <a data-t="243">you specify the shape, and then you take that input object and you pipe it through a couple</a> <a data-t="248">of layers to get back an output object.</a> <a data-t="252">And once you have your input and your output, you have all you need to create your model,</a> <a data-t="256">and you have a model.</a></p>

<p><a data-t="259">One of the things you can do with a model is print it, where you get this colorful summary</a> <a data-t="262">table showing you what's in there.</a> <a data-t="264">You can also plot it to see a flowchart of the path your data will take through the model.</a> <a data-t="270">You can also plot it as a not-so-simple flowchart full of annotations.</a></p>

<p><a data-t="275">And then once you've created your model and you're ready to train it, this is a two-step</a> <a data-t="280">process.</a> <a data-t="281">You call compile, where you specify the loss function, the optimizer, and the metrics.</a> <a data-t="286">And then you call fit with your dataset, and your model starts learning.</a></p>

<p><a data-t="292">Now this example model that I'm showing here is the simplest practical model you could</a> <a data-t="297">make.</a> <a data-t="298">It only has one trainable layer, it has no fancy tricks, but this remarkably simple model</a> <a data-t="303">already trains to 92% accuracy on the MNIST image dataset.</a> <a data-t="308">And this is what I mean when I say that deep learning is forgiving, because there are plenty</a> <a data-t="312">of contexts I can think of where 92% accuracy is already plenty useful.</a> <a data-t="318">And you don't have to be a deep learning expert to walk away with a model that's useful.</a></p>

<h3>Progressive disclosure of complexity</h3>

<p><a data-t="366">Okay, so that gives you a flavor for what first steps with Keras looks like.</a> <a data-t="370">There is a lot of effort to make sure there is a smooth initial experience, and that might</a> <a data-t="375">give you the impression that Keras is like a set of training wheels, where it's really</a> <a data-t="378">helpful at first but will become constraining later on.</a> <a data-t="382">But that's not the case.</a> <a data-t="384">Keras is a framework that you can continue to use as your skills and your expertise develops.</a> <a data-t="389">And in my opinion, where Keras truly shines is in research environments, when you're trying</a> <a data-t="393">to do something new and custom because of how flexible the API is there.</a> <a data-t="397">And that's because Keras ultimately has its roots in research.</a></p>

<p><a data-t="402">The way Keras can accommodate these different user profiles is through this idea that's</a> <a data-t="407">baked into the API called progressive disclosure of complexity, which is another way of saying</a> <a data-t="413">there is no one true way to do anything.</a> <a data-t="415">For any given task, Keras offers multiple ways to do it, each of them striking a different</a> <a data-t="420">point in the balance between flexibility and exposed complexity.</a></p>

<p><a data-t="427">So I can't illustrate all the ways that this is true in the API, the API surface is large,</a> <a data-t="432">but to illustrate the point, I want to show just one.</a> <a data-t="435">So let's say you want to create a custom layer, Keras comes with 100 layers, but none of them</a> <a data-t="440">do exactly what you want, you want to make your own.</a> <a data-t="442">The simplest way to do this is to call layer lambda and pass in a function there that takes</a> <a data-t="447">a tensor, and you can do whatever work you want on that tensor in that function.</a></p>

<p><a data-t="452">Another way is to use another model as a layer.</a> <a data-t="456">So models are layers too, which means that you can compose layers both sequentially and</a> <a data-t="461">recursively.</a></p>

<p><a data-t="464">The next step up in complexity is to subclass the base layer class.</a> <a data-t="467">And here you can implement some or all of the methods that are called at different points</a> <a data-t="471">in the layer's lifecycle.</a> <a data-t="473">And a custom layer created in this way has identical semantics to the built-in layers.</a> <a data-t="479">And as your code base grows, you can use inheritance to help manage your code.</a> <a data-t="483">So you can subclass not just the base layer class, but also other built-in layers and</a> <a data-t="488">other custom layers.</a></p>

<p><a data-t="491">And this subclassing API exists not just for layers, but for all the object types that</a> <a data-t="497">you might work with in Keras, including training callbacks, metrics, learning rate schedules,</a> <a data-t="503">and so on.</a></p>

<p><a data-t="507">For R users, along this path of progressive disclosure of complexity is the fact that</a> <a data-t="512">under the hood, Keras in R is implemented using </a><a href="/software/reticulate/">reticulate</a><a data-t="512">.</a> <a data-t="517">And reticulate is an R package that embeds Python in R. What this means is that everything</a> <a data-t="524">available in Python is also available in R. So you can choose to use Keras in R or</a> <a data-t="531">Keras — you can choose to use Keras from R or Keras from Python, and because of reticulate,</a> <a data-t="537">that choice is simply a matter of preference for language syntax and development environment.</a> <a data-t="542">You don't have to worry about one interface getting out of sync with the other, or one</a> <a data-t="546">having more features than the other, or one being faster than the other, because when</a> <a data-t="551">you're training your model, it's literally the same exact code that's running.</a></p>

<p><a data-t="557">Reticulate itself has had a steady stream of updates over the past few years.</a> <a data-t="561">There's improved error tracebacks, improved autocomplete, improved support for operators</a> <a data-t="566">and generics, and probably most exciting of all, updated Python environment and discovery</a> <a data-t="571">tools.</a></p>

<p><a data-t="572">Quick aside, I've heard from multiple people tell me that they work primarily in Python,</a> <a data-t="578">but they use reticulate to set up their Python environments, which I just think is so neat.</a></p>

<h3>What's new in Keras 3</h3>

<p><a data-t="589">Back to Keras 3, what's new in Keras 3?</a> <a data-t="594">The biggest thing to know is the return of multi-backend support, and that comes with</a> <a data-t="599">a set of new features that let you write backend agnostic code.</a></p>

<p><a data-t="600">A backend is something that Keras uses to compute gradients and to interface with a</a> <a data-t="606">GPU.</a> <a data-t="607">And for most of its life, up to about three years ago, Keras supported multiple backends,</a> <a data-t="612">including </a><a href="/software/tensorflow/">TensorFlow</a><a data-t="612">, Theano, MXNet, and CNTKit.</a> <a data-t="615">And then over time, those last three were retired, and three years ago, TensorFlow became</a> <a data-t="621">the only supported backend when Keras became an official part of the TensorFlow project.</a></p>

<p><a data-t="627">Now with Keras 3, Keras is once again a standalone package that's not part of TensorFlow.</a> <a data-t="633">When you first load the package, you can call use-backend with the name of your backend,</a> <a data-t="637">and the supported backends today are Jax, PyTorch, and TensorFlow.</a> <a data-t="641">You can also use NumPy in inference-only mode, and there's some work to add MLX as a backend</a> <a data-t="646">ongoing.</a></p>

<p><a data-t="649">The benefits of multi-backend support are that it lets you write future-proof code.</a> <a data-t="654">The high-level Keras API insulates you a little bit from some of the churn in the low-level</a> <a data-t="659">framework space.</a> <a data-t="661">You can easily switch backends at any time in a project, which means that if you encounter</a> <a data-t="665">a bug with one backend, you can just sidestep it by switching to another.</a> <a data-t="669">And more generally, it enables workflows where you can take advantage of the best that each</a> <a data-t="672">backend has to offer.</a></p>

<p><a data-t="674">So for example, you can develop your model with Torch as your backend, which gives you</a> <a data-t="678">a nice interactive experience.</a> <a data-t="680">And then you can switch to Jax to train it, which will give you the fastest performance</a> <a data-t="685">typically.</a> <a data-t="686">And then when you're getting ready to deploy, you can switch to TensorFlow, which has the</a> <a data-t="689">most mature deployment options.</a></p>

<p><a data-t="693">One of the things that makes this possible is the new saving and loading API that comes</a> <a data-t="698">with a new backend agnostic file format.</a> <a data-t="701">So say you train your model with Jax, you can then save that to a file with the .keras</a> <a data-t="706">extension, and then in a different R session with a different backend, the TensorFlow backend,</a> <a data-t="710">you can load that model, and you've effectively converted your Jax model to a TensorFlow</a> <a data-t="715">model.</a></p>

<p><a data-t="716">One reason you might want to do this is because you are getting ready to deploy.</a> <a data-t="721">And one of the easiest ways to deploy a Keras model is to connect.</a> <a data-t="726">Connect recently gained the ability to serve Keras 3 models using TensorFlow serving, which</a> <a data-t="735">gives you access to a scalable concurrent serving with no R or Python runtime, with</a> <a data-t="741">just these two function calls.</a></p>

<h3>The Ops family</h3>

<p><a data-t="747">Another thing that comes with the multibackend support in Keras is the Ops family.</a> <a data-t="751">This is a suite of 200 lower level functions for working with multidimensional arrays.</a> <a data-t="758">These functions all work the same regardless of if you're working with TensorFlow, Torch</a> <a data-t="762">or Jax tensors, if you're working in eager mode or in graph mode, and you can use them</a> <a data-t="767">to write backend agnostic code.</a></p>

<p><a data-t="769">You can use these anywhere where you operate with tensors, including at the top level REPL,</a> <a data-t="774">and custom layers, custom losses, custom metrics, and so on.</a> <a data-t="777">You can also use them as standalone layers.</a> <a data-t="781">The Ops API surface aims to be comprehensive.</a> <a data-t="784">It includes the NumPy API, the TFNN API, a set of linear algebra operations, image operations</a> <a data-t="791">and more.</a></p>

<p><a data-t="794">I think our users in particular are going to appreciate the Ops family because it provides</a> <a data-t="798">a more ergonomic alternative to using the backend through reticulate.</a> <a data-t="803">You get things like consistent one-based indexing, automatic coercion of arguments,</a> <a data-t="808">and best of all, built-in R documentation with lots and lots of R examples.</a></p>

<p><a data-t="813">Another reason why I think our users will like it is a little more abstract, but all</a> <a data-t="819">the functions are pure, so they have no side effects.</a> <a data-t="822">That makes for a much more consistent experience.</a> <a data-t="824">It encourages experimentation at the REPL, and it aligns better with the mental model</a> <a data-t="829">that R encourages.</a></p>
