<h3>HTML canvas graphics device</h3>

<p><a data-t="444">So that's the</a> <a data-t="444">REPL application.</a> <a data-t="444">I'll move on now to talking about the HTML canvas graphics device, which</a> <a data-t="452">I briefly showed in the lower portion of the screen.</a> <a data-t="459">So the canvas graphics device is built</a> <a data-t="459">into the WebR application and allows you to render graphics in a web browser.</a> <a data-t="466">We've</a> <a data-t="466">made big improvements since the last release by moving from the canvas API to this off-screen</a> <a data-t="474">canvas API.</a> <a data-t="474">So here's a video of producing a plot with 300,000 points from the last version</a> <a data-t="479">of WebR.</a> <a data-t="479">And you can see it's quite slow.</a> <a data-t="483">The reason for that is because each of those</a> <a data-t="483">points is drawn by sending a message from the worker thread where computation happens</a> <a data-t="489">in the WebAssembly environment to the main thread where that plot is being drawn.</a> <a data-t="495">So</a> <a data-t="495">dispatching and handling all of those messages for each of those points separately creates</a> <a data-t="499">a big overhead.</a> <a data-t="499">And that's what the main cause of those performance issues were.</a></p>

<p><a data-t="504">In the latest</a> <a data-t="504">version of WebR, it's much improved.</a> <a data-t="509">It's an order of magnitude faster.</a> <a data-t="509">And it's all</a> <a data-t="509">thanks to this off-screen canvas API.</a> <a data-t="514">What it does is it lets us draw the image on the</a> <a data-t="514">worker thread and then just send a single message with the finished image to the main</a> <a data-t="519">thread to be shown.</a></p>

<blockquote><a data-t="504">In the latest version of WebR, it's much improved.</a> <a data-t="509">It's an order of magnitude faster.</a> <a data-t="509">And it's all thanks to this off-screen canvas API.</a></blockquote>

<p><a data-t="527">Using off-screen canvas has enabled us to enable much more advanced</a> <a data-t="527">text rendering features too.</a> <a data-t="532">We can now take advantage of the text rendering provided by</a> <a data-t="532">the web browser that WebR is running in.</a> <a data-t="537">So here's an example of some code, again, running</a> <a data-t="537">live in this slide, showing off some of those advanced text features.</a> <a data-t="545">The biggest benefit of</a> <a data-t="545">this is probably that WebR now supports much better international scripts when plotting.</a> <a data-t="553">So you can see there's some Arabic text here, some Hebrew and some Japanese.</a> <a data-t="558">And they're all</a> <a data-t="558">handled automatically now by the WebR canvas graphics device.</a> <a data-t="565">That font fallback that enables</a> <a data-t="565">this also allows you to use emoji in your plot should you want to.</a></p>

<h3>Bitmap graphics and the lazy virtual file system</h3>

<p><a data-t="627">In addition to the canvas graphics device, WebR also has updates to</a> <a data-t="633">support the standard bitmap graphics devices built into R.</a> <a data-t="639">So if you're running</a> <a data-t="639">in a browser that doesn't have a, if you're running an environment that's not a web browser,</a> <a data-t="643">such as node server side, or you're running R code that isn't aware of the canvas graphics device,</a> <a data-t="649">you can now do so with WebR.</a> <a data-t="654">So here I have an example that this is just loading WebR into the</a> <a data-t="654">page.</a> <a data-t="661">And then there's some R plotting code here, writing a PNG to the virtual file system.</a> <a data-t="662">And once that PNG has been written to the virtual file system with WebR,</a> <a data-t="666">there are a file system APIs that let you read that file into a JavaScript variable.</a> <a data-t="672">And once</a> <a data-t="672">you have that data in JavaScript, you can do what you need to do with it.</a> <a data-t="677">You can offer it to the</a> <a data-t="677">user as a download, you can display it in the page, or if you're running server side, you could</a> <a data-t="681">send that data over the network.</a></p>

<p><a data-t="688">The way this works is that we've compiled the Cairo graphics</a> <a data-t="688">library and Pango text layout engine to WebAssembly as part of the WebR build process.</a> <a data-t="695">And by compiling those libraries to WebAssembly and linking them in, we can enable those advanced</a> <a data-t="701">text features that I already talked about for the built-in graphics devices, as well as the canvas</a> <a data-t="706">graphics device.</a> <a data-t="714">The main difference between the two is that for built-in Cairo graphics with</a> <a data-t="714">PNG, for example, those fonts can't be read off the user's file system.</a> <a data-t="720">And instead, they must be</a> <a data-t="720">provided to Cairo through the WebR virtual file system.</a> <a data-t="727">So WebR ships with a minimal selection</a> <a data-t="727">of Noto fonts.</a> <a data-t="733">But if you do have other requirements for different scripts or different font</a> <a data-t="733">typefaces, you can upload those fonts to the virtual file system.</a> <a data-t="739">And WebR has been configured</a> <a data-t="739">to search that directory, home, web, user, fonts, for font files, and use them.</a></p>

<p><a data-t="747">So here's an example,</a> <a data-t="747">very similar to the previous example I showed you.</a> <a data-t="751">The only difference is this time it's creating a</a> <a data-t="751">PNG file instead of displaying it in a browser.</a> <a data-t="757">And you can see the results are very, very similar.</a> <a data-t="757">And all those text features that I talked about, things like ligatures in Arabic text and the font</a> <a data-t="763">fallback for different glyphs, they're all supported, including the right-to-left text and</a> <a data-t="770">various other features.</a> <a data-t="777">This works in this example because I've uploaded some fonts for those</a> <a data-t="777">alternative scripts to the virtual file system in the correct place, and WebR has automatically</a> <a data-t="783">then used them to produce this plot.</a></p>

<p><a data-t="790">So the last thing I want to talk about in this video</a> <a data-t="790">is something we're calling the lazy virtual file system.</a> <a data-t="795">So everything I've shown you so far comes</a> <a data-t="795">with a price.</a> <a data-t="802">Increased web assembly, download size, data size.</a> <a data-t="802">If you're bundling fonts,</a> <a data-t="802">they'll have to be downloaded.</a> <a data-t="808">But not every user will want all of those features.</a> <a data-t="808">Not every user</a> <a data-t="808">will want to use Cairo graphics and download those font files.</a> <a data-t="812">Some people will just want</a> <a data-t="812">to do some computation and return some numbers.</a> <a data-t="819">The same thing applies to files built into R already.</a> <a data-t="819">So R ships with lots of documentation and examples and data sets, and they're not necessarily</a> <a data-t="826">needed for every user.</a> <a data-t="832">So it would be nice if WebR only downloaded the data it actually needs</a> <a data-t="832">to run your R code.</a></p>

<p><a data-t="841">And WebR 0.2.0 does just that.</a> <a data-t="841">It uses an API provided by Emscripten</a> <a data-t="849">for their file system environment that allows the file system to only download the files</a> <a data-t="849">when they're requested.</a> <a data-t="854">So here in this example, I'll just open the dev tools and you'll see exactly</a> <a data-t="854">what I mean.</a> <a data-t="861">So if I refresh the page, you can see the R files are downloaded to run R and its</a> <a data-t="861">base packages.</a> <a data-t="869">But then if I run the command png to use Cairo graphics to build a bitmap graphics</a> <a data-t="869">image, you can see it's only at that point that those Cairo libraries are downloaded to the</a> <a data-t="876">device.</a> <a data-t="884">And a similar thing, if I plot something now and I draw some text, you'll see that the</a> <a data-t="884">font files are only downloaded at the point of actually producing that plot.</a> <a data-t="891">So if I don't use</a> <a data-t="891">Cairo graphics and produce a plot that requires those fonts, they won't be downloaded by the user.</a></p>

<p><a data-t="897">That's really nice because it reduces the initial download size of WebR by a lot.</a> <a data-t="902">For the application that I've been showing, the demo REPL app in the previous version of WebR with</a> <a data-t="909">less features, that was 40 megabytes of download.</a> <a data-t="915">In the new version, it's only 9.5 megabytes and</a> <a data-t="915">that's with the extra features.</a> <a data-t="922">So that's only 23% of the previous download size.</a> <a data-t="922">So that's a</a> <a data-t="922">really good improvement for the download size of an initial WebR loading of the engine into a page.</a></p>
