<h3>Work lazy, not hard</h3>

<p><a data-t="1094">Brings me to the second lesson.</a> <a data-t="1096">Work lazy, not hard.</a> <a data-t="1098">I bet in a room full of programmers,</a> <a data-t="1100">that's something that will resonate.</a> <a data-t="1103">So we're in the beginning to understand Polars better.</a> <a data-t="1106">We started working with the more eager ways,</a> <a data-t="1108">but lazy makes a big difference.</a> <a data-t="1110">As Orsan already explained why,</a> <a data-t="1113">this is one of the differences you can see,</a> <a data-t="1115">so it's the same type of benchmark.</a> <a data-t="1117">Again, with the log scale, you can see that even Polars,</a> <a data-t="1117">compared to itself, makes a massive jump in performance</a> <a data-t="1120">when you see the lazy engine compared to the eager,</a> <a data-t="1122">and that's just because a lot of work doesn't have to be done.</a> <a data-t="1125">So work lazy, not hard.</a></p>

<p><a data-t="1219">So one example of that would be,</a> <a data-t="1222">you're going to need a little bit of imagination</a> <a data-t="1224">to keep it possible to do on a slide.</a> <a data-t="1227">So you take a lazy frame,</a> <a data-t="1229">say you have some heavy computation</a> <a data-t="1231">that you need to apply to that.</a> <a data-t="1233">In this case, with every collect,</a> <a data-t="1236">you run the lazy frame query all the way from scratch.</a> <a data-t="1239">So that means that this heavy computation that's in there</a> <a data-t="1241">is being run twice, which is a waste.</a> <a data-t="1244">So instead, you can kind of cache it like this.</a> <a data-t="1247">So you just run this heavy computation once,</a> <a data-t="1249">you cache it to RAM,</a> <a data-t="1251">and from there these two calculations that are following</a> <a data-t="1253">are just taking that from the sub-result,</a> <a data-t="1255">from the intermittent stage from RAM.</a></p>

<h3>Consistency and fewer surprises</h3>
