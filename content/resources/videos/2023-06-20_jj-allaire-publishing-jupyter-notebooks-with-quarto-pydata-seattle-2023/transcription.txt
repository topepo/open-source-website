<p><a data-t="397">So first, how do we support computational narratives?</a> <a data-t="400">Well, the core requirements,</a> <a data-t="402">it's kind of what I just described.</a> <a data-t="404">You've got to be able to render executable content from Jupyter.</a> <a data-t="406">You've got to be able to include</a> <a data-t="407">these kind of technical content types.</a> <a data-t="409">You've got to be able to then take Markdown</a> <a data-t="411">and turn it into a lot of different output formats.</a> <a data-t="415">This is, you know, Quarto does this,</a> <a data-t="417">but it's not the first tool to do it.</a> <a data-t="419">In fact, there's a tool called MVConvert</a> <a data-t="421">that's been around since the dawn of Jupyter, I think,</a> <a data-t="424">that does this sort of thing.</a> <a data-t="425">And then there's also a couple newer projects,</a> <a data-t="427">JupyterBook and Mist.js, that also do it.</a> <a data-t="430">So this talk, I'm going to focus very much,</a> <a data-t="432">kind of talk about the tooling for computational narratives</a> <a data-t="434">through the lens of Quarto,</a> <a data-t="435">but it's important to know these other tools exist.</a> <a data-t="438">They share a lot of the same features,</a> <a data-t="439">and they're definitely all worth taking a look at.</a></p>

<h3>Semantic authoring</h3>

<p><a data-t="1301">All right, a little bit more about this idea</a> <a data-t="1303">of semantic authoring.</a> <a data-t="1305">What do I mean by semantic authoring?</a> <a data-t="1307">Well, I'll define it a little bit by saying</a> <a data-t="1309">what is literal authoring?</a> <a data-t="1311">Literal authoring is writing or creating</a> <a data-t="1313">technical content or report or analysis</a> <a data-t="1315">kind of in the native format of a given medium.</a> <a data-t="1316">So I'm writing a dissertation in LaTeX.</a> <a data-t="1318">I'm creating a presentation with Keynote.</a> <a data-t="1320">I'm publishing a confluence wiki.</a> <a data-t="1322">And that's a lot of the authoring that we do.</a> <a data-t="1323">But it has challenges.</a></p>

<p><a data-t="1326">I think one of the biggest challenges,</a> <a data-t="1328">especially when you kind of like have notebooks</a> <a data-t="1330">or like really high-value technical content,</a> <a data-t="1332">they each have their own proprietary format</a> <a data-t="1334">and it's quite awkward and time-consuming</a> <a data-t="1336">to take what you've got in a notebook, for example,</a> <a data-t="1339">and mark down and repurpose it</a> <a data-t="1341">across many different mediums.</a> <a data-t="1344">These tools typically have</a> <a data-t="1347">mostly non-existent support</a> <a data-t="1349">for the kind of technical content that I just talked about.</a> <a data-t="1352">They usually don't have any mechanism</a> <a data-t="1354">for including live code in its output.</a> <a data-t="1356">So usually you're ending up copying and pasting.</a> <a data-t="1359">And then when your notebook changes</a> <a data-t="1361">or your analysis changes,</a> <a data-t="1363">you've got to hopefully correctly update all your outputs.</a> <a data-t="1366">And that's what I'm just alluding to.</a> <a data-t="1368">There's no real straightforward way</a> <a data-t="1370">to sort of automate or reproduce</a> <a data-t="1372">computationally derived content.</a></p>

<p><a data-t="1374">So the idea, I think, is to move to,</a> <a data-t="1377">and this is really what Jupyter notebooks do</a> <a data-t="1379">and what Markdown does,</a> <a data-t="1381">move to semantic authoring</a> <a data-t="1383">where you don't literally double-click a word</a> <a data-t="1385">and say, this is bold,</a> <a data-t="1387">but you sort of write textually,</a> <a data-t="1389">this is a heading, this is italicized,</a> <a data-t="1391">this is code, this is a block quote.</a> <a data-t="1393">And then the underlying rendering engine</a> <a data-t="1396">can give it the right appearance</a> <a data-t="1399">in a given target format.</a></p>

<p><a data-t="1401">So again, semantic structure,</a> <a data-t="1404">really important when we get into talking about</a> <a data-t="1406">how the system can kind of be hacked and customized.</a> <a data-t="1409">Pandoc parses the document</a> <a data-t="1411">into this abstract syntax tree</a> <a data-t="1413">so the document is something we can actually compute on.</a> <a data-t="1415">We can filter it, we can change it,</a> <a data-t="1417">we can amend it,</a> <a data-t="1419">we can do lots and lots of things to the document</a> <a data-t="1421">after it's been parsed into this abstract syntax tree.</a> <a data-t="1423">That is not something you can do, again,</a> <a data-t="1425">with all these literal output formats.</a> <a data-t="1427">But here, since we've got the document in this generic form,</a> <a data-t="1429">it's very easy to compute on.</a> <a data-t="1431">Many of the features of Quarto</a> <a data-t="1433">and many of the extensions that you might want to use or write</a> <a data-t="1435">take advantage of this fact.</a></p>
