<h3>Async and <a href="/software/mirai/">Mirai</a></h3>

<p><a data-t="213">So first, in terms of async, I'm mainly going to talk about async in the context of Shiny itself.</a> <a data-t="222">On the left here is the documentation for using promises in Shiny.</a> <a data-t="228">Some of you will have come across that, because it's been up for seven or eight years.</a> <a data-t="233">It's written by </a><a href="/people/joe-cheng/">Joe Cheng</a><a data-t="233"> himself.</a> <a data-t="235">What we've done earlier this year is update it to assume that when you're launching these async tasks, you're using Mirai to do so,</a> <a data-t="241">which is right at the bottom.</a> <a data-t="246">And then on the right, what is Mirai?</a></p>

<p><a data-t="249">Mirai is an RLib package.</a> <a data-t="252">It's the package that I created to essentially bring modern async to R.</a> <a data-t="261">And Mirai means future in Japanese, by the way, in case you're wondering.</a> <a data-t="266">So it's a Japanese flavor of future, it's a Japanese future.</a></p>

<p><a data-t="271">So what do I mean by modern async?</a> <a data-t="274">Well, I like to use this analogy, email, because everyone should get this, everyone uses email,</a> <a data-t="279">and basically before Mirai, what we had was what I call a scheduled fetch,</a> <a data-t="286">but I mean it doesn't really have a name, but it's when you have a desktop client and it just checks for email every 15 minutes in the background, something like that.</a> <a data-t="295">And we've all had this, you know, we've requested an OTP, we're waiting for the code,</a> <a data-t="300">so we're just constantly clicking refresh, it's like, you know, why haven't I got my code yet?</a></p>

<p><a data-t="305">And that's sort of what we were doing before we had Mirai,</a> <a data-t="311">and you can imagine when we actually get an email, we get, the client says there is new email,</a> <a data-t="316">then it actually goes to the server and fetches that back.</a> <a data-t="318">So if you have an email and it actually contains large attachments, you could actually be waiting some time for that attachment to be downloaded before you can open up your email.</a> <a data-t="330">So that's essentially the type of async, the experience you had before you had Mirai.</a></p>

<p><a data-t="335">What Mirai brings you is basically push notification.</a> <a data-t="338">So again, this is the newer mobile experience.</a> <a data-t="343">We all know what happens, we get a notification on our phone, as soon as there's email on the server, notification pops up,</a> <a data-t="349">and the difference here as well is when we get that notification, the email is already sitting on your phone.</a> <a data-t="359">So when you click into notification, the email will open up immediately.</a></p>

<p><a data-t="364">So this is just an analogy, but this actually corresponds very well to what's actually happening under the hood when we talk about async in R and in Shiny.</a> <a data-t="376">So this is pretty much what actually happens with Mirai versus not.</a> <a data-t="380">So this is what I mean by modern async.</a></p>

<p><a data-t="384">So if you use Mirai within an extended task in Shiny, you get all of that, but you also get all the other advantages of Mirai for free.</a> <a data-t="395">And I'm just going to spend a couple of minutes talking about some of the other advantages.</a> <a data-t="399">If this goes over your head, just think, oh, wow, this sounds really advanced.</a></p>

<p><a data-t="405">But Mirai was designed thoughtfully, I'd like to say, I like to think, and on these four pillars</a> <a data-t="412">in that it has a modern foundation, which gives it the performance that it does, and it was designed for production.</a> <a data-t="419">This is important.</a> <a data-t="420">We'll get back to that.</a> <a data-t="421">So you have the confidence to deploy it everywhere.</a></p>

<p><a data-t="424">So in terms of modern foundation, it's built on NNG, which stands for Nano Message Next Generation.</a> <a data-t="430">This is a high-performance messaging library.</a> <a data-t="433">What this means is that we get the most optimal types of connections out of the box.</a> <a data-t="438">So this is inter-process communications, TCP, or even secure TLS, where we need that.</a> <a data-t="445">We've also engineered BaseR's serialization mechanism to better support custom serialization of newer cross-language data formats,</a> <a data-t="453">such as Apache Array, such as if you're working with Torch tensors.</a></p>

<p><a data-t="462">Because we have this foundation, Mirai can scale to millions of tasks over thousands of connected processes,</a> <a data-t="469">and it can do this all at 1,000 times the efficiency of anything that was available prior to Mirai.</a></p>

<blockquote><a data-t="462">Because we have this foundation, Mirai can scale to millions of tasks over thousands of connected processes,</a> <a data-t="469">and it can do this all at 1,000 times the efficiency of anything that was available prior to Mirai.</a></blockquote>

<p><a data-t="479">The zero latency promises are what I've just talked about, and this is probably the most important point.</a> <a data-t="486">Mirai was designed for production, and because it was designed for production, it's designed to be 100% reliable.</a> <a data-t="493">So it has this clear evaluation model, which matches, again, it matches what's actually happening under the hood.</a> <a data-t="501">So that means the code that you write with Mirai, you can expect it to be executed consistently and transparently and reliably.</a> <a data-t="511">And we've minimized the complexity in the package itself, and we don't have any hidden state.</a></p>

<p><a data-t="520">And finally, you can really deploy this everywhere, so you can use Mirai to parallelize on your local machine,</a> <a data-t="528">on remote machines where you have SSH, so this is any compute on your local network or any cloud instance that you spin up.</a> <a data-t="536">And of course, if you have access to a high-performance compute cluster, you can use your scheduler of choice.</a> <a data-t="541">And Mirai has this concept of compute profiles, so these work in a modular way,</a> <a data-t="548">so you can be connected to all three types of resources at the same time,</a> <a data-t="554">and you can do things like send different portions of compute to different destinations.</a></p>

<p><a data-t="561">So that's all I'm going to be talking about on Mirai itself.</a> <a data-t="565">Mirai is an RLIT package.</a> <a data-t="568">It is now the primary async backend for Shiny.</a> <a data-t="572">It is the built-in async evaluator in </a><a href="/software/plumber/">Plumber</a><a data-t="572"> 2.</a> <a data-t="576">It powers Parallel Per, and it's used in other parts of the Tidyverse as well,</a> <a data-t="583">such as in an upcoming release of </a><a href="/software/ragnar/">Ragnar</a><a data-t="583">,</a> <a data-t="591">and it's also used in Tidy models for things like hyperparameter tuning.</a></p>

<h3>OpenTelemetry and observability</h3>

<p><a data-t="601">So moving on to my second topic, this is open telemetry.</a> <a data-t="607">This is something that's probably going to be new to most people in the audience.</a> <a data-t="612">And open telemetry is all about observability at scale.</a> <a data-t="616">This is especially important for something like a Shiny app, because a Shiny app can be quite complicated.</a> <a data-t="621">I mean, you can be doing a lot in a Shiny app.</a> <a data-t="624">So you could be ingesting data from a database.</a> <a data-t="627">You could be making a call to an API using HIDA2, and you can be doing computation using Mirai on another machine, for example.</a> <a data-t="634">And to be able to see what's happening through all the layers of packages can be a challenge.</a> <a data-t="640">And this is the problem that open telemetry is designed to solve.</a></p>

<p><a data-t="645">So open telemetry has this concept of traces.</a> <a data-t="648">A trace is just what happens in response and action.</a> <a data-t="652">So in the context of a Shiny app, this could be someone clicking a button, which sets off a reactive update.</a> <a data-t="657">So when that happens, then everything that's the result of that action, you can see in spans,</a> <a data-t="667">which is what happens in the various packages as a result of the action.</a> <a data-t="673">And I won't spend too long explaining this, because I have a live demo where you can see this in action.</a></p>

<p><a data-t="678">But first of all, why might you want to use data like this?</a> <a data-t="685">Firstly, if you want to improve the performance of your Shiny app, you can easily see how long these spans are,</a> <a data-t="691">and you can look at minimizing the span length.</a> <a data-t="698">And also, if they're very heavily nested, you might look at reducing the amount of nesting you have.</a></p>

<p><a data-t="708">Secondly, is you can see errors immediately.</a> <a data-t="716">And these are errors that actually happen in actual use.</a> <a data-t="719">So not just theoretical errors where you're just testing.</a> <a data-t="723">If there's an error, they will show up in your spans.</a> <a data-t="726">Third point, all this data will be centralized.</a> <a data-t="730">So even if you're doing things in other processes or even on other machines, you can receive this data in one place,</a> <a data-t="736">in one dashboard that's easy for you to look at.</a></p>

<p><a data-t="741">And this final point is probably the best point.</a> <a data-t="745">You can leave this on in production.</a> <a data-t="749">So this isn't just when you're profiling your apps in development.</a> <a data-t="754">For an actual deployed Shiny app, you can leave this on.</a> <a data-t="757">And what this gives you is real-time monitoring.</a> <a data-t="761">So for example, if your Shiny app makes an API call and that API goes down, you'd be able to see that in real-time,</a> <a data-t="768">and you can actually choose to get alerted to that.</a> <a data-t="772">So if you wanted to mitigate that, that's possibly something that you can do.</a></p>

<blockquote><a data-t="745">You can leave this on in production.</a> <a data-t="749">So this isn't just when you're profiling your apps in development.</a> <a data-t="754">For an actual deployed Shiny app, you can leave this on.</a> <a data-t="757">And what this gives you is real-time monitoring.</a></blockquote>

<p><a data-t="779">Right, so you might be wondering, how then do I enable this?</a> <a data-t="786">How can I make use of this?</a> <a data-t="788">Well, the good news is you don't need to do anything.</a> <a data-t="791">You don't need to change any of your code.</a> <a data-t="794">We at Puzzit have done all the hard work.</a> <a data-t="796">We've instrumented all the key packages where we think this will be useful to you.</a></p>

<p><a data-t="801">So OpenTelemetry is already integrated in Mirai.</a> <a data-t="808">Versions of Shiny, HIDA2, and Elmer are going to be released imminently, not yet, but imminently, with this enabled.</a> <a data-t="816">After that, we're going to be instrumenting </a><a href="/software/plumber2/">Plumber2</a><a data-t="816"> and other packages where we think this will be useful.</a> <a data-t="823">For you as a user or developer of a package or a Shiny app, all you need to do is install the two packages, otel and otel-sdk,</a> <a data-t="829">and then you set some environment variables.</a> <a data-t="835">And these just tell the packages where to send the data.</a> <a data-t="841">So these are the actual environment variables I'm using for the demo I'm just going to show you shortly.</a> <a data-t="848">This sends the data to an online service called LogFire.</a> <a data-t="854">But for you, you don't have to send any data over the internet.</a> <a data-t="858">You can have a collector that's on your local network or even on the same machine.</a></p>

<h3>Performance workflow</h3>

<p><a data-t="1156">So you can use this in a lot of ways, as I sort of briefly went through before.</a> <a data-t="1163">But one possible way you can use this is to improve the performance of your Shiny app.</a> <a data-t="1169">So this performance workflow was given by Joe Cheng back in 2019, and it was updated earlier this year by my colleague Barrett.</a></p>

<p><a data-t="1179">And essentially what we're seeing now with all the new tools that we have since 2019 is,</a> <a data-t="1185">firstly, you can enable open telemetry, so you can see how long these spans are.</a> <a data-t="1192">And for the long spans, you can then profile using ProfViz or another profiler to see where your code is slow.</a> <a data-t="1199">And to optimize slow code, I mean, this has always been true.</a> <a data-t="1203">You can try and move as much work out of Shiny as possible.</a> <a data-t="1208">So pre-compute a lot of stuff before you even start a Shiny server, and this will quite often work.</a></p>

<p><a data-t="1216">You can also try to make code faster.</a> <a data-t="1219">And again, very often this actually works.</a> <a data-t="1221">And we have a lot more tools now versus in 2019.</a> <a data-t="1226">So instead of reading a CSV file, you can now use something like Duck Plier to read a Parquet file.</a> <a data-t="1233">So you can easily make your code faster.</a> <a data-t="1236">You can use caching, and this works sometimes.</a> <a data-t="1239">This is a feature that's built into Shiny itself.</a> <a data-t="1242">And you can use non-blocking reactivity.</a> <a data-t="1245">So this is async with something like Mirai.</a> <a data-t="1248">And again, this will work sometimes, but this is going to work in many more cases now,</a> <a data-t="1255">because Mirai, if you remember, has 1,000 times the efficiency that you had access to before.</a> <a data-t="1269">So you can use this in a lot more contexts.</a></p>
