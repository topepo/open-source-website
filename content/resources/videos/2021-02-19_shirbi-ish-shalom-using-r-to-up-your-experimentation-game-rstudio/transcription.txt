<h3>Taking big swings: lessons from a first experiment</h3>

<p><a data-t="166">The first big experiment I ever ran, I fell prey to this exact fallacy.</a> <a data-t="171">We were testing a new version of a recommendation model, and we wanted to test our recommendation</a> <a data-t="176">model out in the wild against an older iteration.</a> <a data-t="180">This was the perfect candidate for an A-B test.</a> <a data-t="184">However, to move our baseline metric of 3%, we needed almost a million samples, and it</a> <a data-t="191">would take us three months to run.</a> <a data-t="194">Guess what?</a> <a data-t="195">No surprise, we cheated.</a></p>

<p><a data-t="198">For the recommendation experiment, we used three-day connection rate as a baseline metric.</a> <a data-t="204">This suffered from being a bottom-of-the-funnel metric, because not only did the users have</a> <a data-t="208">to see the recommendation and click on it, but then they had to connect and also hold</a> <a data-t="213">on to that connection for at least three days for us to register that as a win.</a> <a data-t="218">That led to our metric being only 2.88%, which is fairly low.</a> <a data-t="224">To make matters worse, we were taking only a tiny swing, only aiming to move our baseline</a> <a data-t="229">metric by 3%.</a> <a data-t="232">With such a low minimum detectable effect of 3%, we needed 12 weeks or three months</a> <a data-t="238">to be able to fully bake our test.</a></p>

<p><a data-t="241">If you think about it, we were looking to move a baseline metric of 2.88% by only 3%,</a> <a data-t="248">which would have led to an absolute lift of less than a tenth of a percent.</a> <a data-t="254">If we had taken a bigger swing of 10%, 7.5%, or even just 5% as our minimum detectable</a> <a data-t="262">effect, we could have reduced our test time from the 12 weeks we needed to just two weeks.</a></p>

<h3>Applying big swings to a second experiment</h3>

<p><a data-t="268">A year later, we ran a second experiment where we wanted to learn from our missteps from</a> <a data-t="273">our first experiment.</a> <a data-t="275">In this second experiment, we wanted to test automating sales tax.</a> <a data-t="279">We wanted to show users' country-specific sales tax rate for their region in their first</a> <a data-t="285">time use experience.</a> <a data-t="288">This time, we wanted to apply the principle of taking big swings to our test.</a> <a data-t="293">This time, we used a metric much closer to the user action, engagement clicks, with a</a> <a data-t="299">relatively high baseline of about 7%.</a> <a data-t="302">More importantly, we declared that we were going to take a big swing and said our minimum</a> <a data-t="308">detectable effect had to be 100%, meaning that our baseline conversion rate of 7% had</a> <a data-t="314">to double to 14%.</a> <a data-t="317">This time, we were able to take our projected timeline to be only two weeks.</a></p>

<p><a data-t="323">The key idea, taking big swings to learn and fail quickly, is one of the biggest levers</a> <a data-t="329">you can push to speed up your experience.</a> <a data-t="332">But we wanted to do even better.</a> <a data-t="334">By taking our experimental philosophy of taking big swings, we were able to layer on an A-B</a> <a data-t="340">testing enhancement, sequential testing, which allowed us to take our two-week test time</a> <a data-t="345">and actually cut it in half to just one week.</a></p>

<h3>What is sequential testing?</h3>

<p><a data-t="348">But what is sequential testing?</a> <a data-t="351">Sequential testing is kind of like doing your taxes.</a> <a data-t="353">It allows you to not overspend and waste time waiting for results when you could have</a> <a data-t="359">already called it.</a> <a data-t="360">But it also ensures that you don't underspend time and actually call a test that's not statistically</a> <a data-t="365">regress.</a> <a data-t="367">Sequential testing allows you to spend that exact right amount of time such that when</a> <a data-t="372">you call the test, you didn't overspend or underspend time to be able to call a winner.</a></p>

<p><a data-t="378">The way that it does that is by building in a checkpoint or multiple checkpoints where</a> <a data-t="384">you can check the results of your test and peak earlier than a standard fixed horizon</a> <a data-t="389">test.</a> <a data-t="391">Sequential testing is also ideal for software situations because software gives us the unique</a> <a data-t="396">ability to monitor results in real time, which means that any moment of a peak, we can go</a> <a data-t="401">ahead and analyze the results in that moment.</a></p>

<p><a data-t="406">Additionally, since we have their experimental philosophy of taking big swings, it really</a> <a data-t="410">lends itself to sequential testing because sequential testing has the unique ability</a> <a data-t="415">to call winners early.</a> <a data-t="417">For our sales text test, we applied two checkpoints, one at the 50% mark where half of the total</a> <a data-t="423">samples had come in and one at the 100% mark where a total or full sample set had come</a> <a data-t="429">in at approximately two weeks.</a> <a data-t="431">At each checkpoint, you can see that we have a z-score acceptance criteria necessary to</a> <a data-t="436">be able to successfully determine a winner.</a> <a data-t="439">The acceptance criteria are much more stringent in the earlier checkpoints to account for</a> <a data-t="443">the smaller sample size that you're using to determine whether there's a winner.</a></p>

<p><a data-t="448">Note that the z-score acceptance criteria is greater than the 1.96 critical value, which</a> <a data-t="453">we normally see during fixed horizon testing.</a> <a data-t="456">It's there because we're paying a small price to be able to peak earlier.</a> <a data-t="461">Namely, the price is that our experiment is slightly less powerful in a statistical sense</a> <a data-t="465">so that we may miss the detection of a true effect or win.</a> <a data-t="470">In this case, we're using something called the O'Brien-Fleming boundary, which we've</a> <a data-t="474">chosen because it tries to minimize that payment at the end of the test because we're trying</a> <a data-t="480">to get as close to the 1.96 critical value as we can with that last peak.</a> <a data-t="486">As an aside, if you want to check out more, there's many more boundaries you can use which</a> <a data-t="489">may suit your needs more.</a></p>
