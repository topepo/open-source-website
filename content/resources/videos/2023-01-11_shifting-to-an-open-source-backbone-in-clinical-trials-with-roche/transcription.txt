<h3>The Ocean platform: building blocks</h3>

<p><a data-t="643">Thank you so much, Kieran.</a> <a data-t="645">So then let's talk about the building blocks of this kind of modern data science platform</a> <a data-t="650">we set out to create, which is really focused on open source as its backbone.</a></p>

<p><a data-t="657">So first of all, I would just like to quickly dive a bit more detail into the requirements</a> <a data-t="661">we and PD data scientists have for our day to day work.</a> <a data-t="664">And generally, we can separate our task into either the regulatory side of things or the</a> <a data-t="670">exploratory side of things.</a> <a data-t="671">And on the regulatory side of things, there's really four main deliverables we need to create.</a> <a data-t="676">The first one is called SDTM.</a> <a data-t="678">So SDTM is an industry standard for the data we collect in clinical trials.</a> <a data-t="682">So regardless of how the data is collected in the end, when you submit it to the health</a> <a data-t="686">authority, it has to be in this data standard such that it's harmonized across companies.</a></p>

<p><a data-t="692">Next up, we have the analysis or ADaM data sets.</a> <a data-t="695">And here you take the SDTM data sets as source and then you transform them in a way that</a> <a data-t="701">suits your analysis downstream.</a> <a data-t="702">So as a quick example, imagine you collect a questionnaire which has 10 questions and</a> <a data-t="707">an overall summary score.</a> <a data-t="708">So the 10 questions you would have in your SDTM data, because that's the raw data as collected.</a> <a data-t="712">But then you would calculate the summary overall score as part of your ADaM transformation.</a></p>

<p><a data-t="719">Next up, we need to perform a bunch of statistical analyses, and that can be something as simple</a> <a data-t="724">and descriptive as counting adverse events or something much more complex, for example,</a> <a data-t="728">fitting a survival model or fitting a mixed measures repeat model.</a> <a data-t="732">And finally, we need to put those statistical analysis into an easily digestible format.</a> <a data-t="737">And that's generally either tables, listings or graphs.</a> <a data-t="741">And all these four components combined make up the packages that we submit to health authorities,</a> <a data-t="745">such as the US FDA, in order to get market approval for novel therapeutics we're working on.</a> <a data-t="750">So this is really at the core of our business.</a></p>

<p><a data-t="753">But of course, we also have the exploratory side of things.</a> <a data-t="756">And here, really, it's highly context dependent what we need to perform.</a> <a data-t="760">And therefore, we need flexible tools and really the best tool for the job.</a> <a data-t="764">So having an open source background where you have access to multiple language, for example, is really a good thing in this context.</a></p>

<p><a data-t="772">We also need to consider that there's novel data modalities we're getting more and more.</a> <a data-t="776">For example, imaging data is becoming rapidly and more in our trials, digital biomarker,</a> <a data-t="781">omics data, et cetera.</a> <a data-t="782">So it's not just the classical rectangular data anymore, which we used to analyze in our commercial software.</a></p>

<p><a data-t="789">And finally, we want not only to enable our data scientists to easily explore our data,</a> <a data-t="795">but also people who don't work in data science, our colleagues in safety or clinical scientists.</a> <a data-t="800">And we want to really enable them to easily explore their data and do so in a reproducible</a> <a data-t="806">manner such that they don't always have to come back to us as data science.</a> <a data-t="809">And we have this long feedback loop.</a> <a data-t="812">And really, all of this combined necessitated that we created a next generation statistical</a> <a data-t="816">computing platform where we could really perform all these analyses.</a></p>

<p><a data-t="821">And this is where Ocean comes into play, our one central analytics environment, which is</a> <a data-t="826">the home for both our exploratory and regulatory work.</a> <a data-t="830">So Ocean is built in the cloud on AWS, and it's a language agnostic platform, which right</a> <a data-t="836">now supports our Python as well as SAS for our legacy work.</a> <a data-t="840">But it is easily extendable because it's based on Docker containers.</a> <a data-t="844">So if you spin up a container with, for example, Julia in it, suddenly you would have access to that language as well.</a></p>

<p><a data-t="850">For version control, we are using Git and GitLab.</a> <a data-t="853">And we're not only using that for our packages we develop internally, but it actually will</a> <a data-t="858">be there to store the code for all our studies.</a> <a data-t="860">So if on a study you program a particular ADaM data set or an adverse event table, the</a> <a data-t="864">code for that will be stored on GitLab.</a></p>

<p><a data-t="869">Next up, and this is really a big change, even though it sounds somewhat silly, but</a> <a data-t="873">we moved away from this proprietary format, SAS 7 BDAT, that used to be the only format</a> <a data-t="880">really that we used to store our CDISC data.</a> <a data-t="882">And instead, we opted for an open source format called Parquet, which is developed by the Apache Foundation.</a> <a data-t="888">And it's a binary format for handling rectangular data, and it's highly efficient in both reading</a> <a data-t="893">and writing, and it has served us already very well so far.</a></p>

<p><a data-t="897">We have an internal </a><a href="/software/rstudio/">RStudio</a><a data-t="897"> Connect server as part of this platform where we house our</a> <a data-t="901">exploratory Shiny apps, and we have a validated R package repository.</a> <a data-t="906">So while typically you would install packages from CRAN, internally, we use this validated</a> <a data-t="911">package repository, and you would find a lot of packages that you know from CRAN, such</a> <a data-t="915">as </a><a href="/software/ggplot2/">ggplot2</a><a data-t="915"> or dplyr over there as well, as well as our own open source packages we develop,</a> <a data-t="921">plus internal packages.</a> <a data-t="922">And it goes through our own internal, really streamlined and efficient validation process,</a> <a data-t="928">which is called autovalidate.</a></p>

<p><a data-t="931">Then we are using another open source tool called SnakeMake to orchestrate our production runs.</a> <a data-t="935">And SnakeMake is very clever in a way because it creates a directed acyclical graph of all</a> <a data-t="942">the artifacts that you need to create.</a> <a data-t="943">So think of your SDTMs, your ADaMs, your TLGs.</a> <a data-t="946">And then it really paralyzes as much as possible, taking dependencies into account.</a> <a data-t="951">So what used to take maybe a whole day, for example, to run everything for a final clinical</a> <a data-t="956">study report now is done in a matter of hours, if not minutes.</a></p>

<h3>Next-gen tools: OAK, Admiral, and RTables</h3>

<p><a data-t="961">And finally, Ocean is the tool, is the home of our next-gen tools for clinical reporting,</a> <a data-t="966">and I will spend the rest of my part talking about these next-gen tools.</a></p>

<p><a data-t="972">So the first one of those is called OAK, and OAK is our solution for automating SDTM mapping.</a> <a data-t="978">So SDTM mapping used to take a considerable amount of our time, and it was a rather labor</a> <a data-t="984">intensive process that was also somewhat mundane.</a> <a data-t="987">So we really opted to streamline this by creating an automation solution.</a> <a data-t="991">And driven by metadata and our global data standards at Roche, OAK can now automate around</a> <a data-t="997">80% of SDTM domains with around 22 reasonable mapping algorithms.</a> <a data-t="1002">And this, in turn, then really saved us a lot of time.</a> <a data-t="1005">We've seen already efficiency gains by at least 50%, so this is really huge.</a> <a data-t="1010">And finally, it's actually a fairly simple tool to use because it's, at the end of the</a> <a data-t="1014">day, a single R package you interact with plus a web app.</a></p>

<p><a data-t="1017">But that being said, under the hood, it's somewhat more complex because it has to interact</a> <a data-t="1023">with several systems, and that's actually why it's called the OAK Garden.</a> <a data-t="1027">And the first component of this garden is called Pistol and Honeybee.</a> <a data-t="1031">So really, what you have to think about is we have a metadata repository at Roche, which is a graph database.</a> <a data-t="1036">And inside that, we store the mappings from the raw data as we collect it to this SDTM</a> <a data-t="1042">standard in a machine-readable format.</a> <a data-t="1044">And that's really key here because that makes it then easy for R, for example, to query</a> <a data-t="1048">this information and make use of it.</a></p>

<p><a data-t="1051">Next up, we have our SDTM spec creator, which is called Mint.</a> <a data-t="1056">So this is a web application in React to create the study-specific SDTM mappings.</a> <a data-t="1061">And when I say that, really, most of it, at least 80% from what I had on a previous slide,</a> <a data-t="1067">should already be automated there if you adhere to standards.</a> <a data-t="1070">And then in this app, you have additional functionality to add any non-standard mappings.</a> <a data-t="1076">If, for example, you collect somewhat new data endpoints, which are not yet in our standards.</a> <a data-t="1082">All of this is then saved in a repository, which is called SEPHRON.</a> <a data-t="1087">And again, in a machine-readable format, namely JSON, such that it may query down the road in R.</a> <a data-t="1092">And very importantly, any non-standards you would define in the Mint application,</a> <a data-t="1097">you would save in SEPHRON, such that you can reuse these non-standard mappings across different studies.</a></p>

<p><a data-t="1105">And finally, we have the actual data transformation engine, which is called OAP.</a> <a data-t="1108">And it's the R package.</a> <a data-t="1110">It's the R package people in the end would use.</a> <a data-t="1112">And really, it's a very streamlined process.</a> <a data-t="1114">You fetch all this metadata from SEPHRON, basically.</a> <a data-t="1117">And then you have a single function call per SDTM domain.</a> <a data-t="1120">And given all this metadata, it will then output for SDTM datasets.</a></p>

<p><a data-t="1126">So the Oak Garden is not open source.</a> <a data-t="1129">And the reason for that being is that it's fairly tailored towards our internal Roche architecture.</a> <a data-t="1134">But that being said, the overall principle is really broadly applicable to anyone who</a> <a data-t="1138">has to map SDTM datasets, which is also the reason why we're in active talks with the</a> <a data-t="1143">CDISC Open Source Alliance to create a proof of concept in which we would use CDASH rather</a> <a data-t="1148">than our own internal data collection standard as a source.</a> <a data-t="1152">And if that's something you and your organization are interested in to collaborate, potentially, then please reach out.</a> <a data-t="1159">I certainly have heard a lot of questions regarding SDTM and R at R in Pharma.</a> <a data-t="1164">So this might be a good opportunity to reach out.</a> <a data-t="1167">And maybe a good collaboration can spring out of that.</a></p>

<p><a data-t="1172">So we cover the first pillar of our regulatory reporting site.</a> <a data-t="1175">And now it's time for the second, which is the ADaM datasets.</a> <a data-t="1178">And here, our solution is called Admiral, which is an open source modular toolkit for ADaM</a> <a data-t="1182">dataset creation in R.</a> <a data-t="1184">And what is really unique about Admiral is that unlike other packages we developed, which</a> <a data-t="1189">are now open source, this did not start out as a closed source internal project.</a> <a data-t="1194">And then later on got open source.</a> <a data-t="1196">Really, from the start of it, we collaborated with another organization, namely GSK, to</a> <a data-t="1201">hit the ground running and build something that would be usable not only for Roche's</a> <a data-t="1206">purposes, but really for the industry as a whole.</a> <a data-t="1208">And I think it's fair to say that this has been quite a success.</a> <a data-t="1210">We certainly want to use this kind of model of collaboration going forward.</a></p>
