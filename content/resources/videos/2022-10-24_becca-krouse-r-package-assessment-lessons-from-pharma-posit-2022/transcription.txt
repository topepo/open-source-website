<h3>The community domain</h3>

<p><a data-t="330">So let's step through these now, beginning with the community domain.</a> <a data-t="338">The purpose of</a> <a data-t="338">this domain is to get an idea of how broadly used a package is in the community, how much</a> <a data-t="346">folks are enjoying it, how strong its roots are.</a> <a data-t="352">So specifically, we can look at something</a> <a data-t="352">like downloads to get an idea of the volume of use to date, and that volume of use is</a> <a data-t="359">an indicator of how much the package has been vetted in real-life scenarios, how much</a> <a data-t="364">it's been stress-tested.</a> <a data-t="370">We are also interested in the overall trajectory of that use.</a> <a data-t="370">Is</a> <a data-t="370">it gaining traction? Is the package maintaining or declining for some reason?</a> <a data-t="376">Maybe there's</a> <a data-t="376">another package that's come along that's superseded it.</a></p>

<p><a data-t="384">We also want to look at how the community</a> <a data-t="384">relates to the package and its maintainers, both how the community relies on this package,</a> <a data-t="391">depends on this package as part of other software or other materials like books, and also, going</a> <a data-t="399">the opposite direction, how much the package maintainer has a vested interest in supporting</a> <a data-t="406">the community and keeping up with the ever-evolving, many moving parts of the community.</a></p>

<p><a data-t="415">Now something to keep in mind is in an industry like pharma, we have some fairly specific</a> <a data-t="421">needs.</a> <a data-t="421">Maybe our packages that we are choosing are a little more niche, or we have some internal</a> <a data-t="426">packages that we're using.</a> <a data-t="432">So by nature, these are going to have a little bit smaller community</a> <a data-t="432">user base, and that's okay.</a> <a data-t="439">We have other domains to look at as well.</a></p>

<h3>The authorship domain</h3>

<p><a data-t="447">With the authorship domain, we are doing some research to learn who the developers are.</a> <a data-t="453">What are their backgrounds, their experience? And really, does that experience, is that</a> <a data-t="459">appropriate match for the complexity of the tool?</a> <a data-t="467">We found this one to be fairly important</a> <a data-t="467">for statistical packages where that level of expertise is quite high, quite specific.</a></p>

<p><a data-t="467">So some things that we look at here are the overall reputation, how the community regards</a> <a data-t="474">this author as an expert in their field, as an R developer, and how much experience they</a> <a data-t="481">have in both those areas.</a> <a data-t="486">So something you could look at is the other packages that that</a> <a data-t="486">person or those individuals might have out there, and any specific qualifications, degrees,</a> <a data-t="494">certificates, things like that, all round out to our picture of who these authors are</a> <a data-t="499">and what their backgrounds are.</a> <a data-t="506">And this domain is quite dependent on what you can find out</a> <a data-t="506">there, which obviously varies from individual to individual.</a> <a data-t="514">So it's important nonetheless.</a></p>

<h3>The documentation domain</h3>

<p><a data-t="522">Our next domain is the documentation domain, which here I intend to cover functionality</a> <a data-t="522">documentation.</a> <a data-t="531">So what is the tool supposed to do? What is the intent?</a> <a data-t="531">And what is the</a> <a data-t="531">scope, what are the limits of that tool to help us understand what is appropriate use,</a> <a data-t="536">what is responsible use?</a> <a data-t="543">This is kind of a prerequisite to being able to prove accuracy.</a> <a data-t="550">You need to understand in which cases your function is going to produce correct results.</a></p>

<p><a data-t="556">So documentation, we are all probably pretty familiar with looking around at help files,</a> <a data-t="556">vignettes, examples.</a> <a data-t="564">So what we're after is, you know, do those things exist, but also</a> <a data-t="564">what is their quality?</a> <a data-t="570">How helpful and clear are they? What types of scenarios are covered?</a> <a data-t="578">And then for the intended use, clear documentation of expected inputs, expected outputs.</a> <a data-t="587">Really</a> <a data-t="587">nice if edge case, the limits piece is covered in there.</a> <a data-t="587">What might we want to watch out</a> <a data-t="587">for when we're using this tool, for instance.</a></p>

<h3>The software development lifecycle domain</h3>

<p><a data-t="596">And this leads us into the software development</a> <a data-t="596">lifecycle domain, which we are aiming to capture the overall process for how the code</a> <a data-t="605">develops and evolves over time.</a> <a data-t="615">How it is cared for, essentially.</a> <a data-t="615">With this one, first</a> <a data-t="615">and foremost, we need transparency into what's happening.</a> <a data-t="623">And luckily for us, this is increasingly</a> <a data-t="623">easy with things like GitHub, which is pretty popular these days.</a> <a data-t="630">Allows us to get a peek</a> <a data-t="630">into what's going on, what kind of branching model is being used, how are things being</a> <a data-t="635">worked on and merged in, and are there any what is the road map of the tool?</a> <a data-t="641">Are there</a> <a data-t="641">any critical open issues?</a> <a data-t="650">All things we can look at just by peeking into that repository.</a></p>

<p><a data-t="658">And to get a sense of how well supported it is, we can also look at the activity.</a> <a data-t="658">How</a> <a data-t="658">quickly the code or how often the code is being worked on, issues being closed, releases</a> <a data-t="665">to CRAN.</a> <a data-t="673">And then any safeguards that are in place to protect the integrity of the code</a> <a data-t="673">as it evolves.</a> <a data-t="680">Protected branches, continuous checking, continuous testing, all to prevent</a> <a data-t="680">against any bugs being introduced as this changes.</a></p>

<p><a data-t="688">Now, something to keep in mind is</a> <a data-t="688">a package could have all the hallmarks of a good package here.</a> <a data-t="694">But maybe it's really,</a> <a data-t="694">really new.</a> <a data-t="701">And it hasn't had a chance to work out some of the bugs.</a> <a data-t="701">And, you know,</a> <a data-t="701">we're not aware of those yet.</a> <a data-t="706">And it hasn't been stress tested enough yet.</a> <a data-t="706">Or it's changing</a> <a data-t="706">a lot.</a> <a data-t="711">On the other end of things, a package could have fairly stalled development, which</a> <a data-t="711">could mean it's been abandoned or it could mean it's just really, really stable.</a> <a data-t="717">So something</a> <a data-t="717">to keep in mind in looking at this domain is what is the history of the project?</a> <a data-t="722">Where</a> <a data-t="722">is it in terms of its life cycle?</a></p>

<h3>The testing domain</h3>

<p><a data-t="730">And our final domain is the testing domain.</a> <a data-t="736">So here, testing, I mean the actual testing inside the package.</a> <a data-t="736">Oftentimes this is your</a> <a data-t="736">test that, your unit tests.</a> <a data-t="742">And what these do is check that the functionality is working</a> <a data-t="742">as expected and that it's producing correct results.</a> <a data-t="751">Which sounds a lot like our definition</a> <a data-t="751">of accuracy that we talked about earlier.</a> <a data-t="757">So a really solid test suite can be essentially</a> <a data-t="757">a gold standard for how you can prove accuracy.</a></p>

<p><a data-t="764">So let's take a look at what we might look</a> <a data-t="764">for inside the package.</a> <a data-t="772">So this is your package directory.</a> <a data-t="772">But we can look at whether a test</a> <a data-t="772">exists and what they're covering.</a> <a data-t="778">So things like code coverage.</a> <a data-t="778">Which functions are being</a> <a data-t="778">covered? And to what extent?</a> <a data-t="784">And from there, we can look specifically at what we're interested</a> <a data-t="784">in.</a> <a data-t="790">What are the key things that we feel we are going to be using the most? <p><a data-t="813">And I recognize this is a fairly technical task.</a> <a data-t="818">So this may look a little bit different</a> <a data-t="818">depending on who you are and what resources you have.</a> <a data-t="824">But even being familiar with the</a> <a data-t="824">first two, the presence, the breadth, and then taking a peek into the depth.</a> <a data-t="831">Maybe some</a> <a data-t="831">of those descriptions in the test that can be very helpful.</a> <a data-t="838">Another thing to keep in</a> <a data-t="838">mind about the testing domain is not everything is super easy to test.</a> <a data-t="844">Sometimes visualizations</a> <a data-t="844">or complex algorithms can be a little bit— really novel complex algorithms can be a little</a> <a data-t="851">bit tougher to test.</a> <a data-t="857">So that's okay.</a> <a data-t="857">It's just— you know, again, we have the rest</a> <a data-t="857">of the domains.</a></p>

<h3>Thinking holistically about assessment</h3>

<p><a data-t="866">So with those in mind, the assessment may</a> <a data-t="866">look a little bit different depending on who you are and what you do.</a> <a data-t="874">For us, we are very</a> <a data-t="874">much working towards our submission work.</a> <a data-t="881">We're very risk averse in this situation.</a> <a data-t="881">We do a pretty in-depth assessment.</a> <a data-t="890">So, you know, if you're just kind of poking around</a> <a data-t="890">exploring, maybe you do a little bit lighter of an assessment.</a> <a data-t="894">But getting to know your</a> <a data-t="894">packages is really important.</a> <a data-t="902">It will serve you well.</a></p>

<p><a data-t="902">Also on a task level, if you're transforming the data, changing the data in some way, or</a> <a data-t="909">performing a statistical analysis, the stakes might be a little bit higher if something</a> <a data-t="917">goes awry as compared to— and also it might be harder to tell when something goes awry</a> <a data-t="923">as compared to something that's more aesthetic in nature.</a> <a data-t="929">So, again, your approach to the</a> <a data-t="929">assessment might change depending on that.</a></p>

<p><a data-t="940">And so you could also perform this whole assessment</a> <a data-t="940">and not be quite satisfied with the result.</a> <a data-t="947">But maybe you have only one option.</a> <a data-t="947">And this</a> <a data-t="947">is the package you need.</a> <a data-t="953">So maybe you're willing to bend a little bit on your risk and build</a> <a data-t="953">some extra safeguards around that for yourself.</a> <a data-t="963">Augment it with some testing or extra QC.</a></p>

<p><a data-t="963">So even if you find yourself in the left side, very much in the stable risk averse, the rest</a> <a data-t="971">of the ecosystem is going to continue to evolve and change.</a> <a data-t="976">And that's the beauty of R, right?</a> <a data-t="976">There's cutting edge newer things every single day.</a> <a data-t="983">So it's important for us, you know, if</a> <a data-t="983">we are in that situation, to keep our eyes up ahead of what's to come.</a> <a data-t="989">And find ways</a> <a data-t="989">to innovate.</a> <a data-t="996">Find ways to experiment.</a> <a data-t="996">And keep our minds open for any gems that come</a> <a data-t="996">along.</a> <a data-t="1002">And we can in turn then grow our tried and true collection of packages that we count</a> <a data-t="1002">on.</a></p>
