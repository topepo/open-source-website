<h3>Challenges of multilingual Shiny apps</h3>

<p><a data-t="64">Having said that, translating Shiny</a> <a data-t="64">apps into multiple languages is not that straightforward to do.</a> <a data-t="73">What I want to talk about today is</a> <a data-t="73">some of the small issues that cropped up when we were developing a bilingual Shiny app to</a> <a data-t="79">present public data.</a> <a data-t="85">Translation affects every single component of your Shiny app.</a> <a data-t="85">Simple</a> <a data-t="85">UI elements suddenly became reactive expressions that relied on a user input.</a> <a data-t="92">There are a couple</a> <a data-t="92">of packages out there that help you to automate the translation process, but what we found</a> <a data-t="96">is that those packages typically only work for a few of the most widely spoken languages,</a> <a data-t="101">and that their built-in dictionaries typically didn't include domain-specific terms.</a> <a data-t="106">So when</a> <a data-t="106">we were developing our bilingual app, we built the translation capabilities directly into</a> <a data-t="110">the app itself.</a></p>

<p><a data-t="116">And during the development process, there were a couple of seemingly</a> <a data-t="116">small problems that suddenly became much bigger problems to face.</a> <a data-t="121">And one of those problems</a> <a data-t="121">was what we thought was easy, ordering a factor for a bar chart.</a> <a data-t="128">Most of you have probably</a> <a data-t="128">come across the fact that R, in the absence of other instructions, will automatically</a> <a data-t="133">sort your character vectors and your factors alphabetically.</a> <a data-t="138">However, this doesn't always</a> <a data-t="138">make sense.</a> <a data-t="145">So in this example here, we have age group encodings as character strings,</a> <a data-t="145">and we might want to sort them from young to old because that's easier for a user to</a> <a data-t="149">interpret.</a> <a data-t="154">So for a single language, this is really easy, right?</a> <a data-t="154">We just create a factor</a> <a data-t="154">and we explicitly set the factor level.</a> <a data-t="161">And we get a nicely ordered bar chart.</a></p>

<h3>Solving factor ordering across languages</h3>

<p><a data-t="161">The problem</a> <a data-t="161">is that when we start doing this with multiple languages, it's not quite so easy.</a> <a data-t="167">First of</a> <a data-t="167">all, we can't rely on this alphabetical ordering at all anymore.</a> <a data-t="171">If we sort character vector</a> <a data-t="171">in one language and we sort it in another, actually they're still in different orders</a> <a data-t="175">now.</a> <a data-t="180">So we can't rely on alphabetical ordering at all.</a> <a data-t="180">We also don't want to have to manually</a> <a data-t="180">specify factor levels for all languages.</a> <a data-t="185">It's a little bit annoying if you have a few words</a> <a data-t="185">in two languages, but if you have hundreds or thousands of words in multiple languages,</a> <a data-t="190">it's just completely unmanageable.</a></p>

<p><a data-t="196">So to solve this problem for our app, what we did was</a> <a data-t="196">we ended up using a sort of default or primary language for everything related to data processing.</a> <a data-t="201">So in our case, we did everything in English and we explicitly set these factor levels</a> <a data-t="207">in English only.</a> <a data-t="212">For our bar chart example, what that meant was essentially creating two</a> <a data-t="212">different functions.</a> <a data-t="218">One function to compute the break point for the x axis.</a> <a data-t="218">Again, this</a> <a data-t="218">is done in English only based on our dataset.</a> <a data-t="223">We then created a second function to compute</a> <a data-t="223">the labels based on the output from that breaks function and from the user's choice</a> <a data-t="227">of language.</a> <a data-t="237">Essentially, we wanted to translate as little as possible.</a></p>
