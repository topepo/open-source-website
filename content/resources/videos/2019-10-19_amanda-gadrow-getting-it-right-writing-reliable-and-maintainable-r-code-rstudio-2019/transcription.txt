<h3>What we want from our code</h3>

<p><a data-t="68">So since the code behind our analysis or prediction is important, we should know what we want</a> <a data-t="73">out of it.</a> <a data-t="74">Ideally we want reliability, the ability to depend on our code producing the correct results,</a> <a data-t="80">and we want confidence that our functions won't break when new data comes in or that</a> <a data-t="84">one area of the code won't break.</a> <a data-t="86">If you make a change to one area of the code, it won't break another area of the code.</a> <a data-t="91">We also want reproducibility so that you and others can run the same code with the same</a> <a data-t="95">data and get the same results every single time.</a></p>

<p><a data-t="99">Flexibility goes hand in hand with extensibility.</a> <a data-t="101">In many cases we want to be able to handle different data sets, inputs, context, things</a> <a data-t="106">like that, and we need to be able to extend what we've already written to make incremental</a> <a data-t="112">improvements to the code and in our projects.</a> <a data-t="117">We want our projects to have longevity, to be relevant for the long term, and to accomplish</a> <a data-t="121">this it has to be straightforward to update.</a> <a data-t="125">We want to make it easy to iterate on the code since small changes over time are easier</a> <a data-t="129">to absorb than a huge code drop in much the same way it's easier to put $20 away a week</a> <a data-t="133">for a college fund rather than to find $1,000 at the end of the year to drop all at once.</a> <a data-t="139">Finally we need the code to be scalable, flexible enough to handle more users, additional</a> <a data-t="144">variables or inputs, and potentially larger data sets as well.</a></p>

<p><a data-t="148">So the overarching need here is to be able to trust the outcome of our analysis, which</a> <a data-t="152">is directly related to the quality of the code that is producing the results.</a></p>

<h3>Measuring code quality</h3>

<p><a data-t="159">Code quality does matter.</a> <a data-t="160">We should be examining it to make sure that it's up to snuff.</a> <a data-t="164">How do we do that?</a> <a data-t="165">Well, we can get some feedback during script execution in the form of console output, logs,</a> <a data-t="171">errors, warnings and things like that when the code runs locally.</a> <a data-t="174">And we also get user feedback from the people who are running the script or launching this</a> <a href="/software/shiny-r/">shiny</a><a data-t="178"> application.</a> <a data-t="180">We could wait until it breaks in the field, but it's better to check it yourself before</a> <a data-t="183">you release it to the rest of the world.</a></p>

<p><a data-t="187">The best way to do this is to create tests to verify the quality of the code.</a> <a data-t="192">Just check the output of the script to make sure it's doing what it should.</a> <a data-t="196">And test themselves, unit tests especially are separate R files that are designed to</a> <a data-t="200">exercise the functional code to see how it handles various inputs and integration points.</a> <a data-t="205">And by that I mean pieces of code that rely on the output of other pieces of code, like</a> <a data-t="209">a function that relies on the output of another function.</a> <a data-t="213">So you want to examine how quickly it runs as well and how well it runs under load to</a> <a data-t="217">gauge its performance.</a></p>

<p><a data-t="221">One of the biggest benefits of testing is that you're able to make faster updates to</a> <a data-t="226">your projects, because you run your tests if you run your tests after you make the changes,</a> <a data-t="230">you'll get quick feedback on the impact of those changes.</a> <a data-t="234">So the act of writing the test increases your visibility into internal dependencies, which</a> <a data-t="238">are those integration points that I just mentioned.</a> <a data-t="241">And you'll potentially gain a greater understanding of the meaning and the intent of the code,</a> <a data-t="245">because you really need to understand the code extremely well in order to be able to</a> <a data-t="248">test it effectively.</a></p>

<p><a data-t="253">Ideally when you write these tests, you're going to want to cover all of your code paths,</a> <a data-t="256">but practically speaking, you're going to want to start with the major functions, inputs</a> <a data-t="259">and integration points.</a> <a data-t="261">You're not going to be able to test everything all at once, so focus on the parts that you</a> <a data-t="264">think may be the most fragile.</a> <a data-t="269">Once these tests are in place, you can then expand to other areas of the code once you've</a> <a data-t="272">gotten sort of those major areas covered first.</a></p>

<h3>Manual vs. coded tests</h3>

<p><a data-t="279">Manual testing.</a> <a data-t="280">So you're going to do this as you're developing your code.</a> <a data-t="282">You're going to run things in the console just to make sure things are working, there's</a> <a data-t="284">spot checks, you want to make sure that the stuff that you're writing is actually producing</a> <a data-t="288">the results you expect.</a> <a data-t="290">The problem with those, though, I mean, they're very good, you want to run them, make sure</a> <a data-t="294">things are working, but the problem is you're not going to remember what you ran three months</a> <a data-t="297">later when you go to change the code.</a> <a data-t="299">So it's really easy to have things break.</a> <a data-t="303">Areas that used to work might break because you don't remember what tests you ran the</a> <a data-t="305">first time to make sure that it was working in the first place.</a></p>

<p><a data-t="309">So to avoid this problem, you should convert some of those manual tests into a set of coded</a> <a data-t="313">tests.</a> <a data-t="317">Coded tests are called unit tests in this context because they exercise small units</a> <a data-t="321">or pieces of a larger whole.</a> <a data-t="323">And you'll want to run these often, again, because you're going to get feedback right</a> <a data-t="326">away when you make changes.</a> <a data-t="329">If you find an issue several weeks later after you've moved on to other things, tracking</a> <a data-t="333">down the cause of that problem is really difficult.</a> <a data-t="336">It's much better to build these in from the beginning if you can rather than to wrestle</a> <a data-t="339">with a fallout later.</a> <a data-t="341">So writing tests now rather than waiting for bugs to appear later is the classic case of</a> <a data-t="345">an ounce of prevention being better than a pound of cure.</a></p>

<h3>Designing good tests</h3>

<p><a data-t="701">So as you're designing these tests, you're going to want to put yourself into the shoes</a> <a data-t="705">of the user, the consumer of each function.</a> <a data-t="708">What would you expect to happen?</a> <a data-t="710">What are some common mistakes or typos that you might anticipate that a user will put</a> <a data-t="715">in there for you?</a> <a data-t="716">You're going to want to focus on your external interfaces, your input types, and integration</a> <a data-t="722">points since those are the areas that tend to be most fragile.</a> <a data-t="726">Write one test for each behavior so that you always know where to find the test for a given</a> <a data-t="730">behavior.</a> <a data-t="731">And when you discover bugs, write tests for those as well so you're notified if they crop</a> <a data-t="735">up again later.</a></p>

<p><a data-t="737">As you're writing these tests, you may find that parts of the functional code would be</a> <a data-t="740">a little easier to test if they were more isolated.</a> <a data-t="743">We'll come back to that later in the talk, but generally speaking, if some area of the</a> <a data-t="747">code is hard to test, it's also going to be hard to maintain.</a></p>

<h3>Designing for testability and maintainability</h3>

<p><a data-t="908">So getting back to something I alluded to earlier, designing for your ease of testing</a> <a data-t="912">is also designing for ease of maintenance and so there's some good software practices</a> <a data-t="916">involved here.</a> <a data-t="917">So let's talk about that quickly.</a> <a data-t="918">Again, your goals are reliability, reproducibility, flexibility, longevity and scalability.</a> <a data-t="924">You want a good user experience and you want a high degree of confidence that everything</a> <a data-t="927">you're doing and all of your changes aren't going to break what you've already done well</a> <a data-t="930">before.</a></p>

<p><a data-t="931">You also want to take a look at a modular design for ease of testing and maintenance.</a> <a data-t="935">Break things down.</a> <a data-t="936">It's much easier to test a small function than it is to test a huge function because</a> <a data-t="941">it's much more easy to control the inputs and outputs that way.</a> <a data-t="944">You want to decouple your code, pull out helper functions, things like that to make it easier</a> <a data-t="948">to maintain later.</a> <a data-t="950">Don't repeat yourself.</a> <a data-t="952">Keep it simple if you can.</a> <a data-t="953">So I don't mean keep your analysis simple, I mean keep the pieces of your analysis, your</a> <a data-t="957">pieces of code as simple as possible, again, so they're isolated and easy to test and maintain.</a> <a data-t="962">And you want a consistent coding style so that it makes it easier for future you to</a> <a data-t="966">read or for your colleagues for that matter.</a></p>

<p><a data-t="969">Then on the shiny test side of things, for example, you want your ease of use user experience</a> <a data-t="973">tested, the state of that application when you've done a certain set of user actions,</a> <a data-t="977">you want that to be as consistent as possible and you can prove that with some shiny tests.</a> <a data-t="982">And finally, you want to make sure that your error handling is good.</a> <a data-t="984">You want to make sure if something does go wrong, that the message you give your user</a> <a data-t="988">is actionable.</a></p>

<p><a data-t="991">So at that point, yay, profit.</a> <a data-t="993">However, you have to remember that the actual tests themselves are not going to ensure the</a> <a data-t="997">high quality of your script or application, they just verify it when it's there.</a> <a data-t="1000">You need to develop a workflow of adding these tests as you go along to ensure that you're</a> <a data-t="1005">actually building in the good code in the first place.</a></p>
