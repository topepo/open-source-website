<h3>A framework for debugging</h3>

<p><a data-t="127">So when I'm trying to fix a bug in some code, I like to let myself be guided by three questions.</a> <a data-t="133">How do I know what the bug is first?</a> <a data-t="136">And then how do I fix it?</a> <a data-t="138">And then how do I know when I'm done fixing it?</a> <a data-t="141">And by answering these questions, I find I can keep myself a little bit on track and</a> <a data-t="146">that there are some specific strategies that I can use in answering each.</a></p>

<p><a data-t="151">So before you can go about fixing a bug, you have to know what the bug is, and the best</a> <a data-t="156">way to figure that out is to start by isolating the problem.</a> <a data-t="160">So at this point, you're not necessarily even trying to think about what the cause of the</a> <a data-t="164">problem is, but rather just trying to pin down exactly when it happens.</a> <a data-t="168">And the best way to convey this is through a minimal reproducible example or a </a><a href="/software/reprex/">reprex</a><a data-t="168">.</a> <a data-t="174">A reprex should be the minimal code and data necessary to produce the problem that you're</a> <a data-t="179">seeing.</a></p>

<p><a data-t="180">And the minimal part is really important here, because if you have thousands of rows of data</a> <a data-t="185">and lots of custom plot themes, there's a lot of distractions and it's hard to tell</a> <a data-t="189">what's really relevant to the problem at hand.</a> <a data-t="192">So paring it down can really help isolate the problem and often can turn up, can help</a> <a data-t="199">you figure out what is causing it through that process.</a> <a data-t="203">Then the reproducible part is important so that you or someone else can know if you fixed</a> <a data-t="209">it.</a> <a data-t="210">If you're not sure when the problem occurs, you can't make it happen reliably, it's hard</a> <a data-t="213">to know if it's really been solved.</a></p>

<p><a data-t="216">So the reprex package by </a><a href="/people/jenny-bryan/">Jenny Bryan</a><a data-t="216"> makes it really easy to produce these nice standalone</a> <a data-t="222">copy-pasteable reprexes and it can even do things like embed figures into the output.</a> <a data-t="228">And fortunately, the person who reported this issue provided a reprex of their problem,</a> <a data-t="233">so they did this step for me of isolating the problem.</a> <a data-t="236">So without any extraneous detail, they identified exactly when the problem occurs.</a> <a data-t="241">When boxes have variable widths, they overlap.</a> <a data-t="244">If they don't have variable widths, they don't.</a> <a data-t="247">And I really like this reprex because every piece of code that is in it is necessary to</a> <a data-t="252">producing this problem.</a> <a data-t="254">They also gave an example of the case where the problem doesn't occur.</a> <a data-t="258">When we have these uniform width boxes, you can see that they get moved side to side and</a> <a data-t="263">they don't overlap.</a> <a data-t="264">So that's what we're aiming for.</a></p>

<h3>Tracing the bug in ggplot2</h3>

<p><a data-t="268">So once you've isolated when the problem occurs, it's time to start digging into the cause.</a> <a data-t="273">And to do that, you need to kind of follow the data through its journey to the point</a> <a data-t="277">where the problem occurs.</a> <a data-t="279">If plotting code in particular, this can be sort of challenging because a lot is going</a> <a data-t="284">on within ggplot2 to turn a data frame that you provide into a plot.</a> <a data-t="290">And at the time that I worked on this, I was not familiar with all of the steps that were</a> <a data-t="294">happening, so I wasn't even sure really where to start looking.</a></p>

<p><a data-t="299">But this was sort of my thought process.</a> <a data-t="301">So looking at the code that was reported, I can see that in the buggy version of the</a> <a data-t="305">plot, we get this warning message.</a> <a data-t="307">Warning, position dodge requires non-overlapping X intervals.</a> <a data-t="311">And this is a little bit confusing to me because, like, yeah, the whole point is that these</a> <a data-t="315">are overlapping and we don't want them to.</a> <a data-t="317">Like, we're not trying to have overlapping X intervals here.</a> <a data-t="320">But I can latch on to this position dodge and sort of try to figure out what is happening</a> <a data-t="327">around there that is probably contributing to this problem, since that's in the warning</a> <a data-t="332">message.</a></p>

<p><a data-t="335">So I'm going to try to find position dodge and see if I can figure out what it's meant</a> <a data-t="339">to do.</a> <a data-t="340">And to do this, I need to search into the ggplot2 code base, which I have a copy of</a> <a data-t="345">locally.</a> <a data-t="346">I could search within the folder of code and find it.</a> <a data-t="349">But honestly, a lot of the time, I'm searching on GitHub, and so I'll type in position dodge</a> <a data-t="355">and find where it's located and see where I go from there.</a></p>

<p><a data-t="362">So then I go through the function, reading it line by line, and seeing when the function</a> <a data-t="368">that I'm on may be calling some other function that I don't know, and sort of repeating that</a> <a data-t="372">process of just reading the code, reading the comments, hoping that I can make some</a> <a data-t="376">sense of it.</a> <a data-t="378">Hopefully the comments in code are clear enough that it's possible to do this.</a> <a data-t="381">And in this case, I was able to figure out just by reading that there's sort of this</a> <a data-t="387">pathway that the data goes through from position dodge to capital P position dodge to a function</a> <a data-t="393">called collide and then posdodge.</a> <a data-t="396">And it appeared that those last two were the ones kind of controlling the most about how</a> <a data-t="402">these boxes were being placed.</a></p>

<p><a data-t="406">So at a high level, what collide does is that it gets information about the boxes before</a> <a data-t="412">they've been moved side to side.</a> <a data-t="414">So when you have these overlap, it gets these overlapping boxes, looks for the overlap,</a> <a data-t="420">and then sends all of the overlapping boxes to posdodge, which will shrink them down and</a> <a data-t="428">move them side by side.</a> <a data-t="429">And so the boxes sort of look like this figure when they arrive to collide, and then in groups,</a> <a data-t="437">they get sent to posdodge.</a> <a data-t="438">So one will go, and then the next will go.</a> <a data-t="442">And posdodge is the one that moves them side to side.</a></p>

<h3>Using debug to inspect the data</h3>

<p><a data-t="447">So that's helpful.</a> <a data-t="448">But now I want to see, OK, what's wrong with the variable width boxes that's making this</a> <a data-t="452">not happen?</a> <a data-t="454">I want to explore what these functions are actually doing and look at what data they</a> <a data-t="458">see so I can try to get more insight into this.</a></p>

<p><a data-t="461">So I'm very regretful that I wrote R code for many years without using debug or trace</a> <a data-t="469">or a lot of the really helpful tools for debugging that are built into R.</a> <a data-t="474">So debug is one that I'll give an example of right now.</a> <a data-t="477">And I use it all the time now because it lets you enter into the function that you're interested</a> <a data-t="482">in and step through it and see all of the data that it's passed and see all of the transformations</a> <a data-t="487">that it does as if you were in a regular R console.</a></p>

<p><a data-t="494">So through calling debug on the collide function, I can enter into collide.</a> <a data-t="499">And that's what this message here is saying, debugging in collide.</a> <a data-t="504">And I can then view the data that's being passed to the data argument of collide.</a> <a data-t="509">So this is roughly what the data looks like for a standard box plot when all the boxes</a> <a data-t="513">are the same size.</a> <a data-t="515">The data, I removed some columns to simplify it.</a> <a data-t="519">But there's basically one row per box that gives information about what X position it's</a> <a data-t="525">centered at, the start and end points of the boxes horizontally, the X min and X max.</a> <a data-t="531">And then in the data that I removed, there's also the Y position, the color and so forth.</a> <a data-t="535">So if you had just an empty plot, this would be all the information that you would need</a> <a data-t="539">to draw boxes on a box plot.</a></p>

<p><a data-t="544">Now we go back to my first point of isolating the problem.</a> <a data-t="547">Because we know that this bug happens when the boxes have variable widths, but not when</a> <a data-t="551">they have uniform widths.</a> <a data-t="553">So I, using this debugging tool, look at the data for both cases.</a> <a data-t="558">And I can see that for the uniform width boxes, the X min and X max for two boxes at the same</a> <a data-t="564">position are the same.</a> <a data-t="566">Which makes sense.</a> <a data-t="567">They're the same size.</a> <a data-t="568">So they're going to extend the same amount around the place where they're centered.</a> <a data-t="572">And for the variable width boxes, they're different.</a> <a data-t="575">Which also makes sense, because, again, they're different sizes.</a> <a data-t="578">So they're going to extend farther or less far around where they're centered.</a></p>

<p><a data-t="584">So by comparing these data sets and viewing the code, I'm able to find the source of the</a> <a data-t="588">bug.</a> <a data-t="590">Because one of the things that Collide does is, again, send these groups of boxes to posdodge</a> <a data-t="597">to be dodged.</a> <a data-t="598">And it is doing this on the basis of their X min value.</a> <a data-t="601">So it treats all boxes that have the same X min as being at the same position.</a> <a data-t="606">And so then they get sent to posdodge, and they get moved from one another.</a> <a data-t="610">But these variable width boxes don't have the same X min, so Collide doesn't realize</a> <a data-t="615">that they occupy the same position.</a></p>

<h3>Attempting a fix</h3>

<p><a data-t="619">So now we know what the problem is.</a> <a data-t="621">But fixing it is a whole other kettle of fish.</a> <a data-t="623">And it may or may not be obvious, once you've found the source of the bug, what the fix</a> <a data-t="627">ought to be.</a> <a data-t="629">So depending on what it is, you may just have to get in there and start changing things.</a> <a data-t="633">I made a lot of minor changes to the code and saved them with commit messages like,</a> <a data-t="638">this doesn't fix position dodge, but it might be in the right direction, before I ever got</a> <a data-t="643">to anything that was even a plausible solution.</a> <a data-t="645">These are real commit messages from my git ref log that I dug up to put on this slide.</a> <a data-t="652">And so I would go in there, make changes, run the reprex again, and see if anything</a> <a data-t="657">was different or if somehow I had fixed the problem.</a></p>

<p><a data-t="663">Since the main source of the problem was that Collide was treating these boxes with different</a> <a data-t="668">X mins as not being at the same position, my first thinking was, well, could it use</a> <a data-t="673">their X position instead?</a> <a data-t="675">Because we have X as a column in the data, so maybe we can just swap that out and everything</a> <a data-t="680">will be great.</a> <a data-t="683">So that's roughly what I did here.</a> <a data-t="685">And at first glance, it sort of works, but there's something wrong with this plot that</a> <a data-t="689">maybe some of you with sharp eyes are spotting, and that's that the boxes are not in the right</a> <a data-t="695">order.</a> <a data-t="696">We have pink, then blue, then blue, then pink, then blue, then pink.</a> <a data-t="700">So that's weird and not great.</a></p>

<p><a data-t="703">And then there are some other problems that are not visible in that plot, but came up</a> <a data-t="707">in other investigations, that this doesn't work for continuous X axes, it only works</a> <a data-t="713">for categorical ones for some reason.</a> <a data-t="716">And then there's this problem where the boxes are actually not being, the approach to scaling</a> <a data-t="722">the boxes is not accurate.</a> <a data-t="725">So when the boxes get moved side to side, they need to be shrunk down in order to fit.</a> <a data-t="730">And POSDodge is getting them in groups and scaling them by however many are in the group.</a> <a data-t="735">So if it gets three boxes, it'll shrink it down so that the three can fit side by side.</a> <a data-t="739">If it gets one, then it may not need to really scale it down.</a> <a data-t="743">And so you could end up in a situation where these variable widths that are supposed to</a> <a data-t="747">convey meaning in the plot are not really accurate at all.</a></p>

<p><a data-t="753">So this is a major problem.</a> <a data-t="756">And one lesson here is that sometimes in trying to fix a bug, you'll end up introducing way</a> <a data-t="760">worse bugs in the process.</a> <a data-t="762">So that's why it's important to test many different scenarios.</a> <a data-t="767">Isolating the problem helped me figure out the cause, but a solution needs to take into</a> <a data-t="770">account the real world complexity of how the code is used in lots of different scenarios,</a> <a data-t="776">lots of different types of data.</a> <a data-t="777">Fortunately, ggplot2 has a lot of automated tests that can alert me if something that</a> <a data-t="783">I'm changing causes unexpected changes elsewhere.</a> <a data-t="787">And as I worked on different possible solutions, I also made sure to add more tests to make</a> <a data-t="791">sure that things that I fixed didn't then get unfixed accidentally.</a></p>

<h3>Rewriting the dodging algorithm</h3>

<p><a data-t="797">So I then from here kind of rewrote this dodging algorithm so that it could see all</a> <a data-t="802">the data at once instead of getting these groups of boxes separately.</a> <a data-t="806">That way, it could scale all the boxes by the same amount and be consistent in that</a> <a data-t="810">way.</a> <a data-t="811">And I made some other tweaks that allowed it to work with continuous x-axes.</a> <a data-t="815">And with these new changes, things are looking pretty good.</a> <a data-t="817">The original issue was fixed.</a> <a data-t="819">We have box plots that work with both continuous and categorical x-axes.</a></p>

<p><a data-t="826">And then once you have something that's working for this original example, it's worth seeing</a> <a data-t="830">if there are ways that you can generalize it to other situations.</a> <a data-t="834">In reviewing my work on this box plot algorithm, Hadley pointed out that with a few minor tweaks,</a> <a data-t="840">we could generalize the algorithm to handle not just boxes, but also bars and arbitrary</a> <a data-t="846">rectangles.</a> <a data-t="847">And I realized that if we did this, we could solve or at least improve three other existing</a> <a data-t="853">open issues.</a> <a data-t="854">So we could really kind of kill a bunch of birds with one stone with this, or feed a</a> <a data-t="858">bunch of birds with one scone.</a></p>

<p><a data-t="861">So we did this.</a> <a data-t="863">The main thing that needed to be changed in order for the dodging algorithm to work with</a> <a data-t="867">arbitrary rectangles is that this approach where we're looking for the x position is</a> <a data-t="873">not really going to work anymore.</a> <a data-t="875">Boxes and bars would be centered around the same x, but for rectangles, you could have</a> <a data-t="881">overlapping rectangles like this one in the bottom left plot here that aren't centered</a> <a data-t="886">at the same position, but still overlap and would need to be moved from one another.</a> <a data-t="892">So rather than basing my determination of what boxes overlap on their x position, I</a> <a data-t="898">wrote a for loop that compares the starting point of one box or rectangle with the ending</a> <a data-t="905">point of the previous one so that it can check and see if there's overlap in that way.</a></p>

<p><a data-t="911">And I should say that Hadley had suggested this for loop approach from the very beginning</a> <a data-t="916">of my work on this pull request, and for some reason, I repeatedly ignored his suggestion,</a> <a data-t="921">which in retrospect was extremely foolish.</a> <a data-t="925">So another takeaway of this for me has been that if Hadley tells me I should write a for</a> <a data-t="930">loop, I probably should write a for loop.</a></p>

<h3>Knowing when to stop</h3>

<p><a data-t="934">But soon we had an algorithm that was working well for boxes, bars, arbitrary rectangles,</a> <a data-t="942">and from here, it was sort of unclear, like, when do we stop?</a> <a data-t="948">Some bugs require only simple fixes, but for more complicated ones, it can be hard to know</a> <a data-t="953">when you're really done.</a> <a data-t="955">This new dodging function seemed to be performing well, so again, we want to be testing a representative</a> <a data-t="961">variety of different cases to ensure that it is, in fact, working before we stop.</a> <a data-t="967">That's kind of at a minimum.</a> <a data-t="969">But once it works, there still might be more that one could do to improve it, so you need</a> <a data-t="973">to weigh the work that it would take to make it way, way, way better or a little bit better</a> <a data-t="980">to how important it really is to make the thing that much better.</a></p>

<p><a data-t="989">With this dodging algorithm, I had started to wonder whether it could, in fact, completely</a> <a data-t="993">replace the original position dodge.</a> <a data-t="997">I was really hoping that it would.</a> <a data-t="998">It would have been really personally satisfying to kind of wrap everything up with a nice</a> <a data-t="1002">neat little bow and be able to just completely replace the old position algorithm with one</a> <a data-t="1009">that could handle a few more situations.</a> <a data-t="1012">But alas, when I tried this out, I found that it still didn't work for violin plots and</a> <a data-t="1017">dot plots, which also sometimes get dodged in ggplot.</a> <a data-t="1021">So violins, which look like this on the left, sorry, Hillary, just become slanty lines with</a> <a data-t="1028">position dodge, too.</a></p>

<p><a data-t="1032">So after weeks of work, we decided to just wrap this up and call it good enough.</a> <a data-t="1037">We solved the original problem and many others besides, and while it would have felt satisfying</a> <a data-t="1041">to come up with one comprehensive dodging algorithm that could do everything, we figured</a> <a data-t="1045">that we could always come back and improve things later and it just wasn't worth the</a> <a data-t="1049">additional effort that it would take to make it completely, completely perfect.</a> <a data-t="1055">So we ended with a second dodging algorithm, position dodge 2, which is similar but slightly</a> <a data-t="1060">different to position dodge but can do some things that position dodge can't.</a></p>
