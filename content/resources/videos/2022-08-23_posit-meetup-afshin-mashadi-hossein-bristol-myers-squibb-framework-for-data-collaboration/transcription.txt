<h3>Defining data product and data as a product</h3>

<p><a data-t="443">So to start things off, I think it's quite critical to provide a definition of data product and data as a product.</a> <a data-t="454">If you have heard of this term, you might have noticed that in different circles, in different groups, it could mean quite different concepts.</a></p>

<p><a data-t="505">How we are defining data product is simply a data object that, much like any other products, come with a set of characteristics or attributes.</a> <a data-t="519">So specifically, we call data product, this notion of data product, to be a data object that is use case driven.</a> <a data-t="531">This is in contrast with, quote unquote, wrangled data that can serve any purpose.</a> <a data-t="536">It's clean data, and you can use it for whatever purpose.</a> <a data-t="539">So this has to be, the data product has to have a specific user story or a set of coherent objectives that it achieves.</a></p>

<p><a data-t="548">Now, if you have that level of specificity, you can imagine that you may have slightly different use cases leading to a variety of different data objects.</a> <a data-t="560">So it becomes even more critical to be able to version control it and track it.</a> <a data-t="603">So making it use case specific and being able, then, to branch it and version track it becomes quite critical.</a></p>

<p><a data-t="621">It needs to be tested.</a> <a data-t="622">It needs to be reproducible.</a> <a data-t="625">And I would say not only reproducible, but easily reproducible, meaning that with a handful of commands, you can just reproduce the data.</a> <a data-t="633">It should be easily maintainable, extensible.</a> <a data-t="636">Think about this as your typical R package in this context.</a></p>

<p><a data-t="642">So in this context of R users, you can use an R package for whatever end that you have in your analysis or otherwise.</a> <a data-t="650">You can also use that to integrate it into a new R package and build upon it and extend it.</a> <a data-t="655">Same should apply to data products.</a> <a data-t="657">You should be able to integrate it into a new data product, bring a few of them together, build upon them, extend them.</a> <a data-t="664">So it should be extensible.</a></p>

<p><a data-t="666">It should be findable, accessible, and portable.</a> <a data-t="669">And notes on that last one is that as data scientists, if I may speak for a lot of us, that we may not necessarily be super conscious or have concerns about whether my data is coming from Azure or AWS S3 or whatever platform, whatever infrastructure I have.</a> <a data-t="730">But our colleagues in IT or other places, they may.</a> <a data-t="734">So as my data, as the requirements of my organization changes, I need to take my data from, let's say, AWS S3 to Azure or vice versa or to some network drive.</a></p>

<p><a data-t="752">All of these characteristics that we talked about should go along.</a> <a data-t="753">So it shouldn't be something that is dependent on the platform or technology outside of the data product.</a> <a data-t="800">So these core set of characteristics and attributes define what we call a data product.</a></p>

<p><a data-t="815">Now, data as a product is a framework to make it practical for data science teams to pull together their data wrangling efforts towards collaboratively building data products.</a> <a data-t="827">So on one hand, we have this product.</a> <a data-t="832">On the other hand, we have a set of processes that allows us to easily, or at least methodically, build these data products.</a></p>

<p><a data-t="842">So the processes could be agreements between the teams, like how we want to use the conventions of coding,</a> <a data-t="852">as well as some automations and tools that allow us to implement those agreements.</a> <a data-t="858">So data as a product, you can think about as a framework, way of thinking, as well as the tool sets that allow us to implement that way of thinking.</a> <a data-t="868">Data product is the output.</a></p>

<h3>Motivation: the 80-20 rule of data science</h3>

<p><a data-t="915">So to focus on one, perhaps you can think about this 80-20 rule of data science, which is basically this observation made by a lot of the data scientists that a typical week could involve four days out of that five working days that you are engaged in data processing or data wrangling, whatever you call it.</a> <a data-t="958">And then on that last day, you may get a chance to actually do your data analysis.</a></p>

<blockquote><a data-t="915">So to focus on one, perhaps you can think about this 80-20 rule of data science, which is basically this observation made by a lot of the data scientists that a typical week could involve four days out of that five working days that you are engaged in data processing or data wrangling, whatever you call it.</a></blockquote>

<p><a data-t="962">So as a data scientist or leader of a data science team, if you are interested in adding more agility to your overall team efforts,</a> <a data-t="1022">it is fair to look at this data processing and try to bring in additional agility to it and efficiency to it.</a></p>

<p><a data-t="1036">And there are a variety of approaches that teams have implemented.</a> <a data-t="1038">Hopefully, we all work in teams that heroic data wrangling is not one of those approaches.</a> <a data-t="1043">No one is advocating for shrinking that 80% to 30% by just willpower and working extra hours.</a> <a data-t="1055">But then there are things like new data agreements that you may put in place with data vendors.</a> <a data-t="1059">You may ask your data platform to do more in the way of data validation.</a> <a data-t="1066">Even pushing that validation of data further upstream into your pipelines, that could help.</a></p>

<p><a data-t="1207">Perhaps a slightly different view would be rather than seeking relief, we can acknowledge that here's my team spending their precious time and resources and expertise.</a> <a data-t="1218">I should emphasize that expertise part.</a> <a data-t="1221">Building something, so there's a lot of value in that.</a> <a data-t="1225">If 80% of my team is building this artifact or output, that has a lot of value in it.</a></p>

<p><a data-t="1234">So instead of thinking about or along the way of thinking about reducing that time to getting that, can I also think about how can I drive more value from this precious resource?</a> <a data-t="1250">And if I'm thinking in that sense, I'm not trying to remove the burden of this effort.</a> <a data-t="1259">I'm just trying to drive more value from it.</a> <a data-t="1261">But to alleviate that burden, help with that, I can also make collaboration easy.</a> <a data-t="1266">So that big lift is shared across the team.</a></p>

<p><a data-t="1274">So you make collaboration easy.</a> <a data-t="1280">When permissible, you can add a dash of automated standards and make, put in place processes, process frameworks for collaboration.</a> <a data-t="1307">So not surprisingly, perhaps at this point, you can imagine that this latter approach of driving more value from data wrangling is what the framework of data as a product, as we are defining it, advocates.</a></p>

<p><a data-t="1324">So the idea is that you're bringing in your usual input data, and you're adding your logic of data wrangling.</a> <a data-t="1328">And then you feed it into this process that out would come a data product, which then can be fed into a number of utilities, analysis, apps, automated reports.</a> <a data-t="1339">And this data product now is reusable and extensible.</a> <a data-t="1346">So that work is essentially something that lives on.</a></p>

<p><a data-t="1351">That in contrast with a typical workflow of a data scientist, which may include data wrangling, be so inseparably intertangled with your utility, be it analysis or otherwise, that once that analysis is done or once that utility has run its course, then that piece of data work is essentially disposable.</a> <a data-t="1409">So you have an expensive, very, very expensive effort that now has no more use.</a></p>

<h3>Requirements for a data product</h3>

<p><a data-t="1418">So up to this point, we have talked about this notion of data product in abstract.</a> <a data-t="1421">So let's think about what's our north star.</a> <a data-t="1424">What are we aiming for here?</a> <a data-t="1426">So as I alluded to earlier, it has to serve a coherent set of objectives.</a> <a data-t="1431">So this is one of the core requirements.</a></p>

<p><a data-t="1451">So that limits the scope of the effort.</a> <a data-t="1453">Even with a limited scope, you don't want to be building it from A to Z all in one shot.</a> <a data-t="1457">So think about, again, a product.</a> <a data-t="1459">Like you may have different components that you build and you assemble.</a> <a data-t="1462">It has to be built modularly.</a></p>

<p><a data-t="1467">And if you are building it modularly and your buddy here is building some parts and you are building some parts of it and you're bringing it all together,</a> <a data-t="1533">it has to have, it shouldn't be something that is an art and everybody does it their own way.</a> <a data-t="1545">So collaboration requires some blueprint of how we are building these elements.</a></p>

<p><a data-t="1552">Now, again, collaboration requires everything to be versioned so we don't get tangled up in what version of what elements we are talking about across the team.</a> <a data-t="1563">So by everything here, I mean code, environment, and data needs to be versioned.</a></p>

<p><a data-t="1570">So all of the things that I've talked about so far, as far as the requirements of the product, has to do with the build process.</a> <a data-t="1576">But from a user perspective, it also has to be easy to use.</a> <a data-t="1579">So connect, list, get is all you need to be able to pull the data down and use it.</a> <a data-t="1584">So as a user, you should have an easy interface.</a> <a data-t="1586">And lastly, as I mentioned, we want to have it be platform agnostic.</a></p>

<p><a data-t="1591">So if you're going from S3, later on you need to go to file systems and RStudio Connect is your next destination.</a> <a data-t="1596">The data science workflow should dictate where your data is or if there are other requirements that can be accommodated and you don't have to worry about the functionality of your data being compromised.</a></p>

<p><a data-t="1612">Now, this is the product's requirement.</a> <a data-t="1614">And there should be a build tool that can enable building that product according to those attributes.</a> <a data-t="1624">So you can see the linkage here that a build tool, given all those attributes that I mentioned are requirements, should naturally support modular development,</a> <a data-t="1629">should minimize boilerplate, should enable reproducibility by capturing everything in the state of the code, data, and environment,</a> <a data-t="1637">should encourage best code style practices, should enable full traceability of everything, code, data, environment.</a> <a data-t="1645">And as a builder, you should not be learning a new language.</a> <a data-t="1650">If you know R, if you know Git, you should be good to go.</a></p>

<h3>Data as code: the conceptual approach</h3>

<p><a data-t="1652">Now, if this was instead of a data as a product, this was software as a product, you would rightly expect that the tool is already around.</a> <a data-t="1665">And in the rich set of tools and experiences and processes that have been developed and implemented by DevOps community, when we are talking about the software, we can accommodate all of those features of a build tool.</a> <a data-t="1683">The difference here is that not only we have code, but we have data.</a> <a data-t="1688">So this gives us an inspiration that if we were able to capture data as code, we would then build a bridge to this rich set of tools and experiences that are available in the DevOps community and put those to use for our end.</a></p>

<p><a data-t="1714">So building with data as a code, what's the approach here?</a> <a data-t="1718">Let me walk through an example of how this could work conceptually.</a></p>

<p><a data-t="1726">So let's say you are here in this point one.</a> <a data-t="1727">You have some data sets.</a> <a data-t="1728">Some of you in pharma are familiar with this nomenclature.</a> <a data-t="1729">So let's say you have some LV data, which is this lab data collected on a cohort of patients.</a> <a data-t="1734">You have some medication data and a bunch more data sets that you have at your disposal.</a></p>

<p><a data-t="1741">Now, you start your project.</a> <a data-t="1743">You set the initialization in motion.</a> <a data-t="1747">It starts capturing the environments that you're developing in.</a> <a data-t="1751">It adds the remote locations for your code, for your data.</a> <a data-t="1756">So that initialization is done.</a></p>

<p><a data-t="1758">Now, you go ahead and onboard your data into your data product.</a> <a data-t="1762">So let's start with this LV, which is this collection of lab measurements on a cohort of patients you have.</a> <a data-t="1769">And so you bring that in.</a> <a data-t="1771">And let's say you can now sync that to your remote location, which you have already defined at the initialization.</a></p>

<p><a data-t="1778">So at this stage, this is the part that perhaps worth attention is that you can capture this data that you're pushing to a remote location with a set of metadata in a way that uniquely identifies that.</a> <a data-t="1792">So you're capturing some metadata, like the label for that data, like some description.</a> <a data-t="1798">And this version, which could be essentially a SHA hash of that data, or any other flavor of getting some sort of a hash for that data, uniquely identifies that data set.</a></p>

<p><a data-t="1812">And some additional set of metadata that then will help you retrieve this data when you have the need for it.</a> <a data-t="1817">So now you have converted your data to a set of metadata that uniquely identifies and gives some description about that data.</a> <a data-t="1824">And if you capture that, as you repeat adding additional data sets, you build a manifest that has all the information you need for your input data.</a> <a data-t="1833">And that is the approach of converting data to code.</a></p>

<p><a data-t="1836">Then as you add additional layers of code for logic and you're capturing your environment, all of this is code.</a> <a data-t="1843">You can commit it.</a> <a data-t="1844">You can push it.</a> <a data-t="1845">And when you have done that pushing, in your remote GitHub repository, now you have the build logic, you have the package dependency, and you have all the information you need to identify the input data.</a></p>

<p><a data-t="1858">Now, this recipe is complete in a way that a colleague sitting at this point four or maybe a CI CD machine that gets triggered by the push sitting downstream can build this from scratch and push it again to the same remote data repository in the same fashion, capturing the same set of metadata.</a> <a data-t="1921">Now, everything here has been converted into code.</a> <a data-t="1925">So your data lives in its data location.</a> <a data-t="1928">Your code lives in the code location.</a> <a data-t="1930">And as long as you have access to both, you can build it from scratch.</a></p>

<h3>The daapr package family</h3>

<p><a data-t="1938">We have implemented this framework in R.</a> <a data-t="1942">So it's a family of R packages we call gapper.</a> <a data-t="1949">And it is available for use to build, essentially, implement the framework that I just shared with you all.</a></p>

<p><a data-t="1952">So this family of package consists of three packages, dpi, dpdeploy, dpbuild.</a> <a data-t="1960">Why three packages, you may ask.</a> <a data-t="1962">I can come up with a couple of different answers, historically, in terms of the bulk and code management.</a> <a data-t="1968">But perhaps the most important one is that you can think about three different persona that would be using these packages.</a></p>

<p><a data-t="1976">So starting with dpi, it means it's meant to simplify access to data products.</a> <a data-t="1981">So if you are just a user, whether it's an application or a report, you only need a handful of functions to be able to access the data products.</a> <a data-t="1990">So you don't need the bulk of the rest of the packages.</a></p>

<p><a data-t="1994">dpbuild is meant to simplify building of data products.</a> <a data-t="1998">This is the bulkiest of the packages.</a> <a data-t="2000">This is used by the data developer.</a> <a data-t="2002">So you can think about the persona here as the data developer.</a> <a data-t="2044">dpdeploy can be a data developer.</a> <a data-t="2049">Or it can be a workflow automation machine.</a> <a data-t="2053">So you have the task of dpdeploys to simplify sync and deployment of data products.</a> <a data-t="2058">So you can have a CI-CD process set up where, as your data comes in, it gets pushed.</a></p>
