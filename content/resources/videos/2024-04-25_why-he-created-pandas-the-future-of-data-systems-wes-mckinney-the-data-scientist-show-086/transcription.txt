<h3>Vertical vs. composable tools</h3>

<p><a data-t="1082">So like you mentioned, pandas is more like vertical and it also has a huge user base.</a> <a data-t="1090">So there is a benefit of having a vertical data processing tool.</a> <a data-t="1093">So when do you think is more appropriate for this vertical type of tools and when</a> <a data-t="1099">do you think it's better for composable data tools?</a></p>

<p><a data-t="1131">And so if people want to create, and there've been a number of projects like</a> <a data-t="1135">Modin, Koalas, that have created pandas emulation layers that translate their</a> <a data-t="1142">other operations onto some other execution environment.</a> <a data-t="1145">So there's a company Ponder Data, which used Modin to create a pandas interface</a> <a data-t="1150">to Snowflake.</a> <a data-t="1151">And now they were acquired by Snowflake and are working actively on that in</a> <a data-t="1155">Snowflake.</a></p>

<p><a data-t="1159">And so the challenge is that you can get 90 or 95% of the way there, but there are</a> <a data-t="1164">always going to be some, you're never going to be able to totally take existing</a> <a data-t="1168">pandas code and run it against another processing engine because there's details</a> <a data-t="1168">of pandas that surface in the API.</a> <a data-t="1171">And that's, what's very challenging is because whenever you want to swap out the</a> <a data-t="1175">execution, the storage, if you want to take pandas code and run it at a thousand</a> <a data-t="1180">times larger scale on like a large data set, I think Modin and Ponder are designed</a> <a data-t="1186">to help with that problem, to be able to run pandas code at scale, but there's</a> <a data-t="1190">always going to be some pandas code that it's not going to, that isn't going to be</a> <a data-t="1193">able to be translated and run because it's relying on internals, internal</a> <a data-t="1198">details of pandas data frame actually contains these NumPy arrays internally.</a></p>
