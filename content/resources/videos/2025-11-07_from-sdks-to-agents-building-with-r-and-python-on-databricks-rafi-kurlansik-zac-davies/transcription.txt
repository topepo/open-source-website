<h3>How Databricks addresses the obstacles</h3>

<p><a data-t="626">So coming back to reality, hopefully that's explained a little bit more about how we actually did things.</a> <a data-t="631">Okay.</a> <a data-t="631">That was pretty compelling.</a> <a data-t="631">I think it's worth revisiting sort of how you did that through our lenses here.</a> <a data-t="638">So when it comes to how Databricks actually gets around some of these obstacles, there's really three products that play a major role.</a></p>

<p><a data-t="651">So the first is Unity Catalog, which is essentially the governance layer in Databricks.</a> <a data-t="651">It provides access control for all of your tables, all of your models.</a> <a data-t="658">It even goes to objects like functions and files.</a> <a data-t="658">It's called Unity for a reason.</a> <a data-t="663">But this is basically whatever access Zach has to those assets in Databricks, those are governed by Unity Catalog.</a> <a data-t="669">So when he makes this connection and he's running with the when the assistant uses his authorization via OAuth, it's respecting all of the Unity Catalog access control.</a></p>

<p><a data-t="681">The next thing in terms of compute, do you have enough compute?</a> <a data-t="690">So Zach's actually running this on serverless compute on Databricks.</a> <a data-t="696">So we were basically able to scale to whatever we need for the data volume for any particular query.</a> <a data-t="700">So even if you're running small queries alongside larger queries, serverless compute handles that all for you in the back end.</a> <a data-t="704">You don't have to worry about that.</a> <a data-t="710">So yes, you do have enough compute.</a></p>

<p><a data-t="710">And then another really big piece here is the AI gateway.</a> <a data-t="714">So the idea of the AI gateway on Databricks is no matter which LLM you're using, whether it's an open source model that's hosted on Databricks model serving or if you're using OpenAI or Anthropic, you can essentially control and govern access to those endpoints through the AI gateway.</a> <a data-t="730">So your authentication to OpenAI or something like that can be handled by the AI gateway.</a> <a data-t="743">Anthropic actually has a partnership with Databricks, so you can use Cloud natively in Databricks, so you don't have to worry about the governance there.</a> <a data-t="743">It's totally handled by the Databricks security perimeter.</a> <a data-t="751">And when it comes to tokens, the AI gateway can really help there too, because you can basically set guardrails for how many tokens your users can consume.</a> <a data-t="761">So we can keep things from getting out of control with the assistant.</a></p>

<p><a data-t="774">Just one last thing on the complexity with the authentication.</a> <a data-t="774">In both cases, Zach used OAuth, so he signed in as himself.</a> <a data-t="778">He was prompted by the assistant to basically sign in as himself.</a> <a data-t="784">And those credentials were brought into the Positron session and were used by the assistant.</a> <a data-t="790">So it wasn't able to go beyond the scope of what Zach has access to across these different software.</a></p>

<p><a data-t="799">And the access is limited, right?</a> <a data-t="799">If it's OAuth, the sessions are not infinite in terms of their span, so they're only going to be short-lived credentials.</a></p>
