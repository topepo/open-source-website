<h3>The reproducibility problem</h3>

<p><a data-t="140">Now that's nice in principle, but in practice, things don't go so nicely.</a> <a data-t="145">There was a paper published in 2022 that took a bunch of research outputs, ones that involved R code, and they</a> <a data-t="156">take the look at the data in the R code and they try to rerun those analyses.</a></p>

<p><a data-t="162">And unfortunately, what they found was that 74% of R files included with open research</a> <a data-t="169">failed to run on the first try.</a> <a data-t="171">They would error or some other problem would occur.</a></p>

<p><a data-t="174">And I remember a colleague of mine, Winston, showed me this paper, and I remember being</a> <a data-t="178">really quite disappointed when I saw it.</a> <a data-t="181">I'm a former academic myself, and I remember thinking</a> <a data-t="184">that surely we can do better than that.</a> <a data-t="188">Who are we to say that we find truth when 74% of R code</a> <a data-t="196">fails to run?</a> <a data-t="198">It was really quite upsetting.</a></p>

<blockquote><a data-t="188">Who are we to say that we find truth when 74% of R code</a> <a data-t="196">fails to run?</a></blockquote>

<p><a data-t="202">This is really important, this idea that when we write code, it should be runnable anywhere.</a> <a data-t="207">And for some research codes, like for some research projects, these trials may be going</a> <a data-t="215">on for years.</a> <a data-t="217">So it's good that any code that we do write should be able to run in the same</a> <a data-t="221">way for years to come.</a></p>

<p><a data-t="222">So what do I mean when I say the word reproducible?</a> <a data-t="226">Now, there's no real hard definition for when a piece of software is reproducible or a piece of analysis is reproducible.</a> <a data-t="236">There's just many guidelines.</a> <a data-t="238">One definition that I like is that something's reproducible</a> <a data-t="241">if anyone can go and rerun and verify the result of some computational procedure.</a></p>

<p><a data-t="252">As a follow-on to that, you also get the ability to modify or extend that procedure.</a> <a data-t="259">Perhaps you go and collect more data, or perhaps there is extended software that you've written or</a> <a data-t="265">you can use to extend that piece of work and gain new insights.</a> <a data-t="271">Now, you'll notice with that definition that there's a level of subjectivity, there's a word easily in there.</a> <a data-t="278">And what's easy for some people is not going to be easy for other people.</a> <a data-t="284">I'm a software engineer and I might find quite technical things very easy,</a> <a data-t="289">while a working researcher may not have that knowledge and experience of really working deep with technical software or procedures.</a> <a data-t="300">And they might find certain reproducibility methods more difficult to handle.</a></p>

<p><a data-t="303">And because of that, there are these levels of reproducibility where some things are quite easily done that can make your software run in more places,</a> <a data-t="313">while there are really deep technical ways to ensure that you have a reproducible environment everywhere that are much harder to do.</a></p>

<h3>Managing software environments</h3>

<p><a data-t="601">Okay, so even if you've done all of that, at some point you'll probably take your code and you'll want to run it somewhere else.</a> <a data-t="611">The example I use in sort of the academic setting is if you have some kind of data analysis and you want to send it to someone for peer review.</a> <a data-t="620">This kind of thing happens in science all of the time.</a> <a data-t="624">Now, unfortunately, if you do have software, everyone's computational environment is going to be slightly different.</a> <a data-t="631">They're going to have different versions of software or packages installed, for example.</a></p>

<p><a data-t="636">Here on the left, I've shown some examples of real situations I've seen where different versions of certain packages have caused problems with somebody's code running on someone else's machine,</a> <a data-t="648">whether it's a version of R or Python.</a> <a data-t="651">And also I saw a case once where the version of Node caused a problem, which is a JavaScript engine.</a> <a data-t="657">But even so, even if you have the same version of R installed on a machine, you might have different packages.</a> <a data-t="664">And again, there are tools to help with this.</a> <a href="/software/rig/">Rig</a><a data-t="667"> and pyenv are R and Python tools respectively for managing the versions of R and Python on your machine.</a> <a data-t="675">And </a><a href="/software/renv/">renv</a><a data-t="675"> and venv are packages that will manage the versions of packages for your projects for, again, R and Python.</a> <a data-t="683">They can kind of take a snapshot of the versions of packages installed in your machine and reproduce that environment on somebody else's machine so that you know they've got the same versions of the packages on their machine.</a> <a data-t="697">So when they run your code, it's going to give the same answer.</a></p>

<p><a data-t="698">In addition to that, I've also added p3m here, which is the sort of short name for Posit Package Manager.</a> <a data-t="705">And I do want to shout this out specifically because this is a great tool for building reproducible R scripts and Python scripts.</a> <a data-t="714">Because what it is, is it's a version of the CRAN and PyPy package managers that you can actually, if you go into the setup and run through the options, you can actually pick a specific date in time.</a> <a data-t="731">And if you set that up as your package repository in R or Python, what you'll find is that the versions of the packages from the repository will be at that point in time.</a> <a data-t="743">So you know that if you come back to a project in a year's time, if you use this repository and select the date that project was written, those packages installed will be exactly the same versions.</a> <a data-t="755">So that's really, really useful if you're trying to freeze a point in time for your particular project.</a></p>

<p><a data-t="761">Saying that, even if you have the same versions of packages installed, again, there's another layer, which is sort of system libraries.</a> <a data-t="770">A lot of packages and software itself like R will actually call out to system software.</a> <a data-t="776">And you can have different versions of system software installed on machines.</a> <a data-t="780">So you could have different implementations of Blas and Lepak, for example, depending on whether you're running a Linux machine or Windows machine.</a> <a data-t="789">And in those cases, you can get slight differences in results because of that.</a></p>

<p><a data-t="795">What you'll notice is as I'm going sort of deeper into the layers, you'll find that you get closer and closer to full reproducibility.</a> <a data-t="805">We're reaching the point now with system libraries where you might find small variations in results,</a> <a data-t="812">but generally the overall result should be pretty much the same by this point.</a></p>

<p><a data-t="813">However, if you find that it really is important that a certain system library has to be the same for your results to be consistent, there are ways to do that.</a> <a data-t="822">There are tools like virtual machines and Docker, which are popular in the cloud space for reproducing system environments.</a> <a data-t="830">And there's also a system there, Nix, which is built to, again, reproduce a system environment.</a> <a data-t="836">But Nix is really interesting because it's sort of you build your environment using code.</a> <a data-t="841">And then because of the way Nix works, it can use cryptographic hashes to actually confirm to yourself that you really do have the same version of a certain piece of source code as somebody else running Nix.</a> <a data-t="853">It's a very exact way of making sure that your source code and environments match.</a></p>

<p><a data-t="859">But there are downsides.</a> <a data-t="861">Anyone who's built the </a><a href="/software/tidyverse/">Tidyverse</a><a data-t="861"> from scratch will know that building tools from source, if there are many of them, can take a very long time.</a> <a data-t="871">And the more you have to build from source and the more you have to reproduce in your environment, the longer this will take, especially without things like binary caching.</a> <a data-t="881">And the other downside is that this can be quite difficult to use, particularly very advanced tools like Nix can be hard to use if you haven't seen it before.</a> <a data-t="891">And there is steep learning curves.</a></p>

<h3>Floating point determinism</h3>

<p><a data-t="895">Even with a fully reproducible environment from source code, there is one more level that I want to talk about.</a> <a data-t="903">This one will affect you only very rarely.</a> <a data-t="906">And that's that when you take a piece of source code and you compile it for a computer, it compiles it into a machine-specific language.</a> <a data-t="914">So if you have something like a Windows desktop workstation or a Mac laptop and something like a phone, each of these things will run slightly different versions of machine language.</a> <a data-t="927">And that means even if you have exactly the same source code, if you compile it for these different types of machines, you can get different answers.</a> <a data-t="935">99% of the time, this doesn't really affect you, but it can.</a> <a data-t="940">It can happen.</a></p>

<p><a data-t="941">And here's an example of such a thing.</a> <a data-t="943">Here's an R console.</a> <a data-t="944">And what I've done here is I'm inverting a matrix essentially.</a> <a data-t="948">And you'll see that if you look at the numbers in this example, they're reasonable numbers.</a> <a data-t="953">They're not too big.</a> <a data-t="954">They're not tiny.</a> <a data-t="955">They're just sort of reasonably sized numbers.</a> <a data-t="957">But in fact, this example, those numbers have been specially chosen to demonstrate this problem.</a> <a data-t="963">In this case, there's two different operating systems running those consoles.</a> <a data-t="967">And you'll see that on the left, the result ends in 3015.</a> <a data-t="972">And on the right, it ends in 3046.</a> <a data-t="975">And you do get slightly different answers with this example.</a></p>

<p><a data-t="977">The reason for this, by the way, is something called floating point determinism.</a> <a data-t="982">So you can demonstrate this easier with this example.</a> <a data-t="985">Basically, the problem comes down to the fact that when you're doing floating point computations on a computer, the order of operations matter.</a> <a data-t="994">So when your source code is compiled into machine language, the exact order that the computer chooses to do things makes a very small difference to the answers.</a> <a data-t="1003">So here, I'm adding 0.1 plus 0.2, and then 0.3.</a> <a data-t="1007">And I get an answer, 0.6 stuff.</a> <a data-t="1009">And here, I'm adding 0.2 plus 0.3, and then 0.1.</a> <a data-t="1013">And I get a slightly different answer.</a> <a data-t="1016">If you want to try this yourself in R, do make sure to turn on this option, which tells R to print more digits when it's answering results.</a></p>

<p><a data-t="1026">So the important thing to note here is that really neither of these are perfectly exact.</a> <a data-t="1031">And no computer is perfectly exact when it's calculating floating point or non-integer calculations.</a> <a data-t="1038">And the exact answer that you get can depend on the system you're running on, even if everything else is the same.</a></p>

<p><a data-t="1046">There's some more information about that in my slides, if you do want to know more about that.</a> <a data-t="1050">I won't go into any more detail about this, other than to say that this effect exists.</a> <a data-t="1055">But I do want to just point out that even though these kinds of things are rare, they do happen.</a> <a data-t="1060">Here's an example in the NumPy GitHub repo, where someone asks, I'm calculating the hyperbolic tan of some function,</a> <a data-t="1067">of some number, and I get two different answers on Linux and Windows.</a> <a data-t="1071">And in this case, Linux is just that little bit more accurate.</a> <a data-t="1075">And they're asking, well, why is this?</a> <a data-t="1077">Is there anything that can be done?</a> <a data-t="1079">And one of the developers answers, in general, NumPy doesn't attempt to provide exactly the same results on different platforms.</a> <a data-t="1086">It's essentially impossible for floating point code.</a></p>

<h3>WebAssembly as a solution</h3>

<p><a data-t="1092">So if you really wanted to make your data analysis to the exact number reproducible anywhere, this is a problem.</a> <a data-t="1101">So we think, OK, well, is there a way that we can compile our software so that it's using the same machine language anywhere we take it?</a> <a data-t="1110">And the answer to that is yes.</a> <a data-t="1112">And that's what I'm going to talk about now, this idea of WebAssembly.</a> <a data-t="1117">And what WebAssembly is, is a portable binary code format.</a> <a data-t="1121">So what binary code format means here is the machine language I talked about.</a> <a data-t="1126">And what portable means is that it's the same machine language on any machine.</a> <a data-t="1130">And WebAssembly has really been designed so that if you compile something into WebAssembly language, it will run anywhere with a modern web browser.</a> <a data-t="1140">So even things like phones and tablets, Chromebooks, laptops, computers, they're all running the exact same machine language.</a> <a data-t="1147">And that really gives you very strong guarantees about the reproducibility of the results.</a></p>

<p><a data-t="1153">WebAssembly enables high performance applications on web pages.</a> <a data-t="1157">It's not the first implementation of this idea of a portable binary code format.</a> <a data-t="1163">There is an older thing called Java, which is probably the most popular example.</a> <a data-t="1169">But where WebAssembly really shines is that it's designed to work in modern web browsers through modern JavaScript APIs.</a> <a data-t="1178">And because of that, it's designed to be secure by default.</a> <a data-t="1182">There are very strong limits on what a WebAssembly application can actually do on your machine.</a> <a data-t="1188">It's very strongly containerized and sandboxed inside your web browser.</a> <a data-t="1193">And web browsers are designed really to be very secure systems because obviously you don't want any random website to be able to see your bank details in another tab, for example.</a> <a data-t="1202">Because of that, WebAssembly gains all of these security benefits essentially for free through the browser.</a></p>

<p><a data-t="1208">So I'm the lead developer of a project called WebR, which is a version of the R interpreter that's built for this WebAssembly system.</a> <a data-t="1216">And that allows you to execute R code directly inside your web browser.</a> <a data-t="1221">And importantly, without a supporting R server, which is traditionally how things like Shiny apps run.</a> <a data-t="1229">You can run WebR server-side too.</a> <a data-t="1231">I know there are some people who are running WebR in a node process so that they can run R code on servers they wouldn't otherwise have access to.</a></p>

<p><a data-t="1238">But today I'm just going to be talking about R in the web browser.</a> <a data-t="1242">In addition to the interpreter, WebR includes a JavaScript and TypeScript library so that you can integrate with the R session from JavaScript.</a></p>

<p><a data-t="1252">So here's an example of something doing just that.</a> <a data-t="1255">It's a WebR demo application.</a> <a data-t="1258">And if you go to the URL shown on the screen in my slides, you'll see this webpage come up.</a> <a data-t="1264">And what this is, it's a version of the R interpreter running in your web browser.</a> <a data-t="1268">So you can actually go and use R on machines that don't have R installed or any packages installed.</a> <a data-t="1275">You can just go to this website and use R.</a> <a data-t="1277">If you've seen something like </a><a href="/software/rstudio/">RStudio</a><a data-t="1277"> before, this will be very familiar.</a> <a data-t="1283">It's sort of this four-pane data science look where you have an editor, a console, and plots.</a></p>

<p><a data-t="1290">And what I really like about this slide in particular, these are </a><a href="/software/quarto/">Quarto</a><a data-t="1290"> slides.</a> <a data-t="1296">They're web-based.</a> <a data-t="1297">And because things on the web plug in together very easily, what you find is that this isn't actually an image.</a> <a data-t="1303">It's a real session I can interact with.</a> <a data-t="1305">So I can really interact with an R session directly inside my web browser through these slides.</a> <a data-t="1311">In the editor, I could type some R code.</a> <a data-t="1313">I could do some analysis.</a> <a data-t="1315">Let's plot a histogram.</a> <a data-t="1317">So we'll do RNorm 1000.</a> <a data-t="1319">I can produce images.</a> <a data-t="1321">I can save those to my local disk.</a> <a data-t="1323">And I can even upload data files into this sort of file browser view, analyze them using R script, and produce outputs.</a></p>

<p><a data-t="1331">What's hopefully clear here is that the real entire cycle of basic data science is possible to do inside a web browser reproducibly without having to install any software in your machine.</a> <a data-t="1344">That's the real benefit of these web assembly systems, I think.</a></p>

<p><a data-t="1350">One of the things we found very popular with WebR is using it in teaching situations, too.</a> <a data-t="1356">So you can imagine here, if I was some kind of educator and I had some R code I was showing, one of the really nice things about having R code directly in the slides like this is that if someone asked me, oh, what if we change something?</a> <a data-t="1371">Like if I just change this code to get a slightly different output, I can rerun that code and the output updates live.</a> <a data-t="1379">This is really great for educational uses, not just because you don't have to go to another window, open RStudio, reproduce your image, and then come back.</a> <a data-t="1390">But also, this is kind of reproducible by default.</a> <a data-t="1393">You kind of get it for free in these slides because the fact that you're seeing that image at all means that your entire R script had to run and it had to produce that image.</a> <a data-t="1402">It's not a picture like a traditional slide with a graph image would be, but it's a real live execution of R code shown on the screen.</a> <a data-t="1411">So it has to be reproducible for this to actually show anything at all.</a></p>

<h3>Shinylive and WebAssembly</h3>

<p><a data-t="1418">And that takes me on to Shiny Live.</a> <a data-t="1421">So what this is, is a system for Shiny for R and </a><a href="/software/shiny-python/">Shiny for Python</a><a data-t="1421"> that allows you to run these reproducible applications in Shiny inside your web browser without the need for a computational server.</a> <a data-t="1432">So I'll spend the rest of the time in this talk showing you how Shiny Live works.</a> <a data-t="1438">And then I have some nice examples.</a></p>

<p><a data-t="1440">So a traditional Shiny app looks like this.</a> <a data-t="1443">You will have some kind of server and then the people who visit your app are the clients and they'll open the app in their web browsers.</a> <a data-t="1451">The important thing to take from this slide is that Shiny runs on the server and in particular that means that your server has to be able to run R or Python code.</a> <a data-t="1462">Now that puts limits on the kind of servers that you can use to host your Shiny app.</a> <a data-t="1468">So if you're something like a researcher who's produced some kind of scientific output and you want to share that output with the world or with a reviewer, you're going to have to set up a Shiny server.</a></p>

<p><a data-t="1481">And there are a few ways to do this.</a> <a data-t="1483">The simplest way is that when you run a Shiny app in something like RStudio, it runs a Shiny server in the background.</a> <a data-t="1491">And in that case, the client machine and the server machine are the same.</a> <a data-t="1495">But obviously that's not great once you actually want to distribute that app so that other people can view it.</a> <a data-t="1501">And in that case, you're going to start having to think about, okay, well now I have to manage some kind of Shiny server infrastructure.</a> <a data-t="1509">There are three ways to do this.</a> <a data-t="1511">There's an open source Shiny server and there's also a more paid for and enterprise ways to do this.</a> <a data-t="1519">So you might upload your Shiny app to </a><a href="/software/shinyapps/">shinyapps</a><a data-t="1519">.io or if your institution has something like a Posit enterprise license, there are Posit tools and cloud tools that can make this easy to share Shiny app.</a></p>

<p><a data-t="1534">However, if you don't have access to those enterprise tools and you have something like three or four projects on the go and you've produced three or four Shiny apps that you want to put somewhere on the internet to share, and you don't want to have to manage these over time, then none of these solutions are really a hundred percent great for you.</a> <a data-t="1558">I mean, shinyapps.io is pretty close.</a> <a data-t="1560">You can host Shiny apps for free, but there are computational limits and there are limits on the numbers of apps.</a> <a data-t="1567">So at some point, costs are going to be involved.</a></p>

<p><a data-t="1586">One of the ways to get around this is that you could bundle your Shiny app and its source code and transfer that data to another machine.</a> <a data-t="1595">So you could send your data science output to someone and they run the Shiny app on their machine and they can view your output.</a> <a data-t="1603">Now that's useful for things like long-term archival or even something like peer review, where you're sending your app to a single other person or one or two other people to review.</a> <a data-t="1613">But the problem is it involves all of that reproducible workflow that I talked about in the first half of the talk.</a> <a data-t="1620">Now, all of those tools and methods to make sure that your output is reproducible needs to be taken care of by the person who's reviewing your Shiny app.</a> <a data-t="1630">And the problem is that the person viewing your Shiny app may not be a Shiny expert.</a> <a data-t="1635">They may not have used something like R or Python before.</a></p>

<p><a data-t="1641">So if something goes wrong because of the lack of reproducibility, they're not going to know how to fix it necessarily.</a> <a data-t="1647">So the idea of Shiny Live is, wouldn't it be great if we could run these Shiny apps locally on our machine without having to use a server, but without having to install any extra software or manage this stack of reproducible software.</a></p>

<p><a data-t="1665">The way it works is that when you have a Shiny Live app, the web server no longer has to run R or Python code.</a> <a data-t="1672">It just becomes a static web server.</a> <a data-t="1674">And that means it just serves files rather than running code.</a> <a data-t="1678">And then anyone who visits your Shiny app, Shiny itself loads on their machine locally inside the web browser.</a> <a data-t="1687">And in the case of Shiny Live through WebAssembly is able to run that Shiny app locally.</a></p>

<p><a data-t="1692">One of the nice things about this too, is that serving static files is very efficient.</a> <a data-t="1697">Serving static files on the web has been around a long time and servers are very good at it.</a> <a data-t="1701">And that means that because that R or Python code is not running on that static web server, it's running on the client, the load associated with having many users is shared over each of those users' machines.</a> <a data-t="1715">So there's no chance of overloading something like a Shiny server if you have lots of users, because they all are running a local version of that computation to produce your data science output.</a></p>

<p><a data-t="1727">There are lots of static web servers available.</a> <a data-t="1729">The one that people are probably most familiar with is GitHub Pages.</a> <a data-t="1733">You can serve static files for free on GitHub Pages, but there are others too, both paid and enterprise software is available to serve static files.</a> <a data-t="1742">And like I say, this is a very efficient thing to do.</a> <a data-t="1745">So that's why companies can provide static web services for free or for very cheap.</a></p>

<p><a data-t="1750">When you load Shiny Live, there's a URL on the screen there if you want to try this out yourself.</a> <a data-t="1756">This is a Shiny Live editor, and it's actually running a live Shiny app in my web browser right now.</a> <a data-t="1762">So I can play with the slider and see reactive results.</a> <a data-t="1766">But one thing that's different between this and the traditional Shiny deployment is that I can edit the code.</a> <a data-t="1772">We found this really useful for teaching.</a> <a data-t="1774">In particular, if someone hasn't seen Shiny before, they don't have to install R, they don't have to install Python, they don't have to download the Shiny package.</a> <a data-t="1783">They can just go to this website and they get an editor and they can change the source code and see the results immediately.</a> <a data-t="1791">And we have done this.</a> <a data-t="1792">We've actually run an introduction to Shiny sessions where users have used the Shiny Live web browser for the first time learning Shiny, and they didn't feel like they needed to install the software.</a> <a data-t="1806">They were perfectly happy to learn the basics directly inside the web browser.</a></p>

<p><a data-t="1811">If you do use Shiny Live, one of the things you might want to do is embed a Shiny application inside a wider document.</a> <a data-t="1819">And as I said, these slides are Quarto slides.</a> <a data-t="1821">Quarto can also produce long form documents.</a> <a data-t="1824">So you could write a data analysis describing the trial that you ran and describing what the data shows.</a> <a data-t="1832">And right in the middle of that, you can actually add a Shiny application directly into the source code of that document.</a> <a data-t="1840">If you do that, in this case, there's some slides.</a> <a data-t="1843">It gives you a slide with your Shiny app.</a> <a data-t="1845">And in the case of something like a website output on a Quarto document, it will just drop that Shiny Live app directly inside your long form document, which is really nice for interactive demonstrations of outputs.</a></p>

<p><a data-t="1860">You don't have to use Quarto though.</a> <a data-t="1862">If you have a Shiny app that already exists, you can install the Shiny Live R package.</a> <a data-t="1867">And with one line of code, this will take your Shiny app and convert it into a Shiny Live app.</a> <a data-t="1874">What this does is give you a bundle of files that's ready to transfer to another machine or to host on a static web service like GitHub.</a> <a data-t="1882">If you are sending this to someone and you give them this bundle of files, in theory, all you need to do to run that Shiny app is to run a static web server.</a> <a data-t="1892">And there's various ways to do this.</a> <a data-t="1894">If you have R installed, there's a one-liner that will give you a static server.</a> <a data-t="1898">But even if you're using Shiny for R, the person at the other end doesn't need to have R installed if they have other ways of starting a server.</a> <a data-t="1907">There's a one-liner in Python and also in Node.</a> <a data-t="1910">Any way that you can start a static web server will work.</a> <a data-t="1913">It doesn't have to be within R.</a></p>

<h3>Packages and reproducibility in Shinylive</h3>

<p><a data-t="1915">One of the questions I get asked is, what about packages?</a> <a data-t="1918">R has a very rich package ecosystem.</a> <a data-t="1920">One of the things we do for WebR is to compile as many of these packages as we can into WebAssembly.</a> <a data-t="1927">And we host a WebAssembly binary package repository at the URL there.</a> <a data-t="1932">Not all packages can be compiled for WebAssembly.</a> <a data-t="1935">There's a lot of restrictions in what you can do, particularly security restrictions.</a> <a data-t="1940">But we are compiling as many as we can.</a> <a data-t="1942">And so far, we have about 60% of CRAN packages available for WebR.</a> <a data-t="1947">So there's a good chance that if you use a bunch of packages in your data analysis, you'll either be able to load them directly from this repository, or you'll be able to find some other way to produce your result in WebAssembly.</a></p>

<p><a data-t="1961">One of the things about R packages is that they update and change often.</a> <a data-t="1965">And despite our best efforts, if we want to do something like create a data analysis script or something like a Shiny app, when those packages change, I mean, how often have you gone away for a year, come back to some project, and then all of a sudden you update all your packages and it doesn't run anymore?</a> <a data-t="1982">It does happen despite our best efforts.</a> <a data-t="1985">And one of the changes we've made with the next version of Shiny live is that WebAssembly R packages will be frozen, downloaded, and bundled with your app automatically.</a> <a data-t="1997">So in the current version, when you load a Shiny app, it will go and download the latest versions of packages from the repository.</a> <a data-t="2005">However, in new versions, those packages will be frozen in time.</a> <a data-t="2009">And that means that your apps will continue to work indefinitely.</a> <a data-t="2013">As long as web browsers support WebAssembly, then they'll be able to load those packages because those packages will not change.</a> <a data-t="2021">They'll be fixed and frozen.</a> <a data-t="2022">This is obviously really good for reproducibility because the less things change, the more reproducible your result is going to be.</a></p>

<blockquote><a data-t="2022">This is obviously really good for reproducibility because the less things change, the more reproducible your result is going to be.</a></blockquote>

<p><a data-t="2030">So here's an example of exactly that.</a> <a data-t="2032">So if I just play this video, here is a Shiny app that loads the </a><a href="/software/dplyr/">dplyr</a><a data-t="2032"> package.</a> <a data-t="2038">And what I wanted to really just show here is that when you export this to WebAssembly, you can see that each of those packages that dplyr depends on is being downloaded.</a> <a data-t="2050">So the dplyr packages and the Shiny packages will be downloaded at the time of producing your Shiny live app.</a> <a data-t="2057">And like I say, that means that they'll stay the same.</a> <a data-t="2060">They won't change.</a> <a data-t="2061">So here's the output of that exact app.</a> <a data-t="2063">And I have my filter based on dplyr.</a> <a data-t="2066">And if for whatever reason, that code that produces that filter was to change, it wouldn't matter.</a> <a data-t="2072">This app would continue to work because that version of the package is now fixed and part of the bundle for that Shiny live app.</a></p>

<p><a data-t="2033">The R Consortium Submissions Working Group is an interesting application of all of these ideas.</a> <a data-t="2042">They're running a pilot at the moment based around producing Shiny apps in WebAssembly for clinical trial submissions to the FDA.</a> <a data-t="2051">So they're looking at the WebAssembly technology and the container technology, things like Docker and Podman, as ways of producing Shiny applications that are reproducible.</a> <a data-t="2061">So that if you're working in, say, the pharma space and you have some clinical trial output, you can send that output to the FDA in the form of a Shiny app that they can interact with.</a> <a data-t="2074">They can open it on their machine without having to install R and they can review that work.</a> <a data-t="2080">There's a good post on the Pharmaverse blog talking about the work they've been doing in this direction.</a> <a data-t="2086">So if you're interested in this, do go and read that.</a> <a data-t="2089">They're doing some really great work.</a></p>

<p><a data-t="2099">Do bear in mind that not all R packages will work under WebAssembly.</a> <a data-t="2103">We are trying to compile as many as we can within the restrictions of the WebAssembly security sandbox, but work is ongoing here.</a> <a data-t="2112">If you do see a package that you use and isn't available in the WebR binary repository, do feel free to go and open a GitHub issue on the WebR repo, and we'll take a look at that when we can.</a> <a data-t="2125">Also, if you have your own R packages, custom R packages can be compiled into WebAssembly, and we're currently working on building GitHub actions to make this easier.</a> <a data-t="2135">So for your own package, you'll be able to attach a GitHub action so that when you release your package, it will go and build it into WebAssembly.</a> <a data-t="2143">The same kind of tools are also available on R Universe if you use R Universe.</a> <a data-t="2148">This is still experimental and we need to do some documentation updates and clarify some things, but as time goes on, this should hopefully become much easier and you'll be able to use your own R packages as part of a ShinyLive app.</a></p>

<p><a data-t="2160">Also, bear in mind that the idea here is not that ShinyLive will replace a traditional Shiny deployment.</a> <a data-t="2166">There'll always be the cases where traditional Shiny servers are needed.</a> <a data-t="2172">One example that someone told me, which I thought was good, was that they use Shiny as a front end to submit jobs to high-performance computing clusters.</a> <a data-t="2181">Now, in that case, something like a laptop or a phone is not going to have the same kind of computational power as those Shiny servers.</a> <a data-t="2190">In those cases, a traditional Shiny deployment is always going to be required.</a> <a data-t="2195">Another reason you might need a traditional Shiny deployment is for browser security restrictions.</a> <a data-t="2200">Things like connecting to a database, for example, can't be done in ShinyLive at the moment because the web browser simply does not give you that level of access to the network.</a> <a data-t="2210">So if you have Shiny apps that do things like connect to a database and do some work on that data, in those cases, you'll want a more traditional Shiny app.</a></p>

<p><a data-t="2223">And the biggest thing to bear in mind when you're building a ShinyLive app is that there are no secrets.</a> <a data-t="2228">Anytime you write some R code or you have some kind of data in your ShinyLive app, that is available to the client.</a> <a data-t="2235">Even if on a normal Shiny deployment, that data would not be sent to the client, with a ShinyLive deployment, everything is sent to the client.</a> <a data-t="2243">So no secrets or tokens should be in your source code when you write a ShinyLive app.</a> <a data-t="2249">That's really, really important.</a></p>
