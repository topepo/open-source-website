<h3>The data access problem in enterprises</h3>

<p><a data-t="282">So I think for much of Posit's early history we mostly saw enterprise customers manage access to data if they were doing so at all via Windows file shares and on-prem database management systems like Oracle SQL Server.</a> <a data-t="295">And over time actually our enterprise products ended up working reasonably well in this context.</a> <a data-t="302">But I think over the last five years or so, because everything is slow in enterprise, we've seen a pretty dramatic shift among our customers to the cloud, surprise.</a> <a data-t="316">And a lot of the fairly simple access control mechanisms that we were accustomed to and the credential paradigms that we used worked in this on-prem context have not served folks well in this brave new cloud world.</a></p>

<p><a data-t="340">So to give a hopefully humorous example of this, suppose you are your team inside of your organization is tasked with analyzing customer churn, so classic problem right.</a> <a data-t="353">So when you get started on this project you know that your company stores basic customer data in Salesforce so you know you'll eventually probably want to access it by its Salesforce API.</a> <a data-t="363">It turns out that this means asking your IT department to register a Salesforce external app for you which takes a few weeks and when it comes back that comes back with a warning not to embed the OAuth client credentials in scripts or reports, whatever that means.</a></p>

<h3>Common themes in enterprise data access challenges</h3>

<p><a data-t="517">So I think that like the common themes that we hear today when we're talking about the challenges of getting access to data inside of organizations would be complexity.</a> <a data-t="527">So although there's a limited number of cloud or data platforms — so this is essentially AWS, Azure, GCP, Snowflake, and Databricks — those are kind of from our vantage point the biggest ones that we see.</a> <a data-t="537">Many customers are using more than one, many customers are actually using all of them together.</a> <a data-t="542">And everyone seems to be in the middle of a years-long migration to an internal data platform that is yet to materialize.</a> <a data-t="547">And so you are even more likely to be pulling data from multiple places in the same projects.</a></p>

<p><a data-t="553">The second thing that we see is new forms of authentication and authorization.</a> <a data-t="559">The native language of these cloud platforms is IAM and OAuth, not usernames and passwords.</a> <a data-t="570">And the third major theme we see is more scrutiny I would say about the use of credentials.</a> <a data-t="575">So when I worked as a data scientist the entire department, actually the company, shared a single database account and password.</a> <a data-t="582">This would be considered wildly, actually wildly, unsecure even at the time.</a> <a data-t="588">But it would be wildly inappropriate today.</a> <a data-t="589">IT folks are increasingly unhappy with data scientists managing long-lived credentials themselves.</a> <a data-t="594">They really do not want you to embed your personal Snowflake username and password inside of a </a><a href="/software/quarto/">Quarto</a><a data-t="594"> report, which is absolutely the formal guidance that Snowflake sent me via email and PDF a year ago.</a></p>

<p><a data-t="612">Not only have the technologies changed, the expectations have changed as well.</a> <a data-t="615">And actually something a co-worker reminded me of this morning, there was a big data breach at Snowflake about six weeks ago.</a> <a data-t="620">It was caused because someone was using a service account that they did not use single sign-on with, unsurprisingly because it's a service account so you can't do single sign-on directly anyway.</a></p>

<p><a data-t="633">I think that something that Posit has always valued both on the open source side and in its professional products is meeting people where they are, helping data scientists be heroes inside of their organizations.</a> <a data-t="644">And right now we think that an increasing amount of heroism is coming from the ability of our users to navigate these complex data platforms.</a></p>

<h3>Workbench managed credentials</h3>

<p><a data-t="656">So with that in mind I think beginning about two years ago we started building deep integrations into Workbench to manage data and cloud platform credentials for you.</a> <a data-t="669">And today, as of the Workbench release this week, I hope it's this week, that includes managed credentials for AWS, for Azure, for Databricks, and Snowflake.</a></p>

<p><a data-t="688">So what this means in practice, because you're all nerds here, is that we've taught Workbench to speak these new languages like IAM and OAuth.</a> <a data-t="695">The idea is that when you arrive in an IDE like RStudio, all of the credentials you need to access your organization's data are already present quietly in the background.</a> <a data-t="705">You don't need to manage an AWS access key, you don't need to manage a Databricks personal access token, you don't need a Snowflake service account.</a> <a data-t="710">You don't as a data scientist need to do this work yourself.</a> <a data-t="716">This makes not only you happy but it also makes your admin and hopefully IT folks very happy as well.</a></p>

<p><a data-t="724">Now on the subject of being nerdy you may be interested to know that the way this works under the hood is that Workbench will manage configurations for files and environment variables for you.</a> <a data-t="732">We will do things like run through this OAuth stuff, we will refresh tokens for you, we will generate these configuration files.</a> <a data-t="739">And we've been very very careful to make this as compatible with the existing tools as possible.</a> <a data-t="745">So you'll find things like the AWS </a><a href="/software/cli/">CLI</a><a data-t="745"> just like works in RStudio, you'll find that the Databricks SDKs just work inside of Visual Studio Code in Python, the Snowflake CLI works.</a> <a data-t="759">Like an amazing percentage of this work was actually just very carefully reading the public source code of these SDKs and figuring out how they worked, figuring out how they manage credentials and teaching Workbench to operate in that paradigm.</a> <a data-t="772">And something actually Edgar talked about a little bit today is that we then took a lot of what we had learned by looking particularly at the Python ecosystem</a> <a data-t="777">and then like ported some of those credential paradigms to R,</a> <a data-t="781">largely through the </a><a href="/software/odbc/">odbc</a><a data-t="781"> package but also through other mechanisms.</a></p>
