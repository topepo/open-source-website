<h3>The tidyverse lifecycle</h3>

<p><a data-t="347">Now let's start to get into some more of the details of dealing with change in code.</a> <a data-t="351">And this is a place where the metaphor of the hardware store starts to break down a</a> <a data-t="355">little.</a> <a data-t="356">Because once you've purchased something from the hardware store, it stays the same.</a> <a data-t="360">But when you use code from a package, that code might work differently when you update</a> <a data-t="365">the package.</a> <a data-t="366">This is a, on the whole, this is normally a good thing.</a> <a data-t="369">This is like the package developer can come into your house, replace all of those expensive,</a> <a data-t="375">inefficient, incandescent bulbs with new LED bulbs that are much more efficient.</a> <a data-t="378">You get new features, you get bug fixes, you get performance improvements.</a> <a data-t="383">But any updates to a package also comes with a risk that it might break your code.</a></p>

<p><a data-t="391">And I want to talk in a little bit more detail about how we think about that, particularly</a> <a data-t="395">in the tidyverse.</a> <a data-t="396">These ideas apply in general, but I'm going to talk about the kind of framework that we</a> <a data-t="400">are using currently in the tidyverse.</a> <a data-t="402">And I want to talk a little bit about the kind of set of promises, the tidyverse money</a> <a data-t="407">back guarantee, if you will, that we want to provide to you so that you can better understand</a> <a data-t="414">what is likely to break your code and what's likely not to break your code.</a> <a data-t="418">And to do that, we first have to talk a bit about the tidyverse lifecycle.</a></p>

<p><a data-t="424">So you might have seen this before, because the documentation of certain functions contains</a> <a data-t="429">these badges, like lifecycle deprecated or lifecycle superseded.</a> <a data-t="435">So the full tidyverse lifecycle is relatively complicated, but today I wanted to talk about</a> <a data-t="440">the four most important components, experimental, deprecated, stable, and superseded.</a> <a data-t="448">And you can kind of roughly divide these in half by whether their functions in these various</a> <a data-t="452">stages are kind of out of warranty or in warranty.</a> <a data-t="457">And I'll talk about what that means precisely shortly.</a> <a data-t="460">But roughly, you know, if a function is out of warranty, it might change at any point.</a> <a data-t="464">If it's in warranty, we have some pretty strict, we make some pretty strict promises about</a> <a data-t="469">how it will change.</a></p>

<p><a data-t="471">So out of warranty, we've got experimental on the one hand.</a> <a data-t="474">These are new functions, new features.</a> <a data-t="477">We don't know exactly how they should work, but we want to get them out into the world</a> <a data-t="480">so that you can try them out and give us feedback.</a> <a data-t="483">This is the type of code that you need to have a pretty close relationship with.</a> <a data-t="487">You need to be working with it a lot because it's going to change based on your feedback.</a> <a data-t="493">Now most functions are on a stable lifecycle.</a> <a data-t="496">That means the developer is basically happy with them and has no major plans to change</a> <a data-t="501">in the future.</a></p>

<p><a data-t="503">There's two ways that a function can exit that stable lifecycle stage.</a> <a data-t="508">It can become deprecated, which means basically that it's a bad idea, you should stop using</a> <a data-t="514">it.</a> <a data-t="515">Or it can become superseded.</a> <a data-t="516">We know that there's a better alternative, but the old function's not going away.</a></p>

<p><a data-t="523">So the first thing we're going to guarantee, the first promise we're going to make you</a> <a data-t="528">is if we're going to try and avoid as much as possible to make breaking changes in any</a> <a data-t="533">stable functions.</a> <a data-t="535">So a breaking change is a change that we expect, we predict, will break the vast majority of</a> <a data-t="541">correctly written code that uses it.</a> <a data-t="544">So what are some examples of breaking changes?</a> <a data-t="547">Well firstly, if you remove a function, obviously any code that uses that function is going</a> <a data-t="552">to break.</a> <a data-t="554">Or if you remove an argument to a function, any code that uses the argument.</a> <a data-t="558">Or if you decrease the set of allowed inputs to an argument, that's also going to break</a> <a data-t="562">some code.</a> <a data-t="564">So in general, making a function, the scope of a function kind of smaller is going to</a> <a data-t="569">be a breaking change because code that works today will not work after this change happens.</a></p>

<p><a data-t="577">Now as well as the input to a function, there's also the output.</a> <a data-t="580">Changing the type of an output, type of the output of a function is also a breaking change.</a> <a data-t="585">If a function previously returned a numeric vector and now it returns a data frame, it's</a> <a data-t="590">likely to break.</a> <a data-t="592">So what are non-breaking changes?</a> <a data-t="594">Well of course we can change the output because we have to be able to fix bugs.</a> <a data-t="599">And this means that technically something that's a non-breaking change can still actually</a> <a data-t="603">break your code if you've accidentally depended on a bug.</a> <a data-t="607">So this is a really important topic that I'm going to come back to later when we talk about</a> <a data-t="611">off-label usage of functions.</a> <a data-t="614">Otherwise, non-breaking changes are generally increasing the scope of things, so increasing</a> <a data-t="620">the possible set of allowed inputs, adding new arguments, or adding new functions.</a> <a data-t="625">So anything that grows the interface of a function or a package is going to be a non-breaking</a> <a data-t="630">change.</a></p>

<h3>Deprecation in practice</h3>

<p><a data-t="633">Now sometimes we will have to make a breaking change to a function that is stable, but we're</a> <a data-t="639">never going to do that suddenly.</a> <a data-t="642">We're going to do it through a gradual process of deprecation.</a> <a data-t="645">We're going to give you a chance to find out that something is changing and respond to</a> <a data-t="650">that change before we take it away for good.</a> <a data-t="653">So what does that look like?</a> <a data-t="654">Well let's take a function from the tbl package called data underscore frame.</a> <a data-t="659">Now we, probably I, created this function when we just started working with the idea</a> <a data-t="665">of tbls, and because tbls are kind of like a modern reimagining of a data frame, I thought</a> <a data-t="670">well let's give it a modern name by taking the dot and replacing it with an underscore.</a> <a data-t="677">Pretty soon afterwards I realized that was a, kind of cutesy, and b, misleading, because</a> <a data-t="682">this function doesn't return a data frame, it returns a tbl.</a> <a data-t="686">So we decided to deprecate that function.</a></p>

<p><a data-t="689">So what happens if you call that function today?</a> <a data-t="692">Well you're going to get a warning that tells you it's deprecated, and it tells you when</a> <a data-t="695">it was deprecated.</a> <a data-t="697">So tbl 1.1 was actually released in 2016, about four years ago.</a> <a data-t="701">So deprecated functions don't go away immediately, they're going to hang around for a while so</a> <a data-t="706">that you can find out about the change.</a> <a data-t="709">Generally more, the more important or the more widely used a function was, the longer</a> <a data-t="714">that deprecation message will hang around.</a> <a data-t="718">Next you're going to find out what you should use instead.</a> <a data-t="721">So here you can use a tbl function instead, it does exactly what it says on the tin, it</a> <a data-t="725">creates a tbl for you.</a></p>

<p><a data-t="728">Now what happens if you're looking at some old code that has hundreds of data underscore</a> <a data-t="734">frame calls in it?</a> <a data-t="736">It would be super annoying if every single one of those generated a message, so we're</a> <a data-t="740">only going to show you this warning every eight hours.</a> <a data-t="744">But it's also important for you to fix these deprecated functions, so we're going to help</a> <a data-t="748">you find out where they were by giving you this advice to call this lifecycle's last</a> <a data-t="754">warnings function.</a></p>

<p><a data-t="757">There's one important thing that you do not see here, and that's an error.</a> <a data-t="761">So a deprecated function does not generate an error, it still works, it still does exactly</a> <a data-t="765">the same thing it used to do, but it is going to go away in the future, and the warning</a> <a data-t="770">message is going to encourage you to update.</a></p>

<p><a data-t="775">So if you do call that last warnings function, you'll get that same message again, and you'll</a> <a data-t="779">also get a backtrace.</a> <a data-t="781">And a backtrace is just a sequence of function calls that eventually lead to that deprecated</a> <a data-t="786">function.</a> <a data-t="787">In this made up example we've got a function f that calls g that calls h that calls i that</a> <a data-t="791">finally calls data underscore frame.</a> <a data-t="793">And this is quite important because it allows you to narrow in exactly where that warning</a> <a data-t="797">is coming from as quickly as possible.</a></p>

<p><a data-t="800">We really do not want deprecated functions to be that one smoke alarm in your house that</a> <a data-t="806">needs its batteries changed, and it feels like the beep is like perfectly calibrated,</a> <a data-t="812">it's fast enough, it happens frequently enough that it like drives you to the point of insanity,</a> <a data-t="817">but not frequently enough that you can ever figure out which of the four smoke alarms</a> <a data-t="822">is actually coming from.</a></p>

<h3>Superseded functions</h3>

<p><a data-t="827">Now the lifecycle stages, I've talked about them mostly in the context of functions, they</a> <a data-t="832">also apply to packages in slightly different ways, and they also apply to arguments of</a> <a data-t="837">functions and even specific values of arguments of functions.</a> <a data-t="843">And so for example in the nest function in </a><a href="/software/tidyr/">tidyr</a><a data-t="843"> 1.0, we wanted to update it to match</a> <a data-t="849">emerging consensus of interface, function interface across the tidyverse, so we deprecated</a> <a data-t="854">one argument and we deprecated another way of using dot dot dot.</a></p>

<p><a data-t="861">So deprecated functions are functions that are kind of moving, deprecated functions,</a> <a data-t="865">features, arguments are things that are moving out of warranty.</a> <a data-t="869">What about things that we just kind of regret a little, they're not wrong, but maybe there's</a> <a data-t="875">a better alternative available now.</a> <a data-t="878">That's the idea of a superseded lifecycle stage, which I'm going to talk about with</a> <a data-t="882">a little example.</a></p>

<p><a data-t="884">Because a few years ago it sort of felt like there was this disturbance in the tidyverse</a> <a data-t="890">force that a lot of people were having problems remembering which of spread and gather was</a> <a data-t="897">which and how do you actually use them.</a> <a data-t="900">Like every time you go to use them you'd have to look up the documentation or do some time</a> <a data-t="903">googling.</a> <a data-t="905">And this just wasn't happening to the community, this was also happening to me, like I forgot</a> <a data-t="910">how to use these functions, which seemed like a really good sign it was time to put some</a> <a data-t="914">work into them.</a> <a data-t="916">And so last year we spent a bunch of time talking to the community, looking at other</a> <a data-t="920">approaches taken by packages like datatable, and we came up with two new functions, pivot</a> <a data-t="925">longer which makes your data longer, and pivot wider which makes your data wider.</a> <a data-t="931">And overall it seems like that those new functions have been successful.</a> <a data-t="937">People have found them, or at least people in this cherry-picked sequence of tweets have</a> <a data-t="941">found them much easier to use and remember.</a></p>

<p><a data-t="946">But just because pivot longer and pivot wider exist and are great and are easier to remember,</a> <a data-t="951">it doesn't mean that spread and gather are wrong or bad.</a> <a data-t="956">They still do what they say in the tin, they still do, they still work the same way that</a> <a data-t="961">they always have, and by any means they're some of the most successful functions I've</a> <a data-t="966">ever written, probably used by hundreds of thousands of people.</a> <a data-t="970">So spread and gather are never going to go away, but we want to make it clear that newer</a> <a data-t="977">approaches are available.</a> <a data-t="979">And that's the idea of this superseded lifecycle badge, which basically says this function,</a> <a data-t="983">the development is now complete, it's never going to get any new features, and it will</a> <a data-t="988">only receive the sorts of critical bug fixes that will keep that function alive and useful.</a></p>

<p><a data-t="995">And to me the way that I think about superseded functions is kind of like the idea of building</a> <a data-t="999">codes.</a> <a data-t="1001">So a building code is a set of best practices that you need to apply when you are building</a> <a data-t="1005">a new home today.</a> <a data-t="1008">And obviously over time those standards evolve.</a> <a data-t="1012">So what happens if you have a 50-year-old home or a 100-year-old home, do you have to</a> <a data-t="1016">go and update your house to meet all of today's standards?</a> <a data-t="1021">That would be crazy.</a> <a data-t="1023">So the idea of building standards is they are important when you are building a new</a> <a data-t="1027">house or whenever you touch something in your existing house.</a> <a data-t="1030">So if you're renovating your kitchen and discovering the wiring is no longer up to code, you're</a> <a data-t="1034">going to have to replace that, but you don't have to rip out all of the walls, all of the</a> <a data-t="1038">wires in your house, elsewhere in your house.</a></p>

<p><a data-t="1041">And the same applies to superseded functions.</a> <a data-t="1044">Really good idea to use them in new projects.</a> <a data-t="1046">If you are touching old projects, if you're renovating old projects, update superseded</a> <a data-t="1052">functions when you need them, but don't otherwise worry about it.</a> <a data-t="1054">It's totally fine for the old and the new to coexist seamlessly.</a> <a data-t="1060">The other place it's worthwhile updating is if you're teaching.</a> <a data-t="1063">Really important when you're teaching to be teaching the latest best practices as much</a> <a data-t="1066">as possible.</a></p>

<p><a data-t="1069">You might wonder, well, when are these superseded functions going to go away?</a> <a data-t="1075">And you know, I don't want to make any promises like I can't commit to.</a> <a data-t="1079">Who knows what our priorities are going to be in five years or 10 years and what our</a> <a data-t="1083">resources will be.</a> <a data-t="1085">But I think you can get some sense of how we care about superseded functions by looking</a> <a data-t="1090">at some superseded packages, like for example, the reshape package, which I wrote in 2005</a> <a data-t="1097">and was superseded in 2010 by reshape2.</a> <a data-t="1100">So because people are still using it, because packages are still using it, we've kept reshape</a> <a data-t="1106">still working.</a> <a data-t="1107">It's not getting any new features, so it's not a lot of work.</a> <a data-t="1110">We're going to keep that alive on CRAN as long as possible.</a> <a data-t="1112">Similarly, same story for plier and reshape2.</a> <a data-t="1116">Both created a long time ago.</a> <a data-t="1117">They've been superseded.</a> <a data-t="1119">We're going to keep those functions, those packages alive and on CRAN for the foreseeable</a> <a data-t="1124">future.</a></p>

<h3>Off-label usage of functions</h3>

<p><a data-t="1129">Now so far, I've talked about these lifecycle stages, which we as the creators of the functions</a> <a data-t="1135">have control over.</a> <a data-t="1138">There's an important thing that you have control over, which is how you use functions.</a> <a data-t="1143">And I want to talk a little bit about the kind of off-label usage of functions.</a> <a data-t="1148">Because in medication, off-label usage is basically prescribing a medicine for something</a> <a data-t="1153">it's not formally approved for.</a> <a data-t="1157">This is not illegal.</a> <a data-t="1160">It's not wrong.</a> <a data-t="1161">It's useful in lots of cases because there's lots of knowledge about medication that hasn't</a> <a data-t="1164">been formally approved yet.</a> <a data-t="1166">But it is a little riskier than using something for its intended purpose.</a> <a data-t="1171">And particularly for functions, it exposes you, I think, to a greater risk of breakage.</a></p>

<p><a data-t="1178">So what exactly do I mean by off-label usage?</a> <a data-t="1181">Let's take a little example.</a> <a data-t="1182">I have got a factor, and I want to extract the underlying numeric levels of the factor.</a> <a data-t="1189">Now one way to do this is to use the C function, because the C function doesn't know how to</a> <a data-t="1196">handle factors, and so it just drops all the levels.</a> <a data-t="1200">But I would say that this is off-label usage because the job of the C function is to concatenate</a> <a data-t="1205">vectors, concatenate or combine vectors together.</a> <a data-t="1209">And here we've only got one vector.</a> <a data-t="1211">So you, or someone else, you in the future, or someone else looking at this code, might</a> <a data-t="1215">scratch your head a little bit and think, well, what does it mean to combine one vector?</a> <a data-t="1221">Here you're really relying on kind of a side effect, an unintended side effect of the C</a> <a data-t="1225">function, which, because it doesn't know about factors, drops those levels.</a> <a data-t="1230">You're going to be much better off explicitly declaring that you want to get the integer,</a> <a data-t="1234">underlying integer levels by using something like as.integer.</a></p>

<p><a data-t="1239">Or maybe you've got a data frame, and you want to find all the rows in this data frame</a> <a data-t="1243">where x equals 1.</a> <a data-t="1245">And you do that using subsetting in R, and because R has this fantastic rule that missing</a> <a data-t="1250">values don't go silently missing, you're going to get a missing row in the output, which</a> <a data-t="1255">a lot of times is not what you want.</a> <a data-t="1257">And so you might think, well, I'll just use which to get rid of this.</a> <a data-t="1261">Now this certainly isn't such an egregious off-label usage as the last example, but the</a> <a data-t="1267">primary purpose of which is to find the location of true values in a logical vector.</a> <a data-t="1274">So this, you know, it's hard to know what else which would do with missing values, but</a> <a data-t="1279">you're not really fully communicating what you're trying to do here.</a> <a data-t="1283">What you're trying to do is remove missing values.</a> <a data-t="1285">And so in this case, I think you're better off switching to a function like </a><a href="/software/dplyr/">dplyr</a><a data-t="1285"> filter</a> <a data-t="1289">or a base subset, explicitly designed for subsetting rows of a data frame, which knows</a> <a data-t="1295">that you don't want to get those Na rows back.</a></p>

<p><a data-t="1299">And the reason you want to avoid this off-label usage is because the original author has no</a> <a data-t="1304">way to anticipate what you're doing.</a> <a data-t="1307">And so when they make changes to the function, it's much more likely to negatively affect</a> <a data-t="1311">these off-label usages than the documented usage.</a> <a data-t="1317">So and another way of saying that is off-label usage is going to void your warranty.</a> <a data-t="1321">All bets are off when you start using a function in a way that the author did not envisage.</a></p>

<blockquote><a data-t="1317">So and another way of saying that is off-label usage is going to void your warranty.</a> <a data-t="1321">All bets are off when you start using a function in a way that the author did not envisage.</a></blockquote>

<p><a data-t="1329">So how do you avoid it?</a> <a data-t="1332">Well, the first thing you need to recognize is that recognizing, understanding off-label</a> <a data-t="1334">usage and what the intended purpose of a function is, is a skill.</a> <a data-t="1338">It's not something that you're born knowing.</a> <a data-t="1340">It's something you're going to develop over the course of your career in R.</a> <a data-t="1344">And in particular, when you start out learning R, your primary goal is to solve the problem</a> <a data-t="1350">in front of you and get the result that you need.</a> <a data-t="1353">And that is 100% OK.</a> <a data-t="1356">But as you get better, it's a really good practice to think, are you using functions</a> <a data-t="1360">not just because they do what you want, but because they do what they say?</a> <a data-t="1366">And the easiest way to discover that is to read the documentation.</a> <a data-t="1371">If you are using a function for a purpose that is not mentioned anywhere in its documentation,</a> <a data-t="1376">you are getting a little, you're likely to be in the danger zone.</a></p>

<p><a data-t="1383">But I think the best technique for learning more about the intended usage of a function</a> <a data-t="1386">is to do code review.</a> <a data-t="1388">So this is going to help you when you get your code reviewed by someone else, because</a> <a data-t="1392">they're going to look at that, they're going to look at you concatenating a single factor</a> <a data-t="1395">and they're going to ask you, like, what are you doing here?</a> <a data-t="1398">Because the chances are they don't know all the same quirks of the functions that you</a> <a data-t="1403">do.</a> <a data-t="1404">They'll know the intended purpose of a function, but they're less likely to know some side</a> <a data-t="1407">effect that you're relying on.</a> <a data-t="1411">But code review also helps you when you're reading other people's code, because you're</a> <a data-t="1414">going to look at that code and you're going to think about it.</a> <a data-t="1416">You're going to say, well, like, what is this doing?</a> <a data-t="1420">Maybe you'll read a little bit of documentation, you'll learn some new functions.</a> <a data-t="1424">And that process of reflecting on other people's code is really going to help you understand</a> <a data-t="1430">and improve your ability to write good code.</a></p>

<p><a data-t="1433">Personally, one of the biggest improvements to my coding style, I think, happened when</a> <a data-t="1437">I started grading code, when I started teaching data analysis at Rice University.</a> <a data-t="1443">That process of having to read other people's code and think about, like, why was this hard</a> <a data-t="1448">to understand, really made my own code much more elegant and clearer.</a></p>

<h3>Avoiding unforeseen consequences</h3>

<p><a data-t="1457">Now, while off-label usage of a function will void your warranty, we as maintainer</a> <a data-t="1463">of the package still want to avoid unforeseen consequences as much as possible.</a> <a data-t="1468">And I think there's a really interesting unforeseen consequence of this transition from incandescent</a> <a data-t="1474">light bulbs to LED light bulbs.</a> <a data-t="1477">And that is, a few years back, Citi started converting their traffic lights from incandescent</a> <a data-t="1483">to LED, because it saved a bunch of time, they don't need to be replaced as frequently,</a> <a data-t="1487">and they used much, much less power.</a> <a data-t="1490">But unfortunately, the first winter after this happened, they discovered a problem.</a> <a data-t="1496">That snow was accumulating in the traffic lights and blocking them so people couldn't</a> <a data-t="1500">see them.</a> <a data-t="1501">And this was because one of the reasons that incandescent bulbs were so inefficient was</a> <a data-t="1509">that they also produced a bunch of heat.</a> <a data-t="1512">And so now, previously, they would have melted the snow, and now with these new LED bulbs,</a> <a data-t="1519">they didn't, snow accumulated, and there were a bunch of car crashes.</a></p>

<p><a data-t="1523">So in the tidyverse, we want to avoid that as much as possible by trying, particularly</a> <a data-t="1528">when we make big changes, to making sure that there aren't common off-label usages that</a> <a data-t="1532">we need to support.</a> <a data-t="1534">And I want to talk about this in the context of a recent change to Magrida, a set of big</a> <a data-t="1539">changes that improved backtraces, so when you get an error message, it's much easier</a> <a data-t="1544">to see where in the pipeline it actually happened, there's much less overhead, so it's faster,</a> <a data-t="1551">and it is much closer to the upcoming base pipe.</a> <a data-t="1555">So I think these are three really big and important improvements, but to get them, we</a> <a data-t="1560">had to fundamentally re-engineer how the pipe worked.</a> <a data-t="1564">And our analysis suggested that it should be fine for, you know, any correct usage of</a> <a data-t="1569">the pipe, or any way that we used the pipe, but we wanted to make sure it's not going</a> <a data-t="1574">to, there weren't a bunch of uses that we hadn't imagined that we still needed to support.</a></p>

<p><a data-t="1580">So we did three things, and you can learn from those three things.</a> <a data-t="1583">So the first thing we always do is we always run our command check on all of the packages</a> <a data-t="1590">that use our package on CRAN.</a> <a data-t="1593">Now code in CRAN packages is not exactly the same as data analysis code, but it is a very</a> <a data-t="1599">large body of R code that is very easy for us to rerun.</a> <a data-t="1602">So this is a great first pass to pick up if there's any major problems with a new version</a> <a data-t="1607">of our package.</a> <a data-t="1608">And this is also, you know, if you can go to the effort of putting a package on CRAN,</a> <a data-t="1613">this also gets you some extra protection before any package in the tidyverse changes, all</a> <a data-t="1618">of your tests will be rerun and will tell you if we've broken something.</a></p>

<p><a data-t="1623">We also tweeted about it, we tweet about pretty much everything we do, following us on Twitter</a> <a data-t="1628">is a great way to keep up with the latest and greatest changes in the tidyverse.</a> <a data-t="1633">But for changes of this magnitude, we also blog about it.</a> <a data-t="1636">And so in this case, Leonel wrote a blog post saying, hey, there's a big new version of</a> <a data-t="1641">Magrida coming along, please try it out and let us know what you think.</a> <a data-t="1646">Does it break anything in your code?</a></p>

<p><a data-t="1650">And in the process of coming up with a slide, I realized if you don't use Twitter and you</a> <a data-t="1654">don't use blogs, how are you supposed to keep up with these things?</a> <a data-t="1658">So we're going to add a mailing list so you can easily sign up for updates over email</a> <a data-t="1662">in the not too distant future.</a></p>

<p><a data-t="1665">Now when we did this for Magrida, we actually uncovered a bunch of smaller and not so small</a> <a data-t="1669">things.</a> <a data-t="1670">So we either fixed Magrida to make this work, or if we couldn't do that, at least make it</a> <a data-t="1676">give an informative error message.</a> <a data-t="1679">But there were still a few things where the bit tradeoff between the amount of time it</a> <a data-t="1683">would take to fix them versus the frequency of the usage in the community.</a> <a data-t="1688">And at the end of the day, we have to kind of make this balance that we are going to</a> <a data-t="1692">pick these large positive improvements for the vast majority of people is an okay tradeoff</a> <a data-t="1698">for a few new problems for a very small amount of people.</a></p>

<h3>Opting out of package updates</h3>

<p><a data-t="1703">Now even after learning about the tidyverse lifecycle and the various promises that we</a> <a data-t="1709">make to you, you might still feel nervous that for particularly critical projects, you</a> <a data-t="1715">might be worried that an otherwise innocuous package update might break your code.</a> <a data-t="1721">And so you have a fallback, which is to opt out of the hurly-burly of package updates.</a> <a data-t="1726">Now this is generally not a good idea because you're not just opting out of changes that</a> <a data-t="1732">might break your code, you're also opting out of bug fixes and improvements in new features.</a> <a data-t="1738">But for some types of code, particularly code that is run unattended like code in production,</a> <a data-t="1745">it can be a good idea to use one of these techniques so that your code is 100% isolated</a> <a data-t="1751">and protected from changes to other packages.</a></p>

<p><a data-t="1755">So the first thing you can do is use the </a><a href="/software/renv/">renv</a><a data-t="1755"> package.</a> <a data-t="1758">Basically what this is going to do is it's going to isolate each project you work on,</a> <a data-t="1761">give it its own library of R packages.</a> <a data-t="1764">So it doesn't matter what you install elsewhere, this project will always use the same set</a> <a data-t="1769">of packages.</a> <a data-t="1771">It also comes with a pair of functions that let you save those package versions to disk</a> <a data-t="1776">and then restore them elsewhere so you can run your code on another computer and it'll</a> <a data-t="1780">hopefully give you exactly the same results because you've installed exactly the same</a> <a data-t="1784">versions of packages.</a> <a data-t="1787">Now because renv works by giving you a custom library of packages, an isolated library of</a> <a data-t="1791">packages, it works with any packages and it doesn't matter where it comes from, whether</a> <a data-t="1796">it's CRAN or GitHub or your company's internal package repository.</a></p>

<p><a data-t="1802">But renv is quite a lot of work to set up, it's not a huge amount of work, but there</a> <a data-t="1805">is a simpler alternative if you only care about CRAN packages and that's to use a CRAN</a> <a data-t="1810">time machine.</a> <a data-t="1811">So there are two alternatives here, you can use the public version of </a><a href="/software/rstudio/">RStudio</a><a data-t="1811">'s package</a> <a data-t="1815">manager or you can use Microsoft's MRAN.</a> <a data-t="1818">Both of these basically work the same way.</a> <a data-t="1821">They take a regular, roughly daily snapshot of CRAN and so you can choose to pick a day</a> <a data-t="1826">in the past and install packages that were on CRAN on that day.</a> <a data-t="1833">Now this is obviously much simpler to set up, you just have to change the repos option</a> <a data-t="1838">in your R profile or your R configuration, but it has the limitation that it only works</a> <a data-t="1843">with CRAN packages.</a></p>
