<h3>Trying existing tools</h3>

<p><a data-t="241">But the tools at hand were feeling not just right.</a> <a data-t="245">The one thing I knew I needed in this documentation system was long format docs, and that's kind</a> <a data-t="250">of to tell people the when and the why to use different packages.</a> <a data-t="255">And the R world really has this down to a science.</a> <a data-t="258">In fact, they have a whole name for it, called vignettes, and vignettes are great for telling</a> <a data-t="263">a story.</a> <a data-t="265">It gives context to some sort of, like, small workflows that a package is useful for doing.</a></p>

<p><a data-t="273">And I tried to use a tool in Python that was really excellent in long format documentation.</a> <a data-t="283">This tool is called make docs, and it was great to get started.</a> <a data-t="286">It felt pretty ergonomic, and that was really exactly what I wanted.</a> <a data-t="290">I could write some markdown.</a> <a data-t="292">But I realized, actually, my users cannot live on vignettes alone.</a> <a data-t="297">While people were fantastically prepared for, like, when to use docs and why they should</a> <a data-t="303">be running this better code, there wasn't much intuition on, like, how to use certain</a> <a data-t="308">functions if I only had vignettes and, like, examples.</a> <a data-t="312">So this documentation tool was not just right.</a> <a data-t="315">In order to get this API docs, I'd have to wrangle a bunch of extensions, and it just</a> <a data-t="319">didn't really feel like something that I wanted to do with this docs tool.</a></p>

<p><a data-t="362">And this is on purpose.</a> <a data-t="364">This is because there are tools out there to programmatically export this doc string</a> <a data-t="369">into beautiful, beautiful websites.</a> <a data-t="372">Here you can see I used a package called Sphinx, and I could get my function name, the package</a> <a data-t="379">name, you know, Vennifer pin metrics in the yellow.</a> <a data-t="382">You don't really have to read it if you don't want to, but there are, like, the parameters</a> <a data-t="386">there as the function signature.</a> <a data-t="389">In green, we have the description of the function, and we have the parameters with the description</a> <a data-t="393">of each parameter in blue.</a> <a data-t="395">It's kind of a lot, but it's all there.</a> <a data-t="397">I didn't have to write this twice.</a> <a data-t="399">It just automatically got created into this website, and it was super cool.</a></p>

<p><a data-t="557">And because this is YAML, it looks and feels like a bulleted point list.</a> <a data-t="561">Which means it's super literate and maintainable for me later on.</a> <a data-t="564">I'm not trying to wrangle scripts.</a> <a data-t="567">I just get to put this in the Quarto.yaml file.</a> <a data-t="572">If you're a Quarto user, you realize this is just the same file you're using anyway</a> <a data-t="576">to create your site.</a> <a data-t="578">It's just got a few extra lines at the bottom.</a></p>

<p><a data-t="580">And if we want to go line by line of how to make a quartodoc website, you're going to start</a> <a data-t="586">with a YAML chunk that's quartodoc specific.</a> <a data-t="591">And if you want, you can have this sidebar that quartodoc will create for you.</a> <a data-t="597">And you actually don't really have to touch this.</a> <a data-t="599">But if you think about how you're ingesting documentation, you kind of have the function</a> <a data-t="603">here and a sidebar of all the other functions on the side.</a> <a data-t="607">This is when it is being created.</a> <a data-t="610">You also have to say the name of the package you would like to document.</a> <a data-t="613">It needs to be imported into the environment you're running this in.</a> <a data-t="617">And then you get to create your index page.</a> <a data-t="620">And this page is kind of like the landing page for all your functions.</a> <a data-t="624">You're going to give it a title, like my golden functions.</a> <a data-t="627">And a description that they are just right.</a> <a data-t="630">It's going to be a scrollable list of all your functions separated by however many headers</a> <a data-t="634">you would want.</a></p>

<p><a data-t="636">And underneath these headers, you list what you would like to document.</a> <a data-t="642">Here we can see I have my house function and then my Goldilocks class.</a> <a data-t="647">And actually, if you just wanted to list your classes out, that would be just fine.</a> <a data-t="651">But if you want to have a less nested structure and see every single attribute and method</a> <a data-t="658">in the sidebar, you can list them out this way for a little bit of a flatter architecture.</a> <a data-t="664">If you just wanted to put Goldilocks alone, if you clicked on Goldilocks in your sidebar,</a> <a data-t="669">it would put all of these attributes and methods automatically in your documentation.</a> <a data-t="673">So you're not missing out on information.</a></p>
