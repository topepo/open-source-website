<h3>How tidymodels fits a model</h3>

<p><a data-t="201">Okay, so quickly some background on what tidymodels is actually doing when you go to fit a model.</a> <a data-t="208">So at the top of this diagram, I have this code reading fit, boost tree.</a> <a data-t="208">We're modeling the outcome as a function of all of the predictors, and we're using this data set called d.</a> <a data-t="221">What actually happens in this situation is that tidymodels, the tidymodels team doesn't actually implement any of the model training</a> <a data-t="228">or prediction code that we're running.</a> <a data-t="236">There are a lot of our users who are intimately familiar with the details of different statistical models, and we trust those practitioners to be the ones</a> <a data-t="242">to implement the modeling code.</a> <a data-t="248">What we do as the tidymodels team is put a wrapper on top of that</a> <a data-t="248">training and prediction code that unifies the interface.</a></p>

<p><a data-t="255">So this code fits a boosted tree model,</a> <a data-t="261">the code to fit a logistic regression or a decision tree or all sorts of other model types,</a> <a data-t="261">and the code to do so with all sorts of different modeling packages is exactly the same regardless.</a> <a data-t="275">So the first step is that we translate that inputted code into a code that the modeling</a> <a data-t="275">engine will understand.</a> <a data-t="280">So the default modeling engine for a boosted tree in tidymodels is actually XGBoost.</a> <a data-t="287">That code will take some amount of time to evaluate, and it will return its result back to tidymodels.</a> <a data-t="295">Tidymodels situates it in a common user interface, and the fit function returns its value.</a></p>

<p><a data-t="304">So this code takes some amount of time to run.</a> <a data-t="304">I've shown the portions of that evaluation time that the tidymodels team is responsible for in green, and I've shown the</a> <a data-t="310">portions that the modeling engine is responsible for that the tidymodels team can't necessarily control in orange.</a> <a data-t="324">In reality, the green portions here are on the order of like a couple milliseconds,</a> <a data-t="324">and they don't tend to scale with the size of the data.</a> <a data-t="332">The orange segment here is, again, the implementer of the modeling engine is responsible for how long that code takes to run,</a> <a data-t="339">and that tends to scale with the size of the data.</a> <a data-t="343">So larger data sets take longer to fit.</a></p>

<p><a data-t="352">So if I'm tuning a boosted tree, that analogous diagram in that situation is that we're fitting</a> <a data-t="352">many, many models, and we're doing the translation step on either side, but the tuning process</a> <a data-t="359">requires trying out a bunch of different arguments to the boosted tree model and fitting it to a bunch of different subsets.</a> <a data-t="370">So in this example, we're trying out 10 possible boosted tree</a> <a data-t="376">argument settings, and for each of those 10 argument settings, we're fitting that model</a> <a data-t="383">to 10 different subsets of the data.</a> <a data-t="383">That's called resampling.</a> <a data-t="390">So there's 100 model fits total happening here.</a> <a data-t="390">This tuning process that we're visualizing right now is the one that took three hours and 41 minutes.</a> <a data-t="403">So how can we cut down on the time to tune that model by over 99%?</a></p>

<h3>Four changes to speed up tuning</h3>

<p><a data-t="404">The first thing that we can do is distribute our computations across CPU cores.</a> <a data-t="415">So modern laptops are pretty incredible. Many have many, many cores.</a> <a data-t="415">The most recent MacBooks are looking at double digits.</a> <a data-t="423">I'm working on a laptop right now that's four years old, and I can use four CPU cores at a single time.</a> <a data-t="431">So instead of fitting 100 models in a row one at a time, we can fit four at a time, and best case scenario, cut down on the amount of</a> <a data-t="439">time to tune this boosted tree by four.</a> <a data-t="448">In tidymodels, all you need to make this happen is one line of code.</a> <a data-t="448">If you're familiar with the future framework, tidymodels integrates with future,</a> <a data-t="454">and so this is just future code as it is, and this one line of code will tell tidymodels</a> <a data-t="459">all it needs to know about how to distribute computations on your laptop.</a></p>

<p><a data-t="468">Next, I'm going to pick a modeling engine that is particularly well suited to the modeling</a> <a data-t="468">problem that I'm working on.</a> <a data-t="479">So by default, the modeling engine in our boosted tree case is XGBoost.</a> <a data-t="487">I happen to know, based on the way that this data set that we're using as an example</a> <a data-t="487">looks, that a modeling engine called LightGBM might fit faster on that example.</a> <a data-t="494">So we've cut down on the width of this diagram by a pretty significant factor already, both by distributing</a> <a data-t="499">our computations across four CPU cores and choosing a model that fits faster than the default model.</a> <a data-t="515">It's not always the case that a given modeling engine will fit faster than another,</a> <a data-t="517">regardless of the data context that you're working with.</a> <a data-t="520">There are situations where XGBoost is more performant, but this is a matter of building your intuition through practice,</a> <a data-t="525">and I'll show you at the end of this talk through some resources that I'll share with you.</a></p>

<p><a data-t="532">In tidymodels, there's no sort of syntax foo that you need to do</a> <a data-t="537">to translate your fitting code from one modeling engine to another.</a> <a data-t="541">All you need to do is set the engine argument to a different value.</a></p>

<p><a data-t="545">Next, we can use something called the sub-model trick.</a> <a data-t="545">This is the third modification that I'll talk about.</a> <a data-t="551">The sub-model trick is a bit of a rabbit hole, so we're not going to go too deep down that rabbit hole.</a> <a data-t="556">But the idea here is that we can evaluate many more models than we actually fit.</a> <a data-t="568">The sub-model trick enables you to fit a single model and predict across several of them,</a> <a data-t="569">and tidymodels has some tricky stuff going on in the back end to make sure that we</a> <a data-t="574">enable that as often as we can.</a> <a data-t="580">The way that you can explicitly opt into that sub-model trick as a user is by using a function called grid regular, and using a regular grid will enable the sub-model trick.</a> <a data-t="595">In cases where you're just tuning a single model parameter, this just works if it's available, so there's nothing you need to do as a user.</a> <a data-t="601">But this is for added lines of code here.</a></p>

<p><a data-t="605">And the fourth technique that we'll use is something called racing.</a> <a data-t="609">So, I said earlier that we're training or we're choosing between 10 different possible values,</a> <a data-t="613">and we're doing so by fitting each of those 10 different possible argument values to 10 different subsets of the data.</a> <a data-t="627">What happens in practice is that we can fit each of those 10 possible models to 3 of those subsets, or even 2, and that will give us enough information to realize</a> <a data-t="644">that some of those models are much more likely to be the most performant, the best possible model we have access to, than others.</a> <a data-t="649">And so, instead of continuing to train all of those models that we're pretty sure are not going to outperform the most performant,</a> <a data-t="656">and we can bound that pretty sure by a significance level, we can just go ahead and give up on those and not fit them to the</a> <a data-t="668">remaining 8 or 7 or 6 resamples, and that will cut down on our time to tune even further.</a></p>

<p><a data-t="678">In tidymodels, to make this happen, all you need to do is change one line of code.</a> <a data-t="682">That default grid search can be transitioned to racing with a repeated measures ANOVA model.</a> <a data-t="689">That's called tune race ANOVA, and that can cut down on your time to tune as well.</a></p>

<p><a data-t="696">And so, altogether, we started with this tuning process that fitted every possible model in sequence, and each of those model fits took a good while, and that resulted in a training process that</a> <a data-t="707">was 3 hours and 40 minutes.</a> <a data-t="714">That will really hinder your ability to iterate as you develop models interactively.</a> <a data-t="721">We can make four small changes that require changes to seven lines of code, give or take, depending on how liberally you add spaces, to transition that code to run</a> <a data-t="728">in 90 seconds, and you don't have to sacrifice predictive performance on your way to doing so.</a></p>
