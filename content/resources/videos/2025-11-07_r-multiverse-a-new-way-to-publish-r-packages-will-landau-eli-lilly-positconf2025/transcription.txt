<p><a data-t="0">We are all in great company here.</a> <a data-t="2">I count myself as a package developer as well.</a> <a data-t="6">And we write an R package, we want, chances are, we want to share it with as many people</a> <a data-t="13">as will find it useful.</a> <a data-t="14">We want it to see the light of day.</a></p>

<p><a data-t="17">And that brings us to the dreaded ordeal of publishing.</a> <a data-t="23">And that's something that we have in common with authors of books, of journal articles,</a> <a data-t="32">and of other kinds of media that are published in general.</a> <a data-t="36">We have the standard practices generally to go through a central publisher, like CRC Press</a> <a data-t="43">for books or Journal of the American Statistical Association for journal articles.</a> <a data-t="48">And for us, we have repositories like CRAN, Bioconductor, and ROpenSci.</a></p>

<p><a data-t="54">This is generally the best way to help our work see the light of day, to make sure people</a> <a data-t="61">use it, to convince people to trust what we put out there.</a> <a data-t="65">But it comes with a challenge.</a></p>

<h3>The challenge of central publishing</h3>

<p><a data-t="69">Inherent to the model of central publishing, whether it's a book or article or package,</a> <a data-t="76">is editorial review.</a> <a data-t="79">And for sure, these review experiences can be extremely rewarding and edifying.</a> <a data-t="84">But they also come with a lot of frustration and a long wait time to wait for feedback.</a> <a data-t="92">And then there's ingesting that feedback, coming up with revisions, and you might even</a> <a data-t="97">get rejected.</a> <a data-t="98">Even if you don't get rejected, you might have to do something to your package that you</a> <a data-t="104">find difficult or maybe not even agree with.</a></p>

<p><a data-t="107">And so, a lot of folks turn to self-publishing, which is possible, again,</a> <a data-t="115">for books and journal articles and packages.</a> <a data-t="119">We all have ways of doing this.</a> <a data-t="121">And the reasons we might want to do this is that, as package developers,</a> <a data-t="126">we control publishing.</a> <a data-t="130">No one has to tell us what to do.</a> <a data-t="130">The freedom is really nice, but it comes at a cost.</a> <a data-t="133">Again, this other extreme of the trade-off, because we have a hard time convincing users</a> <a data-t="139">to trust our work.</a></p>

<p><a data-t="140">If on a self-published repository, as a user, I might express more skepticism</a> <a data-t="145">of something that's self-published rather than something that's formally vetted</a> <a data-t="149">in a central repository.</a> <a data-t="151">And also, your work is harder to find if you self-publish.</a> <a data-t="155">And speaking of obscurity, let's talk a little bit about siloing.</a> <a data-t="159">If I'm a user and I install a package from GitHub or even from our universe,</a> <a data-t="166">not only do I need to know the name of each package I want to install,</a> <a data-t="169">I need to know every package's individual repository.</a> <a data-t="173">And for a user with hundreds of downloaded installed packages on my system,</a> <a data-t="178">I don't want to have to do that for every single one.</a></p>

<h3>Introducing R-multiverse</h3>

<p><a data-t="183">Which brings me to R Multiverse, which is a new dual repository built on top</a> <a data-t="190">of the R Universe project.</a> <a data-t="192">And what it aims to do is provide a centralized place for self-published</a> <a data-t="198">releases, and on top of that, automated checks and quality control</a> <a data-t="204">for production scenarios.</a> <a data-t="205">What we're trying to do is meet this middle ground to try to combine the best</a> <a data-t="211">of both worlds of self-publishing and centralized publishing with review.</a></p>

<blockquote><a data-t="205">What we're trying to do is meet this middle ground to try to combine the best</a> <a data-t="211">of both worlds of self-publishing and centralized publishing with review.</a></blockquote>

<p><a data-t="218">Now, let's get into how this works.</a> <a data-t="222">With R Multiverse, it's very new.</a> <a data-t="225">It's a dual repository, so it's not just one repository, it's two.</a> <a data-t="230">And our first repository is this community repository.</a> <a data-t="235">It's just plain and simple, an R Universe with all the latest releases</a> <a data-t="239">of registered packages.</a> <a data-t="241">And on top of that, we have a production repository with quarterly snapshots</a> <a data-t="246">of the healthy releases from the community repository.</a></p>

<p><a data-t="250">So my goal for the rest of this talk is to help you understand each</a> <a data-t="253">of these repositories, what they mean for users, and what they mean for developers</a> <a data-t="259">who want to participate and contribute packages.</a></p>

<h3>The community repository</h3>

<p><a data-t="263">So a bit about this community repository.</a> <a data-t="267">So it's, like I said, plain and simple, an R Universe,</a> <a data-t="271">just like any other R Universe.</a> <a data-t="273">It's got a nice dashboard with package documentation, metrics,</a> <a data-t="277">and test results, and it's transparently available.</a> <a data-t="282">We have a nice custom URL for it, both for the dashboard and</a> <a data-t="285">for downloading packages.</a> <a data-t="288">And the latest releases come directly from the individual source,</a> <a data-t="293">GitHub and GitLab repositories, just like any other universe does</a> <a data-t="298">on the R Universe platform.</a></p>

<p><a data-t="301">And the implications of this, because this is a single universe that is offered</a> <a data-t="307">to the entire community to participate in.</a> <a data-t="310">Usually, individual universes are maintained by individual users,</a> <a data-t="314">individual labs, or individual developers, I should say,</a> <a data-t="316">who want to publish their own work.</a> <a data-t="318">This is something that we all share, and it's also focused on releases.</a> <a data-t="322">What that means is it's a middle ground between development on the one hand</a> <a data-t="327">and production on the other.</a></p>

<p><a data-t="329">There's this term QA, or quality assurance, in industry and</a> <a data-t="333">software development.</a> <a data-t="334">And to my knowledge, we've never had this for R before,</a> <a data-t="338">but that's exactly what this community repository aims to do.</a> <a data-t="342">It's this middle ground.</a> <a data-t="344">It's releases that the developers endorse for production,</a> <a data-t="349">but are not necessarily ready for production because we don't enforce tests</a> <a data-t="354">or guarantees of compatibility at this level precisely.</a> <a data-t="357">That's for production later on.</a></p>

<p><a data-t="359">So, that's what the community repository is.</a> <a data-t="363">You can imagine that you would want to install, maybe if I'm a user,</a> <a data-t="368">I would install a small number of packages from the community universe where I want</a> <a data-t="374">just the latest and greatest from a small number of packages that I'm familiar</a> <a data-t="378">with, that I trust, and developers can just very quickly and easily get releases</a> <a data-t="383">there very fast.</a></p>

<h3>Registering a package</h3>

<p><a data-t="392">Now, how do you, as a package developer, get your package into the community</a> <a data-t="397">universe and registered with R multiverse in general?</a> <a data-t="402">So, we keep our submission requirements very minimal at this level.</a> <a data-t="408">So, we want to make this experience as close to self-publishing as we can.</a> <a data-t="413">And so, what really this boils down to is the artifact that you register has</a> <a data-t="419">to be an R package with at least one release on GitHub or GitLab.</a> <a data-t="425">It has to be devoid of malicious activity, has to be, you know,</a> <a data-t="430">in good faith and not have malware, et cetera, et cetera.</a> <a data-t="434">And it's got to have a free and open-source license and the most</a> <a data-t="440">obvious standard place.</a></p>

<p><a data-t="440">So, we, as the administrators of R multiverse, have permission</a> <a data-t="445">to distribute the code.</a> <a data-t="447">And so, all we're asking is that it's an R package and it's legal and safe.</a> <a data-t="452">We do have policies, and there's maybe a bit more to than that,</a> <a data-t="457">but we generally take after the GitHub terms of service and acceptable</a> <a data-t="462">use policies.</a> <a data-t="463">So, very straightforward and, you know, for well-meaning authors to abide</a> <a data-t="465">by this.</a></p>

<p><a data-t="466">And that is all we ask for registered packages in community.</a> <a data-t="472">So, the registration procedure, again, is simple and it's open,</a> <a data-t="477">it's open-source.</a> <a data-t="478">We have this contributions repository with a collection of registered listings</a> <a data-t="485">of packages.</a> <a data-t="486">And these are just text files where it's one text file per package and the text</a> <a data-t="490">file usually has the URL of the source code repository of the package.</a> <a data-t="495">And all you'll do is you'll go into the web interface, you will submit a pull</a> <a data-t="500">request to add a new text file where the title of the file is your package and the</a> <a data-t="505">contents is just one line with the source code URL.</a></p>

<p><a data-t="509">And then what happens in the pull request is there is a bot that runs about every</a> <a data-t="514">hour and it scans the pull requests and it just checks, is this a package?</a> <a data-t="519">Does it have a license?</a> <a data-t="520">Are there any conflicts with an existing package with the same name on CRAN,</a> <a data-t="524">et cetera, et cetera?</a> <a data-t="525">And if those very, very light checks pass and there's a high degree of trust,</a> <a data-t="538">let's say the maintainer is a member of a trusted GitHub organization,</a> <a data-t="544">then the bot can automatically merge and register the package.</a> <a data-t="548">In a lot of cases, you know, if there's some finding with a check,</a> <a data-t="552">it doesn't necessarily mean a rejection, it just means a manual review is</a> <a data-t="555">required.</a></p>

<p><a data-t="561">And again, according to these very light submission requirements.</a> <a data-t="566">Downstream of that, another automated process compiles all these listings and</a> <a data-t="570">automatically generates an R universe package manifest with, again,</a> <a data-t="574">the package URLs and it sets the branch field equal to star release.</a> <a data-t="577">And what that means is instead of pulling the latest commit,</a> <a data-t="580">we're pulling the latest release off of GitHub and GitLab.</a></p>

<p><a data-t="585">The implication of that is that new releases,</a> <a data-t="589">if you want to update a registered package that already exists in our multiverse,</a> <a data-t="594">you do not have to go through human.</a> <a data-t="597">You can just straightforwardly create a new release on GitHub.</a> <a data-t="601">This is what a lot of us as package developers do anyway when we create a</a> <a data-t="607">release of a package and we want to submit it to other standard repositories.</a> <a data-t="612">It's great practice to create a release on GitHub and GitLab anyway.</a> <a data-t="616">And this is all that it takes to update a new version in the community repository.</a> <a data-t="622">And then your package will be straightforwardly available to install.</a></p>

<p><a data-t="627">No matter what package is in community, again, for users,</a> <a data-t="629">it's this one unified URL that users can install from with install.packages.</a> <a data-t="635">It's very simple and easy.</a> <a data-t="638">The dependencies might come from other repos like CRAN, and so</a> <a data-t="642">this getOptionsRepos is useful to make sure the dependencies are also installed</a> <a data-t="647">if needed.</a></p>

<h3>The production repository</h3>

<p><a data-t="651">So that's the community repository.</a> <a data-t="652">The production repository has in mind that we use R for</a> <a data-t="659">really tightly regulated, tightly controlled situations and</a> <a data-t="664">things that are not just exploratory analyses.</a> <a data-t="667">We run R for high stakes analyses that really need to be correct and</a> <a data-t="671">based on software that we trust.</a> <a data-t="673">And that's what production is aimed at.</a> <a data-t="675">And our approach is a bit like Debian.</a></p>

<p><a href="/people/jeroen-ooms/">Jeroen Ooms</a><a data-t="680"> came up with the spark for this idea,</a> <a data-t="684">borrowing from Debian's philosophy of gradually building snapshots at intervals.</a> <a data-t="690">And we're optimistic that this way of thinking will turn out to not only</a> <a data-t="696">provide the same quality for users, but also provide the same convenience for</a> <a data-t="700">developers.</a></p>

<p><a data-t="702">So how this works is we take quarterly snapshots of the community universe, and</a> <a data-t="707">we only include healthy releases with healthy dependencies</a> <a data-t="711">from that community universe.</a> <a data-t="713">And we gradually build each snapshot from the ground up,</a> <a data-t="716">gradually to give developers enough time to make the adjustments they need</a> <a data-t="722">leading up to a snapshot, and also to avoid last minute volatility and panic.</a></p>

<p><a data-t="731">What we require from production, it's a bit more.</a> <a data-t="735">All the requirements of community apply, plus we enforce warnings and</a> <a data-t="740">errors in R command check for Mac, Linux, and Windows.</a> <a data-t="744">For a particular version of R, we require that the version number of</a> <a data-t="750">the current release is higher than the others, so this is a benefit for users.</a> <a data-t="755">And we also require that the dependencies of this package in our multiverse</a> <a data-t="760">also pass the same checks.</a></p>

<p><a data-t="762">Our universe actually has this really nice feature that as soon as the checks</a> <a data-t="767">of a package are finished, all the downstream packages of degree one that</a> <a data-t="771">depend on it in the same universe are also rechecked.</a> <a data-t="776">And what our multiverse does is it waits for those checks to complete, and</a> <a data-t="779">effectively what that means is we have the same reverse dependency check</a> <a data-t="784">guarantees that we're used to and that we rely on, but</a> <a data-t="788">coming at it from a different operating model.</a></p>

<p><a data-t="792">Now, our operating model is kind of looks like this in terms of</a> <a data-t="797">where do we want to go with a snapshot and what does it look like?</a> <a data-t="800">We want a collection of releases from our multiverse set in stone,</a> <a data-t="805">built on a particular version of base R, and</a> <a data-t="810">a snapshot of packages from CRAN also set in stone.</a></p>

<p><a data-t="814">That's where we want to arrive at, and</a> <a data-t="817">what we end up starting with is not a pyramid of stone,</a> <a data-t="823">more like a pyramid of water, where any package could change at any time.</a> <a data-t="828">That's where we start, and we gradually freeze this ecosystem from the ground up,</a> <a data-t="832">and we start by fixing a version of base R and</a> <a data-t="835">a snapshot of CRAN packages from Posit Public Package Manager.</a> <a data-t="839">And we say, those aren't gonna change for the entirety of the snapshot, and</a> <a data-t="845">we give an entire month for developers to adjust their packages and checks.</a></p>

<p><a data-t="849">And after that, we start to freeze in the next month</a> <a data-t="853">the packages that already work in multiverse.</a> <a data-t="856">We freeze the healthy packages, and we give time for bugs to be fixed.</a> <a data-t="861">So, and this is a gradual, nice, soft landing and</a> <a data-t="865">soft freeze that allows the changes that enforce quality to happen, but</a> <a data-t="870">we provide stability along the way.</a></p>

<p><a data-t="873">And what we end up with is maybe not all the packages from community, but</a> <a data-t="877">certainly a lot of them.</a> <a data-t="879">And it's this nice model that guarantees this compatibility of</a> <a data-t="884">the packages and their dependencies in this ecosystem.</a></p>

<p><a data-t="890">And for user installing from production, you would supply a dated snapshot URL,</a> <a data-t="896">and the Posit Public Package Manager snapshot from two months prior.</a></p>
