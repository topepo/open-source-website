<h3>What is async?</h3>

<p><a data-t="72">So going back to the title of the presentation, what do we actually mean when we say next-generation</a> <a data-t="78">async? Or what do we mean by async in the first place? Well, let me give you a very simple</a> <a data-t="84">definition, which is if we say that parallelism is the ability to do multiple things at once,</a> <a data-t="91">then async is just not waiting around while that's happening.</a></p>

<p><a data-t="97">Now, this definition, this concept isn't very controversial in a lot of other programming</a> <a data-t="105">languages. So even if you've never programmed in Rust or Go or JavaScript, you might have heard</a> <a data-t="112">that async generally works very well in those contexts.</a> <a data-t="121">Most modern websites have some JavaScript in them. So if you open a web page,</a> <a data-t="128">perhaps on your mobile, and you click a button, you generally expect it to just work. When you're</a> <a data-t="135">hitting that button, you're not likely to be wondering whether that thing is going to hang on</a> <a data-t="141">you.</a></p>

<p><a data-t="150">And that conveniently brings us to where we are now in R. So if I bring back this definition</a> <a data-t="158">of async, so we have this in mind, then I suggest that perhaps most of us are missing what I term a</a> <a data-t="167">first class async experience.</a> <a data-t="167">And why do I say that? Well, I think for most of us, our typical</a> <a data-t="172">experience with parallelism is with the parallel package, which has been part of base R for over</a> <a data-t="178">20 years now. And that is simply parallel and not async. So you send tasks to a lot of parallel</a> <a data-t="185">workers, but you're waiting until they all happen, and then you collect the results at the end.</a></p>

<p><a data-t="184">Of course, there are many other excellent packages, such as Polar, which is used for</a> <a data-t="191">local parallelism because it relies on writing and reading files from disk. But again, that's not</a> <a data-t="199">always async. If you create what are called persistent sessions, then if those sessions are</a> <a data-t="204">all busy doing jobs, you cannot send another job onto those sessions.</a> <a data-t="212">And that actually is the</a> <a data-t="212">same thing with the future package, which also promises async. <h3>crew: extending mirai to high-performance computing</h3>

<p><a data-t="522">Thanks. I'm Will. I am a statistician and package developer. And I am mirai's all time</a> <a data-t="530">biggest fan. <p><a data-t="566">And we throw at this all the advanced modeling we can because these are some of the</a> <a data-t="572">most important decisions that can be made with statistics. And those models take a long time to</a> <a data-t="578">run, potentially hours for a single model. And when we design the clinical trial, it gets even</a> <a data-t="585">more challenging because to design a trial, we often have to simulate it. And to simulate a trial,</a> <a data-t="593">we often need to run these models thousands of times. If you were to try this on your local</a> <a data-t="602">laptop, you would never get it done.</a> <a data-t="610">We need entire armies of computers to do this. Armies</a> <a data-t="610">of computers that are working together on a cluster that's managed with a resource manager</a> <a data-t="616">like Slurm or Grid Engine or a cloud system or a service like Batch.</a></p>

<p><a data-t="616">And these systems are really</a> <a data-t="622">tricky to use. For the uninitiated, they're difficult to even access. And so it's tempted to keep it running</a> <a data-t="647">because what if you need to submit work? <p><a data-t="668">crew is designed to navigate these kinds of tricky waters. So the solution to the</a> <a data-t="676">cost overhead trade-off that crew provides is autoscaling. crew plugs into high-performance</a> <a data-t="683">computing environments through the crew cluster package or the crew AWS Batch package or other</a> <a data-t="691">plugins to make it easier to access these systems. And thanks to mirai, which does by far</a> <a data-t="699">the most difficult part of all of this, when we manage those tasks, the overhead is really low.</a></p>

<h3>How crew works</h3>

<p><a data-t="706">So how does all this work? So in crew, there's this thing called a controller, which is an</a> <a data-t="712">interface that sits locally. And then there are workers, and each worker is an R process.</a> <a data-t="719">And these R processes could live on different computers on the local network, and they do one</a> <a data-t="725">or more tasks. Now, these workers need to get the instructions somehow. They need to know what to do,</a> <a data-t="732">and they need to be able to send results back. So there's got to be communication.</a> <a data-t="739">And this is exactly the challenge that mirai automatically solves.</a></p>

<p><a data-t="756">So the controller calls a function in mirai that says, here are a bunch of WebSockets on</a> <a data-t="762">the local network. All you workers out there, come find me. And then each worker dials into</a> <a data-t="769">one of those WebSockets with another function in mirai, and each worker accepts instructions,</a> <a data-t="755">does tasks, sends the results back. And so this way, mirai doesn't care where the workers are,</a> <a data-t="762">but it manages the tasks, and crew gets the workers up and running.</a></p>

<h3>Autoscaling</h3>

<p><a data-t="784">And the degree that mirai allows crew to operate to accomplish this allows flexibility</a> <a data-t="790">for autoscaling. And to demonstrate, let's take this pipeline for simulation. This is a</a> <a data-t="796">directed acyclic graph like the one Shatish mentioned in the previous presentation.</a> <a data-t="810">And each of these tasks, from simulated datasets to models, is kind of like a unit of demand</a> <a data-t="819">for a worker. So if you have tasks to do, you require workers to do them. When crew sees a</a> <a data-t="825">pipeline like this, one of the first things it does is it spins up the number of workers</a> <a data-t="830">required to accomplish the tasks that can be done. And so a worker starts. And if the task</a> <a data-t="837">load increases in the pipeline, if you can fan out and accomplish parallel tasks simultaneously,</a> <a data-t="844">then crew will increase the number of workers to meet that demand.</a></p>

<p><a data-t="848">When you proceed along the pipeline, crew reuses the workers that may have been running previously,</a> <a data-t="856">but are just finishing up tasks. And that way you avoid the overhead of requesting</a> <a data-t="861">brand new resources. And when the workload subsides, demand decreases, some workers may</a> <a data-t="868">be sleeping on the job, doing nothing. And so to avoid the extra cost, we evict them from the team.</a></p>

<p><a data-t="879">So this fluctuation in demand, it can happen any number of times over the course of a pipeline.</a> <a data-t="885">And crew will scale up and scale down automatically.</a> <a data-t="889">And you can control that with settings when you create the controller. How many seconds do you</a> <a data-t="895">wait here? How many idle seconds do you wait before you terminate a worker? How many tasks</a> <a data-t="902">can a worker run before it retires? And this allows us a continuum between perfectly transient</a> <a data-t="909">workers that only run one task and exit, and fully persistent workers, on the other hand,</a> <a data-t="915">that keep running until there's no more work to do. This continuum between transient persistent</a> <a data-t="921">workers is something we've never had in R before. And with crew, you can hit any point in the middle.</a></p>

<blockquote><a data-t="921">This continuum between transient persistent workers is something we've never had in R before. And with crew, you can hit any point in the middle.</a></blockquote>

<h3>Managing tasks and plugins</h3>
