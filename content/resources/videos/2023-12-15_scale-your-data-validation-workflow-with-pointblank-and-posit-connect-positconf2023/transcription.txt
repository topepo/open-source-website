<h3>Introducing pointblank</h3>

<p><a data-t="236">pointblank allows you to validate your data using these simple yet powerful validation functions.</a> <a data-t="242">For example, if you wanted to test if a set of values in a column were less than or equal to another value or set of values in another column, there's a function for that.</a> <a data-t="251">The way that it works is you create what is known as an agent object.</a> <a data-t="256">You pass to it a single target table that you want to validate and some other data, like how you want to be alerted if a test case would fail.</a> <a data-t="262">You then pass this object to these validation functions, and these act as sort of instructions for what your agent should do when it's ready to validate.</a> <a data-t="272">And then when you are ready to validate, you call the interrogate function.</a> <a data-t="276">So in essence, you're interrogating the agent for the information that you want.</a> <a data-t="282">And the result is a really nice-looking, easy-to-digest HTML report.</a></p>

<p><a data-t="282">On the left, you have a nice color-coding scheme that tells you what passed, what failed, or if you hit another threshold that you set when creating your agent.</a> <a data-t="291">Each record represents a validation function that you set and the columns next to it that you apply them to.</a> <a data-t="298">And on the right, you have some descriptive statistics telling you how many cases were run for each function, how many passes and fails, and their proportions.</a> <a data-t="308">And at first glance, it was perfect. It was everything that we needed.</a> <a data-t="312">The way that it integrated with packages that we used already and technologies, and also its intuitive API and its source code was easy to digest.</a> <a data-t="322">However, once we actually started to implement it for our processes and data, we noticed there were four specific challenges that we had to address before we could do that.</a></p>

<h3>Challenge 1: operating on multiple tables</h3>

<p><a data-t="330">So, the first challenge is we wanted pointblank to operate on more than just a single target table.</a> <a data-t="341">Here's an example of a data model for our study, for a very simple study that we may see.</a> <a data-t="345">And if we wanted to test each individual table, we would need to create each individual agents and apply them to the tables.</a> <a data-t="350">And that's at the core of pointblank. And we didn't want to change that for our needs.</a> <a data-t="355">Instead, we wanted to bundle all of these agents together into a collection that we can then operate on that collection.</a></p>

<p><a data-t="366">And so, in typical R fashion, we wrapped everything into a list, and we applied some processing to it that's specific to our needs, and we called it a test plan.</a> <a data-t="371">And so, essentially, it's a list of agents with some extra goodies that I'll talk about in a little bit.</a> <a data-t="374">If you're familiar with pointblank, this is very much like a multi-agent report, and we borrowed a lot from that.</a> <a data-t="379">We're just extending upon it to perform operations on that actual multi-agent instead of it being just a reporting function.</a> <a data-t="389">So, in the background, this is a very simple concept, but we modified lots of pointblank and pins functions to actually do what we wanted to do with this.</a> <a data-t="399">And also, we integrated it with the DM package, which we use for our data modeling needs.</a></p>

<p><a data-t="405">So, when we want to create a test plan, it's very straightforward. You pass in your individual agents, and then you pass to it also a test name, which we will reference later on.</a> <a data-t="411">And some of the data that gets stored with the test plan are the name that we just passed through, as well as a test version that we get from pins and Connect.</a> <a data-t="425">Now, when we're ready to validate our collection of agents or our test plan, we have to run interrogate on each individual function, so each individual agent.</a> <a data-t="431">So, we basically map the interrogate function to all the agents, and we also force the user to specifically pass in the data that they're going to validate.</a> <a data-t="436">And the result is what's known as a multi-agent report.</a></p>

<h3>Challenge 2: sharing and reusability</h3>

<p><a data-t="477">The second challenge is we wanted to implement sharing and reusability of the test plans that our team is creating.</a> <a data-t="482">So, I showed a simple data model before, but in reality, they look more like this, and you can guess that if we're validating each table, the code, the pointblank code to generate all that can get pretty large and pretty unwieldy quickly.</a> <a data-t="501">So, one feedback that we heard from the team is, how can we reuse these test plans so that we're not recreating the wheel every single time we start a new study or want to iterate on an existing test plan?</a> <a data-t="511">So, copying-pasting is an option, but it's not really a practical one in a production setting.</a></p>

<p><a data-t="517">So, if we revisit our create test plan function again, we could do a little more in it.
