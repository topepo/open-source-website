<h3>Keeping the project going</h3>

<p><a data-t="719">Now, after the coding task is</a> <a data-t="719">all done, we can hand off the process and move on.</a> <a data-t="725">But there are still things we need to look</a> <a data-t="725">out for in order to keep the project going.</a> <a data-t="731">Overall, there are three deadly situations</a> <a data-t="731">that could affect your project.</a></p>

<p><a data-t="738">The first situation is that after you handed off your</a> <a data-t="738">automated processes, someone ran the code.</a> <a data-t="744">But the result isn't what's expected.</a> <a data-t="745">Maybe because</a> <a data-t="745">the script wasn't handled properly, like running it on a new machine or a new environment without</a> <a data-t="751">the proper setup.</a> <a data-t="751">Believe it or not, after all these years, it still happens to me.</a></p>

<p><a data-t="757">So, to avoid this situation, what I would recommend is to always have a handout document</a> <a data-t="762">that discusses the requirements of the job, instructions of running the process,</a> <a data-t="766">testing the process, and what to do to maintain.</a> <a data-t="773">The second scenario is that as you build more and</a> <a data-t="773">more automated processes, chances are multiple processes may run into issues at the same time.</a> <a data-t="780">And sometimes someone will come up and need your help to fix it.</a> <a data-t="786">Sometimes multiple people will</a> <a data-t="786">show up, which could look pressing.</a> <a data-t="793">So, my tip for you is this.</a> <a data-t="793">Always discuss the fail safe</a> <a data-t="793">in your handoff document.</a> <a data-t="800">In most cases, a line like this will do the trick.</a> <a data-t="800">This is almost like</a> <a data-t="800">when the autopilot stopped working.</a> <a data-t="804">You don't stop flying or driving.</a> <a data-t="804">You switch back to manual.</a> <a data-t="805">And this often reduces the urgency and gives us the time to properly fix the issue.</a></p>

<p><a data-t="812">Lastly, everything went well.</a> <a data-t="812">You just got a new feature request.</a> <a data-t="812">Should you directly jump in?</a> <a data-t="820">And my recommendation here, well, two recommendations.</a> <a data-t="824">The short term is that</a> <a data-t="824">always treat it like a brand new task so that you can start from gathering requirements and</a> <a data-t="830">best determine when to work on it.</a> <a data-t="836">And then the longer term solution here is that it is also a</a> <a data-t="836">perfect opportunity for you to train your individual team members to gradually take on the task and</a> <a data-t="841">accumulate more knowledge.</a></p>
