<p><a data-t="0">Hi, my name is Emily Riederer, and today I'm here to talk about how to design packages</a> <a data-t="11">for your organization.</a> <a data-t="13">Think about the last time that you joined a new organization.</a> <a data-t="16">There was so much you needed to learn before you could start contributing.</a> <a data-t="20">Remember the frustration when you couldn't figure out how to access data, the lost hours</a> <a data-t="25">trying to answer the wrong question before you built up your intuition, and the awkwardness</a> <a data-t="29">of figuring out team norms through trial and error.</a></p>

<p><a data-t="33">When we join a new organization, we must learn these things only once before we can hit the</a> <a data-t="38">ground running.</a> <a data-t="39">However, the off-the-shelves tools that we use can't preserve this context.</a> <a data-t="45">Every day is like their first day at work.</a> <a data-t="48">Unlike open-source packages, internal packages can embrace institutional knowledge more like</a> <a data-t="53">a colleague in two important ways.</a> <a data-t="56">First and foremost, they can aim to solve far more domain-specific and concrete problems</a> <a data-t="61">than open-source packages.</a> <a data-t="62">Second, although narrower is the scope of the defined problem, their insight into our</a> <a data-t="68">organization allows them to be more holistic in the solutions they propose by internalizing</a> <a data-t="73">more steps of the necessary workflow to answer that question, covering areas like pulling</a> <a data-t="78">data and communicating with colleagues.</a></p>

<p><a data-t="81">Because of these two factors, internal packages can add a lot of value by interfacing internal</a> <a data-t="87">utilities, streamlining the analysis process, and empowering data scientists to create high-quality</a> <a data-t="93">results.</a> <a data-t="94">But, more than the tasks they can do, today I want to talk about how to design such internal</a> <a data-t="100">packages to be as rich in context as possible.</a></p>

<p><a data-t="104">To do this, I like to think about the popular jobs-to-be-done framework for product development.</a> <a data-t="110">This asserts that we hire a product to do a job that helps us make progress towards</a> <a data-t="115">the goal.</a> <a data-t="115">And, to me, it's that notion of progress, and truly knowing the ins and outs of what</a> <a data-t="121">sort of progress our users need to make that sets internal packages apart.</a> <a data-t="126">Additionally, these jobs can have functional, social, and emotional aspects.</a></p>

<p><a data-t="131">For the rest of this discussion, I'll tweak this framework just slightly.</a> <a data-t="137">Let's build a team of packages to do the jobs that help our org answer impactful questions</a> <a data-t="143">with efficient workflows.</a> <a data-t="145">So what sort of teammates should our packages be?</a> <a data-t="148">Let's meet a few.</a></p>

<h3>The IT guy: abstracting infrastructure</h3>

<p><a data-t="149">First, I'd like to introduce you to the IT guy.</a> <a data-t="153">Think of a helpful or friendly IT or DevOps colleague that you may know.</a> <a data-t="157">They're fantastic at handling the quirks of infrastructure and abstracting away all the</a> <a data-t="162">sorts of things that data scientists don't want to spend their time thinking about, like</a> <a data-t="165">proxies, connections, servers, etc.</a> <a data-t="169">In that abstraction process, they also take on additional responsibilities to promote</a> <a data-t="174">good practices like data security and credential management.</a> <a data-t="178">Ideally, they can save us a lot of time and frustration in navigating organization-specific</a> <a data-t="183">roadblocks that no amount of searching on Stack Overflow can help.</a></p>

<p><a data-t="188">So how can we encode these characteristics in a package?</a> <a data-t="192">Writing utility functions can help achieve that same type of abstraction.</a> <a data-t="196">And in these functions, we can take an opinionated stance and craft helpful error messages.</a> <a data-t="202">Let's take a look at an example.</a> <a data-t="204">Suppose we want to write a function to connect ourselves to a database.</a> <a data-t="209">First, we might start out with a rather boilerplate piece of code using the </a><a href="/software/dbi/">DBI</a><a data-t="209"> package.</a> <a data-t="215">We take in the username and password, hardcode the driver name, server location, port, and</a> <a data-t="220">return the connection object.</a></p>

<p><a data-t="223">Now let's suppose our organization has strict rules against putting secure credentials in</a> <a data-t="228">plain text, as well they should.</a> <a data-t="231">In an open-source package, I wouldn't presume to force users' hands to use one specific</a> <a data-t="236">system setup.</a> <a data-t="237">However, in this case, we can make strong assumptions based on our knowledge of an organization's</a> <a data-t="243">rules and norms.</a> <a data-t="245">And this sort of function can be great leverage to incentivize users to do the right thing,</a> <a data-t="250">like storing their credentials in environment variables, because it's the only way they</a> <a data-t="254">can get the function to work.</a></p>

<p><a data-t="257">Of course, that's only helpful if we provide descriptive error messages when the user does</a> <a data-t="261">not have their credentials set up this way.</a> <a data-t="264">Otherwise, they'll get an error that dbPaths is missing, find nothing useful online to</a> <a data-t="269">help them troubleshoot, since this is an internal choice that we made.</a> <a data-t="273">So we can enhance this function with an extremely custom and prescriptive error message explaining</a> <a data-t="278">what went wrong, and either how to fix it, or where we can get more information.</a></p>

<blockquote><a data-t="257">Of course, that's only helpful if we provide descriptive error messages when the user does</a> <a data-t="261">not have their credentials set up this way.</a> <a data-t="264">Otherwise, they'll get an error that dbPaths is missing, find nothing useful online to</a> <a data-t="269">help them troubleshoot, since this is an internal choice that we made.</a></blockquote>

<p><a data-t="285">Of course, even better than explaining errors is preventing them from occurring at all.</a> <a data-t="290">We might also know at our specific organization that non-alphanumeric characters are required</a> <a data-t="296">in passwords, and that dbConnect doesn't natively encode these correctly when passing them to</a> <a data-t="301">the database.</a> <a data-t="302">Instead of troubling the user and telling them how to pick their password, we can instead</a> <a data-t="307">handle this silently inside of the function.</a></p>

<h3>The junior analyst: proactive yet responsive functions</h3>

<p><a data-t="316">However, strong opinions and complete independence don't always make a great colleague.</a> <a data-t="320">Other times, we might want someone more like a junior analyst.</a> <a data-t="325">They know a good bit about the organization, and we can trust them to execute calculations</a> <a data-t="328">correctly and make reasonable assumptions.</a> <a data-t="328">At the same time, we want them to be responsive to our feedback and willing to try things</a> <a data-t="333">out in more ways than one.</a></p>

<p><a data-t="335">So how can we capture these jobs in an internal package?</a> <a data-t="339">We can build in proactive yet responsive functions with default arguments, reserved keywords,</a> <a data-t="345">and the </a><a href="/software/ellipsis/">ellipsis</a><a data-t="345">.</a> <a data-t="346">To illustrate, imagine a basic visualization function that wraps </a><a href="/software/ggplot2/">ggplot2</a><a data-t="346"> code but allows</a> <a data-t="353">users to input their preferred x, y, and grouping variables to draw cohort groups.</a> <a data-t="360">This function is fine, but we can probably draw on institutional knowledge to push our</a> <a data-t="365">junior analyst to be a little bit more proactive.</a></p>

<p><a data-t="369">If we relied on the same opinionated design as the IT guy, we might consider hardcoding</a> <a data-t="374">some of the variables inside of the function.</a> <a data-t="379">Here though, that isn't a great approach.</a> <a data-t="382">It isn't the junior analyst's job to give edicts.</a> <a data-t="382">We might know what the desired x-axis will be 80% of the time, but hardcoding here is</a> <a data-t="387">too strong of an assumption and decreases the usefulness of the function for that other</a> <a data-t="392">20%.</a> <a data-t="393">Instead, we can put our best guess 80% right names as the default arguments in the function</a> <a data-t="399">header, ordered by decreasing likelihood of the need to override.</a> <a data-t="403">This means when users do not provide their own value, a default is used.</a> <a data-t="407">That's the junior analyst's best guess, but users retain complete control to change</a> <a data-t="413">it as they see fit.</a></p>

<p><a data-t="416">This approach becomes even more powerful if we can abstract out a small set of incredibly</a> <a data-t="421">common, frequently occurring, assumed variable names or other values.</a> <a data-t="427">We can define and document this set of keywords or special variable names that span all of</a> <a data-t="432">our internal packages.</a> <a data-t="434">If these are well-known and well-documented, users will get into the habit of shaping their</a> <a data-t="438">data so it plays nice with the package ecosystem and saves a lot of manual typing.</a> <a data-t="444">This type of incentive to standardize file names can have other convenient consequences</a> <a data-t="449">in making data extracts more shareable and code more readable.</a></p>

<p><a data-t="454">Finally, one other nice trick in making our functions responsive to feedback is the ellipsis,</a> <a data-t="459">or passing the dots.</a> <a data-t="461">This allows users to provide any number of arbitrary additional arguments beyond what</a> <a data-t="466">was specified by the developer, and plugs them in at a designated place in the function</a> <a data-t="471">body.</a> <a data-t="473">This way, users can extend functions based on needs that the developer could not have</a> <a data-t="478">anticipated, like customizing the color, size, and line type.</a></p>

<h3>The tech lead: vignettes and templates</h3>

<p><a data-t="484">So far, we've mostly focused on that first dimension, making our package teammates targeted</a> <a data-t="490">to solving specific internal problems.</a> <a data-t="493">But there's just as much value in that second dimension.</a> <a data-t="496">Using internal packages as a way to ship not just calculations, but workflows, and to share</a> <a data-t="501">an understanding of how the broader organization operates.</a> <a data-t="505">To illustrate this, consider our intrepid tech lead.</a> <a data-t="509">We value this type of teammate because they can draw from a breadth of past experience</a> <a data-t="513">and institutional knowledge to help you weigh tradeoffs, learn from collected wisdom, and</a> <a data-t="518">inspire you to do your best work.</a></p>

<p><a data-t="521">So that's a pretty tall order to put in a package, but conscientious use of vignettes</a> <a data-t="526">and templates can help us towards this goal.</a> <a data-t="530">Vignettes often help introduce basic functionality of a package with a toy example, as found</a> <a data-t="535">in the declier vignette.</a> <a data-t="537">Or less commonly, they may discuss a statistical method that's implemented, as done in the</a> <a data-t="542">survival package.</a> <a data-t="543">Vignettes of an internal R package can do more diverse and numerous jobs.</a></p>

<p><a data-t="549">These vignettes can accumulate the hard-won experience and domain knowledge like an experienced</a> <a data-t="555">tech lead's photographic memory, and they hand-deliver these insights to anyone currently</a> <a data-t="560">working on a related analysis.</a> <a data-t="563">Just as a few examples, you can consider having completely code-free vignettes that conceptually</a> <a data-t="570">introduce you to a common problem that your package solves, explain the workflow and key</a> <a data-t="575">questions you should be asking yourself along the way, and even potentially get into procedural</a> <a data-t="581">weeds like what you need to do to add features or deploy a model.</a></p>

<p><a data-t="586">Then after aligning with your users on a conceptual framework, you may introduce the package's</a> <a data-t="591">functionality and explain how those two overlap.</a> <a data-t="595">When your package contains functions for many different ways to do a task, you can also</a> <a data-t="599">compare pros and cons and explain different situations where different ones have proven</a> <a data-t="603">more or less effective.</a> <a data-t="605">Finally, vignettes can also include lessons learned, reflections from past users, and</a> <a data-t="610">references to past examples to help analysts learn about similar projects.</a></p>

<p><a data-t="616">In fact, all of that context may be so helpful even to people that are not direct users of</a> <a data-t="621">your package, they may want to seek this mentorship.</a> <a data-t="625">In this case, you can use the Packagedown package to automatically create a package</a> <a data-t="629">website to share these vignettes with anyone who needs to learn more about a specific problem</a> <a data-t="634">space.</a> <a data-t="635">With a single function call, your package can share its wisdom more broadly.</a> <a data-t="640">And, unlike their human counterparts, the package tech lead can always find time for</a> <a data-t="645">another meeting.</a></p>

<p><a data-t="648">Similar to vignettes, embedded templates take on a more important and distinctive role for</a> <a data-t="652">internal packages.</a> <a data-t="654">In open-source packages, our Markdown templates provide a pre-populated file instead of the</a> <a data-t="660">default.</a> <a data-t="661">This is most commonly used to demonstrate proper formatting syntax.</a> <a data-t="664">For example, the Flux Dashboard package uses a template to show users how to set up the</a> <a data-t="670">YAML metadata and section headers.</a></p>

<p><a data-t="673">Instead, internal packages can use templates to coach users through workflows because they</a> <a data-t="679">understand the problems users are facing and the progress they hope to achieve.</a> <a data-t="684">Internal packages can mentor users and structure their work in two different ways.</a> <a data-t="689">Package walkthroughs can serve as interactive notebooks that coach users through common</a> <a data-t="695">analyses.</a> <a data-t="695">As an example, if a type of analysis requires manual data cleaning and curation, a template</a> <a data-t="702">notebook can guide users to ask the right questions of their data and generate common</a> <a data-t="706">views they need to analyze.</a></p>

<p><a data-t="709">We can also include full end-to-end analysis outlines which include placeholder text, commentary,</a> <a data-t="715">and code if the type of analysis that a package supports usually results in a specific type</a> <a data-t="720">of report.</a> <a data-t="722">Similarly, our package can include a project template, not just in our Markdown template.</a> <a data-t="728">These templates can pre-define a standard file structure and a boilerplate set of files</a> <a data-t="733">for a new project to give users a helping hand and drive the kind of consistency across</a> <a data-t="738">projects that any tech lead dreams of when doing a code review.</a></p>

<h3>The project manager: modularizing workflows</h3>

<p><a data-t="743">Speaking of collaboration, brings us to the last teammate whose traits we want our package</a> <a data-t="748">to evoke, the project manager.</a> <a data-t="750">One of the biggest differences between task-doing versus problem-solving packages is understanding</a> <a data-t="756">the whole workflow and helping coordinate the project across different components.</a> <a data-t="762">When writing open-source packages, we rightly assume that our intended audience is our users,</a> <a data-t="767">but on a true cross-functional internal team, not everyone will be, so we can intentionally</a> <a data-t="773">modularize the workflow and think about how to augment </a><a href="/software/rstudio/">RStudio</a><a data-t="773">'s IDE to make sure our</a> <a data-t="779">tools work well with all of our colleagues.</a></p>

<p><a data-t="783">One way to do this is with modularizing the parts of the workflow that really do or do</a> <a data-t="788">not require R code.</a> <a data-t="790">For example, in the templates we just discussed, we could actually make separate template components</a> <a data-t="795">for the parts that require R code and for the parts that are just English text and commentary.</a> <a data-t="802">The commentary files can be plain vanilla markdown files that any collaborator could</a> <a data-t="807">edit without even having to have R installed, and the main R markdown can pull in this plain</a> <a data-t="812">text output using child documents.</a></p>

<p><a data-t="816">This approach is now made even easier with advances in the RStudio IDE.</a> <a data-t="820">The visual markdown editor provides a great graphical user interface to support word processing</a> <a data-t="826">in markdown for those plain text narrative documents that we just discussed.</a> <a data-t="831">We can also use RStudio add-ins to extend the RStudio interface and ship interactive</a> <a data-t="837">widgets in our internal packages.</a> <a data-t="839">To illustrate, I've shown the point-and-click plot coding assistant from the open-source</a> <a data-t="844">Esquisse package.</a> <a data-t="846">Add-ins require more investment up front, but they're much easier to maintain than a</a> <a data-t="850">full application, and they can help convert teammates to R users.</a> <a data-t="855">Besides, a good project manager is willing to go that extra mile to support their team.</a></p>

<h3>Building a coherent package ecosystem</h3>

<p><a data-t="861">Now speaking of collaboration, we've talked about what can make an individual package</a> <a data-t="867">a good teammate.</a> <a data-t="869">Another major opportunity when building a suite of internal tools is that we can think</a> <a data-t="873">about how multiple packages on our team can best work together.</a> <a data-t="878">We want teammates that are clear communicators, have defined responsibilities, and keep their</a> <a data-t="884">promises.</a> <a data-t="886">We can help packages be these sorts of good teammates with naming conventions, clearly</a> <a data-t="891">defined scopes, and careful attention to dependencies in testing.</a></p>

<p><a data-t="898">Clear function naming conventions and consistent method signatures help packages effectively</a> <a data-t="903">communicate with both package and human collaborators.</a> <a data-t="907">Internally, we can give our suite of internal packages a richer language by defining a consistent</a> <a data-t="914">set of prefixes for the namesteps that indicate how each function is to be used.</a> <a data-t="920">One approach I like is that each function prefix can denote the type of object that</a> <a data-t="924">that function will return.</a> <a data-t="926">For example, using viz prefixes for functions that return ggplot2 objects.</a> <a data-t="932">This way, past experience working with one internal package seamlessly translates to</a> <a data-t="937">the next.</a></p>

<p><a data-t="939">Another aspect of good team communication is having clearly defined roles and responsibilities.</a> <a data-t="945">Again, since we own our whole internal stack of packages, we have more freedom in how we</a> <a data-t="950">can choose to divide up functionality and responsibilities across packages.</a> <a data-t="956">Take for example the data science workflow as described in the R for Data Science book.</a> <a data-t="962">Open-source packages inevitably have overlapping functionality, which forces users to compare</a> <a data-t="967">alternatives and decide which is best.</a> <a data-t="970">Internally, we can use some amount of central planning to ensure each package teammate has</a> <a data-t="975">a clearly defined role, whether that be to provide a horizontal utility or to enable</a> <a data-t="980">progress on a specific workstream.</a> <a data-t="984">And just like one team ought to work well with another, that central planning can include</a> <a data-t="988">the curation and promotion of open-source tools along with our internal ones.</a> <a data-t="994">After all, no one team can do it all alone.</a></p>

<p><a data-t="998">When assigning these roles to our team of packages, we should consider how to manage</a> <a data-t="1003">the dependencies between them when different functionality needs to be shared across packages.</a> <a data-t="1010">Packages often have direct dependencies where a function in one package calls a function</a> <a data-t="1014">in another.</a> <a data-t="1015">This is not necessarily bad, but especially with internal packages, which might sometimes</a> <a data-t="1020">have a shorter shelf life and fewer developers, this can sometimes create a domino effect.</a> <a data-t="1026">If one package is deprecated or decides to retire or take a vacation, we don't want</a> <a data-t="1031">the rest of our ecosystem affected.</a></p>

<p><a data-t="1034">Alternatively, we can use the fact that both packages A and B are under our control to</a> <a data-t="1040">see if we can eliminate the explicit dependency by promoting a clean handoff.</a> <a data-t="1045">We can see if a function in A can produce an output that B can consume instead of calling</a> <a data-t="1050">A's function directly.</a> <a data-t="1052">Additionally, because we own the full stack, we may also consider if there are shared needs</a> <a data-t="1058">in A and B that should be extracted into a common building block package C. For instance,</a> <a data-t="1065">a set of common visualization function primitives.</a> <a data-t="1068">This way, we at least have a clear hierarchy of dependencies instead of a spider web and</a> <a data-t="1073">can identify a small number of truly essential ones.</a></p>

<p><a data-t="1077">Regardless of the type of dependencies we end up with, we can use tests to make sure</a> <a data-t="1082">our packages are reliable teammates who do what they promised.</a> <a data-t="1086">Typically, if I write package B that depends on package A, I can only control package B's</a> <a data-t="1092">tests, so I could write tests to see if A continues to perform as B is expecting.</a> <a data-t="1098">This is a good safeguard, but it means that we will only detect problems after they've</a> <a data-t="1103">already been introduced in A. There's nothing in place to actually stop package A from getting</a> <a data-t="1108">distracted in the first place.</a></p>

<p><a data-t="1110">Instead, we prefer that both A and B be conscientious of the promises they've made and stay committed</a> <a data-t="1117">to working towards their shared goal.</a> <a data-t="1120">We can formalize that shared vision with integration tests.</a> <a data-t="1123">That is, we can add tests to both the upstream and the downstream packages to ensure they</a> <a data-t="1129">continue to check in with each other and inquire if any of the changes they're planning could</a> <a data-t="1133">be disruptive.</a> <a data-t="1134">Now, just imagine having such rigorous and ongoing communication and prioritization with</a> <a data-t="1139">your own teammates.</a></p>

<blockquote><a data-t="1110">Instead, we prefer that both A and B be conscientious of the promises they've made and stay committed</a> <a data-t="1117">to working towards their shared goal.</a> <a data-t="1120">We can formalize that shared vision with integration tests.</a></blockquote>

<p><a data-t="1142">In summary, we all know the joy of working with a great team.</a> <a data-t="1146">And given that you're here right now, I suspect you know the pleasure of cracking</a> <a data-t="1150">open a new R package.</a> <a data-t="1152">By taking advantage of the unique opportunities of designing internal packages, we can achieve</a> <a data-t="1157">the best of both worlds.</a> <a data-t="1159">We can share the fun of working with good tools with the teammates we care about.</a> <a data-t="1164">And we can elevate those tools to full-fledged teammates by giving them the skills they need</a> <a data-t="1169">to succeed.</a> <a data-t="1170">Thank you.</a></p>
