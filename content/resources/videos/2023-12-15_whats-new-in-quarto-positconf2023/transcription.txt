<h3>Quarto versions overview</h3>

<h3>New content features</h3>

<p><a data-t="135">The first feature I want to show you is the embed shortcode.</a> <a data-t="140">Imagine you've got a Quarto document and you've imported some data, done some processing,</a> <a data-t="145">done some exploration and you've landed at a visualization that you really want to include</a> <a data-t="150">in another document.</a> <a data-t="152">I'll call this document with all the code that gets to the visualization, the source</a> <a data-t="156">document.</a> <a data-t="157">What you want to end up with is another document, I'll call it the destination, with just this</a> <a data-t="162">visualization in it.</a></p>

<p><a data-t="166">The embed shortcode lets you do that with one line.</a> <a data-t="171">Here's that source document and the code chunk that generates the visualization.</a> <a data-t="177">We'll need a way to identify the cell, so let's add a label.</a> <a data-t="181">I'll call it fig-scatter.</a> <a data-t="185">Then in the destination document, I'll use the shortcode.</a> <a data-t="188">The embed shortcode requires a path to the source file, followed by a cell identifier.</a> <a data-t="195">That's it.</a></p>

<p><a data-t="197">The cell options come from the source document, so if you want to add a caption, you'll need</a> <a data-t="203">to do that in the source document.</a> <a data-t="206">When rendered, the output from that source document will be included in the destination</a> <a data-t="211">document.</a> <a data-t="212">In HTML formats, you also get a link back to a rendered version of that source document.</a> <a data-t="219">The embed shortcode was introduced in Quarto 1.3, but only when the source was a Jupyter</a> <a data-t="224">notebook.</a> <a data-t="225">In Quarto 1.4, you'll be able to do embeds from QMD documents too.</a></p>

<h3>Cross-references in Quarto 1.4</h3>

<p><a data-t="234">Quarto 1.4 is bringing some big changes to cross-references.</a> <a data-t="239">In particular, you'll be able to make almost anything a cross-referenceable element.</a> <a data-t="244">All the current cross-reference syntax will still work, but Quarto 1.4 adds some new syntax</a> <a data-t="250">that's more flexible.</a></p>

<p><a data-t="253">To define a cross-referenceable element, you'll need to create a div with an ID that has a</a> <a data-t="259">prefix that Quarto recognizes as a cross-reference.</a> <a data-t="264">So for instance, here, the triple colons indicate the div, followed by an ID starting</a> <a data-t="270">with fig, that lets Quarto know that this is a figure cross-reference.</a> <a data-t="275">Then comes the content for this element, and then the final line will be interpreted as</a> <a data-t="280">the caption.</a> <a data-t="282">Then I can refer back to this element using the usual syntax, an at symbol followed by</a> <a data-t="287">the ID of the element.</a> <a data-t="290">When rendered, Quarto adds figure, along with a counter, and generates the text and link</a> <a data-t="296">wherever I refer back to this element.</a></p>

<p><a data-t="300">With this new syntax, I can put almost anything inside this div.</a> <a data-t="304">I could put a code cell inside and call it a figure, or I could put an image inside,</a> <a data-t="309">and by changing the prefix to tbl, call it a table.</a></p>

<p><a data-t="314">The other new thing you'll be able to do is define custom types of cross-reference in</a> <a data-t="318">your YAML.</a> <a data-t="321">The keynames here are still a work in progress, but essentially, you decide what the prefix</a> <a data-t="325">will be that will identify this as a cross-reference in the div ID, and the labels used in the</a> <a data-t="331">caption, and for the in-text references.</a> <a data-t="335">Here I'm defining one that will represent diagrams.</a> <a data-t="339">Then you can switch to this new type of cross-reference, and now you have control over the text used</a> <a data-t="345">to represent this reference.</a> <a data-t="349">Each type of cross-reference maintains its own counter, so defining a new type of cross-reference</a> <a data-t="354">also gives you a way to restart a counter.</a></p>

<h3>Inline execution for Jupyter</h3>

<p><a data-t="361">This next feature is specific to the Jupyter engine, and it adds a feature that Nitar users</a> <a data-t="366">have had for a long time, an easy way to get computed values inline.</a> <a data-t="372">So for example, imagine you had a document with a Python code cell that defined the variable</a> <a data-t="377">radius, and what you want in your output is a sentence that refers to the value in that</a> <a data-t="384">variable.</a> <a data-t="385">It used to be that you'd need a code cell, and inside that code cell, you would have</a> <a data-t="390">to generate the markdown for that entire sentence, and then feed it into the markdown function.</a></p>

<p><a data-t="398">Quarto 1.4 introduces a new syntax for inline execution, single backticks, followed by Python</a> <a data-t="405">and curly braces, followed by the code to be executed.</a> <a data-t="409">In this case, we're just grabbing the variable itself.</a> <a data-t="413">Or if you happen to use the Julia Jupyter kernel, you could swap this out for a Julia</a> <a data-t="418">example as well.</a></p>

<h3>Adding context to code</h3>

<p><a data-t="423">The final content features I want to talk about are really useful if you tend to write</a> <a data-t="428">about code, and you want to add additional context for your readers.</a> <a data-t="433">For example, imagine seeing this code in a rendered document.</a> <a data-t="437">It happens to be YAML, and it was produced with this markdown in its source.</a> <a data-t="442">I think the first piece of context people need when they're looking at code is where</a> <a data-t="447">this code is.</a> <a data-t="449">In this case, this is code you'd find in the _quarto.yaml file in a Quarto project.</a> <a data-t="456">You can add that by adding the filename attribute to the code cell, which adds a header to the</a> <a data-t="462">code cell in the rendered output.</a> <a data-t="464">Here, the code lives in an actual file, but the filename attribute doesn't have to be</a> <a data-t="469">a filename.</a> <a data-t="471">For instance, you might say, this code is written in the terminal, or this code is written</a> <a data-t="477">on the R console.</a></p>

<p><a data-t="480">The other feature for adding context to code is code annotation.</a> <a data-t="484">Code annotation lets you add comments about specific lines of code.</a> <a data-t="488">For HTML output, I really like the hover setting, where the annotation is only revealed on hover</a> <a data-t="494">so that it doesn't distract from the code.</a> <a data-t="498">Adding code annotation is a matter of adding specially formatted comments to the code itself,</a> <a data-t="503">along with an ordered list directly following the code.</a></p>

<h3>Project features</h3>

<p><a data-t="513">Because they involve many files, Quarto projects can sometimes be a little bit more complicated</a> <a data-t="518">to get started.</a> <a data-t="520">Create is a Quarto command introduced in 1.2 that makes it really easy to produce a minimal</a> <a data-t="526">working project.</a> <a data-t="528">Here I am on the terminal running quarto create.</a> <a data-t="533">This then runs through an interactive prompt, where I can select I want to create a project</a> <a data-t="539">of type website in the directory test-website.</a> <a data-t="544">Quarto then creates that directory and populates it with a minimal working set of files.</a></p>

<p><a data-t="551">This works in any terminal, but because I use the </a><a href="/software/rstudio/">RStudio</a><a data-t="551"> terminal, Quarto will create</a> <a data-t="556">an RStudio project file and open up that project for me.</a> <a data-t="560">Then I might build it just to see what I've got.</a> <a data-t="563">I love using the interactive prompt, but you can also specify all those options as part</a> <a data-t="568">of the command.</a> <a data-t="570">This is a really handy way to build small reproducible examples of projects, or to get</a> <a data-t="575">started with a project type that's new to you.</a></p>

<h3>Project profiles</h3>

<p><a data-t="581">The other project feature I want to show you are what's called project profiles.</a> <a data-t="587">Imagine you've got a project where you need variations of basically the same content,</a> <a data-t="592">but for different audiences or different venues.</a> <a data-t="595">So for example, maybe you want a version of a report to share with external stakeholders,</a> <a data-t="601">and another version for internal use that has a little bit more detail.</a></p>

<p><a data-t="606">Your project might start like this, a quarto file report.qmd, where I've just got a placeholder</a> <a data-t="613">for some content that everyone might see.</a> <a data-t="616">_quarto.yaml, with some project level information, like the project</a> <a data-t="621">title, and a location for the rendered output to go, in this case a folder called external.</a> <a data-t="629">So if I was to render this project, I'd get this report dot html in the folder external.</a> <a data-t="638">This represents rendering without a profile activated.</a></p>

<p><a data-t="642">To add a profile, you add another quarto configuration file, but with an extra suffix.</a> <a data-t="648">I'm going to call mine _quarto-internal.yaml.</a> <a data-t="654">To activate the profile, you can add the profile option to the render command.</a> <a data-t="660">So here I'm running quarto render, with profile set to internal.</a> <a data-t="666">When activated, the options in the profile configuration file are merged with those in</a> <a data-t="671">the default configuration file.</a> <a data-t="673">I'm going to add a different output directory to my internal profile.</a> <a data-t="678">Let me say internal.</a> <a data-t="680">Then, when I render with this profile activated, I get this report dot html in the internal</a> <a data-t="686">folder.</a></p>

<p><a data-t="687">Not very exciting yet, but I can use this profile to start making changes that distinguish</a> <a data-t="693">the internal output from my default output.</a> <a data-t="697">So just to visually make a distinction, I could add a theme to my internal profile.</a> <a data-t="705">In addition to having a profile apply different options, you can also use it to control content.</a> <a data-t="713">Let's take another look at report dot qmd.</a> <a data-t="715">I'm going to add some content that I want only visible in the internal report.</a> <a data-t="721">To have it show up only when the internal profile is activated, I can put it inside</a> <a data-t="725">a div and add the content visible class with the when profile attribute set to internal.</a></p>

<p><a data-t="735">Now if I render both with and without the internal profile activated, I get these two</a> <a data-t="740">versions.</a> <a data-t="742">Report dot html in the external folder has only one section, and report dot html in the</a> <a data-t="747">internal folder has two.</a></p>

<h3>Authoring tools: VS Code and JupyterLab</h3>

<p><a data-t="754">The biggest changes in authoring tools are to our extensions for Visual Studio Code and</a> <a data-t="760">JupyterLab.</a> <a data-t="761">The Visual Studio Code extension now has a visual editor, just like RStudio.</a> <a data-t="768">When you enter the visual editor, you get a what-you-see-is-what-you-get editing experience</a> <a data-t="772">for your Quarto documents.</a> <a data-t="775">For example, rather than seeing markdown syntax, you see formatted text.</a></p>

<p><a data-t="779">One of the nicest features of the visual editor is the insert anything tool.</a> <a data-t="785">Simply type a forward slash, and you'll get a dropdown of all the content types you might</a> <a data-t="789">insert.</a> <a data-t="791">A nice example is insert table.</a> <a data-t="794">And if you only need one reason to use the visual editor, it should be for editing tables</a> <a data-t="798">without having to worry about markdown syntax.</a></p>
