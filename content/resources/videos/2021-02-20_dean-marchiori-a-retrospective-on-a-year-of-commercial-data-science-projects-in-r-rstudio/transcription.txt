<p><a data-t="0">G'day, I'm Dean Marchiori.</a> <a data-t="0">I'm a statistician from Sydney, Australia, and I'm interested</a> <a data-t="9">in reproducible analysis and workflow choices in R.</a> <a data-t="9">But in my day-to-day practice in industry</a> <a data-t="15">as a commercial data scientist, is my work actually reproducible?</a> <a data-t="21">To answer this, I conducted</a> <a data-t="21">a reproducibility retrospective, or ReproRetro.</a> <a data-t="28">I went back through the last year of my commercial</a> <a data-t="28">data science projects, and rated each project across a number of dimensions that are valued</a> <a data-t="32">in my practice.</a></p>

<p><a data-t="37">You can pick aspects that suit your work, but the dimensions I picked</a> <a data-t="37">cover if an analysis were replicable, modular, auditable, automated, and collaborative.</a> <a data-t="44">So</a> <a data-t="44">what were the results?</a> <a data-t="51">Out of 55 projects, I scored an average of 3.8 out of 5.</a> <a data-t="51">This</a> <a data-t="51">is good, but not great.</a> <a data-t="58">So if I care so much about this, why wasn't my work rated higher?</a> <a data-t="61">Here are the lessons I took from my ReproRetro.</a></p>

<h3>Lesson one: not everything needs to be reproducible</h3>

<p><a data-t="61">Controversial opinion number one, not everything needs to be reproducible.</a> <a data-t="66">While all projects</a> <a data-t="66">benefited from some aspect of reproducibility, the most important consideration for me was,</a> <a data-t="72">how do you tell what needs to be reproducible?</a></p>

<p><a data-t="77">Enter my Fiji test.</a> <a data-t="77">So if you have a task that's sufficiently important, you may want</a> <a data-t="82">to consider the Fiji test.</a> <a data-t="88">Aussies like Fiji because it's only a few hours away, but if</a> <a data-t="88">you're lying on a beach in holidays in Fiji, just spare a thought for your colleagues back</a> <a data-t="92">in the office.</a> <a data-t="97">If they need to rerun some of your analysis, will they be able to find</a> <a data-t="97">it, understand your work, run it without too much fuss, and make reasonable enhancements</a> <a data-t="103">or small changes?</a> <a data-t="107">The last thing you want while you're sipping your pina colada is to</a> <a data-t="107">be disturbed by a weird phone call about some model that you trained.</a></p>

<blockquote><a data-t="107">The last thing you want while you're sipping your pina colada is to</a> <a data-t="107">be disturbed by a weird phone call about some model that you trained.</a></blockquote>

<p><a data-t="112">When I looked at the type of analytics task I completed, not everything needed to pass</a> <a data-t="116">the Fiji test.</a> <a data-t="123">A lot of work in industry is ad hoc and quite simple.</a> <a data-t="123">Commercial or a high</a> <a data-t="123">pressure environment, I think it's perfectly reasonable to have different standards on</a> <a data-t="127">workflow choices depending on the type of work and the return on investment that you</a> <a data-t="131">need.</a> <a data-t="138">So long as a reasonable standard is maintained and as long as your team is consistent</a> <a data-t="138">with these choices.</a> <a data-t="143">However, this may mean getting better at anticipating the full scope</a> <a data-t="143">of the project before rushing into solving a problem, which can be a challenge.</a> <a data-t="149">In addition,</a> <a data-t="149">it gives you the freedom to pick a more lightweight and flexible workflow choice if desired.</a></p>

<h3>Picking the right tools for the job</h3>

<p><a data-t="154">My next lesson was picking the right tools for the job.</a> <a data-t="158">You know, I've been searching</a> <a data-t="158">for the biggest and best workflow choice, but I found it paid to have options.</a> <a data-t="163">When</a> <a data-t="163">I examined my tooling and workflow options, the results of my repro retro varied.</a> <a data-t="169">Using</a> <a data-t="169">a basic directory structure template to organize my projects was okay all round, but quickly</a> <a data-t="174">got cumbersome.</a> <a data-t="180">Notebooks like R Markdown were nicely replicable, but tended to be a</a> <a data-t="180">bit of a dumping ground for code.</a> <a data-t="185">Drake, a more formal workflow system, carried a bit</a> <a data-t="185">more overhead and more admin to get used to, but overall it enforced much better standards,</a> <a data-t="191">particularly around writing functional and modular code.</a></p>

<p><a data-t="196">While any system can be made</a> <a data-t="196">arbitrarily sophisticated and do a good job, I found it more effective to have a few different</a> <a data-t="200">workflow choices available rather than try a one size fits all approach.</a> <a data-t="206">All workflow</a> <a data-t="206">choices are naturally optimized for certain applications, and I found it better to only</a> <a data-t="210">take on additional complexity when the job really needed it.</a></p>

<h3>Replication vs. reproduction</h3>

<p><a data-t="214">Next, I found in my work I had a tendency to mistake replication for reproduction, and</a> <a data-t="220">the same goes for automation.</a> <a data-t="226">Getting your code to rerun is only part of the solution,</a> <a data-t="226">so consider the factors that matter in your practice and be deliberate about what you</a> <a data-t="230">optimize for, and use that to guide your workflow choice.</a> <a data-t="236">And how do you do that?</a> <a data-t="236">Well, do a</a> <a data-t="237">repro-retro.</a> <a data-t="243">The process wasn't and isn't intended to be scientific or unbiased.</a> <a data-t="243">The</a> <a data-t="243">true value of a repro-retro is in the self-reflection and improvement.</a> <a data-t="249">So go and dig through your</a> <a data-t="249">old code and think about how you might do things better.</a></p>
